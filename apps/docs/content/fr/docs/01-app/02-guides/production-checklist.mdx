Avant de mettre votre application Next.js en production, il existe certaines optimisations et bonnes pratiques que vous devriez envisager d'implémenter pour offrir la meilleure expérience utilisateur, performance et sécurité.

Cette page fournit des bonnes pratiques que vous pouvez utiliser comme référence lors du [développement de votre application](#pendant-le-développement) et [avant la mise en production](#avant-la-mise-en-production), ainsi que les [optimisations automatiques de Next.js](#optimisations-automatiques) dont vous devriez être conscient.

## Optimisations automatiques

Ces optimisations Next.js sont activées par défaut et ne nécessitent aucune configuration :

<AppOnly>

- **[Composants Serveur (Server Components)](/docs/app/getting-started/server-and-client-components) :** Next.js utilise par défaut les Composants Serveur. Ces composants s'exécutent sur le serveur et ne nécessitent pas de JavaScript pour le rendu côté client. Ainsi, ils n'ont aucun impact sur la taille de vos bundles JavaScript côté client. Vous pouvez ensuite utiliser des [Composants Client (Client Components)](/docs/app/getting-started/server-and-client-components) si nécessaire pour l'interactivité.
- **[Découpage du code (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works) :** Les Composants Serveur permettent un découpage automatique du code par segments de route. Vous pouvez également envisager le [chargement différé (lazy loading)](/docs/app/guides/lazy-loading) des Composants Client et des bibliothèques tierces, lorsque cela est approprié.
- **[Préchargement (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching) :** Lorsqu'un lien vers une nouvelle route entre dans le champ de vision de l'utilisateur, Next.js précharge la route en arrière-plan. Cela rend la navigation vers de nouvelles routes quasi instantanée. Vous pouvez désactiver le préchargement si nécessaire.
- **[Rendu statique (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering) :** Next.js effectue un rendu statique des Composants Serveur et Client sur le serveur au moment du build et met en cache le résultat pour améliorer les performances de votre application. Vous pouvez opter pour un [Rendu dynamique (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) pour des routes spécifiques si nécessaire. {/* TODO: Mettre à jour lorsque PPR est stable */}
- **[Mise en cache (Caching)](/docs/app/deep-dive/caching) :** Next.js met en cache les requêtes de données, le résultat du rendu des Composants Serveur et Client, les assets statiques et plus encore, afin de réduire le nombre de requêtes réseau vers votre serveur, base de données et services backend. Vous pouvez désactiver la mise en cache si nécessaire.

</AppOnly>

<PagesOnly>

- **[Découpage du code (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts) :** Next.js découpe automatiquement votre code d'application par pages. Cela signifie que seul le code nécessaire pour la page actuelle est chargé lors de la navigation. Vous pouvez également envisager le [chargement différé (lazy loading)](/docs/pages/guides/lazy-loading) des bibliothèques tierces si nécessaire.
- **[Préchargement (Prefetching)](/docs/pages/api-reference/components/link#prefetch) :** Lorsqu'un lien vers une nouvelle route entre dans le champ de vision de l'utilisateur, Next.js précharge la route en arrière-plan. Cela rend la navigation vers de nouvelles routes quasi instantanée. Vous pouvez désactiver le préchargement si nécessaire.
- **[Optimisation statique automatique (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization) :** Next.js détermine automatiquement qu'une page est statique (peut être pré-rendue) si elle n'a pas de besoins en données bloquantes. Les pages optimisées peuvent être mises en cache et servies à l'utilisateur final depuis plusieurs emplacements CDN. Vous pouvez opter pour un [Rendu côté serveur (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props) si nécessaire.

</PagesOnly>

Ces paramètres par défaut visent à améliorer les performances de votre application et à réduire le coût et la quantité de données transférées à chaque requête réseau.

## Pendant le développement

Lors du développement de votre application, nous recommandons d'utiliser les fonctionnalités suivantes pour garantir les meilleures performances et expérience utilisateur :

### Routage et rendu

<AppOnly>

- **[Layouts](/docs/app/api-reference/file-conventions/layout) :** Utilisez les layouts pour partager l'interface utilisateur entre les pages et activer le [rendu partiel (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering) lors de la navigation.
- **[Composant `<Link>`](/docs/app/building-your-application/routing/linking-and-navigating#link-component) :** Utilisez le composant `<Link>` pour la [navigation côté client et le préchargement](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works).
- **[Gestion des erreurs (Error Handling)](/docs/app/getting-started/error-handling) :** Gérez élégamment les [erreurs globales (catch-all errors)](/docs/app/getting-started/error-handling) et les [erreurs 404](/docs/app/api-reference/file-conventions/not-found) en production en créant des pages d'erreur personnalisées.
- **[Composants Client et Serveur (Client and Server Components)](/docs/app/getting-started/server-and-client-components#examples) :** Suivez les modèles de composition recommandés pour les Composants Serveur et Client, et vérifiez le placement de vos [limites `"use client"`](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree) pour éviter d'augmenter inutilement la taille de votre bundle JavaScript côté client.
- **[APIs dynamiques (Dynamic APIs)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) :** Soyez conscient que les APIs dynamiques comme [`cookies`](/docs/app/api-reference/functions/cookies) et la prop [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional) feront basculer toute la route en [Rendu dynamique (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) (ou toute votre application si utilisées dans le [Layout racine (Root Layout)](/docs/app/api-reference/file-conventions/layout#root-layout)). Assurez-vous que l'utilisation des APIs dynamiques est intentionnelle et encapsulez-les dans des limites `<Suspense>` si nécessaire.

> **Bon à savoir :** Le [Pré-rendu partiel (Partial Prerendering) (expérimental)](/blog/next-14#partial-prerendering-preview) permettra à des parties d'une route d'être dynamiques sans basculer toute la route en rendu dynamique.

</AppOnly>

<PagesOnly>

- **[Composant `<Link>`](/docs/pages/building-your-application/routing/linking-and-navigating) :** Utilisez le composant `<Link>` pour la navigation côté client et le préchargement.
- **[Erreurs personnalisées (Custom Errors)](/docs/pages/building-your-application/routing/custom-error) :** Gérez élégamment les [erreurs 500](/docs/pages/building-your-application/routing/custom-error#500-page) et [404](/docs/pages/building-your-application/routing/custom-error#404-page)

</PagesOnly>

### Récupération et mise en cache des données

<AppOnly>

- **[Composants Serveur (Server Components)](/docs/app/getting-started/fetching-data) :** Tirez parti des avantages de la récupération de données côté serveur en utilisant les Composants Serveur.
- **[Gestionnaires de route (Route Handlers)](/docs/app/building-your-application/routing/route-handlers) :** Utilisez les gestionnaires de route pour accéder à vos ressources backend depuis les Composants Client. Mais n'appelez pas les gestionnaires de route depuis les Composants Serveur pour éviter une requête serveur supplémentaire.
- **[Streaming](/docs/app/building-your-application/routing/loading-ui-and-streaming) :** Utilisez l'interface de chargement (Loading UI) et React Suspense pour envoyer progressivement l'interface du serveur vers le client, et empêcher que toute la route ne soit bloquée pendant la récupération des données.
- **[Récupération parallèle des données (Parallel Data Fetching)](/docs/app/getting-started/fetching-data#parallel-data-fetching) :** Réduisez les cascades réseau en récupérant les données en parallèle si nécessaire. Envisagez également de [précharger les données (preloading data)](/docs/app/getting-started/fetching-data#preloading-data) si approprié.
- **[Mise en cache des données (Data Caching)](/docs/app/deep-dive/caching#data-cache) :** Vérifiez si vos requêtes de données sont mises en cache ou non, et activez la mise en cache si nécessaire. Assurez-vous que les requêtes n'utilisant pas `fetch` sont [mises en cache](/docs/app/api-reference/functions/unstable_cache).
- **[Images statiques (Static Images)](/docs/app/api-reference/file-conventions/public-folder) :** Utilisez le répertoire `public` pour mettre automatiquement en cache les assets statiques de votre application, par exemple les images.

</AppOnly>

<PagesOnly>

- **[Routes API (API Routes)](/docs/pages/building-your-application/routing/api-routes) :** Utilisez les gestionnaires de route pour accéder à vos ressources backend et empêcher l'exposition de secrets sensibles au client.
- **[Mise en cache des données (Data Caching)](/docs/pages/building-your-application/data-fetching/get-static-props) :** Vérifiez si vos requêtes de données sont mises en cache ou non, et activez la mise en cache si nécessaire. Assurez-vous que les requêtes n'utilisant pas `getStaticProps` sont mises en cache si approprié.
- **[Regénération statique incrémentielle (Incremental Static Regeneration)](/docs/pages/guides/incremental-static-regeneration) :** Utilisez la regénération statique incrémentielle pour mettre à jour les pages statiques après leur construction, sans reconstruire l'intégralité de votre site.
- **[Images statiques (Static Images)](/docs/pages/api-reference/file-conventions/public-folder) :** Utilisez le répertoire `public` pour mettre automatiquement en cache les assets statiques de votre application, par exemple les images.

</PagesOnly>

### Interface utilisateur et accessibilité

<AppOnly>

- **[Formulaires et validation (Forms and Validation)](/docs/app/guides/forms) :** Utilisez les Actions Serveur (Server Actions) pour gérer les soumissions de formulaires, la validation côté serveur et la gestion des erreurs.

</AppOnly>

- **[Module de polices (Font Module)](/docs/app/api-reference/components/font) :** Optimisez les polices en utilisant le module de polices, qui héberge automatiquement vos fichiers de police avec les autres assets statiques, supprime les requêtes réseau externes et réduit le [décalage de mise en page (layout shift)](https://web.dev/articles/cls).
- **[Composant `<Image>`](/docs/app/api-reference/components/image) :** Optimisez les images en utilisant le composant Image, qui optimise automatiquement les images, empêche le décalage de mise en page et les sert dans des formats modernes comme WebP.
- **[Composant `<Script>`](/docs/app/guides/scripts) :** Optimisez les scripts tiers en utilisant le composant Script, qui diffère automatiquement les scripts et les empêche de bloquer le thread principal.
- **[ESLint](/docs/architecture/accessibility#linting) :** Utilisez le plugin intégré `eslint-plugin-jsx-a11y` pour détecter les problèmes d'accessibilité dès le début.

### Sécurité

<AppOnly>

- **[Marquage (Tainting)](/docs/app/api-reference/config/next-config-js/taint) :** Empêchez l'exposition de données sensibles au client en marquant les objets de données et/ou des valeurs spécifiques.
- **[Actions Serveur (Server Actions)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization) :** Assurez-vous que les utilisateurs sont autorisés à appeler les Actions Serveur. Consultez les [bonnes pratiques de sécurité](/blog/security-nextjs-server-components-actions) recommandées.

</AppOnly>

- **[Variables d'environnement (Environment Variables)](/docs/app/guides/environment-variables) :** Assurez-vous que vos fichiers `.env.*` sont ajoutés à `.gitignore` et que seules les variables publiques sont préfixées par `NEXT_PUBLIC_`.
- **[Politique de sécurité du contenu (Content Security Policy)](/docs/app/guides/content-security-policy) :** Envisagez d'ajouter une politique de sécurité du contenu pour protéger votre application contre diverses menaces de sécurité telles que le cross-site scripting, le clickjacking et d'autres attaques par injection de code.

### Métadonnées et SEO

<AppOnly>

- **[API Métadonnées (Metadata API)](/docs/app/getting-started/metadata-and-og-images) :** Utilisez l'API Métadonnées pour améliorer le référencement (SEO) de votre application en ajoutant des titres de page, des descriptions et plus encore.
- **[Images Open Graph (OG)](/docs/app/api-reference/file-conventions/metadata/opengraph-image) :** Créez des images OG pour préparer votre application au partage sur les réseaux sociaux.
- **[Sitemaps](/docs/app/api-reference/functions/generate-sitemaps) et [Robots](/docs/app/api-reference/file-conventions/metadata/robots) :** Aidez les moteurs de recherche à explorer et indexer vos pages en générant des sitemaps et fichiers robots.

</AppOnly>

<PagesOnly>

- **[Composant `<Head>`](/docs/pages/api-reference/components/head) :** Utilisez le composant `next/head` pour ajouter des titres de page, des descriptions et plus encore.

</PagesOnly>

### Sécurité des types

- **TypeScript et [Plugin TS](/docs/app/api-reference/config/typescript) :** Utilisez TypeScript et le plugin TypeScript pour une meilleure sécurité des types et pour détecter les erreurs plus tôt.

## Avant la mise en production

Avant la mise en production, vous pouvez exécuter `next build` pour construire votre application localement et détecter les erreurs de build, puis exécuter `next start` pour mesurer les performances de votre application dans un environnement similaire à la production.

### Core Web Vitals

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse) :** Exécutez Lighthouse en mode navigation privée pour mieux comprendre comment vos utilisateurs vivront votre site et identifier les axes d'amélioration. Ceci est un test simulé et devrait être complété par l'analyse des données de terrain (comme les Core Web Vitals).

<AppOnly>

- **[Hook `useReportWebVitals`](/docs/app/api-reference/functions/use-report-web-vitals) :** Utilisez ce hook pour envoyer les données des [Core Web Vitals](https://web.dev/articles/vitals) à vos outils d'analyse.

</AppOnly>

### Analyse des bundles

Utilisez le plugin [`@next/bundle-analyzer`](/docs/app/guides/package-bundling#analyzing-javascript-bundles) pour analyser la taille de vos bundles JavaScript et identifier les gros modules et dépendances qui pourraient impacter les performances de votre application.

De plus, les outils suivants peuvent vous aider à comprendre l'impact de l'ajout de nouvelles dépendances à votre application :

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
