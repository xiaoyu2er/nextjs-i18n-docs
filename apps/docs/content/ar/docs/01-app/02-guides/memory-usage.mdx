مع نمو التطبيقات وزيادة ثراء الميزات، قد تطلب موارد أكثر عند التطوير محليًا أو إنشاء بنائات للإنتاج.

دعونا نستكشف بعض الاستراتيجيات والتقنيات لتحسين الذاكرة ومعالجة مشكلات الذاكرة الشائعة في Next.js.

## تقليل عدد التبعيات

التطبيقات التي تحتوي على عدد كبير من التبعيات ستستخدم ذاكرة أكثر.

يمكن لـ [محلل الحزم (Bundle Analyzer)](/docs/app/guides/package-bundling) مساعدتك في التحقيق في التبعيات الكبيرة في تطبيقك والتي قد يكون من الممكن إزالتها لتحسين الأداء واستخدام الذاكرة.

## جرب `experimental.webpackMemoryOptimizations`

بدءًا من الإصدار `v15.0.0`، يمكنك إضافة `experimental.webpackMemoryOptimizations: true` إلى ملف `next.config.js` الخاص بك لتغيير سلوك Webpack مما يقلل من الحد الأقصى لاستخدام الذاكرة ولكن قد يزيد أوقات التجميع قليلاً.

> **معلومة مفيدة**: هذه الميزة حالياً تجريبية لاختبارها على المزيد من المشاريع أولاً، ولكنها تعتبر منخفضة المخاطر.

## تشغيل `next build` مع `--experimental-debug-memory-usage`

بدءًا من الإصدار `14.2.0`، يمكنك تشغيل `next build --experimental-debug-memory-usage` لتشغيل البناء في وضع يقوم فيه Next.js بطباعة معلومات حول استخدام الذاكرة بشكل مستمر طوال عملية البناء، مثل استخدام الكومة (heap) وإحصائيات جمع القمامة (garbage collection). كما سيتم التقاط لقطات الكومة تلقائيًا عندما يقترب استخدام الذاكرة من الحد الم configured.

> **معلومة مفيدة**: هذه الميزة غير متوافقة مع خيار عامل بناء Webpack الذي يتم تمكينه تلقائيًا إلا إذا كان لديك تكوين Webpack مخصص.

## تسجيل ملف تعريف للكومة (heap profile)

للبحث عن مشكلات الذاكرة، يمكنك تسجيل ملف تعريف للكومة من Node.js وتحميله في Chrome DevTools لتحديد المصادر المحتملة لتسرب الذاكرة.

في طرفيتك، مرر علم `--heap-prof` إلى Node.js عند بدء بناء Next.js الخاص بك:

```sh
node --heap-prof node_modules/next/dist/bin/next build
```

في نهاية عملية البناء، سيتم إنشاء ملف `.heapprofile` بواسطة Node.js.

في Chrome DevTools، يمكنك فتح علامة تبويب الذاكرة (Memory) والنقر على زر "تحميل الملف الشخصي" (Load Profile) لتصور الملف.

## تحليل لقطة للكومة (heap snapshot)

يمكنك استخدام أداة مفتش (inspector) لتحليل استخدام الذاكرة للتطبيق.

عند تشغيل أمر `next build` أو `next dev`، أضف `NODE_OPTIONS=--inspect` إلى بداية الأمر. سيعرض هذا وكيل المفتش على المنفذ الافتراضي.
إذا كنت ترغب في التوقف قبل بدء أي كود للمستخدم، يمكنك تمرير `--inspect-brk` بدلاً من ذلك. أثناء تشغيل العملية، يمكنك استخدام أداة مثل Chrome DevTools للاتصال بمنفذ التصحيح لتسجيل وتحليل لقطة للكومة لمعرفة ما يتم الاحتفاظ به في الذاكرة.

بدءًا من الإصدار `14.2.0`، يمكنك أيضًا تشغيل `next build` مع علم `--experimental-debug-memory-usage` لتسهيل التقاط لقطات الكومة.

أثناء التشغيل في هذا الوضع، يمكنك إرسال إشارة `SIGUSR2` إلى العملية في أي وقت، وسوف تأخذ العملية لقطة للكومة.

سيتم حفظ لقطة الكومة في جذر مشروع تطبيق Next.js ويمكن تحميلها في أي محلل كومة، مثل Chrome DevTools، لمعرفة ما يتم الاحتفاظ به في الذاكرة. هذا الوضع غير متوافق بعد مع عمال بناء Webpack.

راجع [كيفية تسجيل وتحليل لقطات الكومة](https://developer.chrome.com/docs/devtools/memory-problems/heap-snapshots) لمزيد من المعلومات.

## عامل بناء Webpack (Webpack build worker)

يسمح عامل بناء Webpack بتشغيل تجميعات Webpack داخل عامل Node.js منفصل مما يقلل استخدام الذاكرة لتطبيقك أثناء عمليات البناء.

يتم تمكين هذا الخيار افتراضيًا إذا لم يكن لتطبيقك تكوين Webpack مخصص بدءًا من الإصدار `v14.1.0`.

إذا كنت تستخدم إصدارًا أقدم من Next.js أو لديك تكوين Webpack مخصص، يمكنك تمكين هذا الخيار عن طريق تعيين `experimental.webpackBuildWorker: true` داخل ملف `next.config.js` الخاص بك.

> **معلومة مفيدة**: قد لا تكون هذه الميزة متوافقة مع جميع ملحقات Webpack المخصصة.

## تعطيل ذاكرة التخزين المؤقت لـ Webpack

[ذاكرة التخزين المؤقت لـ Webpack](https://webpack.js.org/configuration/cache/) تحفظ وحدات Webpack التي تم إنشاؤها في الذاكرة و/أو على القرص لتحسين سرعة البناء. يمكن أن يساعد هذا في الأداء، ولكنه سيزيد أيضًا من استخدام الذاكرة لتطبيقك لتخزين البيانات المخزنة مؤقتًا.

يمكنك تعطيل هذا السلوك عن طريق إضافة [تكوين Webpack مخصص](/docs/app/api-reference/config/next-config-js/webpack) إلى تطبيقك:

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  webpack: (
    config,
    { buildId, dev, isServer, defaultLoaders, nextRuntime, webpack }
  ) => {
    if (config.cache && !dev) {
      config.cache = Object.freeze({
        type: 'memory',
      })
    }
    // مهم: إرجاع التكوين المعدل
    return config
  },
}

export default nextConfig
```

## تعطيل التحليل الثابت (static analysis)

قد يتطلب التحقق من الأنواع (Typechecking) والتدقيق (linting) الكثير من الذاكرة، خاصة في المشاريع الكبيرة.
ومع ذلك، فإن معظم المشاريع لديها منفذ تنفيذ CI مخصص يتولى بالفعل هذه المهام.
عندما ينتج عن البناء مشكلات نفاد الذاكرة أثناء خطوة "التدقيق والتحقق من صحة الأنواع"، يمكنك تعطيل هذه المهام أثناء عمليات البناء:

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  eslint: {
    // تحذير: يسمح هذا بإكمال بنائات الإنتاج بنجاح حتى إذا
    // كان مشروعك يحتوي على أخطاء ESLint.
    ignoreDuringBuilds: true,
  },
  typescript: {
    // !! تحذير !!
    // يسمح بشكل خطير بإكمال بنائات الإنتاج بنجاح حتى إذا
    // كان مشروعك يحتوي على أخطاء في الأنواع.
    // !! تحذير !!
    ignoreBuildErrors: true,
  },
}

export default nextConfig
```

- [تجاهل أخطاء TypeScript](/docs/app/api-reference/config/typescript#disabling-typescript-errors-in-production)
- [ESLint في تكوين Next.js](/docs/pages/api-reference/config/next-config-js/eslint)

ضع في اعتبارك أن هذا قد ينتج عنه نشرات معيبة بسبب أخطاء في الأنواع أو مشكلات في التدقيق.
نوصي بشدة فقط بترقية البنائات إلى الإنتاج بعد اكتمال التحليل الثابت.
إذا كنت تقوم بالنشر على Vercel، يمكنك الاطلاع على [دليل النشر المؤقت](https://vercel.com/docs/deployments/managing-deployments#staging-and-promoting-a-production-deployment) لمعرفة كيفية ترقية البنائات إلى الإنتاج بعد نجاح المهام المخصصة.

## تعطيل خرائط المصدر (source maps)

يستهلك إنشاء خرائط المصدر ذاكرة إضافية أثناء عملية البناء.

يمكنك تعطيل إنشاء خرائط المصدر عن طريق إضافة `productionBrowserSourceMaps: false` و `experimental.serverSourceMaps: false` إلى تكوين Next.js الخاص بك.

> **معلومة مفيدة**: قد تقوم بعض الملحقات بتشغيل خرائط المصدر وقد تتطلب تكوينًا مخصصًا لتعطيلها.

## مشكلات الذاكرة في Edge runtime

أصلح Next.js `v14.1.3` مشكلة في الذاكرة عند استخدام Edge runtime. يرجى التحديث إلى هذا الإصدار (أو أحدث) لمعرفة ما إذا كان يحل مشكلتك.
