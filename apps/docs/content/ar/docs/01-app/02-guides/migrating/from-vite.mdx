سيساعدك هذا الدليل في نقل تطبيق Vite موجود إلى Next.js.

## لماذا التبديل؟

هناك عدة أسباب قد تدفعك للتبديل من Vite إلى Next.js:

### بطء وقت تحميل الصفحة الأولي

إذا قمت ببناء تطبيقك باستخدام [إضافة Vite الافتراضية لـ React](https://github.com/vitejs/vite-plugin-react/tree/main/packages/plugin-react)، فإن تطبيقك هو تطبيق يعمل فقط على جانب العميل. التطبيقات التي تعمل فقط على جانب العميل، والمعروفة أيضًا باسم التطبيقات ذات الصفحة الواحدة (SPAs)، غالبًا ما تعاني من بطء في وقت تحميل الصفحة الأولي. يحدث هذا بسبب عدة أسباب:

1. يحتاج المتصفح إلى انتظار تحميل وتشغيل كود React وحزمة التطبيق بالكامل قبل أن يتمكن الكود من إرسال طلبات لتحميل بعض البيانات.
2. ينمو كود التطبيق الخاص بك مع كل ميزة جديدة وإضافة تبعية إضافية.

### عدم وجود تقسيم تلقائي للكود

يمكن إدارة مشكلة بطء أوقات التحميل السابقة إلى حد ما باستخدام تقسيم الكود. ومع ذلك، إذا حاولت إجراء تقسيم الكود يدويًا، فغالبًا ما ستجعل الأداء أسوأ. من السهل إدخال شلالات الشبكة عن غير قصد عند تقسيم الكود يدويًا. يوفر Next.js تقسيمًا تلقائيًا للكود مدمجًا في جهاز التوجيه الخاص به.

### شلالات الشبكة

أحد الأسباب الشائعة لضعف الأداء هو عندما تقوم التطبيقات بعمل طلبات متتابعة بين العميل والخادم لجلب البيانات. أحد الأنماط الشائعة لجلب البيانات في تطبيق SPA هو عرض عنصر نائب في البداية، ثم جلب البيانات بعد تحميل المكون. لسوء الحظ، هذا يعني أن المكون الفرعي الذي يجلب البيانات لا يمكنه البدء في الجلب حتى ينتهي المكون الأصلي من تحميل بياناته الخاصة.

بينما يتم دعم جلب البيانات على العميل باستخدام Next.js، فإنه يمنحك أيضًا خيار نقل جلب البيانات إلى الخادم، مما يمكن أن يلغي شلالات العميل-الخادم.

### حالات تحميل سريعة ومقصودة

مع دعم مدمج لـ [البث عبر React Suspense](/docs/app/building-your-application/routing/loading-ui-and-streaming#streaming-with-suspense)، يمكنك أن تكون أكثر تحديدًا بشأن أجزاء واجهة المستخدم التي تريد تحميلها أولًا وبأي ترتيب دون إدخال شلالات الشبكة.

هذا يمكّنك من بناء صفحات يتم تحميلها بسرعة أكبر وتجنب [تحولات التخطيط](https://vercel.com/blog/how-core-web-vitals-affect-seo).

### اختر استراتيجية جلب البيانات

حسب احتياجاتك، يسمح لك Next.js باختيار استراتيجية جلب البيانات على أساس الصفحة والمكون. يمكنك أن تقرر الجلب في وقت البناء، في وقت الطلب على الخادم، أو على العميل. على سبيل المثال، يمكنك جلب البيانات من نظام إدارة المحتوى الخاص بك وعرض مشاركات مدونتك في وقت البناء، والتي يمكن بعد ذلك تخزينها بكفاءة على CDN.

### البرمجيات الوسيطة

[برمجيات Next.js الوسيطة](/docs/app/building-your-application/routing/middleware) تسمح لك بتشغيل الكود على الخادم قبل اكتمال الطلب. هذا مفيد بشكل خاص لتجنب ومضة من المحتوى غير المصادق عليه عندما يزور المستخدم صفحة تتطلب مصادقة عن طريق توجيه المستخدم إلى صفحة تسجيل الدخول. البرمجيات الوسيطة مفيدة أيضًا للتجارب و[التدويل](/docs/app/guides/internationalization).

### تحسينات مدمجة

غالبًا ما يكون للـ[الصور](/docs/app/api-reference/components/image)، [الخطوط](/docs/app/api-reference/components/font)، و[نصوص الطرف الثالث](/docs/app/guides/scripts) تأثير كبير على أداء التطبيق. يأتي Next.js مع مكونات مدمجة تحسن تلك تلقائيًا لك.

## خطوات النقل

هدفنا من هذا النقل هو الحصول على تطبيق Next.js يعمل في أسرع وقت ممكن، بحيث يمكنك بعد ذلك تبني ميزات Next.js تدريجيًا. في البداية، سنحافظ عليه كتطبيق يعمل فقط على جانب العميل (SPA) دون نقل جهاز التوجيه الحالي الخاص بك. هذا يساعد في تقليل فرص مواجهة مشكلات أثناء عملية النقل ويقلل من تعارضات الدمج.

### الخطوة 1: تثبيت تبعية Next.js

أول شيء تحتاج إلى فعله هو تثبيت `next` كتبعية:

```bash filename="Terminal"
npm install next@latest
```

### الخطوة 2: إنشاء ملف تكوين Next.js

قم بإنشاء ملف `next.config.mjs` في جذر مشروعك. سيحتوي هذا الملف على [خيارات تكوين Next.js](/docs/app/api-reference/config/next-config-js).

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export', // يخرج تطبيقًا ذا صفحة واحدة (SPA).
  distDir: './dist', // يغير دليل إخراج البناء إلى `./dist/`.
}

export default nextConfig
```

> **جيد أن تعرف:** يمكنك استخدام إما `.js` أو `.mjs` لملف تكوين Next.js الخاص بك.

### الخطوة 3: تحديث تكوين TypeScript

إذا كنت تستخدم TypeScript، فأنت بحاجة إلى تحديث ملف `tsconfig.json` الخاص بك بالتغييرات التالية لجعله متوافقًا مع Next.js. إذا كنت لا تستخدم TypeScript، يمكنك تخطي هذه الخطوة.

1. قم بإزالة [مرجع المشروع](https://www.typescriptlang.org/tsconfig#references) إلى `tsconfig.node.json`
2. أضف `./dist/types/**/*.ts` و `./next-env.d.ts` إلى [مصفوفة `include`](https://www.typescriptlang.org/tsconfig#include)
3. أضف `./node_modules` إلى [مصفوفة `exclude`](https://www.typescriptlang.org/tsconfig#exclude)
4. أضف `{ "name": "next" }` إلى [مصفوفة `plugins` في `compilerOptions`](https://www.typescriptlang.org/tsconfig#plugins): `"plugins": [{ "name": "next" }]`
5. اضبط [`esModuleInterop`](https://www.typescriptlang.org/tsconfig#esModuleInterop) على `true`: `"esModuleInterop": true`
6. اضبط [`jsx`](https://www.typescriptlang.org/tsconfig#jsx) على `preserve`: `"jsx": "preserve"`
7. اضبط [`allowJs`](https://www.typescriptlang.org/tsconfig#allowJs) على `true`: `"allowJs": true`
8. اضبط [`forceConsistentCasingInFileNames`](https://www.typescriptlang.org/tsconfig#forceConsistentCasingInFileNames) على `true`: `"forceConsistentCasingInFileNames": true`
9. اضبط [`incremental`](https://www.typescriptlang.org/tsconfig#incremental) على `true`: `"incremental": true`

إليك مثال على ملف `tsconfig.json` يعمل مع هذه التغييرات:

```json filename="tsconfig.json"
{
  "compilerOptions": {
    "target": "ES2020",
    "useDefineForClassFields": true,
    "lib": ["ES2020", "DOM", "DOM.Iterable"],
    "module": "ESNext",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "moduleResolution": "bundler",
    "allowImportingTsExtensions": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "noEmit": true,
    "jsx": "preserve",
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noFallthroughCasesInSwitch": true,
    "allowJs": true,
    "forceConsistentCasingInFileNames": true,
    "incremental": true,
    "plugins": [{ "name": "next" }]
  },
  "include": ["./src", "./dist/types/**/*.ts", "./next-env.d.ts"],
  "exclude": ["./node_modules"]
}
```

يمكنك العثور على مزيد من المعلومات حول تكوين TypeScript في [وثائق Next.js](/docs/app/api-reference/config/typescript#ide-plugin).

### الخطوة 4: إنشاء تخطيط الجذر

يجب أن يتضمن تطبيق [موجه تطبيق Next.js](/docs/app) ملف [تخطيط جذر](/docs/app/api-reference/file-conventions/layout#root-layout)، وهو [مكون خادم React](/docs/app/getting-started/server-and-client-components) الذي سيلف جميع الصفحات في تطبيقك. يتم تعريف هذا الملف في المستوى العلوي من دليل `app`.

أقرب ما يعادل ملف تخطيط الجذر في تطبيق Vite هو ملف [`index.html`](https://vitejs.dev/guide/#index-html-and-project-root)، الذي يحتوي على علامات `<html>` و`<head>` و`<body>` الخاصة بك.

في هذه الخطوة، ستقوم بتحويل ملف `index.html` الخاص بك إلى ملف تخطيط جذر:

1. قم بإنشاء دليل `app` جديد في مجلد `src` الخاص بك.
2. قم بإنشاء ملف `layout.tsx` جديد داخل دليل `app`:

```tsx filename="app/layout.tsx" switcher
export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return '...'
}
```

```jsx filename="app/layout.js" switcher
export default function RootLayout({ children }) {
  return '...'
}
```

> **جيد أن تعرف**: يمكن استخدام امتدادات `.js` أو `.jsx` أو `.tsx` لملفات التخطيط.

3. انسخ محتوى ملف `index.html` الخاص بك إلى مكون `<RootLayout>` الذي تم إنشاؤه مسبقًا مع استبدال علامات `body.div#root` و`body.script` بـ `<div id="root">{children}</div>`:

```tsx filename="app/layout.tsx" switcher
export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <head>
        <meta charset="UTF-8" />
        <link rel="icon" type="image/svg+xml" href="/icon.svg" />
        <meta name="viewport" content="width=device-width, initial-scale=1.0" />
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

```jsx filename="app/layout.js" switcher
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <head>
        <meta charset="UTF-8" />
        <link rel="icon" type="image/svg+xml" href="/icon.svg" />
        <meta name="viewport" content="width=device-width, initial-scale=1.0" />
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

4. يتضمن Next.js بالفعل بشكل افتراضي علامات [meta charset](https://developer.mozilla.org/docs/Web/HTML/Element/meta#charset) و [meta viewport](https://developer.mozilla.org/docs/Web/HTML/Viewport_meta_tag)، لذا يمكنك إزالتها بأمان من `<head>` الخاص بك:

```tsx filename="app/layout.tsx" switcher
export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <head>
        <link rel="icon" type="image/svg+xml" href="/icon.svg" />
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

```jsx filename="app/layout.js" switcher
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <head>
        <link rel="icon" type="image/svg+xml" href="/icon.svg" />
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

5. أي [ملفات بيانات وصفية](/docs/app/getting-started/metadata-and-og-images#file-based-metadata) مثل `favicon.ico`، `icon.png`، `robots.txt` تتم إضافتها تلقائيًا إلى علامة `<head>` للتطبيق طالما قمت بوضعها في المستوى العلوي من دليل `app`. بعد نقل [جميع الملفات المدعومة](/docs/app/getting-started/metadata-and-og-images#file-based-metadata) إلى دليل `app`، يمكنك حذف علامات `<link>` الخاصة بها بأمان:

```tsx filename="app/layout.tsx" switcher
export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <head>
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

```jsx filename="app/layout.js" switcher
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <head>
        <title>My App</title>
        <meta name="description" content="My App is a..." />
      </head>
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

6. أخيرًا، يمكن لـ Next.js إدارة علامات `<head>` الأخيرة الخاصة بك باستخدام [واجهة برمجة تطبيقات البيانات الوصفية](/docs/app/getting-started/metadata-and-og-images). انقل معلومات البيانات الوصفية النهائية الخاصة بك إلى كائن [`metadata` مُصدّر](/docs/app/api-reference/functions/generate-metadata#metadata-object):

```tsx filename="app/layout.tsx" switcher
import type { Metadata } from 'next'

export const metadata: Metadata = {
  title: 'My App',
  description: 'My App is a...',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

```jsx filename="app/layout.js" switcher
export const metadata = {
  title: 'My App',
  description: 'My App is a...',
}

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        <div id="root">{children}</div>
      </body>
    </html>
  )
}
```

مع التغييرات المذكورة أعلاه، انتقلت من الإعلان عن كل شيء في `index.html` إلى استخدام النهج القائم على الاصطلاحات في Next.js المدمج في الإطار ([واجهة برمجة تطبيقات البيانات الوصفية](/docs/app/getting-started/metadata-and-og-images)). يتيح لك هذا النهج تحسين تحسين محركات البحث (SEO) وإمكانية مشاركة صفحاتك على الويب بسهولة أكبر.

### الخطوة 5: إنشاء صفحة نقطة الدخول

في Next.js، يمكنك تحديد نقطة دخول لتطبيقك عن طريق إنشاء ملف `page.tsx`. أقرب ما يعادله في Vite هو ملف `main.tsx` الخاص بك. في هذه الخطوة، ستقوم بإعداد نقطة الدخول لتطبيقك.

1. **قم بإنشاء دليل `[[...slug]]` داخل دليل `app` الخاص بك.**

نظرًا لأننا في هذا الدليل نهدف أولاً إلى إعداد Next.js كتطبيق صفحة واحدة (SPA)، تحتاج إلى أن تكون نقطة دخول الصفحة قادرة على التقاط جميع المسارات المحتملة لتطبيقك. لهذا، قم بإنشاء دليل جديد `[[...slug]]` داخل دليل `app`.

هذا الدليل يسمى [قطاع مسار اختياري شامل](/docs/app/api-reference/file-conventions/dynamic-routes#optional-catch-all-segments). يستخدم Next.js موجهًا يعتمد على نظام الملفات حيث تُستخدم المجلدات لتحديد المسارات. هذا الدليل الخاص سيتأكد من توجيه جميع مسارات تطبيقك إلى ملف `page.tsx` الموجود بداخله.

2. **قم بإنشاء ملف جديد `page.tsx` داخل دليل `app/[[...slug]]` بالمحتوى التالي:**

```tsx filename="app/[[...slug]]/page.tsx" switcher
import '../../index.css'

export function generateStaticParams() {
  return [{ slug: [''] }]
}

export default function Page() {
  return '...' // سنقوم بتحديث هذا لاحقًا
}
```

```jsx filename="app/[[...slug]]/page.js" switcher
import '../../index.css'

export function generateStaticParams() {
  return [{ slug: [''] }]
}

export default function Page() {
  return '...' // سنقوم بتحديث هذا لاحقًا
}
```

> **معلومة مفيدة**: يمكن استخدام امتدادات `.js` أو `.jsx` أو `.tsx` لملفات الصفحات.

هذا الملف هو [مكون خادم (Server Component)](/docs/app/getting-started/server-and-client-components). عند تشغيل `next build`، يتم تقديم الملف مسبقًا كأصل ثابت. لا يحتاج إلى أي كود ديناميكي.

هذا الملف يستورد CSS العام الخاص بنا ويخبر [`generateStaticParams`](/docs/app/api-reference/functions/generate-static-params) أننا سننشئ مسارًا واحدًا فقط، وهو مسار الفهرس عند `/`.

الآن، لننقل بقية تطبيق Vite الذي سيعمل فقط على جانب العميل.

```tsx filename="app/[[...slug]]/client.tsx" switcher
'use client'

import React from 'react'
import dynamic from 'next/dynamic'

const App = dynamic(() => import('../../App'), { ssr: false })

export function ClientOnly() {
  return <App />
}
```

```jsx filename="app/[[...slug]]/client.js" switcher
'use client'

import React from 'react'
import dynamic from 'next/dynamic'

const App = dynamic(() => import('../../App'), { ssr: false })

export function ClientOnly() {
  return <App />
}
```

هذا الملف هو [مكون عميل (Client Component)](/docs/app/getting-started/server-and-client-components)، محدد بتوجيه `'use client'`. مكونات العميل لا تزال [تُقدم مسبقًا إلى HTML](/docs/app/getting-started/server-and-client-components#how-do-server-and-client-components-work-in-nextjs) على الخادم قبل إرسالها إلى العميل.

نظرًا لأننا نريد تطبيقًا يعمل فقط على العميل في البداية، يمكننا تكوين Next.js لتعطيل التقديم المسبق من مكون `App` فما دون.

```tsx
const App = dynamic(() => import('../../App'), { ssr: false })
```

الآن، قم بتحديث صفحة نقطة الدخول لاستخدام المكون الجديد:

```tsx filename="app/[[...slug]]/page.tsx" switcher
import '../../index.css'
import { ClientOnly } from './client'

export function generateStaticParams() {
  return [{ slug: [''] }]
}

export default function Page() {
  return <ClientOnly />
}
```

```jsx filename="app/[[...slug]]/page.js" switcher
import '../../index.css'
import { ClientOnly } from './client'

export function generateStaticParams() {
  return [{ slug: [''] }]
}

export default function Page() {
  return <ClientOnly />
}
```

### الخطوة 6: تحديث استيرادات الصور الثابتة

يتعامل Next.js مع استيرادات الصور الثابتة بشكل مختلف قليلاً عن Vite. مع Vite، استيراد ملف صورة يعيد عنوان URL العام كسلسلة نصية:

```tsx filename="App.tsx"
import image from './img.png' // `image` سيكون '/assets/img.2d8efhg.png' في الإنتاج

export default function App() {
  return <img src={image} />
}
```

مع Next.js، استيرادات الصور الثابتة تعيد كائنًا. يمكن بعد ذلك استخدام هذا الكائن مباشرة مع مكون Next.js [`<Image>`](/docs/app/api-reference/components/image)، أو يمكنك استخدام خاصية `src` للكائن مع وسم `<img>` الحالي.

مكون `<Image>` لديه ميزات إضافية مثل [تحسين الصور التلقائي](/docs/app/api-reference/components/image). يقوم مكون `<Image>` تلقائيًا بتعيين سمات `width` و `height` لوسم `<img>` الناتج بناءً على أبعاد الصورة. هذا يمنع حدوث تغييرات في التخطيط عند تحميل الصورة. ومع ذلك، يمكن أن يسبب هذا مشاكل إذا كان تطبيقك يحتوي على صور بأبعاد غير متساوية دون تعيين الآخر إلى `auto`. عندما لا يتم تعيينه إلى `auto`، سيعتمد البعد على قيمة سمة البعد لوسم `<img>`، مما قد يجعل الصورة تظهر مشوهة.

الحفاظ على وسم `<img>` سيقلل من عدد التغييرات في تطبيقك ويمنع المشاكل المذكورة أعلاه. يمكنك لاحقًا الانتقال إلى مكون `<Image>` للاستفادة من تحسين الصور عن طريق [تكوين محمل (loader)](/docs/app/api-reference/components/image#loader)، أو الانتقال إلى خادم Next.js الافتراضي الذي لديه تحسين صور تلقائي.

1. **تحويل مسارات الاستيراد المطلقة للصور المستوردة من `/public` إلى استيرادات نسبية:**

```tsx
// قبل
import logo from '/logo.png'

// بعد
import logo from '../public/logo.png'
```

2. **تمرير خاصية `src` للصورة بدلاً من كائن الصورة بالكامل إلى وسم `<img>`:**

```tsx
// قبل
<img src={logo} />

// بعد
<img src={logo.src} />
```

بدلاً من ذلك، يمكنك الرجوع إلى عنوان URL العام لأصل الصورة بناءً على اسم الملف. على سبيل المثال، `public/logo.png` سيقدم الصورة عند `/logo.png` لتطبيقك، والذي سيكون قيمة `src`.

> **تحذير:** إذا كنت تستخدم TypeScript، قد تواجه أخطاء نوع عند الوصول إلى خاصية `src`. يمكنك تجاهلها بأمان الآن. سيتم إصلاحها بنهاية هذا الدليل.

### الخطوة 7: ترحيل متغيرات البيئة

يدعم Next.js متغيرات البيئة `.env` [متغيرات البيئة](/docs/app/guides/environment-variables) بشكل مشابه لـ Vite. الفرق الرئيسي هو البادئة المستخدمة لعرض متغيرات البيئة على جانب العميل.

- قم بتغيير جميع متغيرات البيئة ذات البادئة `VITE_` إلى `NEXT_PUBLIC_`.

يعرض Vite بعض متغيرات البيئة المدمجة على كائن `import.meta.env` الخاص والتي لا يدعمها Next.js. تحتاج إلى تحديث استخدامها كما يلي:

- `import.meta.env.MODE` ⇒ `process.env.NODE_ENV`
- `import.meta.env.PROD` ⇒ `process.env.NODE_ENV === 'production'`
- `import.meta.env.DEV` ⇒ `process.env.NODE_ENV !== 'production'`
- `import.meta.env.SSR` ⇒ `typeof window !== 'undefined'`

Next.js أيضًا لا يوفر متغير بيئة `BASE_URL` مدمج. ومع ذلك، يمكنك تكوين واحد إذا كنت بحاجة إليه:

1. **أضف ما يلي إلى ملف `.env` الخاص بك:**

```bash filename=".env"
# ...
NEXT_PUBLIC_BASE_PATH="/some-base-path"
```

2. **قم بتعيين [`basePath`](/docs/app/api-reference/config/next-config-js/basePath) إلى `process.env.NEXT_PUBLIC_BASE_PATH` في ملف `next.config.mjs`:**

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export', // يُخرج تطبيق صفحة واحدة (SPA).
  distDir: './dist', // يغير دليل إخراج البناء إلى `./dist/`.
  basePath: process.env.NEXT_PUBLIC_BASE_PATH, // يحدد المسار الأساسي إلى `/some-base-path`.
}

export default nextConfig
```

3. **قم بتحديث استخدامات `import.meta.env.BASE_URL` إلى `process.env.NEXT_PUBLIC_BASE_PATH`**

### الخطوة 8: تحديث الأوامر في `package.json`

يجب أن تكون الآن قادرًا على تشغيل تطبيقك لاختبار ما إذا كنت قد نجحت في ترحيله إلى Next.js. ولكن قبل ذلك، تحتاج إلى تحديث `scripts` في `package.json` بأوامر متعلقة بـ Next.js، وإضافة `.next` و `next-env.d.ts` إلى `.gitignore`:

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start"
  }
}
```

```txt filename=".gitignore"
# ...
.next
next-env.d.ts
dist
```

الآن قم بتشغيل `npm run dev`، وافتح [`http://localhost:3000`](http://localhost:3000). يجب أن ترى تطبيقك يعمل الآن على Next.js.

> **مثال:** تحقق من [طلب السحب هذا](https://github.com/inngest/vite-to-nextjs/pull/1) للحصول على مثال عمل لتطبيق Vite تم ترحيله إلى Next.js.

### الخطوة 9: التنظيف

يمكنك الآن تنظيف قاعدة التعليمات البرمجية من متعلقات Vite:

- احذف `main.tsx`
- احذف `index.html`
- احذف `vite-env.d.ts`
- احذف `tsconfig.node.json`
- احذف `vite.config.ts`
- أزل تبعيات Vite
