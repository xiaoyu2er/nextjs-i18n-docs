تسمح لك المسارات المتوازية (Parallel Routes) بعرض صفحة واحدة أو أكثر داخل نفس التخطيط (layout) بشكل متزامن أو مشروط. وهي مفيدة للأجزاء عالية الديناميكية في التطبيق، مثل لوحات التحكم (dashboards) وتغذيات المواقع الاجتماعية.

على سبيل المثال، بالنظر إلى لوحة تحكم، يمكنك استخدام المسارات المتوازية لعرض صفحات `team` و `analytics` في نفس الوقت:

## الاصطلاح

### الفتحات (Slots)

يتم إنشاء المسارات المتوازية باستخدام **فتحات** مسماة. يتم تعريف الفتحات باستخدام اصطلاح `@folder`. على سبيل المثال، هيكل الملفات التالي يعرف فتحتين: `@analytics` و `@team`:

<Image
  alt="هيكل نظام الملفات للمسارات المتوازية"
  srcLight="/docs/light/parallel-routes-file-system.png"
  srcDark="/docs/dark/parallel-routes-file-system.png"
  width="1600"
  height="687"
/>

يتم تمرير الفتحات كخصائص (props) إلى تخطيط الأب المشترك. بالنسبة للمثال أعلاه، المكون في `app/layout.js` يقبل الآن خصائص فتحات `@analytics` و `@team`، ويمكن عرضها بالتوازي مع خاصية `children`:

```tsx filename="app/layout.tsx" switcher
export default function Layout({
  children,
  team,
  analytics,
}: {
  children: React.ReactNode
  analytics: React.ReactNode
  team: React.ReactNode
}) {
  return (
    <>
      {children}
      {team}
      {analytics}
    </>
  )
}
```

```jsx filename="app/layout.js" switcher
export default function Layout({ children, team, analytics }) {
  return (
    <>
      {children}
      {team}
      {analytics}
    </>
  )
}
```

ومع ذلك، الفتحات **ليست** مقاطع مسار (route segments) ولا تؤثر على هيكل URL. على سبيل المثال، بالنسبة لـ `/@analytics/views`، سيكون URL هو `/views` لأن `@analytics` هي فتحة. يتم دمج الفتحات مع مكون [الصفحة](/docs/app/api-reference/file-conventions/page) العادي لتشكيل الصفحة النهائية المرتبطة بمقطع المسار. بسبب هذا، لا يمكنك أن يكون لديك فتحات [ثابتة](/docs/app/getting-started/partial-prerendering#static-rendering) و [ديناميكية](/docs/app/getting-started/partial-prerendering#dynamic-rendering) منفصلة في نفس مستوى مقطع المسار. إذا كانت إحدى الفتحات ديناميكية، فيجب أن تكون جميع الفتحات في ذلك المستوى ديناميكية.

> **معلومة مفيدة**:
>
> - خاصية `children` هي فتحة ضمنية لا تحتاج إلى تعيينها لمجلد. هذا يعني أن `app/page.js` تعادل `app/@children/page.js`.

### `default.js`

يمكنك تعريف ملف `default.js` ليتم عرضه كحل بديل للفتحات غير المتطابقة أثناء التحميل الأولي أو إعادة تحميل الصفحة بالكامل.

خذ بعين الاعتبار هيكل المجلدات التالي. فتحة `@team` تحتوي على صفحة `/settings`، لكن `@analytics` لا تحتوي عليها.

<Image
  alt="مسارات متوازية مع مسارات غير متطابقة"
  srcLight="/docs/light/parallel-routes-unmatched-routes.png"
  srcDark="/docs/dark/parallel-routes-unmatched-routes.png"
  width="1600"
  height="930"
/>

عند التنقل إلى `/settings`، ستعرض فتحة `@team` صفحة `/settings` مع الحفاظ على الصفحة النشطة حاليًا لفتحة `@analytics`.

عند التحديث، سيقوم Next.js بعرض `default.js` لـ `@analytics`. إذا لم يوجد `default.js`، سيتم عرض `404` بدلاً من ذلك.

بالإضافة إلى ذلك، بما أن `children` هي فتحة ضمنية، فأنت بحاجة أيضًا إلى إنشاء ملف `default.js` لعرض حل بديل لـ `children` عندما لا يتمكن Next.js من استعادة الحالة النشطة للصفحة الأصلية.

## السلوك

بشكل افتراضي، يحتفظ Next.js بتتبع الحالة _النشطة_ (أو الصفحة الفرعية) لكل فتحة. ومع ذلك، المحتوى المعروض داخل الفتحة سيعتمد على نوع التنقل:

- [**التنقل اللين (Soft Navigation)**](/docs/app/building-your-application/routing/linking-and-navigating#5-soft-navigation): أثناء التنقل من جانب العميل، سيقوم Next.js بتنفيذ [عرض جزئي](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)، تغيير الصفحة الفرعية داخل الفتحة، مع الحفاظ على الصفحات الفرعية النشطة للفتحات الأخرى، حتى لو لم تتطابق مع URL الحالي.
- **التنقل الصعب (Hard Navigation)**: بعد تحميل الصفحة بالكامل (تحديث المتصفح)، لا يمكن لـ Next.js تحديد الحالة النشطة للفتحات التي لا تتطابق مع URL الحالي. بدلاً من ذلك، سيعرض ملف [`default.js`](#defaultjs) للفتحات غير المتطابقة، أو `404` إذا لم يوجد `default.js`.

> **معلومة مفيدة**:
>
> - تساعد صفحة `404` للمسارات غير المتطابقة على ضمان عدم عرض مسار متوازي على صفحة لم يُقصد له أن يكون عليها.

## أمثلة

### مع `useSelectedLayoutSegment(s)`

كل من [`useSelectedLayoutSegment`](/docs/app/api-reference/functions/use-selected-layout-segment) و [`useSelectedLayoutSegments`](/docs/app/api-reference/functions/use-selected-layout-segments) يقبلان معلمة `parallelRoutesKey`، والتي تتيح لك قراءة مقطع المسار النشط داخل فتحة.

```tsx filename="app/layout.tsx" switcher
'use client'

import { useSelectedLayoutSegment } from 'next/navigation'

export default function Layout({ auth }: { auth: React.ReactNode }) {
  const loginSegment = useSelectedLayoutSegment('auth')
  // ...
}
```

```jsx filename="app/layout.js" switcher
'use client'

import { useSelectedLayoutSegment } from 'next/navigation'

export default function Layout({ auth }) {
  const loginSegment = useSelectedLayoutSegment('auth')
  // ...
}
```

عندما ينتقل المستخدم إلى `app/@auth/login` (أو `/login` في شريط العناوين)، سيكون `loginSegment` مساويًا للسلسلة `"login"`.

### مسارات مشروطة

يمكنك استخدام المسارات المتوازية لعرض مسارات مشروطة بناءً على شروط معينة، مثل دور المستخدم. على سبيل المثال، لعرض صفحة لوحة تحكم مختلفة لدور `/admin` أو `/user`:

<Image
  alt="رسم توضيحي للمسارات المشروطة"
  srcLight="/docs/light/conditional-routes-ui.png"
  srcDark="/docs/dark/conditional-routes-ui.png"
  width="1600"
  height="898"
/>

```tsx filename="app/dashboard/layout.tsx" switcher
import { checkUserRole } from '@/lib/auth'

export default function Layout({
  user,
  admin,
}: {
  user: React.ReactNode
  admin: React.ReactNode
}) {
  const role = checkUserRole()
  return role === 'admin' ? admin : user
}
```

```jsx filename="app/dashboard/layout.js" switcher
import { checkUserRole } from '@/lib/auth'

export default function Layout({ user, admin }) {
  const role = checkUserRole()
  return role === 'admin' ? admin : user
}
```

### مجموعات علامات التبويب (Tab Groups)

يمكنك إضافة `layout` داخل فتحة للسماح للمستخدمين بالتنقل داخل الفتحة بشكل مستقل. هذا مفيد لإنشاء علامات تبويب.

على سبيل المثال، فتحة `@analytics` تحتوي على صفحتين فرعيتين: `/page-views` و `/visitors`.

### النوافذ المنبثقة (Modals)

يمكن استخدام المسارات المتوازية مع [مسارات الاعتراض (Intercepting Routes)](/docs/app/api-reference/file-conventions/intercepting-routes) لإنشاء نوافذ منبثقة تدعم الربط العميق (deep linking). هذا يسمح لك بحل التحديات الشائعة عند بناء النوافذ المنبثقة، مثل:

- جعل محتوى النافذة **قابلاً للمشاركة عبر URL**.
- **الحفاظ على السياق** عند تحديث الصفحة، بدلاً من إغلاق النافذة.
- **إغلاق النافذة عند التنقل للخلف** بدلاً من الذهاب إلى المسار السابق.
- **إعادة فتح النافذة عند التنقل للأمام**.

خذ بعين الاعتبار نمط واجهة المستخدم التالي، حيث يمكن للمستخدم فتح نافذة تسجيل دخول من تخطيط باستخدام التنقل من جانب العميل، أو الوصول إلى صفحة `/login` منفصلة:

<Image
  alt="رسم توضيحي للمسارات المتوازية مع نافذة تسجيل دخول"
  srcLight="/docs/light/parallel-routes-auth-modal.png"
  srcDark="/docs/dark/parallel-routes-auth-modal.png"
  width="1600"
  height="687"
/>

لتنفيذ هذا النمط، ابدأ بإنشاء مسار `/login` الذي يعرض صفحة تسجيل الدخول **الرئيسية**.

<Image
  alt="رسم توضيحي للمسارات المتوازية مع صفحة تسجيل دخول"
  srcLight="/docs/light/parallel-routes-modal-login-page.png"
  srcDark="/docs/dark/parallel-routes-modal-login-page.png"
  width="1600"

/>

```tsx filename="app/login/page.tsx" switcher
import { Login } from '@/app/ui/login'

export default function Page() {
  return <Login />
}
```

```jsx filename="app/login/page.js" switcher
import { Login } from '@/app/ui/login'

export default function Page() {
  return <Login />

```

ثم، داخل فتحة `@auth`، أضف ملف [`default.js`](/docs/app/api-reference/file-conventions/default) الذي يُرجع `null`. هذا يضمن عدم عرض النافذة عندما لا تكون نشطة.

```tsx filename="app/@auth/default.tsx" switcher
export default function Default() {
  return null
}
```

```jsx filename="app/@auth/default.js" switcher
export default function Default() {
  return null
}
```

داخل فتحة `@auth` الخاصة بك، اعترض مسار `/login` عن طريق تحديث مجلد `/(.)login`. استورد مكون `<Modal>` وأطفاله إلى ملف `/(.)login/page.tsx`:

```tsx filename="app/@auth/(.)login/page.tsx" switcher
import { Modal } from '@/app/ui/modal'
import { Login } from '@/app/ui/login'

export default function Page() {
  return (
    <Modal>
      <Login />
    </Modal>
  )
}
```

```jsx filename="app/@auth/(.)login/page.js" switcher
import { Modal } from '@/app/ui/modal'
import { Login } from '@/app/ui/login'

export default function Page() {
  return (
    <Modal>
      <Login />
    </Modal>
  )
}
```

> **معلومة مفيدة:**
>
> - الاصطلاح المستخدم لاعتراض المسار، مثل `(.)`، يعتمد على هيكل نظام الملفات الخاص بك. راجع [اصطلاح مسارات الاعتراض](/docs/app/api-reference/file-conventions/intercepting-routes#convention).
> - بفصل وظيفة `<Modal>` عن محتوى النافذة (`<Login>`)، يمكنك التأكد من أن أي محتوى داخل النافذة، مثل [النماذج](/docs/app/guides/forms)، هي مكونات خادم (Server Components). راجع [دمج مكونات العميل والخادم](/docs/app/getting-started/server-and-client-components#examples#supported-pattern-passing-server-components-to-client-components-as-props) لمزيد من المعلومات.

#### فتح النافذة

الآن، يمكنك الاستفادة من موجه Next.js لفتح وإغلاق النافذة. هذا يضمن تحديث URL بشكل صحيح عند فتح النافذة، وعند التنقل للخلف والأمام.

لفتح النافذة، قم بتمرير فتحة `@auth` كخاصية إلى تخطيط الأب وعرضها بالتوازي مع خاصية `children`.

```tsx filename="app/layout.tsx" switcher
import Link from 'next/link'

export default function Layout({
  auth,
  children,
}: {
  auth: React.ReactNode
  children: React.ReactNode
}) {
  return (
    <>
      <nav>
        <Link href="/login">فتح النافذة</Link>
      </nav>
      <div>{auth}</div>
      <div>{children}</div>
    </>
  )
}
```

```jsx filename="app/layout.js" switcher
import Link from 'next/link'

export default function Layout({ auth, children }) {
  return (
    <>
      <nav>
        <Link href="/login">فتح النافذة</Link>
      </nav>
      <div>{auth}</div>
      <div>{children}</div>
    </>
  )
}
```

عندما ينقر المستخدم على `<Link>`، ستفتح النافذة بدلاً من الانتقال إلى صفحة `/login`. ومع ذلك، عند التحديث أو التحميل الأولي، الانتقال إلى `/login` سيأخذ المستخدم إلى صفحة تسجيل الدخول الرئيسية.

#### إغلاق النافذة

يمكنك إغلاق النافذة عن طريق استدعاء `router.back()` أو باستخدام مكون `Link`.

```tsx filename="app/ui/modal.tsx" switcher
'use client'

import { useRouter } from 'next/navigation'

export function Modal({ children }: { children: React.ReactNode }) {
  const router = useRouter()

  return (
    <>
      <button
        onClick={() => {
          router.back()
        }}
      >
        إغلاق النافذة
      </button>
      <div>{children}</div>
    </>
  )
}
```

```jsx filename="app/ui/modal.js" switcher
'use client'

import { useRouter } from 'next/navigation'

export function Modal({ children }) {
  const router = useRouter()

  return (
    <>
      <button
        onClick={() => {
          router.back()
        }}
      >
        إغلاق النافذة
      </button>
      <div>{children}</div>
    </>
  )
}
```

عند استخدام مكون `Link` للتنقل بعيدًا عن صفحة لا يجب أن تعرض فتحة `@auth` بعد الآن، نحتاج إلى التأكد من أن المسار المتوازي يتطابق مع مكون يُرجع `null`. على سبيل المثال، عند التنقل للخلف إلى الصفحة الرئيسية، ننشئ مكون `@auth/page.tsx`:

```tsx filename="app/ui/modal.tsx" switcher
import Link from 'next/link'

export function Modal({ children }: { children: React.ReactNode }) {
  return (
    <>
      <Link href="/">إغلاق النافذة</Link>
      <div>{children}</div>
    </>
  )
}
```

```jsx filename="app/ui/modal.js" switcher
import Link from 'next/link'

export function Modal({ children }) {
  return (
    <>
      <Link href="/">إغلاق النافذة</Link>
      <div>{children}</div>
    </>
  )
}
```

```tsx filename="app/@auth/page.tsx" switcher
export default function Page() {
  return null
}
```

```jsx filename="app/@auth/page.js" switcher
export default function Page() {
  return null
}
```

أو إذا كنت تنتقل إلى أي صفحة أخرى (مثل `/foo`، `/foo/bar`، إلخ)، يمكنك استخدام فتحة catch-all:

```tsx filename="app/@auth/[...catchAll]/page.tsx" switcher
export default function CatchAll() {
  return null
}
```

```jsx filename="app/@auth/[...catchAll]/page.js" switcher
export default function CatchAll() {
  return null
}
```

> **معلومة مفيدة:**
>
> - نستخدم مسار catch-all في فتحة `@auth` لإغلاق النافذة بسبب كيفية تصرف المسارات المتوازية(#behavior). نظرًا لأن عمليات التنقل من جانب العميل إلى مسار لم يعد يتطابق مع الفتحة ستبقى مرئية، نحتاج إلى مطابقة الفتحة مع مسار يُرجع `null` لإغلاق النافذة.
> - يمكن أن تتضمن الأمثلة الأخرى فتح نافذة صورة في معرض مع وجود صفحة مخصصة `/photo/[id]`، أو فتح عربة تسوق في نافذة جانبية.
> - [عرض مثال](https://github.com/vercel-labs/nextgram) للنوافذ المنبثقة مع مسارات الاعتراض والمسارات المتوازية.
