Next.js 透過快取渲染工作和資料請求來提升應用程式效能並降低成本。本頁將深入探討 Next.js 的快取機制、可用的配置 API 以及它們之間的互動方式。

> **須知事項**：本頁內容幫助您理解 Next.js 的底層運作原理，但並非使用 Next.js 的必備知識。Next.js 的大部分快取啟發式規則由您的 API 使用方式決定，並預設為最佳效能且無需或只需極少配置。若您想直接查看範例，[請從這裡開始](/docs/app/building-your-application/data-fetching/fetching)。

## 概覽

以下是不同快取機制及其用途的高層次概覽：

| 機制                                   | 快取內容                   | 位置   | 用途                                         | 持續時間                        |
| ------------------------------------------- | -------------------------- | ------ | ----------------------------------------------- | ------------------------------- |
| [請求記憶化 (Request Memoization)](#request-memoization) | 函數回傳值 | 伺服器端 | 在 React 元件樹中重複使用資料           | 單次請求生命週期           |
| [資料快取 (Data Cache)](#data-cache)                   | 資料                       | 伺服器端 | 跨使用者請求與部署儲存資料 | 持久性（可重新驗證） |
| [完整路由快取 (Full Route Cache)](#full-route-cache)       | HTML 與 RSC 負載       | 伺服器端 | 降低渲染成本並提升效能   | 持久性（可重新驗證） |
| [路由快取 (Router Cache)](#client-side-router-cache)   | RSC 負載                | 客戶端 | 減少導航時的伺服器請求            | 使用者會話或基於時間      |

## 請求記憶化 (Request Memoization)

Next.js 擴展了 [`fetch` API](#fetch) 以自動**記憶化**具有相同 URL 和選項的請求。這意味著您可以在 React 元件樹的多個位置呼叫相同的 fetch 函數，而實際上只會執行一次。

<Image
  alt="去重複的 Fetch 請求"
  srcLight="/docs/light/deduplicated-fetch-requests.png"
  srcDark="/docs/dark/deduplicated-fetch-requests.png"
  width="1600"
  height="857"
/>

例如，如果您需要在路由中多處使用相同資料（如在 Layout、Page 和多個元件中），您不必在樹的頂層獲取資料並透過 props 傳遞。相反，您可以在需要資料的元件中直接獲取，而無需擔心因多次網路請求相同資料而影響效能。

```tsx filename="app/example.tsx" switcher
async function getItem() {
  // `fetch` 函數會自動記憶化，結果會被快取
  const res = await fetch('https://.../item/1')
  return res.json()
}

// 此函數被呼叫兩次，但僅首次執行
const item = await getItem() // cache MISS

// 第二次呼叫可以位於路由的任何位置
const item = await getItem() // cache HIT
```

```jsx filename="app/example.js" switcher
async function getItem() {
  // `fetch` 函數會自動記憶化，結果會被快取
  const res = await fetch('https://.../item/1')
  return res.json()
}

// 此函數被呼叫兩次，但僅首次執行
const item = await getItem() // cache MISS

// 第二次呼叫可以位於路由的任何位置
const item = await getItem() // cache HIT
```

**請求記憶化的工作原理**

<Image
  alt="圖表展示 React 渲染期間 fetch 記憶化的工作方式。"
  srcLight="/docs/light/request-memoization.png"
  srcDark="/docs/dark/request-memoization.png"
  width="1600"
  height="742"
/>

- 在渲染路由時，首次呼叫特定請求時，其結果不會在記憶體中，因此會是快取 `MISS`。
- 因此，函數會被執行，資料會從外部來源獲取，結果會儲存在記憶體中。
- 同一渲染過程中的後續請求呼叫會是快取 `HIT`，資料會從記憶體中回傳而不執行函數。
- 一旦路由渲染完成且渲染過程結束，記憶體會被「重置」，所有請求記憶化條目會被清除。

> **須知事項**：
>
> - 請求記憶化是 React 的功能，而非 Next.js 的功能。此處提及是為了展示它與其他快取機制的互動。
> - 記憶化僅適用於 `fetch` 請求中的 `GET` 方法。
> - 記憶化僅適用於 React 元件樹，這意味著：
>   - 它適用於 `generateMetadata`、`generateStaticParams`、Layouts、Pages 和其他伺服器元件中的 `fetch` 請求。
>   - 它不適用於路由處理器 (Route Handlers) 中的 `fetch` 請求，因為它們不是 React 元件樹的一部分。
> - 對於不適合使用 `fetch` 的情況（如某些資料庫客戶端、CMS 客戶端或 GraphQL 客戶端），您可以使用 [React `cache` 函數](#react-cache-function) 來記憶化函數。

### 持續時間

快取持續時間為伺服器請求的生命週期，直到 React 元件樹完成渲染。

### 重新驗證 (Revalidating)

由於記憶化不會跨伺服器請求共享且僅在渲染期間適用，因此無需重新驗證。

### 選擇退出

記憶化僅適用於 `fetch` 請求中的 `GET` 方法，其他方法如 `POST` 和 `DELETE` 不會被記憶化。此預設行為是 React 的優化，我們不建議選擇退出。

若要管理個別請求，您可以使用 [`AbortController`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController) 的 [`signal`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController/signal) 屬性。然而，這不會讓請求退出記憶化，而是中止進行中的請求。

```js filename="app/example.js"
const { signal } = new AbortController()
fetch(url, { signal })
```

## 資料快取 (Data Cache)

Next.js 內建了資料快取，可**持久化**資料獲取的結果，跨**伺服器請求**和**部署**。這是因為 Next.js 擴展了原生 `fetch` API，允許伺服器上的每個請求設置自己的持久快取語義。

> **須知事項**：在瀏覽器中，`fetch` 的 `cache` 選項表示請求如何與瀏覽器的 HTTP 快取互動；在 Next.js 中，`cache` 選項表示伺服器端請求如何與伺服器的資料快取互動。

您可以使用 `fetch` 的 [`cache`](#fetch-optionscache) 和 [`next.revalidate`](#fetch-optionsnextrevalidate) 選項來配置快取行為。

**資料快取的工作原理**

<Image
  alt="圖表展示快取和非快取的 fetch 請求如何與資料快取互動。快取的請求會儲存在資料快取中並被記憶化，非快取的請求會從資料來源獲取，不儲存在資料快取中，但仍會被記憶化。"
  srcLight="/docs/light/data-cache.png"
  srcDark="/docs/dark/data-cache.png"
  width="1600"
  height="661"
/>

- 當渲染期間首次呼叫帶有 `'force-cache'` 選項的 `fetch` 請求時，Next.js 會檢查資料快取中是否有快取的回應。
- 若找到快取的回應，會立即回傳並[記憶化](#request-memoization)。
- 若未找到快取的回應，會向資料來源發出請求，結果會儲存在資料快取中並被記憶化。
- 對於非快取資料（如未定義 `cache` 選項或使用 `{ cache: 'no-store' }`），結果總是從資料來源獲取並被記憶化。
- 無論資料是否被快取，請求都會被記憶化，以避免在 React 渲染過程中對相同資料發出重複請求。

> **資料快取與請求記憶化的差異**
>
> 雖然兩種快取機制都透過重複使用快取資料來提升效能，但資料快取跨請求與部署持久化，而記憶化僅在單次請求的生命週期中有效。

### 持續時間

資料快取跨請求與部署持久化，除非您重新驗證或選擇退出。

### 重新驗證

快取資料可透過兩種方式重新驗證：

- **基於時間的重新驗證 (Time-based Revalidation)**：在指定時間間隔後重新驗證資料。這適用於變更不頻繁且即時性不高的資料。
- **按需重新驗證 (On-demand Revalidation)**：根據事件（如表單提交）重新驗證資料。按需重新驗證可使用標籤或路徑來一次重新驗證一組資料。這適用於需要盡快顯示最新資料的情境（如 headless CMS 內容更新時）。

#### 基於時間的重新驗證

要在時間間隔後重新驗證資料，您可以使用 `fetch` 的 `next.revalidate` 選項來設定資源的快取生命週期（以秒為單位）。

```js
// 最多每小時重新驗證一次
fetch('https://...', { next: { revalidate: 3600 } })
```

或者，您可以使用[路由區段配置選項](#segment-config-options)來配置區段中的所有 `fetch` 請求，或在不使用 `fetch` 的情況下進行配置。

**基於時間的重新驗證工作原理**

<Image
  alt="圖表展示基於時間的重新驗證如何運作，在重新驗證期間後，首次請求會回傳過期資料，隨後資料會被重新驗證。"
  srcLight="/docs/light/time-based-revalidation.png"
  srcDark="/docs/dark/time-based-revalidation.png"
  width="1600"
  height="1252"
/>

- 首次呼叫帶有 `revalidate` 的 `fetch` 請求時，資料會從外部資料來源獲取並儲存在資料快取中。
- 在指定時間範圍內（如 60 秒）的任何請求都會回傳快取的資料。
- 時間範圍過後，下一次請求仍會回傳快取的（此時已過期）資料。
  - Next.js 會在背景觸發資料的重新驗證。
  - 資料成功獲取後，Next.js 會用新資料更新資料快取。
  - 若背景重新驗證失敗，則保留舊資料不變。

這類似於 [**stale-while-revalidate**](https://web.dev/articles/stale-while-revalidate) 行為。

#### 按需重新驗證

資料可透過路徑 ([`revalidatePath`](#revalidatepath)) 或快取標籤 ([`revalidateTag`](#fetch-optionsnexttags-and-revalidatetag)) 按需重新驗證。

**按需重新驗證工作原理**

<Image
  alt="圖表展示按需重新驗證如何運作，重新驗證請求後，資料快取會更新為新資料。"
  srcLight="/docs/light/on-demand-revalidation.png"
  srcDark="/docs/dark/on-demand-revalidation.png"
  width="1600"
  height="1082"
/>

- 首次呼叫 `fetch` 請求時，資料會從外部資料來源獲取並儲存在資料快取中。
- 觸發按需重新驗證時，相應的快取條目會從快取中清除。
  - 這與基於時間的重新驗證不同，後者會保留過期資料直到新資料獲取完成。
- 下一次請求時會再次成為快取 `MISS`，資料會從外部資料來源獲取並儲存在資料快取中。

### 選擇退出

若您**不**想快取 `fetch` 的回應，可以這樣做：

```js
let data = await fetch('https://api.vercel.app/blog', { cache: 'no-store' })
```

## 完整路由快取 (Full Route Cache)

> **相關術語**：
>
> 您可能會看到**自動靜態優化 (Automatic Static Optimization)**、**靜態網站生成 (Static Site Generation)** 或**靜態渲染 (Static Rendering)** 這些術語交替使用，指在建置時渲染和快取應用程式路由的過程。

Next.js 會在建置時自動渲染和快取路由。這是一種優化，允許您提供快取的路由，而非在每次請求時在伺服器上渲染，從而實現更快的頁面載入。

要理解完整路由快取的工作原理，有助於了解 React 如何處理渲染，以及 Next.js 如何快取結果：

### 1. 伺服器端的 React 渲染

在伺服器端，Next.js 使用 React 的 API 來協調渲染。渲染工作被拆分為多個區塊：按個別路由區段和 Suspense 邊界。

每個區塊的渲染分為兩個步驟：

1. React 將伺服器元件渲染為一種特殊的資料格式，稱為 **React 伺服器元件負載 (React Server Component Payload)**，專為串流優化。
2. Next.js 使用 React 伺服器元件負載和客戶端元件的 JavaScript 指令在伺服器上渲染 **HTML**。

這意味著我們不必等待所有內容渲染完成後才快取工作或發送回傳。相反，我們可以在工作完成時串流回傳。

> **什麼是 React 伺服器元件負載？**
>
> React 伺服器元件負載是渲染後的 React 伺服器元件樹的緊湊二進位表示。React 在客戶端使用它來更新瀏覽器的 DOM。React 伺服器元件負載包含：
>
> - 伺服器元件的渲染結果
> - 客戶端元件應渲染位置的佔位符及其 JavaScript 檔案的參考
> - 從伺服器元件傳遞給客戶端元件的任何 props
>
> 欲了解更多，請參閱[伺服器元件](/docs/app/getting-started/server-and-client-components)文件。

### 2. Next.js 在伺服器端的快取（完整路由快取）

<Image
  alt="完整路由快取的預設行為，展示 React 伺服器元件負載和 HTML 如何在伺服器上為靜態渲染的路由快取。"
  srcLight="/docs/light/full-route-cache.png"
  srcDark="/docs/dark/full-route-cache.png"
  width="1600"
  height="888"
/>

Next.js 的預設行為是在伺服器上快取路由的渲染結果（React 伺服器元件負載和 HTML）。這適用於建置時靜態渲染的路由，或在重新驗證期間。

### 3. 客戶端的 React 水合與協調

在請求時，客戶端會：

1. 使用 HTML 立即顯示客戶端和伺服器元件的快速非互動式初始預覽。
2. 使用 React 伺服器元件負載來協調客戶端和已渲染的伺服器元件樹，並更新 DOM。
3. 使用 JavaScript 指令來[水合 (hydrate)](https://react.dev/reference/react-dom/client/hydrateRoot) 客戶端元件，使應用程式可互動。

### 4. Next.js 在客戶端的快取（路由快取）

React 伺服器元件負載儲存在客戶端的[路由快取 (Router Cache)](#client-side-router-cache) 中——一個按個別路由區段分割的獨立記憶體快取。此路由快取用於儲存先前訪問的路由和預取未來路由，以提升導航體驗。

### 5. 後續導航

在後續導航或預取時，Next.js 會檢查 React 伺服器元件負載是否儲存在路由快取中。若是，則會跳過向伺服器發送新請求。

若路由區段不在快取中，Next.js 會從伺服器獲取 React 伺服器元件負載，並在客戶端填充路由快取。

### 靜態與動態渲染

路由是否在建置階段被快取取決於它是靜態還是動態渲染。靜態路由預設會被快取，而動態路由則會在請求時渲染且不被快取。

### 持續時間

預設情況下，完整路由快取是持久性的。這表示渲染輸出會在使用者請求之間被快取。

### 失效

有兩種方式可以讓完整路由快取失效：

- **[重新驗證資料](/docs/app/deep-dive/caching#revalidating)**：重新驗證[資料快取](#data-cache)會連帶讓路由快取失效，透過在伺服器重新渲染元件並快取新的渲染輸出。
- **重新部署**：與跨部署持續存在的資料快取不同，完整路由快取會在每次新部署時被清除。

### 選擇退出

您可以選擇退出完整路由快取，換句話說，針對每個傳入請求動態渲染元件，方法如下：

- **使用[動態 API](#dynamic-apis)**：這會讓該路由退出完整路由快取，並在請求時動態渲染。資料快取仍可使用。
- **使用路由區段設定選項 `dynamic = 'force-dynamic'` 或 `revalidate = 0`**：這會跳過完整路由快取和資料快取。表示元件會在每個傳入的伺服器請求時重新渲染並獲取資料。路由快取仍會生效，因為它是客戶端快取。
- **選擇退出[資料快取](#data-cache)**：如果路由有一個未被快取的 `fetch` 請求，這會讓該路由退出完整路由快取。特定 `fetch` 請求的資料會在每個傳入請求時重新獲取。其他未選擇退出快取的 `fetch` 請求仍會被快取在資料快取中。這允許快取與未快取資料的混合使用。

## 客戶端路由快取

Next.js 有一個記憶體中的客戶端路由快取，用於儲存路由區段的 RSC 負載，按佈局、載入狀態和頁面分割。

當使用者在路由之間導航時，Next.js 會快取已造訪的路由區段，並[預取](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)使用者可能導航的路由。這會實現即時的返回/前進導航、導航間無需完整頁面重新載入，並保留 React 狀態和瀏覽器狀態。

使用路由快取：

- **佈局**會被快取並在導航時重複使用（[部分渲染](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)）。
- **載入狀態**會被快取並在導航時重複使用，以實現[即時導航](/docs/app/building-your-application/routing/loading-ui-and-streaming#instant-loading-states)。
- **頁面**預設不會被快取，但在瀏覽器後退和前進導航時會被重複使用。您可以透過實驗性的 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定選項啟用頁面區段的快取。

### 持續時間

快取儲存在瀏覽器的臨時記憶體中。兩個因素決定路由快取的持續時間：

- **工作階段**：快取在導航間持續存在，但在頁面重新載入時會被清除。
- **自動失效週期**：佈局和載入狀態的快取會在特定時間後自動失效。持續時間取決於資源如何被[預取](/docs/app/api-reference/components/link#prefetch)，以及資源是否為[靜態生成](/docs/app/getting-started/partial-prerendering#static-rendering)：
  - **預設預取** (`prefetch={null}` 或未指定)：動態頁面不會快取，靜態頁面為 5 分鐘。
  - **完整預取** (`prefetch={true}` 或 `router.prefetch`)：靜態和動態頁面均為 5 分鐘。

雖然頁面重新載入會清除**所有**快取區段，但自動失效週期僅影響從預取時間起的個別區段。

> **須知**：實驗性的 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定選項可用於調整上述的自動失效時間。

### 失效

有兩種方式可以讓路由快取失效：

- 在**伺服器動作**中：
  - 透過路徑 ([`revalidatePath`](/docs/app/api-reference/functions/revalidatePath)) 或快取標籤 ([`revalidateTag`](/docs/app/api-reference/functions/revalidateTag)) 按需重新驗證資料。
  - 使用 [`cookies.set`](/docs/app/api-reference/functions/cookies#setting-a-cookie) 或 [`cookies.delete`](/docs/app/api-reference/functions/cookies#deleting-cookies) 會讓路由快取失效，以防止使用 cookies 的路由變得過時（例如驗證）。
- 呼叫 [`router.refresh`](/docs/app/api-reference/functions/use-router) 會讓路由快取失效，並為目前路由向伺服器發送新請求。

### 選擇退出

從 Next.js 15 開始，頁面區段預設選擇退出。

> **須知**：您也可以透過將 `<Link>` 元件的 `prefetch` 屬性設為 `false` 來[選擇退出預取](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)。

## 快取互動

在設定不同的快取機制時，了解它們如何互相影響非常重要：

### 資料快取與完整路由快取

- 重新驗證或選擇退出資料快取**會**讓完整路由快取失效，因為渲染輸出依賴資料。
- 讓完整路由快取失效或選擇退出**不會**影響資料快取。您可以動態渲染一個同時包含快取和未快取資料的路由。這在頁面大部分使用快取資料，但有少數元件依賴需在請求時獲取的資料時非常有用。您可以動態渲染，無需擔心重新獲取所有資料對效能的影響。

### 資料快取與客戶端路由快取

- 要立即讓資料快取和路由快取失效，您可以在[伺服器動作](/docs/app/building-your-application/data-fetching/server-actions-and-mutations)中使用 [`revalidatePath`](#revalidatepath) 或 [`revalidateTag`](#fetch-optionsnexttags-and-revalidatetag)。
- 在[路由處理程式](/docs/app/building-your-application/routing/route-handlers)中重新驗證資料快取**不會**立即讓路由快取失效，因為路由處理程式不綁定特定路由。這表示路由快取會繼續提供先前的負載，直到強制重新載入或自動失效週期結束。

### `<Link>`

預設情況下，`<Link>` 元件會自動從完整路由快取預取路由，並將 React 伺服器元件負載加入路由快取。

要停用預取，您可以將 `prefetch` 屬性設為 `false`。但這不會永久跳過快取，當使用者造訪路由時，路由區段仍會在客戶端被快取。

深入了解 [`<Link>` 元件](/docs/app/api-reference/components/link)。

### `router.prefetch`

`useRouter` 鉤子的 `prefetch` 選項可用於手動預取路由。這會將 React 伺服器元件負載加入路由快取。

參閱 [`useRouter` 鉤子](/docs/app/api-reference/functions/use-router) API 參考。

### `router.refresh`

`useRouter` 鉤子的 `refresh` 選項可用於手動重新整理路由。這會完全清除路由快取，並為目前路由向伺服器發送新請求。`refresh` 不會影響資料或完整路由快取。

渲染結果會在客戶端進行協調，同時保留 React 狀態和瀏覽器狀態。

參閱 [`useRouter` 鉤子](/docs/app/api-reference/functions/use-router) API 參考。

### `fetch`

從 `fetch` 回傳的資料**不會**自動快取在資料快取中。

`fetch` 的預設快取行為（例如未指定 `cache` 選項時）等同於將 `cache` 選項設為 `no-store`：

```js
let data = await fetch('https://api.vercel.app/blog', { cache: 'no-store' })
```

參閱 [`fetch` API 參考](/docs/app/api-reference/functions/fetch) 以獲取更多選項。

### `fetch options.cache`

您可以透過將 `cache` 選項設為 `force-cache` 來讓個別 `fetch` 選擇加入快取：

```jsx
// 選擇加入快取
fetch(`https://...`, { cache: 'force-cache' })
```

參閱 [`fetch` API 參考](/docs/app/api-reference/functions/fetch) 以獲取更多選項。

### `fetch options.next.revalidate`

您可以使用 `fetch` 的 `next.revalidate` 選項來設定個別 `fetch` 請求的重新驗證週期（以秒為單位）。這會重新驗證資料快取，進而重新驗證完整路由快取。會獲取新資料，並在伺服器重新渲染元件。

```jsx
// 最多 1 小時後重新驗證
fetch(`https://...`, { next: { revalidate: 3600 } })
```

參閱 [`fetch` API 參考](/docs/app/api-reference/functions/fetch) 以獲取更多選項。

### `fetch options.next.tags` 與 `revalidateTag`

Next.js 有一個快取標籤系統，用於細粒度的資料快取與重新驗證。

1. 使用 `fetch` 或 [`unstable_cache`](/docs/app/api-reference/functions/unstable_cache) 時，您可以選擇用一個或多個標籤標記快取項目。
2. 接著，您可以呼叫 `revalidateTag` 來清除與該標籤相關的快取項目。

例如，您可以在獲取資料時設定標籤：

```jsx
// 使用標籤快取資料
fetch(`https://...`, { next: { tags: ['a', 'b', 'c'] } })
```

然後，呼叫 `revalidateTag` 並帶入標籤來清除快取項目：

```jsx
// 重新驗證帶有特定標籤的項目
revalidateTag('a')
```

根據您想達到的目標，有兩個地方可以使用 `revalidateTag`：

1. [路由處理程式](/docs/app/building-your-application/routing/route-handlers) - 在第三方事件（例如 webhook）回應中重新驗證資料。這不會立即讓路由快取失效，因為路由處理程式不綁定特定路由。
2. [伺服器動作](/docs/app/building-your-application/data-fetching/server-actions-and-mutations) - 在使用者動作（例如表單提交）後重新驗證資料。這會讓相關路由的路由快取失效。

### `revalidatePath`

`revalidatePath` 允許您手動重新驗證資料**並**在單一操作中重新渲染特定路徑下的路由區段。呼叫 `revalidatePath` 方法會重新驗證資料快取，進而讓完整路由快取失效。

```jsx
revalidatePath('/')
```

根據您想達到的目標，有兩個地方可以使用 `revalidatePath`：

1. [路由處理程式](/docs/app/building-your-application/routing/route-handlers) - 在第三方事件（例如 webhook）回應中重新驗證資料。
2. [伺服器動作](/docs/app/building-your-application/data-fetching/server-actions-and-mutations) - 在使用者互動（例如表單提交、點擊按鈕）後重新驗證資料。

參閱 [`revalidatePath` API 參考](/docs/app/api-reference/functions/revalidatePath) 以獲取更多資訊。

> **`revalidatePath`** 與 **`router.refresh`** 的差異：
>
> 呼叫 `router.refresh` 會清除路由快取，並在伺服器重新渲染路由區段，而不會讓資料快取或完整路由快取失效。
>
> 差異在於 `revalidatePath` 會清除資料快取和完整路由快取，而 `router.refresh()` 不會變更資料快取和完整路由快取，因為它是客戶端 API。

### 動態 API

像 `cookies` 和 `headers` 這樣的動態 API，以及頁面中的 `searchParams` 屬性，依賴於運行時傳入的請求資訊。使用它們會讓路由退出完整路由快取，換句話說，路由會動態渲染。

#### `cookies`

在伺服器動作中使用 `cookies.set` 或 `cookies.delete` 會讓路由快取失效，以防止使用 cookies 的路由變得過時（例如反映驗證變更）。

參閱 [`cookies`](/docs/app/api-reference/functions/cookies) API 參考。

### 路由區段設定選項

路由區段設定選項可用於覆寫預設的路由區段設定，或是當您無法使用 `fetch` API 時（例如使用資料庫客戶端或第三方函式庫）。

以下路由區段設定選項將會排除完整路由快取：

- `const dynamic = 'force-dynamic'`

此設定選項會讓所有擷取操作排除資料快取（即 `no-store`）：

- `const fetchCache = 'default-no-store'`

查看 [`fetchCache`](/docs/app/api-reference/file-conventions/route-segment-config#fetchcache) 以瞭解更多進階選項。

查看[路由區段設定](/docs/app/api-reference/file-conventions/route-segment-config)文件以取得更多選項。

### `generateStaticParams`

對於[動態區段](/docs/app/building-your-application/routing/dynamic-routes)（例如 `app/blog/[slug]/page.js`），由 `generateStaticParams` 提供的路徑會在建置時快取至完整路由快取中。在請求時，Next.js 也會在首次造訪時快取建置階段未知的路徑。

若要在建置時靜態渲染所有路徑，請提供完整路徑清單給 `generateStaticParams`：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))
}
```

若要在建置時靜態渲染部分路徑，並在執行階段首次造訪時渲染其餘路徑，請回傳部分路徑清單：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  // 在建置時渲染前 10 篇文章
  return posts.slice(0, 10).map((post) => ({
    slug: post.slug,
  }))
}
```

若要在首次造訪時靜態渲染所有路徑，請回傳空陣列（建置時不會渲染任何路徑）或使用 [`export const dynamic = 'force-static'`](/docs/app/api-reference/file-conventions/route-segment-config#dynamic)：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  return []
}
```

> **須知：** 您必須從 `generateStaticParams` 回傳一個陣列，即使是空陣列。否則，路由將會被動態渲染。

```jsx filename="app/changelog/[slug]/page.js"
export const dynamic = 'force-static'
```

若要停用請求時的快取，請在路由區段中加入 `export const dynamicParams = false` 選項。使用此設定選項時，僅會提供由 `generateStaticParams` 產生的路徑，其他路由將會回傳 404 或匹配（在[萬用路由](/docs/app/building-your-application/routing/dynamic-routes#catch-all-segments)的情況下）。

### React `cache` 函式

React `cache` 函式可讓您記憶化函式的回傳值，讓您能多次呼叫相同函式卻只執行一次。

由於 `fetch` 請求會自動記憶化，您不需要將其包裹在 React `cache` 中。然而，當 `fetch` API 不適用時，您可以使用 `cache` 手動記憶化資料請求。例如某些資料庫客戶端、CMS 客戶端或 GraphQL 客戶端。

```ts filename="utils/get-item.ts" switcher
import { cache } from 'react'
import db from '@/lib/db'

export const getItem = cache(async (id: string) => {
  const item = await db.item.findUnique({ id })
  return item
})
```

```js filename="utils/get-item.js" switcher
import { cache } from 'react'
import db from '@/lib/db'

export const getItem = cache(async (id) => {
  const item = await db.item.findUnique({ id })
  return item
})
```
