`unstable_rethrow` 可用於避免在處理應用程式程式碼拋出的錯誤時，捕捉到 Next.js 的內部錯誤。

例如，呼叫 `notFound` 函式會拋出 Next.js 內部錯誤並渲染 [`not-found.js`](/docs/app/api-reference/file-conventions/not-found) 元件。然而，如果在 `try/catch` 區塊中使用，該錯誤會被捕捉，導致 `not-found.js` 無法渲染：

```tsx filename="@/app/ui/component.tsx"
import { notFound } from 'next/navigation'

export default async function Page() {
  try {
    const post = await fetch('https://.../posts/1').then((res) => {
      if (res.status === 404) notFound()
      if (!res.ok) throw new Error(res.statusText)
      return res.json()
    })
  } catch (err) {
    console.error(err)
  }
}
```

您可以使用 `unstable_rethrow` API 重新拋出內部錯誤，讓預期行為繼續執行：

```tsx filename="@/app/ui/component.tsx"
import { notFound, unstable_rethrow } from 'next/navigation'

export default async function Page() {
  try {
    const post = await fetch('https://.../posts/1').then((res) => {
      if (res.status === 404) notFound()
      if (!res.ok) throw new Error(res.statusText)
      return res.json()
    })
  } catch (err) {
    unstable_rethrow(err)
    console.error(err)
  }
}
```

以下 Next.js API 依賴於拋出錯誤，這些錯誤應由 Next.js 本身重新拋出和處理：

- [`notFound()`](/docs/app/api-reference/functions/not-found)
- [`redirect()`](/docs/app/building-your-application/routing/redirecting#redirect-function)
- [`permanentRedirect()`](/docs/app/building-your-application/routing/redirecting#permanentredirect-function)

如果路由段被標記為除非是靜態否則會拋出錯誤，動態 API 呼叫也會拋出開發者不應捕捉的錯誤。請注意，部分預渲染 (PPR) 也會影響此行為。這些 API 包括：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional)
- `fetch(..., { cache: 'no-store' })`
- `fetch(..., { next: { revalidate: 0 } })`

> **須知**：
>
> - 此方法應在 catch 區塊的頂部呼叫，並將錯誤物件作為其唯一參數傳遞。它也可以在 Promise 的 `.catch` 處理程序中使用。
> - 如果您確保呼叫會拋出錯誤的 API 時未包裹在 try/catch 中，則不需要使用 `unstable_rethrow`
> - 任何資源清理（如清除間隔、計時器等）必須在呼叫 `unstable_rethrow` 之前或在 `finally` 區塊中進行。
