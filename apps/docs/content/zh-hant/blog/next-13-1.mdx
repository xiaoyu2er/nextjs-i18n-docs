[`app` 目錄的可靠性與支援改進](#improved-reliability-and-support-for-app-directory)
-----------------------------------------------------------------------------------------------------------

在 Next.js 13 中，我們推出了新的 `app` 目錄（測試版）。這個新的路由和資料獲取系統可以與現有的 `pages` 目錄逐步共存使用。

`app` 目錄提供了許多優勢，包括增強版佈局(layouts)、元件與測試和樣式的共置(co-location)、元件級資料獲取等。感謝您的回饋和早期測試，我們針對 `app` 目錄的可靠性進行了多項改進：

*   **不再有佈局 div 元素：** 先前，`app` 目錄會添加額外的 `<div>` 元素以便在導航時將佈局滾動至可見區域。在 13.1 版本中，不再產生這些額外元素，同時保留了滾動行為。
*   **TypeScript 插件：** 我們開發了一個新的 TypeScript 插件，提供頁面和佈局配置選項的建議，將文件直接整合到您的 IDE 中，並針對伺服器元件(Server Components)和客戶端元件(Client Components)提供使用提示（例如防止在伺服器元件中使用 `useState`）。[了解更多](https://www.youtube.com/watch?v=pqMqn9fKEf8)。
*   **可靠性改進：** 我們修復了許多錯誤，包括改進 CSS 模組支援、正確為佈局和頁面去重複製 `cache()` 和 `fetch()`、記憶體洩漏等問題。
*   **更少的客戶端 JavaScript：** `app` 目錄現在比 `pages` 目錄減少了 `9.3kB` 的客戶端 JavaScript。這個基準不會因您添加 1 個或 1000 個伺服器元件到應用中而增加。React 運行時暫時稍微變大，這是由 React 伺服器元件運行時引起的，它處理了 Next.js 先前處理的機制。我們正在努力進一步減少這部分。

[內建模組轉譯（穩定版）](#built-in-module-transpilation-stable)
-------------------------------------------------------------------------------

現在您可以標記來自本地套件（如 monorepos）或外部依賴項（`node_modules`）的依賴項進行轉譯和打包。這項內建支援取代了流行的 `next-transpile-modules` 套件。

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ['@acme/ui', 'lodash-es'],
};
 
module.exports = nextConfig;
```

我們感謝 Pierre de la Martinière（[@martpie](https://twitter.com/martpie_)）對這個套件的貢獻以及協助確保內建支援滿足社群需求。

[導入解析以產生更小的打包檔](#import-resolution-for-smaller-bundles)
-------------------------------------------------------------------------------

許多流行的 npm 套件使用「barrel 檔案」來提供一個重新導出其他模組的單一檔案。例如：

```ts filename="@acme/ui/index.ts"
export { default as Button } from './dist/Button';
export { default as Slider } from './dist/Slider';
export { default as Dropdown } from './dist/Dropdown';
```

這使得套件的使用者可以在一行程式碼中使用具名導出：

```
import { Button, Slider, Dropdown } from '@acme/ui';
```

雖然打包工具理解這些 barrel 檔案並可以移除未使用的重新導出（稱為「死碼消除」），但這個過程涉及解析/編譯所有重新導出的檔案。對於已發布的函式庫，某些 npm 套件提供的 barrel 檔案可能重新導出了數千個模組，這會減慢編譯時間。這些函式庫建議使用 `babel-plugin-transform-imports` 來避免這個問題，但對於使用 SWC 的使用者，之前沒有支援。我們在 Next.js 中新增了一個稱為 `modularizeImports` 的內建 SWC 轉換功能。

這個新設定啟用 SWC 轉換，根據定義的模式變更您的導入語句。例如，上述使用三個元件的程式碼將自動轉換為使用直接導入，開發者無需手動編寫這段程式碼：

```
// 轉換前（使用 barrel 檔案）
import { Button, Slider, Dropdown } from '@acme/ui';
 
// 轉換後（使用插件的模組化導入）
import Button from '@acme/ui/dist/Button';
import Slider from '@acme/ui/dist/Slider';
import Dropdown from '@acme/ui/dist/Dropdown';
```

這項轉換可透過 `next.config.js` 中的 `modularizeImports` 選項實現：

```js filename="next.config.js"
module.exports = {
  modularizeImports: {
    '@acme/ui': {
      transform: '@acme/ui/dist/{{member}}',
    },
  },
};
```

利用這項轉換功能搭配 `@mui/icons-material` 或 `lodash` 可以跳過未使用檔案的編譯。[了解更多](/docs/architecture/nextjs-compiler#modularize-imports)。

[查看示範](https://twitter.com/nextjs/status/1610288783311732737)以實際了解其運作方式。

[適用於邊緣的輕量 Node.js 運行時，現已穩定支援 API 路由](#a-light-nodejs-runtime-for-the-edge-now-stable-for-api-routes)
---------------------------------------------------------------------------------------------------------------------------------

Next.js 中的邊緣運行時(Edge Runtime)使用 Node.js API 的嚴格子集（如 `Request`、`Response` 等），這些 API 與 Vercel 等邊緣計算平台或自行託管環境相容。這些 API 可在任何地方運行，包括瀏覽器，讓開發者學習一次即可到處使用。

```ts filename="pages/api/hello.ts"
// 不再需要 "experimental-" 前綴
export const config = {
  runtime: 'edge',
};
 
export default function handler(req: Request) {
  return new Response('Hello World');
}
```

Next.js 中介軟體(Middleware)預設已使用這個輕量邊緣運行時以獲得更好的效能。由於中介軟體可以在應用程式的每個請求之前運行，擁有一個輕量運行時對於確保低延遲至關重要。在 Next.js 12.2 中，我們新增了可選使用此運行時於 [API 路由](https://nextjs.org/blog/next-12-2#edge-api-routes-experimental)的功能。

隨著 13.1 版本的發布，Next.js 中的邊緣運行時現已**穩定**支援 API 路由。在自行託管時，使用邊緣運行時的中介軟體和 API 路由預設將作為單一區域工作負載運行，作為 `next start` 的一部分。在 Vercel 上，Next.js 中介軟體和 API 路由會[使用 Vercel 邊緣函式(Edge Functions)進行全球部署](https://vercel.com/docs/concepts/next.js/overview)，以實現最低延遲。Vercel 邊緣函式也已[正式全面可用](https://vercel.com/blog/edge-functions-generally-available)。

[Turbopack 改進](#turbopack-improvements)
-------------------------------------------------

在 Next.js 13 中發布 [Turbopack](https://nextjs.org/docs/app/api-reference/turbopack) alpha 版本後，我們一直專注於提升可靠性、增加對最常請求功能的支援，並制定在其他框架中使用插件和 Turbopack 的計劃。

自 Next.js 13.0.0 以來，Turbopack：

*   支援 PostCSS，包括 Tailwind CSS
*   支援 `next/image`
*   支援 `@next/font`（Google 字體）
*   支援從動態 `import()` 語句載入 CSS
*   支援 CSS 原始碼映射（感謝 `@ahabhgk` 的[貢獻](https://github.com/vercel/turbo/pull/2839)）
*   改進 `next dev` 錯誤覆蓋層的錯誤處理
*   改進記憶體使用
*   改進 CSS 模組支援
*   改進 HMR 更新的分塊演算法
*   改進 HMR 原始碼映射的可靠性

我們感謝 Evan You 和 Vite 社群的回饋和貢獻，確保 Turbopack 基準測試盡可能準確。我們與 Vite 團隊合作驗證最新的 Turbopack 基準測試，並對我們的測試方法進行了多項改進。

作為這次合作的結果，我們現在使用更準確的指標，包括 React 更新機制所花費的時間。我們能夠將 Turbopack 以及 Next.js 13.1 在 webpack 上的 React 快速刷新(Fast Refresh)時間改善 30 毫秒。我們還新增了一個使用 SWC 的 Vite 新基準測試，顯示與預設使用 Babel 的 Vite 相比效能有所提升。查看[更新後的基準測試](https://nextjs.org/docs/app/api-reference/turbopack)或閱讀[測試方法](https://nextjs.org/docs/app/api-reference/turbopack)。

立即在 Next.js 13 中使用 `next dev --turbo` 試用 Turbopack alpha 版本。如果您有任何回饋，請在 [GitHub 討論區](https://github.com/vercel/turbo/discussions/2312)告訴我們。

[Next.js 進階中介軟體](#nextjs-advanced-middleware)
----------------------------------------------------------

感謝您的回饋，我們讓 Next.js 中介軟體(Middleware)比以往更強大。在 13.1 版本中，您現在可以從中介軟體返回回應，以及在請求上設定標頭。

這些 API 改進為您提供了強大的新靈活性，可自訂 Next.js 路由生命週期的每個部分。`next.config.js` 中的 `experimental.allowMiddlewareResponseBody` 配置選項不再需要。

您現在可以更輕鬆地在請求上設定標頭，以及直接回應而無需 `rewrite` 或 `redirect`：

```ts filename="middleware.ts"
import { NextResponse } from 'next/server';
 
export function middleware(request: Request) {
  // 檢查使用者是否有權限...
  if (!isAuthorized(request)) {
    return NextResponse.json({ message: 'Unauthorized' });
  }
 
  // 新增標頭，這將變更傳入請求標頭
  // 您可以在 getServerSideProps 和 API 路由中讀取
  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-version', '13.1');
 
  return NextResponse.next({
    request: {
      // 套用新的請求標頭
      headers: requestHeaders,
    },
  });
}
```

了解更多關於 [Next.js 進階中介軟體](/docs/pages/building-your-application/routing/middleware)。
