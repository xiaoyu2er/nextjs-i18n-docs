[Webpack 4](#webpack-4)
-----------------------

從最初版本開始，Next.js 就採用 webpack 來打包您的程式碼，並重用豐富的插件與擴充生態系統。我們很興奮地宣布，Next.js 現在由最新的 webpack 4 驅動，帶來眾多改進與錯誤修正。

其中包括：

*   支援 `.mjs` 原始檔
*   程式碼分割改進
*   更好的 tree-shaking（移除未使用程式碼）支援

另一項新功能是 WebAssembly 支援，Next.js 甚至能伺服器渲染 WebAssembly，這裡有個[範例](https://github.com/vercel/next.js/tree/canary/examples/with-webassembly)。

**注意：** 此升級**完全向下相容**。然而，如果您透過 [next.config.js](/docs#customizing-webpack-config) 使用自訂 webpack 載入器或插件，可能需要升級它們。

### [CSS 導入](#css-imports)

隨著 webpack 4 的推出，引入了一種從套件中提取 CSS 的新方法，稱為 [mini-extract-css-plugin](https://github.com/webpack-contrib/mini-css-extract-plugin)。

[`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`@zeit/next-less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less)、[`@zeit/next-sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) 和 [`@zeit/next-stylus`](https://github.com/vercel/next-plugins/tree/master/packages/next-stylus) 現在都由 `mini-extract-css-plugin` 驅動。

這些 Next.js 插件的新版本**解決了 20 個與 CSS 導入相關的現有問題**；例如，現在支援在動態 `import()` 中導入 CSS：

```js filename="components/my-dynamic-component.js"
import './my-dynamic-component.css';
 
export default function MyDynamicComponent() {
  return <h1>我的動態組件</h1>;
}
```

```js filename="pages/index.js"
import dynamic from 'next/dynamic'
 
const MyDynamicComponent = dynamic(import('../components/my-dynamic-component'))
 
export default function Index() {
  return () {
    <div>
      <MyDynamicComponent/>
    </div>
  }
}
```

一項重大改進是您不再需要在 `pages/_document.js` 中添加以下內容：

```
<link rel="stylesheet" href="/_next/static/style.css" />
```

取而代之的是，Next.js 會自動注入 CSS 檔案。在生產環境中，Next.js 還會自動在 CSS URL 中添加內容哈希值，以確保您的終端用戶永遠不會獲得過時的檔案版本，並能實現不可變的永久快取。

簡而言之，要在 Next.js 專案中支援導入 `.css` 檔案，您只需在 `next.config.js` 中**註冊 withCSS 插件**：

```
const withCSS = require('@zeit/next-css');
module.exports = withCSS({
  /* 我的 next 配置 */
});
```

### [標準化動態導入](#standardized-dynamic-imports)

Next.js 從第 3 版開始就透過 `next/dynamic` 支援動態導入。

作為這項技術的早期採用者，我們必須自行編寫解決方案來處理 `import()`。

因此，Next.js 開始與 webpack 後續對其的支援產生分歧，且缺少某些功能。

由於這個原因，Next.js 不支援 webpack 後來引入的一些 `import()` 功能。

例如，無法手動為特定檔案命名並將它們打包在一起：

```
import(/* webpackChunkName: 'my-chunk' */ '../lib/my-library');
```

另一個例子是使用 `import()` 時沒有將其包裹在 `next/dynamic` 模組中。

從 Next.js 7 開始，我們不再修改預設的 `import()` 行為。這意味著您能**直接獲得完整的 import() 支援**。

此變更也完全向下相容。使用動態組件仍然非常簡單：

```js filename="pages/index.js"
import dynamic from 'next/dynamic';
 
const MyComponent = dynamic(import('../components/my-component'));
 
export default function Index() {
  return (
    <div>
      <MyComponent />
    </div>
  );
}
```

此範例會為 `my-component` 創建一個新的 JavaScript 檔案，並僅在 `<MyComponent />` 渲染時載入它。

最重要的是，如果它沒有被渲染，`<script>` 標籤**不會包含在初始 HTML 文件負載中**。

為了進一步簡化我們的程式碼庫並利用優秀的 React 生態系統，Next.js 7 中的 `next/dynamic` 被重寫為在底層使用 [`react-loadable`](https://github.com/jamiebuilds/react-loadable)（並進行了一些小修改）。這也為動態組件引入了兩項出色的新功能：

*   使用 `next/dynamic` 的 `timeout` 選項設定超時
*   使用 `next/dynamic` 的 `delay` 選項設定載入組件的延遲。此延遲讓您的 `loading` 組件能在渲染載入狀態前等待 x 時間，例如當導入速度非常快時。

[Babel 7](#babel-7)
-------------------

Next.js 6 在 Babel 7 仍處於測試階段時就引入了它。自那時起，Babel 7 的穩定版本已經發布，而 Next.js 7 現在使用此版本。

有關所有變更的完整列表，您可以參考 Babel 的[發布說明](http://babeljs.io/blog/2018/08/27/7.0.0)。

一些主要功能包括：

*   支援 Typescript，對於 Next.js 您可以使用 [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript)
*   支援片段語法 `<>`
*   支援 `babel.config.js`
*   `overrides` 屬性，僅對部分檔案或目錄應用預設集/插件

如果您的 Next.js 專案中沒有自訂 Babel 配置，則沒有破壞性變更。

但如果您有自訂 Babel 配置，則需要將相應的自訂插件/預設集升級至最新版本。

如果您是從 Next.js 6 以下版本升級，可以執行優秀的 [`babel-upgrade`](http://babeljs.io/blog/2018/08/27/7.0.0#babel-upgrade) 工具。

除了升級至 Babel 7 外，Next.js Babel 預設集（[`next/babel`](https://github.com/vercel/next.js#customizing-babel-config)）現在預設在 `NODE_ENV` 設為 `test` 時將 `modules` 選項設為 `commonjs`。

此配置選項通常是 Next.js 專案中創建自訂 `.babelrc` 的唯一原因：

```bash filename=".babelrc"
{
  "env": {
    "development": {
      "presets": ["next/babel"]
    },
    "production": {
      "presets": ["next/babel"]
    },
    "test": {
      "presets": [["next/babel", { "preset-env": { "modules": "commonjs" } }]]
    }
  }
}
```

在 Next.js 7 中，這變為：

```bash filename=".babelrc"
{
  "presets": ["next/babel"]
}
```

此時，可以移除 `.babelrc`，因為 Next.js 在沒有 Babel 配置時會自動使用 `next/babel`。

[更小的初始 HTML 負載](#smaller-initial-html-payload)
-------------------------------------------------------------

由於 Next.js 會預渲染 HTML，它會將頁面包裹在預設結構中，包含 `<html>`、`<head>`、`<body>` 以及渲染頁面所需的 JavaScript 檔案。

此初始負載先前約為 1.62kB。在 Next.js 7 中，我們優化了初始 HTML 負載，現在為 1.5kB，減少了 7.4%，讓您的頁面更精簡。

*   當未使用時，編譯掉未使用的 `__NEXT_DATA__` 屬性，例如 `nextExport` 和 `assetPrefix` 屬性。

[靜態 CDN 支援](#static-cdn-support)
-----------------------------------------

在 Next.js 5 中，我們引入了 `assetPrefix` 支援，這是一種讓 Next.js 自動從特定位置（通常是 CDN）載入資源的方式。如果您的 CDN 支援代理，此選項效果很好，您會請求像這樣的 URL：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
```

通常，CDN 會檢查其快取中是否有該檔案，否則直接從來源請求。

代理資源正是 [Edge Network](https://vercel.com/edge-network) 的運作方式。

然而，有些解決方案需要直接將目錄預先上傳至 CDN。這樣做的問題在於 Next.js 的 URL 結構與 `.next` 資料夾內的結構不匹配。例如我們之前的範例：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// 對應至：
.next/page/index.js
```

在 Next.js 7 中，我們已變更 `.next` 的目錄結構以匹配 URL 結構：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// 對應至：
.next/static/<buildid>/pages/index.js
```

雖然我們推薦使用代理類型的 CDN，但此新結構允許使用不同類型 CDN 的用戶將 `.next` 目錄上傳至其 CDN。

[Styled JSX v3](#styled-jsx-v3)
-------------------------------

我們很興奮地推出 styled-jsx 3，Next.js 中預設包含的 CSS-in-JS 解決方案現在已準備好支援 [React Suspense](https://www.youtube.com/watch?v=z-6JC0_cOns)。

一個常見的困惑是如何為非當前組件範圍的子組件設定樣式，例如當您包含一個子組件，且該子組件僅在父組件中使用時需要特定樣式：

```js filename="pages/index.js"
const ChildComponent = () => (
  <div>
    <p>一些文字</p>
  </div>
);
 
export default function Index() {
  return (
    <div>
      <ChildComponent />
      <style jsx>{`
        p {
          color: black;
        }
      `}</style>
    </div>
  );
}
```

上述程式碼嘗試選擇 `p` 標籤但無效，因為 styled-jsx 的樣式僅限於當前組件，它們不會洩漏至子組件。解決此問題的一種方法是使用 `:global` 方法，移除 `p` 標籤的前綴。然而，這會引入新問題，即樣式會洩漏至整個頁面。

在 styled-jsx 3 中，此問題已透過引入新 API `css.resolve` 解決，它會為給定的 styled-jsx 字串生成 `className` 和 `<style>` 標籤（`styles` 屬性）：

```js filename="pages/index.js"
import css from 'styled-jsx/css';
 
const ChildComponent = ({ className }) => (
  <div>
    <p className={className}>一些文字</p>
  </div>
);
 
const { className, styles } = css.resolve`
  p {
    color: black;
  }
`;
 
export default function Index() {
  return (
    <div>
      <ChildComponent className={className} />
      {styles}
    </div>
  );
}
```

此新 API 允許透明地將自訂樣式傳遞至子組件。

由於這是 styled-jsx 的重大版本發布，如果您使用 `styles-jsx/css`，有一項破壞性變更能改善套件大小。在 styled-jsx 2 中，我們會生成外部樣式的「scoped」和「global」版本，即使僅使用「scoped」版本，我們仍會包含這些外部樣式的「global」版本。

在 styled-jsx 3 中，全域樣式必須使用 `css.global` 而非 `css` 標記，以便 styled-jsx 能優化套件大小。

有關所有變更，請參閱[發布說明](https://github.com/vercel/styled-jsx/releases/tag/v3.0.0)。

[App 與 Pages 間的 SSR React Context 支援](#react-context-with-ssr-between-app-and-pages)
---------------------------------------------------------------------------------------------

從 Next.js 7 開始，我們現在支援在 `pages/_app.js` 與頁面組件間使用新的 React context API。

先前在伺服器端無法在頁面間使用 React context。原因是 webpack 使用內部模組快取而非 require.cache，我們編寫了一個自訂 webpack 插件來變更此行為，以在頁面間共享模組實例。
