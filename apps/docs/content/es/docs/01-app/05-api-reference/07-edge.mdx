Next.js tiene dos entornos de ejecución (runtimes) de servidor que puedes usar en tu aplicación:

- El **Entorno de ejecución Node.js** (predeterminado), que tiene acceso a todas las APIs de Node.js y se usa para renderizar tu aplicación.
- El **Edge Runtime** que contiene un [conjunto más limitado de APIs](#reference), usado en [Middleware](/docs/app/api-reference/file-conventions/middleware).

## Consideraciones

- El Edge Runtime no soporta todas las APIs de Node.js. Algunos paquetes pueden no funcionar como se espera.
- El Edge Runtime no soporta Regeneración Estática Incremental (ISR).
- Ambos entornos de ejecución pueden soportar [streaming](/docs/app/api-reference/file-conventions/loading) dependiendo de tu adaptador de despliegue.

### APIs no soportadas

El Runtime de Edge tiene algunas restricciones, incluyendo:

- Las APIs nativas de Node.js **no son compatibles**. Por ejemplo, no puede leer ni escribir en el sistema de archivos.
- Se _pueden_ usar `node_modules`, siempre que implementen Módulos ES y no utilicen APIs nativas de Node.js.
- Llamar directamente a `require` **no está permitido**. Utilice Módulos ES en su lugar.

Las siguientes características del lenguaje JavaScript están deshabilitadas y **no funcionarán:**

| API                                                                                                                             | Descripción                                                         |
| ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| [`eval`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/eval)                                       | Evalúa código JavaScript representado como una cadena              |
| [`new Function(evalString)`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Function)               | Crea una nueva función con el código proporcionado como argumento  |
| [`WebAssembly.compile`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/WebAssembly/compile)         | Compila un módulo WebAssembly desde una fuente de buffer           |
| [`WebAssembly.instantiate`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/WebAssembly/instantiate) | Compila e instancia un módulo WebAssembly desde una fuente de buffer |

En casos raros, su código podría contener (o importar) algunas sentencias de evaluación dinámica de código que _no se pueden alcanzar en tiempo de ejecución_ y que no pueden eliminarse mediante treeshaking.
Puede relajar la verificación para permitir archivos específicos con la configuración de su Middleware:

```javascript filename="middleware.ts"
export const config = {
  unstable_allowDynamic: [
    // permite un solo archivo
    '/lib/utilities.js',
    // usa un glob para permitir cualquier cosa en el módulo de terceros function-bind
    '**/node_modules/function-bind/**',
  ],
}
```

`unstable_allowDynamic` es un [glob](https://github.com/micromatch/micromatch#matching-features), o un array de globs, que ignora la evaluación dinámica de código para archivos específicos. Los globs son relativos a la carpeta raíz de su aplicación.

Tenga en cuenta que si estas sentencias se ejecutan en el Edge, _lanzarán un error y causarán un fallo en tiempo de ejecución_.
