Antes de llevar tu aplicación Next.js a producción, hay algunas optimizaciones y patrones que deberías considerar implementar para la mejor experiencia de usuario, rendimiento y seguridad.

Esta página proporciona buenas prácticas que puedes usar como referencia cuando [construyas tu aplicación](#during-development) y [antes de ir a producción](#before-going-to-production), así como las [optimizaciones automáticas de Next.js](#automatic-optimizations) que deberías conocer.

## Optimizaciones automáticas

Estas optimizaciones de Next.js están habilitadas por defecto y no requieren configuración:

<AppOnly>

- **[Componentes del servidor (Server Components)](/docs/app/getting-started/server-and-client-components):** Next.js usa Componentes del Servidor por defecto. Los Componentes del Servidor se ejecutan en el servidor y no requieren JavaScript para renderizarse en el cliente. Por lo tanto, no impactan el tamaño de tus paquetes JavaScript del lado del cliente. Puedes usar [Componentes del Cliente (Client Components)](/docs/app/getting-started/server-and-client-components) según sea necesario para interactividad.
- **[División de código (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** Los Componentes del Servidor permiten la división automática de código por segmentos de ruta. También puedes considerar [carga diferida (lazy loading)](/docs/app/guides/lazy-loading) de Componentes del Cliente y bibliotecas de terceros cuando sea apropiado.
- **[Precarga (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** Cuando un enlace a una nueva ruta entra en el viewport del usuario, Next.js precarga la ruta en segundo plano. Esto hace que la navegación a nuevas rutas sea casi instantánea. Puedes optar por no usar precarga cuando sea apropiado.
- **[Renderizado estático (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js renderiza estáticamente Componentes del Servidor y del Cliente en el servidor durante el tiempo de construcción y almacena en caché el resultado para mejorar el rendimiento de tu aplicación. Puedes optar por [Renderizado dinámico (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) para rutas específicas cuando sea apropiado. {/* TODO: Actualizar cuando PPR sea estable */}
- **[Almacenamiento en caché (Caching)](/docs/app/deep-dive/caching):** Next.js almacena en caché solicitudes de datos, el resultado renderizado de Componentes del Servidor y del Cliente, activos estáticos y más, para reducir el número de solicitudes de red a tu servidor, base de datos y servicios backend. Puedes optar por no usar caché cuando sea apropiado.

</AppOnly>

<PagesOnly>

- **[División de código (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js divide automáticamente el código de tu aplicación por páginas. Esto significa que solo se carga el código necesario para la página actual durante la navegación. También puedes considerar [carga diferida (lazy loading)](/docs/pages/guides/lazy-loading) de bibliotecas de terceros cuando sea apropiado.
- **[Precarga (Prefetching)](/docs/pages/api-reference/components/link#prefetch):** Cuando un enlace a una nueva ruta entra en el viewport del usuario, Next.js precarga la ruta en segundo plano. Esto hace que la navegación a nuevas rutas sea casi instantánea. Puedes optar por no usar precarga cuando sea apropiado.
- **[Optimización estática automática (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** Next.js determina automáticamente si una página es estática (puede ser prerrenderizada) si no tiene requisitos de datos bloqueantes. Las páginas optimizadas pueden almacenarse en caché y servirse al usuario final desde múltiples ubicaciones CDN. Puedes optar por [Renderizado del lado del servidor (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props) cuando sea apropiado.

</PagesOnly>

Estos valores predeterminados buscan mejorar el rendimiento de tu aplicación y reducir el costo y la cantidad de datos transferidos en cada solicitud de red.

## Durante el desarrollo

Mientras construyes tu aplicación, recomendamos usar las siguientes características para garantizar el mejor rendimiento y experiencia de usuario:

### Enrutamiento y renderizado

<AppOnly>

- **[Diseños (Layouts)](/docs/app/api-reference/file-conventions/layout):** Usa diseños para compartir UI entre páginas y habilitar [renderizado parcial (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering) en la navegación.
- **[Componente `<Link>`](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** Usa el componente `<Link>` para [navegación del lado del cliente y precarga (client-side navigation and prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works).
- **[Manejo de errores (Error Handling)](/docs/app/getting-started/error-handling):** Maneja adecuadamente [errores catch-all](/docs/app/getting-started/error-handling) y [errores 404](/docs/app/api-reference/file-conventions/not-found) en producción creando páginas de error personalizadas.
- **[Componentes del Cliente y del Servidor (Client and Server Components)](/docs/app/getting-started/server-and-client-components#examples):** Sigue los patrones de composición recomendados para Componentes del Servidor y del Cliente, y verifica la ubicación de tus [límites `"use client"`](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree) para evitar aumentar innecesariamente tu paquete JavaScript del lado del cliente.
- **[APIs dinámicas (Dynamic APIs)](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** Ten en cuenta que APIs dinámicas como [`cookies`](/docs/app/api-reference/functions/cookies) y la prop [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional) harán que toda la ruta opte por [Renderizado dinámico (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) (o toda tu aplicación si se usan en el [Diseño raíz (Root Layout)](/docs/app/api-reference/file-conventions/layout#root-layout)). Asegúrate de que el uso de APIs dinámicas sea intencional y envuélvelas en límites `<Suspense>` cuando sea apropiado.

> **Nota importante**: [Prerrenderizado parcial (experimental)](/blog/next-14#partial-prerendering-preview) permitirá que partes de una ruta sean dinámicas sin optar por renderizado dinámico en toda la ruta.

</AppOnly>

<PagesOnly>

- **[Componente `<Link>`](/docs/pages/building-your-application/routing/linking-and-navigating):** Usa el componente `<Link>` para navegación del lado del cliente y precarga.
- **[Errores personalizados (Custom Errors)](/docs/pages/building-your-application/routing/custom-error):** Maneja adecuadamente [errores 500](/docs/pages/building-your-application/routing/custom-error#500-page) y [404](/docs/pages/building-your-application/routing/custom-error#404-page)

</PagesOnly>

### Obtención de datos y almacenamiento en caché

<AppOnly>

- **[Componentes del Servidor (Server Components)](/docs/app/getting-started/fetching-data):** Aprovecha los beneficios de obtener datos en el servidor usando Componentes del Servidor.
- **[Manejadores de ruta (Route Handlers)](/docs/app/building-your-application/routing/route-handlers):** Usa Manejadores de Ruta para acceder a tus recursos backend desde Componentes del Cliente. Pero no llames a Manejadores de Ruta desde Componentes del Servidor para evitar una solicitud adicional al servidor.
- **[Streaming](/docs/app/building-your-application/routing/loading-ui-and-streaming):** Usa UI de carga y React Suspense para enviar progresivamente UI desde el servidor al cliente, y evitar que toda la ruta se bloquee mientras se obtienen los datos.
- **[Obtención paralela de datos (Parallel Data Fetching)](/docs/app/getting-started/fetching-data#parallel-data-fetching):** Reduce las cascadas de red obteniendo datos en paralelo cuando sea apropiado. También considera [precargar datos (preloading data)](/docs/app/getting-started/fetching-data#preloading-data) cuando sea apropiado.
- **[Almacenamiento en caché de datos (Data Caching)](/docs/app/deep-dive/caching#data-cache):** Verifica si tus solicitudes de datos están siendo almacenadas en caché o no, y opta por el almacenamiento en caché cuando sea apropiado. Asegúrate de que las solicitudes que no usan `fetch` estén [almacenadas en caché](/docs/app/api-reference/functions/unstable_cache).
- **[Imágenes estáticas (Static Images)](/docs/app/api-reference/file-conventions/public-folder):** Usa el directorio `public` para almacenar automáticamente en caché los activos estáticos de tu aplicación, como imágenes.

</AppOnly>

<PagesOnly>

- **[Rutas API (API Routes)](/docs/pages/building-your-application/routing/api-routes):** Usa Manejadores de Ruta para acceder a tus recursos backend y evitar que secretos sensibles se expongan al cliente.
- **[Almacenamiento en caché de datos (Data Caching)](/docs/pages/building-your-application/data-fetching/get-static-props):** Verifica si tus solicitudes de datos están siendo almacenadas en caché o no, y opta por el almacenamiento en caché cuando sea apropiado. Asegúrate de que las solicitudes que no usan `getStaticProps` estén almacenadas en caché cuando sea apropiado.
- **[Regeneración estática incremental (Incremental Static Regeneration)](/docs/pages/guides/incremental-static-regeneration):** Usa Regeneración Estática Incremental para actualizar páginas estáticas después de que se hayan construido, sin reconstruir todo tu sitio.
- **[Imágenes estáticas (Static Images)](/docs/pages/api-reference/file-conventions/public-folder):** Usa el directorio `public` para almacenar automáticamente en caché los activos estáticos de tu aplicación, como imágenes.

</PagesOnly>

### UI y accesibilidad

<AppOnly>

- **[Formularios y validación (Forms and Validation)](/docs/app/guides/forms):** Usa Acciones del Servidor (Server Actions) para manejar envíos de formularios, validación del lado del servidor y manejo de errores.

</AppOnly>

- **[Módulo de fuentes (Font Module)](/docs/app/api-reference/components/font):** Optimiza fuentes usando el Módulo de Fuentes, que aloja automáticamente tus archivos de fuente con otros activos estáticos, elimina solicitudes de red externas y reduce [cambio de diseño (layout shift)](https://web.dev/articles/cls).
- **[Componente `<Image>`](/docs/app/api-reference/components/image):** Optimiza imágenes usando el Componente Image, que optimiza automáticamente las imágenes, evita cambios de diseño y las sirve en formatos modernos como WebP.
- **[Componente `<Script>`](/docs/app/guides/scripts):** Optimiza scripts de terceros usando el Componente Script, que difiere automáticamente los scripts y evita que bloqueen el hilo principal.
- **[ESLint](/docs/architecture/accessibility#linting):** Usa el plugin incorporado `eslint-plugin-jsx-a11y` para detectar problemas de accesibilidad temprano.

### Seguridad

<AppOnly>

- **[Tainting](/docs/app/api-reference/config/next-config-js/taint):** Evita que datos sensibles se expongan al cliente marcando objetos de datos y/o valores específicos.
- **[Acciones del Servidor (Server Actions)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** Asegúrate de que los usuarios estén autorizados para llamar a Acciones del Servidor. Revisa las [prácticas de seguridad recomendadas](/blog/security-nextjs-server-components-actions).

</AppOnly>

- **[Variables de entorno (Environment Variables)](/docs/app/guides/environment-variables):** Asegúrate de que tus archivos `.env.*` estén agregados a `.gitignore` y solo las variables públicas tengan el prefijo `NEXT_PUBLIC_`.
- **[Política de seguridad de contenido (Content Security Policy)](/docs/app/guides/content-security-policy):** Considera agregar una Política de Seguridad de Contenido para proteger tu aplicación contra varias amenazas de seguridad como cross-site scripting, clickjacking y otros ataques de inyección de código.

### Metadatos y SEO

<AppOnly>

- **[API de Metadatos (Metadata API)](/docs/app/getting-started/metadata-and-og-images):** Usa la API de Metadatos para mejorar el SEO de tu aplicación agregando títulos de página, descripciones y más.
- **[Imágenes Open Graph (OG)](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** Crea imágenes OG para preparar tu aplicación para compartir en redes sociales.
- **[Mapas del sitio (Sitemaps)](/docs/app/api-reference/functions/generate-sitemaps) y [Robots](/docs/app/api-reference/file-conventions/metadata/robots):** Ayuda a los motores de búsqueda a rastrear e indexar tus páginas generando mapas del sitio y archivos robots.

</AppOnly>

<PagesOnly>

- **[Componente `<Head>`](/docs/pages/api-reference/components/head):** Usa el componente `next/head` para agregar títulos de página, descripciones y más.

</PagesOnly>

### Seguridad de tipos

- **TypeScript y [Plugin TS](/docs/app/api-reference/config/typescript):** Usa TypeScript y el plugin de TypeScript para mayor seguridad de tipos y para ayudarte a detectar errores temprano.

## Antes de ir a producción

Antes de ir a producción, puedes ejecutar `next build` para construir tu aplicación localmente y detectar cualquier error de construcción, luego ejecuta `next start` para medir el rendimiento de tu aplicación en un entorno similar a producción.

### Core Web Vitals

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** Ejecuta Lighthouse en modo incógnito para entender mejor cómo experimentarán los usuarios tu sitio e identificar áreas de mejora. Esta es una prueba simulada y debe combinarse con datos de campo (como Core Web Vitals).

<AppOnly>

- **[Hook `useReportWebVitals`](/docs/app/api-reference/functions/use-report-web-vitals):** Usa este hook para enviar datos de [Core Web Vitals](https://web.dev/articles/vitals) a herramientas de análisis.

</AppOnly>

### Analizando paquetes

Usa el plugin [`@next/bundle-analyzer`](/docs/app/guides/package-bundling#analyzing-javascript-bundles) para analizar el tamaño de tus paquetes JavaScript e identificar módulos y dependencias grandes que puedan estar afectando el rendimiento de tu aplicación.

Adicionalmente, las siguientes herramientas pueden ayudarte a entender el impacto de agregar nuevas dependencias a tu aplicación:

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
