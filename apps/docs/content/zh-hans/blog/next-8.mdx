[无服务器 Next.js](#serverless-nextjs)
----------------------------------------

Next.js 的无服务器构建目标会将页面输出为无服务器函数 (Serverless functions)

无服务器部署通过将应用拆分为更小的部分（也称为 lambda 函数）显著提高了可靠性和可扩展性。在 Next.js 中，`pages` 目录中的每个页面都会成为一个无服务器 lambda 函数。

无服务器架构有诸多优势。参考链接中讨论了在 Express 上下文中的一些优势，但这些原则具有普适性：无服务器架构支持分布式故障点、无限扩展性，并且采用"按使用付费"模式，极具成本效益。

要在 Next.js 中启用无服务器模式，请在 `next.config.js` 中添加 `serverless` 构建目标：

```js filename="next.config.js"
module.exports = {
  target: 'serverless',
};
```

`serverless` 目标会为每个页面输出一个独立的 lambda 函数。这些文件完全独立，无需任何依赖即可运行：

*   `pages/index.js` => `.next/serverless/pages/index.js`
*   `pages/about.js` => `.next/serverless/pages/about.js`

Next.js 无服务器函数的签名与 Node.js HTTP 服务器回调类似：

```
type Function = (req: http.IncomingMessage, res: http.ServerResponse) => void;
```

*   [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
*   [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)
*   `void` 表示函数没有返回值，等同于 JavaScript 的 `undefined`。调用函数将结束请求。

由于托管平台的函数签名各不相同，Next.js 提供了用于无服务器部署的低级 API。通常您需要使用兼容层来封装 Next.js 无服务器构建的输出。

例如，如果平台支持 Node.js [http.Server](https://nodejs.org/api/http.html#http_class_http_server) 类：

```js filename="server.js"
const http = require('http');
const page = require('./.next/serverless/about.js');
const server = new http.Server((req, res) => page.render(req, res));
server.listen(3000, () => console.log('Listening on http://localhost:3000'));
```

### [**总结**](#summary)

*   用于实现无服务器部署的低级 API
*   `pages` 目录中的每个页面都会成为一个无服务器函数 (lambda)
*   创建尽可能小的无服务器函数（基础 zip 包仅 50 KB）
*   针对函数的快速冷启动进行了优化
*   无服务器函数零依赖（所有依赖已包含在函数包中）
*   使用 Node.js 的 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage) 和 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)
*   通过 `next.config.js` 中的 `target: 'serverless'` 启用
*   仍然完全支持并维护 `server` 目标
*   `serverless` 模式下不支持 `publicRuntimeConfig` 和 `serverRuntimeConfig`，请改用构建时配置

[大幅减少构建时内存占用](#massive-build-time-memory-usage-reduction)
---------------------------------------------------------------------------------------

我们为 webpack 做出了贡献，以改进 Next.js（以及整个 webpack 生态系统）的构建性能和资源利用率。

这些改进使得内存使用量最高可减少 16 倍，且性能不受影响。

内存释放速度更快，在多页面场景下进程也不再崩溃。

我们将在 [Next.js 博客](/blog) 上详细介绍这些优化实现，敬请关注。

[构建时环境配置](#build-time-environment-configuration)
-----------------------------------------------------------------------------

在审查 Next.js 应用时，我们经常发现开发者会添加 `babel-plugin-transform-define` 或 `webpack.DefinePlugin` 来为应用提供配置值。

Next.js 8 在 `next.config.js` 中引入了新的 `env` 键，以向后兼容的方式提供相同功能：

```js filename="next.config.js"
module.exports = {
  env: {
    customKey: 'MyValue',
  },
};
```

这允许您在代码中使用 `process.env.customKey`。例如：

```js filename="pages/index.js"
export default function IndexPage() {
  return <h1>customKey 的值为：{process.env.customKey}</h1>;
}
```

`process.env.customKey` 将在构建时被替换为 `'MyValue'`。

[预取性能优化](#prefetch-performance-improvements)
-----------------------------------------------------------------------

Next.js 路由器允许您预取页面以实现更快导航：

```js filename="pages/index.js"
import Link from 'next/link';
 
export default function IndexPage() {
  return (
    <>
      <Link href="/about" prefetch>
        <a>前往关于页面</a>
      </Link>
    </>
  );
}
```

其工作原理是预取带有 `prefetch` 属性的链接对应的 JavaScript 包。

在 Next.js 8 之前的版本中，这会通过向文档 `<body>` 中注入 `<script>` 标签实现。

然而，这会导致页面打开时产生一些额外开销，最明显的是即使页面已经可以交互，浏览器"加载"指示仍会显示较长时间。

在 Next.js 8 中，`prefetch` 改用 `<link rel="preload">` 而非 `<script>` 标签。并且仅在 `onload` 后开始预取，以便浏览器管理资源。

此外，Next.js 现在会检测 2G 网络和 `navigator.connection.saveData` 模式，在较慢网络连接下禁用预取。

[更小的初始 HTML 体积](#smaller-initial-html-size)
-------------------------------------------------------

由于 Next.js 会预渲染 HTML，它会将页面包裹在包含 `<html>`、`<head>`、`<body>` 和渲染页面所需 JavaScript 文件的默认结构中。

[Next.js 7 将初始负载优化至 1.50KB](https://nextjs.org/blog/next-7/#smaller-initial-html-payload)，比前一版本减少了 7.4%。

### [按需加载 /\_error](#on-demand-loading-of-_error)

当生产环境中发生错误时，会渲染 `/_error` 页面来显示错误信息。

自 Next.js 首个版本以来，`/_error` 页面脚本标签一直是初始 HTML 的一部分，这意味着即使没有运行时错误也会加载它。

从 Next.js 8 开始，`/_error` 页面将在错误发生时按需加载。

这意味着默认情况下需要加载、解析和执行的代码更少。

[开发者体验改进](#dx-improvements)
-----------------------------------

Next.js 的主要目标之一是在提供最佳生产性能的同时，提供最佳的开发者体验。此版本包含许多基于用户反馈的细微改进。

### [改进的按需入口](#improved-on-demand-entries)

开箱即用，Next.js 会自动仅编译正在开发的页面。运行 `next dev` 时，Next.js 不会每次都编译 pages 目录中的所有页面，而是按需编译您访问的页面。

例如，当访问 `http://localhost:3000/my-page` 时，`pages/my-page.js` 文件会被按需编译，然后渲染页面。

这确保开发者在启动开发服务器时无需等待所有页面编译完成，这在大型应用中可能耗时较长。它保持内存使用率低且编译器快速，因为打包时编译器不需要考虑所有页面。

![按需入口流程](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-8/flow.jpg)

按需入口流程

当页面超过 25 秒未被访问时，它将被从编译器的构建缓存中清除，以保持编译器快速并减少内存使用。

Next.js 通过轮询机制跟踪页面访问情况。每 5 秒发送一次"on-demand-entries-ping"，让 Next.js 开发服务器知道某个页面正在被访问。

自该功能首次发布以来，ping 是通过 `window.fetch` 调用实现的，这意味着每次触发 ping 时，它都会出现在浏览器开发者工具的 `console` 和 `network` 标签页中。

最常被请求的功能之一是能够从浏览器开发者工具中隐藏这些请求，因为这些请求会增加不必要的干扰。

我们很高兴地宣布，在 Next.js 8 中，基于 `fetch` 的 ping 已被基于 WebSocket 的方法取代，这意味着 ping 仍然会发生，但仅在检查 WebSocket 连接时可见。

特别感谢 [JJ Kasper](https://github.com/ijjk) 在转换为 WebSocket 方面的合作。

### [开发模式下更快的端口监听](#faster-port-listening-in-development)

启动 Next.js 开发服务器时，它需要运行一些初始编译才能处理请求。默认情况下，Next.js 会等待此编译步骤完成后再启动 HTTP 服务器，这意味着如果您运行 `next dev` 然后立即打开浏览器，有时可能会看到"无法访问此网站"的消息，因为 HTTP 服务器尚未开始监听连接。

在 Next.js 8 中，HTTP 服务器将在编译开始前就开始监听连接，这意味着如果您在编译完成前访问 `http://localhost:3000/`，请求将等待初始编译完成后再响应，而不必反复刷新页面直到可用。

特别感谢 [Brian Beck](https://github.com/exogen) 实现此功能。

[更快的静态导出](#faster-static-export)
---------------------------------------------

Next.js 专注于预渲染作为实现高性能的手段。预渲染有两种形式：

*   **服务端渲染**：每个请求触发一次渲染。最终用户无需等待任何 JavaScript 下载即可开始消费数据
*   **静态渲染**：输出可直接提供的静态文件，无需在服务器上执行任何代码

从 Next.js 8 开始，如果您的机器有多个 CPU，通过 `next export` 进行的静态渲染将获得速度提升。

基于 4 核 MacBook 的测试，通过利用所有核心预渲染页面，导出速度从每秒约 25 页提升至每秒 75 页。

Next.js 会自动检测 CPU 核心数并相应分配页面，无需任何代码更改。

特别感谢 [Benjamin Kniffler](https://github.com/bkniffler) 实现此功能。

[Head 元素去重](#head-element-deduplication)
---------------------------------------------------------

构建应用时的常见需求是更新页面的 `<head>` 元素。例如设置响应式设计的 `<title>` 或 `<meta name="viewport">`。

Next.js 提供了内置组件来修改 `<head>`：

```js filename="pages/index.js"
import Head from 'next/head';
 
export default function IndexPage() {
  return (
    <>
      <Head>
        <title>我的页面标题</title>
      </Head>
    </>
  );
}
```

`<Head>` 组件甚至可以在不同组件中多次使用，例如您的布局组件可以设置一些默认的 head 标签。

但是，您可能希望用不同的值覆盖默认 head 标签，在旧版 Next.js 中，这会导致输出中出现重复标签，因为没有去重机制。

因此，现在可以为 `<Head>` 组件中的每个元素提供 `key` 属性，这将自动对具有相同 `key` 值的标签进行去重。

当在两个标签上设置 `key="viewport"` 时，只有最后一个会被渲染。

```js filename="pages/index.js"
import Head from 'next/head';
export default function IndexPage() {
  return (
    <>
      <Head>
        <title>我的页面标题</title>
        <meta
          name="viewport"
          content="initial-scale=1.0, width=device-width"
          key="viewport"
        />
      </Head>
      <Head>
        <meta
          name="viewport"
          content="initial-scale=1.2, width=device-width"
          key="viewport"
        />
      </Head>
    </>
  );
}
```

[安全性改进](#security-improvements)
-----------------------------------------------

### [新增 `crossOrigin` 配置选项](#new-crossorigin-config-option)

在 Next.js 6 中，我们引入了在 `pages/_document.js` 中为 `<Head>` 和 `<NextScript>` 添加 `crossOrigin` 属性的选项，但这并未涵盖设置跨域的所有用例。

Next.js 有一个客户端路由器，会动态注入 `<script>` 标签，这些标签在注入时缺少 `cross-origin` 属性。

为确保所有 `<script>` 标签都设置了 `cross-origin`，我们在 `next.config.js` 中引入了一个新的配置选项：

```js filename="next.config.js"
module.exports = {
  crossOrigin: 'anonymous',
};
```

引入此选项的另一个好处是，您的应用不再需要自定义 `pages/_document.js` 来设置跨域。

旧行为仍然支持，但会在开发时发出警告，以帮助开发者迁移到新引入的选项。

### [移除内联 JavaScript](#removed-inline-javascript)

在使用 Next.js 7 及以下版本时，为了启用 [内容安全策略 (CSP)](https://developer.mozilla.org/docs/Web/HTTP/CSP)，用户必须在策略中包含 `script-src 'unsafe-inline'`，因为 Next.js 会创建一个内联的 `<script>` 标签来传递数据，例如将 `getInitialProps` 的结果传递到客户端。

从 Next.js 8 开始，我们将这个内联脚本标签改为了 JSON 标签，以便安全地传输到客户端。这意味着 Next.js 不再包含任何内联脚本。

经过仔细考虑，现在可以使用 `script-src 'self'` 了。

### [API 认证示例](#example-of-api-authentication)

[有史以来最受欢迎的示例之一](https://github.com/vercel/next.js/issues/153) 是如何在 Next.js 中针对外部 API 进行认证，无论是什么 API，使用什么编程语言。

随着 Next.js 8 的发布，我们还引入了一个新创建的示例：[with-cookie-auth](https://github.com/vercel/next.js/tree/canary/examples/with-cookie-auth)

这个示例展示了如何针对外部 Node.js API 进行认证，但所应用的概念适用于任何无状态 API。

该示例使用 cookie 在服务端渲染和客户端渲染之间共享令牌。

这样，如果应用在服务端渲染，它仍然可以代表用户获取认证数据。

特别感谢贡献这个示例的 [Juan Olvera](https://github.com/j0lv3r4)。
