[Next.js 9](/blog/next-9) 发布于六个月前，随后三个月前推出了 [Next.js 9.1](/blog/next-9-1)。

这两个版本为 Next.js 添加了非常强大的新功能，同时没有增加基础客户端运行时体积。

此后，我们重点对整个框架进行了优化和改进：[9.1.1](https://github.com/vercel/next.js/releases/tag/v9.1.1)、[9.1.2](https://github.com/vercel/next.js/releases/tag/v9.1.2)、[9.1.3](https://github.com/vercel/next.js/releases/tag/v9.1.3)、[9.1.4](https://github.com/vercel/next.js/releases/tag/v9.1.4)、[9.1.5](https://github.com/vercel/next.js/releases/tag/v9.1.5)、[9.1.6](https://github.com/vercel/next.js/releases/tag/v9.1.6) 和 [9.1.7](https://github.com/vercel/next.js/releases/tag/v9.1.7)。

让我们深入了解这些版本带来的改进！

*   **[客户端 JavaScript 体积减少 3%-8%+](#3--8-smaller-client-side-javascript-size)**：我们进一步优化了应用输出，"Hello World" 应用减少了 7.5kB。更复杂的应用可节省高达 8% 或更多。
*   **[重新设计的生产构建 CLI 输出](#redesigned-production-build-cli-output)**：生产构建输出现在以更易理解的格式显示 gzip 压缩后的文件大小。
*   **[新增内置 Polyfill：fetch()、URL 和 Object.assign](#new-built-in-polyfills-fetch-url-and-objectassign)**：应用可在旧版浏览器中无忧使用 `fetch()` API、`URL` 和 `Object.assign`。
*   **[优化页面加载：提升 FCP 和 TTI](#optimized-page-loading-better-fcp-and-tti)**：我们与 Google Chrome 团队紧密合作，最大化页面加载性能，显著改善终端用户体验。
*   **[支持最新 JavaScript 特性](#support-for-the-latest-javascript-features)**：我们承诺您始终可以使用最新 JavaScript 特性，包括 [可选链式调用](https://github.com/tc39/proposal-optional-chaining)、[空值合并运算符](https://github.com/tc39/proposal-nullish-coalescing) 等稳定的 ES2020 特性。
*   **[零配置部署支持 `next export` 应用](#zero-config-deployment-support-for-next-export-applications)**：`next export` 生成的应用现在可以零配置部署到 [Vercel](https://vercel.com)。
*   **[React 严格模式兼容与启用选项](#react-strict-mode-compliance-and-opt-in)**：Next.js 所有客户端运行时现已兼容 React 严格模式，并新增配置选项供您全局启用。
*   **[针对 React 每日构建版本的自动化测试](#automated-testing-against-nightly-react-builds)**：Next.js 现在自动测试 [React 的 next 渠道版本](https://reactjs.org/blog/2019/10/22/react-release-channels.html)，确保与未来版本兼容。

所有这些改进都是非破坏性且完全向后兼容的。只需运行以下命令即可更新：

```bash filename="终端"
npm i next@latest react@latest react-dom@latest
```

### [客户端 JavaScript 体积减少 3%-8%+](#3---8-smaller-client-side-javascript-size)

与 Google Chrome 团队合作，所有 Next.js 应用都将受益于 **7.5kB 或更多** 的体积缩减。

基础应用可减少 3-4% 的体积，更复杂的应用可能减少 6-8%（或更多）！

部分节省来自于 [替换 `url` 包的客户端版本](https://github.com/GoogleChromeLabs/native-url) 为基于 [URL API](https://developer.mozilla.org/docs/Web/API/URL) 构建的实现。

更多体积缩减通过为常用包提供内置的小型 polyfill 实现。您可 [在此阅读关于这些 polyfill 的更多信息](#new-built-in-polyfills-fetch-url-and-objectassign)。

最后，我们优化了 JSX 输出，节省的体积与您应用中 JSX 的数量直接相关。

### [重新设计的生产构建 CLI 输出](#redesigned-production-build-cli-output)

为提升清晰度，我们重新设计了 CLI 的生产构建输出。由于 Next.js 是混合应用框架，每个页面可能具有不同特性。

新输出将每个页面分类为以下类型之一：

*   **服务端渲染 (Server)**：页面在运行时通过服务端渲染，意味着使用了 `getInitialProps` 或 [`getServerProps` (提案)](https://github.com/vercel/next.js/issues/9524)
*   **自动静态优化 (Static)**：页面在构建时生成为静态 HTML，将作为静态文件提供（不使用初始 props）
*   **使用计算数据静态生成 (SSG)**：页面在构建时生成为静态 HTML/JSON，将作为静态文件提供（使用 [`getStaticProps` (提案)](https://github.com/vercel/next.js/issues/9524)）

此外，新输出显示每个页面的 Gzip 压缩大小——这些大小排除了所有页面共享的文件（单独显示）。

每个页面的大小将根据其感知用户体验着色：

*   **小于 130kB**：_绿色_——您的应用在大多数网络和设备条件下都应表现良好。
*   **130kB 至 170kB 之间**：_黄色_——您的应用在 [全球基准设备 + 网络条件](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 下接近 5-6 秒加载时间。
*   **大于 170kB**：_红色_——您的应用在 [相同条件](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 下可能需要超过 6 秒加载。

### [新增内置 Polyfill：fetch()、URL 和 Object.assign](#new-built-in-polyfills-fetch-url-and-objectassign)

在检查许多用户应用和示例时，我们发现大多数应用都附带了一组相似的 polyfill。某些情况下，应用甚至为同一特性重复加载 polyfill。

为解决此问题，我们与 Google Chrome 团队合作，内置了三个最常用 API 的 polyfill。

通过差异化加载，这些 polyfill **不会为全球 83% 的网络流量加载**。这意味着大多数用户不会下载这些 polyfill 相关的字节——仅在必要时才会下载。

此外，任何我们现已内置的知名 polyfill 将从您的生产构建中完全消除。这意味着您不会因为某个依赖项无意中导入这些 API 的 polyfill 而付出代价。

内置 API 及其替代的 polyfill 列表如下：

*   [**fetch()**](https://developer.mozilla.org/docs/Web/API/Fetch_API) — 替代：`whatwg-fetch` 和 `unfetch`。
*   [**URL**](https://developer.mozilla.org/docs/Web/API/URL) — 替代：[`url` 包 (Node.js API)](https://nodejs.org/api/url.html)。
*   [**Object.assign()**](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Object/assign) — 替代：`object-assign`、`object.assign` 和 `core-js/object/assign`。

如果您在服务端使用 `fetch()`，仍需导入 `isomorphic-fetch` 或 `isomorphic-unfetch`。

此更改完全非破坏性，所有 polyfill 均采用最符合规范的可用版本实现。结果是在现代浏览器上可 **减少高达 18kB 的 JavaScript**。

### [优化页面加载：提升 FCP 和 TTI](#optimized-page-loading-better-fcp-and-tti)

Next.js 优化了其预加载实现，以减少 FCP 和整体 TTI。

通过修复浏览器问题，CSS（当使用时）现在正确优先于 JavaScript。这带来了更快的首次内容绘制 (FCP)，为终端用户提供视觉上更快的网站体验。

此外，我们优化了页面预取，使用更低优先级的网络请求。

这些请求仅在浏览器空闲时发生，从而缩短可交互时间 (TTI)。这是因为浏览器会优先使您的应用可交互，而非乐观预取。

### [支持最新 JavaScript 特性](#support-for-the-latest-javascript-features)

Next.js 拥有先进、高度优化的编译管道，允许您使用最新 JavaScript 特性。我们近期引入的优化直接促成了 [3-8% 的应用体积缩减](#3--8-smaller-client-side-javascript-size)。

这些 JavaScript 特性可以无忧使用，无需担心浏览器兼容性，因为我们会自动将您的代码编译为支持所有浏览器（不包括已终止支持的版本）。这包括 ES6+ 特性，如 [async/await (ES2017)](https://github.com/tc39/ecmascript-asyncawait)、[对象剩余/展开属性 (ES2018)](https://github.com/tc39/proposal-object-rest-spread)、[动态 `import()` (ES2020)](https://v8.dev/features/dynamic-import) 等！

在此版本中，我们很高兴宣布支持 [可选链式调用 (Stage 4)](https://github.com/tc39/proposal-optional-chaining) 和 [空值合并运算符 (Stage 4)](https://github.com/tc39/proposal-nullish-coalescing)。

```js filename="pages/index.js"
function Page(props) {
  return (
    <p>{props?.deeply?.nested?.value}</p>
    /* ⬆ 如果 deeply.nested.value 不可用则不会渲染 */
  );
}
 
export default Page;
```

> 可选链式操作符 (`?.`)

```js filename="pages/index.js"
function Page(props) {
  return (
    <p>{props.something ?? '默认值'}</p>
    /* ⬆ 结果为 "默认值" */
  );
}
 
export default Page;
```

> 空值合并操作符 (`??`)

未来，我们计划通过自动 [module / nomodule 构建](https://nextjs.org/blog/next-9-1#module--nomodule) 输出更优化的包。

### [零配置部署支持 `next export` 应用](#zero-config-deployment-support-for-next-export-applications)

Next.js 的 `next export` 命令现在开箱即用地支持 Vercel 的 [零配置](https://vercel.com/blog/zero-config) 部署。在此更改之前，使用 `next export` 的用户需要创建自定义 `now.json`。

在 [Vercel](https://vercel.com) 上使用 Next.js 的导出模式非常简单，只需在 `package.json` 中包含以下 `build` 脚本：

```json filename="package.json"
{
  "scripts": {
    "build": "next build && next export"
  }
}
```

然后，您可以通过 **仅一个命令** 将 Next.js 应用部署到 [Vercel](https://vercel.com)。如果尚未安装 [Vercel](https://vercel.com)，您可以通过安装 [Vercel CLI](https://vercel.com/cli) 实现。

```bash filename="终端"
now
```

### [React 严格模式兼容与启用选项](#react-strict-mode-compliance-and-opt-in)

完整的 Next.js 运行时现已兼容严格模式。这包括对 Next.js 的 head 管理器 (`<Head>`)、`next/dynamic` 和其他核心功能的更新。这意味着运行时现在使用钩子或已弃用生命周期，并采用 React 的新 Context API。

我们还添加了一个便捷的启用选项，供您为整个应用启用严格模式。

为此，在 `next.config.js` 中配置以下选项：

```js filename="next.config.js"
module.exports = {
  reactStrictMode: true,
};
```

如果您或您的团队尚未准备好在整个应用中使用严格模式，没关系！您可以 [使用 `<React.StrictMode>`](https://reactjs.org/docs/strict-mode.html) 逐页渐进迁移。

虽然 **非必需**，但严格模式将在未来解锁许多优化。因此，我们建议您尽早了解它！

### [针对 React 每日构建版本的自动化测试](#automated-testing-against-nightly-react-builds)

在与 React 核心团队 [密切合作](https://github.com/vercel/next.js/pull/9165) 下，我们现在每 12 小时 [测试](https://github.com/vercel/next.js/pull/9151) Next.js 与 React 的 next 渠道版本。

这些测试有助于确保我们为 React 的未来版本做好准备并保持兼容。兼容性是 Next.js 非常重视的事项，我们致力于 Next.js 的长期 API 稳定性。

React 核心团队已 [记录此过程](https://reactjs.org/blog/2019/10/22/react-release-channels.html)，希望 React 生态系统的其他项目效仿。
