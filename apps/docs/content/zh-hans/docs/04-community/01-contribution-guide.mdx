欢迎阅读 Next.js 文档贡献指南！我们很高兴您的加入。

本页面提供编辑 Next.js 文档的说明。我们的目标是让社区中的每个人都能轻松贡献力量，共同改进文档。

## 为什么参与贡献？

开源工作永无止境，文档建设亦是如此。贡献文档是初学者参与开源的好途径，也是经验丰富的开发者分享知识、澄清复杂概念的好机会。

通过为 Next.js 文档做贡献，您正在帮助所有开发者构建更强大的学习资源。无论是发现拼写错误、存在疑惑的章节，还是意识到某些主题缺失，我们都欢迎并感谢您的贡献。

## 如何贡献

文档内容位于 [Next.js 代码库](https://github.com/vercel/next.js/tree/canary/docs)。您可以直接在 GitHub 上编辑文件，或克隆代码库后在本地编辑。

### GitHub 工作流

如果您是 GitHub 新手，建议阅读 [GitHub 开源指南](https://opensource.guide/how-to-contribute/#opening-a-pull-request) 了解如何复刻仓库、创建分支和提交拉取请求。

> **须知**：文档底层代码位于私有代码库，会同步到 Next.js 公共仓库。这意味着您无法在本地预览文档，但合并拉取请求后可在 [nextjs.org](https://nextjs.org/docs) 查看变更。

### 编写 MDX

文档使用 [MDX](https://mdxjs.com/) 格式编写，这种 Markdown 格式支持 JSX 语法，允许我们在文档中嵌入 React 组件。可参阅 [GitHub Markdown 指南](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax) 快速了解 Markdown 语法。

### VSCode 配置

#### 本地预览更改

VSCode 内置 Markdown 预览器可用于本地查看编辑效果。要为 MDX 文件启用预览器，需在用户设置中添加配置项。

打开命令面板（Mac 使用 `⌘ + ⇧ + P`，Windows 使用 `Ctrl + Shift + P`），搜索 `Preferences: Open User Settings (JSON)`。

然后在 `settings.json` 文件中添加：

```json filename="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

再次打开命令面板，搜索 `Markdown: Preview File` 或 `Markdown: Open Preview to the Side`，这将打开预览窗口显示格式化后的更改。

#### 推荐扩展

推荐 VSCode 用户安装以下扩展：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx)：提供 MDX 的智能提示和语法高亮
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode)：保存时自动格式化 MDX 文件

### 审核流程

提交贡献后，Next.js 或开发者体验团队将审核您的更改、提供反馈，并在准备就绪时合并拉取请求。

如有疑问或需要协助，请在 PR 评论中告知。感谢您为 Next.js 文档做出贡献，成为社区一员！

> **提示**：提交 PR 前运行 `pnpm prettier-fix` 执行 Prettier 格式化。

## 文件结构

文档采用**文件系统路由**。[`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 内的每个文件夹和文件代表一个路由段，这些分段用于生成 URL 路径、导航和面包屑导航。

文件结构反映网站导航，默认按字母顺序排序导航项。但可通过在文件夹/文件名前添加两位数字（`00-`）调整顺序。

例如在 [函数 API 参考](/docs/app/api-reference/functions) 中，页面按字母排序便于开发者查找特定函数：

```txt
04-functions
├── after.mdx
├── cacheLife.mdx
├── cacheTag.mdx
└── ...
```

而在 [路由章节](/docs/app) 中，文件添加了两位数字前缀，按学习顺序排列：

```txt
01-routing
├── 01-defining-routes.mdx
├── 02-pages.mdx
├── 03-layouts-and-templates.mdx
└── ...
```

在 VSCode 中可使用 `⌘ + P` (Mac) 或 `Ctrl + P` (Windows) 打开搜索栏快速定位页面，例如输入 `defining-routes`。

> **为何不使用清单文件？**
>
> 我们考虑过使用清单文件（另一种流行的文档导航生成方式），但发现清单容易与文件不同步。文件系统路由促使我们思考文档结构，更符合 Next.js 原生特性。

## 元数据

每个页面顶部都有由三条短横线分隔的元数据块。

### 必填字段

以下字段为**必填项**：

| 字段          | 说明                                       |
| ------------- | ------------------------------------------ |
| `title`       | 页面主标题 `<h1>`，用于 SEO 和 OG 图片     |
| `description` | 页面描述，用于 SEO 的 `<meta name="description">` 标签 |

```yaml filename="required-fields.mdx"
---
title: 页面标题
description: 页面描述
---
```

建议页面标题限制在 2-3 个词（如"优化图片"），描述控制在 1-2 句话（如"了解如何在 Next.js 中优化图片"）。

### 可选字段

以下字段为**可选项**：

| 字段        | 说明                                                                 |
| ----------- | -------------------------------------------------------------------- |
| `nav_title` | 覆盖导航中显示的标题。当页面标题过长时特别有用，未提供时使用 `title` |
| `source`    | 从共享页面拉取内容。参见[共享页面](#shared-pages)                    |
| `related`   | 文末相关页面列表，会自动转换为卡片。参见[相关链接](#related-links)   |
| `version`   | 开发阶段标识，如 `experimental`,`legacy`,`unstable`,`RC`            |

```yaml filename="optional-fields.mdx"
---
nav_title: 导航项标题
source: app/building-your-application/optimizing/images
related:
  description: 查看图片组件 API 参考。
  links:
    - app/api-reference/components/image
version: experimental
---
```

## `App` 和 `Pages` 文档

由于 **App Router** 和 **Pages Router** 的功能大多完全不同，它们的文档分别存放在独立章节（`02-app` 和 `03-pages`）。但部分功能是两者共享的。

### 共享页面

为避免内容重复和不同步风险，我们使用 `source` 字段将内容从一个页面拉取到另一个页面。例如 `<Link>` 组件在 **App** 和 **Pages** 中的行为*基本*相同。我们不复制内容，而是从 `app/.../link.mdx` 拉取内容到 `pages/.../link.mdx`：

```mdx filename="app/.../link.mdx"
---
title: <Link>
description: <Link> 组件的 API 参考。
---

此 API 参考将帮助您理解如何使用 Link 组件可用的属性和配置选项。
```

```mdx filename="pages/.../link.mdx"
---
title: <Link>
description: <Link> 组件的 API 参考。
source: app/api-reference/components/link
---

{/* 请勿编辑此页面 */}
{/* 此页面内容来自上方源文件 */}
```

这样我们只需在一处编辑内容，变更就会同步到两个章节。

### 共享内容

在共享页面中，有时会存在 **App Router** 或 **Pages Router** 专属内容。例如 `<Link>` 组件的 `shallow` 属性仅在 **Pages** 中可用。

为确保内容只显示在正确的路由器文档中，可用 `<AppOnly>` 或 `<PagesOnly>` 组件包裹内容块：

```mdx filename="app/.../link.mdx"
此内容在 App 和 Pages 文档中共享。

<PagesOnly>

此内容仅在 Pages 文档中显示。

</PagesOnly>

此内容在 App 和 Pages 文档中共享。
```

这些组件常用于示例和代码块。

## 代码块

代码块应包含可复制粘贴的最小工作示例，意味着代码无需额外配置即可运行。

例如展示 `<Link>` 组件用法时，应包含 `import` 语句和组件本身：

```tsx filename="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">关于</Link>
}
```

提交前务必在本地运行示例，确保代码最新且可用。

### 语言与文件名

代码块应包含语言和 `filename` 的标题。添加 `filename` 属性会显示特殊终端图标，帮助用户定位输入位置：

````mdx filename="code-example.mdx"
```bash filename="Terminal"
npx create-next-app
```
````

文档中多数示例使用 `tsx` 和 `jsx`，少数用 `bash`。但您可以使用任何[支持的语言](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)。

## 相关链接

相关链接通过添加逻辑后续步骤引导用户学习路径。

- 链接以卡片形式显示在正文下方
- 包含子页面的页面会自动生成链接

在页面元数据的 `related` 字段中创建相关链接：

```yaml filename="example.mdx"
---
related:
  description: 了解如何快速开始构建首个应用。
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### 嵌套字段说明

| 字段          | 是否必填 | 说明                                                                                     |
| ------------- | -------- | ---------------------------------------------------------------------------------------- |
| `title`       | 可选     | 卡片列表标题，默认为 **下一步**                                                          |
| `description` | 可选     | 卡片列表描述                                                                             |
| `links`       | 必填     | 其他文档页面的链接列表。每项应为相对 URL 路径（无前导斜杠），如 `app/api-reference/file-conventions/page` |

## 风格指南

本节为技术写作新手提供文档撰写指南。

### 页面模板

虽然没有严格的页面模板，但以下章节在文档中反复出现：

- **概述**：页面首段应说明功能定义和用途，附带最小工作示例或 API 参考
- **约定**：若功能有使用约定，应在此说明
- **示例**：展示不同场景下的功能用法
- **API 表格**：API 页面应在末尾提供带跳转链接的概览表（如适用）
- **后续步骤（相关链接）**：添加相关页面链接引导学习路径

请根据需要添加这些章节。

### 页面类型

文档页面分为两类：概念性页面和参考性页面。

- **概念性页面**用于解释概念或功能，通常较长且信息丰富。在 Next.js 文档中位于**构建应用**章节
- **参考性页面**用于说明特定 API，通常更简短聚焦。在 Next.js 文档中位于**API 参考**章节

> **须知**：根据贡献的页面类型，可能需要采用不同的语调和风格。例如概念性页面更具指导性，使用"您"称呼用户；参考性页面更技术化，多用"创建、更新、接受"等祈使词，常省略"您"。

### 行文风格

以下指南帮助保持文档风格与语调的一致性：

- 语句清晰简洁，避免离题
  - 若频繁使用逗号，考虑拆分成多句或用列表
  - 用简单词替换复杂词，如用"使用"替代"利用"
- 谨慎使用"这"字，避免歧义。表意不清时可重复主语
  - 例如用"Next.js 使用 React"而非"Next.js 使用这个"
- 使用主动语态更易阅读
  - 例如用"Next.js 使用 React"而非"React 被 Next.js 使用"。若出现"被"、"由"等词可能是被动语态
- 避免使用"简单"、"快速"、"只需"等主观词汇
- 避免"不要"、"不能"、"不会"等否定词
  - 例如用"您可使用 `Link` 组件创建页面链接"而非"不要用 `<a>` 标签创建页面链接"
- 使用第二人称（您/您的）更亲切
- 使用中性称呼，如"开发者"、"用户"或"读者"
- 添加代码示例时确保格式正确且可运行

这些指南虽不全面，但能帮助您入门。如需深入了解技术写作，请参阅 [Google 技术写作课程](https://developers.google.com/tech-writing/overview)。
