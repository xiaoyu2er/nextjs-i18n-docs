在将 Next.js 应用部署至生产环境前，您应当考虑实施以下优化措施和模式，以获得最佳的用户体验、性能和安全性。

本页提供了最佳实践指南，可作为[应用开发阶段](#during-development)和[上线前准备](#before-going-to-production)的参考，同时介绍了您需要了解的[Next.js自动优化机制](#automatic-optimizations)。

## 自动优化

以下 Next.js 优化功能默认启用且无需配置：

<AppOnly>

- **[服务端组件 (Server Components)](/docs/app/getting-started/server-and-client-components):** Next.js 默认使用服务端组件。这些组件在服务器端运行，无需客户端 JavaScript 即可渲染，因此不会增加客户端 JavaScript 包的大小。您可按需使用[客户端组件 (Client Components)](/docs/app/getting-started/server-and-client-components)来实现交互功能。
- **[代码分割 (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** 服务端组件支持按路由段自动代码分割。您还可以考虑在适当场景下对客户端组件和第三方库进行[懒加载 (lazy loading)](/docs/app/guides/lazy-loading)。
- **[预获取 (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** 当用户视图中出现新路由链接时，Next.js 会在后台预获取该路由，实现近乎即时的导航体验。您可以在必要时禁用此功能。
- **[静态渲染 (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js 会在构建时静态渲染服务端和客户端组件，并缓存渲染结果以提升性能。您可以为特定路由选择启用[动态渲染 (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。{/* TODO: 待PPR稳定后更新 */}
- **[缓存机制 (Caching)](/docs/app/deep-dive/caching):** Next.js 会自动缓存数据请求、服务端/客户端组件渲染结果、静态资源等，减少对服务器、数据库和后端服务的网络请求。您可以在必要时禁用缓存。

</AppOnly>

<PagesOnly>

- **[代码分割 (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js 会按页面自动分割应用代码，确保导航时只加载当前页面所需的代码。您还可以考虑在适当场景下对第三方库进行[懒加载 (lazy loading)](/docs/pages/guides/lazy-loading)。
- **[预获取 (Prefetching)](/docs/pages/api-reference/components/link#prefetch):** 当用户视图中出现新路由链接时，Next.js 会在后台预获取该路由，实现近乎即时的导航体验。您可以在必要时禁用此功能。
- **[自动静态优化 (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** 对于没有阻塞数据需求的页面，Next.js 会自动判定为静态页面（可预渲染）。优化后的页面可被缓存，并通过多个CDN节点分发给终端用户。您也可以根据需要选择启用[服务端渲染 (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props)。

</PagesOnly>

这些默认优化旨在提升应用性能，减少每次网络请求传输的数据量和成本。

## 开发阶段

在构建应用时，我们建议使用以下功能来确保最佳性能和用户体验：

### 路由与渲染

<AppOnly>

- **[布局 (Layouts)](/docs/app/building-your-application/routing/layouts-and-templates#layouts):** 使用布局在页面间共享UI，并启用导航时的[部分渲染 (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)功能。
- **[`<Link>`组件](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** 使用该组件实现[客户端导航和预获取](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works)功能。
- **[错误处理 (Error Handling)](/docs/app/building-your-application/routing/error-handling):** 通过创建自定义错误页面，优雅处理生产环境中的[全局错误](/docs/app/building-your-application/routing/error-handling)和[404错误](/docs/app/api-reference/file-conventions/not-found)。
- **[客户端与服务端组件](/docs/app/getting-started/server-and-client-components#examples):** 遵循服务端与客户端组件的推荐组合模式，检查[`"use client"`边界](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree)的放置位置，避免不必要地增加客户端JavaScript包体积。
- **[动态API](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** 注意使用如[`cookies`](/docs/app/api-reference/functions/cookies)和[`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional)等动态API会将整个路由（若在[根布局](/docs/app/building-your-application/routing/layouts-and-templates#root-layout-required)中使用则会影响整个应用）切换至[动态渲染模式](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。确保动态API的使用是必要的，并在适当时用`<Suspense>`边界包裹。

> **须知**: [部分预渲染 (实验性功能)](/blog/next-14#partial-prerendering-preview)将允许路由部分内容动态化，而无需将整个路由转为动态渲染。

</AppOnly>

<PagesOnly>

- **[`<Link>`组件](/docs/pages/building-your-application/routing/linking-and-navigating):** 使用该组件实现客户端导航和预获取功能。
- **[自定义错误页](/docs/pages/building-your-application/routing/custom-error):** 优雅处理[500](/docs/pages/building-your-application/routing/custom-error#500-page)和[404错误](/docs/pages/building-your-application/routing/custom-error#404-page)。

</PagesOnly>

### 数据获取与缓存

<AppOnly>

- **[服务端组件](/docs/app/building-your-application/data-fetching/fetching):** 充分利用服务端组件在服务器端获取数据的优势。
- **[路由处理器 (Route Handlers)](/docs/app/building-your-application/routing/route-handlers):** 通过路由处理器从客户端组件访问后端资源。但避免在服务端组件中调用路由处理器，以防止额外的服务器请求。
- **[流式传输 (Streaming)](/docs/app/building-your-application/routing/loading-ui-and-streaming):** 使用加载UI和React Suspense逐步从服务端向客户端发送UI，防止数据获取期间阻塞整个路由。
- **[并行数据获取 (Parallel Data Fetching)](/docs/app/getting-started/fetching-data#parallel-data-fetching):** 在适当时通过并行获取数据减少网络瀑布流。同时考虑在必要时[预加载数据](/docs/app/getting-started/fetching-data#preloading-data)。
- **[数据缓存 (Data Caching)](/docs/app/deep-dive/caching#data-cache):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用`fetch`的请求也能被[缓存](/docs/app/api-reference/functions/unstable_cache)。
- **[静态资源 (Static Images)](/docs/app/api-reference/file-conventions/public-folder):** 使用`public`目录自动缓存应用的静态资源（如图像）。

</AppOnly>

<PagesOnly>

- **[API路由](/docs/pages/building-your-application/routing/api-routes):** 使用路由处理器访问后端资源，防止敏感信息暴露给客户端。
- **[数据缓存](/docs/pages/building-your-application/data-fetching/get-static-props):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用`getStaticProps`的请求在适当时被缓存。
- **[增量静态再生 (Incremental Static Regeneration)](/docs/pages/guides/incremental-static-regeneration):** 使用该功能在构建后更新静态页面，而无需重新构建整个站点。
- **[静态资源](/docs/pages/api-reference/file-conventions/public-folder):** 使用`public`目录自动缓存应用的静态资源（如图像）。

</PagesOnly>

### UI与可访问性

<AppOnly>

- **[表单与验证](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#forms):** 使用服务端动作 (Server Actions) 处理表单提交、服务端验证和错误处理。

</AppOnly>

- **[字体模块](/docs/app/api-reference/components/font):** 通过字体模块优化字体，自动托管字体文件与其他静态资源，消除外部网络请求，减少[布局偏移 (layout shift)](https://web.dev/articles/cls)。
- **[`<Image>`组件](/docs/app/api-reference/components/image):** 使用该组件优化图像，自动转换现代格式（如WebP），防止布局偏移。
- **[`<Script>`组件](/docs/app/guides/scripts):** 使用该组件优化第三方脚本，自动延迟加载避免阻塞主线程。
- **[ESLint](/docs/architecture/accessibility#linting):** 使用内置的`eslint-plugin-jsx-a11y`插件早期发现可访问性问题。

### 安全性

<AppOnly>

- **[数据污染防护 (Tainting)](/docs/app/api-reference/config/next-config-js/taint):** 通过标记数据对象或特定值，防止敏感数据暴露给客户端。
- **[服务端动作](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** 确保用户有权调用服务端动作。查阅推荐的[安全实践](/blog/security-nextjs-server-components-actions)。

</AppOnly>

- **[环境变量](/docs/app/guides/environment-variables):** 确保`.env.*`文件已加入`.gitignore`，且只有公共变量以`NEXT_PUBLIC_`为前缀。
- **[内容安全策略 (Content Security Policy)](/docs/app/guides/content-security-policy):** 考虑添加CSP策略防范XSS、点击劫持等代码注入攻击。

### 元数据与SEO

<AppOnly>

- **[元数据API](/docs/app/getting-started/metadata-and-og-images):** 使用该API添加页面标题、描述等信息，提升搜索引擎优化(SEO)。
- **[开放图谱(OG)图片](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** 创建OG图片优化社交分享效果。
- **[站点地图](/docs/app/api-reference/functions/generate-sitemaps)与[robots文件](/docs/app/api-reference/file-conventions/metadata/robots):** 生成这些文件帮助搜索引擎抓取和索引页面。

</AppOnly>

<PagesOnly>

- **[`<Head>`组件](/docs/pages/api-reference/components/head):** 使用该组件添加页面标题、描述等元信息。

</PagesOnly>

### 类型安全

- **TypeScript与[TS插件](/docs/app/api-reference/config/typescript):** 使用TypeScript及其插件增强类型安全，早期发现错误。

## 上线前准备

在上线前，可运行`next build`本地构建应用并捕获构建错误，然后运行`next start`在类生产环境中测量性能。

### 核心Web指标

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** 在隐身模式下运行测试，了解用户体验并发现改进点（需结合真实用户的核心Web指标数据）。

<AppOnly>

- **[`useReportWebVitals`钩子](/docs/app/api-reference/functions/use-report-web-vitals):** 使用该钩子向分析工具发送[核心Web指标](https://web.dev/articles/vitals)数据。

</AppOnly>

### 包分析

使用[`@next/bundle-analyzer`插件](/docs/app/guides/package-bundling#analyzing-javascript-bundles)分析JavaScript包大小，识别可能影响性能的大型模块和依赖。

以下工具可帮助评估新依赖的影响：
- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
