在将 Next.js 应用部署至生产环境前，您应当考虑实施以下优化措施和模式，以获得最佳的用户体验、性能和安全性。

本页提供了最佳实践指南，可作为[应用开发阶段](#during-development)和[上线前准备](#before-going-to-production)的参考，同时介绍了您需要了解的[Next.js自动优化机制](#automatic-optimizations)。

## 自动优化

以下 Next.js 优化功能默认启用且无需配置：

<AppOnly>

- **[服务端组件 (Server Components)](/docs/app/getting-started/server-and-client-components):** Next.js 默认使用服务端组件。这些组件在服务端运行，无需客户端 JavaScript 即可渲染，因此不会增加客户端 JavaScript 包的大小。您可以根据需要使用[客户端组件 (Client Components)](/docs/app/getting-started/server-and-client-components) 来实现交互功能。
- **[代码分割 (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** 服务端组件支持按路由段自动进行代码分割。您还可以考虑在适当场景下[懒加载 (lazy loading)](/docs/app/guides/lazy-loading) 客户端组件和第三方库。
- **[预取 (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** 当用户视图中出现新路由链接时，Next.js 会在后台预取该路由，实现近乎即时的导航体验。您可以在必要时选择禁用预取功能。
- **[静态渲染 (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js 在构建时会在服务端静态渲染服务端和客户端组件，并缓存渲染结果以提升性能。您可以为特定路由选择启用[动态渲染 (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。{/* TODO: 待PPR稳定后更新 */}
- **[缓存 (Caching)](/docs/app/deep-dive/caching):** Next.js 会缓存数据请求、服务端和客户端组件的渲染结果、静态资源等，以减少对服务器、数据库和后端服务的网络请求。您可以在必要时选择退出缓存机制。

</AppOnly>

<PagesOnly>

- **[代码分割 (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js 会按页面自动分割应用代码，确保导航时只加载当前页面所需的代码。您还可以考虑在适当场景下[懒加载 (lazy loading)](/docs/pages/guides/lazy-loading) 第三方库。
- **[预取 (Prefetching)](/docs/pages/api-reference/components/link#prefetch):** 当用户视图中出现新路由链接时，Next.js 会在后台预取该路由，实现近乎即时的导航体验。您可以在必要时选择禁用预取功能。
- **[自动静态优化 (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** 如果页面没有阻塞性数据需求，Next.js 会自动判定其为静态页面（可预渲染）。优化后的页面可被缓存，并通过多个CDN节点分发给终端用户。您可以在必要时选择启用[服务端渲染 (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props)。

</PagesOnly>

这些默认优化旨在提升应用性能，减少每次网络请求传输的数据量和成本。

## 开发阶段

在构建应用时，我们建议使用以下功能来确保最佳性能和用户体验：

### 路由与渲染

<AppOnly>

- **[布局 (Layouts)](/docs/app/api-reference/file-conventions/layout):** 使用布局在页面间共享UI，并启用导航时的[部分渲染 (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
- **[`<Link>`组件](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** 使用`<Link>`组件实现[客户端导航和预取](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works)。
- **[错误处理 (Error Handling)](/docs/app/getting-started/error-handling):** 通过创建自定义错误页面，优雅处理生产环境的[全局错误](/docs/app/getting-started/error-handling)和[404错误](/docs/app/api-reference/file-conventions/not-found)。
- **[客户端与服务端组件](/docs/app/getting-started/server-and-client-components#examples):** 遵循服务端与客户端组件的推荐组合模式，检查[`"use client"`边界](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree)的放置位置，避免不必要地增加客户端JavaScript包大小。
- **[动态API](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** 注意如[`cookies`](/docs/app/api-reference/functions/cookies)和[`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional)等动态API会使整个路由进入[动态渲染模式](/docs/app/getting-started/partial-prerendering#dynamic-rendering)（如果在[根布局](/docs/app/api-reference/file-conventions/layout#root-layout)中使用则会影响整个应用）。确保动态API的使用是经过深思熟虑的，并在适当时用`<Suspense>`边界包裹它们。

> **须知**: [部分预渲染 (Partial Prerendering)（实验性功能）](/blog/next-14#partial-prerendering-preview)将允许路由部分内容保持动态，而无需使整个路由进入动态渲染模式。

</AppOnly>

<PagesOnly>

- **[`<Link>`组件](/docs/pages/building-your-application/routing/linking-and-navigating):** 使用`<Link>`组件实现客户端导航和预取功能。
- **[自定义错误页](/docs/pages/building-your-application/routing/custom-error):** 优雅处理[500](/docs/pages/building-your-application/routing/custom-error#500-page)和[404错误](/docs/pages/building-your-application/routing/custom-error#404-page)。

</PagesOnly>

### 数据获取与缓存

<AppOnly>

- **[服务端组件](/docs/app/getting-started/fetching-data):** 充分利用服务端组件在服务端获取数据的优势。
- **[路由处理器 (Route Handlers)](/docs/app/building-your-application/routing/route-handlers):** 使用路由处理器从客户端组件访问后端资源。但不要从服务端组件调用路由处理器，以避免额外的服务端请求。
- **[流式传输 (Streaming)](/docs/app/building-your-application/routing/loading-ui-and-streaming):** 使用加载UI和React Suspense逐步将UI从服务端发送到客户端，防止数据获取时阻塞整个路由。
- **[并行数据获取 (Parallel Data Fetching)](/docs/app/getting-started/fetching-data#parallel-data-fetching):** 在适当时通过并行获取数据减少网络瀑布流。同时考虑在适当时[预加载数据](/docs/app/getting-started/fetching-data#preloading-data)。
- **[数据缓存 (Data Caching)](/docs/app/deep-dive/caching#data-cache):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用`fetch`的请求也能被[缓存](/docs/app/api-reference/functions/unstable_cache)。
- **[静态图片](/docs/app/api-reference/file-conventions/public-folder):** 使用`public`目录自动缓存应用的静态资源（如图片）。

</AppOnly>

<PagesOnly>

- **[API路由](/docs/pages/building-your-application/routing/api-routes):** 使用路由处理器访问后端资源，防止敏感信息暴露给客户端。
- **[数据缓存](/docs/pages/building-your-application/data-fetching/get-static-props):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用`getStaticProps`的请求在适当时也能被缓存。
- **[增量静态再生 (Incremental Static Regeneration)](/docs/pages/guides/incremental-static-regeneration):** 使用增量静态再生功能在构建后更新静态页面，而无需重建整个站点。
- **[静态图片](/docs/pages/api-reference/file-conventions/public-folder):** 使用`public`目录自动缓存应用的静态资源（如图片）。

</PagesOnly>

### UI与可访问性

<AppOnly>

- **[表单与验证](/docs/app/guides/forms):** 使用服务端操作 (Server Actions) 处理表单提交、服务端验证和错误处理。

</AppOnly>

- **[字体模块](/docs/app/api-reference/components/font):** 使用字体模块优化字体，自动托管字体文件与其他静态资源，消除外部网络请求，减少[布局偏移 (layout shift)](https://web.dev/articles/cls)。
- **[`<Image>`组件](/docs/app/api-reference/components/image):** 使用图片组件优化图片，自动优化图片、防止布局偏移，并提供WebP等现代格式。
- **[`<Script>`组件](/docs/app/guides/scripts):** 使用脚本组件优化第三方脚本，自动延迟加载脚本，防止阻塞主线程。
- **[ESLint](/docs/architecture/accessibility#linting):** 使用内置的`eslint-plugin-jsx-a11y`插件及早发现可访问性问题。

### 安全性

<AppOnly>

- **[数据污染防护 (Tainting)](/docs/app/api-reference/config/next-config-js/taint):** 通过标记数据对象或特定值，防止敏感数据暴露给客户端。
- **[服务端操作 (Server Actions)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** 确保用户有权调用服务端操作。查阅推荐的[安全实践](/blog/security-nextjs-server-components-actions)。

</AppOnly>

- **[环境变量](/docs/app/guides/environment-variables):** 确保`.env.*`文件已添加到`.gitignore`，且只有公共变量才添加`NEXT_PUBLIC_`前缀。
- **[内容安全策略 (Content Security Policy)](/docs/app/guides/content-security-policy):** 考虑添加内容安全策略，保护应用免受跨站脚本、点击劫持等代码注入攻击。

### 元数据与SEO

<AppOnly>

- **[元数据API](/docs/app/getting-started/metadata-and-og-images):** 使用元数据API通过添加页面标题、描述等提升应用的搜索引擎优化(SEO)。
- **[Open Graph (OG)图片](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** 创建OG图片为社交分享做好准备。
- **[站点地图 (Sitemaps)](/docs/app/api-reference/functions/generate-sitemaps)和[robots文件](/docs/app/api-reference/file-conventions/metadata/robots):** 通过生成站点地图和robots文件帮助搜索引擎抓取和索引页面。

</AppOnly>

<PagesOnly>

- **[`<Head>`组件](/docs/pages/api-reference/components/head):** 使用`next/head`组件添加页面标题、描述等信息。

</PagesOnly>

### 类型安全

- **TypeScript和[TS插件](/docs/app/api-reference/config/typescript):** 使用TypeScript和TypeScript插件获得更好的类型安全，帮助及早发现错误。

## 上线前准备

在上线前，您可以运行`next build`在本地构建应用并捕获构建错误，然后运行`next start`测量应用在生产环境下的性能表现。

### 核心Web指标

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** 在隐身模式下运行Lighthouse，更好地理解用户体验并发现改进点。这是模拟测试，应与实际数据（如核心Web指标）结合分析。

<AppOnly>

- **[`useReportWebVitals`钩子](/docs/app/api-reference/functions/use-report-web-vitals):** 使用此钩子向分析工具发送[核心Web指标](https://web.dev/articles/vitals)数据。

</AppOnly>

### 包分析

使用[`@next/bundle-analyzer`插件](/docs/app/guides/package-bundling#analyzing-javascript-bundles)分析JavaScript包大小，识别可能影响性能的大型模块和依赖项。

此外，以下工具可帮助您了解添加新依赖对应用的影响：

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
