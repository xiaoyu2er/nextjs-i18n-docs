在将 Next.js 应用部署至生产环境前，您应考虑实施以下优化措施和模式，以获得最佳用户体验、性能和安全性。

本文档提供了在[应用开发阶段](#during-development)和[上线前准备](#before-going-to-production)可参考的最佳实践，以及您需要了解的[Next.js 自动优化机制](#automatic-optimizations)。

## 自动优化

以下 Next.js 优化功能默认启用且无需配置：

<AppOnly>

- **[服务端组件 (Server Components)](/docs/app/getting-started/server-and-client-components):** Next.js 默认使用服务端组件。这些组件在服务端运行，无需客户端 JavaScript 即可渲染，因此不会增加客户端 JavaScript 包体积。您可按需使用[客户端组件 (Client Components)](/docs/app/getting-started/server-and-client-components) 实现交互功能。
- **[代码分割 (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** 服务端组件支持按路由段自动代码分割。您还可考虑在适当时[懒加载 (lazy loading)](/docs/app/guides/lazy-loading) 客户端组件和第三方库。
- **[预获取 (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** 当用户视图中出现新路由链接时，Next.js 会在后台预获取该路由，实现近乎即时的导航体验。您可在适当时选择禁用此功能。
- **[静态渲染 (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js 在构建时于服务端静态渲染服务端和客户端组件，并缓存渲染结果以提升性能。您可为特定路由选择启用[动态渲染 (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。{/* TODO: 待 PPR 稳定后更新 */}
- **[缓存 (Caching)](/docs/app/deep-dive/caching):** Next.js 会缓存数据请求、服务端/客户端组件渲染结果、静态资源等，以减少对服务器、数据库和后端服务的网络请求。您可在适当时选择禁用缓存。

</AppOnly>

<PagesOnly>

- **[代码分割 (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js 自动按页面分割应用代码，确保导航时仅加载当前页面所需代码。您还可考虑在适当时[懒加载 (lazy loading)](/docs/pages/guides/lazy-loading) 第三方库。
- **[预获取 (Prefetching)](/docs/pages/api-reference/components/link#prefetch):** 当用户视图中出现新路由链接时，Next.js 会在后台预获取该路由，实现近乎即时的导航体验。您可在适当时选择禁用此功能。
- **[自动静态优化 (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** 若页面无阻塞数据需求，Next.js 会自动判定为静态页面（可预渲染）。优化后的页面可被缓存，并通过多个 CDN 节点分发给终端用户。您可为特定路由选择启用[服务端渲染 (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props)。

</PagesOnly>

这些默认优化旨在提升应用性能，减少每次网络请求传输的数据量和成本。

## 开发阶段

在构建应用时，我们建议使用以下功能确保最佳性能和用户体验：

### 路由与渲染

<AppOnly>

- **[布局 (Layouts)](/docs/app/building-your-application/routing/layouts-and-templates#layouts):** 使用布局在页面间共享 UI，并启用导航时的[部分渲染 (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
- **[`<Link>` 组件](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** 使用 `<Link>` 组件实现[客户端导航和预获取 (client-side navigation and prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works)。
- **[错误处理 (Error Handling)](/docs/app/building-your-application/routing/error-handling):** 通过创建自定义错误页优雅处理生产环境的[全局错误 (catch-all errors)](/docs/app/building-your-application/routing/error-handling) 和 [404 错误](/docs/app/api-reference/file-conventions/not-found)。
- **[客户端与服务端组件 (Client and Server Components)](/docs/app/getting-started/server-and-client-components#examples):** 遵循服务端与客户端组件的推荐组合模式，检查 [`"use client"` 边界](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree)的放置位置，避免不必要地增加客户端 JavaScript 包体积。
- **[动态 API (Dynamic APIs)](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** 注意 [`cookies`](/docs/app/api-reference/functions/cookies) 等动态 API 和 [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional) 属性会使整个路由进入[动态渲染 (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) 模式（若在[根布局 (Root Layout)](/docs/app/building-your-application/routing/layouts-and-templates#root-layout-required) 中使用则会影响整个应用）。确保动态 API 的使用是必要的，并在适当时用 `<Suspense>` 边界包裹。

> **须知**: [部分预渲染 (Partial Prerendering)（实验性）](/blog/next-14#partial-prerendering-preview) 将允许路由部分内容动态化，而无需使整个路由进入动态渲染模式。

</AppOnly>

<PagesOnly>

- **[`<Link>` 组件](/docs/pages/building-your-application/routing/linking-and-navigating):** 使用 `<Link>` 组件实现客户端导航和预获取。
- **[自定义错误页 (Custom Errors)](/docs/pages/building-your-application/routing/custom-error):** 优雅处理 [500](/docs/pages/building-your-application/routing/custom-error#500-page) 和 [404 错误](/docs/pages/building-your-application/routing/custom-error#404-page)。

</PagesOnly>

### 数据获取与缓存

<AppOnly>

- **[服务端组件 (Server Components)](/docs/app/building-your-application/data-fetching/fetching):** 充分利用服务端组件在服务端获取数据的优势。
- **[路由处理器 (Route Handlers)](/docs/app/building-your-application/routing/route-handlers):** 通过路由处理器从客户端组件访问后端资源。但避免从服务端组件调用路由处理器，以免产生额外的服务端请求。
- **[流式传输 (Streaming)](/docs/app/building-your-application/routing/loading-ui-and-streaming):** 使用加载 UI 和 React Suspense 逐步从服务端向客户端发送 UI，防止数据获取时阻塞整个路由。
- **[并行数据获取 (Parallel Data Fetching)](/docs/app/building-your-application/data-fetching/fetching#parallel-and-sequential-data-fetching):** 在适当时通过并行获取数据减少网络瀑布流。同时考虑在适当时[预加载数据 (preloading data)](/docs/app/building-your-application/data-fetching/fetching#preloading-data)。
- **[数据缓存 (Data Caching)](/docs/app/deep-dive/caching#data-cache):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用 `fetch` 的请求被[缓存](/docs/app/api-reference/functions/unstable_cache)。
- **[静态资源 (Static Images)](/docs/app/api-reference/file-conventions/public-folder):** 使用 `public` 目录自动缓存应用的静态资源（如图片）。

</AppOnly>

<PagesOnly>

- **[API 路由 (API Routes)](/docs/pages/building-your-application/routing/api-routes):** 使用路由处理器访问后端资源，防止敏感信息暴露给客户端。
- **[数据缓存 (Data Caching)](/docs/pages/building-your-application/data-fetching/get-static-props):** 验证数据请求是否被缓存，并在适当时启用缓存。确保未使用 `getStaticProps` 的请求在适当时被缓存。
- **[增量静态再生 (Incremental Static Regeneration)](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration):** 使用增量静态再生在构建后更新静态页面，而无需重建整个站点。
- **[静态资源 (Static Images)](/docs/pages/api-reference/file-conventions/public-folder):** 使用 `public` 目录自动缓存应用的静态资源（如图片）。

</PagesOnly>

### UI 与可访问性

<AppOnly>

- **[表单与验证 (Forms and Validation)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#forms):** 使用服务端操作 (Server Actions) 处理表单提交、服务端验证和错误处理。

</AppOnly>

- **[字体模块 (Font Module)](/docs/app/api-reference/components/font):** 使用字体模块优化字体，该模块会自动托管字体文件与其他静态资源，消除外部网络请求并减少[布局偏移 (layout shift)](https://web.dev/articles/cls)。
- **[`<Image>` 组件](/docs/app/api-reference/components/image):** 使用 Image 组件优化图片，该组件会自动优化图片、防止布局偏移，并以 WebP 等现代格式提供。
- **[`<Script>` 组件](/docs/app/guides/scripts):** 使用 Script 组件优化第三方脚本，该组件会自动延迟脚本加载，防止阻塞主线程。
- **[ESLint](/docs/architecture/accessibility#linting):** 使用内置的 `eslint-plugin-jsx-a11y` 插件及早发现可访问性问题。

### 安全性

<AppOnly>

- **[数据标记 (Tainting)](/docs/app/building-your-application/data-fetching/fetching#preventing-sensitive-data-from-being-exposed-to-the-client):** 通过标记数据对象或特定值，防止敏感数据暴露给客户端。
- **[服务端操作 (Server Actions)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** 确保用户有权调用服务端操作。查阅推荐的[安全实践](/blog/security-nextjs-server-components-actions)。

</AppOnly>

- **[环境变量 (Environment Variables)](/docs/app/guides/environment-variables):** 确保 `.env.*` 文件已添加到 `.gitignore`，且仅公开变量以 `NEXT_PUBLIC_` 为前缀。
- **[内容安全策略 (Content Security Policy)](/docs/app/guides/content-security-policy):** 考虑添加内容安全策略，防止应用遭受跨站脚本、点击劫持等代码注入攻击。

### 元数据与 SEO

<AppOnly>

- **[元数据 API (Metadata API)](/docs/app/getting-started/metadata-and-og-images):** 使用元数据 API 添加页面标题、描述等信息，提升应用的搜索引擎优化 (SEO)。
- **[Open Graph (OG) 图片](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** 创建 OG 图片为社交分享做准备。
- **[站点地图 (Sitemaps)](/docs/app/api-reference/functions/generate-sitemaps) 和 [Robots 文件](/docs/app/api-reference/file-conventions/metadata/robots):** 通过生成站点地图和 robots 文件帮助搜索引擎抓取和索引页面。

</AppOnly>

<PagesOnly>

- **[`<Head>` 组件](/docs/pages/api-reference/components/head):** 使用 `next/head` 组件添加页面标题、描述等信息。

</PagesOnly>

### 类型安全

- **TypeScript 和 [TS 插件](/docs/app/api-reference/config/typescript):** 使用 TypeScript 和 TypeScript 插件提升类型安全性，及早发现错误。

## 上线前准备

在上线前，可运行 `next build` 在本地构建应用并捕获构建错误，然后运行 `next start` 测量应用在生产环境下的性能表现。

### 核心 Web 指标

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** 在隐身模式下运行 Lighthouse，更好地理解用户体验并识别改进点。此为模拟测试，应与实际数据（如核心 Web 指标）结合分析。

<AppOnly>

- **[`useReportWebVitals` 钩子](/docs/app/api-reference/functions/use-report-web-vitals):** 使用此钩子向分析工具发送[核心 Web 指标 (Core Web Vitals)](https://web.dev/articles/vitals) 数据。

</AppOnly>

### 包分析

使用 [`@next/bundle-analyzer` 插件](/docs/app/guides/package-bundling#analyzing-javascript-bundles) 分析 JavaScript 包大小，识别可能影响性能的大型模块和依赖。

此外，以下工具可帮助您了解新增依赖对应用的影响：

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
