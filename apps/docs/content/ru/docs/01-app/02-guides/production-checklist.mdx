Перед выгрузкой Next.js приложения в продакшен следует рассмотреть ряд оптимизаций и шаблонов, которые обеспечат лучший пользовательский опыт, производительность и безопасность.

На этой странице представлены лучшие практики, которые можно использовать в качестве справочника при [разработке приложения](#во-время-разработки) и [перед выгрузкой в продакшен](#перед-выгрузкой-в-продакшен), а также [автоматические оптимизации Next.js](#автоматические-оптимизации), о которых следует знать.

## Автоматические оптимизации

Эти оптимизации Next.js включены по умолчанию и не требуют настройки:

<AppOnly>

- **[Серверные компоненты (Server Components)](/docs/app/getting-started/server-and-client-components):** Next.js по умолчанию использует серверные компоненты. Они выполняются на сервере и не требуют JavaScript для рендеринга на клиенте, что не влияет на размер клиентских JavaScript-бандлов. Для интерактивности можно использовать [клиентские компоненты (Client Components)](/docs/app/getting-started/server-and-client-components).
- **[Разделение кода (Code-splitting)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** Серверные компоненты позволяют автоматически разделять код по сегментам маршрутов. Также можно использовать [ленивую загрузку (lazy loading)](/docs/app/guides/lazy-loading) клиентских компонентов и сторонних библиотек, где это уместно.
- **[Предзагрузка (Prefetching)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** Когда ссылка на новый маршрут попадает в область видимости пользователя, Next.js предзагружает маршрут в фоновом режиме, что делает навигацию практически мгновенной. При необходимости можно отключить предзагрузку.
- **[Статический рендеринг (Static Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js статически рендерит серверные и клиентские компоненты на сервере во время сборки и кэширует результат для повышения производительности. Для отдельных маршрутов можно включить [динамический рендеринг (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering). {/* TODO: Обновить, когда PPR станет стабильным */}
- **[Кэширование (Caching)](/docs/app/deep-dive/caching):** Next.js кэширует запросы данных, результаты рендеринга серверных и клиентских компонентов, статические ресурсы и другое, чтобы уменьшить количество сетевых запросов к серверу, базе данных и бэкенд-сервисам. При необходимости можно отключить кэширование.

</AppOnly>

<PagesOnly>

- **[Разделение кода (Code-splitting)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js автоматически разделяет код приложения по страницам, загружая только код, необходимый для текущей страницы. Также можно использовать [ленивую загрузку (lazy loading)](/docs/pages/guides/lazy-loading) сторонних библиотек, где это уместно.
- **[Предзагрузка (Prefetching)](/docs/pages/api-reference/components/link#prefetch):** Когда ссылка на новый маршрут попадает в область видимости пользователя, Next.js предзагружает маршрут в фоновом режиме, что делает навигацию практически мгновенной. При необходимости можно отключить предзагрузку.
- **[Автоматическая статическая оптимизация (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** Next.js автоматически определяет, что страница статическая (может быть предварительно отрендерена), если у неё нет блокирующих требований к данным. Оптимизированные страницы можно кэшировать и обслуживать пользователям из нескольких CDN-локаций. При необходимости можно включить [рендеринг на стороне сервера (Server-side Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props).

</PagesOnly>

Эти настройки по умолчанию направлены на улучшение производительности приложения и сокращение объёма передаваемых данных при каждом сетевом запросе.

## Во время разработки

При разработке приложения рекомендуется использовать следующие функции для обеспечения наилучшей производительности и пользовательского опыта:

### Маршрутизация и рендеринг

<AppOnly>

- **[Макеты (Layouts)](/docs/app/api-reference/file-conventions/layout):** Используйте макеты для совместного использования UI между страницами и включения [частичного рендеринга (partial rendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering) при навигации.
- **[Компонент `<Link>`](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** Используйте компонент `<Link>` для [клиентской навигации и предзагрузки](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works).
- **[Обработка ошибок (Error Handling)](/docs/app/getting-started/error-handling):** Грамотно обрабатывайте [все ошибки (catch-all errors)](/docs/app/getting-started/error-handling) и [404 ошибки](/docs/app/api-reference/file-conventions/not-found) в продакшене, создавая пользовательские страницы ошибок.
- **[Клиентские и серверные компоненты (Client and Server Components)](/docs/app/getting-started/server-and-client-components#examples):** Следуйте рекомендуемым шаблонам композиции для серверных и клиентских компонентов и проверяйте расположение ваших [`"use client"` границ](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree), чтобы избежать неоправданного увеличения клиентского JavaScript-бандла.
- **[Динамические API (Dynamic APIs)](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** Учтите, что динамические API, такие как [`cookies`](/docs/app/api-reference/functions/cookies) и пропс [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional), переведут весь маршрут в [динамический рендеринг (Dynamic Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) (или всё приложение, если они используются в [корневом макете (Root Layout)](/docs/app/api-reference/file-conventions/layout#root-layout)). Убедитесь, что использование динамических API обосновано, и при необходимости оборачивайте их в границы `<Suspense>`.

> **Полезно знать**: [Частичный предварительный рендеринг (Partial Prerendering, экспериментальная функция)](/blog/next-14#partial-prerendering-preview) позволит сделать части маршрута динамическими без перевода всего маршрута в динамический рендеринг.

</AppOnly>

<PagesOnly>

- **[Компонент `<Link>`](/docs/pages/building-your-application/routing/linking-and-navigating):** Используйте компонент `<Link>` для клиентской навигации и предзагрузки.
- **[Пользовательские ошибки (Custom Errors)](/docs/pages/building-your-application/routing/custom-error):** Грамотно обрабатывайте [500](/docs/pages/building-your-application/routing/custom-error#500-page) и [404 ошибки](/docs/pages/building-your-application/routing/custom-error#404-page).

</PagesOnly>

### Загрузка данных и кэширование

<AppOnly>

- **[Серверные компоненты (Server Components)](/docs/app/getting-started/fetching-data):** Используйте преимущества загрузки данных на сервере с помощью серверных компонентов.
- **[Обработчики маршрутов (Route Handlers)](/docs/app/building-your-application/routing/route-handlers):** Используйте обработчики маршрутов для доступа к бэкенд-ресурсам из клиентских компонентов. Не вызывайте обработчики маршрутов из серверных компонентов, чтобы избежать дополнительных серверных запросов.
- **[Стриминг (Streaming)](/docs/app/building-your-application/routing/loading-ui-and-streaming):** Используйте загрузочный UI и React Suspense для постепенной отправки UI с сервера на клиент, предотвращая блокировку всего маршрута во время загрузки данных.
- **[Параллельная загрузка данных (Parallel Data Fetching)](/docs/app/getting-started/fetching-data#parallel-data-fetching):** Уменьшайте водопады запросов, загружая данные параллельно, где это уместно. Также рассмотрите возможность [предзагрузки данных (preloading data)](/docs/app/getting-started/fetching-data#preloading-data).
- **[Кэширование данных (Data Caching)](/docs/app/deep-dive/caching#data-cache):** Проверяйте, кэшируются ли ваши запросы данных, и включайте кэширование, где это уместно. Убедитесь, что запросы, не использующие `fetch`, [кэшируются](/docs/app/api-reference/functions/unstable_cache).
- **[Статические изображения (Static Images)](/docs/app/api-reference/file-conventions/public-folder):** Используйте директорию `public` для автоматического кэширования статических ресурсов приложения, например, изображений.

</AppOnly>

<PagesOnly>

- **[API-маршруты (API Routes)](/docs/pages/building-your-application/routing/api-routes):** Используйте обработчики маршрутов для доступа к бэкенд-ресурсам и предотвращения раскрытия чувствительных данных клиенту.
- **[Кэширование данных (Data Caching)](/docs/pages/building-your-application/data-fetching/get-static-props):** Проверяйте, кэшируются ли ваши запросы данных, и включайте кэширование, где это уместно. Убедитесь, что запросы, не использующие `getStaticProps`, кэшируются при необходимости.
- **[Инкрементальная статическая регенерация (Incremental Static Regeneration)](/docs/pages/guides/incremental-static-regeneration):** Используйте инкрементальную статическую регенерацию для обновления статических страниц после их сборки без пересборки всего сайта.
- **[Статические изображения (Static Images)](/docs/pages/api-reference/file-conventions/public-folder):** Используйте директорию `public` для автоматического кэширования статических ресурсов приложения, например, изображений.

</PagesOnly>

### UI и доступность

<AppOnly>

- **[Формы и валидация (Forms and Validation)](/docs/app/guides/forms):** Используйте серверные действия (Server Actions) для обработки отправки форм, валидации на сервере и обработки ошибок.

</AppOnly>

- **[Модуль шрифтов (Font Module)](/docs/app/api-reference/components/font):** Оптимизируйте шрифты с помощью модуля шрифтов, который автоматически размещает ваши файлы шрифтов вместе с другими статическими ресурсами, удаляет внешние сетевые запросы и уменьшает [сдвиг макета (layout shift)](https://web.dev/articles/cls).
- **[Компонент `<Image>`](/docs/app/api-reference/components/image):** Оптимизируйте изображения с помощью компонента Image, который автоматически оптимизирует изображения, предотвращает сдвиг макета и обслуживает их в современных форматах, таких как WebP.
- **[Компонент `<Script>`](/docs/app/guides/scripts):** Оптимизируйте сторонние скрипты с помощью компонента Script, который автоматически откладывает скрипты и предотвращает их блокировку основного потока.
- **[ESLint](/docs/architecture/accessibility#linting):** Используйте встроенный плагин `eslint-plugin-jsx-a11y` для раннего выявления проблем с доступностью.

### Безопасность

<AppOnly>

- **[Загрязнение данных (Tainting)](/docs/app/api-reference/config/next-config-js/taint):** Предотвращайте раскрытие чувствительных данных клиенту, помечая данные или конкретные значения как загрязнённые.
- **[Серверные действия (Server Actions)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** Убедитесь, что пользователи авторизованы для вызова серверных действий. Ознакомьтесь с рекомендуемыми [практиками безопасности](/blog/security-nextjs-server-components-actions).

</AppOnly>

- **[Переменные окружения (Environment Variables)](/docs/app/guides/environment-variables):** Убедитесь, что ваши файлы `.env.*` добавлены в `.gitignore` и только публичные переменные имеют префикс `NEXT_PUBLIC_`.
- **[Политика безопасности контента (Content Security Policy)](/docs/app/guides/content-security-policy):** Рассмотрите возможность добавления политики безопасности контента для защиты приложения от различных угроз, таких как межсайтовый скриптинг, кликджекинг и другие атаки с внедрением кода.

### Метаданные и SEO

<AppOnly>

- **[API метаданных (Metadata API)](/docs/app/getting-started/metadata-and-og-images):** Используйте API метаданных для улучшения поисковой оптимизации (SEO) вашего приложения, добавляя заголовки страниц, описания и другое.
- **[Open Graph (OG) изображения](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** Создавайте OG-изображения для подготовки вашего приложения к публикации в соцсетях.
- **[Карты сайта (Sitemaps)](/docs/app/api-reference/functions/generate-sitemaps) и [файлы robots](/docs/app/api-reference/file-conventions/metadata/robots):** Помогите поисковым системам сканировать и индексировать ваши страницы, генерируя карты сайта и файлы robots.

</AppOnly>

<PagesOnly>

- **[Компонент `<Head>`](/docs/pages/api-reference/components/head):** Используйте компонент `next/head` для добавления заголовков страниц, описаний и другого.

</PagesOnly>

### Типобезопасность

- **TypeScript и [TS Plugin](/docs/app/api-reference/config/typescript):** Используйте TypeScript и плагин TypeScript для улучшения типобезопасности и раннего выявления ошибок.

## Перед выгрузкой в продакшен

Перед выгрузкой в продакшен можно выполнить `next build` для локальной сборки приложения и выявления ошибок сборки, а затем `next start` для измерения производительности приложения в среде, приближенной к продакшену.

### Core Web Vitals

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** Запустите Lighthouse в режиме инкогнито, чтобы лучше понять, как пользователи будут воспринимать ваш сайт, и выявить области для улучшения. Это симулированный тест, который следует дополнять анализом реальных данных (например, Core Web Vitals).

<AppOnly>

- **[Хук `useReportWebVitals`](/docs/app/api-reference/functions/use-report-web-vitals):** Используйте этот хук для отправки данных [Core Web Vitals](https://web.dev/articles/vitals) в инструменты аналитики.

</AppOnly>

### Анализ бандлов

Используйте плагин [`@next/bundle-analyzer`](/docs/app/guides/package-bundling#analyzing-javascript-bundles) для анализа размера ваших JavaScript-бандлов и выявления крупных модулей и зависимостей, которые могут влиять на производительность приложения.
