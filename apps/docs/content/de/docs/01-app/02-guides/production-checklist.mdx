Bevor Sie Ihre Next.js-Anwendung in die Produktion überführen, sollten Sie einige Optimierungen und Muster implementieren, um die beste Benutzererfahrung, Leistung und Sicherheit zu gewährleisten.

Diese Seite bietet Best Practices, die Sie als Referenz nutzen können, während Sie [Ihre Anwendung entwickeln](#während-der-entwicklung) und [vor dem Livegang](#vor-dem-livegang), sowie Informationen zu den [automatischen Next.js-Optimierungen](#automatische-optimierungen), die Sie kennen sollten.

## Automatische Optimierungen

Diese Next.js-Optimierungen sind standardmäßig aktiviert und erfordern keine Konfiguration:

<AppOnly>

- **[Server Components (Server-Komponenten)](/docs/app/getting-started/server-and-client-components):** Next.js verwendet standardmäßig Server Components. Server Components werden auf dem Server ausgeführt und benötigen kein JavaScript für das Rendering auf dem Client. Daher haben sie keine Auswirkungen auf die Größe Ihrer clientseitigen JavaScript-Bundles. Sie können bei Bedarf [Client Components (Client-Komponenten)](/docs/app/getting-started/server-and-client-components) für Interaktivität verwenden.
- **[Code-Splitting (Code-Aufteilung)](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works):** Server Components ermöglichen eine automatische Code-Aufteilung nach Routensegmenten. Sie können auch [Lazy Loading (verzögertes Laden)](/docs/app/guides/lazy-loading) für Client Components und Drittanbieter-Bibliotheken in Betracht ziehen, wo dies sinnvoll ist.
- **[Prefetching (Vorabrufen)](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching):** Wenn ein Link zu einer neuen Route in den Sichtbereich des Benutzers gelangt, ruft Next.js die Route im Hintergrund vorab. Dies macht die Navigation zu neuen Routen nahezu verzögerungsfrei. Sie können das Prefetching bei Bedarf deaktivieren.
- **[Static Rendering (Statisches Rendering)](/docs/app/getting-started/partial-prerendering#static-rendering):** Next.js rendert Server- und Client-Komponenten beim Build statisch auf dem Server und speichert das gerenderte Ergebnis zwischen, um die Leistung Ihrer Anwendung zu verbessern. Sie können bei Bedarf [Dynamic Rendering (Dynamisches Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) für bestimmte Routen aktivieren. {/* TODO: Update when PPR is stable */}
- **[Caching (Zwischenspeicherung)](/docs/app/deep-dive/caching):** Next.js speichert Datenanfragen, das gerenderte Ergebnis von Server- und Client-Komponenten, statische Assets und mehr zwischen, um die Anzahl der Netzwerkanfragen an Ihren Server, Ihre Datenbank und Backend-Dienste zu reduzieren. Sie können die Zwischenspeicherung bei Bedarf deaktivieren.

</AppOnly>

<PagesOnly>

- **[Code-Splitting (Code-Aufteilung)](/docs/pages/building-your-application/routing/pages-and-layouts):** Next.js teilt Ihren Anwendungscode automatisch nach Seiten auf. Das bedeutet, dass bei der Navigation nur der Code für die aktuelle Seite geladen wird. Sie können auch [Lazy Loading (verzögertes Laden)](/docs/pages/guides/lazy-loading) für Drittanbieter-Bibliotheken in Betracht ziehen, wo dies sinnvoll ist.
- **[Prefetching (Vorabrufen)](/docs/pages/api-reference/components/link#prefetch):** Wenn ein Link zu einer neuen Route in den Sichtbereich des Benutzers gelangt, ruft Next.js die Route im Hintergrund vorab. Dies macht die Navigation zu neuen Routen nahezu verzögerungsfrei. Sie können das Prefetching bei Bedarf deaktivieren.
- **[Automatic Static Optimization (Automatische statische Optimierung)](/docs/pages/building-your-application/rendering/automatic-static-optimization):** Next.js erkennt automatisch, ob eine Seite statisch ist (vorgerendert werden kann), wenn sie keine blockierenden Datenanforderungen hat. Optimierte Seiten können zwischengespeichert und dem Endbenutzer von mehreren CDN-Standorten aus bereitgestellt werden. Sie können bei Bedarf [Server-side Rendering (Serverseitiges Rendering)](/docs/pages/building-your-application/data-fetching/get-server-side-props) aktivieren.

</PagesOnly>

Diese Standardeinstellungen zielen darauf ab, die Leistung Ihrer Anwendung zu verbessern und die Kosten sowie die Menge der übertragenen Daten bei jeder Netzwerkanfrage zu reduzieren.

## Während der Entwicklung

Während Sie Ihre Anwendung entwickeln, empfehlen wir die Verwendung der folgenden Funktionen, um die beste Leistung und Benutzererfahrung zu gewährleisten:

### Routing und Rendering

<AppOnly>

- **[Layouts](/docs/app/api-reference/file-conventions/layout):** Verwenden Sie Layouts, um UI über Seiten hinweg zu teilen und [Partial Rendering (Teilrendering)](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering) bei der Navigation zu ermöglichen.
- **[`<Link>`-Komponente](/docs/app/building-your-application/routing/linking-and-navigating#link-component):** Verwenden Sie die `<Link>`-Komponente für [Client-seitige Navigation und Prefetching](/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works).
- **[Error Handling (Fehlerbehandlung)](/docs/app/getting-started/error-handling):** Behandeln Sie [Catch-All-Fehler](/docs/app/getting-started/error-handling) und [404-Fehler](/docs/app/api-reference/file-conventions/not-found) in der Produktion elegant, indem Sie benutzerdefinierte Fehlerseiten erstellen.
- **[Client und Server Components (Client- und Server-Komponenten)](/docs/app/getting-started/server-and-client-components#examples):** Befolgen Sie die empfohlenen Kompositionsmuster für Server- und Client-Komponenten und überprüfen Sie die Platzierung Ihrer [`"use client"`-Grenzen](/docs/app/getting-started/server-and-client-components#examples#moving-client-components-down-the-tree), um eine unnötige Vergrößerung Ihres clientseitigen JavaScript-Bundles zu vermeiden.
- **[Dynamic APIs (Dynamische APIs)](/docs/app/getting-started/partial-prerendering#dynamic-rendering):** Beachten Sie, dass dynamische APIs wie [`cookies`](/docs/app/api-reference/functions/cookies) und die [`searchParams`](/docs/app/api-reference/file-conventions/page#searchparams-optional)-Prop die gesamte Route für [Dynamic Rendering (Dynamisches Rendering)](/docs/app/getting-started/partial-prerendering#dynamic-rendering) aktivieren (oder Ihre gesamte Anwendung, wenn sie im [Root Layout (Stamm-Layout)](/docs/app/api-reference/file-conventions/layout#root-layout) verwendet werden). Stellen Sie sicher, dass die Verwendung dynamischer APIs absichtlich erfolgt, und umschließen Sie sie bei Bedarf mit `<Suspense>`-Grenzen.

> **Gut zu wissen**: [Partial Prerendering (Teilweises Vorrendern, experimentell)](/blog/next-14#partial-prerendering-preview) ermöglicht es, Teile einer Route dynamisch zu gestalten, ohne die gesamte Route für dynamisches Rendering zu aktivieren.

</AppOnly>

<PagesOnly>

- **[`<Link>`-Komponente](/docs/pages/building-your-application/routing/linking-and-navigating):** Verwenden Sie die `<Link>`-Komponente für clientseitige Navigation und Prefetching.
- **[Custom Errors (Benutzerdefinierte Fehler)](/docs/pages/building-your-application/routing/custom-error):** Behandeln Sie [500-](/docs/pages/building-your-application/routing/custom-error#500-page) und [404-Fehler](/docs/pages/building-your-application/routing/custom-error#404-page) elegant.

</PagesOnly>

### Datenabruf und Caching

<AppOnly>

- **[Server Components (Server-Komponenten)](/docs/app/getting-started/fetching-data):** Nutzen Sie die Vorteile des Datenabrufs auf dem Server mit Server Components.
- **[Route Handlers (Routen-Handler)](/docs/app/building-your-application/routing/route-handlers):** Verwenden Sie Route Handlers, um von Client Components auf Ihre Backend-Ressourcen zuzugreifen. Rufen Sie jedoch keine Route Handlers von Server Components auf, um zusätzliche Serveranfragen zu vermeiden.
- **[Streaming](/docs/app/building-your-application/routing/loading-ui-and-streaming):** Verwenden Sie Loading UI und React Suspense, um UI schrittweise vom Server an den Client zu senden und zu verhindern, dass die gesamte Route blockiert wird, während Daten abgerufen werden.
- **[Parallel Data Fetching (Paralleler Datenabruf)](/docs/app/getting-started/fetching-data#parallel-data-fetching):** Reduzieren Sie Netzwerk-Wasserfälle durch parallelen Datenabruf, wo dies sinnvoll ist. Ziehen Sie auch [Preloading (Vorabladen) von Daten](/docs/app/getting-started/fetching-data#preloading-data) in Betracht, wo dies sinnvoll ist.
- **[Data Caching (Daten-Zwischenspeicherung)](/docs/app/deep-dive/caching#data-cache):** Überprüfen Sie, ob Ihre Datenanfragen zwischengespeichert werden oder nicht, und aktivieren Sie die Zwischenspeicherung bei Bedarf. Stellen Sie sicher, dass Anfragen, die nicht `fetch` verwenden, [zwischengespeichert](/docs/app/api-reference/functions/unstable_cache) werden.
- **[Static Images (Statische Bilder)](/docs/app/api-reference/file-conventions/public-folder):** Verwenden Sie das `public`-Verzeichnis, um statische Assets Ihrer Anwendung automatisch zwischenzuspeichern, z.B. Bilder.

</AppOnly>

<PagesOnly>

- **[API Routes (API-Routen)](/docs/pages/building-your-application/routing/api-routes):** Verwenden Sie Route Handlers, um auf Backend-Ressourcen zuzugreifen und zu verhindern, dass sensible Geheimnisse dem Client preisgegeben werden.
- **[Data Caching (Daten-Zwischenspeicherung)](/docs/pages/building-your-application/data-fetching/get-static-props):** Überprüfen Sie, ob Ihre Datenanfragen zwischengespeichert werden oder nicht, und aktivieren Sie die Zwischenspeicherung bei Bedarf. Stellen Sie sicher, dass Anfragen, die nicht `getStaticProps` verwenden, bei Bedarf zwischengespeichert werden.
- **[Incremental Static Regeneration (Inkrementelle statische Regenerierung)](/docs/pages/guides/incremental-static-regeneration):** Verwenden Sie Incremental Static Regeneration, um statische Seiten nach dem Build zu aktualisieren, ohne Ihre gesamte Website neu zu erstellen.
- **[Static Images (Statische Bilder)](/docs/pages/api-reference/file-conventions/public-folder):** Verwenden Sie das `public`-Verzeichnis, um statische Assets Ihrer Anwendung automatisch zwischenzuspeichern, z.B. Bilder.

</PagesOnly>

### UI und Barrierefreiheit

<AppOnly>

- **[Forms and Validation (Formulare und Validierung)](/docs/app/guides/forms):** Verwenden Sie Server Actions, um Formularübermittlungen, serverseitige Validierung und Fehlerbehandlung zu verwalten.

</AppOnly>

- **[Font Module (Schriftarten-Modul)](/docs/app/api-reference/components/font):** Optimieren Sie Schriftarten mit dem Font Module, das Ihre Schriftartendateien automatisch mit anderen statischen Assets hostet, externe Netzwerkanfragen entfernt und [Layout Shift (Layoutverschiebung)](https://web.dev/articles/cls) reduziert.
- **[`<Image>`-Komponente](/docs/app/api-reference/components/image):** Optimieren Sie Bilder mit der Image-Komponente, die Bilder automatisch optimiert, Layoutverschiebungen verhindert und sie in modernen Formaten wie WebP bereitstellt.
- **[`<Script>`-Komponente](/docs/app/guides/scripts):** Optimieren Sie Drittanbieter-Skripte mit der Script-Komponente, die Skripte automatisch verzögert und verhindert, dass sie den Hauptthread blockieren.
- **[ESLint](/docs/architecture/accessibility#linting):** Verwenden Sie das integrierte `eslint-plugin-jsx-a11y`-Plugin, um Barrierefreiheitsprobleme frühzeitig zu erkennen.

### Sicherheit

<AppOnly>

- **[Tainting (Markierung)](/docs/app/api-reference/config/next-config-js/taint):** Verhindern Sie, dass sensible Daten dem Client preisgegeben werden, indem Sie Datenobjekte und/oder bestimmte Werte markieren.
- **[Server Actions (Server-Aktionen)](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#authentication-and-authorization):** Stellen Sie sicher, dass Benutzer berechtigt sind, Server Actions aufzurufen. Lesen Sie die empfohlenen [Sicherheitspraktiken](/blog/security-nextjs-server-components-actions).

</AppOnly>

- **[Environment Variables (Umgebungsvariablen)](/docs/app/guides/environment-variables):** Stellen Sie sicher, dass Ihre `.env.*`-Dateien in `.gitignore` aufgeführt sind und nur öffentliche Variablen mit `NEXT_PUBLIC_` beginnen.
- **[Content Security Policy (Inhalts-Sicherheitsrichtlinie)](/docs/app/guides/content-security-policy):** Erwägen Sie das Hinzufügen einer Content Security Policy, um Ihre Anwendung vor verschiedenen Sicherheitsbedrohungen wie Cross-Site-Scripting, Clickjacking und anderen Code-Injection-Angriffen zu schützen.

### Metadaten und SEO

<AppOnly>

- **[Metadata API (Metadaten-API)](/docs/app/getting-started/metadata-and-og-images):** Verwenden Sie die Metadata API, um die Suchmaschinenoptimierung (SEO) Ihrer Anwendung durch das Hinzufügen von Seitentiteln, Beschreibungen und mehr zu verbessern.
- **[Open Graph (OG)-Bilder](/docs/app/api-reference/file-conventions/metadata/opengraph-image):** Erstellen Sie OG-Bilder, um Ihre Anwendung für das Teilen in sozialen Medien vorzubereiten.
- **[Sitemaps](/docs/app/api-reference/functions/generate-sitemaps) und [Robots](/docs/app/api-reference/file-conventions/metadata/robots):** Helfen Sie Suchmaschinen beim Crawlen und Indizieren Ihrer Seiten, indem Sie Sitemaps und Robots-Dateien generieren.

</AppOnly>

<PagesOnly>

- **[`<Head>`-Komponente](/docs/pages/api-reference/components/head):** Verwenden Sie die `next/head`-Komponente, um Seitentitel, Beschreibungen und mehr hinzuzufügen.

</PagesOnly>

### Typsicherheit

- **TypeScript und [TS Plugin (TypeScript-Plugin)](/docs/app/api-reference/config/typescript):** Verwenden Sie TypeScript und das TypeScript-Plugin für bessere Typsicherheit und um Fehler frühzeitig zu erkennen.

## Vor dem Livegang

Vor dem Livegang können Sie `next build` ausführen, um Ihre Anwendung lokal zu erstellen und Build-Fehler zu erkennen, und dann `next start` ausführen, um die Leistung Ihrer Anwendung in einer produktionsähnlichen Umgebung zu messen.

### Core Web Vitals

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** Führen Sie Lighthouse im Inkognito-Modus aus, um ein besseres Verständnis dafür zu erhalten, wie Ihre Benutzer Ihre Website erleben werden, und um Verbesserungsmöglichkeiten zu identifizieren. Dies ist ein simulierter Test und sollte mit Felddaten (wie Core Web Vitals) kombiniert werden.

<AppOnly>

- **[`useReportWebVitals`-Hook](/docs/app/api-reference/functions/use-report-web-vitals):** Verwenden Sie diesen Hook, um [Core Web Vitals](https://web.dev/articles/vitals)-Daten an Analyse-Tools zu senden.

</AppOnly>

### Analyse der Bundles

Verwenden Sie das [`@next/bundle-analyzer`-Plugin](/docs/app/guides/package-bundling#analyzing-javascript-bundles), um die Größe Ihrer JavaScript-Bundles zu analysieren und große Module und Abhängigkeiten zu identifizieren, die die Leistung Ihrer Anwendung beeinträchtigen könnten.

Zusätzlich können die folgenden Tools Ihnen helfen, die Auswirkungen des Hinzufügens neuer Abhängigkeiten zu Ihrer Anwendung zu verstehen:

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
