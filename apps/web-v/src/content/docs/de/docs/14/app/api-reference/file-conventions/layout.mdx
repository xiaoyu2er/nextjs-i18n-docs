Ein **Layout** ist eine BenutzeroberflĂ¤che, die zwischen Routen gemeinsam genutzt wird.

```tsx filename="app/dashboard/layout.tsx" switcher
  children,
}: {
  children: React.ReactNode
```

```tsx filename="app/layout.tsx" switcher
      <body>{children}</body>

  )

```

```jsx filename="app/layout.js" switcher

  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  )

```

## Props

### `children` (erforderlich)

Layout-Komponenten sollten eine `children`-Prop akzeptieren und verwenden. WĂ¤hrend des Renderings wird `children` mit den Routensegmenten gefĂĽllt, die das Layout umschlieĂźt. Dies sind hauptsĂ¤chlich die Komponenten eines untergeordneten [Layouts](/docs/14/app/building-your-application/routing/pages-and-layouts#pages) (falls vorhanden) oder einer [Seite](/docs/14/app/building-your-application/routing/pages-and-layouts#pages), aber kĂ¶nnen auch andere spezielle Dateien wie [Loading](/docs/14/app/building-your-application/routing/loading-ui-and-streaming) oder [Error](/docs/14/app/building-your-application/routing/error-handling) sein, wenn zutreffend.

### `params` (optional)

Das Objekt der [dynamischen Routenparameter](/docs/14/app/building-your-application/routing/dynamic-routes) vom Wurzelsegment bis zu diesem Layout.

| Beispiel                          | URL            | `params`                  |
| --------------------------------- | -------------- | ------------------------- |
| `app/dashboard/[team]/layout.js`  | `/dashboard/1` | `{ team: '1' }`           |
| `app/shop/[tag]/[item]/layout.js` | `/shop/1/2`    | `{ tag: '1', item: '2' }` |
| `app/blog/[...slug]/layout.js`    | `/blog/1/2`    | `{ slug: ['1', '2'] }`    |

Beispiel:

```tsx filename="app/shop/[tag]/[item]/layout.tsx" switcher
export default function ShopLayout({
  children,
  params,
}: {
  children: React.ReactNode
  params: {
    tag: string
    item: string
  }
}) {
  // URL -> /shop/shoes/nike-air-max-97
  // `params` -> { tag: 'shoes', item: 'nike-air-max-97' }
  return <section>{children}</section>
}
```

```jsx filename="app/shop/[tag]/[item]/layout.js" switcher
export default function ShopLayout({ children, params }) {
  // URL -> /shop/shoes/nike-air-max-97
  // `params` -> { tag: 'shoes', item: 'nike-air-max-97' }
  return <section>{children}</section>
}
```

## Wissenswertes

### Layouts erhalten keine `searchParams`

Im Gegensatz zu [Seiten](/docs/14/app/api-reference/file-conventions/page) erhalten Layout-Komponenten **keine** `searchParams`-Prop. Dies liegt daran, dass ein gemeinsames Layout [wĂ¤hrend der Navigation nicht neu gerendert wird](/docs/14/app/building-your-application/routing/linking-and-navigating#4-partial-rendering), was zu veralteten `searchParams` zwischen Navigationen fĂĽhren kĂ¶nnte.

Bei clientseitiger Navigation rendert Next.js automatisch nur den Teil der Seite unterhalb des gemeinsamen Layouts zwischen zwei Routen.

Bei der Navigation von `/dashboard/settings` zu `/dashboard/analytics` wird `page.tsx` in `/dashboard/analytics` auf dem Server neu gerendert, wĂ¤hrend `dashboard/layout.tsx` **nicht** neu gerendert wird, da es eine gemeinsame BenutzeroberflĂ¤che zwischen den beiden Routen ist.

Da `dashboard/layout.tsx` nicht neu gerendert wird, kĂ¶nnte die `searchParams`-Prop in der Layout-Server-Komponente nach der Navigation **veraltet** sein.

- Verwenden Sie stattdessen die [`searchParams`](/docs/14/app/api-reference/file-conventions/page#searchparams-optional)-Prop der Seite oder den [`useSearchParams`](/docs/14/app/api-reference/functions/use-search-params)-Hook in einer Client-Komponente, die auf dem Client mit den neuesten `searchParams` neu gerendert wird.

### Root-Layouts

- Das `app`-Verzeichnis **muss** ein Root-`app/layout.js` enthalten.
- Das Root-Layout **muss** `<html>`- und `<body>`-Tags definieren.
  - Sie sollten **keine** manuell `<head>`-Tags wie `<title>` und `<meta>` zu Root-Layouts hinzufĂĽgen. Verwenden Sie stattdessen die [Metadata-API](/docs/14/app/api-reference/functions/generate-metadata), die erweiterte Anforderungen wie Streaming und Deduplizierung von `<head>`-Elementen automatisch handhabt.
- Sie kĂ¶nnen [Route-Gruppen](/docs/14/app/building-your-application/routing/route-groups) verwenden, um mehrere Root-Layouts zu erstellen.
  - Die Navigation **zwischen mehreren Root-Layouts** verursacht einen **vollstĂ¤ndigen Seitenladevorgang** (im Gegensatz zu einer clientseitigen Navigation). Dies gilt **nur** fĂĽr mehrere Root-Layouts.
