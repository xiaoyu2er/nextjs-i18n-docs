1\. 始め方 [](#1-getting-started)
-----------------------------------------

### 1.1 Next.js アプリの作成 [](#11-create-a-nextjs-app)

これから新規プロジェクトを始める場合は、以下を使用して新しい Next.js プロジェクトを作成できます。

ターミナル

```
npx create-next-app@latest --api
```

> **注:** `--api` フラグは、プロジェクトの新しい `app/` フォルダ内に例の `route.ts` を自動的に含めます。これにより、API エンドポイントの作成方法が示されます。

### 1.2 App ルーター vs Pages ルーター [](#12-app-router-vs-pages-router)

*   **Pages ルーター**: 歴史的に、Next.js は API に `pages/api/*` を使用していました。このアプローチは Node.js のリクエスト/レスポンスオブジェクトと Express 型の API に依存しています。
*   **App ルーター (既定)**: Next.js 13 で導入された App ルーターは、Web 標準の Request/Response API を完全に採用しています。`pages/api/*` の代わりに、`route.ts` や `route.js` ファイルを `app/` ディレクトリ内のどこでも配置できます。

> **なぜ移行するか?** App ルーターの「Route Handlers」は、Node.js 固有の API ではなく、[Web プラットフォームの Request/Response API](https://developer.mozilla.org/en-US/docs/Web/API) に依存しています。これにより学習が簡素化され、摩擦が減少し、異なるツール間で知識を再利用できるようになります。

2\. なぜ (いつ) Next.js で API を構築すべきか [](#2-why-and-when-to-build-apis-with-nextjs)

1.  **複数クライアント向けの公開 API**
    
    *   Next.js ウェブアプリ、別のモバイルアプリ、サードパーティのいずれにも消費される公開 API を構築できます。例えば、React ウェブサイトと React Native モバイルアプリの両方から `/api/users` からフェッチする場合があります。
2.  **既存のバックエンドへのプロキシ**
    
    *   外部 [マイクロサービス](https://vercel.com/blog/how-vercel-adopted-microfrontends) を単一のエンドポイントの背後に統合または隠蔽したい場合があります。Next.js Route Handlers は、別の既存バックエンドへのプロキシまたはミドル層として機能できます。例えば、リクエストを拾い上げ、認証を処理し、データを変換し、リクエストをアップストリーム API に転送できます。
3.  **Webhooks とインテグレーション**
    
    *   外部からのコールバック or Webhooks (例：Stripe, GitHub, Twilio から) を受け取った場合、Route Handlers で処理できます。
4.  **カスタム認証**
    
    *   セッション、トークン、その他認証ロジックが必要な場合、クッキーに保存し、ヘッダーを読み取り、Next.js API レイヤーで適切なデータで応答できます。

> **注:** 次の場合にのみサーバーサイドデータフェッチが必要 (自分の Next.js アプリ内部で使用し、外部でデータ共有する必要がない) の場合、レンダリング中にデータに直接フェッチするだけで Server Components で十分である可能性があり、別の API レイヤーは必要ありません。

3\. Route Handlers を使用した API の作成 [](#3-creating-an-api-with-route-handlers)

### 3.1 基本ファイルセットアップ [](#31-basic-file-setup)

App ルーター (`app/`) で、ルートを表すフォルダを作成し、その中に `route.ts` ファイルを作成します。

例として、`/api/users` にエンドポイントを作成する場合。

```
app
└── api
    └── users
        └── route.ts
```

### 3.2 1 つのファイル内での複数の HTTP メソッド [](#32-multiple-http-methods-in-one-file)

Pages ルーター API ルート (デフォルトのエクスポートが単一) とは異なり、同じファイルから異なる HTTP メソッドを表す複数の関数をエクスポートできます。

app/api/users/route.ts

```
export async function GET(request: Request) {
  // For example, fetch data from your DB here
  const users = [
    { id: 1, name: 'Alice' },
    { id: 2, name: 'Bob' }
  ];
  return new Response(JSON.stringify(users), {
    status: 200,
    headers: { 'Content-Type': 'application/json' }
  });
}
 
export async function POST(request: Request) {
  // Parse the request body
  const body = await request.json();
  const { name } = body;
 
  // e.g. Insert new user into your DB
  const newUser = { id: Date.now(), name };
 
  return new Response(JSON.stringify(newUser), {
    status: 201,
    headers: { 'Content-Type': 'application/json' }
  });
}
```

これで、`/api/users` に GET リクエストを送信するとユーザーリストが返され、同 URL への POST リクエストが新しいユーザーを挿入します。

4\. Web API の活用 [](#4-working-with-web-apis)

### 4.1 Request と Response の直接使用 [](#41-directly-using-request--response)

既定では、Route Handler メソッド (`GET`, `POST` など) は標準的な [Request](https://developer.mozilla.org/docs/Web/API/Request) オブジェクトを受け取り、標準的な [Response](https://developer.mozilla.org/docs/Web/API/Response) オブジェクトを返さなければなりません。

### 4.2 クエリパラメータ [](#42-query-parameters)

app/api/search/route.ts

```
import { NextRequest } from 'next/server';
 
export function GET(request: NextRequest) {
  const searchParams = request.nextUrl.searchParams;
  const query = searchParams.get('query'); // e.g. `/api/search?query=hello`
 
  return new Response(
    JSON.stringify({ result: `You searched for: ${query}` }),
    {
      headers: { 'Content-Type': 'application/json' },
    },
  );
}
```

### 4.3 ヘッダーとクッキー [](#43-headers-and-cookies)

app/api/auth/route.ts

```
import { NextRequest } from 'next/server';
import { cookies, headers } from 'next/headers';
 
export async function GET(request: NextRequest) {
  // 1. Using 'next/headers' helpers
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
 
  const headersList = await headers();
  const referer = headersList.get('referer');
 
  // 2. Using the standard Web APIs
  const userAgent = request.headers.get('user-agent');
 
  return new Response(JSON.stringify({ token, referer, userAgent }), {
    headers: { 'Content-Type': 'application/json' },
  });
}
```

他のサーバーサイドコードの共有ロジックを再利用する予定の場合、`cookies()` と `headers()` 関数が役立ちます。Next.js には Base Web API を拡張した `NextRequest` と `NextResponse` も提供されていることに注意してください。

5\. 動的ルート [](#5-dynamic-routes)

動的パス (例: `/api/users/:id`) を作成するには、フォルダ構造に **動的セグメント** を使用します：

```
app
└── api
    └── users
        └── [id]
            └── route.ts
```

このファイルは、`123` がパラメータとして捕捉された URL `/api/users/123` に対応します。

app/api/users/[id]/route.ts

```
import { NextRequest } from 'next/server';
 
export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  const id = (await params).id;
  // e.g. Query a database for user with ID `id`
  return new Response(JSON.stringify({ id, name: `User ${id}` }), {
    status: 200,
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export async function DELETE(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  const id = (await params).id;
  // e.g. Delete user with ID `id` in DB
  return new Response(null, { status: 204 });
}
```

ここで、`params.id` が動的セグメントを受け取ります。

6\. Next.js をプロキシまたは転送レイヤーとして使用する [](#6-using-nextjs-as-a-proxy-or-forwarding-layer)

一般的なシナリオは、既存のバックエンドサービスの **プロキシ** です。リクエストの認証、ロギングの処理、またはリモートサーバーまたはバックエンドへの送信前のデータ変換を行います。

app/api/external/route.ts

```
import { NextRequest } from 'next/server';
 
export async function GET(request: NextRequest) {
  const response = await fetch('https://example.com/api/data', {
    // Optional: forward some headers, add auth tokens, etc.
    headers: { Authorization: `Bearer ${process.env.API_TOKEN}` },
  });
 
  // Transform or forward the response
  const data = await response.json();
  const transformed = { ...data, source: 'proxied-through-nextjs' };
 
  return new Response(JSON.stringify(transformed), {
    headers: { 'Content-Type': 'application/json' },
  });
}
```

これでクライアントは `/api/external` への呼び出しだけで済み、Next.js がその余分を処理します。これを「Backend for Frontend」または BFF とも呼びます。

7\. 共有「ミドルウェア」ロジックの構築 [](#7-building-shared-middleware-logic)

複数の Route Handler に同じロジック (例：認証チェック、ロギング) を適用したい場合、Handler をラップする再利用可能な関数を作成できます。

lib/with-auth.ts

```
import { NextRequest } from 'next/server';
 
type Handler = (req: NextRequest, context?: any) => Promise<Response>;
 
export function withAuth(handler: Handler): Handler {
  return async (req, context) => {
    const token = req.cookies.get('token')?.value;
    if (!token) {
      return new Response(JSON.stringify({ error: 'Unauthorized' }), {
        status: 401,
        headers: { 'Content-Type': 'application/json' },
      });
    }
 
    // If authenticated, call the original handler
    return handler(req, context);
  };
}
```

その後、Route Handler 内で：

app/api/secret/route.ts

```
import { NextRequest } from 'next/server';
import { withAuth } from '@/lib/with-auth';
 
async function secretGET(request: NextRequest) {
  return new Response(JSON.stringify({ secret: 'Here be dragons' }), {
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export const GET = withAuth(secretGET);
```

8\. デプロイと「SPA Mode」の考慮事項 [](#8-deployment-and-spa-mode-considerations)

### 8.1 標準的な Node.js デプロイ [](#81-standard-nodejs-deployment)

標準的な Next.js サーバーデプロイ (next start を使用) を使用すると、Route Handlers、Server Components、Middleware などを特徴として使用しつつ、動的なリクエスト時の情報を活用できます。

追加の構成は必要ありません。詳細については [デプロイ](/docs/app/building-your-application/deploying) を参照してください。

### 8.2 SPA/Static Export [](#82-spastatic-export)

Next.js では、[静的シングルページアプリケーション (SPA) として](/docs/app/building-your-application/upgrading/single-page-applications) 全体のサイトを出力することもサポートしています。

これは以下を設定することで有効にできます：

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  output: 'export',
};
 
export default nextConfig;
```

**静的エクスポートモード**で、Next.js は真に静的な HTML、CSS、JS を生成します。 **サーバーサイドコード (API エンドポイントなど) は実行できません**。API が必要な場合は、別途ホストする必要があります (例：スタンドアロンの Node.js サーバー)。

> **注:**
> 
> *   **GET Route Handlers** は動的なリクエストデータに依存しない場合 [静的エクスポートが可能です](/docs/app/building-your-application/deploying/static-exports#route-handlers)。out フォルダ内の静的ファイルになります。
> *   **すべての他のサーバー機能** (動的リクエスト、クッキーの書き換えなど) は、純粋な SPA エクスポートでは **サポートされていません**。

### 8.3 Vercel での API デプロイ [](#83-deploying-apis-on-vercel)

Next.js アプリケーションを Vercel にデプロイする場合、[API デプロイに関するガイド](https://vercel.com/guides/hosting-backend-apis) があります。これには Vercel Firewall を介した [プログラム的レート制限](https://vercel.com/docs/security/vercel-waf/rate-limiting-sdk) などの他の Vercel 機能も含まれます。Vercel はまた、API アプローチには一般的に必要な [Cron Jobs](https://vercel.com/docs/cron-jobs/manage-cron-jobs) も提供しています。

9\. API エンドポイント作成をスキップするタイミング [](#9-when-to-skip-creating-an-api-endpoint)

App ルーターの **React Server Components** を使用すると、公開エンドポイントを公開せずにサーバーで直接データをフェッチできます。

app/users/page.tsx

```
// (Server Component)
export default async function UsersPage() {
  // This fetch runs on the server (no client-side code needed here)
  const res = await fetch('https://api.example.com/users');
  const data = await res.json();
 
  return (
    <main>
      <h1>Users</h1>
      <ul>
        {data.map((user: any) => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    </main>
  );
}
```

データが Next.js アプリ内でのみ使用される場合、公開 API は不要な場合があります。

10\. まとめ [](#10-putting-it-all-together)

1.  **新しい Next.js プロジェクトを作成**: `npx create-next-app@latest --api`。
2.  `app/` ディレクトリ内に **Route Handlers** を追加します (例：`app/api/users/route.ts`)。
3.  同じファイルで **HTTP メソッド (GET, POST, PUT, DELETE など)** をエクスポートします。
4.  `Request` オブジェクトとのインタラクションと `Response` への返却に **Web 標準 API** を使用します。
5.  他のクライアントがデータを消費する必要がある、またはバックエンドサービスのプロキシが必要な場合は、**公開 API を構築します**。
6.  クライアントから新しい API ルートを **フェッチ** します (例：Client コンポーネント内または `fetch('/api/...')` で)。
7.  Server Component でデータをフェッチするだけで済む場合、**API の作成をスキップします**。
8.  認証または他の繰り返しロジックのために共有「**ミドルウェア**」パターン (例：`withAuth()`) を追加します。
9.  サーバー機能を持つ Node.js 環境に **デプロイ**するか、静的な SPA が必要な場合のみ **静的エクスポート**します。

結論 [](#conclusion)

Next.js の **App ルーター** と **Route Handlers** を使用すると、**Web プラットフォーム** に直接対応した、柔軟で現代的な API 構築方法を提供します。あなたは：


*   Web、モバイル、サードパーティクライアントで共有するための **完全な公開 API を作成する**
*   既存の外部サービスへのコールを **プロキシ** しカスタマイズする
*   認証、ロギング、またはその他繰り返しロジックのために再利用可能な「**ミドルウェア**」レイヤーを **実装する**
*   `[id]` セグメントフォルダ構造を使用してリクエストを **動的にルート処理する**。

### Server Actions については？ [](#what-about-server-actions)

[Server Actions](/docs/app/building-your-application/data-fetching/server-actions-and-mutations) は、クライアントから呼び出せる自動生成された `POST` API ルートと考えることができます。

データ作成、更新、削除などのミューテーション操作に設計されています。Server Action を呼び出すのは、明示的な `fetch` を定義された API ルートに対して行うのと同じように、通常の JavaScript 関数のように呼び出します。

確かにネットワークリクエストが発生していますが、明示的に管理する必要はありません。URL パスは自動生成され [暗号化されています](/docs/app/building-your-application/data-fetching/server-actions-and-mutations#security)、ブラウザで `/api/users` のようなルートを直接アクセスすることはできません。

Server Actions かつ公開 API を暴露する予定の場合、コアロジックを [Data Access Layer](/blog/security-nextjs-server-components-actions) に移動し、Server Action と API ルートの両方から同じロジックを呼び出すことをお勧めします。
