Next.js 16（ベータ版）が利用可能になりました。この早期リリースは、安定版リリースに先立ち、Turbopack、キャッシング、Next.js アーキテクチャの最新改善にアクセスできます。今回のリリースのハイライトは以下の通りです：

*   [**Turbopack (安定版)**](#turbopack-stable): 全アプリのデフォルトバンドラーで、Fast Refresh が最大 5-10 倍高速、ビルドが 2-5 倍高速
*   [**Turbopack ファイルシステムキャッシング (ベータ版)**](#turbopack-file-system-caching-beta): 最大規模のアプリでもさらに高速な起動とコンパイル時間
*   [**React Compiler のサポート (安定版)**](#react-compiler-support-stable): 自動メモ化のためのビルトイン統合
*   [**ビルドアダプター API (アルファ版)**](#build-adapters-api-alpha): ビルドプロセスを変更するカスタムアダプターを作成
*   [**強化されたルーティング**](#enhanced-routing-and-navigation): レイアウト重複排除と累積プリフェッチによる最適化されたナビゲーションとプリフェッチ
*   [**改善されたキャッシング API**](#improved-caching-apis): 新しい `updateTag()` と洗練された `revalidateTag()`
*   [**React 19.2**](#react-192-and-canary-features): View Transitions、`useEffectEvent()`、`<Activity/>`
*   [**破壊的変更**](#breaking-changes-and-other-updates): 非同期パラメータ、`next/image` のデフォルトなど

**ベータ版をお試しいただき、フィードバックをご共有いただくことをお勧めします。** あなたのテストは、安定版リリース前に Next.js の問題を特定し改善するのに役立ちます。遭遇したバグや問題があれば、[GitHub](https://github.com/vercel/next.js/issues) で教えてください。

ベータ版へアップグレード：

ターミナル

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade beta
 
# ...or upgrade manually
npm install next@beta react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@beta
```

## 開発者体験 [](#developer-experience)
---------------------------------------------

### Turbopack (安定版) [](#turbopack-stable)

Turbopack は開発ビルドおよび本番ビルドの両方で安定版に到達し、すべての新しい Next.js プロジェクトのデフォルトバンドラーになりました。今年初夏のベータ版リリース以来、採用が急拡大しており、Next.js 15.3 以上の開発セッションの 50% 以上と本番ビルドの 20% 以上がすでに Turbopack で実行されています。

Turbopack を使用すると、以下が期待できます：

*   本番ビルドが 2–5 倍高速化
*   Fast Refresh が最大 10 倍高速化

これらのパフォーマンス向上をすべての Next.js 開発者に提供するために、設定なしで Turbopack をデフォルトにしています。カスタム webpack セットアップを持つアプリの場合、以下を実行することで webpack の使用を続けることができます：

ターミナル

```
next dev --webpack
next build --webpack
```

### Turbopack ファイルシステムキャッシング (ベータ版) [](#turbopack-file-system-caching-beta)

Turbopack は開発環境でファイルシステムキャッシングをサポートするようになりました。起動間隔でコンパイラーの出力物をディスクに保存し、特に大規模プロジェクトで起動間でのコンパイル時間を大幅に短縮します。

設定でファイルシステムキャッシングを有効化します：

next.config.ts

```
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
};
 
export default nextConfig;
```

すべての内部 Vercel アプリはこの機能をすでに使用しており、大規模リポジトリで開発者の生産性の顕著な向上を確認しています。

ファイルシステムキャッシングの改善についてフィードバックをお待ちしています。試していただき、あなたの経験を共有してください。

### 簡素化された `create-next-app` [](#simplified-create-next-app)

`create-next-app` は、簡素化されたセットアップフロー、更新されたプロジェクト構造、改善されたデフォルトと共に redesign されました。新しいテンプレートはデフォルトで App Router を含み、TypeScript 最優先の設定、Tailwind CSS、ESLint を提供します。

### ビルドアダプター API (アルファ版) [](#build-adapters-api-alpha)

[ビルドアダプター RFC](https://github.com/vercel/next.js/discussions/77740) に続いて、私たちはコミュニティおよびデプロイプラットフォームと連携して、ビルドアダプター API の最初のアルファ版を提供しました。

ビルドアダプターは、ビルドプロセスにフックするカスタムアダプターを作成できるため、デプロイプラットフォームとカスタムビルドインテグレーションが Next.js 設定を変更したりビルド出力を処理したりできます。

next.config.js

```
const nextConfig = {
  experimental: {
    adapterPath: require.resolve('./my-adapter.js'),
  },
};
 
module.exports = nextConfig;
```

[RFC ディスカッション](https://github.com/vercel/next.js/discussions/77740) でフィードバックを共有してください。

### React Compiler のサポート (安定版) [](#react-compiler-support-stable)

React Compiler のビルトインサポートは、React Compiler の 1.0 リリースに続いて Next.js 16 で安定版になりました。React Compiler はコンポーネントを自動的にメモ化し、手間のかかるコード変更なしで不要な再レンダリングを削減します。

reactCompiler 設定オプションは experimental から安定版に昇格しました。さまざまなアプリケーションタイプでビルドパフォーマンスデータを収集し続けているため、デフォルトでは有効化されていません。React Compiler は Babel に依存しているため、これを有効化すると開発ビルドおよびビルド時のコンパイル時間が長くなります。

next.config.ts

```
const nextConfig = {
  reactCompiler: true,
};
 
export default nextConfig;
```

React Compiler プラグインの最新バージョンをインストールしてください：

ターミナル

```
npm install babel-plugin-react-compiler@latest
```

### ルーティングとナビゲーションの強化 [](#enhanced-routing-and-navigation)

Next.js 16 には、ページ遷移をより軽快かつ高速にするため、ルーティングとナビゲーションシステム全体の刷新が含まれています。

**レイアウトの重複排除**: 共通レイアウトを持つ複数の URL をプリフェッチする際、レイアウトは各 Link ごとに個別にダウンロードされるのではなく一度だけダウンロードされます。例えば、50 個の製品リンクを持つページは、レイアウトを 50 回ではなく 1 回だけダウンロードし、ネットワーク転送サイズを劇的に削減します。

**段階的プリフェッチ**: Next.js は既にキャッシュにある部分をスキップし、キャッシュにない部分のみをプリフェッチします。プリフェッチキャッシュは現在次のようになります：

*   リンクが表示領域から外れるとリクエストをキャンセルします
*   ホバー時や表示領域への再入力時にリンクのプリフェッチを優先します
*   データが無効化されたときリンクを再プリフェッチします
*   Cache Components などの今後の機能とシームレスに動作します

**トレードオフ**: より個別のプリフェッチリクエストが表示される場合がありますが、総転送サイズは大幅に低くなります。このトレードオフはほぼすべてのアプリケーションに適切だと考えています。リクエスト数の増加に問題がある場合はお知らせください。データチャンクをより効率的にインライン化する追加の最適化に取り組んでいます。

これらの変更はコード修正を必要とせず、すべてのアプリの性能向上を目的として設計されています。

### PPR と Cache Components [](#ppr-and-cache-components)

Next.js 16 では、実験的な**部分プリレンダリング (PPR)** フラグと設定オプションが削除されました。PPR は Cache Components に統合されています。

Next.js 16 以降は、experimental.cacheComponents 設定を使用して PPR にオプトインできます。実装に違いがあり、Cache Components は Next.js Conf と Next.js 16 安定版リリースに先立ち文書化・発表される追加機能と動作も提供します。

> **アプリケーションが PPR（`experimental.ppr = true`）に依存している場合**: 現在使用している Next.js `canary` の特定バージョンに固定してください。移行に問題がある場合は、一旦現在のバージョンに固定してください。安定版リリースに先立ち移行ガイドを提供します。

### 改善されたキャッシング APIs [](#improved-caching-apis)

Next.js 16 では、キャッシュ動作に対するより明示的な制御を提供する、洗練されたキャッシング API が導入されました。

#### `revalidateTag()` (更新版) [](#revalidatetag-updated)

`revalidateTag()` は、stale-while-revalidate (SWR) 動作を有効にするために、第二引数として**[`cacheLife` プロファイル]**(https://nextjs.org/docs/app/api-reference/functions/cacheLife#reference) を要求するようになりました：

```
import { revalidateTag } from 'next/cache';
 
// ✅ Use built-in cacheLife profile (we recommend 'max' for most cases)
revalidateTag('blog-posts', 'max');
 
// Or use other built-in profiles
revalidateTag('news-feed', 'hours');
revalidateTag('analytics', 'days');
 
// Or use an inline object with a custom revalidation time
revalidateTag('products', { revalidate: 3600 });
 
// ⚠️ Deprecated - single argument form
revalidateTag('blog-posts');
```

プロファイル引数は、組み込みの `cacheLife` プロファイル名（例：`'max'`, `'hours'`, `'days'`）または `next.config` に定義された [カスタムプロファイル](https://nextjs.org/docs/app/api-reference/functions/cacheLife#defining-reusable-cache-profiles) を受け取ります。インラインの `{ revalidate: number }` オブジェクトも指定できます。長期間生存するコンテンツのバックグラウンド再検証を有効にするため、ほとんどすべての場合に `'max'` を使用することを推奨します。ユーザーがタグ付きコンテンツをリクエストすると、Next.js がバックグラウンドで再検証している間、キャッシュデータが即座に返されます。

タグ付きキャッシュエントリのみを SWR 動作で無効化したい場合は `revalidateTag()` を使用してください。これは最終的な整合性を許容できる静的コンテンツに理想的です。

> **移行ガイド:** SWR 動作のため `cacheLife` プロファイル（`'max'` を推奨）の第二引数を追加するか、読み書きセマンティクスが必要であれば Server Actions で `updateTag()` を使用してください。

#### `updateTag()` (新機能) [](#updatetag-new)

`updateTag()` は、**読み書きセマンティクス**を提供する新しい Server Actions のみ API です。リクエスト内でキャッシュデータを無効化し、即座に更新します：

```
'use server';
 
import { updateTag } from 'next/cache';
 
export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile);
 
  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`);
}
```

これにより、インタラクティブな機能が変更を即座に反映します。フォーム、ユーザー設定、ユーザーが変更を即座に確認するあらゆるワークフローに最適です。

#### `refresh()` (新機能) [](#refresh-new)

`refresh()` は、**キャッシュされていないデータのみ**をリフレッシュするための新しい Server Actions のみ API です。キャッシュには一切影響しません：

```
'use server';
 
import { refresh } from 'next/cache';
 
export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId);
 
  // Refresh the notification count displayed in the header
  // (which is fetched separately and not cached)
  refresh();
}
```

この API はクライアントサイドの `router.refresh()` と補完関係にあります。アクションを実行した後にページ内の他の場所で表示されるキャッシュされていないデータをリフレッシュする必要がある場合に使用します。キャッシュされたページシェルと静的コンテンツは高速なまま、通知数、リアルタイムメトリクス、またはステータスインジケーターなどの動的データが更新されます。

### React 19.2 と Canary 機能 [](#react-192-and-canary-features)

Next.js 16 の App Router は、新しくリリースされた React 19.2 の機能と段階的に安定化されているその他機能を含む、最新の React [Canary リリース](https://react.dev/blog/2023/05/03/react-canaries) を使用します。ハイライトは以下の通りです：

*   **[View Transitions]**(https://react.dev/reference/react/ViewTransition): トランジションまたはナビゲーション内で更新される要素をアニメーションします
*   **[useEffectEvent]**(https://react.dev/reference/react/useEffectEvent): 効果から非反応的逻辑を抽出し、再利用可能な Effect Event 関数として定義します
*   **[Activity]**(https://react.dev/reference/react/Activity): UI を `display: none` で隠しつつ、状態を維持し Effect をクリーンアップして「バックグラウンドアクティビティ」をレンダリングします

[React 19.2 アナウンスメント](https://react.dev/blog/2025/10/01/react-19-2) で詳しく学びましょう。

### 削除 [](#removals)

これらの機能は以前に非推奨でしたが、現在削除されました：

| 削除 | 代替 |
| --- | --- |
| **AMP サポート** | すべての AMP API および設定が削除されました（`useAmp`、`export const config = { amp: true }`） |
| **`next lint` コマンド** | Biome を直接使用するか ESLint を使用してください。`next build` ではもはやリンティングは実行されません。Codemod が利用可能です：`npx @next/codemod@canary next-lint-to-eslint-cli .` |
| **`devIndicators` オプション** | `appIsrStatus`、`buildActivity`、`buildActivityPosition` を設定から削除しました。インジケーターは残ります。 |
| **`serverRuntimeConfig`、`publicRuntimeConfig`** | 環境変数（`.env` ファイル）を使用してください |
| **`experimental.turbopack` 場所** | 設定はトップレベルの `turbopack` に移動しました（experimental 内ではなくなりました） |
| **`experimental.dynamicIO` フラグ** | `experimental.cacheComponents` として名称変更 |
| **`experimental.ppr` フラグ** | PPR フラグ削除; Cache Components プログラミングモデルへ移行 |
| **`export const experimental_ppr`** | ルートレベルの PPR エクスポート削除; Cache Components プログラミングモデルへ移行 |
| **自動の `scroll-behavior: smooth`** | HTML ドキュメントに `data-scroll-behavior="smooth"` を追加してオプトインしてください |
| **`unstable_rootParams()`** | 今後のマイナーアップデートで別の API を提供しています |
| **同期の `params`、`searchParams` props アクセス** | 非同期を使用する必要があります：`await params`、`await searchParams` |
| **同期の `cookies()`、`headers()`、`draftMode()` アクセス** | 非同期を使用する必要があります：`await cookies()`、`await headers()`、`await draftMode()` |
| **Metadata image route `params` 引数** | 非同期 `params` に変更; `generateImageMetadata` からの `id` は現在 `Promise<string>` |
| **`next/image` `local` src クエリストリング** | Enumeration 対策を防ぐために `images.localPatterns` 設定を必要とします |

### 動作変更 [](#behavior-changes)

これらの機能は、Next.js 16 で新しいデフォルト動作があります：

| 変更された動作 | 詳細 |
| --- | --- |
| **デフォルトバンドラー** | Turbopack はすべてのアプリのデフォルトバンドラーになりました。`next build --webpack`でオプトアウトできます |
| **`images.minimumCacheTTL` デフォルト** | 60s から 4 時間 (14400s) に変更; キャッシュ制御ヘッダーがない画像の再検証コストを削減 |
| **`images.imageSizes` デフォルト** | デフォルトサイズから `16` を削除（4.2% のプロジェクトのみが使用）；`srcset` サイズと API 変動を削減 |
| **`images.qualities` デフォルト** | `[1..100]` から `[75]` に変更; `quality` プロパティは `images.qualities` の最も近い値に型変換されます |
| **`images.dangerouslyAllowLocalIP`** | 新しいセキュリティ制限により、デフォルトでローカル IP 最適化がブロックされます。プライベートネットワークのみの設定には `true` を設定 |
| **`images.maximumRedirects` デフォルト** | 無制限から最大 3 回のリダイレクトに変更; リダイレクトを無効化するか、稀なエッジケースのために `0` に設定して増加 |
| **`@next/eslint-plugin-next` デフォルト** | ESLint Flat Config 形式にデフォルト。ESLint v10 がレガシー設定サポートを削除することに合わせます |
| **プリフェッチキャッシュの動作** | レイアウト重複排除および累積プリフェッチによる完全再書き込み |
| **`revalidateTag()` 署名** | ステイルウィズリバレイデータ動作のため第 2 引数に `cacheLife` プロファイルを要求 |
| **Turbopack 内の Babel 設定** | Babel 設定が見つかった場合自動的に有効化（以前はハードエラーで終了） |
| **ターミナル出力** | より明確なフォーマット、より良いエラーメッセージ、改善されたパフォーマンスメトリクスで redesign |
| **開発およびビルド出力ディレクトリ** | `next dev` および `next build` は現在別々の出力ディレクトリを使用し、同時実行を許可 |
| **ロックファイルの動作** | 同じプロジェクト上の複数の `next dev` または `next build` インスタンスを防ぐためにロックファイルメカニズムを追加 |
| **並列ルート `default.js`** | すべての並列ルートスロットに明示的な `default.js` ファイルが必要; なければビルド失敗。以前の動作については `notFound()` を呼び出すか `null` を返す `default.js` を作成 |
| **モダン Sass API** | `sass-loader` v16 に bump。モダン Sass 構文および新機能に対応 |

### 非推奨 [](#deprecations)

これらの機能は Next.js 16 で非推奨となり、将来のバージョンで削除されます：

| 非推奨 | 詳細 |
| --- | --- |
| `middleware.ts` ファイル名 | ネットワーク境界とルーティングの焦点を明確にするために `proxy.ts` に変更ください |
| `next/legacy/image` コンポーネント | 性能向上と機能強化のために `next/image` を使用してください |
| `images.domains` 設定 | 強化されたセキュリティ制限のために `images.remotePatterns` 設定を使用してください |
| `revalidateTag()` 単一引数 | SWR のためには `revalidateTag(tag, profile)` を、読み書きのためには Actions で `updateTag(tag)` を使用してください |

### その他の改善 [](#additional-improvements)

*   **性能改善**: `next dev` および `next start` コマンドの大幅な性能最適化
*   **`next.config.ts` 用の Node.js ネイティブ TypeScript**: `--experimental-next-config-strip-types` フラグで `next dev`、`next build`、`next start` コマンドを実行し、`next.config.ts` のネイティブ TypeScript を有効化

安定版リリースに先立ち、より包括的な移行ガイドをドキュメントで共有する予定です。
