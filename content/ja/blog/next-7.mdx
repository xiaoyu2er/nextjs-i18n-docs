26 本の canary リリースと 340 万回のダウンロードの後、製品版として準備が整った Next.js 7 を発表できることを我々は大変光栄に思っています。

* [DX 改善](#dx-improvements): 起動が 57% 高速、再コンパイルが 42% 高速
* [react-error-overlay](https://www.npmjs.com/package/react-error-overlay) を使用した [より良いエラーレポート](#better-error-reporting-with-react-error-overlay)
* [Webpack 4](#webpack-4) と [Babel 7](#babel-7) による [アップグレードされたコンパイルパイプライン](#webpack-4)
* [標準化された動的インポート](#standardized-dynamic-imports)
* [静的 CDN サポート](#static-cdn-support)
* [最初の HTML ペイロードの縮小](#smaller-initial-html-payload)
* [App と Pages 間の SSR を伴う React Context](#react-context-with-ssr-between-app-and-pages)

ついに、新しい Nextjs.org 上でこのニュースを共有できることを嬉しく思っています

## DX 改善 [](#dx-improvements)
-----------------------------------

Next.js の主な目標の一つは、**最高の開発者体験**を提供しながら最適な本番パフォーマンスを実現することです。このリリースでは、ビルドとデバッグの両方のパイプラインに多くの重要な改善がもたらされます。

### コンパイル速度 [](#compilation-speed)

webpack 4、Babel 7、およびコードベースにおける多くの改善と最適化のおかげで、開発時の Next.js の起動が最大 57% 高速化しました。

新しい増分コンパイルキャッシュのおかげで、コードに対する変更は現在 40% 高速にビルドされます。

これらは私たちが収集した例の数値です。

|  | 6.0 | 7.0 | delta |
| --- | --- | --- | --- |
| 起動時間 (基本アプリケーション) | 1947ms | 835ms | **57%** _速い_ |
| コード変更 (基本アプリケーション) | 304ms | 178ms | **42%** _速い_ |

ボーナスとして、開発時およびビルド時には、[webpackbar](https://github.com/nuxt/webpackbar) によって、より良いリアルタイムフィードバックが確認できるようになりました：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-7/webpackbar.gif)

* サーバーとクライアントの両方のエラーに対する正確なエラー場所
* 文脈を提供するソースのハイライト
* 完全で豊富なスタックトレース

Webpack 4 [](#webpack-4)
-----------------------

最初のリリース以来、Next.js はコードのバンドルおよび豊富なプラグインおよび拡張機能エコシステムの再利用に webpack を使用してきました。Next.js が最新の webpack 4 で支えられていることを発表できることを嬉しく思います。これは多くの改善とバグ修正をもたらします。

その中で私たちが得られるのは:

* `.mjs` ソースファイルのサポート
* コード分割の改善
* 効果的なツリーシェイキング (使用されないコードの削除) のサポート

別の新機能は WebAssembly サポートです。Next.js は WebAssembly を SSR することもできます。ここに [例](https://github.com/vercel/next.js/tree/canary/examples/with-webassembly) があります。

**注:** このアップグレードは**完全に後方互換性**があります。ただし、[next.config.js](/docs#customizing-webpack-config) を経由してカスタム webpack ローダーまたはプラグインを使用している場合は、それらをアップグレードする必要があるかもしれません。

### CSS インポート [](#css-imports)

Webpack 4 で、バンドルから CSS を抽出する新しい方法 [mini-css-extract-plugin](https://github.com/webpack-contrib/mini-css-extract-plugin) が導入されました。

[`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`@zeit/next-less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less)、[`@zeit/next-sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass)、[`@zeit/next-stylus`](https://github.com/vercel/next-plugins/tree/master/packages/next-stylus) は現在 `mini-extract-css-plugin` を使用しています。

これらの Next.js プラグインの新しいバージョンは、CSS インポートに関する 20 の既存の問題を解決します。例として、動的 `import()` での CSS のインポートがサポートされています。

components/my-dynamic-component.js

```
import './my-dynamic-component.css';
 
export default function MyDynamicComponent() {
  return <h1>My dynamic component</h1>;
}
```

pages/index.js

```
import dynamic from 'next/dynamic'
 
const MyDynamicComponent = dynamic(import('../components/my-dynamic-component'))
 
export default function Index() {
  return () {
    <div>
      <MyDynamicComponent/>
    </div>
  }
}
```

大きな改善点は、`pages/_document.js` に以下を追加する必要がなくなったことです。

```
<link rel="stylesheet" href="/_next/static/style.css" />
```

代わりに、Next.js は CSS ファイルを自動的に注入します。本番環境では、Next.js も CSS URL にコンテンツハッシュを自動的に追加するため、変更があった場合、エンドユーザーが常に古いバージョンを受け取らず、不変の永続的なキャッシングを導入する能力を確保できます。

要するに、Next.js プロジェクトで `.css` ファイルをインポートするサポートに必要なことは、`next.config.js` で `withCSS` プラグインを登録することだけです。

```
const withCSS = require('@zeit/next-css');
module.exports = withCSS({
  /* my next config */
});
```

### 標準化された動的インポート [](#standardized-dynamic-imports)

Next.js は、バージョン 3 を通じて `next/dynamic` を介して動的インポートをサポートしていました。

この技術の早期採用者として、私たちは `import()` を処理するための独自のソリューションを作成する必要がありました。

その結果、Next.js は webpack が後にサポートとして導入したことと離れ始め、いくつかの機能が不足し始めていました。

このため Next.js は、それ以来 webpack が紹介したいくつかの `import()` 機能をサポートしませんでした。

例えば、特定のファイルを名前で手動でバンドルすることはできません。

```
import(/* webpackChunkName: 'my-chunk' */ '../lib/my-library');
```

もう一つの例は、`next/dynamic` モジュールでラップされていない `import()` の使用です。

Next.js 7 からは、デフォルトの `import()` 動作を変更しないようになりました。これは、**`import()` 機能が標準でフルサポートされている**ことを意味します。

この変更も完全に後方互換性があります。動的コンポーネントの利用は引き続き簡単なままです。

pages/index.js

```
import dynamic from 'next/dynamic';
 
const MyComponent = dynamic(import('../components/my-component'));
 
export default function Index() {
  return (
    <div>
      <MyComponent />
    </div>
  );
}
```

この例は `my-component` の新しい JavaScript ファイルを作成し、`<MyComponent />` がレンダリングされたときにのみロードすることです。

最も重要なのは、レンダリングされていない場合、`<script>` タグは**最初の HTML ドキュメントのペイロードに含まれない**ことです。

さらにコードベースを簡素化し、優れた React エコシステムを利用するために、Next.js 7 では `next/dynamic` が裏側で [`react-loadable`](https://github.com/jamiebuilds/react-loadable)の使用のために再作成されました（いくつかのマイナーな変更があります）。これにより、動的コンポーネントのための 2 つの素晴らしい新機能も導入されます。

* `next/dynamic` の `timeout` オプションを使用するタイムアウト
* `next/dynamic` の `delay` オプションを使用する loading コンポーネントの遅延。この遅延により、`loading` コンポーネントが読み込み状態をレンダリングする前に x 時間待機できます。例えば、インポートが非常に速い場合。

[Babel 7](#babel-7)
-------------------

Next.js 6 はベータ版の間に Babel 7 を導入しました。その後、Babel 7 の安定版がリリースされ、Next.js 7 は現在このバージョンを使用しています。

完全な変更リストについては、Babel の [リリースノート](http://babeljs.io/blog/2018/08/27/7.0.0) を参照してください。

主な機能の一部は次の通りです。

* TypeScript サポート。Next.js では [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript) を使用できます
* フラグメント構文 `<>` サポート
* `babel.config.js` サポート
* 特定のファイルまたはディレクトリのサブセットにのみプリセット/プラグインを適用するための `overrides` プロパティ

Next.js プロジェクトにカスタム Babel 設定がない場合、-breaking 変更はありません。

ただし、カスタム Babel 設定がある場合、それぞれの個別のカスタムプラグイン/プリセットを最新バージョンにアップグレードする必要があります。

Next.js 6 未満のバージョンからアップグレードする場合は、優れた [`babel-upgrade`](http://babeljs.io/blog/2018/08/27/7.0.0#babel-upgrade) ツールを実行できます。

Babel 7 へのアップグレードに加えて、Next.js Babel プリセット ([`next/babel`](https://github.com/vercel/next.js#customizing-babel-config)) は、`NODE_ENV` が `test` に設定されているときに、`modules` オプションをデフォルトで `commonjs` に設定します。

この設定オプションは、Next.js プロジェクトでカスタム `.babelrc` を作成する唯一の理由であることが多いです。

.babelrc

```
{
  "env": {
    "development": {
      "presets": ["next/babel"]
    },
    "production": {
      "presets": ["next/babel"]
    },
    "test": {
      "presets": [["next/babel", { "preset-env": { "modules": "commonjs" } }]]
    }
  }
}
```

Next.js 7 では次のようになります。

.babelrc

```
{
  "presets": ["next/babel"]
}
```

この時点で、`.babelrc` を削除できます。Next.js は Babel 設定がない場合 `next/babel` を自動的に使用するためです。

Smaller Initial HTML Payload[](#smaller-initial-html-payload)
-------------------------------------------------------------

Next.js は HTML をプリレンダリングするため、ページを `<html>`, `<head>`, `<body>`でラップし、ページをレンダリングするために必要な JavaScript ファイルをデフォルトの構造に格納します。

この初期ペイロードは以前は約 1.62kB でした。Next.js 7 では初期 HTML ペイロードを最適化し、1.5kB になり、7.4% の削減により、ページを軽量にしました。

|  | 6.0 | 7.0 | delta |
| --- | --- | --- | --- |
| ドキュメントサイズ (サーバーレンダリング) | 1.62kb | 1.50kb | **7.4%** _小さい_ |

サイズを削減した主な方法は以下の通りです：

* `__next-error` div が削除されました
* インラインスクリプトは最小化され、今後のリリースでは完全に削除されます
* 使用されない場合、例えば `nextExport` と `assetPrefix` プロパティは `__NEXT_DATA__` プロパティも削除されます。

Static CDN Support[](#static-cdn-support)
-----------------------------------------

Next.js 5 で `assetPrefix` サポートを導入しました。これは Next.js が CD の特定の場所から自動的にアセットをロードする方法です。このオプションは、CDN がプロキシをサポートする場合、URL をリクエストする場合など素晴らしい機能です。

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
```

通常、CDN はキャッシュ内のファイルを確認するか、そうでない場合はオリジンから直接リクエストします。

プロキシはまさに [Edge Network](https://vercel.com/edge-network) です。

ただし、一部のソリューションでは、ディレクトリを直接 CDN にアップロードする必要があります。これを行う問題点は、Next.js の URL 構造が `.next` フォルダ内のフォルダ構造と一致しなかったことです。例えば以前の例

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/page/index.js
```

Next.js 7 では URL 構造に一致するように `.next` のディレクトリ構造を変更しました。

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/static/<buildid>/pages/index.js
```

プロキシタイプの CDN の使用を推奨しますが、この新しい構造により、異なるタイプの CDN ユーザーは `.next` ディレクトリを CDN にアップロードできます。

Styled JSX v3[](#styled-jsx-v3)
-------------------------------

デフォルトで Next.js に含まれている CSS-in-JS ソリューションが `styled-jsx 3` で React Suspense に準備を整えたことを発表できることを嬉しく思います。

よく不明だったことは、子コンポーネントが現在のコンポーネントスコープの一部でない場合、例えば親コンポーネント内で使用される場合にのみ特定のスタイルが必要な子コンポーネントを含める場合に、どのようにスタイルするかでした。

pages/index.js

```
const ChildComponent = () => (
  <div>
    <p>some text</p>
  </div>
);
 
export default function Index() {
  return (
    <div>
      <ChildComponent />
      <style jsx>{`
        p {
          color: black;
        }
      `}</style>
    </div>
  );
}
```

上記のコードは `p` タグを選択しようとしますが、styled-jsx のスタイルは現在のコンポーネントにスコープされているため、子コンポーネントにリークしません。この問題を回避する 1 つの方法は `p` タグのプレフィックスを除去する `:global` メソッドを使用することでした。しかし、これは、スタイルがページ全体にリークするという新たな問題をもたらします。

styled-jsx 3 では、この問題は `css.resolve` という新しい API を導入することで解決され、指定された styled-jsx ストリングに対して `className` と `<style>` タグ (styles プロパティ) を生成します。

pages/index.js

```
import css from 'styled-jsx/css';
 
const ChildComponent = ({ className }) => (
  <div>
    <p className={className}>some text</p>
  </div>
);
 
const { className, styles } = css.resolve`
  p {
    color: black;
  }
`;
 
export default function Index() {
  return (
    <div>
      <ChildComponent className={className} />
      {styles}
    </div>
  );
}
```

この新しい API は、子コンポーネントへのカスタムスタイルの透過的伝達を可能にします。

styled-jsx のメジャーリリースであるため、バウンドサイズを改善する 1 つの breaking change があります。styled-jsx 2 では、「スコープ」と「グローバル」バージョンの両方を生成していましたが、「スコープ」バージョンのみを使用していたとしても、外部スタイルの「グローバル」バージョンも含んでいました。

styled-jsx 3 では、グローバルスタイルは `css` ではなく `css.global` でタグ付けする必要があります。これにより、styled-jsx がバンドルサイズを最適化できます。

すべての変更について、[リリースノート](https://github.com/vercel/styled-jsx/releases/tag/v3.0.0) を参照してください。

React Context with SSR between App and Pages[](#react-context-with-ssr-between-app-and-pages)
---------------------------------------------------------------------------------------------

Next.js 7 から `pages/_app.js` とページコンポーネント間の新しい React context API をサポートするようになりました。

これまでサーバーサイドのページ間で React context を使用することはできませんでした。その理由は、webpack は require.cache を使用するのではなく内部モジュールキャッシュを保持していたためです。モジュールインスタンスをページ間で共有するこの動作を変更するカスタム webpack プラグインを作成しました。

これにより、新しい React context の使用だけでなく、ページ間でコードを共有する際の Next.js のメモリ使用量も削減できます。

|  | 6.0 | 7.0 | delta |
| --- | --- | --- | --- |
| メモリ使用量 | 57.5MB | 48.1MB | **\-16%** メモリ |
