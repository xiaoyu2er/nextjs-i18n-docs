Next.js 16.1 專注於更快速的開發工作流程和更高的穩定性，針對 Turbopack 和工具進行了重大更新。

*   [**針對 `next dev` 的 Turbopack 文件系統緩存（穩定版）](#turbopack-file-system-caching-for-next-dev): 預設情況下改善 `next dev` 的編譯時間。
*   [**Next.js Bundle Analyzer（實驗性）](#nextjs-bundle-analyzer-experimental): 使用我們新的互動工具來優化您的程式碼。
*   [**更輕鬆除錯**](#easier-debugging-with-next-dev---inspect): 使用 `next dev --inspect` 除錯您的 Next.js 應用程式。
*   [**傳遞外部依賴**](#improved-handling-of-serverexternalpackages): Turbopack 現在可以自動處理傳遞外部依賴，無需警告。

Next.js Bundle Analyzer（實驗性）[](#nextjs-bundle-analyzer-experimental)
------------------------------------------------------------------------------

Next.js 16.1 包含了一個與 Turbopack 配合運作的新實驗性 [Next.js Bundle Analyzer (套件分析器)](/docs/app/guides/package-bundling#nextjs-bundle-analyzer-experimental)。它有助於優化工具箱（伺服器端與客戶端代碼）的大小——有助於改善 Core Web Vitals、減少 Lambda 冷啟動時間，並識別臃腫的依賴。

終端機

```
next experimental-analyze
```

執行此命令會啟動互動式 UI 以檢查生產構建套件、識別大模塊，並查看它們被包含的原因。

Next.js Bundle Analyzer 顯示選取 TopNav.tsx 的矩形樹圖，揭示了其匯入鏈。

> **親自試一試：** [開啟互動套件分析器 Demo](https://turbopack-bundle-analyzer-demo.vercel.sh/) 以探索模塊圖表。

套件分析器 (Bundle Analyzer) 深度整合進 Next.js，允許你：

*   按路由篩選套件
*   查看完整的匯入鏈，顯示模塊被包含的原因
*   追蹤跨越伺服器到客戶端元件邊界及動態匯入的引用
*   檢視 CSS 和其他導入資產的大小
*   切換客戶端和伺服器視圖

套件分析器處於早期開發阶段，將在未來的發布中進一步改進。請在專用的 [GitHub 討論](https://github.com/vercel/next.js/discussions/86731) 分享您的反饋。

使用 `next dev --inspect` 更輕鬆除錯 [](#easier-debugging-with-next-dev---inspect)
---------------------------------------------------------------------------------------

您可以透過向 `next dev` 傳遞 `--inspect` 參數來啟用 [Node.js debugger](https://nodejs.org/en/learn/getting-started/debugging)。之前這需要傳遞 `NODE_OPTIONS=--inspect`，並且會將檢查器附加到 Next.js 啟動的所有過程中，而不僅僅是運行您程式碼的過程。

改善 `serverExternalPackages` 處理方式 [](#improved-handling-of-serverexternalpackages)
---------------------------------------------------------------------------------------------

Next.js 允許您使用 [`serverExternalPackages`](https://nextjs.org/docs/app/api-reference/config/next-config-js/serverExternalPackages) 保持依賴不打包。之前，這僅能可靠地用於直接依賴。如果您使用了內部依賴 `sqlite` 等的庫，並且需要外部化 `sqlite`，則必須將其添加到您自己的 `package.json` 中——即使它不是您的直接依賴。這種繞過方式洩漏了內部實作細節，增加了維護負擔，並且當多個包需要同一依賴的不同版本時，可能導致無法解決的版本衝突。

Next.js 16.1 為 Turbopack 修正了這個問題，現在它可以正確解析並將 `serverExternalPackages` 中的傳遞依賴外部化，而無需額外配置。

其他更新 [](#other-updates)
-------------------------------

*   **20MB 更小的安裝**：Next.js 安裝包小了約 20MB，這要歸功於 Turbopack 文件系統緩存層的精簡。
*   **新的 `next upgrade` 命令**：新的 [`next upgrade`](/docs/app/getting-started/upgrading#latest-version) 命令讓升級更簡單。往後，您可以直接執行此命令來升級 Next.js 版本。
*   **MCP `get_routes` 工具**：[Next.js DevTools MCP 伺服器](/docs/app/guides/mcp) 現在有了 `get_routes` 工具來獲取應用程式中的所有路由列表。
*   **`generateStaticParams` 計時**：花費在 [`generateStaticParams`](/docs/app/api-reference/functions/generate-static-params) 上的時間現在會被記錄為開發時顯示請求計時的一部分。
*   **構建執行緒日誌**：`next build` 的 "Collecting page data" 和 "Generating static pages" 現在會記錄所使用的 worker 執行緒數量。
*   **改善非同步匯入打包**：Turbopack 改善了開發環境中 async 匯入的打包方式，以減少產生的 chunk 數量，避免某些病態但真實存在的案例。
*   **相對原始碼圖檔路徑**：Turbopack 現在為伺服器端代碼生成具有相對檔案路徑的原始碼圖檔，提高了與 Node.js 和其他生態繫統工具的相容性。
