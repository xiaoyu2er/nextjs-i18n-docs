Next.js 15 已正式穩定並適用於生產環境。此版本建立在 [RC1](/blog/next-15-rc) 和 [RC2](/blog/next-15-rc2) 的更新之上。我們專注於穩定性，同時加入了一些您可能會喜愛的更新。今天就試試 Next.js 15：

終端機

```
# Use the new automated upgrade CLI
npx @next/codemod@canary upgrade latest
 
# ...or upgrade manually
npm install next@latest react@rc react-dom@rc
```

我們也非常高興分享 Next.js Conf 於 10 月 24 日（星期四）即將帶來的更多內容。

以下是 Next.js 15 的新功能：

*   [**@next/codemod CLI：**|](#smooth-upgrades-with-nextcodemod-cli) 輕鬆升級至最新的 Next.js 和 React 版本。
*   [**非同步請求 API（重大變更）：|](#async-request-apis-breaking-change) 邁向簡化渲染和快取模型的一步。

*   [**Turbopack 開發（穩定版）:|](#turbopack-dev) 效能和穩定性改進。
*   [**靜態指示器：|](#static-route-indicator) 開發期間顯示靜態路由的新視覺指示器。

*   [**打包外部套件（穩定版）：|](#optimizing-bundling-of-external-packages-stable) App 和 Pages Router 的新配置選項。
*   [**ESLint 9 支援：|](#eslint-9-support) 新增對 ESLint 9 的支援。
*   [**開發和建置效能：|](#development-and-build-improvements) 改進建置時間和更快的 Fast Refresh。

使用 `@next/codemod` CLI 進行平滑升級 [](#smooth-upgrades-with-nextcodemod-cli)
---------------------------------------------------------------------------------

我們在每個主要的 Next.js 發布中都會包含 codemods（自動代碼轉換），以幫助自動化升級破壞性變更。

為了讓升級更順暢，我們發布了一個增強版的 codemod CLI：

終端機

```
npx @next/codemod@canary upgrade latest
```

此工具協助您將程式碼庫升級至最新的穩定版或預發版。CLI 會更新您的相依套件、顯示可用的 codemod，並指導您套用這些程式。

`canary` 標籤使用 codemod 的最新版本，而 `latest` 指定 Next.js 版本。我們建議使用 codemod 的金屬版本，即使您升級至最新 Next.js 版本，因為我們計劃根據您的回饋持續改進此工具。

進一步了解 [Next.js codemod CLI](/docs/app/building-your-application/upgrading/codemods)。

異步請求 API（破壞性變更） [](#async-request-apis-breaking-change)
---------------------------------------------------------------------------

在傳統的伺服器端渲染中，伺服器會在渲染任何內容之前等待請求。然而，並非所有元件都依賴特定請求的數據，因此不需要等待請求來渲染它們。理想情況下，伺服器會在請求到達之前盡可能準備。為了實現這一點，並為未來優化奠定基礎，我們需要知道何時等待請求。

因此，我們正在將依賴特定請求數據的 API（例如 `headers`、`cookies`、`params` 和 `searchParams`）轉變為異步。

```
import { cookies } from 'next/headers';
 
export async function AdminPanel() {
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
 
  // ...
}
```

這是一種破壞性變更，會影響以下 API：

* `cookies`
* `headers`
* `draftMode`
* `layout.js`、`page.js`、`route.js`、`default.js`、`generateMetadata` 和 `generateViewport` 中的 `params`
* `page.js` 中的 `searchParams`

為了更易於遷移，這些 API 可以暫時以同步方式存取，但會在開發和生產環境中顯示警告，直到下一個主要版本。已提供 [codemod](/docs/app/building-your-application/upgrading/codemods) 以自動化遷移：

終端機

```
npx @next/codemod@canary next-async-request-api .
```

在 codemod 無法完全遷移程式碼的情況中，請閱讀 [升級指南](/docs/app/building-your-application/upgrading/version-15)。我們 также 提供了 [範例](https://github.com/leerob/next-saas-starter/pull/62) 說明如何將 Next.js 應用程式遷移至新 API。

快取語義 [](#caching-semantics)
---------------------------------------

Next.js App Router 啟動時帶有預設的緩存策略。這些策略旨在預設提供最具效能的選項，並在需要時能夠選擇退出。

基於您的迴響，我們重新評估了我們的 [緩存啟發式演算法](https://x.com/feedthejim/status/1785242054773145636) 以及它們如何與 Partial Prerendering (PPR) 等專案以及使用 `fetch` 的第三方函式庫互動。

隨著 Next.js 15，我們正在更改 `GET` Route Handlers 和 Client Router Cache 的預設快取設定，從預設快取改為預設不快速。如果您想保留之前的行為，可以繼續選擇加入快取。

我們在未來幾個月中將繼續改善 Next.js 中的快取，並將盡快分享更多細節。

### 客戶端路由緩存不再預設緩存 Page 組件[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next 14 中，使用 `GET` HTTP 方法的 路由處理器 (Route Handlers) 預設會被緩存，除非它們使用了動態函數或動態配置選項。在 Next.js 15 中，`GET` 函數**預設不會被緩存**。

您仍可使用靜態路由配置選項來選擇啟用緩存，例如 `export dynamic = 'force-static'`。

特殊的 Route Handlers，如 [`sitemap.ts`](/docs/app/api-reference/file-conventions/metadata/sitemap)、[`opengraph-image.tsx`](/docs/app/api-reference/file-conventions/metadata/opengraph-image) 和 [`icon.tsx`](/docs/app/api-reference/file-conventions/metadata/app-icons)，以及其他 [metadata 檔案](/docs/app/api-reference/file-conventions/metadata) 預設為靜態，除非它們使用動態函式或動態配置選項。

### 客戶端路由緩存 (Client Router Cache) 預設不再緩存頁面組件 (Page Components)[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next.js 14.2.0 中，我們引入了實驗性的 [`staleTimes`](/docs/app/api-reference/next-config-js/staleTimes) 旗標，允許自訂配置 [Router Cache](/docs/app/building-your-application/caching#client-side-router-cache)。

在 Next.js 15 中，此旗標仍然可用，但我們正在更改預設行為，使 Page segments 的 `staleTime` 為 `0`。這意味著當您在應用程式中導航時，客戶端將總是反映作為導航部分變得活動的 Page 組件(s) 的最新數據。然而，仍有一些重要行為保持不變：

*   共用版面配置資料不會從伺服器重新取用，以繼續支援 [局部渲染](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
*   返回/前進導航仍將從快取還原，以確保瀏覽器可以還原捲動位置。
*   [`loading.js`](/docs/app/api-reference/file-conventions/loading) 將快取 5 分鐘（或 `staleTimes.static` 配置的數值）。

您可以通過設定以下配置來選擇加入以前的客戶端路由緩存行為：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
    },
  },
};
 
export default nextConfig;
```

React 19 [#react-19]
---------------------

作為 Next.js 15 版本的一部分，我們已決定配合即將釋出的 React 19。

在第 15 版中，App Router 使用 React 19 RC，我們也根據社群回饋引入了對 React 18 的向下相容性。如果您使用 Pages Router，這允許您在準備就緒時升級至 React 19。

雖然 React 19 仍處於 RC 階段，但我們在真實應用程式中的廣泛測試以及與 React 團隊的密切合作使我們對其穩定性充滿信心。核心重大變更已經過充分測試，不會影響現有的 App Router 使用者。因此，我們決定現在將 Next.js 15 作為穩定版釋出，讓您的專案完全為 React 19 GA 做好準備。

為了確保過渡盡可能順利，我們提供了 [codemod 和自動化工具](#smooth-upgrades-with-codemod-cli) 以協助減輕遷移流程。

閱讀 [Next.js 15 升級指南](/docs/app/building-your-application/upgrading/version-15)、[React 19 升級指南](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)，以及觀看 [React Conf Keynote](https://www.youtube.com/live/T8TZQ6k4SLE?t=1788) 以了解更多。

### Pages Router 上 React 18[](#pages-router-on-react-18)

Next.js 15 為 Pages Router 保持對 React 18 的向下相容性，允許用戶繼續使用 React 18 同時受益於 Next.js 15 的改進。

自第一個 Release Candidate (RC1) 以來，我們已將重心轉向納入對 React 18 的支援，這是基於社群回饋。這種靈活性使您能夠採用 Next.js 15 同時使用帶有 React 18 的 Pages Router，為您提供對升級路徑的更大控制。

> **注意：** 雖然可以在同一應用程式中讓 Pages Router 運行在 React 18 上，而 App Router 運行在 React 19 上，但我們不建議這種設定。這樣做可能會導致不可預測的行為和類型不一致，因為兩個版本之間的底層 API 和渲染邏輯可能無法完全對齊。

### React 編譯器（實驗性質）[](#react-compiler-experimental)

[React Compiler](https://react.dev/learn/react-compiler) 是 React 團隊在 Meta 創建的新實驗性編譯器。該編譯器透過對純 JavaScript 語法和 [React 規則](https://react.dev/reference/rules) 的深度理解來理解您的程式碼，這使得它能夠為您的程式碼添加自動最佳化。它透過 `useMemo` 和 `useCallback` 等 API 減少開發者必須執行的手動 memoization 數量，使程式碼更簡單、更易於維護，且不易出錯。

隨著 Next.js 15，我們已新增對 [React 編譯器](https://react.dev/learn/react-compiler) 的支援。進一步了解 React 編譯器，以及 [可用的 Next.js 配置選項](https://react.dev/learn/react-compiler#usage-with-nextjs)。

> **注意：** React 編譯器目前僅可作為 Babel 插件使用，這將導致開發和建置時間變慢。

使用 `unstable_after` 執行回應後的程式碼（實驗性質）[](#executing-code-after-a-response-with-unstable_after-experimental)
-----------------------------------------------------------------------------------------------------------------------------------------

在處理使用者請求時，伺服器通常會執行與計算回應直接相關的任務。然而，您可能需要執行記錄、分析和其他外部系統同步等任務。

由於這些任務與回應不直接相關，使用者不應該等待它們完成。在回應使用者後延遲工作會帶來挑戰，因為 serverless 函式會在回應關閉後立即停止計算。

`after()` 是一個新的實驗性 API，它解決了這個問題，允許您排定工作以在回應完成串流後處理，使得次要任務能夠執行而不阻擋主要回應。

要使用它，請在 `next.config.js` 中新增 `experimental.after`：

next.config.ts

```
const nextConfig = {
  experimental: {
    after: true,
  },
};
 
export default nextConfig;
```

然後，在 Server Components、Server Actions、Route Handlers 或 Middleware 中匯入該函數。

```
import { unstable_after as after } from 'next/server';
import { log } from '@/app/utils';
 
export default function Layout({ children }) {
  // Secondary task
  after(() => {
    log();
  });
 
  // Primary task
  return <>{children}</>;
}
```

`instrumentation.js`（穩定版） [](#instrumentationjs-stable)
----------------------------------------------------------

`instrumentation` 檔案配合 `register()` API，允許用戶介入 Next.js 伺服器生命週期以監控效能、追蹤錯誤來源，並與 [OpenTelemetry](https://opentelemetry.io/) 等可觀測性函式庫深度整合。

此功能現在已 **穩定**，`experimental.instrumentationHook` 配置選項可以移除。

此外，我們還與 [Sentry](https://sentry.io/) 合作設計了新的 `onRequestError` 鉤子，可用於：

*   捕捉伺服器上拋出的所有錯誤的重要上下文，包括：
    *   **Router：** Pages Router 或 App Router
    *   **伺服器上下文：** Server Component、Server Action、Route Handler 或 Middleware
*   將錯誤報告給您喜愛的可觀測性提供者。

```
export async function onRequestError(err, request, context) {
  await fetch('https://...', {
    method: 'POST',
    body: JSON.stringify({ message: err.message, request, context }),
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export async function register() {
  // init your favorite observability provider SDK
}
```

進一步了解 `onRequestError` [函數](/docs/app/api-reference/file-conventions/instrumentation#onrequesterror-optional)。

`<Form>` 組件 [](#form-component)
-------------------------------------

新的 `<Form>` 元件擴充了 HTML `<form>` 元素，帶有 [預先取用](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)、[客戶端導航](/docs/app/building-your-application/routing/linking-and-navigating#5-soft-navigation) 和漸進式增強。

它適用於導航到新頁面的表單，例如導向結果頁面的搜尋表單。

app/page.jsx

```
import Form from 'next/form';
 
export default function Page() {
  return (
    <Form action="/search">
      <input name="query" />
      <button type="submit">Submit</button>
    </Form>
  );
}
```

`<Form>` 元件配備了：

*   **預先取用：** 當表單在視圖範圍內時，[版面配置](/docs/app/api-reference/file-conventions/layout) 和 [快速載入](/docs/app/api-reference/file-conventions/loading) UI 會被預先取用，使導航快速。
*   **客戶端導航：** 提交時，共用版面配置和客戶端狀態保留。
*   **漸進式增強：** 如果 JavaScript 尚未載入，表單仍然可以通過全頁導航工作。

過去，實現這些功能需要大量手動的範本碼。例如：

範例

```
// Note: This is abbreviated for demonstration purposes.
// Not recommended for use in production code.
 
'use client'
 
import { useEffect } from 'react'
import { useRouter } from 'next/navigation'
 
export default function Form(props) {
  const action = props.action
  const router = useRouter()
 
  useEffect(() => {
    // if form target is a URL, prefetch it
    if (typeof action === 'string') {
      router.prefetch(action)
    }
  }, [action, router])
 
  function onSubmit(event) {
    event.preventDefault()
 
    // grab all of the form fields and trigger a `router.push` with the data URL encoded
    const formData = new FormData(event.currentTarget)
    const data = new URLSearchParams()
 
    for (const [name, value] of formData) {
      data.append(name, value as string)
    }
 
    router.push(`${action}?${data.toString()}`)
  }
 
  if (typeof action === 'string') {
    return <form onSubmit={onSubmit} {...props} />
  }
 
  return <form {...props} />
}
```

next.config.ts 支援 [](#support-for-nextconfigts)
---------------------------------------------------------

Next.js 現在支援 TypeScript `next.config.ts` 檔案類型，並提供 `NextConfig` 型別以用於自動完成和類型安全選項：

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  /* config options here */
};
```

自主託管改進 [](#improvements-for-self-hosting)
---------------------------------------------------------------

當自主託管應用程式時，您可能需要對 `Cache-Control` 指令有更多控制權。

一個常見的案例是控制發送到 ISR 頁面的 `stale-while-revalidate` 期間。我們實作了兩項改進：

1.  您現在可以在 `next.config` 中配置 [`expireTime`](/docs/app/api-reference/next-config-js/expireTime) 數值。這以前是 `experimental.swrDelta` 選項。
2.  已更新預設值為一年，確保大多數 CDN 可以完全按預期套用 `stale-while-revalidate` 語義。

我們也不再覆蓋自訂 `Cache-Control` 值為我們的預設值，允許完全控制並確保與任何 CDN 設定相容。

最後，我們改進了自主託管時的圖片優化。過去，我們建議您安裝 `sharp` 以在 Next.js 伺服器上優化圖片。此建議有時會被遺漏。隨著 Next.js 15，您不再需要手動安裝 `sharp` — `next start` 或使用 [單獨輸出模式](/docs/app/api-reference/next-config-js/output) 執行時，Next.js 將自動使用 `sharp`。

要了解更多詳情，請參閱我們關於自主託管 Next.js 的 [新示範和教學影片](https://x.com/leeerob/status/1843796169173995544)。

伺服器操作的增強安全性 [](#enhanced-security-for-server-actions)
-----------------------------------------------------------------------------

[伺服器操作](https://react.dev/reference/rsc/server-actions) 是在伺服器端定義、可從客戶端呼叫的函數。它們通过在文件頂部添加 'use server' 指令並匯出異步函數來定義。

即使 Server Action 或函式未在您的程式碼中的其他地方導入，它仍然是公共可存取的 HTTP 端點。雖然這種行為在技術上是正確的，但這可能會導致這些功能的意外洩露。

為了提升安全性，我們引入了以下改進：

*   **死碼消除：** 未使用的 Server Action 不會將其 ID 暴露給客戶端 JavaScript 套包，減少套包大小並提升效能。
*   **安全的動作 ID：** Next.js 現在創建不可猜測的非確定性 ID，允許客戶端參照和呼叫 Server Action。這些 ID 會在建置之間定期重新計算以增強安全性。

```
// app/actions.js
'use server';
 
// This action **is** used in our application, so Next.js
// will create a secure ID to allow the client to reference
// and call the Server Action.
export async function updateUserAction(formData) {}
 
// This action **is not** used in our application, so Next.js
// will automatically remove this code during `next build`
// and will not create a public endpoint.
export async function deleteUserAction(formData) {}
```

您仍應將 Server Actions 視為公共 HTTP 端點。進一步了解 [保護 Server Actions](/blog/security-nextjs-server-components-actions#write)。

最佳化外部套件打包 (穩定)[](#optimizing-bundling-of-external-packages-stable)

打包外部套件可以改進應用程式的冷啟動效能。在 **App Router** 中，外部套件預設打包，您可以使用新的 [`serverExternalPackages`](/docs/app/api-reference/next-config-js/serverExternalPackages) 配置選項取消特定套包的捆綁。

在 **Pages Router** 中，預設不打包外部套件，但您可以提供一個套件清單使用現有的 [`transpilePackages`](/docs/pages/api-reference/next-config-js/transpilePackages) 選項。使用此配置選項時，您需要指定每個套件。

為了統一 App 和 Pages Router 之間的配置，我們新增了選項 [`bundlePagesRouterDependencies`](/docs/pages/api-reference/next-config-js/bundlePagesRouterDependencies) 以匹配 App Router 的預設自動打包。然後您可以使用 [`serverExternalPackages`](/docs/app/api-reference/next-config-js/serverExternalPackages) 來取消特定套包的捆綁（如果需要）。

next.config.ts

```
const nextConfig = {
  // Automatically bundle external packages in the Pages Router:
  bundlePagesRouterDependencies: true,
  // Opt specific packages out of bundling for both App and Pages Router:
  serverExternalPackages: ['package-name'],
};
 
export default nextConfig;
```

ESLint 9 支援 [](#eslint-9-support)
-------------------------------------

Next.js 15 也引入了對 [ESLint 9](https://eslint.org/blog/2024/04/eslint-v9.0.0-released) 的支援，隨後 ESLint 8 於 2024 年 10 月 5 日終止支援。

為了確保平穩過渡，Next.js 仍保留向下相容，意味著您仍可使用 ESLint 8 或 9。

如果您升級至 ESLint 9，且我們偵測到您尚未採用 [新配置格式](https://eslint.org/blog/2024/04/eslint-v9.0.0-released/#flat-config-is-now-the-default-and-has-some-changes)，Next.js 將自動套用 `ESLINT_USE_FLAT_CONFIG=false` 的escape hatch 以減輕遷移。

此外，運行 `next lint` 時將移除已棄用的選項，如 `—ext` 和 `—ignore-path`。請注意，ESLint 最終将在 ESLint 10 中不再允許這些舊配置，因此我們建議您盡快開始遷移。

有關這些變更的更多詳情，請參閱 [遷移指南](https://eslint.org/docs/latest/use/migrate-to-9.0.0)。

作為此更新的一部分，我們也升級了 `eslint-plugin-react-hooks` 至 `v5.0.0`，這引入了用於 React Hooks 使用的新規則。您可以在 [eslint-plugin-react-hooks@5.0.0 的變更日誌](https://github.com/facebook/react/releases/tag/eslint-plugin-react-hooks%405.0.0) 中檢視所有變更。

開發與建置改進 [](#development-and-build-improvements)
-------------------------------------------------------------------------

### Server Components HMR [](#server-components-hmr)

在開發期間，Server 元件在儲存時會重新執行。這意味著，對 API 端點或第三方服務的 `fetch` 請求也會被呼叫。

為了改進本地開發效能並降低已計費 API 呼叫的潛在成本，我們現在確保 Hot Module Replacement (HMR) 可以重用先前渲染中的 `fetch` 回應。

進一步了解 [Server Components HMR 快取](/docs/app/api-reference/next-config-js/serverComponentsHmrCache)。

### App Router 的靜態生成速度更快 [](#faster-static-generation-for-the-app-router)

我們已優化靜態生成以改善建置時間，特別是對於有慢速網路請求的頁面。

過去，我們靜態優化流程會渲染兩次頁面——一次用於生成用戶端導航數據，二次用於渲染初始頁面訪問的 HTML。現在，我們重用第一次渲染，省去第二次遍歷，減少工作量和建置時間。

此外，靜態產生工作線程現在在頁面之間共用 `fetch` 快取。如果 `fetch` 呼叫沒有選擇退出快取，其結果將由由同一工作線程處理的其他頁面重用。這減少了相同資料的請求數量。

### 進階靜態生成控制（實驗性） [](#advanced-static-generation-control-experimental)

我們已新增實驗性支援，提供更多對靜態產生流程的控制，適合那些需要更大控制力的進階使用案例。

我們建議除非您有特定需求，否則使用目前的預設值，因為這些選項可能會導致資源使用量增加，並因並發性增加而出現記憶體耗盡錯誤。

next.config.ts

```
const nextConfig = {
  experimental: {
    // how many times Next.js will retry failed page generation attempts
    // before failing the build
    staticGenerationRetryCount: 1
    // how many pages will be processed per worker
    staticGenerationMaxConcurrency: 8
    // the minimum number of pages before spinning up a new export worker
    staticGenerationMinPagesPerWorker: 25
  },
}
 
export default nextConfig;
```

進一步了解 [靜態產生選項](/docs/app/api-reference/next-config-js/staticGeneration)。
