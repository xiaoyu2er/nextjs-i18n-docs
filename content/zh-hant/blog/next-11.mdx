我們曾在 [Next.js Conf](/conf) 上宣布，建立最佳開發者體驗的使命將透過 Next.js 11 得以延續，包含：

*   [**合規性 (Conformance)**](#conformance): 一個精心策劃解決方案以支援最佳使用者體驗 (UX) 的系統。
*   [**效能改進 (Improved Performance)**](#improved-performance): 進一步優化以改善冷啟動時間，讓您可以更快地開始編碼。
*   [**`next/script`**](#script-optimization): 自動優先載入第三方腳本以改善效能。
*   [**`next/image`**](#image-improvements): 減少版面佈局偏移並透過自動大小偵測和支援模糊置換佔位符來建立更順暢的視覺體驗。
*   [**Webpack 5**](#webpack-5): 已預設啟用於所有 Next.js 應用程式，將 [這些優勢](/blog/next-10-2#webpack-5) 帶給所有 Next.js 開發者。
*   [**Create React App 遷移 (Experimental)**](#cra-migration): 自動將 Create React App 轉換為與 Next.js 相容。
*   [**Next.js Live（預覽版）**](#nextjs-live-preview-release): 在瀏覽器中與您的團隊即時編碼。

請執行 `npm i next@latest react@latest react-dom@latest` 以立即更新，並參考下方的 [遷移指南](#upgrade-guide)。

合規性 (Conformance)[](#conformance)
---------------------------

即使框架中有優秀的工具和本機優化，網站負責人和應用程式開發者經常被要求成為 UX 品質主題（例如效能、安全性和無障礙性）的專家。隨著功能增加和團隊擴大，開發者需要改變思維方式。

透過建立大型 Web 應用程式（如 Search 和 Maps）的工作，Google 已證明框架在團隊和應用程式擴展時維持品質方面可發揮關鍵作用。透過利用強大預設值和保護措施的系統，他們能賦能開發者更專注於功能和產品。

今日，Google 的 Web 平台團隊已開始公開其 **Next.js 合規性 (Conformance for Next.js)** 系統。

[合規性 (Conformance)](http://web.dev/conformance) 是一個提供精心策劃解決方案和規則的系統，以支援最佳載入和 Core Web Vitals，未來將增加更多支援安全性和無障礙性等品質面向的功能。這些解決方案讓您的團隊無需背誦所有最新規則即可擁最佳載入效能，同時也提供彈性讓您為應用程式做出正確選擇。

除了許多基於效能研究的基礎優化外，**Next.js 11 現在預裝 [ESLint](https://eslint.org/) 支援**，讓您在開發期間更容易找出框架特定問題，並為您的團隊設定指南，以確保在擴展時仍符合最佳實踐。

開始使用 ESLint，請在升級到 Next.js 11 後執行 `npx next lint`。ESLint 整合適用於新和現有的 Next.js 應用程式，提供一套新規則協助開發者建構更好的應用程式。

效能改進 [](#improved-performance)
---------------------------------------------

從 Next.js 10 開始，我們一直致力于進一步改進 Next.js 的開發者體驗。在 [10.1](/blog/next-10-1) 和 [10.2](/blog/next-10-2) 中，我們透過 React Fast Refresh 將啟動時間改善了 **多達 24%**，並減少 **40% 的處理時間**用於變更。您只需持續更新 Next.js 就能獲得這些驚人的效能提升。

Next.js 11 包含另一項針對 Babel 的優化，進一步減少啟動時間。我們為 webpack 創建了一個全新的 Babel loader 實現，優化載入並新增記憶體配置快取層。實際上，這意味著**對開發者無需變更**，但最終將帶來更快速的開發體驗。

腳本最佳化 (Script Optimization)[](#script-optimization)
-------------------------------------------

新的 Next.js Script 元件 (Script Component) 是一個基礎優化，可讓開發者設定第三方腳本的載入優先順序，以節省開發者時間並改善載入效能。

網站經常需要第三方服務，例如分析、廣告、客戶支援小工具 (widgets) 和同意管理。然而，這些腳本往往會增加載入效能負擔，並拖慢使用者體驗。開發者通常難以決定在應用程式中最佳化載入時應將其放置何處。

使用 `next/script`，您可以定義 `strategy` 屬性，Next.js 將自動優化它們的載入順序以改善效能：

*   `beforeInteractive`：用於需要在頁面可互動**之前**抓取並執行的關鍵腳本，例如機器人偵測和同意管理。這些腳本由伺服器注入至初始 HTML 中，並在自行打包的 JavaScript 執行前運行。
*   `afterInteractive`（**預設值**）：用於可在頁面可互動**之後**抓取並執行的腳本，例如標籤管理員和分析工具。這些腳本注入在客戶端，並在水合 (hydration) 後運行。
*   `lazyOnload`：用於可於閒置時間等待載入的腳本，例如聊天諮詢和社交媒體小工具。

```
<Script
  src={url}
  strategy="beforeInteractive" // lazyOnload, afterInteractive
/>
```

您也可以在載入後執行代碼。例如，您可以等待直到使用者回答同意後再執行代碼：

```
<Script
  src={url} // consent mangagement
  strategy="beforeInteractive"
  onLoad={() => {
    // If loaded successfully, then you can load other scripts in sequence
  }}
/>
```

我們也將 Next.js 11 中的預設腳本載入體驗從預載 (`preloading`) 和 `async` 更改為 `defer`。第三方腳本通常會與更高優先順序的資源（例如 CSS、字型和圖片）競爭。維持與這些資源以及其他腳本的適當排序，對開發者來說是過度的負擔。

透過提供預設載入策略為 `afterInteractive` 的 Script Component，開發者現在擁有更好的預設值以達到最佳效能，同時仍可根據需求選擇 `beforeInteractive`。

欲了解更多關於切換預設值背後的技術選擇，請參閱 Google Chrome 團隊的 [RFC](https://github.com/vercel/next.js/discussions/24938) 和 [關於預載的挑戰](https://docs.google.com/document/u/0/d/1ZEi-XXhpajrnq8oqs5SiW-CXR3jMc20jWIzN5QRy1QA/mobilebasic#)。

圖片改進 (Image Improvements)[](#image-improvements)
-----------------------------------------

我們很興奮分享社群最受歡迎的兩個 `next/image` 元件功能：減少 [累積版面佈局偏移 (CLS)](https://vercel.com/blog/core-web-vitals#cumulative-layout-shift)，並建立更順暢的視覺體驗。

### 自動偵測大小（本地圖片） (Automatic Size Detection (Local Images))[](#automatic-size-detection-local-images)

對圖片 `src` 使用 `import` 關鍵字，以自動為靜態圖片定義 `width` 和 `height`。

例如，使用 [內建 Image 元件](/docs/pages/building-your-application/optimizing/images) 現在更加簡易：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Image from 'next/image';
import author from '../public/me.png';
 
export default function Home() {
  return (
    // When importing the image as the source, you
    // don't need to define `width` and `height`.
    <Image src={author} alt="Picture of the author" />
  );
}
```

### 圖片置換符 (Image Placeholders)[](#image-placeholders)

`next/image` 現在支援模糊置換佔位符 (blur-up placeholders)，以緩和從空白空間過渡到圖片的過程，並減少感知載入時間，特別是對於網際網路連線較慢的使用者。

要使用模糊置換符，請將 `placeholder="blur"` 新增到您的圖片中。

```
<Image src={author} alt="Picture of the author" placeholder="blur" />
```

Next.js 也支援透過允許您提供自訂 `blurDataURL` 來模糊化動態圖片，這是由您的後端提供的。例如，您可以在伺服器上產生 [blurha.sh](http://blurha.sh)。

Webpack 5[](#webpack-5)
-----------------------

在 Next.js 10.2 中，我們將 webpack 5 的推廣擴展至所有在 `next.config.js` 中沒有自訂 webpack 配置的應用程式。今日，我們將 **webpack 5 設為所有 Next.js 應用程式的預設值**，這將提供 [多方面的功能和改進](/blog/next-10-2#webpack-5)。

我們與社群密切合作，確保順利過渡至 webpack 5，超過 3,400 個現有的 Next.js 整合測試會在每次合併要求 (pull request) 上運行，並啟用 webpack 5。

如果您的應用程式擁有自訂 webpack 配置，我們建議遵循 [webpack 5 的升級文件](/docs/messages/webpack5)。如果您遇到任何問題，[請與我們分享回饋](https://github.com/vercel/next.js/discussions/23498)。

CRA 遷移 (CRA Migration)[](#cra-migration)
-------------------------------

過去六個月中，我們看到更多應用程式 [從 Create React App 遷移至 Next.js](/docs/app/building-your-application/upgrading/from-create-react-app) 的成長，以利用 Next.js 提供的許多開發者體驗和最終用戶效能改進。

為了協助開發者將應用程式轉換為 Next.js，我們為 `@next/codemod` 引進了一個新工具，可將 Create React App 應用程式自動轉換為與 Next.js 相容。

轉換程式會自動新增 `pages/` 目錄並移動 CSS 匯入至正確的位置。它也會在 Next.js 中啟用 Create React App 相容模式，以確保 Create React App 使用的某些範式能在 Next.js 中運行。

透過運用新轉換程式，您可以在維持現有 Create React App 應用程式功能的同时，[逐步採用 Next.js](/docs/migrating/incremental-adoption)。

開始遷移您的 Create React App 專案，請使用以下命令：

終端機

```
npx @next/codemod cra-to-next
```

此功能目前處於實驗階段，請在 [本討論區](https://github.com/vercel/next.js/discussions/25858) 分享任何回饋。

Next.js Live（預覽版） (Next.js Live (Preview Release))[](#nextjs-live-preview-release)
--------------------------------------------------------------

Next.js Live 延續了我們讓開發不僅 **更快** 且更愉快，但更重要的是 **更包容整個組織** 的使命。透過運用 ServiceWorker、WebAssembly 和 ES Modules 等尖端技術，Next.js Live 將整個開發流程置於網站瀏覽器中。這開啟了可能性，如使用 URL 即時協作和分享，**無需建置步驟**。對於開發者，這意味著更快速的回饋迴圈，減少等待建置的時間，以及在瀏覽器中進行即時同儕編碼和編輯。

升級指南 (Upgrade Guide)[](#upgrade-guide)
-------------------------------

Next.js 11 引入了一些破壞性變更，但應該不會影響大多數使用者。這些傳統功能已以向後相容的方式維護多年，有些可追溯至 `v4.0`。

移除這些功能是為了減少套件大小，並確保程式碼庫未來可維持維護。要了解從版本 10 升級至 11 的相關資訊，請參閱 [升級指南](https://github.com/vercel/next.js/blob/canary/docs/pages/building-your-application/upgrading.md)。

隨著 Next.js 11，最低 React 版本已更新為 `17.0.2`。詳情請參閱 [React 17 部落格文章](https://reactjs.org/blog/2020/08/10/react-v17-rc.html)。我們也與 React 團隊緊密合作，因為他們正在推出 [React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html)。當使用 React 18 alpha 時，Next.js 11 使用 `createRoot`。
