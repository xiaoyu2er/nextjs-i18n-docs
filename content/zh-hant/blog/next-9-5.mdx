穩定增量靜態生成 (Stable Incremental Static Regeneration)[](#stable-incremental-static-regeneration)
---------------------------------------------------------------------------------

Next.js 在 9.3 版引入了 [靜態網站生成方法](https://nextjs.org/blog/next-9-3#next-gen-static-site-generation-ssg-support)，目標明確：我們應該獲得 [靜態的好處](https://rauchg.com/2020/2019-in-review#static-is-the-new-dynamic)（始終快速、始終在線上、[全域複製](https://rauchg.com/2020/static-hoisting#hoist-to-the-edge)），同時獲得 Next.js 以之聞名的動態資料優秀支援。

為了兼顧兩者優點，Next.js 引入了 **增量靜態生成**，在您已建置完網站後更新靜態內容。使用 [`getStaticPaths` 中的 `fallback: true` 選項](/docs/api-reference/data-fetching/get-static-paths#fallback-false)，您可以在 _執行時間_ **註冊** **新的靜態頁面**。

Next.js 可以以這種方式靜態預先渲染任意數量的頁面，無論您的資料集有多大。

今天，我們宣布 **增量靜態重新生成 (Incremental Static Re-generation)** 的全面可用性，這是一種在背景重新渲染頁面，以響應流量時 **更新 _現有_ 頁面** 的機制。

受 [stale-while-revalidate](https://tools.ietf.org/html/rfc5861) 啟發，背景重新生成確保流量持續獲得服務，始終來自靜態儲存庫，並且新建置的頁面僅在其 **生成完成後** 才推送。

```
export async function getStaticProps() {
  return {
    props: await getDataFromCMS(),
    // we will attempt to re-generate the page:
    // - when a request comes in
    // - at most once every second
    revalidate: 1,
  };
}
```

> `revalidate` 旗標是指在不引發 [緩衝區沖刷 (Cache stampede)](https://en.wikipedia.org/wiki/Cache_stampede) 的情況下，最多只會發生一次生成的秒數。

與傳統 SSR 不同，增量靜態重新生成確保您保留靜態的好處：

*   不會出現延遲尖峰。頁面始終保持一致的快速。
*   頁面絕不會離線。如果背景頁面重新生成失敗，舊頁面保持未更改狀態。
*   低資料庫和後端負載。頁面在 *並行* 的情況下最多重新計算一次。

漸進式功能（新增頁面和延遲更新它們），以及 [預覽模式](/docs/pages/building-your-application/configuring/preview-mode)，現在都已經穩定，並且已受 `next start` 和 [Vercel Edge 平台](https://vercel.com) 完全支援，開箱即用。

為了展示此新功能，我們建立了一個範例，顯示重新生成靜態頁面，顯示特定問題上各種 GitHub 反應的計數：[https://reactions-demo.vercel.app/](https://reactions-demo.vercel.app/)

![在我們 emoji 反應後的第一次訪問後，新的頁面生成在背景中啟動。整個過程中的所有請求都來自靜態快取。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9-5/reactions-demo.gif)

在我們 emoji 反應後的第一次訪問後，新的頁面生成在背景中啟動。整個過程中的所有請求都來自靜態快取。

接下來，我們將針對另外兩個額外增量靜態生成功能撰寫補充 RFC：

*   一次性重新生成和無效化多個頁面（例如部落格主頁和特定部落格文章）
*   聆聽事件（例如 CMS Webhooks）進行重新生成，而非用戶流量之前

更多詳情，請查看 [`getStaticProps` 文件](/docs/pages/building-your-application/data-fetching#getstaticprops-static-generation)。

可自訂基礎路徑 (Customizable Base Path)[](#customizable-base-path)
-------------------------------------------------

Next.js 專案並非總是從網域的根級別服務。有時您希望在 `/docs` 這樣的子路徑下託管 Next.js 專案，使 Next.js 專案僅涵蓋網域的該子部分。

雖然這一直是可能的，但代價是相當額外的配置。例如，將前綴新增到每個 `<Link>` 並確保 Next.js 從正確路徑服務 JavaScript 彙編。

為解決此痛點，我們正在引入新的配置選項。`basePath` 允許您輕鬆在網域子路徑上託管 Next.js 專案。

開始使用 `basePath` 時，您可以將其新增到 `next.config.js`：

next.config.js

```
module.exports = {
  basePath: '/docs',
};
```

配置 `basePath` 後，您的專案將自動從提供的路徑進行路由。在此情況下為 `/docs`。

使用 `next/link` 或 `next/router` 連結到專案中的其他頁面時，`basePath` 將自動加上前綴。這讓您可以變更 `basePath` 而無需修改您的專案。

範例如下所示，使用 `next/link` 路由至另一個頁面：

```
import Link from 'next/link';
 
export default function HomePage() {
  return (
    <>
      <Link href="/documentation-page">
        <a>Documentation page</a>
      </Link>
    </>
  );
}
```

以這種方式使用 `next/link` 會導致以下 HTML 在網頁瀏覽器中渲染：

```
<a href="/docs/documentation-page">Documentation page</a>
```

如需更多詳細資訊，請參閱 [`basePath` 文件](/docs/pages/api-reference/next-config-js/basePath)。

重寫 (Rewrites)、重定向 (Redirects) 和標頭 (Headers)[#support-for-rewrites-redirects-and-headers]
-------------------------------------------------------------------------------------------

### 重寫 (Rewrites)[#rewrites]

構建 Next.js 專案時，您可能想要將特定路由代理到其他 URL。例如，如果您想要逐步將 Next.js 引入您的技術棧，您可能會希望將 Next.js 專案中已存在的頁面與舊專案（您打算遷移的專案）中未被匹配的內容進行路由。

隨著 Next.js 9.5，我們引入了一個名為 `rewrites` 的新配置選項，它允許您將請求路徑映射到不同的目的地路徑，包含外部 URL。

例如，您可能會想要將特定路由重寫至 `example.com`。

next.config.js

```
module.exports = {
  async rewrites() {
    return [
      { source: '/backend/:path*', destination: 'https://example.com/:path*' },
    ];
  },
};
```

在這種情況下，`/backend` 下的所有路徑將被路由至 `example.com`。

您還可以先檢查 Next.js 專案的路由是否匹配，若無匹配則重寫至之前的專案。這對於 **Next.js 的逐步採用** 非常有幫助：

```
module.exports = {
  async rewrites() {
    return [
      // check if Next.js project routes match before we attempt proxying
      {
        source: '/:path*',
        destination: '/:path*',
      },
      {
        source: '/:path*',
        destination: `https://example.com/:path*`,
      },
    ];
  },
};
```

在此情況下，我們首先匹配所有路徑。如果沒有匹配項，則代理至 `example.com`，這將是之前的專案。

要了解有關 `rewrites` 功能的更多資訊，請參閱 [重寫 (Rewrites) 文件](/docs/pages/api-reference/next-config-js/rewrites)。

### 重定向 (Redirects)[#redirects]

大多數網站至少需要一些重定向。特別是當變更專案路由結構時。例如，將 `/blog` 移動到 `/news` 或類似的遷移。

以往在 Next.js 專案中列出重定向需要設置自訂伺服器或自訂 `_error` 頁面來檢查路由是否已設置重定向。然而，這代價是無效了關鍵的靜態和無伺服器優化（通過設置伺服器）或者不夠實用。

從 Next.js 9.5 開始，現在您可以在 `next.config.js` 下的 `redirects` 鍵中創建重定向列表：

next.config.js

```
module.exports = {
  async redirects() {
    return [
      {
        source: '/about',
        destination: '/',
        permanent: true,
      },
    ];
  },
};
```

要了解有關 `redirects` 功能的更多資訊，請參閱 [重定向 (Redirects) 文件](/docs/pages/api-reference/next-config-js/redirects)。

### 標頭 (Headers)[#headers]

Next.js 允許您構建同時使用靜態生成 (Static Generation) 和伺服器端渲染 (Server-Side Rendering) 的混合專案。使用伺服器端渲染時，您可以為請求設置標頭。對於靜態頁面，設置標頭在此之前是不可能的。

現在我們在 `next.config.js` 中引入了 `headers` 屬性，適用於所有 Next.js 路由：

next.config.js

```
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*',
        headers: [
          {
            key: 'Feature-Policy',
            // Disable microphone and geolocation
            value: "microphone 'none'; geolocation 'none'",
          },
        ],
      },
    ];
  },
};
```

`headers` 選項允許您設置常用的標頭，例如 [`Feature-Policy`](https://developer.mozilla.org/docs/Web/HTTP/Headers/Feature-Policy) 和 [`Content-Security-Policy`](https://developer.mozilla.org/docs/Web/HTTP/CSP)。

要了解有關 `headers` 功能的更多資訊，請參閱 [標頭 (Headers) 文件](/docs/pages/api-reference/next-config-js/headers)。

URL 末尾可選的斜線 (#optional-trailing-slash-in-urls)
-------------------------------------------------------------------

3 年前引入 Next.js 時，其預設行為是將帶有結尾斜線的所有 URL 均返回 404 頁面。

雖然有效，但一些用戶要求更改此行為的選項。例如，將此前嚴格要求結尾斜線的現有專案遷移至 Next.js 時。

隨著 Next.js 9.5，我們在 `next.config.js` 中引入了名為 `trailingSlash` 的新選項。

此新選項確保 Next.js 自動處理結尾斜線行為：

*   自動將帶有結尾斜線的 URL 重定向至沒有結尾斜線的 URL，例如：`/about/` 轉為 `/about`
*   當 `trailingSlash` 設置為 `true` 時，沒有結尾斜線的 URL 將被重定向至帶有結尾斜線的 URL，例如：`/about` 轉為 `/about/`
*   確保 `next/link` 會自動應用/移除結尾斜線，以避免無謂的重定向。

next.config.js

```
module.exports = {
  // Force a trailing slash, the default value is no trailing slash (false)
  trailingSlash: true,
};
```

配置 `basePath` 後，您的專案將自動從提供的路徑路由。在這種情況下為 `/docs`。

```

```

隨著 Next.js 9.5，我們引入名為 `rewrites` 的新配置選項，允許您將進來的請求路徑映射到不同的目標路徑，包括外部 URL。

例如，您可能希望重寫特定路由到 `example.com`：

next.config.js

在這種情況下，`/backend` 下的所有路徑都會路由到 `example.com`。

您也可以檢查 Next.js 專案路由是否匹配，然後如果沒有匹配則重寫到先前的專案。這對於 **Next.js 的漸進式採用** 至關重要：

```
/_next/static/chunks/pages/about.qzfS4o5gIEXRME6sTEahL.js
```

在這種情況下，我們首先匹配所有路徑。如果沒有匹配則代理到 `example.com`，這是之前專案。

### 重定向 (Redirects)[](#redirects)

大多數網站至少需要一些重定向。特別是當變更專案路由結構時。例如，當將 `/blog` 移到 `/news` 或類似轉換時。

先前在 Next.js 專案中擁有一組重定向清單需要設置自訂伺服器或自訂 `_error` 頁面來檢查是否存在路由重定向。然而，這代價是無效化關鍵的靜態和無伺服器最佳化（通過擁有伺服器）或者不夠方便。

*   自動將尾隨斜線 URL 重定向到無尾隨斜線的 URL，例如：`/about/` 到 `/about`
*   當 `trailingSlash` 設置為 `true` 時，無尾隨斜線的 URL 將重定向到帶尾隨斜線的 URL，例如：`/about` 到 `/about/`
*   確保 `next/link` 自動套用/移除尾隨斜線，避免無謂的重定向。

---------------------------------------------------------------------------

撰寫 Next.js 頁面時，所有腳本彙編、CSS 樣式表和 HTML 的創建完全是自動的，並對您抽象。如果您在 Next.js 9.5 之前檢查生成的 `<script>` 標籤，您會注意到它們的 URL 遵循此模式：

上述路徑段 `ovgxWYrvKyjnlM15qtz7h` 是我們所稱之為 _建置 ID_。雖然這些檔案在邊緣和使用者機器上容易快取，但重新建置您的應用程式後，建置 ID 會變更，所有快取都會失效。


在 Next.js 9.2 中引入的 [改進的程式碼分割策略](https://nextjs.org/blog/next-9-2#improved-code-splitting-strategy)，與 Google Chrome 團隊合作開發，為 Next.js 頁面彙編生成這些改進奠定了一些基礎。

可選的 Catch All 路由 (Optional Catch All Routes)[](#optional-catch-all-routes)
-------------------------------------------------------

Next.js 9.2 加入了 [Catch All 動態路由支援](https://nextjs.org/blog/next-9-2#catch-all-dynamic-routes)，已被社群廣泛採用於各種案例。Catch All 路由為您提供靈活性，建立高度動態的路由結構，由 Headless CMS、GraphQL APIs、檔案系統等驅動。

*   [改進的 Tree-Shaking](https://github.com/webpack/changelog-v5#nested-tree-shaking)： nested exports, inner-module, and CommonJS 進行 Tree-Shaking
*   [持久快取](https://github.com/webpack/changelog-v5#persistent-caching)：允許重用先前建置的工作
*   [確定性的 Chunk 和模組 ID](https://github.com/webpack/changelog-v5#deterministic-chunk-and-module-ids)：解決 Webpack 模組 ID 在建置之間會變更的案例

我們今天很高興地宣布 Next.js 的 Webpack 5 測試版可用性。

要試用 Webpack 5，您可以在 `package.json` 中使用 [Yarn resolutions](https://classic.yarnpkg.com/en/docs/selective-version-resolutions/)：

package.json

```
{
  "resolutions": {
    "webpack": "^5.0.0-beta.30"
  }
}
```

Webpack 5 beta 已經在 [nextjs.org](https://nextjs.org) 和 [vercel.com](https://vercel.com) 生產環境中部署。我們鼓勵您以漸進的方式試用，並在 [GitHub](https://github.com/vercel/next.js/issues/13341) 回報您的發現。

### 編譯基礎設施改進 (Compilation infrastructure improvements)[](#compilation-infrastructure-improvements)

為了支援 Webpack 5，我們重寫了很多編譯管線，以更適合 Next.js：

*   Next.js 不再依賴 `webpack-hot-middleware` 和 `webpack-dev-middleware`，相反，我們現在直接使用 Webpack 並針對 Next.js 專案優化。這轉化為更簡單的架構和更快的開發編譯。
*   [即時入口](https://nextjs.org/blog/next-8#improved-on-demand-entries)，這是 Next.js 允許它在您發展時訪問的頁面進行編譯的系統，也已重寫，現在更有可靠性，透過利用新 Webpack 行為專為我們的案例設計。
*   [React Fast Refresh 和 Next.js 錯誤覆蓋層](https://nextjs.org/blog/next-9-4#fast-refresh) 現在完全與 Webpack 5 相容
*   磁碟快取將在未來的測試版版本中啟用。

### 向後相容性 (Backwards compatibility)[](#backwards-compatibility)

我們始終致力於確保 Next.js 與之前的版本向後相容。

Webpack 4 將繼續受到完全支援。我們與 Webpack 團隊緊密合作，確保從 Webpack 4 到 5 的遷移盡可能順利。

如果您的 Next.js 專案沒有自訂 Webpack 配置，無需變更專案即可完全利用 Webpack 5。

**重要**：如果您的專案有 [自訂 Webpack 配置](/docs/pages/api-reference/next-config-js/webpack)，可能需要一些變更來過渡到 Webpack 5。建議關注我們的遷移指引，或盡量減少使用 Webpack 擴張，以實現未來的無縫升級。

### 在 macOS 上改進檔案監聽 (Improved file watching on macOS)[](#improved-file-watching-on-macos)

我們最近發現 Webpack 的問題，macOS 上的檔案監聽會在對程式碼進行幾處變更後停止。您必須手動重新啟動專案才能再次看到更新。經過幾處變更，週期會重複。

此外，我們發現這個問題不僅發生在 Next.js 專案中，還發生在所有建立在 Webpack 之上的專案和框架。

經過幾天的除錯，我們將根本原因定位到 Webpack 使用的檔案監聽實現，稱為 chokidar，這是 Node.js 生態系中廣泛使用的檔案監聽實現。

我們發送了 [patch 到 chokidar](https://github.com/paulmillr/chokidar/pull/1018) 來修正這個問題。Patch 釋出後，我們與 [Tobias Koppers](https://github.com/sokra) 合作，在 [新的 Webpack 版本](https://github.com/webpack/webpack/releases/tag/v4.44.0) 中部署此 Patch。

This patched webpack version is automatically used when you upgrade to Next.js 9.5.
