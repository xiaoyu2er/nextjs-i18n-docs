Next.js 15.3 包含用於構建的 Turbopack、新的客戶端監測掛鉤和導航掛鉤等：

*   [**用於構建的 Turbopack（測試版）**](/blog/next-15-3#turbopack-builds-alpha): 通過 8000+ 測試（99%）的 faster 生產構建
*   [**Rspack 的社區支持（實驗性）**](/blog/next-15-3#community-support-for-rspack-experimental): 具有 Webpack 兼容性的替代打包器
*   [**客戶端監測掛鉤**](/blog/next-15-3#client-instrumentation-hook): 早期監測和分析設置
*   [**導航掛鉤**](/blog/next-15-3#navigation-hooks): 使用 `onNavigate` 和 `useLinkStatus` 控制路由
*   [**TypeScript 插件改進**](/blog/next-15-3#typescript-plugin-performance-improvements): 改進對大型代碼庫的支援

Turbopack 構建（測試版）[](#turbopack-builds-alpha)
---------------------------------------------------

`next dev --turbopack` 穩定版發布後，現在 Next.js 15 上 **超過 50% 的開發會話** 正在使用 Turbopack。

此次發布包含 `next build --turbopack` 的 **測試版**，將本地開發的性能改進帶到生產構建中。

通過升級到 15.3 並運行來嘗試 Turbopack 用於生產構建：

終端機

```
next build --turbopack
```

### 功能 [](#functionality)

`next build` 的 [99.3%](https://areweturboyet.com/) 集成測試已經通過。您可以在 [areweturboyet.com](http://areweturboyet.com/) 跟蹤我們達到 100% 的進度。如果您的應用已在 Turbopack 上為 `dev` 運行，那麼它應該可以按原樣用於 `build`。

Turbopack 構建處於測試階段。我們不建議在當前階段針對關鍵業務應用使用它們用於生產。相反，請在預覽或測試環境中嘗試，或在本地運行構建以觀察打包體積和性能的差異。

我們正在積極通過作用域提升、改進分塊和其他優化來填補這些性能差距。

### 構建性能 [](#build-performance)

我們一直在 Vercel 的大型內部 monorepo 和合作夥伴代碼庫中驗證 Turbopack。與我們之前的 Webpack 實現相比，Turbopack 架構的一個優勢是，性能在添加 CPU 核心時會擴展：

*   在 4 核心時：比 Webpack 快 28%
*   在 16 核心時：比 Webpack 快 60%
*   在 30 核心時：比 Webpack 快 83%

隨著我們關於持久緩存的實驗性工作，這些構建時間會進一步減少。我們將在未來的發布中分享更多詳情。

`next.config.ts` 中的 Turbopack 配置（穩定版）[](#turbopack-configuration-in-nextconfigts-stable)
-------------------------------------------------------------------------------------------------------

`next.config.ts` 中的 Turbopack 配置已從 `experimental.turbo` 移到頂層 `turbopack` 鍵：

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  turbopack: {
    rules: {
      '*.svg': {
        loaders: ['@svgr/webpack'],
        as: '*.js',
      },
    },
  },
};
 
export default nextConfig;
```

為了兼容性，`experimental.turbo` 選項將在 Next.js 的下一個主要發布之前繼續得到支援。

完整的 Turbopack 配置選項列表，請參閱 [Turbopack API 引用](https://nextjs.org/docs/app/api-reference/config/next-config-js/turbopack)。

Rspack 的社區支持（實驗性）[](#community-support-for-rspack-experimental)
-----------------------------------------------------------------------------------------

Rspack 團隊為 Next.js 創建了一個社區插件。

這為需要極高 Webpack API 兼容性但尚無法遷移到 Turbopack 的 Next.js 用戶提供了一個選項，以改善他們的本地編譯和構建時間。我們對 Turbopack 的進展有信心，並將繼續為 Webpack 用戶提供漸進的向前路徑。

雖然這不是官方 Next.js 插件，但我們正在與 Rspack 團隊合作。兩個團隊將就 SWC 和 Lightning CSS 等共享基礎設施進行協作，使所有 Next.js 用戶和更廣泛的生態系統受益。

如果您想探索在 Next.js 中使用 Rspack，可以使用 `next-rspack` 適配器。我們正在對集成測試套件運行該適配器。它目前通過了約 96% 的測試。

[查看範例](https://github.com/vercel/next.js/tree/canary/examples/with-rspack) 或在 [Rspack 文檔](https://rspack.dev/guide/tech/next) 中了解更多。

客戶端監測掛鉤 [](#client-instrumentation-hook)
-----------------------------------------------------------

`instrumentation-client.js|ts` 文件允許您添加監控和分析代碼，這些代碼會在應用程序前端代碼開始執行之前運行。

這非常適合盡早建立性能追蹤、錯誤監控或其他客戶端可觀測性工具。

instrumentation-client.js

```
// Set up performance monitoring
performance.mark('app-init');
 
// Initialize analytics
console.log('Analytics initialized');
 
// Set up error tracking
window.addEventListener('error', (event) => {
  // Send to your error tracking service
  reportError(event.error);
});
```

將此文件放置在項目根目錄，類似於 [服務端監測](https://nextjs.org/docs/app/building-your-application/optimizing/instrumentation)。

在 [instrumentation-client 文件文檔](https://nextjs.org/docs/app/api-reference/file-conventions/instrumentation-client) 中了解更多。

導航掛鉤 [](#navigation-hooks)
-------------------------------------

我們正在推出新的導航掛鉤，增強 Next.js 15.3 中的客戶端路由功能，使您能夠更輕鬆地開發本地加載狀態並實現複雜控制（如導航取消）。

### `onNavigate`[](#onnavigate)

此事件處理器是 `Link` 組件的新屬性，並在客戶端導航時執行，讓您精準控制應用程序的路由行為。

與針對所有點擊觸發的 `onClick` 事件不同，`onNavigate` 可用於 [單頁應用 (SPA)](https://nextjs.org/docs/app/building-your-application/upgrading/single-page-applications) 導航，允許您執行代碼或使用 `preventDefault()` 取消導航。

此 API 可用於實施僅在實際頁面過渡時運行的過渡動畫、導航保護或分析追蹤。

通過訪問 [`onNavigate` 文檔](https://nextjs.org/docs/app/api-reference/components/link#onnavigate) 了解更多。

### `useLinkStatus`[](#uselinkstatus)

`useLinkStatus` 客戶端組件掛鉤返回一個 `pending` 布爾值，指示導航是否進行中，為您提供本地加載狀態的控制權。

此 API 基於 React 的 [`useFormStatus`](https://react.dev/reference/react-dom/hooks/useFormStatus)，有助於在頁面過渡期間添加自定義加載指示器，特別是在禁用預取或您鏈接的路由沒有專用加載狀態時。

將使用 `useLinkStatus` 的組件作為 `<Link>` 組件的子代，您可以創建對導航事件做出即時響應的響應式 UI 元素。

通過訪問 [`useLinkStatus` 文檔](https://nextjs.org/docs/app/api-reference/functions/use-link-status) 了解更多。

TypeScript 插件性能改進 [](#typescript-plugin-performance-improvements)
-----------------------------------------------------------------------------------------

Next.js TypeScript 語言服務器插件 (LSP) 現在更快了。

LSP 提供內聯 Intellisense 功能，如服務器/客戶端邊界驗證、組件 prop 提示、配置自動完成以及 React 服務器組件中禁用 API 的錯誤檢測。在非常大的代碼庫中，插件以前可能會導致 TypeScript 語言服務器掛起或崩潰。

我們已對性能進行了重大改進以解決這些問題。在我們內部的測試中，插件響應時間加快了約 60%，且無凍結或崩潰。
