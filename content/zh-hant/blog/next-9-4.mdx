我們今天很高興地宣布推出 Next.js 9.4，特點包括：

*   **[快速刷新](#fast-refresh)**: 快速且可靠的即時編輯體驗，已在 [Facebook 的規模](https://twitter.com/dan_abramov/status/1152689338553131008)得到驗證
*   **[遞增式靜態再生 (beta)](#incremental-static-regeneration-beta)**: 部署後可在毫秒級別重新生成靜態頁面
*   **[CMS 範例](#cms-examples)**: [Contentful](https://github.com/vercel/next.js/tree/canary/examples/cms-contentful)、[DatoCMS](https://github.com/vercel/next.js/tree/canary/examples/cms-datocms)、[Prismic](https://github.com/vercel/next.js/tree/canary/examples/cms-prismic)、[Sanity](https://github.com/vercel/next.js/tree/canary/examples/cms-sanity) 和 [TakeShape](https://github.com/vercel/next.js/tree/canary/examples/cms-takeshape) 的範例，使用我們新的 [下一代靜態站點生成](/blog/next-9-3)
*   **[新環境變量支持](#new-environment-variables-support)**: 內建支援 `.env` 和 `NEXT_PUBLIC_` 前綴，如 CRA 中所見
*   **[改進的內置 Fetch 支持](#improved-built-in-fetch-support)**: 棄用 `node-fetch` 和 `isomorphic-fetch` 導入，改用內建的 `fetch` 多型填充，適用於 Node.js 和所有瀏覽器（構建和執行時）
*   **[集成網頁核心指標報告](#integrated-web-vitals-reporting)**: 捕捉[Lighthouse](https://web.dev/vitals/)評分的指標，但從您的真實流量中
*   **[絕對引入和別名](#absolute-imports-and-aliases)**: 更清晰和更短的引入，避免 `../../../` 意面代碼
*   **[可配置的 Sass 支持](#configurable-sass-support)**: [配置 `includePaths`](https://sass-lang.com/documentation/js-api#includepaths) 和我們 [內建 Sass 支持](/docs/pages/building-your-application/styling#sass-support) 的其他選項
*   **[改進的日誌輸出](#improved-log-output)**: 更易讀的 `console` 輸出，格式一致且更簡潔

快速刷新 [](#fast-refresh)
-----------------------------

快速刷新是一種全新的熱更新體驗，讓您可以即時獲得對 React 組件所做的編輯的反饋。它現已在 Next.js 9.4 或更新版本的所有項目中**默認啟用**。

熱更新已經[存在了很久](https://github.com/gaearon/react-hot-loader)，但[歷史上一直](https://github.com/gaearon/react-hot-loader/blob/master/docs/Troubleshooting.md) [過於脆弱](https://github.com/gaearon/react-hot-loader/issues?q=is%3Aissue)，無法在您工作流程中默認啟用。正因如此，Next.js 之前實現了一種粗粒度的熱更新形式，會重置整個應用程序狀態。

舊的熱更新實作無法適應編譯或運行時錯誤，如果您在編輯 CSS 或 JavaScript 時輸入錯誤，它將執行應用程序的全載入。這是不理想的，並打斷了您的思路。

快速刷新深度整合到 React 內部 ([透過 React Refresh](https://github.com/facebook/react/tree/master/packages/react-refresh))，允許 Next.js 對您的 React 組件樹執行可預測的精確更新。

這意味著 Next.js 只會更新您編輯文件中的程式碼，並且只會重新渲染該組件，**而不丟失組件狀態**。這包括樣式（內聯、CSS-in-JS 或 CSS/Sass Modules）、標記、事件處理程序和效果（透過 `useEffect`）。

包含編譯器和運行時錯誤的編輯會話（可快速恢復），以及保留狀態的編輯。

作為此體驗的一部分，我們完全重新設計了錯誤覆蓋層，使其更具幫助並使您的應用程序能夠適應輸入錯誤或運行時錯誤。這包括但不限於：

*   **精確的錯誤位置**，解析為代碼的**原始行和列**，在編譯之前
*   與上下文相關的 **源代碼片段**，能夠 **點擊在您的編輯器中打開**
*   **語法錯誤修復後的會話恢復**，**不丟失應用程序狀態**
*   **未處理的運行時錯誤自動Dismiss**，當您修復錯誤時

我們感謝 [Dan Abramov](https://twitter.com/dan_abramov) 對實施此功能的無價貢獻和協助。

遞增式靜態再生 (beta) [](#incremental-static-regeneration-beta)
-------------------------------------------------------------------------------

Next.js 在 9.3 中引入了靜態站點生成方法，其目標很明確：我們應該獲得 [靜態的好處](https://rauchg.com/2020/2019-in-review#static-is-the-new-dynamic)（永遠快、永遠在線、[全球分佈](https://rauchg.com/2020/static-hoisting#hoist-to-the-edge)），但同時擁有 Next.js 以之聞名的強大動態數據支持。

要兼得兩者好處，Next.js 支援**遞增式靜態生成**，在您構建網站後更新靜態內容。例如，在 9.3 中我們在 `getStaticPaths` 中引入了 `fallback: true` 選項，允許您在運行時添加**新頁面**。

我們最近[構建了一個範例](https://on-demand-isr.vercel.app/)展示了 How Next.js 可以這樣靜態預渲染無限數量的頁面。

今天，我們還推出了**遞增式靜態再生 (beta)**，這是一種機制，可以通過在流量進入時在背景重新渲染來**更新現有頁面**。受 [stale-while-revalidate](https://tools.ietf.org/html/rfc5861) 啟發，這確保流量無縫服務，始終為靜態，並且新構建的頁面僅在完成生成後才推送。

pages/blog/[slug].js

```
export async function getStaticProps() {
  return {
    props: await getDataFromCMS(),
    // we will attempt to re-generate the page:
    // - when a request comes in
    // - at most once every second
    unstable_revalidate: 1,
  };
}
```

與 SSR 不同，遞增式靜態再生確保您保留靜態的好處：

*   沒有延遲尖峰。頁面始終快速提供。
*   頁面從不離線。如果背景頁面重新生成失敗，舊頁面保持不變。
*   低數據庫和後端負載。頁面最多並行重新計算一次。

兩個增量功能（添加頁面和惰更新它們），以及[預覽模式](/docs/pages/building-your-application/configuring/preview-mode)，都已經完全由 `next start` 和 [Vercel 邊緣平台](https://vercel.com) 支持。

接下來，我們將針對另外兩個額外增量靜態生成功能撰寫補充 RFC：

*   一次性重新生成和無效化多個頁面（例如部落格主頁和特定部落格文章）
*   聆聽事件（例如 CMS Webhooks）進行重新生成，而非用戶流量之前

新環境變量支持 [](#new-environment-variables-support)
-----------------------------------------------------------------------

我們從使用 Next.js 的公司獲得的常見反饋是，不清楚何時將環境變量插入到瀏覽器捆綁包中，何時僅在 Node.js 環境中可用。

今天，我們宣布了兩個完全後向兼容的功能，將有助於簡化此過程。

首先，您可以現在用 `NEXT_PUBLIC_` 前綴環境變量來將環境變量暴露給瀏覽器。當使用該環境變量時，它將被插入到瀏覽器 JavaScript 捆綁包中。

您不再需要添加 `next.config.js` 並添加 `env` 鍵來暴露這些變量。

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
// The environment variable will be exposed to the browser
console.log('My Application Version', process.env.NEXT_PUBLIC_VERSION);
 
export default function HomePage() {
  return <h1>Hello World</h1>;
}
```

第二個變化是 Next.js 現在默認支援 `.env` 加載。允許您輕鬆定義開發和生產環境變量。

您可以在 [環境變量文檔](/docs/pages/building-your-application/configuring/environment-variables) 中閱讀更多關於 `.env` 加載的內容。

這些新功能將通過遵循以下約定簡化使用環境變量：

*   環境變量默認僅在 Node.js 環境中可用
*   以 `NEXT_PUBLIC_` 前綴的環境變量暴露給瀏覽器

改進的內置 Fetch 支持 [](#improved-built-in-fetch-support)
-------------------------------------------------------------------

在 [Next.js 9.1.7](https://nextjs.org/blog/next-9-1-7#new-built-in-polyfills-fetch-url-and-objectassign) 中，我們宣布了 [`fetch()`](https://developer.mozilla.org/docs/Web/API/Fetch_API) API 在瀏覽器中的多型填充。今天這個多型填充已擴展到 Node.js 環境。

在實踐中，您不再需要使用任何類型的服務器端 fetch 多型（例如 `isomorphic-unfetch` 或 `node-fetch`），因為 Next.js 將在所有環境中自動提供 `fetch()`。

例如，當使用在 Next.js 構建時執行的 `getStaticProps` 時：

pages/blog.js

```
export async function getStaticProps() {
  // fetch no longer needs to be imported from isomorphic-unfetch
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  return {
    props: {
      posts,
    },
  };
}
 
function Blog({ posts }) {
  // Render posts...
}
 
export default Blog;
```

集成網頁核心指標報告 [](#integrated-web-vitals-reporting)
-------------------------------------------------------------------

上周 Google Chrome 團隊推出了 [Core Web Vitals](https://web.dev/vitals/)。Core Web Vitals 是在網路上提供偉大 UX 的關鍵質量信號，在其之上構建了 [著名的 Lighthouse 報告](https://twitter.com/rauchg/status/1259701306387656704)。

如果您希望您的網站或 Web 應用程序盡可能快，跟蹤這些指標就非常有用了，這是 Next.js 的核心目標之一。

![網頁核心性能指標](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9-4/core-web-vitals.jpg)

Chrome 團隊發布了一個 [Core Web Vitals Chrome 擴展](https://twitter.com/addyosmani/status/1258416376433131520)，允許您作為開發者獲得視覺反饋頁面表現如何。

在構建生產 Web 應用程序時，您還想知道您的網站對訪問者和（潛在）客戶的表現。您甚至可能想要跟蹤這些指標隨時間推移的改進或回歸，以查看您的更改是否對您的受眾產生預期影響。

為了協助向您的分析服務報告 Core Web Vitals，我們 [與 Google 合作](https://www.techrepublic.com/article/insiders-look-at-googles-web-framework-contributions-to-next-js-and-more/) 引入了新的方法 `reportWebVitals`，可以從 `pages/_app.js` 導出：

pages/_app.js

```
// Will be called once for every metric that has to be reported.
export function reportWebVitals(metric) {
  // These metrics can be sent to any analytics service
  console.log(metric);
}
 
function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
 
export default MyApp;
```

絕對引入和別名 [](#absolute-imports-and-aliases)
-------------------------------------------------------------

要將此方法與您的分析服務結合使用，請參考文檔的 [“發送結果到分析”](/docs/pages/building-your-application/optimizing/analytics#sending-results-to-analytics) 部分。如果您想了解更多關於 Core Web Vitals 的信息，您可以參考 [web.dev/vitals](https://web.dev/vitals/)。

如果您正在處理大型項目，您的 `import` 語句可能會受到 `../../../` 意面代碼的影響：

```
import Button from '../../../../components/button';
```

在這種情況下，我們可能想使用**絕對引入**而不是相對引入。假設 `components` 目錄存在於根目錄下，我們可能希望 `import` 語句看起來像：

```
import Button from 'components/button';
```

我們很興奮地宣布，Next.js 9.4 讓設置 JavaScript 和 TypeScript 項目的絕對引入變得非常簡單。您只需將 `baseUrl` 配置添加到 [`jsconfig.json` (JS 項目)](https://code.visualstudio.com/docs/languages/jsconfig#_jsconfig-options) 或 [`tsconfig.json` (TS 項目)](https://www.typescriptlang.org/docs/handbook/module-resolution.html#base-url)。

jsconfig.json / tsconfig.json

```
{
  "compilerOptions": {
    "baseUrl": "."
  }
}
```

這將允許從 `.` （根目錄）進行絕對引入。它還與 VSCode 和其他編輯器集成，支援程式碼導航和其他編輯器功能。

**注意：** 如果您以前修改了 [Webpack 模組別名配置](https://webpack.js.org/configuration/resolve/#resolvealias) 來啟用絕對引入，現在可以移除該配置。

此外，Next.js 9.4 也支援 `paths` 選項，允許您創建自定義模組別名。例如，以下允許您使用 `@/design-system` 而不是 `components/design-system`：

jsconfig.json / tsconfig.json

```
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/design-system/*": ["components/design-system/*"]
    }
  }
}
```

然後您可以像這樣使用您的別名：

```
// Imports 'components/design-system/button'
import Button from '@/design-system/button';
```

如果您指定 `paths`，您必須指定 `baseUrl`。您可以在 [TypeScript 文檔](https://www.typescriptlang.org/docs/handbook/module-resolution.html#path-mapping) 中了解有關 `paths` 選項的更多信息。

可配置的 Sass 支持 [](#configurable-sass-support)
-------------------------------------------------------

當在 [Next.js 9.3](https://nextjs.org/blog/next-9-3#built-in-sass-support-for-global-stylesheets) 中推出內建 Sass 支持時，我們收到反饋說部分用戶想要配置 sass 編譯器。例如配置 `includePaths`。

這現在可以透過使用 `next.config.js` 中的 `sassOptions` 鍵來實現：

next.config.js

```
const path = require('path');
 
module.exports = {
  sassOptions: {
    includePaths: [path.join(__dirname, 'styles')],
  },
};
```
