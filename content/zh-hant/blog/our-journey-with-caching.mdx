前端效能 (Frontend performance) 可能難以掌握。即使在高度最佳化的應用程式中，目前最常見的問題仍然是客戶端 / 服務器 (client-server) 瀑布流 (waterfalls)。在引入 Next.js App Router 時，我們就知道我們想要解決這個問題。為了做到這一點，我們需要將客戶端 / 服務器 (client-server) REST fetches 移至服務器端，使用 React Server Components 於單個往返中完成。這意味著服務器有時必須是動態的，犧牲了 Jamstack 的出色初始加載效能。我們構建了部分預渲染 (Partial Prerendering) 來解決這個權衡，並兼顧兩者的優點。

然而，在此過程中，開發者體驗 (Developer Experience) 因我們提供的預設緩存和 controls 而受損。`fetch()` 的預設值改為傾向效能而預設緩存，但快速原型設計和高度動態的應用程式卻受損了。我們沒有提供足夠的控制權來管理未使用 `fetch()` 的本機資料庫存取。我們有 `unstable_cache()`，但它不夠優雅。這導致了需要區段層級的配製，例如 `export const dynamic, runtime, fetchCache, dynamicParams, revalidate = ...`, 作為逃生艙 (escape hatch)。

當然，我們會繼續支援該功能以確保向後相容。但此刻，我希望你能忘掉所有這些。我們認為我們有個點子可以做得更好，更簡單。

我們正在研發一種新實驗模式，它建立在兩個觀念上：`<Suspense>`和 `use cache`。

選擇你的冒險路徑 [](#choose-your-adventure)
-----------------------------------------------

你會發現，當你把資料新增到你的組件時，現在會收到錯誤。

app/page.tsx

```
async function Component() {
  return fetch(...) // error
}
 
export default async function Page() {
  return <Component />
}
```

若要使用資料、Cookies、Headers、當前時間或隨機值，你現在有個選擇：你想要資料緩存 (緩存服務器端或客戶端) 還是每次請求都執行？我以 `fetch()` 作為範例，但這適用於任何非同步 Node API，例如資料庫或計時器。

動態 [](#dynamic)
-------------------

如果你仍在迭代或建構高度動態的儀表板，你可以用 `<Suspense>` 邊界包裝組件。`<Suspense>` 選擇進入動態資料擷取和串流 (streaming)。

app/page.tsx

```
async function Component() {
  return fetch(...) // no error
}
 
export default async function Page() {
  return <Suspense fallback="..."><Component /></Suspense>
}
```

你也可以在根佈局中執行，或使用 `loading.tsx`。

這確保你的應用程式外殼保持即時。你可以繼續在 Page 中加入更多資料，知道預設情況下所有資料都是動態的。預設不緩存任何東西。不再隱藏緩存。

靜態 [](#static)
-----------------

如果你在建構靜態內容且不想要使用動態功能，你可以使用新的 `use cache` 指令。

app/page.tsx

```
"use cache"
 
export default async function Page() {
  return fetch(...) // no error
}
```

透過 `use cache`標示 Page，你標明整個區段都應緩存。這表示任何你擷取的資料現在都可以緩存，允許頁面被靜態渲染。靜態內容不使用 `<Suspense>` 邊界。你可以新增更多資料到頁面，它們全都將被緩存。

部分模式 [](#partial)
-------------------

你也可以混合搭配。例如，你可以在根佈局中加入 `use cache` 以確保它被緩存。每個佈局或頁面都可以獨立緩存。

app/layout.tsx

```
"use cache"
 
export default async function Layout({ children }) {
  const response = await fetch(...)
  const data = await response.json()
  return <html>
    <body>
      <div>{data.notice}</div>
      {children}
    </body>
  </html>
}
```

當在特定 Page 中使用動態資料時:

app/page.tsx

```
import { Suspense } from 'react'
async function Component() {
  return fetch(...) // no error
}
 
export default async function Page() {
  return <Suspense fallback="..."><Component /></Suspense>
}
```

緩存函數 [](#cached-functions)
-------------------------------------

使用這種混合方法時，將緩存新增到更接近 API 呼叫的位置可能會更方便。

你可以像 `use server` 一樣，將 `use cache` 新增到任何非同步函數。想像它是一個 Server Action，但你呼叫的是 Cache 而不是 Server。它支援同樣豐富類型的參數和回應值，不僅限於 JSON。緩存鍵自動包含任何參數和封閉變數 (Closures)，所以你不需要手動指定緩存鍵。

app/layout.tsx

```
async function getNotice() {
  "use cache"
  const response = await fetch(...)
  const data = await response.json()
  return data.notice;
}
 
export default async function Layout({ children }) {
  return <html>
    <body>
      <h1>{await getNotice()}</h1>
      {children}
    </body>
  </html>
}
```

由於此佈局中未使用其他資料，它可以保持靜態。這種方法的好處是，如果你意外在佈局中新增動態資料，它會在建置期間觸發錯誤，迫使你再做一次選擇。如果你將 `use cache` 新增到整個佈局，它將被緩存且無錯誤。你選擇哪種方法取決於你的使用案例。

標記緩存 [](#tagging-a-cache)
-----------------------------------

如果你想透過標籤明確清除緩存項目，可以在 `use cache` 函數中使用新的 `cacheTag()` API。

app/utils.ts

```
import { cacheTag } from 'next/cache';
 
async function getNotice() {
  'use cache';
  cacheTag('my-tag');
}
```

然後，如同先前一樣，從 Server Action 呼叫 `revalidateTag('my-tag')`。

由於此 API 可以在資料載入後來呼叫，你現在可以使用資料來標記你的緩存項目。

app/actions.ts

```
import { cacheTag } from 'next/cache';
 
async function getBlogPosts(page) {
  'use cache';
  const posts = await fetchPosts(page);
  for (let post of posts) {
    cacheTag('blog-post-' + post.id);
  }
  return posts;
}
```

## 定義緩存的生命週期 [](#defining-the-lifetime-of-a-cache)
---------------------------------------------------------------------

如果你想要控制特定項目或頁面在緩存中存在的時間長度，可以使用 `cacheLife()` API:

app/page.tsx

```
"use cache"
import { cacheLife } from 'next/cache'
 
export default async function Page() {
  cacheLife("minutes")
  return ...
}
```

預設情況下，它接受以下值:

*   "seconds"
*   "minutes"
*   "hours"
*   "days"
*   "weeks"
*   "max"

選擇一個最符合你的使用案例的粗略範圍。不需要指定精確數字並計算一週中有多少秒 (還是毫秒？)。你也可以指定具體值或配置你自己的命名緩存配置文件。

除了 `revalidate` 功能，此 API 還可以控制客戶端緩存的 `stale` (不新鮮度) 時間以及 `expire` (過期時間)，這決定了 Page 在多長時間沒有流量時應過期。

## 實驗性功能 [](#experimental)
-----------------------------

這仍然是一個非常實驗性的專案。它尚未準備好用於生產環境，仍有缺失的功能和錯誤。特別是，我們知道我們需要改進此新類型錯誤的錯誤堆疊 (Error stacks)。然而，如果你很喜歡冒險，我們很歡迎你的早期反饋。

我們將發布更詳細的升級路徑。除了早期錯誤外，這裡的主要破壞性變更是否決了 `fetch()` 的預設緩存。不過，我們建議在這早期的實驗階段僅在綠地專案 (Greenfield projects) 上進行實驗。如果結果良好，我們希望能透過次要 (Minor) 版本推出選用版本，並在未來的重大 (Major) 版本中成為預設。

若要嘗試，你必須使用 Next.js 的 `canary` 版本:

```
npx create-next-app@canary
```

你也必須在 `next.config.ts` 中啟用實驗性的 `dynamicIO` 標誌:

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  experimental: {
    dynamicIO: true,
  }
};
 
export default nextConfig;
```

在我們的文件中閱讀更多關於 [`use cache`](/docs/canary/app/api-reference/directives/use-cache), [`cacheLife`](/docs/canary/app/api-reference/next-config-js/cacheLife), 和 [`cacheTag`](/docs/canary/app/api-reference/functions/cacheTag) 的資訊。
