我們花了一年的時間改進 Next.js 的代理體驗。在此過程中，我們建構並停用了一個瀏覽器內代理，推出了 MCP 整合，並學到更好的代理支援的真正答案是從代理的觀點出發。

### 代理無法看到瀏覽器 [](#agents-couldnt-see-the-browser)

今年夏天，我們正在專注改進 Next.js 開發者工具時注意到了一種模式。開發者會看到瀏覽器中的錯誤，複製詳細資訊，貼上到 AI 編輯器中，並要求代理修復它。

問題在於代理無法看到瀏覽器。執行時錯誤、客戶端警告和已渲染的元件對它們來說都是不可見的。當使用者說「修復錯誤」時，代理不知道是指哪個錯誤。

我們的第一個回應是更新複製按鈕以擷取結構化錯誤資料。接著我們加入了一項功能，將瀏覽器記錄轉發至終端。這是小修正，但它們指向了一個更大的認識。我們需要讓 Next.js 本身對代理可見。

### 嘗試使用瀏覽器內代理 [](#experimenting-with-an-in-browser-agent)

這引出一個大膽的想法。如果我們直接在 Next.js 內建構一個像智能開發者工具那樣的代理呢？

我們建構了一個名為 Vector 的瀏覽器內聊天代理。類似於 `react-grab` 但與 Next.js 整合，Vector 讓您可以選取頁面上的元素，查看其原始程式碼，並提示更改。它內建了 Next.js 最佳實踐，以幫助代理避免幻覺。

![Vector 的介面顯示錯誤偵測與建議修正。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/agentic-future/vector-both-theme.png)

_副標題：Vector 的介面顯示錯誤偵測與建議修正。_

Vector 很有用，但它與 Cursor 和 Claude Code 等一般程式碼代理重疊了。大多數開發者已經在使用這些工具處理所有專案，而不僅限於 Next.js。UI 選取讓您可以容易指向想要更改的確切內容，但這並不是人們每天都要使用的功能。

我們停用了 Vector，但保留了它對有用的部分（結構化可見性和特定框架知識），並決定將這些組建到 Next.js 本身中。

### MCP 讓 Next.js 狀態對代理可見 [](#mcp-made-nextjs-state-visible-to-agents)

2025 年 10 月 Next.js v16 發布前後，使用者在使用代理除錯時遇到困難。常見的提示是「修復錯誤」，要求代理解決瀏覽器疊層中的問題。但代理會要求頁面 HTML 卻發現沒有問題。

執行時失效、瀏覽器 JavaScript 錯誤和非同步錯誤都存在於瀏覽器中，而非 HTML 中。已渲染的頁面、佈局段落、路由和其他內建狀態對代理來說都是不可見的。

MCP 讓我們能暴露這些資料。第一版展示了內建狀態如錯誤、路由和已渲染段落，但僅暴露資料是不夠的。代理也需要發現正在運行的開發伺服器並與之溝通，這導致了 `next-devtools-mcp`。

MCP 也整合了提示和工具以利升級和緩衝區元件遷移。如果您想了解更多，有一場 [關於 MCP 整合的詳細演講](https://www.youtube.com/watch?v=WT7_TJS0tFE)。

### 答案在於像代理一樣思考 [](#the-answer-is-to-think-like-an-agent)

MCP 證實了 Vector 教導我們的事。代理需要能看到 Next.js 在做的內容，但這只是故事的一部分。更深的教訓是將代理視為 Next.js 的一等公民使用者，並從它們的觀點出發思考。它們需要什麼資訊？何時需要？它們如何消費它？

這種思維帶來了實際的改變。如果代理在開發期間讀取終端輸出，那麼記錄 Server Action 呼叫並轉發瀏覽器錯誤就給了它們需要的線索。如果代理難於理解不在它們訓練數據中的框架概念，那麼嵌入壓縮文件索引 `agents.md` 或提供結構化工作流程（[Next.js skills](https://github.com/vercel-labs/next-skills)）能提供比單獨文件更好的上下文。

這些問題貫穿我們建構的一切。對可見性的需求帶來更好的記錄，對知識的需求帶來 agents.md，對發現的需求帶來 MCP。當您將代理視為一等公民使用者並在其所在之處與之互動時，除錯就成為程式碼、執行時和 AI 之間的緊密回饋迴路。

### 接下來 [](#whats-next)

我們目前正在努力讓這變得更容易採用。您現在可以執行 `npx @next/codemod` 來為您的專案產生最新的文件索引，我們將擴展評估套件以涵蓋更多 Next.js 16 API，以便我們可以衡量實際上對代理幫助什麼。長期來看，我們希望這內建於 `next dev` 中，讓代理自動獲得正確的上下文，無需任何設定。

我們期待收到您關於如何讓 Next.js 與代理配合得更好的回饋和想法。
