Next.js 編譯器：Turbocharged[](#nextjs-compiler-turbocharged)
---------------------------------------------------------------

自 Next.js 13 以來，我們致力於提升 Next.js 的本地開發效能，適用於 Pages 與 App Router。

之前，我們重寫了 `next dev` 和 Next.js 的其他部分以支援這項工作。我們已改變方法，使其更具漸進性。這意味著基於 Rust 的編譯器將很快達到 stable，因為我們已重新聚焦於優先支援所有 Next.js 功能。

`next dev` 的 5,000 個整合測試現在已通過，搭配 [Turbopack](https://nextjs.org/docs/app/api-reference/turbopack)，這是我們底層的 Rust 引擎。這些測試包含 7 年的錯誤修復與重現案例。

在 `vercel.com`（一個大型 Next.js 應用程式）上測試時，我們看到：

*   本地伺服器啟動快達 53.3%
*   搭配 Fast Refresh 的程式碼更新快達 94.7%

此基準測試是你在使用大型應用程式（以及大型模組圖）時應該期許的效能改進實例。由於 `next dev` 目前已有 90% 的測試通過，使用 `next dev --turbo` 時應能看到更快且更可靠的效能。

一旦我們達到 100% 的測試通過，我們將在下一次次要版本中將 Turbopack 移至穩定。我們將繼續支援使用 webpack 進行自訂配置和生態系外掛。

您可以在 [areweturboyet.com](https://areweturboyet.com) 觀看測試通過的百分比。

表單與變異操作 [](#forms-and-mutations)
-------------------------------------------

Next.js 9 引入了 API Routes——這是一種與前端程式碼併建後端端點的方法。

例如，您會在 `api/` 目錄中建立新檔案：

pages/api/submit.ts

```
import type { NextApiRequest, NextApiResponse } from 'next';
 
export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  const data = req.body;
  const id = await createItem(data);
  res.status(200).json({ id });
}
```

然後，在客戶端側，您可以使用 React 和 `onSubmit` 等事件處理器來對您的 API Route 發出 `fetch` 請求。

pages/index.tsx

```
import { FormEvent } from 'react';
 
export default function Page() {
  async function onSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();
 
    const formData = new FormData(event.currentTarget);
    const response = await fetch('/api/submit', {
      method: 'POST',
      body: formData,
    });
 
    // Handle response if necessary
    const data = await response.json();
    // ...
  }
 
  return (
    <form onSubmit={onSubmit}>
      <input type="text" name="name" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

現在隨著 Next.js 14，我們想要簡化開發者撰寫資料變異操作的體驗。此外，我們想改善用戶在網路連線較慢時，或從效能較低裝置提交表單時的使用者體驗。

### Server Actions (穩定) [](#server-actions-stable)

如果不需要手動建立 API Route 會怎樣？取而代之，您可以定義一個在伺服器上安全執行的函數，然後由您的 React 元件直接呼叫。

App Router 建立在 React 的 `canary` 通道上，這對於框架採用新功能而言是 [穩定的](https://react.dev/blog/2023/05/03/react-canaries)。從 v14 開始，Next.js 已升級至最新的 React `canary`，其中包含穩定的 Server Actions。

Pages Router 的前一個範例可以簡化為一個檔案：

app/page.tsx

```
export default function Page() {
  async function create(formData: FormData) {
    'use server';
    const id = await createItem(formData);
  }
 
  return (
    <form action={create}>
      <input type="text" name="name" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

對於以前使用過以伺服器為中心框架的開發者來說，Server Actions 應該會感覺很熟悉。它是建立在表單和 [FormData Web API](https://developer.mozilla.org/en-US/docs/Web/API/FormData) 等網路基礎設施之上的。

透過表單使用 Server Actions 有助於漸進增強，但這不是必要的。您也可以直接作為函數呼叫，無需表單。使用 TypeScript 時，這讓客戶端和伺服器之間具有完整的端到端類型安全性。

變異資料、重新渲染頁面或重新導向可以發生在 **一次網路往返** 中，確保客戶端顯示正確的資料，即使上游提供者較慢。此外，您還可以組合和重用不同的操作，包括在同一個路由中使用許多不同的操作。

### 快取、重新驗證、重新導向等 [](#caching-revalidating-redirecting-and-more)

Server Actions 與整個 App Router 模型深度整合。您可以：

*   使用 `revalidatePath()` 或 `revalidateTag()` 重新驗證快取資料
*   透過 `redirect()` 導向不同路由
*   透過 `cookies()` 設定和讀取 Cookie
*   使用 `useOptimistic()` 處理樂觀 UI 更新
*   使用 `useFormState()` 攔截並顯示來自伺服器的錯誤
*   使用 `useFormStatus()` 在客戶端顯示載入狀態

了解更多關於 [表單與變異操作以及 Server Actions](https://nextjs.org/docs/app/building-your-application/data-fetching/forms-and-mutations) 的內容，或關於 [安全模型](https://nextjs.org/blog/security-nextjs-server-components-actions) 以及 Server Components 和 Server Actions 的最佳實踐。

部分預渲染 (Partial Prerendering) (預覽) [](#partial-prerendering-preview)
---------------------------------------------------------------

我們想分享部分預渲染的預覽版——這是一種為動態內容提供初始靜態回應的快速編譯器優化功能——我們正為 Next.js 開發此功能。

部分預渲染基於十多年來針對伺服器端渲染 (SSR)、靜態網站生成 (SSG) 和增量靜態重新驗證 (ISR) 的研究與開發。

### 目標 [](#motivation)

我們聽到了您的反饋。目前有太多執行環境、配置選項和渲染方法需要考慮。您想要靜態的速度和可靠性，同時也支援完整動態且個人化的回應。

要在全球擁有優異效能 _並_ 提供個人化，不應以複雜性為代價。

我們的挑戰在於創造更好的開發者體驗，在簡化現有模型的同時，不引入開發者需要學習的新 API。雖然伺服端內容的部分快取已經存在，但這些方法仍需要滿足我們旨在實現的開發者體驗和組合性目標。

**部分預渲染不需要學習新的 API。**

### 基於 React Suspense 建構 [](#built-on-react-suspense)

部分預渲染由您的 Suspense 邊界定義。以下是它的工作原理。考慮以下電子商務頁面：

app/page.tsx

```
export default function Page() {
  return (
    <main>
      <header>
        <h1>My Store</h1>
        <Suspense fallback={<CartSkeleton />}>
          <ShoppingCart />
        </Suspense>
      </header>
      <Banner />
      <Suspense fallback={<ProductListSkeleton />}>
        <Recommendations />
      </Suspense>
      <NewProducts />
    </main>
  );
}
```

啟用部分預渲染後，此頁面會根據您的 `<Suspense />` 邊界生成靜態外殼。React Suspense 的 `fallback` 會被預渲染。

外殼中的 Suspense fallback 隨後會由動態元件取代，例如讀取 Cookie 以確定購物車，或根據使用者顯示橫幅。

當提出請求時，靜態 HTML 外殼會立即送出：

```
<main>
  <header>
    <h1>My Store</h1>
    <div class="cart-skeleton">
      <!-- Hole -->
    </div>
  </header>
  <div class="banner" />
  <div class="product-list-skeleton">
    <!-- Hole -->
  </div>
  <section class="new-products" />
</main>
```

由於 `<ShoppingCart />` 從 `cookies` 讀取以查看使用者工作階段，此元件隨即作為靜態外殼相同 HTTP 請求的一部分進行串流傳輸。無需額外的網路往返。

app/cart.tsx

```
import { cookies } from 'next/headers'
 
export default function ShoppingCart() {
  const cookieStore = cookies()
  const session = cookieStore.get('session')
  return ...
}
```

要有最細粒度的靜態外殼，這可能需要新增額外的 Suspense 邊界。然而，如果您今天已經使用 `loading.js`，這就是一個隱式的 Suspense 邊界，因此不需要更改即可生成靜態外殼。

元數據改進 [](#metadata-improvements)
-----------------------------------------------

在您的頁面內容可以從伺服器串流之前，必須先傳送有關視口、色彩方案和主題的元數據至瀏覽器。

確保這些 `meta` 標籤與初始頁面內容一起傳送，有助於流暢的使用者體驗，防止頁面因主題顏色改變而閃爍，或因視口變動導致版面位移。

在 Next.js 14 中，我們解耦了阻塞和非阻塞元數據。只有一小部分元數據選項是阻塞的，我們希望確保非阻塞元數據不會阻止部分預渲染的頁面送出靜態外殼。

以下元數據選項現在已廢棄，將在未來主要版本中從 `metadata` 移除：

*   `viewport`: 設定視口的縮放和其他屬性
*   `colorScheme`: 設定視口的支援模式 (淺色/深色)
*   `themeColor`: 設定視口周圍的 Chrome 應渲染的顏色

從 Next.js 14 開始，有 [`viewport` 和 `generateViewport`](https://nextjs.org/docs/app/api-reference/functions/generate-viewport) 等新選項取代這些選項。所有其他 `metadata` 選項保持不變。

您可以從今天開始採用這些新 API。現有的 `metadata` 選項將繼續運作。
