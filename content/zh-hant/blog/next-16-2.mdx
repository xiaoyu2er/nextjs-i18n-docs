Next.js 16.2 包含效能改進、更好的調試、Agents 適配，以及超過 200 個 Turbopack 修復與改進。

*   [**更快的 URL 載入時間**](#faster-time-to-url-in-development): `next dev` 啟動時間快約 400%
*   [**更快的渲染**](#faster-rendering): 渲染快約 50%
*   [**新的預設錯誤頁面**](#new-default-error-page): 重新設計的內建 500 頁面
*   [**服務器函數日誌記錄**](#server-function-logging): 服務器函數執行的開發終端日誌
*   [**水合差異指標**](#hydration-diff-indicator): 錯誤覆蓋層中清晰顯示服務器/用戶端差異
*   [**`next start` 的 `--inspect`**](#--inspect-for-next-start): 將 Node.js 除錯器掛載到您的生產服務器

我們也為此次發佈的兩個主要領域發布了專用的深入剖析文章：

*   [**Turbopack**](/blog/next-16-2-turbopack): 更快的構建、SRI 支持、`postcss.config.ts`、樹搖優化、服務器快速更新，以及 200 多個 Bug 修復
*   [**AI 改進**](/blog/next-16-2-ai): `create-next-app` 中的 `AGENTS.md`、瀏覽器日誌轉發和 `next-browser` (實驗性)

開發中更快的 URL 載入時間[](#faster-time-to-url-in-development)
-----------------------------------------------------------------------

我們在 `next dev` 期間顯著縮短了 `localhost:3000` 準備就緒所需的時間。在同一台機器和專案上，與默認應用的 Next.js 16.1 相比，啟動速度加快了約 87%。

更快的渲染[](#faster-rendering)
-------------------------------------

我們 [貢獻了一個變更] 到 React，使服務器端組件的數據負載反序列化速度快了高達 350%。之前的實作使用了 `JSON.parse` 的迴歸處理回調，這會導致在 V8 中為 JSON 中的每一個鍵值對跨越 C++/JavaScript 邊界。即使是微不足道的無操作迴歸處理回調也會使 `JSON.parse` 變慢約 4 倍。

新方法使用兩步流程：純 `JSON.parse()` 後跟 JavaScript 中的遞迴遍歷。這消除了邊界跨越開銷，並添加了優化，例如短路處理不需要轉換的純字串。

在實際的 Next.js 應用程式中，這意味著 HTML 渲染速度快了 25% 到 60%，具體取決於 RSC 數據負載的大小。

新的預設錯誤頁面[](#new-default-error-page)
-------------------------------------------------

生產環境中顯示的預設錯誤頁面已重新設計。當您的應用程式遇到錯誤且您未定義自訂的 `global-error.tsx` 或 `error.tsx` 時，Next.js 會呈遞一個內建的後備頁面。此後備頁面已更新為更乾淨、更現代的設計。

`next start` 的 `--inspect`[](#--inspect-for-next-start)
---------------------------------------------------------

Next.js 16.1 引入了 `next dev --inspect` 用於在開發期間掛載 Node.js 除錯器。在 16.2 中，這擴展到 `next start`，允許您在生產服務器上掛載 Node.js 除錯器。

終端機

```
next start --inspect
```

這對於調試問題或分析 CPU 和記憶體佔用情況很有用。

關於如何啟用 Node.js 除錯器的更多資訊，請參閱 [Chrome 說明文件](https://developer.chrome.com/docs/devtools/performance/nodejs)。

`next/link` 的 `transitionTypes` 屬性[](#transitiontypes-prop-for-nextlink)
----------------------------------------------------------------------------

`<Link>` 組件現在接受 [`transitionTypes`](https://react.dev/reference/react/addTransitionType) 屬性——一組指定在導航時應用何種類型視圖轉換的字串陣列。每個類型在導航 Transition 期間傳入 `React.addTransitionType`，允許您根據導航方向或上下文觸發不同的動畫。

```
<Link href="/about" transitionTypes={['slide']}>
  About
</Link>
```

這僅在 App Router 中支援，因為 Pages Router 不使用 React Transition 進行導航。Pages Router 鏈接上的 `transitionTypes` 會被靜音忽略，因此共用鏈接組件可在兩個路由器之間工作。

要了解有關視圖轉換的更多資訊，請參閱 [說明文件](/docs/app/api-reference/config/next-config-js/viewTransition)。

更快的 ImageResponse[](#faster-imageresponse)
-----------------------------------------------

`ImageResponse` 已更新，包含重大改進：

*   **快 2 倍** `ImageResponse` 用於基本圖像，最多快 **20 倍** 用於複雜圖像
*   改進 CSS 和 SVG 支援，包括內聯 CSS 變數、`text-indent`、`text-decoration-skip-ink`、`box-sizing`、`display: contents`、`position: static`，以及 `gap` 的百分比值
*   預設字體從 Noto Sans 更改為 Geist Sans

瞭解更多關於 [ImageResponse](/docs/app/api-reference/functions/image-response)。

適配器[](#adapters)
---------------------

[適配器](/docs/app/api-reference/config/next-config-js/adapterPath) 現在已穩定。這是一種新 API，允許平台自訂構建流程。

這對於部署平台或需要修改 Next.js 配置或處理構建輸出的自訂構建整合很有用。

我們將在下週分享適配器的更詳細概述。

多圖示格式[](#multiple-icon-formats)
-----------------------------------------------

現在自動處理具有相同基本名稱但不同延伸檔名的多個圖示檔案（例如 `icon.png` 和 `icon.svg`）。這對於瀏覽器後備支援很有用。現代瀏覽器可以使用 SVG 圖示，而較舊的瀏覽器則回退到 PNG。兩種格式都渲染為單獨的 `<link>` 標籤。

實驗性功能[](#experimental-features)
-----------------------------------------------

### `unstable_catchError()`[](#unstable_catcherror)

透過 [`error.js`](/docs/app/api-reference/file-conventions/error) 檔案慣例擴展錯誤恢復體驗，[`unstable_catchError()`](/docs/app/api-reference/functions/catchError) 提供組件層級錯誤邊界的更細粒度控制。通過使用 `unstable_catchError()` 建立自訂錯誤邊界，您可以將其放置在組件樹的任何位置。

`unstable_catchError()` 建立的錯誤邊界接收從呼叫點傳入的屬性以及 `ErrorInfo` 值作為第二個參數，這與傳入 `error.js` 檔案慣例的屬性形狀相同。

如同 `error.js`，`unstable_catchError()` 只能從 [用戶端組件](/docs/app/getting-started/server-and-client-components) 呼叫。

app/custom-error-boundary.tsx

```
'use client';
 
import { unstable_catchError, type ErrorInfo } from 'next/error';
 
function CustomErrorBoundary(
  props: { title: string },
  { error, unstable_retry }: ErrorInfo,
) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
 
export default unstable_catchError(CustomErrorBoundary);
```

app/page.tsx

```
import CustomErrorBoundary from './custom-error-boundary';
 
export default function Page() {
  return (
    <CustomErrorBoundary title="Oops! Something went wrong!">
      <Component />
    </CustomErrorBoundary>
  );
}
```

### `error.tsx` 中的 `unstable_retry()`[](#unstable_retry-in-errortsx)

`error.tsx` 組件現在提供新 [`unstable_retry()`](/docs/app/api-reference/file-conventions/error#unstable_retry) 屬性。之前 `reset()` 屬性僅清除錯誤狀態並重新執行子元素，這適用於暫時性渲染錯誤，但在錯誤源自資料抓取或 RSC 階段時無助。

`unstable_retry()` 在 `startTransition()` 內調用 `router.refresh()` 和 `reset()`，提供內建的重新嘗試邏輯，可重新抓取資料並重新執行區段。這預期在大多數錯誤恢復情況下優先於 `reset()`。

app/error.tsx

```
'use client';
 
import type { ErrorInfo } from 'next/error';
 
export default function Error({ error, unstable_retry }: ErrorInfo) {
  return (
    <div>
      <h2>Something went wrong</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
```

### `experimental.prefetchInlining`[](#experimentalprefetchinlining)

Next.js 16 引入了每區段預取，其中客戶端針對路由樹中的每個區段發出個別請求。這提高了快取效率，因為共享區段之間的佈局會抓取一次並重複使用，但這增加了請求數量。

新的 `experimental.prefetchInlining` 選項將路線的所有區段數據打包成單一響應，將預取請求數量減少到每個鏈接一個。代價是共享佈局數據被複製到多個內聯響應中，而不是被快取和重複使用。

next.config.ts

```
const nextConfig = {
  experimental: {
    prefetchInlining: true,
  },
};
 
export default nextConfig;
```

這是邁向基於大小的啟發式的跳板，其中小區段自動內聯，而大型區段保持分開。

### `experimental.cachedNavigations`[](#experimentalcachednavigations)

此標誌獨立控制緩存導航行為，它快取靜態和動態服務器端組件資料，從導航和初始 HTML 載入中獲取，以便重複訪問可以立即服務。需要啟用 `cacheComponents`。

### `experimental.appNewScrollHandler`[](#experimentalappnewscrollhandler)

App Router 中重構 scroll 和焦點管理系統，使用 React Fragment refs。新處理器改進了導航後焦點的管理方式。它不再聚焦於區段內部深處的某個可聚焦的第一個後代（可能跳過內容），而是模糊活躍元素，模擬瀏覽器導航行為。

next.config.ts

```
const nextConfig = {
  experimental: {
    appNewScrollHandler: true,
  },
};
 
export default nextConfig;
```
