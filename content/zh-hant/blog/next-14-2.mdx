Next.js 14.2 包含開發、生產和緩存改進。

*   [**Turbopack 開發用（發布候選人）:**](#turbopack-for-development-release-candidate) `next dev --turbo` 的測試通過率為 99.8%。
*   [**構建與生產改進:**](#build-and-production-improvements) 減少構建內存使用並優化 CSS。
*   [**緩存改進:**](#caching-improvements) 使用 `staleTimes` 配置無效化週期。
*   [**錯誤 DX (開發者體驗) 改進:**](#errors-dx-improvements) 更好的水合不匹配錯誤和設計更新。

立即升級或開始使用：

終端機

```
npx create-next-app@latest
```

Turbopack 開發用（發布候選人）[#turbopack-for-development-release-candidate]
---------------------------------------------------------------------------------------------

過去幾個月來，我們一直致力於通過 Turbopack 改進本地開發性能。在 14.2 版本中，Turbopack 發布候選人版（Release Candidate）現在可用於本地開發：

*   **99.8%** 的 [整合測試](https://areweturboyet.com/) 現在通過。
*   我們已驗證 Next.js 應用中使用的前 300 個 npm 包可以使用 Turbopack 進行編譯。
*   所有 [Next.js 示例](https://github.com/vercel/next.js/tree/canary/examples) 與 Turbopack 兼容。
*   我們已集成 Lightning CSS，這是一個用 Rust 編寫的快速 CSS 打包器及混淆器。

我們已在 Vercel 的應用程序上廣泛試用 Turbopack。例如，對於 `vercel.com`（一個大型 Next.js 應用），我們觀察到：

*   本地服務器啟動速度最高可快 **76.7%**。
*   使用 Fast Refresh 代碼更新速度最高可快 **96.3%**。
*   首次路由編譯速度（無緩存）最高可快 **45.8%**（Turbopack 尚未有磁盤緩存）。

Turbopack 繼續保持可選，您可以使用以下方式嘗試：

終端機

```
next dev --turbo
```

我們將專注於改進內存使用、實現持久化緩存，以及 `next build --turbo`。

*   **內存使用** - 我們構建了用於調查內存使用的底層工具。您現在可以 [生成軌跡](/docs/architecture/turbopack#generating-trace-files)，其中包含性能指標和廣泛的內存使用信息。這些軌跡使我們能夠調查性能和內存使用情況，而無需訪問應用程序的源代碼。
*   **持久化緩存** - 我們也在探索最好的架構選項，預計會在下個版本中分享更多細節。
*   **`next build`** - 雖然 Turbopack 尚未可用於構建，但 **74.7%** 的測試目前通過。您可以跟隨進度 at [areweturboyet.com/build](https://areweturboyet.com/build)。

若要查看 Turbopack 中 [支持](/docs/architecture/turbopack#supported-features) 和 [不支持的功能](/docs/architecture/turbopack#unsupported-features)，請參閱我們的 [文檔](/docs/architecture/turbopack)。

構建與生產改進 [#build-and-production-improvements]
------------------------------------------------------------------------

除了 Turbopack 的打包改進外，我們還致力於改進所有 Next.js 應用的整體構建和生產性能（包括 Pages 和 App Router）。

### Tree-shaking（樹搖）[#tree-shaking]

我們發現了伺服器組件和客戶端組件邊界的優化，允許對未使用的導出進行樹搖。例如，從包含 `"use client"` 的文件中導入單個 `Icon` 組件時，不再包含該組件包中的所有其他圖標。這可以大幅減少生產 JavaScript 束包大小。

在 `react-aria-components` 這樣的流行庫上測試此優化，最終束包大小減少了 **\-51.3%**。

> **注意：** 此優化目前與桶文件（barrel files）不兼容。同時，您可以使用 [`optimizePackageImports`](/docs/app/api-reference/next-config-js/optimizePackageImports) 配置選項：
> 
> next.config.ts
> 
> ```
> module.exports = {
>   experimental: {
>     optimizePackageImports: ['package-name'],
>   },
> };
> ```

### 構建內存使用 [#build-memory-usage]

對於極大規模的 Next.js 應用，我們發現生產構建期間會發生記憶體耗尽錯誤（OOM）。在調查用戶報告和復現問題後，我們發現根本原因是打包過多和混淆（Next.js 創建了較少、較大的 JavaScript 文件，並存在重複）。我們重構了打包邏輯並針對這些情況優化了編譯器。

我們早期的測試顯示，在一個最小化的 Next.js 應用中，內存使用和緩存文件大小平均從 2.2GB 降至 190MB 以下。

為了更方便地調試內存性能，我們為 `next build` 引入了 `--experimental-debug-memory-usage` 標誌。在我們的 [文檔](/docs/app/building-your-application/optimizing/memory-usage) 中了解更多信息。

### CSS [#css]

我們更新了 Next.js 構建中優化 CSS 的方式，通過將 CSS 分塊來避免在頁面導航時產生衝突樣式。

CSS 塊的順序和合併現在由導入順序定義。例如，`base-button.module.css` 將排在 `page.module.css` 之前：

base-button.tsx

```
import styles from './base-button.module.css';
 
export function BaseButton() {
  return <button className={styles.primary} />;
}
```

page.tsx

```
import { BaseButton } from './base-button';
import styles from './page.module.css';
 
export function Page() {
  return <BaseButton className={styles.primary} />;
}
```

為了保持正確的 CSS 順序，我們建議：

*   優先使用 [CSS 模組](/docs/app/building-your-application/styling/css-modules) 而非 [全局樣式](/docs/app/building-your-application/styling/css-modules#global-styles)。
*   僅在單一 JS/TS 文件中導入 CSS 模組。
*   如果使用全局類名，也在同一個 JS/TS 中導入全局樣式。

我們預期此更改不會對大多數應用程序產生負面影響。然而，如果您在升級時看到任何意外樣式，請按照 [文檔](/docs/app/building-your-application/styling/css#ordering-and-merging) 中的建議審查您的 CSS 導入順序。

緩存改進 [#caching-improvements]
---------------------------------------------

緩存是構建快速可靠網絡應用的關鍵部分。執行變異操作時，用戶和開發人員都希望緩存能夠更新以反映最新更改。我們一直在探索如何改進 App Router 中的 Next.js 緩存體驗。

### `staleTimes`（實驗性）[#staletimes-experimental]

[客戶端路由緩存](/docs/app/building-your-application/caching#data-cache-and-client-side-router-cache) 是一個緩存層，旨在通過緩存客戶端上訪問和預處理的路由來提供快速的導航體驗。

基於社區反饋，我們添加了實驗性 `staleTimes` 選項，以允許配置 [客戶端路由緩存](/docs/app/building-your-application/caching#router-cache) 的無效化週期。

預設情況下，預處理路由（使用 `<Link>` 組件且不設置 `prefetch` 屬性）將緩存 30 秒，如果 `prefetch` 屬性設置為 `true`，則為 5 分鐘。您可以在 `next.config.js` 中定義自定義 [重新驗證時間](/docs/app/building-your-application/caching#duration-3) 來覆蓋這些默認值：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
};
 
module.exports = nextConfig;
```

`staleTimes` 旨在改善那些想要更多緩存邏輯控制權的用戶的當前排驗，但它並非完整的解決方案。在接下來的版本中，我們將專注於改進整體緩存語義並提供更靈活的解決方案。

在我們的 [文檔](/docs/app/api-reference/next-config-js/staleTimes) 中了解有關 `staleTimes` 的更多信息。

### 平行和攔截路由 [#parallel-and-intercepting-routes]

我們繼續迭代 [平行](/docs/app/building-your-application/routing/parallel-routes) 和 [攔截](/docs/app/building-your-application/routing/intercepting-routes) 路由，現在改進與客戶端路由緩存的集成。

*   調用帶有 [`revalidatePath`](/docs/app/api-reference/functions/revalidatePath) 或 [`revalidateTag`](/docs/app/api-reference/functions/revalidateTag) 的伺服器操作的平行和攔截路由將重新驗證緩存並刷新可見槽位，同時保持用戶當前視圖。
*   同樣，現在調用 [`router.refresh`](/docs/app/building-your-application/caching#routerrefresh) 會正確刷新可見槽位，保持當前視圖。

錯誤開發者體驗（DX）改進 [#errors-dx-improvements]
-------------------------------------------------

在 14.1 版本中，我們開始致力於 [改進錯誤信息和堆棧追蹤的可讀性](/blog/next-14-1#improved-error-messages-and-fast-refresh)，當運行 `next dev` 時。這些工作已繼續至 14.2，現在包括更好的錯誤消息、App Router 和 Pages Router 的重疊設計改進、明暗模式支持，以及更清晰的 `dev` 和 `build` 日誌。

例如，React 水合錯誤是我們社區常見的困惑來源。雖然我們進行了改進以幫助用戶定位水合不匹配的來源（見下文），但我們正在與 React 團隊合作改進底層錯誤消息並顯示發生錯誤的文件名。

React 19 [#react-19]
---------------------

2 月份，React 團隊宣布了即將發佈的 [React 19](https://react.dev/blog/2024/02/15/react-labs-what-we-have-been-working-on-february-2024#the-next-major-version-of-react)。為了準備 React 19，我們正致力於將最新的功能和改進集成到 Next.js 中，並計劃發布一個主要版本來協調這些更改。

像 Actions 及其相關的 Hooks 這類新功能，之前在 Next.js 中的 [React canary channel](https://react.dev/blog/2023/05/03/react-canaries)，現在將為所有 React 應用程序（包括僅客戶端應用程序）可用。我們很興奮看到這些功能在 React 生態系統中的更廣泛採用。
