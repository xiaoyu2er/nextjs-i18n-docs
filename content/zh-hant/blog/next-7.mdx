開發者體驗 (DX) 改進[](#dx-improvements)
-----------------------------------

Next.js 的主要目標之一是提供最佳的生產級性能，同時確保**最佳的開發者體驗**。此次發布對構建和除錯管道帶來了許多重大改進

### 編譯速度[](#compilation-speed)

多虧了 webpack 4、Babel 7 以及代碼庫的許多改進和優化，Next.js 現在在開發期間的啟動速度提高了 57%。

多虧我們新的增量編譯緩存，您對代碼所做的更改現在可以**構建快 40%**。

這是我們收集的一些示例數據：

|  | 6.0 | 7.0 | 差異 |
| --- | --- | --- | --- |
| 啟動時間 (基本應用) | 1947ms | 835ms | **57%** _更快_ |
| 代碼變更 (基本應用) | 304ms | 178ms | **42%** _更快_ |

作為額外福利，在開發和構建時，您將能夠通過 [webpackbar](https://github.com/nuxt/webpackbar) 看到更好的實時反饋：

Webpack 4[](#webpack-4)
-----------------------

自首次發布以來，Next.js 一直由 webpack 為您的代碼打包提供支持，並重用豐富的外掛和擴展生態系統。我們興奮地宣布，Next.js 現在由最新的 webpack 4 提供支持，其中包含眾多改進和修復。

其中包括：

*   對 `.mjs` 源文件的支持
*   代碼分割改進
*   更好的樹搖晃 (tree-shaking/移除未使用代碼) 支持

另一項新功能是在 WebAssembly 支持，Next.js 甚至可以進行 WebAssembly 的伺服器端渲染 (SSR)，這是一個 [示例](https://github.com/vercel/next.js/tree/canary/examples/with-webassembly)。

**注意：** 此次升級**完全向後兼容**。但是，如果您使用自定義的 webpack 載入器或外掛，通過 [next.config.js](/docs#customizing-webpack-config)，您可能需要升級它們。

### CSS 導入[](#css-imports)

通過 webpack 4，引入了從構件中提取 CSS 的新方法，名為 [mini-extract-css-plugin](https://github.com/webpack-contrib/mini-css-extract-plugin)。

[`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`@zeit/next-less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less)、[`@zeit/next-sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) 和 [`@zeit/next-stylus`](https://github.com/vercel/next-plugins/tree/master/packages/next-stylus) 現在由 `mini-extract-css-plugin` 提供支持。

這些 Next.js 外掛的新版本**解決了 20 個現有與 CSS 導入相關的問題**；例如，現在支持在動態 `import()` 中導入 CSS：

components/my-dynamic-component.js

```
import './my-dynamic-component.css';
 
export default function MyDynamicComponent() {
  return <h1>My dynamic component</h1>;
}
```

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import dynamic from 'next/dynamic'
 
const MyDynamicComponent = dynamic(import('../components/my-dynamic-component'))
 
export default function Index() {
  return () {
    <div>
      <MyDynamicComponent/>
    </div>
  }
}
```

主要的改進是，您不再需要將以下內容添加到 `pages/_document.js` 中：

```
<link rel="stylesheet" href="/_next/static/style.css" />
```

相反，Next.js 會自動注入 CSS 文件。在生產環境中，Next.js 還會自動在 CSS URL 中添加內容雜湊 (content hash)，因此如有變更，可確保最終用戶永遠不會收到過舊的文件版本，並具有引入不可變永久緩存的 ability。

簡而言之，要在您的 Next.js 項目中支持導入 `.css` 文件，您只需要在 `next.config.js` 中**註冊 withCSS 外掛**：

```
const withCSS = require('@zeit/next-css');
module.exports = withCSS({
  /* my next config */
});
```

### 標準化的動態導入[](#standardized-dynamic-imports)

Next.js 自第 3 版起就支持通過 `next/dynamic` 進行動態導入。

作為這一技術的早期採用者，我們不得不為處理 `import()` 寫出自己的解決方案。

因此，Next.js 開始偏離 webpack 稍後引入的支持，並且缺少一些功能。

因此，Next.js 不支持自那以後 webpack 引入的一些 `import()` 功能。

例如，無法手動命名和打包某些文件：

```
import(/* webpackChunkName: 'my-chunk' */ '../lib/my-library');
```

另一個示例是未包裝在 `next/dynamic` 模組中使用 `import()`。

從 Next.js 7 開始，我們不再修改默認的 `import()` 行為。這意味著您**開箱即用獲得完整的 import() 支持**。

此更改同樣完全向後兼容。使用動態組件仍然簡單如：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import dynamic from 'next/dynamic';
 
const MyComponent = dynamic(import('../components/my-component'));
 
export default function Index() {
  return (
    <div>
      <MyComponent />
    </div>
  );
}
```

此示例所做的內容是為 `my-component` 創建新的 JavaScript 文件，並且僅在渲染 `<MyComponent />` 時加載它。

最重要的是，如果沒有渲染，則 `<script>` 標籤**不會包含在初始 HTML 文檔負載中**。

為了進一步簡化代碼庫並利用優秀的 React 生態系統，在 Next.js 7 中，`next/dynamic` 被重寫為在後台使用 [`react-loadable`](https://github.com/jamiebuilds/react-loadable)（帶有少量修改）。這還為動態組件引入了兩個非常棒的新功能：

*   使用 `next/dynamic` 上的 `timeout` 選項設置超時
*   使用 `next/dynamic` 上的 `delay` 選項設置加載組件延遲。此延遲允許您的 `loading` 組件在加載狀態渲染前等待 x 秒，例如，如果導入非常快。

[Babel 7](#babel-7)
-------------------

Next.js 6 在 Babel 7 仍然處於 beta 階段時引入了 Babel 7。從此以來，Babel 7 的穩定版本已發布，而 Next.js 7 現在使用此版本。

對於完整的更改列表，您可以參考 Babel 的 [發行說明](http://babeljs.io/blog/2018/08/27/7.0.0)。

其中一些主要功能包括：

*   Typescript 支持，對於 Next.js 您可以使用 [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript)
*   Fragment 語法 `<>` 支持
*   `babel.config.js` 支持
*   `overrides` 屬性，僅將預設/外掛應用於子集文件或目錄

如果您在 Next.js 項目中沒有自定義的 Babel 配置，則沒有衝突的更改。

如果您確實有自定義的 Babel 配置，則必須將相應的自定義外掛/预设升級到其最新版本。

如果您從低於 Next.js 6 的版本升級，您可以運行優秀的 [`babel-upgrade`](http://babeljs.io/blog/2018/08/27/7.0.0#babel-upgrade) 工具。

除了升級到 Babel 7，Next.js Babel 预设 ([`next/babel`](https://github.com/vercel/next.js#customizing-babel-config)) 現在默認在 `NODE_ENV` 設置為 `test` 時將 `modules` 選項設置為 `commonjs`。

此配置選項通常是 Next.js 項目中創建自定義 `.babelrc` 的唯一原因：

.babelrc

```
{
  "env": {
    "development": {
      "presets": ["next/babel"]
    },
    "production": {
      "presets": ["next/babel"]
    },
    "test": {
      "presets": [["next/babel", { "preset-env": { "modules": "commonjs" } }]]
    }
  }
}
```

使用 Next.js 7 這變成：

.babelrc

```
{
  "presets": ["next/babel"]
}
```

此時，可以刪除 `.babelrc`，因為當沒有 Babel 配置時，Next.js 將自動使用 `next/babel`。

更小的初始 HTML 負載[](#smaller-initial-html-payload)
-------------------------------------------------------------

由於 Next.js 預先呈現 HTML，它將頁面包裝在默認結構中，包括 `<html>`、`<head>`、`<body>` 和渲染頁面所需的 JavaScript 文件。

此初始負載之前約為 1.62kB。使用 Next.js 7，我們優化了初始 HTML 負載，現在為 1.5kB，減少了 7.4%，使頁面更緊湊。

|  | 6.0 | 7.0 | 差異 |
| --- | --- | --- | --- |
| 文檔大小 (伺服器端呈現) | 1.62kb | 1.50kb | **7.4%** _更小_ |

我們減少大小的主要方式包括：

*   移除 `__next-error` div
*   將內聯腳本進行最小化，在未來的發布中它們將被完全移除
*   當未使用已編譯的 `__NEXT_DATA__` 屬性時將其移除（例如 `nextExport` 和 `assetPrefix` 屬性）。

靜態 CDN 支援[](#static-cdn-support)
-----------------------------------------

在 Next.js 5 中，我們引入了 `assetPrefix` 支持，這是一種讓 Next.js 自動從某個位置加載資源的方法（通常是 CDN）。如果您的 CDN 支持代理，此選項效果很好，您請求 URL 如：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
```

通常，CDN 檢查其緩存中是否有該文件，否則直接向源請求。

代理資源正是 [邊緣網絡 (Edge Network)](https://vercel.com/edge-network) 的工作方式。

然而，某些解決方案要求將目錄直接預先上傳到 CDN。這樣做的問題是，Next.js 的 URL 結構與 `.next` 文件夾內的文件結構不匹配。例如我們之前的示例

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/page/index.js
```

使用 Next.js 7，我們更改了 `.next` 的文件夾結構以匹配 URL 結構：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/static/<buildid>/pages/index.js
```

雖然我們建議使用代理類型的 CDN，但此新結構允許不同類型 CDN 的使用者將 `.next` 文件夾上傳到他們的 CDN。

Styled JSX v3[](#styled-jsx-v3)
-------------------------------

我們很興奮地介紹 styled-jsx 3，Next.js 默認包含的 CSS-in-JS 解決方案現在已準備好用於 [React Suspense](https://www.youtube.com/watch?v=z-6JC0_cOns)。

通常不清楚的一件事是，如果組件不是當前組件範圍的一部分（例如，如果包含了一個子組件，且該子組件僅在父組件中使用時才需要特定樣式），如何對子組件進行樣式設置：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
const ChildComponent = () => (
  <div>
    <p>some text</p>
  </div>
);
 
export default function Index() {
  return (
    <div>
      <ChildComponent />
      <style jsx>{`
        p {
          color: black;
        }
      `}</style>
    </div>
  );
}
```

上述代碼試圖選擇 `p` 標籤不起作用，因為 styled-jsx 樣式作用於當前組件，它們不會泄漏到子組件中。解決這一問題的一種方法是使用 `:global` 方法，去掉 `p` 標籤的前綴。然而，這又引入了新問題，即樣式會在頁面間泄漏。

在 styled-jsx 3 中，通過引入新 API `css.resolve` 解決了此問題，該 API 會為給定的 styled-jsx 字符串生成 `className` 和 `<style>` 標籤（`styles` 屬性）：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import css from 'styled-jsx/css';
 
const ChildComponent = ({ className }) => (
  <div>
    <p className={className}>some text</p>
  </div>
);
 
const { className, styles } = css.resolve`
  p {
    color: black;
  }
`;
 
export default function Index() {
  return (
    <div>
      <ChildComponent className={className} />
      {styles}
    </div>
  );
}
```

此新 API 允許透明地向子組件傳遞自定義樣式。

由於這是 styled-jsx 的重大發布，如果使用 `styles-jsx/css`，則有一個會改善捆包大小的衝突更改。在 styled-jsx 2 中，即使僅使用了 "作用域" (scoped) 版本，我們也會為外部樣式生成 "作用域" 和 "全局" (global) 版本，我們仍會包含這些外部樣式的 "全局" 版本。

使用 styled-jsx 3，全局樣式必須標記為 `css.global` 而不是 `css`，以便 styled-jsx 可以優化捆包大小。

對於所有更改，請參考 [發行說明](https://github.com/vercel/styled-jsx/releases/tag/v3.0.0)。

### App 與 Pages 間的 React Context 與 SSR 支援[](#react-context-with-ssr-between-app-and-pages)

從 Next.js 7 開始，我們現在支持 `pages/_app.js` 與頁面組件之間的新的 React Context API。

在此之前，在服務器端在頁面之間使用 React Context 是不可能的。原因是 webpack 使用內部模組緩存而不是使用 require.cache，我們已編寫了一個自定義的 webpack 外掛，將此行為改為在頁面之間共享模組實例。

通過這樣做，我們不僅允許使用新的 React Context，還減少了 Next.js 在頁面之間共享代碼時的記憶體佔用。

|  | 6.0 | 7.0 | 差異 |
| --- | --- | --- | --- |
| 記憶體使用 | 57.5MB | 48.1MB | **\-16%** _記憶體_ |
