This RFC (Request for Comment) outlines the biggest update to Next.js since it was introduced in 2016:

*   **Nested Layouts（嵌套佈局）**：使用嵌套路由構建複雜應用。
*   **專為服務器組件設計**：針對子樹導航進行了優化。
*   **改進數據獲取**：在佈局中獲取數據，同時避免瀑布流。
*   **使用 React 18 功能**：串流、轉換和 Suspense。
*   **客戶端與服務器路由**：以服務器為中心的路由，具有 _類 SPA_ 行為。
*   **100% 增量採用**：沒有破壞性更改，因此您可以逐步採用。
*   **進階路由模式**：平行路由、攔截路由等。

新的 Next.js 路由器將建立在 [最近發布的 React 18](https://reactjs.org/blog/2022/03/29/react-v18.html) 功能之上。我們計劃引入預設值和約定，讓您能夠輕鬆採用這些新功能並充分利用其釋放的優勢。

> 本 RFC 的工作目前正在進行中，我們將在新的功能可用時進行公告。
> 如有反饋，請加入 [GitHub Discussions](https://github.com/vercel/next.js/discussions/37136) 上的對話。

動機 [](#motivation)
-------------------------

我們一直從 GitHub、Discord、Reddit 和我們的開發者調查中收集關於 Next.js 當前路由限制的社區反饋。我們發現：

*   **創建佈局的開發者體驗可以改進。** 應輕鬆建立可以嵌套、在路由間共享，並在導航時保留其狀態的佈局。
*   **許多 Next.js 應用程式是儀表板或控制台**；許多 Next.js 應用程式是儀表板或控制台，這會從更進階的路由解決方案中受益。

While the current routing system has worked well since the beginning of Next.js, we want to make it easier for developers to build more performant and feature-rich web applications.

作為框架維護者，我們也希望構建一個與後向兼容且與 React 未來一致的路由系統。

> **注意：** 一些路由約定受到 Meta 基於 Relay 的路由器的啟發，其中服務器組件的一些功能最初是在那裡開發的，以及像 React Router 和 Ember.js 這樣的客戶端路由器。`layout.js` 文件約定是受到 SvelteKit 工作的啟發。我們也感謝 [Cassidy](https://twitter.com/cassidoo) 開啟了 [早期關於佈局的 RFC](https://github.com/vercel/next.js/discussions/26389)。

術語 [](#terminology)
---------------------------

本 RFC 引入了新的路由約定和語法。此術語基於 React 和標準 Web 平台術語。在整本 RFC 中，您將看到這些術語連結回其下方的定義。

*   **樹 (Tree)**：用於可視化層次結構的約定。例如，具有父組件和子組件的組件樹、文件夾結構等。
*   **子樹 (Subtree)**：樹的一部分，從根節點（第一個）開始到葉節點（最後一個）。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/terminology.png)

*   **URL 路徑 (URL Path)**：URL 中域名之後的部分。
*   **URL 片段 (URL Segment)**：由斜線分隔的 URL 路徑部分的片段。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/url-anatomy.png)

當前路由如何運作 [](#how-routing-currently-works)
-----------------------------------------------------------

今天，Next.js 使用文件系統將 [頁面](/docs/pages/building-your-application/routing/pages-and-layouts) 目錄中的各個文件夾和檔案映射為可通過 URL 訪問的路由。每個 **頁面 (page)** 檔案匯出一個 React 組件，並根據其檔案名稱具有關聯的 **路由 (route)**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/routing-today.png)

*   **動態路由**：Next.js 支援 [動態路由](/docs/pages/building-your-application/routing/dynamic-routes)（包括 catch all 變體），使用 `[param].js`、`[...param].js` 和 `[[...param]].js` 約定。
*   **佈局**：Next.js 提供簡單的 [基於組件的](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern) 佈局支援、使用組件 [屬性模式](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern#per-page-layouts) 的每頁面佈局，以及使用 [自定義應用](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern#single-shared-layout-with-custom-app) 的全局佈局。
*   **數據獲取**：Next.js 提供數據獲取方法 ([`getStaticProps`](/docs/pages/building-your-application/data-fetching/get-static-props)、[`getServerSideProps`](/docs/pages/building-your-application/data-fetching/get-server-side-props))，可在頁面（路由）層級使用。這些方法用於決定頁面是否應靜態生成 ([`getStaticProps`](/docs/pages/building-your-application/data-fetching/get-static-props)) 或服務端渲染 ([`getServerSideProps`](/docs/pages/building-your-application/data-fetching/get-server-side-props))。此外，您可以使用 [增量靜態重新生成 (ISR)](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration) 在構建網站後創建或更新靜態頁面。
*   **渲染**：Next.js 提供三種渲染選項：[靜態生成](https://nextjs.org/learn/foundations/how-nextjs-works/rendering)、[服務端渲染](https://nextjs.org/learn/foundations/how-nextjs-works/rendering) 和 **客戶端渲染 (CSR)**。預設情況下，頁面是靜態生成的，除非它們有阻塞的數據獲取需求 (`getServerSideProps`)。

介紹「app」目錄 [](#introducing-the-app-directory)
-----------------------------------------------------------------

為了確保這些新的改進可以增量採用並避免破壞性更改，我們提出一個名為 `app` 的新目錄。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/app-folder.png)

`app` 目錄將與 `pages` 目錄並行工作。您可以將應用程式的部分遷移到新的 `app` 目錄，以利用新功能。為了後向兼容，`pages` 目錄的行為將保持不變，繼續支援。

定義路由 [](#defining-routes)
-----------------------------------

您可以使用 `app` 中的 **文件夾** 層級結構來定義路由。**路由** 是嵌套文件夾的單一路徑，遵循從 **根文件夾** 到最終 **葉文件夾** 的層級結構。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/routes.png)

例如，您可以在 `app` 目錄中嵌套兩個新文件夾來添加新的 `/dashboard/settings` 路由。

> **注意：**
>
> * 使用此系統時，您將使用文件夾定義路由，使用檔案定義 UI（使用新的檔案約定，如 `layout.js`、`page.js`，以及在 RFC 的第二部分 `loading.js`）。
> * 這允許您在 `app` 目錄中放置您自己的專案檔案（UI 組件、測試檔案、故事等）。目前這僅能透過 [pageExtensions 配置](/docs/pages/api-reference/next-config-js/pageExtensions#including-non-page-files-in-the-pages-directory) 完成。

### 路由片段 [](#route-segments)

每個 [子樹](#terminology) 中的文件夾代表一個 **路由片段**。每個路由片段都映射到 **[URL 路徑](#terminology)** 中的相應 **片段**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/route-segments.png)

例如，`/dashboard/settings` 路由由 3 個片段組成：

* `/` 根片段
* `dashboard` 片段
* `settings` 片段

> **注意：** 名稱 **路由片段** 是為了與現有的 [URL 路徑](#terminology) 術語相匹配而選擇的。

佈局 [](#layouts)
-------------------

**新檔案約定：** `layout.js`

到目前為止，我們一直使用文件夾來定義應用程式的路由。但空文件夾本身沒有做任何事情。讓我們討論如何如何使用新的檔案約定來定義這些路由將呈現的 UI。

**佈局** 是 [子樹](#terminology) 中路由片段之間共享的 UI。佈局不影響 [URL 路徑](#terminology)，當用戶在兄弟片段之間導航時不會重新渲染（React 狀態已保留）。

佈局可以通過從 `layout.js` 檔案匯出預設的 React 組件來定義。該組件應接受一個 `children` 屬性，該屬性將填充佈局包裝的片段。

有 2 種類型的佈局：

* **根佈局 (Root layout)：** 適用於所有路由
* **一般佈局 (Regular layout)：** 適用於特定路由

您可以將兩個或更多佈局嵌套在一起以形成 **嵌套佈局**。

### 根佈局 (Root Layout) [](#root-layout)

您可以在 `app` 文件夾中添加 `layout.js` 檔案來建立根佈局，該佈局將應用於您的所有應用程式路由。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/root-layout.png)

> **注意：**
>
> * 根佈局取代了 [自定義應用 (`_app.js`)](/docs/pages/building-your-application/routing/custom-app) 和 [自定義文檔 (`_document.js`)](/docs/pages/building-your-application/routing/custom-document) 的需求，因為它適用於所有路由。
> * 您將能夠使用根佈局自定義初始文檔外殼（例如 `<html>` 和 `<body>` 標籤）。
> * 您將能夠在根佈局中（和其他佈局中）獲取數據。

### 一般佈局 [](#regular-layouts)

您也可以通過在特定文件夾中添加 `layout.js` 檔案，建立僅應用於應用程式一部分的佈局。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/regular-layouts.png)

例如，您可以在 `dashboard` 文件夾中建立 `layout.js` 檔案，該檔案將僅應用於 `dashboard` 內的路由片段。

### 嵌套佈局 [](#nesting-layouts)

佈局預設 **嵌套**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/nested-layouts.png)

例如，如果我們結合上述兩個佈局。根佈局 (`app/layout.js`) 將應用於 `dashboard` 佈局，這也將應用於 `dashboard/*` 內的所有路由片段。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/nested-layouts-example.png)

頁面 [](#pages)
---------------

**新檔案約定：** `page.js`

頁面是獨一無二的 UI，專門用於路由片段。您可以在文件夾中添加 `page.js` 檔案來建立頁面。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page.png)

例如，要建立 `/dashboard/*` 路由的頁面，您可以在每個文件夾中添加 `page.js` 檔案。當用戶訪問 `/dashboard/settings` 時，Next.js 將渲染 `settings` 文件夾的 `page.js` 檔案，並包含位於 [子樹](#terminology) 更上層的佈局。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page-example.png)

您可以在 dashboard 文件夾中直接建立 `page.js` 檔案以匹配 `/dashboard` 路由。dashboard 佈局也適用於此頁面：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page-nested.png)

此路由由 2 個片段組成：

* `/` 根片段
* `dashboard` 片段

> **注意：**
>
> * 路由必須在葉片段中擁有頁面才能有效。如果沒有，路由將拋出錯誤。

### 佈局和頁面行為 [](#layout-and-page-behavior)

* 檔案延伸模組 `js|jsx|ts|tsx` 可用於 Pages 和 Layouts。
* 頁面組件是 `page.js` 的預設匯出。
* 佈局組件是 `layout.js` 的預設匯出。
* 佈局組件 **必須** 接受 `children` 屬性。

當渲染佈局組件時，`children` 屬性將填充子佈局（如果存在於 [子樹](#terminology) 下方）或頁面。

將它視為佈局 [樹](#terminology) 會更容易理解，其中父佈局會選擇最接近的子佈局，直到達到頁面。

**範例：**

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/basic-example.png)

`app/layout.js`

```
// Root layout
// - Applies to all routes
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <Header />
        {children}
        <Footer />
      </body>
    </html>
  );
}
```

app/dashboard/layout.js

```
// Regular layout
// - Applies to route segments in app/dashboard/*
export default function DashboardLayout({ children }) {
  return (
    <>
      <DashboardSidebar />
      {children}
    </>
  );
}
```

app/dashboard/analytics/page.js

```
// Page Component
// - The UI for the `app/dashboard/analytics` segment
// - Matches the `acme.com/dashboard/analytics` URL path
export default function AnalyticsPage() {
  return <main>...</main>;
}
```

上述佈局和頁面的組合將渲染以下組件層級結構：

組件層級結構

```
<RootLayout>
  <Header />
  <DashboardLayout>
    <DashboardSidebar />
    <AnalyticsPage>
      <main>...</main>
    </AnalyticsPage>
  </DashboardLayout>
  <Footer />
</RootLayout>
```

React 服務器組件 [](#react-server-components)
---------------------------------------------------

> **注意：** React 引入了新的組件類型：服務器、客戶端（傳統 React 組件）和共享。要了解有關這些新類型的更多資訊，我們建議閱讀 React [服務器組件 RFC](https://reactjs.org/blog/2020/12/21/data-fetching-with-react-server-components.html)。

透過本 RFC，您可以開始使用 React 功能並逐步採用 React 服務器組件到您的 Next.js 應用程式中。

新路由系統的内部也将利用最近发布的 React 功能，例如串流、Suspense 和轉換。這些是 React 服務器組件的構建塊。

### 預設為服務器組件 [](#server-components-as-the-default)

`pages` 和 `app` 目錄之間最大的變化之一是，預設情況下，**`app` 內的檔案將會在服務器上渲染為 React 服務器組件。**

這將允許您在從 `pages` 遷移到 `app` 時採用 React 服務器組件。

> **注意：** 服務器組件可用於 `app` 文件夾或您自己的文件夾，但不能用於 `pages` 目錄，以支援後向兼容。

### 客戶端和服務器組件約定 [](#client-and-server-components-convention)

`app` 文件夾將支援服務器、客戶端和共享組件，您將能夠 [在樹中混排這些組件](#interleaving-client-and-server-components-in-a-route)。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/server-components.png)

目前正在就客戶端組件和服務器組件的定義約定 [進行討論](https://github.com/reactjs/rfcs/pull/189#issuecomment-1116482278)。我們將遵循此討論的決議。

* 目前，服務器組件可以通過在檔案名稱後附加 `.server.js`來定義。例如 `layout.server.js`
* 客戶端組件可以通過在檔案名稱後附加 `.client.js` 來定義。例如 `page.client.js`。
* `.js` 檔案被視為共享組件。由於它們可以在服務器和客戶端上渲染，因此它們需要符合每個上下文的約束。

> **注意：**
>
> * 客戶端和服務器組件具有需要尊重的 [約束](https://github.com/josephsavona/rfcs/blob/server-components/text/0000-server-components.md#capabilities--constraints-of-server-and-client-components)。在決定使用客戶端或服務器組件時，我們建議使用服務器組件（預設），直到您需要使用客戶端組件。

### Hooks[](#hooks)

我們將添加客戶端和服務器組件 Hooks，讓您能夠訪問 headers 對象、cookies、路徑、搜索參數等。未來我們將有包含更多資訊的文檔。

### 渲染環境 [](#rendering-environments)

您將可以使用客戶端和服務器組件約定，對客戶端 JavaScript 捆綁包中有哪些組件進行粒度控制。

預設情況下，`app` 中的路由將使用靜態生成，並且當路由片段使用需要請求上下文的 [服務端 Hooks](#hooks) 時，將切換為動態渲染。

### 路由中混排客戶端和服務器組件 [](#interleaving-client-and-server-components-in-a-route)

在 React 中，存在在客戶端組件內部匯入服務器組件的限制，因為服務器組件可能具有僅服務器代碼（例如數據庫或檔案系統工具）。

例如，匯入服務器組件將不起作用：

ClientComponent.js

```
import ServerComponent from './ServerComponent.js';
 
export default function ClientComponent() {
  return (
    <>
      <ServerComponent />
    </>
  );
}
```

然而，服務器組件可以作為客戶端組件的子組件。您可以通過 **包裝** 它們到另一個服務器組件來完成。例如：

ClientComponent.js

```
export default function ClientComponent({ children }) {
  return (
    <>
      <h1>Client Component</h1>
      {children}
    </>
  );
}
 
// ServerComponent.js
export default function ServerComponent() {
  return (
    <>
      <h1>Server Component</h1>
    </>
  );
}
 
// page.js
// It's possible to import Client and Server components inside Server Components
// because this component is rendered on the server
import ClientComponent from "./ClientComponent.js";
import ServerComponent from "./ServerComponent.js";
 
export default function ServerComponentPage() {
  return (
    <>
      <ClientComponent>
        <ServerComponent />
      </ClientComponent>
    </>
  );
}
```

使用此模式，React 將知道它需要在將結果（不包含僅服務器代碼）發送到客戶端之前在服務器上渲染 `ServerComponent`。從客戶端組件的角度來看，其子組件已渲染。

在佈局中，此模式通過 `children` 屬性套用，因此您不需要建立額外的包裝組件。

例如，`ClientLayout` 組件將接受 `ServerPage` 組件作為其子組件：

app/dashboard/layout.js

```
// The Dashboard Layout is a Client Component
export default function ClientLayout({ children }) {
  // Can use useState / useEffect here
  return (
    <>
      <h1>Layout</h1>
      {children}
    </>
  );
}
 
// The Page is a Server Component that will be passed to Dashboard Layout
// app/dashboard/settings/page.js
export default function ServerPage() {
  return (
    <>
      <h1>Page</h1>
    </>
  );
}
```

> **注意：** 這種組合風格是渲染服務器組件 inside 客戶端組件的重要模式。它確定了要學習的一種模式的優先順序，也是我們決定使用 `children` 屬性原因之一。

數據獲取 [](#data-fetching)
-------------------------------

在路由的多個片段中獲取數據是可能的。這與 `pages` 目錄不同，後者的數據獲取限制在頁面層級。

### 佈局中的數據獲取 [](#data-fetching-in-layouts)

您可以通過使用 Next.js 數據獲取方法 `getStaticProps` 或 `getServerSideProps` 在 `layout.js` 檔案中獲取數據。

例如，部落格佈局可以使用 `getStaticProps` 從 CMS 獲取分類，用於填充側邊欄組件：

app/blog/layout.js

```
export async function getStaticProps() {
  const categories = await getCategoriesFromCMS();
 
  return {
    props: { categories },
  };
}
 
export default function BlogLayout({ categories, children }) {
  return (
    <>
      <BlogSidebar categories={categories} />
      {children}
    </>
  );
}
```

### 路由中的多個數據獲取方法 [](#multiple-data-fetching-methods-in-a-route)

您也可以獲取路由多個片段的數據。例如，獲取數據的 `layout` 可以包裝獲取其自身數據的 `page`。

使用上述部落格範例，單個帖子頁面可以使用 `getStaticProps` 和 `getStaticPaths` 從 CMS 獲取帖子數據：

app/blog/[slug]/page.js

```
export async function getStaticPaths() {
  const posts = await getPostSlugsFromCMS();
 
  return {
    paths: posts.map((post) => ({
      params: { slug: post.slug },
    })),
  };
}
 
export async function getStaticProps({ params }) {
  const post = await getPostFromCMS(params.slug);
 
  return {
    props: { post },
  };
}
 
export default function BlogPostPage({ post }) {
  return <Post post={post} />;
}
```

由於 `app/blog/layout.js` 和 `app/blog/[slug]/page.js` 都使用 `getStaticProps`，Next.js 將整個 `/blog/[slug]` 路由在構建時間靜態生成為 **React 服務器組件** - 這導致客戶端 JavaScript 減少和水合速度加快。

靜態生成路由進一步改進了這一點，因為客戶端導航重用緩存（服務器組件數據）而不重新計算工作，導致 CPU 時間減少，因為您正在渲染服務器組件的快照。

### 行為和優先級 [](#behavior-and-priority)

Next.js 數據獲取方法 (`getServerSideProps` 和 `getStaticProps`) 僅可用於 `app` 文件夾中的服務器組件。單一路由中片段的不同數據獲取方法會相互影響。

在一個片段中使用 `getServerSideProps` 會影響其他片段中的 `getStaticProps`。由於請求已經必須前往服務器以獲取 `getServerSideProps` 片段，服務器也將渲染任何 `getStaticProps` 片段。它將重用構建時獲取的屬性，因此數據仍然是靜態的，**渲染** 是在每次請求時按需進行，屬性是在 `next build` 期間生成的。

在一個片段中使用帶有 **revalidate (ISR)** 的 `getStaticProps` 會影響其他片段中使用 `revalidate` 的 `getStaticProps`。如果路由中有兩個 revalidate 週期，較短的 revalidate 將優先。

> **注意：** 未來，這可能會優化以允許在路由中進行完整的數據獲取粒度。

### 搭配 React 服務器組件的數據獲取 [](#data-fetching-with-react-server-components)

服務端路由、React 服務器組件、Suspense 和串流的組合對 Next.js 的數據獲取和渲染有幾個含義：

### 平行數據獲取 [](#parallel-data-fetching)

Next.js 將積極以平行方式啟動數據獲取以最大限度地減少瀑布流。例如，如果數據獲取是 **串行的**，每個嵌套片段都無法開始獲取數據，直到前一個片段完成。但是 **平行** 獲取，每個片段都可以同時積極開始數據獲取。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel.png)

由於渲染可能依賴上下文，每個片段的渲染將在獲取其數據且父段完成渲染後開始。

未來，通過 Suspense，渲染甚至可以立即開始，即使數據未完全加載。如果在數據可用之前讀取數據，將觸發 Suspense。React 將積極開始渲染服務器組件，在請求完成之前，並將結果插入請求解析。

### 部分獲取和渲染 [](#partial-fetching-and-rendering)

在兄弟路由片段之間導航時，Next.js 僅從該片段開始獲取和渲染。它無需重新獲取或重新渲染上方的任何內容。這意味著，在共享佈局的頁面中，當用戶在兄弟頁面之間導航時，佈局將保留，Next.js 僅會從該片段開始獲取和渲染。

這對於 React 服務器組件特別有用，因為否則每次導航都會導致頁面在服務器上重新渲染，而不是在服務器上僅渲染頁面的一部分。這減少了數據傳輸量和執行時間，從而提高性能。

例如，如果用戶在 `/analytics` 和 `/settings` 頁面之間導航，React 將重新渲染頁面片段，但保留佈局：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/partial-example.png)

> **注意：** 將有可能強制重新獲取更上層 [樹](#terminology) 的數據。我們仍在討論詳細資訊，並將更新 RFC。

路由群組 [](#route-groups)
-----------------------------

`app` 文件夾的層級結構直接映射到 URL 路徑。但透過建立路由群組，可以打破這種模式。路由群組可用於：

* 組織路由而不影響 URL 結構。
* 讓路由片段退出佈局。
* 透過分開應用程式來建立多個根佈局。

> **注意：** 路由群組的命名僅用於組織目的，因為它們不影響 URL 路徑。

**範例：讓路由退出佈局**

要讓路由退出佈局，請建立新的路由群組（例如 `(shop)`）並將共享相同佈局的路由移入群組（例如 `account` 和 `cart`）。群組外的路由將不會共享佈局（例如 `checkout`）。

之後：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/route-group-opt-out-after.png)

**範例：組織路由而不影響 URL 路徑**

同樣，為了組織路由，請建立群組以保持相關路由在一起。括號內的文件夾將從 URL 中省略（例如 `(marketing)` 或 `(shop)`）。

以服務器為中心的路由 [](#server-centric-routing)
-------------------------------------------------

目前，Next.js 使用客戶端路由。在首次載入和後續導航之後，會向服務器發出請求以獲取新頁面的資源。這包括 **每個組件的 JavaScript**（包括僅在特定條件下顯示的組件）和它們的屬性（來自 `getServerSideProps` 或 `getStaticProps` 的 JSON 數據）。一旦 JavaScript 和數據都從服務器加載，**React 將在客戶端渲染組件。**

在新模型中，Next.js 將使用以服務器為中心的路由，同時保持客戶端轉換。這與正在服務器上評估的服務器組件一致。

導航時，獲取數據並讓 React 在 **服務器上** 渲染組件。服務器的輸出是特殊的指令（不是 HTML 或 JSON），供客戶端上的 React 更新 DOM。這些指令包含渲染的服務器組件的 **結果**，這意味著無需將該組件的 JavaScript 加載到瀏覽器中即可渲染結果。

與客戶端組件的預設值相反，客戶端組件需要將組件 JavaScript 發送到瀏覽器以在客戶端渲染。

以服務器為中心的路由搭配 React 服務器組件的好處包括：

* 路由使用與服務器組件相同的請求（不發出額外的服務器請求）。
* 服務器上的工作較少，因為路由之間僅獲取和渲染更改的片段。
* 當客戶端導航且不使用新的客戶端組件時，瀏覽器中不會加載額外的 JavaScript。
* 路由器利用新的串流協議，這使得渲染可以在加載所有數據之前開始。

當用戶在應用程式中導航時，路由器會在記憶體中的客戶端緩存中存儲 React 服務器組件 _有效負載_ 的結果。緩存按路由片段劃分，這允許在任何層級無效化，並確保並發渲染的一致性。這意味著在某些情況下，先前獲取的片段的緩存可以被重用。

> **注意**
>
> * 靜態生成和服務器端緩存可用於優化數據獲取。
> * 上述資訊描述了後續導航的行為。初始載入是涉及服務端渲染以生成 HTML 的不同程序。
> * 雖然客戶端路由對於 Next.js 一直運作良好，但它在大規模潛在路由數量時擴展性不佳，因為客戶端必須下載 **路由圖**。
> * 總之，通過使用 React 服務器組件，客戶端導航速度更快，因為我們加載和渲染的瀏覽器組件較少。

即時載入狀態 [](#instant-loading-states)
-------------------------------------------------

透過服務端路由，導航發生在 **數據獲取和渲染後**，因此在數據獲取時顯示載入 UI 很重要，否則應用程式將無響應。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/server-side-routing.png)

新路由器將使用 [Suspense](https://reactjs.org/docs/react-api.html#reactsuspense) 進行即時載入狀態和預設骨架。這意味著可以在新片段內容加載時立即顯示載入 UI。一旦服務器渲染完成，新內容將交換到顯示中。

渲染進程中：

* 導航到新路由將即時完成。
* 共享佈局將保持互動，同時新路由片段加載。
* 導航將可中斷 — 意味著用戶可以在一個路由內容加載的同時在不同路由之間導航。

### 預設載入骨架 [](#default-loading-skeletons)

Suspense 邊界將透過稱為 `loading.js` 的新檔案約定在幕后自動處理。

**範例：**

您可以通過在文件夾中添加 `loading.js` 檔案來建立預設載入骨架。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/loading.png)

`loading.js` 應匯出 React 組件：

loading.js

```
export default function Loading() {
  return <YourSkeleton />
}
 
// layout.js
export default function Layout({children}) {
  return (
    <>
      <Sidebar />
      {children}
    </>
  )
}
 
// Output
<>
  <Sidebar />
  <Suspense fallback={<Loading />}>{children}</Suspense>
</>
```

這將導致文件夾中的所有片段包裝在 suspense 邊界內。當首次載入佈局和導航之間的兄弟頁面時，將使用預設骨架。

錯誤處理 [](#error-handling)
---------------------------------

[錯誤邊界](https://reactjs.org/docs/error-boundaries.html) 是 React 組件，用於捕獲其子組件樹中任何地方的 JavaScript 錯誤。

### 約定 [](#convention-1)

您可以透過添加 `error.js` 檔案並預設匯出 React 組件，建立能捕獲子樹中錯誤的錯誤邊界。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/error.png)

如果是在該子樹中拋出錯誤，此組件將作為回退顯示。此組件可用於記錄錯誤、顯示有關錯誤的有用資訊，以及嘗試恢復錯誤的功能。

由於片段和佈局的嵌套性質，建立錯誤邊界可讓您將錯誤隔離到 UI 的這些部分。在錯誤期間，邊界上方的佈局將保持互動狀態，並且狀態將保留。

error.js

```
export default function Error({ error, reset }) {
  return (
    <>
      An error occurred: {error.message}
      <button onClick={() => reset()}>Try again</button>
    </>
  );
}
 
// layout.js
export default function Layout({children}) {
  return (
    <>
      <Sidebar />
      {children}
    </>
  )
}
 
// Output
<>
  <Sidebar />
  <ErrorBoundary fallback={<Error />}>{children}</ErrorBoundary>
</>
```

> 注意：
>
> * `layout.js` 檔案中的錯誤與 `error.js` 在同一片段中將不會被捕獲，因為自動錯誤邊界包裝佈局的子組件，而不是佈局本身。

範本 [](#templates)
-----------------------

範本類似於佈局，因為它們包裝每個子佈局或頁面。

與跨路由持久化並保留狀態的佈局不同，範本為其每個子組件建立新實例。這意味著，當用戶在共享範本的路由片段之間導航時，組件的新實例將被掛載。

> **注意：** 我們建議使用佈局，除非您有特定原因需要使用範本。

### 約定 [](#convention-2)

可以通過從 `template.js` 檔案匯出預設 React 組件來定義範本。該組件應接受一個 `children` 屬性，該屬性將填充嵌套片段。

### [範例](#example)

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/template.png)

template.js

```
export default function Template({ children }) {
  return <Container>{children}</Container>;
}
```

帶有佈局和範本的路由片段的渲染輸出如下所示：

```
<Layout>
  {/* Note that the template is given a unique key. */}
  <Template key={routeParam}>{children}</Template>
</Layout>
```

### 行為 [](#behavior)

在某些情況下，您可能需要掛載和卸載共享 UI，而範本會提供更合適的選項。例如：

* 使用 CSS 或.animation 庫的進入/退出動畫
* 依賴 `useEffect`（例如記錄頁面檢視）和 `useState`（例如每頁反饋表）的功能
* 更改預設框架行為。例如，只有在佈局首次載入時 Suspense 邊界才顯示回退，而不顯示頁面切換時。對於範本，每次導航時顯示回退。

例如，考慮帶有一個邊框容器的嵌套佈局設計，該容器應包圍每個子頁面。

您可以將容器放入父佈局 (`shop/layout.js`) 中：

shop/layout.js

```
export default function Layout({ children }) {
  return <div className="container">{children}</div>;
}
 
// shop/page.js
export default function Page() {
  return <div>...</div>;
}
 
// shop/categories/layout.js
export default function CategoryLayout({ children }) {
  return <div>{children}</div>;
}
```

但當切換頁面時，進入/退出動畫不會播放，因為共享的父佈局不會重新渲染。

您可以將容器放在每個嵌套佈局或頁面：

shop/layout.js

```
export default function Layout({ children }) {
  return <div>{children}</div>;
}
 
// shop/page.js
export default function Page() {
  return <div className="container">...</div>;
}
 
// shop/categories/layout.js
export default function CategoryLayout({ children }) {
  return <div className="container">{children}</div>;
}
```

但這時您必須在每個嵌套佈局或頁面中手動放置，這在更複雜的應用程式中可能會很繁瑣且容易出錯。

使用此約定，您可以在路由之間共享範本，在導航時建立新實例。這意味著 DOM 元素將被重新建立，狀態將不會保留，並且將重新同步效果。

進階路由模式 [](#advanced-routing-patterns)
-------------------------------------------------------

我們計劃引入約定來覆蓋邊界情況，並允許您實現更進階的路由模式。以下是我們積極思考的一些範例：

### 攔截路由 [](#intercepting-routes)

有時，從其他路由中攔截路由片段可能是有用的。導航時，URL 將像常規一樣更新，但攔截片段將顯示在當前路由的佈局中。

#### 範例 [](#example-1)

**之前：** 點擊圖像會導致導航到具有自身佈局的新路由。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-routes-before.png)

**之後：** 透過攔截路由，點擊圖像現在在當前路由的佈局中載入片段。例如，作為模態框。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-routes-after.png)

要從 `/[username]` 片段攔截 `/photo/[id]` 路由，請在 `/[username]` 文件夾內建立重複的 `/photo/[id]` 文件夾，並在其前面加上 `(..)` 約定。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-route.png)

#### 約定 [](#convention-3)

* `(..)` — 將匹配高層一級的路由片段（父目錄的兄弟節點）。類似於相對路徑中的 `../`。
* `(..)(..)` — 將匹配高兩層的路由片段。類似於相對路徑中的 `../../`。
* `(...)` — 將匹配根目錄中的路由片段。

> **注意：** 刷新或分享頁面將以預設佈局載入路由。

### 動態平行路由 [](#dynamic-parallel-routes)

有時在相同視圖中顯示兩個或更多葉片段 (`page.js`) 並且可以獨立導航可能是有用的。

舉例說明同一儀表板內的兩個或多個標籤組。導航一個標籤組不應對另一個標籤組產生影響。在向后和向前導航時，標籤的組合也應正確恢復。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes.png)

#### 約定 [](#convention-4)

預設情況下，佈局接受一個名為 `children` 的屬性，其中包含嵌套佈局或頁面。您可以透過建立命名「槽」（包含 `@` 前綴的文件夾並將片段嵌套在其中）來重命名屬性。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes-children.png)

此更改後，佈局將接收名為 `customProp` 的屬性，而不是 `children`。

analytics/layout.js

```
export default function Layout({ customProp }) {
  return <>{customProp}</>;
}
```

您可以透過在同一層級添加多個命名槽來創建平行路由。在下方的範例中，`@views` 和 `@audience` 都作為屬性傳遞給分析佈局。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes-complete.png)

您可以使用命名槽同時顯示葉片段。

analytics/layout.js

```
export default function Layout({ views, audience }) {
  return (
    <>
      <div>
        <ViewsNav />
        {views}
      </div>
      <div>
        <AudienceNav />
        {audience}
      </div>
    </>
  );
}
```

當用戶首次導航至 `/analytics` 時，每個文件夾（`@views` 和 `@audience`）中的 `page.js` 片段將顯示。

導航至 `/analytics/subscribers` 時，僅更新 `@audience`。同理，導航至 `/analytics/impressions` 時，僅更新 `@views`。

向后和向前導航將恢復平行路由的正確組合。

結合攔截和並行路由 [](#combining-intercepting-and-parallel-routes)
-----------------------------------------------------------------------------------------

您可以結合攔截和並行路由來實現應用程式中的特定路由行為。

**範例**

例如，建立模態框時，您通常需要了解一些常見挑戰，例如：

* 模態框無法通過 URL 訪問。
* 頁面刷新時模態框關閉。
* 向后導航導航到上一個路由，而不是模態框後面的路由。
* 向前導航沒有重新開啟模態框。

您可能希望模態框在開啟時更新 URL，並能夠通過向前/向后導航開啟和關閉模態框。此外，分享 URL 時，您可能希望頁面載入時開啟模態框及其後面的上下文，或者可能希望頁面載入內容而不開啟模態框。

一個很好的範例是社交媒體網站上的照片。通常，照片可以在用戶的資訊流或個人資料中通過模態框訪問。但是當分享照片時，它們直接在各自的頁面顯示。

透過使用約定，我們可以預設將模態框行為對應到路由行為。

考慮這個文件夾結構：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-route-modal.png)

使用此模式：

* `/photo/[id]` 的內容可以透過其自身上下文中的 URL 訪問。它也可以在 `/[username]` 路由中的模態框中訪問。
* 使用客戶端導航的向前和向後導航應關閉和重新開啟模態框。
* 刷新頁面（服務器端導航）應將用戶帶回到原始的 `/photo/id` 路由，而不是顯示模態框。

在 `/@modal/(..)photo/[id]/page.js` 中，您可以返回包裝在模態框組件中的頁面內容。

/@modal/(..)photo/[id]/page.js

```
export default function PhotoPage() {
  const router = useRouter();
 
  return (
    <Modal
      // the modal should always be shown on page load
      isOpen={true}
      // closing the modal should take user back to the previous page
      onClose={() => router.back()}
    >
      {/* Page Content */}
    </Modal>
  );
}
```

> **注意：** 此解決方案不是建立 Next.js 模態框的唯一方式，但旨在展示您可以如何結合約定以實現更複雜的路由行為。

條件路由 [](#conditional-routes)
-----------------------------------------

有時，您可能需要類似數據或上下文的動態資訊來決定顯示哪個路由。您可以使用平行路由來條件加載一個路由或另一個路由。

**範例**

layout.js

```
export async function getServerSideProps({ params }) {
  const { accountType } = await fetchAccount(params.slug);
  return { props: { isUser: accountType === 'user' } };
}
 
export default function UserOrTeamLayout({ isUser, user, team }) {
  return <>{isUser ? user : team}</>;
}
```

在上面的範例中，取決於 slug 可以返回 `user` 或 `team` 路由。這允許條件加載數據並針對一個選項或另一個選項匹配子路由。
