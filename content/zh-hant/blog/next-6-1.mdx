我們很榮幸今天推出生產環境可用的 **Next.js 6.1**，其包含：

*   提升熱重載可靠性
*   程式碼庫改進
*   Next.js 代碼修改

除發布 **Next.js 6.1** 外，我們很興奮地宣布 [nextjs.org](https://nextjs.org) 現在已 [開源](https://github.com/vercel/next-site)

熱重載可靠性提升[](#improved-hot-reloading-reliability)
-------------------------------------------------------------------------

在 **Next.js 6.1** 之前的版本中，Next.js 替用戶實現了 `react-hot-loader`。該庫會在熱重載之間保持 React 狀態。

這樣做會讓 `react-hot-loader` 為 React 添加一些非標準行為，例如，它會忽略 `shouldComponentUpdate` 並且元素的 `type` 最終變成了一個代理組件，而不是實際的 React 組件。

為了讓 Next.js 盡可能接近默認 React，我們移除了 `react-hot-loader` 作為依賴項，這確保開發和生產模式在行為上更一致。請注意，Next.js 的熱重載功能並**沒有**被移除，熱重載一直是由 Next.js 內部處理的。

熱重載 Typescript 和其他自定義擴展 [](#hot-reloading-typescript-and-other-custom-extensions)
-------------------------------------------------------------------------------------------------------------

默認情況下，Next.js 會自動查找 `pages` 目錄中的所有 `.js` 或 `.jsx` 文件以定義路由。

隨著 Next.js 5 引入通用 Webpack，現在擁有可能性編譯為 JS 的頂層頁面。一個好的例子是 Typescript，它使用 `.ts` 和 `.tsx`。

`pageExtensions` 是 `next.config.js` 中的配置鍵，目的是允許 Next.js 插件定義應視為頁面的擴展。例如 [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript) 定義了 `.ts` 和 `.tsx` 或 [`@zeit/next-mdx`](https://github.com/vercel/next-plugins/tree/master/packages/next-mdx)，其 [文件](https://github.com/vercel/next-plugins/tree/master/packages/next-mdx#top-level-mdx-pages) 展示了如何擁有頂層 `.mdx` 頁面。

過去在實現 `pageExtensions` 時，Next.js 插件需要實現用於熱重載的 `hot-self-accept-loader`。這不再需要，當添加擴展到 `pageExtensions` 時，`hot-self-accept-loader` 會自動應用。

程式碼庫改進 [](#codebase-improvements)
-----------------------------------------------

最近我們一直在為即將到來的功能鋪平道路，這涉及一些底層變化，將從長期來看提高代碼質量。

這些變化之一是 `server/build` 目錄被移動到頂層 `build`。這對新貢獻者來說更容易找到 Webpack 和 Babel 配置。

我們也一直致力於在整個代碼庫中添加 [Flow](https://flow.org/) 類型。

我們做出的一項對用戶更明顯的變更 `.next/dist` 已重命名為 `.next/server`。`.next` 目录用於存放構建輸出。例如當你運 `next build` 時，結果將存儲在 `.next` 中。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-6-1/server-dist.png)

> 預渲染文件現在位於 `server` 目錄中

相同常量的出現已移动到一個公共文件：[`constants.js`](https://github.com/vercel/next.js/blob/canary/lib/constants.js)

Next.js 代碼修改 [](#nextjs-codemods)
------------------------------------

當 Next.js 6.0 發布時，頁面組件上神奇注入的 `url` 屬性已被棄用。`url` 要消失的原因是我們希望讓事情更加可預測和明確。無緣無故出現一個神奇的 `url` 屬性無助於實現這一目標。

獲取 `url` 屬性原有相同屬性 的推薦方式是使用 `withRouter`：

page.js

```
// old
class Page extends React.Component {
  render() {
    const { url } = this.props;
    return <div>{url.pathname}</div>;
  }
}
export default Page;
```

> 在 Next.js 6 之前的版本中如何使用 `url` 訪問 pathname

page.js

```
// new
import { withRouter } from 'next/router';
class Page extends React.Component {
  render() {
    const { router } = this.props;
    return <div>{router.pathname}</div>;
  }
}
export default withRouter(Page);
```

> 在 Next.js 6 之後的版本中，應如何使用 `withRouter` 注入的 `router` 訪問 pathname

由於我們致力於保持 Next.js 應用程序的升級路徑簡單，因此我們制定了簡單的升級方式，將 `url` 的使用升級為 `withRouter`。

這項努力的結果是 [next‑codemod](https://github.com/vercel/next-codemod)，這是一個代碼修改 (codemod) 庫，讓將特定的棄用功能升級到新用法變得像在一個命令下運行一樣簡單。

我們提供的第一個代碼修改 (codemod) 是 `url-to-withrouter`，它會自動將大量使用 `url` 的情況轉換為 `withRouter`。

終端機

```
  jscodeshift -t ./url-to-withrouter.js pages/**/*.js
```

> 這將把 `url` 的使用轉換為 `withRouter`。

[在此閱讀詳情](https://github.com/vercel/next-codemod#url-to-withrouter)

未來 [](#future)
-----------------

我們一直在努力開發一些新功能以提高可靠性和性能，以下是幾個重點：

### Webpack 4[](#webpack-4)

Webpack 4 帶來了很多改進：更好的代碼分割、默認需要的配置更少，最重要的是構建時間更快。在我們對一個擁有 200 多個頁面的應用程序進行的初始測試中，`next build` 的耗時從平均 100 秒降至 70 秒。第二次運行（使用緩存）`next build` 平均耗時 21 秒。

### Serverless Next.js[](#serverless-nextjs)

我們正在逐步進行更改，以準備將 `next start` 移出成為自己的包：`next-server`。此包將在安裝大小和啟動時間上進行高度優化。這些優化是「無服務器」用例所需要的，其中每個請求或每幾個請求就會執行一次新的應用實例。這意味著應用程序的「冷啟動」必須優化到最快的速度。
