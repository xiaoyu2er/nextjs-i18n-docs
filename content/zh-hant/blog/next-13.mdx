正如我們在 [Next.js Conf](https://nextjs.org/conf) 所宣布的那樣，Next.js 13（穩定版）奠定了無限制動態應用的基礎：

*   [**`app` Directory (beta):**](#new-app-directory-beta) 更簡易、更快速、更少的客戶端 JS。
    *   [**佈局**](#layouts)
    *   [**React Server Components**](#server-components)
    *   [**串流**](#streaming)
*   **[Turbopack (alpha):](#introducing-turbopack-alpha)** 最多 700 倍更快的基於 Rust 的 Webpack 替代品。
*   **[新 `next/image`:](#nextimage)** 透過原生瀏覽器延遲載入實現更快效能。
*   **[新 `@next/font` (beta):](#nextfont)** 預設自動自我託管字型，零佈局偏移。
*   **[改進 `next/link`:](#breaking-changes)** 使用自動 `<a>` 的簡化 API。

Next.js 13 和 `pages` 目錄穩定且適用於生產環境。今天運行以下指令進行更新：

終端機

```
npm i next@latest react@latest react-dom@latest eslint-config-next@latest
```

New `app` Directory (Beta)[](#new-app-directory-beta)
-----------------------------------------------------

今天，我們通過引入 `app` 目錄改善 Next.js 中的路由和佈局體驗，並與 React 的未來對齊。這是之前發布的 [佈局 RFC](https://nextjs.org/blog/layouts-rfc) 的後續，旨在收集社群反饋。

`app` 目錄目前處於 Beta 階段，我們還不建议在生產環境中使用。您可以使用 Next.js 13 搭配 `pages` 目錄，使用穩定的功能（如改進後的 [`next/image`](#nextimage) 和 [`next/link`](#nextlink) 元件），並以自己的速度選擇加入 `app` 目錄。`pages` 目錄在可見的未來將繼續獲支持。

`app` 目錄支援：

*   **[佈局：](#layouts)** 輕鬆在不同路由之間共享 UI，同時保留狀態並避免昂貴的重新渲染。
*   **[伺服器元件：](#server-components)** 使伺服器優先成為最動態應用的預設。
*   **[串流：](#streaming)** 顯示即時載入狀態，並在渲染時串流 UI 單元。
*   **[支援資料獲取：](#data-fetching)** `async` 伺服器元件和擴充的 `fetch` API 支援元件級別獲取。

![`app` 目錄可以從您現有的 pages/ 目錄中逐步採用。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/app-folder.png)

`app` 目錄可以從您現有的 pages/ 目錄中逐步採用。

### Layouts[](#layouts)

`app/` 目錄讓建立複雜介面變得輕鬆，這些介面可在導航之間保留狀態，避免昂貴的重新渲染，並支援進階路由模式。此外，您可以嵌套佈局，並**將應用程式代碼置於同一路徑**以與您的路由在一起，例如元件、測試和樣式。

![`app/` 目錄可以從您現有的 pages/ 目錄中逐步採用。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-13/colocating-assets-in-the-app-directory.png)

`app/` 目錄可以從您現有的 pages/ 目錄中逐步採用。

在 `app/` 中建立路由需要單一檔案 `page.js`：

app/page.js

```
// This file maps to the index route (/)
export default function Page() {
  return <h1>Hello, Next.js!</h1>;
}
```

接著您可以透過檔案系統**定義佈局**。佈局在多個頁面上共享 UI。在導航時，佈局保留狀態，保持互動性，且不會重新渲染。

app/blog/layout.js

```
export default function BlogLayout({ children }) {
  return <section>{children}</section>;
}
```

[了解更多關於佈局和頁面的資訊](/docs/app/building-your-application/routing) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/app-directory)。

### Server Components[](#server-components)

`app/` 目錄引入對 React 新 [伺服器元件 (Server Components)](/docs/app/building-your-application/rendering/server-components) 架構的支援。伺服器元件和 [客戶端元件 (Client Components)](/docs/app/building-your-application/rendering/client-components) 分別利用伺服器和客戶端專長的領域 - 讓您能夠建立快速、高度互動的應用程式，並使用單一程式設計模型提供優異的開發者體驗。

透過伺服器元件，我們奠定了建立複雜介面的基礎，同時**減少發送給客戶端的 JavaScript 數量**，實現更快的初始頁面載入。

當載入路由時，Next.js 和 React 執行緒載量會隨之載入，這是**可快取**且**大小可預測**的。此執行緒大小不會隨應用程式成長而增加。此外，執行緒是異步載入的，允許您的伺服器 HTML 在客戶端進行漸進式增強。

[了解更多關於伺服器元件的資訊](/docs/app/building-your-application/rendering/server-components) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/app-directory)。

### Streaming[](#streaming)

`app/` 目錄引入了漸進式渲染並逐步串流 UI 渲染單元到客戶端的能力。

透過 Next.js 中的伺服器元件和 [嵌套佈局](/docs/app/building-your-application/routing/pages-and-layouts)，您能夠立即渲染不需要特定資料的頁面部分，並顯示 [載入狀態](/docs/app/building-your-application/routing/loading-ui-and-streaming) 給正在獲取資料的頁面部分。透過這種方法，使用者不需要等待整個頁面載入後才能開始與其互動。

![您可以將應用程式代碼（如元件、測試和樣式）與您的路由置於同一路徑。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-13/streaming.png)

您可以將應用程式代碼（如元件、測試和樣式）與您的路由置於同一路徑。

部署到 Vercel 時，使用 `app/` 目錄的 Next.js 13 應用程式在 Node.js 和 Edge 執行緒中將預設串流回應，以改善效能。

[了解更多關於串流的資訊](/docs/app/building-your-application/routing/loading-ui-and-streaming#what-is-streaming) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/app-directory)。

### Data Fetching[](#data-fetching)

React 最近的 [Promises RFC](https://github.com/acdlite/rfcs/blob/first-class-promises/text/0000-first-class-support-for-promises.md) 引入了一種在元件內部獲取資料和處理 promises 的**強大新方式**：

app/page.js

```
async function getData() {
  const res = await fetch('https://api.example.com/...');
  // The return value is *not* serialized
  // You can return Date, Map, Set, etc.
  return res.json();
}
 
// This is an async Server Component
export default async function Page() {
  const data = await getData();
 
  return <main>{/* ... */}</main>;
}
```

原生的 `fetch` Web API 也已在 React 和 Next.js 中擴充。它 [自動去重 fetch 請求](/docs/app/api-reference/functions/fetch)，並提供**一種**靈活的方式在元件級別獲取、快取和重新驗證資料。這意味著靜態網站生成 (SSG)、伺服器端渲染 (SSR) 和增量靜態再生 (ISR) 的所有好處現在都可用於一種 API：

```
// This request should be cached until manually invalidated.
// Similar to `getStaticProps`.
// `force-cache` is the default and can be omitted.
fetch(URL, { cache: 'force-cache' });
 
// This request should be refetched on every request.
// Similar to `getServerSideProps`.
fetch(URL, { cache: 'no-store' });
 
// This request should be cached with a lifetime of 10 seconds.
// Similar to `getStaticProps` with the `revalidate` option.
fetch(URL, { next: { revalidate: 10 } });
```

在 `app` 目錄中，您可以在 [佈局、頁面](/docs/app/building-your-application/routing/pages-and-layouts) 和元件內部獲取資料 - 包括支援來自伺服器的 [串流回應](#streaming)。

我們提供易用的方式來處理載入和錯誤狀態，並串流渲染中的 UI。在未来的版本中，我們也將改善和簡化資料變更。

![透過 `app/` 目錄，您可以使用新的特殊檔案 `loading.js` 自動建立具有 Suspend 邊界 (Suspense boundaries) 的即時載入 UI。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-13/streaming.png)

透過 `app/` 目錄，您可以使用新的特殊檔案 `loading.js` 自動建立具有 Suspend 邊界 (Suspense boundaries) 的即時載入 UI。

我們興奮地與開放原始碼社群、套件維護者和其他貢獻於 React 生態系統的公司合作，以建立這個 React 和 Next.js 的新時代。能夠在元件內部放置資料獲取並减少对客戶端傳輸的 JavaScript 數量，是我們興奮納入 `app/` 目錄的兩項重要社群反饋。

[了解更多關於資料獲取的資訊](/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/app-directory)。

### 引入 Turbopack (Alpha)[](#introducing-turbopack-alpha)
-------------------------------------------------------------

Next.js 13 包含 [Turbopack](https://vercel.com/blog/turbopack)，這是新的基於 Rust 的 Webpack 繼承者。

Webpack 已經被下載了超過 **30 億次**。雖然它是建構 Web 的不可或缺的一部分，但我們已經觸及了基於 JavaScript 的工具有所所能的最大效能極限。

在 Next.js 12 中，我們開始轉向原生的 Rust 驅動工具。我們通過移轉 Babel 開始，帶來了 17 倍更快的編譯。接著，我們替換了 Terser，帶來了 6 倍更快的壓縮。現在是徹底轉向原生進行封裝的時候了。

使用 Next.js 13 的 Turbopack alpha 版本將導致：

*   **700 倍更快** 更新速度勝過 Webpack
*   **10 倍更快** 更新速度勝過 Vite
*   **4 倍更快** 冷啟動速度勝過 Webpack

![Turbopack 是我們基於 Rust 的 Webpack 繼承者，大型應用程式的熱模組換載 (HMR) 速度快 700 倍。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-13/turbopack.png)

Turbopack 是我們基於 Rust 的 Webpack 繼承者，大型應用程式的熱模組換載 (HMR) 速度快 700 倍。

[Turbopack](https://nextjs.org/docs/app/api-reference/turbopack) 僅在開發中封裝所需的最小資源，因此啟動時間極其快速。在具有 3000 個模組的應用中，**Turbopack 啟動耗時 1.8 秒**。**Vite 耗時 11.4 秒**，**Webpack 耗時 16.5 秒**。

Turbopack 出廠即支援伺服器元件、TypeScript、JSX、CSS 等。在 alpha 階段期間，[許多功能](https://nextjs.org/docs/app/api-reference/turbopack#supported-features) 尚未支援。我們很想聽到您使用 Turbopack 加速本地迭代反饋。

> **注意：** Next.js 中的 Turbopack 目前僅支援 `next dev`。請參閱 [支援功能](https://nextjs.org/docs/app/api-reference/turbopack#supported-features)。我們也致力於透過 Turbopack 增加對 `next build` 的支援。

今天在 Next.js 13 中使用 `next dev --turbo` 體驗 Turbopack alpha。

`next/image`[](#nextimage)
--------------------------

Next.js 13 引入了**強大的新圖片元件**，允許您輕鬆顯示圖片而無佈局偏移，並按需優化檔案以提升效能。

在 Next.js 社群調查期間，**70% 的受訪者**告訴我們他們在生產環境中使用 Next.js 圖片元件，從而看到**改善的 Core Web Vitals**。隨著 Next.js 13，我們將進一步改善 `next/image`。

新圖片元件：

*   發送較少的客戶端 JavaScript
*   更容易樣式和配置
*   更易於無障礙，預設需要 `alt` 標籤
*   與 Web 平台對齊
*   更快，因為原生延遲載入不需要水合 (hydration)

app/page.js

```
import Image from 'next/image';
import avatar from './lee.png';
 
export default function Home() {
  // "alt" is now required for improved accessibility
  // optional: image files can be colocated inside the app/ directory
  return <Image alt="leeerob" src={avatar} placeholder="blur" />;
}
```

[了解更多關於圖片元件的資訊](/docs/app/building-your-application/optimizing/images) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/nextjs-boilerplate)。

`@next/font`[](#nextfont)
-------------------------

Next.js 13 引入了**全新的字型系統**，該系統：

*   自動優化您的字型，包括自定字型
*   移除外部網路請求以改善隱私和效能
*   內建任何字型檔案的自動自我託管 (self-hosting)
*   使用 CSS 的 [`size-adjust`](https://web.dev/css-size-adjust/) 屬性自動實現零佈局偏移

這個新字型系統允許您方便地使用所有 Google Fonts，同時考慮效能和隱私。CSS 和字型檔案在建置時下載並與其餘靜態資源一起自我託管。**瀏覽器不會向 Google 發送任何請求。**

app/layout.js / pages/_app.js

```
import { Inter } from '@next/font/google';
 
const inter = Inter();
 
<html className={inter.className}></html>;
```

自定字型也獲支援，包括對字型檔案的自動自我託管、快取和預載入的支援。

app/layout.js / pages/_app.js

```
import localFont from '@next/font/local';
 
const myFont = localFont({ src: './my-font.woff2' });
 
<html className={myFont.className}></html>;
```

您可以客製化字型載入體驗的每個部分，同時確保良好的效能和零佈局偏移，包括 `font-display`、預載入、回退以及更多。

[了解更多關於新字型元件的資訊](/docs/app/building-your-application/optimizing/fonts) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/loading-web-fonts)。

`next/link`[](#nextlink)
------------------------

`next/link` 不再需要手動添加 `<a>` 作為子項目。

這是在 12.2 中作為實驗選項添加的，現在是預設設定。在 Next.js 13 中，`<Link>` 始終渲染為 `<a>` 並允許您將屬性傳遞給底層標籤。例如：

```
import Link from 'next/link'
 
// Next.js 12: `<a>` has to be nested otherwise it's excluded
<Link href="/about">
  <a>About</a>
</Link>
 
// Next.js 13: `<Link>` always renders `<a>`
<Link href="/about">
  About
</Link>
```

[了解更多關於改進的 Link 元件的資訊](/docs/app/api-reference/components/link) 或 [部署範例以嘗試](https://vercel.com/templates/next.js/nextjs-boilerplate)。

OG 圖片生成[](#og-image-generation)
-------------------------------------------

社群卡（也稱為 開放圖形圖片 (Open Graph images)）可以大幅提高您內容的點擊參與率，一些實驗顯示轉換率最高可提升 40%。

靜態社群卡耗時、容易出錯且難以維護。因此，社群卡經常匱乏甚至是跳過的。直到今天，需要即時個性化和運算的動態社群卡一直難以實現且昂貴。

我們建立了一個新函式庫 **`@vercel/og`**，可與 Next.js 無縫合作以生成動態社群卡。

pages/api/og.jsx

```
import { ImageResponse } from '@vercel/og';
 
export const config = {
  runtime: 'experimental-edge',
};
 
export default function () {
  return new ImageResponse(
    (
      <div
        style={{
          display: 'flex',
          fontSize: 128,
          background: 'white',
          width: '100%',
          height: '100%',
        }}
      >
        Hello, World!
      </div>
    ),
  );
}
```

這種方法使用 Vercel Edge Functions、WebAssembly 和新核心函式庫來轉換 HTML 和 CSS 為圖片，並利用 React 元件抽象化，速度比現有解決方案快 **5 倍**。

[了解更多關於 OG 圖片生成的資訊](https://vercel.com/docs/og-image-generation) 或 [部署範例以嘗試](https://vercel.com/new/clone?repository-url=https://github.com/vercel/examples/tree/main/edge-functions/vercel-og-nextjs&project-name=vercel-og-nextjs&repository-name=vercel-og-nextjs)。

Middleware API 更新[](#middleware-api-updates)
-------------------------------------------------

在 Next.js 12 中，我們引入了 Middleware 以實現 Next.js 路由器的全靈活性。我們聽取了您對初始 API 設計的意見，並增加了一些增進開發者體驗和添加強大新功能的加項。

您現在可以更輕鬆地在請求上設定標頭：

middleware.ts

```
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
 
export function middleware(request: NextRequest) {
  // Clone the request headers and set a new header `x-version`
  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-version', '13');
 
  // You can also set request headers in NextResponse.rewrite
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  });
 
  // Set a new response header `x-version`
  response.headers.set('x-version', '13');
  return response;
}
```

您現在也可以直接從 Middleware 提供回應，而不必 `rewrite` 或 `redirect`。

middleware.ts

```
import { NextRequest, NextResponse } from 'next/server';
import { isAuthenticated } from '@lib/auth';
 
// Limit the middleware to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
};
 
export function middleware(request: NextRequest) {
  // Call our authentication function to check the request
  if (!isAuthenticated(request)) {
    // Respond with JSON indicating an error message
    return NextResponse.json(
      {
        success: false,
        message: 'Auth failed',
      },
      {
        status: 401,
      },
    );
  }
}
```

目前從 Middleware 發送回應需要在 `next.config.js` 中的 `experimental.allowMiddlewareResponseBody` 設定選項。

破壞性變更[](#breaking-changes)
-------------------------------------

*   最低 React 版本已從 17.0.2 提升至 18.2.0。
*   最低 Node.js 版本已從 12.22.0 提升至 14.6.0，因為 12.x 已達生命周期終止 ([PR](https://github.com/vercel/next.js/pull/41482))。
*   `swcMinify` 設定屬性從 `false` 更改為 `true`。請參閱 [Next.js 編譯器](/docs/architecture/nextjs-compiler) 獲取更多資訊。
*   `next/image` 匯入已改名為 `next/legacy/image`。`next/future/image` 匯入已改名為 `next/image`。有一個 [代碼轉換](/docs/app/building-your-application/upgrading/codemods#next-image-to-legacy-image) 可安全且自動地重命名您的匯入。
*   `next/link` 子項目不再可以是 `<a>`。添加 `legacyBehavior` 屬性以使用舊版行為或移除 `<a>` 以升級。有一個 [代碼轉換](/docs/app/building-your-application/upgrading/codemods#new-link) 可自動升級您的程式碼。
*   當 `User-Agent` 是機器人時，路由不再會被預取。
*   `next.config.js` 已棄用的 `target` 選項已移除。
*   支援的瀏覽器已更改為放棄 Internet Explorer 並鎖定現代瀏覽器。您仍可使用 Browserslist 來變更目標瀏覽器。
    *   Chrome 64+
    *   Edge 79+
    *   Firefox 67+
    *   Opera 51+
    *   Safari 12+

要了解更多信息，請查閱 [升級指南](/docs/app/building-your-application/upgrading)。
