我們正在為 Next.js 開發一個簡單且強大的緩存模型。在之前的文章中，我們談論了 [關於我們與緩存的旅程](/blog/our-journey-with-caching) 以及我們是如何來到 'use cache' 指令的。

本文將討論 'use cache' 的 API 設計和好處。

什麼是 'use cache'？[](#what-is-use-cache)
-------------------------------------------

「use cache」讓您的應用程式透過按需緩存數據或組件而變得更快。

它是一個 JavaScript「指令」——一個您代碼中添加的字面量字符串——這會向 Next.js 編譯器發出信號進入不同的「邊界」。例如，從服務器到客戶端。

這與 React 指令類似，例如 'use client' 和 'use server'。指令是編譯器指示，定義代碼應該在哪裡運行，讓框架為您優化並協調各個部分。

它是如何運作的？[](#how-does-it-work)
--------------------------------------

讓我們從一個簡單的示例開始：

```
async function getUser(id) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

在後台，Next.js 由於 'use cache' 指令將此代碼轉換為服務器函數。在編譯期間，此緩存條目的「依賴項」被找到並用作緩存鍵的一部分。

例如，`id` 成為緩存鍵的一部分。如果我們多次調用 `getUser(1)`，我們返回緩存服務器函數的記憶化輸出。更改此值將會在緩存中創建新條目。

讓我們來看一個示例，在其中一個服務器組件中使用緩存函數並有一個 [閉包](https://v0.app/chat/5kD47RIecQK?b=b_rCP4CvfbFFW)。

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    'use cache';
    return await db
      .select()
      .from(notifications)
      .limit(limit)
      .offset(index)
      .where(eq(notifications.userId, id));
  }
 
  return <User notifications={getNotifications} />;
}
```

這個示例更複雜。你能指出所有需要成為緩存鍵一部分的依賴項嗎？

參數 `index` 和 `limit` 是合理的——如果這些值改變，我們會選擇通知的不同部分。但是，用戶 `id` 呢？它的值來自父組件。

編譯器能夠理解 `getNotifications` 也依賴於 `id`，並且它的值自動包含在緩存鍵中。這防止了整個類別的緩存問題，即由於緩存鍵中的依賴項錯誤或缺失而產生的問題。

為什麼不使用緩存函數？[](#why-not-use-a-cache-function)
--------------------------------------------------------------

讓我們重新檢查最後一個示例。我們是應該使用 `cache()` 函數而不是指令嗎？

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    return await cache(async () => {
      return await db
        .select()
        .from(notifications)
        .limit(limit)
        .offset(index)
        // Oops! Where do we include id in the cache key?
        .where(eq(notifications.userId, id));
    });
  }
 
  return <User notifications={getNotifications} />;
}
```

`cache()` 函數將無法查看閉包並看到 `id` 值應該作為緩存鍵的一部分。您需要手動指定 `id` 是您鍵的一部分。如果您忘記這樣做，或執行錯誤，您可能會面臨緩存碰撞或數據過期的風險。

閉包可以捕捉各種局部變量。一個幼稚的方法可能會无意中烘焙進（或遺漏）您原本未打算包含的變量。這會導致緩存錯誤的數據，或者如果敏感信息洩露到緩存鍵中，可能會危及緩存投毒。

'use cache' 為編譯器提供了足夠的上下文以安全處理閉包並正確生成緩存鍵。一個僅運行時的解決方案，例如 `cache()`，將需要您手動執行所有操作——很容易犯錯。相比之下，指令可以被靜態分析以在底層可靠地處理所有您的依賴項。

未序列化的輸入值是如何處理的？[](#how-are-non-serialized-input-values-handled)
--------------------------------------------------------------------------------------------

我們有兩種不同類型的輸入值用於緩存：

*   **可序列化**: 在此，「可序列化」意味著輸入可以轉換為穩定的基於字符串的格式而不丟失含義。雖然許多人首先想到的是 `JSON.stringify`，但我們實際上使用 React 的序列化（例如，通過服務器組件）來處理更廣泛範圍的輸入——包括 Promise、循環數據結構和其他複雜對象。這超出了純 JSON 所能做到的範圍。
*   **不可序列化**: 這些輸入不是緩存鍵的一部分。當我們嘗試緩存這些值時，我們返回服務器「引用」。此引用隨後被 Next.js 用於在運行時還原原始值。

讓我們假設我們記得將 `id` 包含在緩存鍵中：

```
await cache(async () => {
  return await db
    .select()
    .from(notifications)
    .limit(limit)
    .offset(index)
    .where(eq(notifications.userId, id));
}, [id, index, limit]);
```

如果輸入值可以序列化，這就可以工作。但是如果 `id` 是 React 元素或更復雜的值，我們必須手動序列化輸入鍵。考慮一個服務器組件，它根據 `id` prop 獲取當前用戶：

```
async function Profile({ id, children }) {
  'use cache';
  const user = await getUser(id);
 
  return (
    <>
      <h1>{user.name}</h1>
      {/* Changing children doesn’t break the cache... why? */}
      {children}
    </>
  );
}
```

讓我們逐步說明這是如何運作的：

1.  在編譯期間，Next.js 看到 'use cache' 指令並轉換代碼以創建一個支持緩存的特殊服務器函數。編譯期間不會發生緩存，而是 Next.js 正在設置運行時緩存所需的機制。
2.  當您的代碼調用「緩存函數」時，Next.js 會序列化函數的參數。任何無法直接序列化的內容，例如 JSX，都會被「引用」替換符替換。
3.  Next.js 檢查是否存在具有給定序列化參數的緩存結果。如果未找到結果，函數將計算要緩存的新值。
4.  函數完成後，返回值被序列化。返回值中不可序列化的部分會重新變成引用。
5.  調用緩存函數的代碼會反序列化輸出並評估引用。這使得 Next.js 能夠將引用與其實際對象或值交換，意味著像 `children` 這樣不可序列化的輸入可以保留其原始、未緩存的值。

這意味著我們可以安全地僅緩存 `<Profile>` 組件而不緩存子組件。在後續渲染中，不會再次調用 `getUser()`。`children` 的值可能是動態的，或者是具有不同緩存生命週期的單獨緩存元素。這就是可組合緩存。

這看起來是不是很熟悉…[](#this-seems-familiar)
--------------------------------------------

如果您想「這感覺像服務器客戶端組合的同一種模型」——您完全正確。這有時被稱為「甜甜圈」模式：

*   甜甜圈**外層**部分是一個處理數據獲取或重邏輯的服務器組件。
*   中間的**孔**是一個可能有某些交互性的子組件

app/page.tsx

```
export default function Page() {
  return (
    <ServerComponent>
      {/* Create a hole to the client */}
      <ClientComponent />
    <ServerComponent />
  );
}
```

'use cache' 也是一樣的。甜甜圈是外層組件的緩存值，孔是在運行時填充的引用。這就是為什麼更改 `children` 不會使整個緩存輸出無效。子組件只是一個稍後填充的引用。

那麼標籤和無效化呢？[](#what-about-tagging-and-invalidation)
----------------------------------------------------------------------------

您可以使用不同的 [配置文件](/docs/app/api-reference/functions/cacheLife) 定義緩存的生命週期。我們包含一組默認配置文件，但如果您希望，也可以定義自己的自定義值。

```
async function getUser(id) {
  'use cache';
  cacheLife('hours');
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

要無效化特定的緩存條目，您可以 [標記緩存](/docs/app/api-reference/functions/cacheTag)，然後調用 `revalidateTag()`。一個強大的模式是您可以在獲取數據（例如來自 CMS）後標記緩存：

```
async function getPost(postId) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/blog/${postId}`);
  let data = await res.json();
  cacheTag(postId, data.authorId);
  return data;
}
```

簡單且強大[](#simple-and-powerful)
-------------------------------------------

我們的目標是讓 'use cache' 編寫緩存邏輯既簡單又強大。

*   **簡單**: 您可以使用局部推理創建緩存條目。您不需要擔心全局副作用，例如遺漏的緩存鍵條目或對代碼庫其他部分的不期變更。
*   **強大**: 您可以緩存比靜態可分析代碼更多的東西。例如，在運行時可能改變的值，但仍希望在評估後緩存輸出結果。

'use cache` 仍然是 Next.js 中的**實驗性**功能。在測試時，我們期待您的早期反饋。
