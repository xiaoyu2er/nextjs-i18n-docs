我們正在為 Next.js 的未來奠定基礎，12.2 版本帶來：

*   [**Middleware (穩定):**](#middleware-stable) 為您的整個應用程式提供動態路由。
*   [**On-Demand ISR (穩定):**](#on-demand-incremental-static-regeneration-stable) 無需重新部署即可更新內容。
*   [**Edge API Routes (實驗性):**](#edge-api-routes-experimental) 高性能 API 端點。
*   [**Edge SSR (實驗性):**](#edge-server-rendering-experimental) 在 Edge 伺服器渲染您的應用程式。
*   [**SWC Plugins (實驗性):**](#swc-plugins-experimental) 使用您自己的插件擴展編譯。
*   [**`next/image` 的改進:**](#improvements-to-nextimage) 包含新的 `next/future/image` 元件。

透過執行 `npm i next@latest` 立即更新。

Middleware (穩定)[](#middleware-stable)
-----------------------------------------

我們很興奮地宣佈，Middleware 在 12.2 中已穩定，並基於用戶反饋更新並優化了 API。

Middleware 允許你在請求完成之前運行代碼。根據進入的請求，你可以通過重寫、重定向、添加標頭或設置 Cookie，來修改響應。

middleware.ts

```
import { NextRequest, NextResponse } from 'next/server';
 
// If the incoming request has the "beta" cookie
// then we'll rewrite the request to /beta
export function middleware(req: NextRequest) {
  const isInBeta = JSON.parse(req.cookies.get('beta') || 'false');
  req.nextUrl.pathname = isInBeta ? '/beta' : '/';
  return NextResponse.rewrite(req.nextUrl);
}
 
// Supports both a single value or an array of matches
export const config = {
  matcher: '/',
};
```

要更新到 Middleware 的最新 API 變更，請參閱 [遷移指南](/docs/messages/middleware-upgrade-guide)。

在 [Vercel](https://vercel.com/edge) 上免費體驗 Middleware，或在使用 `next start` [自託管](/docs/pages/building-your-application/deploying#self-hosting) 時進行嘗試。

On-Demand Incremental Static Regeneration (穩定)[](#on-demand-incremental-static-regeneration-stable)
-------------------------------------------------------------------------------------------------------

按需增量靜態生成（ISR）允許你在不需要重新部署的情況下更新網站內容。這使得當頭部 CMS 或電商平台數據發生變化時，立即更新你的網站變得容易。這是由社區請求次數最多的功能之一，我們很興奮它現在已經穩定。

pages/api/revalidate.js

```
export default async function handler(req, res) {
  // Check for secret to confirm this is a valid request
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' });
  }
 
  try {
    await res.revalidate('/path-to-revalidate');
    return res.json({ revalidated: true });
  } catch (err) {
    // If there was an error, Next.js will continue
    // to show the last successfully generated page
    return res.status(500).send('Error revalidating');
  }
}
```

增量靜態生成適用於支持 [Next.js Build API](/docs/pages/building-your-application/deploying#nextjs-build-api)（`next build`）的任何提供者。當部署到 [Vercel](https://vercel.com/) 時，將頁面推送到 Edge 後，按需重新驗證的全球傳播時間約為 ~300 毫秒。

更多資訊，請查看 [文件](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration) 或 [我們的演示](https://on-demand-isr.vercel.app/) 以查看按需重新驗證的實際運作。

Edge API Routes (實驗性)[](#edge-api-routes-experimental)
---------------------------------------------------------------

Next.js 現在也支持在 API Routes 中使用 [Edge Runtime](/docs/pages/api-reference/edge)。Edge Runtime 是比 Node.js 輕量級的運行時，提供低延遲的快速啟動。此外，Edge API Routes 支持從伺服器串流傳輸響應。

你可以在 `config` 中為 API Route 設定 runtime，指定 `nodejs`（預設）或 `experimental-edge`：

pages/api/hello.js

```
import type { NextRequest } from 'next/server';
 
export default (req: NextRequest) => {
  return new Response(`Hello, from ${req.url} I'm now an Edge API Route!`);
};
 
export const config = {
  runtime: 'experimental-edge',
};
```

由於 Edge Runtime 是輕量級的，為了適應快速啟動，它有一些限制——例如，它不支持 `fs` 等 Node.js 專屬 API。因此，API Routes 的預設 runtime 仍為 `nodejs`。

更多資訊，請查看 [文件](/docs/pages/building-your-application/routing/api-routes)。

Edge 伺服器渲染（實驗性）[](#edge-server-rendering-experimental)
---------------------------------------------------------------------------

Next.js 現在支持使用 [Edge Runtime](/docs/pages/api-reference/edge) 進行伺服器渲染。

如上所述，Edge Runtime 是比 Node.js 輕量級的運行時，提供低延遲的快速啟動。搭配 React 18 使用時，它為 Pages 啟用伺服器渲染串流傳輸。

Next.js 使用 Node.js 作為伺服器端渲染頁面的預設 runtime。從 12.2 開始，如果你使用 React 18，可以選擇使用 Edge Runtime。

你可以在 `next.config.js` 中全域設定 runtime，指定 `nodejs` 或 `experimental-edge`：

next.config.js

```
module.exports = {
  experimental: {
    runtime: 'experimental-edge',
  },
};
```

更改預設頁面 runtime 會影響所有頁面，包括 [SSR 串流](/docs/app/building-your-application/routing/loading-ui-and-streaming) 和 [Server Components](/docs/getting-started/react-essentials) 功能。你也可以基於單個頁面覆蓋此預設，透過匯出 `runtime` 配置：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
export const config = {
  runtime: 'nodejs',
};
 
export default function Home() {}
```

你可以在運行時通過查看 `process.env.NEXT_RUNTIME` 環境變量來檢測你使用的是哪個 runtime，並在 webpack 編譯期間檢查 `options.nextRuntime` 變量。

更多資訊，請查看 [文件](/docs/pages/building-your-application/rendering/edge-and-nodejs-runtimes)。

「`next/image`」的改進 [](#improvements-to-nextimage)
----------------------------------------------------------

### `next/future/image` 元件（實驗性）[](#nextfutureimage-component-experimental)

我們收到了您對目前 Image 元件的反饋，很興奮能分享新 `next/image` 的早期預覽。這個新的改進版 Image 元件需要更少的前端 JavaScript，並簡化了你樣式化圖片的方式：

*   只渲染單一的 `<img>`，不含 `<div>` 或 `<span>` 包裝
*   加入對標準 `style` prop 的支持
*   移除 `layout`, `objectFit`, 和 `objectPosition` props，改用 `style` 或 `className`
*   移除 `IntersectionObserver` 實現，改用 [原生惰性加載](https://caniuse.com/loading-lazy-attr)
*   移除 `loader` 配置，改用 `loader` prop
*   注意：尚未有 `fill` 模式，因此需要 `width` 與 `height` prop

這提升了性能，因為原生的 `loading="lazy"` 不需要等待 React hydration 和客戶端 JavaScript。

更多資訊，請查看 [文件](/docs/pages/building-your-application/optimizing/images)。

### Remote Patterns (實驗性)[](#remote-patterns-experimental)

`next/image` 現在支持實驗性配置選項 `remotePatterns`，允許你在使用內建的 Image Optimization API 時為遠端圖片指定通配符。這允許比現有 [`images.domains`](http://images.domains) 配置更強大的匹配，後者僅對域名進行精確匹配。

next.config.js

```
module.exports = {
  experimental: {
    images: {
      remotePatterns: [
        {
          // The `src` property hostname must end with `.example.com`,
          // otherwise this will respond with 400 Bad Request.
          protocol: 'https',
          hostname: '**.example.com',
        },
      ],
    },
  },
};
```

更多資訊，請查看 [文件](/docs/pages/api-reference/components/image#remotepatterns)。

### 停用圖像優化 [](#disable-image-optimization)

零配置的 Image Optimization API 禁止使用 `next export`，因為它需要伺服器在請求時即時優化圖像。直到今天，針對 `next export` 的用戶需要配置 `loader` 使用第三方圖像優化提供者，但如果沒有可用提供者，則沒有明確解決方案。從今天開始，`next export` 用戶可以使用新的配置屬性地停用所有 `next/image` 實例的圖像優化：

next.config.js

```
module.exports = {
  experimental: {
    images: {
      unoptimized: true,
    },
  },
};
```

SWC Plugins (實驗性)[](#swc-plugins-experimental)
-------------------------------------------------------

[Next.js Compiler](/docs/architecture/nextjs-compiler) 使用 [SWC](https://swc.rs/) 來轉換和壓縮您的 JavaScript 代碼以供生產環境使用。SWC 於 Next.js 12.0 引入，以改進本地開發和構建性能。

現在你可以添加插件（使用 WebAssembly 編寫）來自定義編譯期間的 SWC 轉換行為：

next.config.js

```
module.exports = {
  experimental: {
    swcPlugins: [
      ['css-variable/swc', { displayName: true, basePath: __dirname }],
    ],
  },
};
```

更多資訊，請查看 [文件](https://github.com/vercel/next.js/blob/canary/docs/architecture/nextjs-compiler.md#experimental-swc-plugin-support)。

React 18 支援改進 [](#react-18-support-improvements)
---------------------------------------------------------------

*   改進了對 `styled-components` 和 `emotion` 等 CSS-in-JS 庫的支持，體驗更流暢且無需破壞性變更。
*   AMP 和 HTML 後期優化（CSS、字體優化）現在得到支持。
*   `next/head` 現在支援 React 18。
*   Next.js 的路由通告器（用於向讀屏軟件和其他輔助技術正確宣告頁面轉換）現在支持 React 18。

[其他改進](#other-improvements)
-----------------------------------------

*   Next.js Compiler 支持 Emotion 轉換。`@emotion/babel-plugin` 的大多功能現在已支持，除非你使用 `importMap`，否則可以移除。更多資訊，請查看 [文件](/docs/architecture/nextjs-compiler#emotion)。
*   Next.js Compiler 對 `styled-components` 轉換有更好的支持，允許自定義預設選項，包括 `cssProp` 選項。更多資訊，請查看 [文件](/docs/architecture/nextjs-compiler#styled-components)。
*   對 JavaScript ES Modules 有更好的支持，以便 `next/image` 和 `next/link` 等元件可以正確 `import`。
*   `next/link` 不再要求手動添加 `<a>` 作為子元件。現在你可以以向后兼容的方式 [選擇此行為](https://github.com/vercel/next.js/pull/36436)。
*   我們添加了實驗性支持，通過修改 `browsersList` 僅發送現代 JavaScript。你可以通過在 `next.config.js` 的 `experimental` 選項中設置 `browsersListForSwc: true` 和 `legacyBrowsers: false` 來選擇此行為。
*   新的 `@swc/helpers` 優化防止各個打包中的重複，在最小配置下減少約 `2KB` 打包大小，大應用中更多。
*   我們顯著減少了 Next.js 安裝大小。這是通過將 monorepo 移到 `pnpm` 實現的，這允許我們在創建用於我們所需的預編譯版本時刪除重複包。這導致安裝大小減少了 14MB。
*   為持續改進自託管 Next.js，我們正將實驗性 `outputStandalone: true` 配置穩定為 `output: 'standalone'`。此配置通過僅包含必要文件和 assets 大幅減少部署大小，包括在構建的部署包中移除安裝所有 `node_modules` 的需求。此配置可在我們的 [`with-docker` 示例](https://github.com/vercel/next.js/blob/canary/examples/with-docker/README.md) 中看到。

Layouts RFC 與高級路由支援 [](#layouts-rfc--advanced-routing-support)
--------------------------------------------------------------------------------

如果錯過了，我們上個月宣佈了 [Layouts RFC](/blog/layouts-rfc)——自 2016 年推出以來 Next.js 最大的一次更新，包括：

*   **嵌套佈局：** 使用嵌套路由構建複雜應用程式。
*   **針對 Server Components 設計：** 優化子樹導航。
*   **改進的數據獲取：** 在佈局中獲取數據，同時避免瀑布流。
*   **使用 React 18 功能：** 串流、過渡和 Suspense。
*   **客戶端與伺服器路由：** 以伺服器為中心的路由，具有 _SPA-like_ 行為。
*   **100% 可增量採用：** 無破壞性變更，因此你可以逐步採用。
*   **高級路由規約：** 離屏存儲、即時過渡等。

更多資訊，請查看 [RFC](/blog/layouts-rfc) 或 [提供反饋](https://github.com/vercel/next.js/discussions/37136)。
