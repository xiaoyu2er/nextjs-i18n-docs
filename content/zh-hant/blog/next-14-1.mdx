Next.js 14.1 包含開發者體驗改進，包括：

* [**自託管改進：**](#improved-self-hosting) 新的文件與自訂快取處理器
* [**Turbopack 改進：**](#turbopack-improvements) `next dev --turbo` 通過 5,600 項測試
* [**開發者體驗 (DX) 改進：**](#developer-experience-improvements) 改進的錯誤訊息、`pushState` 和 `replaceState` 支援
* [**並行與介接路由：**](#parallel--intercepted-routes) 基於反饋的 20 項錯誤修復
* [**`next/image` 改進：**](#nextimage-support-for-picture-and-art-direction) `<picture>`、藝術導向與深色模式支援

立即升級或開始使用：

npx create-next-app@latest
```

自託管改進 [](#improved-self-hosting)
-----------------------------------------------

我們已收到您的反饋，希望能更清晰地說明如何使用 Node.js 伺服器、Docker 容器或靜態導出来自建 Next.js。我們對以下自託管文件進行了全面 overhaul：

* [運行情境環境變數](/docs/app/building-your-application/deploying#environment-variables)
* [ISR 自訂快取設定](/docs/app/building-your-application/deploying#caching-and-isr)
* [自訂影像最佳化](/docs/app/building-your-application/deploying#image-optimization)
* [中間層 (Middleware)](/docs/app/building-your-application/deploying#middleware)

隨著 Next.js 14.1，我們也已穩定化提供自訂快取處理器服務，用於靜態增量重新編譯 (Incremental Static Regeneration) 以及應用程式路由 (App Router) 更細粒度的資料快取：

next.config.js

```
module.exports = {
  cacheHandler: require.resolve('./cache-handler.js'),
  cacheMaxMemorySize: 0, // disable default in-memory caching
};
```

在使用容器編排平台（如 Kubernetes）時，使用自訂快取處理器對於自託管至關重要，因為每個 Pod 都會有一份快取副本。使用自訂快取處理器可以確保所有托管您 Next.js 應用程式的 Pod 之間的一致性。

例如，您可以將暫存的值儲存在任何地方，像是 Redis 或 Memcached。我們感謝 [`@neshca`](https://github.com/caching-tools/next-shared-cache) 的 [Redis 快取處理器配接器](https://github.com/vercel/next.js/tree/canary/examples/cache-handler-redis) 和範例。

Turbopack 改進 (Turbopack Improvements) [](#turbopack-improvements)
-------------------------------------------------

我們持續專注於本地 Next.js 開發的穩定性與效能：

* **穩定性：** Turbopack 通過所有 Next.js 開發測試套組，並進行 Vercel 應用程式的內部測試 (dogfooding)
* **效能：** 改進 Turbopack 初始編譯時間與 Fast Refresh 時間
* **記憶體使用量：** 改進 Turbopack 記憶體使用量

我們計劃在未來的發布版本中穩定 `next dev --turbo`，目前仍為選定加入。

### 穩定性 [](#reliability)

Next.js 搭配 Turbopack 現在通過了 **5,600 項開發測試 (94%)**，比上次更新多了 600 項。您可以在 [areweturboyet.com](https://areweturboyet.com/) 追蹤進度。

我們持續在所有 Vercel 的 Next.js 應用程式上進行內部測試 (dogfooding) `next dev --turbo`，包括 [vercel.com](https://vercel.com) 和 [v0.app](https://v0.app)。在這些應用程式上工作的一切工程師都每天使用 Turbopack。

我們發現並修復了許多使用 Turbopack 的大型 Next.js 應用程式的問題。對於這些修復，我們已在 Next.js 現有的開發測試套組中新增了測試。

### [效能表現](#performance)

對於 [vercel.com](https://vercel.com) 這樣的大型 Next.js 應用程式，我們看到了：

* 本地伺服器啟動速度高達 **76.7% 更快**
* 使用 Fast Refresh 進行程式碼更新速度快達 **96.3%**
* 無快取的初始路由編譯速度快達 **45.8%** (Turbopack 尚未有磁碟快取)

在 [v0.app](https://v0.app) 中，我們發現在 Turbopack 中發現和打包 React 客戶端元件 (React Client Components) 的方式有優化機會，結果**初始編譯時間速度快達 61.5%**。這種效能改進在 [vercel.com](https://vercel.com) 也有觀察到。

### 未來改進 [](#future-improvements)

Turbopack 目前具記憶體快取 (in-memory caching)，可改善 Fast Refresh 的增量編譯時間。

然而，重新啟動 Next.js 開發伺服器時，目前快取尚未保留。Turbopack 效能的下一大步是**磁碟快取 (disk caching)**，這將允許在重新啟動開發伺服器時保留快取。

### 改進的錯誤訊息與 Fast Refresh [](#improved-error-messages-and-fast-refresh)

我們知道清晰的錯誤訊息對您的本地開發體驗至關重要。我們進行了一系列修復，以改善執行 `next dev` 時看到的堆疊追蹤 (stack traces) 和錯誤訊息的品質。

* 先前顯示 `webpack-internal` 等打包程式錯誤的錯誤訊息，現在現在正確顯示錯誤的來源代碼及受影響的檔案。
* 在客戶端元件中看到錯誤後，然後在編輯器中修復錯誤，Fast Refresh 不會清除錯誤畫面。這需要重新整頁重新載入。我們已修復許多這些情況。例如，嘗試從客戶端元件匯出 `metadata`。

### `window.history.pushState` 與 `window.history.replaceState`[](#windowhistorypushstate-and-windowhistoryreplacestate)

應用程式路由 (App Router) 現在允許使用原生的 [`pushState`](https://developer.mozilla.org/en-US/docs/Web/API/History/pushState) 和 [`replaceState`](https://developer.mozilla.org/en-US/docs/Web/API/History/replaceState) 方法來更新瀏覽器歷史堆疊，而無需重新載入頁面。

`pushState` 和 `replaceState` 呼叫整合進 Next.js 應用程式路由 (App Router)，允許您與 [`usePathname`](/docs/app/api-reference/functions/use-pathname) 和 [`useSearchParams`](/docs/app/api-reference/functions/use-search-params) 同步。

這在需要立即更新 URL 時很有幫助，例如儲存狀態像是篩選器、排序順序，或其他希望在重新載入之間持續的資訊。

```
'use client';
 
import { useSearchParams } from 'next/navigation';
 
export default function SortProducts() {
  const searchParams = useSearchParams();
 
  function updateSorting(sortOrder: string) {
    const params = new URLSearchParams(searchParams.toString());
    params.set('sort', sortOrder);
    window.history.pushState(null, '', `?${params.toString()}`);
  }
 
  return (
    <>
      <button onClick={() => updateSorting('asc')}>Sort Ascending</button>
      <button onClick={() => updateSorting('desc')}>Sort Descending</button>
    </>
  );
}
```

了解更多關於使用原生的 [History API](/docs/app/building-your-application/routing/linking-and-navigating#using-the-native-history-api) 搭配 Next.js。

### 資料快取記錄 [](#data-cache-logging)

為了在執行 `next dev` 時提高您的 Next.js 應用程式中暫存資料的可觀察性 (observability)，我們對 `logging` [設定選項](/docs/app/api-reference/next-config-js/logging) 進行了多項改進。

您現在可以顯示是否有快取 `HIT` 或 `SKIP`，以及請求的完整 URL：

終端機

```
GET / 200 in 48ms
 ✓ Compiled /fetch-cache in 117ms
 GET /fetch-cache 200 in 165ms
  │ GET https://api.vercel.app/products/1 200 in 14ms (cache: HIT)
 ✓ Compiled /fetch-no-store in 150ms
 GET /fetch-no-store 200 in 548ms
  │ GET https://api.vercel.app/products/1 200 in 345ms (cache: SKIP)

這可以透過 `next.config.js` 啟用：

```
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
};
```

`next/image` 對 `<picture>` 與藝術導向 (Art Direction) 的支援 [](#nextimage-support-for-picture-and-art-direction)

Next.js 影像元件現在支援更多進階使用案例，透過 `getImageProps()` (已穩定) 不需要直接使用 `<Image>`。這包括：

*   與 [`background-image`](https://developer.mozilla.org/docs/Web/CSS/background-image) 或 [`image-set`](https://developer.mozilla.org/docs/Web/CSS/image/image-set) 配合使用
*   與 canvas [`context.drawImage()`](https://developer.mozilla.org/docs/Web/API/Canvas_API/Tutorial/Using_images) 或 `new Image()` 配合使用
*   與 [`<picture>`](https://developer.mozilla.org/docs/Web/HTML/Element/picture) 媒體查詢配合實現 [藝術指導](https://developer.mozilla.org/docs/Learn/HTML/Multimedia_and_embedding/Responsive_images#art_direction) 或 淺色/深色模式圖片

```
import { getImageProps } from 'next/image';
 
export default function Page() {
  const common = { alt: 'Hero', width: 800, height: 400 };
  const {
    props: { srcSet: dark },
  } = getImageProps({ ...common, src: '/dark.png' });
  const {
    props: { srcSet: light, ...rest },
  } = getImageProps({ ...common, src: '/light.png' });
 
  return (
    <picture>
      <source media="(prefers-color-scheme: dark)" srcSet={dark} />
      <source media="(prefers-color-scheme: light)" srcSet={light} />
      <img {...rest} />
    </picture>
  );
}
```

了解更多關於 [`getImageProps()`](/docs/app/api-reference/components/image#getimageprops)。

並行與介接路由 [](#parallel--intercepted-routes)
--------------------------------------------------------------

在 Next.js 14.1 中，我們針對並行與介接路由進行了 **20 項改進**。

過去兩個發布版本中，我們專注於改進 Next.js 的效能與穩定性。現在我們已能根據您的反饋，針對 [並行路由](https://nextjs.org/docs/app/building-your-application/routing/parallel-routes) 與 [介接路由](https://nextjs.org/docs/app/building-your-application/routing/intercepting-routes) 進行許多改進。 notably，我們已新增萬用路由 (catch-all routes) 和伺服器動作 (Server Actions) 的支援。

* **並行路由**允許您在同一版面配置 (layout) 中同時或條件式顯示一或多個頁面。對於應用程式高度動態的部分，例如社交網站上的儀表板和動態訊息串列，並行路由可用於實現複雜的導航模式。
* **介接路由**允許您在當前版面配置中載入來自應用程式其他部分的路由。例如，當點擊動態訊息串列中的照片時，您可以在模態視窗 (modal) 中顯示照片，覆蓋在動態訊息串列之上。在此情況下，Next.js 介接 `/photo/123` 路由，掩碼 URL，並將其覆蓋在 `/feed` 上。

了解更多關於 [並行路由](https://nextjs.org/docs/app/building-your-application/routing/parallel-routes) 與 [介接路由](https://nextjs.org/docs/app/building-your-application/routing/intercepting-routes) 或 [檢視範例](https://github.com/vercel/nextgram)。
