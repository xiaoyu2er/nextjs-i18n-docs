Next.js 16.2 包含多項專為 AI 輔助開發設計的改進。這些變更讓 AI 代理更容易了解您的專案、從終端機除錯問題、並檢查正在執行的應用程式——所有操作都無需使用瀏覽器。

*   [**準備好用於 AI 的 `create-next-app`**](#ai-ready-project-setup): 預先建立準備好用於 AI 的專案
*   [**瀏覽器日誌轉發**](#browser-log-forwarding): 將瀏覽器錯誤轉發至終端機，讓 AI 代理協助除錯
*   [**開發伺服器鎖定檔案**](#dev-server-lock-file): 當第二個開發伺服器嘗試啟動時顯示可執行的錯誤訊息
*   [**實驗性 Agent 開發工具**](#experimental-agent-devtools): 讓 AI 代理能透過終端機存取 React DevTools 和 Next.js 診斷工具

準備好用於 AI 的 `create-next-app` [](#agent-ready-create-next-app)
-------------------------------------------------------------

`create-next-app` 預設包含 `AGENTS.md` 檔案，讓 AI 程式設計代理從專案開始就能存取版本對應的 Next.js 文件。

這是基於我們對 [`AGENTS.md`](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals) 的研究，該研究發現，讓代理存取打包的文件能讓 Next.js 評估達到 100% 通過率——優於技能為本的方法，後者最高僅達 79%。關鍵要點：隨時可用的環境比按需檢查找運作得更好，因為代理經常無法定位何時需要搜尋文件。

`AGENTS.md` 檔案是一個簡短的指示，告知代理在編寫任何程式碼之前先讀取打包在 `node_modules/next/dist/docs/` 的文件。Next.js npm 套件現在包含完整的文件作為純 Markdown 檔案，讓代理能在本地區取得準確的版本對應參考，而無需抓取外部資料。

對於現有專案，設定取決於您的 Next.js 版本。

**在 16.2 或更高版本上**，文件已打包在 `next` 套件中。將這兩個檔案新增至您專案的根目錄：

AGENTS.md

```
<!-- BEGIN:nextjs-agent-rules -->
 
# Next.js: ALWAYS read docs before coding
 
Before any Next.js work, find and read the relevant doc in `node_modules/next/dist/docs/`. Your training data is outdated — the docs are the source of truth.
 
<!-- END:nextjs-agent-rules -->
```

註解標記區隔了 Next.js 控管的區段。您可以在它們之外新增您專用的專案指令——未來的更新只會取代標記之間的内容。

`CLAUDE.md` 是 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 的指示檔案。`@` 指示它包含 `AGENTS.md` 作為其他上下文：

CLAUDE.md

```
@AGENTS.md
```

**在更早的版本上**，使用 codemod 自動產生這些檔案：

終端機

```
npx @next/codemod@latest agents-md
```

有關更多詳細資訊，請參閱 [AI 代理設定指南](/docs/app/guides/ai-agents#existing-projects)。

瀏覽器日誌轉發 [](#browser-log-forwarding)
-------------------------------------------------

Next.js 現在在開發期間預設將瀏覽器錯誤轉發至終端機，因此您可以查看客戶端錯誤而無需切換到瀏覽器控制台。這對於主要透過終端機操作且無法存取瀏覽器控制台的 AI 代理特別有幫助。

預設情況下，只有錯誤會轉發至終端機。您可以在 `next.config.ts` 中使用 `logging.browserToTerminal` 來控管轉發層級：

next.config.ts

```
const nextConfig = {
  logging: {
    browserToTerminal: true,
    // 'error' — errors only (default)
    // 'warn'  — warnings and errors
    // true    — all console output
    // false   — disabled
  },
};
 
export default nextConfig;
```

開發伺服器鎖定檔案[](#dev-server-lock-file)
---------------------------------------------

Next.js 現在將正在執行的開發伺服器的 PID、連接埠與 URL 寫入 `.next/dev/lock` 檔案。當第二個 `next dev` 進程在相同專案目錄中啟動時，Next.js 會讀取鎖定檔案並列印具可操作性的錯誤：

終端機

```
Error: Another next dev server is already running.
 
- Local:        http://localhost:3000
- PID:          12345
- Dir:          /path/to/project
- Log:          .next/dev/logs/next-development.log
 
Run kill 12345 to stop it.
```

這對於 AI 編碼代理特別有益，因為它們經常在不知道伺服器已執行的情況下嘗試啟動 `next dev`。結構化的錯誤資訊給出了代理停止既有進程所需的 PID，或連接至其的 URL — 無需手動干預。

鎖定檔案也可防止兩個 `next build` 進程同時執行，否則可能會損壞生成檔案。

實驗性代理開發人員工具[](#experimental-agent-devtools)
-----------------------------------------------------------

上述功能協助代理了解您的專案與除錯問題。[`@vercel/next-browser`](https://github.com/vercel-labs/next-browser) 則進一步透過讓代理檢視執行中的 Next.js 應用程式延伸了這項功能。

`next-browser` 是一個實驗性 CLI，它公開瀏覽器層級資料 — 螢幕擷圖、網路請求、控制台日誌 — 搭配來自 React DevTools 與 Next.js 開發控制臺的框架專有洞察，如元件樹、屬性、鉤子、[部分預渲染 (PPR)](/docs/app/getting-started/cache-components) 外殼與錯誤。所有內容透過 shell 命令以結構化文字回傳。

LLM 無法讀取 DevTools 面板，但它能執行 `next-browser tree`、解析輸出，並決定下一次要檢視什麼。每個命令都是針對持久化瀏覽器會話的單次請求，因此代理可以重複查詢應用程式，無需管理瀏覽器狀態。這將瀏覽器變成了代理可推理的對象，而非無法存取的 UI。

### 它目前能做到什麼[](#what-it-can-do-today)

功能集正在快速演變。截至此次版本，`next-browser` 支援：

*   **檢視 React 元件樹** — 檢視屬性、鉤子、狀態與來源對應檔案位置
*   **分析 PPR 外殼** — 識別靜態與動態區域，以及封鎖的 [懸吊邊界 (Suspense boundaries)](/docs/app/glossary#suspense-boundary)
*   **存取錯誤與日誌** — 從開發伺服器擷取生成與執行期問題
*   **監控網路活動** — 追蹤導航以來的請求，包含伺服器動作
*   **擷取視覺資料** — 拍攝螢幕擷圖或記錄載入影片條

### 開始使用[](#getting-started)

安裝它為一個 [技能](https://skills.sh)（AI 代理的可重覆功能）：

終端機

```
npx skills add vercel-labs/next-browser
```

然後在 Claude Code、Cursor 或任何支援技能的 AI 代理中輸入 `/next-browser`。CLI 會管理預先載入 React DevTools 的 Chromium 實體 — 無需瀏覽器設定。

### 範例：擴展靜態外殼[](#example-growing-the-static-shell)

透過 [部分預渲染 (PPR)](/docs/app/getting-started/cache-components)，Next.js 可以從邊緣立即提供 [靜態外殼](/docs/app/glossary#static-shell) — 您的頁面中不依賴每次請求資料的部分 — 然後串流其餘內容。放入靜態外殼的內容越多，使用者看到有意義頁面的速度就越快。

在實務上，單次的每次請求 fetch 可能無意中使整個頁面變成動態。考慮一下帶有訪客計數器的主頁：

app/blog/\[slug\]/page.tsx

```
export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map((post) => ({ slug: post.slug }));
}
 
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
  const views = await getVisitorCount(params.slug); // per-request
 
  return (
    <article>
      <h1>{post.title}</h1>
      <span>{views} views</span>
      <div>{post.content}</div>
    </article>
  );
}
```

每個 slug 都在 `generateStaticParams` 中列舉，因此主頁内容可以在編譯時預先產生。但 `getVisitorCount` 在每次請求時執行 — 而且因為它位於元件的頂層，它使 _整個頁面_ 變成動態。結果，整個頁面在載入骨架後等待，而非逐級串流。

代理可以使用 `next-browser` 診斷這個狀況。鎖定 PPR 模式僅顯示靜態外殼 — 在此情況下，`app/blog/[slug]/loading.tsx` 骨架，因為頁面中任何內容都未進入外殼：

終端機

```
next-browser ppr lock          # Enter PPR mode
next-browser goto /blog/hello  # The entire page is a loading skeleton
```

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-16-2-ai/ppr_before_light.png)![](/_next/image?url=https%3A%2F%2Fh8DxKfmAPhn8O0p3.public.blob.vercel-storage.com%2Fstatic%2Fblog%2Fnext-16-2-ai%2Fppr_before_dark.png&w=1920&q=75)

接著代理執行 `ppr unlock` 以找出出了什麼問題：

終端機

```
next-browser ppr unlock
```

終端機

```
# PPR Shell Analysis
# 1 dynamic hole, 1 static
 
blocked by:
  - getVisitorCount (server-fetch)
    owner: BlogPost at app/blog/[slug]/page.tsx:5
    next step: Push the fetch into a smaller Suspense leaf
```

報表明確告訴代理該做什麼：`getVisitorCount` 是封鎖者，它位於 `BlogPost` 中，修復方式是將其推入專屬的 [懸停邊界 (Suspense boundary)](/docs/app/glossary#suspense-boundary)。代理僅包裹計數器：

app/blog/\[slug\]/page.tsx

```
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
 
  return (
    <article>
      <h1>{post.title}</h1>
      <Suspense fallback={<span>— views</span>}>
        <VisitorCount slug={params.slug} />
      </Suspense>
      <div>{post.content}</div>
    </article>
  );
}
```

再次執行 `ppr lock` 確認外殼已擴大 — 主頁内容現在立即預先產生。僅訪客計數器顯示載入：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-16-2-ai/ppr_after_light.png)![](/_next/image?url=https%3A%2F%2Fh8DxKfmAPhn8O0p3.public.blob.vercel-storage.com%2Fstatic%2Fblog%2Fnext-16-2-ai%2Fppr_after_dark.png&w=1920&q=75)

`next-browser` 仍在演變，但它指向了一個未來，代理能像開發人員一樣除錯與優化應用程式。
