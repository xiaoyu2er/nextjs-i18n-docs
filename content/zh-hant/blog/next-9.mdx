經過 70 次 canary 發布後，我們很高興地推出 Next.js 9，特點如下：

*   **[內建零設定 TypeScript 支援](#built-in-zero-config-typescript-support)**：得益于自動 TypeScript 支援與整合類型檢查，讓您在建立應用程式時擁有更高的信心。
*   **[基於檔案系統的動態路由](#dynamic-route-segments)**：無需自訂伺服器即可透過檔案系統表達複雜的應用程式路由需求。
*   **[自動靜態優化](#automatic-static-optimization)**：預設建立結合 *伺服器端渲染 (SSR)* 與 *靜態預渲染* 的超高速網站，且不犧牲功能。
*   **[API 路由](#api-routes)**：利用熱重載與統一的建置流程，快速建立後端應用程式端點。
*   **[更多生產環境優化](#production-optimizations)**：藉由視窗內預取及其他優化措施，應用程式比以往更靈敏。

內建零設定 TypeScript 支援 [](#built-in-zero-config-typescript-support)
-----------------------------------------------------------------------------------

一年前 [Next.js 6](https://nextjs.org/blog/next-6) 透過名為 `@zeit/next-typescript` 的插件引入基本 TypeScript 支援。使用者還需要自訂其 `.babelrc` 並在 `next.config.js` 中啟用它。

配置後，插件允許 `.ts` 和 `.tsx` 檔案由 Next.js 建構。但是，它沒有整合類型檢查，且 Next.js 核心也沒有提供類型。這意味著需要一個維護在 DefinitelyTyped 中的社群套件，且該套件可能會與發布不同步。

在與許多現有和新使用者進行交流後，我們很清楚地了解到大多數使用者都非常有興趣使用 TypeScript。他們希望有一個更可靠且標準的解決方案，能夠輕鬆地將 TypeScript 整合到他們現有或新穎的程式碼庫中。

出於這個原因，我們開始將 TypeScript 支援整合到 Next.js 核心，改善開發體驗，同時讓它變得更快。

##### 自動設定 [](#automated-setup)

在 Next.js 中使用 TypeScript 來開始很簡單：將任何檔案、頁面或元件從 `.js` 重命名為 `.tsx`。然後，運行 `next dev`！

這將導致 Next.js 偵測到您的專案中正在使用 TypeScript。Next.js CLI 將引導您安裝 React 和 Node.js 所需的類型。

如果尚未存在，Next.js 還會建立預設的 `tsconfig.json` 與合理的預設值。此檔案允許在編輯器（如 [Visual Studio Code](https://code.visualstudio.com/)）中進行整合類型檢查。

動態路由區段 [](#dynamic-route-segments)
-------------------------------------------------

動態路由（也稱為 URL Slugs 或 Pretty/Clean URLs）是在 Next.js 發布後大約 2.5 年，GitHub 上第一個功能請求之一！

此問題在 Next.js 2.0 透過引入自訂伺服器 API 以程式化方式使用 Next.js 來「解決」。這允許將 Next.js 用作渲染引擎，啟用抽象化和將進入的 URL 映射到渲染某些頁面。

我們與使用者交談並檢視了許多他們的應用程式，發現許多應用程式都有自訂伺服器。出現了一種模式：使用自訂伺服器最根本的原因就是動態路由。

然而，自訂伺服器有其自身的陷阱：路由是在伺服器層級而非代理層處理，它作為單體進行部署和擴展，且容易出現效能問題。

由於自訂伺服器要求整個應用程式必須在一例中可用，因此部署到解決這些問題的 Serverless 環境通常很困難。Serverless 請求是在代理層路由，並獨立擴展/執行以避免效能瓶頸。

此外，我們相信我們可以提供更好的開發體驗 (DX)！Next.js 的大部分魔法是從您建立檔案 `pages/blog.js` 並突地擁有可透過 `/blog` 存取的頁面開始的。

為什麼使用者需要建立自己的伺服器並學習 Next.js 的程式化 API 以支援 `/blog/my-first-post` (`/blog/:id`) 這樣的 route？

基於此回饋和願景，我們開始調查 route 映射解決方案，由使用者已經知道的 `pages/` 資料夾驅動。

##### 建立動態路由頁面 [](#creating-a-dynamically-routed-page)

Next.js 支援以基本命名參數建立 routes，此模式由 [`path-to-regexp`](https://www.npmjs.com/package/path-to-regexp) (支援 [Express](https://expressjs.com) 的函式庫) 普及化。

建立與 route `/post/:pid` 匹配的頁面現在可以透過在 `pages` 資料夾中建立名為 `pages/post/[pid].js` 的檔案來達成！

Next.js 將自動匹配如 `/post/1`、`/post/hello-nextjs` 等的請求，並渲染在 `pages/post/[pid].js` 中定義的頁面。匹配的 URL 區段將作為查詢參數傳遞到您的頁面，名稱指定在 `[square-brackets]` 之間。

例如：假設以下頁面和請求 `/post/hello-nextjs`，`query` 物件將是 `{ pid: 'hello-nextjs' }`：

```
static async getInitialProps({ query }) {
  // pid = 'hello-nextjs'
  const { pid } = query
 
  const postContent = await fetch(
    `https://api.example.com/post/${encodeURIComponent(pid)}`
  ).then(r => r.text())
 
  return { postContent }
}
```

也支援多個動態 URL 區段！

`[param]` 語法支援資料夾名稱和檔案名稱，意味著以下範例有效：

```
./pages/blog/[blogId]/comments/[commentId].js
./pages/posts/[pid]/index.js
```

您可以在 [Next.js 文件](https://github.com/vercel/next.js#dynamic-routing) 或 [Next.js 學習區段](https://nextjs.org/learn/basics/dynamic-routes) 閱讀更多關於此功能的資訊。

自動靜態最佳化 [](#automatic-static-optimization)
---------------------------------------------------------------

Next.js 在 v3（大約兩年前發布）中增加了靜態網站生成支援。當時，這是最多被請求加入 Next.js 的功能。

理由很充分：毫無疑問，靜態網站*非常快速*！它們不需要伺服器端運算，可以立即從 CDN 位置串流到最終使用者。

然而，伺服器端渲染或靜態生成的應用程式之間的選擇是二進制的，您必須選擇伺服器端渲染或靜態生成。沒有中間地帶。

在現實中，應用程式可能有不同的需求。這些需求需要不同的渲染策略和權衡。

例如，首頁和行銷頁面通常包含靜態內容，是靜態最佳化的絕佳候選。

另一方面，產品儀表板可能受益於伺服器端渲染，因為其中的數據經常更新。

我們開始探索如何給予使用者两者的最佳體驗，並能夠**預設快速**。我們如何給予使用者靜態行銷頁面和動態伺服器渲染頁面？

從 Next.js 9 開始，使用者不再需要在完全伺服器渲染或靜態導出應用程式之間做出選擇。在每個頁面的基礎上給予您兩者最好的體驗。

#### 自動部分靜態導出 [](#automatic-partial-static-export)

引入了一種啟發式方法，以自動確定頁面是否可以預渲染為靜態 HTML。

此決定是基於頁面是否透過使用 [`getInitialProps`](https://github.com/vercel/next.js#fetching-data-and-component-lifecycle) 具有阻塞數據需求。

此啟發式方法允許 Next.js 發出包含**伺服器端渲染和靜態生成頁面**的混合應用程式。

內建的 Next.js 伺服器 (`next start`) 和程式化 API (`app.getRequestHandler()`) 都透明支援此建置輸出。不需要配置或特別處理。

靜態生成的頁面仍然是互動的：Next.js 將在客戶端水合您的應用程式以賦予其完整互動性。

此外，如果頁面依賴於 URL 中的查詢參數，Next.js 將在水合後更新您的應用程式。

如果頁面將在開發期間靜態生成，Next.js 將視覺化地通知您。這種視覺殘影可以透過點擊隱藏。

API 路由 [](#api-routes)
-------------------------

在建立 React 應用程式的多種情況下，您需要某種方式來獲取後端。無論是从資料庫獲取數據還是處理您的使用者提供的數據（例如聯絡表單）。

我們發現許多需要建立後端的使用者使用自訂伺服器建立他們的 API。在這種情況下，他們遇到了許多問題。例如，Next.js 不會編譯自訂伺服器程式碼，這意味著您無法使用 `import` / `export` 或 TypeScript。

出於這個原因，許多使用者最終在自訂伺服器上實現了他們自己的自訂編譯流程。雖然這解決了他們的目标，但也容易陷入許多陷阱：例如，當配置不正確時，樹搖動 (tree shaking) 將為整個應用程式禁用。

這引發了一個問題：如果我們將 Next.js 提供的開發體驗帶到 API 後端建立中呢？

今天我們很興奮地介紹 API 路由 (API Routes)，這是 Next.js 最優秀的開發體驗，用於建立您的後端。

要開始使用 API 路由，您需要在 `pages/` 資料夾內建立一個名為 `api/` 的資料夾。

此資料夾中的任何檔案將自動映射到 `/api/<your route>`，與其他頁面檔案映射到路由的方式相同。

例如，`pages/api/contact.js` 將映射到 `/api/contact`。

> **注意**：API 路由也支援 [動態路由](#dynamic-route-segments)！

`pages/api/` 資料夾內的所有檔案都匯出一個請求處理器函式，而不是 React 組件：

```
export default function handle(req, res) {
  res.end('Hello World');
}
```

*   `req` 參照 [NextApiRequest](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L143-L158) 它延伸 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
*   `res` 參照 [NextApiResponse](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L168-L178) 它延伸 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)

通常 API 端點會接收一些進入數據，例如查詢字串、請求主體或 Cookie，並回應其他數據。

在調查為 Next.js 添加 API 路由支援時，我們注意到在多個情況下使用者並沒有直接使用 Node.js 請求和響應物件。相反，他們使用了如 [Express](https://expressjs.com) 等服务庫提供的抽象。

這樣做的理由是，在多個情況下，進入數據是某種形式的文本，必須先解析才能有用。因此，這些特定的伺服器庫有助於去除手動解析數據的負擔，通常透過中間件。最常用的提供者查詢字串、主體和 Cookie 解析，然而它們仍然需要一些設置才能開始。

Next.js 中的 API 路由預設將提供這些中間件，以便您可以立即建立 API 端點並投入生產：

```
export default function handle(req, res) {
  console.log(req.body); // The request body
  console.log(req.query); // The url querystring
  console.log(req.cookies); // The passed cookies
  res.end('Hello World');
}
```

除了使用進入數據外，您的 API 端點通常也會返回數據。通常此響應將是 JSON。Next.js 預設提供 `res.json()` 以方便發送數據：

```
export default function handle(req, res) {
  res.json({ title: 'Hello World' });
}
```

在開發期間更改 API 端點的程式碼時，程式碼將自動重新載入，因此無需重新啟動伺服器。

生產環境最佳化 [](#production-optimizations)
-----------------------------------------------------

#### 視窗內 `Link` 預取 [](#prefetching-in-viewport-links)

Next.js 9 將自動預取出現在視窗內的 `<Link>` 組件。

此功能透過使新頁面的導航更快來改善您的應用程式的響應速度。

Next.js 使用 [Intersection Observer](https://developer.mozilla.org/docs/Web/API/Intersection_Observer_API) 在背景中 [預取資產](https://www.w3.org/TR/resource-hints/#prefetch)。

這些請求為低優先級並讓位給 `fetch()` 或 XHR 請求。如果使用者啟用了數據節省模式，Next.js 將避免自動預取。

您可以透過將 `prefetch` 屬性設定為 `false` 來選擇退出此功能，適用於很少訪問的頁面：

```
<Link href="/terms" prefetch={false}>
  <a>Terms of Service</a>
</Link>
```

#### 預設最佳化 AMP [](#optimized-amp-by-default)

Next.js 9 現在預設為 AMP 優先和混合 AMP 頁面渲染最佳化 AMP。

雖然 AMP 頁面是選擇性的，但 Next.js 將自動最佳化其輸出。這些最佳化可導致 [50% 更快的](https://blog.amp.dev/2018/10/08/how-to-make-amp-even-faster/) 渲染速度！

此變更由 [Sebastian Benz](https://twitter.com/sebabenz) 在 [AMP Optimizer](https://github.com/ampproject/amp-toolbox/tree/master/packages/optimizer) 上的極佳工作所實現。

#### `typeof window` 分支的死代碼消除 [](#dead-code-elimination-for-typeof-window-branches)

Next.js 9 在伺服器端和客戶端建置期間將 `typeof window` 替換為其適當的值（`undefined` 或 `object`）。此變更允許 Next.js 自動從您的生產建構應用程式中移除死代碼。

如果您在 `getInitialProps` 或應用程式的其他部分具有僅伺服器端程式碼，用戶應看到他們的客戶端套件大小減少。

開發者體驗改進 [](#developer-experience-improvements)
-----------------------------------------------------------------------

#### 編譯指示器 [](#compiling-indicator)

在 9 之前的版本中，知道熱代碼置換要發生（且 Next.js 編譯工具鏈正在工作）的唯一方法是查看開發者控制台。

然而，許多時候人們是在看生成的渲染結果，很難知道 Next.js 是否仍在進行編譯工作或沒有。例如，您可能正在更改頁面的樣式，這些樣式很細微，您不會立即知道它們是否已更新。

出於這個原因，我們建立了一個 [RFC / "good first issue"](https://github.com/vercel/next.js/issues/4626) 以討論指示工作的通知解決方案的潛在問題。

我們收到了許多設計師和工程師在 RFC 上的回饋，例如他們偏好的內容以及指示器設計的潛在方向。

[Rafael Almeida](https://github.com/rafaelalmeidatk) 抓住了這個機會與我們的團隊合作，實施了一個全新的指示器，現已默認在 Next.js 9 中提供。

Whenever Next.js is doing compilation work you will see a small triangle show up in the bottom right corner of the page!

Whenever Next.js is doing compilation work you will see a small triangle show up in the bottom right corner of the page!

Whenever Next.js is doing compilation work you will see a small triangle show up in the bottom right corner of the page!

Traditionally when making changes in development Next.js would show a compiling indicator state with loading state bars filling up and would continuously clear the screen as you made changes.

This behavior causes some issues. Most notably it would clear console output from both your application code, for example when you add `console.log` to your components. But also when using external tools that stitch log output together like the [Vercel CLI](https://vercel.com/cli) or `docker-compose`.

Starting from Next.js 9 the log output jumps less and no longer clears the screen. The allows for a better overall experience as your terminal window will have more relevant information and flicker less while Next.js will integrate better with tools that you might already be using.

Per-Page Configuration Object[](#per-page-configuration-object)
---------------------------------------------------------------

Every page can now export a configuration object. Initially this configuration allows you to opt-into [AMP](https://nextjs.org/blog/next-8-1), but in the future you will be able to configure more page specific options.

pages/about.js

```
export const config = { amp: true };
 
export default function AboutPage(props) {
  return <h3>My AMP About Page!</h3>;
}
```

To opt into hybrid AMP rendering you can use the value `'hybrid'`:

pages/about.js

```
import { useAmp } from 'next/amp';
 
export const config = { amp: 'hybrid' };
 
export default function AboutPage(props) {
  const isAmp = useAmp();
  return <h3>My About Page!{isAmp ? <> Powered by AMP!</> : ''}</h3>;
}
```

The `withAmp` higher order component was removed in favor of this new configuration.

We've provided a [codemod](https://github.com/vercel/next-codemod#withamp-to-config) that automatically converts usage of `withAmp` to the new configuration object. You can read more about this in the [upgrade guide](https://github.com/vercel/next.js/blob/canary/UPGRADING.md).

Codebase Improvements[](#codebase-improvements)
-----------------------------------------------

We've recently made some changes to our tooling to provide a better experience while contributing to the codebase and ensure stability as the codebase grows.

As you've read under the TypeScript section the Next.js core is now written in TypeScript and types are automatically generated for Next.js applications to use. Besides this being useful for applications built using Next.js, it's also useful when working on the core codebase. As you get type errors and autocompletion automatically.

Next.js already had quite a large integration test suite that consists of 50+ Next.js applications with tests that run against them. These tests ensure that when a new version is released upgrading is smooth as the features that were available before were tested against the same test suite.

Most of our tests are integration tests because in many cases they replicate "real" developers using Next.js in development. For example we have tests that replicate making changes to a Next.js application to see if hot module replacement works.

Our integration tests are mostly based on Selenium webdriver, which we combined with chromedriver to test in headless Chrome. However as time passed certain issues would arise in other browsers, especially older browsers like Internet Explorer 11.

Because we used Selenium we were able to run our tests automatically on multiple browsers.

As of right now we are running our test suite on Chrome, Firefox, Safari and Internet Explorer 11.

### Google Chrome 合作計畫 [](#google-chrome-collaboration)

The Google Chrome team has been working on improving Next.js by contributing RFCs and pull-requests.

The goal of this collaboration is large-scale performance improvements, focused on bundle sizes, bootup and hydration time.

For example these changes will improve the experience of small websites, but also that of massive applications like [Hulu](https://hulu.com), [Twitch](https://m.twitch.tv), and [Deliveroo](https://deliveroo.co.uk/).

#### Module / Nomodule[](#module--nomodule)

The first area of focus is shipping modern JavaScript to browsers that support modern JavaScript.

For example currently Next.js has to provide polyfills for [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) syntax as code might be executed in browsers that do not support [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) which would break.

![Next.js Module/Nomodule Collaboration RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/first-collab-rfc.png)

Next.js Module/Nomodule Collaboration RFC

To avoid breaking older browsers while still sending modern JavaScript to browsers that support it Next.js will utilize the [module/nomodule pattern](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/). The module/nomodule pattern provides a reliable mechanism for serving modern JavaScript to modern browsers while still allowing older browsers to fall back to polyfilled ES5.

The RFC for module/nomodule in Next.js can be [found here](https://github.com/vercel/next.js/issues/7563).

#### Improved Bundle Splitting[](#improved-bundle-splitting)

The current bundle splitting strategy in Next.js is based around a ratio-based heuristic for including modules in a single "commons" chunk. Because there is very little granularity as there is only one bundle, code is either downloaded unnecessarily (because the commons chunk could include code that's not actually required for a particular route) or the code is duplicated across multiple page bundles.

![Next.js Chunking Collaboration RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/chunking-rfc.png)

Next.js Chunking Collaboration RFC

The RFC for improved bundle splitting can be [found here](https://github.com/vercel/next.js/issues/7631).
