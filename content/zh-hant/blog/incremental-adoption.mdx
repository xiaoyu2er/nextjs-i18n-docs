[Next.js](https://nextjs.org/) 是為漸進式採用而設計的。使用 Next.js，您可以繼續使用既有代碼，並根據需要添加適量的 React。透過起步時從少量開始並逐步增加更多頁面，您可以避免全面重寫而導致功能開發受阻。

許多公司需要現代化其技術堆棧，以降低成本、提高開發者生產力，並為客戶提供最優質的體驗。基於組件的開發大幅提升了現代程式碼庫的部署速度和可重用性。

且擁有每月超過 [800 萬次下載](https://www.npmtrends.com/react)，React 是開發者首選的基於組件選擇。Next.js 是專為生產環境打造的 React 框架，讓您可以逐步採用 React。

動機 [](#motivation)
-------------------------

在日益移動化的世界中，改善和追蹤您的 [核心網頁指標](/analytics) 對成功至關重要。您的客戶很可能分佈在全球各地且網絡速度各異。每多等待一頁載入或操作完成所花的一秒（或毫秒），都可能成為銷售、曝光或轉化的關鍵差異。

如果您正在現代化您的技術堆棧，您可能會面臨以下挑戰：

* 您的應用程式擁有數年的舊代碼，難以理解，重寫將耗費數年（且數百萬美元）。
* 隨著應用程式規模和複雜度增加，您的頁面加載時間不斷增加。簡單的營銷頁面與最複雜的頁面一樣慢。
* 您試圖擴展開發團隊，但在向既有程式碼庫添加更多開發人員時遇到問題。
* 您使用過時的 CI/CD 和 DevOps 流程，這降低了開發者生產力，且使得安全可靠地發布新更改變得困難。
* 您的應用程式對移動設備不具備響應性，且無法在不破壞應用程式其他部分的同時更新全局頁面樣式。

您知道需要做 _一些事_，但要去理解 [從何開始](https://www.psychologytoday.com/us/blog/mindfully-present-fully-alive/201804/the-only-way-eat-elephant) 可能會令人感到不知所措。透過逐步採用 Next.js，您可以開始解決上述問題並徹底改造您的應用程式。讓我們討論幾種不同策略，以便將 Next.js 整合到您現有的技術堆棧中。

策略 [](#strategies)
-------------------------

### 子路徑 [](#subpath)

第一種策略是設定您的伺服器或代理器，使特定子路徑下的所有內容都指向 Next.js 應用程式。例如，您的現有網站可能位於 `example.com`，而您可以設定您的代理器，使 `example.com/store` 提供 Next.js 電子商務店面。

使用 [`basePath`](/docs/pages/api-reference/next-config-js/basePath)，您可以設定 Next.js 應用程式的資源和鏈結，使其自動與您的新子路徑 `/store` 配合運作。由於 Next.js 中的每個頁面都有自己的 [獨立路由](/docs/pages/building-your-application/routing)，像 `pages/products.js` 這類的頁面在您的應用程式中將路由至 `example.com/store/products`。

next.config.js

```
module.exports = {
  basePath: '/store',
};
```

若要進一步了解 `basePath`，請參閱我們的 [文件](/docs/pages/api-reference/next-config-js/basePath)。

（**注意：** 此功能於 Next.js 9.5 及更高版本引入。如果您使用的是舊版 Next.js，請在嘗試使用前進行升級。）

### 重寫 (Rewrites)[#rewrites]

第二種策略是建立一個指向網域根 URL 的新 Next.js 應用程式。接著，您可以在 `next.config.js` 中使用 [`rewrites`](/docs/pages/api-reference/next-config-js/rewrites)，使某些子路徑被代理到您的現有應用程式。

例如，假設您要從 `example.com` 服務一個 Next.js 應用程式，並使用以下的 `next.config.js`。現在，對於您加入此 Next.js 應用程式的頁面（例如：如果您已加入 `pages/about.js`，則為 `/about`）的請求將由 Next.js 處理，而對於任何其他路由（例如：`/dashboard`）的請求將被代理至 `proxy.example.com`。

next.config.js

```
module.exports = {
  async rewrites() {
    return [
      // we need to define a no-op rewrite to trigger checking
      // all pages/static files before we attempt proxying
      {
        source: '/:path*',
        destination: '/:path*',
      },
      {
        source: '/:path*',
        destination: `https://proxy.example.com/:path*`,
      },
    ];
  },
};
```

若要進一步了解 rewrites（重新設定），請參閱我們的 [文件](/docs/pages/api-reference/next-config-js/rewrites)。

### 使用 Monorepos 與子網域的微前端 [](#micro-frontends-with-monorepos-and-subdomains)

Next.js 與 [Vercel](https://vercel.com) 讓您能輕鬆採用 [微前端](https://martinfowler.com/articles/micro-frontends.html)，並作為 [Monorepo](https://vercel.com/blog/monorepos) 進行部署。這讓您可以利用 [子網域](https://en.wikipedia.org/wiki/Subdomain) 來逐步採用新應用程式。微前端的一些好處：

* 更小、凝聚力更強且更容易維護的程式碼庫。
* 擁有一個解耦和自治團隊的更具擴展性的組織。
* 能夠以更漸進的方式升級、更新，甚至重寫前端的部分功能。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/incremental-adoption/light-arch.png)

> 部署到 Vercel 的 Monorepo 架構。

設定好您的 Monorepo 後，像平常一樣將更改推送到您的 Git 庫，您將會看到提交內容部署至您已連接的 Vercel 專案。告別過時的 CI/CD 流程。

結論 [](#conclusion)
-------------------------

Next.js 是為逐步採用到您現有的技術堆棧而設計的。Vercel 平台透過與 GitHub、GitLab 和 Bitbucket 無縫整合，為每一次代碼更改提供部署預覽，使其成為協作體驗。

* 透過 [Fast Refresh](/docs/architecture/fast-refresh) 即時在本地預覽更改，提高開發者生產力。
* 推送更改以建立 [分支預覽](https://vercel.com/github)，優化與利害關係人協作。
* 透過合併 PR 使用 [Vercel](https://vercel.com) 部署到位。無需複雜的 DevOps。

若要了解更多，請閱讀關於 [子路徑](/docs/pages/api-reference/next-config-js/basePath) 和 [重新設定](/docs/pages/api-reference/next-config-js/rewrites)，或 [部署微前端範例](https://vercel.com/import/project?template=https://github.com/vercel/next.js/tree/canary/examples/with-zones)。
