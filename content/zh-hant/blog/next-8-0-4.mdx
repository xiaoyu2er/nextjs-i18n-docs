建置效能 [](#build-performance)
---------------------------------------

Next.js 建置現在更具確定性，這意味著如果程式碼未更改，建置輸出每次都會保持相同結果。

這使得建置過程中的更多工作可以被快取，從而使得在第一次建置後生產程式碼的重新建置更快。

我們的測量顯示，大量建置時間都花在壓縮 JavaScript 上，由於建置輸出現在更具確定性，壓縮後的檔案已在快取中的機率更大。

使用者 [部署到 Vercel](https://vercel.com/guides/deploying-nextjs-with-vercel) 將自動體驗這些快取改進。

我們簡化了伺服器端 ``next/dynamic`` manifest，僅包含以非同步方式載入的模組。這個簡化的 manifest 計算速度更快，並產生更少的 JavaScript。

CircleCI 使用者應該會注意到建置時間更快。之前，由於 CircleCI 環境無法準確反映可用 CPU 數量，導致工作節點過度排程。現在 Next.js 會檢測 CircleCI 並根據可用資源設定適當的 CPU 數量。

更小的執行時載著 [](#smaller-runtime-footprint)
-------------------------------------------------------

Next.js 現在生成的**Client-side JavaScript 少了 5.58KB**，且 HTTP 負載比上一個版本更小。

``withRouter`` 不再依賴 ``hoist-non-react-statics``，套件大小減少**3KB**。``withRouter`` 仍會提升 ``getInitialProps``，但會提升其他靜態屬性。

``next/babel`` preset 已最佳化為產生更 _輕盈_ 的 _更快_ JavaScript。

`X-Powered-By` 標頭已移除，減少 HTTP 負載大小。我們調查了社群，發現此標頭在生產環境中經常被停用，因此我們決定將其移除。這也意味著如果您的專案中啟用了 ``poweredByHeader`` 選項，則可從您的 ``next.config.js`` 中移除它。

我們已對 Next.js 相依性樹和整體程式碼庫進行了許多最佳化，這樣我們能夠將每個 Serverless 函數減少**44KB (5.44KB gzip)**。

Serverless 函數大小直接影響 Serverless 啟動效能，更小的函數意味著啟動更快。

|  | 8.0 | 8.0.4 | 差異 |
| --- | --- | --- | --- |
| Serverless 頁面大小 | 259 KB | 215 KB | **17%** _更小_ |
| Serverless 頁面大小 (gzip) | 62.3 KB | 56.8 KB | **9%** _更小_ |

Next.js 8 發行之後，我們收到少數使用者的回報，指出他們在 Next.js 外部（例如在測試套件中）匯入 React 元件時遇到了困難。這是因為對 `next` 的匯入被重寫成了 Next.js 程式碼庫內正確的檔案，然而，此最佳化方案套用了所有使用 `next/babel` 預設套件的使用者。此最佳化方案已移至 Next.js 建置流程本身，因此不再會與使用者的 Babel 設定產生衝突。

預設視窗 Meta Tag [](#default-viewport-meta-tag)
-------------------------------------------------------

Next.js 的目標之一是為撰寫網路應用程式提供最佳可能的預設值。為了減少實施 CSS 媒體查詢時需要執行的設定數量。

預設情況下，瀏覽器不會以您預期它們運作的方式處理 CSS ``@media`` 查詢和縮放，這可能導致撰寫 CSS ``@media`` 查詢時出現意外不一致。

在幾乎所有情況下，Next.js 使用者都會向他們的應用程式新增 ``viewport`` Meta Tag 來解決這些不一致。

從版本 8.0.4 開始，在大多數情況下不再需要此 ``viewport`` 標籤。如果應用程式未設定 ``viewport``，將套用預設視窗：

```
<meta
  name="viewport"
  content="width=device-width,minimum-scale=1,initial-scale=1"
/>
```

視窗標籤仍可以使用 ``next/head`` 覆蓋：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Head from 'next/head';
 
function HomePage() {
  return (
    <>
      <Head>
        <meta
          name="viewport"
          content="width=device-width,minimum-scale=0.5,initial-scale=1"
        />
      </Head>
    </>
  );
}
 
export default HomePage;
```

感謝 [Jason Miller](https://twitter.com/_developit) 合作並實施此變更。

全新 @next/mdx 插件 ([](#new-nextmdx-plugin)
---------------------------------------------

[MDX](https://mdxjs.com/) 允許您在 Markdown 文件中撰寫 JSX。您可以使用一般的 Markdown 語法撰寫內容，並匯入 React 組件以透過互動和動態內容增強文件。

為 Next.js 提供 MDX 支援的插件 ``@zeit/next-mdx`` 已移至 [Next.js GitHub 儲存庫](https://github.com/vercel/next.js/tree/canary/packages/next-mdx)，現在可作為獨立套件 [``@next/mdx``](https://www.npmjs.com/package/@next/mdx) 取得。

安裝：

終端機

```
npm i @next/mdx @mdx-js/loader
```

要使其可用於您的 Next.js 應用程式，請建立 ``next.config.js`` 檔案並包含：

```
const withMDX = require('@next/mdx')();
module.exports = withMDX();
```

隨著時間推移，更多的 Next.js 插件將移至 Next.js 儲存庫，以便與 Next.js 核心一同發布，並由 Next.js 測試套件測試。這樣我們可以確保熱模組替換、生產建置和其他功能與插件合作良好。
