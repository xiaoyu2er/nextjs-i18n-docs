下一代靜態網站生成 (SSG) 支援 [#next-gen-static-site-generation-ssg-support]
---------------------------------------------------------------------------------------------
在建立網站或 Web 應用程式時，您通常必須在兩個策略之間選擇：靜態生成 (**SSG**) 或 伺服器端渲染 (**SSR**)。

Next.js 是第一個混合框架，允許您根據每頁的需求，選擇最適合您使用案例的技術。

Next.js 9.0 引入了 [自動靜態最佳化](https://nextjs.org/blog/next-9#automatic-static-optimization) 的概念。當頁面沒有像 `getInitialProps` 這樣的阻擋性資料取得需求時，它將在建置時自動渲染為 HTML。

還有一些情況您可能希望在建置時將頁面渲染為靜態 HTML，即使有阻擋性資料取得需求。這方面的範例是由 (無頭) 內容管理系統 (**CMS**)  powering 的行銷頁面，或網站的部落格區塊。

我們與大量的 SSG 和 `next export` 使用者（如 HashiCorp）合作，並在 Next.js 歷史上 [評論數最多的 RFC](https://github.com/vercel/next.js/issues/9524) 中與社群廣泛討論了適當的約束，以建立統一的数据取得和靜態生成的新方法。

今天我們非常興奮地宣佈兩項新的資料取得方法：[`getStaticProps`](/docs/pages/building-your-application/data-fetching) 和 [`getServerSideProps`](/docs/pages/building-your-application/data-fetching)。我們還提供了一種方法，為動態路由靜態生成靜態頁面提供參數：[`getStaticPaths`](/docs/pages/building-your-application/data-fetching)。

這些新方法相比 `getInitialProps` 模型有許多優點，因為 SSG 和 SSR 之間有明確的區分。

*   `getStaticProps`（靜態生成）：在 **建置時** 取得資料。
*   `getStaticPaths`（靜態生成）：指定 [動態路由](/docs/pages/building-your-application/routing/dynamic-routes) 以根據資料進行預渲染。
*   `getServerSideProps`（伺服器端渲染）：在 **每次請求** 時抓取數據。
*   這些改進是 API 新增。所有新功能完全向後兼容，可以增量採用。沒有引入任何棄用，`getInitialProps` 仍會像現在這樣運作。我們鼓勵在新頁面和專案中採用這些新方法。
    

### getStaticProps [#getstaticprops]
如果您從頁面匯出一个呼叫 `getStaticProps` 的 `async` 函式，Next.js 將在建置時預渲染此頁面。這在您想從 CMS 渲染特定的靜態頁面時特別有用。

`getStaticProps` 始終在 Node.js 環境中執行，程式碼會自動從瀏覽器捆綁中搖樹 (tree-shaken)，確保較少的程式碼被傳送到瀏覽器。這樣您就不必擔心 Node.js 和瀏覽器環境（這兩個環境有些不一致）中資料取得程式碼的執行。

這允許您使用任何非同步甚至同步的資料取得技術，包括 `fetch`、REST、GraphQL，甚至是直接存取資料庫。

pages/posts/[id].js

```
export async function getStaticProps(context) {
  return {
    props: {}, // will be passed to the page component as props
  };
}
```

`context` 參數是一個包含以下鍵的物件：

*   `params`: `params` 包含使用動態路由的頁面的路由參數。例如，如果頁面名稱是 `[id].js` ，那麼 `params` 看起來會像 `{ id: ... }`。要了解更多信息，請查看 [動態路由文件](/docs/pages/building-your-application/routing/dynamic-routes)。您應該將此與 `getStaticPaths` 結合使用，我們稍後會解釋。

這是一個使用 `getStaticProps` 從 CMS 獲取部落格文章清單的範例：

pages/blog.js

```
// You can use any data fetching library
import fetch from 'node-fetch';
 
// posts will be populated at build time by getStaticProps()
function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li>{post.title}</li>
      ))}
    </ul>
  );
}
 
// This function gets called at build time in the Node.js environment.
// It won't be called on client-side, so you can even do
// direct database queries. See the "Technical details" section.
export async function getStaticProps() {
  // Call an external API endpoint to get posts.
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  // By returning { props: posts }, the Blog component
  // will receive `posts` as a prop at build time
  return {
    props: {
      posts,
    },
  };
}
 
export default Blog;
```

### 我何時應使用 getStaticProps? [#when-should-i-use-getstaticprops]

在以下情況下應使用 `getStaticProps`：

*   渲染頁面所需的資料在用戶請求之前就已在建置時可用。
*   資料來自無頭 CMS。
*   資料可以公共快取（不特定於用戶）。
*   頁面必須預渲染（針對 SEO）且非常快速 — `getStaticProps` 生成 HTML 和 JSON 檔案，這兩者都可以由 CDN 快取以獲取性能。

有關 `getStaticProps` 的更多信息，請參閱 [資料取得文件](/docs/pages/building-your-application/data-fetching)。

### getStaticPaths [#getstaticpaths]

如果頁面有 [動態路由](/docs/pages/building-your-application/routing/dynamic-routes) 並使用 `getStaticProps`，它需要定義一個路徑列表，這些路徑必須在建置時渲染為 HTML。

如果您從使用動態路由的頁面匯出呼叫 `getStaticPaths` 的 `async` 函式，Next.js 將靜態預渲染 `getStaticPaths` 指定的所有路徑。

pages/posts/[id].js

```
export async function getStaticPaths() {
  return {
    paths: [
      { params: { ... } } // See the "paths" section below
    ],
    fallback: true or false // See the "fallback" section below
  };
}
```

### 路徑鍵 (必填) [#the-paths-key-required]

路徑鍵 determines 哪些路徑將被預渲染。例如，假設您有一個頁面使用動態路由命名為 `pages/posts/[id].js`。如果您從這個頁面匯出 `getStaticPaths` 並返回以下路徑：

```
return {
  paths: [
    { params: { id: 1 } },
    { params: { id: 2 } }
  ],
  fallback: ...
}
```

然後 Next.js 將使用 `pages/posts/[id].js` 中的頁面組件，在建置時靜態生成 `posts/1` 和 `posts/2`。

請注意每個 `params` 的值必須與頁面名稱中使用的參數匹配：

*   如果頁面名稱是 `pages/posts/[postId]/[commentId]`，則 `params` 應包含 `postId` 和 `commentId`。
*   如果頁面名稱使用 catch-all 路由，例如 `pages/[...slug]`，則 `params` 應包含 `slug`，這是一個陣列。例如，如果該陣列是 `['foo', 'bar']`，則 Next.js 將靜態生成頁面 `/foo/bar`。

### 支援鍵 (fallback key) [#the-fallback-key-required]

`getStaticPaths` 返回的物件必須包含一個布林值 `fallback` 鍵。

### 支援：false [#fallback-false]

如果 `fallback` 是 `false`，則任何沒有由 `getStaticPaths` 返回的路徑將產生 **404 頁面**。如果您知道所有路徑在建置時都已知，這很有用。

這是一個預先渲染每個部落格文章在 `pages/posts/[id].js` 的範例。部落格文章清單將從 CMS 取得並由 `getStaticPaths` 返回。然後，對於每個頁面，它使用 `getStaticProps` 從 CMS 獲取文章數據。

pages/posts/[id].js

```
import fetch from 'node-fetch';
 
function Post({ post }) {
  // Render post...
}
 
// This function gets called at build time
export async function getStaticPaths() {
  // Call an external API endpoint to get posts
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  // Get the paths we want to pre-render based on posts
  const paths = posts.map((post) => `/posts/${post.id}`);
 
  // We'll pre-render only these paths at build time.
  // { fallback: false } means other routes should 404.
  return { paths, fallback: false };
}
 
// This also gets called at build time
export async function getStaticProps({ params }) {
  // params contains the post `id`.
  // If the route is like /posts/1, then params.id is 1
  const res = await fetch(`https://.../posts/${params.id}`);
  const post = await res.json();
 
  // Pass post data to the page via props
  return { props: { post } };
}
 
export default Post;
```

### 支援：true [#fallback-true]

如果 `fallback` 是 `true`，則 `getStaticProps` 的行為會改變，Next.js 將在建置時將提供給定的路徑渲染為 HTML。當路徑在建置時未生成時，它會在用戶請求頁面時按需生成。

**這對於擁有許多可以靜態生成的路由的應用程式非常有用，但您不希望因為僅在建置時生成子集而導致頁面建置時間增加。**

觸發頁面生成的用戶將接收隨附的 HTML，這通常是一個帶有載入狀態的頁面。這是因為靜態 HTML 可以從 CDN 提供，確保頁面始終快速，即使尚未生成。

按需靜態生成額外頁面的範例：

pages/posts/[id].js

```
import { useRouter } from 'next/router';
import fetch from 'node-fetch';
 
function Post({ post }) {
  const router = useRouter();
 
  // If the page is not yet generated, this will be displayed
  // initially until getStaticProps() finishes running
  if (router.isFallback) {
    return <div>Loading...</div>;
  }
 
  // Render post...
}
 
// This function gets called at build time
export async function getStaticPaths() {
  return {
    // Only `/posts/1` and `/posts/2` are generated at build time
    paths: [{ params: { id: 1 } }, { params: { id: 2 } }],
    // Enable statically generating additional pages
    // For example: `/posts/3`
    fallback: true,
  };
}
 
// This also gets called at build time
export async function getStaticProps({ params }) {
  // params contains the post `id`.
  // If the route is like /posts/1, then params.id is 1
  const res = await fetch(`https://.../posts/${params.id}`);
  const post = await res.json();
 
  // Pass post data to the page via props
  return { props: { post } };
}
 
export default Post;
```

有關 `getStaticPaths` 的更多信息，請參閱 [資料取得文件](/docs/pages/building-your-application/data-fetching)。

### getServerSideProps [#getserversideprops]

如果您從頁面匯出呼叫 `getServerSideProps` 的 `async` 函式，Next.js 將在每次請求時渲染此頁面（SSR）。

`getServerSideProps` 始終在伺服器端執行，程式碼會自動從瀏覽器捆綁中搖樹 (tree-shaken)，確保較少的程式碼被傳送到瀏覽器。這樣您就不必擔心資料取得程式碼在伺服器和瀏覽器環境（這兩個環境有些不一致）中的執行。這在許多情況下增加了效能，因為伺服器通常具有與數據來源更快的連線。它也通過減少公開資料取得邏輯來增加安全性。

這允許您使用任何非同步甚至同步的資料取得技術，包括 `fetch`、REST、GraphQL，甚至是直接存取資料庫。

使用 `next/link` 在頁面之間導航而不是在瀏覽器中執行 `getServerSideProps` 時，Next.js 將向伺服器進行請求，該請求將返回呼叫 `getServerSideProps` 的結果。

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
export async function getServerSideProps(context) {
  return {
    props: {}, // will be passed to the page component as props
  };
}
```

`context` 參數是一個包含以下鍵的物件：

*   `params`: 如果此頁面使用動態路由，則 `params` 包含路由參數。如果頁面名稱是 `[id].js` ，則 `params` 看起來會像 `{ id: ... }`。要了解更多信息，請查看 [動態路由文件](/docs/pages/building-your-application/routing/dynamic-routes)。
*   `req`: [HTTP 請求對象](https://nodejs.org/api/http.html#http_class_http_incomingmessage)。
*   `res`: [HTTP 響應對象](https://nodejs.org/api/http.html#http_class_http_serverresponse)。
*   `query`: 查詢字串。

這是一個使用 `getServerSideProps` 在請求時間獲取數據並進行渲染的範例：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
function Page({ data }) {
  // Render data...
}
 
// This gets called on every request
export async function getServerSideProps() {
  // Fetch data from external API
  const res = await fetch(`https://.../data`);
  const data = await res.json();
 
  // Pass data to the page via props
  return { props: { data } };
}
 
export default Page;
```

有關 `getServerSideProps` 的更多信息，請參閱 [資料取得文件](/docs/pages/building-your-application/data-fetching)。

預覽模式 [#preview-mode]
-----------------------------
正如本文 earlier 討論的，當您的頁面從無頭 CMS 獲取資料時，靜態生成非常有用。然而，當您在無頭 CMS 上撰寫草稿並希望在頁面上立即預覽草稿時，這並不是最理想的選擇。由於輸出是靜態的，預覽變更會變得困難，因為您必須重新生成該靜態頁面。

`getStaticProps` 在 Next.js 中的引進開啟了新的可能性，例如在特定條件下利用 Next.js 的按需渲染功能。

例如，當從您的無頭 CMS 預覽草稿時，您希望迴避靜態渲染並根據草稿內容按需渲染頁面，而不是發布的內容。您希望 Next.js 僅針對此特定情況迴避靜態生成。

**我們很高興宣佈 Next.js 的一項新的內建功能以滿足這一需求：預覽模式。**

預覽模式允許用戶迴避靜態生成頁面，以按需渲染（SSR）例如來自 CMS 的草稿頁面。

然而，您不僅限於特定的 CMS 系統。預覽模式直接與 `getStaticProps` 和 `getServerSideProps` 整合，因此它可以用於任何類型的資料取得解決方案。

使用 `next start` 時，或者透過 [部署](/docs/pages/building-your-application/deploying) 到 [Vercel Edge Network](https://vercel.com/docs/edge-network)，預覽模式即可用。

在 [https://next-preview.vercel.app/](https://next-preview.vercel.app/) 上親身體驗預覽模式。

有關 [預覽模式](/docs/pages/building-your-application/configuring/preview-mode) 的更多信息，請參閱 [文件](/docs/pages/building-your-application/configuring/preview-mode)。

全域樣式表內建 Sass 支援 [#built-in-sass-support-for-global-stylesheets]
---------------------------------------------------------------------------------------------
[Next.js 9.2](https://nextjs.org/blog/next-9-2) 引入了 [全域 CSS 樣式表的內建支援](https://nextjs.org/blog/next-9-2#built-in-css-support-for-global-stylesheets)，以使用更好的預設值替換 `next-css` 插件，以提供更優化的結果。

在發布之後，我們越來越多人要求整合 Sass 支援，因為許多轉向 Next.js 的企業都有基於 Sass 的現有設計系統。

在調查 Next.js 插件使用情況時，我們發現大約 30% 的 Next.js 應用程式目前使用 `next-sass`。相比之下，44% 使用原生 CSS，6% 使用 Less。

此外，`next-sass` 與 `next-css` 具有相同的缺失約束。這意味著您可以在專案的每個檔案中匯入 Sass 檔案，然而，此匯入的 Sass 檔案對於整個應用程式來說是全域的。

在考慮這些統計數據和反饋後，我們很高興地宣佈 Next.js 現在具有匯入 Sass 樣式表的內建支援。

要在應用程式中使用全域 Sass 匯入，請安裝 [`sass`](https://github.com/sass/sass)：

終端機

```
npm install sass
```

然後，在 `pages/_app.js` 內匯入 Sass 檔案。

例如，考慮以下 stylesheet 名稱為 `styles.scss` 位於您的專案根目錄：

```
$primary-color: #333;
 
body {
  padding: 20px 20px 60px;
  margin: 0;
  color: $primary-color;
}
```

如果您已經存在，建立 `pages/_app.js` 檔案。然後，匯入 `styles.scss` 檔案：

pages/_app.js

```
import '../styles.scss';
 
// This default export is required in a new `pages/_app.js` file.
export default function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
```

由於樣式表本質上是全域的，因此必須在 [自訂 `<App>` 組件](/docs/pages/building-your-application/routing/custom-app) 中匯入。這對於避免全域樣式的類別名稱和排序衝突是必需的。

在開發中，這樣表達樣式表允許您在編輯它們時自動更新頁面上的樣式。

在生產環境中，所有 Sass 和 CSS 檔案將自動連接成單個壓縮 `.css` 檔案。此 CSS 檔案將通過 `<link>` 標籤載入，並自動注入 Next.js 生成的預設 HTML 標記中。

此新功能完全向後兼容。如果您正在使用 `@zeit/next-sass` 或其他 CSS 相關插件，該功能已停用以避免衝突。

如果您目前使用 `@zeit/next-sass`，建議您從 `next.config.js` 和 `package.json` 中移除插件，從而將升級移至內建 Sass 支援。

組件層級樣態內建 CSS Module Sass 支援 [#built-in-sass-css-module-support-for-component-level-styles]
---------------------------------------------------------------------------------------------------------------------------
Next.js 現在使用 `[name].module.scss` 檔案命名慣例支援 [CSS Modules](https://github.com/css-modules/css-modules) 與 Sass 檔案。

與 Next.js 5+ 期間使用 `next-sass` 之前可用的支援不同，全域 Sass 和 CSS 模組現在可以 **共存** — `next-sass` 要求應用程式中的所有 `.scss` 檔案處理為全域或本機，但不能同時為兩者。

CSS 模組透過自動建立唯一類別名稱來本機限域 Sass。這允許您在不同檔案中使用相同的 Sass 類別名稱，而無需擔心碰撞。

此行為使 CSS 模組成為包含組件層級 SASS 的理想方式。CSS 模組檔案 **可以在您的應用程式的任何地方匯入**。

要在應用程式中使用 Sass CSS 模組，請確保您安装了 [`sass`](https://github.com/sass/sass)：

終端機

```
npm install sass
```

現在，考慮 `components/` 文件夾中的可重用了 `Button` 組件：

首先，建立 `components/Button.module.scss`，內容如下：

```
/*
You do not need to worry about .error {} colliding with any other `.css` or
`.module.css` files!
*/
$color: white;
 
.error {
  color: $color;
  background-color: red;
}
```

然後，建立 `components/Button.js`，匯入和使用上方 CSS 檔案：

components/Button.js

```
import styles from './Button.module.scss';
 
export function Button() {
  return (
    <button
      type="button"
      // Note how the "error" class is accessed as a property on the imported
      // `styles` object.
      className={styles.error}
    >
      Destroy
    </button>
  );
}
```

針對 Sass 檔案的 CSS 模組是一個 **可選** 功能，並且僅對具有 `.module.scss` 副檔名的檔案啟用。Regular [`<link>` stylesheets](/docs/pages/api-reference/components/head) 和 [global Sass styles](https://nextjs.org/blog/next-9-2#built-in-css-support-for-global-stylesheets) 仍支援。

在生產環境中，所有 CSS 模組檔案將自動連接成 **許多壓縮和拆分的 `.css` 檔案**。這些 `.css` 檔案代表應用程式中的熱執行路徑，確保應用程式的每頁載入最小數量的 CSS 以供塗裝。

正如上面那樣，此新功能完全向後兼容。如果您正在使用 `@zeit/next-sass` 或其他 CSS 相關插件，該功能已停用以避免衝突。

如果您目前使用 `@zeit/next-sass`，建議您從 `next.config.js` 和 `package.json` 中移除插件，從而將升級移至內建 Sass 支援。

404 頁面自動靜態最佳化 [#automatic-static-optimization-for-404]
-------------------------------------------------------------------------------
Next.js 9 的發布引入了 [自動靜態最佳化](https://nextjs.org/blog/next-9#automatic-static-optimization) 的概念，當頁面沒有阻擋性資料需求時，Next.js 將在建置時自動將頁面生成為靜態 HTML。然而，有一個頁面沒有自動渲染為靜態 HTML：404 頁面。404 頁面沒有自動變成靜態的主要原因是由 `/_error` 頁面提供 404 功能的頁面不僅處理 404，還處理例如錯誤。

由於 404 頁面會為不存在的路由渲染，按需渲染頁面可能會導致成本增加和伺服器負載增加。

我們旨在以兩種方式讓您在成功的深淵中：

*   預設 Next.js 體驗會生成靜態 404 頁面
*   在自訂 404 頁面時，它仍然確保您最終會得到一個靜態頁面

此功能完全向後兼容，所以如果您目前有一個自訂的 `pages/_error.js`，只要您添加 `pages/404.js`，它將繼續用於 404 頁面。

### 默認靜態 404 頁面 [#static-404-page-by-default]

當您的應用程式沒有自訂 `pages/_error.js` 頁面時，Next.js 將自動靜態生成 404 頁面並在使用到 404 需要服務時使用它。這是自動發生的，不需要進行任何變更。

### 使用 pages/404.js 自訂 404 頁面 [#custom-404-page-using-pages404js]

要覆寫預設 404 頁面，您現在可以建立 `pages/404.js`，這仍將在建置時自動靜態最佳化。如果您的應用程式有一個，這個頁面將用於 `pages/_error.js` 渲染 404。

pages/404.js

```
export default () => <h1>This is the 404 page</h1>;
```

32+ kB **更小執行環境 (15 kB+ Gzip)** [#32-kb-smaller-runtime-15-kb-gzip]
-----------------------------------------------------------------------------
Next.js 支援 [與 React 本身相同的使用者](https://reactjs.org/docs/react-dom.html#browser-support) 瀏覽器，且不需要配置。這包括 Internet Explorer 11 (IE11) 和所有流行瀏覽器（Edge、Firefox、Chrome、Safari、Opera 等）。

作為此相容性的一部分，我們也將您的應用程式編譯為 IE11 相容：這允許您安全地使用 [ES6](https://github.com/lukehoban/es6features)\+ 語法功能、Async/Await、Object Rest/Spread Properties 等 — 所有都無需配置。

此編譯過程的一部分還涉及透明注入必要的功能補丁（例如 `Array.from` 或 `Symbol`）。然而，這些補丁僅對 [不到 10% 的網路流量](https://caniuse.com/#feat=es6-module) 必要，在大多數情況下支援 IE11。

Next.js 9.3 開始，Next.js 將自動載入支援復古瀏覽器所需的補丁，並且僅在這些復古瀏覽器中載入補丁。

實踐上，這意味著從您的 **首次載入 (First Load)** 大小將消除 32 kB 或更多，[對於 90% 以上的您的用戶](https://caniuse.com/#feat=es6-module)。

對於依賴更多瀏覽器功能的大型應用程式來說，這些大小節省甚至更大。

此優化的完全自動且不需要任何應用程式變更即可利用它！
