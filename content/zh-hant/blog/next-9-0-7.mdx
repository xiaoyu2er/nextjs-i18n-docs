Next.js 9.0 約兩個月前發布。從那時起，我們一直在忙於 7 個較小但相當重要的版本更新：[9.0.1](https://github.com/vercel/next.js/releases/tag/v9.0.1), [9.0.2](https://github.com/vercel/next.js/releases/tag/v9.0.2), [9.0.3](https://github.com/vercel/next.js/releases/tag/v9.0.3), [9.0.4](https://github.com/vercel/next.js/releases/tag/v9.0.4), [9.0.5](https://github.com/vercel/next.js/releases/tag/v9.0.5), [9.0.6](https://github.com/vercel/next.js/releases/tag/v9.0.6), 和 [9.0.7](https://github.com/vercel/next.js/releases/tag/v9.0.7).

讓我們深入了解這些更新為您的網站和應用帶來了什麼，且完全不包含破壞性變更。

*   **[Windows 環境的並發改進](#improved-concurrency-in-windows-environments)**: ``next build`` 流程現在在 Windows 上更可靠，並能更好地並行處理工作。
*   **[預設啟用 Gzip 壓縮](#gzip-compression-by-default)**: Gzip 壓縮現在預設新增，以減少所需的優化步驟。
*   **[僅針對活動頁面的 TypeScript 報告](#typescript-report-on-active-pages-only)**: 內建的 TypeScript 支援已擴展為僅監控活動頁面的變更，使其更快且更可靠。
*   **[遥測 (Telemetry)](#telemetry)**: 將幫助我們專注於 Next.js 哪些部分需要優化並驗證優化確實達到預期效果。
*   **[改進的 ``next/head`` 元件追蹤](#improved-nexthead-element-tracking)**: ``next-head`` 類別已移除，這使得實現某些驗證其實現的工具更加容易。
*   **[防止 Pages 目錄中的非頁面檔案](#preventing-non-pages-in-the-pages-directory)**: 優化您的應用程式安全性及 ``next build`` 時間，防止意外發布非頁面檔案。
*   **[執行時改進](#runtime-improvements)**: 當 Next.js 的某些部分未使用時，例如 ``next/dynamic``，它們在執行時將不再需要，使得打包大小更小。

### Windows 環境的並發改進[](#improved-concurrency-in-windows-environments)

Next.js 在 ``next build`` 過程的許多地方進行並發工作。主要用途是並行壓縮構建輸出，使用 Terser。

先前，此工作是透過名爲 ``worker-farm`` 的套件Across 多個 CPU 處理。然而，我們注意到許多 Windows 使用者已使用自訂 Webpack 設定停用壓縮。通過進一步檢查，我們發現 ``worker-farm`` 在 Windows 機器上無法一致工作。

為了解決此問題，我們從 ``worker-farm`` 遷移至 ``jest-worker``。這確保了構建在 macOS、Linux 和 Windows 機器上都可靠且一致。

``jest-worker`` 如其名稱所示，是 Jest 測試運行器的一部分。它是 Jest 用來並行化測試案例的套件。這意味著該套件經過嚴格測試、可靠且持續維護。

``jest-worker`` 也支援 [`worker_threads`](https://github.com/facebook/jest/tree/master/packages/jest-worker#experimental-worker)，這是 Node 12 的新功能。與 ``child_process`` 不同，``worker_threads`` 可以共享記憶體 —— 這意味著在較新的 Node 版本上構建速度更快。

透過切換至 ``jest-worker``，我們能夠為 Windows 用戶重新啟用構建並發。

### 預設啟用 Gzip 壓縮[](#gzip-compression-by-default)

在調查為何公司使用自訂服務器時，我們發現大多數情況是為了壓縮。公司會新增一個名為 ``compression`` 的 Express 中間件，用於處理 HTTP 回應的 Gzip 壓縮。

此 middleware 壓縮回應，以便減少傳輸到用戶手中的位元組數量。通常這應由反向代理如 Nginx 處理。反向代理將 CPU 密集工作從單一线程 Node 流程中移除。

然而，在檢查 Web 上的 Next.js 使用情況時，我們發現有相當大部分的公司並沒有配置壓縮。

在像 [Vercel](https://vercel.com) 這樣的平台上，`gzip` 和 `brotli` 已在反向代理層面上自動處理。

在自行託管時，公司必須自行添加 gzip (透過 `compression` 或反向代理)。

從 Next.js 9.0.4 開始，使用 `next start` 或自訂 `server.js` 時，`gzip` 壓縮將預設包含。

`brotli` 支援將 [即將到來](https://github.com/expressjs/compression/pull/150) ，因為 Node.js 現在原生支援 Brotli。

如果壓縮已透過自訂服務器在應用程式中啟用，Next.js 將不會添加自己的壓縮器。

Next.js 預設不包含針對 serverless 目標的壓縮，因為使用 serverless 目標時，資源是單獨上傳的，不是透過 Node.js 服務器。

如果您在處理壓縮的平台上部署，例如 [Vercel](https://vercel.com)，則不需要任何更改。

### TypeScript 僅針對活動頁面的報告[](#typescript-report-on-active-pages-only)

Next.js 9 提供了內建的 TypeScript 支援。只要將單一頁面從 `.js` 重新命名為 `.tsx`。Next.js 將自動處理或引導您完成任何剩餘的設定。

Next.js 也會透過在開發流程並行執行 `tsc --watch` 來進行類型檢查。在開發期間，Next.js 有一個稱為 [按需條目](https://nextjs.org/blog/next-8#improved-on-demand-entries) 的概念。此功能只編譯您正在積極工作的頁面。

![按需條目流程](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-8/flow.jpg)

按需條目流程

從 9.0.4 開始，Next.js 現在僅報告由按需條目正在編譯的頁面的類型錯誤。這減少了大量類型檢查雜訊，同時專注於特定的一組頁面。

完整應用程式類型檢查仍會在 ``next build`` 期間執行，或由您的編輯器處理。

### 遥測 (Telemetry)[](#telemetry)

Next.js 已經發布近 3 年了，框架在這 3 年間顯著成長，包括新功能及為所有用戶提供更好的預設值。

我們一直以來處理這種改進過程的方式非常手動。

Vercel 有幾大型 Next.js 應用程式。例如，[vercel.com](https://vercel.com), [vercel.com/docs](https://vercel.com/docs), 以及 [https://nextjs.org](https://nextjs.org)。我們一直在 Vercel 內部 [吃自己的狗糧 (dogfooding)](https://en.wikipedia.org/wiki/Eating_your_own_dog_food) 並根據我們的經驗改進 Next.js。

除此之外，我們積極與社群互動以獲取反饋。您以前很可能與 Tim 談過以提供關於您如何在公司使用 Next.js 的反饋。

例如，如果您使用自訂服務器、自訂 `webpack` 設定，以及更多。這些反饋對於引導 Next.js 的功能開發至關重要。

然而，這種方法有一個問題，那就是我們只能從部分用戶那裡收集反饋。此子集的需求及使用案例可能與您/您的公司不同。

其中一個例子是匯入 CSS 檔案，這雖非標準，但相當多用戶似乎在使用它，透過 ``next-css`` (或 Sass/Less) 或自訂設定。如果我們知道多少百分比的用戶使用該特定方法，我們可以優先考慮改進它。

出於此原因，我們引入了一種無名、更自動化的方法來收集這些反饋點，以便我們能在近期進一步改進 Next.js。

這也將允許我們驗證框架上的改進是否改善了所有應用程式的基準。

您可以在 [nextjs.org/telemetry](https://nextjs.org/telemetry) 閱讀更多關於遥測的內容。在那裡您也會找到如何可選退出如果您不想參與。

### 帶動作指示點的構建反饋[](#build-feedback-with-dots-indicating-activity)

與 Next.js 用戶談話時，一個小反饋項目是它有時看起來 `next build` 沒有做任何事，主要在視覺上。

為了解決這個問題，我們在運行 `next build` 時為控制台輸出添加了載入指示器。此輸出給視覺指示命令仍在運行，且進程未凍結。

我們計劃在可能的情況下擴展此構建輸出以顯示更多構建階段。

### 改進的 ``next/head`` 元件追蹤[](#improved-nexthead-element-tracking)

Next.js 提供了一種管理 `<head>` 元件的內建方式，因為它負責渲染應用程式的 HTML。此 API 透過 `next/head` 模組暴露。

例如，要為頁面新增標題:

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Head from 'next/head';
 
export default function MyPage() {
  return (
    <>
      <Head>
        <title>My Title</title>
      </Head>
      <h1>Hello World</h1>
    </>
  );
}
```

當渲染至 HTML 時，Next.js 會收集在 `<Head>` 內渲染的所有元件並將標籤渲染到您頁面的 `<head>` 中。

然而，Next.js 允許使用 `<Link>` 元件進行單頁應用程式 (SPA) 類型的路由過渡。

當您點擊 `<Link>` 時，Next.js 會獲取所需的 JavaScript 檔案以在客戶端渲染頁面。然後，它會渲染與該檔案相關的 React 元件。

因為此過渡在客戶端發生，我們必須清理從上一頁注入的 `<head>` 元件，否則另一頁上可能會有過時的元件。

先前，Next.js 透過向每個 `<Head>`-提供的元件新增類別名稱來追蹤這些元件。

以上面的例子來說，`<head>` 會像:

```
<head>
  <title class="next-head">My Title</title>
</head>
```

此解決方案運作良好，因為每個透過 `next/head` 注入的元件都有明顯標記且易於清理。

然而，小部分用戶回報問題，指出元件上額外的 `class` 屬性有時會導致從外部服務新增的腳本無法驗證。

[Gerald Monaco](https://github.com/devknoll)，來自 Google Chrome 團隊，[想出了一種方法](https://github.com/vercel/next.js/pull/8020) 來保留清理行為而不需要在元件上使用類別名稱。

新行為是 Next.js 將插入另一個 `<meta>` 標籤，儲存它 (`next/head`) 渲染的元件數量。有了這個，Next.js 可以使用計數來清理現有元件。

因此，當多個元件注入到頁面的 `<head>` 時，此方法會減少初始 HTML payload 大小。

### 防止 Pages 目錄中的非頁面檔案[](#preventing-non-pages-in-the-pages-directory)

開始使用 Next.js 時，您做的第一件事是建立 `pages` 目錄。

慣例是 `pages` 目錄中的每個檔案都會變成應用程式中的路由。一個簡單的例子是 `pages/about.js` 會變成 `/about`。

隨著時間推移，我們收到了偶爾的報告，指出使用者的應用程式，無論大小，都有不佳的構建性能。

經進一步調查，這些使用者將整個元件結構建立於 `pages` 目錄中。

因為 `pages` 目錄中的每個檔案都被視為頁面，所以每個元件都被編譯為應用程式中的頁面。這造成了大量構建時開銷，為無效頁面生成 2 個以上的 JavaScript 檔案。

此外，這也會部分影響 Next.js 決定如何生成拆分代碼區塊的方式。這是因為 Next.js 使用不同頁面之間的庫使用启发式方法。

由於這個原因，我們必須確保使用者不會將此陷阱引入他們的 Next.js 應用程式。

Next.js 9 現在驗證 `pages` 目錄中的檔案是否匯出 React 元件。

在實際操作中，這意味著 Next.js 會顯示訊息提醒您，在 `pages` 目錄中發現潛在的非頁面。

這鼓勵使用者將不是頁面的檔案移至另一個目錄。從而，開發、生產環境和程式碼分割會更快且更準確。

### 執行時改進[](#runtime-improvements)

Next.js 框架由許多部分組成。其中之一是客戶端執行時環境。此執行時環境負責 hydration、客戶端路由及更多。

Hydration (水合)，也是這個改進專注的部分，這是使服務端渲染或多預先渲染的 HTML 變爲互動式所必需的。Hydration 新增事件處理程式並呼叫生命週期方法如 `useEffect()` 或 `componentDidMount` —— 使您的應用程式準備好給最終用戶。

此外，Next.js 處理的功能不僅限於基本 hydration —— 例如，設定客戶端路由、配置 `next/head`、以及透過 `next/dynamic` 載入更多應用程式邏輯。

這些責任每個都有自己特定的執行時部分。

在 `next/dynamic` 的情況下，Next.js 必須確保在服務端渲染的惰性載入元件已準備好在客戶端就緒。每個 `next/dynamic` 的使用都會生成額外的 JavaScript 包，且這些檔案必須在 hydration 之前載入以避免 hydration 不匹配。

先前，此執行時環境總是包含在 Next.js 執行時環境包中。現在，只有當您在應用程式中使用 `next/dynamic` 時，才會包含它。這意味著對於不需要使用 `next/dynamic` 的應用程式，下載、解析和執行的 JavaScript 更少。

### AppTree 支援[](#apptree-support)

React 生態系統中的一些庫以非常特定的方式實現服務端渲染。其中最值得注意的是 Apollo 的服務端渲染解決方案，稱為 `getDataFromTree`，透過渲染 React 樹並等待每個找到的 `Query` 結果，然後重新渲染 React 樹。

預設情況下，Next.js 會在 React 樹中新增一些上下文值，例如可使用 `useRouter` 讀取的路由。

`with-apollo` 範例過去渲染 React 樹的方式是透過渲染 `<App>` 並嘗試手動填補缺失的屬性。隨著 React Context 的加入，這已不再可能，因為上下文提供者是單獨的元件。

從 Next.js 9.0.4 開始，在 `getInitialProps` 的上下文物件中新增了名為 `AppTree` 的新屬性。它是專門為外部庫必須遍歷整個 React 樹的情況而新增的，例如 Apollo 的 `getDataFromTree`。

`with-apollo` 範例已 [更新以反映變更](https://github.com/vercel/next.js/blob/v9.0.4/examples/with-apollo/lib/with-apollo-client.js#L20-L34)。如果您已在應用程式中實現了 Apollo，建議更新以採用 `AppTree` 方式，以便在您的 Next.js 應用程式中正確工作 `useRouter` 和未來添加的其他 API。

如果您未使用 Apollo 或類似庫，我們建議嘗試避免使用 AppTree，因為 Next.js 團隊也不建議遍歷 React 樹。這會增加相當多的效能開銷，因為 React 樹會被渲染多次，而不是一次。

### `next-server` 套件的移除[](#removal-of-the-next-server-package)

當我們開始為 serverless 部署優化 Next.js 超過一年前，我們建立了一個名為 `next-server` 的套件。這個套件是實驗性的，並與 `next` 套件一起發布。它從未公開文件過，但是一個實驗，旨在建立最小可能的 Next.js 服務器執行時環境。

最終，這個套件取得了成功，確實讓生產服務器執行時環境變小。然而，我們想出了一種創新的新方法，利用 Next.js 編譯器和 [靜態分析](https://github.com/vercel/node-file-trace) 使執行時環境更小。

這樣做後，`next-server` 變得過時，並被 [serverless 目標](/docs#serverless-deployment) 取代。此目標具有比使用 `next-server` 套件作為 `next` 的替代者更優化的輸出。

雖然這個套件過時了，且不能直接使用，我們仍保留著它。這是因為它有在不同套件之間共享的內部功能，移動程式碼將需要非平凡的努力。

我們最近完成了這個努力，將程式碼從 `next-server` 移回了 `next` 套件。這意味著 Next.js 框架的所有程式碼現在都位於 `next` 套件中。

這使得初學者和資深貢獻者都更容易為 Next.js 做出貢獻。現在有單一的編譯流程和統一的構建配置。先前，`next` 和 `next-server` 有單獨的設定，以及對哪些程式碼屬於每個套件的任意約束。

### 升級 Next.js[](#upgrading-nextjs)

如果您的項目正在在較舊版本的 Next.js 上運行，我們建議升級到 Next.js 9。

在大多數情況下，升級不需要更改。您可以遵循 [升級指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md#migrating-from-v8-to-v9) 以確保流暢的升級體驗。

我們感謝所有社區貢獻者更新了自發布以來的指南。
