React 伺服器端組件 (React Server Components, RSC) 在 App Router 是一種新穎的範例模式，消除了與傳統方法相關的大量冗餘和潛在風險。鑑於其新穎性，開發人員並進而安全團隊可能會發現將其現有的安全協議與此模型對齊具有一定挑戰性。

本文旨在強調幾個需要注意的區域、內建了哪些防護措施，並包含應用程式稽核指南。我們特別關注意外資料暴露的風險。

選擇您的資料處理模型 [](#choosing-your-data-handling-model)
-----------------------------------------------------------------------

[React 伺服器端組件](/docs/app/building-your-application/rendering/server-components) 模糊了伺服器端與客戶端的界限。資料處理是了解資訊在哪裡處理以及隨後可供使用的關鍵。

我們首先要做的是選擇適合我們專案的資料處理方法。

*   [HTTP APIs](#http-apis) (適用於現有大型專案/組織，推薦)
*   [資料存取層](#data-access-layer) (適用於新專案，推薦)
*   [組件層級資料存取](#component-level-data-access) (適用於原型設計與學習，推薦)

我們建議您堅持使用一種方法，不要混合搭配過多。這對於在您程式碼庫中工作的開發人員和安全稽核人員來說，可以清楚地預期他們會看到什麼。例外情況會顯得可疑。

### HTTP APIs[](#http-apis)

如果您在現有專案中採用伺服器端組件，推薦的做法是在執行時將伺服器端組件預設視為不安全/不可信，如同 SSR 或在客戶端中執行。因此，不假設內部網路或信任區域，工程師可以應用零信任 (Zero Trust) 概念。此外，您只像從客戶端執行一樣，從伺服器端組件使用 `fetch()` 呼叫自訂 API 端點，如 REST 或 GraphQL。傳遞任何 Cookie。

如果您之前有連接資料庫的現有 `getStaticProps`/`getServerSideProps`，您可能想合併模型並將這些移至 API 端點，這樣您就有了一種做事的方法。

找出任何假設來自內部網路的 `fetch` 是安全的存取控制。

這種方法讓您能保留現有的組織架構，在這種架構下，專注於安全的現有後端團隊能套用現有的安全實務。如果這些團隊使用 JavaScript 以外的語言，這種方法相當合適。

它仍然可以利用伺服器端組件的許多好處，例如向客戶端發送較少的程式碼，以及固有的資料階梯可以以低延遲執行。

### 資料存取層 [](#data-access-layer)

我們建議的新專案做法是在您的 JavaScript 程式碼庫內建立一個獨立的資料存取層 (Data Access Layer)，並在該處合併所有資料存取。此方法確保一致的資料存取並降低發生授權錯誤的機會。此外，由於您將資料合併到單一套件中，因此更容易維護。可能還提供了更好的團隊凝聚力，使用單一流行語言。您也可以獲得更好的效能，具有較低的執行時開銷，並能夠在請求的不同部分之間共享記憶體快取。

您建置一個內部 JavaScript 套件，該套件在向呼叫者提供之前提供自訂的資料存取檢查。類似於 HTTP 端點，但在相同的記憶體模型中。每個 API 應接受目前的用戶並檢查用戶是否有权查看此數據，然後再傳回它。原則在於，伺服器端組件函數的函式體只能看到目前發出請求的用戶有權訪問的數據。

從這一點開始，實施 API 的標準安全慣例就接管了。

data/auth.tsx

```
import { cache } from 'react';
import { cookies } from 'next/headers';
 
// Cached helper methods makes it easy to get the same value in many places
// without manually passing it around. This discourages passing it from Server
// Component to Server Component which minimizes risk of passing it to a Client
// Component.
export const getCurrentUser = cache(async () => {
  const token = cookies().get('AUTH_TOKEN');
  const decodedToken = await decryptAndValidate(token);
  // Don't include secret tokens or private information as public fields.
  // Use classes to avoid accidentally passing the whole object to the client.
  return new User(decodedToken.id);
});
```

data/user-dto.tsx

```
import 'server-only';
import { getCurrentUser } from './auth';
 
function canSeeUsername(viewer: User) {
  // Public info for now, but can change
  return true;
}
 
function canSeePhoneNumber(viewer: User, team: string) {
  // Privacy rules
  return viewer.isAdmin || team === viewer.team;
}
 
export async function getProfileDTO(slug: string) {
  // Don't pass values, read back cached values, also solves context and easier to make it lazy
 
  // use a database API that supports safe templating of queries
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`;
  const userData = rows[0];
 
  const currentUser = await getCurrentUser();
 
  // only return the data relevant for this query and not everything
  // <https://www.w3.org/2001/tag/doc/APIMinimization>
  return {
    username: canSeeUsername(currentUser) ? userData.username : null,
    phonenumber: canSeePhoneNumber(currentUser, userData.team)
      ? userData.phonenumber
      : null,
  };
}
```

這些方法應暴露物件，以便可以直接安全地傳輸到客戶端。我們喜歡稱這些資料傳輸物件 (Data Transfer Objects, DTO)，以釐清它們已由客戶端消費準備就緒。

在實際情況下，它們可能僅由伺服器端組件消費。這會創建一層架構，使安全稽核可以專注於資料存取層，而使用者介面可以快速迭代。較小的暴露面積和較少的程式碼範圍使得更容易發現安全問題。

```
import {getProfile} from '../../data/user'
export async function Page({ params: { slug } }) {
  // This page can now safely pass around this profile knowing
  // that it shouldn't contain anything sensitive.
  const profile = await getProfile(slug);
  ...
}
```

金鑰可以儲存在環境變數中，但在這種方法下，只有資料存取層應存取 `process.env`。

### 組件層級資料存取 [](#component-level-data-access)

另一種做法是在您的伺服器端組件 (Server Components) 中直接放置您的資料庫查詢。此僅適用於快速迭代與原型設計。例如，對於小產品與小團隊，每個人都了解風險及如何監視。

在這種方法中，您將需要仔細稽核您的 `"use client"` 檔案。在稽核與審查 Pull Requests 時，查看所有導出函數，以及類型簽章是否接受過於寬泛的物件（如 `User`），或包含 `token` 或 `creditCard` 等屬性。即使是隱私敏感欄位（如 `phoneNumber`）也需要額外審查。客戶端組件不應接受比執行其工作所需的最小數據更多的數據。

```
import Profile from './components/profile.tsx';
 
export async function Page({ params: { slug } }) {
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`;
  const userData = rows[0];
  // EXPOSED: This exposes all the fields in userData to the client because
  // we are passing the data from the Server Component to the Client.
  // This is similar to returning `userData` in `getServerSideProps`
  return <Profile user={userData} />;
}
```

```
'use client';
// BAD: This is a bad props interface because it accepts way more data than the
// Client Component needs and it encourages server components to pass all that
// data down. A better solution would be to accept a limited object with just
// the fields necessary for rendering the profile.
export default async function Profile({ user }: { user: User }) {
  return (
    <div>
      <h1>{user.name}</h1>
      ...
    </div>
  );
}
```

永遠使用參數化查詢，或會為您這樣做的資料庫套件，以避免 SQL 注入攻擊。

唯伺服器端執行 [](#server-only)
---------------------------

應該僅在伺服器端執行的程式碼可以標記為:

```
import 'server-only';
```

這將導致建構錯誤，如果客戶端組件試圖匯入此模組。這可用於確保專屬/敏感程式碼或內部業務邏輯不會意外洩漏到客戶端。

轉移資料的主要方式是使用 React 伺服器端組件協定，當將 prop 傳遞給客戶端組件時會自動執行。此序列化和支援的 JSON 超集。不支援轉移自訂類別，並會導致錯誤。

因此，避免過於大型物件意外暴露在客戶端的一個好技巧是使用 `class` 作為您的資料存取記錄。

在即將發行的 Next.js 14 版中，您還可以嘗試實驗性的 [React Taint APIs](https://react.dev/reference/react/experimental_taintObjectReference)，透過在 `next.config.js` 中啟用 `taint` 旗標。

next.config.js

```
module.exports = {
  experimental: {
    taint: true,
  },
};
```

這允許您標記不允許作為本身傳遞給客戶端的物件。

app/data.ts

```
import { experimental_taintObjectReference } from 'react';
 
export async function getUserData(id) {
  const data = ...;
  experimental_taintObjectReference(
    'Do not pass user data to the client',
    data
  );
  return data;
}
```

app/page.tsx

```
import { getUserData } from './data';
 
export async function Page({ searchParams }) {
  const userData = getUserData(searchParams.id);
  return <ClientComponent user={userData} />; // error
}
```

這並不能保護防止從該物件中提取資料欄位並傳遞下去：

app/page.tsx

```
export async function Page({ searchParams }) {
  const { name, phone } = getUserData(searchParams.id);
  // Intentionally exposing personal data
  return <ClientComponent name={name} phoneNumber={phone} />;
}
```

對於像是權杖的唯一字串，也可以透過 [`taintUniqueValue`](https://react.dev/reference/react/experimental_taintUniqueValue) 來阻擋原始值。

app/data.ts

```
import { experimental_taintObjectReference, experimental_taintUniqueValue } from 'react';
 
export async function getUserData(id) {
  const data = ...;
  experimental_taintObjectReference(
    'Do not pass user data to the client',
    data
  );
  experimental_taintUniqueValue(
    'Do not pass tokens to the client',
    data,
    data.token
  );
  return data;
}
```

然而，即使這樣也無法阻擋衍生值。

更好的是避免資料首先进入伺服器端組件——使用資料存取層。變異檢查提供了一層額外的保護，以針對錯誤，指定該值，請注意，函數和類別已被禁止傳遞給客戶端組件。更多的層級可以將某物錯過的風險降至最低。

預設情況下，環境變數僅在伺服器端可用。根據約定，Next.js 也會將任何前綴為 `NEXT_PUBLIC_` 的環境變數暴露給客戶端。這允許您暴露某些應該對客戶端可用的明確配置。

SSR 與 RSC[](#ssr-vs-rsc)
-------------------------

為了初始載入，Next.js 將在伺服器端執行伺服器端組件與客戶端組件以產出 HTML。

伺服器端組件 (RSC) 在從客戶端組件分開的模組系統中執行，以避免在兩個模組之間意外暴露資訊。

透過伺服器端渲染 (SSR) 渲染的客戶端組件應視為與瀏覽器客戶端相同的安全政策。它不應該獲得以獲得任何特權數據或私有 API 的訪問權限。強烈不建議使用技巧來嘗試繞過此保護（例如將數據暫存在全域物件上）。原則在於此程式碼應能夠在伺服器上與客戶端一樣執行。為了符合預設安全慣例，如果客戶端組件匯入 `server-only` 模組，Next.js 會使建構失敗。

讀取 [](#read)
-------------

在 Next.js App Router 中，從資料庫或 API 讀取資料是透過呈現伺服器端組件頁面來實現的。

頁面的輸入是 URL 中的 `searchParams`、從 URL 映射的动态參數以及標頭。這些可能被客戶端操縱為不同的值。它們不應受到信任，並且每次讀取時都需要重新驗證。例如，搜索參數應用於追蹤 `?isAdmin=true` 等內容是不恰當的。僅僅因為使用者在 `/[team]/` 上，並不意味著他們有權訪問該團隊，這需要在讀取資料時進行驗證。原則是在讀取資料時始終重新讀取存取控制和 `cookies()`。不要將其作為 prop 或參數傳遞。

呈現伺服器端組件絕不應執行副作用，例如變更。這不僅限於伺服器端組件。React 甚至當呈現客戶端組件（`useEffect` 之外）時，自然 discourages 副作用，例如透過雙重呈現 (double-rendering) 的方式。

此外，在 Next.js 中，沒有在呈現期間設定 Cookie 或觸發快取重新驗證的方法。這也 discourages 使用呈現來執行變更。

例如：`searchParams` 應用於執行副作用，例如儲存變更或登出。應改用 Server Actions。

這意味著當按預期使用時，Next.js 模型從未使用 GET 請求來執行副作用。這有助於避免 CSRF 問題的大量來源。

Next.js 確實支援自訂路由處理常式 (`route.tsx`)，這可以在 GET 時設定 Cookie。被視為逃生通道，不屬於一般模型的一部分。這些必須明確選擇接受 GET 請求。沒有可能會意外收到 GET 請求的萬用字串處理常式。如果您決定建立自訂 GET 處理常式，可能需要額外的稽核。

寫入 [](#write)
---------------

在 Next.js App Router 中執行寫入或變更的慣用方式是使用 [Server Actions](/docs/app/building-your-application/data-fetching/forms-and-mutations)。

actions.ts

```
'use server';
 
export function logout() {
  cookies().delete('AUTH_TOKEN');
}
```

`"use server"` 注解暴露了一個端點，使所有導出的函數都可被客户端調用。識別碼目前為原始程式碼位置的字串。只要使用者獲得操作 ID 的處理，就可以使用任何參數調用它。

因此，這些函數應該始終首先驗證目前用戶是否允許執行此操作。函數還應驗證每個參數的完整性。這可以手動完成，或使用 `zod` 等工具完成。

actions.ts

```
"use server";
 
export async function deletePost(id: number) {
  if (typeof id !== 'number') {
    // The TypeScript annotations are not enforced so
    // we might need to check that the id is what we
    // think it is.
    throw new Error();
  }
  const user = await getCurrentUser();
  if (!canDeletePost(user, id)) {
    throw new Error();
  }
  ...
}
```

### 閉包 [](#closures)

Server Actions 也可以編碼在 [閉包](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures) 中。這允許操作與渲染時使用的數據快照關聯，以便您可以在調用此操作時使用：

app/page.tsx

```
export default function Page() {
  const publishVersion = await getLatestVersion();
  async function publish() {
    "use server";
    if (publishVersion !== await getLatestVersion()) {
      throw new Error('The version has changed since pressing publish');
    }
    ...
  }
  return <button action={publish}>Publish</button>;
}
 
```

必須將閉包的快照傳送到客戶端，並在伺服器端被調用時傳回。

在 Next.js 14 中，閉包變數在傳送至客戶端之前會使用操作 ID 進行加密。預設情況下，在 Next.js 專案建構期間會自動產生私密金鑰。每次重新建構都會產生新的私密金鑰，這意味著每個 Server Action 僅能針對特定建構進行調用。您可能想使用 [版本跳斜保護](https://vercel.com/blog/version-skew-protection)，以確保在重新部署時調用正確版本。

如果您需要更頻繁旋轉或在多次建構之間永久性的金鑰，可以使用 `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 環境變數手動配置它。

透過加密所有閉包變數，您不會在它們中意外暴露任何機密資訊。透過簽署它，可以讓攻擊者更難以干擾操作的輸入。

使用閉包的另一種替代方法是使用 JavaScript 的 `.bind(...)` 函數。**這些未加密。** 這提供了效能的選擇，並且與 `client` 上的 `.bind()` 一致。

app/page.tsx

```
async function deletePost(id: number) {
  "use server";
  // verify id and that you can still delete it
  ...
}
 
export async function Page({ slug }) {
  const post = await getPost(slug);
  return <button action={deletePost.bind(null, post.id)}>
    Delete
  </button>;
}
```

原則在於，Server Actions (`"use server"`) 的參數列表必須始終視為敵意的，並且輸入必須經過驗證。

### CSRF[](#csrf)

所有 Server Actions 都可以透過純 `<form>` 調用，這可能會讓他們容易受到 CSRF 攻擊。在幕後，Server Actions 總是使用 POST 實作，並且僅允許此 HTTP 方法調用它們。這本身就防止了現代瀏覽器中大多數的 CSRF 漏洞，特別是由於 Same-Site Cookies 為預設值。

作為額外保護，Next.js 14 中的 Server Actions 也會比較 `Origin` 標頭與 `Host` 標頭（或 `X-Forwarded-Host`）。如果它們不匹配，Action 將被拒絕。換句話說，Server Actions 只能在使用它的頁面的相同主機上調用。非常古老的不受支援且過時的瀏覽器，如果它們不支援 `Origin` 標頭，可能會存在風險。

Server Actions 不使用 CSRF 權杖，因此 HTML 清理至關重要。

當改用自訂路由處理常式 (`route.tsx`)，可能需要額外的稽核，因為 CSRF 保護必須在那裏手動完成。傳統規則在那裏適用。

錯誤處理 [](#error-handling)
---------------------------------

發生錯誤。當錯誤在伺服器端擲回時，它們最終會在客戶端程式碼中重擲回，以在 UI 中處理。錯誤訊息和堆疊追蹤可能會包含敏感資訊。例如：`[信用卡號碼] 不是有效的電話號碼。`

在生產模式中，React 不會將錯誤或拒絕的 promises 發送到客戶端。相反，傳送代表錯誤的雜湊。此雜湊可用於關聯多個相同錯誤並與伺服器日誌關聯錯誤。React 將錯誤訊息替換為其自己的通用訊息。

在開發模式中，伺服器錯誤仍會以純文字發送到客戶端，以協助除錯。

對於生產工作負載，始終在生產模式下執行 Next.js 非常重要。開發模式並不針對安全性和效能進行最佳化。

自訂路由和中介軟體 [](#custom-routes-and-middleware)
-------------------------------------------------------------

[自訂路由處理常式](/docs/app/building-your-application/routing/route-handlers) 和 [中介軟體](/docs/app/building-your-application/routing/middleware) 被視為針對無法使用任何其內建功能實現的功能的低級逃生通道。這也打開了框架原本會保護的潛在陷阱。能力越大，責任越大。

如上所述，`route.tsx` 路由可以實作自訂的 GET 和 POST 處理常式，如果您沒有正確執行，可能會受到 CSRF 問題的影響。

中介軟體可用於限制對特定頁面的存取。通常最好以允許清單 (allow list) 而非拒絕清單 (deny list) 執行此操作。這是因為知道所有存取資料的不同方式可能很棘手，例如是否有重新寫入或客戶端請求。

例如，通常只會想到 HTML 頁面。Next.js 也支援客戶端導航，該導航可以載入 RSC/JSON 有效負載。在 Pages Router 中，這過去是在自訂 URL 中。

為了使撰寫匹配器更容易，Next.js App Router 總是使用頁面的純 URL，用於初始 HTML、客戶端導航和 Server Actions。客戶端導航使用 `?_rsc=...` 搜索參數作為快取斷開。

Server Actions 位於它們使用的頁面上，因此繼承相同的存取控制。如果 Middleware 允許讀取頁面，您還可以呼叫該頁面上的操作。要限制對頁面上 Server Actions 的存取，您可以禁止該頁面上的 POST HTTP 方法。

稽核 [](#audit)
---------------

如果您正在稽核 Next.js App Router 專案，以下是一些我們建議特別查看的內容：

*   **資料存取層 (Data Access Layer)**。是否有已建立的資料存取層 (Data Access Layer) 慣例？驗證資料庫套件和環境變數是否未在資料存取層之外匯入。
*   `"use client"` 文件。組件 prop 是否預期私有數據？類型簽章是否過於寬泛？
*   `"use server"` 文件。操作參數是否在操作中或資料存取層內進行驗證？使用者是否在操作內重新授權？
*   `/[param]/`。帶有括號的資料夾是使用者輸入。參數是否已驗證？
*   `middleware.tsx` 和 `route.tsx` 具有大量功能。花更多時間使用傳統技術稽核這些工具。進行滲透測試或漏洞掃描，定期執行或與您團隊的軟體開發生命週期一致。
