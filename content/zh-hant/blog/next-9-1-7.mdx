[Next.js 9](/blog/next-9) 六（6）個月前發布，隨後在 [Next.js 9.1](/blog/next-9-1) 三（3）個月前發布。

這兩次發布為 Next.js 添加了功能強大的新功能，同時沒有增加我們的基線客戶端運行時大小。

此後，我們致力於改進整個框架：[9.1.1](https://github.com/vercel/next.js/releases/tag/v9.1.1), [9.1.2](https://github.com/vercel/next.js/releases/tag/v9.1.2), [9.1.3](https://github.com/vercel/next.js/releases/tag/v9.1.3), [9.1.4](https://github.com/vercel/next.js/releases/tag/v9.1.4), [9.1.5](https://github.com/vercel/next.js/releases/tag/v9.1.5), [9.1.6](https://github.com/vercel/next.js/releases/tag/v9.1.6), 和 [9.1.7](https://github.com/vercel/next.js/releases/tag/v9.1.7)。

讓我們深入了解這些發布的改進！

*   [**3% - 8%+ 客戶端 JavaScript 體積減小**](#3--8-smaller-client-side-javascript-size): 我們進一步優化了應用程序輸出，為 Hello World 應用程序減少了 7.5kB。更複雜的應用程序可節省高達 8% 或更多。
*   [**重設計的生產環境構建 CLI 輸出**](#redesigned-production-build-cli-output): 生產環境構建輸出現在以更易理解的格式顯示壓縮文件的大小。
*   [**新的內建填充程式 (Polyfills): fetch(), URL 和 Object.assign**](#new-built-in-polyfills-fetch-url-and-objectassign): 應用程序可以在舊瀏覽器中使用 `fetch()` API、`URL` 和 `Object.assign` 而無需考慮兼容性问题。
*   [**頁面加載優化：更好的 FCP 和 TTI**](#optimized-page-loading-better-fcp-and-tti): 我們與 Google Chrome 團隊密切合作，以最大化頁面加載性能。這為最終用戶提供了更好的體驗。
*   [**支持最新的 JavaScript 功能**](#support-for-the-latest-javascript-features): 我們致力於確保您始终可以使用最新的 JavaScript 功能，包括 [可選鏈](https://github.com/tc39/proposal-optional-chaining)、[空值合併](https://github.com/tc39/proposal-nullish-coalescing) 及其他穩定的 ES2020 功能。
*   [**`next export`應用程序的零配置部署支持**](#zero-config-deployment-support-for-next-export-applications): `next export`驅動的應用程序現在可以在 [Vercel](https://vercel.com) 上以零配置的方式部署。
*   [**React Strict Mode 合規性及可選**](#react-strict-mode-compliance-and-opt-in): Next.js 的所有客戶端運行時現在都與 React 的 Strict Mode 兼容。我們還添加了配置選項以啟用整個應用程序的此模式。
*   [**針對 React Nightly 構建的自動測試**](#automated-testing-against-nightly-react-builds): Next.js 已針對 [React 的 next 通道](https://reactjs.org/blog/2019/10/22/react-release-channels.html) 進行自動測試，確保與未來版本兼容。

這些益處皆為非破壞性且完全後向相容。更新時只需執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

### 3% - 8%+ 客戶端 JavaScript 體積減小[](#3---8-smaller-client-side-javascript-size)

與 Google Chrome 團隊合作，所有 Next.js 應用程序將受益於 **7.5kB 或更多**的體積減少。

基本應用程序應能看到 3-4% 的體積減少，而更先進的應用程序可能會看到 6-8% 或更多的體積減少！

|  | 9.0.x | 9.1.x | delta |
| --- | --- | --- | --- |
| 基本應用程序 | 68.9kB | 66.1kB | **4.1%** _更小_ |

這些節省部分歸因於 [取代 `url` 套件的前端版本](https://github.com/GoogleChromeLabs/native-url)，改用基於 [URL API 應用程式介面 (URL API)](https://developer.mozilla.org/docs/Web/API/URL) 的版本。

通過提供常用套件的內建、小型填充程式，實現了更多的體積減少。您可以在此[閱讀更多關於這些填充程式的信息](#new-built-in-polyfills-fetch-url-and-objectassign)。

最後，我們優化了 JSX 輸出，這將根據您應用程序中的 JSX 數量帶來相應的節省。

### 重設計的生產環境構建 CLI 輸出[](#redesigned-production-build-cli-output)

CLI 的生產環境構建輸出經過了重新設計以提高清晰度。由於 Next.js 是一個混合應用程序框架，每個頁面可能具有不同的特徵。

新輸出將每個頁面分類為以下之一：

*   **服務器端渲染（伺服器）**: 頁面在運行時服務器端渲染，意味著它使用 `getInitialProps` 或 [`getServerProps (proposal)`](https://github.com/vercel/next.js/issues/9524)
*   **自動靜態優化（靜態）**: 頁面在構建時渲染為靜態 HTML，將作為靜態文件提供（不使用初始屬性）
*   **使用計算數據靜態生成（SSG）**: 頁面在構建時生成為靜態 HTML/JSON，將作為靜態文件提供（使用 [`getStaticProps (proposal)`](https://github.com/vercel/next.js/issues/9524))

此外，新輸出顯示每個頁面的 Gzip 大小——這些大小排除了所有頁面共享的文件（單獨顯示）。

每個頁面的大小將根據其感知到的用戶體驗進行着色：

*   **小於 130kB**: _綠色_ — 您的應用程序在大多數網絡和設備條件下應該是可操作的。
*   **介於 130kB 和 170kB 之間**: _黃色_ — 您的應用程序在 [全球基準設備 + 網絡條件](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 下接近 5-6 秒的加載時間。
*   **超過 170kB**: _紅色_ — 您的應用程序在 [相同條件下](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 可能需要超過 6 秒來加載。

### 新的內建填充程式 (Polyfills): fetch(), URL 和 Object.assign[](#new-built-in-polyfills-fetch-url-and-objectassign)

在檢查了許多用戶的應用和我們的示例後，我們發現大多數應用都帶有一套類似的填充程式。在某些情況下，應用程序甚至為同一功能擁有重複的填充程式。

為了解決這個問題，我們與 Google Chrome 團隊合作，為我們觀察到的三個最常用的構建了填充程式。

通過差異加載，這些填充程式在全球上 **不為 83% 的網絡流量加載**。這意味著您的大多數用戶將不會下載與這些填充程式相關的字節——只有在必要時才會下載它們。

此外，我們現在構建的任何知名填充程式都將從您的生產環境構建中完全消除。這意味著您不會因為您的依賴項間接導入這些 API 之一的填充程式而付出代價。

內建 API 列表及它們過時的填充程式如下：

*   [**fetch()**](https://developer.mozilla.org/docs/Web/API/Fetch_API) — 替換：`whatwg-fetch` 和 `unfetch`。
*   [**URL**](https://developer.mozilla.org/docs/Web/API/URL) — 替換：[`url` 套件（Node.js API）](https://nodejs.org/api/url.html)。
*   [**Object.assign()**](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Object/assign) — 替換：`object-assign`, `object.assign`, 和 `core-js/object/assign`。

如果您在服務器上使用 `fetch()`，您仍然需要導入 `isomorphic-fetch` 或 `isomorphic-unfetch`。

此更改完全非破壞性，所有填充程式都是使用可用的最規範版本製作的。其結果是 **消除了最多 18kB 的 JavaScript** 在現代瀏覽器的生產環境綁定中。

### 頁面加載優化：更好的 FCP 和 TTI[](#optimized-page-loading-better-fcp-and-tti)

Next.js 已優化其預加載實現，以減少 FCP 和整體 TTI。

通過繞過瀏覽器錯誤，CSS（當使用時）現在會正確優先於 JavaScript。這導致更快的首次內容繪制，為最終用戶帶來更快的視覺完備網站。

此外，我們優化了下頁面預取，使用低優先級網絡請求。

此外，這些請求僅在瀏覽器閒置時間發生，從而實現更快的交互時間。這是因為瀏覽器會優先讓您的應用程序可互動，而不是樂觀地預取。

### 支持最新的 JavaScript 功能[](#support-for-the-latest-javascript-features)

Next.js 具有先進且高度優化的編譯管道，使您能夠使用最新的 JavaScript 功能。我們最近引入的優化直接促成了 [3-8% 的體積減少](#3--8-smaller-client-side-javascript-size)。

這些 JavaScript 功能可以不考慮瀏覽器兼容性地利用，因為我們會自動編譯您的代碼以支持所有瀏覽器（不包括終止支持版本）。這包括 ES6+ 功能，例如 [async/await (ES2017)](https://github.com/tc39/ecmascript-asyncawait)、[对象 Rest/Spread 屬性 (ES2018)](https://github.com/tc39/proposal-object-rest-spread)、[动态 `import()` (ES2020)](https://v8.dev/features/dynamic-import) 以及更多！

在這次發布中，我們很高興地宣布支持 [可選鏈 (Stage 4)](https://github.com/tc39/proposal-optional-chaining) 和 [空值合併 (Stage 4)](https://github.com/tc39/proposal-nullish-coalescing)。

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
function Page(props) {
  return (
    <p>{props?.deeply?.nested?.value}</p>
    /* ⬆ If deeply.nested.value is not available it won't render */
  );
}
 
export default Page;
```

> 可選鏈運算子 (`?.`)

### `next export`應用程序的零配置部署支持[](#zero-config-deployment-support-for-next-export-applications)

Next.js 的 `next export` 命令現在可以與 Vercel 的 [零配置](https://vercel.com/blog/zero-config) 即開即用。在此更改之前，使用 `next export` 的用戶必須創建自定義 `now.json`。

在 [Vercel](https://vercel.com) 上利用 Next.js 的導出模式與在 `package.json` 中具有以下 `build` 腳本一樣簡單：

package.json

```
{
  "scripts": {
    "build": "next build && next export"
  }
}
```

然後，您可以使用 **單一命令** 將應用程序部署到 [Vercel](https://vercel.com)。如果您尚未安裝 [Vercel](https://vercel.com)，可以通過安裝 [Vercel CLI](https://vercel.com/cli) 來完成。

終端機

```
now
```

### React Strict Mode 合規性及可選[](#react-strict-mode-compliance-and-opt-in)

完整的 Next.js 運行時現在符合 Strict Mode。這包括對 Next.js 的 head 管理器 (`<Head>`)、`next/dynamic` 和其他核心功能的更新。這意味著運行時現在利用 Hooks 或已消除棄用的生命周期，並使用 React 的新 Context API。

我們還為您添加了一個方便的選項，以啟用在整個應用程序中使用 Strict Mode。

要執行此操作，請在 `next.config.js` 中配置以下選項：

next.config.js

```
module.exports = {
  reactStrictMode: true,
};
```

如果您或您的團隊尚未準備好在整個應用程序中使用 Strict Mode，這沒問題！您可以逐頁面地增量遷移 [使用 `<React.StrictMode>`](https://reactjs.org/docs/strict-mode.html)。

雖然 **非強制**，但 Strict Mode 將在未來帶來大量優化。因此，我們建議您盡快探討這一可能性！

### 針對 React Nightly 構建的自動測試[](#automated-testing-against-nightly-react-builds)

在與 React Core Team 的 [密切合作](https://github.com/vercel/next.js/pull/9165) 下，我們現在每 12 小時[測試](https://github.com/vercel/next.js/pull/9151) Next.js 與 React 的 next 通道。

這些測試有助於確保我們已準備好並與 React 的未來版本兼容。兼容性是 Next.js 非常重視的事情，我們致力於 Next.js 的長期 API 穩定性。

此過程已由 React Core Team [記錄在案](https://reactjs.org/blog/2019/10/22/react-release-channels.html)，希望 React 生態系統中的其他項目跟進。
