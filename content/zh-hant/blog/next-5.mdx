我們很榮幸地向世界介紹 Next.js 5.0。即刻起適用於 npm。若要升級，請執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

> 除了更新 Next.js 版本外，我們也升級了相依套件 (peer dependencies) `react` 和 `react-dom`

Next.js 是適用於通用、伺服器渲染（或靜態預渲染）React.js 應用程式的工具包。開發任何規模的應用程式，起步過程就像執行 `next` 一樣簡單。([繼續閱讀](https://vercel.com/blog/next)。)

每次發布新版本時，我們都致力於保留向後兼容性，提供簡單的升級路徑，僅在絕對必要時進行 API 變更。Next.js 5.0 也不例外。

然而，在底層，Next.js 經歷了根本性的轉變，以實現強大的新用例和可擴展性。我們首先讓 Next.js 為服務器和客戶端代碼共用一套普遍的 Webpack 流程。

通用 Webpack 與 Next 插件[](#universal-webpack-and-next-plugins)
-------------------------------------------------------------------------

Next.js 利用現有的強大工具，如 Webpack、Babel 和 Uglify，為最終用戶呈現為極其簡單的介面：`next`（開發）、`next build`（為生產環境準備）和 `next start`（服務）或 `next export` 以預渲染為靜態文件。

我們早期做出的決定之一，就是針對這些工具的配置提供非常強大的擴展點。我們不僅僅追求易用性，還希望能實現靈活性，讓您能夠按照自己的意願擴展這個工具包。

例如，您可以在您的 `next.config.js` 中設置一個 `webpack` 屬性 來 [擴展 Next.js webpack 配置](https://github.com/vercel/next.js#customizing-webpack-config)。

因為 webpack 在生產和開發環境下執行方式不同，我們當時決定將其製成一個 **函數** 來裝飾我們的預設 webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    // modify it!
    return config;
  },
};
```

> `next.config.js` 文件的範例

然而，Webpack 僅會在客戶端（瀏覽器）捆綁包上執行，您將無法利用這個優秀的工具鏈進行伺服器渲染。

我們很高興地宣布，我們已對代碼庫進行了大量重構，以**讓 Webpack 通用運作**。

從您的角度看，唯一變化的就是有一個額外的 `isServer` 屬性傳遞給上方的裝飾函數。不過，新的語義意味著 Webpack loader 的廣泛生態系統現在可供您使用。

### CSS、LESS、SASS、SCSS 與 CSS Modules[](#css-less-sass-scss-and-css-modules)

我們最常被請求的功能之一是匯入 CSS 文件並利用 Webpack loader 的能力：

```
import './index.css';
 
export default function Index() {
  return (
    <div>
      <p>I love CSS!</p>
    </div>
  );
}
```

> 通用 Webpack 支援下使用 CSS 匯入的範例頁面（`pages/index.js`）

為了實現這功能，您可以引入需要的 loader 作為相依套件 (peer dependencies)：

終端機

```
npm i --save css-loader style-loader postcss-loader
```

> Next.js 讓您可以自由選擇需要的 loader，並隨意升級至不同版本。

接著擴展配置以設定您的 loader。在 `next.config.js` 中：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dev, isServer } = options;
    const extractCSSPlugin = new ExtractTextPlugin({
      filename: 'static/style.css',
      disable: dev,
    });
    config.module.rules.push({
      test: /\\.css$/,
      use: cssLoaderConfig(extractCSSPlugin, {
        cssModules,
        dev,
        isServer,
      }),
    });
    return config;
  },
};
```

> 擴展原始 webpack 配置為您提供極大的靈活性與控制權

雖然我們的一般建議是使用元件級別（component-local）的樣式解決方案，例如 [內建的 `styled-jsx` Babel 插件](https://github.com/vercel/next.js#built-in-css-support)，但我們相信 CSS loader 有許多重要優勢，例如輕鬆重複使用現有的 CSS 代碼庫，以及大程度簡化舊代碼庫遷移至 Next.js。

我們並非預設啟用所有能想到的功能和 loader，而是引入了 [**Next.js 插件**](https://github.com/vercel/next-plugins)，它們是 `簡單函數，用於裝飾您的配置`。不像上面那樣手動擴展配置來設定 loader，您只需執行以下操作：

```
const withCss = require('next-css');
module.exports = withCss({
  /* extra optional config */
});
```

> 啟用匯入 .css 文件只需引入 next-css

閱讀更多關於 [CSS Loaders](https://github.com/vercel/next-plugins/tree/master/packages/next-css) 與 Next.js 的使用說明，或參考我們已為您創作的套件：

| 加載器 | 套件 |
| --- | --- |
| CSS | [next-css](https://github.com/vercel/next-plugins/tree/master/packages/next-css) |
| LESS | [next-less](https://github.com/vercel/next-plugins/tree/master/packages/next-less) |
| SASS | [next-sass](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) |

我們的目標是賦權社群開發並成長一個實用簡易的擴展生態系統。為此，我們開放了 [next-plugins](https://github.com/vercel/next-plugins) monorepo，讓 Next.js 社群維護。所有 PR 均歡迎！

### TypeScript 支援[](#typescript-support)

JavaScript 生態系統中 [發展最迅速的技術](https://stateofjs.com/2017/flavors/results) 之一就是 TypeScript。它甚至 [已獲 Babel 7 官方支援](https://github.com/babel/babel/tree/master/packages/babel-preset-typescript)，這意味著 Next.js 只需 [自訂您的 .babelrc](https://github.com/vercel/next.js#customizing-babel-config) 即可自然支援它。

與此同時，感謝我們新的通用 Webpack 支援，您今天即可獲得完整的 TypeScript 支援！

您可以這樣擴展您的 webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dir, defaultLoaders } = options;
    config.resolve.extensions.push('.ts', '.tsx');
    config.module.rules.push({
      test: /\\.+(ts|tsx)$/,
      include: [dir],
      exclude: /node_modules/,
      use: [
        defaultLoaders.babel,
        { loader: 'ts-loader', options: { transpileOnly: true } },
      ],
    });
    return config;
  },
};
```

> 我們只需要啟用 `ts-loader`

與 CSS loader 和預處理器一樣，TypeScript 是最常被請求的功能之一。為了讓將其併入專案能像其他 loader 一樣簡單，我們現在有一個 [next-typescript 插件](/) 您可以放入您的 next.config.js 文件中：

next.config.js

```
const withTs = require('next-typescript');
module.exports = withTs({
  /* additional config*/
});
```

> 插件可以輕鬆組合：它們只是函數

### React 替代庫與模組替換[](#better-support-for-react-altlibs--module-overloading)

隨著時間推移，出現了許多 React 的即用替換實作方案。其中一些著名的有 [`preact`](https://preactjs.com/)、[`nervjs`](https://github.com/NervJS/nerv) 和 [`inferno`](https://github.com/infernojs/inferno)。

其他一些庫專注於替換 DOM 渲染器，例如 [`react-dom-lite`](https://github.com/jquense/react-dom-lite)，旨在通過引入一些關於瀏覽器相容性的輕微妥協來生成更小的 React 構建。

通用 Webpack 支援讓將這些庫作為即用替換方案整合的過程更加容易。與其它插件一樣，要在使用 Next.js 搭配 preact 時只需這樣做：

終端機

```
npm i @zeit/next-preact preact preact-compat
```

> 我們安裝 preact 插件與必要的相依套件

```
const withPreact = require('@zeit/next-preact');
module.exports = withPreact();
```

> 我們適用於 preact 的新的 next.config.js

查看非常簡單的 [@zeit/next-preact](https://github.com/vercel/next-plugins/tree/master/packages/next-preact) 模組，或建立您自己的！

### 生產環境中的可選外部 Source Maps[](#optional-external-sourcemaps-in-production)

既然 webpack 已由 Next.js 用於客戶端和伺服器代碼，在生產環境構建中啟用 Source Maps 只是對其配置的小調整。

在開發環境中自動啟用 Source Maps，因此我們針對生產環境使用不同的配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
    }
    return config;
  },
};
```

> 可選的 `next.config.js` 檔案範例

我們最終得到的是 _兩個區域_，透過我們的 [路徑別名功能](/docs/pages/building-your-application/configuring/absolute-imports-and-module-aliases) 將它們結合到父網域 `https://vercel.com` 中。看起來大概是這樣：

```
{
  "rules": [
    { "pathname": "/docs", "dest": "our-docs.vercel.app" },
    { "pathname": "/api", "dest": "our-docs.vercel.app" },

```

> 這些簡單規則讓您將微服務和區域組合在一起


動態匯入的快取改進 [](#improved-caching-for-dynamic-imports)
-----------------------------------------------------------------------------

無論何時您使用動態 `import()`，這都向 Webpack 發出信號表示存在新的程式碼分割入口點。

在編譯時，這意味著為對應的模組子樹生成特定的封裝。

在 Next.js 5.0 之前，動態封裝會接收到類似以下的 URL：

```
/_next/1517592683901/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6.js
```

現在，我們將動態匯入轉化為子樹內容的內容可尋址散列 (content-addressable hashes)：

```
/_next/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6-b7874680a9e21fb6eb89.js
```

這意味著在部署之間，您的使用者和客戶不 needlessly 重新下載他們已經使用的程式碼。

Fragments [](#fragments)
-----------------------

Next.js 建立一層的 `<Document>` 組件，每個頁面都會伺服器端渲染。 [重載此組件](/) 讓您可以完全掌控您的標記，實現許多 [進階用例](https://github.com/vercel/next.js/tree/canary/examples)。

部分初始標記是 Next.js 需要在客戶端評估的腳本列表。自訂的 `_document` 看起來像這樣：

pages/\_document.js

```
import Document, { Head, Main, NextScript } from 'next/document';
export default class extends Document {
  render() {
    return (
      <html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </html>
    );
  }
}
```

> `Document` 允許您自訂頁面的整份伺服器端渲染輸出

直到最近，我們被迫在 `<div>` 內包裝腳本。

透過 Next.js 5.0，我們現在利用新的 `Fragment` 支援，這轉化為 **更輕量的頁面** 和對頁面樣式的完全控制，且無多餘的標記。

準確的錯誤提示 [](#more-accurate-errors)
---------------------------------------------

Node.js 不支援原始碼映射，發生在伺服器端的錯誤會附上指向編譯程式碼的堆疊追蹤。

透過 Next 5 我們改進了伺服器端的原始碼映射支援。發生在伺服器端渲染時的錯誤現在指向正確的函式和行號。

結論 [](#conclusion)
-------------------------

通用 Webpack 鞏固了 Next.js 的基礎並使其更具備未來兼容性 (future proof)。基本上，已不再有所謂人為分隔哪些插件或載入器適用於 Next.js，哪些不適用。

為了符合 _零配置_ (zero configuration) 的精神，我們很高興介紹 [Next 插件](https://github.com/vercel/next-plugins)，它是用於自動擴展 Next.js 功能的食譜的社群倉庫，無需調整特定開關。

透過這個，我們現在支援整個 CSS 解決方案、像 TypeScript 的編譯為 JS 的語言以及像 [Nerve](https://github.com/NervJS/nerv) 的 React 替代方案，只需引入額外模組並在 `next.config.js` 中明確包含其。簡單而不晦澀。

區域 (Zones) 允許互連 Next.js 應用，它們甚至不屬於同一倉庫或伺服器 (server(s))。我們認為這是一個非常重要的里程碑，在「團隊可伸縮性」 (team scalability) 類別的改進中。

Next.js 因此成為由多個團隊維護的大型應用程式的理想候選。他們現在可以並行部署更新，減少錯誤面積，提高迭代速度，甚至嘗試除核心外不同的技術，例如 [許多不同的方法](https://github.com/vercel/next.js/tree/canary/examples) 來進行狀態管理或數據獲取。

我們想把握這次機會感謝 Deep Varma 和 [Trulia](https://trulia.com) 工程團隊，他們貢獻了關鍵見解、代碼和測試，導致了此功能的設計。

一如既往，此次發布沒有眾多開源貢獻者 (open-source contributors) 和我們精彩社群 (wonderful community) 的支持是不會可能的。
