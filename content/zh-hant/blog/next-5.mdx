我們很高興向世界介紹 Next.js 5.0。它即日起在 npm 上可用。要升級，請執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

> 除了更新 Next.js，我們也升級了相依套件 `react` 和 `react-dom`

Next.js 是用於通用、伺服器端渲染 (server-rendered)（或靜態預先渲染）的 React.js 應用程式的工具套件。開發任何規模的應用程式的起步就像執行 `next` 一樣簡單。 ([閱讀更多](https://vercel.com/blog/next)。)

每次發佈新版本，我們都致力於保留後向兼容性 (backwards compatibility)、提供簡單的升級路徑，並且僅在絕對必要時才進行 API 變更。Next.js 5.0 也不例外。

不過，在底層，Next.js 經歷了重大的轉變以實現強大的新用例和擴展性。我們首先讓 Next.js 為伺服器端和客戶端代碼共用通用的 Webpack 管道。

通用 Webpack 和 Next 插件 [](#universal-webpack-and-next-plugins)
-------------------------------------------------------------------------

Next.js 利用現有的強大工具，如 Webpack、Babel 和 Uglify，向最終使用者呈現一個非凡簡易的介面：`next`（開發）、`next build`（準備生產環境）和 `next start`（服務）或 `next export` 以預渲染為靜態檔案。

早期決策之一是如何配置這些工具提供非常強大的擴展點。我們不僅想要易用性，我們還希望能夠靈活性地根據您的需要延伸工具套件。

例如，您可以在 `next.config.js` 中設置 `webpack` 屬性來 [延伸 Next.js Webpack 設定](https://github.com/vercel/next.js#customizing-webpack-config)。

由於 Webpack 在生產環境和開發環境中的執行方式不同，我們當時決定將其設為 **一個函式** 來裝飾其預設 Webpack 設定：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    // modify it!
    return config;
  },
};
```

> 可選的 `next.config.js` 檔案範例

不過，Webpack 僅會在客戶端（瀏覽器）封裝上執行，您將錯失使用此強大工具鏈進行伺服器端渲染的可能性。

我們很高興地宣布，我們已大幅重構程式碼庫以 **讓 Webpack 通用運行**。

從您的角度看，發生改變的只是額外傳遞 `isServer` 屬性給上述裝飾函式。不過，新的語義意味著廣泛的 Webpack 載入器生態系現在可供您使用。

### CSS、LESS、SASS、SCSS 和 CSS Modules [](#css-less-sass-scss-and-css-modules)

我們最受歡迎的功能之一是匯入 CSS 檔案並利用 Webpack 載入器：

```
import './index.css';
 
export default function Index() {
  return (
    <div>
      <p>I love CSS!</p>
    </div>
  );
}
```

> 感謝通用 Webpack，使用 CSS 匯入的範例頁面 (`pages/index.js`)

為了實現此功能，您可以將需要的載入器作為相依套件引入：

終端機

```
npm i --save css-loader style-loader postcss-loader
```

> Next.js 讓您可以自由選擇所需的載入器，並隨心所欲地升級它們至不同版本。

然後延伸設定以配置您的載入器。在 `next.config.js` 中：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dev, isServer } = options;
    const extractCSSPlugin = new ExtractTextPlugin({
      filename: 'static/style.css',
      disable: dev,
    });
    config.module.rules.push({
      test: /\.css$/,
      use: cssLoaderConfig(extractCSSPlugin, {
        cssModules,
        dev,
        isServer,
      }),
    });
    return config;
  },
};
```

> 延伸原始 Webpack 設定為您提供強大的靈活性和控制

雖然我們的一般建議是使用元件本機樣式解決方案，例如 [內建的 `styled-jsx` Babel 插件](https://github.com/vercel/next.js#built-in-css-support)，但我們相信 CSS 載入器具有許多重要的優點，如輕鬆重用於現有 CSS 程式碼庫，並大幅簡化遷移舊程式碼庫到 Next.js。

我們沒有預設啟用每個可能的功能和載入器，而是引入了 [**Next.js 插件**](https://github.com/vercel/next-plugins)，它們是 **裝飾您配置的簡單函式**。與手動延伸配置以設置載入器如我們之前所做的不同，您只需執行：

```
const withCss = require('next-css');
module.exports = withCss({
  /* extra optional config */
});
```

> 只需引入 `next-css` 即可啟用 `.css` 檔案匯入

閱讀更多關於 Next.JS 上 [CSS 載入器](https://github.com/vercel/next-plugins/tree/master/packages/next-css) 的使用，或參考我們已經為您創建的一些套件：

| Loader | Package |
| --- | --- |
| CSS | [next-css](https://github.com/vercel/next-plugins/tree/master/packages/next-css) |
| LESS | [next-less](https://github.com/vercel/next-plugins/tree/master/packages/next-less) |
| SASS | [next-sass](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) |

我們的目標是賦能社群開發和成長實用簡單擴充功能的生態系。我們已開放 [next-plugins](https://github.com/vercel/next-plugins) 單倉庫供 Next.js 社群維護。所有 PR 都受到歡迎！

### TypeScript 支援 [](#typescript-support)

JavaScript 生態系中 [成長最快的技術](https://stateofjs.com/2017/flavors/results) 之一就是 TypeScript。它的成長甚至使其成為 [Babel 7 官方支援](https://github.com/babel/babel/tree/master/packages/babel-preset-typescript) 的技術，這意味著 Next.js 僅需 [自訂您的 `.babelrc`](https://github.com/vercel/next.js#customizing-babel-config) 即可自然支援。

與此同時，感謝我們的新通用 Webpack 支援，您 **今天** 就能獲得完整的 TypeScript 支援！

您可以像這樣延伸您的 Webpack 設定：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dir, defaultLoaders } = options;
    config.resolve.extensions.push('.ts', '.tsx');
    config.module.rules.push({
      test: /\.(ts|tsx)$/,
      include: [dir],
      exclude: /node_modules/,
      use: [
        defaultLoaders.babel,
        { loader: 'ts-loader', options: { transpileOnly: true } },
      ],
    });
    return config;
  },
};
```

> 我們只需啟用 `ts-loader`

像 CSS 載入器和前置處理器一樣，TypeScript 是最受關注的功能之一。為了讓將其整合到專案中與其他載入器一樣容易，我們現在有了 [`next-typescript` 插件](/)，您可以將其包含在 `next.config.js` 檔案中：

next.config.js

```
const withTs = require('next-typescript');
module.exports = withTs({
  /* additional config*/
});
```

> 插件可以輕易組合：它們只是函式

### 對 React 替代庫及模組重載的更好支援 [](#better-support-for-react-altlibs--module-overloading)

隨著時間推移，許多 React 的即插即用替代實現湧現。在其之中，一些值得注意的有 [`preact`] ([https://preactjs.com/](https://preactjs.com/))、[`nervjs`](https://github.com/NervJS/nerv) 和 [`inferno`](https://github.com/infernojs/inferno)。

其他庫專注於替換 DOM 渲染器，如 [`react-dom-lite`](https://github.com/jquense/react-dom-lite)，旨在通過引入一些瀏覽器相容性的次要交換條件來縮小 React 構建。

通用 Webpack 支援使 **整合這些庫作為即插即用替代方案的過程更加容易**。與其他插件相同，這是使用 Next.js 和 preact 只需做的：

終端機

```
npm i @zeit/next-preact preact preact-compat
```

> 我們安裝 preact 插件和必要的相依套件

```
const withPreact = require('@zeit/next-preact');
module.exports = withPreact();
```

> 我們的 `next.config.js` 準備好支援 preact

檢視非常簡單的 [@zeit/next-preact](https://github.com/vercel/next-plugins/tree/master/packages/next-preact) 模組或建立您自己的！

### 生產環境中的可選外部原始碼映射 [](#optional-external-sourcemaps-in-production)

現在 Webpack 已被 Next.js 用於客戶端和伺服器代碼，在生產構建中啟用原始碼映射僅需要一小部分設定調整。

在開發模式下，原始碼映射會自動啟用，所以我們針對生產環境配置為不同：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
    }
    return config;
  },
};
```

> 我們僅在未在開發模式下時配置 `devtool` 選項不同

區域 (Zones) [](#zones)
---------------

從 Very 開始，[Next.js 的既定目標](https://deck.vercel.app) 之一就是恢復並保留 Web 的簡潔性。

伺服器端渲染，一種資料獲取和基於檔案系統結構的宣告式頁面的通用方法是我们基於此思維引入的功能之一。

經常被忽略的網路服務和網站面向是如何 _天然可組合和可伸縮 (scalable)_ 。

例如，`mydomain.com/settings` 和 `mydomain.com/` 可以是兩個完全不同的應用程式，獨立部署、獨立伸縮，甚至運行相同軟體的不同版本。

將它們 _"粘結在一起"_ 以為使用者提供統一體驗所需的只是 [一些簡單配置](/docs/pages/building-your-application/configuring/absolute-imports-and-module-aliases) 後端路由層或暴露它們向外界的負載平衡器。我們很高興現在帶來 **組建多個使用 Next.js 構建的應用程式** 的能力，使用普通的 `<Link>` 組件連接在一起。我們稱此功能為 **區域 (Zones)**。

作為範例，考慮這兩個獨立的 Next.js 應用程式部署到 [Vercel](https://vercel.com)：

*   [https://front.vercel.app](https://front.vercel.app)
*   [https://zeit-docs-zbqbsrucga.vercel.app/docs](https://zeit-docs-zbqbsrucga.vercel.app/docs) [https://zeit-docs-zbqbsrucga.vercel.app/api](https://zeit-docs-zbqbsrucga.vercel.app/api)

![我們的頁面均有流暢的體驗，但它们屬於獨立的應用程式](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/zones.png)

我們的頁面均有流暢的體驗，但它们屬於獨立的應用程式

當我們重整文檔時，我們希望讓接受社群貢獻盡可能簡單。

我們將文檔「迷你網站」分離到其 [獨立倉庫](https://github.com/vercel/docs)。此外，每次提交 Pull Request 並提出變更時，我們都會自動隔離部署它：

![每次變更發生在 PR 內部，我們的機器人自動部署它](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/github.png)

每次變更發生在 PR 內部，我們的機器人自動部署它

我們最終得到的是 _兩個區域_，透過我們的 [路徑別名功能](/docs/pages/building-your-application/configuring/absolute-imports-and-module-aliases) 將它們結合到父網域 `https://vercel.com` 中。看起來大概是這樣：

```
{
  "rules": [
    { "pathname": "/docs", "dest": "our-docs.vercel.app" },
    { "pathname": "/api", "dest": "our-docs.vercel.app" },
    { "dest": "my-main-website.vercel.app" }
  ]
}
```

> 這些簡單規則讓您將微服務和區域組合在一起

剩下的是執行 `now alias` 命令：

終端機

```
now alias -r rules.json my-domain.com
```

我們的使命是讓部署盡可能通用和開放。為了協助本地開發，我們最近開源了 `micro-proxy`，此工具與上述相同的配置格式運作。

您可能同樣能夠使用其他解決方案，如 Nginx、HAProxy 或 API Gateway 組合區域。

動態匯入的快取改進 [](#improved-caching-for-dynamic-imports)
-----------------------------------------------------------------------------

無論何時您使用動態 `import()`，這都向 Webpack 發出信號表示存在新的程式碼分割入口點。

在編譯時，這意味著為對應的模組子樹生成特定的封裝。

在 Next.js 5.0 之前，動態封裝會接收到類似以下的 URL：

```
/_next/1517592683901/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6.js
```

現在，我們將動態匯入轉化為子樹內容的內容可尋址散列 (content-addressable hashes)：

```
/_next/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6-b7874680a9e21fb6eb89.js
```

這意味著在部署之間，您的使用者和客戶不 needlessly 重新下載他們已經使用的程式碼。

Fragments [](#fragments)
-----------------------

Next.js 建立一層的 `<Document>` 組件，每個頁面都會伺服器端渲染。 [重載此組件](/) 讓您可以完全掌控您的標記，實現許多 [進階用例](https://github.com/vercel/next.js/tree/canary/examples)。

部分初始標記是 Next.js 需要在客戶端評估的腳本列表。自訂的 `_document` 看起來像這樣：

pages/\_document.js

```
import Document, { Head, Main, NextScript } from 'next/document';
export default class extends Document {
  render() {
    return (
      <html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </html>
    );
  }
}
```

> `Document` 允許您自訂頁面的整份伺服器端渲染輸出

直到最近，我們被迫在 `<div>` 內包裝腳本。

透過 Next.js 5.0，我們現在利用新的 `Fragment` 支援，這轉化為 **更輕量的頁面** 和對頁面樣式的完全控制，且無多餘的標記。

準確的錯誤提示 [](#more-accurate-errors)
---------------------------------------------

Node.js 不支援原始碼映射，發生在伺服器端的錯誤會附上指向編譯程式碼的堆疊追蹤。

透過 Next 5 我們改進了伺服器端的原始碼映射支援。發生在伺服器端渲染時的錯誤現在指向正確的函式和行號。

![錯誤現在顯示正確的行、文件和函式名稱](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/errors.png)

結論 [](#conclusion)
-------------------------

通用 Webpack 鞏固了 Next.js 的基礎並使其更具備未來兼容性 (future proof)。基本上，已不再有所謂人為分隔哪些插件或載入器適用於 Next.js，哪些不適用。

為了符合 _零配置_ (zero configuration) 的精神，我們很高興介紹 [Next 插件](https://github.com/vercel/next-plugins)，它是用於自動擴展 Next.js 功能的食譜的社群倉庫，無需調整特定開關。

透過這個，我們現在支援整個 CSS 解決方案、像 TypeScript 的編譯為 JS 的語言以及像 [Nerve](https://github.com/NervJS/nerv) 的 React 替代方案，只需引入額外模組並在 `next.config.js` 中明確包含其。簡單而不晦澀。

區域 (Zones) 允許互連 Next.js 應用，它們甚至不屬於同一倉庫或伺服器 (server(s))。我們認為這是一個非常重要的里程碑，在「團隊可伸縮性」 (team scalability) 類別的改進中。

Next.js 因此成為由多個團隊維護的大型應用程式的理想候選。他們現在可以並行部署更新，減少錯誤面積，提高迭代速度，甚至嘗試除核心外不同的技術，例如 [許多不同的方法](https://github.com/vercel/next.js/tree/canary/examples) 來進行狀態管理或數據獲取。

我們想把握這次機會感謝 Deep Varma 和 [Trulia](https://trulia.com) 工程團隊，他們貢獻了關鍵見解、代碼和測試，導致了此功能的設計。

一如既往，此次發布沒有眾多開源貢獻者 (open-source contributors) 和我們精彩社群 (wonderful community) 的支持是不會可能的。
