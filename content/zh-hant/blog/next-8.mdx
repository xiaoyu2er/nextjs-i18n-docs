Serverless Next.js[](#serverless-nextjs)
----------------------------------------

Next.js 的 serverless 目標會從頁面輸出 Serverless 函式

 Serverless 部署通過將應用程式拆分為了較小的部分（也被稱為 **Lambda**）來大幅提高可靠性和可擴展性。在 Next.js 的情況下，`pages` 目錄中的每個頁面都會變成一個 Serverless Lambda.

Serverless 有多 [數個好處](https://vercel.com/docs/serverless-functions/conceptual-model)。參考連結討論了以 Express 為背景的其中一些優點，但原則適用於所有情況：Serverless 允許分散的故障點、無限的可擴展性，並以「按需付費」模式提供極低的價格。

要在 Next.js 中啟用 **Serverless 模式**，請在 `next.config.js` 中添加 `serverless` 構建 `target`：

next.config.js

```
module.exports = {
  target: 'serverless',
};
```

 `serverless` 目標將為每個頁面输出 一個 Lambda。此檔案完全獨立運行，無需任何依賴即可執行：

*   `pages/index.js` => `.next/serverless/pages/index.js`
*   `pages/about.js` => `.next/serverless/pages/about.js`

Next.js Serverless 函式的簽名與 Node.js HTTP 伺服器呼叫腳本相似：

```
type Function = (req: http.IncomingMessage, res: http.ServerResponse) => void;
```

*   [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
*   [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)
*   `void` 指函式不返回值，等同於 JavaScript 的 `undefined`。呼叫函式將終止該請求。

Next.js 為 Serverless 部署提供底層 API，因為託管平台的函式簽名不同。通常您需要為 Next.js Serverless 構建的輸出封裝一個兼容性層。

例如，如果平台支援 Node.js [http.Server](https://nodejs.org/api/http.html#http_class_http_server) 類別：

server.js

```
const http = require('http');
const page = require('./.next/serverless/about.js');
const server = new http.Server((req, res) => page.render(req, res));
server.listen(3000, () => console.log('Listening on http://localhost:3000'));
```

### **摘要**[](#summary)

*   用於實作 Serverless 部署的底層 API
*   `pages` 目錄中的每個頁面都變成 Serverless 函式 (Lambda)
*   建立最小的 Serverless 函式 (**50 KB** 基本 .zip 大小)
*   針對函式的快速 [冷啟動](https://vercel.com/docs/serverless-functions/conceptual-model) 進行了優化
*   Serverless 函式具有 0 依賴項（它們包含在函式打包檔中）
*   使用 Node.js 的 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage) 和 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)
*   使用 `next.config.js` 中的 `target: 'serverless'` 選擇加入
*   `server` 目標仍完全支援並獲得維護
*   `serverless` 模式中不支援 `publicRuntimeConfig` 和 `serverRuntimeConfig`。請改用建構時間配置。

我們已為 `webpack` 做出貢獻，以改進 Next.js（及 `webpack` 生態系統的其餘部分！）的建構效能和資源利用率。

建構時間環境配置 [](#build-time-environment-configuration)
-----------------------------------------------------------------------------

在審查 Next.js 應用程式時，我們觀察到一個經常重複的模式是添加 `babel-plugin-transform-define` 或 `webpack.DefinePlugin` 來為應用程式提供配置值。

隨著 Next.js 8，我們將在 `next.config.js` 中新增名為 `env` 的鍵，以便以向後兼容的方式提供相同的功能：

next.config.js

```
module.exports = {
  env: {
    customKey: 'MyValue',
  },
};
```

這將允許您在程式碼中使用 `process.env.customKey`。例如：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
export default function IndexPage() {
  return <h1>The value of customKey is: {process.env.customKey}</h1>;
}
```

 `process.env.customKey` 將在構建時替換為 `'MyValue'`。

預取效能改進 [](#prefetch-performance-improvements)
-----------------------------------------------------------------------

Next.js 路由允許您預先加載頁面以實現更快的導航：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Link from 'next/link';
 
export default function IndexPage() {
  return (
    <>
      <Link href="/about" prefetch>
        <a>To About Page</a>
      </Link>
    </>
  );
}
```

它的工作原理是預先加載具有 `prefetch` 屬性的每個連結的 JavaScript 打包檔。

在 Next.js 8 之前的版本中，這意味著將 `<script>` 標籤注入到文件 `<body>` 中。

然而，這在開啟頁面時引入了一些開銷，最顯著的是瀏覽器的「加載」指示器顯示的時間長於預期，即使頁面已經可以互動。

在 Next.js 8 中，`prefetch` 使用 `<link rel="preload">` 代替 `<script>` 標籤。它也在 `onload` 之後才開始預取，以允許瀏覽器管理資源。

此外，Next.js 現在偵測 2G 網路和 `navigator.connection.saveData` 模式，以在較慢的網路連接上停用預取。

初始 HTML 體積更小 [](#smaller-initial-html-size)
-------------------------------------------------------

由於 Next.js 預先呈現 HTML，它將頁面包裝在默認結構中，包括 `<html>`、`<head>`、`<body>` 和渲染頁面所需的 JavaScript 文件。

隨著 [Next.js 7 我們將初始載荷優化為 1.50KB](https://nextjs.org/blog/next-7/#smaller-initial-html-payload)，這比上一個版本減少了 7.4%。

我們將初始載荷大小進一步降至 1.16KB，進一步減少了 23%：

|  | 7.0 | 8.0 | delta |
| --- | --- | --- | --- |
| 文件大小 (伺服器渲染) | 1.50KB | 1.16KB | **23%** _更小_ |

我們減少大小的主要方式包括：

*   移除了頁面初始化內聯腳本
*   `/_error` 頁面不再包含在每次頁面載入中

### /_error 的非同步載入 [](#on-demand-loading-of-_error)

當生產環境發生錯誤時，`/_error` 頁面將被渲染以顯示錯誤發生。

自 Next.js 首次發佈以來，`/_error` 頁面腳本標籤一直屬於初始 HTML 的一部分，這意味著如果沒有運行時錯誤，它即使不被使用也會被載入。

從 Next.js 8 開始，當發生錯誤時，`/_error` 頁面會按需載入。

這意味著預設情況下載入、解析和運行的程式碼更少。

開發者體驗 (DX) 改進[](#dx-improvements)
-----------------------------------

Next.js 的主要目標之一是在**最佳開發者體驗下**提供最佳生產效能。此次更新包含許多基於用戶反饋的細微改進。

### 改進的按需編譯 [](#improved-on-demand-entries)

開箱即用，Next.js 僅自動編譯 _活躍_ 開發中的頁面。下次運行 `next dev` 時，Next.js 不會編譯 `pages` 目錄中的所有頁面。相反，它會在您訪問頁面時進行編譯。

例如，當訪問 `http://localhost:3000/my-page` 時，會按需編譯 `pages/my-page.js` 檔案，然後渲染頁面。

這確保開發者啟動開發伺服器時不必等待所有頁面編譯完成，在大型應用程式中這可能耗時。它保持低記憶體使用量並使編譯器快速，因為編譯器在打包時不需要考慮所有頁面。

![按需條目流程](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-8/flow.jpg)

按需條目流程

如果頁面在 25 秒內未被訪問，它將從編譯器的構建緩存中刪除，以保持編譯器快速並減少記憶體使用量。

Next.js 跟蹤頁面訪問的方式是使用輪詢機制。每隔 5 秒，會發送一個「on-demand-entries-ping」，讓 Next.js 開發伺服器知道特定頁面被訪問了。

自該功能首次發佈以來，ping 是使用 `window.fetch` 呼叫完成的，這意味著每次觸發 ping 時，它都會在瀏覽器開發工具的 `console` 和 `network` 標籤上顯示。

最請求的功能之一是能夠向瀏覽器開發工具隱藏這些請求，因為這些請求會增加不必要的雜訊。

我們很高興地宣佈，在 Next.js 8 中，基於 `fetch` 的 ping 已替換為基於 WebSockets 的方法，這意味著 ping 仍然發生，但僅在檢查 WebSocket 連接時可見。

特別感謝 [JJ Kasper](https://github.com/ijjk) 協助轉為 WebSockets。

### 開發階段端口監聽更快 [](#faster-port-listening-in-development)

當啟動 Next.js 開發伺服器時，它必須運行一些初始編譯才能處理請求。預設情況下，Next.js 會在開始 HTTP 伺服器之前等待此編譯步驟完成，這意味著如果您運行 `next dev` 然後進入瀏覽器，有時可能會收到「無法存取此網站」的訊息，因為 HTTP 伺服器尚未監聽連接。

隨著 Next.js 8，HTTP 將在編譯開始前開始監聽連接，這意味著如果在編譯完成前去 `http://localhost:3000/`，請求將等待初始編譯完成再處理請求，而不必刷新頁面直到可用。

特別感謝 [Brian Beck](https://github.com/exogen) 實作了此功能。

更快速靜態導出 [](#faster-static-export)
---------------------------------------------

Next.js 的核心概念是透過**預渲染 (pre-rendering)** 來實現高效能。預渲染有兩種形式：

*   **伺服器渲染 (Server Rendering)**，每個請求都會觸發渲染。結果是，最終用戶無需等待任何 JS 下載即可開始處理資料
*   **靜態渲染**，我們輸出靜態檔案，可直接提供服務，無需伺服器上的任何程式碼執行

從 Next.js 8 開始，如果電腦有多個 CPU，通過 `next export` 的靜態渲染將擁有速度提升。

根據 4 核心 MacBook 的測試，通過利用所有核心進行預渲染頁面，導出速度從每秒約 25 個頁面提高到 75 個頁面。

Next.js 將自動偵測 CPU 核心數量並分發頁面，無需任何程式碼更改。

特別感謝 [Benjamin Kniffler](https://github.com/bkniffler) 實作了此功能。

Head 元素去重 [](#head-element-deduplication)
---------------------------------------------------------

構建應用程式時的常見需求是更新頁面的 `<head>` 元素。例如設置 `<title>` 或 `<meta name="viewport">` 用於響應式設計。

Next.js 暴露了一個內建元件用於對 `<head>` 進行更改：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Head from 'next/head';
 
export default function IndexPage() {
  return (
    <>
      <Head>
        <title>My page title</title>
      </Head>
    </>
  );
}
```

 ``<Head>` 元件甚至可以多次在不同的元件中使用，例如您的佈局元件可以設置一些默認的 head 標籤。

但是，您可能想以不同的值覆蓋默認的 head 標籤。在 Next.js 的舊版本中，這會導致標籤在輸出中重複，因為沒有方法去重標籤。

出於這個原因，現在可以為 `<Head>` 元件內的每個元素提供 `key` 屬性，將自動去重具有相同 `key` 值的標籤。

當在兩個標籤上設置 `key="viewport"` 時，只有最後一個標籤會被渲染。

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Head from 'next/head';
export default function IndexPage() {
  return (
    <>
      <Head>
        <title>My page title</title>
        <meta
          name="viewport"
          content="initial-scale=1.0, width=device-width"
          key="viewport"
        />
      </Head>
      <Head>
        <meta
          name="viewport"
          content="initial-scale=1.2, width=device-width"
          key="viewport"
        />
      </Head>
    </>
  );
}
```

安全性改進 [](#security-improvements)
-----------------------------------------------

### 新的 `crossOrigin` 配置選項 [](#new-crossorigin-config-option)

在 Next.js 6 中，我們引入了向 `pages/_document.js` 中的 `<Head>` 和 `<NextScript>` 添加 `crossOrigin` 屬性的選項，但這並未涵蓋設置 `cross-origin` 的所有用例。

Next.js 具有客戶端路由器，它動態注入 `<script>` 標籤，這些標籤在注入時缺少 `cross-origin` 屬性。

為了確保所有 `<script>` 標籤都設置了 `cross-origin`，我們在 `next.config.js` 中引入了一個新的配置選項

next.config.js

```
module.exports = {
  crossOrigin: 'anonymous',
};
```

引入此選項的另一個好處是，不再需要自定義 `pages/_document.js` 來在應用程式中設置 `cross-origin`。

之前的行為仍被支援，但將在開發環境中發出警告，以幫助開發者轉移到新引入的選項。

### 移除內聯 JavaScript [](#removed-inline-javascript)

當使用 Next.js 7 及以下版本時，為了啟用 [內容安全政策 (CSP)](https://developer.mozilla.org/docs/Web/HTTP/CSP)，用戶必須在策略中包含 `script-src 'unsafe-inline'`，因為 Next.js 會創建一個內聯 `<script>` 標籤來傳遞資料，例如將 `getInitialProps` 的結果傳遞到客戶端。

隨著 Next.js 8，我們已將此內聯腳本標籤更改為 JSON 標籤，以便安全地傳遞到客戶端。這意味著 Next.js 不再包含任何內聯腳本。

經過仔細考慮，現在可以使用 `script-src 'self'`。

### API 認證範例 [](#example-of-api-authentication)

[史上最受請求的範例之一](https://github.com/vercel/next.js/issues/153) 是如何在 Next.js 中對外部 API、任何 API、使用任何程式設計語言進行認證。

隨著 Next.js 8 的推出，我們也引入了新創建的範例：[with-cookie-auth](https://github.com/vercel/next.js/tree/canary/examples/with-cookie-auth)

此範例展示了如何對外的 Node.js API 進行認證，但應用的概念適用於任何無狀態 API。

範例使用 cookie 在伺服器端和客戶端渲染之間共享令牌。

這樣，如果應用程式在伺服器上渲染，它仍然可以代表用戶獲取認證資料。

特別感謝 [Juan Olvera](https://github.com/j0lv3r4) 貢獻了此範例。
