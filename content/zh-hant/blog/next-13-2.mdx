Next.js 13.2 針對 App Router (`app`) 帶來重大改進，為穩定性做準備：

*   [**內建搜尋引擎最佳化 (SEO) 支援：**](#built-in-seo-support-with-new-metadata-api) 新的 Metadata API 用於設定靜態和動態 `meta` 標籤。
*   [**路由處理函式 (Route Handlers)：**](#custom-route-handlers) 自訂請求處理函式，基於 Web `Request` 和 `Response`。
*   [**供 Server Components 使用的 MDX：**](#mdx-for-server-components) 在 Markdown 內部使用 React 元件，僅限伺服器端。
*   [**Rust MDX 解析器：**](#rust-mdx-parser) 使用全新的 Rust 套件加快 Markdown 解析。
*   [**改進的錯誤覆蓋層 (Improved Error Overlay)：**](#improved-error-overlay) 將 Next.js 和 React 堆疊追蹤分開，提高可讀性。
*   [**靜態型別連結 (Statically Typed Links) (Beta)：**](#statically-typed-links) 使用 `next/link` 和 TypeScript 防止連結損壞。
*   [**Turbopack 改進 (Turbopack Improvements) (Alpha)：**](#turbopack-improvements) 與 Webpack loaders 相容性與改進的支援。
*   [**Next.js 快取 (Next.js Cache) (Beta)：**](#nextjs-cache) 漸進式 ISR 和更快重複部署程式碼變更。

今天更新，請運行:

終端機

```
npm i next@latest react@latest react-dom@latest eslint-config-next@latest
```

內建搜尋引擎最佳化 (SEO) 支援與新的 Metadata API[](#built-in-seo-support-with-new-metadata-api)
-----------------------------------------------------------------------------------------

Next.js 從一開始就設計為 [支援搜尋引擎最佳化](https://nextjs.org/learn/seo/introduction-to-seo)。

提供預渲染的 HTML 內容不僅有助於提升搜尋引擎的索引，還能改善應用程式的效能。雖然 Next.js 在過去多個版本中提供了修改應用程式元資料的簡單 API (`next/head`)，但我們想重新設計並改進您使用 App Router (`app`) 優化搜尋引擎的方式。

新的 Metadata API 讓您在任何作為 Server Components 的文字版面或頁面中，透過明確的元資料設定來定義元資料（例如 HTML `head` 元素內部的 `meta` 和 `link` 標籤）。

app/layout.tsx

```
import type { Metadata } from 'next';
 
export const metadata: Metadata = {
  title: 'Home',
  description: 'Welcome to Next.js',
};
```

此 API 簡單、可組合，並設計為與串流伺服器渲染相容。例如，您可以在根版面中設定整個應用程式的共用元資料屬性，並為應用程式中的其他路由組合和合併元資料物件。

這包括支援動態元資料以及靜態元資料：

layout.js / page.js

```
// Static metadata
export const metadata = {
  title: '...',
};
 
// Dynamic metadata
export async function generateMetadata({ params, searchParams }) {
  const product = await getProduct(params.id);
  return { title: product.title };
}
```

所有元資料選項均可用，包括提供自訂元資料的能力，透過 [TypeScript 插件](/docs/app/building-your-application/configuring/typescript#using-the-typescript-plugin) 支援或加入 `Metadata` 型別。

例如，您可以透過元資料定義 Open Graph 圖片：

app/layout.tsx

```
export const metadata = {
  openGraph: {
    title: 'Next.js',
    description: 'The React Framework for the Web',
    url: 'https://nextjs.org',
    siteName: 'Next.js',
    images: [
      {
        url: 'https://nextjs.org/og.png',
        width: 800,
        height: 600,
      },
    ],
    locale: 'en-US',
    type: 'website',
  },
};
 
export default function Layout({ children }) {}
```

Metadata API 在 13.2 中支援 App Router (`app`)，取代了以往的 `head.js` 特殊檔案。它不適用於 `pages` 目錄。

[深入了解 SEO](/learn/seo/introduction-to-seo) 或檢視 [Metadata API 參考](/docs/app/api-reference/file-conventions/metadata)。我們感謝 [next-seo](https://github.com/garmeeh/next-seo) 對社群套件的工作以及對初始 API 設計的回饋。

自訂路由處理函式 (Custom Route Handlers)[](#custom-route-handlers)
-----------------------------------------------

App Router (`app`) 原始 Beta 版本缺失的部分之一是指 API Routes，它們存在於 `pages/api` 目錄中。我們想藉此機會建立新的、更現代的 API Routes 版本，與 `app` 的新路由系統深度整合。

路由處理器允許您使用 Web [Request](https://developer.mozilla.org/docs/Web/API/Request) 和 [Response](https://developer.mozilla.org/docs/Web/API/Response) API 為指定路由建立自訂請求處理器。

app/example/route.ts

```
export async function GET(request: Request) {}
```

Route Handlers 擁有異質 API 以無縫支援 Edge 和 Node.js 運行環境，包括串流回應的支援。由於 Route Handlers 與頁面和版面使用相同的 [路由段設定](/docs/app/building-your-application/routing)，它們支援長期期盼的功能，例如通用 [靜態渲染 (Static Rendering)](/docs/app/building-your-application/routing/route-handlers#static-and-dynamic-route-handlers) 和 [重新驗證 (Revalidation)](/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating#revalidating-data)。

一個 `route.ts` 檔案可以匯出以 HTTP 動詞命名的異步函式：`GET`, `HEAD`, `OPTIONS`, `POST`, `PUT`, `DELETE`, 和 `PATCH`。這些函式隨後可以進行封裝和抽象化，以建立自訂路由邏輯的幫助函式 / 可使用邏輯。

其他伺服器函式，如 `cookies` 和 `headers`，可在 Route Handlers 內部使用 – 以及這些抽象化構建所用的任何 Web APIs。這允許在 Server Components 和 Route Handlers 之間共享程式碼。

app/example/route.ts

```
import { cookies } from 'next/headers';
 
export async function GET(request: Request) {
  const cookieStore = cookies();
  const token = cookieStore.get('token');
 
  return new Response('Hello, Next.js!', {
    status: 200,
    headers: { 'Set-Cookie': `token=${token}` },
  });
}
```

Route Handlers 在 13.2 中可用於 App Router (`app`)，使用 `route.ts` 特殊檔案。它們在 `pages` 目錄中不可用，因為它們是 API Routes 的替代方案。

[深入了解 Route Handlers](/docs/app/building-your-application/routing/route-handlers) 或 [檢視 API 參考文件](/docs/app/api-reference/file-conventions/route)。我們在此感謝 SvelteKit 所提供的 [參考先驅和靈感](https://kit.svelte.dev/docs/routing#server)。

供 Server Components 使用的 MDX [](#mdx-for-server-components)
-------------------------------------------------------

MDX 是 Markdown 的超集，讓您能直接在 Markdown 檔案中撰寫 JSX。這是添加動態互動性並將 React 元件嵌入內容中的強大方式。

透過 13.2，您現在可以完全使用 React Server Components 搭配 MDX – 這意味著較少的客戶端 JavaScript 以實現更快速的頁面載入，同時保留 React 用於建立動態 UI 的強大功能。您可以根據需要將互動性點綴到您的 MDX 內容中。

`@next/mdx` 插件已更新，支援新的特殊檔案 `mdx-components.js|ts`，定義在應用程式以提供自訂元件：

your-project/mdx-components.js

```
// This file allows you to provide custom React components
// to be used in MDX files. You can import and use any
// React component you want, including components from
// other libraries.
function H1({ children }) {
  // ...
}
 
function H2({ children }) {
  // ...
}
 
export function useMDXComponents(components) {
  return { h1: H1, h2: H2, ...components };
}
```

此外，我們與社群套件合作以取得 MDX 內容 [`next-mdx-remote`](https://github.com/hashicorp/next-mdx-remote#react-server-components-rsc--nextjs-app-directory-support) 和 [`contentlayer`](https://www.contentlayer.dev/)，以新增 React Server Components 的支援。

[深入了解如何搭配 Server Components 設定 MDX](/docs/app/building-your-application/configuring/mdx) 或 [部署我們的範例](https://vercel.com/new/clone?repository-url=https%3A%2F%2Fgithub.com%2Fvercel%2Fnext.js%2Ftree%2Fcanary%2Fexamples%2Fapp-dir-mdx&project-name=app-dir-mdx&repository-name=app-dir-mdx)。

Rust MDX 解析器 [](#rust-mdx-parser)
-----------------------------------

在啟用 Server Components 的 MDX 的同時，我們也用 Rust 重寫了 MDX 解析器以改進效能。這比之前的 JavaScript 為基礎的解析器有顯著改進，後者在處理大量 MDX 檔案時出現了明顯的延遲。

您可以在 `next.config.js` 中選擇使用 Rust 解析器。例如，使用 `@next/mdx`。

next.config.js

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    appDir: true,
    mdxRs: true,
  },
};
 
const withMDX = require('@next/mdx')();
module.exports = withMDX(nextConfig);
```

我們感謝 [Titus Wormer](https://wooorm.com)，我們贊助了他專心於此專案。如果您想在 Next.js 之外使用，請參考新的套件 [mdxjs-rs](https://github.com/wooorm/mdxjs-rs)。

靜態型別連結 (St statically Typed Links) [](#statically-typed-links)
-------------------------------------------------

Next.js 現在可以為 `app` 目錄中的連結進行靜態型別檢查，以在使用 `next/link` 時防止打字錯誤和其他錯誤，提高頁間導航的型別安全性。

```
import Link from 'next/link'
 
// ✅
<Link href="/about" />
// ✅
<Link href="/blog/nextjs" />
// ✅
<Link href={`/blog/${slug}`} />
 
// ❌ TypeScript errors if href is not a valid route
<Link href="/aboot" />
```

此功能需要使用新的 App Router 以及 TypeScript。

next.config.js

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    appDir: true,
    typedRoutes: true,
  },
};
 
module.exports = nextConfig;
```

此功能現已提供 Beta 版。`rewrites` 和 `redirects` 尚未支援。

[深入了解靜態型別路由](/docs/app/building-your-application/configuring/typescript#statically-typed-links)。

Turbopack 改進 (Turbopack Improvements) [](#turbopack-improvements)
-------------------------------------------------

[Turbopack](https://nextjs.org/docs/app/api-reference/turbopack)，隨 Next.js 13 以 Alpha 版宣布，是一種增量打包器，設計用於加速本地開發，以及未來的生產構建。

我們專注於在 Turbopack 中支援現有的 Next.js 功能，並隨著向 Beta 邁進而改進整體穩定性。自上次發布以來，我們已新增：

*   支援 `next/dynamic`
*   支援 `next.config.js` 中的 `rewrites`、`redirects`、`headers` 和 `pageExtensions`
*   支援 `pages` 中的 404 和錯誤
*   支援 CSS 模組 `composes: ... from ...`
*   改進 Fast Refresh 的可靠性和錯誤恢復
*   改進 CSS 優先級處理
*   改進編譯時評估

我們也在內部使用大型 Next.js 應用程式和一些早期 Vercel 客戶測試 Turbopack 時修復了許多錯誤並改進穩定性。

### 使用 Webpack Loaders 自訂檔案轉換 [](#custom-file-transformation-with-webpack-loaders)

Turbopack 現在已支援並相容於部分 Webpack loaders。這意味著您可以使用 Webpack 生態系統中的許多 loaders 將不同類型的檔案轉換為 JavaScript。支援像 `@mdx-js/loader`、`@svgr/webpack` 和 `babel-loader` 這樣的 loaders。[了解更多](https://nextjs.org/docs/app/api-reference/config/next-config-js/turbopack) 關於自訂 Turbopack。

例如，使用 `experimental.turbo.loaders` 為每個檔案副檔名稱配置 loader 列表：

next.config.js

```
module.exports = {
  experimental: {
    turbo: {
      loaders: {
        '.md': [
          {
            // Option format
            loader: '@mdx-js/loader',
            options: {
              format: 'md',
            },
          },
        ],
        '.svg': ['@svgr/webpack'],
      },
    },
  },
};
```

請參閱 [使用 loaders 的 Turbopack 範例](https://github.com/vercel/next.js/tree/canary/examples/with-turbopack-loaders) 以獲取完整範例。

### Webpack 風格的解析別名 [](#webpack-style-resolve-aliases)

Turbopack 現在可以配置為透過別名修改模組解析，類似於 webpack 的 [`resolve.alias`](https://webpack.js.org/configuration/resolve/#resolvealias)。透過 `experimental.turbo.resolveAlias` 配置此功能：

next.config.js

```
module.exports = {
  experimental: {
    turbo: {
      resolveAlias: {
        underscore: 'lodash',
        mocha: { browser: 'mocha/browser-entry.js' },
      },
    },
  },
};
```

Next.js 快取 (Next.js Cache) [](#nextjs-cache)
------------------------------

Next.js 13.2 引入新的 Next.js 快取 (Beta)，是 [ISR](/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating#revalidating-data) 的演進，解鎖：

*   元件級的漸進式 ISR
*   無需網路請求的快速重新整理
*   變更至靜態頁面的程式碼快速重新部署

對於完全靜態的頁面，ISR 的運作方式與今天相同。對於擁有更細粒度資料擷取、混合靜態和動態的頁面，Next.js 快取使用更細粒度的易失性快取。

配合 React Server Components 和 Next.js App Router (`app`) 中的共置資料擷取基礎，您現在可以將靜態或動態資料與消耗其的元件封裝在一起。

app/page.jsx

```
export default async function Page() {
  const [staticData, dynamicData, revalidatedData] = await Promise.all([
    // Cached until manually invalidated
    fetch(`https://...`),
    // Refetched on every request
    fetch(`https://...`, { cache: 'no-store' }),
    // Cached with a lifetime of 10 seconds
    fetch(`https://...`, { next: { revalidate: 10 } }),
  ]);
 
  return <div>...</div>;
}
```

當使用 App Router 在本地開發時，您現在會在 `next dev` 中看到與 `next start` 生產環境相同的快取行為。這改善了當任何 Server Component 或資料載入程式碼變更時的 Fast Refresh 速度。

有了 Next.js 快取，您的應用程式控制快取，而非第三方 API。這與 `cache-control` headers 不同，後者由上游控制值快取的時間長度。

### 搭配 Vercel Cache API 的 Next.js 快取 [](#nextjs-cache-with-the-vercel-cache-api)

[Vercel 上的 Next.js](https://vercel.com/docs/concepts/next.js/overview) 提供了由框架定義的基礎設施。您撰寫應用程式程式碼，例如使用 `fetch` 進行的元件級資料擷取，而我們則為您建置全球分佈式的基礎設施，無需額外的努力。

新的 Next.js 快取使變更程式碼與變更資料獨立。這可以大幅加速靜態頁面的重新部署，因為這些頁面的生成可以使用現有快取。

這個新的 Vercel Cache API 設計為與任何框架配合使用，但與 Next.js 快取有原生整合。[深入了解 ISR 如何演變為 Next.js 快取](https://vercel.com/blog/vercel-cache-api-nextjs-cache)，以及 Next.js 快取在部署到 Vercel 時如何運作。

### 自架設環境的 Next.js 快取 [](#nextjs-cache-when-self-hosted)

自架設時，使用 LRU 快取，預設為 50 MB。所有快取項目預設會自動寫入磁碟。如果這些節點具有相同的快取鍵，此檔案系統快取可在節點間共享，類似於 [ISR 如何運作](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration#self-hosting-isr)。

對於希望進一步自訂和修改 Next.js 快取核心的開發人員而言，他們可以修改底層快取鍵並更改快取項目的儲存方式和位置，包括完全禁用儲存。

[其他改進](#other-improvements)
-----------------------------------------

*   **字型：** 隨著社群驚人的採用，`@next/font` 現已內建為 Next.js 的 `next/font`。
*   **字型：** 基於社群回饋，`next/font` 的預設 `font-display` 屬性已從 `optional` 更改為 `font-display: swap`。
*   **效能：** 最佳化建構流程以使用較少記憶體，我們的測試節省了 ~550MB ([PR](https://github.com/vercel/next.js/pull/45639))。
*   **效能：** 避免多次載入專案設定，使我們的測試建構速度加快 ~400ms (平均) ([PR](https://github.com/vercel/next.js/pull/45637))。
*   **效能：** 最佳化錯誤元件以減少 0.4kb 的 HTML 負載，而不改變樣式 ([PR](https://github.com/vercel/next.js/pull/45586))。
*   **效能：** 將 Edge 套件大小減少 ~130KB，幾乎是一半的大小，以便在部署到 Vercel 等 Edge 環境時進一步減少冷啟動大小 ([PR](https://github.com/vercel/next.js/pull/45601))。
*   **安全性：** 新增設定 `images.contentDispositionType: "attachment"`，在直接造訪影像最佳化 API 時強制下載影像 ([PR](https://github.com/vercel/next.js/pull/46254))。
