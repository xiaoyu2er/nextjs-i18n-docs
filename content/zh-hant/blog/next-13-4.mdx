Next.js 13.4 是一個奠基性版本，標誌著應用程式路由 (App Router) 的穩定:

*   [**應用程式路由 (App Router) 穩定**](#nextjs-app-router):
    *   React 伺服器元件
    *   嵌套路由與佈局
    *   簡化的資料載入
    *   串流與 Suspense
    *   內建 SEO 支援
*   [**Turbopack Beta**](#turbopack-beta): 您的本地開發伺服器，更快且具有改進的穩定性
*   [**伺服器動作 (Server Actions) Alpha**](#server-actions-alpha): 在伺服端變革資料，零客戶端 JavaScript

自從 [Next.js 13](/blog/next-13) 六個月前發布以來，我們一直致力於為 Next.js 的未來——App Router——建構基礎，並以無需不必要破壞性變化的方式進行漸進式採用。

今天，隨著 13.4 版本的發布，您現在可以開始將 App Router 導入生產環境。

終端機

```
npm i next@latest react@latest react-dom@latest eslint-config-next@latest
```

## Next.js 應用程式路由 (App Router)[](#nextjs-app-router)
----------------------------------------

我們在 2016 年發布了 Next.js，目的是提供一種簡單的方式來伺服端渲染 React 應用程式，我們的目標是創造一個更動態、更個人化以及更全球化的網頁。

[在最初的公告文章](https://vercel.com/blog/next) 中，我們分享了一些 Next.js 的設計原則:

*   **無需設定。將檔案系統視為 API**
*   **僅限 JavaScript。一切皆是函式**
*   **自動伺服端渲染和程式碼分割**
*   **資料載入由開發者主導**

Next.js 現在已經六歲了。我們最初的設計原則一直存在，並且隨著更多開發者和公司採用 Next.js，我們一直在致力於對框架進行基礎升級，以更好地實現這些原則。

我們一直致力於開發 Next.js 的下一代，而今天隨著 `13.4`，這一代產品已穩定並可投入使用。這篇文章將分享更多關於我們為 App Router 所做的設計決策和選擇。

### 無需設定。將檔案系統視為 API[](#zero-setup-use-the-filesystem-as-an-api)

[基於檔案系統的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在我們最初的帖子中，我們展示了如何從單一 React 元件建立路由的範例:

pages/about.js

```
// Pages Router
 
import React from 'react';
export default () => <h1>About us</h1>;
```

無需額外的設定。將檔案放入 `pages/` 目錄中，Next.js 的路由就會處理其餘部分。我們依然熱愛這種路由的簡潔性。但隨著框架用量的增長，開發者希望用它建立的介面類型也不斷變化。

開發者要求改進定義佈局的支持，將 UI 部件作為佈局嵌套，並對定義載入和錯誤狀態有更多靈活性。這並非易事，很難在現有 Next.js 路由中後貼。

框架的每個部分都必須圍繞路由進行設計。頁面轉換、資料載入、快取、變革和重新驗證資料、串流、內容樣式化等等。

為了使我們的路由與串流相容，並解決這些對佈局增強支持的需求，我們開始構建新版本的路由。

這就是我們在初始發布 [佈局 RFC](/blog/layouts-rfc) 後得到的結果。

`app/layout.js`

```
// New: App Router ✨
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

app/page.js

```
export default function Page() {
  return <h1>Hello, Next.js!</h1>;
}
```

比你所看到的更重要的是你 _沒看到_ 的。這個新路由（可以透過 `app/` 目錄漸進式採用）擁有完全不同的架構，建立在 [React 伺服器元件 (Server Components)](/docs/getting-started/react-essentials) 和 [Suspense](/docs/app/building-your-application/routing/loading-ui-and-streaming) 的基礎上。

這個基礎使我們能夠移除最初為擴充 React 原語而開發的 Next.js 特定 API。例如，您不再需要匯入自訂的 `_app` 檔案來設定全域共用佈局:

pages/_app.js

```
// Pages Router
 
// This "global layout" wraps all routes. There's no way to
// compose other layout components, and you cannot fetch global
// data from this file.
export default function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
```

使用頁面路由 (Pages Router)，佈局無法組合，資料載入也無法與元件並置。使用新的應用程式路由 (App Router)，現在已支援這些功能。

`app/layout.js`

```
// New: App Router ✨
// The root layout is shared for the entire application
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

app/dashboard/layout.js

```
// Layouts can be nested and composed
export default function DashboardLayout({ children }) {
  return (
    <section>
      <h1>Dashboard</h1>
      {children}
    </section>
  );
}
```

在使用 Pages Router 時，`_document` 用於自訂來自伺服器的初始有效載荷 (initial payload)。

pages/\_document.js

```
// Pages Router
 
// This file allows you to customize the <html> and <body> tags
// for the server request, but adds framework-specific features
// rather than writing HTML elements.
import { Html, Head, Main, NextScript } from 'next/document';
 
export default function Document() {
  return (
    <Html>
      <Head />
      <body>
        <Main />
        <NextScript />
      </body>
    </Html>
  );
}
```

使用應用程式路由，您不再需要從 Next.js 匯入 `<Html>`、`<Head>` 和 `<Body>`。相反，您只需要使用 React。

`app/layout.js`

```
// New: App Router ✨
// The root layout is shared for the entire application
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

建構新檔案系統路由的機會，也是解決我們路由系統許多其他相關功能請求的恰當時機。例如:

*   之前，您只能在 `_app.js` 中匯入來自外部 npm 套件（例如元件函式庫）的全域樣式表。這不是理想的開發者體驗 (DX)。使用 App Router，您可以在任何元件中匯入（並並置）任何 CSS 檔案。
*   以往，選擇採用 Next.js 的伺服端渲染（透過 `getServerSideProps`）意味著必須等待頁面完全水合，才能與應用程式互動。使用 App Router，我們重構了架構以深度整合 React Suspense，這意味著我們可以選取水合頁面的部分，而不影響 UI 中其他元件的互動性。內容可以立即從伺服器串流，改善頁面的感知載入效能。

[路由](/docs/app/building-your-application/routing) 是讓 Next.js 運作的核心。但這不是關於路由本身，而是它如何整合框架的其他部分——例如 [資料載入](/docs/app/building-your-application/data-fetching)。

### 僅限 JavaScript。一切皆是函式[](#only-javascript-everything-is-a-function)

Next.js 和 React 開發者希望撰寫 JavaScript 和 TypeScript 程式碼並將應用元件組合在一起。根據我們最初的帖子:

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import React from 'react';
import Head from 'next/head';
 
export default () => (
  <div>
    <Head>
      <meta name="viewport" content="width=device-width, initial-scale=1" />
    </Head>
    <h1>Hi. I'm mobile-ready!</h1>
  </div>
);
```

> 在未來版本的 Next.js 中，我們新增了一個開發者體驗 (DX) 改進，可讓您自動匯入 React。

這個元件封裝了可以在您的應用程式任何地方重複使用和組合的邏輯。配合基於檔案系統的路由，這意味著一種簡單的方式來開始建立 React 應用程式，感覺就像在撰寫 JavaScript 和 HTML。

例如，如果您想抓取一些資料，Next.js 的原始版本看起來像這樣:

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import React from 'react';
import 'isomorphic-fetch';
 
export default class extends React.Component {
  static async getInitialProps() {
    const res = await fetch('https://api.company.com/user/123');
    const data = await res.json();
    return { username: data.profile.username };
  }
}
```

> 在未來版本的 Next.js 中，我們新增了一個開發者體驗 (DX) 改進，多模態了 `fetch`，因此您不需要匯入 `isomorphic-fetch` 或 `node-fetch`，並且可以在客戶端和伺服端使用 Web `fetch API`。

隨著採用量增長和框架的成熟，我們探索了新的資料載入模式。

`getInitialProps` 在伺服端 _和_ 客戶端運行。此 API 擴充了 React 元件，允許您建立 `Promise` 並將結果轉發給元件的 `props`。

雖然 `getInitialProps` 至今仍然有效，但我們隨後根據客戶反饋向前迭代了下一代資料載入 API：`getServerSideProps` 和 `getStaticProps`。

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
// Generate a static version of the route
export async function getStaticProps(context) {
  return { props: {} };
}
// Or dynamically server-render the route
export async function getServerSideProps(context) {
  return { props: {} };
}
```

這些 API 讓您的程式碼運行位置（客戶端或伺服端）更加明確，並允許 Next.js 應用程式進行 [自動靜態優化](/docs/pages/building-your-application/rendering/automatic-static-optimization)。此外，它允許 [靜態匯出](/docs/app/building-your-application/deploying/static-exports)，使 Next.js 能夠部署到不支援伺服端的地方（例如 AWS S3 儲存桶）。

然而，這不是「只是 JavaScript」，我們希望更貼近我們最初設計原則。

自 Next.js 創建以來，我們一直與 Meta 的 React 核心團隊密切合作，在 React 原語之上建構框架功能。我們的合作，加上 React 核心團隊多年的研究和開發，為 Next.js 通過 React 架構的最新版本實現了我們的目標，包括 [伺服器元件 (Server Components)](/docs/getting-started/react-essentials)。

使用 App Router，您可以使用熟悉的 `async` 和 `await` 語法 [抓取資料](/docs/app/building-your-application/data-fetching)。無需學習新的 API。預設情況下，所有元件都是 React 伺服器元件，因此資料載入在伺服端安全地進行。例如:

app/page.js

```
export default async function Page() {
  const res = await fetch('https://api.example.com/...');
  // The return value is *not* serialized
  // You can use Date, Map, Set, etc.
  const data = res.json();
 
  return '...';
}
```

關鍵的是，「資料載入由開發者主導」的原則已實現。您可以抓取資料並組合 _任何_ 元件。不僅是第一方的元件，而是伺服器元件生態系統中的 _任何_ 元件，例如 [Twitter 嵌入](https://github.com/vercel-labs/react-tweet) `react-tweet`，它已設計為與伺服器元件整合並完全在伺服端運行。

app/page.js

```
import { Tweet } from 'react-tweet';
 
export default async function Page() {
  return <Tweet id="790942692909916160" />;
}
```

由於路由與 [React Suspense](https://react.dev/reference/react/Suspense) 整合，您可以在內容部分載入時更流暢地顯示回退內容，並按需求逐步揭露內容。

app/page.js

```
import { Suspense } from 'react';
import { PostFeed, Weather } from './components';
 
export default function Page() {
  return (
    <section>
      <Suspense fallback={<p>Loading feed...</p>}>
        <PostFeed />
      </Suspense>
      <Suspense fallback={<p>Loading weather...</p>}>
        <Weather />
      </Suspense>
    </section>
  );
}
```

此外，路由將頁面導航標記為 [轉換](https://react.dev/reference/react/useTransition)，啟用了路由轉換可中斷。

### 自動伺服端渲染和程式碼分割[](#automatic-server-rendering-and-code-splitting)

當我們創建 Next.js 時，開發者手動配置 webpack、babel 和其他工具以使 React 應用運行仍很常見。添加進一步的優化，如伺服端渲染或程式碼分割，通常未在自訂解決方案中實施。Next.js 以及其他 React 框架創建了抽象層來實施並強制執行這些最佳實踐。

基於路由的程式碼分割意味著您的 `pages/` 目錄中的每個檔案都會被分割到自己的 JavaScript 捆綁包中，有助於減少檔案系統並改善初始頁面載入效能。

這對伺服端渲染的應用程式以及 Next.js 的單頁應用程式 (SPA) 都有益，因為後者往往在應用程式啟動時加載一個大型 JavaScript 捆綁包。但是，要實施元件級程式碼分割，開發者需要使用 `next/dynamic` 動態匯入元件。

app/page.tsx

```
import dynamic from 'next/dynamic';
 
const DynamicHeader = dynamic(() => import('../components/header'), {
  loading: () => <p>Loading...</p>,
});
 
export default function Home() {
  return <DynamicHeader />;
}
```

使用 App Router，伺服器元件不包含在瀏覽器的 JavaScript 捆綁包中。 [客戶端元件 (Client Components)](/docs/getting-started/react-essentials#client-components) 預設會自動進行程式碼分割（使用 Next.js 中的 webpack 或 Turbopack）。此外，由於整個路由架構是串流和 Suspense 啟用的，您可以從伺服端逐步發送您的 UI 部分給客戶端。

例如，您可以使用條件邏輯分割整個程式碼路徑。在這個範例中，您不需要為未登入用戶載入儀表板的客戶端 JavaScript。

app/layout.tsx

```
import { getUser } from './auth';
import { Dashboard, Landing } from './components';
 
export default async function Layout() {
  const isLoggedIn = await getUser();
  return isLoggedIn ? <Dashboard /> : <Landing />;
}
```

Turbopack (Beta)[](#turbopack-beta)
-----------------------------------

[Turbopack](https://nextjs.org/docs/app/api-reference/turbopack)，這是我們正在測試並穩定的新打包器，透過 Next.js 幫助加快在您的 Next.js 應用程式上工作時的本地迭代速度（透過 `next dev --turbo`）以及即將到來的生產構建（`next build --turbo`）。

自 Next.js 13 的 Alpha 版本發布以來，隨著我們修復錯誤並添加缺少的功能支援，我們的採用量穩步增長。我們一直在 [Vercel.com](https://vercel.com) 上使用 Turbopack 並與許多運營大型 Next.js 網站的 Vercel 客戶一起使用，以收集改進穩定性的反饋。我們要感謝社群在測試和向我們的團隊報告錯誤方面的支持。

六個月後，我們已準備好進入 Beta 階段。

Turbopack 尚未具有與 webpack 和 Next.js 的完整功能對等性。我們在 [此問題](https://github.com/vercel/next.js/issues/49174) 中追蹤這些功能的支援情況。但是，大多數用例現在應該已獲支援。此 Beta 的目標是繼續解決來自採用量增加的剩餘錯誤，並為未來版本的穩定性做好準備。

我們在改善 Turbopack 的增量引擎和快取層上的投資不僅將加快本地開發速度，還將很快加快生產構建速度。請關注未來的 Next.js 版本，您將能夠運行 `next build --turbo` 進行瞬間構建。

今天就在 Next.js 13.4 中嘗試 [Turbopack](/docs/architecture/turbopack) Beta，使用 `next dev --turbo`。

伺服器動作 (Server Actions) (Alpha)[](#server-actions-alpha)
-----------------------------------------------

React 生態系統在表單、管理表單狀態以及資料快取和重新驗證方面經歷了大量創新和想法探索。隨著時間的推移，React 對其中一些模式變得更加有明確的意見。例如，表單狀態推薦 [「無控制元件」](https://react.dev/learn/sharing-state-between-components#controlled-and-uncontrolled-components)。

目前的解決方案生態系統要么是可重用的客戶端解決方案，要么是内置于框架中的原語。到目前为止，还没有一种方法来组合伺服器变革和资料原語。React 團隊 [一直在努力](https://react.dev/blog/2023/03/22/react-labs-what-we-have-been-working-on-march-2023) 變革的第一方解決方案。

我們很高興宣佈在 Next.js 中支援實驗性 **伺服器動作 (Server Actions)**，使您能夠在伺服端變革資料，直接呼叫函式_無需_建立中間 API 層。

app/post/[id]/page.tsx (伺服器元件)

```
import kv from './kv';
 
export default function Page({ params }) {
  async function increment() {
    'use server';
    await kv.incr(`post:id:${params.id}`);
  }
 
  return (
    <form action={increment}>
      <button type="submit">Like</button>
    </form>
  );
}
```

使用伺服器動作，您擁有強大的伺服器優先資料變革、較少的客戶端 JavaScript 以及逐步增強的表單。

app/post/new/page.tsx (伺服器元件)

```
import db from './db';
import { redirect } from 'next/navigation';
 
async function create(formData: FormData) {
  'use server';
  const post = await db.post.insert({
    title: formData.get('title'),
    content: formData.get('content'),
  });
  redirect(`/blog/${post.slug}`);
}
 
export default function Page() {
  return (
    <form action={create}>
      <input type="text" name="title" />
      <textarea name="content" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

Next.js 中的伺服器動作已設計為與資料生命週期的其餘部分深度整合，包括 Next.js 快取、增量靜態再生 (ISR) 以及用戶端路由。

透過新 API `revalidatePath` 和 `revalidateTag` 重新驗證資料意味著變革、重新呈現頁面或重新導向可以發生在 **一次網路往返** 中，確保在客戶端顯示正確的資料，即使上游提供者較慢。

app/dashboard/posts/page.tsx (伺服器元件)

```
import db from './db';
import { revalidateTag } from 'next/cache';
 
async function update(formData: FormData) {
  'use server';
  await db.post.update({
    title: formData.get('title'),
  });
  revalidateTag('posts');
}
 
export default async function Page() {
  const res = await fetch('https://...', { next: { tags: ['posts'] } });
  const data = await res.json();
  // ...
}
```

伺服器動作設計為可組合的。React 社群中的任何人都可以建立和發布伺服器動作，並在生態系統中分發它們。就像伺服器元件一樣，我們對客戶端和伺服端的新世代可組合原語感到興奮。

[伺服器動作](/docs/app/building-your-application/data-fetching/server-actions) 在 Next.js 13.4 中已於 Alpha 階段提供。通過選擇使用伺服器動作，Next.js 將使用 React 的實驗性發布頻道。

next.config.js

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    serverActions: true,
  },
};
 
module.exports = nextConfig;
```

常見問題 [](#frequently-asked-questions)
---------------------------------------------------------

### App Router 穩定性意味著什麼？[](#what-does-app-router-stability-mean)

今天將 App Router 標定為穩定並不意味著我們的工作完成了。穩定性意味著 App Router 的核心已準備就緒並投入生產，並且已通過我們自己的內部測試以及許多 Next.js 早期採用者的測試驗證。

未來我們仍有一些額外的優化想進行，包括伺服器動作達到完全穩定。我們將核心穩定性推向前沿對我們來說很重要，以幫助社群明確他們今天應該從哪開始學習和構建應用程式。

App Router 建立在 React `canary` 頻道之上，現在已準備好針對伺服器元件等功能進行框架採用。[了解更多](https://react.dev/blog/2023/05/03/react-canaries)。

### 這意味著 Next.js Beta 文件或什麼意思？[](#what-does-this-mean-for-the-nextjs-beta-docs)

從今天開始，我們建議使用 App Router 建置新的應用程式。Next.js Beta 文件說明（已從頭開始重新撰寫用於解釋 App Router），現已合併回 [穩定版 Next.js 文件](/docs)。您現在可以輕鬆切換 App 或 Pages 路由。

我們建議閱讀 [App Router 漸進式採用指南](/docs/app/building-your-application/upgrading/app-router-migration) 以了解如何採用 App Router。

### Pages Router 將消失嗎？[](#is-the-pages-router-going-away)

不。我們致力於支援 `pages/` 開發，包括在多代版本中的錯誤修復、改進和安全修補。我們希望確保開發者有足夠的時間按準備程度漸進式採用 App Router。

同時在生產中使用 `pages/` 和 `app/` 是得到支援且受鼓勵的。App Router 可以按 [每路由基礎](/docs/app/building-your-application/upgrading/app-router-migration) 採用。

### 這意味著伺服器元件已「完成」嗎？[](#does-this-mean-server-components-are-complete)

Next.js 是一個選擇建立在 React 架構（包括伺服器元件）上的框架。我们希望 App Router 提供的體驗能鼓勵其他框架（或新框架）也考慮使用此架構。

這個生態系統中仍有待定義的模式，例如處理無限捲動。暫時，我們建議使用客戶端解決方案進行這些模式，直到生態系統成長並建立或更新函式庫。
