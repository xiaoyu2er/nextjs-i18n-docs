Following the announcement of the first Next.js 15 Release Candidate [back in May,](https://nextjs.org/blog/next-15-rc) we've been preparing a second Release Candidate based on your feedback. Here's what we've been working on:

*   [**`@next/codemod upgrade`**](#smooth-upgrades-with-codemod-cli): 輕鬆升級至最新的 Next.js 和 React 版本。
*   [**Turbopack 用於開發**](#turbopack-for-development): 效能改進和 Next.js 15 穩定性目標。
*   [**異步請求 API（破壞性變更）**](#async-request-apis-breaking-change): 朝向簡化渲染和快取模型的漸進式步驟。
*   [**伺服器操作**](#enhanced-security-for-server-actions): 增強安全性，包含不可猜測的端點和移除未使用的操作。
*   [**靜態指示器**](#static-route-indicator): 新的視覺指示器在開發期間顯示靜態路由。
*   [**`next/form`**](#form-component): 使用用戶端導航增強 HTML 表單。
*   [**`next.config.ts`**](#support-for-nextconfigts): Next.js 配置檔案的 TypeScript 支援。
*   [**`instrumentation.js`（穩定版）**](#instrumentationjs-stable): 伺服器生命週期可觀測性的新 API。
*   [**開發和建置改進**](#development-and-build-improvements): 改進建置時間和更快速的快閃重新整理。
*   [**自主託管**](#improvements-for-self-hosting): 對 `Cache-Control` 標頭有更多的控制權。
*   [**程式碼檢查**](#eslint-9-support): 新增對 ESLint 9 的支援。

今天試試 Next.js 15 預發版本（RC2）：

```
# Use the new automated upgrade CLI
npx @next/codemod@canary upgrade rc
 
# ...or upgrade manually
npm install next@rc react@rc react-dom@rc
```

> **注意：** 此預發版本（RC）包含 [上一個 RC](/blog/next-15-rc) 的所有變更。

使用 Codemod CLI 無縫升級 [](#smooth-upgrades-with-codemod-cli)
---------------------------------------------------------------------

我們在每個主要的 Next.js 發布中都會包含 codemods（自動代碼轉換），以幫助自動化升級破壞性變更。

為了讓升級更順暢，我們發布了一個增強版的 codemod CLI：

```
npx @next/codemod@canary upgrade rc
```

此工具可協助您將程式碼庫升級至最新的穩定版本或預 releases 版本。CLI 將更新依賴項、顯示可用的 codemods，並引導您應用它們。命令列上指定的 dist 標籤（`@rc`、`@canary` 等）決定了要升級的版本。

更多詳情請查阅 [Next.js codemods](/docs/canary/app/building-your-application/upgrading/codemods)。


*   **記憶體使用量減少 25–35%**。
*   **編譯速度提升 30–50%**（針對擁有數千個模組的大頁面）。

我們將在未來的發布中持續優化這些領域。

展望未来，Turbopack 團隊在持久化緩存、記憶體使用量減少以及 Turbopack for `next build` 方面取得了顯著進展，[96% 的測試通過](https://areweturboyet.com/build)。

異步請求 API（破壞性變更） [](#async-request-apis-breaking-change)
---------------------------------------------------------------------------

在傳統的伺服器端渲染中，伺服器會在渲染任何內容之前等待請求。然而，並非所有元件都依賴特定請求的數據，因此不需要等待請求來渲染它們。理想情況下，伺服器會在請求到達之前盡可能準備。為了實現這一點，並為未來優化奠定基礎，我們需要知道何時等待請求。

因此，我們正在將依賴特定請求數據的 API（例如 `headers`、`cookies`、`params` 和 `searchParams`）轉變為異步。

```
import { cookies } from 'next/headers';
 
export async function AdminPanel() {
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
 
  // ...
}
```

這是一種破壞性變更，會影響以下 API：

* `cookies`
* `headers`
* `draftMode`
* `layout.js`、`page.js`、`route.js`、`default.js`、`generateMetadata` 和 `generateViewport` 中的 `params`
* `page.js` 中的 `searchParams`

為了方便遷移，這些 API 可以暫時通過同步方式存取，但在下一個主要版本之前，開發和生產環境將顯示警告。有一個 codemod 可用於自動化遷移：

```
npx @next/codemod@canary next-async-request-api .
```

對於 codemod 無法完全遷移代碼的情況，請參閱 [升級指南](/docs/canary/app/building-your-application/upgrading/version-15)。我們也提供了如何將 Next.js 應用程式遷移至新 API 的 [範例](https://github.com/leerob/next-saas-starter/pull/62)。

伺服器操作的增強安全性 [](#enhanced-security-for-server-actions)
-----------------------------------------------------------------------------

[伺服器操作](https://react.dev/reference/rsc/server-actions) 是在伺服器端定義、可從客戶端呼叫的函數。它們通过在文件頂部添加 'use server' 指令並匯出異步函數來定義。

即使 Server Action 或工具函數未在程式碼的其他地方匯入，它仍然是一個可公開存取的 HTTP 端點。雖然此行為在技術上是正確的，但它可能會導致此類函數意外暴露。

為了提高安全性，我們引入了以下增強功能：

*   **Dead code elimination（代碼消除）：** 未使用的 Server Action 不會將其 ID 暴露給客戶端 JavaScript 捆綁包，減少捆綁包大小並提高性能。
*   **Secure action IDs（安全操作 ID）：** Next.js 現在生成不可猜測、非確定性的 ID，以允許客戶端參照和呼叫 Server Action。這些 ID 會在構建之間定期重新計算以增強安全性。

```
// app/actions.js
'use server';
 
// This action **is** used in our application, so Next.js
// will create a secure ID to allow the client to reference
// and call the Server Action.
export async function updateUserAction(formData) {}
 
// This action **is not** used in our application, so Next.js
// will automatically remove this code during `next build`
// and will not create a public endpoint.
export async function deleteUserAction(formData) {}
```

您仍應將 Server Actions 視為公開的 HTTP 端點。更多詳情請查阅如何 [保護 Server Actions](https://nextjs.org/blog/security-nextjs-server-components-actions#write)。

`<Form>` 組件 [](#form-component)
-------------------------------------

新的 `<Form>` 元件在 HTML `<form>` 元素的基礎上擴充了預取、客戶端導航和漸進式增強功能。

它適用於導航到新頁面的表單，例如導向結果頁面的搜尋表單。

```
import Form from 'next/form';
 
export default function Page() {
  return (
    <Form action="/search">
      <input name="query" />
      <button type="submit">Submit</button>
    </Form>
  );
}
```

`<Form>` 元件配備了：

*   **預取：** 當表單在視圖中時，[佈局](/docs/app/api-reference/file-conventions/layout) 和 [載入](/docs/app/api-reference/file-conventions/loading) UI 會被預取，使導航變快。
*   **用戶端導航：** 提交時，共享佈局和用戶端狀態會被保留。
*   **漸進式增強：** 如果 JavaScript 尚未載入，表單仍可透過整頁導航運作。

過去，實現這些功能需要大量手動的範本碼。例如：

範例

```
// Note: This is abbreviated for demonstration purposes.
// Not recommended for use in production code.
 
'use client'
 
import { useEffect } from 'react'
import { useRouter } from 'next/navigation'
 
export default function Form(props) {
  const action = props.action
  const router = useRouter()
 
  useEffect(() => {
    // if form target is a URL, prefetch it
    if (typeof action === 'string') {
      router.prefetch(action)
    }
  }, [action, router])
 
  function onSubmit(event) {
    event.preventDefault()
 
    // grab all of the form fields and trigger a `router.push` with the data URL encoded
    const formData = new FormData(event.currentTarget)
    const data = new URLSearchParams()
 
    for (const [name, value] of formData) {
      data.append(name, value as string)
    }
 
    router.push(`${action}?${data.toString()}`)
  }
 
  if (typeof action === 'string') {
    return <form onSubmit={onSubmit} {...props} />
  }
 
  return <form {...props} />
}
```

更多詳情請查阅 [`<Form>` 元件](/docs/canary/app/api-reference/components/form)。

`instrumentation.js`（穩定版） [](#instrumentationjs-stable)
----------------------------------------------------------

`instrumentation` 檔案配合 `register()` API，允許用戶介入 Next.js 伺服器生命週期以監控效能、追蹤錯誤來源，並與 [OpenTelemetry](https://opentelemetry.io/) 等可觀測性函式庫深度整合。

此功能現在已 **穩定**，`experimental.instrumentationHook` 配置選項可以移除。

此外，我們與 [Sentry](https://sentry.io/) 合作設計了一個新的 `onRequestError` 函式庫，可用於：

*   捕捉伺服器上拋出的所有錯誤的重要上下文，包括：
    *   **Router：** Pages Router 或 App Router
    *   **伺服器上下文：** Server Component、Server Action、Route Handler 或 Middleware
*   將錯誤報告給您喜愛的可觀測性提供者。

```
export async function onRequestError(err, request, context) {
  await fetch('https://...', {
    method: 'POST',
    body: JSON.stringify({ message: err.message, request, context }),
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export async function register() {
  // init your favorite observability provider SDK
}
```

更多詳情請查阅 `onRequestError` [函式](/docs/canary/app/api-reference/file-conventions/instrumentation#onrequesterror-optional)。

開發與建置改進 [](#development-and-build-improvements)
-------------------------------------------------------------------------

### Server Components HMR [](#server-components-hmr)

在開發期間，Server 元件在儲存時會重新執行。這意味著，對 API 端點或第三方服務的 `fetch` 請求也會被呼叫。

為了改善本地開發效能並減少計費 API 呼叫的潛在成本，我們現在確保熱模組更換（HMR）可以重用先前渲染的 `fetch` 回應。

更多詳情請查阅 [Server Components HMR Cache](/docs/canary/app/api-reference/next-config-js/serverComponentsHmrCache)。

### App Router 的靜態生成速度更快 [](#faster-static-generation-for-the-app-router)

我們已優化靜態生成以改善建置時間，特別是對於有慢速網路請求的頁面。

過去，我們靜態優化流程會渲染兩次頁面——一次用於生成用戶端導航數據，二次用於渲染初始頁面訪問的 HTML。現在，我們重用第一次渲染，省去第二次遍歷，減少工作量和建置時間。

此外，靜態生成工作程序現在跨頁面共用 `fetch` 緩存。如果 `fetch` 呼叫沒有選擇退出緩存，其結果將由同一工作程序處理的其他頁面重用。這減少了相同數據的請求數量。

### 進階靜態生成控制（實驗性） [](#advanced-static-generation-control-experimental)

我們已為需要更多控制的進階使用案例新增實驗性支援，以對靜態生成過程進行更多控制。

我們建議除非您有特定需求，否則使用目前的預設值，因為這些選項可能會導致資源使用量增加，並因並發性增加而出現記憶體耗盡錯誤。

```
const nextConfig = {
  experimental: {
	  // how many times Next.js will retry failed page generation attempts
	  // before failing the build
    staticGenerationRetryCount: 1
    // how many pages will be processed per worker
    staticGenerationMaxConcurrency: 8
    // the minimum number of pages before spinning up a new export worker
    staticGenerationMinPagesPerWorker: 25
  },
}
 
export default nextConfig;
```

更多詳情請查阅 [靜態生成選項](/docs/canary/app/api-reference/next-config-js/staticGeneration)。

自主託管改進 [](#improvements-for-self-hosting)
---------------------------------------------------------------

當自主託管應用程式時，您可能需要對 `Cache-Control` 指令有更多控制權。

一個常見的案例是控制發送到 ISR 頁面的 `stale-while-revalidate` 期間。我們實作了兩項改進：

1. 您現在可以在 `next.config` 中配置 `expireTime` 值。這過去是 `experimental.swrDelta` 選項。
2. 將預設值更新為一年，確保大多數 CDNs 能完全按意圖套用最過時重驗證語義。

我們也不再覆蓋自訂 `Cache-Control` 值為我們的預設值，允許完全控制並確保與任何 CDN 設定相容。

最後，我們改進了自主託管時的圖片優化。過去，我們建議您安裝 `sharp` 以在 Next.js 伺服器上優化圖片。此建議有時會被遺漏。隨著 Next.js 15，您不再需要手動安裝 `sharp` — `next start` 或使用 [單獨輸出模式](/docs/app/api-reference/next-config-js/output) 執行時，Next.js 將自動使用 `sharp`。

要了解更多詳情，請參閱我們關於自主託管 Next.js 的 [新示範和教學影片](https://x.com/leeerob/status/1843796169173995544)。

ESLint 9 支援 [](#eslint-9-support)
-------------------------------------

Next.js 15 也引入了對 [ESLint 9](https://eslint.org/blog/2024/04/eslint-v9.0.0-released) 的支援，隨後 ESLint 8 於 2024 年 10 月 5 日終止支援。

為了確保平穩過渡，Next.js 仍保留向下相容，意味著您仍可使用 ESLint 8 或 9。

如果您升級至 ESLint 9，且我們檢測到您尚未採用 [新的配置格式](https://eslint.org/blog/2024/04/eslint-v9.0.0-released/#flat-config-is-now-the-default-and-has-some-changes)，Next.js 將自動套用 `ESLINT_USE_FLAT_CONFIG=false` 逃生孔以簡化遷移。

此外，運行 `next lint` 時將移除已棄用的選項，如 `—ext` 和 `—ignore-path`。請注意，ESLint 最終将在 ESLint 10 中不再允許這些舊配置，因此我們建議您盡快開始遷移。

有關這些變更的更多詳情，請參閱 [遷移指南](https://eslint.org/docs/latest/use/migrate-to-9.0.0)。

作為此更新的一部分，我們也將 `eslint-plugin-react-hooks` 升級至 `v5.0.0`，其中引入了適用於 React Hooks 使用的新規則。您可以在 `eslint-plugin-react-hooks@5.0.0` 的 [變更記錄](https://github.com/facebook/react/releases/tag/eslint-plugin-react-hooks%405.0.0) 中檢視所有變更。
