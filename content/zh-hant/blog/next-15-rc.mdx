Next.js 15 的发布候选版 (RC) 现已提供。這個早期版本允許您在即將到來的穩定版本发布之前，測試最新的功能。

* [**React：**](#react-19-rc) 支援 React 19 RC（发布候选版）、React Compiler（實驗性）以及水合錯誤改進
* [**緩存 (Caching)：**](#caching-updates) `fetch` 請求、`GET` 路由處理函數和客戶端導航默認不再緩存
* [**部分預渲染 (實驗性)：**](#incremental-adoption-of-partial-prerendering-experimental) 新增用於漸進式導入的 Layout 和 Page 配置選項
* [**`next/after`（實驗性）：**](#executing-code-after-a-response-with-nextafter-experimental) 新增在回應完成串流後執行程式碼的 API
* [**`create-next-app`：**](#create-next-app-updates) 設計更新以及新增可在本地開發中啟用 Turbopack 的旗標
* [**外部套件打包 (穩定)：**](#optimizing-bundling-of-external-packages-stable) 新增 App Router 和 Pages Router 的配置選項

今天即可試用 Next.js 15 RC：

終端機

```
npm install next@rc react@rc react-dom@rc
```

React 19 RC（发布候选版）[](#react-19-rc)
--------------------------------------------

Next.js App Router 是建立在 React 的 [canary 通道](https://react.dev/blog/2023/05/03/react-canaries) 之上，適用於框架，這使得開發者能夠在 v19 版本发布之前使用和提供關於這些新 React API 的迴響。

Next.js 15 RC 目前支援 React 19 RC，其中包含適用於客戶端和服務器的新功能，例如 Actions。

請閱讀 [Next.js 15 升級指南](https://nextjs.org/docs/app/building-your-application/upgrading/version-15)、[React 19 升級指南](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)，並觀看 [React Conf Keynote](https://www.youtube.com/live/T8TZQ6k4SLE?t=1788) 以了解更多。

> **注意：** 某些第三方函式庫可能尚未與 React 19 相容。

React Compiler（實驗性）[](#react-compiler-experimental)
-------------------------------------------------------

[React Compiler](https://react.dev/learn/react-compiler) 是 React 團隊在 Meta 創建的新實驗性編譯器。該編譯器透過對純 JavaScript 語法和 [React 規則](https://react.dev/reference/rules) 的深度理解來理解您的程式碼，這使得它能夠為您的程式碼添加自動最佳化。它透過 `useMemo` 和 `useCallback` 等 API 減少開發者必須執行的手動 memoization 數量，使程式碼更簡單、更易於維護，且不易出錯。

有了 Next.js 15，我們已新增對 [React Compiler](https://react.dev/learn/react-compiler) 的支援。

安裝 `babel-plugin-react-compiler`：

終端機

```
npm install babel-plugin-react-compiler
```

然後，在 `next.config.js` 中新增 `experimental.reactCompiler` 選項：

next.config.ts

```
const nextConfig = {
  experimental: {
    reactCompiler: true,
  },
};
 
module.exports = nextConfig;
```

您可以選擇性地將編譯器配置為以「opt-in」模式運行，如下所示：

next.config.ts

```
const nextConfig = {
  experimental: {
    reactCompiler: {
      compilationMode: 'annotation',
    },
  },
};
 
module.exports = nextConfig;
```

> **注意：** React Compiler 目前僅能透過 Babel plugin 在 Next.js 中使用，這可能會導致構建時間變慢。

請了解更多關於 [React Compiler](https://react.dev/learn/react-compiler) 和 [可用的 Next.js 配置選項](https://react.dev/learn/react-compiler#usage-with-nextjs)。

緩存 (Caching) 更新[](#caching-updates)
----------------------------------

Next.js App Router 啟動時帶有預設的緩存策略。這些策略旨在預設提供最具效能的選項，並在需要時能夠選擇退出。

基於您的迴響，我們重新評估了我們的 [緩存啟發式演算法](https://x.com/feedthejim/status/1785242054773145636) 以及它們如何與 Partial Prerendering (PPR) 等專案以及使用 `fetch` 的第三方函式庫互動。

在 Next.js 15 中，我們將 `fetch` 請求、`GET` 路由處理函數和客戶端路由緩存的預設緩存從「預設緩存」改為「預設不緩存」。如果您想保留以前的行為，您可以繼續選擇加入緩存。

我們將在未來幾個月繼續改進 Next.js 的緩存，並將透過 Next.js 15 GA 公告分享更多詳細資訊。

### `fetch` 請求不再預設緩存[](#fetch-requests-are-no-longer-cached-by-default)

Next.js 使用 [Web `fetch` API](https://developer.mozilla.org/docs/Web/API/Fetch_API) 緩存選項來配置服務器端的 fetch 請求如何與框架的持久性 HTTP 緩存互動：

```
fetch('https://...', { cache: 'force-cache' | 'no-store' });
```

* `no-store` - 每次請求都從遠端伺服器獲取資源並不要更新緩存
* `force-cache` - 從緩存（如果存在）或遠端伺服器獲取資源並更新緩存

在 Next.js 14 中，如果未提供 `cache` 選項，除非使用了 dynamic function 或 dynamic config 選項，否則默認使用 `force-cache`。

在 Next.js 15 中，如果未提供 `cache` 選項，則預設使用 `no-store`。這意味著 **fetch 請求預設不會被緩存**。

您仍然可以通過以下方式選擇加入緩存 `fetch` 請求：

* 在單個 `fetch` 呼叫中將 [`cache` 選項](https://nextjs.org/docs/app/api-reference/functions/fetch#optionscache) 設定為 `force-cache`
* 在單個網路透路中將 [`dynamic` 路由配置選項](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#dynamic) 設定為 `'force-static'`
* 將 [`fetchCache` 路由配置選項](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#fetchcache) 設定為 `'default-cache'` 以覆蓋 Layout 或 Page 中的所有 `fetch` 請求以使用 `force-cache`，除非它們明確指定了自己的 `cache` 選項

### 客戶端路由緩存不再預設緩存 Page 組件[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next 14 中，使用 `GET` HTTP 方法的 路由處理器 (Route Handlers) 預設會被緩存，除非它們使用了動態函數或動態配置選項。在 Next.js 15 中，`GET` 函數**預設不會被緩存**。

您仍可使用靜態路由配置選項來選擇啟用緩存，例如 `export dynamic = 'force-static'`。

特殊路由處理器 (Route Handlers)，如 [`sitemap.ts`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/sitemap)、[`opengraph-image.tsx`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/opengraph-image) 和 [`icon.tsx`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/app-icons)，以及其他 [元數據文件 (metadata files)](https://nextjs.org/docs/app/api-reference/file-conventions/metadata)，除非使用動態函數或動態配置選項，否則預設保持靜態。

### 客戶端路由緩存 (Client Router Cache) 預設不再緩存頁面組件 (Page Components)[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next.js 14.2.0 中，我們引入了一個實驗性的 [`staleTimes`](https://nextjs.org/docs/app/api-reference/next-config-js/staleTimes) 旗標，以允許對 [Router Cache](https://nextjs.org/docs/app/building-your-application/caching#client-side-router-cache) 進行自訂配置。

在 Next.js 15 中，此旗標仍然可用，但我們正在更改預設行為，使 Page segments 的 `staleTime` 為 `0`。這意味著當您在應用程式中導航時，客戶端將總是反映作為導航部分變得活動的 Page 組件(s) 的最新數據。然而，仍有一些重要行為保持不變：

* 共用佈局數據將不會從伺服器重新獲取，以繼續支援 [部分渲染](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
* 返回 / 前進導航仍將從緩存還原，以確保瀏覽器可以還原捲動位置。
* [Loading.js](https://nextjs.org/docs/app/api-reference/file-conventions/loading) 將緩存 5 分鐘（或 `staleTimes.static` 配置的數值）。

您可以通過設定以下配置來選擇加入以前的客戶端路由緩存行為：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
    },
  },
};
 
module.exports = nextConfig;
```

Partial Prerendering (實驗性) 的漸進式導入[](#incremental-adoption-of-partial-prerendering-experimental)

在 Next.js 14 中，我們 [引入了 Partial Prerendering (PPR)](/blog/next-14#partial-prerendering-preview) - 一項在同一頁面中結合 [靜態和動態渲染](https://nextjs.org/docs/app/building-your-application/rendering/server-components#server-rendering-strategies) 的優化技術。

Next.js 目前預設使用靜態渲染，除非您使用 [dynamic functions](https://nextjs.org/docs/app/building-your-application/routing/route-handlers#dynamic-functions) 例如 `cookies()`、`headers()` 和未緩存的資料請求。這些 APIs 將整個路由選擇為動態渲染。透過 PPR，您可以將任何動態 UI 包裝在 Suspense boundary 中。當新請求進入時，Next.js 將立即提供靜態 HTML shell，然後在同一 HTTP 請求中渲染和串流動態部分。

為了允許漸進式導入，我們已新增 `experimental_ppr` 路由配置選項，用於將特定的 Layouts 和 Pages 選擇進入 PPR：

app/page.jsx

```
import { Suspense } from "react"
import { StaticComponent, DynamicComponent } from "@/app/ui"
 
export const experimental_ppr = true
 
export default function Page() {
  return {
     <>
	     <StaticComponent />
	     <Suspense fallback={...}>
		     <DynamicComponent />
	     </Suspense>
     </>
  };
}
```

要使用此新選項，您需要將 `next.config.js` 檔案中的 `experimental.ppr` 配置設定為 `'incremental'`：

next.config.ts

```
const nextConfig = {
  experimental: {
    ppr: 'incremental',
  },
};
 
module.exports = nextConfig;
```

一旦所有 segments 都啟用了 PPR，您就可以設定 `ppr` 值為 `true` 並為整個應用程式和所有未來路線啟用它。

我們將會在 Next.js 15 GA 部落格文章中分享更多關於 PPR 路线图的資訊。

請了解更多關於 [Partial Prerendering](https://nextjs.org/docs/app/building-your-application/rendering/partial-prerendering)。

執行回應後的程式碼 `next/after`（實驗性）[](#executing-code-after-a-response-with-nextafter-experimental)

在處理使用者請求時，伺服器通常會執行與計算回應直接相關的任務。然而，您可能需要執行記錄、分析和其他外部系統同步等任務。

由於這些任務與回應不直接相關，使用者不應該等待它們完成。在回應使用者後延遲工作會帶來挑戰，因為 serverless 函式會在回應關閉後立即停止計算。

`after()` 是一個新的實驗性 API，它解決了這個問題，允許您排定工作以在回應完成串流後處理，使得次要任務能夠執行而不阻擋主要回應。

要使用它，請在 `next.config.js` 中新增 `experimental.after`：

next.config.ts

```
const nextConfig = {
  experimental: {
    after: true,
  },
};
 
module.exports = nextConfig;
```

然後，在 Server Components、Server Actions、Route Handlers 或 Middleware 中匯入該函數。

```
import { unstable_after as after } from 'next/server';
import { log } from '@/app/utils';
 
export default function Layout({ children }) {
  // Secondary task
  after(() => {
    log();
  });
 
  // Primary task
  return <>{children}</>;
}
```

請了解更多關於 [`next/after`](https://nextjs.org/docs/app/api-reference/functions/unstable_after)。

最佳化外部套件打包 (穩定)[](#optimizing-bundling-of-external-packages-stable)

打包外部套件可以改善應用程式的冷啟動效能。在 **App Router** 中，外部套件預設會被打包，您可以使用新的 [`serverExternalPackages`](https://nextjs.org/docs/app/api-reference/next-config-js/serverExternalPackages) 配置選項選擇退出特定的套件。

在 **Pages Router** 中，外部套件預設不會被打包，但您可以使用現有的 [`transpilePackages`](https://nextjs.org/docs/pages/api-reference/next-config-js/transpilePackages) 選項提供要打包的套件清單。使用此配置選項，您需要指定每個套件。

為了統一 App 和 Pages Router 之間的配置，我們正在引入一個新的選項 [`bundlePagesRouterDependencies`](https://nextjs.org/docs/pages/api-reference/next-config-js/bundlePagesRouterDependencies) 以匹配 App Router 的預設自動打包。然後您可以使用 [`serverExternalPackages`](https://nextjs.org/docs/app/api-reference/next-config-js/serverExternalPackages) 選擇退出特定的套件（如果需要）。

next.config.ts

```
const nextConfig = {
  // Automatically bundle external packages in the Pages Router:
  bundlePagesRouterDependencies: true,
  // Opt specific packages out of bundling for both App and Pages Router:
  serverExternalPackages: ['package-name'],
};
 
module.exports = nextConfig;
```

請了解更多關於 [最佳化外部套件](https://nextjs.org/docs/app/building-your-application/optimizing/package-bundling)。
