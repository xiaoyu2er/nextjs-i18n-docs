Next.js 13.1 包含對 `pages/`(穩定版) 和 `app/`(測試版) 目錄的改進:

*   [**`app` 目錄(測試版)改進:**](#improved-reliability-and-support-for-app-directory) 改進可靠性與性能。
*   [**內建模組編譯:**](#built-in-module-transpilation-stable) 將 `next-transpile-modules` 的功能整合到核心中。
*   [**Edge 運行時(穩定版):**](#a-light-nodejs-runtime-for-the-edge-now-stable-for-api-routes) 專為 Edge 設計的輕量級 Node.js 運行時。
*   [**Turbopack 更新:**](#turbopack-improvements) 支援 Tailwind CSS、`next/image`、`@next/font` 等更多功能。
*   [**Middleware 改進:**](#nextjs-advanced-middleware) 返回響應並設置請求標頭。
*   [**SWC 導入解析:**](#import-resolution-for-smaller-bundles) 使用 Barrel 文件時生成更小的 JavaScript 打包文件。
*   [內存使用改進、新模板等!](#other-improvements)

今天更新，請運行:

終端機

```
npm i next@latest react@latest react-dom@latest eslint-config-next@latest
```

改進 `app` 目錄的可靠性與支援 [](#improved-reliability-and-support-for-app-directory)
-----------------------------------------------------------------------------------------------------------

在 Next.js 13 中，我們宣布推出新的 `app` 目錄(測試版)。這個新的路由和數據獲取系統可以與您現有的 `pages` 目錄一起逐步採用。

`app` 目錄提供許多好處，包括增强的佈局、元件、測試和樣式的本地化放置、元件級數據獲取等。感謝您的反饋和早期測試，我們對 `app` 目錄的可靠性進行了幾項改進:

*   **無需佈局 Div:** 過去，`app` 目錄在導航時會添加額外的 `<div>` 元素以將佈局滚动到視圖中。在 13.1 中，這些額外元素不再創建。滚动行為將保留。
*   **TypeScript 插件:** 我們開發了一個新的 TypeScript 插件，為頁面和佈局配置選項提供支持建議，將文檔直接導入您的 IDE，並提供有關服務器端元件 (Server Components) 和客戶端元件 (Client Components) 的有益使用提示 (例如，防止在服務器端元件中使用 `useState`)。[了解更多](https://www.youtube.com/watch?v=pqMqn9fKEf8)。
*   **可靠性改進:** 我們修復了多個錯誤，包括改進 CSS 模組支持、正確去重 `cache()` 和 `fetch()` 以處理佈局和頁面，記憶體遺漏等。
*   **更少的客戶端 JavaScript:** `app` 目錄現在比 `pages` 目錄少包含 `9.3 kB` 的客戶端 JavaScript。無論您在應用程序中添加 1 還是在 1000 個服務器端元件，此基準都不會增加。React 運行時暫時稍大一些，增加是由 React 服務器端元件運行時引起的，它處理 Next.js 之前處理的機制。我們正在努力進一步減少這一點。

| | `pages/` | `app/` | delta |
| --- | --- | --- | --- |
| 總首次加載 JS | `基準值` | `-9.3 kB` | `12.1%` 更小 |
| Next.js 運行時 | `基準值` | `-12 kB` | `56.8%` 更小 |
| React 運行時 | `基準值` | `+2.7 kB` | `5.2%` 更大 |

我們對繼續為 `app` 目錄的穩定性取得進展感到興奮。`app` 目錄的 [測試版文檔](https://beta.nextjs.org/) 已經有數百項更新 [基於您的反饋](https://vercel.com/blog/using-vercel-comments-to-improve-the-next-js-13-documentation)。

內建模組編譯(穩定版)[](#built-in-module-transpilation-stable)
-------------------------------------------------------------------------------

現在您可以標記來自本地包(如 monorepos)或來自外部依賴(`node_modules`)的依賴項進行編譯和打包。這項內建支持取代了流行的 `next-transpile-modules` 包。

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ['@acme/ui', 'lodash-es'],
};
 
module.exports = nextConfig;
```

我們感謝 Pierre de la Martinière([@martpie](https://twitter.com/martpie_)) 在該包方面的工作以及他們在確保內建支持滿足社區需求方面提供的協助。

用於更小打包的導入解析 [](#import-resolution-for-smaller-bundles)
-------------------------------------------------------------------------------

許多流行的 npm 包使用「預覽檔 (barrel files)」來提供一個重新導出其他模組的單文件。例如:

@acme/ui/index.ts

```
export { default as Button } from './dist/Button';
export { default as Slider } from './dist/Slider';
export { default as Dropdown } from './dist/Dropdown';
```

這允許包的消費者在一行中使用命名導出:

```
import { Button, Slider, Dropdown } from '@acme/ui';
```

雖然打包工具理解這些 Barrel 文件並可以刪除未使用的重新導出(稱為「死代碼消除」)，但此過程涉及解析/編譯所有重新導出的文件。在某些已發布的庫中，一些 npm 包提供的 Barrel 文件具有數千個重新導出的模組，這會減慢編譯速度。這些庫推薦 `babel-plugin-transform-imports` 以避免此問題，但對於使用 SWC 的開發人員來說，之前沒有支持。我們已在 Next.js 中添加了一個新的內建 SWC 轉換器 `modularizeImports`。

這項新設定啟用了 SWC 轉換器，根據定義的模式更改您的導入語句。例如，上述使用三個組件的代碼將自動轉換為使用直接導入，無需開發人員手動編寫此代碼:

```
// Before (with barrel file)
import { Button, Slider, Dropdown } from '@acme/ui';
 
// After (with modularized imports from plugin)
import Button from '@acme/ui/dist/Button';
import Slider from '@acme/ui/dist/Slider';
import Dropdown from '@acme/ui/dist/Dropdown';
```

透過 `next.config.js` 中的 `modularizeImports` 選項即可實現此轉換:

next.config.js

```
module.exports = {
  modularizeImports: {
    '@acme/ui': {
      transform: '@acme/ui/dist/{{member}}',
    },
  },
};
```

結合 `@mui/icons-material` 或 `lodash` 使用此轉換器可跳過未使用文件的編譯。[了解更多](/docs/architecture/nextjs-compiler#modularize-imports)。

[查看演示](https://twitter.com/nextjs/status/1610288783311732737) 看看其運作方式。

Edge 的輕量 Node.js 運行時，现在 API 路由穩定 [](#a-light-nodejs-runtime-for-the-edge-now-stable-for-api-routes)
---------------------------------------------------------------------------------------------------------------------------------

Next.js 內部的 Edge Runtime 使用 Node.js API 的嚴格子集(如 `Request`, `Response` 等)，這些與 Vercel 等 Edge 計算平台或本地部署兼容。這些 API 在所有地方運行，包括瀏覽器中，讓開發人員只需學習一次即可在各地編寫代碼。

pages/api/hello.ts

```
// "experimental-" prefix is no longer needed
export const config = {
  runtime: 'edge',
};
 
export default function handler(req: Request) {
  return new Response('Hello World');
}
```

Next.js Middleware 已經預設使用此輕量 Edge Runtime 以獲得更好的性能。由於 Middleware 可以在您的應用程序中的每個請求之前運行，擁有輕量級運行時對於確保低延遲至關重要。在 Next.js 12.2 中，我們添加了可選為 [API Routes](https://nextjs.org/blog/next-12-2#edge-api-routes-experimental) 使用此運行時的能力。

在 13.1 中，Next.js 內部的 Edge Runtime **现在已穩定** 支持 API 路由。在本地部署時，使用 Edge Runtime 的 Middleware 和 API 路由將預設作為單區域工作負載運行，作為 `next start` 的一部分。在 Vercel 上，Next.js Middleware 和 API 路由 [使用 Vercel Edge Functions 全球部署](https://vercel.com/docs/concepts/next.js/overview) 以實現最低的延遲。Vercel Edge Functions 也 [現在一般可用](https://vercel.com/blog/edge-functions-generally-available)。

Turbopack 改進 [](#turbopack-improvements)
-------------------------------------------------

在隨 Next.js 13 發布 [Turbopack](https://nextjs.org/docs/app/api-reference/turbopack) alpha 版後，我們一直專注於提高可靠性、添加最受歡迎的功能支持，並制定插件和使用於其他框架的計劃。

自 Next.js 13.0.0 起，Turbopack:

*   支持 PostCSS，包括 Tailwind CSS
*   支持 `next/image`
*   支持 `@next/font` (Google Fonts)
*   支持從動態 `import()` 語句加載 CSS
*   支持 CSS 源映射 (感謝 `@ahabhgk` 的 [貢獻](https://github.com/vercel/turbo/pull/2839))
*   改進了 `next dev` 錯誤覆蓋中的錯誤處理
*   改進記憶體使用
*   改進 CSS 模組支持
*   改進 HMR 更新的分塊算法
*   改進 HMR 源映射的可靠性

我們感謝 Evan You 和 Vite 社區的反饋和貢獻，以確保 Turbopack 基準測試盡可能準確。我們已與 Vite 團隊合作驗證最新的 Turbopack 基準測試，並對我們的測試方法進行了多項改進。

作為此合作的結果，我們現在使用更準確的指標，其中包括 React 更新機制所花費的時間。我們在 Turbopack 中以及 Next.js 13.1 on webpack 中將 React Fast Refresh 時間改進了 30 毫秒。我們還添加了一個新指標，使用 Vite 與 SWC 進行比較，顯示出比使用默認 Vite 與 Babel 更好的性能。查看 [更新後的基準測試](https://nextjs.org/docs/app/api-reference/turbopack) 或了解 [測試方法](https://nextjs.org/docs/app/api-reference/turbopack)。

今天就在 Next.js 13 中試用 Turbopack alpha 版本，使用 `next dev --turbo`。如果您有任何反饋，請在 [GitHub Discussion](https://github.com/vercel/turbo/discussions/2312) 中告訴我們。

Next.js 高級 Middleware[](#nextjs-advanced-middleware)
----------------------------------------------------------

感謝您的反饋，我們正在讓 Next.js Middleware 比以往更強大。在 13.1 中，您現在可以從 Middleware 返回響應，並設置請求標頭。

這些 API 改進為您提供了強大的新靈活性，以自訂 Next.js 路由生命周期的每一個部分。`next.config.js` 內的 `experimental.allowMiddlewareResponseBody` 配置項不再是必需的。

您現在可以更輕鬆地設置請求標頭，並直接響應，無需 `rewrite` 或 `redirect`:

middleware.ts

```
import { NextResponse } from 'next/server';
 
export function middleware(request: Request) {
  // Check if a user has access...
  if (!isAuthorized(request)) {
    return NextResponse.json({ message: 'Unauthorized' });
  }
 
  // Add a new header, this will change the incoming request headers
  // that you can read in getServerSideProps and API routes
  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-version', '13.1');
 
  return NextResponse.next({
    request: {
      // Apply new request headers
      headers: requestHeaders,
    },
  });
}
```

了解有關 [Next.js 高級 Middleware](/docs/pages/building-your-application/routing/middleware) 的更多信息。

其他改進 [](#other-improvements)
-----------------------------------------

*   `@next/font` 現在支持在同一字體聲明中添加多個字體粗細和樣式。[了解更多](/docs/pages/building-your-application/optimizing/fonts#google-fonts)。
*   `next/dynamic` 現在使用 React 原始物件 `lazy()` 和 `<Suspense>`。之前的 `suspense` 選項不再是必需的。通過這些更改，`next/dynamic` 現在與 `app` 目錄兼容。
*   `create-next-app` 已更新設計，現在默認包含 [`@next/font`](/docs/pages/building-your-application/optimizing/fonts) 以實現字體的自動本地部署且零佈局偏移。請嘗試使用 `npx create-next-app@latest` 或 [部署模板](https://vercel.com/templates/next.js/nextjs-boilerplate)。
*   我們對 [App 目錄沙盒](https://app-dir.vercel.app/) 進行了多項改進，展示了 Next.js 13 中 `app` 目錄 (測試版) 的一些最新功能和約定。[部署您自己的](https://vercel.com/templates/next.js/app-directory)。
*   我們創建了一個 [高性能圖庫模板](https://vercel.com/templates/next.js/image-gallery-starter)，其中包括圖像佔位符、延遲加載、自動優化、鍵盤支持等。[部署您自己的](https://vercel.com/templates/next.js/image-gallery-starter)。
*   我們創建了一個資源用於了解如何 [遷移大型開源 React 和 Express.js 應用程序至 Next.js](https://vercel.com/blog/migrating-a-large-open-source-react-application-to-next-js-and-vercel)，包括詳細的逐步引導和鏈接到特定的提交。
