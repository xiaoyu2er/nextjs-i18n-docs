Next.js 11.1 改善了整個堆疊的建置效能，包括：

*   [**安全修補**](#security-patch): 修復潛在開重定向的重要更新。
*   [**ES 模組 (ES Modules) 支援**](#es-modules-support): 透過實驗性旗標即可啟用。
*   [**導入基於 Rust 的工具 (SWC)**](#adopting-rust-based-swc): SWC 整合工具，取代 JS 工具 (Babel 和 Terser)。
*   [**更快的資料獲取**](#builds--data-fetching): 預渲染時搭配 HTTP `keep-alive` 使資料獲取速度快 2 倍。
*   [**更快的 Source Maps**](#source-maps): 使用 Source Maps 時，建置速度快 70%，記憶體使用量減少 67%。
*   [**ESLint 整合改進**](#eslint-improvements): 更友善的預設值和拼字檢查。
*   [**`next/image` 改進**](#nextimage-improvements): 可選 Sharp 使用、對 `next export` 的支援更好。

透過執行 `npm i next@latest` 立即更新。

安全修補 [](#security-patch)
---------------------------------

Next.js 團隊與安全研究人員和審計人員合作，以防止漏洞。我們感謝 Robinhood 的 Gabriel Benmergui 他們針對 `pages/_error.js` 發現了開重定向問題並進行了負責任的揭露。

報告的問題並未直接傷害使用者，但可能會透過從信任的網域名稱重新導向到攻擊者的網域名稱來允許網路釣魚攻擊。我們已在 Next.js 11.1 中發布修補程式以防止發生此開重定向，以及 [安全回歸測試](https://github.com/vercel/next.js/blob/canary/test/integration/repeated-slashes/test/index.test.js)。

要了解更多詳情，請閱讀 [公共 CVE](https://github.com/vercel/next.js/security/advisories/GHSA-vxf5-wxwp-m7g9)。我們建議升級到 Next.js 最新版本，以改善應用程式的整體安全性。未來關於報告的負責任揭露，請發送電子郵件至 `security@vercel.com`。

**注意：** 託管在 [Vercel](https://vercel.com) 上的 Next.js 應用程式**不受此漏洞影響**（因此，在 Vercel 上執行的 Next.js 應用程式**不需要採取任何操作**。

ES 模組 (ES Modules) 支援 [](#es-modules-support)
-----------------------------------------

我們正在 Next.js 中擴大 [ES Modules](https://nodejs.org/docs/latest/api/esm.html) 支援，將其作為輸入模組和輸出目標。從 Next.js 11.1 開始，您現在可以透過實驗性旗標使用 ES Modules 匯入 npm 套件（例如在其 `package.json` 中使用 `"type": "module"`）。

next.config.js

```
module.exports = {
  // Prefer loading of ES Modules over CommonJS
  experimental: { esmExternals: true },
};
```

ES Modules 支援包含向下相容性，以支援 [CommonJS](https://nodejs.org/docs/latest/api/modules.html) 的先前匯入行為。在 Next.js 12 中，`esmExternals: true` 將成為預設值。我們建議您嘗試新功能，如果您有任何改進建議，請在 [GitHub 討論版留言](https://github.com/vercel/next.js/discussions/27876)。

導入基於 Rust 的 SWC [](#adopting-rust-based-swc)
---------------------------------------------------

我們正在整合 [SWC](https://swc.rs/)，這是一款用 Rust 撰寫的超級快速的 JavaScript 和 TypeScript 編譯器，將取代 Next.js 中使用的兩個工具列：Babel 用於個別檔案，Terser 用於輸出組件的壓縮。

作為以 SWC 取代 Babel 的一部分，我們將 Next.js 使用的所有自訂程式碼轉換移植到用 Rust 撰寫的 SWC 轉換，以最大化效能。例如，將 `getStaticProps`、`getStaticPaths` 和 `getServerSideProps` 內部未使用的程式碼進行樹狀搖晃 (Tree Shaking)。

作為取代 Terser 的一部分，我們正在確保 SWC 壓縮器的輸出與 Terser 類似，同時大幅提升壓縮的效能和平行化。

在早期測試中，使用 Babel 的先前程式碼轉換時間從 **~500ms 降至 ~10ms**，使用 SWC 的 Terser 程式碼壓縮時間從 **~250ms 降至 ~30ms**。整體而言，這導致了 **Build 速度快兩倍**。

我們很高興宣佈 [SWC](https://swc.rs/) 的創立者 [DongYoon Kang](https://twitter.com/kdy1dev) 和 [Parcel](https://parceljs.org/) 的貢獻者 [Maia Teegarden](https://twitter.com/padmaia) 已加入 Vercel 的 Next.js 團隊，致力於改進 `next dev` 和 `next build` 的效能。當 SWC 採用工作穩定後，我們將在下一個版本公布更多結果。

效能改進 [](#improved-performance)
---------------------------------------------

### 建置與資料獲取 [](#builds--data-fetching)

使用 `next build` 並發出大量 HTTP 請求時，我們已平均**將效能提高約 2 倍**。例如，如果您使用 `getStaticProps` 和 `getStaticPaths` 從 Headless CMS 獲取內容，您應該會看到明顯更快的建置。

Next.js 自動 polyfill [node-fetch](/docs/architecture/supported-browsers#polyfills) 並預設啟用 [HTTP Keep-Alive](https://en.wikipedia.org/wiki/HTTP_persistent_connection)。根據 [外部基準測試](https://github.com/Ethan-Arrowood/undici-fetch/blob/main/benchmarks.md#fetch)，這應能使預渲染**速度快約 2 倍**。

若要停用特定 `fetch()` 呼叫的 HTTP Keep-Alive，您可以新增 agent 選項：

```
import { Agent } from 'https';
const url = '<https://example.com>';
const agent = new Agent({ keepAlive: false });
fetch(url, { agent });
```

若要全域覆蓋所有 `fetch()` 呼叫，您可以使用 `next.config.js`：

next.config.js

```
module.exports = {
  httpAgentOptions: {
    keepAlive: false,
  },
};
```

### Source Maps [](#source-maps)

在 Next.js 應用程式中包含瀏覽器 Source Maps 現在性能成本減少約 70%，記憶體成本減少約 67%，這是因為 webpack 資源和 Source Maps 處理方面的優化。

這僅影響在其 `next.config.js` 檔案中具有 `productionBrowserSourceMaps: true` 的 Next.js 應用程式。在 Next.js 11.1 中，啟用 Source Maps 時建置時間僅增加 11%。

我們也與 Sentry 合作，透過 [Sentry Next.js 外掛](https://docs.sentry.io/platforms/javascript/guides/nextjs/) 增加上傳 Source Maps 的 [效能](https://github.com/vercel/next.js/issues/26588#issuecomment-894329188)。

ESLint 改進 [](#eslint-improvements)
-------------------------------------------

在 Next.js 11 中，我們透過 `next lint` 引入了內建 [ESLint 支援](/docs/pages/building-your-application/configuring/eslint)。自首次發布以來，我們繼續增加規則以協助開發人員修正應用程式中的常見錯誤。

*   [aria-props](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/HEAD/docs/rules/aria-props.md?rgh-link-date=2021-06-04T02%3A10%3A36Z)
*   [aria-proptypes](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/HEAD/docs/rules/aria-proptypes.md?rgh-link-date=2021-06-04T02%3A10%3A36Z)
*   [aria-unsupported-elements](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/HEAD/docs/rules/aria-unsupported-elements.md?rgh-link-date=2021-06-04T02%3A10%3A36Z)
*   [role-has-required-aria-props](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/HEAD/docs/rules/role-has-required-aria-props.md?rgh-link-date=2021-06-04T02%3A10%3A36Z)
*   [role-supports-aria-props](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/HEAD/docs/rules/role-supports-aria-props.md?rgh-link-date=2021-06-04T02%3A10%3A36Z)

### 常見錯誤 [](#common-typos)

對 `getStaticProps`、`getStaticPaths` 和 `getServerSideProps` 中的常見錯別字進行語意檢查現在預設會發出警告。這將有助於處理因小錯別字導致資料獲取未被呼叫的情況。例如 `getstaticprops` 或 `getStaticprops` 將顯示警告。

特別感謝社群貢獻者 [kaykdm](https://github.com/kaykdm) 建立此規則。

[`next/image` 改進](#nextimage-improvements)
----------------------------------------------------

我們一直在收集關於 `next/image` 和內建 [影像最佳化](/docs/pages/building-your-application/optimizing/images) 的社群回饋，並很高興分享多個在效能、開發者經驗和使用者經驗方面的改進。

### 影像最佳化 [](#image-optimization)

預設情況下，Next.js 使用 WebAssembly 來執行影像最佳化，這透過顯著較小的套件大小且不具安裝後步驟來抵消 Next.js 套件的安裝時間。在 Next.js 11.1 中，您可以選擇安裝 `sharp`，它能優化未快取影像的生成時間，代價是安裝速度較慢。

基於 WebAssembly 的影像優化程式已更新，以支援搭配 Node.js 16 的 ARM 晶片（例如 Apple M1）。

內建影像優化程式現在會根據回應內容的主體自動偵測外部影像的 Content-Type。這使 Next.js 能夠在回應標頭為 `Content-Type: application/octet-stream` 時最佳化託管在 AWS S3 上的影像。

### 開發環境中模糊遮罩佔位元的延遲產生 [](#lazy-generation-of-blur-up-placeholders-in-development)

在 `next dev` 期間，使用 `placeholder="blur"` 的 [靜態影像匯入](/docs/pages/building-your-application/optimizing/images#image-imports) 現在會自動延遲產生，提高了具有許多靜態影像匯入的應用程式的開發伺服器啟動時間：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import Image from 'next/image';
import author from '../public/me.png';
 
export default function Home() {
  return (
    // The placeholder for this image is lazy-generated during development
    <Image src={author} alt="Picture of the author" placeholder="blur" />
  );
}
```

### 其他影像改進 [](#other-image-improvements)

*   **已載入的影像不再延遲載入**：若影像已透過用戶端導航或在頁面其他點載入，Next.js 現在會自動跳過延遲載入以避免顯示影像前的快速閃爍。
*   **支援 `next export` 的自訂影像載入器**： `next/image` 現在支援使用 `next export` 和任何 [第三方影像最佳化服務](/docs/pages/building-your-application/optimizing/images#loader)。如果您打算向 `next/image` 提供 [自訂 `loader` 屬性](/docs/pages/api-reference/components/image#loader)，您可以在 `next.config.js` 中設定 `images.loader: "custom"`。
*   **影像載入完成時的新事件**：基於使用者回饋，我們已在 `next/image` 上新增屬性 [`onLoadingComplete`](/docs/pages/api-reference/components/image#onloadingcomplete)。這允許在影像完全載入後註冊回呼。
*   **預設影像快取 TTL（生存時間）的設定**：您現在可以在 `next.config.js` 中設定 [`images.minimumCacheTTL`](/docs/pages/building-your-application/optimizing/images#minimumcachettl) 以變更最佳化影像的預設快取 TTL。在可能的情況下，我們建議使用 [靜態影像 `import`](/docs/pages/building-your-application/optimizing/images#image-imports)，因為它們使用最大 TTL，因為影像內容雜湊值在 URL 中。
