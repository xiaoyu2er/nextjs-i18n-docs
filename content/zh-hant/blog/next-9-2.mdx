我們很開心今天介紹 Next.js 9.2，特色包括：

*   **[全局樣式表的內建 CSS 支援 (Built-In CSS Support for Global Stylesheets)](#built-in-css-support-for-global-stylesheets)**: 應用程式現可直接匯入 `.css` 檔案作為全局樣式表。
*   **[元件級樣式的內建 CSS 模組支援 (Built-In CSS Module Support for Component-Level Styles)](#built-in-css-module-support-for-component-level-styles)**: 利用 `.module.css` 慣例，本地範圍的 CSS 可以被匯入並在應用程式的任何地方使用。
*   **[程式碼分割策略改善 (Improved Code-Splitting Strategy)](#improved-code-splitting-strategy)**: Google Chrome 團隊極大程度地優化了 Next.js 的程式碼分割策略，結果導致客戶端束包 (client-side bundles) 明顯更小。此外，他們最大化了 [HTTP/2](https://developers.google.com/web/fundamentals/performance/http2) 的利用率，以減輕 HTTP/1.1 效能損耗的同時改善頁面載入速度。
*   **[捕獲全域動態路由 (Catch-All Dynamic Routes)](#catch-all-dynamic-routes)**: Next.js 的動態路由現支援捕獲全域路由，支援各種新的使用案例，例如 CMS 驅動的網站。

這些益處皆為非破壞性且完全後向相容。更新時只需執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

### 全局樣式表的內建 CSS 支援 [](#built-in-css-support-for-global-stylesheets)

Next.js 5 透過稱為 `next-css` 的自訂外掛引入了匯入 CSS 的支援，這擴充了 Next.js 的行為。

隨著時間推移，我們收到了來自 Next.js 公司和用戶的一致反饋，提到他們往往會將 `next-css` 加入其應用程式中。

此外，`next-css` 在匯入 CSS 時缺少了一些限制。例如，您可以在專案的每個檔案中匯入 CSS 檔案，然而此匯入的 CSS 檔案將對整個應用程式是全局的。

為了改善開發者體驗並解決這些不一致之處，我們開始著手在 Next.js 預設中引入 CSS 匯入支援。

我們很榮幸宣布 Next.js 現已內建支援匯入樣式表到您的應用程式。

要在應用程式中使用 CSS 匯入，請在 `pages/_app.js` 內匯入 CSS 檔案。

例如，考慮您的專案根目錄中名為 `styles.css` 的以下樣式表：

```
body {
  padding: 20px 20px 60px;
  margin: 0;
}
```

如果尚不存在，請建立 [`pages/_app.js` 檔案](/docs/pages/building-your-application/routing/custom-app)。

然後，匯入 `styles.css` 檔案：

pages/_app.js

```
import '../styles.css';
 
// This default export is required in a new `pages/_app.js` file.
export default function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
```

由於樣式表本質上是全域的，因此必須在 [自訂 `<App>` 組件](/docs/pages/building-your-application/routing/custom-app) 中匯入。這對於避免全域樣式的類別名稱和排序衝突是必需的。

在開發中，這樣表達樣式表允許您在編輯它們時自動更新頁面上的樣式。

在生產環境中，所有 CSS 檔案將自動連接並組合成單一最小化的 `.css` 檔案。此 CSS 檔案將透過 `<link>` 標籤載入，並自動注入到 Next.js 生成的預設 HTML 標記中。

此新功能完全後向相容。如果您正在使用 `@zeit/next-css` 或其他與 CSS 相關的外掛，則功能會被停用以避免衝突。

如果您目前正在使用 `@zeit/next-css`，我們建議從您的 `next.config.js` 和 `package.json` 中移除該外掛，從而於升級時轉向內建 CSS 支援。

### 元件級樣式的內建 CSS 模組支援 [](#built-in-css-module-support-for-component-level-styles)

Next.js 現支援 [CSS 模組 (CSS Modules)](https://github.com/css-modules/css-modules) 使用 `[name].module.css` 檔案命名慣例。

與之前在 Next.js 5 中透過 `next-css` 取得的支援不同，[全局 CSS](#built-in-css-support-for-global-stylesheets) 和 CSS 模组現在可以**並存** — `next-css` 要求應用程式中所有的 `.css` 檔案必須作為全局或本地方式處理，但不能兩者皆是。

CSS 模组透過自動建立唯一的類別名稱，將 CSS 限定在本地範圍。這允許您在不同檔案中使用相同的 CSS 類別名稱而無需擔心衝突。

此行為使 CSS 模組成為包含元件級 CSS 的理想方式。CSS 模組檔案**可在您的應用程式中任何地方匯入**。

例如，考慮 `components/` 資料夾中可複用的 `Button` 元件：

首先，建立 `components/Button.module.css` 並包含以下內容：

```
/*
You do not need to worry about .error {} colliding with any other `.css` or
`.module.css` files!
*/
.error {
  color: white;
  background-color: red;
}
```

然後，建立 `components/Button.js`，匯入和使用上方 CSS 檔案：

components/Button.js

```
import styles from './Button.module.css';
 
export function Button() {
  return (
    <button
      type="button"
      // Note how the "error" class is accessed as a property on the imported
      // `styles` object.
      className={styles.error}
    >
      Destroy
    </button>
  );
}
```

CSS 模组是*可選的*功能，僅對 `.module.css` 副檔名的檔案啟用。一般的 `<link>` 樣式表](/docs/pages/api-reference/components/head) 和 [全局 CSS 檔案](#built-in-css-support-for-global-stylesheets) 仍然獲得支援。

在生產環境中，所有 CSS 模組檔案將自動連接成 **許多壓縮和拆分的 `.css` 檔案**。這些 `.css` 檔案代表應用程式中的熱執行路徑，確保應用程式的每頁載入最小數量的 CSS 以供塗裝。

如上所述，此新功能完全後向相容。如果您正在使用 `@zeit/next-css` 或其他與 CSS 相關的外掛，則功能會被停用以避免衝突。

如果您目前正在使用 `@zeit/next-css`，我們建議從您的 `next.config.js` 和 `package.json` 中移除該外掛，從而轉向內建 CSS 支援。

### 程式碼分割策略優化 [](#improved-code-splitting-strategy)

Next.js 9.2 之前的版本具有一組固定需要的 JavaScript 束包，用於載入和使頁面互動：

*   頁面的 JavaScript 檔案
*   包含通用 JavaScript 的檔案
*   Next.js 客戶端執行階段束包
*   Webpack 客戶端執行階段束包
*   動態匯入 (透過 `next/dynamic` 加入，當使用時)

要使頁面互動，這些束包都必須載入，因為它們依賴互相啟動瀏覽器中的 React。

由於所有這些束包都是應用程式變得互動所必需的，所以重要的是它們盡可能優化。實際上，這意味著不要從應用程式的其他部分下載過多的程式碼。

為此，Next.js 使用了一個 `commons` 束包，其中包含頁面之間的通用 JavaScript。舊束包分割策略計算生成 `commons` 的方式是使用頻率高度的啟發式算法。如果一个模組在所有頁面中使用超過 50%，它將被標記為通用模組。否則，它會被編譯到頁面的 JavaScript 檔案中。

然而，應用程式可以包含許多不同类型的頁面。例如，行銷頁面、部落格和控制台。如果行銷頁面的數量與其它頁面類型相比較多，commons 計算的結果將是優化高度集中在行銷頁面上。

我們的目標是在一個應用程式中針對所有頁面類型進行優化。

[Alex Castle](https://twitter.com/atcastle) 提出了一種新的分塊 (creation of separate JavaScript files) 方法，允許使用多個檔案進行最佳化的 commons 分塊，即使涉及許多頁面類型也是如此。

今天，我們很高興宣布此新的分塊預設在 Next.js 9.2 中啟用。我們想向 [Google Chrome 團隊](/blog/next-9#google-chrome-collaboration) 和 [Alex Castle](https://twitter.com/atcastle) 表達深深的謝意，他們貢獻了此變更。此變更反映了數週的研究、實驗室測試、實際測試和實施的累積努力。

新的分塊實作利用 [HTTP/2](https://developers.google.com/web/fundamentals/performance/http2) 來傳遞更多數量但較小尺寸的區塊。

在新的啟發式算法下，會以下列方式建立區塊：

*   每個頁面的最小區塊。
*   包含 React、ReactDOM、React 計畫程表等的框架區塊。
*   任何超過 160kb (壓縮前/minify/gzip) 的 `node_module` 依賴項的函式庫區塊。
*   用於跨頁面程式碼的通用區塊。
*   盡可能多的共享區塊 (由 2 個或多個頁面使用)，最佳化整體應用程式大小和初始載入速度。
*   Next.js 的客戶端執行階段。
*   Webpack 執行階段。

讓我們來看看這在真實世界的應用程式中意味著什麼：

早期採用業界合作夥伴 [Barnebys®](https://www.barnebys.com/) 看到整體應用程式大小減少了 23%。

此外，他們最大的 JS 束包減少了 30% — 從 605kB 減至 425kB — 無需更改程式碼。

另一個業界合作夥伴 [SumUp®](https://sumup.com/) 看到他們最大的 JS 束包減少了 70% — 從 395kB 減至 122kB — 無需更改程式碼。

|  | 之前 | 之後 | 差異 |

新的分塊行為不僅減少了您的整體和初始載入大小，還減少了後續的客戶端導航。[Barnebys®](https://www.barnebys.com/) 在六 (6) 次頁面導航後，看到的 JavaScript 載入量減少了 87%：

### 捕獲全域動態路由 [](#catch-all-dynamic-routes)

隨著 Next.js 9 的發布，我們引入了 [動態路由分割段](https://nextjs.org/blog/next-9#dynamic-route-segments)，目標是簡化 Next.js 中的動態分割段，而無需自訂伺服器。此功能已被 Next.js 用戶廣泛採用。

動態路由分割段功能仍有一些未涵蓋的情況。

其中一種情況是捕獲全域路由。例如，將像 `/post/**` 這樣的萬用字元路由為頁面。這對於具有由內容來源 (如 CMS) 定義的嵌套結構時特別有用。

您現在可以使用 `[...name]` 語法建立捕獲全域動態路由。

例如，`pages/post/[...slug].js` 將匹配 `/post/a`、`/post/a/b`、`/post/a/b/c`，以此類推。

`slug` 將作為個別路徑部分的陣列提供給 router query 物件。所以，對於路徑 `/post/foo/bar`，query 物件將為 `{ slug: ['foo', 'bar'] }`。
