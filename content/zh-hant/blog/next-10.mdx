### 內建圖像元件與自動圖像最佳化 [](#built-in-image-component-and-automatic-image-optimization)

Next.js 的目標在於改善兩件事：開發者體驗與使用者體驗。

今年我們已在開發者體驗和為所有 Next.js 應用程式提升效能方面投入了巨資。我們專注於減少瀏覽器需載入的 JavaScript 數量。

我們引進了**20 多個新功能**，改善效能和開發者體驗。與此同時，Next.js 核心的 JavaScript 大小已**減少 16%**。

一月份，我們與 Google Chrome 團隊合作，引進了一項最佳化的 JavaScript 程式碼分割策略。

例如，Barnebys 應用程序大小**減少了 23%**，Sumup 的最大 JavaScript 打包大小則**減少了 70%**。這些改進是在**不需更改 Next.js 應用程式中任何程式碼**的情況下實現的。

公司只需升級 Next.js 至最新版本，即可自動採用這一新策略。

### 網頁上的圖像 [](#images-on-the-web)

雖然我們專注於減少瀏覽器需載入的 JavaScript 數量的努力已見成效，但網路不僅是 JavaScript：它還包含標記和圖像。

圖像佔據網頁總字節數的 50%。

圖像對「最大內容繪製 (Largest Contentful Paint)」影響很大，因為它們通常在頁面載入時是視覺上最大的元素。最大內容繪製是「核心網頁指標 (Core Web Vital)」之一，Google 將在[很快](https://webmasters.googleblog.com/2020/05/evaluating-page-experience.html)將其用於搜尋排名。

一半的圖像大小超過 1 MB，這意味著它們未針對在網路上顯示進行最佳化。

如今使用者使用手機、平板電腦和筆記型電腦瀏覽網路，但圖像仍然是一體化的。例如：網站載入 2000 x 2000 像素的圖像，但手機僅將其顯示為 100 x 100 像素。

此外，網頁上有 30% 的圖像位於初始視口 (Viewport) 之外，這意味著瀏覽器會在使用者向下滾動頁面之前載入他們看不到的圖像。

圖像通常沒有 `width` 和 `height` 屬性，導致頁面載入時它們會跳動。這會損害「累積佈局偏移 (Cumulative Layout Shift)」核心網頁指標。

99.7% 的網站图像未使用 WebP 等現代圖像格式。

要高效地於網頁上使用圖像，需考量許多層面：大小、重量、延遲載入與現代圖像格式。

開發者必須設定複雜的建構工具鏈以最佳化圖像，然而這些工具通常不包含來自外部數據源的使用者上傳圖像，這使得無法最佳化所有圖像。

這項不可能的開發任務必然導致令人沮喪的終端使用者體驗。

### Next.js 圖像元件 [](#nextjs-image-component)

我們很高興地宣佈我們的解決方案，用於網路上的高效圖像：Next.js 圖像元件與自動圖像最佳化。

在最基本的層面上，Next.js 圖像元件只是 HTML `<img>` 元件的隨插即用替代品，已針對現代網路進行了演進。

```
<img
  src="/profile-picture.jpg"
  width="400"
  height="400"
  alt="Profile Picture"
/>
```

```
import Image from 'next/image';
<Image
  src="/profile-picture.jpg"
  width="400"
  height="400"
  alt="Profile Picture"
/>;
```

Google Chrome 團隊協助建立了這個 React 元件，以將最佳實踐設為預設值來提升頁面效能。

使用 `next/image` 元件時，圖像會自動延遲載入，這意味著只有當使用者接近查看圖像時才會渲染。這可防止載入初始視口之外的 30% 圖像。

強制執行圖像尺寸，使瀏覽器能夠立即渲染圖像所需空間，而非在載入時跳動，從而防止佈局偏移。

雖然 HTML `<img>` 元件上的 `width` 和 `height` 會導致響應式佈局的問題，但使用 `next/image`**則不然**。使用 `next/image` 時，圖像會根據提供的 `width` 和 `height` 自動響應。

開發者可以標記初始視口中的圖像，讓 Next.js 自動預載這些圖像。在初始視口中預載圖像可顯示大幅改善「最大內容繪製 (Largest Contentful Paint)」達 50%。

### 自動圖像最佳化 [](#automatic-image-optimization)

即使與 HTML `<img>` 元件相比有這些改進，仍存在重大問題。2000 x 2000 像素的圖像被發送到位於手機上渲染較小圖形的設備。

透過 Next.js 10 我們也在解決這個問題。`next/image` 元件將透過內建圖像最佳化自動生成較小的尺寸。

內建圖像最佳化會在瀏覽器支援情況下，自動以 WebP 等現代圖像格式提供圖像，這比 JPEG 約小 30%。它還允許 Next.js 自動採用**未來**圖像格式並向支援這些格式的瀏覽器提供。

圖像最佳化適用於任何圖像源。即使圖像來自外部數據源（如 CMS），也會進行最佳化。

Next.js 10 不在建構時最佳化圖像，而是在使用者請求時按需最佳化。與靜態站點生成器和僅靜態解決方案不同，無論發送 10 張圖像還是 1000 萬張圖像，建構時間都不會增加。

### 結論 [](#conclusion)

新的 `next/image` 元件和自動圖像最佳化是強大的新功能，將大幅改善使用者體驗。

`next/image` 元件處理自動延遲載入、關鍵圖像的預載、跨裝置的正確大小，並自動支援現代格式。這些功能適用於來自任何源的圖像。

我們期待看到這些新原語使用者的體驗變得有多快。

更多詳情請參閱 [Next.js 圖像元件與自動圖像最佳化文件](/docs/pages/building-your-application/optimizing/images)。

### 國際化路由 [](#internationalized-routing)

今年，幾家企業和社群成員幫助我們團隊了解國際化的重要性。

例如，我們了解到 72% 的消費者更有可能停留在已翻譯的網站上，55% 的消費者說他們只會在母語的電子商務網站購物。

如果您計劃在其他國家進入市場，將專案本地化對於成功至關重要。

將專案國際化的兩個主要支柱：翻譯和路由。

許多 React 函式庫應用程式準備好翻譯，但它們大多數期望您手動處理路由，且通常只支援一種渲染策略。

這就是為什麼，作為 Next.js 10 的一部分，我們正在發布對國際化路由和語言偵測的內建支援。

此國際化路由的內建支援支援 Next.js 的混合策略，因此您可以選擇按頁面進行靜態生成或伺服器渲染。

Next.js 10 支援兩種最常用的路由策略：子路徑路由和網域路由。

對於兩種策略，您都從 Next.js 配置中設定區域語言 (locales) 開始。

next.config.js

```
module.exports = {
  i18n: {
    locales: ['en', 'nl'],
    defaultLocale: 'en',
  },
};
```

語言環境採用 [UTS 語言環境識別碼](https://www.unicode.org/reports/tr35/tr35-59/tr35.html#Identifiers)，這是定義語言環境的標準化格式。

通常語言環境識別碼由語言、地區和文字系統組成，以連字號分隔：`語言-地區-文字系統`。地區和文字系統為可選項目。例如：

*   `en-US` - 美國英語
*   `nl-NL` - 荷蘭語（荷蘭）
*   `nl` - 荷蘭語（無特定地區）

配置好區域語言後，您可以選擇子路徑或網域路由。

### 子路徑路由 [](#subpath-routing)

子路徑路由將區域語言放入網址。這使所有語言都能位於單一網域上。

例如，您可以在網址中插入區域語言，如 `/nl-nl/blog` 和 `/en/blog`。

### 網域路由 [](#domain-routing)

網域路由允許您將區域語言映射到頂級網域。例如 [`example.nl`](http://example.nl) 可映射到 `nl` 語言，[`example.com`](http://example.com) 可映射到 `en` 語言。

網域路由需要一些額外的配置來知道如何路由網域：

next.config.js

```
module.exports = {
  i18n: {
    locales: ['en', 'nl'],
    domains: [
      {
        domain: 'example.com',
        defaultLocale: 'en',
      },
      {
        domain: 'example.nl',
        defaultLocale: 'nl',
      },
    ],
  },
};
```

### 語言偵測 [](#language-detection)

Next.js 10 在 `/` 路由上具有基於 `Accept-Language` 標頭的內建語言偵測，所有現代瀏覽器都支援。配置的區域語言將與 `Accept-Language` 標頭配對，然後根據配置的策略進行重新導向。

### 搜尋引擎最佳化 [](#search-engine-optimization)

由於 Next.js 知道使用者所訪問頁面的語言，它將自動向 `<html>` 標記添加 `lang` 屬性。

Next.js 不知道頁面的變體，因此您需使用 `next/head` 添加 `hreflang` Meta 標記。您可以在 [Google Webmasters 文件](https://support.google.com/webmasters/answer/189077) 了解更多關於 `hreflang` 的資訊。

### Next.js 國際化的未來 [](#the-future-of-internationalization-in-nextjs)

國際化路由是系列功能中的第一個，將使國際化和本地化您的專案變得更容易。國際化路由允許整合到大多數 React 國際化函式庫中。

要深入了解國際化路由，請參閱 [國際化路由文件](/docs/pages/building-your-application/routing/internationalization)。

### Next.js 效能洞察 [](#nextjs-speed-insights)

在 Vercel，我們知道無法度量就無法修正。

您的訪客對網站效能越來越敏感。超過 50% 的訪客會放弃您的網站 [如果載入時間超過 3 秒鐘](https://www.blog.google/products/admanager/the-need-for-mobile-speed/)。如果您從事電子商務，許多人發現將載入時間改善 1/10 秒會帶來 [1% 的轉換率提升](http://robotics.stanford.edu/~ronnyk/2007IEEEComputerOnlineExperiments.pdf)。

由於效能對您的成功至關重要，我們很自豪地推出 Next.js 效能洞察。這是追蹤真實世界效能指標並將這些洞察反饋到您的開發工作流程中的解決方案。

配合 Next.js 效能洞察：

不再是一次測量，**您現在將持續測量。**

不再於開發裝置上測量，**測量數據將來自訪客實際使用的裝置。**

**Next.js 效能洞察專注於整體畫面，深入了解您的受眾，以及您的應用程式對使用者的表現。**

我們非常堅持收集實測數據，因為效能不佳的原因並不總是顯而易見的。效能衰退可能來自多個方面——第三方腳本和樣式表，或過大或過慢的第一方字體、圖像和影片。

### 核心網頁指標 [](#core-web-vitals)

Google 與 [Web Performance Working Group](https://www.w3.org/webperf/) 合作，確立了一套指標來準確測量使用者如何體驗網站的效能：恰如其名，為 Web Vitals。Web Vitals 是追蹤使用者感知的載入速度、響應性和視覺穩定性的指標——這三者對於網站的整體健康度至關重要！

感知的載入速度可以由「最大內容繪製 (Largest Contentful Paint)」來測量，或者當頁面所有內容都顯示出來時。例如，當我打開購買一雙運動鞋的連結時——在我看到運動鞋、價格和加購按鈕之前經過的時間就是 LCP。

頁面響應性可以用「首次輸入延遲 (First Input Delay)」來測量，它測量使用者必須等待多久才能看到他們與頁面第一次互動的反應。例如，我點擊加到購物車和購物車物品數量增加之間的時間就是 FID。

最後，視覺穩定性可以用「累積佈局偏移 (Cumulative Layout Shift)」來測量，即元素在顯示給使用者後移動了多少。例如，我們都曾經歷過嘗試點擊因圖像延遲載入而移動的按鈕的挫敗感——這就是佈局偏移。

對您的真實使用者圍繞這些 Web Vitals 的持續測量與符合是至關重要的。這是真正了解您的網站如何為使用者表現的唯一方法。您的網站效能可能會根據使用者的裝置和網路條件，或他們如何與頁面互動而有顯著差異。載入個人化內容或廣告的網站也可能經歷從使用者到使用者截然不同的效能。

**模擬測試無法捕捉這些重要信號。**

Next.js 效能洞察允許您捕捉真實世界的洞察，而非合成基準。它支援持續的測量資料，而非依賴偶然的測試，確保這是您開發工作流程的一部分。

Next.js 效能洞察允許你捕捉**真實世界的洞察**，而非合成基準。它支援**持續的測量資料**，而非依賴偶然的測試，**確保這是您開發工作流程的一部分。`

請造訪 [nextjs.org/analytics](https://nextjs.org/analytics) 以了解如何立即在我的應用程式中啟用。

### React 17 支援 [](#react-17-support)

React 17 對 Next.js 沒有破壞性的更改，但需要一些維護更改，例如更新對等相依性。當使用 React 17 時，[新的 JSX 轉換](https://reactjs.org/blog/2020/09/22/introducing-the-new-jsx-transform.html) 會自動啟用，無需配置更改。

要開始使用 React 17，您需要升級 Next.js 和 React：

終端機

```
npm install next@latest react@latest react-dom@latest
```

`getStaticProps` / `getServerSideProps` 快速重新整理 [](#getstaticprops--getserversideprops-fast-refresh)

編輯您的 `getStaticProps` 和 `getServerSideProps` 函式時，Next.js 現在將自動重新執行該函式並套用新資料。這使您可以在無需重新整理頁面的情況下更快地進行迭代。

要了解更多關於 `getStaticProps` 和 `getServerSideProps` 的資訊，您可以閱讀 [數據獲取文件](/docs/pages/building-your-application/data-fetching)。

### MDX 的快速重新整理 [](#fast-refresh-for-mdx)

當透過 `@next/mdx` 使用 Next.js 搭配 MDX 時，現在更改 MDX 內容將利用快速重新整理，確保瀏覽器無需重新整理頁面。

[`@next/mdx` 文件](https://github.com/vercel/next.js/tree/canary/packages/next-mdx) 將指導您如何搭配 Next.js 設定 MDX。

### 從第三方 React 組件匯入 CSS [](#importing-css-from-third-party-react-components)

您現在可以在 React 組件內部匯入第三方 CSS。這允許僅針對單一組件的 CSS 進行程式碼分割。例如，您現在可以使用 `react-datepicker` 函式庫，而無需匯入 `_app.js` 中的 CSS：

components/MyComponent.js

```
import DatePicker from 'react-datepicker';
import 'react-datepicker/dist/react-datepicker.css';
 
function MyComponent() {}
```

您可以參考 [內建 CSS 支援文件](/docs/pages/building-your-application/styling) 以了解更多關於 Next.js 如何處理 CSS 匯入的資訊。

### `href` 的自動解析 [](#automatic-resolving-of-href)

如果您之前使用過動態路由，您可能會遇到一種情況，您必須向 `next/link` 提供 `href` 和 `as` 屬性。看起來如下所示：

```
<Link href="/categories/[slug]" as="/categories/books" />
```

這雖然允許 Next.js 動態參數插值 href，但也造成了問題——開發者如果忘記添加 `as` 或在 `href` 中添加 `as`，會導致頁面過渡未使用客戶端路由。

幾個月前，我們着手為開發者解決此問題，主要目的是改善開發者體驗和最終使用者體驗。我們逐步工作，尋求一種允許 `href` 自動解析的解決方案。

我們興奮地宣佈，作為 Next.js 10 的一部分，您不再需要在大多數用例中使用 `as` 屬性。消除開發者的阻礙，改善最終使用者體驗。

此變更完全向後兼容，如果您目前同時使用 `href` 和 `as`，現有行為將保留。

為了採用自動 `href` 解析，您只需將 `next/link` 的使用更改為僅使用 `href`，並保留您之前在 `as` 屬性中的值。

```
<Link href="/categories/books" />
```

要了解有關 `next/link` 和客戶端路由的更多資訊，您可以參考 [`next/link` 文件](/docs/pages/api-reference/components/link)。

`@next/codemod` CLI [](#nextcodemod-cli)

我們致力於通過廣泛的向後兼容性確保 Next.js 升級盡可能平滑。這一承諾從節制廢棄功能開始，同時用新且更好的解決方案替換它們。此外，我們還為所有 Next.js 功能進行了廣泛的整合測試，包括模擬本地開發的測試。

當 Next.js 中的功能被廢棄並需要大量程式庫更改時，我們的團隊會為其創建 codemod。Codemod 是一種自動代碼轉換，您可以在專案上運行以更新原始碼。

例如：我們发布了 codemod 來更替箭頭函式和匿名函式為命名函式。[此轉換](/docs/pages/building-your-application/upgrading/codemods#name-default-component) 是必要的，否則 React Fast Refresh 不會將函式識別為有效的 React 組件。同樣地，React hooks eslint 規則也不會識別該函式為 React 組件。

透過 Next.js 10，我們发布了新的 Next.js codemods CLI 工具，允許您運行單一命令來更新您的應用程式：`npx @next/codemod <transform> <path>`。

若要了解更多關於 codemods 的資訊，您可以參閱 [Next.js Codemods 文件](/docs/pages/building-your-application/upgrading/codemods)。

`getStaticPaths` 的阻斷式預載 [](#blocking-fallback-for-getstaticpaths)

在 Next.js 9.3 中，我們引進了 `getStaticProps` 和 `getStaticPaths`，以及能夠在 `getStaticPaths` 中返回 `fallback` 屬性。`fallback` 屬性允許在不需完全重新建構的情況下生成額外的靜態頁面，初始時提供靜態 HTML 文件，然後在後續請求中由完全渲染的內容替換。過去幾個月，我們收到了許多公司的反饋，他們希望有類似但略有不同的行為：當使用者首次請求頁面時進行阻斷式預渲染。那初始渲染後，頁面將為後續請求重新使用。

透過 Next.js 10 我們正在解決此情況。

我們很高興地宣佈 `getStaticPaths` 的新 `fallback: 'blocking'` 模式，它啟用阻斷行為，即不向瀏覽器發送靜態預載。而是等待初始請求進行預渲染。

pages/posts/[id].js

```
export function getStaticPaths() {
  return {
    // enables blocking mode for the fallback behavior
    fallback: 'blocking',
  };
}
```

若要了解更多關於增量生成額外靜態頁面的 `fallback` 行為，您可以參考 [`fallback` 文件](/docs/pages/building-your-application/data-fetching#fallback-true)。

`getStaticProps` / `getServerSideProps` 的 `Redirect` 和 `notFound` 支援 [](#redirect-and-notfound-support-for-getstaticprops--getserversideprops)

自引入 `getStaticProps` 和 `getServerSideProps` 以來，我們注意到使用者需要返回重新導向和 404 回應的情況。為了幫助簡化這些情況，我們現在允許 `getStaticProps` 和 `getServerSideProps` 返回兩個新欄位：`notFound` 和 `redirect`。

### `notFound` 支援 [](#notfound-support)

當返回 `notFound` 欄位並設為 `true` 時，預設 404 頁面將以 `404` 狀態碼返回。這可避免使用 SSG 生成額外頁面，並需要手動處理 404 頁面渲染。

pages/posts/[id].js

```
export function getStaticProps() {
  return {
    // returns the default 404 page with a status code of 404
    notFound: true,
  };
}
```

### `redirect` 支援 [](#redirect-support)

現在可以返回重新導向，包含 `destination` 和重新導向是否應為永久，例如 `permanent: true`。可選的 `statusCode` 也可以取代 `permanent` 欄位返回，用於您需要在默認之外使用特定狀態的情況。

pages/posts/[id].js

```
export function getStaticProps() {
  return {
    // returns a redirect to an internal page `/another-page`
    redirect: {
      destination: '/another-page',
      permanent: false,
    },
  };
}
```

pages/posts/[id].js

```
export function getServerSideProps() {
  return {
    // returns a redirect to an external domain `example.com`
    redirect: {
      destination: 'https://example.com',
      permanent: false,
    },
  };
}
```
