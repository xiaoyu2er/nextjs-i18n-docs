今天，我們很高興宣布 Next.js 9.1 支援 `src` 和 `public` 目錄。

**本次發行新功能**

*   **[`src`](#src-directory-support) 目錄支援**：`pages` 目錄現在可以嵌套在 `src` 資料夾下，支援更多種應用程式設定。
*   **[`public`](#public-directory-support) 目錄支援**：定義要掛載在應用程式 URL 根目錄的檔案（例如 `favicon.ico`）。

**本次發行預覽功能**

*   **[內建 CSS 支援](#built-in-css-support)**：應用程式很快就可以匯入全域 CSS，並利用開發環境的熱重新載入、生產環境進階最佳化、編譯和 polyfilling。
*   **[靜態錯誤頁面](#static-error-pages)**：靜態匯出預期的錯誤頁面（如 404），以獲得更好的可用性（CDN）。
*   **[Module / Nomodule](#module--nomodule)**：將較小的 JavaScript 封包發送給在 evergreen 瀏覽器上執行的終端使用者。
*   **[改進的封包分割](#improved-bundle-splitting)**：向終端使用者發送更少的位元組，改善 TTI 和頁面轉換速度。大型程式庫區塊在不同的部署之間也會長期快取。

一如既往，我們致力於確保這些功能皆有後向相容性。進行更新時您需要做的是執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

如果您的應用程式使用的是 9 之前的 Next.js 版本，您可以參考 [升級指南 ](https://github.com/vercel/next.js/blob/canary/UPGRADING.md) 以查看您可能需要升級的變更。

自上次重大發行以來，我們很高興看到像 [TikTok](https://www.tiktok.com/en/)、[Hilton](https://www.hilton.com/en/hilton/)、[Elastic](https://www.elastic.co)、[Realtor](https://www.realtor.com/) 和 [JW Player](https://www.jwplayer.com/) 等公司利用 Next.js 推出產品。查看更多 [展示案例](https://nextjs.org/showcase)！

### `src` 目錄支援 [](#src-directory-support)

Next.js 有一個特殊的 `pages` 目錄，其中每個檔案都變成一個獨立的路由，遵循慣例超過設定（Convention over Configuration）的方法，此目錄必須位於 Next.js 應用程式的根目錄中。

在與使用 Next.js 的公司交流並檢視一些封閉原始碼程式庫後，我們發現開發者希望建立一個 `src` 目錄來存放程式碼，並在裡面包含 `pages` 目錄，這是一種常見的模式。

從 Next.js 9.1 開始，現在可以建立 `src/pages` 目錄，而不是在應用程式根目錄建立 `pages` 目錄。

使用 `src` 目錄是可選的，涵蓋了公司已經設立此標準的情況。

### `public` 目錄支援 [](#public-directory-support)

除了 `pages` 目錄外，Next.js 還有一個其他特殊目錄稱為 `static`，其中檔案會映射到 `/static` 路由。

例如，`static/my-image.png` 將映射為 `/static/my-image.png`。

自從首次發行 Next.js 以來，這個慣例一直運作良好，並沒有特別的問題。

然而，經過一段時間我們發明了 `/static` 無法涵蓋網路應用程式所需的一切。例如，`robots.txt` 必須從網域的根目錄提供。

從 Next.js 9.1 開始，我們正在引入一個名為 `public` 的新目錄。

`public` 目錄中的任何檔案將映射到網域的根目錄。

例如，`public/robots.txt` 將映射為 `/robots.txt`。

由於 `public` 也涵蓋 `static` 目錄的使用案例，我們決定棄用 `static` 目錄，改為建立具有相同功能的 `public/static` 資料夾。

一如既往，我們致力於後向相容，所以 `static` 目錄將繼續運作，並會顯示棄用訊息。

### 內建 CSS 支援 [](#built-in-css-support)

目前，Next.js 內建了一個名為 [styled-jsx](http://github.com/vercel/styled-jsx) 的 CSS-in-JS 解決方案。此解決方案用於樣式設定個別 React 元件的效果良好。

然而，在與公司溝通時，我們發現需要重複使用一些現有的樣式或基於 CSS 的設計系統，這是一個共同的需求。

一般來說，這意味著要加入 `next-css` 插件以支援 CSS 匯入。

我們發現約有 50% 的所有 Next.js 使用者會將此插件加入他們的應用程式。

由於這種廣泛的使用，我們正在新增內建的 CSS 匯入支援，包含開發環境的樣式自動重新載入，以及生產環境最佳化，這些在 `next-css` 插件先前無法達成。

初步實作目前正在生產環境的 Next.js 應用程式中進行測試。

首先將引入全域 CSS 匯入：

pages/_app.js

```
// Global styles can be imported from _app.js
import '../styles/global.css';
import App from 'next/app';
 
export default App;
```

全域 CSS 匯入之後，我們將透過 `.module.css` 副檔名引入 CSS modules 支援：

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
// Scoped styles are imported through .module.css
import styles from '../styles/index.module.css';
 
export default function HomePage() {
  return <h1 className={styles.heading}>Hello World</h1>;
}
```

這將允許我們在使用 CSS 匯入時提供更好得多的開發人員體驗。

您可以閱讀在 [GitHub 上的 RFC](https://github.com/vercel/next.js/issues/8626) 上進行的持續工作。

### 靜態錯誤頁面 [](#static-error-pages)

Next.js 在發生錯誤時有一個特殊頁面，這個頁面在內部稱為 `/_error`。此頁面可以通過建立 `pages/_error.js` 檔案並匯出 React 元件來自定義。

渲染的錯誤通常分為 2 種情況：預期錯誤和未預期的錯誤。

預期錯誤例如 `404` 頁面。

未預期的錯誤例如在 `getInitialProps` 中拋出錯誤或在渲染 React 樹時，會渲染 `500`。

我們計劃為預期錯誤新增 [自動靜態最佳化](https://nextjs.org/blog/next-9#automatic-static-optimization)，因為一般來說，它們不必動態呈現。

如果使用者想要動態呈現這些頁面，可以選擇退出，但預設會是 `404` 為靜態頁面。這減少了使用 `server` 目標時的伺服器載入量，並減少了使用 `serverless` 目標時的成本。

使頁面靜態的另一個好處是，當使用 CDN 時，它可以被自動快取。

### Module / Nomodule [](#module--nomodule)

當在 Next.js 中編寫程式碼時，您通常編寫「現代」JavaScript。您可以使用所有最新的穩定功能，Next.js 將確保這些功能透過使用 Babel 編譯程式碼來進行轉換或編入 polyfill。

此時，許多這些 JavaScript 功能已在大多數瀏覽器中支援。然而，一般來說（包括在 Next.js 中），程式碼是作為單一 JavaScript 封包發布的，此封包在您應用程式支援的所有瀏覽器上運行。

就 Next.js 而言，這意味著將現代 JavaScript 編譯為與 Internet Explorer 11 相容的格式。

例如，目前，Next.js 必須為 async/await 語法提供 polyfill，因為程式碼可能會在 not 支援 async/await 的瀏覽器中執行，這會導致中斷。

為了避免破壞舊版瀏覽器，同時向支援新語法的瀏覽器發送現代 JavaScript，Next.js 將利用 [module/nomodule](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/) 模式。module/nomodule 模式提供了向現代瀏覽器提供現代 JavaScript 的可靠機制，同時允許舊版瀏覽器回退到 polyfilled 的 ES5 程式碼。

此新功能目前正由多個大型 Next.js 應用程式在生產環境中進行測試，以收集實際資料。這些測試的結果看起來很有希望，關於此變更的性能改進的更多細節將很快分享。

### 改進的封包分割 [](#improved-bundle-splitting)

Next.js 目前有多個 JavaScript 封包，它載入以確保頁面具有互動性。最為明顯的是：

*   頁面 JavaScript 封包。
*   包含通用 JavaScript 的檔案。
*   Next.js 客戶端封裝封包。
*   Dynamic imports (generally added through `next/dynamic`).

要確保頁面具有互動性，所有這些封包都必須載入，因為它們都相互依賴，才能啟動瀏覽器中的 React。

由於 React 啟動需要載入這些封包，因此很重要，它們盡可能最佳化，並且不會從應用程式的其他部分過載下載太多程式碼。

出於這個原因，有一個 `commons` 封包，用於保存頁面之間的通用 JavaScript。用於生成 `commons` 的當前封包分割策略的計算是基於比率啟發式的，如果所有頁面中有 50% 的模組被使用，它將被標記為通用模組。

然而，應用程式可能由多個不同部分組成。例如，行銷頁面、博客和儀表板。如果行銷頁面與儀表板相比數量眾多，則 commons 計算將為行銷頁面帶來更优化的結果。

我們的目標是在一個應用程式中同時優化這兩個方面。

Alex Castle 定義了一個分塊（創建單獨 JavaScript 檔案）的新層，允許具有多個檔案的更优化的 commons 分塊，特別是當涉及很多頁面時。

與 module/nomodule 支援類似，改進的封包分割正在由多個大型 Next.js 應用程式在生產環境中進行測試，以收集實際資料。這些測試的結果和關於此變更性能改進的更多細節將很快在單獨的部落格文章中分享。
