正如我們在 [Next.js Conf](https://nextjs.org/conf) 所宣布的，Next.js 12 是我們有史以來最大的版本：

*   [**Rust 編譯器**](#faster-builds-and-fast-refresh-with-rust-compiler)：~3 倍更快的 Fast Refresh 和 ~5 倍更快的構建
*   [**Middleware（測試版）**](#introducing-middleware)：透過程式碼取代配置，為 Next.js 提供完整靈活性
*   [**React 18 支援**](#preparing-for-react-18)：原生的 Next.js API 現已支援，以及 Suspense
*   [**`<Image />` AVIF 支援**](#smaller-images-using-avif)：啟用可獲得 20% 更小的圖片
*   [**機器人感知的 ISR 回退**](#bot-aware-isr-fallback)：為網路爬蟲優化 SEO
*   [**原生 ES 模組支援**](#es-modules-support-and-url-imports)：與標準化的模組系統對齊
*   [**URL 引入（alpha）**](#url-imports)：從任何 URL 引入套件，無需安裝
*   [**React Server Components（alpha）**](#react-server-components)：今天就試試看，包括 SSR 串流

透過執行 `npm i next@latest` 立即更新。

使用 Rust 編譯器的更快構建和 Fast Refresh[](#faster-builds-and-fast-refresh-with-rust-compiler)
-------------------------------------------------------------------------------------------------------

我們希望讓每個 Next.js 應用在生產時構建更快，並在本地開發時獲得即時反饋。Next.js 12 包含了一個全新基於 Rust 的編譯器，利用了原生編譯的優勢。

我們的 Rust 編譯器基於 [**SWC**](http://swc.rs/)，這是一個為下一代快速工具打造的外掛平台。我們優化了打包和編譯，本地獲得 ~**3 倍更快的刷新**，生產獲得~**5 倍更快的構建**。其他改進和功能包括：

![使用新的 Rust 編譯器處理大型 Next.js 代碼庫的結果。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-12/swc.png)

使用新的 Rust 編譯器處理大型 Next.js 代碼庫的結果。

*   **大型代碼庫的進一步速度改進**：我們已驗證 Rust 編譯器適用於世界上一些最大的 Next.js 代碼庫。
*   **性能可觀測性改進**：Next.js 現在在控制台中輸出 Fast Refresh 時間，包括客戶端和伺服器的編譯，以及編譯的模組和文件數量。
*   **底層 webpack 改進**：我們對 webpack 進行了多項改進，包括優化 Fast Refresh 並使按需入口更加可靠。

使用 Rust 編譯的速度**比 Babel 快 17 倍**，並在 Next.js 12 中默認啟用，替換了 JavaScript 和 TypeScript 文件的轉換。這意味著我們必須將 Next.js 中的 Babel 轉換移植到 Rust，包括用於實現 `styled-jsx` 轉換的**基於 Rust 的全新 CSS 解析器**。

新的 Rust 編譯器與舊版本兼容。如果您已有 Babel 配置，將自動禁用。我們計劃不久後移植對 `styled-components`、`emotion` 和 `relay` 等流行庫的解析。如果您使用自訂 Babel 設置，[請分享您的配置](https://github.com/vercel/next.js/discussions/30174)。

您也可以啟用使用 Rust 編譯器進行壓縮，這**比 Terser 快 7 倍**。壓縮功能需要啟用，直到經過徹底驗證，因為它替換了多年的基礎設施。

next.config.js

```
module.exports = {
  swcMinify: true,
};
```

除了聘請 SWC 的創始人 [DongYoon Kang](https://twitter.com/kdy1dev) 和 [Parcel](https://parceljs.org/) 的貢獻者 [Maia Teegarden](https://twitter.com/padmaia)，我們繼續投資 Rust 生態。如果您有 Rust 工作經驗，[請申請加入我們的團隊](https://vercel.com/careers)。

更多資訊，請觀看我們 [Next.js Conf 的演示影片](https://www.youtube.com/watch?v=_WNeAubn92U)。

介紹 Middleware[](#introducing-middleware)
-------------------------------------------------

Middleware 讓您能夠使用程式碼取代配置。這為您在 Next.js 中提供了完整的靈活性，因為您可以在請求完成之前執行程式碼。根據用戶的 incoming 請求，您可以透過重新路由、重定向、添加請求頭，甚至串流 HTML 來修改回應。

![Middleware 讓您在 Next.js 內部擁有完全的靈活性。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-12/middleware.png)

Middleware 讓您在 Next.js 內部擁有完全的靈活性。

Middleware 可用於任何形式的共享一組頁面邏輯，包括：

*   [身份驗證](https://vercel.com/templates/next.js/jwt-authentication)
*   [機器人保護](https://vercel.com/templates/next.js/bot-protection-datadome)
*   [重定向](https://vercel.com/templates/next.js/edge-redirects-upstash)
*   [重新路由](https://vercel.com/templates/next.js/hostname-rewrites)
*   [功能旗標和 A/B 測試](https://vercel.com/templates/next.js/ab-testing-simple)
*   [高級 i18n 路由需求](https://vercel.com/templates/next.js/edge-functions-i18n)
*   和 [更多](https://vercel.com/templates?type=edge-functions)！

Middleware 使用嚴格的運行環境，支援標準 Web APIs，如 **`fetch`**。在 `next start` 中即可使用，也支援 Vercel 等 Edge 平台，其中使用 [Edge Middleware](http://www.vercel.com/edge)。

要在 Next.js 中使用 Middleware，可以創建一個 `pages/_middleware.js` 文件。在這個範例中，我們使用標準 Web API Response ([MDN](https://developer.mozilla.org/docs/Web/API/Response))：

pages/\_middleware.js

```
export function middleware(req, ev) {
  return new Response('Hello, world!');
}
```

更多資訊，請觀看我們 [Next.js Conf 的演示影片](https://www.youtube.com/watch?v=WlP2TB2ORL4) 和 [查看文件說明](/docs/pages/building-your-application/routing/middleware)。

準備 React 18[](#preparing-for-react-18)
-------------------------------------------------

[React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html) 將加入 Suspense、自動更新批次處理、像 `startTransition`，以及支援 `React.lazy` 的全新串流伺服器渲染 API。

我們一直與 Facebook 的 React 團隊密切合作，為 React 18 穩定發行前準備 Next.js。我們正讓這些功能在 Next.js 12 中透過實驗旗標可供測試。

終端機

```
npm install react@alpha react-dom@alpha
```

### 伺服器端串流[](#server-side-streaming)

[React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html) 中的並行功能包括內建的伺服器端 Suspense 和 SSR 串流支援。這允許您使用 HTTP 串流伺服器渲染頁面。這是 Next.js 12 中的實驗性功能，但一旦啟用，SSR 將使用與 Middleware 相同的嚴格運行環境。

要啟用，使用實驗旗標 `concurrentFeatures: true`：

next.config.js

```
module.exports = {
  experimental: {
    concurrentFeatures: true,
  },
};
```

### React Server Components[](#react-server-components)

React Server Components 允許我們在伺服器端渲染一切，包括組件本身。這與伺服器端渲染根本上不同，因為後者是在伺服器上預生成 HTML。使用 Server Components，**不需要任何客戶端 JavaScript**，使頁面渲染更快。這改善了您應用的使用者體驗，將伺服器渲染與客戶端互動的最佳部分結合。

next.config.js

```
module.exports = {
  experimental: {
    concurrentFeatures: true,
    serverComponents: true,
  },
};
```

Next.js 現在允許您以 _組件層級_ 進行資料抓取，全部以 JSX 表達。使用 React Server components，我們可以簡化事物。像 `getServerSideProps` 或 `getStaticProps` 的特殊功能不再需要。這與 React Hooks 模型一致，在組件中並置資料抓取。

您可以將任何 Next.js 頁面重新命名為 `.server.js` 以創建 Server Component，並直接在 Server Components 中引入客戶端組件。這些客戶端組件將水合並變得互動，因此您可以添加功能，比如上票。

我們目前正在 Next.js 中開發伺服器端 Suspense、選取水合和串流渲染，並將在下篇部落格文章中分享進展。

特別感謝我們的合作夥伴 [Kara Erickson](https://twitter.com/karaforthewin) 和 [Gerald Monaco](https://twitter.com/devknoll)，來自 [Google Aurora](https://web.dev/introducing-aurora/) 團隊在 React 18 和 Server Components 方面的工作。

更多資訊，請觀看我們 [Next.js Conf 的演示影片](https://www.youtube.com/watch?v=WlP2TB2ORL4) 和 [查看文件說明](https://vercel.link/react-18)。

ES 模組支援和 URL 引入[](#es-modules-support-and-url-imports)
-------------------------------------------------------------------------

ES 模組為 JavaScript 帶來官方標準化的模組系統。所有主要瀏覽器以及 Node.js 都支援。

這個標準推動了 Web 生態向前發展，通過實現更小的套件大小和 JavaScript 打包，最終導致更好的使用者體驗。隨著 JavaScript 生態從 CommonJS（舊標準）過渡到 ES 模組，我們致力於幫助開發者逐步採用這些改進，無需不必要的破壞性變更。

從 [Next.js 11.1](https://nextjs.org/blog/next-11-1) 開始，我們添加了實驗性支援 [ES 模組](https://nodejs.org/docs/latest/api/esm.html) 優先於 CommonJS 模組。在 Next.js 12 中，這現在是默認的。引入只提供 CommonJS 的 NPM 模組仍然支援。

URL 引入[](#url-imports)
---------------------------

Next.js 12 包括實驗性支援透過 URL 引入 ES 模組，無需安裝或單獨的構建步驟。

URL 引入允許您透過 URL 直接使用 _任何_ 套件。這使 Next.js 能夠將远程 HTTP(S) 資源處理與本地依賴完全相同。

如果檢測到 URL 引入，Next.js 將生成 `next.lock` 文件來跟踪远程資源。URL 引入在本地緩存，確保您可以離線工作。Next.js 支援客戶端和服务器端的 URL 引入。

若要參與，請在 `next.config.js` 中加入允許的 URL 前綴：

next.config.js

```
module.exports = {
  experimental: {
    urlImports: ['https://cdn.skypack.dev'],
  },
};
```

接著，您就可以直接從網址匯入模組：

```
import confetti from 'https://cdn.skypack.dev/canvas-confetti';
```

任何提供 ES 模組的 CDN 都可以工作，包括無代碼和設計工具，比如 Framer：

*   [Skypack](https://skypack.dev)
*   [esm.sh](https://esm.sh/)
*   [jsDelivr](https://www.jsdelivr.com/)
*   [JSPM](https://jspm.org/)
*   [unpkg](https://unpkg.com/)

更多資訊，請觀看我們 [Next.js Conf 的演示影片](https://www.youtube.com/watch?v=_WNeAubn92U) 和 [查看文件說明](/docs/pages/api-reference/next-config-js/urlImports)。

機器人感知的 ISR 回退[](#bot-aware-isr-fallback)
-------------------------------------------------

目前，[Incremental Static Regeneration](https://vercel.com/docs/concepts/incremental-static-regeneration/overview) 使用 `fallback: true` 在首次請求未生成的頁面時，渲染回退狀態，然後渲染頁面內容。要阻止頁面加載（伺服器渲染），您需要使用 `fallback: 'blocking'`。

在 Next.js 12 中，[網路爬蟲（例如搜尋機器人）](https://nextjs.org/learn/seo/introduction-to-seo/webcrawlers) 將自動使用 `fallback: true` 伺服器渲染 ISR 頁面，同時仍為非爬蟲 User-Agent 提供回退狀態的前一個行為。這可以防止爬蟲索引加載狀態。

使用 AVIF 的更小圖片[](#smaller-images-using-avif)
-------------------------------------------------------

內建的圖片優化 API 現在支援 AVIF 圖片，相比 WebP 可獲得 20% 更小的圖片。

與 WebP 相比，AVIF 圖片可能需要更長時間進行優化，因此我們透過 `next.config.js` 中的新 `images.formats` 屬性提供此功能需啟用：

next.config.js

```
module.exports = {
  images: {
    formats: ['image/avif', 'image/webp'],
  },
};
```

此格式列表用於使用請求的 `Accept` 標頭按需決定優化的圖片格式。由於 AVIF 排在首位，如果 [瀏覽器支援 AVIF](https://caniuse.com/avif)，將提供 AVIF。否則，如果 [瀏覽器支援 WebP](https://caniuse.com/webp)，將提供 WebP。如果都不支援，將提供原始圖片格式。

輸出文件跟蹤[](#output-file-tracing)
-------------------------------------------

在 Next.js 8 中，我們引入了 `target` 選項。這允許使用 webpack 打包所有依賴項進行構建時，將 Next.js 頁面作為獨立的 JavaScript 打包輸出。我們很快意識到這不是理想的，因此創建了 `@vercel/nft`。`@vercel/nft` 已在 Vercel 平台上的所有部署中使用超過 2 年。

現在，我們將這些改進直接導入 Next.js 框架中，作為默認設置，**針對所有部署平台**，提供顯著優化的解決方案，超越了 `target` 選項。

Next.js 12 自動使用 `@vercel/nft` 跟蹤每個頁面和 API 路由需要的文件，並在 `next build` 輸出旁輸出這些跟蹤結果，允許集成人員利用 Next.js 提供的自動跟蹤。

這些變更也優化了使用 Docker 等工具部署的應用，通過 `next start`。通過利用 `@vercel/nft`，未來的 Next.js 輸出將獨立。不需要安裝依賴項即可運行應用，大幅減少 Docker 镜像大小。

將 `@vercel/nft` 引入 Next.js 使 `target` 方法成為歷史，在 Next.js 12 中 `target` 棄用。[查看文件說明](/docs/pages/api-reference/next-config-js/output) 獲取更多資訊。

[其他改進](#other-improvements)
-----------------------------------------

*   將 `pages/_app.js` 或 `pages/_document.js` 添加到您的應用現在自動替換內建版本，無需重新啟動 Next.js CLI。
*   ESLint 集成現在支援 `next lint` 的 [單文件檢查](https://github.com/vercel/next.js/pull/28042)，使用 `--file` 旗標。
*   Next.js 12 現在支援設置自訂 `tsconfig.json` 路徑。
*   `next.config.mjs` 現在支援作為 ES 模組編寫配置。
*   getStaticProps 現在對未完成的請求進行去重。
*   檢查靜態頁面現在使用共享工作線程池運行。
*   Fast Refresh 現在使用 WebSocket 連線，而不是 EventSource 連線。

破壞性變更[](#breaking-changes)
-------------------------------------

*   在將 webpack 5 設為默認後 [Next.js 11](https://nextjs.org/blog/next-11#webpack-5)，我們現在正式移除了 webpack 4。我們與社區密切合作，確保無縫過渡到 webpack 5。
*   `next.config.js` 中的 `target` 不再需要。
*   `next/image` 現在使用 `span` 作為包裝元素，而不是 `div`。
*   最低 Node.js 版本從 `12.0.0` 提升到 `12.22.0`，這是 Node.js 中第一個支援原生 ES 模組的版本。

要了解更多，請查看 [升級指南](/docs/pages/building-your-application/upgrading)。
