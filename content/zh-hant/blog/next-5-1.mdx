很高興向大家介紹 Next.js 5.1，它支援環境配置、階段、源碼映射以及新的 Next.js 外掛。

帶來重大的效能改進：頁面解析速度提升了 **102 倍**，錯誤頁面載入也更加高效。

若要升級或安裝，請執行：

終端機

```
npm i next@latest react@latest react-dom@latest
```

> 除了更新 Next.js 版本外，我們也升級了相依套件 (peer dependencies) `react` 和 `react-dom`

請務必同時升級 `@zeit/next-css`、`@zeit/next-sass`、`@zeit/next-less` 或 `@zeit/next-typescript` 等 [next-plugins](https://github.com/vercel/next-plugins)。

頁面解析更快 [](#faster-page-resolution)
-------------------------------------------------

得益於 Next.js 5.0 的架構改進，我們能夠 [簡化邏輯](https://github.com/vercel/next.js/pull/3776)，以根據 URL 路徑解析頁面。這些改進基於 [@oliviertassinari](https://github.com/oliviertassinari) 的 [研究](https://github.com/vercel/next.js/issues/3682)。過去解析一個頁面平均需要 **2.347ms**。使用新邏輯解析同一頁面平均只需 **0.023ms**。這對 Next.js 應用中最常被調用的方法之一提供了 **102 倍** 的速度提升。

每個請求顯示頁面解析。左側為 Next.js 5.0，右側為 Next.js 5.1。

環境配置 [](#environment-configuration)
-------------------------------------------------------

一般的 Node.js 環境通常依賴傳遞環境變數給應用程式，例如：`API_URL=https://api.vercel.com node index.js`，然後你就可以在應用程式中使用 `process.env.API_URL` 來使用 `API_URL`。

由於通用渲染，`process.env` 在客戶端不可用。因此，在 Next.js 5.1 中，我們引入了一項新功能：`publicRuntimeConfig` 和 `serverRuntimeConfig`。這些可以在 `next.config.js` 中設置，然後通過 `next/config` 模組使用。

next.config.js

```
module.exports = {
  serverRuntimeConfig: {
    // Will only be available on the server side
    mySecret: 'secret',
  },
  publicRuntimeConfig: {
    // Will be available on both server and client
    staticFolder: '/static',
  },
};
```

> 兩者 `serverRuntimeConfig` 和 `publicRuntimeConfig` 都是在 `next.config.js` 中定義的

混合式 AMP 模式在生產環境中的一個使用範例是 Reddit。每個線程都存儲在 [Google 的 AMP 緩存](https://developers.google.com/amp/cache/) 中，以在移動端網絡上提供快速的用戶體驗，同時仍為桌面端提供 Reddit 的完整版本和後續導航。

```
import getConfig from 'next/config';
const { serverRuntimeConfig, publicRuntimeConfig } = getConfig();
 
console.log(serverRuntimeConfig.mySecret); // Will only be available on the server side
console.log(publicRuntimeConfig.staticFolder); // Will be available on both server and client
 
export default function Index() {
  return (
    <div>
      <img src={`${publicRuntimeConfig.staticFolder}/logo.png`} />
    </div>
  );
}
```

> `next/config` 模組中的 `getConfig` 方法用於獲取配置值

錯誤處理改進 [](#improved-error-handling)
---------------------------------------------------

以前，Next.js 有一個專門的錯誤處理機制，用於檢測加載頁面包時的服務端錯誤。頁面包是加載到客戶端以載入頁面的 JavaScript 文件，例如 `/_next/-/page/index.js`。

如果出現錯誤，例如構建 ID 不匹配，頁面包仍會以 200 HTTP 狀態碼提供，但內容將是 Next.js 伺服器產生的錯誤的 JSON 表示。這樣做的原因是，存在客戶端的錯誤處理機制，其不僅僅依賴於頁面是否是 404。這個解決方案運作得非常好，直到你嘗試將資源上傳到不支援回退的靜態文件托管服務器或 CDN 時。

隨著 Next.js 5.1，我們 [完全重構了](https://github.com/vercel/next.js/pull/3849) 錯誤處理邏輯，當頁面包返回 404 HTTP 狀態時，路由器將自動檢測並重新載入頁面，以確保 [多區域](https://github.com/vercel/next.js#multi-zones) 之間的導航是可能的。

在重寫此邏輯時，我們 [刪除了 `Router.onAppUpdated` 鉤子](https://github.com/vercel/next.js/blob/canary/errors/no-on-app-updated-hook.md)；該鉤子主要用於觸發頁面重新載入。現在頁面將自動重新載入。

此外，我們添加了一組新的 [集成測試](https://github.com/vercel/next.js/pull/3895)，關於開發模式下的錯誤恢復，以避免在未來的版本中出現錯誤恢復方面的回歸。

階段 / 配置函數 [](#phases--config-function)
----------------------------------------------------

一些 [next-plugins](https://github.com/vercel/next-plugins)，例如 [`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)，僅在 Next.js 處於開發模式或運行 `next build` 時需要。

您現在可以導出一個返回配置對象的函數，而不是立即導出該對象。

```
module.exports = (phase, { defaultConfig }) => config;
```

> `next.config.js` 導出一個返回用戶配置的函數

導出函數將讓您獲取 Next.js 運行的 `phase`，例如開發、生產、構建、導出。這允許插件僅在需要時載入，但也允許訪問默認配置。

我們引入了一個名為 [`next/constants`](https://github.com/vercel/next.js/blob/canary/lib/constants.js) 的新模組，用於存放常用常量（包括階段）。

```
const {PHASE_DEVELOPMENT_SERVER} = require('next/constants')
module.exports = (phase, {defaultConfig}){
  if(phase === PHASE_DEVELOPMENT_SERVER) {
    return { /* development only config options here */ }
  }
 
  return { /* config options for all phases except development here */ }
}
```

> 一個檢查開發階段的 `next.config.js`

改進的生產環境源碼映射生成 [](#improved-production-source-map-generation)
---------------------------------------------------------------------------------------

隨著 [Next.js 5](/blog/next-5) 中通用 Webpack 的引入，將源碼映射到您的生產環境變得只需在 `next.config.js` 中添加幾行代碼一樣簡單：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
      for (const plugin of config.plugins) {
        if (plugin['constructor']['name'] === 'UglifyJsPlugin') {
          plugin.options.sourceMap = true;
          break;
        }
      }
    }
    return config;
  },
};
```

> 在 `next.config.js` 中手動啟用源碼映射

可以將 [@zeit/next-source-maps](https://github.com/vercel/next-plugins/tree/master/packages/next-source-maps) 添加到項目中以自動為您啟用生產環境源碼映射，請將以下內容添加到 `next.config.js`：

```
const withSourceMaps = require('@zeit/next-source-maps');
module.exports = withSourceMaps();
```

> 使用 `@zeit/next-source-maps` 在 `next.config.js` 中啟用源碼映射

這使得除了一個文件 `app.js` 之外的所有文件都能輸出源碼映射，原因是 `app.js` 由多個文件（`manifest.js` 和 `commons.js`）與 Webpack 插件組合而成。這樣產生的副作用是 Webpack 無法為組合後的文件生成源碼映射。

感謝 [@ptomasroos](https://github.com/ptomasroos) 的 [拉取請求](https://github.com/vercel/next.js/pull/3793)，`app.js` 文件被 `main.js` 替換。此文件將保留之前編譯到 `manifest.js` 和 `commons.js` 的代碼，Webpack 將為 `main.js` 生成源碼映射。源碼映射將自動提供服務，允許外部錯誤追蹤工具在檢測到錯誤時顯示實際文件和行號。

源碼顯示在源碼面板中。

新外掛 / 對現有外掛的改進 [](#new-plugins--improvements-to-existing-ones)
------------------------------------------------------------------------------------------

我們引入了兩個新的官方外掛。[@zeit/next-bundle-analyzer](https://github.com/vercel/next-plugins/tree/master/packages/next-bundle-analyzer) 允許簡單地設置 `webpack-bundle-analyzer` 以分別分析服務端和客戶端的包。

此外，官方 [`css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less) 和 [`sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) 外掛在熱裝載和打包方面也有了許多改進。例如，在開發模式中不再顯示未樣式內容閃現。子組件中的樣式也能被識別。
