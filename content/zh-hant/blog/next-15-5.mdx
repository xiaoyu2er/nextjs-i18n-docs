Next.js 15.5 包含 beta 版本的 Turbopack 構建、穩定的 Node.js 中間件、TypeScript 改進、`next lint` 棄用，以及針對 Next.js 16 的棄用警告。本次發布的亮點包括：

*   [**Turbopack 構建（beta）**](/blog/next-15-5#turbopack-builds-beta): 生產環境 Turbopack 構建（`next build --turbopack`）現在進入 beta 階段
*   [**Node.js 中間件（穩定）**](/blog/next-15-5#nodejs-middleware-stable): 中間件的 Node.js 運行時支持現在已穩定
*   [**TypeScript 改進**](/blog/next-15-5#typescript-improvements): 類型化路由、路由導出驗證和路由類型輔助函數
*   [**`next lint`**](/blog/next-15-5#next-lint-deprecation): `next lint` 命令棄用
*   [**Next.js 16**](/blog/next-15-5#deprecation-warnings-for-nextjs-16): Next.js 16 的棄用警告

立即升級，或開始使用：

終端機

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade latest
 
# ...or upgrade manually
npm install next@latest react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@latest
```

Turbopack 構建（beta）[](#turbopack-builds-beta)
-------------------------------------------------

我們很高興宣布 `next build --turbopack` 現在已進入 beta 階段。Turbopack 現在驅動 Vercel 網站，包括 [vercel.com](https://vercel.com)、[v0.app](https://v0.app) 和 [nextjs.org](https://nextjs.org)，通過更快的預覽和生產部署構建來加速迭代速度。

這些由 Turbopack 驅動的應用程序已在推出後經過實戰測試，服務超過 **12 億** 次請求。

### 性能與生產結果 [](#performance--production-results)

Turbopack 的原設計目標之一，是能夠幫助開發人員隨著應用程序代碼庫的擴展而擴展構建性能。為此，Turbopack 從根本上設計為利用多個 CPU 核心來優化構建的各個階段。

我們已在許多 Vercel 產品中部署了使用 Turbopack 的構建，並看到編譯時間的一致性改進：

*   **客戶網站**：4 核心機器上快 2 倍
*   **客戶網站**：14 核心機器上快 2.2 倍
*   **小型網站（10K 模塊）**：30 核心機器上快 4 倍
*   **中型網站（40K 模塊）**：30 核心機器上快 2.5 倍
*   **大型網站（70K 模塊）**：30 核心機器上快 5 倍

當我們將 Turbopack 部署到主要 Web 應用時，我們密切監控生產環境的性能回歸問題和損壞。

與 Webpack 相比，我們的生產指標監測顯示，使用 Turbopack 構建的網站在較少的請求中具有相似或更少的 JavaScript 和 CSS 內容，導致 FCP、LCP 和 TTFB 指標相當或更好。

對於在開發中採用 Turbopack 的團隊，我們現在建議嘗試使用 Turbopack 進行構建。

### 已知差異 [](#known-differences)

**小型專案**：在較小的機器或較小的專案上，由於 Webpack 內建的持久化緩存，我們測量到構建時間的改善幅度有限或中立。我們目前正在為 Turbopack 開發 [持久化緩存](https://nextjs.org/docs/15/app/api-reference/config/next-config-js/turbopackPersistentCaching) 解決方案，以實現我們的所有構建都應具有增量和快速設計目標。

**Bundle 最佳化**：在某些邊緣情況下，我們測量到 Webpack 產生的 Bundle 更優化。我們正在追蹤這些情況，並致力於在穩定版發布前縮小差距。請參閱 [關於 bundle sizes 的文檔](https://nextjs.org/docs/app/api-reference/turbopack#bundle-sizes) 以獲取更多信息。

**CSS 排列順序**：由於 Turbopack 在側邊效應處理方面的不同啟發式方法，CSS 文件的連接順序可能與 Webpack 不同，導致潛在的視覺差異。請參閱 [文檔](https://nextjs.org/docs/app/api-reference/turbopack#css-module-ordering) 以獲取詳細說明和建議的解決方案。

> **注意**：這些差異已記錄在案，我們正在積極改善。如需詳細信息和變通方法，請參閱我們的 [Turbopack 文檔](https://nextjs.org/docs/app/api-reference/turbopack#known-gaps-with-webpack).

在我們迭代至穩定版的過程中，如有任何問題，請在 [GitHub](https://github.com/vercel/next.js/discussions/77721) 上報告。

Node.js 中間件（穩定）[](#nodejs-middleware-stable)
--------------------------------------------------------

在向我們的 [15.2 版本](/blog/next-15-2) 引入 Node.js 運行時的實驗性功能並對我們的生產應用進行廣泛測試後，我們很高興宣布 Node.js 中間件運行時已獲得穩定支持。

此前，Next.js 中間件僅支持 Edge 運行時，雖然提供了更好的性能和隔離性，但在整合 Node.js 特定庫和 API 時存在限制。

middleware.ts

```
import { NextRequest, NextResponse } from 'next/server';
 
export const config = {
  runtime: 'nodejs', // Now stable!
};
 
export function middleware(request: NextRequest) {
  // Access to full Node.js APIs and npm packages
  const fs = require('fs');
  const crypto = require('crypto');
 
  // Complex authentication logic
  const token = request.headers.get('authorization');
 
  if (!isValidToken(token)) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
 
  return NextResponse.next();
}
 
function isValidToken(token: string | null): boolean {
  // Use Node.js crypto for validation
  // Access file system, databases, etc.
  return true;
}
```

此變更使中間件能夠處理更複雜的使用場景，同時保持相同的開發者體驗。

雖然 Node.js 運行時在 Next.js 16 中不會預設啟用，但我們正根據社區反饋和使用模式，探索從 Next.js 17 開始將其設為默認。

TypeScript 改進 [](#typescript-improvements)
---------------------------------------------------

Next.js 15.5 為 App Router 帶來了重大的 TypeScript 改進，實現了完整的 Turbopack 兼容性，並為路由提供了全面的類型安全。

### 類型化路由（穩定）[](#typed-routes-stable)

類型化路由為應用程序的路由提供了編譯時類型安全，在鏈接到達生產環境之前攔截無效鏈接。此功能根據您的文件結構自動生成類型，確保每個 `<Link>` 組件指向有效的路由。

此功能位於 `typedRoutes` 標誌後，該標誌之前是實驗性功能，現在已穩定。靜態類型化的路由現在通過新的實現與 Turbopack 一起工作，為 Webpack 和 Turbopack 構建提供類型安全：

next.config.ts

```
const nextConfig = {
  typedRoutes: true, // Now stable!
};
 
export default nextConfig;
```

```
import Link from 'next/link'
 
// Full type safety for route paths
<Link href="/blog/example-slug?ui=dark">Read Post</Link>
 
// TypeScript catches invalid routes at compile time
<Link href="/invalid-route">Broken Link</Link> // ← Type error
```

### 路由導出驗證（Turbopack）[](#route-export-validation-turbopack)

路由導出驗證也通過新的系統在 Turbopack 上工作，該系統生成類型檢查器文件，使用 TypeScript 的 `satisfies` 操作符驗證頁面、佈局和路由處理函數。

這在運行 `next build` 編譯期間會捕捉到錯誤的 `dynamic` 值等無效導出，用一種跨兩種構建系統更高效的解決方案取代了舊的 Webpack 插件。

### 路由 Props 輔助函數 [](#route-props-helpers)

Next.js 自動生成全域可用的 `PageProps`、`LayoutProps` 和 `RouteContext` 類型，具有完整的參數類型檢查，無需導入：

**之前：手動類型檢查和導入**

```
import { Metadata } from 'next';
 
interface Props {
  params: Promise<{ slug: string }>;
  children: React.ReactNode;
  analytics: React.ReactNode; // Manual parallel route typing
  team: React.ReactNode; // Manual parallel route typing
}
 
export default function DashboardLayout(props: Props) {
  return (
    <div>
      {props.children}
      {props.analytics} {/* No type safety for parallel routes */}
      {props.team} {/* No type safety for parallel routes */}
    </div>
  );
}
```

**之後：自動類型檢查並支持並行路由**

```
// No need to import LayoutProps - globally available
export default function DashboardLayout(props: LayoutProps<'/dashboard'>) {
  return (
    <div>
      {props.children}
      {props.analytics} {/* Fully typed parallel route slot */}
      {props.team} {/* Fully typed parallel route slot */}
    </div>
  );
}
```

該系統自動從您的文件結構中發現路由，支持動態路由、並行路由以及來自 `next.config.js` 的自定義路由。類型生成在開發和構建模式下運行，在開發中更改文件結構時立即重新生成類型，並且通過僅生成幾個優化文件而不是先前實現中使用的許多單獨文件，有效地擴展到大型項目。

### `next typegen`[](#next-typegen)

我們添加了 `next typegen` 命令，用於在不運行 `next dev` 或 `next build` 的情況下手動生成類型。這對於外部類型驗證場景特別有用。

終端機

```
next typegen [directory]
```

此前，路由類型僅在 `next dev` 或 `next build` 期間生成，這意味著直接運行 `tsc --noEmit` 不會驗證您的路由類型。現在您可以獨立生成類型並在外部驗證它們：

終端機

```
# Generate route types first, then validate with TypeScript
next typegen && tsc --noEmit
 
# Or in CI workflows for type checking without building
next typegen && npm run type-check
```

`next lint` 棄用 [](#next-lint-deprecation)
-------------------------------------------------

從 Next.js 15.5 開始，`next lint` 命令會顯示棄用警告，並將在 Next.js 16 中移除。這現代化了代碼檢查（Linting）體驗，轉向明確的 ESLint 配置，並引入 Biome 作為快速替代品。

創建新的 Next.js 項目時，您現在可以在 ESLint（綜合規則）、Biome（快速且規則較少）或不使用 Linter 之間選擇。ESLint 項目現在生成明確的 `eslint.config.mjs` 文件，而不是依賴 `next lint` 命令封裝，使您對 Linting 規則有完整的透明度。

Biome 項目獲得優化配置，包含 Next.js 和 React 規則以及內建格式化功能。生成的 `package.json` 腳本現在直接調用 Linters：

```
{
  "scripts": {
    // ESLint projects
    "lint": "eslint", // Instead of "next lint"
    "lint:fix": "eslint --fix",
 
    // For Biome projects
    "lint": "biome check",
    "format": "biome format --write"
  }
}
```

對於現有項目，新的 Codemod 自動將從 `next lint` 到 ESLint CLI 的遷移：

終端機

```
npx @next/codemod@latest next-lint-to-eslint-cli .
```

Codemod 智能地轉換 `package.json` 腳本，同時保留功能，將 Next.js 特定標誌如 `--strict` 映射為 `--max-warnings 0`，並自動安裝所需依賴項。

此轉變讓開發者可以直接控制他們的 Linting 設置，並具備更好的生態系統兼容性。

> **注意**：如果存在 ESLint 配置，`next build` 仍會運行 Linting 驗證步驟。構建期間的此自動 Linting 也將在 Next.js 16 中移除，讓您完全控制 Linting 何時以及如何運行。

Next.js 16 的棄用警告 [](#deprecation-warnings-for-nextjs-16)
--------------------------------------------------------------------------

Next.js 15.5 引入了棄用警告，幫助您為即將到來的 Next.js 16 版本做好準備。這些警告會出現在開發和構建日誌中，讓您在這些功能被移除之前有時間進行遷移。

### `legacyBehavior` 用於 `next/link` [](#legacybehavior-for-nextlink)

用於 `next/link` 的 `legacyBehavior` Prop 將在 Next.js 16 中移除。此 Prop 是在 Next.js 12 到 13 過渡期間作為臨時兼容性層引入的。

```
// ❌ Will be removed in Next.js 16
<Link href="/about" legacyBehavior>
  <a>About</a>
</Link>
 
// ✅ Modern approach (no changes needed)
<Link href="/about">About</Link>
```

**遷移**：移除 `legacyBehavior` Prop 和任何子元素 `<a>`。`Link` 組件現在自動處理樣式和可訪問性。

### AMP 支持 [](#amp-support)

Next.js 的 AMP 支持將在 Next.js 16 中移除。AMP 採用率顯著下降，維護此功能增加了框架的複雜性。所有與 AMP 相關的 API 和配置都將被移除。

pages/amp-page.js

```
// ❌ Will be removed in Next.js 16
import { useAmp } from 'next/amp';
 
export const config = { amp: true };
 
export default function AmpPage() {
  const isAmp = useAmp();
  return <div>AMP Page: {isAmp ? 'AMP' : 'HTML'}</div>;
}
```

next.config.ts

```
const nextConfig = {
  amp: {
    // ❌ Will be removed in Next.js 16
    canonicalBase: 'https://example.com',
  },
};
 
export default nextConfig;
```

**遷移**：移除所有與 AMP 相關的代碼，包括：

*   頁面中的 `export const config = { amp: true }`
*   `next.config.ts` 中的 `amp` 配置
*   `next/amp` 鉤子導入和用法（`useAmp`）
*   任何其他 AMP 特定的 API

評估 AMP 對您的使用場景是否仍然必要。大多數性能優勢現在可以通過 Next.js 的內建優化和新穎的 Web 標準實現。

### `next/image` 質量設置 [](#nextimage-quality-settings)

從 Next.js 16 開始，`quality` Prop 預設僅限制為 `75`。目前 Next.js 15 中，您可以使用從 1 到 100 的任何整數，但 Next.js 16 將要求為 75 以外的質量值進行明確配置。

```
// ⚠️ Will show deprecation warning in Next.js 15.5
// when images.qualities is undefined and quality !== 75
<Image src="/photo.jpg" quality={100} alt="Photo" />
 
// ✅ Explicit configuration required for Next.js 16
```

next.config.ts

```
const nextConfig = {
  images: {
    qualities: [75, 100], // Explicitly allow quality={100}
  },
};
 
export default nextConfig;
```

**遷移**：如果您使用 75 以外的 `quality` Prop，请在 `next.config.ts` 中明確配置 `images.qualities`，以包含 Next.js 16 所需的质量值。

### `next/image` 本地模式 [](#nextimage-local-patterns)

從 Next.js 16 開始，在本地圖像 `src` 路徑中使用查詢字符串將需要在 `images.localPatterns` 進行明確配置。這會影響具有查詢參數的圖像，例如緩存破壞或版本控制。

```
// ⚠️ Will show deprecation warning in Next.js 15.5
// when images.localPatterns is not configured
<Image src="/photo.jpg?v=1" alt="Test" />
 
// ✅ Explicit configuration required for Next.js 16
```

next.config.ts

```
const nextConfig = {
  images: {
    localPatterns: [
      {
        pathname: '/photo.jpg', // allow exact path
        // omitting "search" will allow all query parameters
      },
      {
        pathname: '/photo.jpg', // allow exact path
        search: '?v=1', // allow exact query parameters
      },
      {
        pathname: '/assets/**', // allow wildcard path
        search: '', // empty search will block all query parameters
      },
    ],
  },
};
 
export default nextConfig;
```

**遷移**：在 `next.config.ts` 中配置 `images.localPatterns`，明確允許圖像路徑中的查詢字符串。這提供了更好的安全性和性能優化。

### 時間表 [](#timeline)

這些棄用警告將從 Next.js 15.5 開始出現。這些功能將在 Next.js 16 中完全移除。我們建議盡快進行遷移，以避免升級期間出現問題。
