Next.js 16 (測試版) 現已推出。這個早期發布版本讓您能在新版 Turbopack、快取以及 Next.js 架構的穩定發布之前，先行體驗最新改進。此次發布的重點包括：

*   [**Turbopack (穩定)**](#turbopack-stable): 所有應用的預設打包工具，Fast Refresh 速度快達 5-10 倍，建置速度快達 2-5 倍
*   [**Turbopack 檔案系統快取 (測試版)**](#turbopack-file-system-caching-beta): 為大型應用提供更快的啟動和編譯時間
*   [**React 編譯器支援 (穩定)**](#react-compiler-support-stable): 內建整合，提供自動記憶化功能
*   [**建置適配器 API (測試版)**](#build-adapters-api-alpha): 建立自定義適配器以修改建置流程
*   [**增強路由**](#enhanced-routing-and-navigation): 優化導航與預取，結合佈局去重和增量預取
*   [**改善的快取 API**](#improved-caching-apis): 新的 `updateTag()` 和精簡的 `revalidateTag()`
*   [**React 19.2**](#react-192-and-canary-features): 查看轉換 (View Transitions), `useEffectEvent()`, `<Activity/>`
*   [**破壞性變更**](#breaking-changes-and-other-updates): 異步參數 (async params), `next/image` 預設值，以及其他

**我們鼓勵您試用測試版並分享您的反饋。** 您的測試幫助我們在穩定發布之前識別問題並改進 Next.js。請在 [GitHub](https://github.com/vercel/next.js/issues) 上報告您遇到的任何錯誤或問題。

將版本升級至測試版：

終端機

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade beta
 
# ...or upgrade manually
npm install next@beta react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@beta
```

開發者體驗
-------------

### Turbopack (穩定)

Turbopack 已經在開發和生產建置中達到了穩定狀態，現在是所有新 Next.js 專案的預設打包工具。自今年夏天初發佈測試版以來，採用率迅速增長：在 Next.js 15.3+ 上，超過 50% 的開發工作階段和 20% 的生產建置已經運行在 Turbopack 上。

使用 Turbopack，您可以預期：

*   生產建置快 2–5 倍
*   Fast Refresh 速度最高可提升 10 倍

我們正使 Turbopack 成為預設，無需任何配置即可為每位 Next.js 開發者帶來這些性能提升。對於使用自定義 webpack 設定的應用，您可以繼續使用 webpack，命令如下：

終端機

```
next dev --webpack
next build --webpack
```

### Turbopack 檔案系統快取 (測試版)

Turbopack 現在支援開發環境中的檔案系統快取，在執行之間將編譯器工件存儲在磁盤上，特別是在大型專案中，顯著加快了重启間的編譯時間。

在您的配置中啟用檔案系統快取：

next.config.ts

```
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
};
 
export default nextConfig;
```

所有內部的 Vercel 應用已經在使用這個功能，我們已經在大型儲存庫中看到開發者生產力的顯著提高。

我們很想聽到您在迭代檔案系統快取時的反饋。請試用並分享您的經歷。

### 簡化的 `create-next-app`

`create-next-app` 已重新設計，擁有簡化的設定流程、更新後的專案結構和改進的預設值。新的範本預設包含 App Router、TypeScript-first 配置、Tailwind CSS 和 ESLint。

### 建置適配器 API (測試版)

根據 [Build Adapters RFC](https://github.com/vercel/next.js/discussions/77740)，我們與社區和部署平台合作，交付了建置適配器 API 的首個測試版。

建置適配器允許您建立自定義適配器以掛入建置流程，使部署平台和自定義建置整合能夠修改 Next.js 配置或處理建置輸出。

next.config.js

```
const nextConfig = {
  experimental: {
    adapterPath: require.resolve('./my-adapter.js'),
  },
};
 
module.exports = nextConfig;
```

在 [RFC 討論](https://github.com/vercel/next.js/discussions/77740) 中分享您的反饋。

### React 編譯器支援 (穩定)

在 React Compiler 1.0 發佈後，Next.js 16 中的 React 編譯器內建支援已達穩定狀態。React 編譯器自動記憶化組件，在無需手動修改任何程式碼的情況下減少不必要的重新渲染。

`reactCompiler` 配置選項已從 `experimental` 提升為穩定狀態。它不在預設開啟，因為我們仍在收集不同應用類型的建置效能數據。啟用此選項時，請預期開發環境和建置過程中的編譯時間會較高，因為 React 編譯器依賴 Babel。

next.config.ts

```
const nextConfig = {
  reactCompiler: true,
};
 
export default nextConfig;
```

安裝 React Compiler 插件的最新版本：

終端機

```
npm install babel-plugin-react-compiler@latest
```

核心功能與架構
----------------

### 增強路由與導航

Next.js 16 包含了路由和導航系統的完全重新設計，使頁面轉換更簡潔、更快。

**佈局去重**：當預取共享佈局的多個 URL 時，佈局只會下載一次，而不是為每個 `Link` 分別下載。例如，擁有 50 個產品連結的頁面現在只會下載一次共享佈局，而不是 50 次，大幅減少了網路傳輸大小。

**增量預取**：Next.js 僅預取快取中不存在的部分，而不是整個頁面。預取快取現在：

*   當連結離開視口時取消請求
*   在懸停或重新進入視口時優先預取連結
*   當其數據無效時重新預取連結
*   與即將推出的功能如快取組件 (Cache Components) 無縫配合

**權衡取捨**：您可能會看到更多個別預取請求，但總傳輸大小更低。我們認為這幾乎是所有應用的正確權衡。如果增加的請求數量造成問題，請告知我們。我們正在努力進一步優化，以更有效地內聯數據區塊。

這些變更不需要任何程式碼修改，旨在改善所有應用的效能。

### PPR 與快取組件

Next.js 16 移除了實驗性的 **部分預渲染 (PPR)** 標記和配置選項。PPR 正在整合到快取組件中。

從 Next.js 16 開始，您可以使用 `experimental.cacheComponents` 配置選擇加入 PPR。實作有所不同，快取組件將帶來額外的功能和行為，這些將在 Next.js Conf 和 Next.js 16 穩定發布之前記錄和宣布。

> **如果您的應用依賴 PPR** (`experimental.ppr = true`)：請停留在您目前使用的固定版本 Next.js `canary`。如果您在遷移時遇到困難，目前請停留在當前版本，我們將在穩定發布之前提供遷移指南。

### 改善的快取 API

Next.js 16 引入了精簡的快取 API，以對快取行為提供更明確的控制。

#### `revalidateTag()` (更新)

`revalidateTag()` 現在需要將 **[`cacheLife` 配置文件](https://nextjs.org/docs/app/api-reference/functions/cacheLife#reference)** 作為第二個引數，以啟用舊數據同時重新驗證 (SWR) 行為：

```
import { revalidateTag } from 'next/cache';
 
// ✅ Use built-in cacheLife profile (we recommend 'max' for most cases)
revalidateTag('blog-posts', 'max');
 
// Or use other built-in profiles
revalidateTag('news-feed', 'hours');
revalidateTag('analytics', 'days');
 
// Or use an inline object with a custom revalidation time
revalidateTag('products', { revalidate: 3600 });
 
// ⚠️ Deprecated - single argument form
revalidateTag('blog-posts');
```

配置文件引數接受內建 `cacheLife` 配置文件名稱（如 `'max'`, `'hours'`, `'days'`）或在您的 `next.config` 中定義的 [自定義配置文件](https://nextjs.org/docs/app/api-reference/functions/cacheLife#defining-reusable-cache-profiles)。您也可以傳遞內聯的 `{ revalidate: number }` 物件。我們建議大多數情況使用 `'max'`，因為它為長期內容啟用背景重新驗證。當用戶請求標記內容時，他們會立即收到快取數據，同時 Next.js 在背景中進行重新驗證。

當您想無效化僅正確標記的快取條目並具有舊數據同時重新驗證 (SWR) 行為時，請使用 `revalidateTag()`。這適合可以容忍最終一致性的靜態內容。

> **遷移指南：** 添加第二個引數，使用 `cacheLife` 配置文件（我們建議使用 `'max'`）以實現 SWR 行為，或者在需要讀寫一致性的語義時使用 Server Actions 中的 `updateTag()`。

#### `updateTag()` (新增)

`updateTag()` 是一個新的僅限 Server Actions 的 API，提供 **讀寫一致性** 語義，在同一請求內過期並立即刷新快取數據：

```
'use server';
 
import { updateTag } from 'next/cache';
 
export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile);
 
  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`);
}
```

這確保了互動功能立即反映更改。適合表單、用戶設定以及用戶期望立即看到更新的任何工作流程。

#### `refresh()` (新增)

`refresh()` 是一個新的僅限 Server Actions 的 API，用於僅刷新 **未快取數據**。它完全不觸碰快取：

```
'use server';
 
import { refresh } from 'next/cache';
 
export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId);
 
  // Refresh the notification count displayed in the header
  // (which is fetched separately and not cached)
  refresh();
}
```

此 API 與客戶端的 `router.refresh()` 相輔相成。當您執行某項操作後需要刷新頁面其他區域顯示的未快取數據時，請使用它。您的快取頁面外殼和靜態內容保持快速，而動態數據如通知計數、實時指標或狀態指示器則會刷新。

### React 19.2 與 Canary 功能

Next.js 16 中的 App Router 使用最新的 React [Canary 發布](https://react.dev/blog/2023/05/03/react-canaries)，其中包括新發佈的 React 19.2 功能和正在逐步穩定的其他功能。重點包括：

*   **[查看轉換 (View Transitions)](https://react.dev/reference/react/ViewTransition)**：在轉換或導航中更新元素的動畫
*   **[`useEffectEvent`](https://react.dev/reference/react/useEffectEvent)**：將非反應式邏輯從 Effects 中提取到可重用的 Effect Event 函數中
*   **[Activity](https://react.dev/reference/react/Activity)**：隱藏 UI (`display: none`) 的同時維護狀態和清理 Effects，渲染 "背景活動"

在 [React 19.2 公告](https://react.dev/blog/2025/10/01/react-19-2) 中了解更多。

破壞性變更與其他更新
-----------------------

### 版本需求

| 變更 | 詳情 |
| --- | --- |
| **Node.js 20.9+** | 最低版本為 20.9.0 (LTS)；Node.js 18 不再支援 |
| **TypeScript 5+** | 最低版本為 5.1.0 |
| **瀏覽器** | Chrome 111+, Edge 111+, Firefox 111+, Safari 16.4+ |

### 移除項

這些功能先前已棄用，現在已移除：

| 已移除 | 替代品 |
| --- | --- |
| **AMP 支援** | 所有 AMP API 和配置已移除 (`useAmp`, `export const config = { amp: true }`) |
| **`next lint` 命令** | 直接使用 Biome 或 ESLint；`next build` 不再執行 eslint。可用 codemod：`npx @next/codemod@canary next-lint-to-eslint-cli .` |
| **`devIndicators` 選項** | 從配置中移除 `appIsrStatus`, `buildActivity`, `buildActivityPosition`。指示器保留。 |
| **`serverRuntimeConfig`, `publicRuntimeConfig`** | 使用環境變數 (`.env` 文件) |
| **`experimental.turbopack` 位置** | 配置移動到頂層 `turbopack` (不再在 `experimental` 中) |
| **`experimental.dynamicIO` 標記** | 重命名為 `experimental.cacheComponents` |
| **`experimental.ppr` 標記** | 移除 PPR 標記；演變成快取組件程式化模型 (programming model) |
| **`export const experimental_ppr`** | 路由級別 PPR 導入已移除；演變成快取組件程式化模型 |
| **自動 `scroll-behavior: smooth`** | 添加 `data-scroll-behavior="smooth"` 到 HTML 文檔以重新選擇加入 |
| **`unstable_rootParams()`** | 我們正在開發替代 API，將在即將推出的次要版本中發行 |
| **同步 `params`, `searchParams` props 訪問** | 必須使用異步：`await params`, `await searchParams` |
| **同步 `cookies()`, `headers()`, `draftMode()` 訪問** | 必須使用異步：`await cookies()`, `await headers()`, `await draftMode()` |
| **Metadata image route `params` 引數** | 改為異步 `params`；`generateImageMetadata` 中的 `id` 現在為 `Promise<string>` |
| **`next/image` 本地 src 與查詢字串** | 現在需要 `images.localPatterns` 配置以防止枚舉攻擊 |

### 行為變更

這些功能在 Next.js 16 中具有新的預設行為：

| 變更的 Behavior | 詳情 |
| --- | --- |
| **預設打包工具** | Turbopack 現在是所有應用的預設打包工具；使用 `next build --webpack` 選擇退出 |
| **`images.minimumCacheTTL` 預設值** | 從 60 秒更改為 4 小時 (14400 秒)；減少沒有 cache-control 標頭圖片的重新驗證成本 |
| **`images.imageSizes` 預設值** | 從默認大小中移除 `16`（僅 4.2% 的專案使用）；減少 srcset 大小和 API 變異 |
| **`images.qualities` 預設值** | 從 `[1..100]` 更改為 `[75]`；`quality` prop 現在強制轉換為 `images.qualities` 中最接近的值 |
| **`images.dangerouslyAllowLocalIP`** | 新安全限制預設阻止本地 IP 優化；僅在私有網絡中設置為 `true` |
| **`images.maximumRedirects` 預設值** | 從無限更改為最多 3 次重定向；設置為 `0` 以禁用或增加以處理罕見邊界情況 |
| **`@next/eslint-plugin-next` 預設值** | 現在預設為 ESLint Flat Config 格式，與 ESLint v10 保持一致，後者將棄用舊配置支援 |
| **預取快取行為** | 完全重新編寫，結合佈局去重和增量預取 |
| **`revalidateTag()` 簽名** | 現在需要 `cacheLife` 配置文件作為第二個引數以實現舊數據同時重新驗證行為 |
| **Turbopack 中的 Babel 配置** | 如果找到 Babel 配置則自動啟用 Babel（之前會以硬錯誤退出） |
| **終端機輸出** | 重新設計，具有更清晰的格式、更好的錯誤訊息和改进的效能指標 |
| **開發和建置輸出目錄** | `next dev` 和 `next build` 現在使用單獨的輸出目錄，實現並行執行 |
| **Lockfile 行為** | 添加了 lockfile 機制以防止在同一專案上運行多個 `next dev` 或 `next build` 實例 |
| **並行路由 `default.js`** | 所有並行路由槽現在都需要明確的 `default.js` 文件；沒有它們建置會失敗。建立 `default.js` 調用 `notFound()` 或返回 `null` 以實現以前行為 |
| **現代 Sass API** | 將 `sass-loader` 提升到 v16，支援現代 Sass 語法和新功能 |

### 棄用項

這些功能在 Next.js 16 中已棄用，將在未來的版本中移除：

| 已棄用 | 詳情 |
| --- | --- |
| `middleware.ts` 檔案名稱 | 重命名為 `proxy.ts` 以明確網路邊界和路由重點 |
| `next/legacy/image` 組件 | 改用 `next/image` 以獲得更好的效能和功能 |
| `images.domains` 配置 | 改用 `images.remotePatterns` 配置以獲得更好的安全限制 |
| `revalidateTag()` 單個引數 | 使用 `revalidateTag(tag, profile)` 實現 SWR，或在 Actions 中使用 `updateTag(tag)` 實現讀寫一致性 |

### 額外改進

*   **效能改進**：`next dev` 和 `next start` 命令的顯著效能優化
*   **Node.js 原生 TypeScript 用於 `next.config.ts`**：使用 `--experimental-next-config-strip-types` 標記執行 `next dev`, `next build` 和 `next start` 命令，為 `next.config.ts` 啟用原生 TypeScript。
