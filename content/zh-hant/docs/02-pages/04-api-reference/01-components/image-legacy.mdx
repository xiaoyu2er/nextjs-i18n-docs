從 Next.js 13 開始，`next/image` 元件被重新撰寫以提升效能和開發者體驗。為了提供相容的升級方案，舊版的 `next/image` 被命名為 `next/legacy/image`。

> **警告**：`next/legacy/image` 已棄用，將在 Next.js 的未來版本中移除。請改用 [`next/image`](/docs/app/api-reference/components/image)。

## 比較

與 `next/legacy/image` 相比，新的 `next/image` 元件有以下變更：

- 移除了 `<img>` 周遭的 `<span>` 包裝，改用 [原生的計算 aspect ratio](https://caniuse.com/mdn-html_elements_img_aspect_ratio_computed_from_attributes)
- 新增支援標準 `style` prop
  - 移除 `layout` prop，改用 `style` 或 `className`
  - 移除 `objectFit` prop，改用 `style` 或 `className`
  - 移除 `objectPosition` prop，改用 `style` 或 `className`
- 移除 `IntersectionObserver` 實作，改用 [原生的 lazy loading](https://caniuse.com/loading-lazy-attr)
  - 移除 `lazyBoundary` prop，因為沒有原生對應
  - 移除 `lazyRoot` prop，因為沒有原生對應
- 移除 `loader` 設定，改用 [`loader`](#loader) prop
- `alt` prop 從可選改為必需
- `onLoadingComplete` 回調改為接收 `<img>` 元件的引用

## 必要屬性

`<Image />` 元件需要以下屬性。

### src

必須為以下其中一項：

- [靜態導入](/docs/pages/api-reference/components/image#src) 的圖片檔案
- 路徑字串。這可以是絕對的外部 URL，或是內部路徑，取決於 [loader](#loader) prop 或 [loader 設定](#loader-configuration)。

使用預設 [loader](#loader) 時，也請考量以下來源圖片的設定：

- 當 src 為外部 URL 時，你也必須設定 [remotePatterns](#remote-patterns)
- 當 src 為 [動態圖片](#animated-images) 或未知格式 (JPEG, PNG, WebP, AVIF, GIF, TIFF) 時，圖片將原樣提供
- 當 src 為 SVG 格式時，它將被阻擋，除非 `unoptimized` 或 `dangerouslyAllowSVG` 已啟用

### width

`width` 屬性可以代表 *呈現* 寬度或 *原始* 寬度（像素），視 [`layout`](#layout) 和 [`sizes`](#sizes) 屬性而定。

使用 `layout="intrinsic"` 或 `layout="fixed"` 時，`width` 屬性代表 *呈現* 寬度（像素），因此會影響圖片顯示的大小。

使用 `layout="responsive"`、`layout="fill"` 時，`width` 屬性代表 *原始* 寬度（像素），因此僅會影響長寬比。

`width` 屬性是必需的，除非是 [靜態導入的圖片](/docs/pages/api-reference/components/image#src)，或那些具有 `layout="fill"` 的圖片。

### height

`height` 屬性可以代表 *呈現* 高度或 *原始* 高度（像素），視 [`layout`](#layout) 和 [`sizes`](#sizes) 屬性而定。

使用 `layout="intrinsic"` 或 `layout="fixed"` 時，`height` 屬性代表 *呈現* 高度（像素），因此會影響圖片顯示的大小。

使用 `layout="responsive"`、`layout="fill"` 時，`height` 屬性代表 *原始* 高度（像素），因此僅會影響長寬比。

`height` 屬性是必需的，除非是 [靜態導入的圖片](/docs/pages/api-reference/components/image#src)，或那些具有 `layout="fill"` 的圖片。

## 選用屬性

`<Image />` 元件接受許多除了必需屬性之外的額外屬性。本節描述 Image 元件最常用的屬性。有關較不常用屬性的詳細資訊，請見 [進階屬性](#advanced-props) 章節。

### layout

視視窗大小變更時的圖片佈局行為。

| `layout`              | 行為                                                 | `srcSet`                                                                                                    | `sizes` | 有包裝和縮放程式 |
| --------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- | ------- | ----------------- |
| `intrinsic` (default) | 縮小以配合容器寬度，至圖片大小 | `1x`, `2x` (基於 [imageSizes](#image-sizes))                                                            | N/A     | yes               |
| `fixed`               | 精確以 `width` 和 `height` 大小 | `1x`, `2x` (基於 [imageSizes](#image-sizes))                                                            | N/A     | yes               |
| `responsive`          | 縮放以配合容器寬度 | `640w`, `750w`, ... `2048w`, `3840w` (基於 [imageSizes](#image-sizes) 和 [deviceSizes](#device-sizes)) | `100vw` | yes               |
| `fill`                | 在 X 和 Y 軸擴展以填滿容器 | `640w`, `750w`, ... `2048w`, `3840w` (基於 [imageSizes](#image-sizes) 和 [deviceSizes](#device-sizes)) | `100vw` | yes               |

- [演示 `intrinsic` 佈局（預設）](https://image-legacy-component.nextjs.gallery/layout-intrinsic)
  - 當 `intrinsic`，圖片會縮小尺寸以適應較小的視窗，但為較大的視窗維持原始尺寸。
- [演示 `fixed` 佈局](https://image-legacy-component.nextjs.gallery/layout-fixed)
  - 當 `fixed`，圖片尺寸不會隨視窗變更而改變（無響應性），類似原生 `img` 元件。
- [演示 `responsive` 佈局](https://image-legacy-component.nextjs.gallery/layout-responsive)
  - 當 `responsive`，圖片會縮小尺寸以適應較小的視窗，並擴展以適應較大的視窗。
  - 確保父元素在樣表中使用 `display: block`。
- [演示 `fill` 佈局](https://image-legacy-component.nextjs.gallery/layout-fill)
  - 當 `fill`，圖片會延伸寬度和高度以符合父元素的尺寸，前提是父元素為 relative。
  - 這通常配合 [`objectFit`](#objectfit) 屬性使用。
  - 確保父元素在樣表中有 `position: relative`。
- [演示背景圖片](https://image-legacy-component.nextjs.gallery/background)

### loader

用於解析 URL 的自訂函數。將 loader 設為 Image 元件的 prop 會覆蓋 [`next.config.js` 的 `images` 章節中定義的預設 loader](#loader-configuration)。

`loader` 是根據以下參數傳回圖片 URL 字串的函式：

- [`src`](#src)
- [`width`](#width)
- [`quality`](#quality)

這裡是使用自訂 loader 的範例：

```js
import Image from 'next/legacy/image'

const myLoader = ({ src, width, quality }) => {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`
}

const MyImage = (props) => {
  return (
    <Image
      loader={myLoader}
      src="me.png"
      alt="Picture of the author"
      width={500}
      height={500}
    />
  )
}
```

### sizes

提供資訊的字串，說明圖片在不同斷點下的寬度。`sizes` 的值會大幅影響使用 `layout="responsive"` 或 `layout="fill"` 的圖片效能。對於使用 `layout="intrinsic"` 或 `layout="fixed"` 的圖片則會被忽略。

`sizes` 屬性在圖片效能方面提供兩個重要目的：

首先，`sizes` 的值由瀏覽器用來決定要從 `next/legacy/image` 自動生成的資源集 (source set) 中下載哪種尺寸的圖片。當瀏覽器選擇時，它還不知道圖片在頁面上的大小，所以會選擇與視窗相同大小或更大的圖片。`sizes` 屬性允許你告訴瀏覽器圖片實際上會比全螢幕小。如果你不指定 `sizes` 值，將使用預設值 `100vw`（全螢幕寬度）。

其次，`sizes` 值會被解析並用來修剪自動創建的資源集中的值。如果 `sizes` 屬性包含如 `50vw` 的大小，代表視窗寬度的百分比，則資源集會被修剪，不包含任何永遠不需要的過小值。

例如，如果你知道你的樣式會導致圖片在行動裝置上為全寬，平板裝置上為 2 欄配置，桌面顯示上為 3 欄配置，你應該包含以下的 sizes 屬性：

```js
import Image from 'next/legacy/image'
const Example = () => (
  <div className="grid-element">
    <Image
      src="/example.png"
      layout="fill"
      sizes="(max-width: 768px) 100vw,
              (max-width: 1200px) 50vw,
              33vw"
    />
  </div>
)
```

這個範例 `sizes` 可能會對效能指標產生巨大影響。没有 `33vw` dimensions，從伺服器選擇的圖片寬度會是其所需寬度的 3 倍。因為檔案大小與寬度的平方成正比，没有 `sizes`，用戶將下載大於必要 9 倍的圖片。

了解更多關於 `srcset` 和 `sizes`：

- [web.dev](https://web.dev/learn/design/responsive-images/#sizes)
- [mdn](https://developer.mozilla.org/docs/Web/HTML/Element/img#attr-sizes)

### quality

最佳化圖片的品質，介於 `1` 到 `100` 的整數，其 `100` 為最佳品質。預設為 `75`。

### priority

當為 true 時，圖片將被視為高優先順序並
[載入優先順序](https://web.dev/preload-responsive-images/)。使用 `priority` 的圖片會自動停用惰性載入。

你應該在任何被偵測為 [largest contentful paint (LCP)](https://nextjs.org/learn/seo/web-performance/lcp) 元素的圖片上使用 `priority` 屬性。可能有適當的圖片擁有多個優先順序圖片，因為不同的圖片可能是不同視窗大小的 LCP 元素。

僅應在圖片顯示在視窗上方時使用。預設為 `false`。

### placeholder

圖片載入時使用的佔位符。可能的值為 `blur` 或 `empty`。預設為 `empty`。

當為 `blur` 時，將使用 [`blurDataURL`](#blurdataurl) 屬性作為佔位符。如果 `src` 是來自 [靜態導入](/docs/pages/api-reference/components/image#src) 的物件，且導入的圖片是 `.jpg`、`.png`、`.webp` 或 `.avif`，則 `blurDataURL` 將自動填充。

對於動態圖片，你必須提供 [`blurDataURL`](#blurdataurl) 屬性。像 [Plaiceholder](https://github.com/joe-bell/plaiceholder) 這樣的解決方案可以協助 `base64` 生成。

當為 `empty` 時，圖片載入時將沒有佔位符，只有空白。

試試看：

- [演示 `blur` 佔位符](https://image-legacy-component.nextjs.gallery/placeholder)
- [演示 `blurDataURL` prop 的 shimmer 效果](https://image-legacy-component.nextjs.gallery/shimmer)
- [演示 `blurDataURL` prop 的色彩效果](https://image-legacy-component.nextjs.gallery/color)

## 進階屬性

在某些情況下，你可能需要更進階的使用方式。`<Image />` 元件選用性地接受以下進階屬性。

### style

允許 [傳遞 CSS 樣式](https://developer.mozilla.org/docs/Web/HTML/Element/style) 至底層的圖片元件。

注意所有 `layout` 模式都會將自己的樣式套用到圖片元件上，這些自動樣式優先於 `style` prop。

也要記住，必需的 `width` 和 `height` prop 可能與你的樣式互動。如果你使用樣式來修改圖片的 `width`，你也必須設定 `height="auto"` 樣式，否則你的圖片會變形。

### objectFit

定義當使用 `layout="fill"` 時圖片如何進入其父容器。

此值傳遞給 `src` 圖片的 [object-fit CSS 屬性](https://developer.mozilla.org/docs/Web/CSS/object-fit)。

### objectPosition

定義當使用 `layout="fill"` 時圖片在其父元素中的位置。

此值傳遞給套用至圖片的 [object-position CSS 屬性](https://developer.mozilla.org/docs/Web/CSS/object-position)。

### onLoadingComplete

一個當圖片完全載入且 [placeholder](#placeholder) 已移除時被呼叫的回調函數。

`onLoadingComplete` 函數接受一個參數，一個包含以下屬性的物件：

- [`naturalWidth`](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/naturalWidth)
- [`naturalHeight`](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/naturalHeight)

### loading

圖片的載入行為。預設為 `lazy`。

當為 `lazy`，延遲載入圖片直到它達到計算的視窗距離。

當為 `eager`，立即載入圖片。

[了解更多](https://developer.mozilla.org/docs/Web/HTML/Element/img#attr-loading)

### blurDataURL

用作 `src` 圖片成功載入前之佔位符圖片的 [Data URL](https://developer.mozilla.org/docs/Web/HTTP/Basics_of_HTTP/Data_URIs)。僅當與 [`placeholder="blur"`](#placeholder) 結合時才生效。

必須為 base64 編碼的圖片。它會被放大並模糊化，因此建議使用非常小的圖片（10px 或更少）。將較大的圖片包含為佔位符可能會損害應用程式效能。

試試看：

- [演示預設 `blurDataURL` prop](https://image-legacy-component.nextjs.gallery/placeholder)
- [演示 `blurDataURL` prop 的 shimmer 效果](https://image-legacy-component.nextjs.gallery/shimmer)
- [演示 `blurDataURL` prop 的色彩效果](https://image-legacy-component.nextjs.gallery/color)

你也可以 [生成單色 Data URL](https://png-pixel.com) 以與圖片匹配。

### lazyBoundary

字串（具有與 margin 屬性相似的語法），作為偵測視窗與圖片交集並觸發惰性 [載入](#loading) 使用的邊界框。預設為 `"200px"`。

如果圖片嵌套在根文件之外的可滚动父元素中，你也會需要指派 [lazyRoot](#lazyroot) prop。

[了解更多](https://developer.mozilla.org/docs/Web/API/IntersectionObserver/rootMargin)

### lazyRoot

指向可滚动父元素的 React [Ref](https://react.dev/learn/referencing-values-with-refs)。預設為 `null`（文件視窗）。

Ref 必須指向指向底層 DOM 元件的 DOM 元件或 React 元件 [forwards the Ref](https://react.dev/reference/react/forwardRef)。

**指向 DOM 元件的範例**

```jsx
import Image from 'next/legacy/image'
import React from 'react'

const Example = () => {
  const lazyRoot = React.useRef(null)

  return (
    <div ref={lazyRoot} style={{ overflowX: 'scroll', width: '500px' }}>
      <Image lazyRoot={lazyRoot} src="/one.jpg" width="500" height="500" />
      <Image lazyRoot={lazyRoot} src="/two.jpg" width="500" height="500" />
    </div>
  )
}
```

**指向 React 元件的範例**

```jsx
import Image from 'next/legacy/image'
import React from 'react'

const Container = React.forwardRef((props, ref) => {
  return (
    <div ref={ref} style={{ overflowX: 'scroll', width: '500px' }}>
      {props.children}
    </div>
  )
})

const Example = () => {
  const lazyRoot = React.useRef(null)

  return (
    <Container ref={lazyRoot}>
      <Image lazyRoot={lazyRoot} src="/one.jpg" width="500" height="500" />
      <Image lazyRoot={lazyRoot} src="/two.jpg" width="500" height="500" />
    </Container>
  )
}
```

[了解更多](https://developer.mozilla.org/docs/Web/API/IntersectionObserver/root)

### unoptimized

當為 true 時，來源圖片將以原本狀態從 `src` 提供，而不是變更品質、大小或格式。預設為 `false`。

這對於無法從最佳化受益的圖片很有用，例如小圖片（小於 1KB）、向量圖片 (SVG) 或動態圖片 (GIF)。

```js
import Image from 'next/image'

const UnoptimizedImage = (props) => {
  return <Image {...props} unoptimized />
}
```

自 Next.js 12.3.0 起，此屬性可以透過更新 `next.config.js` 來指派給所有圖片：

```js filename="next.config.js"
module.exports = {
  images: {
    unoptimized: true,
  },
}
```

## 其他屬性

`<Image />` 元件上的其他屬性將傳遞至底層
`img` 元件，除了以下情況：

- `srcSet`。改用
  [Device Sizes](#device-sizes)。
- `ref`。改用 [`onLoadingComplete`](#onloadingcomplete)。
- `decoding`。它永遠為 `"async"`。

## 設定選項

### 遠端模式 (Remote Patterns)

為防止你的應用程式受到惡意用戶攻擊，必須設定配置才能使用外部圖片。這確保只有來自你的帳戶的外部圖片才能從 Next.js 影像最佳化 API 提供。這些外部圖片可以在 `next.config.js` 檔案中使用 `remotePatterns` 屬性進行設定，如下所示：

```js filename="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'example.com',
        port: '',
        pathname: '/account123/**',
        search: '',
      },
    ],
  },
}
```

> **好知道**：上述範例將確保 `next/legacy/image` 的 `src` 屬性必須以 `https://example.com/account123/` 開頭，且不得有查詢字串。任何其他的協定、主機名、埠或不相配的路徑將回應 400 錯誤要求。

以下是 `next.config.js` 檔案中 `remotePatterns` 屬性使用主機名中萬用字元範例的字串：

```js filename="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '**.example.com',
        port: '',
        search: '',
      },
    ],
  },
}
```

> **好知道**：上述範例將確保 `next/legacy/image` 的 `src` 屬性必須以 `https://img1.example.com` 或 `https://me.avatar.example.com` 或任何數量的子網域名稱開頭。它不能有埠或查詢字串。任何其他的協定或不相匹配的主機名將回應 400 錯誤要求。

萬用字元模式可用於 `pathname` 和 `hostname`，具有以下的語法：

- `*` match a single path segment or subdomain
- `**` match any number of path segments at the end or subdomains at the beginning

`**` 語法在模式中間不起起作用。

> **小提示**：當省略 `protocol`、`port`、`pathname` 或 `search` 時，則隱含萬用字元 `**`。这不推薦，因為它可能會允許惡意行為者最佳化您不想要的 url。

Below is an example of the `remotePatterns` property in the `next.config.js` file using `search`:

```js filename="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'assets.example.com',
        search: '?v=1727111025337',
      },
    ],
  },
}
```

> **Good to know**: The example above will ensure the `src` property of `next/legacy/image` must start with `https://assets.example.com` and must have the exact query string `?v=1727111025337`. Any other protocol or query string will respond with 400 Bad Request.

### 網域 (Domains)

> **Warning**: Deprecated since Next.js 14 in favor of strict [`remotePatterns`](#remote-patterns) in order to protect your application from malicious users. Only use `domains` if you own all the content served from the domain.

Similar to [`remotePatterns`](#remote-patterns), the `domains` configuration can be used to provide a list of allowed hostnames for external images.

However, the `domains` configuration does not support wildcard pattern matching and it cannot restrict protocol, port, or pathname.

Below is an example of the `domains` property in the `next.config.js` file:

```js filename="next.config.js"
module.exports = {
  images: {
    domains: ['assets.acme.com'],
  },
}
```

### Loader Configuration

If you want to use a cloud provider to optimize images instead of using the Next.js built-in Image Optimization API, you can configure the `loader` and `path` prefix in your `next.config.js` file. This allows you to use relative URLs for the Image [`src`](#src) and automatically generate the correct absolute URL for your provider.

```js filename="next.config.js"
module.exports = {
  images: {
    loader: 'imgix',
    path: 'https://example.com/myaccount/',
  },
}
```

#### Customizing the Built-in Image Path

If you want to change or prefix the default path for the built-in Next.js image optimization, you can do so with the `path` property. The default value for `path` is `/_next/image`.

```js filename="next.config.js"
module.exports = {
  images: {
    path: '/my-prefix/_next/image',
  },
}
```

### Built-in Loaders

The following Image Optimization cloud providers are included:

- Default: Works automatically with `next dev`, `next start`, or a custom server
- [Vercel](https://vercel.com): Works automatically when you deploy on Vercel, no configuration necessary. [Learn more](https://vercel.com/docs/concepts/image-optimization?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)
- [Imgix](https://www.imgix.com): `loader: 'imgix'`
- [Cloudinary](https://cloudinary.com): `loader: 'cloudinary'`
- [Akamai](https://www.akamai.com): `loader: 'akamai'`
- Custom: `loader: 'custom'` use a custom cloud provider by implementing the [`loader`](#loader) prop on the `next/legacy/image` component

If you need a different provider, you can use the [`loader`](#loader) prop with `next/legacy/image`.

> 圖片無法在構建時使用 [`output: 'export'`](/docs/pages/guides/static-exports) 進行優化，僅能按需優化。若要使用 `output: 'export'` 搭配 `next/legacy/image`，您需要與預設不同的載入器 (loader)。[在討論中閱讀更多。](https://github.com/vercel/next.js/discussions/19065)

## 進階

The following configuration is for advanced use cases and is usually not necessary. If you choose to configure the properties below, you will override any changes to the Next.js defaults in future updates.

### 裝置大小 (Device Sizes)

If you know the expected device widths of your users, you can specify a list of device width breakpoints using the `deviceSizes` property in `next.config.js`. These widths are used when the `next/legacy/image` component uses `layout="responsive"` or `layout="fill"` to ensure the correct image is served for user's device.

If no configuration is provided, the default below is used.

```js filename="next.config.js"
module.exports = {
  images: {
    deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
  },
}
```

### 圖片大小 (Image Sizes)

You can specify a list of image widths using the `images.imageSizes` property in your `next.config.js` file. These widths are concatenated with the array of [device sizes](#device-sizes) to form the full array of sizes used to generate image [srcset](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/srcset)s.

The reason there are two separate lists is that imageSizes is only used for images which provide a [`sizes`](#sizes) prop, which indicates that the image is less than the full width of the screen. **Therefore, the sizes in imageSizes should all be smaller than the smallest size in deviceSizes.**

If no configuration is provided, the default below is used.

```js filename="next.config.js"
module.exports = {
  images: {
    imageSizes: [32, 48, 64, 96, 128, 256, 384],
  },
}
```

### 可接受格式 (Acceptable Formats)

The default [Image Optimization API](#loader-configuration) will automatically detect the browser's supported image formats via the request's `Accept` header in order to determine the best output format.

If the `Accept` header matches more than one of the configured formats, the first match in the array is used. Therefore, the array order matters. If there is no match (or the source image is [animated](#animated-images)), the Image Optimization API will fallback to the original image's format.

If no configuration is provided, the default below is used.

```js filename="next.config.js"
module.exports = {
  images: {
    formats: ['image/webp'],
  },
}
```

您可以啟用 AVIF 支援，如果瀏覽器 [不支援 AVIF](https://caniuse.com/avif)，則會回退到源圖片的原始格式：

```js filename="next.config.js"
module.exports = {
  images: {
    formats: ['image/avif'],
  },
}
```

您也可以同時啟用 AVIF 和 WebP 格式。AVIF 將用於支援它的瀏覽器，WebP 則作為回退：

```js filename="next.config.js"
module.exports = {
  images: {
    formats: ['image/avif', 'image/webp'],
  },
}
```

> **值得注意**:
>
> - 我們仍建議在大多數情況下使用 WebP。
> - AVIF 編碼時間通常較長約 50%，但壓縮後尺寸比 WebP 小 20%。這意味著首次請求圖片時通常會較慢，但後續緩存的請求則會較快。
> - 使用多種格式時，Next.js 會分別緩存每種格式。這意味著與使用單一格式相比，儲存需求較高，因為不同瀏覽器支援需儲存 AVIF 和 WebP 兩種格式的圖片。
> - 如果您在 Next.js 前端自建 Proxy/CDN，您必須配置 Proxy 以轉發 `Accept` 標頭。

## 快取行為

The following describes the caching algorithm for the default [loader](#loader). For all other loaders, please refer to your cloud provider's documentation.

Images are optimized dynamically upon request and stored in the `<distDir>/cache/images` directory. The optimized image file will be served for subsequent requests until the expiration is reached. When a request is made that matches a cached but expired file, the expired image is served stale immediately. Then the image is optimized again in the background (also called revalidation) and saved to the cache with the new expiration date.

The cache status of an image can be determined by reading the value of the `x-nextjs-cache` (`x-vercel-cache` when deployed on Vercel) response header. The possible values are the following:

- `MISS` - 路徑不在快取中 (最多發生一次，在首次訪問時)
- `STALE` - 路徑在快取中但已超過重新驗證時間，因此會在背景更新
- `HIT` - 路徑在快取中且未超過重新驗證時間

The expiration (or rather Max Age) is defined by either the [`minimumCacheTTL`](#minimum-cache-ttl) configuration or the upstream image `Cache-Control` header, whichever is larger. Specifically, the `max-age` value of the `Cache-Control` header is used. If both `s-maxage` and `max-age` are found, then `s-maxage` is preferred. The `max-age` is also passed-through to any downstream clients including CDNs and browsers.

- You can configure [`minimumCacheTTL`](#minimum-cache-ttl) to increase the cache duration when the upstream image does not include `Cache-Control` header or the value is very low.
- You can configure [`deviceSizes`](#device-sizes) and [`imageSizes`](#image-sizes) to reduce the total number of possible generated images.
- You can configure [formats](#acceptable-formats) to disable multiple formats in favor of a single image format.

### 最小快取 TTL

You can configure the Time to Live (TTL) in seconds for cached optimized images. In many cases, it's better to use a [Static Image Import](/docs/pages/api-reference/components/image#src) which will automatically hash the file contents and cache the image forever with a `Cache-Control` header of `immutable`.

If no configuration is provided, the default below is used.

```js filename="next.config.js"
module.exports = {
  images: {
    minimumCacheTTL: 14400, // 4 hours
  },
}
```

You can increase the TTL to reduce the number of revalidations and potentially lower cost:

```js filename="next.config.js"
module.exports = {
  images: {
    minimumCacheTTL: 2678400, // 31 days
  },
}
```

The expiration (or rather Max Age) of the optimized image is defined by either the `minimumCacheTTL` or the upstream image `Cache-Control` header, whichever is larger.

If you need to change the caching behavior per image, you can configure [`headers`](/docs/pages/api-reference/config/next-config-js/headers) to set the `Cache-Control` header on the upstream image (e.g. `/some-asset.jpg`, not `/_next/image` itself).

There is no mechanism to invalidate the cache at this time, so its best to keep `minimumCacheTTL` low. Otherwise you may need to manually change the [`src`](#src) prop or delete `<distDir>/cache/images`.

### 停用靜態導入

The default behavior allows you to import static files such as `import icon from './icon.png'` and then pass that to the `src` property.

In some cases, you may wish to disable this feature if it conflicts with other plugins that expect the import to behave differently.

You can disable static image imports inside your `next.config.js`:

```js filename="next.config.js"
module.exports = {
  images: {
    disableStaticImages: true,
  },
}
```

### 危險地允許 SVG

The default [loader](#loader) does not optimize SVG images for a few reasons. First, SVG is a vector format meaning it can be resized losslessly. Second, SVG has many of the same features as HTML/CSS, which can lead to vulnerabilities without proper [Content Security Policy (CSP) headers](/docs/app/api-reference/config/next-config-js/headers#content-security-policy).

Therefore, we recommended using the [`unoptimized`](#unoptimized) prop when the [`src`](#src) prop is known to be SVG. This happens automatically when `src` ends with `".svg"`.

However, if you need to serve SVG images with the default Image Optimization API, you can set `dangerouslyAllowSVG` inside your `next.config.js`:

```js filename="next.config.js"
module.exports = {
  images: {
    dangerouslyAllowSVG: true,
    contentDispositionType: 'attachment',
    contentSecurityPolicy: "default-src 'self'; script-src 'none'; sandbox;",
  },
}
```

In addition, it is strongly recommended to also set `contentDispositionType` to force the browser to download the image, as well as `contentSecurityPolicy` to prevent scripts embedded in the image from executing.

### `contentDispositionType`

The default [loader](#loader) sets the [`Content-Disposition`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Disposition#as_a_response_header_for_the_main_body) header to `attachment` for added protection since the API can serve arbitrary remote images.

The default value is `attachment` which forces the browser to download the image when visiting directly. This is particularly important when [`dangerouslyAllowSVG`](#dangerously-allow-svg) is true.

You can optionally configure `inline` to allow the browser to render the image when visiting directly, without downloading it.

```js filename="next.config.js"
module.exports = {
  images: {
    contentDispositionType: 'inline',
  },
}
```

### 動態圖片

The default [loader](#loader) will automatically bypass Image Optimization for animated images and serve the image as-is.

Auto-detection for animated files is best-effort and supports GIF, APNG, and WebP. If you want to explicitly bypass Image Optimization for a given animated image, use the [unoptimized](#unoptimized) prop.
