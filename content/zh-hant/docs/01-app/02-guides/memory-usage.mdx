隨著應用程式變得越來越大且功能日益豐富，在本地開發或建立生產環境版本時，它們可能需要更多的資源。

讓我們探索一些策略和技巧，以優化 Next.js 中的記憶體並處理常見的記憶體問題。

## 減少相依套件數量

擁有大量相依套件的應用程式會消耗更多的記憶體。

[Bundle Analyzer](/docs/app/guides/package-bundling) 可以幫助您調查應用程式中可能可以移除的大型相依套件，以改善效能和記憶體使用量。

## 嘗試 `experimental.webpackMemoryOptimizations`

從 `v15.0.0` 開始，您可以在 `next.config.js` 檔案中加入 `experimental.webpackMemoryOptimizations: true`，以改變 Webpack 的行為，從而降低最大記憶體使用量，但這可能會略微增加編譯時間。

> **值得了解**：此功能目前為實驗性質，旨在先在更多專案上進行測試，但被認為是低風險的。

## 使用 `--experimental-debug-memory-usage` 執行 `next build`

從 `14.2.0` 開始，您可以執行 `next build --experimental-debug-memory-usage`，以便在下一種模式下執行建構：在此模式下，Next.js 會在整個建構過程中持續列印關於記憶體使用量的資訊，例如堆記憶體使用量和垃圾收集統計資料。當記憶體使用量接近設定的限制時，也會自動建立堆快照。

> **值得了解**：此功能不與 Webpack 建構工作處理選項相容，除非您有自訂的 webpack 設定，否則該選項會自動啟用。

## 記錄堆疊配置檔 (heap profile)

為了尋找記憶體問題，您可以從 Node.js 記錄堆疊配置檔 (heap profile)，並在 Chrome DevTools 中載入它，以識別潛在的記憶體洩漏來源。

在終端機中，當啟動 Next.js 建構時，將 `--heap-prof` 參數傳遞給 Node.js：

```sh
node --heap-prof node_modules/next/dist/bin/next build
```

建構結束後，Node.js 將會建立一個 `.heapprofile` 檔案。

在 Chrome DevTools 中，您可以開啟 Memory 標籤頁，並點擊「Load Profile」按鈕來檢視該檔案。

## 分析堆快照 (heap snapshot)

您可以使用檢視工具來分析應用程式的記憶體使用量。

執行 `next build` 或 `next dev` 指令時，在指令開頭加入 `NODE_OPTIONS=--inspect`。這將在預設連接埠上啟用檢視工具代理程式。如果您希望在任何使用者程式碼開始執行的中斷執行點之前，可以改為傳遞 `--inspect-brk` 參數。當程式執行時，您可以使用 Chrome DevTools 等工具連接到調試連接埠，以記錄和分析堆快照，查看哪些記憶體被保留。

從 `14.2.0` 開始，您也可以使用 `--experimental-debug-memory-usage` 參數執行 `next build`，以更方便地建立堆快照。

在此模式下執行時，您可以在任何時間點向該程式傳送 `SIGUSR2` 訊號，該程式將會建立堆快照。

堆快照將儲存至 Next.js 應用程式的專案根目錄，並可在任何堆分析器（例如 Chrome DevTools）中載入，以查看保留的記憶體。此模式目前尚不與 Webpack 建構工作處理相容。

請見 [如何記錄和分析堆快照](https://developer.chrome.com/docs/devtools/memory-problems/heap-snapshots) 以獲取更多資訊。

## Webpack 建構工作處理

Webpack 建構工作處理允許您在獨立的 Node.js 工作處理中執行 Webpack 編譯，這將降低應用程式在建構期間的記憶體使用量。

從 `v14.1.0` 開始，如果您的應用程式沒有自訂的 Webpack 設定，此選項將預設啟用。

如果您使用的是較舊版本的 Next.js 或您有自訂的 Webpack 設定，您可以在 `next.config.js` 中設定 `experimental.webpackBuildWorker: true` 來啟用此選項。

> **值得了解**：此功能可能不與所有自訂 Webpack 外掛程式相容。

## 停用 Webpack 快取

[Webpack 快取](https://webpack.js.org/configuration/cache/) 會將產生的 Webpack 模組儲存於記憶體和/或磁碟中，以加快建構速度。這有助於效能，但也會增加應用程式儲存快取資料的記憶體使用量。

您可以透過向應用程式中加入 [自訂 Webpack 設定](/docs/app/api-reference/config/next-config-js/webpack) 來停用此行為：

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  webpack: (
    config,
    { buildId, dev, isServer, defaultLoaders, nextRuntime, webpack }
  ) => {
    if (config.cache && !dev) {
      config.cache = Object.freeze({
        type: 'memory',
      })
    }
    // Important: return the modified config
    return config
  },
}

export default nextConfig
```

## 停用靜態分析

類型檢查 (Typechecking) 可能需要大量記憶體，特別是在大型專案中。
然而，大多數專案都有專門的 CI 執行器 (CI runner) 來處理這些任務。
當建構在「執行 TypeScript」步驟期間產生記憶體不足問題時，您可以停用此任務：

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  typescript: {
    // !! WARN !!
    // Dangerously allow production builds to successfully complete even if
    // your project has type errors.
    // !! WARN !!
    ignoreBuildErrors: true,
  },
}

export default nextConfig
```

- [忽略 TypeScript 錯誤](/docs/app/api-reference/config/typescript#disabling-typescript-errors-in-production)

請記住，這可能會因類型錯誤而導致部署錯誤。
我們強烈建議僅在靜態分析完成後，才將建構發布至生產環境。
如果您部署至 Vercel，您可以查看 [暫存部署指南](https://vercel.com/docs/deployments/managing-deployments#staging-and-promoting-a-production-deployment) 以學習如何在自訂任務成功後將建構發布至生產環境。

## 停用原始碼地圖 (Source maps)

在建構過程中產生原始碼地圖會消耗額外的記憶體。

您可以在 Next.js 設定中加入 `productionBrowserSourceMaps: false` 和 `experimental.serverSourceMaps: false` 來停用原始碼地圖產生。

使用 `cacheComponents` 功能時，Next.js 預設會在 `next build` 的預渲染 (prerender) 階段使用來源映射 (source maps)。
如果在該階段（在「產生靜態頁面」(Generating static pages) 之後）持續遇到記憶體問題，
您可以嘗試透過在 Next.js 配置中新增 `enablePrerenderSourceMaps: false` 來停用該階段的來源映射。

> **值得了解**：某些外掛程式可能會開啟原始碼地圖，並可能需要自訂設定才能停用。

## Edge 記憶體問題

Next.js `v14.1.3` 已修復使用 Edge 執行階段時發生的記憶體問題。請更新到此版本（或更新版本），查看是否解決了您的問題。

## 預載項目 (Preloading Entries)

當 Next.js 伺服器啟動時，它會將每個頁面的 JavaScript 模組預先載入記憶體，而不是在請求時載入。

此最佳化可實現更快的回應時間，代價是較大的初始記憶體佔用。

要停用此最佳化，請將 `experimental.preloadEntriesOnStart` 旗標設定為 `false`。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const config: NextConfig = {
  experimental: {
    preloadEntriesOnStart: false,
  },
}

export default config
```

```js filename="next.config.mjs" switcher
/** @type {import('next').NextConfig} */
const config = {
  experimental: {
    preloadEntriesOnStart: false,
  },
}

export default config
```

Next.js 不會卸載這些 JavaScript 模組，這意味著即使停用此最佳化，如果所有頁面最終都被請求，您的 Next.js 伺服器的記憶體佔用最終也會相同。
