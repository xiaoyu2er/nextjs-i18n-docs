打包 (Bundling) 指的是將您的應用程式程式碼及其相依性組合為針對用戶端與伺服器的最佳化輸出文件的處理過程。較小的套件載入速度更快，減少 JavaScript 執行時間，改善 [核心網頁指標 (Core Web Vitals)](https://web.dev/articles/vitals)，並降低伺服器冷啟動時間。

Next.js 會透過程式碼分割 (code splitting)、搖樹 (tree-shaking) 和其他技術自動最佳化套件。不過，在某些情況下您可能需要手動最佳化您的套件。

有兩種工具可用於分析您的應用程式套件：

- [Next.js Bundle Analyzer for Turbopack（實驗性質）](#nextjs-bundle-analyzer-experimental)
- [`@next/bundle-analyzer` 插件，用於 Webpack](#nextbundle-analyzer-for-webpack)

本指南將帶您了解如何使用每個工具，以及如何 [優化大型套件](#optimizing-large-bundles)。

## Next.js Bundle Analyzer（實驗性質）

> 適用於 v16.1 及之後的版本。您可以在 [專屬的 GitHub 討論](https://github.com/vercel/next.js/discussions/86731) 中分享反饋，並在 [turbopack-bundle-analyzer-demo.vercel.sh](https://turbopack-bundle-analyzer-demo.vercel.sh/) 查看演示。

Next.js Bundle Analyzer 已整合至 Turbopack 的模組圖譜。您可以透過精確的匯入追蹤來檢查用戶端與伺服器模組，這讓尋找大型相依套件變得更容易。開啟互動式 Bundle Analyzer 演示以探索模組圖譜。

### 步驟 1：執行 Turbopack Bundle Analyzer

要開始使用，請執行以下命令並在瀏覽器開啟互動式視圖。

```bash filename="Terminal" package="npm"
npx next experimental-analyze
```

```bash filename="Terminal" package="yarn"
yarn next experimental-analyze
```

```bash filename="Terminal" package="pnpm"
pnpm next experimental-analyze
```

```bash filename="Terminal" package="bun"
bunx next experimental-analyze
```

### 步驟 2：篩選並檢查模組

在 UI 中，您可以依據路由、環境（用戶端或伺服器）和類型（JavaScript、CSS、JSON）進行篩選，或按檔案搜尋：

如果您想與隊友分享分析內容，或比對最佳化前後/後/前的套件大小，您可以跳過互動式視圖，並使用 `--output` 旗標將分析儲存為靜態檔案：

```bash filename="Terminal" package="npm"
npx next experimental-analyze --output
```

```bash filename="Terminal" package="yarn"
yarn next experimental-analyze --output
```

```bash filename="Terminal" package="pnpm"
pnpm next experimental-analyze --output
```

```bash filename="Terminal" package="bun"
bunx next experimental-analyze --output
```

```bash filename="Terminal"
cp -r .next/diagnostics/analyze ./analyze-before-refactor
```

## Webpack 的 `@next/bundle-analyzer`

[`@next/bundle-analyzer`](https://www.npmjs.com/package/@next/bundle-analyzer) 是一個外掛，可協助您管理應用程式套件的規模。它會為每個套件及其相依性產生視覺化報告。您可以利用這些資訊移除大型相依套件、分割，或 [延遲載入](/docs/app/guides/lazy-loading) 您的程式碼。

### 步驟 1：安裝

執行以下命令以安裝插件：

```bash package="pnpm"
pnpm add @next/bundle-analyzer
```

```bash package="npm"
npm install @next/bundle-analyzer
```

```bash package="yarn"
yarn add @next/bundle-analyzer
```

```bash package="bun"
bun add @next/bundle-analyzer
```

然後，在您的 `next.config.js` 中加入打包分析器的設定。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {}

const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})

module.exports = withBundleAnalyzer(nextConfig)
```

### 步驟 2：產生報告

執行以下命令以分析您的打包：

```bash
ANALYZE=true npm run build
# or
ANALYZE=true yarn build
# or
ANALYZE=true pnpm build
```

報表會在瀏覽器中開啟三個新分頁，您可以檢視它們。

## 優化大型套件

一旦您已識別出大型模組，解決方案將取決於您的使用案例。以下是常見原因及其修復方法：

### 具有多個導出的套件

如果您正在使用匯出數百個模組的套件（例如圖示和工具函式庫），您可以在 `next.config.js` 檔案中使用 [`optimizePackageImports`](/docs/app/api-reference/config/next-config-js/optimizePackageImports) 選項來最佳化這些匯入的解析方式。此選項僅會載入您 _實際上_ 使用的模組，同時仍提供撰寫具多個具名匯出的匯入陳述式的便利性。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    optimizePackageImports: ['icon-library'],
  },
}

module.exports = nextConfig
```

> **好知：** Next.js 也會自動最佳化某些函式庫，因此它們無需包含在 `optimizePackageImports` 清單中。請參見 [支援套件的完整清單](/docs/app/api-reference/config/next-config-js/optimizePackageImports)。

### 重型用戶端工作負載

用戶端套件龐大的常見原因之一，是在用戶端元件中執行昂貴的渲染工作。這通常發生在僅用於將資料轉換為 UI 的函式庫上，例如語法高亮顯示、圖表渲染或 Markdown 解析。

如果該工作不需要瀏覽器 API 或使用者互動，則可以在伺服器元件中執行。

在此範例中，基於 Prism 的高亮顯示功能在用戶端元件中執行。即使最終輸出只是 `<code>` 區塊，整個高亮顯示函式庫仍包含在用戶端 JavaScript 套件中：

```tsx filename="app/blog/[slug]/page.tsx"
'use client'

import Highlight from 'prism-react-renderer'
import theme from 'prism-react-renderer/themes/github'

export default function Page() {
  const code = `export function hello() {
    console.log("hi")
  }`

  return (
    <article>
      <h1>Blog Post Title</h1>

      {/* The prism package and its tokenization logic are shipped to the client */}
      <Highlight code={code} language="tsx" theme={theme}>
        {({ className, style, tokens, getLineProps, getTokenProps }) => (
          <pre className={className} style={style}>
            <code>
              {tokens.map((line, i) => (
                <div key={i} {...getLineProps({ line })}>
                  {line.map((token, key) => (
                    <span key={key} {...getTokenProps({ token })} />
                  ))}
                </div>
              ))}
            </code>
          </pre>
        )}
      </Highlight>
    </article>
  )
}
```

這會增加套件大小，因為用戶端必須下載並執行高亮顯示函式庫，即使結果是靜態 HTML。

相反地，將高亮顯示邏輯移至伺服器元件，並在伺服器上渲染最終 HTML。用戶端只會收到渲染過後的標記。

```tsx filename="app/blog/[slug]/page.tsx"
import { codeToHtml } from 'shiki'

export default async function Page() {
  const code = `export function hello() {
    console.log("hi")
  }`

  // The Shiki package runs on the server and is never bundled for the client.
  const highlightedHtml = await codeToHtml(code, {
    lang: 'tsx',
    theme: 'github-dark',
  })

  return (
    <article>
      <h1>Blog Post Title</h1>

      {/* Client receives plain markup */}
      <pre>
        <code dangerouslySetInnerHTML={{ __html: highlightedHtml }} />
      </pre>
    </article>
  )
}
```

### 將特定套件排除在打包之外

在伺服器元件和路由處理常式中匯入的套件會由 Next.js 自動打包。

您可以在 `next.config.js` 檔案中使用 [`serverExternalPackages`](/docs/app/api-reference/config/next-config-js/serverExternalPackages) 選項，將特定套件排除在打包之外。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  serverExternalPackages: ['package-name'],
}

module.exports = nextConfig
```

### 未預先打包的外部套件

預設情況下，匯入應用程式的套件不會被打包。如果外部套件未預先打包（例如，從 monorepo 或 `node_modules` 匯入），可能會影響效能。

要打包特定套件，您可以在 `next.config.js` 檔案中使用 [`transpilePackages`](/docs/app/api-reference/config/next-config-js/transpilePackages) 選項。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ['package-name'],
}

module.exports = nextConfig
```

要自動打包所有套件，您可以在 `next.config.js` 檔案中使用 [`bundlePagesRouterDependencies`](/docs/pages/api-reference/config/next-config-js/bundlePagesRouterDependencies) 選項。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  bundlePagesRouterDependencies: true,
}

module.exports = nextConfig
```

### 將特定套件排除在打包之外

如果您識別出不應包含在套件中的套件，您可以在 `next.config.js` 中使用 [`serverExternalPackages`](/docs/pages/api-reference/config/next-config-js/serverExternalPackages) 選項，將特定套件排除在自動打包之外：

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  // Automatically bundle external packages:
  bundlePagesRouterDependencies: true,
  // Opt specific packages out of bundling:
  serverExternalPackages: ['package-name'],
}

module.exports = nextConfig
```
