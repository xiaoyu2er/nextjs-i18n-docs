[React 伺服器組件 (React Server Components)](https://react.dev/reference/rsc/server-components) 可提升性能並簡化數據獲取，但也會改變數據的存取位置與方式，改變前端應用中處理數據的某些傳統安全假設。

本指南將協助您了解如何在 Next.js 中思考數據安全性，並如何實施最佳實踐。

## 數據獲取方法

在 Next.js 中獲取數據，我們推薦三種主要方法，取決於您專案的大小與使用年限：

- [HTTP API](#external-http-apis)：適用於既有大型應用與組織。
- [數據訪問層 (Data Access Layer)](#data-access-layer)：適用於新專案。
- [組件級數據存取 (Component-Level Data Access)](#component-level-data-access)：適用於原型與學習。

我們建議選擇其中一種數據獲取方法，避免混合使用。這可使開發人員與安全審核人員對您的程式碼庫預期更清楚。

### 外部 HTTP API

在既有專案中採用伺服器組件時，您應該遵循 **零信任 (Zero Trust)** 模型。您仍可以使用 [`fetch`](/docs/app/api-reference/functions/fetch) 從伺服器組件中呼叫現有 API 端點（如 REST 或 GraphQL），就像您在客戶端組件中那樣。

```tsx filename="app/page.tsx"
import { cookies } from 'next/headers'

export default async function Page() {
  const cookieStore = cookies()
  const token = cookieStore.get('AUTH_TOKEN')?.value

  const res = await fetch('https://api.example.com/profile', {
    headers: {
      Cookie: `AUTH_TOKEN=${token}`,
      // Other headers
    },
  })

  // ....
}
```

這種方法在以下情況下效果良好：

- 您已有既定的安全實踐。
- 後端團隊使用其他語言或獨立管理 API。

### 數據訪問層

對於新專案，我們建議建立專門的 **數據訪問層 (Data Access Layer)**。這是一種內部函式庫，控制數據如何獲取、何時獲取，以及什麼會傳遞到您的渲染環境。

數據訪問層應該：

- 僅在伺服器上執行。
- 執行授權檢查。
- 返回安全、最小化的 **數據傳輸物件 (Data Transfer Objects / DTOs)**。

這種方法集中所有數據存取邏輯，使強制一致性數據存取更容易，並減少授權錯誤的風險。您還可以享有在請求的不同部分共用記憶體快取的好處。

```ts filename="data/auth.ts"
import { cache } from 'react'
import { cookies } from 'next/headers'

// Cached helper methods makes it easy to get the same value in many places
// without manually passing it around. This discourages passing it from Server
// Component to Server Component which minimizes risk of passing it to a Client
// Component.
export const getCurrentUser = cache(async () => {
  const token = cookies().get('AUTH_TOKEN')
  const decodedToken = await decryptAndValidate(token)
  // Don't include secret tokens or private information as public fields.
  // Use classes to avoid accidentally passing the whole object to the client.
  return new User(decodedToken.id)
})
```

```tsx filename="data/user-dto.tsx"
import 'server-only'
import { getCurrentUser } from './auth'

function canSeeUsername(viewer: User) {
  // Public info for now, but can change
  return true
}

function canSeePhoneNumber(viewer: User, team: string) {
  // Privacy rules
  return viewer.isAdmin || team === viewer.team
}

export async function getProfileDTO(slug: string) {
  // Don't pass values, read back cached values, also solves context and easier to make it lazy

  // use a database API that supports safe templating of queries
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`
  const userData = rows[0]

  const currentUser = await getCurrentUser()

  // only return the data relevant for this query and not everything
  // <https://www.w3.org/2001/tag/doc/APIMinimization>
  return {
    username: canSeeUsername(currentUser) ? userData.username : null,
    phonenumber: canSeePhoneNumber(currentUser, userData.team)
      ? userData.phonenumber
      : null,
  }
}
```

```tsx filename="app/page.tsx"
import { getProfile } from '../../data/user'

export async function Page({ params: { slug } }) {
  // This page can now safely pass around this profile knowing
  // that it shouldn't contain anything sensitive.
  const profile = await getProfile(slug);
  ...
}
```

> **小提示：** 機密金鑰應儲存在環境變數中，但只有數據訪問層應存取 `process.env`。這可防止機密內容曝露於應用程式的其他部分。

### 組件級數據存取

對於快速原型與迭代，資料庫查詢可直接放置在伺服器組件中。

然而，這種方法更容易意外將私人資料曝露至客戶端，例如：

```tsx filename="app/page.tsx"
import Profile from './components/profile.tsx'

export async function Page({ params: { slug } }) {
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`
  const userData = rows[0]
  // EXPOSED: This exposes all the fields in userData to the client because
  // we are passing the data from the Server Component to the Client.
  return <Profile user={userData} />
}
```

```tsx filename="app/ui/profile.tsx"
'use client'

// BAD: This is a bad props interface because it accepts way more data than the
// Client Component needs and it encourages server components to pass all that
// data down. A better solution would be to accept a limited object with just
// the fields necessary for rendering the profile.
export default async function Profile({ user }: { user: User }) {
  return (
    <div>
      <h1>{user.name}</h1>
      ...
    </div>
  )
}
```

您應該在將資料傳遞至客戶端組件前進行資料清理：

```ts filename="data/user.ts"
import { sql } from './db'

export async function getUser(slug: string) {
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`
  const user = rows[0]

  // Return only the public fields
  return {
    name: user.name,
  }
}
```

```tsx filename="app/page.tsx"
import { getUser } from '../data/user'
import Profile from './ui/profile'

export default async function Page({
  params: { slug },
}: {
  params: { slug: string }
}) {
  const publicProfile = await getUser(slug)
  return <Profile user={publicProfile} />
}
```

## 讀取數據

### 從伺服器傳遞資料至客戶端

在初始載入時，伺服器與客戶端組件均在伺服器上運行以產生 HTML。但它們在隔離的模組系統中執行。這確保伺服器組件可存取私人資料與 API，而客戶端組件則不能。

**伺服器組件：**

- 僅在伺服器上運行。
- 可安全存取環境變數、機密金鑰、資料庫與內部 API。

**客戶端組件：**

- 在預渲染期間於伺服器上執行，但必須遵循與在瀏覽器中執行之程式碼相同的 security assumptions (安全假設)。
- 不得存取 privileged data 或 server-only 模組。

這確保應用程式預設為安全，但透過如何獲取資料或傳遞資料至組件的方式，仍可能意外曝露私人資料。

### 污染 (Tainting)

为防止私人資料意外曝露至客戶端，您可以使用 React Taint API：

- [`experimental_taintObjectReference`](https://react.dev/reference/react/experimental_taintObjectReference) 用於資料物件。
- [`experimental_taintUniqueValue`](https://react.dev/reference/react/experimental_taintUniqueValue) 用於特定值。

您可以在 Next.js 應用中透過 `next.config.js` 的 [`experimental.taint`](/docs/app/api-reference/config/next-config-js/taint) 選項啟用使用：

```js filename="next.config.js"
module.exports = {
  experimental: {
    taint: true,
  },
}
```

這可防止被污染的物件或值傳遞至客戶端。儘管如此，這是一種額外的保護層，您仍應在遞交至 React 的渲染環境前於 [數據訪問層 (DAL)](#data-access-layer) 中過濾與清理資料。

> **小提示：**
>
> - 預設情況下，環境變數僅在伺服器上使用。Next.js 會將任何以 `NEXT_PUBLIC_` 為前綴的環境變數曝露至客戶端。[了解更多](/docs/app/guides/environment-variables)。
> - 函式與類別已預設禁止傳遞至客戶端組件。

### 防止客戶端執行伺服器專有程式碼

为防止伺服器專有程式碼在客戶端執行，您可以使用 [`server-only`](https://www.npmjs.com/package/server-only) 套件標記模組：

```bash package="npm"
npm install server-only
```

```bash package="yarn"
yarn add server-only
```

```bash package="pnpm"
pnpm add server-only
```

```bash package="bun"
bun add server-only
```

```ts filename="lib/data.ts"
import 'server-only'

//...
```

這可確保自有程式碼或內部業務邏輯保留在伺服器上，若該模組在客戶端環境中被匯入則會導致建置錯誤。

## 修改數據

Next.js 使用 [伺服器操作 (Server Actions)](https://react.dev/reference/rsc/server-functions) 處理修改。

### 內建伺服器操作安全功能

預設情況下，當 Server Action 被建立並匯出時，它可透過直接的 POST 請求存取，而不僅透過您的應用程式 UI。這表示，即使 Server Action 或工具函式在您的程式碼其他地方未被匯入，它仍可能被外部呼叫。

為提升安全性，Next.js 具有以下內建功能：

- **安全操作 ID**：Next.js 建立加密、非確定性 ID，讓客戶端可參考與呼叫伺服器操作。這些 ID 在構建間會定期重新計算以提升安全性。
- **死碼消除**：未使用的伺服器操作（由其 ID 引用）會從客戶端套件中移除，以避免公開存取。

> **值得注意**：
>
> 這些 ID 是在編譯期間建立的，並快取最長 14 天。當啟動新構建或使構建快取無效時，它們將被重新生成。
> 此安全改進降低了在缺乏 authentication layer (驗證層) 時的風險。不過，您仍應將 Server Action 視為可透過直接 POST 請求存取，並在每個 Server Action 內部驗證 authentication (驗證) 和 authorization (授權)。

```jsx
// app/actions.js
'use server'

// If this action **is** used in our application, Next.js
// will create a secure ID to allow the client to reference
// and call the Server Action.
export async function updateUserAction(formData) {}

// If this action **is not** used in our application, Next.js
// will automatically remove this code during `next build`
// and will not create a public endpoint.
export async function deleteUserAction(formData) {}
```

### 客戶端輸入驗證

您應該始終驗證來自客戶端的輸入，因為它們容易被修改。例如：表單資料、URL 參數、標頭與 searchParams：

```tsx filename="app/page.tsx"
// BAD: Trusting searchParams directly
export default async function Page({ searchParams }) {
  const isAdmin = searchParams.get('isAdmin')
  if (isAdmin === 'true') {
    // Vulnerable: relies on untrusted client data
    return <AdminPanel />
  }
}

// GOOD: Re-verify every time
import { cookies } from 'next/headers'
import { verifyAdmin } from './auth'

export default async function Page() {
  const token = cookies().get('AUTH_TOKEN')
  const isAdmin = await verifyAdmin(token)

  if (isAdmin) {
    return <AdminPanel />
  }
}
```

### 身份驗證與授權

頁面層級的 authentication (驗證) 檢查不會延伸到在其中定義的 Server Actions。務必在 action 內部重新驗證：

```tsx filename="app/admin/page.tsx" highlight={13,14,15,16}
import { auth } from '@/lib/auth'
import { redirect } from 'next/navigation'

export default async function AdminPage() {
  const session = await auth()
  if (!session?.user?.isAdmin) {
    redirect('/login')
  }

  return (
    <form
      action={async () => {
        'use server'
        const session = await auth()
        if (!session?.user?.isAdmin) {
          throw new Error('Unauthorized')
        }
        await db.record.deleteMany()
      }}
    >
      <button>Delete Records</button>
    </form>
  )
}
```

action 內部強調的 `auth()` 檢查至關重要。第 6 行的頁面層級 redirect (重新導向) 控制呈現哪些 UI，但 Server Action 是獨立的 entry point (入口點)，必須自行驗證呼叫者。

除了 authentication (驗證)（使用者是否已登入？）之外，記得要檢查 **authorization** (授權)（此使用者是否擁有權限對特定資源執行動作？）。這可防止 [Insecure Direct Object Reference (IDOR)](https://cheatsheetseries.owasp.org/cheatsheets/Insecure_Direct_Object_Reference_Prevention_Cheat_Sheet.html) 漏洞：

```tsx filename="app/actions.ts"
'use server'

import { auth } from '@/lib/auth'
import { db } from '@/lib/db'

export async function deletePost(postId: string) {
  const session = await auth()
  if (!session?.user) {
    throw new Error('Unauthorized')
  }

  const post = await db.post.findUnique({ where: { id: postId } })

  // Check that the user owns this resource
  if (post.authorId !== session.user.id) {
    throw new Error('Forbidden')
  }

  await db.post.delete({ where: { id: postId } })
}
```

了解更多關於 [身份驗證 (Authentication)](/docs/app/guides/authentication) 的 Next.js 資訊。

### 使用 Data Access Layer (資料存取層) 處理 mutations

就像我們建議為讀取數據使用 [Data Access Layer (資料存取層)](#data-access-layer) 一樣，您可以將相同的模式應用於 mutations。這可將 authentication (驗證)、authorization (授權) 和資料庫邏輯保留在專屬的 `server-only` 模組中，同時 `"use server"` actions 保持簡潔。

```ts filename="data/posts.ts"
import 'server-only'

import { auth } from '@/lib/auth'
import { db } from '@/lib/db'

export async function deletePost(postId: string) {
  const session = await auth()
  if (!session?.user) {
    throw new Error('Unauthorized')
  }

  const post = await db.post.findUnique({ where: { id: postId } })

  if (post.authorId !== session.user.id) {
    throw new Error('Forbidden')
  }

  await db.post.delete({ where: { id: postId } })
}
```

接著，`"use server"` action 會委派給 DAL：

```ts filename="app/actions.ts"
'use server'

import { deletePost } from '@/data/posts'
import { revalidatePath } from 'next/cache'

export async function deletePostAction(postId: string) {
  await deletePost(postId) // Auth + authz happen inside the DAL
  revalidatePath('/posts')
}
```

> **值得注意**：您可以在 Data Access Layer 和 `"use server"` 檔案本身使用 `import 'server-only'`。當 action 被匯入 Client Component（例如，傳遞給 `useActionState`）時，兩者皆可運作，因為 `"use server"` 模組是在 server-only webpack layer 中解析的。

### 控制回傳值

Server Action 回傳值會被序列化和傳送給客戶端。僅回傳 UI 需要的內容，而非原始資料庫記錄。

```tsx filename="app/actions.ts"
'use server'

import { auth } from '@/lib/auth'
import { db } from '@/lib/db'

// BAD: Returns the full database record, which may include
// internal fields the client should not see.
export async function updateUser(data: FormData) {
  const session = await auth()
  if (!session?.user) {
    throw new Error('Unauthorized')
  }
  return db.user.update({
    where: { id: session.user.id },
    data: { name: data.get('name') as string },
  })
}

// GOOD: Returns only what the client needs.
export async function updateUserSafe(data: FormData) {
  const session = await auth()
  if (!session?.user) {
    throw new Error('Unauthorized')
  }
  await db.user.update({
    where: { id: session.user.id },
    data: { name: data.get('name') as string },
  })
  return { success: true }
}
```

### 速率限制

對於高成本的作業（寄送電子郵件、寫入資料庫），請考慮新增 rate limiting 以防止濫用。詳見 [Rate limiting](/docs/app/guides/backend-for-frontend#rate-limiting) 範例於 Backend for Frontend 指南中。

### 閉包與加密

在組件中定義伺服器操作會建立 [閉包 (closure)](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures)，允許操作存取外部函式的範圍。例如，`publish` 操作可存取 `publishVersion` 變數：

```tsx filename="app/page.tsx" switcher
export default async function Page() {
  const publishVersion = await getLatestVersion();

  async function publish() {
    "use server";
    if (publishVersion !== await getLatestVersion()) {
      throw new Error('The version has changed since pressing publish');
    }
    ...
  }

  return (
    <form>
      <button formAction={publish}>Publish</button>
    </form>
  );
}
```

```jsx filename="app/page.js" switcher
export default async function Page() {
  const publishVersion = await getLatestVersion();

  async function publish() {
    "use server";
    if (publishVersion !== await getLatestVersion()) {
      throw new Error('The version has changed since pressing publish');
    }
    ...
  }

  return (
    <form>
      <button formAction={publish}>Publish</button>
    </form>
  );
}
```

閉包在您需要在呈現時捕捉資料的 _快照_（例如 `publishVersion`），以便在操作被呼叫時使用時很有用。

然而，為達到此目的，捕捉的變數會在操作被呼叫時傳送至客戶端並返回至伺服器。為防止敏感資料曝露至客戶端，Next.js 會自動加密被閉包的變數。每次 Next.js 應用程式建置時，每個操作會重新生成新的私密金鑰。這意味著操作只能為特定構建呼叫。

> **小提示：** 我們不建議單獨依賴加密來防止敏感值在客戶端曝露。

### 覆寫加密金鑰（進階）

當在多台伺服器上 **self-hosting** Next.js 應用程式時，每個伺服器例項最後可能擁有不同的加密金鑰，導致潛在的不一致。

為減輕這個問題，您可以使用 `process.env.NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 環境變數覆寫加密金鑰。指定此變數可確保您的加密金鑰在構建之間保持一致，並且所有伺服器例項使用相同的金鑰。

金鑰必須是 base64 編碼的值，其解碼長度必須符合有效的 AES 金鑰大小（16、24 或 32 位元組）。Next.js 預設生成 32 位元組的金鑰。您可以使用平台上的加密工具生成相容的金鑰，例如：

```bash
openssl rand -base64 32
```

這是一個進階使用案例，在此案例中，跨多個部署的加密行為一致性對於您的應用程式至關重要。請遵循標準的安全實踐，例如金鑰旋轉和簽署。請參閱 [Self-Hosting guide](/docs/app/guides/self-hosting#server-functions-encryption-key) 以了解部署特定的考量。

### 允許來源（進階）

由於伺服器操作可在 `<form>` 元素中呼叫，此操作將它們暴露於 [CSRF 攻擊 (CSRF attacks)](https://developer.mozilla.org/en-US/docs/Glossary/CSRF)。

在後台，伺服器操作使用 `POST` 方法，且僅此 HTTP 方法可用以呼叫它們。這可防止現代瀏覽器中大多數 CSRF 漏洞，特別是當 [SameSite  Cookie](https://web.dev/articles/samesite-cookies-explained) 為預設時。

作為額外保護，Next.js 中的伺服器操作也會將 [來源標頭 (Origin header)](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Origin) 與 [Host 標頭 (Host header)](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Host)（或 `X-Forwarded-Host`）進行比較。若不匹配，請求將被中止。換句話說，伺服器操作只能在與其頁面相同的主機上呼叫。

對於使用反向代理伺服器或多層後端架構的大型應用程式（伺服器 API 與生產網域不同），建議使用配置選項 [`serverActions.allowedOrigins`](/docs/app/api-reference/config/next-config-js/serverActions) 指定安全來源列表。選項接受字串陣列。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
module.exports = {
  experimental: {
    serverActions: {
      allowedOrigins: ['my-proxy.com', '*.my-proxy.com'],
    },
  },
}
```

了解更多關於 [安全性與伺服器操作](https://nextjs.org/blog/security-nextjs-server-components-actions) 的資訊。

### 避免在呈現期間產生副作用

修改（例如登出使用者、更新資料庫、無效化快取）永遠不應產生副作用，在伺服器或客戶端組件中皆然。Next.js 明確禁止在呈現方法中設定 Cookie 或觸發快取重新驗證，以避免非預期的副作用。

```tsx filename="app/page.tsx"
// BAD: Triggering a mutation during rendering
export default async function Page({ searchParams }) {
  if (searchParams.get('logout')) {
    cookies().delete('AUTH_TOKEN')
  }

  return <UserProfile />
}
```

相反，您應該使用伺服器操作來處理修改。

```tsx filename="app/page.tsx"
// GOOD: Using Server Actions to handle mutations
import { logout } from './actions'

export default function Page() {
  return (
    <>
      <UserProfile />
      <form action={logout}>
        <button type="submit">Logout</button>
      </form>
    </>
  )
}
```

> **小提示：** Next.js 使用 `POST` 請求處理修改。這可防止從 GET 請求產生意外的副作用，減少跨站請求偽造 (CSRF) 風險。

## 安全評估

如果您正在對 Next.js 專案進行審計，以下是我們建議額外注意的一些事項：

- **Data Access Layer (資料存取層)**: 針對獨立的 Data Access Layer，是否已有既定的最佳實踐？確認資料庫套件和環境變數未在 Data Access Layer 外匯入。
- **`"use client"` 檔案**: Component 屬性 (props) 是否期待私有資料？Type signatures 是否過於寬泛？
- **`"use server"` 檔案**: Action 參數是在 action 內或 Data Access Layer 內部驗證的嗎？使用者是否是在 action 內部重新授權的？Action 是否檢查資源的所有權（authorization (授權)、authentication (驗證)）？回傳值是否已被過濾，僅保留客戶端 (Client) 需要的部分？資料庫存取是否已委派給 `server-only` Data Access Layer？
- **`/[param]/.`**: 使用括號的資料夾代表使用者輸入。參數 (params) 有經過驗證嗎？
- **`proxy.ts` 和 `route.ts`**: 擁有極大的權限。多花時間使用傳統技術審核這些檔案。定期或在符合團隊軟體開發生命週期時執行 Penetration Testing (滲透測試) 或 Vulnerability Scanning (弱點掃描)。
