公共頁面為每位用戶顯示相同的內容。常見範例包括landing pages、行銷頁面和產品頁面。

由於數據是共享的，這類頁面可以提前 [預渲染 (prerendering)](/docs/app/glossary#prerendering) 並重複使用。這能加快頁面加載速度並降低服務器成本。

本指南將向您展示如何構建在不同用戶之間共享數據的公共頁面。

## 範例

作為範例，我們將構建一個產品列表頁面。

我們將從一個靜態 Header 開始，添加帶有異步外部數據的產品列表，並學習如何在不阻塞響應的情況下渲染它。最後，我們將添加工戶特定的促銷橫幅，而不用將整個頁面切換到 [動態渲染 (dynamic rendering)](/docs/app/glossary#dynamic-rendering)。

您可以在這裡找到本例中使用的資源：

- [影片](https://youtu.be/F6romq71KtI)
- [演示](https://cache-components-public-pages.labs.vercel.dev/)
- [代碼](https://github.com/vercel-labs/cache-components-public-pages)

### 步驟 1：添加簡單 Header

讓我們從簡單的 Header 開始。

```tsx filename="app/products/page.tsx"
// Static component
function Header() {
  return <h1>Shop</h1>
}

export default async function Page() {
  return (
    <>
      <Header />
    </>
  )
}
```

#### 靜態 (Static) 組件

The `<Header />` 組件不依賴於任何在請求之間會變化的輸入，例如：外部數據、請求標頭、路由參數、當前時間或隨機值。

由於其輸出從永不會改變且可以提前確定，這類組件被稱為 **靜態 (static)** 組件。由於無需等待請求，Next.js 可以在 [構建時間 (build time)](/docs/app/glossary#build-time) 安全地 **預渲染 (prerender)** 頁面。

我們可以通過運行 [`next build`](/docs/app/api-reference/cli/next#next-build-options) 來確認這一點。

```bash filename="Terminal"
Route (app)      Revalidate  Expire
┌ ○ /products           15m      1y
└ ○ /_not-found

○  (Static)  prerendered as static content
```

請注意，產品路由被標記為靜態，即使我們沒有添加任何明確配置。

### 步驟 2：添加產品列表

現在，讓我們獲取並渲染我們的產品列表。

```tsx filename="app/products/page.tsx"
import db from '@/db'
import { List } from '@/app/products/ui'

function Header() {}

// Dynamic component
async function ProductList() {
  const products = await db.product.findMany()
  return <List items={products} />
}

export default async function Page() {
  return (
    <>
      <Header />
      <ProductList />
    </>
  )
}
```

與 Header 不同，產品列表依賴於外部數據。

#### 動態 (Dynamic) 組件

由於此數據在未來會變化，渲染後的輸出不再保證穩定。這使得產品列表成為一個 **動態 (dynamic)** 組件。

在沒有指導的情況下，框架假設您希望在每次用戶請求時獲取 **最新 (fresh)** 數據。此設計選擇反映了標準 Web 行為，即新的服務器請求渲染頁面。

但是，如果此組件在請求時渲染，獲取其數據將延遲 **整個 (entire)** 路由的響應。如果我們刷新頁面，我們可以看到這發生。

即使 Header 是即時渲染的，但在產品列表完成獲取之前，它無法發送給瀏覽器。

為了保護我們免受此性能崩潰的影響，Next.js 會在我們第一次 **await (await)** 數據時顯示 [警告](/docs/messages/blocking-route)：`Blocking data was accessed outside of Suspense`

此時，我們必須決定如何 **解鎖 (unblock)** 響應。或者：

- [**緩存 (Cache)**](/docs/app/glossary#cache-components) 該組件，使其成為 **穩定 (stable)** 的，並能與頁面的其餘部分一起被預渲染。
- [**流式 (Stream)**](/docs/app/glossary#streaming) 該組件，使其成為 **非阻塞 (non-blocking)** 的，頁面的其餘部分無需等待。

在我們的情況下，產品目錄在所有用戶之間共享，因此緩存是正確的選擇。

### 緩存 (Cache) 組件

我們可以通過 [`'use cache'`](/docs/app/api-reference/directives/use-cache) 指令將函數標記為可緩存。

```tsx filename="app/products/page.tsx"
import db from '@/db'
import { List } from '@/app/products/ui'

function Header() {}

// Cache component
async function ProductList() {
  'use cache'
  const products = await db.product.findMany()
  return <List items={products} />
}

export default async function Page() {
  return (
    <>
      <Header />
      <ProductList />
    </>
  )
}
```

這將其變為 [緩存組件 (cache component)](/docs/app/glossary#cache-components)。第一次運行時，我們返回的任何內容都將被緩存並重用。

如果緩存組件的輸入在請求到達 **之前 (before)** 可用，它可以像靜態組件一樣被預渲染。

如果我們再次刷新，可以看到頁面加載瞬間完成，因為緩存組件不會阻塞響應。並且，如果我們再次運行 `next build`，我們可以確認頁面仍然是靜態的：

```bash filename="Terminal"
Route (app)      Revalidate  Expire
┌ ○ /products           15m      1y
└ ○ /_not-found

○  (Static)  prerendered as static content
```

但是，頁面很少永遠保持靜態。

### 步驟 3：添加動態促銷橫幅

遲早，即使是簡單頁面也需要一些動態內容。為了演示這一點，讓我們添加一個促銷橫幅：

```tsx filename="app/products/page.tsx"
import db from '@/db'
import { List, Promotion } from '@/app/products/ui'
import { getPromotion } from '@/app/products/data'

function Header() {}

async function ProductList() {}

// Dynamic component
async function PromotionContent() {
  const promotion = await getPromotion()
  return <Promotion data={promotion} />
}

export default async function Page() {
  return (
    <>
      <PromotionContent />
      <Header />
      <ProductList />
    </>
  )
}
```

再次，它從動態開始。和之前一樣，引入阻塞行為會觸發 Next.js 警告。

上次，數據是共享的，所以可以緩存。這次，促銷取決於特定請求的輸入，如用戶位置和 A/B 測試，所以我們無法通過緩存來消除阻塞行為。

### 部分預渲染 (Partial Prerendering)

添加動態內容並不意味著我們必須回到完全阻塞的渲染。我們可以使用流式傳輸來解除響應的阻塞。

Next.js 默認支持流式傳輸。我們可以使用 [Suspense boundary (掛起邊界)](/docs/app/glossary#suspense-boundary) 來告訴框架在哪裡將流式響應拆分成 _chunks_，以及在內容加載時顯示什麼回退 UI。

```tsx filename="app/products/page.tsx"
import { Suspense } from 'react'
import db from '@/db'
import { List, Promotion, PromotionSkeleton } from '@/app/products/ui'
import { getPromotion } from '@/app/products/data'

function Header() {}

async function ProductList() {}

// Dynamic component (streamed)
async function PromotionContent() {
  const promotion = await getPromotion()
  return <Promotion data={promotion} />
}

export default async function Page() {
  return (
    <>
      <Suspense fallback={<PromotionSkeleton />}>
        <PromotionContent />
      </Suspense>
      <Header />
      <ProductList />
    </>
  )
}
```

回退 UI 與我們靜態和緩存內容的其餘部分一起預渲染。內部組件稍後流式傳輸，一旦其異步工作完成。

通過此更改，Next.js 可以將可預渲染的工作與請求時的工作分開，路由變為 [部分預渲染 (partially prerendered)](/docs/app/glossary#partial-prerendering-ppr)。

同樣，我們可以通過運行 `next build` 來確認這一點：

```bash filename="Terminal"
Route (app)      Revalidate  Expire
┌ ◐ /products    15m      1y
└ ◐ /_not-found

◐  (Partial Prerender)  Prerendered as static HTML with dynamic server-streamed content
```

在 [**構建時間 (build time)**](/docs/app/glossary#build-time)，大多數頁面內容，包括 Header、產品列表和促銷回退，均已渲染、緩存並推送到內容傳輸網絡。

在 [**請求時間 (request time)**](/docs/app/glossary#dynamic-rendering)，預渲染部分從靠近用戶的 CDN 節點即時提供。

與此同時，用戶特定的促銷在服務器上渲染，流式傳輸到客戶端，並替換放入回退槽位中。

如果我們最後一次刷新頁面，可以看到大部分頁面加載瞬間完成，同時動態部分在變得可用時流式傳輸進入。
