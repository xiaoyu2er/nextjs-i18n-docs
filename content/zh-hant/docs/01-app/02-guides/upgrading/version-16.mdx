## 從 15 升級至 16

### 使用 Next.js 開發工具 MCP 與 AI 智慧代理

如果您使用的 AI 程式碼助理支援 [模型上下文協議 (MCP)](https://modelcontextprotocol.io)，則可以使用 **Next.js 開發工具 MCP** 來自動化升級流程和遷移任務。

#### 設定

將以下配置添加到您的 MCP 客戶端，對於每個程式碼助理，您可以閱讀 [此部分](https://github.com/vercel/next-devtools-mcp#mcp-client-configuration) 以獲取配置詳情。

**範例：**

```json filename=".mcp.json"
{
  "mcpServers": {
    "next-devtools": {
      "command": "npx",
      "args": ["-y", "next-devtools-mcp@latest"]
    }
  }
}
```

如需更多資訊，請造訪 [`next-devtools-mcp`](https://github.com/vercel/next-devtools-mcp) 文件以與您的 MCP 客戶端進行配置。

> **注意：** 使用 `next-devtools-mcp@latest` 確保您的 MCP 客戶端始終使用最新的 Next.js 開發工具 MCP 伺服器版本。

#### 範例提示

完成配置後，您可以使用自然語言提示來升級您的 Next.js 應用程式：

**升級至 Next.js 16：**

連接到您的程式碼助理，然後發出提示：

```txt
Next Devtools, help me upgrade my Next.js app to version 16
```

**遷移至快取元件（升級至 v16 後）：**

連接到您的程式碼助理，然後發出提示：

```txt
Next Devtools, migrate my Next.js app to cache components
```

請在文件中 [此處](/docs/app/guides/mcp) 了解更多。

### 使用 Codemod

要更新為 Next.js 版本 16，您可以使用 `upgrade` [codemod](/docs/app/guides/upgrading/codemods#160)：

```bash package="pnpm"
pnpm dlx @next/codemod@canary upgrade latest
```

```bash package="npm"
npx @next/codemod@canary upgrade latest
```

```bash package="yarn"
yarn dlx @next/codemod@canary upgrade latest
```

```bash package="bun"
bunx @next/codemod@canary upgrade latest
```

[codemod](/docs/app/guides/upgrading/codemods#160) 能夠：

- 更新 `next.config.js` 以使用新的 `turbopack` 配置
- 從 `next lint` 遷移至 ESLint CLI
- 從已棄用的 `middleware` 慣例遷移至 `proxy`
- 從已穩定化的 API 中移除 `unstable_` 前綴
- 從頁面和佈局中移除 `experimental_ppr` 路由段配置

如果您偏好手動操作，請安裝最新的 Next.js 和 React 版本：

```bash package="pnpm"
pnpm add next@latest react@latest react-dom@latest
```

```bash package="npm"
npm install next@latest react@latest react-dom@latest
```

```bash package="yarn"
yarn add next@latest react@latest react-dom@latest
```

```bash package="bun"
bun add next@latest react@latest react-dom@latest
```

如果您使用 TypeScript，請確保您也將 `@types/react` 和 `@types/react-dom` 升級至最新版本。

## 預設使用 Turbopack

從 **Next.js 16** 開始，Turbopack 已穩定並預設搭配 `next dev` 和 `next build` 使用

之前您必須使用 `--turbopack` 或 `--turbo` 啟用 Turbopack。

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev --turbopack",
    "build": "next build --turbopack",
    "start": "next start"
  }
}
```

這已不再必要。您可以更新您的 `package.json` 腳本：

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start"
  }
}
```

如果您的專案擁有 [自訂 `webpack`](/docs/app/api-reference/config/next-config-js/webpack) 配置並執行 `next build`（現預設使用 Turbopack），建構將**失敗**以防止配置問題。

您有幾種不同的方式來解決這個問題：

- **無論如何使用 Turbopack：** 使用 `next build --turbopack` 執行以使用 Turbopack 進行建構並忽略您的 `webpack` 配置。
- **完全切換至 Turbopack：** 將您的 `webpack` 配置遷移至 Turbopack 相容的選項。
- **繼續使用 Webpack：** 使用 `--webpack` 標誌以退出 Turbopack 並使用 Webpack 進行建構。

> **了解重點**：如果您看到因找到 `webpack` 配置而失敗的建構，但您自己沒有定義，這很可能是外掛正在添加 `webpack` 選項

### 退出 Turbopack

如果您需要繼續使用 Webpack，可以使用 `--webpack` 標誌退出。例如，要在開發時使用 Turbopack 但生產建構使用 Webpack：

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build --webpack",
    "start": "next start"
  }
}
```

我們建議開發和生產都使用 Turbopack。如果您無法切換至 Turbopack，請在此 [討論串](https://github.com/vercel/next.js/discussions/77721) 提交意見。

### Turbopack 配置位置

`experimental.turbopack` 配置已結束實驗階段。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

// Next.js 15 - experimental.turbopack
const nextConfig: NextConfig = {
  experimental: {
    turbopack: {
      // options
    },
  },
}

export default nextConfig
```

您可以將其作為頂層 `turbopack` 選項使用：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

// Next.js 16 - turbopack at the top level of nextConfig
const nextConfig: NextConfig = {
  turbopack: {
    // options
  },
}

export default nextConfig
```

請務必檢視 `Turbopack` 配置 [選項](/docs/app/api-reference/config/next-config-js/turbopack)。**Next.js 16** 引入了各種改進和新選項，例如：

- [進階 Webpack 載入器條件](/docs/app/api-reference/config/next-config-js/turbopack#advanced-webpack-loader-conditions)
- [debugIds](/docs/app/api-reference/config/next-config-js/turbopack#debug-ids)

### 解析別名後備

在某些專案中，客戶端程式碼可能會匯入包含 Node.js 原生模組的檔案。這將導致 `找不到模組：無法解析 'fs'` 類型的錯誤。

當發生此情況時，您應該重構您的程式碼，使您的客戶端打包程式不引用這些 Node.js 原生模組。

然而，在某些情況下，這可能不可能。在 Webpack 中，`resolve.fallback` 選項通常用於**靜音**該錯誤。Turbopack 提供了一個類似的選項，使用 `turbopack.resolveAlias`。在這種情況下，告訴 Turbopack 當為瀏覽器請求 `fs` 時載入空模組。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  turbopack: {
    resolveAlias: {
      fs: {
        browser: './empty.ts', // We recommend to fix code imports before using this method
      },
    },
  },
}

export default nextConfig
```

最好重構您的模組，使客戶端程式碼從不使用 Node.js 原生模組的模組中匯入。

### Sass node_modules 導入

Turbopack 完全支援從 `node_modules` 匯入 Sass 檔案。請注意，雖然 Webpack 允許舊版波浪號 (`~`) 前綴，但 Turbopack 不支援此語法。

在 Webpack 中：

```scss filename="styles/globals.scss"
@import '~bootstrap/dist/css/bootstrap.min.css';
```

在 Turbopack 中：

```scss filename="styles/globals.scss"
@import 'bootstrap/dist/css/bootstrap.min.css';
```

如果無法變更匯入，您可以使用 `turbopack.resolveAlias`。例如：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  turbopack: {
    resolveAlias: {
      '~*': '*',
    },
  },
}

export default nextConfig
```

### Turbopack 檔案系統快取 ( beta )

Turbopack 現在支援開發時的檔案系統快取，在執行之間將編譯器產物儲存在磁碟上，使重啟時的編譯時間顯著加快。

在您的配置中啟用檔案系統快取：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
}

module.exports = nextConfig
```

## 非同步請求 API (重大變更)

版本 15 輸入了 [Async Request APIs](https://nextjs.org/docs/app/guides/upgrading/version-15#async-request-apis-breaking-change) 作為重大變更，並具有 **临时** 的同步相容性。

從 **Next.js 16** 開始，同步存取已完全移除。這些 API 只能以非同步方式存取。

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout)、[`page.js`](/docs/app/api-reference/file-conventions/page)、[`route.js`](/docs/app/api-reference/file-conventions/route)、[`default.js`](/docs/app/api-reference/file-conventions/default)、[`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image#opengraph-image)、[`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image#twitter-image)、[`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons#icon) 和 [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons#apple-icon) 中的 `params`。
- [`page.js`](/docs/app/api-reference/file-conventions/page) 中的 `searchParams`

使用 [codemod](/docs/app/guides/upgrading/codemods#migrate-to-async-dynamic-apis) 遷移至非同步請求時 API。

### 遷移至非同步請求時 API 的類型

為了協助遷移至非同步 `params` 和 `searchParams`，您可以執行 [`npx next typegen`](/docs/app/api-reference/cli/next#next-typegen-options) 來自動產生這些全域可用類型的輔助：

- [`PageProps`](/docs/app/api-reference/file-conventions/page#page-props-helper)
- [`LayoutProps`](/docs/app/api-reference/file-conventions/layout#layout-props-helper)
- [`RouteContext`](/docs/app/api-reference/file-conventions/route#route-context-helper)

> **了解重點**：`typegen` 引入於 Next.js 15.5

這簡化了新非同步 API 模式的類型安全遷移，並使您能夠以完整的類型安全性更新您的元件，例如：

```tsx filename="/app/blog/[slug]/page.tsx"
export default async function Page(props: PageProps<'/blog/[slug]'>) {
  const { slug } = await props.params
  const query = await props.searchParams
  return <h1>Blog Post: {slug}</h1>
}
```

此方法使您能夠在頁面內直接完全類型安全地存取 `props.params`（包括 `slug`）和 `searchParams`。

## 圖示和 Open Graph 圖片的非同步參數 (重大變更)

> 傳遞給 `opengraph-image`、`twitter-image`、`icon` 和 `apple-icon` 中的圖片產生函式的 props 現在是 Promise。

在之前的版本中，`Image`（圖片產生函式）和 `generateImageMetadata` 都接收 `params` 物件。`generateImageMetadata` 返回的 `id` 作為字串傳遞給圖片產生函式。

```js filename="app/shop/[slug]/opengraph-image.js"
// Next.js 15 - synchronous params access
export function generateImageMetadata({ params }) {
  const { slug } = params
  return [{ id: '1' }, { id: '2' }]
}

// Next.js 15 - synchronous params and id access
export default function Image({ params, id }) {
  const slug = params.slug
  const imageId = id // string
  // ...
}
```

從 **Next.js 16** 開始，為了配合 [Async Request APIs](#async-request-apis-breaking-change) 變更，圖片產生函式現在接收作為 Promise 的 `params` 和 `id`。`generateImageMetadata` 函式繼續接收同步 `params`。

```js filename="app/shop/[slug]/opengraph-image.js"
export async function generateImageMetadata({ params }) {
  const { slug } = params
  return [{ id: '1' }, { id: '2' }]
}

// Next.js 16 - asynchronous params and id access
export default async function Image({ params, id }) {
  const { slug } = await params // params now async
  const imageId = await id // id is now Promise<string> when using generateImageMetadata
  // ...
}
```

## Sitemap 的非同步 `id` 參數 (重大變更)

之前，從 [`generateSitemaps`](/docs/app/api-reference/functions/generate-sitemaps) 返回的 `id` 值直接傳遞給 `sitemap` 產生函式。

```js filename="app/product/sitemap.js"
export async function generateSitemaps() {
  return [{ id: 0 }, { id: 1 }, { id: 2 }, { id: 3 }]
}

// Next.js 15 - synchronous id access
export default async function sitemap({ id }) {
  const start = id * 50000 // id is a number
  // ...
}
```

從 **Next.js 16** 開始，`sitemap` 產生函式現在接收 `id` 作為 Promise。

```js filename="app/product/sitemap.js"
export async function generateSitemaps() {
  return [{ id: 0 }, { id: 1 }, { id: 2 }, { id: 3 }]
}

// Next.js 16 - asynchronous id access
export default async function sitemap({ id }) {
  const resolvedId = await id // id is now Promise<string>
  const start = Number(resolvedId) * 50000
  // ...
}
```

## React 編譯器支援

隨著 React 編譯器 1.0 發布，React 編譯器的內建支援在 **Next.js 16** 中已穩定。React 編譯器會自動快取元件，以零手動程式碼變更減少不必要的重新渲染。

`reactCompiler` 配置選項已從 `experimental` 提升至穩定。由於我們繼續在不同應用程式類型中收集建構效能資料，它預設未啟用。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

  reactCompiler: true,
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  reactCompiler: true,
}

module.exports = nextConfig
```

## 快取 API

### revalidateTag

[`revalidateTag`](/docs/app/api-reference/functions/revalidateTag) 具有新的函式簽章。您可以將 [`cacheLife`](/docs/app/api-reference/functions/cacheLife#reference) 配置文件作為第二個參數傳遞。

```ts filename="app/actions.ts" switcher
'use server'

import { revalidateTag } from 'next/cache'

export async function updateArticle(articleId: string) {
  // Mark article data as stale - article readers see stale data while it revalidates
  revalidateTag(`article-${articleId}`, 'max')
}
```

```js filename="app/actions.js" switcher
'use server'

import { revalidateTag } from 'next/cache'

export async function updateArticle(articleId) {
  // Mark article data as stale - article readers see stale data while it revalidates
  revalidateTag(`article-${articleId}`, 'max')
}
```

對於允許更新略有延遲的內容（如部落格文章、產品目錄或文件），請使用 `revalidateTag`。使用者會在背景載入新資料時收到較舊的內容。

### updateTag

[`updateTag`](/docs/app/api-reference/functions/updateTag) 是一個新的僅限 [Server Actions](/docs/app/getting-started/mutating-data#what-are-server-functions) 的 API，提供 **讀取寫入** 語意，使用者進行變更後，UI 會立即顯示變更，而不是較舊的資料。

它透過在相同請求內使資料過期並立即重新整理來達成此目的。

```ts filename="app/actions.ts" switcher
'use server'

import { updateTag } from 'next/cache'

export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile)

  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`)
}
```

```js filename="app/actions.js" switcher
'use server'

import { updateTag } from 'next/cache'

export async function updateUserProfile(userId, profile) {
  await db.users.update(userId, profile)

  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`)
}
```

這確保了互動功能立即反映更改。適合表單、用戶設定以及用戶期望立即看到更新的任何工作流程。

請於 [此處](/docs/app/api-reference/functions/updateTag#when-to-use-updatetag) 了解更多關於何時使用 `updateTag` 或 `revalidateTag` 的資訊。

### refresh

[`refresh`](/docs/app/api-reference/functions/refresh) 允許您在 Server Action 內部重新整理客戶端路由器。

```ts filename="app/actions.ts" switcher
'use server'

import { refresh } from 'next/cache'

export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId)

  // Refresh the notification count displayed in the header
  refresh()
}
```

```js filename="app/actions.js" switcher
'use server'

import { refresh } from 'next/cache'

export async function markNotificationAsRead(notificationId) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId)

  // Refresh the notification count displayed in the header
  refresh()
}
```

當您需要在执行動作後重新整理客戶端路由器時使用它。

### cacheLife 和 cacheTag

[`cacheLife`](/docs/app/api-reference/functions/cacheLife) 和 [`cacheTag`](/docs/app/api-reference/functions/cacheTag) 現在已穩定。 `unstable_` 前綴不再需要。

無論您在哪裡使用過類似於以下的別名匯入：

```ts
import {
  unstable_cacheLife as cacheLife,
  unstable_cacheTag as cacheTag,
} from 'next/cache'
```

您可以更新您的匯入為：

```ts
import { cacheLife, cacheTag } from 'next/cache'
```

## 增強的路由和導航

**Next.js 16** 包含路由和導航系統的全面重構，使頁面轉換更輕盈和更快。這優化了 Next.js 預抓和快取導航數據的方式：

- **佈局去重**：當預抓使用共用佈局的多個 URL 時，佈局僅下載一次。
- **增量預抓**：Next.js 僅預抓快取中未包含的部份，而非整個頁面。

這些變更需要 **零程式碼修改**，旨在改善所有應用程式的效能。

然而，您可能會看到更多個別預抓請求，但總傳輸大小更低。我們認為這對絕大多數應用程式來說是正確的權衡。

如果增加的請求數量導致問題，請透過建立 [issue](https://github.com/vercel/next.js/issues) 或 [discussion](https://github.com/vercel/next.js/discussions) 項目告訴我們。

## 部分預渲染 (PPR)

**Next.js 16** 移除了實驗性的 **Partial Prerendering (PPR)** 標誌和配置選項，包括路由層級段 `experimental_ppr`。

從 **Next.js 16** 開始，您可以使用 [`cacheComponents`](/docs/app/api-reference/config/next-config-js/cacheComponents) 配置選擇加入 PPR。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  cacheComponents: true,
}

module.exports = nextConfig
```

## `middleware` 改為 `proxy`

`middleware` 檔案名稱已棄用，並更名為 `proxy` 以明確化網路邊界和路由焦點。

### `minimumCacheTTL` 預設值 (重大變更)

`images.minimumCacheTTL` 的預設值已從 `60 秒` 改為 `4 小時` (14400 秒)。這減少了沒有 cache-control 標頭的圖片的重驗證成本。

對於一些 Next.js 使用者，圖片重驗證經常發生，通常是因為上游來源圖片缺少 `cache-control` 標頭。這導致每 `60` 秒發生一次重驗證，增加了 CPU 使用率和成本。

由於大多數圖片不常變更，此短時間間隔並不理想。將預設值設定為 4 小時可在預設情況下提供更耐久的快取，同時仍允許圖片根據需要在每天更新幾次。

如果您需要之前的行為，將 `minimumCacheTTL` 更改為較低的值，例如變回 `60` 秒：

```ts filename="next.config.ts" switcher

  },
}

export default nextConfig
```

## 捲動行為覆蓋

在 **Next.js 的先前版本**中，如果您透過 CSS 在全域 `<html>` 元件上設定 `scroll-behavior: smooth`，Next.js 會在 SPA 路由轉換期間覆蓋此設定，如下所示：

1. 暫時將 `scroll-behavior` 設定為 `auto`
2. 執行導航（導致立即捲動至頂端）
3. 还原您当前的 `scroll-behavior` 值

這確保了頁面導航始終感覺迅速且即時，即使您啟用了頁面內導航的平順捲動。然而，此操作可能成本高昂，特別是在每次導航的開始。

在 **Next.js 16** 中，此行為已變更。預設情況下，Next.js 在導航期間將 **不再覆蓋** 您的 `scroll-behavior` 設定。

**如果您希望 Next.js 執行此覆蓋**（先前的預設行為），請將 `data-scroll-behavior="smooth"` 屬性添加到您的 `<html>` 元件：

```tsx filename="app/layout.tsx"
export default function RootLayout({ children }) {
  return (
    <html lang="en" data-scroll-behavior="smooth">
      <body>{children}</body>
    </html>
  )
}
```

### `next dev` 配置載入

在先前版本中，Next 配置文件在開發期間載入了兩次：

- 執行 `next dev` 命令時
- 當 `next dev` 命令啟動 Next.js 伺服器時

這效率不彰，因為 `next dev` 命令不需要配置文件來啟動 Next.js 伺服器。

此變更的後果是，當執行 `next dev` 並檢查您的 Next.js 配置文件中的 `process.argv` 是否包含 `'dev'` 時，將返回 `false`。

> **了解重點**：`typegen` 和 `build` 命令仍然可見於 `process.argv`。

這對於在 `next dev` 上觸發副作用的外掛特別重要。如果是這樣，檢查 `NODE_ENV` 是否設定為 `development` 可能就足夠了。

```js filename="next.config.js"
import { startServer } from 'docs-lib/dev-server'

const isDev = process.env.NODE_ENV === 'development'

if (isDev) {
  startServer()
}

const nextConfig = {
  /* Your config options */
}

module.exports = nextConfig
```

或者，使用載入配置時的 [`phase`](/docs/app/api-reference/config/next-config-js#phase)。

### 執行時期配置

`serverRuntimeConfig` 和 `publicRuntimeConfig` 已移除。請改用環境變數。

**之前（Next.js 15）：**

```js filename="next.config.js"
module.exports = {
  serverRuntimeConfig: {
    dbUrl: process.env.DATABASE_URL,
  },
  publicRuntimeConfig: {
    apiUrl: '/api',
  },

```

```tsx filename="pages/index.tsx"
import getConfig from 'next/config'

export default function Page() {
  const { publicRuntimeConfig } = getConfig()
  return <p>API URL: {publicRuntimeConfig.apiUrl}</p>
}
```

**之後（Next.js 16）：**

對於僅伺服器值，在 Server Components 中直接存取環境變數：

```tsx filename="app/page.tsx"
async function fetchData() {
  const dbUrl = process.env.DATABASE_URL
  // Use for server-side operations only
  return await db.query(dbUrl, 'SELECT * FROM users')
}

export default async function Page() {
  const data = await fetchData()
  return <div>{/* render data */}</div>
}
```

> **了解重點**：使用 [taint API](/docs/app/api-reference/config/next-config-js/taint) 以防止意外將敏感的伺服器值傳遞給 Client Components。

對於可供客戶端讀取的變數，請使用 `NEXT_PUBLIC_` 前綴：

```bash filename=".env.local"
NEXT_PUBLIC_API_URL="/api"
```

```tsx filename="app/components/client-component.tsx"
'use client'

export default function ClientComponent() {
  const apiUrl = process.env.NEXT_PUBLIC_API_URL
  return <p>API URL: {apiUrl}</p>
}
```

要確保環境變數在執行時期被讀取（而非在建構時期打包），在從 `process.env` 讀取之前使用 [`connection()`](/docs/app/api-reference/functions/connection) 函式：

```tsx filename="app/page.tsx"
import { connection } from 'next/server'

export default async function Page() {
  await connection()
  const config = process.env.RUNTIME_CONFIG
  return <p>{config}</p>
}
```

請了解 [環境變數](/docs/app/guides/environment-variables) 的更多資訊。
