## 什麼是串流？

在传统伺服器端渲染 (SSR) 中，伺服器會先生成完整的 HTML 文件再發送任何內容。單一較慢的資料庫查詢或 API 呼叫可能會阻塞整個頁面。串流會透過 [區塊傳輸編碼](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Transfer-Encoding) 變更此行為，在回應各部分就緒時發送回應的區塊。瀏覽器會開始渲染 HTML，同時伺服器仍在產生剩餘部分。

這對於結合快速靜態內容（標頭、導航、佈局）與較慢動態內容（個人化資料、分析、推薦）的頁面尤其重要。靜態部分可以預先渲染並從 CDN 發送至畫面，而動態部分會隨著準備就緒隨伺服器串流入畫面。

React 的伺服器渲染器會產生與 `<Suspense>` 邊界對齊的區塊 HTML。Next.js 將其整合到 App Router，讓串流無需額外的配置即可運作。

## App Router 如何交付頁面

當瀏覽器請求頁面時，兩個串流會在初始頁面載入期間共同運作：

### HTML 串流

React 的伺服器渲染器會產生漸進的 HTML 區塊。頁面的靜態部分（佈局、導航、Suspense 預留位置）會先渲染並立即发送。當非同步 [伺服器組件](/docs/app/glossary#server-component) 解析完成後，React 會串流其完成的 HTML，連同內嵌的 `<script>` 標籤：一個用於將預留位置的 DOM 節點交換為新內容，另一個攜帶 [組件負載](#the-component-payload) 以便 React 稍後活化它。瀏覽器會立即執行交換，無需等待頁面的 JavaScript 函數庫載入或活化完成。這就是使用者所 _看見_ 的：頁面隨著每一節段逐步畫出。

### 組件負載

組件負載是用於 React 活化頁面並處理用戶端更新的組件樹序列化表示。在初始載入時，它嵌入在 HTML 串流中（如上所述）。在 **用戶端導航** 期間，僅會獲取組件負載（帶有 `rsc: 1` 請求標頭），完全不傳輸 HTML。React 使用它來更新原地中的組件樹。

### 靜態外殼 (Static Shell)

在任非同步工作解析之前所渲染的一切都被稱為 **靜態外殼 (Static Shell)**：您的佈局、導航，以及您的 `<Suspense>` 邊界所定義的預留位置 UI。它會立即发送，讓使用者在動態內容串流入畫面時有東西可以看和互動。配合 [Cache Components](/docs/app/getting-started/caching)，靜態外殼會在建置時預先渲染，並從邊緣即時發出。

## 使用 `<Suspense>` 進行細粒度串流

`<Suspense>` 讓您可以精確控制頁面中哪些部分會獨立串流。與其使用全頁骨架圖案，您可以將預留位置推入特定區塊，讓靜態外殼包含更多真實內容。

### 使用相鄰邊界進行平行串流

當多個組件執行非同步工作（獲取資料、從資料庫讀取）時，將每一個包裹在獨立的 `<Suspense>` 邊界中。每個邊界會隨著其非同步工作完成而獨立串流，無論順序如何，且不會相互阻塞：

```tsx filename="app/dashboard/page.tsx" switcher
import { Suspense } from 'react'
import { Revenue } from './revenue'
import { RecentOrders } from './recent-orders'
import { Recommendations } from './recommendations'

export default function Dashboard() {
  return (
    <div>
      <h1>Dashboard</h1>
      <div className="grid grid-cols-2 gap-4">
        <Suspense fallback={<p>Loading revenue...</p>}>
          <Revenue />
        </Suspense>
        <Suspense fallback={<p>Loading orders...</p>}>
          <RecentOrders />
        </Suspense>
      </div>
      <Suspense fallback={<p>Loading recommendations...</p>}>
        <Recommendations />
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/dashboard/page.js" switcher
import { Suspense } from 'react'
import { Revenue } from './revenue'
import { RecentOrders } from './recent-orders'
import { Recommendations } from './recommendations'

  return (
    <div>
      <h1>Dashboard</h1>
      <div className="grid grid-cols-2 gap-4">
        <Suspense fallback={<p>Loading revenue...</p>}>
          <Revenue />
        </Suspense>
        <Suspense fallback={<p>Loading orders...</p>}>
          <RecentOrders />

```

在此範例中，如果 `Revenue` 在 200ms 內解析完成，`RecentOrders` 在 1s 內，`Recommendations` 在 3s 內，則使用者會在每節段資料就緒時看到它們出現。

### 用於漸進詳細資訊的嵌套邊界

您可以嵌套 `<Suspense>` 邊界以建立分層的載入體驗。例如，產品頁面可能會立即串流標頭，接著是產品詳細資訊，最後是評論：

```tsx filename="app/product/[id]/page.tsx" switcher
import { Suspense } from 'react'
import { ProductDetails } from './product-details'
import { Reviews } from './reviews'

export async function generateStaticParams() {
  const products = await getTopProducts()
  return products.map((product) => ({ id: product.id }))
}

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params

  return (
    <div>
      <h1>Product</h1>
      <Suspense fallback={<p>Loading product details...</p>}>
        <ProductDetails id={id} />
        <Suspense fallback={<p>Loading reviews...</p>}>
          <Reviews productId={id} />
        </Suspense>
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/product/[id]/page.js" switcher

import { ProductDetails } from './product-details'
```

外層邊界會顯示「正在匯入產品詳細資訊...」直到 `ProductDetails` 解析完成。解析後，內層邊界會可見，顯示「正在載入評論...」直到 `Reviews` 解析完成。這會造成漸進揭示。

### 將動態存取推下

最大化即時串流內容的關鍵在於將動態資料存取延遲到實際需要的組件。這適用於 `params`、`searchParams`、`cookies()`、`headers()` 和資料抓取。如果您在佈局或頁面的頂部 `await` 任何這些內容，則該點下方的所有內容都會變成動態的，無法作為靜態外殼的一部分預先渲染。

相反地，傳遞 promise，讓消費組件在 `<Suspense>` 邊界內解析它：

```tsx filename="app/dashboard/layout.tsx" switcher
import { Suspense } from 'react'
import { Nav } from './nav'
import { UserMenu } from './user-menu'
import { cookies } from 'next/headers'

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  const cookieStore = cookies() // Start the work, but don't await

  return (
    <div>
      <Nav>
        <Suspense fallback={<p>Loading user...</p>}>
          <UserMenu cookiePromise={cookieStore} />
        </Suspense>
      </Nav>
      {children}
    </div>
  )
}
```

```jsx filename="app/dashboard/layout.js" switcher
import { Suspense } from 'react'
import { Nav } from './nav'
import { UserMenu } from './user-menu'
import { cookies } from 'next/headers'

export default function DashboardLayout({ children }) {
  const cookieStore = cookies() // Start the work, but don't await

  return (
    <div>
      <Nav>
        <Suspense fallback={<p>Loading user...</p>}>
          <UserMenu cookiePromise={cookieStore} />
        </Suspense>
      </Nav>
      {children}
    </div>
  )
}
```

在此範例中，`<Nav>` 和 `{children}` 會作為靜態外殼的一部分渲染，因為佈局中沒有任何內容會被 await。只有 `<UserMenu>` 會在解析 cookie promise 時暫停。如果佈局頂部調用了 `await cookies()`，則整個佈局及其所有子項都將無法預先渲染。

`params` 和 `searchParams` 也適用相同原則。不要在頁面層級解構它們，而是將 promise 傳遞給需要該值的組件：

```tsx filename="app/shop/[category]/page.tsx" switcher
import { Suspense } from 'react'
import { Hero } from './hero'
import { ProductGrid } from './product-grid'

export async function generateStaticParams() {
  const categories = await getCategories()
  return categories.map((c) => ({ category: c.slug }))
}

export default function ShopPage({
  params,
}: {
  params: Promise<{ category: string }>
}) {
  return (
    <div>
      <Hero />
      <Suspense fallback={<p>Loading products...</p>}>
        <ProductGrid paramsPromise={params} />
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/shop/[category]/page.js" switcher
import { Suspense } from 'react'
import { Hero } from './hero'
import { ProductGrid } from './product-grid'

export async function generateStaticParams() {
  const categories = await getCategories()
  return categories.map((c) => ({ category: c.slug }))
}

      <Hero />
      <Suspense fallback={<p>Loading products...</p>}>

      </Suspense>
    </div>
  )
}
```

`<Hero />` 會作為靜態外殼畫出。`<ProductGrid>` 會在需要類別時解析 `params`，僅在其邊界內懸掛。

您也可以與 `.then()` 一起解壓 promise，讓子組件接收單純的值而非 promise：

```tsx filename="app/shop/[category]/page.tsx" switcher
<Suspense fallback={<p>Loading products...</p>}>
  {params.then(({ category }) => (
    <ProductGrid category={category} />
  ))}
</Suspense>
```

```jsx filename="app/shop/[category]/page.js" switcher
<Suspense fallback={<p>Loading products...</p>}>
  {params.then(({ category }) => (
    <ProductGrid category={category} />
  ))}
</Suspense>
```

這會讓 `ProductGrid` 保持簡單（它接收 `string`，而非 `Promise`），同時仍能將 `params` 存取延遲到 Suspense 邊界內。

### 使用 `loading.js` 或 `<Suspense>` 的時機

| | `loading.js` | `<Suspense>` |
| --- | --- | --- |
| **範圍** | 整個頁面 | 任何組件 |
| **設置** | 放入文件中 | 明確包裹組件 |
| **導航** | 預先抓取為立即預留位置 | 預預先抓取 |
| **最佳用途** | 無資料則不渲染的頁面 | 大多數頁面，用於細粒度控制 |

優先使用靠近動態存取位置的明確 `<Suspense>` 邊界。當預先渲染器遇到動態工作時，它會沿著樹往上尋找最近的 Suspense 邊界。如果未找到，建置會失敗並顯示 [阻塞路由錯誤](/docs/messages/blocking-route)。`loading.js` 高層級的樹是有效的邊界，所以框架會找到並停止，但現在整個頁面會回退到全頁骨架圖案，而非細粒度串流。

如果組件在串流開始後拋出錯誤，最近的 [`error.js`](/docs/app/api-reference/file-conventions/error) 邊界會捕捉到它，並在原地渲染錯誤 UI 以取代失敗的組件。頁面的其餘部分保持完整，僅取代出錯的區塊。

因為 HTTP 狀態碼 (`200 OK`) 已隨第一個區塊发送，所以無法變更 為 `4xx` 或 `5xx`。錯誤完全在串流的 HTML 中處理。請參閱 [HTTP 合約](#the-http-contract) 了解更多關於此限制。

## 向用戶端串流資料

您可以在 [伺服器組件](/docs/app/glossary#server-component) 中開始抓取，並將未解析的 promise 作為 prop 傳遞給 [用戶端組件](/docs/app/glossary#client-component)。promise 可以傳遞多層。只有調用 React 的 [`use`](https://react.dev/reference/react/use) API 來讀取值且周圍有 `<Suspense>` 邊界的組件：

```tsx filename="app/dashboard/page.tsx" switcher
import { Suspense } from 'react'
import { StatsChart } from './stats-chart'

type Stats = { revenue: number; orders: number }

async function getStats(): Promise<Stats> {
  const res = await fetch('https://api.example.com/stats')
  return res.json()
}

export default function Dashboard() {
  // Start the fetch during server render, don't await it
  const statsPromise = getStats()

  return (
    <Suspense fallback={<p>Loading chart...</p>}>
      <StatsChart dataPromise={statsPromise} />
    </Suspense>
  )
}
```

```jsx filename="app/dashboard/page.js" switcher
import { Suspense } from 'react'
import { StatsChart } from './stats-chart'

async function getStats() {
  const res = await fetch('https://api.example.com/stats')

}

  const statsPromise = getStats()

  return (
    <Suspense fallback={<p>Loading chart...</p>}>
      <StatsChart dataPromise={statsPromise} />
    </Suspense>
  )
}
```

```tsx filename="app/dashboard/stats-chart.tsx" switcher
'use client'

import { use } from 'react'

type Stats = { revenue: number; orders: number }

export function StatsChart({ dataPromise }: { dataPromise: Promise<Stats> }) {

  return <div>{/* render chart with stats */}</div>
}
```

```jsx filename="app/dashboard/stats-chart.js" switcher

```

預留位置會立即隨靜態外殼发送。當 promise 解析完成後，React 會將完成的 HTML 串流入畫面。

### 樹狀結構間共享 promise

當多個組件需要相同的資料時，開始一次抓取，並透過 context provider 傳遞 promise，讓子樹中的任何組件都可以用 `use()` 解析它：

```tsx filename="app/layout.tsx"
import { getUser } from '@/lib/data'
// Stores the promise in React context for the subtree
import { UserProvider } from './user-provider'

export default function Layout({ children }: { children: React.ReactNode }) {
  const userPromise = getUser()

  return <UserProvider userPromise={userPromise}>{children}</UserProvider>
}
```

請參閱 [使用 context 和 React.cache 共享資料](/docs/app/getting-started/fetching-data#sharing-data-with-context-and-reactcache) 以了解完整的模式，包括 provider 和 consumer 組件。

## Route Handler 串流

上述模式依賴 React 和 Suspense 來串流 UI。在 React 渲染之外，[Route Handler](/docs/app/api-reference/file-conventions/route) 可以 Web Streams API 串流原始回應。這適用於 Server-Sent Events、大型檔案生成，或任何希望資料逐步到達的回應：

```ts filename="app/api/stream/route.ts" switcher
export async function GET() {
  const encoder = new TextEncoder()

  const stream = new ReadableStream({
    async start(controller) {
      for (let i = 0; i < 10; i++) {
        controller.enqueue(encoder.encode(`Chunk ${i + 1}\n`))
        await new Promise((resolve) => setTimeout(resolve, 200))
      }
      controller.close()
    },
  })

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/plain; charset=utf-8',
      'X-Content-Type-Options': 'nosniff',
    },
  })
}
```


您也可以在不完全載入檔案的情況下串流檔案。使用 `FileHandle.readableWebStream()` 從檔案直接獲取 Web `ReadableStream`：

```ts filename="app/api/download/route.ts" switcher
import { open } from 'node:fs/promises'

export async function GET() {
  const file = await open('/path/to/large-file.csv')

  return new Response(file.readableWebStream(), {
    headers: {
      'Content-Type': 'text/csv',
      'Content-Disposition': 'attachment; filename="data.csv"',
    },
  })
}
```

## 串流與 Web Vitals

[Web Vitals](https://web.dev/articles/vitals) 是 Google 用於衡量使用者體驗的指標。串流會直接影響其中幾項。

### TTFB 和 FCP

沒有串流時，伺服器會等待所有資料後才發送任何 HTML，因此 TTFB 等同於最慢的查詢。有了串流，只要伺服器準備就緒就會立即發送靜態外殼。TTFB 會降至渲染佈局和預留位置所需的時間。瀏覽器會立即畫出靜態外殼，所以 FCP 與資料抓取時間分離。

### LCP (最大內容渲染)

如果您的 LCP 元素（Hero 圖像、主要標頭、產品照片）位於 Suspense 邊界內，它無法畫出直到該邊界解析。若要保持 LCP 快速：

- 保持 LCP 元素 **在** 或 **低於** Suspense 邊界之外，讓它們作為靜態外殼的一部分渲染。
- 在 LCP 圖像上使用 [`preload`](/docs/app/api-reference/components/image#preload) prop 在 `next/image`。這會將 `<link rel="preload">` 注入 `<head>`，因此瀏覽器會从第一個區塊開始抓取圖像，甚至出現在 HTML 中的 `<img>` 標簽之前。
- 對於非圖像 LCP 元素（文字、標頭），確保它們不在依賴慢速資料的 Suspense 邊界內。

### CLS (版面配置跳動)

當 Suspense 預留位置被解析內容取代時，瀏覽器會重新排列頁面。如果預留位置與解析內容大小不同，周圍的佈局會移動。若要最小化 CLS：

- 設計能 **匹配尺寸** 的骨架預留位置它們代表的內容。與最終卡片網格相同高寬的骨架圖案可防止移動。
- 在 Suspense 邊界周圍使用固定或 min-height 容器，以便在內容到達之前預留空間。

### INP (互動至下一個繪製)

串流 enabling [selective hydration](https://react.dev/reference/react-dom/client/hydrateRoot)：React 會獨立活化串流入畫面的組件，並優先於使用者互動whatever的活化。每個 `<Suspense>` 邊界都是一個活化單位。沒有它們，React 會阻塞式地活化整個頁面。有了它們，活化被分割成任務，讓出給瀏覽器，保持主線程響應。[附屬範例](https://streaming-demo.labs.vercel.dev/hydration-single) 讓您可以比較單一阻塞活化步驟與使用 Suspense 邊界的 [分割活化](https://streaming-demo.labs.vercel.dev/hydration-split)。

### 早期資源發現

靜態外殼包含 `<link>` 和 `<script>` 標签在第一個 HTML 區塊中。瀏覽器會立即發現並開始抓取 CSS、JavaScript 和字體，同時伺服器仍在生成內容。資源是在伺服器思考時間期間抓取，而非事後。

在上面的 [儀表板範例](#parallel-streaming-with-sibling-boundaries) 中，`<h1>` 在靜態外殼中渲染（對 LCP 好），每個資料區塊在其各自的 Suspense 邊界後面獨立串流（對 INP 好，因為活化是分割的），骨架預留位置預留空間（對 CLS 好）。

## HTTP 合約

一旦串流開始，HTTP 回應標頭（包括狀態碼）已經发送到用戶端。**您無法在串流開始後變更狀態碼或標頭。** 本節中的所有內容都源自此基本限制。

### 狀態碼

當 `<Suspense>` 預留位置呈現或當組件懸掛時，伺服器必須提交 `200 OK` 以開始發送 HTML 串流。如果 [`notFound()`](/docs/app/api-reference/functions/not-found) 在串流期間發射，Next.js 無法回到並更改狀態為 404。相反地，它會将 `<meta name="robots" content="noindex">` 注入串流的 HTML 中，以防止搜索引擎索引該頁面。同樣地，串流中的 [`redirect()`](/docs/app/api-reference/functions/redirect) 會成為用戶端導航，而非 HTTP 重導向標頭。

### 串流何時開始？

回應body開始串流時，會有一個 Suspense 預留位置呈現（例如 `loading.tsx`）或組件在 `<Suspense>` 邊界下懸掛。若要獲得錯誤的真實 HTTP 狀態碼，請在 `await` 或 `<Suspense>` 邊界 **之前** 放置 `notFound()`：

```tsx filename="app/post/[slug]/page.tsx" switcher
import { Suspense } from 'react'
import { notFound } from 'next/navigation'
import { PostContent } from './post-content'

export async function generateStaticParams() {
  const posts = await getPublishedPosts()
  return posts.map((post) => ({ slug: post.slug }))
}

export default async function PostPage({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const exists = await checkSlugExists(slug) // Fast existence check
  if (!exists) notFound() // Real 404, before any Suspense boundary

  return (
    <Suspense fallback={<p>Loading post...</p>}>
      <PostContent slug={slug} />
    </Suspense>
  )
}
```

```jsx filename="app/post/[slug]/page.js" switcher

import { PostContent } from './post-content'
```

> **了解事項：** 您也可以早期拒絕請求使用 [`proxy`](/docs/app/api-reference/file-conventions/proxy) (用於重導向、重新寫入或返回回應) 或 [`next.config.js` redirects](/docs/app/api-reference/config/next-config-js/redirects)。兩者在頁面渲染之前執行，因此 HTTP 狀態碼仍可取得。

### 元數據與 BOT

[`generateMetadata`](/docs/app/api-reference/functions/generate-metadata) 會在串流開始前解析，針對僅抓取靜態 HTML 的 BOT（例如 Twitterbot 或 Slackbot）。對於支援瀏覽器與能力爬蟲，元數據可以隨頁面內容 [串流](/docs/app/api-reference/functions/generate-metadata#streaming-metadata)。

Next.js 會自動偵測使用者代理以選擇正確的行為。您可以使用 [`htmlLimitedBots`](/docs/app/api-reference/config/next-config-js/htmlLimitedBots) 配置選項自定義哪些 BOT 會收到阻擋式元數據。

詳細資訊請參見 [`loading.js` SEO 章節](/docs/app/api-reference/file-conventions/loading#seo)。

## 影響串流的因素

伺服器與用戶端之間的任意外層只要緩衝回應，就會削弱串流的好處。HTML 可能在伺服器上漸進生成，但如果代理、CDN 或甚至客戶端本身收集所有區塊再渲染，使用者看到的是延遲的單一回應，而非逐步渲染。

### Reverse Proxies

Nginx 和其他反向代理預設會緩衝回應。設定 `X-Accel-Buffering` 標頭為 `no` 以停用緩衝：

```js filename="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*{/}?',
        headers: [
          {
            key: 'X-Accel-Buffering',
            value: 'no',
          },
        ],
      },
    ]
  },
}
```

### Serverless Platforms

並非所有 Serverless 環境都支援串流。例如，AWS Lambda 需要明確啟用 [回應串流模式](https://docs.aws.amazon.com/lambda/latest/dg/configuration-response-streaming.html)（它不是預設）。Vercel 原生支援串流。

### 壓縮

Gzip 和 Brotli 壓縮可以在 flush 之前內部緩衝區塊，因為壓縮演算法需要足夠的資料才能有效率地壓縮。這可能會增加第一個可見區塊的延遲。如果您注意串流延遲，請檢查您的壓縮層是否足夠激進地 flush。

### 用戶端

緩衝也會在用戶端發生。[Safari/WebKit](https://bugs.webkit.org/show_bug.cgi?id=252413) 會緩衝串流回應直到收到 1024 位元組，因此非常小的回應會一次畫出，而非逐步畫出。實際應用很容易超過此門檻（佈局、樣式、腳本），因此它只影響極小演示或微小的 Route Handler 回應。

命令列工具如 `curl` 也會預設緩衝。`-N` 標記會停用輸出緩衝，但 `curl` 仍需依賴換行字元 flush 線到終端機。發送區塊且無換行的串流即使有 `-N` 也似乎會卡住。

### 驗證串流是否運作

此章節關於確認 HTTP 回應是否實際透過您的基礎架構分區塊到達。關於設計有意義的裝載狀態並有效放置 Suspense 邊界的指導，請參見 [使用 `<Suspense>` 進行細粒度串流](#granular-streaming-with-suspense) 和 [Cache Components](/docs/app/getting-started/caching) 指南。

**檢查 Network 選項卡。** 在 Chrome DevTools 中，選取文件請求並查看「Timing」分類。長期的「Content Download」階段與早期的「Time to First Byte」確認回應正在串流，而非一次性到達。

**觀察原始區塊。** 為了查看伺服器發送的内容和時序，請使用輕量 script 以串流方式讀取回應。這比 `curl` 觀察時序區塊更可靠，因為 `curl` 有自己的緩衝行為：

```js filename="stream-observer.mjs"
const res = await fetch(
  'https://streaming-demo.labs.vercel.dev/suspense-demo',
  {
    headers: { 'Accept-Encoding': 'identity' },
  }
)

const reader = res.body.getReader()
const decoder = new TextDecoder()
let i = 0
const start = Date.now()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  console.log(`\nchunk ${i++} (+${Date.now() - start}ms)\n`)
  console.log(decoder.decode(value))
}
```

使用 `node stream-observer.mjs` 執行。對於具有兩個相鄰 Suspense 邊界的頁面（如 [companion demo 的 Suspense 頁面](https://streaming-demo.labs.vercel.dev/suspense-demo))，您會看到類似輸出：

```text filename="Terminal"
chunk 0 (+0ms)    # Static shell: <head>, CSS, nav, fallback skeletons,
                  # <template id="B:0"> and <template id="B:1"> placeholders,
                  # bootstrap scripts
chunk 1 (+170ms)  # Component payload (self.__next_f.push) for hydration
chunk 2 (+1000ms) # Weather widget: payload + <div hidden id="S:0"> (swaps B:0)
chunk 3 (+3000ms) # Analytics dashboard: payload + <div hidden id="S:1"> (swaps B:1)
```

`<template id="B:0">` 標記是 Suspense 預留位置的佔位符。當邊界解析完成，React 會串流`<div hidden id="S:0">`包含完成的 HTML 及將它交換到頁面的 script。時間戳顯示每個邊界獨立解析。

> **了解事項：** 標頭 `Accept-Encoding: identity` 會停用壓縮，所以區塊不會被壓縮層緩衝。

## 摘要

觸發是 **您的程式碼**：非同步工作、非確定性輸出或執行期資料。當框架遇到這些時，它會沿著樹往上尋找 `<Suspense>` 邊界以作為殘缺。那些邊界上方的所有內容構成 [靜態外殼](#the-static-shell)，會立即发送。隨著每個邊界解析，React 會將結果串流入畫面。

關鍵決定是 **要快取什麼** 和 **要放置 Suspense 邊界的位置**。使用 [`"use cache"`](/docs/app/api-reference/directives/use-cache) 快取您能做的以擴大靜態外殼。將動態存取推送到需要它的組件，並用 `<Suspense>` 包裹它們。其他所有內容都成為外殼的一部分。
