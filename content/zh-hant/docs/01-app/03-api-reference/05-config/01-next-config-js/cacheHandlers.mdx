「`cacheHandlers`」設定允許您針對 [`使用快取 (use cache)`](/docs/app/api-reference/directives/use-cache) 和 [`使用快取：遠端 (use cache: remote)`](/docs/app/api-reference/directives/use-cache-remote) 定義自訂快取儲存實作。這使您能夠將快取的元件和函式儲存於外部服務，或自訂快取行為。 [`使用快取：私密 (use cache: private)`](/docs/app/api-reference/directives/use-cache-private) 無法自訂。

## 何時使用自訂快取處理程式

**大多數應用程式不需要自訂快取處理程式。** 預設的記憶體快取在典型使用情況下運作良好。

自訂快取處理程式適用於您需要在多個實例間共用快取，或更改快取儲存位置等進階情境。例如，您可以為外部儲存 ( 如鍵值儲存 ) 設定自訂 `remote` 處理程式，接著在程式碼中使用 `use cache` 進行記憶體快取及 `use cache: remote` 進行外部儲存，允許在同一應用程式中使用不同的快取策略。

**在不同實例間共用快取**

預設的記憶體快取與每個 Next.js 進程隔離。如果您運行多個伺服器或容器，每個實例將擁有自己的快取，不與其他實例共用，且在重新啟動時會遺失。

自訂處理程式允許您整合共用的儲存系統 ( 如 Redis、Memcached 或 DynamoDB )，您的所有 Next.js 實例均可存取。

**更改儲存類型**

您可能希望以不同於預設記憶體方案的方式儲存快取。您可以實作自訂處理程式，將快取儲存於磁碟、資料庫或外部快取服務。理由包括：重新啟動後的持久性、降低記憶體用量，或整合既有基礎設施。

## 使用方法

設定自訂快取處理程式：

1. 在單獨的檔案中定義您的快取處理程式，詳細實作請參見 [範例](#examples)。
2. 在您的 Next 設定檔中參考檔案路徑

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  cacheHandlers: {
    default: require.resolve('./cache-handlers/default-handler.js'),
    remote: require.resolve('./cache-handlers/remote-handler.js'),
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
module.exports = {
  cacheHandlers: {
    default: require.resolve('./cache-handlers/default-handler.js'),
    remote: require.resolve('./cache-handlers/remote-handler.js'),
  },
}
```

### 處理程式類型

- **`default`**: 為 `use cache` 指令所用
- **`remote`**: 為 `use cache: remote` 指令所用

如果您沒有配置 `cacheHandlers`，Next.js 會同時為 `default` 和 `remote` 使用記憶體 LRU ( Least Recently Used ) 快取。您可以參閱 [預設實作](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/default.ts) 作為參考。

您也可以定義其他命名處理程式 ( 例如 `sessions`、`analytics` )，並使用 `use cache: <name>` 引用它們。

注意 `use cache: private` 不使用快取處理程式，且無法自訂。

## API 參考

快取處理程式必須實作 [`CacheHandler`](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/types.ts) 介面，包含以下方法：

### `get()`

為給定的快取金鑰取得快取項目。

```ts
get(cacheKey: string, softTags: string[]): Promise<CacheEntry | undefined>
```

| 參數 | 類型 | 說明 |
| ---------- | ---------- | ------------------------------------------------------------ |
| `cacheKey` | `string` | 快取項目的唯一金鑰。
| `softTags` | `string[]` | 用於檢查過期的標籤 ( 用於某些快取策略 )。 |

如果找到則回傳 `CacheEntry` 物件，若未找到或已過期則回傳 `undefined`。

您的 `get` 方法應從儲存中取得快取項目，根據 `revalidate` 時間檢查是否已過期，並對於遺失或過期的項目回傳 `undefined`。

```js
const cacheHandler = {
  async get(cacheKey, softTags) {
    const entry = cache.get(cacheKey)
    if (!entry) return undefined

    // Check if expired
    const now = Date.now()
    if (now > entry.timestamp + entry.revalidate * 1000) {
      return undefined
    }

    return entry
  },
}
```

### `set()`

儲存給定快取金鑰的快取項目。

```ts
set(cacheKey: string, pendingEntry: Promise<CacheEntry>): Promise<void>
```

| 參數 | 類型 | 說明 |
| -------------- | --------------------- | ------------------------------------------- |
| `cacheKey` | `string` | 儲存該項目要使用的唯一天然金鑰。
| `pendingEntry` | `Promise<CacheEntry>` | 解析為快取項目的 Promise。 |

呼叫時，項目可能仍在 Pending ( 例如值串流可能仍在寫入中 )。您的處理程式應在處理項目前先等待 Promise。

回傳 `Promise<void>`。

您的 `set` 方法必須在儲存前等待 `pendingEntry` Promise，因為快取項目在此方法呼叫時可能仍在產生中。解析後，將項目儲存到您的快取系統中。

```js
const cacheHandler = {
  async set(cacheKey, pendingEntry) {
    // Wait for the entry to be ready
    const entry = await pendingEntry

    // Store in your cache system
    cache.set(cacheKey, entry)
  },
}
```

### `refreshTags()`

在起始新要求前定期呼叫，以與外部標籤服務同步。

```ts
refreshTags(): Promise<void>
```

如果您要在多個實例或服務間協調快取無效，這很有用。對於記憶體快取，這可以是無操作。

回傳 `Promise<void>`。

對於記憶體快取，這可以是一個無操作。對於分散式快取，請利用它在使用要求前，從外部服務或資料庫同步標籤狀態。

```js
const cacheHandler = {
  async refreshTags() {
    // For in-memory cache, no action needed
    // For distributed cache, sync tag state from external service
  },
}
```

### `getExpiration()`

取得一組標籤的最大重新驗證時間戳記。

```ts
getExpiration(tags: string[]): Promise<number>
```

| 參數 | 類型 | 說明 |
| --------- | ---------- | -------------------------------------- |
| `tags` | `string[]` | 用於檢查過期的標籤陣列。 |

回傳：

- 若從未重新驗證任何標籤則為 `0`
- 代表最近重新驗證的時間戳記 ( 毫秒 )
- `Infinity` 表示應在 `get` 方法中檢查軟式標籤

如果您不追蹤標籤重新驗證時間戳記，請回傳 `0`。否則，請尋找所有提供的標籤中最近的重新驗證時間戳記。如果您希望在 `get` 方法中處理軟式標籤檢查，請回傳 `Infinity`。

```js
const cacheHandler = {
  async getExpiration(tags) {
    // Return 0 if not tracking tag revalidation
    return 0

    // Or return the most recent revalidation timestamp
    // return Math.max(...tags.map(tag => tagTimestamps.get(tag) || 0));
  },
}
```

### `updateTags()`

當標籤重新驗證或過期時呼叫。

```ts
updateTags(tags: string[], durations?: { expire?: number }): Promise<void>
```

| 參數 | 類型 | 說明 |
| ----------- | --------------------- | ---------------------------------------- |
| `tags` | `string[]` | 待更新的標籤陣列。
| `durations` | `{ expire?: number }` | 選用的過期秒數。 |

您的處理程式應更新其內部狀態，將這些標籤標記為無效。

回傳 `Promise<void>`。

當標籤重新驗證時，您的處理程式應使所有具有這些標籤的快取項目無效。遍歷您的快取，移除標籤與提供清單相符的項目。

```js
const cacheHandler = {
  async updateTags(tags, durations) {
    // Invalidate all cache entries with matching tags
    for (const [key, entry] of cache.entries()) {
      if (entry.tags.some((tag) => tags.includes(tag))) {
        cache.delete(key)
      }
    }
  },
}
```

## CacheEntry 型別

[`CacheEntry`](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/types.ts) 物件具有以下結構：

```ts
interface CacheEntry {
  value: ReadableStream<Uint8Array>
  tags: string[]
  stale: number
  timestamp: number
  expire: number
  revalidate: number
}
```

| 屬性 | 類型 | 說明 |
| ------------ | ---------------------------- | ------------------------------------------------------------ |
| `value` | `ReadableStream<Uint8Array>` | 快取資料的串流形式。
| `tags` | `string[]` | 快取標籤 ( 不含軟式標籤 )。
| `stale` | `number` | 客戶端過期持續時間 ( 秒 )。
| `timestamp` | `number` | 項目建立時間 ( 秒時間戳 )。
| `expire` | `number` | 允許使用項目的時間長度 ( 秒 )。
| `revalidate` | `number` | 項目應重新驗證前的時間長度 ( 秒 )。 |

> **值得注意**：
>
> - `value` 是 [`ReadableStream`](https://developer.mozilla.org/docs/Web/API/ReadableStream)。如果您需要讀取並儲存串流資料，請使用 [`.tee()`](https://developer.mozilla.org/docs/Web/API/ReadableStream/tee)。
> - 如果串流錯誤且只包含部分資料，您的處理程式必須決定保留部分快取或放棄它。

## 範例

### 基本記憶體快取處理程式

這是使用 `Map` 儲存的最小化實作。此範例展示了核心概念，但若是具備 LRU 淘汰、錯誤處理及標籤管理的生產環境實作，請參見 [預設快取處理程式](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/default.ts)。

```js filename="cache-handlers/memory-handler.js"
const cache = new Map()
const pendingSets = new Map()

module.exports = {
  async get(cacheKey, softTags) {
    // Wait for any pending set operation to complete
    const pendingPromise = pendingSets.get(cacheKey)
    if (pendingPromise) {
      await pendingPromise
    }

    const entry = cache.get(cacheKey)
    if (!entry) {
      return undefined
    }

    // Check if entry has expired
    const now = Date.now()
    if (now > entry.timestamp + entry.revalidate * 1000) {
      return undefined
    }

    return entry
  },

  async set(cacheKey, pendingEntry) {
    // Create a promise to track this set operation
    let resolvePending
    const pendingPromise = new Promise((resolve) => {
      resolvePending = resolve
    })
    pendingSets.set(cacheKey, pendingPromise)

    try {
      // Wait for the entry to be ready
      const entry = await pendingEntry

      // Store the entry in the cache
      cache.set(cacheKey, entry)
    } finally {
      resolvePending()
      pendingSets.delete(cacheKey)
    }
  },

  async refreshTags() {
    // No-op for in-memory cache
  },

  async getExpiration(tags) {
    // Return 0 to indicate no tags have been revalidated
    return 0
  },

  async updateTags(tags, durations) {
    // Implement tag-based invalidation
    for (const [key, entry] of cache.entries()) {
      if (entry.tags.some((tag) => tags.includes(tag))) {
        cache.delete(key)
      }
    }
  },
}
```

### 外部儲存模式

對於 Redis 或資料庫等耐久儲存，您必須序列化快取項目。以下是簡單的 Redis 範例：

```js filename="cache-handlers/redis-handler.js"
const { createClient } = require('redis')

const client = createClient({ url: process.env.REDIS_URL })
client.connect()

module.exports = {
  async get(cacheKey, softTags) {
    // Retrieve from Redis
    const stored = await client.get(cacheKey)
    if (!stored) return undefined

    // Deserialize the entry
    const data = JSON.parse(stored)

    // Reconstruct the ReadableStream from stored data
    return {
      value: new ReadableStream({
        start(controller) {
          controller.enqueue(Buffer.from(data.value, 'base64'))
          controller.close()
        },
      }),
      tags: data.tags,
      stale: data.stale,
      timestamp: data.timestamp,
      expire: data.expire,
      revalidate: data.revalidate,
    }
  },

  async set(cacheKey, pendingEntry) {
    const entry = await pendingEntry

    // Read the stream to get the data
    const reader = entry.value.getReader()
    const chunks = []

    try {
      while (true) {
        const { done, value } = await reader.read()
        if (done) break
        chunks.push(value)
      }
    } finally {
      reader.releaseLock()
    }

    // Combine chunks and serialize for Redis storage
    const data = Buffer.concat(chunks.map((chunk) => Buffer.from(chunk)))

    await client.set(
      cacheKey,
      JSON.stringify({
        value: data.toString('base64'),
        tags: entry.tags,
        stale: entry.stale,
        timestamp: entry.timestamp,
        expire: entry.expire,
        revalidate: entry.revalidate,
      }),
      { EX: entry.expire } // Use Redis TTL for automatic expiration
    )
  },

  async refreshTags() {
    // No-op for basic Redis implementation
    // Could sync with external tag service if needed
  },

  async getExpiration(tags) {
    // Return 0 to indicate no tags have been revalidated
    // Could query Redis for tag expiration timestamps if tracking them
    return 0
  },

  async updateTags(tags, durations) {
    // Implement tag-based invalidation if needed
    // Could iterate over keys with matching tags and delete them
  },
}
```
