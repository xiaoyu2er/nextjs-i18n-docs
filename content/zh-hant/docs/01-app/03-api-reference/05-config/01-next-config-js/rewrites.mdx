重寫允許您將传入的請求路徑映射到不同的目的路徑。

<AppOnly>

重寫功能作為 URL 代理，隱藏目的路徑，使看起來使用者並未更改網站上的位置。相比之下，[重定向 (redirects)](/docs/app/api-reference/config/next-config-js/redirects) 會重新導向到新頁面並顯示 URL 的變更。

</AppOnly>

<PagesOnly>

重寫功能作為 URL 代理，隱藏目的路徑，使看起來使用者並未更改網站上的位置。相比之下，[重定向 (redirects)](/docs/pages/api-reference/config/next-config-js/redirects) 會重新導向到新頁面並顯示 URL 的變更。

</PagesOnly>

要使用重寫，您可以在 `next.config.js` 中使用 `rewrites` 鍵：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/about',
        destination: '/',
      },
    ]
  },
}
```

重寫適用於客戶端路由。在上方的範例中，導航至 `<Link href="/about">` 將從 `/` 提供內容，同時保持 URL 為 `/about`。

`rewrites` 是一個非同步函數，預期回傳陣列或包含帶有 `source` 和 `destination` 屬性的物件之陣列物件（參見下方）：

- `source`: `String` - 传入請求路徑範式。
- `destination`: `String` - 您想要路由到的路徑。
- `basePath`: `false` 或 `undefined` - 若為 false，則在匹配時不會包含 basePath，僅可用於外部重寫。
- `locale`: `false` 或 `undefined` - 匹配時是否不包含區域設定。
- `has` 陣列為 [has 物件](#header-cookie-and-query-matching)，含有 `type`、`key` 和 `value` 屬性。
- `missing` 陣列為 [missing 物件](#header-cookie-and-query-matching)，含有 `type`、`key` 和 `value` 屬性。

當 `rewrites` 函數回傳陣列時，重寫功能在檢查檔案系統（頁面和 `/public` 文件）後、動態路由前套用。當 `rewrites` 函數回傳特定格式的陣列物件時，此行為可變更並更精細地控制，適用於 Next.js v10.1 起：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return {
      beforeFiles: [
        // These rewrites are checked after headers/redirects
        // and before all files including _next/public files which
        // allows overriding page files
        {
          source: '/some-page',
          destination: '/somewhere-else',
          has: [{ type: 'query', key: 'overrideMe' }],
        },
      ],
      afterFiles: [
        // These rewrites are checked after pages/public files
        // are checked but before dynamic routes
        {
          source: '/non-existent',
          destination: '/somewhere-else',
        },
      ],
      fallback: [
        // These rewrites are checked after both pages/public files
        // and dynamic routes are checked
        {
          source: '/:path*',
          destination: `https://my-old-site.com/:path*`,
        },
      ],
    }
  },
}
```

> **要知道**：`beforeFiles` 中的重寫在匹配 source 後不會立即檢查檔案系統/動態路由，它們會繼續直到檢查完所有 `beforeFiles`。

Next.js 檢查路由的順序如下：

<AppOnly>

1. [headers](/docs/app/api-reference/config/next-config-js/headers) 會被檢查與套用
2. [redirects](/docs/app/api-reference/config/next-config-js/redirects) 會被檢查與套用
3. [proxy](/docs/app/api-reference/file-conventions/proxy) 會被檢查與套用
4. `beforeFiles` 重寫 (rewrites) 會被檢查與套用
5. `靜態檔案 (static files)` 來自 [public directory](/docs/app/api-reference/file-conventions/public-folder), `_next/static` 檔案，以及 `非動態頁面 (non-dynamic pages)` 會被檢查或提供
6. `afterFiles` 重寫 (rewrites) 會被檢查與套用，如果匹配到其中一個重寫，則在每次匹配後我們檢查 `動態路由 (dynamic routes)`/`靜態檔案 (static files)`
7. `fallback` 重寫 (rewrites) 會被檢查與套用，這些是在渲染 `404 頁面` 之前以及在 `動態路由`/`所有靜態資源` 都經過檢查之後套用。如果您在 `getStaticPaths` 中使用 `fallback: true/'blocking'`，則您 `next.config.js` 中定義的 `fallback` `重寫 (rewrites)` 將*_不會执行。`

</AppOnly>

<PagesOnly>

1. [`headers`](/docs/pages/api-reference/config/next-config-js/headers) 被檢查/套用
2. [`重定向 (redirects)`](/docs/pages/api-reference/config/next-config-js/redirects) 被檢查/套用
3. `beforeFiles` 重寫被檢查/套用
4. 來自 [`public 目錄`](/docs/pages/api-reference/file-conventions/public-folder) 的靜態文件、`_next/static` 文件和非動態頁面被檢查/服務
5. `afterFiles` 重寫被檢查/套用，若匹配到其中一個重寫，我們每次匹配後檢查動態路由/靜態文件
6. `fallback` 重寫被檢查/套用，這些在渲染 404 頁面之前以及動態路由/所有靜態資源檢查之後套用。如果您在 `getStaticPaths` 中使用 [`fallback: true/'blocking'`](/docs/pages/api-reference/functions/get-static-paths#fallback-true)，您在 `next.config.js` 中定義的 fallback `rewrites` 將 _不會_ 執行。

</PagesOnly>

## 重寫參數

當在重寫中使用參數時，若 destination 中未使用任何參數，預情況下參數將透過 query 傳遞。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/old-about/:path*',
        destination: '/about', // The :path parameter isn't used here so will be automatically passed in the query
      },
    ]
  },
}
```

若 destination 中使用了參數，則沒有任何參數會自動透過 query 傳遞。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/docs/:path*',
        destination: '/:path*', // The :path parameter is used here so will not be automatically passed in the query
      },
    ]
  },
}
```

即使 destination 中已使用參數，您仍可透過在 destination 中指定 query 來手動傳遞參數。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/:first/:second',
        destination: '/:first?second=:second',
        // Since the :first parameter is used in the destination the :second parameter
        // will not automatically be added in the query although we can manually add it
        // as shown above
      },
    ]
  },
}
```

> **要知道**：來自 [`Automatic Static Optimization`](/docs/pages/building-your-application/rendering/automatic-static-optimization) 或 [`prerendering`](/docs/pages/building-your-application/data-fetching/get-static-props) 的靜態頁面，來自重寫的參數將在用戶端 hydration 後解析，並在 query 中提供。

## 路徑匹配

路徑匹配是被允許的，例如 `/blog/:slug` 會將 `/blog/first-post` (不包含巢狀路徑) 匹配：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/blog/:slug',
        destination: '/news/:slug', // Matched parameters can be used in the destination
      },
    ]
  },
}
```

模式 `/blog/:slug` 匹配 `/blog/first-post` 和 `/blog/post-1`，但不匹配 `/blog/a/b` (不包含巢狀路徑)。模式將從開頭進行錨定：`/blog/:slug` 不會匹配 `/archive/blog/first-post`。

您可以在參數上使用修飾符：`*` (零個或多個)、`+` (一個或多個)、`?` (零個或一個)。例如，`/blog/:slug*` 會匹配 `/blog`, `/blog/a` 和 `/blog/a/b/c`。

請參閱 [path-to-regexp](https://github.com/pillarjs/path-to-regexp) 文件以獲取更多詳細資訊。

### 萬用字元路徑匹配

要匹配萬用字元路徑，您可以在參數後使用 `*`，例如 `/blog/:slug*` 將匹配 `/blog/a/b/c/d/hello-world`：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/blog/:slug*',
        destination: '/news/:slug*', // Matched parameters can be used in the destination
      },
    ]
  },
}
```

### 正則表達式路徑匹配

要匹配正規表達式路徑，您可以在參數後用括號將正規表達式括起來，例如 `/blog/:slug(\\d{1,})` 將匹配 `/blog/123` 但不匹配 `/blog/abc`：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/old-blog/:post(\\d{1,})',
        destination: '/blog/:post', // Matched parameters can be used in the destination
      },
    ]
  },
}
```

以下字符 `(`、`)`、`{`、`}`、`[`、`]`、`|`、`\`、`^`、`.`、`:`、`*`、`+`、`-`、`?`、`$` 用於路徑正則表達式匹配，因此當在 source 中作為非特殊值使用时，必須在其前添加 `\` 進行轉義：

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        // this will match `/english(default)/something` being requested
        source: '/english\\(default\\)/:slug',
        destination: '/en-us/:slug',
      },
    ]
  },
}
```

## 標頭、Cookie 和 查詢 匹配

要僅在標頭、cookie 或 查詢 值匹配 `has` 欄位或不匹配 `missing` 欄位時才匹配重寫，可以使用此方式。要套用重寫，`source` 與所有 `has` 項目都必須匹配，且所有 `missing` 項目都不得匹配。

`has` 和 `missing` 項目可以有下列欄位：

- `type`: `String` - 必須為 `header`、`cookie`、`host` 或 `query` 之一。
- `key`: `String` - 選定類型中要匹配的鍵。
- `value`: `String` 或 `undefined` - 要檢查的值，若為 undefined 則任意值皆可匹配。可用類似正則的字串來抓取值中的特定部分，例如若 `first-(?<paramName>.*)` 用於 `first-second`，則 `second` 可在 destination 中搭配 `:paramName` 使用。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      // if the header `x-rewrite-me` is present,
      // this rewrite will be applied
      {
        source: '/:path*',
        has: [
          {
            type: 'header',
            key: 'x-rewrite-me',
          },
        ],
        destination: '/another-page',
      },
      // if the header `x-rewrite-me` is not present,
      // this rewrite will be applied
      {
        source: '/:path*',
        missing: [
          {
            type: 'header',
            key: 'x-rewrite-me',
          },
        ],
        destination: '/another-page',
      },
      // if the source, query, and cookie are matched,
      // this rewrite will be applied
      {
        source: '/specific/:path*',
        has: [
          {
            type: 'query',
            key: 'page',
            // the page value will not be available in the
            // destination since value is provided and doesn't
            // use a named capture group e.g. (?<page>home)
            value: 'home',
          },
          {
            type: 'cookie',
            key: 'authorized',
            value: 'true',
          },
        ],
        destination: '/:path*/home',
      },
      // if the header `x-authorized` is present and
      // contains a matching value, this rewrite will be applied
      {
        source: '/:path*',
        has: [
          {
            type: 'header',
            key: 'x-authorized',
            value: '(?<authorized>yes|true)',
          },
        ],
        destination: '/home?authorized=:authorized',
      },
      // if the host is `example.com`,
      // this rewrite will be applied
      {
        source: '/:path*',
        has: [
          {
            type: 'host',
            value: 'example.com',
          },
        ],
        destination: '/another-page',
      },
    ]
  },
}
```

## 重寫至外部 URL

<details>

  <summary>範例</summary>

- [使用多個區域 (Using Multiple Zones)](https://github.com/vercel/next.js/tree/canary/examples/with-zones)

</details>

重寫允許您將請求重寫至外部 URL。這對於逐步採用 Next.js 特別有用。以下範例為將主應用程式的 `/blog` 路由重寫至外部網站。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return [
      {
        source: '/blog',
        destination: 'https://example.com/blog',
      },
      {
        source: '/blog/:slug',
        destination: 'https://example.com/blog/:slug', // Matched parameters can be used in the destination
      },
    ]
  },
}
```

如果您使用 `trailingSlash: true`，您還需要在 `source` 參數中插入結尾斜線。如果目標伺服器也期待結尾斜線，則也應包含在 `destination` 參數中。

```js filename="next.config.js"
module.exports = {
  trailingSlash: true,
  async rewrites() {
    return [
      {
        source: '/blog/',
        destination: 'https://example.com/blog/',
      },
      {
        source: '/blog/:path*/',
        destination: 'https://example.com/blog/:path*/',
      },
    ]
  },
}
```

### Next.js 的逐步採用

您還可以让 Next.js 在檢查所有 Next.js 路由後回退至代理至現有網站。

這樣當您將更多頁面遷移至 Next.js 時，就不必更改重寫配置。

```js filename="next.config.js"
module.exports = {
  async rewrites() {
    return {
      fallback: [
        {
          source: '/:path*',
          destination: `https://custom-routes-proxying-endpoint.vercel.app/:path*`,
        },
      ],
    }
  },
}
```

### 支援 basePath 的重寫

當在重寫中使用 [`basePath 支援`](/docs/app/api-reference/config/next-config-js/basePath) 時，每個 `source` 和 `destination` 都會自動加上 `basePath` 前綴，除非您在重寫中添加 `basePath: false`：

```js filename="next.config.js"
module.exports = {
  basePath: '/docs',

  async rewrites() {
    return [
      {
        source: '/with-basePath', // automatically becomes /docs/with-basePath
        destination: '/another', // automatically becomes /docs/another
      },
      {
        // does not add /docs to /without-basePath since basePath: false is set
        // Note: this cannot be used for internal rewrites e.g. `destination: '/another'`
        source: '/without-basePath',
        destination: 'https://example.com',
        basePath: false,
      },
    ]
  },
}
```

### 與 i18n 支援的重寫

當在重寫中使用 [`i18n 支援`](/docs/pages/guides/internationalization) 時，每個 `source` 和 `destination` 都會自動加上前綴以處理配置的 `locales`，除非您在重寫中添加 `locale: false`。若使用 `locale: false`，您必須在 `source` 和 `destination` 前加上區域設定以正確匹配。

```js filename="next.config.js"
module.exports = {
  i18n: {
    locales: ['en', 'fr', 'de'],
    defaultLocale: 'en',
  },

  async rewrites() {
    return [
      {
        source: '/with-locale', // automatically handles all locales
        destination: '/another', // automatically passes the locale on
      },
      {
        // does not handle locales automatically since locale: false is set
        source: '/nl/with-locale-manual',
        destination: '/nl/another',
        locale: false,
      },
      {
        // this matches '/' since `en` is the defaultLocale
        source: '/en',
        destination: '/en/another',
        locale: false,
      },
      {
        // it's possible to match all locales even when locale: false is set
        source: '/:locale/api-alias/:path*',
        destination: '/api/:path*',
        locale: false,
      },
      {
        // this gets converted to /(en|fr|de)/(.*) so will not match the top-level
        // `/` or `/fr` routes like /:path* would
        source: '/(.*)',
        destination: '/another',
      },
    ]
  },
}
```

</PagesOnly>
