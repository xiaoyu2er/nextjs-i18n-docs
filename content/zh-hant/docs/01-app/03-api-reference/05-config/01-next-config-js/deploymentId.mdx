「deploymentId」選項允許您為部署設定識別符。此識別符用於 [版本偏移](/docs/app/guides/self-hosting#version-skew) 保護以及滾動部署期間的快取失效。

```js filename="next.config.js"
module.exports = {
  deploymentId: 'my-deployment-id',
}
```

您也可以使用 `NEXT_DEPLOYMENT_ID` 環境變數來設定部署 ID：

```bash
NEXT_DEPLOYMENT_ID=my-deployment-id next build
```

> **提示**：如果兩者都設定，`next.config.js` 中的 `deploymentId` 值會優先於 `NEXT_DEPLOYMENT_ID` 環境變數。

## 運作原理

當設定 `deploymentId` 時，Next.js 會：

1. 在靜態資源 URL（JavaScript、CSS、圖片）後附加 `?dpl=<deploymentId>`
2. 為客戶端導航請求添加 `x-deployment-id` 標頭
3. 為導航回應添加 `x-nextjs-deployment-id` 標頭
4. 在 `<html>` 元素上注入 `data-dpl-id` 屬性

當客戶端偵測到其部署 ID 與伺服器（透過回應標頭）之間出現不匹配時，會觸發硬式導航（整頁重新整理），而不是客戶端導航。這確保用戶總是從一致的部署版本接收資源 <AppOnly>和伺服器函數</AppOnly>。

> **提示**：Next.js 不會讀取對請求上的 `?dpl=` 查詢參數。查詢參數是用於快取失效（確保瀏覽器與 CDN 取得最新資源），而非用於路由。若您需具備版本感知路由，請查閱您的託管提供者或 CDN 的技術文件以實作基於部署的路由。

## 使用情境

### 滾動部署

在滾動部署期間，某些伺服器執行個體可能正在執行新版本，而其他執行個體則仍在執行舊版本。若未設定部署 ID，使用者可能會收到舊版與新版資源的混合，導致錯誤。

為每次部署設定一致的 `deploymentId` 可確保：

<AppOnly>

- 客戶端始終從配對的部署版本請求資源
- 不匹配會觸發完全重新整理以獲取正確的資源
- 伺服器函數跨部署邊界能正確運作

</AppOnly>

<PagesOnly>

- 客戶端始終從配對的部署版本請求資源
- 不匹配會觸發完全重新整理以獲取正確的資源

</PagesOnly>

### 多伺服器環境

在負載平衡器後執行多個 Next.js 應用程式執行個體時，同一部署的所有執行個體應使用相同的 `deploymentId`。

```js filename="next.config.js"
module.exports = {
  deploymentId: process.env.DEPLOYMENT_VERSION || process.env.GIT_SHA,
}
```
