## 使用方法

關於在 `<head>` 中內嵌 CSS 的實驗性支援。啟用此標誌時，所有原本會生成 `<link>` 標籤的地方將改為使用生成的 `<style>` 標籤。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    inlineCss: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    inlineCss: true,
  },
}

module.exports = nextConfig
```

## 取捨 (Trade-Offs)

- **啟用 (Enable)** 如果您使用原子化 CSS（如 Tailwind）並希望優化新訪客的首次加載性能
- **跳過 (Skip)** 如果回頭訪客常見，並且希望他們能充分利用已緩存的樣式表

### 內聯 CSS 何時有益

通常情況下，瀏覽器必須下載 HTML、解析它，發現 CSS <link> 標籤，然後請求樣式表才能渲染。使用內聯 CSS [消除了此請求瀑布](https://web.dev/learn/performance/optimize-resource-loading#inline_critical_css)，樣式會與 HTML 同時到達，這樣瀏覽器即可立即渲染。

這一優勢在以下情況下最顯著：

- **首次訪客**: 由於 CSS 文件是渲染阻塞的，內聯消除了首次訪客體驗到的初始加載延遲。擁有已緩存樣式表的回頭訪客不會看到這一優勢。

- **性能指標**: 通過去除額外的 CSS 文件網絡請求，內聯可以顯著改善 [首次內容渲染 (FCP)](https://web.dev/articles/fcp) 和 [最大內容渲染 (LCP)](https://web.dev/articles/lcp)。

- **連接速度慢**: 對於處於高延遲網絡中的用戶，每個額外請求都會增加延遲。內聯減少了往返次數，當連接速度慢時這一點尤為重要。

- **原子化 CSS (Tailwind)**: 基於工具類優先 (Utility-first) 的框架僅生成您使用的 CSS 類，保持 CSS 體積小。頁面的樣式不會隨頁面複雜度成比例增長——無論構建多少 UI，它們通常都很緊湊。這使得內聯變得實用，因為您可以在不顯著膨脹 HTML 的同時獲得性能優勢。

### 何時使用外部 CSS 更佳

內聯樣式無法與 HTML 分開緩存。每次頁面加載都會重新下載相同的 CSS。

這一權衡在以下情況下最為關鍵：

- **回頭訪客**: 反覆訪問您網站的用戶會受益于已緩存的外部樣式表。使用內聯時，他們每次訪問都會重新下載樣式。

- **大型 CSS 彙編包**: 外部樣式表獨立緩存，並在現代基礎設施上高效加載。內聯的 CSS 會隨每個 HTML 響應到達，增加 [首字節時間 (TTFB)](https://web.dev/articles/ttfb)，並阻止瀏覽器將樣式分開緩存。這一權衡適用於小型 CSS（基於原子化的框架，如 Tailwind），但會增加大型彙編包（如 Bootstrap 或 Material UI 等組件庫）的開銷。

- **多個頁面共享樣式**: 在一個頁面緩存的外部樣式表可加速導航到其他頁面。內聯樣式無跨頁面緩存優勢。

> **值得了解**:
>
> 此功能目前處於實驗階段，且存在一些已知限制：
>
> - CSS 內聯是全局應用的，無法基於每頁進行配置
> - 樣式在初次加載頁面時會被重複——一次在 SSR 的 <style> 標籤內，一次在 RSC 負載中
> - 當導航到預渲染頁面時，樣式將使用 <link> 標籤而不是內聯 CSS，以避免重複
> - 此功能在開發模式下不可用，僅在生產構建中有效
