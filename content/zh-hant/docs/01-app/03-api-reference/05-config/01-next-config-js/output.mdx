構建期間，Next.js 會自動追蹤每個頁面及其相依性，以確定部署應用程式生產版本所需的所有檔案。

此功能有助於大幅縮小部署的大小。過去，當使用 Docker 部署時，您需要安裝套件 `dependencies` 中的所有檔案才能執行 `next start`。自 Next.js 12 起，你可以利用 `.next/` 目錄中的輸出檔案追蹤機能，僅包含必要的檔案。

此外，這也消除了對已棄用的 `serverless` 目標的需求，這可能會導致各種問題，並且會產生不必要的重複。

## 運作方式

在 `next build` 期間，Next.js 將使用 [`@vercel/nft`](https://github.com/vercel/nft) 靜態分析 `import`、`require` 和 `fs` 的使用情況，以確定頁面可能會載入的所有檔案。

Next.js 的生產伺服器所需的檔案也會被追蹤，並輸出至 `.next/next-server.js.nft.json`，可在生產環境中利用。

若要利用發送至 `.next` 輸出目錄的 `.nft.json` 檔案，您可以讀取每個追蹤中相對於 `.nft.json` 檔案的檔案清單，然後將它們複製到您的部署位置。

## 自動複製追蹤檔案

Next.js 可以自動建立 `standalone` 資料夾，其中僅複製部署生產版本所需的必要檔案，包括 `node_modules` 中的部分檔案。

若要利用此自動複製功能，您可以在 `next.config.js` 中啟用它：

```js filename="next.config.js"
module.exports = {
  output: 'standalone',
}
```

這會建立一個位於 `.next/standalone` 的資料夾，然後即可單獨部署，無需安裝 `node_modules`。

此外，還會輸出一個最小的 `server.js` 檔案，可用作 `next start` 的替代方案。此最小伺服器預設不會複製 `public` 或 `.next/static` 資料夾，因為這些檔案本應由 CDN 處理，但這些資料夾可以手動複製到 `standalone/public` 和 `standalone/.next/static` 資料夾中，之後 `server.js` 檔案將在這些資料夾中自動提供服務。

若要手動複製這些檔案，您可以在執行 `next build` 後使用 `cp` 命令列工具：

```bash filename="Terminal"
cp -r public .next/standalone/ && cp -r .next/static .next/standalone/.next/
```

要在本地環境啟動您的最小 `server.js` 檔案，請執行以下命令：

```bash filename="Terminal"
node .next/standalone/server.js
```

## 注意事項

- 在 monorepo 設定中進行追蹤時，預設使用專案資料夾進行追蹤。對於 `next build packages/web-app`，`packages/web-app` 將作為追蹤根目錄，該資料夾外的任何檔案都不會包含在內。若要在該資料夾外包含檔案，您可以在 `next.config.js` 中設定 `outputFileTracingRoot`。

```js filename="packages/web-app/next.config.js"
const path = require('path')

module.exports = {
  // this includes files from the monorepo base two directories up
  outputFileTracingRoot: path.join(__dirname, '../../'),
}
```

- 在某些情況下，Next.js 可能無法包含所需的檔案，或者可能錯誤地包含未使用的檔案。在這些情況下，您可以分別在 `next.config.js` 中使用 `outputFileTracingExcludes` 和 `outputFileTracingIncludes`。每個選項接受一個物件，其鍵為 **路徑通配符 (route globs)**（使用 [picomatch](https://www.npmjs.com/package/picomatch#basic-globbing) 與 Route 路徑匹配，例如 `/api/hello`），其值為 **從專案根目錄解析的全域模式 (glob patterns)**，用於指定要在追蹤中包含或排除的檔案。

> **值得注意**：
> 在混合儲存庫 (Monorepo) 中，`project root` 指的是 Next.js 專案根目錄（包含 `next.config.js` 的資料夾，例如 `packages/web-app`），而非混合儲存庫 (Monorepo) 的根目錄。

```js filename="next.config.js"
module.exports = {
  outputFileTracingExcludes: {
    '/api/hello': ['./un-necessary-folder/**/*'],
  },
  outputFileTracingIncludes: {
    '/api/another': ['./necessary-folder/**/*'],
    '/api/login/\\[\\[\\.\\.\\.slug\\]\\]': [
      './node_modules/aws-crt/dist/bin/**/*',
    ],
  },
}
```

使用 `src/` 資料夾不會改變您撰寫這些選項的方式：

- **鍵 (Keys)** 仍然匹配 Route 路徑（`'/api/hello'`、`'/products/[id]'` 等）。
- **值 (Values)** 可以引用 `src/` 下的路徑，因為它們是相對於專案根目錄解析的。

```js filename="next.config.js"
module.exports = {
  outputFileTracingIncludes: {
    '/products/*': ['src/lib/payments/**/*'],
    '/*': ['src/config/runtime/**/*.json'],
  },
  outputFileTracingExcludes: {
    '/api/*': ['src/temp/**/*', 'public/large-logs/**/*'],
  },
}
```

您也可以使用全域鍵（如 `'/*'`）來鎖定所有 Route：

```js filename="next.config.js"
module.exports = {
  outputFileTracingIncludes: {
    '/*': ['src/i18n/locales/**/*.json'],
  },
}
```

這些選項套用至伺服器追蹤，且不會影響不會產生伺服器追蹤檔案的 Route：

- Edge Runtime Route 不受影響。
- 完全靜態頁面不受影響。

在 monorepo 環境或當您需要包含應用資料夾外的檔案時，請將 `outputFileTracingRoot` 與 includes 結合使用：

```js filename="next.config.js"
const path = require('path')

module.exports = {
  // Trace from the monorepo root
  outputFileTracingRoot: path.join(__dirname, '../../'),
  outputFileTracingIncludes: {
    '/route1': ['../shared/assets/**/*'],
  },
}
```

> **小知識**：
>
> - 在模式中使用正斜線 (`/`) 以確保跨平台相容性。
> - 將模式保持得愈窄愈好，以避免過大的追蹤（避免在 Repo 根目錄使用 `**/*`）。

原始碼/執行時資源常用的包含模式：

```js filename="next.config.js"
module.exports = {
  outputFileTracingIncludes: {
    '/*': ['node_modules/sharp/**/*', 'node_modules/aws-crt/dist/bin/**/*'],
  },
}
```
