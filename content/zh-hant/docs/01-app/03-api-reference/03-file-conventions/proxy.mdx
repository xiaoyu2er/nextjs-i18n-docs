> **注意**：`middleware` 檔案慣例已停用，並已重新命名為 `proxy`。詳細資訊請參閱 [遷移至 Proxy](#migration-to-proxy)。

`proxy.js|ts` 檔案用於撰寫 [Proxy](/docs/app/getting-started/proxy) 並在被請求完成之前在伺服器上執行程式碼。接著，根據傳入的請求，您可以透過重新寫入、重新導向、修改請求或回應標頭，或直接回應來修改回應。

Proxy 會在路由渲染之前執行。它特別適合用於實現自訂的伺服器端邏輯，例如身份驗證、記錄日誌或處理重新導向。

> **知識小補**:
>
> Proxy 預定要與您的渲染程式碼分開調用，並且在最佳化情況下會部署到您的 CDN 以快速處理重新導向/重新寫入，您不應嘗試依賴共用模組或全域變數。
>
> 若要將資訊從 Proxy 傳遞到您的應用程式，請使用 [標頭](#setting-headers)、[Cookie](#using-cookies)、[重新寫入](/docs/app/api-reference/functions/next-response#rewrite)、[重新導向](/docs/app/api-reference/functions/next-response#redirect) 或 URL。

在專案根目錄或 `src`（如適用）內建立 `proxy.ts`（或 `.js`）檔案，使其與 `pages` 或 `app` 位於同一層級。

如果您已自訂 [`pageExtensions`](/docs/app/api-reference/config/next-config-js/pageExtensions)，例如設定為 `.page.ts` 或 `.page.js`，請將您的檔案名稱對應改為 `proxy.page.ts` 或 `proxy.page.js`。

```tsx filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

```js filename="proxy.js" switcher
import { NextResponse } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

## 匯出

### Proxy 函式

檔案必須匯出一個函式，可為預設匯出或命名為 `proxy`。請注意，不支援從同一檔案匯出多個 proxy。

```js filename="proxy.js"
// Example of default export
export default function proxy(request) {
  // Proxy logic
}
```

### 設定物件（選用）

選用，可以與 Proxy 函式一起匯出配置物件。此物件包含 [matcher](#matcher) 以指定 Proxy 適用的路徑。

### 比對器 (Matcher)

`matcher` 選項允許您指定 Proxy 運行所針對的特定路徑。您可以透過以下方式指定這些路徑：

- 針對單一路徑：直接使用方法字串定義路徑，例如 `'/about'`。
- 針對多個路徑：使用陣列列出多個路徑，例如 `matcher: ['/about', '/contact']`，這將 Proxy 套用至 `/about` 和 `/contact` 兩者。

```js filename="proxy.js"
export const config = {
  matcher: ['/about/:path*', '/dashboard/:path*'],
}
```

此外，`matcher` 選項支援使用正規表示式來指定複雜路徑。例如，您可以使用正規表示式 matcher 來排除特定路徑：

```js filename="proxy.js"
export const config = {
  matcher: [
    // Exclude API routes, static files, image optimizations, and .png files
    '/((?!api|_next/static|_next/image|.*\\.png$).*)',
  ],
}
```

這提供了精確控制要包含或排除哪些路徑的能力。

`matcher` 選項接受包含以下鍵的物件陣列：

- `source`：用於匹配請求路徑的路徑或模式。它可以是用於直接路徑匹配的語言字串，或是用於較複雜匹配的模式。
- `locale` (選用)：布林值，當設定為 `false` 時，在路徑匹配中忽略基於區域的語系路由。
- `has` (選用)：基於特定請求元素的存在來指定條件，例如標頭、查詢參數或 Cookie。
- `missing` (選用)：專注於特定請求元素缺席的條件，例如缺失標頭或 Cookie。

```js filename="proxy.js"
export const config = {
  matcher: [
    {
      source: '/api/:path*',
      locale: false,
      has: [
        { type: 'header', key: 'Authorization', value: 'Bearer Token' },
        { type: 'query', key: 'userId', value: '123' },
      ],
      missing: [{ type: 'cookie', key: 'session', value: 'active' }],
    },
  ],
}
```

`source` 路徑模式：

1. 必須以 `/` 開頭
2. 可包含命名參數：`/about/:path` 匹配 `/about/a` 和 `/about/b`，但不匹配 `/about/a/c`
3. 命名參數（以 `:` 開頭）可有修飾符：`/about/:path*` 匹配 `/about/a/b/c`，因為 `*` 是 _零或多個_。`?` 是 _零或一個_，`+` 是 _一或多個_
4. 可使用括號內的正規表示式：`/about/(.*)` 等同於 `/about/:path*`
5. 固定於路徑開頭：`/about` 匹配 `/about` 和 `/about/team`，但不匹配 `/blog/about`

有關更多詳細資訊，請參閱 [path-to-regexp](https://github.com/pillarjs/path-to-regexp#path-to-regexp-1) 文件。

> **重要資訊**：
>
> - `matcher` 值必須是常數，以便在編譯時進行靜態分析。動態值（如變數）將被忽略。
> - 為了向後相容性，Next.js 總是將 `/public` 視為 `/public/index`。因此，`/public/:path` 的比對器會匹配。

## Params

### `request`

定義 Proxy 時，預設匯出函式接受單一參數 `request`。此參數為 `NextRequest` 的實例，代表傳入的 HTTP 請求。

```tsx filename="proxy.ts" switcher
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Proxy logic goes here
}
```

```js filename="proxy.js" switcher
export function proxy(request) {
  // Proxy logic goes here
}
```

如果您偏好捷徑，可以使用 `NextProxy` 型別。它會自動推斷 `request` (`NextRequest`) 和 `event` (`NextFetchEvent`) 的參數型別：

```tsx filename="proxy.ts"
import type { NextProxy } from 'next/server'

export const proxy: NextProxy = (request, event) => {
  event.waitUntil(Promise.resolve())
  return Response.json({ pathname: request.nextUrl.pathname })
}
```

> **知識小補**:
>
> - `NextRequest` 是代表 Next.js Proxy 中傳入 HTTP 請求的型別，而 [`NextResponse`](#nextresponse) 是用來操作和傳回 HTTP 回應的類別。

## NextResponse

`NextResponse` API 允許您：

- `redirect` 將傳入請求重導向到不同的 URL
- `rewrite` 重寫回應，顯示給定的 URL
- 設定 API Routes、`getServerSideProps` 和 `rewrite` 目的地的請求標頭
- 設定回應 Cookie
- 設定回應標頭

<AppOnly>

若要從 Proxy 產生回應，您可以：

1. `rewrite` 重寫到產生回應的路由 ([Page](/docs/app/api-reference/file-conventions/page) 或 [Route Handler](/docs/app/api-reference/file-conventions/route))
2. 直接返回 `NextResponse`。請參閱 [產生回應](#producing-a-response)

> **重要資訊**：對於重導向，您也可以使用 `Response.redirect` 代替 `NextResponse.redirect`。

</AppOnly>

<PagesOnly>

若要從 Proxy 產生回應，您可以：

1. `rewrite` 重寫到產生回應的路由 ([Page](/docs/pages/building-your-application/routing/pages-and-layouts) 或 [Edge API Route](/docs/pages/building-your-application/routing/api-routes))
2. 直接返回 `NextResponse`。請參閱 [產生回應](#producing-a-response)

</PagesOnly>

## 執行順序

Proxy 將會在您的專案中 **每一個路由** 上被調用。鑑於此，使用 [matchers](#matcher) 來精確針對或排除特定路由至關重要。以下是執行順序：

1. `next.config.js` 中的 `headers`
2. `next.config.js` 中的 `redirects`
3. Proxy (`rewrites`, `redirects`, 等)
4. `next.config.js` 中的 `beforeFiles` (`rewrites`)
5. 檔案系統路由 (`public/`, `_next/static/`, `pages/`, `app/`, 等)
6. `next.config.js` 中的 `afterFiles` (`rewrites`)
7. 動態路由 (`/blog/[slug]`)
8. `next.config.js` 中的 `fallback` (`rewrites`)

> **知識小補**：[Server Functions](/docs/app/api-reference/directives/use-server) 在此鏈中不是單獨的路由。它們被處理為對使用它們的路由的 POST 請求，因此排除某路徑的 Proxy matcher 也會跳過該路徑上的 Server Function 調用。
>
> Matcher 變更或將 Server Function 移至不同路由的重構可能會靜默移除 Proxy 涵蓋範圍。請務必在每個 Server Function 內部驗證身份驗證和授權，而不是僅依賴 Proxy。請參閱 [資料安全指南](/docs/app/guides/data-security#authentication-and-authorization) 查看建議模式。

## 執行階段

Proxy 預設使用 Node.js 執行階段。`runtime` 配置選項在 Proxy 檔案中不可用。在 Proxy 中設定 `runtime` 配置選項會引發錯誤。

## 進階 Proxy 標誌

在 Next.js `v13.1` 中，為 proxy 新增了兩個額外標誌：`skipProxyUrlNormalize`（原名 `skipMiddlewareUrlNormalize`）和 `skipTrailingSlashRedirect`，用以處理進階使用案例。

`skipTrailingSlashRedirect` 停用 Next.js 為新增或移除結尾斜線而產生的重新導向。這允許在 proxy 內部進行自訂處理，以便某些路徑保留結尾斜線而有些不保留，這能使漸進式遷移更容易。

```js filename="next.config.js"
module.exports = {
  skipTrailingSlashRedirect: true,
}
```

```js filename="proxy.js"
const legacyPrefixes = ['/docs', '/blog']

export default async function proxy(req) {
  const { pathname } = req.nextUrl

  if (legacyPrefixes.some((prefix) => pathname.startsWith(prefix))) {
    return NextResponse.next()
  }

  // apply trailing slash handling
  if (
    !pathname.endsWith('/') &&
    !pathname.match(/((?!\.well-known(?:\/.*)?)(?:[^/]+\/)*[^/]+\.\w+)/)
  ) {
    return NextResponse.redirect(
      new URL(`${req.nextUrl.pathname}/`, req.nextUrl)
    )
  }
}
```

`skipProxyUrlNormalize` 允許停用 Next.js 中的 URL 正規化，以便直接參訪和客戶端轉換的處理方式相同。在某些進階情況下，此選項透過使用原始 URL 來提供完全控制。

```js filename="next.config.js"
module.exports = {
  skipProxyUrlNormalize: true,
}
```

```js filename="proxy.js"
export default async function proxy(req) {
  const { pathname } = req.nextUrl

  // GET /_next/data/build-id/hello.json

  console.log(pathname)
  // with the flag this now /_next/data/build-id/hello.json
  // without the flag this would be normalized to /hello
}
```

### 使用 Cookie

Cookie 是常規標頭。在 `Request` 上，它們儲存在 `Cookie` 標頭中。在 `Response` 上，它們位於 `Set-Cookie` 標頭中。Next.js 提供了通過 `NextRequest` 和 `NextResponse` 上的 `cookies` 擴充功能來訪問和操作這些 Cookie 的便捷方式。

1. 對於傳入請求，`cookies` 帶有以下方法：`get`, `getAll`, `set` 和 `delete` Cookie。您可以使用 `has` 檢查 Cookie 是否存在，或與 `clear` 移除所有 Cookie。
2. 對於傳出回應，`cookies` 具有 `get`, `getAll`, `set` 和 `delete` 方法。

```ts filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Assume a "Cookie:nextjs=fast" header to be present on the incoming request
  // Getting cookies from the request using the `RequestCookies` API
  let cookie = request.cookies.get('nextjs')
  console.log(cookie) // => { name: 'nextjs', value: 'fast', Path: '/' }
  const allCookies = request.cookies.getAll()
  console.log(allCookies) // => [{ name: 'nextjs', value: 'fast' }]

  request.cookies.has('nextjs') // => true
  request.cookies.delete('nextjs')
  request.cookies.has('nextjs') // => false

  // Setting cookies on the response using the `ResponseCookies` API
  const response = NextResponse.next()
  response.cookies.set('vercel', 'fast')
  response.cookies.set({
    name: 'vercel',
    value: 'fast',
    path: '/',
  })
  cookie = response.cookies.get('vercel')
  console.log(cookie) // => { name: 'vercel', value: 'fast', Path: '/' }
  // The outgoing response will have a `Set-Cookie:vercel=fast;path=/` header.

  return response
}
```


### 設定標頭

您可以使用 `NextResponse` API 設定請求和回應標頭（設定 _請求_ 標頭自 Next.js v13.0.0 起可用）。

```ts filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Clone the request headers and set a new header `x-hello-from-proxy1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-proxy1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-proxy2`
  response.headers.set('x-hello-from-proxy2', 'hello')
  return response
}
```

```js filename="proxy.js" switcher
import { NextResponse } from 'next/server'

export function proxy(request) {
  // Clone the request headers and set a new header `x-hello-from-proxy1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-proxy1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-proxy2`
  response.headers.set('x-hello-from-proxy2', 'hello')
  return response
}
```

注意此程式碼片段使用：

- `NextResponse.next({ request: { headers: requestHeaders } })` 使 `requestHeaders` 對上游可用
- **非** `NextResponse.next({ headers: requestHeaders })`，這使 `requestHeaders` 對客戶端可用

請參閱 [Proxy 中的 NextResponse 標頭](/docs/app/api-reference/functions/next-response#next) 以獲取更多資訊。

> **重要資訊**：避免設定大型標頭，這可能會導致根據您的後端 web 伺服器設定而出現 [431 Request Header Fields Too Large](https://developer.mozilla.org/docs/Web/HTTP/Status/431) 錯誤。

#### RSC 請求與重新寫入

在 RSC 請求期間，Next.js 會從 Proxy 中的 `request` 實例中移除內部 Flight 標頭。例如，`rsc`、`next-router-state-tree` 和 `next-router-prefetch` 等標頭無法透過 `request.headers` 暴露。這是為了防止意外地以不同方式處理 RSC 請求與 HTML 請求，因為兩者需要一致。

當您使用 `NextResponse.rewrite()` 時，Next.js 會自動向上游傳播所需的 RSC 重新寫入標頭。

如果您使用 `fetch()` 而非 `NextResponse.rewrite()` 實作自訂重新寫入邏輯，除非您手動轉發它們，否則可能會遇到缺失的 RSC 標頭。

對於自訂 `fetch` 重新寫入設定，您也可以啟用 `next.config.js` 中的 `skipProxyUrlNormalize`，讓您的重新寫入邏輯能從提供的請求物件接收必要的 URL 形狀和 RSC 標頭：

```js filename="next.config.js"
module.exports = {
  skipProxyUrlNormalize: true,
}
```

### CORS

您可以設定 Proxy 中的 CORS 標頭，以允許跨 origin 請求，包括 [簡單](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#simple_requests) 和 [預檢](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#preflighted_requests) 請求。

```tsx filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

const allowedOrigins = ['https://acme.com', 'https://my-app.org']

const corsOptions = {
  'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type, Authorization',
}

export function proxy(request: NextRequest) {
  // Check the origin from the request
  const origin = request.headers.get('origin') ?? ''
  const isAllowedOrigin = allowedOrigins.includes(origin)

  // Handle preflighted requests
  const isPreflight = request.method === 'OPTIONS'

  if (isPreflight) {
    const preflightHeaders = {
      ...(isAllowedOrigin && { 'Access-Control-Allow-Origin': origin }),
      ...corsOptions,
    }
    return NextResponse.json({}, { headers: preflightHeaders })
  }

  // Handle simple requests
  const response = NextResponse.next()

  if (isAllowedOrigin) {
    response.headers.set('Access-Control-Allow-Origin', origin)
  }

  Object.entries(corsOptions).forEach(([key, value]) => {
    response.headers.set(key, value)
  })

  return response
}

export const config = {
  matcher: '/api/:path*',
}
```

### 產生回應

您可以透過回傳 `Response` 或 `NextResponse` 實例直接從 Proxy 回應。（此功能自 [Next.js v13.1.0](https://nextjs.org/blog/next-13-1#nextjs-advanced-proxy) 起可用）

```ts filename="proxy.ts" switcher
import type { NextRequest } from 'next/server'
import { isAuthenticated } from '@lib/auth'

// Limit the proxy to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
}

export function proxy(request: NextRequest) {
  // Call our authentication function to check the request
  if (!isAuthenticated(request)) {
    // Respond with JSON indicating an error message
    return Response.json(
      { success: false, message: 'authentication failed' },
      { status: 401 }
    )
  }
}
```

```js filename="proxy.js" switcher
import { isAuthenticated } from '@lib/auth'

// Limit the proxy to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
}

export function proxy(request) {
  // Call our authentication function to check the request
  if (!isAuthenticated(request)) {
    // Respond with JSON indicating an error message
    return Response.json(
      { success: false, message: 'authentication failed' },
      { status: 401 }
    )
  }
}
```

### 負向匹配

`matcher` 設定允許完整的正則表達式，因此支援像負向預覽或字元匹配的匹配。以下是用於匹配特定路徑以外所有路徑的負向預覽範例：

```js filename="proxy.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
  ],
}
```

您也可以透過使用 `missing` 或 `has` 陣列，或同時使用兩者，讓特定請求跳過 Proxy：

```js filename="proxy.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      missing: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [{ type: 'header', key: 'x-present' }],
      missing: [{ type: 'header', key: 'x-missing', value: 'prefetch' }],
    },
  ],
}
```

> **知識小補**:
>
> 即使在負向 matcher 模式中排除了 `_next/data`，proxy 仍會針對 `_next/data` 路由被調用。這是預期的行為，以防止潛在的安全問題，您可能保護了頁面卻忘記保護對應的資料路由。

```js filename="proxy.js"
export const config = {
  matcher:
    '/((?!api|_next/data|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
}

// Proxy will still run for /_next/data/* routes despite being excluded
```

### `waitUntil` 和 `NextFetchEvent`

`NextFetchEvent` 物件擴充了原生的 [`FetchEvent`](https://developer.mozilla.org/docs/Web/API/FetchEvent) 物件，並包含 [`waitUntil()`](https://developer.mozilla.org/docs/Web/API/ExtendableEvent/waitUntil) 方法。

`waitUntil()` 方法接受 promise 作為參數，並延長 Proxy 的生命週期直到 promise 執行完畢。這對於在背景執行工作很有用。

```ts filename="proxy.ts"
import { NextResponse } from 'next/server'
import type { NextFetchEvent, NextRequest } from 'next/server'

export function proxy(req: NextRequest, event: NextFetchEvent) {
  event.waitUntil(
    fetch('https://my-analytics-platform.com', {
      method: 'POST',
      body: JSON.stringify({ pathname: req.nextUrl.pathname }),
    })
  )

  return NextResponse.next()
}
```

### 單元測試 (實驗性)

從 Next.js 15.1 開始，`next/experimental/testing/server` 套件包含用於幫助測試 proxy 檔案的單個單元測試工具。單個單元測試 proxy 可確保它僅在所需路徑上運行，並在程式碼送達生產環境之前確保自訂路由邏輯按預期運作。

`unstable_doesProxyMatch` 函式可用於斷言 proxy 是否會為提供的 URL、標頭和 Cookie 執行。

```js
import { unstable_doesProxyMatch } from 'next/experimental/testing/server'

expect(
  unstable_doesProxyMatch({
    config,
    nextConfig,
    url: '/test',
  })
).toEqual(false)
```

整個 proxy 函式也可以進行測試。

```js
import { isRewrite, getRewrittenUrl } from 'next/experimental/testing/server'

const request = new NextRequest('https://nextjs.org/docs')
const response = await proxy(request)
expect(isRewrite(response)).toEqual(true)
expect(getRewrittenUrl(response)).toEqual('https://other-domain.com/docs')
// getRedirectUrl could also be used if the response were a redirect
```

## 遷移至 Proxy

### 變更原因

將 `middleware` 重新命名的原因是「middleware」一詞通常容易與 Express.js middleware 混淆，導致誤解其用途。此外，Middleware 功能強大，可能會鼓勵使用；然而，建議將此功能作為最後手段使用。

Next.js 正在邁向提供更好的 API 和更佳的易用性，以使開發人員能夠在沒有 Middleware 的情況下達成目標。這就是重新命名 `middleware` 的原因。

### 為何使用 "Proxy"

Proxy 名稱說明了 Middleware 具備的功能。「proxy」一詞暗示它在應用程式前方具有網路邊界，這是 Middleware 的行為。此外，Middleware 預設在 [Edge Runtime](/docs/app/api-reference/edge) 執行，可以更靠近客戶端執行，與應用程式的區域分開。這些行為與「proxy」一词更契合，並提供了更明確的功能目的。

### 如何遷移

我們建議用戶避免依賴 Middleware，除非無其他選項可用。我們的目標是提供更具易用性的 API，讓他們無需 Middleware 即可達成目標。

「middleware」一詞通常會讓用戶與 Express.js middleware 混淆，這可能鼓勵誤用。為了釐清方向，我們將檔案慣例重新命名為「proxy」。這強調我們正遠離 Middleware，拆解其過度負載的功能，並明確 Proxy 的用途。

Next.js 提供 codemod 以從 `middleware.ts` 遷移至 `proxy.ts`。您可以執行以下命令進行遷移：

```bash
npx @next/codemod@canary middleware-to-proxy .
```

Codemod 將把檔案和函式名稱從 `middleware` 重新命名為 `proxy`。

```diff
// middleware.ts -> proxy.ts

- export function middleware() {
+ export function proxy() {
```
