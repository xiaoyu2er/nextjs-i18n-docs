當您無法提前得知確切的路由區段名稱，並希望從動態資料建立路由時，您可以使用在請求時填充或建置時預先渲染的動態區段。

## 慣例

動態區段可以透過將資料夾名稱用方括號括起來建立：`[folderName]`。例如，部落格可能包含以下路由 `app/blog/[slug]/page.js`，其中 `[slug]` 是部落格文章的動態區段。

```tsx filename="app/blog/[slug]/page.tsx" switcher
export default async function Page({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  return <div>My Post: {slug}</div>
}
```

```jsx filename="app/blog/[slug]/page.js" switcher
export default async function Page({ params }) {
  const { slug } = await params
  return <div>My Post: {slug}</div>
}
```

動態區段作為 `params` prop 傳遞給 [`layout`](/docs/app/api-reference/file-conventions/layout)、[`page`](/docs/app/api-reference/file-conventions/page)、[`route`](/docs/app/api-reference/file-conventions/route) 和 [`generateMetadata`](/docs/app/api-reference/functions/generate-metadata#generatemetadata-function) 函式。

| Route                     | Example URL | params        |
| ------------------------- | ----------- | ------------- |
| `app/blog/[slug]/page.js` | `/blog/a`   | `{ slug: 'a' }` |
| `app/blog/[slug]/page.js` | `/blog/b`   | `{ slug: 'b' }` |
| `app/blog/[slug]/page.js` | `/blog/c`   | `{ slug: 'c' }` |

### In Client Components

在 用戶端元件 (Client Component) **頁面** 中，可以透過 [`use`](https://react.dev/reference/react/use) API 從 props 存取動態路由段 (dynamic segments)。

```tsx filename="app/blog/[slug]/page.tsx" switcher
'use client'
import { use } from 'react'

export default function BlogPostPage({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = use(params)

  return (
    <div>
      <p>{slug}</p>
    </div>
  )
}
```

```jsx filename="app/blog/[slug]/page.js" switcher
'use client'
import { use } from 'react'

export default function BlogPostPage({ params }) {
  const { slug } = use(params)

  return (
    <div>
      <p>{slug}</p>
    </div>
  )
}
```

或者，客戶端元件也可使用 [`useParams`](/docs/app/api-reference/functions/use-params) hook 來在整個客戶端元件樹狀結構中存取 `params`。

### Catch-all Segments

可以透過在方括號內加入省略號 `[...folderName]` 將動態區段擴展至 **全捕捉** 後續區段。

例如，`app/shop/[...slug]/page.js` 將匹配 `/shop/clothes`，但也匹配 `/shop/clothes/tops`、`/shop/clothes/tops/t-shirts` 等等。

| Route                        | Example URL   | params                    |
| ---------------------------- | ------------- | ------------------------- |
| `app/shop/[...slug]/page.js` | `/shop/a`     | `{ slug: ['a'] }`         |
| `app/shop/[...slug]/page.js` | `/shop/a/b`   | `{ slug: ['a', 'b'] }`    |
| `app/shop/[...slug]/page.js` | `/shop/a/b/c` | `{ slug: ['a', 'b', 'c'] }` |

### Optional Catch-all Segments

可透過將參數包含在雙重方括號中，將全捕捉區段設定為 **可選**：`[[...folderName]]`。

例如，`app/shop/[[...slug]]/page.js` 除了匹配 `/shop/clothes`、`/shop/clothes/tops`、`/shop/clothes/tops/t-shirts` 之外，也將匹配 `/shop`。

**全捕捉**與**可選全捕捉**區段的區別在於，可選版本也會匹配不帶參數的路由（如上例中的 `/shop`）。

| Route                          | Example URL   | params                    |
| ------------------------------ | ------------- | ------------------------- |
| `app/shop/[[...slug]]/page.js` | `/shop`       | `{ slug: undefined }`     |
| `app/shop/[[...slug]]/page.js` | `/shop/a`     | `{ slug: ['a'] }`         |
| `app/shop/[[...slug]]/page.js` | `/shop/a/b`   | `{ slug: ['a', 'b'] }`    |
| `app/shop/[[...slug]]/page.js` | `/shop/a/b/c` | `{ slug: ['a', 'b', 'c'] }` |

### TypeScript

使用 TypeScript 時，您可以根據配置的路由區段為 `params` 新增類型 — 使用 [`PageProps<'/route'>`](/docs/app/api-reference/file-conventions/page#page-props-helper)、[`LayoutProps<'/route'>`](/docs/app/api-reference/file-conventions/layout#layout-props-helper) 或 [`RouteContext<'/route'>`](/docs/app/api-reference/file-conventions/route#route-context-helper) 分別為 `page`、`layout` 和 `route` 中的 `params` 輸入類型。

路由 `params` 值定義為 `string`、`string[]` 或 `undefined`（用於可選全捕捉區段），因為這些值直到執行時才知道。使用者可以在網址列輸入任何 URL，這些寬鬆的類型有助於確保您的應用程式程式碼能處理所有這些可能的案例。

| Route                               | params Type Definition |
| ----------------------------------- | ---------------------- |
| `app/blog/[slug]/page.js`           | `{ slug: string }`     |
| `app/shop/[...slug]/page.js`        | `{ slug: string[] }`   |
| `app/shop/[[...slug]]/page.js`      | `{ slug?: string[] }`  |
| `app/[categoryId]/[itemId]/page.js` | `{ categoryId: string, itemId: string }` |

如果您正在處理一個 `params` 只能具有有限有效值的路由，例如具有已知語言代碼集之 `[locale]` 參數，您可以使用執行時驗證來處理使用者可能輸入的任何無效參數，並讓應用程式其餘部分使用您已知集合中的較窄類型。

```tsx filename="/app/[locale]/page.tsx"
import { notFound } from 'next/navigation'
import type { Locale } from '@i18n/types'
import { isValidLocale } from '@i18n/utils'

function assertValidLocale(value: string): asserts value is Locale {
  if (!isValidLocale(value)) notFound()
}

export default async function Page(props: PageProps<'/[locale]'>) {
  const { locale } = await props.params // locale is typed as string
  assertValidLocale(locale)
  // locale is now typed as Locale
}
```

## 行為

- 由於 `params` prop 是 Promise。您必須使用 `async`/`await` 或 React 的 use function 來存取值。
  - 在版本 14 及更早版本中，`params` 是同步 prop。為了協助向後相容性，您可以在 Next.js 15 中繼續同步存取它，但此行為未來將被棄用。

### 與快取元件 (Cache Components)

當配合 [緩存元件 (Cache Components)](/docs/app/getting-started/caching) 搭配動態路由段 (dynamic route segments) 時，處理 params 的方式取決於您是否使用 [`generateStaticParams`](/docs/app/api-reference/functions/generate-static-params)。

若不使用 `generateStaticParams`，在 預渲染 期間參數值未知，使得 params 成為運行時數據 (runtime data)。您必須將參數存取包裝在 `<Suspense>` 邊界內以提供回退 UI (fallback UI)。

使用 `generateStaticParams` 時，您可提供範例參數值供構建時 (build time) 使用。構建過程會驗證動態內容和其他運行時 API 是否已正確處理，然後為範例生成靜態 HTML 檔案。透過運行時 params 渲染的頁面，在首次請求成功後會儲存至磁盤 (disk)。

以下章節將展示這兩種模式。

#### 不使用 `generateStaticParams`

所有 params 均為運行時數據。參數存取必須由 Suspense 回退 UI 包裝。Next.js 會在構建時生成靜態外殼，內容則在每個請求時載入。

> **小提示**：您也可以使用 [`loading.tsx`](/docs/app/api-reference/file-conventions/loading) 作為頁面級別的回退 UI。

```tsx filename="app/blog/[slug]/page.tsx"
import { Suspense } from 'react'

export default function Page({ params }: PageProps<'/blog/[slug]'>) {
  return (
    <div>
      <h1>Blog Post</h1>
      <Suspense fallback={<div>Loading...</div>}>
        {params.then(({ slug }) => (
          <Content slug={slug} />
        ))}
      </Suspense>
    </div>
  )
}

async function Content({ slug }: { slug: string }) {
  const res = await fetch(`https://api.vercel.app/blog/${slug}`)
  const post = await res.json()

  return (
    <article>
      <h2>{post.title}</h2>
      <p>{post.content}</p>
    </article>
  )
}
```

#### 使用 `generateStaticParams`

提前提供 params 以在構建時預渲染頁面。您可以根據需求預渲染所有路由或其中一部分。

在構建過程中，會使用每個範例參數執行路由以收集 HTML 結果。如果動態內容或運行時數據存取方式錯誤，構建將失敗。

```tsx filename="app/blog/[slug]/page.tsx" highlight={3-5,8,19}
import { Suspense } from 'react'

export async function generateStaticParams() {
  return [{ slug: '1' }, { slug: '2' }, { slug: '3' }]
}

export default async function Page({ params }: PageProps<'/blog/[slug]'>) {
  const { slug } = await params

  return (
    <div>
      <h1>Blog Post</h1>
      <Content slug={slug} />
    </div>
  )
}

async function Content({ slug }: { slug: string }) {
  const post = await getPost(slug)
  return (
    <article>
      <h2>{post.title}</h2>
      <p>{post.content}</p>
    </article>
  )
}

async function getPost(slug: string) {
  'use cache'
  const res = await fetch(`https://api.vercel.app/blog/${slug}`)
  return res.json()
}
```

構建時驗證僅涵蓋配合範例參數執行的程式碼路徑。如果您的路由包含條件邏輯，針對範例中未包含的某些參數值存取運行時 API，則這些分支不會在構建時驗證：

```tsx filename="app/blog/[slug]/page.tsx"
import { cookies } from 'next/headers'

export async function generateStaticParams() {
  return [{ slug: 'public-post' }, { slug: 'hello-world' }]
}

export default async function Page({ params }: PageProps<'/blog/[slug]'>) {
  const { slug } = await params

  if (slug.startsWith('private-')) {
    // This branch is never executed at build time
    // Runtime requests for 'private-*' slugs will error
    return <PrivatePost slug={slug} />
  }

  return <PublicPost slug={slug} />
}

async function PrivatePost({ slug }: { slug: string }) {
  const token = (await cookies()).get('token')
  // ... fetch and render private post using token for auth
}
```

對於 `generateStaticParams` 未返回的運行時 params，驗證會在首次請求時發生。在上面的範例中，針對以 `private-` 開頭的 slugs 的請求將失敗，因為 `PrivatePost` 存取 `cookies()` 時未使用 Suspense 邊界。其他未觸發條件分支的運行時 params 將成功渲染，並儲存至磁盤供後續請求使用。

為了解決這個問題，請將 `PrivatePost` 包裹在 Suspense 中：

```tsx filename="app/blog/[slug]/page.tsx" highlight={13-15}
import { Suspense } from 'react'
import { cookies } from 'next/headers'

export async function generateStaticParams() {
  return [{ slug: 'public-post' }, { slug: 'hello-world' }]
}

export default async function Page({ params }: PageProps<'/blog/[slug]'>) {
  const { slug } = await params

  if (slug.startsWith('private-')) {
    return (
      <Suspense fallback={<div>Loading...</div>}>
        <PrivatePost slug={slug} />
      </Suspense>
    )
  }

  return <PublicPost slug={slug} />
}

async function PrivatePost({ slug }: { slug: string }) {
  const token = (await cookies()).get('token')
  // ... fetch and render private post using token for auth
}
```

## 範例

### With `generateStaticParams`

`generateStaticParams` 函數可用於在構建時 [靜態生成](/docs/app/glossary#prerendering) 路由，而不是在請求時按需生成。

```tsx filename="app/blog/[slug]/page.tsx" switcher
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))
}
```

```jsx filename="app/blog/[slug]/page.js" switcher
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))
}
```

當在 `generateStaticParams` 函數中使用 `fetch` 時，請求會 [自動去重](/docs/app/glossary#memoization)。這可避免佈局、頁面和其他 `generateStaticParams` 函數為相同數據進行多次網絡調用，從而加快構建速度。

### 配合 `generateStaticParams` 的動態 GET 路由處理函數

`generateStaticParams` 也適用於動態 [路由處理函數](/docs/app/api-reference/file-conventions/route)，以便在構建時靜態生成 API 回應：

```ts filename="app/api/posts/[id]/route.ts" switcher
export async function generateStaticParams() {
  const posts: { id: number }[] = await fetch(
    'https://api.vercel.app/blog'
  ).then((res) => res.json())

  return posts.map((post) => ({
    id: `${post.id}`,
  }))
}

export async function GET(
  request: Request,
  { params }: RouteContext<'/api/posts/[id]'>
) {
  const { id } = await params
  const res = await fetch(`https://api.vercel.app/blog/${id}`)

  if (!res.ok) {
    return Response.json({ error: 'Post not found' }, { status: 404 })
  }

  const post = await res.json()
  return Response.json(post)
}
```

```js filename="app/api/posts/[id]/route.js" switcher
export async function generateStaticParams() {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )

  return posts.map((post) => ({
    id: `${post.id}`,
  }))
}

export async function GET(request, { params }) {
  const { id } = await params
  const res = await fetch(`https://api.vercel.app/blog/${id}`)

  if (!res.ok) {
    return Response.json({ error: 'Post not found' }, { status: 404 })
  }

  const post = await res.json()
  return Response.json(post)
}
```

在此範例中，`generateStaticParams` 返回的所有部落格貼文 ID 的路由處理函數，都將在構建時靜態生成。針對其他 ID 的請求將在請求時動態處理。
