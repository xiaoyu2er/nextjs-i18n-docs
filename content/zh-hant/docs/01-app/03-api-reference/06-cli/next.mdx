Next.js CLI 允許您開發、構建、啟動您的應用程序等操作。

基本用法：

```bash package="pnpm"
pnpm next [command] [options]
```

```bash package="npm"
npx next [command] [options]
```

```bash package="yarn"
yarn next [command] [options]
```

```bash package="bun"
bunx next [command] [options]
```

> **注意**：使用 `npm run` 時，在 CLI 參數前加上 `--`，以便 npm 將它們轉發給 `next`。`pnpm`、`yarn` 或 `bun` 則不需要這樣做。

## 參考

下列選項可用：

| Options             | Description                        |
| ------------------- | ---------------------------------- |
| `-h` or `--help`    | 顯示所有可用選項        |
| `-v` or `--version` | 輸出 Next.js 版本號 |

### 命令

下列命令可用：

| Command                                                      | Description                                                                                                   |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------- |
| [`dev`](#next-dev-options)                                   | 以開發模式啟動 Next.js，包含熱模塊重新載入 (HMR)、錯誤報告等。                      |
| [`build`](#next-build-options)                               | 建立應用程式的優化生產構建。顯示每個路由的資訊。           |
| [`start`](#next-start-options)                               | 以生產模式啟動 Next.js。應用程式應先使用 `next build` 進行編譯。                |
| [`info`](#next-info-options)                                 | 列印與當前系統相關的詳細資訊，可用於報告 Next.js 錯誤。                    |
| [`telemetry`](#next-telemetry-options)                       | 允許啟用或停用 Next.js 完全匿名的遙測收集。                           |
| [`typegen`](#next-typegen-options)                           | 在不運行完整構建的狀況下為路由、頁面、佈局和路由處理函數生成 TypeScript 定義。 |
| [`upgrade`](#next-upgrade-options)                           | 將您的 Next.js 應用程式升級至最新版本。                                                      |
| [`experimental-analyze`](#next-experimental-analyze-options) | 使用 Turbopack 分析捆綁輸出。不產生構建 artifacts。                                     |

> **小知識**: 運行 `next` 而沒有命令是 `next dev` 的別名。

### `next dev` 選項

`next dev` 以開發模式啟動應用程式，包含熱模塊重新載入 (HMR)、錯誤報告等功能。

> **注意**：開發構建的輸出位於 `.next/dev` 而非 `.next`。這允許您同時運行 `next dev` 和 `next build` 而不會產生衝突。

運行 `next dev` 時可使用以下選項：

| Option                                   | Description                                                                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-h, --help`                             | 顯示所有可用選項。                                                                                                                          |
| `[directory]`                            | 構建應用程式的目錄。若未提供，則使用當前目錄。                                                           |
| `--turbopack`                            | 強制啟用 [Turbopack](/docs/app/api-reference/turbopack) (預設已啟用)。也可使用 `--turbo`。                                       |
| `--webpack`                              | 使用 Webpack 而非預設的 [Turbopack](/docs/app/api-reference/turbopack) 打包器進行開發。                                           |
| `-p` or `--port <port>`                  | 指定啟動應用程式的埠號。預設值：3000，環境變數：PORT                                                                    |
| `-H` or `--hostname <hostname>`           | 指定啟動應用程式的主機名。適用於使應用程式可在網路上的其他裝置上使用。預設值：0.0.0.0 |
| `--experimental-https`                   | 以 HTTPS 啟動伺服器並生成自簽名憑證。                                                                                |
| `--experimental-https-key <path>`        | HTTPS 密鑰檔案的路徑。                                                                                                                            |
| `--experimental-https-cert <path>`       | HTTPS 憑證檔案的路徑。                                                                                                                    |
| `--experimental-https-ca <path>`         | HTTPS 憑證授權單位檔案的路徑。                                                                                                          |
| `--experimental-upload-trace <traceUrl>` | 將除錯追蹤的子集報告至遠端 HTTP URL。                                                                                        |
| `--experimental-cpu-prof`                | 使用 V8 的檢查器啟用 CPU 效能分析。退出時將效能分析檔案儲存至 `.next/cpu-profiles/`。                                                     |

### `next build` 選項

`next build` 創建應用程序的優化生產構建。輸出顯示有關每個路由的信息。例如：

```bash filename="Terminal"
Route (app)
┌ ○ /_not-found
└ ƒ /products/[id]

○  (Static)   prerendered as static content
ƒ  (Dynamic)  server-rendered on demand
```

下列選項可用於 `next build` 命令：

| Option                             | Description                                                                                                                                                                        |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-h, --help`                       | 顯示所有可用選項。                                                                                                                                                        |
| `[directory]`                      | 構建應用程式的目錄。若未提供，則使用當前目錄。                                                                                |
| `--turbopack`                      | 強制啟用 [Turbopack](/docs/app/api-reference/turbopack) (預設已啟用)。也可使用 `--turbo`。                                                                     |
| `--webpack`                        | 使用 Webpack 進行構建。                                                                                                                                                               |
| `-d` or `--debug`                  | 啟用更詳細的構建輸出。啟用此旗標時，將顯示額外的構建輸出，如重寫規則、重定向標頭等。                                           |
| |                                    |
| `--profile`                        | 啟用生產環境的 [React 效能分析](https://react.dev/reference/react/Profiler)。                                                                                              |
| `--no-lint`                        | 停用除錯檢查。 _注意：除錯檢查將於 Next 16 從 `next build` 移除。如果您使用 Next 15.5+ 且除错器非 `eslint`，構建期間將不會進行除錯檢查。_ |
| `--no-mangling`                    | 停用名稱混淆。這可能影響效能，應僅用於除錯目的。                                                                                                                                 |
| `--experimental-app-only`          | 僅建構 App Router 路由。                                                                                                                                                     |
| `--experimental-build-mode [mode]` | 使用實驗性構建模式。 (選項："compile"、"generate"，預設："default")                                                                                              |
| `--debug-prerender`                | 在開發期間除錯預渲染錯誤。                                                                                                                                             |
| `--debug-build-paths=<patterns>` | 僅為除錯目的構建特定路由。                                                                                                                                          |
| `--experimental-cpu-prof`          | 使用 V8 的檢查器啟用 CPU 效能分析。退出時將效能分析檔案儲存至 `.next/cpu-profiles/`                                                                                   |

### `next start` 選項

`next start` 以生產模式啟動應用程序。應用程序應先使用 [`next build`](#next-build-options) 進行編譯。

下列選項可用於 `next start` 命令：

| Option                                  | Description                                                                                                     |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `-h` or `--help`                        | 顯示所有可用選項。                                                                                     |
| `[directory]`                           | 啟動應用程式的目錄。若未提供目錄，則使用當前目錄。 |
| `-p` or `--port <port>`                 | 指定啟動應用程式的埠號。 (預設值：3000，環境變數：PORT)                             |
| `-H` or `--hostname <hostname>`         | 指定啟動應用程式的主機名 (預設值：0.0.0.0)。                                        |
| `--keepAliveTimeout <keepAliveTimeout>` | 指定在關閉非作用連線之前等待的最大毫秒數。                     |
| `--experimental-cpu-prof`               | 使用 V8 的檢查器啟用 CPU 效能分析。退出時將效能分析檔案儲存至 `.next/cpu-profiles/`。                |

### `next telemetry` 選項

Next.js 會收集有關一般使用情况的**完全匿名**遥測數據。參與此匿名計劃是可選的，如果您不希望共享信息，可以選擇退出。

下列選項可用於 `next telemetry` 命令：

| Option       | Description                             |
| ------------ | --------------------------------------- |
| `-h, --help` | 顯示所有可用選項。             |
| `--enable`   | 啟用 Next.js 的遥測收集。  |
| `--disable`  | 禁用 Next.js 的遥測收集。 |

了解有關 [Telemetry](/telemetry) 的更多資訊。

### `next typegen` 選項

`next typegen` 無需執行完整構建即可為您的應用程序路由生成 TypeScript 定義。這對於 IDE 自動完成和 CI 路由使用類型檢查很有用。

此前，路由類型僅在 `next dev` 或 `next build` 期間生成，這意味著直接運行 `tsc --noEmit` 不會驗證您的路由類型。現在您可以獨立生成類型並在外部驗證它們：

```bash filename="Terminal"
# Generate route types first, then validate with TypeScript
next typegen && tsc --noEmit

# Or in CI workflows for type checking without building
next typegen && npm run type-check
```

下列選項可用於 `next typegen` 命令：

| Option        | Description                                                                                  |
| ------------- | -------------------------------------------------------------------------------------------- |
| `-h, --help`  | 顯示所有可用選項。                                                                  |
| `[directory]` | 生成類型時使用的目錄。如果未提供，則使用當前目錄。 |

輸出檔案會寫入 `<distDir>/types` (通常：開發環境為 `.next/dev/types` 或生產環境為 `.next/types`)：

```bash filename="Terminal"
next typegen
# or for a specific app
next typegen ./apps/web
```

此外，`next typegen` 會生成 `next-env.d.ts` 檔案。我們建議將 `next-env.d.ts` 加入您的 `.gitignore` 檔案。

`next-env.d.ts` 檔案會包含在您的 `tsconfig.json` 檔案中，以便您的專案可以使用 Next.js 類型。

為確保在類型檢查前存在 `next-env.d.ts`，請運行 `next typegen`。`next dev` 和 `next build` 指令也會生成 `next-env.d.ts` 檔案，但為了類型檢查而運行這些命令通常並不理想，例如在 CI/CD 環境中。

> **小知識**: `next typegen` 使用生產構建階段加載您的 Next.js 配置 (`next.config.js`, `next.config.mjs`, or `next.config.ts`)。確保任何必需的環境變量和依賴項可用，以便配置可以正確加載。

### `next upgrade` 選項

`next upgrade` 將您的 Next.js 應用程式升級至最新版本。

以下選項適用於 `next upgrade` 指令：

| Option                  | Description                                                                                                                                        |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-h, --help`            | 顯示所有可用選項。                                                                                                                        |
| `[directory]`           | 包含要升級的 Next.js 應用程式的目錄。若未提供，則使用當前目錄。                                          |
| `--revision <revision>` | 指定要升級到的 Next.js 版本或標籤 (例如，`latest`、`canary`、`15.0.0`)。預設值為您目前安裝的發行通道版本。 |
| `--verbose`             | 顯示升級過程中的詳細輸出。                                                                                                    |

### `next experimental-analyze` 選項

`next experimental-analyze` 使用 [Turbopack](/docs/app/api-reference/turbopack) 分析您應用程式的捆綁輸出。此指令可協助您了解捆綁的大小和組合成分，包括 JavaScript、CSS 和其他資源。此指令不會產生應用程式構建。

```bash package="pnpm"
pnpm next experimental-analyze
```

```bash package="npm"
npx next experimental-analyze
```

```bash package="yarn"
yarn next experimental-analyze
```

```bash package="bun"
bunx next experimental-analyze
```

預設情況下，指令會在分析完成後啟動本地伺服器，允許您在瀏覽器中探索您的捆綁組合成分。分析器讓您能夠：

- 按路由篩選捆綁，並在客戶端和伺服器視圖之間切換
- 檢視顯示為何包含模組的全套匯編串接
- 追蹤伺服器至客戶端元件邊界及動態匯入之間的匯入

關於最佳化策略，請參閱 [套件捆綁](/docs/app/guides/package-bundling#optimizing-large-bundles)。

若要將分析輸出寫入磁碟而無需啟動伺服器，請使用 `--output` 旗標。輸出會寫入 `.next/diagnostics/analyze`，其中包含可複製到其他位置或與他人分享的靜態檔案：

```bash filename="Terminal"
# Write output to .next/diagnostics/analyze
npx next experimental-analyze --output

# Copy the output for comparison with a future analysis
cp -r .next/diagnostics/analyze ./analyze-before-refactor
```

以下選項適用於 `next experimental-analyze` 指令：

| Option          | Description                                                                                                                                   |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `-h, --help`    | 顯示所有可用選項。                                                                                                                   |
| `[directory]`   | 分析應用程式的目錄。若未提供，則使用當前目錄。                                         |
| `--no-mangling` | 停用名稱混淆。這可能影響效能，應僅用於除錯目的。 |
| `--profile`     | 啟用生產環境的 [React 效能分析](https://react.dev/reference/react/Profiler)。這可能影響效能。                            |
| `-o, --output`  | 將分析檔案寫入磁碟而無需啟動伺服器。輸出會寫入 `.next/diagnostics/analyze`.                                   |
| `--port <port>` | 指定用於提供分析器的埠號。 (預設值：4000，環境變數：PORT)                                                                    |

## 範例

### 調試預渲染錯誤

如果您在 `next build` 期間遇到預渲染錯誤，可以通過傳遞 `--debug-prerender` 標誌來獲取更詳細的輸出：

```bash filename="Terminal"
next build --debug-prerender
```

這啟用了幾個實驗性選項以方便調試：

- 停用伺服器程式碼壓縮：
  - `experimental.serverMinification = false`
  - `experimental.turbopackMinify = false`
- 為伺服器捆綁生成原始碼地圖：
  - `experimental.serverSourceMaps = true`
- 啟用預渲染中使用之子進程中的原始碼地圖消耗：
  - `enablePrerenderSourceMaps = true`
- 即使在第一個預渲染錯誤後仍繼續構建，以便您可一次看到所有問題：
  - `experimental.prerenderEarlyExit = false`

這有助於在構建輸出中顯示更易讀的堆棧追蹤和代碼框架。

> **警告**: `--debug-prerender` 僅用於開發環境調試。切勿將使用 `--debug-prerender` 生成的構建部署到生產環境，因為它可能會影響性能。

### 建置特定路由

您可以使用 `--debug-build-paths` 選項僅在 App 和 Pages Routers 中構建特定路由。這對於在處理大型應用程式時進行快速除錯很有用。`--debug-build-paths` 選項接受以逗號分隔的檔案路徑，並支援 glob 模式：

```bash filename="Terminal"
# Build a specific route
next build --debug-build-paths="app/page.tsx"

# Build more than one route
next build --debug-build-paths="app/page.tsx,pages/index.tsx"

# Use glob patterns
next build --debug-build-paths="app/**/page.tsx"
next build --debug-build-paths="pages/*.tsx"
```

### 更改默認端口

默認情況下，Next.js 在開發期間和與 `next start` 一起使用 `http://localhost:3000`。默認端口可以使用 `-p` 選項更改，如下所示：

```bash filename="Terminal"
next dev -p 4000
```

或者使用 `PORT` 環境變量：

```bash filename="Terminal"
PORT=4000 next dev
```

> **小知識**: `PORT` 不能設置在 `.env` 中，因為 HTTP 服務器啟動發生在任何其他代碼初始化之前。

### 開發期間使用 HTTPS

對於某些使用情況，如 webhooks 或身份驗證，您可以使用 [HTTPS](https://developer.mozilla.org/en-US/docs/Glossary/HTTPS) 在 `localhost` 上擁有安全環境。Next.js 可以使用 `--experimental-https` 標誌通過 `next dev` 生成自簽名證書：

```bash filename="Terminal"
next dev --experimental-https
```

使用生成的證書，Next.js 開發服務器將存在於 `https://localhost:3000`。默認端口 `3000` 用於指定 `-p`, `--port` 或 `PORT` 之外。

您也可以使用 `--experimental-https-key` 和 `--experimental-https-cert` 提供自定義證書和密鑰。此外，您也可以使用 `--experimental-https-ca` 提供自定義 CA 證書。

```bash filename="Terminal"
next dev --experimental-https --experimental-https-key ./certificates/localhost-key.pem --experimental-https-cert ./certificates/localhost.pem
```

`next dev --experimental-https` 僅打算用於開發，並使用 [`mkcert`](https://github.com/FiloSottile/mkcert) 創建本地可信證書。在生產環境中，請使用可信機構發放的適當證書。

### 為下游代理配置超時時間

當在下游代理後部署 Next.js 時（例如 AWS ELB/ALB 等負載平衡器），建議將 Next 的底層 HTTP 服務器配置為 [keep-alive timeouts](https://nodejs.org/api/http.html#http_server_keepalivetimeout)，這些超時時間應**大於**下游代理的超時時間。否則，一旦達到給定 TCP 連接的 keep-alive 超時時間，Node.js 將立即終止該連接而不會通知下游代理。這會導致代理在嘗試重用 Node.js 已經終止的連接時出現代理錯誤。

要為生產 Next.js 服務器配置超時值，將 `--keepAliveTimeout`（以毫秒為單位）傳遞給 `next start`，如下所示：

```bash filename="Terminal"
next start --keepAliveTimeout 70000
```

### 傳遞 Node.js 參數

您可以將任何 [node 參數](https://nodejs.org/api/cli.html#cli_node_options_options) 傳遞給 `next` 命令。例如：

```bash filename="Terminal"
NODE_OPTIONS='--throw-deprecation' next
NODE_OPTIONS='-r esm' next
NODE_OPTIONS='--inspect' next
```

### CPU 效能分析

您可以捕捉 CPU 效能分析檔案以分析 Next.js 應用程式中的效能瓶頸。`--experimental-cpu-prof` 旗標會啟用 V8 內建的 CPU 分析器，並在程式退出時將效能分析檔案保存至 `.next/cpu-profiles/`：

```bash filename="Terminal"
# Profile the build process
next build --experimental-cpu-prof

# Profile the dev server (profile saved on Ctrl+C or SIGTERM)
next dev --experimental-cpu-prof

# Profile the production server
next start --experimental-cpu-prof
```

生成的 `.cpuprofile` 檔案可以在 Chrome DevTools (效能標籤頁 → 載入分析檔案) 或其他 V8 相容的效能分析工具中開啟。

> **注意**：分析檔案以描述性前綴和時間戳記命名。生成的分析檔案取決於指令：
>
> **`next dev`**：
>
> - `dev-main-*` - 父程序 (開發伺服器編排)
> - `dev-server-*` - 子伺服器程序 (請求處理和渲染) - 這通常是您想要分析的內容
>
> **`next build` (Turbopack)**：
>
> - `build-main-*` - 主構建編排程序
> - `build-turbopack-*` - Turbopack 編譯工作程序
>
