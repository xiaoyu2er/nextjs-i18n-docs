歡迎來到 Next.js 文檔貢獻指南！我們很高興你能來到這裡。

本頁提供了如何編輯 Next.js 文檔的指南。我們的目標是確保社群中的每個人都能感到有能力貢獻並改善我們的文件。

## 為什麼要貢獻？

開源工作永遠沒有做完，文檔工作亦然。貢獻文檔是初學者參與開源的好方法，也是資深開發者澄清更複雜主題的途徑，同時將知識分享給社區。

通過貢獻 Next.js 文檔，您正在幫助我們為所有開發者構建更完善的學習資源。無論您發現了錯別字、令人困惑的章節，或是意識到某個主題缺失，您的貢獻都受到歡迎和感謝。

## 如何貢獻

文檔內容可在 [Next.js 倉庫](https://github.com/vercel/next.js/tree/canary/docs) 找到。要貢獻，您可以直接在 GitHub 上編輯文件，或複製倉庫並在本地編輯文件。

### GitHub 工作流程

如果您是新 GitHub 用戶，我們建議閱讀 [GitHub 開源指南](https://opensource.guide/how-to-contribute/#opening-a-pull-request) 以了解如何分叉倉庫、創建分支和提交拉取請求。

> **要知道**：底層的文檔代碼存在於一個與 Next.js 公共倉庫同步的私有代碼庫中。這意味著您無法在本地預覽文檔。但是，在合併拉取請求後，您將在 [nextjs.org](https://nextjs.org/docs) 上看到您的更改。

### 撰寫 MDX

文檔是使用 [MDX](https://mdxjs.com/) 撰寫的，這是一個支持 JSX 語法的 Markdown 格式。這使我們能夠在文檔中嵌入 React 組件。請參閱 [GitHub Markdown 指南](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax) 以快速了解 Markdown 語法。

### VSCode

#### 在本地預覽更改

VSCode 內建 Markdown 預覽器，您可以在本地查看編輯內容。要為 MDX 文件啟用預覽器，您需要在用戶設置中添加配置選項。

打開命令面板（`⌘ + ⇧ + P` 在 Mac 上或 `Ctrl + Shift + P` 在 Windows 上）並搜索 `Preferences: Open User Settings (JSON)`。

然後，將以下行添加到您的 `settings.json` 文件中：

```json filename="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

接下來，再次打開命令面板，並搜索 `Markdown: Preview File` 或 `Markdown: Open Preview to the Side`。這將打開一個預覽窗口，您可以在其中查看格式化的更改。

#### 擴充功能

我們也為 VSCode 用戶推薦以下擴充功能：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx): MDX 的 IntelliSense 和語法高亮。
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode): 保存時格式化 MDX 文件。

### 審核流程

一旦您提交了貢獻，Next.js 或開發者體驗團隊將審核您的更改，提供反饋，並在準備好時合併拉取請求。

如果在您的 PR 評論中有任何問題或需要進一步協助，請告訴我們。感謝您為 Next.js 文檔貢獻並成為我們社群的一員！

> **提示**：在提交 PR 之前運行 `pnpm prettier-fix` 以運行 Prettier。

## 文件結構

文檔使用 **文件系統路由**。[`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 內的每個文件和文件夾代表一個路由段。這些段用於生成 URL 路徑、導航和麵包屑。

文件結構反映了您在網站上看到的導航，預設情況下，導航項目按字母順序排序。但是，我們可以通過在文件夾或文件名前添加兩位數 (`00-`) 來更改項目的順序。

例如，在 [函數 API 參考](/docs/app/api-reference/functions) 中，頁面按字母順序排序，因為這使得開發者更容易找到特定函數：

```txt
04-functions
├── after.mdx
├── cacheLife.mdx
├── cacheTag.mdx
└── ...
```

但是，在 [應用路由 (app router) 部分](/docs/app)，檔案名稱的開頭會加上兩位數，順序是依照開發人員學習這些概念的順序排列：

```txt
01-getting-started
├── 01-installation.mdx
├── 02-project-structure.mdx
├── 03-layouts-and-pages.mdx
└── ...
```

若要快速找到頁面，您可以使用 `⌘ + P` (Mac) 或 `Ctrl + P` (Windows) 來開啟 VSCode 中的搜尋列。接著，輸入您正在尋找的頁面的 slug。例如：`installation`

> **為什麼不使用清單 (manifest)？**
>
> 我們曾考慮使用清單文件 (manifest file)（另一種生成文檔導航的流行方式），但我們發現清單會很快與文件不同步。文件系統路由迫使我們思考文件的結構，感覺更貼近 Next.js。

## 元數據

每個頁面在文件頂部都有一個元數據塊，由三個連字號分隔。

### 必要欄位

以下欄位為 **必要**：

| Field         | Description                                                                |
| ------------- | ---------------------------------------------------------------------------- |
| `title`       | 頁面的 `<h1>` 標題，用於 SEO 和 OG Images。                         |
| `description` | 頁面的描述，用於 `<meta name="description">` 標籤以進行 SEO。                |

```yaml filename="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

限制頁面標題為 2-3 個字（例如：Optimizing Images），描述為 1-2 個句子（例如：Learn how to optimize images in Next.js）是一種良好的做法。

### 可選欄位

以下欄位為 **可選**：

| Field       | Description                                                                                                                                        |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | 覆蓋導航中的頁面標題。當頁面標題太長無法適應時很有用。如果未提供，則使用 `title` 欄位。 |
| `source`    | 將內容拉入共享頁面。查看 [Shared Pages](#shared-pages)。                                                                               |
| `related`   | 文件底部的相關頁面列表。這些將自動轉換為卡片。查看 [Related Links](#related-links)。         |
| `version`   | 開發階段。例如 `experimental`, `legacy`, `unstable`, `RC`                                                                               |

```yaml filename="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
version: experimental
---
```

## `App` 和 `Pages` 文檔

由於 **App Router** 和 **Pages Router** 中的大多數功能完全不同，它們各自的文檔保留在單獨的部分 (`02-app` 和 `03-pages`) 中。但是，也有一些功能是它們共用的。

### 共享頁面

為了避免內容複製和導致內容不同步的風險，我們使用 `source` 欄位從一個頁面將內容拉入另一個頁面。例如，`<Link>` 組件在 **App** 和 **Pages** 中行為 _大部分_ 相同。與其複製內容，我們可以直接從 `app/.../link.mdx` 拉取內容到 `pages/.../link.mdx`：

```mdx filename="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx filename="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* DO NOT EDIT THIS PAGE. */}
{/* The content of this page is pulled from the source above. */}
```

因此，我們可以在一個地方編輯內容，並在兩部分中反映出來。

### 共享內容

在共享頁面中，有時可能存在僅針對 **App Router** 或 **Pages Router** 的內容。例如，`<Link>` 組件擁有一個 `shallow` 屬性，該屬性僅在 **Pages** 中可用，而在 **App** 中不可用。

為了確保內容僅顯示在正確的路由器中，我們可以在 `<AppOnly>` 或 `<PagesOnly>` 組件包裝內容塊：

```mdx filename="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

This content will only be shown on the Pages docs.

</PagesOnly>

This content is shared between App and Pages.
```

您可能會將這些組件用於示例和程式碼塊。

## 程式碼塊

程式碼塊應包含一個可複製和貼上的最小可運行示例。這意味著代碼應該能夠無需任何額外配置即可運行。

例如，如果您正在展示如何使用 `<Link>` 組件，您應該包含 `import` 語句和 `<Link>` 組件本身。

```tsx filename="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

提交之前請務必先在本地運行示例。這將確保代碼是更新的且可以工作的。

### 語言和文件名稱

程式碼區塊應該包含一個標頭，標頭中需說明語言和 `filename`。例如：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx"
export default function Page() {
  return <h1>Hello, Next.js!</h1>
}
```
````

對於 CLI 指令，請使用 `package` prop 來顯示每個套件管理器的指令：

````mdx filename="code-example.mdx"
```bash package="pnpm"
pnpm create next-app
```

```bash package="npm"
npx create-next-app@latest
```

```bash package="yarn"
yarn create next-app
```

```bash package="bun"
bun create next-app
```
````

文檔中的大多數示例是用 `tsx` 和 `jsx` 撰寫的，少數是 `bash`。但是，您可以使用任何支持的語言，這是 [完整列表](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)。

撰寫 JavaScript 程式碼塊時，我們使用以下語言和擴展組合。

|                                | Language | Extension |

| JavaScript files with JSX code | ```jsx   | .js       |
| JavaScript files without JSX   | ```js    | .js       |
| TypeScript files with JSX      | ```tsx   | .tsx      |
| TypeScript files without JSX   | ```ts    | .ts       |

### TS 和 JS 切換器

添加一個語言切換器可在 TypeScript 和 JavaScript 之間切換。程式碼塊應以 TypeScript 優先，並附有 JavaScript 版本以适应用戶。

目前，我們連續編寫 TS 和 JS 示例，並使用 `switcher` 屬性將它們鏈接起來：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx" switcher

```

```jsx filename="app/page.js" switcher

```
````

> **要知道**：我們計劃未來自動將 TypeScript 代碼片段編譯為 JavaScript。與此同時，您可以使用 [transform.tools](https://transform.tools/typescript-to-javascript)。

### 行高亮

程式碼行可以高亮顯示。當您想吸引人們關注代碼的特定部分時這很有用。您可以通過將數字傳遞給 `highlight` 屬性來高亮行。

**單行：** `highlight={1}`

```tsx filename="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**多行：** `highlight={1,3}`

```tsx filename="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**行範圍：** `highlight={1-5}`

```tsx filename="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## 註記

對於重要但不緊急的資訊，使用註記。註記是添加資訊而不分散使用者對主要內容注意力的好方法。

```mdx filename="notes.mdx"
> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.
```

**輸出：**

> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.

## 相關連結

相關連結通過添加邏輯後續步驟的連結來引導使用者的學習旅程。

- 連結將顯示在頁面主要內容下方的卡片中。
- 具有子頁面的頁面將自動生成連結。

使用頁面元數據中的 `related` 欄位創建相關連結。

```yaml filename="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### 嵌套欄位

| Field         | Required? | Description                                                                                                                                               |
| ------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | Optional  | The title of the card list. Defaults to **Next Steps**.                                                                                                   |
| `description` | Optional  | The description of the card list.                                                                                                                         |
| `links`       | Required  | A list of links to other doc pages. Each list item should be a relative URL path (without a leading slash) e.g. `app/api-reference/file-conventions/page` |

## 風格指南

本節包含針對剛接觸技術寫作的撰寫文檔指南。

### 頁面模板

雖然我們沒有嚴格的頁面模板，但有一些頁面章節會在文件重複看到：

- **概覽**：頁面的第一段應告訴使用者該功能是什麼及用於什麼。接著是最小可運行示例或其 API 參考。
- **慣例**：如果功能有慣例，應在此解釋。
- **示例**：展示功能如何在不同使用案例中使用。
- **API 表格**：API 頁面應在頁面末尾有概覽表格並包含跳轉章節連結（如果可能）。
- **下一步（相關連結）**：添加相關頁面連結以引導使用者的學習旅程。

請隨意根据需要添加這些章節。

### 頁面類型

文件頁面也分為兩類：概念性和參考性。

- **概念性** 頁面用於解釋概念或功能。它們通常較長且包含比參考頁面更多的資訊。在 Next.js 文件檔中，概念性頁面可在 **Building Your Application** 部分找到。
- **參考性** 頁面用於解釋特定 API。它們通常較短且更集中。在 Next.js 文件中，參考性頁面可在 **API Reference** 部分找到。

> **要知道**：根據您貢獻的頁面，您可能需要遵循不同的語氣和風格。例如，概念性頁面更具指導性，使用 _you_ 一詞來稱呼使用者。參考性頁面更具技術性，它們使用更多命令詞，如 "create, update, accept"，並且往往省略 _you_ 一詞。

### 語氣

以下是保持文件風格和語氣一致的一些指南：

- 撰寫清晰、簡潔的句子。避免枝節。
  - 如果您發現自己使用了很多逗號，考慮將句子拆分成多個句子或使用清單。
  - 用簡單詞替換複雜詞。例如，_use_ 而不是 _utilize_。
- 注意 _this_ 的用法。它可能含糊不清，如果 unclear，不要害怕重複句子的主詞。
  - 例如，_Next.js uses React_ 而不是 _Next.js uses this_。
- 使用主動語態而不是被動。主動句更容易閱讀。
  - 例如，_Next.js uses React_ 而不是 _React is used by Next.js_。如果您發現自己使用 _was_ 和 _by_ 等詞，您可能在使用被動語態。
- 避免使用 _easy_, _quick_, _simple_, _just_ 等詞。這主觀且可能讓使用者感到沮喪。
- 避免 _don't_, _can't_, _won't_ 等負面詞。這可能會讓讀者感到沮喪。
  - 例如，_"You can use the `Link` component to create links between pages"_ 而不是 _"Don't use the `<a>` tag to create links between pages"。
- 使用第二人稱 (you/your)。這更具個人化和吸引力。
- 使用中性語言。使用 _developers_, _users_，或 _readers_，指代受眾。
- 如果添加程式碼示例，確保它們已正確格式化且可運行。

雖然這些指南並非詳盡無遺，但它應該能幫助您開始。如果您想更深入地了解技術寫作，請查看 [Google Technical Writing Course](https://developers.google.com/tech-writing/overview)。
