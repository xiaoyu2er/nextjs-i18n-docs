Next.js 16 (إصدار تجريبي) متاح الآن. يوفر هذا الإصدار المبكر الوصول لأحدث التحسينات الخاصة بـ Turbopack التخزين، والبنية المعمارية لـ Next.js قبل الإصدار المستقر. تشمل أبرز ما في هذا الإصدار:

*   [**Turbopack (مستقر)**](#turbopack-stable): حزمة أولية لجميع التطبيقات مع تسريع Fast Refresh يصل إلى 5-10 أضعاف، وبناء أسرع بنسبة 2-5 أضعاف
*   [**تخزين نظام ملفات Turbopack (تجريبي)**](#turbopack-file-system-caching-beta): بدء تشغيل وأوقات compilation أسرع للتطبيقات الكبيرة
*   [**دعم React Compiler (مستقر)**](#react-compiler-support-stable): تكامل مدمج للاستدعاء التلقائي للذاكرة
*   [**واجهة برمجة تطبيقات مجرّدات البناء (تجريبي)**](#build-adapters-api-alpha): إنشاء مجرّدات مخصصة لتعديل عملية البناء
*   [**التوجيه المحسن**](#enhanced-routing-and-navigation): التنقلات و prefetching محسّنة مع إزالة تخطيط المكرر و Prefetch التدريجي
*   [**واجهات برمجة تطبيقات تخزين محسّنة**](#improved-caching-apis): واجهات `updateTag()` الجديدة و `revalidateTag()` المُحسّنة
*   [**React 19.2**](#react-192-and-canary-features): View Transitions، `useEffectEvent()`، `<Activity/>`
*   [**تغييرات كاسرة (Breaking Changes)**](#breaking-changes-and-other-updates): متغيرات params Async، افتراضيات `next/image`، والمزيد

**نشجعكم على تجربة الإصدار التجريبي ومشاركة ملاحظاتكم.** يساعد اختباركم في تحديد المشكلات وتحسين Next.js قبل الإصدار المستقر. يرجى الإبلاغ عن أي أخطاء أو مشكلات تواجهها على [GitHub](https://github.com/vercel/next.js/issues).

ترقية إلى الإصدار التجريبي:

Terminal

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade beta
 
# ...or upgrade manually
npm install next@beta react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@beta
```

## تجربة المطور[](#developer-experience)
---------------------------------------------

### Turbopack (مستقر)[](#turbopack-stable)

وصل Turbopack لمرحلة الاستقرار بالنسبة لبناءات التطوير والإنتاج، وهو الآن الحزمة الافتراضية لكل المشاريع الجديدة في Next.js. منذ صدوره التجريبي في الصيف الماضي، نمت عملية التبني بسرعة: أكثر من 50% من جلسات التطوير و 20% من بنود الإنتاج على Next.js 15.3+ تعمل بالفعل باستخدام Turbopack.

باستخدام Turbopack، يمكنك توقع:


*   بناء إنتاج أسرع بنسبة 2–5x
*   Fast Refresh أسرع يصل إلى 10x

نحن نجعل Turbopack هو الافتراضي لنجلب هذه المكاسب في الأداء لكل مطور Next.js، دون الحاجة لأي إعداد. للتطبيقات ذات إعدادات webpack المخصصة، يمكنك الاستمرار باستخدام webpack عن طريق تشغيل:

Terminal

```
next dev --webpack
next build --webpack
```

### تخزين نظام ملفات Turbopack (تجريبي)[](#turbopack-file-system-caching-beta)

يدعم Turbopack الآن تخزين ملفات النظام في التطوير، مع تخزين أدوات المترجم على القرص بين التشغيلات، مما يحقق أوقات compilation أسرع بشكل كبير عبر إعادة التشغيل، خاصة في المشاريع الكبيرة.

تفعيل تخزين نظام الملفات في إعداداتك:

next.config.ts

```
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
};
 
export default nextConfig;
```

جميع التطبيقات الداخلية لـ Vercel تستخدم هذه الميزة بالفعل، وقد لاحظنا تحسينات كبيرة في إنتاجية المطورين عبر المستودعات الكبيرة.

نود أن نسمع ملاحظاتك مع تكراراتنا على تخزين نظام الملفات. يرجى تجربتها ومشاركة خبراتك.

### تبسيط create-next-app[](#simplified-create-next-app)

تم إعادة تصميم `create-next-app` مع تدفق إعدادات مبسط، وهيكل مشروع محدث، وافتراضيات محسّنة. يحتوي القالب الجديد على App Router كافتراضي، وإعدادات تركز على TypeScript، وTailwind CSS، و ESLint.

### واجهة برمجة تطبيقات مجرّدات البناء (تجريبي)[](#build-adapters-api-alpha)

تبعًا لـ [Build Adapters RFC](https://github.com/vercel/next.js/discussions/77740)، عملنا مع المجتمع ومنصات النشر لتقديم أول إصدار تجريبي من واجهة برمجة تطبيقات مجرّدات البناء.

تسمح مجرّدات البناء بإنشاء مجرّدات مخصصة يتم توصيلها بعملية البناء، مما يسمح لمنصات النشر والتكاملات البنائية المخصصة بتعديل إعدادات Next.js أو معالجة مخرجات البناء.

next.config.js

```
const nextConfig = {
  experimental: {
    adapterPath: require.resolve('./my-adapter.js'),
  },
};
 
module.exports = nextConfig;
```

شاركونا ملاحظاتكم في [مناقشة RFC](https://github.com/vercel/next.js/discussions/77740).

### دعم React Compiler (مستقر)[](#react-compiler-support-stable)

تم إخراج دعم مدمج لـ React Compiler ليكون مستقرًا في Next.js 16 بعد إصدار 1.0 من React Compiler. يقوم React Compiler تلقائيًا بتأليف المكونات، مما يقلل من التحديثات اللازمية بدون تغييرات يدوية على الكود.

تم نقل خيار الإعداد `reactCompiler` من `experimental` إلى مستقر. إنه ليس مفعلًا بشكل افتراضي لأننا نواصل جمع بيانات أداء البناء عبر أنواع التطبيقات المختلفة. من المتوقع أن تكون أوقات compilation في التطوير والبناء أعلى عند تمكين هذا الخيار لأن React Compiler يعتمد على Babel.

next.config.ts

```
const nextConfig = {
  reactCompiler: true,
};
 
export default nextConfig;
```

تثبيت أحدث إصدار من ملحق React Compiler:

Terminal

```
npm install babel-plugin-react-compiler@latest
```

### الميزات الأساسية والبنية المعمارية[](#core-features--architecture)

### التوجيه والتنقل المحسن[](#enhanced-routing-and-navigation)

تتضمن Next.js 16 إعادة تصميم كاملة لنظام التوجيه والتنقل، مما يجعل انتقالات الصفحات أخف وأسرع.

**إزالة تكرار التخطيط**: عند تحميل صفحات متعددة بمواقع مشتركة، يتم تحميل التخطيط مرة واحدة بدلاً من تحميله بشكل منفصل لكل Link. على سبيل المثال، الصفحة التي تحتوي على 50 رابط منتج تحمل الآن التخطيط المشترك مرة واحدة بدلاً من 50 مرة، مما يقلل بشكل كبير من حجم نقل الشبكة.

**التحميل المسبق التدريجي**: تقوم Next.js بتحميل أجزاء فقط غير موجودة حاليًا في الذاكرة المؤقتة، بدلاً من الصفحات الكاملة. ذاكرة التخزين المؤقتية للتحميل المسبق الآن:


*   تلغي الطلبات عندما يغادر الرابط منطقة العرض
*   تعطي أولوية لتحميل الروابط أثناء التمرير أو عند العودة لمنطقة العرض
*   تعيد تحميل الروابط عندما يتم إلغاء صلاحية بياناتها
*   تعمل بسلاسة مع الميزات القادمة مثل مكونات التخزين

**المفاضلة**: قد تلاحظ المزيد من طلبات التحميل المسبق الفردية، ولكن مع إجمالي أحجام نقل أقل بكثير. نعتقد أن هذا هو الخيار الصحيح لتطبيق تقريبا جميع التطبيقات. إذا سبب زيادة عدد الطلبات مشكلات، يرجى إعلامنا. نحن نعمل على تحسينات إضافية لتعبئة قطع البيانات بكفاءة.

لا تتطلب هذه التغييرات تعديلات على الكود وهي مصممة لتحسين الأداء عبر جميع التطبيقات.

### PPR ومكونات التخزين[](#ppr-and-cache-components)

تزيل Next.js 16 خيار `PPR` التجريبي **التجريبي من التحضير الجزئي (PPR)** وخيارات الإعداد. يتم دمج PPR في مكونات التخزين.

بدءًا من Next.js 16، يمكنك الاشتراك في PPR باستخدام خيار الإعداد `experimental.cacheComponents`. توجد اختلافات في التنفيذ تجلب مكونات التخزين ميزات وسلوكيات إضافية سيتم توثيقها والإعلان عنها قبل Next.js Conf والإصدار المستقر لـ Next.js 16.

"> **إذا كان تطبيقك يعتمد على PPR** (`experimental.ppr = true`): البق على النسخة المحددة من Next.js `canary` التي تستخدمها حاليًا. في حالة وجود صعوبة في الترحيل، البق على نسختك الحالية في الوقت الحالي، وسنوفر دليل ترحيل قبل الإصدار المستقر.

### واجهات برمجة تطبيقات التخزين المحسّنة[](#improved-caching-apis)

تقدم Next.js 16 واجهات برمجة تطبيقات تخزين محسّنة للتحكم أكثر صراحةً بسلوك الذاكرة المؤقتة.

#### `revalidateTag()` (محدث)[](#revalidatetag-updated)

تتطلب `revalidateTag()` الآن **[ملف `cacheLife` profile](https://nextjs.org/docs/app/api-reference/functions/cacheLife#reference)** كالمعطى الثاني لتمكين سلوك SWR (stale-while-revalidate):


```
import { revalidateTag } from 'next/cache';
 
// ✅ Use built-in cacheLife profile (we recommend 'max' for most cases)
revalidateTag('blog-posts', 'max');
 
// Or use other built-in profiles
revalidateTag('news-feed', 'hours');
revalidateTag('analytics', 'days');
 
// Or use an inline object with a custom revalidation time
revalidateTag('products', { revalidate: 3600 });
 
// ⚠️ Deprecated - single argument form
revalidateTag('blog-posts');
```

يقبل المعطى profile أسماء ملفات التخزين المؤقت المضمنة (`cacheLife` profiles) مثل `'max'`، `'hours'`، `'days'` أو [ملفات مخصصة](https://nextjs.org/docs/app/api-reference/functions/cacheLife#defining-reusable-cache-profiles) محددة في `next.config`. يمكنك أيضًا تمرير كائن مدمج `{ revalidate: number }`. نوصي باستخدام `'max'` لمعظم الحالات، لأنه يتيح التحديث في الخلفية للمحتوى طويل الأمد. عند طلب المستخدمين لمحتوى موصوف، يتلقون البيانات المؤقتة فورًا بينما تقوم Next.js بتجدها في الخلفية.

استخدم `revalidateTag()` عندما ترغب في إلغاء صلاحية عناصر الذاكرة المؤقتة المرتومة بشكل صحيح بسلوك SWR. هذا مثالي للمحتوى الثابت الذي يمكنه تحمل الاتساق النهائي.

"> **إرشادات الترحيل**: أضف المعطى الثاني بملف `cacheLife` (نوصي بـ `'max'`) لسلوك SWR، أو استخدم `updateTag()` في Server Actions إذا كنت بحاجة لسلوك 'اكْتُب-اقرأ-اكتب'.

#### `updateTag()` (جديد)[](#updatetag-new)

`updateTag()` هي واجهة برمجة تطبيقات جديدة خاصة بإجراءات الخادم فقط توفر سلوك **اكْتُب-اقرأ-اكتب** (read-your-writes)، مما يؤدي لإلغاء صلاحية وتجديد البيانات المؤقتة فورًا داخل الطلب نفسه:

```
'use server';
 
import { updateTag } from 'next/cache';
 
export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile);
 
  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`);
}
```

هذا يضمن أن الميزات التفاعلية تعكس التغييرات فورًا. مثالي للنماذج وإعدادات المستخدم وأي سير عمل يتوقع المستخدم رؤية تحديثاته على الفور.

#### `refresh()` (جديد)[](#refresh-new)

`refresh()` هي واجهة برمجة تطبيقات جديدة خاصة بإجراءات الخادم فقط لتجديد **المعلومات غير المؤقتة فقط**. لا تمس هذه الذاكرة المؤقتة على الإطلاق:

```
'use server';
 
import { refresh } from 'next/cache';
 
export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId);
 
  // Refresh the notification count displayed in the header
  // (which is fetched separately and not cached)
  refresh();
}
```

هذه الواجهة مكملة لـ `router.refresh()` من جانب العميل. استخدمها عند الحاجة لتجديد المعلومات غير المؤقتة المعروضة في مكان آخر في الصفحة بعد تنفيذ إجراء. تظل أصداف الصفحة المؤقتة والمحتوى الثابت سريعة بينما يتم تحديث البيانات الديناميكية مثل عدادات الإشعارات والمقاييس الحية ومؤشرات الحالة.

### React 19.2 وميزات Canary[](#react-192-and-canary-features)

يستخدم App Router في Next.js 16 أحدث إصدار [Canary release](https://react.dev/blog/2023/05/03/react-canaries) لـ React، الذي يتضمن ميزات React 19.2 الجديدة وغيرها من الميزات التي يتم تثبيتها تدريجيًا. تشمل أبرز ما في ذلك:

*   **[انتقالات العرض (View Transitions)](https://react.dev/reference/react/ViewTransition)**: تحريك العناصر التي يتم تحديثها داخل Transition أو التنقل
*   **[`useEffectEvent`](https://react.dev/reference/react/useEffectEvent)**: استخرج المنطق غير التفاعلي من Effects إلى دوال Event Effect قابلة لإعادة الاستخدام
*   **[Activity](https://react.dev/reference/react/Activity)**: تشغيل "نشاط في الخلفية" عن طريق إخفاء الواجهة باستخدام `display: none` مع الحفاظ على الحالة وتنظيف Effects

تعلم المزيد في [إعلان React 19.2](https://react.dev/blog/2025/10/01/react-19-2).

| تم الحذف | البديل |
| --- | --- |
| **دعم AMP** | جميع APIs وإعدادات AMP محذوفة (`useAmp`، `export const config = { amp: true }`) |
| **أمر `next lint`** | استخدم Biome أو ESLint مباشرةً؛ لم يعد `next build` يقوم بتشغيل Linting. يوجد codemod متاح: `npx @next/codemod@canary next-lint-to-eslint-cli .` |
| **خيارات `devIndicators`** | `appIsrStatus`، `buildActivity`، `buildActivityPosition` محذوفة من الإعداد. المؤشر يظل موجودًا. |
| **`serverRuntimeConfig`، `publicRuntimeConfig`** | استخدم متغيرات البيئة (ملفات `.env`) |
| **موقع `experimental.turbopack`** | انتقل الإعداد إلى `turbopack` على مستوى المستوى الأول (لم يعد في `experimental`) |
| **علم `experimental.dynamicIO`** | أُعيد تسميته إلى `experimental.cacheComponents` |
| **علم `experimental.ppr`** | علم PPR محذوف؛ يتطور إلى نموذج برمجة مكون التخزين |
| **`export const experimental_ppr`** | تصدير PPR على مستوى المسار محذوف؛ يتطور إلى نموذج برمجة مكون التخزين |
| **`scroll-behavior: smooth` تلقائي** | أضف `data-scroll-behavior="smooth"` إلى وثيقة HTML لاختيار العودة |
| **`unstable_rootParams()`** | نحن نعمل على واجهة API بديلة سنطلقها في تحديث فرعي قادم |
| **الوصول المتزامن لخصائص `params`، `searchParams`** | يجب استخدام async: `await params`، `await searchParams` |
| **الوصول المتزامن لـ `cookies()`، `headers()`، `draftMode()`** | يجب استخدام async: `await cookies()`، `await headers()`، `await draftMode()` |
| **وسيط `params` لمسار صورة Metadata** | تغير إلى `params` async؛ `id` من `generateImageMetadata` هو الآن `Promise<string>` |
| **`next/image` src محلي مع سلاسل الاستعلام** | يتطلب الآن إعداد `images.localPatterns` لمنع هجمات الاستعلام |

### تغييرات السلوك[](#behavior-changes)

هذه الميزات لها سلوكيات افتراضية جديدة في Next.js 16:

| التغيير في السلوك | التفاصيل |
| --- | --- |
| **حزمة الافتراضية** | Turbopack هو الآن الحزمة الافتراضية لجميع التطبيقات؛ اختيار الخروج باستخدام `next build --webpack` |
| **افتراض `images.minimumCacheTTL`** | تغير من 60 ثانية إلى 4 ساعات (14400 ثانية)؛ يقلل من تكلفة التحديث للصور دون ترويسات cache-control |
| **افتراض `images.imageSizes`** | أزيلت `16` من الأحجام الافتراضية (يستخدمها فقط 4.2% من المشاريع)؛ يقلل حجم srcset وتنوع API |
| **افتراض `images.qualities`** | تغير من `[1..100]` إلى `[75]`؛ يتم الآن قمع `prop` الجودة إلى أقرب قيمة في `images.qualities` |
| **`images.dangerouslyAllowLocalIP`** | قيد أمني جديد يوقف تحسين محلي IP افتراضيًا؛ ضعه على `true` للشبكات الخاصة فقط |
| **افتراض `images.maximumRedirects`** | تغير من غير محدود إلى الحد الأقصى 3 إعادة توجيه؛ اضبطه على `0` لتعطيله أو زيادة الحالات النادرة |
| **`@next/eslint-plugin-next` الافتراضي** | الآن ينفعل بتنسيق ESLint Flat Config، ليتوافق مع ESLint v10 الذي سيحذف الدعم القديم للإعداد |
| **سلوك ذاكرة التخزين المؤقتة للتحميل المسبق** | إعادة كتابة كاملة مع إزالة تكرار التخطيط والتحميل المسبق التدريجي |
| **توقيع `revalidateTag()`** | يتطلب الآن ملف `cacheLife` كالمعطى الثاني لسلوك SWR |
| **إعدادات Babel في Turbopack** | يتم تمكين Babel تلقائيًا إذا تم العثور على إعداد babel (لم يعد يوقف بمشكلة قاسية سابقًا) |
| **مخرجات Terminal** | إعادة تصميم مع تنسيق أوضح، رسائل خطأ أفضل، ومقاييس أداء محسّنة |
| **مجلدات مخرجات التطوير والبناء** | `next dev` و `next build` يستخدمان الآن مجلدات مخرجات منفصلة، مما يتيح التنفيذ المتزامن |
| **سلوك ملف القفل** | تمت إضافة آلية ملف القفل لمنع عدة إصدارات من `next dev` أو `next build` على نفس المشروع |
| **`default.js` للمسارات المتوازية** | جميع خيوط المسارات المتوازية تتطلب الآن ملفات `default.js` صريحة؛ يفشل البناء بدونها. أنشئ `default.js` الذي يستدعي `notFound()` أو يعيد `null` للسلوك السابق |
| **واجهة Sass الحديثة** | تم ترقية `sass-loader` إلى v16، التي تدعم Syntax Sass الحديث والميزات الجديدة |

### التبعيات[](#deprecations)

هذه الميزات قديمة في Next.js 16 وسيتم حذفها في إصدار قادم:

| المستبعد | التفاصيل |
| --- | --- |
| اسم ملف `middleware.ts` | أعد التسمية إلى `proxy.ts` لتوضيح الحد الشبكة و تركيز التوجيه |
| مكون `next/legacy/image` | استخدم `next/image` للحصول على أداء وتحسينات أفضل |
| إعداد `images.domains` | استخدم إعداد `images.remotePatterns` بدلاً منه للتحسين في قيود الأمان |
| المعطى الواحد لـ `revalidateTag()` | استخدم `revalidateTag(tag, profile)` لـ SWR، أو `updateTag(tag)` في Actions لـ 'اكْتُب-اقرأ-اكتب' |

### تحسينات إضافية[](#additional-improvements)

*   **تحسينات الأداء**: تحسينات أداء كبيرة لأوامر `next dev` و `next start` | **TypeScript الأصلي لـ Node.js لـ `next.config.ts`**: قم بتشغيل أوامر `next dev` و `next build` و `next start` مع علم `--experimental-next-config-strip-types` لتمكين TypeScript الأصلي لـ `next.config.ts`.
