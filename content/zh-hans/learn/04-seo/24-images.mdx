### 未优化图片 [](#unoptimized-images)

使用常规 HTML，我们添加了以下 Hero 图片：

```
<img src="large-image.jpg" alt="Large Image" />
```

然而，这意味着我们必须手动优化一些内容，例如：

*   确保我们的图片在不同屏幕尺寸上是响应式的。
*   使用第三方工具或库优化我们的图片。
*   当图片进入视口时进行懒加载

### Image 组件 [](#the-image-component)

Next 提供了一个 [Image 组件](/docs/api-reference/next/image)，可以为我们开箱即用处理这些优化。

`next/image` 是 HTML `img` 元素的扩展，已针对现代网络进行了改进。

这意味着，使用 `next/image` 可以自动调整大小、优化和以现代格式（如 WebP，当浏览器支持时）提供图像。

该组件避免将大图片发送到较小视口的设备，并允许 Next.js 采用未来的图像格式，并为支持这些格式浏览器提供这些图片。

自动图片优化适用于任何图片来源。即使图片托管在外部数据源（如 CMS）上，仍然可以进行优化。

### 自动图片优化是如何工作的？[](#how-does-automatic-image-optimization-work)

#### 按需优化 [](#on-demand-optimization)

Next.js 不是在构建时优化图片，而是在用户请求时按需优化图片。与静态站点生成器和仅限静态的解决方案不同，无论交付十张图片还是一千万张图片，构建时间都不会增加。

#### 懒加载图片 [](#lazy-loaded-images)

默认情况下，图片会进行懒加载。视口外的图片不会影响页面速度得分。图片仅在进入视野时加载。

#### 避免累积布局偏移 (CLS) [](#avoids-cls)

图片始终会被渲染，以避免累积布局偏移 (CLS)。

### [使用 Image 组件](#using-the-image-component)

让我们使用 `next/image` 更新应用以显示我们的 Hero 图片。高度和宽度属性应为期望的渲染大小，且宽高比与源图片相同。

打开 `pages/index.js` 文件并添加以下导入：

在文件开头从 `next/image` 导入 `Image`：

```
import Image from 'next/image';
```

然后，用 `Image` 组件替换 Hero `img`：

```
// Before
<img src="large-image.jpg" alt="Large Image" />
 
// After
<Image src="/large-image.jpg" alt="Large Image" width={3048} height={2024} />
```

页脚中还有一张图片。我们也替换它：

```
// Before
<img src="/vercel.svg" alt="Vercel Logo" />
 
// After
<Image src="/vercel.svg" alt="Vercel Logo" width={72} height={16} />
```

最后，在 Chrome DevTools 中运行另一个 Lighthouse 报告并比较结果。
