正如我们在 [Next.js Conf](/conf) 上宣布的，我们创造最佳开发体验的使命通过 Next.js 11 延续，其中包括：

*   [**符合性 (Conformance)**](#conformance): 提供精心设计的解决方案以支持最佳用户体验的系统。
*   [**性能提升**](#improved-performance): 进一步优化的冷启动时间，让您能够更快地开始编码。
*   [**`next/script`**](#script-optimization): 自动优先加载第三方脚本，提高性能。
*   [**`next/image`**](#image-improvements): 减少布局偏移，并通过自动尺寸检测和模糊占位符支持创建更流畅的视觉体验。
*   [**Webpack 5**](#webpack-5): 现在默认启用所有 Next.js 应用，为所有 Next.js 开发者带来 [这些好处](/blog/next-10-2#webpack-5)。
*   [**Create React App 迁移 (Experimental)**](#cra-migration): 自动将 Create React App 转换为与 Next.js 兼容。
*   [**Next.js Live (预览版本)**](#nextjs-live-preview-release): 与团队在浏览器中实时协作编码。

今日请使用 `npm i next@latest react@latest react-dom@latest` 进行更新，并参考下方的 [升级指南](#upgrade-guide)。

符合性 (Conformance)[](#conformance)
---------------------------

即使拥有优秀的工具及框架内的自动优化，站点拥有者和应用开发者仍常被要求成为 UX 质量主题（如性能、安全性和无障碍性）方面的专家。当功能增加及团队扩大时，开发者需要以不同的方式思考。

Google 通过构建 Search 和 Maps 等大型 Web 应用的经历证明，随着团队和应用的扩展，框架在维持质量方面可以发挥关键作用。通过利用强默认值和保障的系统，他们赋能开发人员更加专注于功能和产品。

当今，Google 的 Web 平台团队已开始开源他们的系统，即 **Next.js 的符合性 (Conformance)**。

[**符合性 (Conformance)**](http://web.dev/conformance) 是一个提供精心设计的解决方案和规则以支持最佳加载和 Core Web Vitals (核心 Web 指标) 的系统，未来将增加更多功能以支持其他质量方面，如安全性和无障碍性。这些解决方案让您的团队不必死记硬背所有最新的最佳加载性能规则，同时仍然赋予您为应用程序做出正确选择的灵活性。

除了许多由性能研究支持的基础优化外，**Next.js 11 现在开箱即支持 [ESLint](https://eslint.org/)**，以便在开发期间更容易捕获特定于框架的问题，并为团队设定指南以确保最佳实践，即使在扩展时也是如此。

要开始使用 ESLint，请在升级到 Next.js 11 后运行 `npx next lint`。ESLint 集成适用于新建及现有的 Next.js 应用，提供一套新规则帮助开发者构建更好的应用。

性能提升[](#improved-performance)
---------------------------------------------

自 Next.js 10 以来，我们一直致力于进一步提升 Next.js 的开发体验。在 [10.1](/blog/next-10-1) 和 [10.2](/blog/next-10-2) 中，我们将启动时间缩短了 **24%**，通过 React Fast Refresh 将处理时间缩短了 **40%**。您只需更新 Next.js 即可获得这些惊人的速度提升。

Next.js 11 包含对 Babel 的另一次优化，以进一步减少启动时间。我们创建了全新的 Webpack Babel loader 实现，优化了加载过程并添加了内存配置缓存层。实际上，这对**开发者没有任何改变**，但最终意味着更快的开发体验。

脚本优化 (Script Optimization)[](#script-optimization)
-------------------------------------------

新的 Next.js Script 组件是一个基础优化，它使开发人员能够设置第三方脚本的加载优先级，从而节省开发人员时间并提高加载性能。

网站通常需要第三方服务来进行分析、广告、客户支持小部件和同意管理等操作。然而，这些脚本往往会占用大量加载性能，可能会拖累用户体验。开发者经常难以决定在应用程序中何处放置它们以实现最佳加载效果。

使用 `next/script`，您可以定义 `strategy` 属性，Next.js 将自动优先处理它们以提高加载性能：

*   `beforeInteractive`: 对于需要在页面交互**之前**获取并执行的**关键**脚本，如机器人检测和同意管理。这些脚本由服务器注入到初始 HTML 中，并在自捆绑 JavaScript 执行之前运行。
*   `afterInteractive` (**默认**): 对于可以在页面交互**之后**获取并执行的脚本，如标签管理器和分析工具。这些脚本在客户端注入，将在水合后运行。
*   `lazyOnload`: 可以在空闲时间等待加载的脚本，如聊天支持和社交媒体小部件。

```
<Script
  src={url}
  strategy="beforeInteractive" // lazyOnload, afterInteractive
/>
```

加载后也可以运行代码。例如，您可以等待用户回答同意后再执行代码：

```
<Script
  src={url} // consent mangagement
  strategy="beforeInteractive"
  onLoad={() => {
    // If loaded successfully, then you can load other scripts in sequence
  }}
/>
```

我们还将 Next.js 11 中的默认脚本加载体验从预加载和 `async` 更改为 `defer`。第三方脚本通常与更高优先级的资源（如 CSS、字体和图像）竞争。与这些资源以及其他脚本保持正确的序列，对开发者来说是不合理的负担。

通过提供具有 `afterInteractive` 作为默认加载策略的 Script 组件，开发者现在拥有了更好的默认值以实现最佳性能，并且仍然可以在需要时选择 `beforeInteractive`。

要了解有关切换默认的决策背后的技术选择，请查看 RFC [RFC](https://github.com/vercel/next.js/discussions/24938) 以及 Google Chrome 团队关于 [预加载挑战](https://docs.google.com/document/u/0/d/1ZEi-XXhpajrnq8oqs5SiW-CXR3jMc20jWIzN5QRy1QA/mobilebasic#) 的资料。

图像改进 (Image Improvements)[](#image-improvements)
-----------------------------------------

我们很高兴分享社区针对 `next/image` 组件的最受欢迎功能请求，减少 [累积布局偏移 (Cumulative Layout Shift)](https://vercel.com/blog/core-web-vitals#cumulative-layout-shift) 并创建更流畅的视觉体验。

### 自动尺寸检测 (Local Images)[](#automatic-size-detection-local-images)

对于图像 `src` 使用 `import` 关键字以自动定义静态图像的 `width` 和 `height`。

例如，使用 [内置的 Image 组件](/docs/pages/building-your-application/optimizing/images) 现在更加容易：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Image from 'next/image';
import author from '../public/me.png';
 
export default function Home() {
  return (
    // When importing the image as the source, you
    // don't need to define `width` and `height`.
    <Image src={author} alt="Picture of the author" />
  );
}
```

### 图像占位符 (Image Placeholders)[](#image-placeholders)

`next/image` 现在支持模糊占位符 (blur-up placeholders) 以缓解从空白空间到图像的过渡并减少感知加载时间，特别是对于互联网连接较慢的用户。

要使用模糊占位符，请在图像中添加 `placeholder="blur"`。

```
<Image src={author} alt="Picture of the author" placeholder="blur" />
```

Next.js 还支持模糊动态图像，您可以提供自定义 `blurDataURL`，该数据由您的后端提供。例如，您可以在服务器端生成 [blurha.sh](http://blurha.sh)。

Webpack 5[](#webpack-5)
-----------------------

随着 Next.js 10.2，我们将 webpack 5 的推广扩大到所有 `next.config.js` 中没有自定义 webpack 配置的应用。今天，我们让 **webpack 5 成为所有 Next.js 应用的默认设置**，这将提供 [各种特性和改进](/blog/next-10-2#webpack-5)。**。**

我们密切与社区合作以确保平滑过渡到 webpack 5，每个启用 webpack 5 的拉取请求都运行了超过 3,400 个现有的 Next.js 集成测试。

如果你的应用程序有自定义 webpack 配置，我们建议遵循 [webpack 5 的升级文档](/docs/messages/webpack5)。如果遇到问题，[请与我们分享反馈](https://github.com/vercel/next.js/discussions/23498)。

CRA 迁移 (CRA Migration)[](#cra-migration)
-------------------------------

在过去的六个月里，我们看到 [从 Create React App 迁移到 Next.js](/docs/app/building-your-application/upgrading/from-create-react-app) 的应用呈增加趋势，以充分利用 Next.js 提供的多项开发体验和最终用户性能改进。

为了帮助开发者将应用程序转换为 Next.js，我们为 `@next/codemod` 引入了一个新工具，可以将 Create React App 应用程序自动转换为与 Next.js 兼容。

该转换会自动添加 `pages/` 目录，并将 CSS 导入移动到正确的位置。它还将在 Next.js 中启用 Create React App 兼容模式，以确保 Create React App 中使用的一些模式与 Next.js 兼容。

通过利用新的转换，您可以 [逐步采用 Next.js](/docs/migrating/incremental-adoption)，同时保持现有 Create React App 应用程序的功能。

要开始迁移 Create React App 项目，请使用以下命令：

终端

```
npx @next/codemod cra-to-next
```

此功能目前是实验性的，请在 [此讨论](https://github.com/vercel/next.js/discussions/25858) 中分享任何反馈。

Next.js Live (预览版本)[](#nextjs-live-preview-release)
--------------------------------------------------------------

Next.js Live 是我们使开发不仅**更快**且更令人愉快，而且关键在于**让整个组织都能参与**的使命的延续。通过利用 ServiceWorker、WebAssembly 和 ES 模块 (ES Modules) 等尖端技术，Next.js Live 将整个开发过程置于 Web 浏览器中。这使得拥有 URL 即可即时协作和共享成为可能，**无需构建步骤**。对于开发者来说，这意味着更快的反馈循环，减少等待构建的时间，以及在浏览器内进行实时的结对编程和编辑。

升级指南 (Upgrade Guide)[](#upgrade-guide)
-------------------------------

Next.js 11 引入了一些破坏性更改，但不应影响大多数用户。这些遗留功能多年来一直以向后兼容的方式维护，有些甚至可追溯到 `v4.0`。

移除这些功能是为了减少包体大小并确保代码库在未来易于维护。要了解从版本 10 升级到 11 的更多信息，请参见 [升级指南](https://github.com/vercel/next.js/blob/canary/docs/pages/building-your-application/upgrading.md)。

随着 Next.js 11，最低 React 版本已更新为 `17.0.2`。请查看 [React 17 博客文章](https://reactjs.org/blog/2020/08/10/react-v17-rc.html) 以获取更多详细信息。我们也与 React 团队紧密合作，因为他们推出了 [React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html)。当使用 React 18 alpha 时，Next.js 11 使用 `createRoot`。
