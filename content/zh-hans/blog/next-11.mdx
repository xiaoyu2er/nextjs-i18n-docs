正如我们在 [Next.js Conf](/conf) 上宣布的那样，Next.js 11 延续了我们要为开发者创造最佳体验的使命，其中包括：

* [**合规性 (Conformance)**](#conformance): 一套精心打造的解决方案，以支持最佳用户体验。
* [**性能提升**](#improved-performance): 进一步优化冷启动时间，让您能更快开始编码。
* [**`next/script`**](#script-optimization): 自动优先加载第三方脚本以提升性能。
* [**`next/image`**](#image-improvements): 通过自动尺寸检测和支持模糊占位符，减少布局偏移并创造更流畅的视觉体验。
* [**Webpack 5**](#webpack-5): 现已作为 Next.js 应用的默认启用项，为所有 Next.js 开发人员带来 [这些好处](/blog/next-10-2#webpack-5)。
* [**Create React App 迁移 (实验性)**](#cra-migration): 自动转换 Create React App 以兼容 Next.js。
* [**Next.js Live (预览版)**](#nextjs-live-preview-release): 在浏览器中与团队实时进行代码编写。

今天就通过运行 `npm i next@latest react@latest react-dom@latest` 进行更新，并参考下面的 [迁移指南](#upgrade-guide)。

合规性 (Conformance)[](#conformance)
---------------------------

即使在框架中拥有优秀的工具和自动优化功能，站点所有者和应用程序开发人员也常被要求成为 UX 质量主题（如性能、安全性和无障碍性）的专家。随着功能的添加和团队的扩展，开发人员需要以不同的思维方式思考。

通过构建像 Search 和 Maps 这样的大型 Web 应用程序，Google 已证明框架在团队和应用程序扩展过程中在保持质量方面发挥着至关重要的作用。通过利用一组强大的默认值和防护措施，他们使开发人员能够更多地关注功能和产品。

如今，Google 的 Web 平台团队已开始将其系统开源，即 Next.js 的合规性系统。

[**合规性**](http://web.dev/conformance) 是一套精心打造的解决方案和规则，旨在支持最佳加载速度和核心 Web 指标，未来还将增加支持其他质量方面（如安全性和无障碍性）。这些解决方案使您的团队无需记住所有最新的最佳加载性能规则，同时仍为您提供灵活性以针对您的应用程序做出正确选择。

除了许多由性能研究支持的基础优化外，**Next.js 11 现已开箱支持 [ESLint](https://eslint.org/)**，以便更容易在开发期间发现框架特定问题，并为您的团队设置指南，即使在扩展过程中也能确保最佳实践。

要开始使用 ESLint，请在升级到 Next.js 11 后运行 `npx next lint`。ESLint 集成适用于新的和现有的 Next.js 应用，提供了一套新的规则来帮助开发人员构建更好的应用。

性能提升[](#improved-performance)
---------------------------------------------

自从 Next.js 10 以来，我们一直专注于进一步改善 Next.js 的开发者体验。在 [10.1](/blog/next-10-1) 和 [10.2](/blog/next-10-2) 中，我们将启动时间缩短了 **高达 24%**，并通过 React Fast Refresh 将更改的处理时间缩短了另外 **40%**。您通过保持 Next.js 更新，一直在享受这些惊人的速度提升。

Next.js 11 包含了对 Babel 的另一个优化，以进一步减少启动时间。我们为 webpack 创建了一个全新的 Babel 加载器实现，优化了加载并添加了内存配置缓存层。在实践中，这对开发人员来说**没有变化**，但最终将意味着更快的开发体验。

脚本优化 [](#script-optimization)
-------------------------------------------

新的 Next.js Script 组件是一项基础性优化，使开发人员能够设置第三方脚本的加载优先级，以节省开发人员的时间并提高加载性能。

网站通常需要第三方服务来处理分析、广告、客户支持小部件和同意管理等功能。然而，这些脚本往往在加载性能上较重，会降低用户体验。开发人员常常难以决定在应用程序中放置它们以实现最佳加载。

使用 `next/script`，您可以定义 `strategy` 属性，Next.js 将自动优先处理它们以提高加载性能：

* `beforeInteractive`：针对需要在页面可交互**之前**获取并执行的脚本，例如机器人检测和同意管理。这些脚本从服务器注入初始 HTML，并在本地捆绑 JavaScript 执行之前运行。
* `afterInteractive` (**默认**)：针对可以在页面可交互**之后**获取和执行的脚本，例如标签管理器和分析工具。这些脚本在客户端注入，将在水合后运行。
* `lazyOnload`：针对可以在空闲时间等待加载的脚本，例如聊天支持和社交媒体小部件。

```
<Script
  src={url}
  strategy="beforeInteractive" // lazyOnload, afterInteractive
/>
```

您也可以在加载后运行代码。例如，您可以等待执行代码直到用户回答同意：

```
<Script
  src={url} // consent mangagement
  strategy="beforeInteractive"
  onLoad={() => {
    // If loaded successfully, then you can load other scripts in sequence
  }}
/>
```

我们还更改了 Next.js 11 中的默认脚本加载体验，从预加载和 `async` 改为 `defer`。第三方脚本通常会与更高优先级的资源（如 CSS、字体和图像）竞争。维护这些资源以及其他脚本的适当序列对开发人员来说是过重的负担。

通过提供一个默认为 `afterInteractive` 的 Script 组件，开发人员现在拥有了针对最佳性能的更好默认值，并且仍然可以在需要时选择 `beforeInteractive`。

要了解切换到默认值背后的技术选择，请查看 [RFC](https://github.com/vercel/next.js/discussions/24938) 和 Google Chrome 团队的 [预加载挑战](https://docs.google.com/document/u/0/d/1ZEi-XXhpajrnq8oqs5SiW-CXR3jMc20jWIzN5QRy1QA/mobilebasic#)。

### 图像占位符 [](#image-placeholders)

`next/image` 现在支持模糊提升占位符，以缓解从空白空间到图像的过渡并减少感知加载时间，特别是对于互联网连接较慢的用户。

要使用模糊占位符，请向您的图像添加 `placeholder="blur"`。

```
<Image src={author} alt="Picture of the author" placeholder="blur" />
```

Next.js 还支持模糊动态图像，通过允许您提供由后端提供的自定义 `blurDataURL`。例如，您可以在服务器上生成 [blurha.sh](http://blurha.sh)。

Webpack 5[](#webpack-5)
-----------------------

随着 Next.js 10.2，我们将 webpack 5 的部署扩展到所有在 `next.config.js` 中没有自定义 webpack 配置的应用程序。今天，我们正将 **webpack 5 设为所有 Next.js 应用的默认值**，这将为所有 Next.js 开发人员提供 [各种功能和改进](/blog/next-10-2#webpack-5)。**

我们与社区密切合作，以确保平滑过渡到 webpack 5，超过 3,400 个现有 Next.js 集成测试在每次拉取请求上运行（启用 webpack 5）。

如果您的应用程序有自定义 webpack 配置，我们建议您遵循 [webpack 5 的升级文档](/docs/messages/webpack5)。如果您遇到任何问题，请 [与我们分享反馈](https://github.com/vercel/next.js/discussions/23498)。

CRA 迁移 [](#cra-migration)
-------------------------------

在过去六个月中，我们已看到更多应用程序从 Create React App 迁移到 Next.js，以利用 Next.js 提供的许多开发者体验和有用户性能改进。

为了帮助开发人员将应用程序转换为 Next.js，我们在 `@next/codemod` 中引入了一个新工具，可将 Create React App 应用程序自动转换为 Next.js 兼容。

该转换自动添加 `pages/` 目录并将 CSS 导入移动到正确的位置。它还将启用 Next.js 中的 Create React App 兼容模式，以确保 Create React App 中使用的一些模式能与 Next.js 配合使用。

通过利用新的转换，您可以 [逐步采用 Next.js](/docs/migrating/incremental-adoption)，同时保留现有 Create React App 应用的功能。

要开始迁移您的 Create React App 项目，请使用以下命令：

终端

```
npx @next/codemod cra-to-next
```

此功能目前处于实验阶段，请在此讨论中 [分享任何反馈](https://github.com/vercel/next.js/discussions/25858)。

Next.js Live (预览发布)[](#nextjs-live-preview-release)
--------------------------------------------------------------

Next.js Live 延续了我们的使命，使开发不仅**更快**、更令人愉快，而且关键的是**更包容整个组织**。通过利用 ServiceWorker、WebAssembly 和 ES Modules 等尖端技术，Next.js Live 将整个开发过程置于 Web 浏览器中。这开启了使用 URL 即时协作和共享的可能性，**无需构建步骤**。对于开发人员来说，这意味着更快的反馈循环，减少等待构建的时间，以及在浏览器内进行实时结对编程和编辑。

升级指南 [](#upgrade-guide)
-------------------------------

Next.js 11 引入了一些破坏性变更，这些变更不应影响大多数用户。这些遗留功能多年来一直向后兼容地维护，有些甚至可以追溯到 `v4.0`。

移除这些功能是为了减小 bundle 大小并确保代码库在未来可维护。要了解有关从版本 10 升级到 11 的更多信息，请参阅 [升级指南](https://github.com/vercel/next.js/blob/canary/docs/pages/building-your-application/upgrading.md)。

随着 Next.js 11，最低 React 版本已更新为 `17.0.2`。更多信息请参阅 [React 17 博客文章](https://reactjs.org/blog/2020/08/10/react-v17-rc.html)。随着 React 团队引入 [React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html)，我们也在与其团队密切合作。当使用 React 18 alpha 时，Next.js 11 使用 `createRoot`。
