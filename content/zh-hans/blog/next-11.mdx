我们在 [Next.js Conf](/conf) 上宣布，打造最佳开发者体验的使命在 Next.js 11 中继续，包括：

*   [**规范系统 (Conformance)**](#conformance): 提供精心打造的解决方案，以支持最佳用户体验的系统。
*   [**性能提升**](#improved-performance): 进一步优化冷启动时间，让您更快地开始编码。
*   [**`next/script`**](#script-optimization): 自动优先加载第三方脚本以提升性能。
*   [**`next/image`**](#image-improvements): 通过自动尺寸检测和 `blur-up` 占位符支持，减少布局偏移并创建更流畅的视觉体验。
*   [**Webpack 5**](#webpack-5): 现已默认启用于所有 Next.js 应用，为所有 Next.js 开发者带来 [这些优势](/blog/next-10-2#webpack-5)。
*   [**Create React App 迁移 (实验性功能)**](#cra-migration): 自动转换 Create React App 以与 Next.js 兼容。
*   [**Next.js Live (预览版)**](#nextjs-live-preview-release): 在浏览器中与团队实时协作编码。

今天通过运行 npm i next@latest react@latest react-dom@latest 进行更新，并参考下方的 [迁移指南](#upgrade guide)。

规范系统 (Conformance)[](#conformance)
-------------------------------

即使拥有优秀的工具和框架内的自动优化，网站所有者和应用程序开发人员也常被要求成为用户体验质量主题方面的专家，例如性能、安全性和可访问性。随着功能的添加和团队的扩展，开发者需要不同的思考方式。

通过构建 Google Search 和 Maps 等大规模 Web 应用，Google 已证明框架在团队和应用程序扩展时保持质量方面发挥着至关重要的作用。通过利用强默认值和防范措施系统，他们让开发者能够更专注于功能和产品。

今天，Google 的 Web 平台团队已开始开源其系统，名为 **Next.js 规范系统**。

[**规范系统 (Conformance)**](http://web.dev/conformance) 是一个提供精心打造的解决方案和规则的系统，以支持最佳加载性能和核心网页指标 (Core Web Vitals)，后续还将增加更多内容以支持安全性、可访问性等其他质量方面。这些解决方案让您的团队无需死记硬背所有最新的加载性能规则，同时仍然为您提供足够的灵活性为您的应用程序做出正确的选择。

伴随着许多由性能研究支持的基础性优化，**Next.js 11 现已开箱支持 [ESLint](https://eslint.org/)**，使您在开发期间更容易发现特定于框架的问题，并为您的团队设定指导原则，确保在扩展时遵循最佳实践。

开始使用 ESLint，请在升级到 Next.js 11 后运行 `npx next lint`。ESLint 集成适用于新的和现有的 Next.js 应用，提供一套新规则以帮助开发者构建更好的应用程序。

性能提升[](#improved-performance)
---------------------------------------------

自 Next.js 10 以来，我们一直致力于进一步改善 Next.js 的开发者体验。在 [10.1](/blog/next-10-1) 和 [10.2](/blog/next-10-2) 中，我们将启动时间缩短了 **最多 24%**，并通过 React Fast Refresh 节省了 **40% 的处理时间**。您只需保持 Next.js 更新，就能获得这些惊人的速度提升。

Next.js 11 对 Babel 进行了另一项优化，以进一步减少启动时间。我们创建了一个全新的 webpack Babel loader 实现，优化了加载过程并添加了内存配置缓存层。实际上，这意味着 **开发者无需更改**，但最终将带来更快的开发体验。

脚本优化 [](#script-optimization)
-------------------------------------------

新的 Next.js Script 组件是一项基础性优化，允许开发者设置第三方脚本的加载优先级，以节省开发者时间并提高加载性能。

网站通常需要第三方服务进行数据分析、广告、客户支持组件和同意管理等。然而，这些脚本往往加载性能开销大，会拖累用户体验。开发者往往难以决定在应用程序中何处放置它们以实现最佳加载。

通过 `next/script`，您可以定义 `strategy` 属性，Next.js 将自动优化它们以提高加载性能：

*   `beforeInteractive`: 需要在页面可交互**之前**获取并执行的**关键脚本**，例如机器人检测和同意管理。这些脚本由服务器注入到初始 HTML 中，并在自打包 JavaScript 执行之前运行。
*   `afterInteractive` (**默认**): 需要在页面可交互**之后**获取并执行的脚本，例如标签管理器和数据分析。这些脚本在客户端注入，并在水合 (hydration) 后运行。
*   `lazyOnload`: 可以在空闲时间等待加载的脚本，例如聊天支持和社交媒体组件。

```
<Script
  src={url}
  strategy="beforeInteractive" // lazyOnload, afterInteractive
/>
```

您还可以在加载后运行代码。例如，您可以等待执行代码直到用户回答同意：

```
<Script
  src={url} // consent mangagement
  strategy="beforeInteractive"
  onLoad={() => {
    // If loaded successfully, then you can load other scripts in sequence
  }}
/>
```

我们还已将 Next.js 11 中的默认脚本加载体验从预加载和 `async` 更改为 `defer`。第三方脚本经常与优先级较高的资源（如 CSS、字体和图像）竞争。维护与这些资源以及其他脚本之间正确的顺序对开发者来说是一个沉重的负担。

通过提供默认加载策略为 `afterInteractive` 的 Script 组件，开发者现在拥有了更好的默认值以实现最佳性能，并在需要时仍可选择 `beforeInteractive`。

了解更多关于切换默认值的背后技术选择，请查看 [RFC](https://github.com/vercel/next.js/discussions/24938) 和 Google Chrome 团队关于 [预加载的挑战](https://docs.google.com/document/u/0/d/1ZEi-XXhpajrnq8oqs5SiW-CXR3jMc20jWIzN5QRy1QA/mobilebasic#)。

图像改进 [](#image-improvements)
-----------------------------------------

我们很高兴地分享社区为 `next/image` 组件提出的两个顶级功能请求：减少 [累积布局偏移](https://vercel.com/blog/core-web-vitals#cumulative-layout-shift) 并创建更流畅的视觉体验。

### 自动尺寸检测（本地图片）[](#automatic-size-detection-local-images)

对于图片 `src` 使用 `import` 关键字以静态图片自动定义 `width` 和 `height`。

例如，使用 [内置的 Image 组件](/docs/pages/building-your-application/optimizing/images) 现在更加简单了：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Image from 'next/image';
import author from '../public/me.png';
 
export default function Home() {
  return (
    // When importing the image as the source, you
    // don't need to define `width` and `height`.
    <Image src={author} alt="Picture of the author" />
  );
}
```

### 图像占位符 [](#image-placeholders)

现在 `next/image` 支持模糊向上占位符 (`blur-up placeholders`)，以缓解从空白区域到图片的过渡，并减少感知加载时间，特别是对于互联网连接较慢的用户。

要使用模糊占位符，请在图片中添加 `placeholder="blur"`。

```
<Image src={author} alt="Picture of the author" placeholder="blur" />
```

Next.js 还支持模糊处理动态图片，允许您提供自定义的 `blurDataURL`，该数据由您的后端提供。例如，您可以在服务器端生成一个 [blurha.sh](http://blurha.sh)。

Webpack 5 [](#webpack-5)
-----------------------

凭借 Next.js 10.2，我们将 webpack 5 的扩展范围扩大到所有 `next.config.js` 中没有自定义 webpack 配置的应用程序。今天，我们要让 **webpack 5 成为所有 Next.js 应用的默认选项**，这将提供 [一系列功能和改进](/blog/next-10-2#webpack-5)。

我们与社区密切合作，以确保平稳过渡到 webpack 5，超过 3,400 个现有的 Next.js 集成测试在每次拉取请求中运行，并且启用了 webpack 5。

如果您的应用程序有自定义 webpack 配置，我们建议您遵循 [webpack 5 的升级文档](/docs/messages/webpack5)。如果您遇到任何问题，[请与我们分享反馈](https://github.com/vercel/next.js/discussions/23498)。

CRA 迁移 [](#cra-migration)
-------------------------------

在过去的六个月里，我们看到越来越多应用程序 [从 Create React App 迁移到 Next.js](/docs/app/building-your-application/upgrading/from-create-react-app)，以利用 Next.js 提供的众多开发者体验和最终用户性能改进。

为了帮助开发者将应用程序转换为 Next.js，我们向 @next/codemod 引入了一款新工具，可自动将 Create React App 应用程序转换为与 Next.js 兼容。

该转换工具会自动添加 `pages/` 目录并将 CSS 导入移至正确位置。它还会在 Next.js 中启用 Create React App 兼容模式，以确保 Create React App 中使用的一些模式与 Next.js 兼容。

通过利用新的转换工具，您可以 [渐进式采用 Next.js](/docs/migrating/incremental-adoption)，同时保持现有 Create React App 应用程序的功能。

要开始迁移您的 Create React App 项目，请使用以下命令：

终端

```
npx @next/codemod cra-to-next
```

该功能目前处于实验阶段，请在 [此讨论中](https://github.com/vercel/next.js/discussions/25858) 分享任何反馈。

Next.js Live (预览版) [](#nextjs-live-preview-release)
--------------------------------------------------------------

Next.js Live 继续致力于让开发不仅变得**更快**、更愉快，而且至关重要地 **更加包容整个组织**。通过利用 ServiceWorker、WebAssembly 和 ES Modules 等尖端技术，Next.js Live 将开发过程置于 Web 浏览器中。这打开了可能性，例如通过 URL 即时协作和共享，**无需构建步骤**。对开发者而言，这意味着更快的反馈循环、更少的构建等待时间，以及在浏览器内进行实时结对编程和编辑。

要了解更多关于 Next.js Live 的内容以及如何将其与 Vercel 的实时协作引擎配合使用，请参阅文档中的 [Next.js Live](/live) 部分。

升级指南 [](#upgrade-guide)
-------------------------------

Next.js 11 引入了一些重大更改，这不会影响大多数用户。这些遗留功能多年来一直以向后兼容的方式维护，有些甚至追溯到 `v4.0`。

移除这些功能是为了减小打包体积并确保代码库在未来可维护。要了解从版本 10 升级到 11 的更多信息，请参阅 [升级指南](https://github.com/vercel/next.js/blob/canary/docs/pages/building-your-application/upgrading.md)。

随着 Next.js 11，最低 React 版本已更新为 `17.0.2`。有关更多详细信息，请参阅 [React 17 博客文章](https://reactjs.org/blog/2020/08/10/react-v17-rc.html)。我们也与 React 团队密切合作，随着他们推出 [React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html)。当使用 React 18 alpha 时，Next.js 11 使用 `createRoot`。
