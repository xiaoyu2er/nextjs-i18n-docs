Next.js 15 发布候选版本 (RC) 现已发布。此早期版本允许您在即将到来的稳定版本发布之前测试最新功能。

*   [**React:**](#react-19-rc) 支持 React 19 RC、React Compiler（实验性）以及错误修复改进
*   [**缓存更新:**](#caching-updates) `fetch` 请求、`GET` 路由处理器和客户端导航默认不再缓存
*   [**部分预 prerendering（实验性）的增量采用:**](#incremental-adoption-of-partial-prerendering-experimental) 用于增量采用的新 Layout 和 Page 配置选项
*   [**`next/after`（实验性）:**](#executing-code-after-a-response-with-nextafter-experimental) 用于在响应流完成后执行代码的新 API
*   [**`create-next-app`：**](#create-next-app-updates) 更新的界面设计以及一个本地开发启用 Turbopack 的新标志
*   [**优化外部包的打包（稳定）：**](#optimizing-bundling-of-external-packages-stable) 新增 App 和 Pages Router 配置选项

今天试用 Next.js 15 RC：

终端

```
npm install next@rc react@rc react-dom@rc
```

React 19 RC[](#react-19-rc)
---------------------------
Next.js App Router 基于 React 的 [canary channel](https://react.dev/blog/2023/05/03/react-canaries) 构建，这使得开发者可以在 v19 版本发布之前使用这些新的 React API 并提供反馈。

Next.js 15 RC 现在支持 React 19 RC，其中包括客户端和服务器端的新功能，如 Actions。

阅读 [Next.js 15 升级指南](https://nextjs.org/docs/app/building-your-application/upgrading/version-15)、[React 19 升级指南](https://react.dev/blog/2024/04/25/react-19-upgrade-guide) 并观看 [React Conf Keynote](https://www.youtube.com/live/T8TZQ6k4SLE?t=1788) 以了解更多。

> **注意**：一些第三方库可能尚不兼容 React 19。

React Compiler（实验性）[](#react-compiler-experimental)
-------------------------------------------------------------
[React Compiler](https://react.dev/learn/react-compiler) 是 React 团队创建的一个新的实验性编译器。该编译器通过在纯 JavaScript 语义和 [React 规则](https://react.dev/reference/rules) 方面的深入理解来理解代码，从而允许它为您的代码添加自动优化。该编译器减少了开发者通过 `useMemo` 和 `useCallback` 等 API 进行手动记忆化的数量，使代码更简单、更易维护且更少错误。

随着 Next.js 15，我们添加了对 [React Compiler](https://react.dev/learn/react-compiler) 的支持。

安装 `babel-plugin-react-compiler`：

终端

```
npm install babel-plugin-react-compiler
```

然后，在 `next.config.js` 中添加 `experimental.reactCompiler` 选项：

next.config.ts

```
const nextConfig = {
  experimental: {
    reactCompiler: true,
  },
};
 
module.exports = nextConfig;
```

可选地，您可以将编译器配置为按以下方式在“opt-in”模式下运行：

next.config.ts

```
const nextConfig = {
  experimental: {
    reactCompiler: {
      compilationMode: 'annotation',
    },
  },
};
 
module.exports = nextConfig;
```

> **注意**：React Compiler 目前只能通过 Babel 插件在 Next.js 中使用，这可能会导致构建时间变慢。

了解更多关于 [React Compiler](https://react.dev/learn/react-compiler) 以及 [可用的 Next.js 配置选项](https://react.dev/learn/react-compiler#usage-with-nextjs) 的信息。

缓存更新[](#caching-updates)
-----------------------------------
Next.js App Router 启动时带有特定缓存默认值。这些默认值旨在默认提供最具性能的选择，并允许在需要时选择退出。

基于您的反馈，我们重新评估了我们的 [缓存启发式规则](https://x.com/feedthejim/status/1785242054773145636) 以及它们如何与项目（如部分预渲染 (PPR)）和使用 `fetch` 的第三方库进行交互。

随着 Next.js 15，我们要将 `fetch` 请求、`GET` 路由处理器和客户端路由器缓存的缓存默认项从“默认缓存”改为“默认不缓存”。如果您想保留以前的行为，您可以继续选择缓存。

在接下来的几个月里，我们将继续改进 Next.js 中的缓存功能，并将在 Next.js 15 GA 公告中分享更多详细信息。

### `fetch` 请求默认不再缓存[](#fetch-requests-are-no-longer-cached-by-default)

Next.js 使用 Web `fetch` API 缓存选项来配置服务器端 `fetch` 请求如何与框架的持久 HTTP 缓存交互：

```
fetch('https://...', { cache: 'force-cache' | 'no-store' });
```

*   `no-store` - 每次都从远程服务器获取资源，且不更新缓存
*   `force-cache` - 从缓存（如果存在）或远程服务器获取资源并更新缓存

在 Next.js 14 中，如果未提供 `cache` 选项，则默认使用 `force-cache`，除非使用了动态函数或动态配置选项。

在 Next.js 15 中，如果未提供 `cache` 选项，则默认使用 `no-store`。这意味着 `fetch` 请求将默认不会缓存。

您仍然可以通过以下方式选择为 `fetch` 请求启用缓存：

*   在单个 `fetch` 调用中将 [`cache` 选项](https://nextjs.org/docs/app/api-reference/functions/fetch#optionscache) 设置为 `force-cache`
*   为单个路由将 [`dynamic` 路由配置选项](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#dynamic) 设置为 `'force-static'`
*   将 [`fetchCache` 路由配置选项](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#fetchcache) 设置为 `'default-cache'` 以覆盖布局或页面中的所有 `fetch` 请求以使用 `force-cache`，除非它们明确指定了各自的 `cache` 选项

### `GET` 路由处理器默认不再缓存[](#get-route-handlers-are-no-longer-cached-by-default)

在 Next 14 中，使用 `GET` HTTP 方法的路由处理器默认会被缓存，除非使用了动态函数或动态配置选项。在 Next.js 15 中，`GET` 函数默认不会被缓存。

您仍然可以使用静态路由配置选项（例如 `export dynamic = 'force-static'`）选择加入缓存。

特殊路由处理器，如 [`sitemap.ts`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/sitemap)、[`opengraph-image.tsx`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/opengraph-image) 和 [`icon.tsx`](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/app-icons)，以及其他 [元数据文件](https://nextjs.org/docs/app/api-reference/file-conventions/metadata)，默认保持静态，除非它们使用了动态函数或动态配置选项。

### 客户端路由器缓存默认不再缓存 Page 组件[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next.js 14.2.0 中，我们引入了一个实验性的 [`staleTimes`](https://nextjs.org/docs/app/api-reference/next-config-js/staleTimes) 标志，以允许自定义配置 [路由器缓存](https://nextjs.org/docs/app/building-your-application/caching#client-side-router-cache)。

在 Next.js 15 中，此标志仍然可以访问，但我们正在更改默认行为，使 Page 段的 `staleTime` 为 `0`。这意味着当您在使用应用内导航时，客户端始终会反映作为导航的一部分变为激活状态的 Page 组件（们）中的最新数据。然而，仍有一些重要行为保持不变：

*   共享布局数据不将重新从服务器获取，继续支持 [部分渲染](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
*   后退/前进导航仍将从缓存中恢复，以确保浏览器可以恢复滚动位置。
*   [Loading.js](https://nextjs.org/docs/app/api-reference/file-conventions/loading) 将保留缓存 5 分钟（或 `staleTimes.static` 配置的值）。

您可以通过设置以下配置选择以前的客户端路由器缓存行为：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
    },
  },
};
 
module.exports = nextConfig;
```

部分预渲染（实验性）的增量采用[](#incremental-adoption-of-partial-prerendering-experimental)
-------------------------------------------------------------------------------------------------------------------------
在 Next.js 14 中，我们 [引入了部分预渲染 (PPR)](/blog/next-14#partial-prerendering-preview)——一种将静态渲染和动态渲染结合在同一页面中的优化方法。

Next.js 当前默认使用静态渲染，除非您使用 [动态函数](https://nextjs.org/docs/app/building-your-application/rendering/server-components#server-rendering-strategies) 例如 `cookies()`、`headers()` 和未缓存的数据请求。这些 API 将整个路由选择为动态渲染。使用 PPR，您可以将任何动态 UI 包装在 Suspense 边界中。当新请求进来时，Next.js 将立即提供静态 HTML 外壳，然后在同一个 HTTP 请求中渲染并流式传输动态部分。

为了允许增量采用，我们添加了一个 `experimental_ppr` 路由配置选项，用于将特定 Layout 和 Page 选择为 PPR：

app/page.jsx

```
import { Suspense } from "react"
import { StaticComponent, DynamicComponent } from "@/app/ui"
 
export const experimental_ppr = true
 
export default function Page() {
  return {
     <>
	     <StaticComponent />
	     <Suspense fallback={...}>
		     <DynamicComponent />
	     </Suspense>
     </>
  };
}
```

要使用此新选项，您需要在 `next.config.js` 文件中将 `experimental.ppr` 配置设置为 `'incremental'`：

next.config.ts

```
const nextConfig = {
  experimental: {
    ppr: 'incremental',
  },
};
 
module.exports = nextConfig;
```

一旦所有段都启用了 PPR，您设置 `ppr` 值为 `true`，并为整个应用和所有未来路由启用它，将被视为安全的。

我们将在 Next.js 15 GA 博客文章中分享更多关于我们 PPR 路线图的信息。

了解更多关于 [部分预渲染](https://nextjs.org/docs/app/building-your-application/rendering/partial-prerendering) 的信息。

使用 `next/after`（实验性）在响应之后执行代码[](#executing-code-after-a-response-with-nextafter-experimental)
--------------------------------------------------------------------------------------------------------------------------------
在处理用户请求时，服务器通常会执行与计算响应直接相关的任务。但是，您可能需要执行记录日志、分析和其他外部系统同步等任务。

由于这些任务与响应不直接相关，用户不应等待它们完成。在响应用户后延迟工作是一个挑战，因为无服务器函数在响应关闭后立即停止计算。

`after()` 是一个新的实验性 API，通过允许您在响应流完成后安排工作来处理，使次要任务可以在不阻塞主响应的情况下运行。

要使用它，请在 `next.config.js` 中添加 `experimental.after`：

next.config.ts

```
const nextConfig = {
  experimental: {
    after: true,
  },
};
 
module.exports = nextConfig;
```

然后，在服务器组件、服务器操作、路由处理器或中间件中导入该函数。

```
import { unstable_after as after } from 'next/server';
import { log } from '@/app/utils';
 
export default function Layout({ children }) {
  // Secondary task
  after(() => {
    log();
  });
 
  // Primary task
  return <>{children}</>;
}
```

了解更多关于 [`next/after`](https://nextjs.org/docs/app/api-reference/functions/unstable_after) 的信息。

优化外部包的打包（稳定）[](#optimizing-bundling-of-external-packages-stable)
-----------------------------------------------------------------------------------------------------
打包外部包可以提高应用程序的冷启动性能。在 **App Router** 中，外部包默认会被打包，您可以使用新的 [`serverExternalPackages`](https://nextjs.org/docs/app/api-reference/next-config-js/serverExternalPackages) 配置选项选择退出特定的包。

在 **Pages Router** 中，外部包默认不会被打包，但您可以使用现有的 [`transpilePackages`](https://nextjs.org/docs/pages/api-reference/next-config-js/transpilePackages) 选项提供要打包的包列表。使用此配置选项时，您需要指定每个包。

为了统一 App 和 Pages Router 之间的配置，我们引入了一项新选项 [`bundlePagesRouterDependencies`](https://nextjs.org/docs/pages/api-reference/next-config-js/bundlePagesRouterDependencies) 以匹配 App Router 的默认自动打包。如果您需要，您可以使用 [`serverExternalPackages`](https://nextjs.org/docs/app/api-reference/next-config-js/serverExternalPackages) 来选择退出特定的包。

next.config.ts

```
const nextConfig = {
  // Automatically bundle external packages in the Pages Router:
  bundlePagesRouterDependencies: true,
  // Opt specific packages out of bundling for both App and Pages Router:
  serverExternalPackages: ['package-name'],
};
 
module.exports = nextConfig;
```

了解更多关于 [优化外部包](https://nextjs.org/docs/app/building-your-application/optimizing/package-bundling) 的信息。
