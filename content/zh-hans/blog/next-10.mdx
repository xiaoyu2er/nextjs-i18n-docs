Built-in Image Component and Automatic Image Optimization[](#built-in-image-component-and-automatic-image-optimization)
-----------------------------------------------------------------------------------------------------------------------

我们的 Next.js 目标是为改善两件事：开发者体验 (Developer Experience) 和用户体验 (User Experience)。

今年我们已经在开发者体验和为所有 Next.js 应用程序提高性能方面投入了大量精力。我们专注于减少浏览器必须加载的 JavaScript 量。

我们推出了超过 **20 项新功能**，以提高性能和开发者体验。同时，Next.js 核心的 JavaScript 大小**减少了 16%**。

今年一月，我们与 Google Chrome 团队合作推出了一种新的业界最佳 JavaScript 代码拆分策略。

例如，Barnebys **减少了 23% 的应用程序大小**，Sumup 的 **最大 JavaScript 包大小减少了 70%**。这些改进是在**不更改任何代码**的情况下在其 Next.js 应用程序中取得的。

公司只需将 Next.js 升级到最新版本，即可自动采用这种新策略。

### Images on the Web[](#images-on-the-web)

虽然我们在减少浏览器必须加载的 JavaScript 量方面的努力已经初见成效，但网络不仅仅是 JavaScript：它还包括标记和图像。

图像占网页总字节的 50%。

图像对最大内容绘制 (Largest Contentful Paint) 有很大影响，因为当页面加载时，它们通常是最显眼的元素。最大内容绘制是一项核心网页指标 (Core Web Vital)，Google 很快将在其搜索排名中使用该指标 [很快](https://webmasters.googleblog.com/2020/05/evaluating-page-experience.html)。

一半以上的图像大小超过 1 兆字节，这意味着它们没有经过优化以适应显示在网络上。

如今用户使用手机、平板电脑和笔记本电脑浏览网络，但图像仍然是一样的。例如：网站加载一张 2000 乘 2000 像素的图像，但手机仅显示 100 乘 100 像素。

此外，网页上 30% 的图像位于初始视口之外，这意味着浏览器加载用户直到滚动到页面更下方才看到的图像。

图像通常没有 `width` 和 `height` 属性，导致页面加载时它们会跳动。这损害了累积布局偏移 (Cumulative Layout Shift) 核心网页指标。

网站上 99.7% 的图像不使用 WebP 等现代图像格式。

为了以高性能的方式在网页上使用图像，必须考虑很多方面：大小、重量、懒加载和现代图像格式。

开发者必须设置复杂的构建工具来优化图像，但这些工具通常不涵盖来自外部数据源的用户上传的图像，使得无法优化所有图像。

这个不可能的开发任务不可避免地导致令人沮丧的最终用户体验。

### Next.js Image Component[](#nextjs-image-component)

我们很兴奋地宣布我们在网络高性能图像的解决方案：Next.js 图像组件和自动图像优化。

在最基本的层面上，Next.js 图像组件只是 HTML `<img>` 元素的即用型替代品，为现代网络演化而来。

```
<img
  src="/profile-picture.jpg"
  width="400"
  height="400"
  alt="Profile Picture"
/>
```

```
import Image from 'next/image';
<Image
  src="/profile-picture.jpg"
  width="400"
  height="400"
  alt="Profile Picture"
/>;
```

Google Chrome 团队帮助创建了这个 React 组件，将最佳实践作为默认值来提高页面性能。

使用 `next/image` 组件时，图像会自动懒加载，这意味着当用户接近看到图像时它们才会渲染。这防止加载初始视口之外的那 30% 的图像。

图像尺寸被强制执行，允许浏览器立即渲染图像所需的空间，而不是在加载时跳动，防止布局偏移。

虽然 HTML `<img>` 元素上的 `width` 和 `height` 可能会导致响应式布局问题，但**使用 `next/image` 时并非如此**。使用 `next/image` 时，图像会根据提供的 `width` 和 `height` 的比例自动变为响应式。

开发者可以标记初始视口中的图像，允许 Next.js 自动预加载这些图像。在初始视口中预加载图像已显示可将最大内容绘制 (Largest Contentful Paint) 提高多达 50%。

### Automatic Image Optimization[](#automatic-image-optimization)

即使与 HTML `<img>` 元素相比有这些改进，仍然存在一个主要问题。2000 乘 2000 像素的图像仍被发送到渲染较小图像的手机上。

通过 Next.js 10，我们也在解决这个问题。`next/image` 组件将通过内置的图像优化自动生成更小的尺寸。

内置图像优化会自动以现代图像格式（如 WebP）提供图像，这比 JPEG 小约 30%，如果浏览器支持的话。它还允许 Next.js 自动采用**未来**的图像格式，并为支持这些格式的浏览器提供服务。

图像优化适用于任何图像源。即使图像来自外部数据源（如内容管理系统），它们也会被优化。

Next.js 10 按需优化图像，即在用户请求时优化，而不是在构建时优化。与静态站点生成器和仅静态解决方案不同，无论发送 10 张图像还是 1000 万张图像，您的构建时间都不会增加。

### Conclusion[](#conclusion)

新的 `next/image` 组件和自动图像优化是强大的新原语，将极大地改善用户体验。

`next/image` 组件处理自动懒加载、关键图像预加载、跨设备的正确大小支持，并自动支持现代格式。这些功能适用于任何来源的图像。

我们很期待看到您的用户在使用这些新原语后体验变得多快。

有关更多详细信息，请查看 [Next.js 图像组件和自动图像优化文档](/docs/pages/building-your-application/optimizing/images)。

Internationalized Routing[](#internationalized-routing)
-------------------------------------------------------

今年，一些企业和社区成员帮助我们团队了解到国际化有多么重要。

例如，我们了解到 72% 的消费者如果网站已翻译更有可能留在您的网站上，55% 的消费者表示他们只以母语从电子商务网站购买。

如果您计划在不同国家进入市场，国际化您的项目对于成功至关重要。

国际化项目主要有两个支柱：翻译和路由。

许多 React 库使应用程序准备翻译，但它们中的大多数期望您手动处理路由，通常仅适用于一种渲染策略。

这就是为什么，作为 Next.js 10 的一部分，我们发布了对国际化路由和语言检测的内置支持。

这种对国际化路由的内置支持支持 Next.js 的混合策略，因此您可以针对每个页面选择静态生成 (Static Generation) 或服务端渲染 (Server-Rendering)。

Next.js 10 支持两种最常见的路由策略：子路径路由 (subpath routing) 和域名路由 (domain routing)。

对于这两种策略，您首先需要在 Next.js 配置中配置本地化。

next.config.js

```
module.exports = {
  i18n: {
    locales: ['en', 'nl'],
    defaultLocale: 'en',
  },
};
```

语言环境需采用 [UTS 语言标识符](https://www.unicode.org/reports/tr35/tr35-59/tr35.html#Identifiers) 这一标准化格式。

通常语言标识符由语言代码、地区代码和文字代码组成，格式为 `语言-地区-文字`。其中地区代码和文字代码为可选项。例如：

*   `en-US` - 美国英语
*   `nl-NL` - 荷兰语
*   `nl` - 荷兰语，无特定地区

一旦配置好本地化，您就可以选择子路径或域名路由。

### Subpath routing[](#subpath-routing)

子路径路由将本地化放入 url。这允许所有语言驻留在单个域上。

例如，您可以在 url 中插入本地化，如 `/nl-nl/blog` 和 `/en/blog`。

### Domain routing[](#domain-routing)

域名路由使您能够将本地化映射到顶级域。例如 [`example.nl`](http://example.nl) 可以映射到 `nl` 本地化，[`example.com`](http://example.com) 可以映射到 `en` 本地化。

域名路由需要一些额外的配置来了解如何路由域：

next.config.js

```
module.exports = {
  i18n: {
    locales: ['en', 'nl'],
    domains: [
      {
        domain: 'example.com',
        defaultLocale: 'en',
      },
      {
        domain: 'example.nl',
        defaultLocale: 'nl',
      },
    ],
  },
};
```

### 语言检测 [](#language-detection)

Next.js 10 具有 / 路由的内置语言检测，基于 `Accept-Language` 标头，所有现代浏览器都支持。配置的本地化将与 `Accept-Language` 标头匹配，然后根据配置的策略重定向。

### 搜索引擎优化 [](#search-engine-optimization)

由于 Next.js 知道用户访问页面的语言，它将自动向 `<html>` 标签添加 lang 属性。

Next.js 不知道页面的变体，因此您需要使用 `next/head` 添加 `hreflang` 元标签。您可以从 [Google Webmasters 文档](https://support.google.com/webmasters/answer/189077) 了解更多关于 `hreflang` 的信息。

### Next.js 中国化未来的[](#the-future-of-internationalization-in-nextjs)

国际化路由是一系列功能中的第一个，将使国际化和本地化您的项目更容易。国际化路由允许集成大多数 React 国际化库。

有关国际化路由的更多信息，请查看 [国际化路由文档](/docs/pages/building-your-application/routing/internationalization)。

Next.js Speed Insights[](#nextjs-speed-insights)
------------------------------------------------

在 Vercel，我们知道您无法修复无法衡量的事物。

您的访客对网站性能的容忍度越来越高。超过 50% 的访客如果加载时间超过 3 秒，将放弃您的网站 [如果它需要超过 3 秒才能加载](https://www.blog.google/products/admanager/the-need-for-mobile-speed/)。如果您从事电子商务，许多人发现将加载时间提高 1/10 秒会导致 [转化率提高 1%](http://robotics.stanford.edu/~ronnyk/2007IEEEComputerOnlineExperiments.pdf)。

由于性能对于您的成功至关重要，我们自豪地发布 Next.js Speed Insights。用于跟踪现实世界性能指标并将这些洞察反馈到您的开发工作流中的解决方案。

通过 Next.js Speed Insights：

您现在将**持续衡量**，而不是只衡量一次。

在您的开发设备上进行衡量，**测量将来自您的访客使用的实际设备**。

**Next.js Speed Insights 关注整个画面，深入了解您的受众以及您的应用程序对您的用户表现如何**。

我们非常坚持收集真实数据，因为性能不佳的原因并不总是最明显的。性能回归可能源于各种地方——第三方脚本和样式表，或过大或缓慢的第一方字体、图像和视频。

### Core Web Vitals[](#core-web-vitals)

Google 与 [Web 性能工作组](https://www.w3.org/webperf/) 一起确立了准确衡量用户如何体验您网站性能的一组指标：恰如其名，Web 指标 (Web Vitals)。Web 指标 (Web Vitals) 是追踪您网站 perceived loading speed（感知加载速度）、响应能力和视觉稳定性的一组指标——这三者对于网站的总体健康都至关重要！

感知加载速度可以通过最大内容绘制 (Largest Contentful Paint) 来衡量，即所有页面内容已显示时。例如，当我打开一个链接购买一双运动鞋——我看到运动鞋、价格和购物车按钮之前经过的时间是 LCP。

页面响应能力可以通过首次输入延迟 (First Input Delay) 来衡量，它测量您的用户必须等待才能看到他们对页面的第一次交互反应的时间。例如，我点击添加到购物车和购物车中物品数量增加之间的时间是 FID。

最后，视觉稳定性可以通过累积布局偏移 (Cumulative Layout Shift) 来衡量，即元素显示给用户后移动了多少。例如，我们都经历过试图点击一个因为图像加载晚而移动的按钮的挫折——那就是布局偏移。

对您的真实用户进行这些 Web 指标 (Web Vitals) 的持续衡量和符合至关重要。这是了解您的网站对您的用户表现如何的唯一真正方式。您的网站性能会根据用户的设备及其网络条件或他们与页面的交互方式而截然不同。加载个性化内容或广告的网站也可能在用户之间表现出截然不同的性能。

**模拟测试无法捕捉这些重要信号**。

Next.js Speed Insights 允许您捕捉现实世界的洞察，而不是合成基准测试。它实现连续的测量流，而不是依赖偶尔的测试，确保它成为您开发者工作流的一部分。

Next.js Speed Insights 允许您捕捉**现实世界的洞察**，而不是合成基准测试。它实现**连续的流**的测量，而不是依赖偶尔的测试，**确保它成为您开发者工作流的一部分**。

访问 [nextjs.org/analytics](https://nextjs.org/analytics) 了解如何在您的应用中立即启用它。

---------------------------------------------------------------------------------------------------

您现在可以在 React 组件内部导入第三方 CSS。这允许对仅适用于单个组件的 CSS 进行代码拆分。例如，您可以现在使用 `react-datepicker` 库而无需在 `_app.js` 中导入 CSS：

components/MyComponent.js

```
import DatePicker from 'react-datepicker';
import 'react-datepicker/dist/react-datepicker.css';
 
function MyComponent() {}
```

您可以参考 [内置 CSS 支持文档](/docs/pages/building-your-application/styling) 了解 Next.js 如何处理 CSS 导入。

Automatic Resolving of `href`[](#automatic-resolving-of-href)
-------------------------------------------------------------

如果您之前使用过动态路由，您可能遇到过必须同时向 `next/link` 提供 `href` 和 `as` 属性的情况。它看起来像这样：

```
<Link href="/categories/[slug]" as="/categories/books" />
```

这允许 Next.js 插值动态参数的 href，但是，当开发者忘记添加 `as` 或在 `href` 中添加 `as` 时，会导致页面过渡不使用客户端路由从而造成摩擦。

我们几个月前着手解决开发者这一摩擦，主要目的是提高开发者体验和最终用户体验。我们逐步朝着允许 `href` 自动解析的解决方案努力。

我们很兴奋宣布，作为 Next.js 10 的一部分，在大多数用例中您不再需要使用 `as` 属性。消除开发者摩擦并提高最终用户体验。

此更改完全向后兼容，如果您当前使用 `href` 和 `as`，现有行为将保留。

为了采用自动 `href` 解析，您只需将您的 `next/link` 用法更改为仅使用 `href`，保留您以前在 `as` 属性中的值。

```
<Link href="/categories/books" />
```

有关 `next/link` 和客户端路由的更多信息，您可以参考 [`next/link` 文档](/docs/pages/api-reference/components/link)。

`@next/codemod` CLI[](#nextcodemod-cli)
---------------------------------------

我们致力于确保 Next.js 升级尽可能顺畅，通过广泛的向后兼容性。这种承诺从一开始就很少弃用功能，同时用新的、更好的解决方案取而代之。除此之外，我们对所有 Next.js 功能都有广泛的集成测试，包括测试复制本地开发。

当 Next.js 中的某个功能被弃用并需要大型代码库更改时，我们的团队会为它创建一个代码转换 (codemod)。代码转换 (codemod) 是一种自动化代码转换，您可以运行它来更新您的项目源代码。

例如：我们发布了一个代码转换来更新箭头函数和匿名函数为命名函数。[此转换](/docs/pages/building-your-application/upgrading/codemods#name-default-component) 是必需的，否则 React Fast Refresh 无法将函数检测为有效的 React 组件。同样，React hooks eslint 规则也无法将函数识别为 React 组件。

通过 Next.js 10，我们正在发布一个新的 Next.js 代码转换 (codemods) 命令行工具，允许您运行单个命令来更新您的应用程序：`npx @next/codemod <transform> <path>`。

有关代码转换 (codemods) 的更多信息，您可以查看 [Next.js 代码转换文档](/docs/pages/building-your-application/upgrading/codemods)。

Blocking Fallback for `getStaticPaths`[](#blocking-fallback-for-getstaticpaths)
-------------------------------------------------------------------------------

在 Next.js 9.3 中，我们引入了 `getStaticProps` 和 `getStaticPaths`，以及能够在 `getStaticPaths` 中返回 `fallback` 属性的能力。`fallback` 属性允许在不进行完整构建的情况下生成额外的静态页面，最初提供静态 HTML 文件，然后在后续请求中被完全渲染的内容替换。过去的几个月，我们收到了很多来自公司的反馈，他们想要类似但略有不同的行为：用户在第一次请求页面时阻塞预渲染。在那次初始渲染之后，页面将在后续请求中被重新使用。

通过 Next.js 10，我们正在解决这个问题。

我们很高兴宣布为 `getStaticPaths` 引入新的 `fallback: 'blocking'` 模式，启用阻塞行为，不向浏览器发送静态回退。而是等待初始请求进行预渲染。

`pages/posts/[id].js`

```
export function getStaticPaths() {
  return {
    // enables blocking mode for the fallback behavior
    fallback: 'blocking',
  };
}
```

有关增量生成额外静态页面的 `fallback` 行为的更多信息，您可以参考 [`fallback` 文档](/docs/pages/building-your-application/data-fetching#fallback-true)。

Redirect and notFound Support for `getStaticProps` / `getServerSideProps`[](#redirect-and-notfound-support-for-getstaticprops--getserversideprops)

自引入 `getStaticProps` 和 `getServerSideProps` 以来，我们注意到用户需要返回重定向和 404 响应的情况。为了帮助简化这些情况，我们现在允许从 `getStaticProps` 和 `getServerSideProps` 返回两个新字段：`notFound` 和 `redirect`。

### `notFound` Support[](#notfound-support)

当返回带有 `true` 的 `notFound` 字段时，将返回状态代码为 `404` 的默认 404 页面。这允许您避免生成额外的使用 SSG 的页面，并手动处理渲染 404 页面。

`pages/posts/[id].js`

```
export function getStaticProps() {
  return {
    // returns the default 404 page with a status code of 404
    notFound: true,
  };
}
```

### `redirect` Support[](#redirect-support)

现在可以返回重定向，包含 `destination` 以及重定向是否永久，例如 `permanent: true`。可以选择返回 `statusCode` 代替 `permanent` 字段，用于需要特定状态而非默认值的情况。

`pages/posts/[id].js`

```
export function getStaticProps() {
  return {
    // returns a redirect to an internal page `/another-page`
    redirect: {
      destination: '/another-page',
      permanent: false,
    },
  };
}
```

`pages/posts/[id].js`

```
export function getServerSideProps() {
  return {
    // returns a redirect to an external domain `example.com`
    redirect: {
      destination: 'https://example.com',
      permanent: false,
    },
  };
}
```
