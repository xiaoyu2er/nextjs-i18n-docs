[Next.js 9](/blog/next-9) 是在六个月前发布的，随后 [Next.js 9.1](/blog/next-9-1) 三个月前发布。

这两次发布为 Next.js 添加了非常有用的新功能，且没有增加我们基线客户端运行时的体积。

此后，我们专注于整体框架的精细化和改进：[9.1.1](https://github.com/vercel/next.js/releases/tag/v9.1.1), [9.1.2](https://github.com/vercel/next.js/releases/tag/v9.1.2), [9.1.3](https://github.com/vercel/next.js/releases/tag/v9.1.3), [9.1.4](https://github.com/vercel/next.js/releases/tag/v9.1.4), [9.1.5](https://github.com/vercel/next.js/releases/tag/v9.1.5), [9.1.6](https://github.com/vercel/next.js/releases/tag/v9.1.6), 以及 [9.1.7](https://github.com/vercel/next.js/releases/tag/v9.1.7)。

让我们深入了解一下这些版本所做的改进！

*   **[3% - 8%+ 更小的客户端 JavaScript 体积](#3--8-smaller-client-side-javascript-size)**：我们进一步优化了应用程序输出，从 hello world 应用程序中减少了 7.5kB。更复杂的应用程序可减少 8% 或更多。
*   **[重新设计的产品构建 CLI 输出](#redesigned-production-build-cli-output)**：产品构建输出现在以更易懂的格式显示 gzipped 文件体积。
*   **[内置的新 Polyfills：fetch(), URL, and Object.assign](#new-built-in-polyfills-fetch-url-and-objectassign)**：应用程序可以在旧版浏览器中利用 `fetch()` API、`URL` 和 `Object.assign`，而无需担心兼容性问题。
*   **[优化的页面加载：更好的 FCP 和 TTI](#optimized-page-loading-better-fcp-and-tti)**：我们与 Google Chrome 团队紧密合作，以最大化页面加载性能。这为用户带来了更好的最终用户体验。
*   **[支持最新的 JavaScript 功能](#support-for-the-latest-javascript-features)**：我们致力于确保您始终可以使用最新的 JavaScript 功能，包括 [可选链 (Optional Chaining)](https://github.com/tc39/proposal-optional-chaining)、[空值合并 (Nullish Coalescing)](https://github.com/tc39/proposal-nullish-coalescing) 以及其他稳定的 ES2020 功能。
*   **[支持 `next export` 应用程序的零配置部署](#zero-config-deployment-support-for-next-export-applications)**：`next export` 支持的应用程序现在可以零配置部署到 [Vercel](https://vercel.com)。
*   **[React Strict Mode 合规性和可选启用](#react-strict-mode-compliance-and-opt-in)**：Next.js 的所有客户端端运行时现在都符合 React 的 Strict Mode 标准。我们还添加了一个配置选项，以在整个应用程序中启用此模式。
*   **[针对 nightly React 构建的自动测试](#automated-testing-against-nightly-react-builds)**：Next.js 会自动针对 [React 的 next 通道](https://reactjs.org/blog/2019/10/22/react-release-channels.html) 进行测试，确保与未来版本的兼容性。

All of these benefits are non-breaking and fully backwards compatible. All you need to do to update is run:

终端

```
npm i next@latest react@latest react-dom@latest
```

### 3% - 8% 以上的客户端 JavaScript 体积减小[](#3---8-smaller-client-side-javascript-size)

与 Google Chrome 团队合作，所有 Next.js 应用程序都将受益于 **7.5kB 或更多** 的体积减少。

基础应用程序应看到 3-4% 的应用程序体积减少，而更高级的应用程序可能会看到 6-8%（或更多）的减少！

|  | 9.0.x | 9.1.x | delta |
| --- | --- | --- | --- |
| Basic Application | 68.9kB | 66.1kB | **4.1%** _smaller_ |

这些节省部分归功于 [替换 `url` 包的客户端版本](https://github.com/GoogleChromeLabs/native-url)，使用 [URL API](https://developer.mozilla.org/docs/Web/API/URL) 构建的替代品。

更多的体积减少是通过提供内置的、常用的包的 polyfills 来实现的。您可以 [在此处阅读有关这些 polyfills 的更多信息](#new-built-in-polyfills-fetch-url-and-objectassign)。

最后，我们优化了我们的 JSX 输出，这将根据您应用程序中 JSX 的数量引入相应的节省。

### 重新设计的产品构建 CLI 输出[](#redesigned-production-build-cli-output)

CLI 的产品构建输出为了清晰度而重新设计。因为 Next.js 是一个混合应用框架，每个页面可能具有不同的特性。

新的输出将每个页面分类为以下之一：

*   **服务端渲染 (Server)**：页面在运行时服务端渲染，这意味着它使用 `getInitialProps` 或 [`getServerProps (proposal)`](https://github.com/vercel/next.js/issues/9524)
*   **自动静态优化 (Static)**：页面在构建时渲染为静态 HTML，并将作为静态文件提供（不使用初始属性）
*   **使用计算数据静态生成 (SSG)**：页面在构建时生成为静态 HTML/JSON，并将作为静态文件提供（使用 [`getStaticProps (proposal)`](https://github.com/vercel/next.js/issues/9524)）

此外，新的输出还显示每个页面的 Gzipped 体积——这些体积不包括所有页面共享的文件（另行显示）。

每个页面的体积将根据其感知用户体验进行着色：

*   **小于 130kB**：_绿色_ —— 您的应用程序在大多数网络和设备条件下应表现良好。
*   **在 130kB 和 170kB 之间**：_黄色_ —— 您的应用程序在 [全球基准设备 + 网络条件](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 下接近 5-6 秒的加载时间。
*   **大于 170kB**：_红色_ —— 您的应用程序在 [相同的条件下](https://infrequently.org/2017/10/can-you-afford-it-real-world-web-performance-budgets/) 可能需要超过 6 秒才能加载。

### 内置的新 Polyfills：fetch(), URL, and Object.assign[](#new-built-in-polyfills-fetch-url-and-objectassign)

在检查了许多用户的应用程序和我们提供的示例后，我们发现大多数打包中带有类似的 polyfills 集。在某些情况下，应用程序甚至对同一功能拥有重复的 polyfills。

为了解决这个问题，我们与 Google Chrome 团队合作，为我们观察到的最常见的三个 API 构建了 polyfills。

通过使用差分加载，这些 polyfills **在全球 83% 的网络流量中不会被加载**。这意味着您的大多数用户将不会下载与这些 polyfills 相关的字节——它们仅会在需要时下载。

此外，我们现在内置的任何知名 polyfills 都将从您的产品构建中完全消除。这意味着您不必为您的依赖项无意中导入其中一个 API 的 polyfills 而付出代价。

内置 API 列表及其变得过时的 polyfills 如下：

*   [**fetch()**](https://developer.mozilla.org/docs/Web/API/Fetch_API) —— 替代：`whatwg-fetch` 和 `unfetch`。
*   [**URL**](https://developer.mozilla.org/docs/Web/API/URL) —— 替代：[`url` 包 (Node.js API)](https://nodejs.org/api/url.html)。
*   [**Object.assign()**](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Object/assign) —— 替代：`object-assign`, `object.assign`, 和 `core-js/object/assign`。

如果您在服务器上使用了 `fetch()`，您仍然需要导入 `isomorphic-fetch` 或 `isomorphic-unfetch`。

这个更改是完全非破坏性的，所有的 polyfills 都使用了可用版本中最符合规范的方式构建。结果是在现代浏览器中，您的产品构建中消除了多达 **18kB 的 JavaScript**。

### 优化的页面加载：更好的 FCP 和 TTI[](#optimized-page-loading-better-fcp-and-tti)

Next.js 优化了其预加载实现以减少 FCP 和整体 TTI。

通过绕过一个浏览器 bug，CSS（在使用时）现在正确优先于 JavaScript。这带来了更快首屏内容绘制 (FCP)，从而为用户带来视觉上更快的完成网站。

此外，我们优化了页面预取以使用较低优先级的网络请求。

此外，这些请求仅在浏览器空闲时间发生，从而实现了更快的可交互时间。这是因为浏览器将优先确保您的应用程序可交互，而不是乐观预取。

### 支持最新的 JavaScript 功能[](#support-for-the-latest-javascript-features)

Next.js 拥有一套先进的、高度优化的编译流水线，允许您使用最新的 JavaScript 功能。我们最近引入的优化直接促成了应用程序体积 **3-8% 的减少** (#3--8-smaller-client-side-javascript-size)。

这些 JavaScript 功能可以在无需担心浏览器兼容性的情况下利用，因为我们自动编译您的代码以支持所有浏览器（排除已停产的版本）。这包括 ES6+ 功能，如 [async/await (ES2017)](https://github.com/tc39/ecmascript-asyncawait), [对象剩余/展开属性 (Object Rest/Spread Properties (ES2018))](https://github.com/tc39/proposal-object-rest-spread), [动态 `import()` (ES2020)](https://v8.dev/features/dynamic-import) 以及更多！

在此次发布中，我们很高兴宣布支持 [可选链 (Optional Chaining Stage 4)](https://github.com/tc39/proposal-optional-chaining) 和 [空值合并 (Nullish Coalescing Stage 4)](https://github.com/tc39/proposal-nullish-coalescing)。

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
function Page(props) {
  return (
    <p>{props?.deeply?.nested?.value}</p>
    /* ⬆ If deeply.nested.value is not available it won't render */
  );
}
 
export default Page;
```

### 支持 `next export` 应用程序的零配置部署[](#zero-config-deployment-support-for-next-export-applications)

Next.js 的 `next export` 命令现在可以开箱即用与 Vercel 的 [零配置](https://vercel.com/blog/zero-config)。在此更改之前，使用 `next export` 的用户必须创建自定义 `now.json`。

在 [Vercel](https://vercel.com) 上利用 Next.js 的导出模式与在 `package.json` 中拥有以下 `build` 脚本一样简单：

package.json

```
{
  "scripts": {
    "build": "next build && next export"
  }
}
```

接下来，您只需**一个命令**即可将 Next.js 应用程序部署到 [Vercel](https://vercel.com)。如果您尚未安装 [Vercel](https://vercel.com)，可以通过安装 [Vercel CLI](https://vercel.com/cli) 进行安装。

终端

```
now
```

### React Strict Mode 合规性和可选启用[](#react-strict-mode-compliance-and-opt-in)

完整的 Next.js 运行时现在符合 Strict Mode 标准。这包括对 Next.js' head manager (`<Head>`)、`next/dynamic` 和其他核心功能的更新。这意味着运行时现在利用 Hooks 消除了已弃用的生命周期，并开始使用 React 的新的 Context API。

我们还为您添加了一个方便的可选启用选项，以启用您整个应用程序的 Strict Mode。

为此，请在您的 `next.config.js` 中配置以下选项：

next.config.js

```
module.exports = {
  reactStrictMode: true,
};
```

如果您或您的团队尚未准备好在整个应用程序中使用 Strict Mode，这也没关系！您可以逐步迁移页面的基础上 [使用 `<React.StrictMode>`](https://reactjs.org/docs/strict-mode.html)。

虽然**不是必需的**，但 Strict Mode 将在未来解锁大量优化。因此，我们建议您尽早研究它！

### 针对 nightly React 构建的自动测试[](#automated-testing-against-nightly-react-builds)

在与 React 核心团队 [密切合作](https://github.com/vercel/next.js/pull/9165) 的情况下，我们现在 [每 12 小时测试](https://github.com/vercel/next.js/pull/9151) Next.js 针对 React 的 next 通道。

这些测试有助于确保我们为 React 的未来版本做好了准备并保持兼容。兼容性是 Next.js 非常重视的，我们致力于 Next.js 的长期 API 稳定性。

React 核心团队已在 [此处记录了该过程](https://reactjs.org/blog/2019/10/22/react-release-channels.html)，希望 React 生态系统中的其他项目也能效仿。
