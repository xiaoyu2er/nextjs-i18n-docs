稳定的增量静态再生(#stable-incremental-static-regeneration)
---------------------------------------------------------------------------------

Next.js 在 9.3 版本中引入了 [静态站点生成 (Static Site Generation) 方法](https://nextjs.org/blog/next-9-3#next-gen-static-site-generation-ssg-support)，其目标明确：我们应该获得 [静态的优势](https://rauchg.com/2020/2019-in-review#static-is-the-new-dynamic)（始终快速，始终在线，[全球复制](https://rauchg.com/2020/static-hoisting#hoist-to-the-edge)），同时拥有 Next.js 赖以闻名的动态数据出色支持。

为了兼顾两者的优点，Next.js 引入了 **增量静态生成 (Incremental Static Generation)**，在您构建完网站后即可更新静态内容。通过使用 [`getStaticPaths`](/docs/api-reference/data-fetching/get-static-paths#fallback-false) 中的 [`fallback: true` 选项]，您可以在 _运行时 (runtime)_ **注册 (register)** **新的静态页面 (new static pages)*。

无论数据集有多大，Next.js 都可以通过这种方式按需静态预渲染无限数量的页面。

今天，我们宣布 **增量静态再生 (Incremental Static Re-generation)** 正式 **发布**，这是一种 **用于更新 _现有 (existing)* 页面**的机制，通过在后台重新渲染页面，在流量进入时进行处理。

Inspired by [stale-while-revalidate](https://tools.ietf.org/html/rfc5861)，后台再生确保流量始终不间断地从静态存储中服务，只有在 _完成生成 (after it's done generating)_ 后才会推送新构建的页面。

```
export async function getStaticProps() {
  return {
    props: await getDataFromCMS(),
    // we will attempt to re-generate the page:
    // - when a request comes in
    // - at most once every second
    revalidate: 1,
  };
}
```

> revalidate 标志表示在最长时间段内的秒数，在此期间最多发生一次生成，以防止 [https://en.wikipedia.org/wiki/Cache\_stampede](https://en.wikipedia.org/wiki/Cache_stampede)。

与传统的 SSR 不同，增量静态再生 (Incremental Static Regeneration) 确保您保留静态的优势：

*   无延迟尖峰。页面始终快速服务。
*   页面永不离线。如果后台页面重新生成失败，旧页面保持不变。
*   低数据库和后端负载。页面重新计算最多一次 `\`concurrently。

这两个增量功能（添加页面和延迟更新它们），以及 [预览模式 (preview mode)](/docs/pages/building-your-application/configuring/preview-mode)，现在已稳定，并得到了 `next start` 和 [Vercel edge 平台 (https://vercel.com)](https://vercel.com) 开箱即用的完全支持。

为了展示这一新功能，我们创建了一个示例，展示如何重新生成一个显示特定问题的各种 GitHub 反应（reactions）数量的静态页面：[https://reactions-demo.vercel.app/](https://reactions-demo.vercel.app/)

![After the first visit following our emoji reaction, a new page generation kicks off in the background. Every single request throughout is served from static cache.](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9-5/reactions-demo.gif)

在我们进行表情符号反应后的第一次访问之后，后台会启动一个新的页面生成。整个过程中的每个请求都是从静态缓存中服务的。

接下来，我们将致力于补充 RFC，以解决另外两个增量静态生成功能：

*   同时重新生成和使多个页面无效（例如您的博客索引和特定博客帖子）
*   通过监听事件重新生成（例如 CMS Webhook），在用户流量之前

更多细节，请查看 [getStaticProps 文档](/docs/pages/building-your-application/data-fetching#getstaticprops-static-generation)。

可自定义的基础路径(#customizable-base-path)
-------------------------------------------------

Next.js 项目并不总是从域根目录服务。有时您可能希望在子路径（例如 `/docs`）下托管您的 Next.js 项目，这样 Next.js 项目仅覆盖该域的该子部分。

虽然到目前为止这已经成为可能，但它需要花费相当多的额外配置。例如，在每个 `<Link>` 中添加前缀，并确保 Next.js 从正确的路径提供 JavaScript 资源包。

为了解决这一痛点，我们引入了一个新的配置选项。`basePath` 使您可以轻松地在域的子路径上托管 Next.js 项目。

要开始使用 `basePath`，您可以将其添加到 `next.config.js` 中：

next.config.js

```
module.exports = {
  basePath: '/docs',
};
```

配置 `basePath` 后，您的项目将自动从提供的路径路由。在这种情况下，`/docs`。

当使用 `next/link` 或 `next/router` 链接到项目中的其他页面时， `basePath` 将被自动添加前缀。这允许您更改 `basePath` 而无需更改项目。

例如，使用 `next/link` 路由到另一页：

```
import Link from 'next/link';
 
export default function HomePage() {
  return (
    <>
      <Link href="/documentation-page">
        <a>Documentation page</a>
      </Link>
    </>
  );
}
```

以这种方式在 `next/link` 使用中会导致以下 HTML 渲染到 Web 浏览器：

```
<a href="/docs/documentation-page">Documentation page</a>
```

更多细节，请查看 [basePath 文档](/docs/pages/api-reference/next-config-js/basePath)。

重写、重定向和标头支持(#support-for-rewrites-redirects-and-headers)
-------------------------------------------------------------------------------------------

### 重写(#rewrites)

在构建 Next.js 项目时，您可能希望将某些路由代理到其他 URL。例如，如果您希望在堆栈中逐步采用 Next.js，则会将存在于 Next.js 项目中的页面路由到那里，然后将所有未匹配到正在迁移的旧项目的页面都路由到那个旧项目。

使用 Next.js 9.5，我们引入了一个新配置选项 `rewrites`，它允许您将传入的请求路径映射到不同的目标路径，包括外部 URL。

例如，您可能希望将某个路由重写为 example.com：

next.config.js

```
module.exports = {
  async rewrites() {
    return [
      { source: '/backend/:path*', destination: 'https://example.com/:path*' },
    ];
  },
};
```

在这种情况下，`/backend` 下的所有路径都将路由到 example.com。

您还可以检查您的 Next.js 项目路由是否匹配，如果没有匹配，则重写回之前的项目。这对于 Next.js 的逐步采用非常有用：

```
module.exports = {
  async rewrites() {
    return [
      // check if Next.js project routes match before we attempt proxying
      {
        source: '/:path*',
        destination: '/:path*',
      },
      {
        source: '/:path*',
        destination: `https://example.com/:path*`,
      },
    ];
  },
};
```

在这种情况下，我们首先匹配所有路径。如果没有任何匹配，我们将代理到 example.com，这将是之前的项目。

要了解有关 `rewrites` 功能的更多信息，请查看 [rewrites 文档](/docs/pages/api-reference/next-config-js/rewrites)。

### 重定向(#redirects)

大多数网站至少需要一些重定向。特别是在更改项目路由结构时。例如，当将 `/blog` 移动到 `/news` 或类似转换时。

以前，在 Next.js 项目中拥有重定向列表需要设置自定义服务器或自定义 `_error` 页面来检查该路由是否设置了重定向。然而，这需要付出代价，即使用静态和无服务器优化无效（因为有了服务器），或者不够便捷。

从 Next.js 9.5 开始，您现在可以在 `next.config.js` 下的 `redirects` 键中创建重定向列表：

next.config.js

```
module.exports = {
  async redirects() {
    return [
      {
        source: '/about',
        destination: '/',
        permanent: true,
      },
    ];
  },
};
```

要了解有关 `redirects` 功能的更多信息，请查看 [redirects 文档](/docs/pages/api-reference/next-config-js/redirects)。

### 标头(#headers)

Next.js 允许您构建同时使用静态生成和服务端渲染 (Server-Side Rendering) 的混合项目。使用服务端渲染，您可以为传入请求设置标头。对于静态页面，直到现在设置标头还是不可能的。

我们现在在 `next.config.js` 中引入了一个 `headers` 属性，它适用于所有 Next.js 路由：

next.config.js

```
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*',
        headers: [
          {
            key: 'Feature-Policy',
            // Disable microphone and geolocation
            value: "microphone 'none'; geolocation 'none'",
          },
        ],
      },
    ];
  },
};
```

`headers` 选项允许您设置常用标头，例如 [`Feature-Policy`](https://developer.mozilla.org/docs/Web/HTTP/Headers/Feature-Policy) 和 [`Content-Security-Policy`](https://developer.mozilla.org/docs/Web/HTTP/CSP)。

要了解有关 `headers` 功能的更多信息，请查看 [headers 文档](/docs/pages/api-reference/next-config-js/headers)。

URL 中可选的尾部斜杠(#optional-trailing-slash-in-urls)
-------------------------------------------------------------------

当 Next.js 在 3 年前推出时，其默认行为是所有带有尾部斜杠的 URL 都始终返回 404 页面。

虽然有效，但一些用户请求能够更改此行为。例如，在将之前一直强制使用尾部斜杠的现有项目迁移到 Next.js 时。

使用 Next.js 9.5，我们在 `next.config.js` 中引入了一个新的选项 `trailingSlash`。

此新选项确保 Next.js 自动处理尾部斜杠行为：

*   自动将带尾部斜杠的 URL 重定向到不带尾部斜杠的 URL，例如：`/about/` 到 `/about`
*   当 `trailingSlash` 设置为 `true` 时，不带尾部斜杠的 URL 将重定向到带尾部斜杠的 URL，例如：`/about` 到 `/about/`
*   确保 `next/link` 自动应用/移除尾部斜杠，以避免不必要的重定向。

next.config.js

```
module.exports = {
  // Force a trailing slash, the default value is no trailing slash (false)
  trailingSlash: true,
};
```

要了解有关 `trailingSlash` 功能的更多信息，请查看 [trailingSlash 文档](/docs/pages/api-reference/next-config-js/trailingSlash)

页面资源包的持久缓存(#persistent-caching-for-page-bundles)
---------------------------------------------------------------------------

在编写 Next.js 页面时，所有脚本资源包、CSS 样式表和 HTML 的创建是完全自动的，并为您抽象化了。如果在 Next.js 9.5 之前检查生成的 `<script>` 标签，您会发现它们的 URL 遵循如下模式：

```
/_next/static/ovgxWYrvKyjnlM15qtz7h/pages/about.js
```

上面的路径段 `ovgxWYrvKyjnlM15qtz7h` 是我们所说的构建 ID。虽然这些文件在边缘和用户的机器上很容易缓存，但在重新构建应用后，构建 ID 会更改，所有缓存都会被破坏。

对于大多数项目来说，这种权衡是可以的，但是，我们希望通过不再使未更改页面的浏览器缓存失效，进一步优化这种行为。

在 Next.js 9.2 中引入的、由 Google Chrome 团队合作开发的改进代码拆分策略，为这些 Next.js 页面资源包生成改进奠定了一些基础。

从 Next.js 9.5 开始，所有页面 JavaScript 资源包都将使用内容哈希，而不是构建 ID。这使得在部署之间未更改的页面可以保留在浏览器和边缘缓存中，而无需再次下载。

相比之下，这些更改后的 URL 模式看起来像这样：

```
/_next/static/chunks/pages/about.qzfS4o5gIEXRME6sTEahL.js
```

代替全局构建 ID，`qzfS4o5gIEXRME6sTEahL` 部分是 `about.js` 资源包的可确定哈希，只要该站点部分的代码不更改，它将保持稳定。此外，现在通过 `Cache-Control: public,max-age=31536000,immutable` 进行长期缓存，该缓存会在重新部署时跨缓存持久化，Next.js 为您自动设置了此缓存。

快速刷新增强(#fast-refresh-enhancements)
-------------------------------------------------------

我们在 Next.js 9.4 中引入了快速刷新 (Fast Refresh)，这是一种新的热重载体验，为您提供对 React 组件编辑的即时反馈。

Next.js 9.5 进一步完善了我们的快速刷新实现，并为您提供成功所需的工具：

*   **易于理解的错误**：所有编译和运行时错误都已更新，仅显示相关信息，包括导致错误代码的代码帧。
*   **保持组件状态的开发时间提示**：Next.js 现在为您提供有用的提示，以确保快速刷新在各种情况下尽可能保留您的组件状态。Next.js 提供的每个提示都是完全可操作的，并附有前后示例！
*   **组件状态重置时的警告**：如果 Next.js 在编辑文件后无法保持组件状态，我们现在将打印详细警告。此警告将帮助您诊断项目为何必须重置组件状态，从而让您能够修复它并充分利用快速刷新。
*   **新文档**：我们添加了广泛的文档，解释了什么是快速刷新，它是如何工作的，以及您可以期待什么！该文档还将通过解释其错误恢复方式教您如何利用快速刷新。
*   **用户代码故障排除指南**：新文档还包括如何在开发中充分利用快速刷新的常见问题故障排除步骤和提示。

生产 React 分析(#production-react-profiling)
---------------------------------------------------------

React 很早就引入了 [Profiler API](https://github.com/reactjs/rfcs/pull/51)，它允许您追踪 React 组件中的性能问题。虽然此功能在开发中自动工作，但它在生产中进行分析时需要使用单独的 ReactDOM 版本。

使用 Next.js 9.5，您现在可以在 `next build` 中使用 `--profile` 标志为 React 启用生产分析：

```
next build --profile
```

之后，您可以像开发中一样使用分析工具。

如需了解更多关于 React 分析的信息，您可以阅读 [React 团队的 React Profiler 文章](https://reactjs.org/blog/2018/09/10/introducing-the-react-profiler.html)。特别感谢 [TODOrTotev](https://github.com/TODOrTotev) 和 [@darshkpatel](https://github.com/darshkpatel) 为此功能的贡献。

可选的通配路由(#optional-catch-all-routes)
-------------------------------------------------------

Next.js 9.2 添加了对通配动态路由的支持，这已被社区广泛采用用于各种用例。通配路由为您提供了灵活性，可以创建由无头 CMS (Headless CMS)、GraphQL API、文件系统等驱动的动态路由结构。

在听取反馈时，我们听到用户希望有更多的灵活性来匹配路由的最顶层。今天，我们很高兴揭晓可选的通配动态路由，用于这些高级场景。

要创建可选的通配路由，您可以使用 `[[...slug]]` 语法创建页面。

例如，`pages/blog/[[...slug]].js` 将匹配 `/blog`，以及其下的任何路由，例如：`/blog/a`、`/blog/a/b/c` 等。

与通配路由一样，`slug` 将作为路径部分的数组在 [路由查询对象](/docs/pages/api-reference/functions/use-router#router-object) 中提供。所以，对于路径 `/blog/foo/bar`，查询对象将是 `{ slug: ['foo', 'bar'] }`。对于路径 `/blog`，查询对象将省略 `slug` 键：`{ }`。

您可以在我们的文档中了解更多关于可选通配路由的信息 ([/docs/pages/building-your-application/routing/dynamic-routes#optional-catch-all-routes](/docs/pages/building-your-application/routing/dynamic-routes#optional-catch-all-routes))。

Webpack 5 支持（beta）(#webpack-5-support-beta)
---------------------------------------------------

Webpack 5 目前处于测试版。它包括一些重大改进：

*   [改进的 Tree-Shaking](https://github.com/webpack/changelog-v5#nested-tree-shaking)：嵌套导出、内部模块和 CommonJS 会被摇树
*   [持久化缓存](https://github.com/webpack/changelog-v5#persistent-caching)：允许重用先前构建的工作
*   [确定的块和模块 ID](https://github.com/webpack/changelog-v5#deterministic-chunk-and-module-ids)：解决了构建之间 webpack 模块 ID 可能更改的情况

我们很兴奋今天宣布 Next.js Webpack 5 的 Beta 版可用性。

要尝试 Webpack 5，您可以在 `package.json` 中使用 [Yarn 分辨率 (Yarn resolutions)](https://classic.yarnpkg.com/en/docs/selective-version-resolutions/)：

package.json

```
{
  "resolutions": {
    "webpack": "^5.0.0-beta.30"
  }
}
```

Webpack 5 Beta 版已在生产中用于 [nextjs.org](https://nextjs.org) 和 [vercel.com](https://vercel.com)。我们鼓励您以渐进的方式尝试，并在 [GitHub](https://github.com/vercel/next.js/issues/13341) 上报告您的发现。

### 编译基础设施改进(#compilation-infrastructure-improvements)

为了支持 Webpack 5，我们重写了大量编译管道，使其更适合 Next.js：

*   Next.js 不再依赖 `webpack-hot-middleware` 和 `webpack-dev-middleware`，我们现在直接使用 Webpack 并针对 Next.js 项目进行优化。这转化为更简单的架构和更快的开发编译。
*   [按需条目 (On-demand-entries)](https://nextjs.org/blog/next-8#improved-on-demand-entries)，这是 Next.js 用于在开发过程中为特定时间访问的页面编译的系统，也已重写，现在通过利用专门针对我们用例的新 Webpack 行为更加可靠。
*   React Fast Refresh 和 Next.js 错误覆盖现在完全兼容 Webpack 5
*   磁盘缓存将在未来的 Beta 版本中启用。

### 向后兼容性(#backwards-compatibility)

我们始终致力于确保 Next.js 与以前的版本向后兼容。

Webpack 4 将继续得到完全支持。我们与 Webpack 团队密切合作，确保从 Webpack 4 到 5 的迁移尽可能平稳。

如果您的 Next.js 项目没有自定义 Webpack 配置，则无需更改项目即可充分利用 Webpack 5。

**重要：**如果项目有 [自定义 Webpack 配置](/docs/pages/api-reference/next-config-js/webpack)，转换到 Webpack 5 可能需要进行一些更改。我们建议留意我们的迁移说明，或者最大限度地减少 webpack 扩展的使用，以实现无缝的未来升级。

### macOS 上改进的文件监控(#improved-file-watching-on-macos)

我们最近发现 Webpack 的一个问题，在 macOS 上，在代码做出几次更改后，文件监控会停止。您必须手动重启项目才能再次查看更新。经过几次更改后，此周期会重复。

此外，我们发现此问题不仅发生在 Next.js 项目中，还发生在构建在 Webpack 之上的所有项目和框架中。

经过几天的调试，我们将根因定位到 Webpack 使用的名为 chokidar 的文件监控实现，它是 Node.js 生态系统中广泛使用的文件监控实现。

我们向 chokidar 发送了一个修复问题的补丁。在补丁发布后，我们与 [Tobias Koppers](https://github.com/sokra) 合作在 [新版本 Webpack](https://github.com/webpack/webpack/releases/tag/v4.44.0) 中推出了此补丁。

当您升级到 Next.js 9.5 时，将自动使用此打补丁的 Webpack 版本。
