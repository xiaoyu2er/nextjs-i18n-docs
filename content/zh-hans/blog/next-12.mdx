正如我们在 [Next.js Conf](https://nextjs.org/conf) 上宣布的那样，Next.js 12 是我们迄今为止最大的发布版本：

* [**Rust 编译器**](#faster-builds-and-fast-refresh-with-rust-compiler): ~3 倍更快的快速刷新 和 ~5 倍更快的构建
* [**中间件 (beta)**](#introducing-middleware): 利用代码而非配置，在 Next.js 中实现完全的灵活性
* [**React 18 支持**](#preparing-for-react-18): 原生 Next.js API 现受支持，并包括 Suspense
* [**`<Image />` AVIF 支持**](#smaller-images-using-avif): 可选的 20% 更小图像
* [**机器人感知的 ISR 回退**](#bot-aware-isr-fallback): 针对网络爬网机器人的 SEO 优化
* [**原生 ES 模块支持**](#es-modules-support-and-url-imports): 与标准化模块系统保持一致
* [**URL 导入 (alpha)**](#url-imports): 从任何 URL 导入包，无需安装
* [**React 服务器组件 (alpha)**](#react-server-components): 今天就试用，包括 SSR 流式传输

今天通过运行 `npm i next@latest` 进行更新。

## 更快的构建与 Rust 编译器的快速刷新
--------------------------------------------------------------------------------

我们致力于加快每个 Next.js 应用程序的生产构建速度，并在本地开发中即时获得反馈。Next.js 12 包含一个全新利用原生编译的 Rust 编译器。

我们的 Rust 编译器是基于 [**SWC**](http://swc.rs/) 构建的，这是一个面向下一代快捷工具平台的开源平台。我们针对打包和编译进行了优化，在本地拥有 **~3 倍更快的刷新**，生产构建拥有 **~5 倍更快的构建速度**。其他改进和功能包括：

![结果来自在大型 Next.js 代码库中使用新的 Rust 编译器。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-12/swc.png)

在大型 Next.js 代码库中使用新的 Rust 编译器的结果。

* 针对大型代码库的进一步速度提升：我们使用世界领先的几个最大规模的 Next.js 代码库验证了 Rust 编译器。
* 增强的性能可观测性：Next.js 现在在控制台输出客户端和服务端编译的快速刷新计时，包括编译的模块数和文件数。
* 底层的 Webpack 改进：我们对 Webpack 进行了多项改进，包括优化快速刷新并提高按需入口的可靠性。

使用 Rust 进行编译比 Babel **快 17 倍**，并在 Next.js 12 中默认启用，替换了 JavaScript 和 TypeScript 文件的转换。这意味着我们将 Next.js 中的 Babel 转换端口到了 Rust，包括用于实现 `styled-jsx` 转换的 **Rust 中的全新 CSS 解析器**。

新的 Rust 编译器向后兼容。如果您已有现有的 Babel 配置，您将自动选择不参与。我们计划很快将流行库如 `styled-components`, `emotion`, 和 `relay` 的解析移植过来。如果您正在使用自定义 Babel 设置，[请分享您的配置](https://github.com/vercel/next.js/discussions/30174)。

您还可以选择使用 Rust 编译器进行压缩。这比 Terser **快 7 倍**。压缩功能是可选的，直到经过充分验证，因为它替换了多年的基础设施。

next.config.js

```
module.exports = {
  swcMinify: true,
};
```

除了雇佣 [SWC 的创造者 DongYoon Kang](https://twitter.com/kdy1dev) 和 [Parcel 贡献者 Maia Teegarden](https://twitter.com/padmaia)，我们继续投资 Rust 生态系统。如果您有 Rust 工作经验，[请申请加入我们的团队](https://vercel.com/careers)。

更多信息，[观看我们在 Next.js Conf 上的演示](https://www.youtube.com/watch?v=_WNeAubn92U)。

## 介绍中间件

中间件使您能够使用代码代替配置。这赋予您在 Next.js 中的完全灵活性，因为您可以在请求完成之前运行代码。根据用户的传入请求，您可以重写、重定向、添加标头，甚至流式传输 HTML 来修改响应。

![中间件在 Next.js 内部给予您完全的灵活性。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-12/middleware.png)

中间件在 Next.js 内部给予您完全的灵活性。

中间件可用于任何为页面集共享逻辑的内容，包括：

* [身份验证](https://vercel.com/templates/next.js/jwt-authentication)
* [机器人保护](https://vercel.com/templates/next.js/bot-protection-datadome)
* [重定向](https://vercel.com/templates/next.js/edge-redirects-upstash)
* [重写](https://vercel.com/templates/next.js/hostname-rewrites)
* [特性标志和 A/B 测试](https://vercel.com/templates/next.js/ab-testing-simple)
* [高级 i18n 路由要求](https://vercel.com/templates/next.js/edge-functions-i18n)
* 还有 [更多](https://vercel.com/templates?type=edge-functions)!

中间件使用一个支持标准 Web API 如 **`fetch`** 的严格运行时。这使得它在使用 `next start` 或 Vercel 等 Edge 平台时开箱即用，后者使用 [Edge 中间件](http://www.vercel.com/edge)。

要在 Next.js 中使用中间件，您可以创建一个文件 `pages/_middleware.js`。在此示例中，我们使用标准 Web API Response ([MDN](https://developer.mozilla.org/docs/Web/API/Response))：

pages/_middleware.js

```
export function middleware(req, ev) {
  return new Response('Hello, world!');
}
```

更多信息，[观看演示](https://www.youtube.com/watch?v=WlP2TB2ORL4) 和 [查看文档](/docs/pages/building-your-application/routing/middleware)。

为 React 18 做准备 [](#preparing-for-react-18)
-------------------------------------------------

[React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html) 将增添诸如 Suspense、更新的自动批处理、`startTransition` 等 API，以及支持 `React.lazy` 的服务端渲染新方法。

我们一直与 Facebook 的 React 团队密切合作，为 Next.js 准备 React 18，使其在向稳定版本过渡。我们使这些功能可以在 Next.js 12 下通过实验性标志尝试。

终端

```
npm install react@alpha react-dom@alpha
```

### 服务端流式传输

[React 18](https://reactjs.org/blog/2021/06/08/the-plan-for-react-18.html) 中的并发功能包括内置支持服务端 Suspense 和 SSR 流式支持。这允许您使用 HTTP 流式传输渲染服务器页面。这是 Next.js 12 中的实验性功能，但一旦启用，SSR 将使用与中间件相同的严格运行时。

要启用，请使用实验性标志 `concurrentFeatures: true`：

next.config.js

```
module.exports = {
  experimental: {
    concurrentFeatures: true,
  },
};
```

### React 服务器组件

React 服务器组件允许我们在服务器端渲染所有内容，包括组件本身。这与服务端渲染不同，服务端渲染是在服务器上预生成 HTML。使用服务器组件，**不需要客户端 JavaScript**，从而使页面渲染更快。这改善了应用程序的用户体验，结合了服务端渲染和客户端交互的最佳部分。

next.config.js

```
module.exports = {
  experimental: {
    concurrentFeatures: true,
    serverComponents: true,
  },
};
```

Next.js 现在允许您在 _组件级别_ 获取数据，全部表示为 JSX。通过使用 React 服务器组件，我们可以简化事情。不再需要 `getServerSideProps` 或 `getStaticProps` 等特殊函数。这与 React Hooks 模型的将数据获取与组件共置保持一致。

您可以将任何 Next.js 页面重命名为 `.server.js` 以创建服务器组件，并在服务器组件中直接导入客户端组件。这些客户端组件将水合并变得可交互，因此您可以添加功能如点赞。

我们目前正在 Next.js 中开发服务端 Suspense、选择性水合和流式渲染，并将在未来的博客文章中分享我们的进展。

特别感谢我们的合作者 [Kara Erickson](https://twitter.com/karaforthewin) 和 [Gerald Monaco](https://twitter.com/devknoll) 在 [Google Aurora](https://web.dev/introducing-aurora/) 团队关于 React 18 和服务器组件方面的工作。

更多信息，[观看演示](https://www.youtube.com/watch?v=WlP2TB2ORL4) 和 [查看文档](https://vercel.link/react-18)。

## ES 模块支持和 URL 导入

ES 模块为 JavaScript 带来了官方的标准化模块系统。它们被所有主流浏览器以及 Node.js 所支持。

这一标准通过实现更小的包大小和 JavaScript 捆绑包，推动着网络生态系统向前发展，最终带来更好的用户体验。随着 JavaScript 生态系统从 Common JS（旧标准）过渡到 ES 模块，我们致力于帮助开发人员逐步采用这些改进，而不需要进行不必要的破坏性更改。

从 [Next.js 11.1](https://nextjs.org/blog/next-11-1) 开始，我们添加了对 [ES 模块](https://nodejs.org/docs/latest/api/esm.html) 的实验性支持，优先于 CommonJS 模块。在 Next.js 12 中，这是默认设置。导入仅提供 CommonJS 的 NPM 模块仍受支持。

## URL 导入

Next.js 12 包括通过 URL 导入 ES 模块的实验性支持，不需要安装或单独的构建步骤。

URL 导入允许您通过 URL 直接使用 _任何_ 包。这使得 Next.js 能够像处理本地依赖项一样处理远程 HTTP(S) 资源。

如果检测到 URL 导入，Next.js 将生成 `next.lock` 文件以跟踪远程资源。URL 导入会缓存在本地，以确保您仍然可以离线工作。Next.js 支持客户端和服务器 URL 导入。

要启用此功能，请在 `next.config.js` 中添加允许的 URL 前缀：

next.config.js

```
module.exports = {
  experimental: {
    urlImports: ['https://cdn.skypack.dev'],
  },
};
```

然后，您就可以直接从 URL 导入模块了：

```
import confetti from 'https://cdn.skypack.dev/canvas-confetti';
```

任何提供 ES 模块的 CDN 都可以，包括无代码和设计工具如 Framer：

* [Skypack](https://skypack.dev)
* [esm.sh](https://esm.sh/)
* [jsDelivr](https://www.jsdelivr.com/)
* [JSPM](https://jspm.org/)
* [unpkg](https://unpkg.com/)

更多信息，[观看演示](https://www.youtube.com/watch?v=_WNeAubn92U) 和 [查看文档](/docs/pages/api-reference/next-config-js/urlImports)。

## 机器人感知的 ISR 回退

目前， [增量静态再生](https://vercel.com/docs/concepts/incremental-static-regeneration/overview) 配合 `fallback: true`，在第一请求到一个尚未生成的页面之前渲染回退状态。要将页面加载 (服务端渲染) 阻塞，您需要使用 `fallback: 'blocking'`。

在 Next.js 12 中，[网络爬虫 (例如搜索机器人)](https://nextjs.org/learn/seo/introduction-to-seo/webcrawlers) 将自动使用 `fallback: true` 服务端渲染 ISR 页面，同时仍然将回退状态的先前行为提供给非爬虫 User-Agent。这防止了爬虫索引加载状态。

## 使用 AVIF 减小图像大小

内置的图像优化 API 现在支持 AVIF 图像，与 WebP 相比可实现 20% 的图像更小。

AVIF 图像可能需要更长时间进行优化，因此我们正在通过 `next.config.js` 中的新 `images.formats` 属性使此功能可选：

next.config.js

```
module.exports = {
  images: {
    formats: ['image/avif', 'image/webp'],
  },
};
```

此格式列表用于根据请求的 `Accept` 标头按需确定优化的图像格式。由于 AVIF 排在首位，如果 [浏览器支持 AVIF](https://caniuse.com/avif)，它将提供。如果不支持，如果 [浏览器支持 WebP](https://caniuse.com/webp)，将提供 WebP。如果两种格式都不支持，将提供原始图像格式。

## 输出文件追踪

在 Next.js 8 中，我们引入了 `target` 选项。这允许通过使用 webpack 在构建期间捆绑所有依赖项，将 Next.js 页面输出为独立的 JavaScript 捆绑包。我们很快意识到这不理想，并创建了 `@vercel/nft`。`@vercel/nft` 已过 2 年在 Vercel 平台的所有部署中使用。

现在，我们直接将这些改进带入 Next.js 框架中，默认应用于 **所有部署平台**，提供了比 `target` 选项显著改进的方法。

Next.js 12 自动使用 `@vercel/nft` 追踪每个页面和 API 路由所需的文件，并在 `next build` 输出旁边输出这些追踪结果，允许集成者利用 Next.js 自动提供的追踪信息。

这些更改还优化使用 Docker 等工具通过 `next start` 部署的应用程序。利用 `@vercel/nft`，我们将能够使 Next.js 输出在未来独立。运行应用程序将不需要安装任何依赖，大大减少 Docker 镜像大小。

将 `@vercel/nft` 引入 Next.js 取代了 `target` 方法，使得 `target` 在 Next.js 12 中已弃用。[查看文档](/docs/pages/api-reference/next-config-js/output) 了解更多信息。

[其他改进](#other-improvements)
-----------------------------------------

* 将 `pages/_app.js` 或 `pages/_document.js` 添加到您的应用程序现在会自动替换内置版本，而无需重新启动 Next.js CLI。
* ESLint 集成现在支持 [单文件 linting](https://github.com/vercel/next.js/pull/28042) 在 `next lint` 中，使用 `--file` 标志。
* Next.js 12 现在支持设置自定义 `tsconfig.json` 路径。
* `next.config.mjs` 现在支持将配置编写为 ES 模块。
* 正在处理的请求现在为 `getStaticProps` 消除重复。
* 检查静态页面现在使用共享工作池运行。
* Fast Refresh 现在使用 WebSocket 连接而不是 EventSource 连接。

破坏性更改[](#breaking-changes)
-------------------------------------

* 在 [Next.js 11](https://nextjs.org/blog/next-11#webpack-5) 中将 webpack 5 设为默认后，我们现在正式移除了 webpack 4。我们一直与社区密切合作，确保平稳过渡到 webpack 5。
* `next.config.js` 中的 `target` 不再需要。
* `next/image` 现在使用 `span` 作为包装元素，而不是 `div`。
* Node.js 最低版本已由 `12.0.0` 提升至 `12.22.0`，这是第一个具有原生 ES 模块支持的 Node.js 版本。

要了解更多信息，请查看 [升级指南](/docs/pages/building-your-application/upgrading)。
