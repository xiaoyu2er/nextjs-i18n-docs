* ** [快速刷新 (Fast Refresh)](#fast-refresh)**：快速可靠的实时编辑体验，经验证 [在 Facebook 规模上](https://twitter.com/dan_abramov/status/1152689338553131008)
* **[增量静态再生 (Incremental Static Regeneration)(Beta)](#incremental-static-regeneration-beta)**：在部署后重新构建静态页面，耗时毫秒级
* **[CMS (内容管理系统) 示例](#cms-examples)**：[Contentful](https://github.com/vercel/next.js/tree/canary/examples/cms-contentful)、[DatoCMS](https://github.com/vercel/next.js/tree/canary/examples/cms-datocms)、[Prismic](https://github.com/vercel/next.js/tree/canary/examples/cms-prismic)、[Sanity](https://github.com/vercel/next.js/tree/canary/examples/cms-sanity) 和 [TakeShape](https://github.com/vercel/next.js/tree/canary/examples/cms-takeshape) 的示例，使用我们新的 [下一代静态站点生成 (next-gen static site generation)](/blog/next-9-3)

* **[集成的 Web Vitals (Web Vitals) 报告](#integrated-web-vitals-reporting)**：捕获驱动 Lighthouse 分数的 [指标](https://web.dev/vitals/)，但来自您的真实流量

* **[可配置的 Sass (Sass) 支持](#configurable-sass-support)**：[配置 `includePaths`](https://sass-lang.com/documentation/js-api#includepaths) 和其他选项，针对我们的 [内置 Sass 支持](/docs/pages/building-your-application/styling#sass-support)
* **[改进的日志输出 (Log Output)](#improved-log-output)**：更容易阅读的控制台输出，格式一致，重复更少

快速刷新 (Fast Refresh)[](#fast-refresh)
-----------------------------
快速刷新是一种新的热重载体验，让您对 React 组件的编辑获得即时反馈。它现在**默认启用**，适用于 Next.js 9.4 或更高版本的所有项目。

热重载已经 [存在了很长时间](https://github.com/gaearon/react-hot-loader)，但 [历史上](https://github.com/gaearon/react-hot-loader/blob/master/docs/Troubleshooting.md) 过于 [脆弱](https://github.com/gaearon/react-hot-loader/issues?q=is%3Aissue)，以至于无法在工作流程中默认启用。因此，Next.js 此前实现了一种粗糙的热重载形式，会重置您应用程序的整个状态。

旧的热重载实现无法抵抗编译或运行时错误，如果你在编辑 CSS 或 JavaScript 时打错字，它将执行应用程序的全量重新加载。这很不理想，并打断了你的思路。

快速刷新深度集成到 React 本身 ([通过 React Refresh](https://github.com/facebook/react/tree/master/packages/react-refresh))，使 Next.js 能够对你的 React 组件树执行可预测的精确更新。

这意味着 Next.js 只会在你编辑的文件中更新代码，并且只重新渲染该组件，**而不会丢失组件状态**。这包括样式（内联、CSS-in-JS 或 CSS/Sass Modules）、标记、事件处理器和效果（通过 `useEffect`）。

具有编译和运行时错误（快速恢复）的编辑会话，以及保留状态的编辑。

作为此体验的一部分，我们彻底重新设计了错误叠加层，使其更具帮助并使您的应用程序能够抵抗拼写错误或运行时错误。这包括但不限于：

* **准确的位置**，解析为代码的**原始行和列**，在编译之前
* 上下文中相关的**源代码片段**，能够**点击在您的编辑器中打开**
* 语法错误修复后的**开发会话恢复**，**不丢失应用程序状态**
* 当错误修复时**自动关闭**未处理的运行时错误

我们要感谢 [Dan Abramov](https://twitter.com/dan_abramov) 为实施此功能所做的宝贵贡献和帮助。

增量静态再生 (Incremental Static Regeneration) (Beta)[](#incremental-static-regeneration-beta)
-------------------------------------------------------------------------------
Next.js 在 9.3 中引入了静态站点生成方法，其目标明确：我们应该获得 [静态的优势](https://rauchg.com/2020/2019-in-review#static-is-the-new-dynamic)（始终快速，始终在线，[全球分布](https://rauchg.com/2020/static-hoisting#hoist-to-the-edge)），但得益于 Next.js 以强大的动态数据支持而闻名。

为了兼顾两者，Next.js 支持 **增量静态生成**，在您构建站点后更新静态内容。例如，在 9.3 中，我们在 `getStaticPaths` 中引入了 `fallback: true` 选项，允许你在运行时添加**新页面**。

我们最近 [构建了一个示例](https://on-demand-isr.vercel.app/)，展示了 Next.js 如何以这种方式静态预渲染无限数量的页面。

今天，我们还推出了 **增量静态再生 (Incremental Static Regeneration) (Beta)**，这是一种**更新现有页面**的机制，通过在流量进入时在后台重新渲染它们。受 [stale-while-revalidate](https://tools.ietf.org/html/rfc5861) 启发，这确保流量可以不间断地服务，始终静态化，并且新构建的页面仅在生成完成后推送。

pages/blog/[slug].js

```
export async function function getStaticProps() {
  return {
    props: await getDataFromCMS(),
    // 我们将尝试重新生成页面：
    // - 当有新请求进来时
    // - 最多每秒一次
    unstable_revalidate: 1,
  };
}
```

与 SSR 不同，增量静态再生确保您保留静态的优势：

* 没有延迟尖峰。页面始终保持快速服务。
* 页面永不离线。如果后台页面重新生成失败，旧页面保持不变。
* 低数据库和后端负载。页面重新计算最多并发一次。

两个增量功能（添加页面以及懒惰更新它们），以及 [预览模式](/docs/pages/building-your-application/configuring/preview-mode)，都已经完全支持 `next start` 和 [Vercel Edge 平台](https://vercel.com) 开箱即用。

接下来，我们将致力于补充 RFC，以解决另外两个增量静态生成功能：

*   同时重新生成和使多个页面无效（例如您的博客索引和特定博客帖子）
*   通过监听事件重新生成（例如 CMS Webhook），在用户流量之前

新的环境变量 (Environment Variables) 支持 [](#new-environment-variables-support)
-----------------------------------------------------------------------
我们从使用 Next.js 的公司那里得到的一个普遍反馈是，不清楚环境变量何时被内联到浏览器 bundle 中，何时仅在 Node.js 环境中可用。

今天我们要宣布两个完全向后兼容的功能，将帮助简化此流程。

首先，您现在可以将环境变量前缀为 `NEXT_PUBLIC_` 以将其暴露给浏览器。当使用该环境变量时，它将被内联到浏览器 JavaScript bundle 中。

你不再需要添加 `next.config.js` 并添加 `env` key 来暴露这些变量。

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
// The environment variable will be exposed to the browser
console.log('My Application Version', process.env.NEXT_PUBLIC_VERSION);
 
export default function HomePage() {
  return <h1>Hello World</h1>;
}
```

第二个变化是 Next.js 现在默认支持 `.env` 加载。允许您轻松定义开发和生产环境变量。

您可以在 [环境变量文档](/docs/pages/building-your-application/configuring/environment-variables) 中阅读更多关于 `.env` 加载的信息。

这些新功能将遵循以下约定来简化使用环境变量：

* 环境变量默认仅在 Node.js 环境中可用
* 前缀为 `NEXT_PUBLIC_` 的环境变量暴露给浏览器

改进的内置 Fetch (Fetch) 支持 [](#improved-built-in-fetch-support)
-------------------------------------------------------------------
在 [Next.js 9.1.7](https://nextjs.org/blog/next-9-1-7#new-built-in-polyfills-fetch-url-and-objectassign) 中，我们宣布了 [`fetch()`](https://developer.mozilla.org/docs/Web/API/Fetch_API) API 在浏览器中的 polyfill。今天，此 polyfill 已扩展到 Node.js 环境。

在实践中，您不再需要使用任何类型的服务器端 fetch polyfill（例如 `isomorphic-unfetch` 或 `node-fetch`），因为 Next.js 将在所有环境中自动提供 `fetch()`。

例如，在使用 `getStaticProps` 时，它是在构建时由 Next.js 执行的：

pages/blog.js

```
export async function getStaticProps() {
  // fetch 不再需要从 isomorphic-unfetch 导入
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  return {
    props: {
      posts,
    },
  };
}
 
function Blog({ posts }) {
  // 渲染帖子...
}
 
export default Blog;
```

集成的 Web Vitals (Web Vitals) 报告 [](#integrated-web-vitals-reporting)
-------------------------------------------------------------------
上周，Google Chrome 团队推出了 [Web 核心指标 (Core Web Vitals)](https://web.dev/vitals/)。Web 核心指标是提供网页卓越用户体验的关键质量信号，在其之上构建了 [著名的 Lighthouse 报告](https://twitter.com/rauchg/status/1259701306387656704)。

如果您希望您的网站或 Web 应用程序尽可能快，跟踪这些指标是非常有用的，这也是 Next.js 的核心目标之一。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9-4/core-web-vitals.jpg)

Chrome 团队发布了一个 [Web Vitals Chrome 扩展](https://twitter.com/addyosmani/status/1258416376433131520)，允许您作为开发者获得页面性能表现的视觉反馈。

在构建生产 Web 应用程序时，您还想知道网站对于您的访问者（和潜在客户）的表现如何。您甚至可能希望跟踪这些指标随时间的改进或倒退，以查看更改是否对您的受众产生了预期影响。

为了协助将 Web Vitals 报告到您的分析服务，我们 [与 Google 合作](https://www.techrepublic.com/article/insiders-look-at-googles-web-framework-contributions-to-next-js-and-more/)，引入了一个新方法 `reportWebVitals`，可以从 `pages/_app.js` 导出：

`pages/_app.js`

```
// 对于每个需要报告的性能指标将被调用一次。
export function reportWebVitals(metric) {
  // 这些指标可以发送到任何分析服务
  console.log(metric);
}
 
function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
 
export default MyApp;
```

有关如何使用此方法与您的分析服务结合使用的说明，请参阅文档中的 ["将结果发送到分析服务"](/docs/pages/building-your-application/optimizing/analytics#sending-results-to-analytics) 部分。如果您想了解更多关于 Web Vitals 的信息，可以参考 [web.dev/vitals](https://web.dev/vitals/)。

绝对导入和别名 (Absolute Imports and Aliases) [](#absolute-imports-and-aliases)
-------------------------------------------------------------
如果您正在处理大型项目，某些 `import` 语句可能会受到 `../../../` 意大利面条代码的困扰：

```
import Button from '../../../../components/button';
```

在这种情况下，我们可能希望使用**绝对导入**而不是相对导入。假设 `components` 目录存在于根目录，我们可能希望 `import` 语句看起来像这样：

```
import Button from 'components/button';
```

我们要宣布 Next.js 9.4 使为 JavaScript 和 TypeScript 项目设置绝对导入变得非常简单。您需要做的就是将 `baseUrl` 配置添加到 [`jsconfig.json` (JS 项目)](https://code.visualstudio.com/docs/languages/jsconfig#_jsconfig-options) 或 [`tsconfig.json` (TS 项目)](https://www.typescriptlang.org/docs/handbook/module-resolution.html#base-url)。

jsconfig.json / tsconfig.json

```
{
  "compilerOptions": {
    "baseUrl": "."
  }
}
```

这将允许从 `.`（根目录）进行绝对导入。它还与 VSCode 和其他编辑器集成，支持代码导航和其他编辑器功能。

**注意：** 如果您之前修改了 [Webpack 模块别名配置](https://webpack.js.org/configuration/resolve/#resolvealias) 来启用绝对导入，该配置现在也可以移除。

此外，Next.js 9.4 还支持 `paths` 选项，允许您创建自定义模块别名。例如，以下内容允许您使用 `@/design-system` 而不是 `components/design-system`：

jsconfig.json / tsconfig.json

```
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/design-system/*": ["components/design-system/*"]
    }
  }
}
```

然后您可以按以下方式使用您的别名：

```
// 导入 'components/design-system/button'
import Button from '@/design-system/button';
```

如果您指定 `paths`，则必须指定 `baseUrl`。您可以在 [TypeScript 文档](https://www.typescriptlang.org/docs/handbook/module-resolution.html#path-mapping) 中了解更多关于 `paths` 选项的信息。

可配置的 Sass (Sass) 支持 [](#configurable-sass-support)
-------------------------------------------------------
当内置 Sass (Sass) 支持在 [Next.js 9.3](https://nextjs.org/blog/next-9-3#built-in-sass-support-for-global-stylesheets) 中发布时，我们收到反馈，部分用户希望配置 Sass 编译器。例如配置 `includePaths`。

现在可以通过在 `next.config.js` 中使用 `sassOptions` key 来实现：

next.config.js

```
const path = require('path');
 
module.exports = {
  sassOptions: {
    includePaths: [path.join(__dirname, 'styles')],
  },
};
```
