*   在不缓存的情况下，初始路由编译速度最高快 **45.8%**（Turbopack 还没有磁盘缓存）。

构建和生产改进[](#build-and-production-improvements)
-------------------------------------------------------

除了 Turbopack 的捆绑改进外，我们还致力于提高所有 Next.js 应用（包括 Pages 和 App Router）的整体构建和生产性能。

### Tree-shaking[](#tree-shaking)

我们识别出服务器组件和客户端组件之间边界的一个优化，允许对未使用的导出进行 Tree-shaking。例如，从包含 `"use client"` 的文件中导入单个 `Icon` 组件不再包含该包中的所有其他图标。这可以大幅减少生产 JavaScript 包大小。

在 `react-aria-components` 这样的大型库上测试此优化，最终包大小减少了 **\-51.3%**。

> **注意：** 此优化目前不适用于桶文件。在此期间，可以使用 [`optimizePackageImports`](/docs/app/api-reference/next-config-js/optimizePackageImports) 配置选项：
> 
> next.config.ts
> 
> ```
> module.exports = {
>   experimental: {
>     optimizePackageImports: ['package-name'],
>   },
> };
> ```

### 构建内存使用[](#build-memory-usage)

对于超大规模的 Next.js 应用，我们注意到在生产构建期间出现内存溢出崩溃 (OOMs)。在调查用户报告和生产环境复现后，我们确定根本原因是过度捆绑和压缩（Next.js 创建了较少、较大的 JavaScript 文件并进行复制）。我们重构了捆绑逻辑并对编译器进行了针对这些情况的优化。

我们的早期测试显示，对于一个最小 Next.js 应用，内存使用和缓存文件大小平均从 **2.2GB 降低到 190MB 以下**。

为了更容易调试内存性能，我们在 `next build` 中引入了 `--experimental-debug-memory-usage` 标志。在 [文档](/docs/app/building-your-application/optimizing/memory-usage) 中了解更多。

### CSS[](#css)

我们在生产模式下 Next.js 构建期间优化 CSS 的方式进行了更新，通过分块 CSS 来避免在页面之间导航时发生样式冲突。

CSS 块的顺序和合并现在由导入顺序决定。例如，`base-button.module.css` 将排在 `page.module.css` 之前：

base-button.tsx

```
import styles from './base-button.module.css';
 
export function BaseButton() {
  return <button className={styles.primary} />;
}
```

page.tsx

```
import { BaseButton } from './base-button';
import styles from './page.module.css';
 
export function Page() {
  return <BaseButton className={styles.primary} />;
}
```

为了保持正确的 CSS 顺序，我们建议：

*   使用 [CSS Modules](/docs/app/building-your-application/styling/css-modules) 代替 [全局样式](/docs/app/building-your-application/styling/css-modules#global-styles)。
*   只在一个 JS/TS 文件中导入 CSS 模块。
*   如果使用全局类名，也在同一个 JS/TS 文件中导入全局样式。

我们预计此更改不会负面影响大多数应用。但是，如果您在升级时看到任何意外样式，请按照 [文档](/docs/app/building-your-application/styling/css#ordering-and-merging) 中的建议检查您的 CSS 导入顺序。

缓存改进[](#caching-improvements)
---------------------------------------------

缓存是构建快速可靠 Web 应用的关键部分。在执行变更操作时，用户和开发者都期望缓存被更新以反映最新变更。我们一直在探索如何改进 App Router 中的 Next.js 缓存体验。

### `staleTimes` (实验性)[](#staletimes-experimental)

[客户端路由器缓存](/docs/app/building-your-application/caching#data-cache-and-client-side-router-cache) 是一种缓存层，旨在通过在客户端缓存访问和预取的路由来提供快速的导航体验。

根据社区反馈，我们添加了一个实验性的 `staleTimes` 选项，以允许配置 [客户端路由器缓存](/docs/app/building-your-application/caching#router-cache) 的失效周期。

默认情况下，预取路由（使用没有 `prefetch` 属性的 `<Link>` 组件）将缓存 30 秒，如果 `prefetch` 属性设置为 `true`，则缓存 5 分钟。您可以通过在 `next.config.js` 中定义自定义 [重新验证时间](/docs/app/building-your-application/caching#duration-3) 来覆盖这些默认值：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
};
 
module.exports = nextConfig;
```

`staleTimes` 旨在改进当前那些希望对缓存启发式规则有更多控制权的用户体验，但它并不是完整的解决方案。在未来版本中，我们将关注改进整体缓存语义并提供更灵活的解决方案。

在 [文档](/docs/app/api-reference/next-config-js/staleTimes) 中了解更多关于 `staleTimes`。

### 并行和拦截路由[](#parallel-and-intercepting-routes)

我们持续迭代 [并行](/docs/app/building-your-application/routing/parallel-routes) 和 [拦截](/docs/app/building-your-application/routing/intercepting-routes) 路由，现在改进与客户端路由器缓存的集成。

*   调用带有 [`revalidatePath`](/docs/app/api-reference/functions/revalidatePath) 或 [`revalidateTag`](/docs/app/api-reference/functions/revalidateTag) 的服务器操作的并行和拦截路由将重新验证缓存并刷新可见槽位，同时保持用户的当前视图。
*   同样，调用 [`router.refresh`](/docs/app/building-your-application/caching#routerrefresh) 现在将正确刷新可见槽位，保持当前视图。

错误开发者体验改进[](#errors-dx-improvements)
-------------------------------------------------

在 14.1 版本中，我们开始着手 [改进运行 `next dev` 时错误消息和堆栈跟踪的可读性](/blog/next-14-1#improved-error-messages-and-fast-refresh)。此项工作在 14.2 中继续，现在包括更好的错误消息、App Router 和 Pages Router 的覆盖层设计改进、浅色和深色模式支持，以及更清晰的 `dev` 和 `build` 日志。

React 19[](#react-19)
---------------------

在二月，React 团队宣布了 [React 19](https://react.dev/blog/2024/02/15/react-labs-what-we-have-been-working-on-february-2024#the-next-major-version-of-react) 的即将发布。为准备 React 19，我们正在将最新功能和改进集成到 Next.js 中，并计划发布一个主要版本来协调这些变更。

如 Actions 和其相关钩子等新功能（目前可在 Next.js 中从 [React Canary 通道](https://react.dev/blog/2023/05/03/react-canaries) 获取）现在将可供所有 React 应用（包括纯客户端应用）使用。我们很兴奋看到这些功能在 React 生态系统中得到更广泛的采用。
