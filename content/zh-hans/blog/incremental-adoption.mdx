[Next.js](https://nextjs.org/) 专为渐进式采用而设计。有了 Next.js，你可以继续使用现有的代码，并根据需要添加任意数量的 React。通过从小处着手并逐步增加更多页面，你可以避免完全重写，从而防止阻碍功能开发。

许多公司需要现代化其技术栈，以降低运营成本、提高开发人员生产力，并为他们的客户提供最佳体验。组件驱动的开发已显著提高了现代代码库的部署速度和可重用性。

并且，拥有超过 [每月 800 万次下载](https://www.npmtrends.com/react)，React 是开发人员首选的组件驱动解决方案。Next.js 作为用于生产的 React 框架，让你能够渐进式地采用 React。

动机 [](#motivation)
-------------------------

在一个日益移动化的世界中，改进和跟踪你的 [核心网页指标](/analytics) 对成功至关重要。你的客户可能分布在世界各地，且互联网速度各不相同。等待页面加载或操作完成所多花的一秒钟（或一毫秒），可能是销售、展示或转化之间的区别。

如果你正在现代化你的技术栈，你可能会面临以下挑战：

* 你的应用程序有数年的遗留代码，难以理解，完全重写需要多年（且花费数百万美元）。
* 随着应用程序的规模和复杂性增加，你的页面加载时间持续增加。简单的营销页面和最复杂的页面加载一样慢。
* 你正试图扩大开发团队，但在向现有代码库添加更多开发人员时遇到了问题。
* 你拥有过时的 CI/CD 和 DevOps 流程，这降低了开发人员生产力，并使得安全可靠地发布新更改变得困难。
* 你的应用程序在移动设备上没有响应，并且在破坏应用程序其他部分的情况下无法更新全局页面样式。

你知道你需要 _做点什么_，但要理解 [从哪里开始](https://www.psychologytoday.com/us/blog/mindfully-present-fully-alive/201804/the-only-way-eat-elephant) 可能会让人感到不知所措。通过渐进式采用 Next.js，你可以开始解决上述问题并转型你的应用程序。让我们讨论几种将 Next.js 集成到你现有技术栈的策略。

策略 [](#strategies)
-------------------------

### 子路径 [](#subpath)

第一种策略是配置服务器或代理，使得特定子路径下的所有内容都指向 Next.js 应用。例如，你的现有网站可能位于 example.com，并且你可以配置代理，使得 example.com/store 提供 Next.js 电子商务商店服务。

使用 [`basePath`](/docs/pages/api-reference/next-config-js/basePath)，你可以配置 Next.js 应用的资源和链接，使之与新的子路径 `/store` 自动配合。由于 Next.js 中的每个页面都是其自身的 [独立路由](/docs/pages/building-your-application/routing)，像 `pages/products.js` 这样的页面将路由到应用中的 `example.com/store/products`。

next.config.js

```
module.exports = {
  basePath: '/store',
};
```

要了解有关 `basePath` 的更多信息，请参阅我们的 [文档](/docs/pages/api-reference/next-config-js/basePath)。

(**注意：** 此功能在 Next.js 9.5 及以上版本中引入。如果你使用的是旧版本的 Next.js，请在尝试之前先进行升级。)

### 重写(#rewrites)

第二种策略是创建一个指向你的域名根 URL 的新 Next.js 应用。然后，你可以在 `next.config.js` 中使用 [`rewrites`](/docs/pages/api-reference/next-config-js/rewrites)，使某些子路径代理到你的现有应用。

例如，假设你创建了一个从 `example.com` 服务的 Next.js 应用，其 `next.config.js` 如下所示。现在，为此 Next.js 应用中添加的页面（例如，如果你添加了 `pages/about.js`，则是 `/about`）的请求将由 Next.js 处理，而其他任何路由（例如 `/dashboard`）的请求将代理到 `proxy.example.com`。

next.config.js

```
module.exports = {
  async rewrites() {
    return [
      // we need to define a no-op rewrite to trigger checking
      // all pages/static files before we attempt proxying
      {
        source: '/:path*',
        destination: '/:path*',
      },
      {
        source: '/:path*',
        destination: `https://proxy.example.com/:path*`,
      },
    ];
  },
};
```

要了解有关 rewrites 的更多信息，请参阅我们的 [文档](/docs/pages/api-reference/next-config-js/rewrites)。

### 利用 Monorepo 和子域名构建的微前端 [](#micro-frontends-with-monorepos-and-subdomains)

Next.js 和 [Vercel](https://vercel.com) 使得采用 [微前端](https://martinfowler.com/articles/micro-frontends.html) 并以 [Monorepo](https://vercel.com/blog/monorepos) 部署变得容易。这让你能够使用 [子域名](https://en.wikipedia.org/wiki/Subdomain) 渐进式地采用新应用程序。微前端的一些好处：

* 更小、更一致且易于维护的代码库。
* 更加可扩展的组织，拥有解耦、独立的团队。
* 能够以更渐进的方式升级、更新甚至重写的部分前端。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/incremental-adoption/light-arch.png)

> 部署到 Vercel 的 Monorepo 架构。

一旦你的 Monorepo 设置完成，像往常一样将更改推送到你的 Git 仓库，你就会看到提交的内容部署到你连接的 Vercel 项目中。向过时的 CI/CD 流程告别。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/incremental-adoption/dark-comment.png)

结论<a id="conclusion"></a>

Next.js 是专为渐进式采用到你现有技术栈而设计的。Vercel 平台通过将 GitHub、GitLab 和 Bitbucket 无缝集成，使得每次代码更改都有部署预览，从而成为一种协作体验。

* 使用 [Fast Refresh](/docs/architecture/fast-refresh) 在本地立即预览更改，提高开发人员生产力。
* 推送更改以创建 [分支预览](https://vercel.com/github)，优化了与利益相关者的协作。
* 通过合并 PR 使用 [Vercel](https://vercel.com) 部署到生产环境。无需复杂的 DevOps。

要了解更多信息，请阅读关于 [子路径](/docs/pages/api-reference/next-config-js/basePath) 和 [rewrites](/docs/pages/api-reference/next-config-js/rewrites) 的内容，或 [通过微前端部署示例](https://vercel.com/import/project?template=https://github.com/vercel/next.js/tree/canary/examples/with-zones)。
