Next.js 15.1 带来了核心升级、新 API 以及对开发者体验的改进。关键更新包括：

*   [**React 19（稳定）**](/blog/next-15-1#react-19-stable): React 19 支持现已在 Pages Router 与 App Router 中正式可用。
*   [**改进的错误调试**](/blog/next-15-1#improved-error-debugging): 增强的 DX 以及对浏览器和终端的更好源映射支持。
*   [**`after`（稳定）**](/blog/next-15-1#after-stable): 用于在响应完成流式传输后执行代码的新 API。
*   [**`forbidden` / `unauthorized`（实验性）**](/blog/next-15-1#forbidden-and-unauthorized-experimental): 用于实现更细粒度身份验证错误处理的新 API。

立即升级，或使用以下内容开始：

终端

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade latest
 
# ...or upgrade manually
npm install next@latest react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@latest
```

React 19（稳定）[](#react-19-stable)
-------------------------------------

Next.js 15.1 现已完全支持 React 19:

*   **对于 Pages Router**：你现在可以使用 React 19 稳定版，无需依赖发布候选版或 Canary 版，同时继续支持 React 18。
*   **对于 App Router**：我们将继续提供内建的 React Canary 发布版。这些版本包含所有稳定的 React 19 变更，以及在新 React 版本发布前在框架中验证的新功能。

自从 Next.js 15 发布以来，React 19 的一个重要新增是"[sibling pre-warming](https://react.dev/blog/2024/04/25/react-19-upgrade-guide#improvements-to-suspense)"。

关于 React 19 更新的综合概述，请参阅 [官方 React 19 博客文章](https://react.dev/blog/2024/12/05/react-19)。

改进的错误调试[](#improved-error-debugging)
-----------------------------------------------------

我们在 Next.js 中改进了错误调试功能，确保您可以快速定位问题的来源，无论是在终端、浏览器还是附加的调试器中。这些改进适用于 Webpack 和 Turbopack（随着 Next.js 15 现已稳定）([now stable with Next.js 15](/blog/turbopack-for-development-stable))。

### 源映射增强 [](#source-maps-enhancements)

错误现在可以通过改进的源映射使用更轻松追溯到其来源。我们实现了源映射的 [`ignoreList` 属性](https://developer.chrome.com/docs/devtools/x-google-ignore-list)，允许 Next.js 隐藏外部依赖的堆栈帧，让您的应用程序代码成为主要焦点。

为了获得稍微更准确的方法名称源映射，建议采用 Turbopack（现已稳定），它在 Webpack 之上改进了源映射的处理和检测。

> **对于库作者**：建议在发布库时填充源映射中的 `ignoreList` 属性，特别是如果它们被配置为外部（例如在 `serverExternalPackages` 配置中）。

### 折叠的堆栈帧 [](#collapsed-stack-frames)

我们改进了折叠堆栈帧的逻辑以突出显示您代码中最相关的部分。

*   **在浏览器和错误覆盖层中**：来自第三方依赖的堆栈帧默认被隐藏，专注于您的应用程序代码。您可通过点击开发者工具或覆盖层中的“Show ignored frames”来显示隐藏的帧。
*   **在终端中**：第三方依赖帧默认也被折叠，错误格式现在与浏览器输出保持一致以确保调试体验的一致性。如果需要整个堆栈跟踪，错误会在浏览器中重播以确保您不会在开发过程中遗漏重要信息。

### 增强的分析功能 [](#enhanced-profiling)

被忽略的堆栈帧也会被内置浏览器分析器识别。这使得分析您的应用程序更加简单，让您能够精确定位代码中的慢函数，而不受外部库的干扰。

### 边缘运行时改进 [](#improved-with-the-edge-runtime)

使用 Edge 运行时，错误现在在开发环境中显示一致，确保无缝调试。以前，记录错误仅包含消息而不包含堆栈。

`after`（稳定）[](#after-stable)
---------------------------------

`after()` API 在其首次引入于 Next.js 15 RC 后现已稳定。

*   **改进的支持** 针对自托管 Next.js 服务器。
*   **Bug 修复** 针对 `after()` 与其他 Next.js 功能交互的场景。
*   **增强扩展性**，允许其他平台注入自己的 `waitUntil()` 原语来驱动 `after()`。
*   **支持运行时的 API**，例如 Server Actions 和 Route Handlers 中的 `cookies()` 和 `headers()`。

app/layout.js

```
import { after } from 'next/server';
import { log } from '@/app/utils';
 
export default function Layout({ children }) {
  // Secondary task
  after(() => {
    log();
  });
 
  // Primary task
  return <>{children}</>;
}
```

阅读更多有关 [`after`](/docs/app/api-reference/functions/after) API 以及如何利用它在文档中的使用方式的文档。

`forbidden` 和 `unauthorized`（实验性）[](#forbidden-and-unauthorized-experimental)
-----------------------------------------------------------------------------------------

Next.js 15.1 包含两个实验性 API，`forbidden()` 和 `unauthorized()`，基于社区反馈。

> **我们需要您的反馈** — 请在您的开发环境中尝试并在 [此讨论线程](https://github.com/vercel/next.js/discussions/73753) 中分享您的想法。

### 概述 [](#overview)

如果您熟悉 App Router，您可能已使用 [`notFound()`](/docs/app/api-reference/file-conventions/not-found) 来触发 404 行为以及可自定义的 `not-found.tsx` 文件。借助 15.1 版本，我们将该方法扩展到身份验证错误：
• `forbidden()` 触发 **403 错误**，通过 `forbidden.tsx` 自定义 UI。
• `unauthorized()` 触发 **401 错误**，通过 `unauthorized.tsx` 自定义 UI。

由于此功能仍处于实验阶段，您需要在 `next.config.ts` 文件中启用它：

• `unauthorized()` triggers a **401 error** with customizable UI via `unauthorized.tsx`.

> **Good to know:** As with `notFound()` errors, the status code will be `200` if the error is triggered after initial response headers have been sent. [Learn more](/docs/app/building-your-application/routing/loading-ui-and-streaming#status-codes).

### Enabling the feature[](#enabling-the-feature)

As this feature is still experimental, you'll need to enable it in your `next.config.ts` file:

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  experimental: {
    authInterrupts: true,
  },
};
 
export default nextConfig;
```

> **注意**：`next.config.ts` 支持是在 Next.js 15 中引入的。 [了解更多](/docs/app/api-reference/config/next-config-js#typescript)。

### 使用 `forbidden()` 和 `unauthorized()` [](#using-forbidden-and-unauthorized)

您可以在 Server Actions、Server Components、Client Components 或 Route Handlers 中使用 `forbidden()` 和 `unauthorized()`。示例如下：

```
import { verifySession } from '@/app/lib/dal';
import { forbidden } from 'next/navigation';
 
export default async function AdminPage() {
  const session = await verifySession();
 
  // Check if the user has the 'admin' role
  if (session.role !== 'admin') {
    forbidden();
  }
 
  // Render the admin page for authorized users
  return <h1>Admin Page</h1>;
}
```

### 创建自定义错误页面 [](#creating-custom-error-pages)

要自定义错误页面，请创建以下文件：

app/forbidden.tsx

```
import Link from 'next/link';
 
export default function Forbidden() {
  return (
    <div>
      <h2>Forbidden</h2>
      <p>You are not authorized to access this resource.</p>
      <Link href="/">Return Home</Link>
    </div>
  );
}
```

app/unauthorized.tsx

```
import Link from 'next/link';
 
export default function Unauthorized() {
  return (
    <div>
      <h2>Unauthorized</h2>
      <p>Please log in to access this page.</p>
      <Link href="/login">Go to Login</Link>
    </div>
  );
}
```

我们要感谢 [Clerk](https://clerk.com) 通过 PR 提出此功能并协助我们进行 API 原型设计。在 15.2 中稳定此功能之前，我们计划向 API 添加更多功能和改进以支持更广泛的应用场景。

阅读 [`unauthorized`](/docs/app/api-reference/functions/unauthorized) 和 [`forbidden`](/docs/app/api-reference/functions/forbidden) API 的文档以了解更多详情。
