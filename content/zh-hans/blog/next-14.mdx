Next.js 编译器：Turbocharged[](#nextjs-compiler-turbocharged)
---------------------------------------------------------------

自从 Next.js 13 以来，我们就一直致力于提升 Next.js 在 Pages 和 App Router 中的本地开发性能。

此前，我们正在重写 `next dev` 及其他 Next.js 部分以支持这项工作。现在我们改变了方法，使其更具增量性。这意味着我们基于 Rust 的编译器很快就会达到稳定状态，因为我们已重新专注于首先支持所有 Next.js 功能。

5,000 个 `next dev` 的集成测试现在正在 [Turbopack](https://nextjs.org/docs/app/api-reference/turbopack) 中通过，这是我们底层的 Rust 引擎。这些测试包含 7 年的错误修复和复现。

在 `vercel.com`（一个大型 Next.js 应用程序）上进行测试时，我们看到：

*   本地服务器启动最多快 53.3%
*   使用 Fast Refresh 的代码更新最多快 94.7%

该基准测试结果是一个大型应用（以及大型模块图）应期望的性能改进的实际结果。随着 `next dev` 90% 的测试现在通过，在使用 `next dev --turbo` 时，您应该看到更快且更可靠的表现。

当我们达到 100% 测试通过时，我们将在接下来的次要版本中将 Turbopack 转变为稳定状态。我们也将继续支持使用 webpack 进行自定义配置和生态系统插件。

您可以在 [areweturboyet.com](https://areweturboyet.com) 跟踪测试通过率。

表单和变更[](#forms-and-mutations)
-------------------------------------------

Next.js 9 引入了 API Routes——一种快速构建与前端代码并存的后端端点的方式。

例如，您会在 `api/` 目录下创建一个新文件：

pages/api/submit.ts

```
import type { NextApiRequest, NextApiResponse } from 'next';
 
export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  const data = req.body;
  const id = await createItem(data);
  res.status(200).json({ id });
}
```

然后，在客户端，您可以使用 React 和像 `onSubmit` 这样的事件处理程序来向您的 API Route 发出 `fetch` 请求。

pages/index.tsx

```
import { FormEvent } from 'react';
 
export default function Page() {
  async function onSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();
 
    const formData = new FormData(event.currentTarget);
    const response = await fetch('/api/submit', {
      method: 'POST',
      body: formData,
    });
 
    // Handle response if necessary
    const data = await response.json();
    // ...
  }
 
  return (
    <form onSubmit={onSubmit}>
      <input type="text" name="name" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

现在有了 Next.js 14，我们希望简化数据变更的开发者体验。此外，当用户在网络缓慢的情况下，或者从性能较低的设备提交表单时，我们希望能改善用户体验。

### Server Actions (Stable)[](#server-actions-stable)

如果您不需要手动创建 API Route 会怎样？您可以定义一个在服务器端安全运行的函数，直接从 React 组件调用。

App Router 基于 React canary 渠道构建，该渠道 [对框架采用新功能而言已稳定](https://react.dev/blog/2023/05/03/react-canaries)。从 v14 开始，Next.js 已升级到最新的 React canary，其中包括稳定的 Server Actions。

之前的 Pages Router 示例可以简化为一个文件：

app/page.tsx

```
export default function Page() {
  async function create(formData: FormData) {
    'use server';
    const id = await createItem(formData);
  }
 
  return (
    <form action={create}>
      <input type="text" name="name" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

对于以前使用过服务端中心框架的开发者来说，Server Actions 应该感觉很熟悉。它建立在表单和 [FormData Web API](https://developer.mozilla.org/en-US/docs/Web/API/FormData) 等 Web 基础之上。

虽然通过表单使用 Server Actions 有助于渐进增强，但不是必需的。您也可以直接将其作为函数调用，而无需表单。在使用 TypeScript 时，这能为您提供从客户端到服务器的完整端到端类型安全。

数据变更、重新渲染页面或重定向可以发生在 **一次网络往返** 中，确保客户端显示正确的数据，即使上游提供商很慢。此外，您可以组合并重用不同的操作，包括同一路由中的多个不同操作。

### 缓存、重新验证、重定向等 [](#caching-revalidating-redirecting-and-more)

Server Actions 与整个 App Router 模型深度集成。您可以：

*   使用 `revalidatePath()` 或 `revalidateTag()` 重新验证缓存数据
*   通过 `redirect()` 重定向到不同的路由
*   通过 `cookies()` 设置和读取 Cookie
*   使用 `useOptimistic()` 处理乐观 UI 更新
*   使用 `useFormState()` 捕获并显示来自服务器的错误
*   使用 `useFormStatus()` 在客户端显示加载状态

更多关于使用 Server Actions 的 [表单和变更](https://nextjs.org/docs/app/building-your-application/data-fetching/forms-and-mutations) 或 [安全模型](https://nextjs.org/blog/security-nextjs-server-components-actions) 以及 Server Components 和 Server Actions 的最佳实践。

### 部分预渲染（预览）[](#partial-prerendering-preview)

我们想分享关于部分预渲染的预览——这是我们正在为 Next.js 开发的针对动态内容的编译器优化，具有快速初始静态响应——。

部分预渲染建立在服务端渲染 (SSR)、静态站点生成 (SSG) 和增量静态重新验证 (ISR) 长达十年的研究和开发之上。

### 动机 [](#motivation)

我们听到了您的反馈。目前，需要考量的运行时、配置选项和渲染方法太多了。您希望拥有静态的速度和可靠性，同时也支持完全动态的个性化响应。

在全球范围内获得出色的性能 _以及_ 个性化不应以复杂性为代价。

我们的挑战是创造更好的开发者体验，简化现有的模型，同时不引入开发者需要学习的新 API。虽然服务端内容的部分缓存储已存在，但这些方法仍需要满足我们旨在实现的开发者体验和组合性目标。

**部分预渲染不需要学习新 API。**

启用部分预渲染后，此页面会根据您的 `<Suspense />`边界生成静态外壳。React Suspense 的 `fallback` 会被预渲染。

随后，外壳中的 Suspense fallback 会被动态组件替换，例如读取 Cookie 以确定购物车，或者根据用户显示横幅。

*   `viewport`：设置视口的初始缩放和其他属性
*   `colorScheme`：为视口设置支持模式（浅色/深色）
*   `themeColor`：设置视口周围 chrome 应渲染的颜色

从 Next.js 14 开始，有新的选项 [`viewport` and `generateViewport`](https://nextjs.org/docs/app/api-reference/functions/generate-viewport) 来替代这些选项。所有其他 `metadata` 选项保持不变。

您可以从今天开始采用这些新的 API。现有的 `metadata` 选项将继续有效。

### Hydration 错误改进[](#hydration-error-improvements)

*   **\[破坏性\]** 最低 Node.js 版本现在为 `18.17`
*   **\[破坏性\]** 移除 `next-swc` 构建的 WASM 目标 ([PR](https://github.com/vercel/next.js/pull/57437))
*   **\[破坏性\]** 弃用 `@next/font` 支持，转而采用 `next/font` ([Codemod](https://nextjs.org/docs/app/building-your-application/upgrading/codemods#built-in-next-font))
*   **\[破坏性\]** 将 `ImageResponse` 导入从 `next/server` 改为 `next/og` ([Codemod](https://nextjs.org/docs/app/building-your-application/upgrading/codemods#next-og-import))
*   **\[破坏性\]** `next export` 命令已被移除，转而采用 `output: 'export'` 配置 ([Docs](https://nextjs.org/docs/app/building-your-application/deploying/static-exports))
*   **\[弃用\]** `onLoadingComplete` 对于 `next/image` 已弃用，转而采用 [`onLoad`](https://nextjs.org/docs/app/api-reference/components/image#onload)
*   **\[弃用\]** `next/image` 的 `domains` 已弃用，转而采用 [`remotePatterns`](https://nextjs.org/docs/app/api-reference/components/image#remotepatterns)
*   **\[特性\]** 可以启用更冗长的 `fetch` 缓存日志 ([Docs](https://nextjs.org/docs/app/api-reference/next-config-js/logging))
*   **\[改进\]** 基础 `create-next-app` 应用程序的函数大小缩小 80%
*   **\[改进\]** 在开发中使用 `edge` 运行时增强的内存管理
