Next.js 16.2 包含多项针对 AI 辅助开发的改进。这些更改使智能体更容易理解您的项目，从终端调试问题，并检查运行中的应用——所有操作都不需要浏览器。

*   [**智能体就绪 (Agent-ready) `create-next-app`**](#ai-ready-project-setup)：开箱即用搭建智能体就绪的项目
*   [**浏览器日志转发**](#browser-log-forwarding)：将浏览器错误转发到终端，以便进行智能体驱动的调试
*   [**开发服务器锁定文件**](#dev-server-lock-file)：当第二个开发服务器尝试启动时，提供可操作错误消息
*   [**实验性智能体开发工具**](#experimental-agent-devtools)：让 AI 智能体通过终端访问 React DevTools 和 Next.js 诊断

智能体就绪的 `create-next-app`[](#agent-ready-create-next-app)
-------------------------------------------------------------

`create-next-app`现在默认包含一个 `AGENTS.md` 文件，让 AI 编码代理从项目开始时即可访问版本匹配的 Next.js 文档。

这基于我们对 [关于 `AGENTS.md` 的研究](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals)，发现在 AI 代理评估中，提供捆绑文档的访问使智能体通过 Next.js 评估的速通率达到 100% —— 优于基于技能的最高 79% 的方法。关键见解：始终可用的上下文优于按需检索，因为代理通常无法识别何时应搜索文档。

`AGENTS.md` 文件是一个简短的指示，告诉代理在编写任何代码之前阅读 `node_modules/next/dist/docs/` 中捆绑的文档。Next.js npm 包现在包括完整的文档作为纯 Markdown 文件，本地提供准确的版本匹配参考，无需获取外部数据。

对于现有项目，设置取决于您的 Next.js 版本。

**在 16.2 或更高版本中**，文档已捆绑在 `next` 包中。将这些两个文件添加到项目根目录：

AGENTS.md

```
<!-- BEGIN:nextjs-agent-rules -->
 
# Next.js: ALWAYS read docs before coding
 
Before any Next.js work, find and read the relevant doc in `node_modules/next/dist/docs/`. Your training data is outdated — the docs are the source of truth.
 
<!-- END:nextjs-agent-rules -->
```

注释标记分隔了 Next.js 管理的部分。您可以在它们之外添加您自己的项目特定指令 —— 更新只会替换标记之间的内容。

`CLAUDE.md` 是 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 的说明文件。`@` 指令告诉它包含 `AGENTS.md` 作为额外上下文：

CLAUDE.md

```
@AGENTS.md
```

**在更早的版本中**，使用 codemod 自动生成这些文件：

终端

```
npx @next/codemod@latest agents-md
```

有关更多详细信息，请参阅 [AI 智能体设置指南](/docs/app/guides/ai-agents#existing-projects)。

浏览器日志转发 [](#browser-log-forwarding)
-------------------------------------------------

Next.js 现在默认在开发期间将浏览器错误转发到终端，这样您可以在不切换到浏览器控制台的条件下查看客户端错误。这对于主要通过终端运行的 AI 智能体特别有用，它们无法访问浏览器控制台。

默认情况下，仅错误被转发到终端。您可以通过 `next.config.ts` 中的 `logging.browserToTerminal` 控制转发级别：

next.config.ts

```
const nextConfig = {
  logging: {
    browserToTerminal: true,
    // 'error' — errors only (default)
    // 'warn'  — warnings and errors
    // true    — all console output
    // false   — disabled
  },
};
 
export default nextConfig;
```

开发服务器锁定文件 [](#dev-server-lock-file)
---------------------------------------------

Next.js 现在将运行中开发服务器的 PID、端口和 URL 写入 `.next/dev/lock` 文件。当同一项目目录中启动第二个 `next dev` 进程时，Next.js 读取锁定文件并打印可操作错误：

终端

```
Error: Another next dev server is already running.
 
- Local:        http://localhost:3000
- PID:          12345
- Dir:          /path/to/project
- Log:          .next/dev/logs/next-development.log
 
Run kill 12345 to stop it.
```

这对 AI 编码智能体特别有用，它们经常尝试启动 `next dev` 而不知道服务器已经在运行。结构化错误为智能体提供了用于终止现有进程的 PID，或用于连接的 URL —— 无需人工干预。

锁定文件还防止两个 `next build` 进程同时运行，否则会导致构建产物损坏。

实验性智能体开发工具 [](#experimental-agent-devtools)
-----------------------------------------------------------

上述功能帮助智能体了解您的项目并调试问题。[`@vercel/next-browser`](https://github.com/vercel-labs/next-browser) 通过让智能体检查正在运行的 Next.js 应用程序扩展了这一功能。

`next-browser` 是一个实验性 CLI，提供浏览器级别的数据 —— 截图、网络请求、控制台日志 —— 以及来自 React DevTools 和 Next.js 开发覆盖层的框架特定见解，例如组件树、属性、钩子、[部分预渲染 (PPR)](/docs/app/getting-started/cache-components) 外壳和错误。所有结果均作为结构化文本通过 shell 命令返回。

LLM 无法读取 DevTools 面板，但可以运行 `next-browser tree`，解析输出，并决定下一步检查什么。每个命令都是针对持久浏览器会话的一次性请求，因此智能体可以反复查询应用程序而无需管理浏览器状态。这使得浏览器成为智能体可以推理的对象，而不是它无法访问的 UI。

### 它今天能做什么 [](#what-it-can-do-today)

功能集正在快速演变。截至本次发布，`next-browser` 支持：

*   **检查 React 组件树** —— 查看属性、钩子、状态和源映射文件位置
*   **分析 PPR 外壳** —— 识别静态与动态区域以及被阻塞的 [Suspense 边界](/docs/app/glossary#suspense-boundary)
*   **访问错误和日志** —— 从开发服务器检索构建和运行时问题
*   **监控网络活动** —— 跟踪导航后的请求，包括服务器操作
*   **捕获视觉效果** —— 截图或记录加载电影胶卷

### 开始使用 [](#getting-started)

将其作为 [技能](https://skills.sh) 安装（AI 智能体的可重用能力）：

终端

```
npx skills add vercel-labs/next-browser
```

然后在 Claude Code、Cursor 或任何支持技能的 AI 智能体中输入 `/next-browser`。CLI 管理带有预加载 React DevTools 的 Chromium 实例 —— 无需浏览器配置。

### 示例：扩大静态外壳 [](#example-growing-the-static-shell)

使用 [部分预渲染 (PPR)](/docs/app/getting-started/cache-components)，Next.js 可以从边缘立即提供 [静态外壳](/docs/app/glossary#static-shell) —— 您的页面中不依赖每个请求数据的部分 —— 然后流式传输其余部分。静态外壳中包含的内容越多，用户看到有意义的页面就越快。

在实践中，单个每个请求的获取可能会意外使整个页面动态化。考虑一个带访问计数器器的博客文章：

app/blog/[slug]/page.tsx

```
export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map((post) => ({ slug: post.slug }));
}
 
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
  const views = await getVisitorCount(params.slug); // per-request
 
  return (
    <article>
      <h1>{post.title}</h1>
      <span>{views} views</span>
      <div>{post.content}</div>
    </article>
  );
}
```

每个 slug 都在 `generateStaticParams` 中枚举，所以文章内容可以在构建时进行预渲染。但 `getVisitorCount` 在每个请求上运行 —— 因为它位于组件的顶层，它使 _整个页面_ 动态化。因此，整个页面等待在加载骨架后面，而不是渐进式流式传输。

智能体可以使用 `next-browser` 来诊断这个问题。锁定 PPR 模式仅显示静态外壳 —— 在本例中是 `app/blog/[slug]/loading.tsx` 骨架，因为页面中的任何内容都未进入外壳：

终端

```
next-browser ppr lock          # Enter PPR mode
next-browser goto /blog/hello  # The entire page is a loading skeleton
```

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-16-2-ai/ppr_before_light.png)![](/_next/image?url=https%3A%2F%2Fh8DxKfmAPhn8O0p3.public.blob.vercel-storage.com%2Fstatic%2Fblog%2Fnext-16-2-ai%2Fppr_before_dark.png&w=1920&q=75)

然后智能体运行 `ppr unlock` 以找出哪里出错了：

终端

```
next-browser ppr unlock
```

终端

```
# PPR Shell Analysis
# 1 dynamic hole, 1 static
 
blocked by:
  - getVisitorCount (server-fetch)
    owner: BlogPost at app/blog/[slug]/page.tsx:5
    next step: Push the fetch into a smaller Suspense leaf
```

该报告告诉智能体确切该做什么：`getVisitorCount` 是阻塞点，它位于 `BlogPost` 中，修复方法是将其推入其自己的 [Suspense 边界](/docs/app/glossary#suspense-boundary)。智能体仅包装计数器：

app/blog/[slug]/page.tsx

```
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
 
  return (
    <article>
      <h1>{post.title}</h1>
      <Suspense fallback={<span>— views</span>}>
        <VisitorCount slug={params.slug} />
      </Suspense>
      <div>{post.content}</div>
    </article>
  );
}
```

再次运行 `ppr lock` 确认外壳已扩大 —— 文章内容现在立即可预渲染。仅访问计数器显示回退：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-16-2-ai/ppr_after_light.png)![](/_next/image?url=https%3A%2F%2Fh8DxKfmAPhn8O0p3.public.blob.vercel-storage.com%2Fstatic%2Fblog%2Fnext-16-2-ai%2Fppr_after_dark.png&w=1920&q=75)

`next-browser` 仍在演进，但它指向了一个智能体能够像开发人员一样以相同的可见性调试和优化应用程序的未来。
