过去一年，我们致力于提升 Next.js 代理体验。在此期间，我们构建并下线了浏览器内代理，推出了 MCP 集成，并认识到更好地支持代理的真正答案在于站在代理的角度思考。

### 代理无法查看浏览器 [](#agents-couldnt-see-the-browser)

今年夏天早些时候，我们正在改进 Next.js 开发工具，注意到了一种模式。开发者会在浏览器中看到错误，复制详细信息，粘贴到 AI 编辑器中，然后要求代理修复它。

问题是代理看不到浏览器。运行时错误、客户端警告和渲染的组件对它们来说都是不可见的。当用户说“修复错误”时，代理不知道他们指的是什么错误。

我们的第一个回应是更新复制按钮以捕获结构化的错误数据。然后我们添加了一个功能，将浏览器日志转发到终端。这些是小的修复，但它们指向了一个更大的认识。我们需要让 Next.js 本身对代理可见。

### 尝试构建浏览器内代理 [](#experimenting-with-an-in-browser-agent)

这引出了一个雄心勃勃的想法。如果我们在 Next.js 内部直接构建一个像智能开发工具一样工作的代理会怎样？

我们构建了一个名为 Vector 的浏览器内聊天代理。类似于 `react-grab` 但与 Next.js 集成，Vector 允许你选择页面上的元素，查看其源代码，并提示更改。它内置了 Next.js 最佳实践，以帮助代理避免幻觉。

![Vector 界面，展示错误检测和建议的修复。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/agentic-future/vector-both-theme.png)

副标题：Vector 界面，展示错误检测和建议的修复。

Vector 很有用，但它与 Cursor 和 Claude Code 等通用编码代理重叠了。大多数开发者本来就会用这些工具来处理所有项目，而不仅仅是 Next.js。UI 选择让用户很容易指出想要更改的确切内容，但这并不是人们每天都需要做的事情。

我们下线了 Vector，但保留了它有用的部分（结构化可视性和框架特定知识），并决定将这些构建到 Next.js 本身中。

### MCP 让 Next.js 状态对代理可见 [](#mcp-made-nextjs-state-visible-to-agents)

2025 年 10 月 Next.js v16 发布前后，用户在使用代理调试时遇到困难。常见的提示是“修复错误”，要求代理解决来自浏览器覆盖层的问题。但代理会请求页面 HTML 却发现没有问题。

运行时故障、浏览器 JavaScript 错误和异步错误都存在于浏览器中，而不是在 HTML 里。渲染的页面、布局段、路由和其他内部状态对代理来说是不可见的。

MCP 给了我们一种暴露这些数据的方法。第一个版本揭示了内部状态，如错误、路由和渲染段，但仅暴露数据还不够。代理还需要发现正在运行的开发服务器并与之通信，这导致了 `next-devtools-mcp`。

![Next.js MCP 响应“什么是 use cache？”的请求，提供文档和上下文。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/agentic-future/answer-dark.png)

副标题：Next.js MCP 响应“什么是 use cache？”的请求，提供文档和上下文。

MCP 还打包了提示和工具，以帮助进行升级和缓存组件迁移。如果你想了解更多，这里有关于 MCP 集成的 [详细演讲](https://www.youtube.com/watch?v=WT7_TJS0tFE)。

### 答案是像代理一样思考 [](#the-answer-is-to-think-like-an-agent)

MCP 证实了 Vector 教给我们的东西。代理需要看到 Next.js 正在做什么，但这只是故事的一部分。更深刻的教训是将代理视为 Next.js 的一等公民，并从他们的角度思考。他们需要什么样的信息？什么时候需要？他们如何消费这些信息？

这种思维模式导致了实际的改进。如果代理在开发期间读取终端输出，那么记录 Server Action 调用并转发浏览器错误可以给它们所需的提示。如果代理在处理训练数据中没有的框架概念时遇到困难，那么嵌入压缩的文档索引 `agents.md` 或提供结构化的工作流（[Next.js skills](https://github.com/vercel-labs/next-skills)）能比仅文档给它们提供更好的上下文。

这些问题贯穿了我们的所有构建。对可视性的需求导致了更好的日志记录，对知识的需求导致了 agents.md，对发现的需求导致了 MCP。当你将代理视为一等公民并在它们所在的地方与它们相遇时，调试就变成了代码、运行时和 AI 之间紧密的反馈循环。

### 接下来要做什么 [](#whats-next)

我们现在致力于使其更易于采用。你现在可以运行 `npx @next/codemod` 为你的项目生成最新的文档索引，我们正在扩展评估套件以涵盖更多的 Next.js 16 API，以便我们可以衡量什么真正有助于代理。长期来看，我们希望将其内置到 `next dev` 中，这样代理无需任何设置即可自动获得正确的上下文。

我们很期待听到你的反馈和关于如何让 Next.js 与代理配合更好的想法。
