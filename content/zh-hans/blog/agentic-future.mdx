我们在过去的一年中不断改进 Next.js 智能体体验。在这个过程中，我们构建并下线了一个在浏览器内的智能体，推出了 MCP 集成，并且学到了真正的解决方案——即为了让智能体得到更好的支持，我们需要从智能体的视角出发。

### 智能体无法看到浏览器 [](#agents-couldnt-see-the-browser)

今年夏天早些时候，我们正在改进 Next.js devtools，此时注意到一个模式。开发者会在浏览器中看到错误，复制详情，粘贴到 AI 编辑器中，并让智能体修复它。

问题是智能体无法看到浏览器。运行时错误、客户端警告和渲染组件对它们来说都是隐形的。当用户说“修复错误”时，智能体不知道他们指的是什么错误。

我们的第一个响应是更新复制按钮以捕获结构化错误数据。接着我们添加了一个功能，将浏览器日志转发到终端。这些都是小修复，但它们指向了一个更大的认识。我们需要让 Next.js 本身对智能体可见。

### 尝试构建浏览器内智能体 [](#experimenting-with-an-in-browser-agent)

这引出了一个雄心勃勃的想法。如果我们直接在 Next.js 内部构建一个像智能 devtools 一样工作的智能体呢？

我们构建了一个名为 Vector 的浏览器内聊天智能体。它类似于 `react-grab` 但是与 Next.js 集成，Vector 允许您选择页面上的元素，查看其源代码，并提示更改。它内置了 Next.js 最佳实践，以帮助智能体避免幻觉。

![Vector 界面显示错误检测和建议修复。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/agentic-future/vector-both-theme.png)

_副标题：Vector 界面显示错误检测和建议修复。_

Vector 是有用的，但它与 Cursor 和 Claude Code 等通用编码智能体重叠。大多数开发者已经在所有项目中使用这些工具了，而不仅仅是在 Next.js 中。UI 选择让它很容易指向您想更改的确切内容，但这并不是人们每天都需要的事情。

我们下线了 Vector，但保留了让它有用的部分（结构化可见性和特定框架的知识），并决定将这些构建到 Next.js 本身中。

### MCP 让 Next.js 状态对智能体可见 [](#mcp-made-nextjs-state-visible-to-agents)

在 2025 年 10 月 Next.js v16 发布前后，用户在尝试使用智能体调试时遇到了困难。常见的提示是“修复错误”，要求智能体从浏览器覆盖层中解决问题。但智能体会请求页面 HTML 并发现没有问题。

运行时故障、浏览器 JavaScript 错误和异步错误都存在于浏览器中，而不是 HTML 中。渲染的页面、布局段、路由和其他内部状态对智能体来说是不可见的。

MCP 为我们提供了一种暴露这些数据的方法。第一版展示了内部状态，如错误、路由和渲染段，但仅暴露数据还不够。智能体还需要发现运行的开发服务器并与它们通信，这导致了 `next-devtools-mcp`。

![Next.js MCP 响应“什么是 use cache？”并带有文档和上下文。](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/agentic-future/answer-dark.png)

_副标题：Next.js MCP 响应“什么是 use cache？”并带有文档和上下文。_

MCP 还打包了解提示和工具，以帮助进行升级和缓存组件迁移。如果您想了解更多，这里有一个关于 MCP 集成的 [详细谈话](https://www.youtube.com/watch?v=WT7_TJS0tFE)。

### 答案是要像智能体一样思考 [](#the-answer-is-to-think-like-an-agent)

MCP 印证了 Vector 教给我们的东西。智能体需要看到 Next.js 在做什么，但这只是故事的一部分。更深层的教训是将智能体视为 Next.js 的一等公民，并从其视角思考。他们需要什么信息？什么时候需要？他们如何消费这些信息？

这种心态促成了实际的改变。如果智能体在开发过程中读取终端输出，那么记录 Server Action 调用和转发浏览器错误就能提供他们需要的提示。如果智能体在训练数据中没有的框架概念上感到挣扎，那么嵌入压缩的文档索引 `agents.md` 或提供结构化的工作流程（[Next.js 技能](https://github.com/vercel-labs/next-skills)）能比仅文档提供更好的上下文。

这些问题贯穿了我们构建的每一个部分。对可见性的需求带来了更好的日志记录，对知识的需求带来了 agents.md，对发现的需求带来了 MCP。当你把智能体视为一等公民并在它们存在的地方与它们相遇时，调试就成为了代码、运行时和 AI 之间的紧密反馈循环。

### 接下来是什么 [](#whats-next)

我们现在致力于使这更容易上手。你已经可以运行 `npx @next/codemod` 来为你的项目生成最新的文档索引，并且我们正在扩展评估套件以覆盖更多的 Next.js 16 API，以便我们可以衡量什么真正帮助了智能体。长期来看，我们想让这个构建到 `next dev` 中，以便智能体能自动获得正确的上下文，而无需任何设置。

我们很期待听到您的反馈和想法，关于如何让 Next.js 与智能体配合得更好。
