## 使用 Codemod CLI 实现平滑升级[](#smooth-upgrades-with-codemod-cli)

我们在每个主要 Next.js 版本发布中都包含 codemod（自动代码转换），以帮助自动化升级破坏性更改。

为了让升级更加平滑，我们发布了增强的 codemod CLI：

```
npx @next/codemod@canary upgrade rc
```

此工具可帮助您将代码库升级到最新的稳定或预发布版本。CLI 将更新您的依赖项、显示可用的 codemod 并指导您应用它们。命令行上指定的 dist tag（`@rc`, `@canary` 等）决定了要升级到的版本。

## Async 请求 APIs（破坏性更改）[](#async-request-apis-breaking-change)

在传统的服务端渲染 (Server-Side Rendering) 中，服务器在渲染任何内容之前会等待请求。然而，并非所有组件都依赖特定请求的数据，因此无需等待请求即可渲染它们。理想情况下，服务器会在请求到达之前尽可能多地做好准备。为了实现这一点，并为未来的优化铺平道路，我们需要知道何时等待请求。

因此，我们将依赖特定请求数据的 API——如 `headers`, `cookies`, `params` 和 `searchParams`——过渡为**异步**。

```
import { cookies } from 'next/headers';
 
export async function AdminPanel() {
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
 
  // ...
}
```

这是一个**破坏性更改**，影响以下 APIs：

*   `cookies`
*   `headers`
*   `draftMode`
*   `params` (在 `layout.js`, `page.js`, `route.js`, `default.js`, `generateMetadata` 和 `generateViewport` 中)
*   `searchParams` (在 `page.js` 中)

为了方便迁移，这些 APIs 可以暂时以同步方式访问，但在下一个主要版本之前，它们将在开发和生产环境中显示警告。有一个 [codemod](/docs/canary/app/building-your-application/upgrading/codemods) 可以自动化迁移：

```
npx @next/codemod@canary next-async-request-api .
```

对于 codemod 无法完全迁移您代码的情况，请阅读 [升级指南](/docs/canary/app/building-your-application/upgrading/version-15)。我们还提供了关于如何将 Next.js 应用程序迁移到新 APIs 的 [示例](https://github.com/leerob/next-saas-starter/pull/62)。

## Server Actions 增强安全性[](#enhanced-security-for-server-actions)

[Server Actions](https://react.dev/reference/rsc/server-actions) 是可从客户端调用的服务器端函数。通过在文件顶部添加 `'use server'` 指令并导出一个异步函数来定义它们。

即使 Server Action 或实用函数未在代码的其他地方导入，它仍然是一个公开可访问的 HTTP 端点。虽然这种行为在技术上正确，但它可能会导致此类函数的意外暴露。

为了提高安全性，我们引入了以下增强功能：

*   **死代码消除**：未使用的 Server Actions 不会将其 ID 暴露给客户端 JavaScript 包，减少包大小并提高性能。
*   **安全操作 ID**：Next.js 现在创建不可猜测的、非确定性的 ID，允许客户端引用和调用 Server Action。这些 ID 会在构建之间定期重新计算以增强安全性。

```
// app/actions.js
'use server';
 
// 此操作 **是** 在我们的应用程序中使用，因此 Next.js
// 会创建一个安全 ID 以允许客户端引用
// 和调用 Server Action。
export async function updateUserAction(formData) {}
 
// 此操作 **未** 在我们的应用程序中使用，因此 Next.js
// 将在 `next build` 期间自动删除此代码
// 并不会创建公开端点。
export async function deleteUserAction(formData) {}
```

您仍应将 Server Actions 视为公开 HTTP 端点。了解有关 [保护 Server Actions](https://nextjs.org/blog/security-nextjs-server-components-actions#write) 的更多信息。

## `<Form>` 组件[](#form-component)

新的 `<Form>` 组件通过 [预取](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)、[客户端导航](/docs/app/building-your-application/routing/linking-and-navigating#5-soft-navigation) 和渐进式增强扩展了 HTML `<form>` 元素。

它适用于导航到新页面的表单，例如引导至结果页面的搜索表单。

```
import Form from 'next/form';
 
export default function Page() {
  return (
    <Form action="/search">
      <input name="query" />
      <button type="submit">Submit</button>
    </Form>
  );
}
```

`<Form>` 组件带有：

*   **预取**: 当表单在视图中时，[layout](/docs/app/api-reference/file-conventions/layout) 和 [loading](/docs/app/api-reference/file-conventions/loading) UI 会被预取，使导航更快。
*   **客户端导航**: 提交时，共享布局和本地状态被保留。
*   **渐进式增强**: 如果 JavaScript 尚未加载，表单仍可通过全页导航工作。

以前，实现这些功能需要大量手动样板代码。例如:

### 示例

```
// 注意：为了演示目的，此为缩写。
// 不建议在生产代码中使用。
 
'use client'
 
import { useEffect } from 'react'
import { useRouter } from 'next/navigation'
 
export default function Form(props) {
  const action = props.action
  const router = useRouter()
 
  useEffect(() => {
    // 如果表单 target 是 URL，预取它
    if (typeof action === 'string') {
      router.prefetch(action)
    }
  }, [action, router])
 
  function onSubmit(event) {
    event.preventDefault()
 
    // 获取所有表单字段并触发带有数据 URL 编码的 `router.push`
    const formData = new FormData(event.currentTarget)
    const data = new URLSearchParams()
 
    for (const [name, value] of formData) {
      data.append(name, value as string)
    }
 
    router.push(`${action}?${data.toString()}`)
  }
 
  if (typeof action === 'string') {
    return <form onSubmit={onSubmit} {...props} />
  }
 
  return <form {...props} />
}
```

## `instrumentation.js`（稳定）[](#instrumentationjs-stable)

`instrumentation` 文件具有 `register()` API，允许用户切入 Next.js 服务器生命周期以监控性能、跟踪错误来源以及深度集成到像 [OpenTelemetry](https://opentelemetry.io/) 这样的可观测性库中。

此功能现在**稳定**，可以移除 `experimental.instrumentationHook` 配置选项。

此外，我们还与 [Sentry](https://sentry.io/) 合作设计了一个新的 `onRequestError` 钩子，可用于：

*   捕获服务器上传抛出的所有错误的重要上下文，包括：
    *   Router：Pages Router 或 App Router
    *   Server context：Server Component、Server Action、Route Handler 或 Middleware
*   向您喜爱的可观测性提供商报告错误。

```
export async function onRequestError(err, request, context) {
  await fetch('https://...', {
    method: 'POST',
    body: JSON.stringify({ message: err.message, request, context }),
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export async function register() {
  // 初始化您喜爱的可观测性提供商 SDK
}
```

了解有关 `onRequestError` [函数](/docs/canary/app/api-reference/file-conventions/instrumentation#onrequesterror-optional) 的更多信息。

## 开发和构建改进[](#development-and-build-improvements)

### Server Components HMR[](#server-components-hmr)

开发期间，Server Components 会在保存时重新执行。这意味着，任何对您 API 端点或第三方服务的 `fetch`请求也会被调用。为了提高本地开发性能并减少已计费 API 调用的潜在成本，我们确保热模块替换 (HMR) 可以重用先前渲染的 `fetch` 响应。

了解有关 [Server Components HMR Cache](/docs/canary/app/api-reference/next-config-js/serverComponentsHmrCache) 的更多信息。

### App Router 更快的静态生成[](#faster-static-generation-for-the-app-router)

我们优化了静态生成以提高构建时间，特别是对于那些具有缓慢网络请求的页面。

此前，我们的静态优化过程渲染页面两次——一次用于生成客户端导航的数据，第二次用于渲染初始页面访问的 HTML。现在，我们重用第一次渲染，取消了第二次遍历，减少了工作负载和构建时间。

此外，静态生成工作线程现在在页面之间共享 `fetch` 缓存。如果 `fetch` 调用没有选择退出缓存，其结果将由同一工作线程处理的其他页面重用。这减少了对相同数据的请求次数。

### 高级静态生成控制（实验）[](#advanced-static-generation-control-experimental)

我们增加了对静态生成流程控制的实验支持，供高级用例从中获益，这些用例需要更大的控制权。

除非您有特定需求，否则建议坚持使用当前的默认值，因为这些选项可能会导致资源使用增加以及由于并发增加而导致的潜在内存不足错误。

```
const nextConfig = {
  experimental: {
	  // Next.js 在构建失败前会重试失败的页面生成尝试次数
    staticGenerationRetryCount: 1
    // 每个工作线程处理多少个页面
    staticGenerationMaxConcurrency: 8
    // 启动新的导出工作线程前最少的页面数量
    staticGenerationMinPagesPerWorker: 25
  },
}
 
export default nextConfig;
```

了解有关 [Static Generation options](/docs/canary/app/api-reference/next-config-js/staticGeneration) 的更多信息。

## 自托管改进[](#improvements-for-self-hosting)

当自托管应用时，您可能需要更多的 `Cache-Control` 指令控制权。

一个常见的用例是控制 ISR 页面发送的 `stale-while-revalidate` 周期。我们实施了两个改进：

1.  您现在可以在 `next.config` 中配置 [`expireTime`](/docs/canary/app/api-reference/next-config-js/expireTime) 值。这以前是 `experimental.swrDelta` 选项。
2.  更新默认值为一年，确保大多数 CDN 可以按预期完全应用 `stale-while-revalidate` 语义。

我们也不再覆盖自定义 `Cache-Control` 值以使用我们的默认值，实现完全控制并确保与任何 CDN 设置兼容。

最后，我们改进了自托管时的图像优化。以前，我们建议您安装 `sharp` 来优化 Next.js 服务器上的图像。这个建议有时会被忽略。随着 Next.js 15，您不再需要手动安装 `sharp`——在使用 `next start` 或运行 [standalone output mode](/docs/app/api-reference/next-config-js/output) 时，Next.js 将自动使用 `sharp`。

有关更多信息，请参阅我们关于自托管 Next.js 的新 [演示和教程视频](https://x.com/leeerob/status/1843796169173995544)。

## ESLint 9 支持[](#eslint-9-support)

Next.js 15 还引入了对 [ESLint 9](https://eslint.org/blog/2024/04/eslint-v9.0.0-released) 的支持，这紧随 2024 年 10 月 5 日 ESLint 8 的生命周期结束。为了确保平滑过渡，Next.js 保持向后兼容，这意味着您可以继续使用 ESLint 8 或 9。

如果您升级到 ESLint 9，并且我们检测到您尚未采用 [新的配置格式](https://eslint.org/blog/2024/04/eslint-v9.0.0-released/#flat-config-is-now-the-default-and-has-some-changes)，Next.js 将自动应用 `ESLINT_USE_FLAT_CONFIG=false` 退出机制以促进迁移。

此外，运行 `next lint` 时，弃用选项如 `—ext` 和 `—ignore-path` 将被移除。请注意 ESLint 最终将在 ESLint 10 中禁止这些较旧的配置，因此我们建议您尽快开始迁移。

有关这些变更的更多详细信息，请查看 [迁移指南](https://eslint.org/docs/latest/use/migrate-to-9.0.0)。

作为此更新的一部分，我们还已将 `eslint-plugin-react-hooks` 升级到 `v5.0.0`，其中引入了针对 React Hooks 使用的新规则。您可以在 [eslint-plugin-react-hooks@5.0.0 的更新日志](https://github.com/facebook/react/releases/tag/eslint-plugin-react-hooks%405.0.0) 中查看所有更改。
