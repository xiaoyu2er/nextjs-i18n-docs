我们为 Next.js 的未来奠定了基础：*   [**中间件（稳定版）(Middleware (Stable)):**](#middleware-stable) 您的整个应用程序的动态路由。

*   [**按需增量静态再生（On-Demand ISR，稳定版）(On-Demand Incremental Static Regeneration (Stable)):**](#on-demand-incremental-static-regeneration-stable) 无需重新部署即可更新内容。
*   [**边缘 API 路由（实验性）(Edge API Routes (Experimental)):**](#edge-api-routes-experimental) 高性能 API 端点。
*   [**边缘服务端渲染（实验性）(Edge SSR (Experimental)):**](#edge-server-rendering-experimental) 在边缘渲染您的应用程序。
*   [**SWC 插件（实验性）(SWC Plugins (Experimental)):**](#swc-plugins-experimental) 使用您自己的插件扩展编译功能。

今天通过运行 `npm i next@latest` 进行更新。

中间件（稳定版）(Middleware (Stable))[](#middleware-stable)
-----------------------------------------

我们很高兴宣布中间件 (Middleware) 在 12.2 中现已稳定，并基于用户反馈改进了 API。

中间件允许您在请求完成之前运行代码。基于传入的请求，您可以通过重写、重定向、添加标头或设置 Cookie 来修改响应。

middleware.ts

```
import { NextRequest, NextResponse } from 'next/server';
 
// If the incoming request has the "beta" cookie
// then we'll rewrite the request to /beta
export function middleware(req: NextRequest) {
  const isInBeta = JSON.parse(req.cookies.get('beta') || 'false');
  req.nextUrl.pathname = isInBeta ? '/beta' : '/';
  return NextResponse.rewrite(req.nextUrl);
}
 
// Supports both a single value or an array of matches
export const config = {
  matcher: '/',
};
```

要更新中间件的最新 API 更改，请参阅 [迁移指南](/docs/messages/middleware-upgrade-guide)。

尝试在 [Vercel 上免费使用](https://vercel.com/edge) 中间件，或使用 `next start` [自行托管](/docs/pages/building-your-application/deploying#self-hosting) 时尝试。

按需增量静态再生（On-Demand ISR，稳定版）(On-Demand Incremental Static Regeneration (Stable))[](#on-demand-incremental-static-regeneration-stable)
-------------------------------------------------------------------------------------------------------

按需增量静态再生 (On-Demand ISR) 允许您更新网站内容而无需重新部署。这使得在数据头内容管理系统或电子商务平台更改时，可以轻松立即更新您的网站。这是社区最请求的功能之一，我们很高兴它现在已稳定。

pages/api/revalidate.js

```
export default async function handler(req, res) {
  // Check for secret to confirm this is a valid request
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' });
  }
 
  try {
    await res.revalidate('/path-to-revalidate');
    return res.json({ revalidated: true });
  } catch (err) {
    // If there was an error, Next.js will continue
    // to show the last successfully generated page
    return res.status(500).send('Error revalidating');
  }
}
```

增量静态再生适用于任何支持 [Next.js 构建 API](/docs/pages/building-your-application/deploying#nextjs-build-api) (`next build`) 的提供者。当部署到 [Vercel](https://vercel.com/) 时，按需验证在将页面推送到边缘时会在 ~300ms 内全球传播。

更多信息，请 [查看文档](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration) 或 [查看我们的演示](https://on-demand-isr.vercel.app/) 以查看按需验证的运行效果。

边缘 API 路由（实验性）(Edge API Routes (Experimental))[](#edge-api-routes-experimental)
---------------------------------------------------------------

Next.js 现在也支持为 API 路由使用 [边缘运行时](/docs/pages/api-reference/edge)。边缘运行时 (Edge Runtime) 是比 Node.js 更轻量的运行时，为低延迟提供快速启动。此外，边缘 API 路由支持来自服务器的流式响应。

您可以在 `config` 中为 API 路由设置运行时，指定 `nodejs` (默认) 或 `experimental-edge`：

pages/api/hello.js

```
import type { NextRequest } from 'next/server';
 
export default (req: NextRequest) => {
  return new Response(`Hello, from ${req.url} I'm now an Edge API Route!`);
};
 
export const config = {
  runtime: 'experimental-edge',
};
```

由于边缘运行时很轻量，它有局限性以适应快速启动 —— 例如，它不支持 Node.js 特定的 API，如 `fs`。因此，API 路由的默认运行时仍然是 `nodejs`。

更多信息，请 [查看文档](/docs/pages/building-your-application/routing/api-routes)。

边缘服务端渲染（实验性）(Edge Server Rendering (Experimental))[](#edge-server-rendering-experimental)
---------------------------------------------------------------------------

Next.js 现在支持为服务端渲染使用 [边缘运行时](/docs/pages/api-reference/edge)。

如上所述，边缘运行时比 Node.js 轻量，为低延迟提供快速启动。与 React 18 一起使用时，它为页面启用流式服务端渲染。

Next.js 使用 Node.js 作为服务端渲染页面的默认运行时。从 12.2 开始，如果您使用 React 18，您可以选择使用边缘运行时。

您可以在 `next.config.js` 中全局设置运行时，指定 `nodejs` 或 `experimental-edge`：

next.config.js

```
module.exports = {
  experimental: {
    runtime: 'experimental-edge',
  },
};
```

更改默认页面运行时会影响到所有页面，包括 [SSR 流式传输](/docs/app/building-your-application/routing/loading-ui-and-streaming) 和 [服务端组件 (Server Components)](/docs/getting-started/react-essentials) 功能。您也可以逐个页面覆盖此默认设置，通过导出 `runtime` 配置：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
export const config = {
  runtime: 'nodejs',
};
 
export default function Home() {}
```

您可以通过查看运行时的 `process.env.NEXT_RUNTIME` 环境变量来检测您当前使用的运行时，并在 webpack 编译期间检查 `options.nextRuntime` 变量。

更多信息，请 [查看文档](/docs/pages/building-your-application/rendering/edge-and-nodejs-runtimes)。

对 `next/image` 的改进 (Improvements to `next/image`)[](#improvements-to-nextimage)
----------------------------------------------------------

### `next/future/image` 组件（实验性）(`next/future/image` component (Experimental))[](#nextfutureimage-component-experimental)

我们听取了您对当前 Image 组件的反馈，并很高兴分享新版 `next/image` 的早期预览。这个新的改进版 Image 组件需要更少的客户端 JavaScript，并简化了如何样图片的方式：

*   渲染单个 `<img>`，不带 `<div>` 或 `<span>` 包装器
*   添加了原生的 `style` 属性支持
*   移除了 `layout`, `objectFit`, 和 `objectPosition` 属性，改用 `style` 或 `className`
*   移除了 `IntersectionObserver` 实现，改用 [原生懒加载](https://caniuse.com/loading-lazy-attr)
*   移除了 `loader` 配置，改用 `loader` 属性
*   注意：暂无 `fill` 模式，所以需要 `width` & `height` 属性

这提高了性能，因为原生的 `loading="lazy"` 不需要等待 React 水合和客户端 JavaScript。

更多信息，请 [查看文档](/docs/pages/building-your-application/optimizing/images)。

### 远程模式（实验性）(Remote Patterns (Experimental))[](#remote-patterns-experimental)

`next/image` 现在支持一个实验性配置选项 `remotePatterns`，允许您在使用内置图像优化 API 时为远程图像指定通配符。这允许更强大的匹配，超越现有的 [`images.domains`](http://images.domains) 配置，该配置仅对域名执行精确匹配。

next.config.js

```
module.exports = {
  experimental: {
    images: {
      remotePatterns: [
        {
          // The `src` property hostname must end with `.example.com`,
          // otherwise this will respond with 400 Bad Request.
          protocol: 'https',
          hostname: '**.example.com',
        },
      ],
    },
  },
};
```

更多信息，请 [查看文档](/docs/pages/api-reference/components/image#remotepatterns)。

### 禁用图像优化 (Disable Image Optimization)[](#disable-image-optimization)

零配置图像优化 API 阻止了 `next export` 的使用，因为它需要一个服务器来在请求时按需优化图像。直到今天，目标是 `next export` 的用户需要配置 `loader` 以使用第三方图像优化提供者，但如果没有任何提供者，则没有清晰的解决方案。从今天开始，`next export` 用户可以使用新的配置属性为所有 `next/image` 实例禁用图像优化：

next.config.js

```
module.exports = {
  experimental: {
    images: {
      unoptimized: true,
    },
  },
};
```

SWC 插件（实验性）(SWC Plugins (Experimental))[](#swc-plugins-experimental)
-------------------------------------------------------

[Next.js 编译器](/docs/architecture/nextjs-compiler) 使用 [SWC](https://swc.rs/) 来转换和压缩您的生产 JavaScript 代码。SWC 是在 Next.js 12.0 中引入的，以改善本地开发和构建性能。

您现在可以添加插件（用 WebAssembly 编写）来在编译期间自定义 SWC 转换行为：

next.config.js

```
module.exports = {
  experimental: {
    swcPlugins: [
      ['css-variable/swc', { displayName: true, basePath: __dirname }],
    ],
  },
};
```

更多信息，请 [查看文档](https://github.com/vercel/next.js/blob/canary/docs/architecture/nextjs-compiler.md#experimental-swc-plugin-support)。

React 18 支持改进 (React 18 Support Improvements)[](#react-18-support-improvements)
---------------------------------------------------------------

*   改进了对 CSS-in-JS 库（如 `styled-components` 和 `emotion`）的支持，升级体验更流畅且无破坏性更改。
*   现在正确支持 AMP 和 HTML 后优化（CSS、字体优化）。
*   `next/head` 现在支持 React 18。
*   Next.js 的路由公告器（用于向屏幕阅读器和辅助技术正确通告页面转换）现在支持 React 18。

[其他改进](#other-improvements)
-----------------------------------------

*   支持 Next.js 编译器中的 Emotion 转换。`@emotion/babel-plugin` 的大部分功能现在都已支持，除非您使用 `importMap`，否则它可以被移除。更多信息，请 [查看文档](/docs/architecture/nextjs-compiler#emotion)。
*   Next.js 编译器中对 `styled-components` 转换的支持更好，允许自定义默认选项，包括 `cssProp` 选项。更多信息，请 [查看文档](/docs/architecture/nextjs-compiler#styled-components)。
*   更好地支持 JavaScript ES 模块，因此 `next/image` 和 `next/link` 等组件可以正确导入。
*   `next/link` 不再需要手动添加 `<a>` 作为子元素。您现在可以以向后兼容的方式 [选择此行为](https://github.com/vercel/next.js/pull/36436)。
*   我们通过修改 `browsersList` 添加了对仅发送现代 JavaScript 的实验性支持。您可以通过在 `next.config.js` 的 `experimental` 选项中设置 `browsersListForSwc: true` 和 `legacyBrowsers: false` 来选择此行为。
*   新的 `@swc/helpers` 优化防止了跨捆绑包的重复，在最小配置中减少了约 `2KB` 的捆绑大小，并且在更大的应用中更多
*   我们显著减少了 Next.js 的安装大小。我们通过将 monorepo 移动到 `pnpm` 来实现这一点，这允许我们在创建我们使用的预编译版本时删除重复的包。这导致安装大小减少了 14MB。
*   在我们不断提高自行托管 Next.js 的持续努力中，我们正在将实验性 `outputStandalone: true` 配置稳定为 `output: 'standalone'`。此配置通过仅包含必要的文件/资产，大幅减少了部署大小，包括无需在安装 `node_modules` 包中安装所有内容。此配置可以在我们的 [`with-docker` 示例](https://github.com/vercel/next.js/blob/canary/examples/with-docker/README.md) 中看到。
布局 RFC 和高级路由支持 (Layouts RFC & Advanced Routing Support)[](#layouts-rfc--advanced-routing-support)

布局 (Layouts) RFC & 高级路由支持 (Advanced Routing Support) [](#layouts-rfc--advanced-routing-support)
--------------------------------------------------------------------------------

如果您错过了，上个月我们宣布了 [布局 RFC](/blog/layouts-rfc) —— 自 2016 年引入以来 Next.js 最大的更新，包括：

*   **嵌套布局 (Nested Layouts)：** 使用嵌套路由构建复杂应用程序。
*   **专为服务端组件设计 (Designed for Server Components)：** 为子树导航优化。
*   **改进的数据获取 (Improved Data Fetching)：** 在布局中获取数据，同时避免瀑布效应。
*   **使用 React 18 功能 (Using React 18 Features)：** 流式传输、转换和 Suspense。
*   **客户端和服务端路由 (Client and Server Routing)：** 以服务端为中心的路由，具有 _SPA-like_ 行为。
*   **100% 可渐进采用 (100% incrementally adoptable)：** 无破坏性更改，因此您可以逐步采用。
*   **高级路由约定 (Advanced Routing Conventions)：** 离线暂存、即时转换等。

更多信息，请 [查看 RFC](/blog/layouts-rfc) 或 [提供反馈](https://github.com/vercel/next.js/discussions/37136)。
