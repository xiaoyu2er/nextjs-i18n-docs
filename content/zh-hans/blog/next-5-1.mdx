我们很高兴介绍 Next.js 5.1，它支持环境配置、阶段、源映射以及新的 Next.js 插件。

引入了重大性能改进：页面解析速度**快了 102 倍**，错误页面加载也更高效。

要升级或安装，请运行：

终端

```
npm i next@latest react@latest react-dom@latest
```

> 除了升级 Next.js，我们还将 peer 依赖项 `react` 和 `react-dom` 一并升级

请务必也升级 [next-plugins](https://github.com/vercel/next-plugins) 类插件，例如 `@zeit/next-css`、`@zeit/next-sass`、`@zeit/next-less` 或 `@zeit/next-typescript`。

更快的页面解析[](#faster-page-resolution)
-------------------------------------------------

得益于 Next.js 5.0 的架构更改，我们得以 [简化了](https://github.com/vercel/next.js/pull/3776) 基于 URL 路径解析页面的逻辑。这些更改基于 [@oliviertassinari](https://github.com/oliviertassinari) 的 [研究](https://github.com/vercel/next.js/issues/3682)。之前解析一个页面平均需要 **2.347 毫秒**。使用新逻辑解析同一个页面平均只需要 **0.023 毫秒**。这意味着 Next.js 应用程序中最常调用的方法之一速度提升了 **102 倍**。

环境配置[](#environment-configuration)
-------------------------------------------------------

典型的 Node.js 环境通常依赖于向应用程序传递环境变量，例如：`API_URL=https://api.vercel.com node index.js`，然后您可以在应用程序的任何地方使用 `process.env.API_URL` 来使用 `API_URL`。

在通用渲染中，客户端不可用 `process.env`。因此，我们在 Next 5.1 中引入了一个新功能：`publicRuntimeConfig` 和 `serverRuntimeConfig`。这些可以在 `next.config.js` 中设置，然后通过 `next/config` 模块使用。

next.config.js

```
module.exports = {
  serverRuntimeConfig: {
    // Will only be available on the server side
    mySecret: 'secret',
  },
  publicRuntimeConfig: {
    // Will be available on both server and client
    staticFolder: '/static',
  },
};
```

> `serverRuntimeConfig` 和 `publicRuntimeConfig` 均在 `next.config.js` 中定义

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import getConfig from 'next/config';
const { serverRuntimeConfig, publicRuntimeConfig } = getConfig();
 
console.log(serverRuntimeConfig.mySecret); // Will only be available on the server side
console.log(publicRuntimeConfig.staticFolder); // Will be available on both server and client
 
export default function Index() {
  return (
    <div>
      <img src={`${publicRuntimeConfig.staticFolder}/logo.png`} />
    </div>
  );
}
```

> `CONFIG` 方法来自 `next/config` 模块，用于获取配置值

改进的错误处理[](#improved-error-handling)
---------------------------------------------------

之前 Next.js 拥有一种特殊的错误处理机制，用于检测加载页面 bundle 时的服务器错误。页面 bundle 是客户端加载以加载页面的 JavaScript 文件，例如 `/_next/-/page/index.js`。

如果发生错误（例如构建 ID 不匹配），页面 bundle 仍会以 200 HTTP 状态码提供，但内容将是 Next.js 服务器生成的错误 JSON 表示。这样做是因为客户端错误处理不仅仅依赖于页面是否为 404。这个解决方案效果很好，直到您尝试将资产上传到不支持回退的静态文件托管服务或 CDN 时。

在 Next.js 5.1 中，我们 [完全重构](https://github.com/vercel/next.js/pull/3849) 了错误处理逻辑，当页面 bundle 返回 404 HTTP 状态码时，路由器将自动检测并重新加载页面，以确保 [多区域](https://github.com/vercel/next.js#multi-zones) 之间的导航是可能的。

在重写此逻辑时，我们 [移除了 `Router.onAppUpdated` 钩子](https://github.com/vercel/next.js/blob/canary/errors/no-on-app-updated-hook.md)；主要用于触发页面重新加载。现在页面将自动重新加载。

此外，我们添加了一组新的 [集成测试](https://github.com/vercel/next.js/pull/3895) 用于开发模式下的错误恢复，以避免未来发布版本的错误恢复回归。

阶段 / 配置文件函数 [](#phases -- config-function)
----------------------------------------------------

一些 [next-plugins](https://github.com/vercel/next-plugins) 如 [`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css) 仅在 Next.js 处于开发模式或运行 `next build` 时需要。

您现在可以导出一个函数来返回配置对象，而不是立即导出对象。

```
module.exports = (phase, { defaultConfig }) => config;
```

> 导出返回用户配置函数的 `next.config.js`

导出函数可以让您访问 Next.js 运行的阶段，例如开发、生产、构建、导出。这允许按需加载插件，但也提供对默认配置的访问。

我们引入了一个新模块 [`next/constants`](https://github.com/vercel/next.js/blob/canary/lib/constants.js)，其中包含常用的常量，包括阶段。

```
const {PHASE_DEVELOPMENT_SERVER} = require('next/constants')
module.exports = (phase, {defaultConfig}){
  if(phase === PHASE_DEVELOPMENT_SERVER) {
    return { /* development only config options here */ }
  }
 
  return { /* config options for all phases except development here */ }
}
```

> 检查开发阶段的 `next.config.js`

改进了生产源映射生成 [](#improved-production-source-map-generation)
---------------------------------------------------------------------------------------

随着 [Next.js 5](/blog/next-5) 中通用 webpack 的引入，向生产环境添加源映射变得像向 `next.config.js` 添加几行代码一样简单：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
      for (const plugin of config.plugins) {
        if (plugin['constructor']['name'] === 'UglifyJsPlugin') {
          plugin.options.sourceMap = true;
          break;
        }
      }
    }
    return config;
  },
};
```

> 在 `next.config.js` 中手动启用源映射

可以添加 [@zeit/next-source-maps](https://github.com/vercel/next-plugins/tree/master/packages/next-source-maps) 到项目中以自动为您启用生产源映射，在 `next.config.js` 中添加以下内容：

```
const withSourceMaps = require('@zeit/next-source-maps');
module.exports = withSourceMaps();
```

> 使用 `@zeit/next-source-maps` 在 `next.config.js` 中启用源映射

这启用了除一个文件 `app.js` 之外所有文件的源映射输出，原因是 `app.js` 由多个文件 (`manifest.js` 和 `commons.js`) 结合 webpack 插件组合而成。这样做有一个副作用，即 webpack 无法为组合文件生成源映射。

得益于 [@ptomasroos](https://github.com/ptomasroos) 的 [Pull Request](https://github.com/vercel/next.js/pull/3793)，`app.js` 文件被 `main.js` 替换。该文件将保存之前编译为 `manifest.js` 和 `commons.js` 的代码，并且 webpack 将为 `main.js` 生成源映射。源映射将自动提供服务，允许外部错误跟踪工具在检测到错误时显示实际的文件和行号。

新插件 / 对现有插件的改进 [](#new-plugins -- improvements-to-existing-ones)
------------------------------------------------------------------------------------------

我们推出了两个新的官方插件。[@zeit/next-bundle-analyzer](https://github.com/vercel/next-plugins/tree/master/packages/next-bundle-analyzer) 允许轻松设置 `webpack-bundle-analyzer` 分别分析服务器端和客户端 bundle。

此外，官方的 [`css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less) 和 [`sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) 插件在热重载和捆绑方面都有了很大的改进。例如，开发模式下不再有未样式内容的闪烁。子组件中的样式也会被捕获。
