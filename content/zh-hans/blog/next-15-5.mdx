Next.js 15.5 包含测试版 Turbopack 构建、稳定的 Node.js 中间件、TypeScript 改进、`next lint` 弃用以及 Next.js 16 弃用警告。本次发布的亮点包括:

*   [**Turbopack 构建 (Beta)**](/blog/next-15-5#turbopack-builds-beta): 生产版 Turbopack 构建 (`next build --turbopack`) 现已进入测试版
*   [**Node.js 中间件 (稳定)**](/blog/next-15-5#nodejs-middleware-stable): 中间件的 Node.js 运行时支持现已稳定
*   [**TypeScript 改进**](/blog/next-15-5#typescript-improvements): 路由类型化、路由导出验证以及路由类型辅助函数
*   [**`next lint`**](/blog/next-15-5#next-lint-deprecation): `next lint` 命令弃用
*   [**Next.js 16**](/blog/next-15-5#deprecation-warnings-for-nextjs-16): Next.js 16 弃用警告

立即升级，或使用以下内容开始：

终端

```
# Use the automated upgrade CLI
npx @next/codemod@canary upgrade latest
 
# ...or upgrade manually
npm install next@latest react@latest react-dom@latest
 
# ...or start a new project
npx create-next-app@latest
```

### Turbopack 构建 (Beta) [](#turbopack-builds-beta)

我们很高兴宣布 `next build --turbopack` 开启测试版。Turbopack 现在已为 Vercel 网站提供动力，包括 [vercel.com](https://vercel.com)、[v0.app](https://v0.app) 和 [nextjs.org](https://nextjs.org)，通过更快的预览和生产部署构建来加速迭代速度。

通过 Turbopack 驱动的这些应用已在上线期间经受过考验，自发布以来已服务超过 **12 亿** 次请求。

### 性能与生产结果 [](#performance--production-results)

Turbopack 最初的设计目标之一是能够帮助开发者随着应用和代码库的增长，扩展构建性能。为此，Turbopack 从底层设计上充分利用了构建所有阶段的多个 CPU 核心。

我们在许多 Vercel 产品中部署了 Turbopack 构建，并观察到编译时间的一致改进：

*   **客户网站**: 在 4 核机器上快 2 倍
*   **客户网站**: 在 14 核机器上快 2.2 倍
*   **小型网站 (10K modules)**: 在 30 核机器上快 4 倍
*   **中型网站 (40K modules)**: 在 30 核机器上快 2.5 倍
*   **大型网站 (70K modules)**: 在 30 核机器上快 5 倍

随着我们将 Turbopack 扩展到主要网站应用，我们密切监控了生产性能回归和破坏性变更。

与 Webpack 相比，我们的生产指标监控显示，使用 Turbopack 构建的网站在更少的请求中服务相同或更少的 JavaScript 和 CSS，从而带来相当或更好的 FCP、LCP 和 TTFB 指标。

对于在开发中已采用 Turbopack 的团队，我们建议现在尝试在构建中使用 Turbopack。

### 已知差异 [](#known-differences)

**小型项目**: 在小型机器或小型项目上，由于 Webpack 内置的持久缓存，我们测量到构建时间的改进很小或中性。我们目前正在为 Turbopack 开发 [持久缓存](https://nextjs.org/docs/15/app/api-reference/config/next-config-js/turbopackPersistentCaching) 解决方案，以实现使所有构建增量和快速的设计目标。

**代码包优化**: 在某些边缘情况下，我们测量到 Webpack 生成了更优化的代码包。我们正在跟踪这些场景，并在稳定发布前努力缩小差距。有关更多信息，请参阅 [关于代码包大小的文档](https://nextjs.org/docs/app/api-reference/turbopack#bundle-sizes)。

**CSS 排序**: 由于 Turbopack 在处理副作用方面的启发式规则不同，CSS 文件的连接顺序可能与 Webpack 不同，导致潜在的视觉差异。有关更详细的解释和建议解决方案，请参阅 [文档](https://nextjs.org/docs/app/api-reference/turbopack#css-module-ordering)。

> **注意**: 这些差异已记录，我们正积极致力于改进。有关详细信息和变通方法，请参阅我们的 [Turbopack 文档](https://nextjs.org/docs/app/api-reference/turbopack#known-gaps-with-webpack)。

随着我们向稳定版本迭代，请务必在 [GitHub](https://github.com/vercel/next.js/discussions/77721) 上报告任何问题。

## Node.js 中间件 (稳定) [](#nodejs-middleware-stable)

在 [15.2 版本](/blog/next-15-2) 引入 Node.js 运行时的实验性支持并在生产应用中广泛测试后，我们很高兴地宣布 Node.js 中间件运行时现已稳定。

以前，Next.js 中间件仅支持 Edge 运行时，虽然提供了更好的性能和隔离性，但在与 Node.js 特定库和 API 集成时存在限制。

middleware.ts

```
import { NextRequest, NextResponse } from 'next/server';
 
export const config = {
  runtime: 'nodejs', // Now stable!
};
 
export function middleware(request: NextRequest) {
  // Access to full Node.js APIs and npm packages
  const fs = require('fs');
  const crypto = require('crypto');
 
  // Complex authentication logic
  const token = request.headers.get('authorization');
 
  if (!isValidToken(token)) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
 
  return NextResponse.next();
}
 
function isValidToken(token: string | null): boolean {
  // Use Node.js crypto for validation
  // Access file system, databases, etc.
  return true;
}
```

这一更改使得中间件能够处理更复杂的用例，同时保持相同的开发者体验。

虽然 Node.js 运行时在 Next.js 16 中不会是默认选项，但我们正在根据社区反馈和使用模式探讨在 Next.js 17 中将其设为默认。

## TypeScript 改进 [](#typescript-improvements)

Next.js 15.5 为 App Router 带来了重大的 TypeScript 改进，具有完全兼容 Turbopack 且全面的代码路由类型安全。

### 路由类型化 (稳定) [](#typed-routes-stable)

路由类型化为您的应用路由提供编译时类型安全，在链接进入生产环境之前就能捕获无效链接。此功能根据您的文件结构自动生成类型，确保每个 `<Link>` 组件都指向有效的路由。

此功能位于 `typedRoutes` 标志的后面，该标志以前是实验性的，现已稳定。静态类型化路由现在通过新的实现方式与 Turbopack 配合使用，该实现方式在 Webpack 和 Turbopack 构建中都提供类型安全：

next.config.ts

```
const nextConfig = {
  typedRoutes: true, // Now stable!
};
 
export default nextConfig;
```

```
import Link from 'next/link'
 
// Full type safety for route paths
<Link href="/blog/example-slug?ui=dark">Read Post</Link>
 
// TypeScript catches invalid routes at compile time
<Link href="/invalid-route">Broken Link</Link> // ← Type error
```

### 路由导出验证 (Turbopack) [](#route-export-validation-turbopack)

路由导出验证也通过在 Turbopack 上运行一个新系统来实现，该系统生成类型守卫文件，验证页面、布局和服务程序，使用 TypeScript 的 `satisfies` 操作符。

这会在运行 `next build` 进行编译期间捕获无效导出，例如错误的 `dynamic` 值，用更高效的解决方案替代旧的 Webpack 插件，该方案在两个构建系统中都有效。

### 路由 Props 辅助函数 [](#route-props-helpers)

Next.js 自动生成全局可用的 `PageProps`、`LayoutProps` 和 `RouteContext` 类型，具有完整的参数类型且无需导入：

**Before: Manual typing and imports**

```
import { Metadata } from 'next';
 
interface Props {
  params: Promise<{ slug: string }>;
  children: React.ReactNode;
  analytics: React.ReactNode; // Manual parallel route typing
  team: React.ReactNode; // Manual parallel route typing
}
 
export default function DashboardLayout(props: Props) {
  return (
    <div>
      {props.children}
      {props.analytics} {/* No type safety for parallel routes */}
      {props.team} {/* No type safety for parallel routes */}
    </div>
  );
}
```

**After: Automatic typing with parallel route support**

```
// No need to import LayoutProps - globally available
export default function DashboardLayout(props: LayoutProps<'/dashboard'>) {
  return (
    <div>
      {props.children}
      {props.analytics} {/* Fully typed parallel route slot */}
      {props.team} {/* Fully typed parallel route slot */}
    </div>
  );
}
```

该系统自动从您的文件结构中发现路由，支持动态路由、并行路由和来自 `next.config.js` 的自定义路由。类型生成在开发模式和构建模式下都会运行，在开发期间文件结构更改时会立即重新生成类型，并通过仅生成几个优化文件而不是之前实现中的许多单个文件，高效地扩展到大型项目。

## `next typegen` [](#next-typegen)

我们已添加 `next typegen` 命令以手动生成类型，而无需运行 `next dev` 或 `next build`。这对于外部类型验证场景特别有用。

终端

```
next typegen [directory]
```

以前，路由类型只在 `next dev` 或 `next build` 期间生成，这意味着直接运行 `tsc --noEmit` 无法验证您的路由类型。现在，您可以独立生成类型并在外部进行验证：

终端

```
# Generate route types first, then validate with TypeScript
next typegen && tsc --noEmit
 
# Or in CI workflows for type checking without building
next typegen && npm run type-check
```

## `next lint` 弃用 [](#next-lint-deprecation)

从 Next.js 15.5 开始，`next lint` 命令显示弃用警告，并将在 Next.js 16 中移除。这通过过渡到显式的 ESLint 配置并引入 Biome 作为更快的替代方案来现代化 Linting 体验。

创建新的 Next.js 项目时，您现在可以在 ESLint（全面的规则）、Biome（快速且规则较少）或无 Linter 之间选择。ESLint 项目现在生成显式的 `eslint.config.mjs` 文件，而不是依赖 `next lint` 命令包装器，为您提供对 Linting 规则的完全透明度。

Biome 项目接收经过优化的配置，包括 Next.js 和 React 规则以及内置的格式化功能。生成的 `package.json` 脚本现在直接调用 Linters：

```
{
  "scripts": {
    // ESLint projects
    "lint": "eslint", // Instead of "next lint"
    "lint:fix": "eslint --fix",
 
    // For Biome projects
    "lint": "biome check",
    "format": "biome format --write"
  }
}
```

对于现有项目，新的 Codemod 自动化从 `next lint` 迁移到 ESLint CLI：

终端

```
npx @next/codemod@latest next-lint-to-eslint-cli .
```

Codemod 智能地转换 `package.json` 脚本同时保留功能，将 Next.js 特定的标志如 `--strict` 映射为 `--max-warnings 0`，并自动安装所需的依赖项。

这一过渡让开发者对其 Linting 设置拥有直接控制权，并具有更好的生态系统兼容性。

> **注意**: 如果存在 ESLint 配置，`next build` 仍将运行 Linting 验证步骤。Next.js 16 中将移除构建过程中的此自动 Linting，让您完全控制 Linting 何时运行以及如何运行。

## Next.js 16 弃用警告 [](#deprecation-warnings-for-nextjs-16)

Next.js 15.5 引入了弃用警告，帮助您为即将到来的 Next.js 16 版本做准备。这些警告出现在开发和构建日志中，让您有时间在这些功能移除之前进行迁移。

### `next/link` 的 `legacyBehavior` [](#legacybehavior-for-nextlink)

`next/link` 的 `legacyBehavior` prop 将在 Next.js 16 中移除。此 prop 在 Next.js 12 到 13 过渡期间作为临时兼容性层引入。

```
// ❌ Will be removed in Next.js 16
<Link href="/about" legacyBehavior>
  <a>About</a>
</Link>
 
// ✅ Modern approach (no changes needed)
<Link href="/about">About</Link>
```

**迁移**: 移除 `legacyBehavior` prop 和任何子 `<a>` 元素。`Link` 组件现在自动处理样式和可访问性。

### AMP 支持 [](#amp-support)

Next.js AMP 支持将在 Next.js 16 中移除。AMP 的使用率已显著下降，且维护此功能增加了框架的复杂性。所有与 AMP 相关的 API 和配置都将被移除。

pages/amp-page.js

```
// ❌ Will be removed in Next.js 16
import { useAmp } from 'next/amp';
 
export const config = { amp: true };
 
export default function AmpPage() {
  const isAmp = useAmp();
  return <div>AMP Page: {isAmp ? 'AMP' : 'HTML'}</div>;
}
```

next.config.ts

```
const nextConfig = {
  amp: {
    // ❌ Will be removed in Next.js 16
    canonicalBase: 'https://example.com',
  },
};
 
export default nextConfig;
```

**迁移**: 移除所有与 AMP 相关的代码，包括：

*   来自页面的 `export const config = { amp: true }`
*   来自 `next.config.ts` 的 `amp` 配置
*   `next/amp` hook 导入和用法 (`useAmp`)
*   任何其他 AMP 特定 API

评估 AMP 是否仍对您的用例必要。大多数性能优势现在可以通过 Next.js 的内置优化和现代 Web 标准来实现。

### `next/image` 质量设置 [](#nextimage-quality-settings)

从 Next.js 16 开始，`quality` prop 将默认仅限制为 `75`。当前在 Next.js 15 中，您可以使用 1 到 100 之间的任何整数，但 Next.js 16 将要求除 75 以外的质量进行显式配置。

```
// ⚠️ Will show deprecation warning in Next.js 15.5
// when images.qualities is undefined and quality !== 75
<Image src="/photo.jpg" quality={100} alt="Photo" />
 
// ✅ Explicit configuration required for Next.js 16
```

next.config.ts

```
const nextConfig = {
  images: {
    qualities: [75, 100], // Explicitly allow quality={100}
  },
};
 
export default nextConfig;
```

**迁移**: 如果您使用除 75 以外的 `quality` props，请在 `next.config.ts` 中显式配置 `images.qualities` 以包含 Next.js 16 所需的quality 值。

### `next/image` 本地模式 [](#nextimage-local-patterns)

从 Next.js 16 开始，使用本地图像 `src` 路径中的查询字符串需要在 `images.localPatterns` 中进行显式配置。这影响带有查询参数的图像，例如缓存清除或版本控制。

```
// ⚠️ Will show deprecation warning in Next.js 15.5
// when images.localPatterns is not configured
<Image src="/photo.jpg?v=1" alt="Test" />
 
// ✅ Explicit configuration required for Next.js 16
```

next.config.ts

```
const nextConfig = {
  images: {
    localPatterns: [
      {
        pathname: '/photo.jpg', // allow exact path
        // omitting "search" will allow all query parameters
      },
      {
        pathname: '/photo.jpg', // allow exact path
        search: '?v=1', // allow exact query parameters
      },
      {
        pathname: '/assets/**', // allow wildcard path
        search: '', // empty search will block all query parameters
      },
    ],
  },
};
 
export default nextConfig;
```

**迁移**: 在 `next.config.ts` 中配置 `images.localPatterns` 以显式允许图像路径中的查询字符串。这提供更好的安全性和性能优化。

### 时间线 [](#timeline)

这些弃用警告将从 Next.js 15.5 开始出现。这些功能将在 Next.js 16 中完全移除。我们建议尽快迁移以避免升级期间出现问题。
