我们随 Next.js 12.3 推出了一些生活质量改进：

*   [**改进的快速刷新 (Fast Refresh)**](#improved-fast-refresh) `.env`、`jsconfig.json` 和 `tsconfig.json` 现在热重载。
*   [**TypeScript 自动安装 (TypeScript Auto-Install)**](#typescript-auto-install) 添加 `.ts` 文件即可自动配置 TypeScript 并安装依赖项。
*   [**图片组件 (Image Component)**](#image-component) `next/future/image` 现在已稳定。
*   [**SWC 压缩器 (SWC Minifier)**](#swc-minifier-stable) 通过 Next.js 编译器进行压缩现在已稳定。
*   [**新路由 + 布局更新 (New Router + Layouts Update)**](#new-router-and-layouts-update) 已开始实施并解释了新特性。

通过运行 `npm i next@latest` 进行更新。

TypeScript 自动安装 [](#typescript-auto-install)
---------------------------------------------------

Next.js 内置支持自动配置 TypeScript。之前，将 TypeScript 添加到现有项目需要手动安装必要的依赖项。

通过此发布，Next.js 会自动检测 TypeScript 文件何时添加到项目中，安装必要的依赖项，并添加一个 `tsconfig.json` 文件。这在运行 `next dev` 和 `next build` 时有效。

改进的快速刷新 (Fast Refresh) [](#improved-fast-refresh)
-----------------------------------------------

快速迭代对于您的本地开发工作流至关重要。Next.js 使用 [快速刷新](/docs/architecture/fast-refresh) 来即时反馈对 React 组件所做的编辑。保存更改并立即在浏览器中看到您的更改，而无需重新加载页面。

随着 Next.js 12.3 的发布，一些配置文件现在在编辑时也会热重载，包括：

*   `.env` 和 `env.*` 变体
*   `jsconfig.json`
*   `tsconfig.json`

您现在**无需重新启动本地开发服务器**即可修改这些配置文件。

图片组件 [](#image-component)
-----------------------------------

### `next/future/image` 组件 (稳定) [](#nextfutureimage-component-stable)

在 Next.js 12.2 中，我们分享了一个**新**图片组件的实验性预览，简化了图片样式，并通过使用原生浏览器懒加载提高了性能。

新的图片组件现在已稳定，不再需要实验性标志。

自上次发布以来，我们做了进一步的改进：

*   添加了对 `fill` 布局模式的支持，使图片填充父元素。
*   改进了模糊占位符，使其适用于透明图片，不再需要 `<noscript>`，且 `next dev` 与 `next start` 行为一致。
*   防止细微的布局偏移（非 Chrome 浏览器），在图片加载期间隐藏 `alt` 文本。
*   要求 `alt` 属性以提供更好的无障碍性。
*   修复了在 hydration 之前发生错误时的 `onError` 属性。
*   修复了仅具有 `width` 或 `height` 属性的静态导入图片的宽高比。
*   在使用不正确的设置或配置改进了错误消息。

在未来，我们计划将 `next/image` 重命名为 `next/legacy/image`，将 `next/future/image` 重命名为 `next/image`。这将为新的 Next.js 项目提供极好的体验，现有项目将能够使用我们的 `next/image` 代码修改工具轻松迁移。我们将在下一个主要版本之前提供更多详细信息。

若要了解如何今天使用它的更多信息，请参阅 [文档](/docs/pages/building-your-application/optimizing/images)，其中包括迁移指南。

### 远程图片模式 (Remote Patterns) (稳定) [](#remote-patterns-stable)

`remotePatterns` 允许您在构建并使用内置图片优化 API 时指定远程图片的通配符。它允许更强大的匹配，超越现有的 `images.domains` 配置，后者仅在域名上执行完全匹配。

此选项现在已稳定，可与以下配置选项一起使用：

next.config.js

```
module.exports = {
  images: {
    remotePatterns: [
      {
        // The `src` property hostname must end with `.example.com`,
        // otherwise the API will respond with 400 Bad Request.
        protocol: 'https',
        hostname: '**.example.com',
      },
    ],
  },
};
```

### 禁用图片优化 (稳定) [](#disable-image-optimization-stable)

图片优化 API 需要一个服务器以按需优化图片。这对于使用 `next start` 进行自托管或部署到 [Vercel](https://vercel.com/solutions/nextjs) 等平台非常有效。但是，这在 `next export` 中不起作用，因为它不包含用于优化图片的服务器。之前，您需要使用 `loader` prop 配置第三方图片优化提供商，但现在您可以使用 `next.config.js` 完全禁用所有 `next/image` 实例的图片优化。

此之前处于实验性状态的选项现在已稳定，可与以下配置选项一起使用：

next.config.js

```
module.exports = {
  images: {
    unoptimized: true,
  },
};
```

SWC 压缩器 (稳定) [](#swc-minifier-stable)
---------------------------------------------

在 Next.js 12 中，我们引入了使用 SWC 的代码压缩，作为 [Next.js 编译器](/docs/architecture/nextjs-compiler) 实验性功能的一部分。早期结果显示，使用 SWC 进行压缩的速度比使用 Terser 的上一版本 **快 7 倍**。

通过此发布，该功能已稳定，您可以使用以下 `next.config.js` 选项进行选择加入：

next.config.js

```
module.exports = {
  swcMinify: true,
};
```

我们计划在下一个主要版本中将 SWC 压缩器设为默认。

新路由和布局更新 [](#new-router-and-layouts-update)
---------------------------------------------------------------

今年早些时候，我们发布了 [布局 RFC](https://nextjs.org/blog/layouts-rfc)，这是自 2016 年发布以来 Next.js 最大的更新。这些更改包括一个基于 React 服务器组件 (React Server Components) 的新路由。实现工作已经开始，我们正在朝着下一个主要版本的公共测试版努力。

今天，我们分享有关即将推出的功能的更多信息，包括：

*   [路由组 (Route Groups)](https://nextjs.org/blog/layouts-rfc#route-groups): 用于选择退出路由的布局，并组织路由而不影响 URL 路径。
*   [即时加载状态 (Instant Loading States)](https://nextjs.org/blog/layouts-rfc#instant-loading-states): 新文件约定，用于轻松添加加载 UI，基于 Suspense 构建。
*   [错误处理 (Error Handling)](https://nextjs.org/blog/layouts-rfc#error-handling): 新文件约定，用于轻松添加嵌套错误边界和错误 UI。
*   [模板 (Templates)](https://nextjs.org/blog/layouts-rfc#templates): 新文件约定，用于添加共享 UI，该 UI 创建不同的实例且不共享状态。
*   [拦截路由 (Intercepting Routes)](https://nextjs.org/blog/layouts-rfc#intercepting-routes), [并行路由 (Parallel Routes)](https://nextjs.org/blog/layouts-rfc#dynamic-parallel-routes), 和 [条件路由 (Conditional Routes)](https://nextjs.org/blog/layouts-rfc#conditional-routes): 允许您实现高级路由模式的新约定。

此外，在 [不同的 RFC](https://github.com/vercel/next.js/discussions/27953) 中，我们正在探索在新 `app` 目录中添加对 **全局 CSS 导入** 的支持。这将允许您使用导入自己 `.css` 文件的软件包，而无需额外配置。

在 [布局 RFC](https://nextjs.org/blog/layouts-rfc) 中阅读有关这些新功能的更多信息。

[其他改进](#other-improvements)
-----------------------------------------

*   Next.js 内置可访问性功能的新文档 ([/docs/accessibility])。
*   `create-next-app` 模板现在使用系统首选项自动包含暗色模式支持。
*   使用 [集成 ESLint 支持](/docs/pages/building-your-application/configuring/eslint) 时，`src/` 现在是一个默认的 linting 目录。
*   `next.config.js` 现在包含带有 `ajv` 的架构验证，帮助防止配置选项误用。
*   `next.config.js` 现在有一个实验性配置标志 `fallbackNodePolyfills: false`，以防止 Next.js 为浏览器 Polyfill Node.js API（这会导致捆绑包大小增加）。此选项将导致 Next.js 显示一个构建时错误，显示不必要的导入来自何处。
