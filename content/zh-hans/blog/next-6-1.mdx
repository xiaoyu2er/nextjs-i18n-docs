我们今天很自豪地推出生产就绪级别的 **Next.js 6.1**，其特性包括：

*   提高了热重载 (hot reloading) 的可靠性
*   代码库改进
*   Next.js 代码转换

除了 Next.js 6.1 发布外，我们兴奋地宣布 [nextjs.org](https://nextjs.org) 现已 [开源](https://github.com/vercel/next-site)

改进的热重载可靠性 [](#improved-hot-reloading-reliability)
-------------------------------------------------------------------------

在 Next.js 6.1 之前的版本中，Next.js 为用户实现了 `react-hot-loader`。该库在热重载期间保留 React 状态。

这样做 `react-hot-loader` 给 React 添加了一些非标准行为，例如，它将忽略 `shouldComponentUpdate`，并且元素 `type` 最终成为了代理组件而不是实际的 React 组件。

为了确保 Next.js 尽可能接近默认的 React，我们移除了 `react-hot-loader` 作为依赖项，这确保了开发模式和生产模式的行为更加一致。请注意，Next.js 的热重载功能**并未**被移除，热重载一直由 Next.js 内部处理。

热重载 TypeScript 和其他自定义扩展 [](#hot-reloading-typescript-and-other-custom-extensions)
-------------------------------------------------------------------------------------------------------------

默认情况下，Next.js 会自动在 `pages` 目录中查找任何 `.js` 或 `.jsx` 文件来定义路由。

随着 Next.js 5 引入通用 webpack，便有了将顶层页面编译为 JS 的可能性。一个很好的例子是 TypeScript，它使用 `.ts` 和 `.tsx`。

`pageExtensions` 是 `next.config.js` 中的一个配置项，旨在允许 Next.js 插件定义应被视为页面的扩展。例如 [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript) 定义了 `.ts` 和 `.tsx`，或者 [`@zeit/next-mdx`](https://github.com/vercel/next-plugins/tree/master/packages/next-mdx)，其 [文档](https://github.com/vercel/next-plugins/tree/master/packages/next-mdx#top-level-mdx-pages) 说明了如何拥有顶层 `.mdx` 页面。

之前，在实现 `pageExtensions` 时，Next.js 插件需要实现用于热重载的 `hot-self-accept-loader`。现在不再需要此项了，当向 `pageExtensions` 添加扩展时，会自动应用 `hot-self-accept-loader`。

代码库改进 [](#codebase-improvements)
-----------------------------------------------

最近，我们正在为即将到来的功能铺平道路，这涉及一些底层更改，将从长远来看提高代码质量。

这些更改之一是将 `server/build` 目录移动到顶层 `build`。这使得 webpack 和 babel 的配置文件对新贡献者来说更容易找到。

我们还专注于在整个代码库中添加 [Flow](https://flow.org/) 类型。

我们做出的一个对用户更明显的更改是 `.next/dist` 已重命名为 `.next/server`。`.next` 目录用于存放构建输出。例如，当你运行 `next build` 时，结果将存储在 `.next` 中。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-6-1/server-dist.png)

> 预渲染文件现在位于 server 目录中

相同常量的出现已移入一个公共文件：[`constants.js`](https://github.com/vercel/next.js/blob/canary/lib/constants.js)

Next.js 代码转换 [](#nextjs-codemods)
------------------------------------

当 Next.js 6.0 发布时，页面上神奇注入的 `url` 属性已被弃用。`url` 消失的原因是我们要让事情非常可预测和明确。拥有一个凭空出现的魔法 url 属性无助于这个目标。

获取与 `url` 属性相同属性的推荐方法是使用 `withRouter`：

page.js

```
// old
class Page extends React.Component {
  render() {
    const { url } = this.props;
    return <div>{url.pathname}</div>;
  }
}
export default Page;
```

> 在 Next.js 6 之前的版本中如何使用 url 访问 pathname

page.js

```
// new
import { withRouter } from 'next/router';
class Page extends React.Component {
  render() {
    const { router } = this.props;
    return <div>{router.pathname}</div>;
  }
}
export default withRouter(Page);
```

> 在 Next.js 6 之后的版本中如何使用由 withRouter 注入的 router 访问 pathname

由于我们致力于保持 Next.js 应用程序的升级路径简单，因此我们着手创建一种简单的方法，将 url 的使用升级到 withRouter。

这项工作的成果是 [next-codemod](https://github.com/vercel/next-codemod)，一个代码转换库，使将特定的已弃用功能升级到其新用法变得像运行一个命令一样简单。

我们要提供的首个代码转换是 `url-to-withrouter`，它自动将许多使用 url 的情况转换为 withRouter。

终端

```
  jscodeshift -t ./url-to-withrouter.js pages/**/*.js
```

> 这将把 url 的使用转换为 withRouter。

[在此阅读更多内容](https://github.com/vercel/next-codemod#url-to-withrouter)

未来 [](#future)
-----------------

我们一直在致力于几个新功能，以提高可靠性和性能，以下是几点亮点：

### Webpack 4 [](#webpack-4)

Webpack 4 带来了许多改进：更好的代码拆分，默认需要的配置更少，最重要的是构建时间更快。在我们最初对一个有超过 200 个页面的应用进行的测试中，`next build` 的平均时间从 100 秒降到了 70 秒。在第二次运行（有缓存）时，`next build` 平均耗时 21 秒。

### Serverless Next.js [](#serverless-nextjs)

我们正在逐步做出更改，以准备将 `next start` 移到其自己的包中：`next-server`。该包将对安装大小和启动时间进行大幅优化。这些优化对于“无服务器”用例是必需的，在该用例中，每个请求或每几个请求都会执行一个新的应用程序实例。这意味着应用程序的冷启动必须进行优化，使其尽可能快。
