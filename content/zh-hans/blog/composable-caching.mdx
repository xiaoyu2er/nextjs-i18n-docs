我们正在为 Next.js 开发一个简单且强大的缓存模型。在之前的一篇文章中，我们谈到了 [我们的缓存之旅](/blog/our-journey-with-caching)，以及我们是如何得出 `'use cache'` 指令的。

本文将讨论 `'use cache'` 的 API 设计和优势。

“use cache”是什么？[](#what-is-use-cache)
--------------------------------------------

`“use cache”` 通过按需缓存数据或组件来让你的应用更快。

它是一个 JavaScript“指令”（directive）—— 你在代码中添加的字符串字面量——这向 Next.js 编译器发出信号进入不同的“边界”。例如，从服务器到客户端。

这与 React 指令（如 `'use client'` 和 `'use server'`）类似。指令是编译器指令，定义了代码的运行位置，从而使框架能够为你优化和编排各个部分。

它是如何工作的？[](#how-does-it-work)
--------------------------------------

让我们从一个简单的例子开始：

```
async function getUser(id) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

在幕后，由于 `'use cache'` 指令，Next.js 将此代码转换为了服务端函数。在编译期间，会查找此缓存项的“依赖项”(dependencies) 并将其用作缓存键 (cache key) 的一部分。

例如，`id` 成为缓存键的一部分。如果多次调用 `getUser(1)`，我们返回来自缓存服务端函数的内存化输出。更改此值将在缓存中创建新条目。

让我们看看一个在服务器组件中使用缓存函数的示例 [闭包](https://v0.app/chat/5kD47RIecQK?b=b_rCP4CvfbFFW)。

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    'use cache';
    return await db
      .select()
      .from(notifications)
      .limit(limit)
      .offset(index)
      .where(eq(notifications.userId, id));
  }
 
  return <User notifications={getNotifications} />;
}
```

这个例子更难。你能发现所有需要作为缓存键一部分的依赖项吗？

参数 `index` 和 `limit` 是有道理的——如果这些值更改，我们将选择不同的通知片段。但是，用户 `id` 呢？它的值来自父组件。

编译器能够理解 `getNotifications` 也取决于 `id`，并且它的值会自动包含在缓存键中。这防止了整个类别的缓存问题，这些问题是由缓存键中不正确或缺少的依赖项引起的。

为什么不使用 cache 函数？[](#why-not-use-a-cache-function)
--------------------------------------------------------------

让我们重新审视一下最后一个例子。我们可以使用 `cache()` 函数而不是指令吗？

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    return await cache(async () => {
      return await db
        .select()
        .from(notifications)
        .limit(limit)
        .offset(index)
        // Oops! Where do we include id in the cache key?
        .where(eq(notifications.userId, id));
    });
  }
 
  return <User notifications={getNotifications} />;
}
```

`cache()` 函数无法查看闭包并看到 `id` 值应该是缓存键的一部分。您将需要手动指定 `id` 是您的键的一部分。如果您忘记这样做，或做错了，您可能会面临缓存冲突 (cache collisions) 或过期数据的风险。

闭包可以捕获各种各样的局部变量。一个简单的方法可能会无意中烘焙进去（或遗漏）您不打算的变量。这可能导致缓存了错误的数据，或者如果敏感信息泄露到缓存键中，可能会危及缓存污染 (cache poisoning)。

`“use cache”` 给予编译器足够的上下文来安全处理闭包并正确生成缓存键。仅运行时的解决方案（如 `cache()`）将要求您手动执行所有操作——很容易出错。相比之下，指令可以进行静态分析以在底层可靠地处理所有依赖项。

如何处理非序列化输入值？[](#how-are-non-serialized-input-values-handled)
--------------------------------------------------------------------------------------------

我们有两种不同类型的输入值需要缓存：

*   **可序列化 (Serializable)**：在这里，“可序列化”意味着输入可以转换为稳定、基于字符串的格式 _without_ 丢失含义。虽然很多人首先想到的是 `JSON.stringify`，但我们实际上使用 React 的序列化（例如，通过 Server Components）来处理更广泛的输入——包括 Promise、循环数据结构和其他复杂对象。这超出了普通 JSON 所能做的范围。
*   **不可序列化 (Non-serializable)**：这些输入不是缓存键的一部分。当我们尝试缓存这些值时，我们返回一个服务器“引用” (reference)。然后 Next.js 使用此引用在运行时恢复原始值。

假设我们记得将 `id` 包含在缓存键中：

```
await cache(async () => {
  return await db
    .select()
    .from(notifications)
    .limit(limit)
    .offset(index)
    .where(eq(notifications.userId, id));
}, [id, index, limit]);
```

如果输入值可以序列化，这就可以运作。但如果 `id` 是一个 React 元素或更复杂的值，我们就必须手动序列化输入键。考虑一个根据 `id` 属性获取当前用户的服务器组件：

```
async function Profile({ id, children }) {
  'use cache';
  const user = await getUser(id);
 
  return (
    <>
      <h1>{user.name}</h1>
      {/* Changing children doesn’t break the cache... why? */}
      {children}
    </>
  );
}
```

让我们逐步了解它的工作原理：

1.  编译期间，Next.js 看到 `'use cache'` 指令并将代码转换为创建支持缓存的特殊服务端函数。编译期间不会发生缓存，而是 Next.js 设置了运行时缓存所需的机制。
2.  当你的代码调用“缓存函数”时，Next.js 会序列化函数的参数。任何不可直接序列化的内容（如 JSX）都将被“引用”占位符替换。
3.  Next.js 检查是否存在给定序列化参数的缓存结果。如果未找到结果，函数计算要缓存的新值。
4.  函数执行完毕后，返回值为序列化。返回值中的不可序列化部分变回引用。
5.  调用缓存函数的代码反序列化输出并评估引用。这允许 Next.js 用其实际对象或值替换引用，意味着像 `children` 这样的非序列化输入可以保持其原始、未缓存的值。

这意味着我们可以安全地缓存 _仅_ `<Profile>` 组件而不包含 children。在后续渲染中，`getUser()` 不会被再次调用。`children` 的值可能是动态的，或者是具有不同缓存生命周期的单独缓存元素。这就是合式缓存 (composable caching)。

这看起来很熟悉……[](#this-seems-familiar)
--------------------------------------------

如果你在想着“这感觉就像服务器和客户端组合的相同模型”——你完全没错。这有时被称为“甜甜圈”模式：

*   甜甜圈的 **外层** 是处理数据获取或重逻辑的服务端组件。
*   中间的 **孔** 是具有某些交互的子组件。

app/page.tsx

```
export default function Page() {
  return (
    <ServerComponent>
      {/* Create a hole to the client */}
      <ClientComponent />
    <ServerComponent />
  );
}
```

“use cache”是一样的。甜甜圈是外层组件的缓存值，孔是运行时填充的引用。这就是为什么更改 `children` 不会使整个缓存输出失效。孩子只是一些随后填充的引用。

关于标记和失效呢？[](#what-about-tagging-and-invalidation)
----------------------------------------------------------------------------

你可以使用不同的 [配置](/docs/app/api-reference/functions/cacheLife) 定义缓存的生命周期。我们包含一组默认配置，但如果你愿意，也可以定义自己的自定义值。

```
async function getUser(id) {
  'use cache';
  cacheLife('hours');
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

要使特定的缓存条目失效，你可以 [标记缓存](/docs/app/api-reference/functions/cacheTag) 然后调用 `revalidateTag()`。一个强大的模式是你可以在获取数据后（例如，从 CMS 获取）之后再标记缓存：

```
async function getPost(postId) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/blog/${postId}`);
  let data = await res.json();
  cacheTag(postId, data.authorId);
  return data;
}
```

简单并且强大[](#simple-and-powerful)
-------------------------------------------

我们的目标是通过 `'use cache'` 让编写缓存逻辑既简单 _又_ 强大。

*   **简单 (Simple)**：你可以使用本地推理创建缓存条目。你不需要担心全局副作用，如忘记缓存键条目或对代码库的其他部分造成意外更改。
*   **强大 (Powerful)**：你可以缓存不仅仅是静态分析代码的内容。例如，可能在运行时更改的值，但你仍然希望在评估后缓存输出结果。

“use cache”在 Next.js 内部仍然是 **实验性 (experimental)** 的。当我们测试它时，我们非常期待您的早期反馈。
