我们正在为 Next.js 构建一个简单且强大的缓存模型。在之前的文章中，我们分享了 [我们的缓存之旅](/blog/our-journey-with-caching)，以及我们是如何得出 `'use cache'` 指令的。

本文讨论 `'use cache'` 的 API 设计和优势。

什么是 `'use cache'`？[](#what-is-use-cache)
--------------------------------------------

`'use cache'` 通过按需缓存数据或组件来加快您的应用程序速度。

它是一个 JavaScript“指令”——您在代码中添加的字符串字面量——它会向 Next.js 编译器发出信号，让编译器进入不同的“边界”。例如，从服务端到客户端。

这与 React 指令类似，比如 `'use client'` 和 `'use server'`。指令是编译器指令，定义代码应在哪里运行，允许框架为您优化和编排各个部分。

它是如何工作的？ [](#how-does-it-work)
--------------------------------------

让我们从一个简单的例子开始：

```
async function getUser(id) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

在幕后，由于 `'use cache'` 指令，Next.js 将此代码转换为服务器函数。在编译期间，找到此缓存条目的“依赖项”并用作缓存键的一部分。

例如，`id` 成为缓存键的一部分。如果我们多次调用 `getUser(1)`，我们将从缓存的服务器函数返回记忆化的输出。更改此值将在缓存中创建新条目。

让我们看一个在服务器组件中使用缓存函数的示例，该示例使用了 [闭包](https://v0.app/chat/5kD47RIecQK?b=b_rCP4CvfbFFW)。

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    'use cache';
    return await db
      .select()
      .from(notifications)
      .limit(limit)
      .offset(index)
      .where(eq(notifications.userId, id));
  }
 
  return <User notifications={getNotifications} />;
}
```

这个例子更复杂。你能找出作为缓存键一部分的所有依赖项吗？

参数 `index` 和 `limit` 说得通——如果这些值改变，我们选择不同的通知切片。但是，用户的 `id` 呢？它的值来自父组件。

编译器能够理解 `getNotifications` 也依赖于 `id`，并且它的值自动包含在缓存键中。这防止了一大类缓存问题，即缓存键中依赖项不正确或缺失的问题。

为什么不使用缓存函数？[](#why-not-use-a-cache-function)
--------------------------------------------------------------

让我们回顾最后一个例子。我们能否改用 `cache()` 函数而不是指令？

```
function Profile({ id }) {
  async function getNotifications(index, limit) {
    return await cache(async () => {
      return await db
        .select()
        .from(notifications)
        .limit(limit)
        .offset(index)
        // Oops! Where do we include id in the cache key?
        .where(eq(notifications.userId, id));
    });
  }
 
  return <User notifications={getNotifications} />;
}
```

`cache()` 函数将无法查看闭包并看到 `id` 值应作为缓存键的一部分。您将需要手动指定 `id` 是密钥的一部分。如果您忘记这样做，或者做得不正确，您将面临缓存冲突或数据过时的风险。

闭包可以捕获各种局部变量。一个简单的方法可能会不知不觉地将您不打算包含（或省略）的变量烘焙进去。这可能导致缓存错误的数据，或者如果敏感信息泄露到缓存键中，可能会带来缓存投毒的风险。

`'use cache'` 给予编译器足够的上下文来安全地处理闭包并正确生成缓存键。仅运行时的解决方案，如 `cache()`，将要求您手动处理所有操作——并且很容易出错。相比之下，指令可以进行静态分析，以在后台可靠地处理所有依赖项。

非序列化输入值如何处理？[](#how-are-non-serialized-input-values-handled)
--------------------------------------------------------------------------------

我们有两种不同类型的输入值需要缓存：

*   **可序列化 (Serializable)**: 在此，“可序列化”意味着输入可以转换为稳定的、基于字符串的格式 _未_ 损失意义。虽然许多人首先想到 `JSON.stringify`，但我们实际上使用 React 的序列化（例如，通过服务端组件）来处理更广泛的输入范围——包括 Promise、循环数据结构和更复杂的对象。这超出了纯 JSON 所能做到的范围。
*   **不可序列化 (Non-serializable)**: 这些输入不是缓存键的一部分。当我们尝试缓存这些值时，我们返回服务器的“引用”。然后 Next.js 使用此引用在运行时还原原始值。

假设我们记得将 `id` 包含在缓存键中：

```
await cache(async () => {
  return await db
    .select()
    .from(notifications)
    .limit(limit)
    .offset(index)
    .where(eq(notifications.userId, id));
}, [id, index, limit]);
```

如果输入值可以被序列化，这就有效。但如果 `id` 是 React 元素或更复杂的值，我们将不得不手动序列化输入键。考虑一个服务器组件，它根据 `id` prop 获取当前用户：

```
async function Profile({ id, children }) {
  'use cache';
  const user = await getUser(id);
 
  return (
    <>
      <h1>{user.name}</h1>
      {/* Changing children doesn’t break the cache... why? */}
      {children}
    </>
  );
}
```

让我们逐步了解它是如何工作的：

1.  编译期间，Next.js 看到 `'use cache'` 指令并将代码转换为创建特殊的服务器函数以支持缓存功能。编译期间不会发生缓存，而是 Next.js 正在设置运行时缓存所需的机制。
2.  当您的代码调用“缓存函数”时，Next.js 序列化函数参数。任何不可直接序列化的内容，例如 JSX，都替换为“引用”占位符。
3.  Next.js 检查是否存在带有相应序列化参数的缓存结果。如果找不到结果，函数将计算新值以缓存。
4.  函数完成后，返回价值被序列化。返回值的不可序列化部分变回引用。
5.  调用缓存函数的代码反序列化输出并评估引用。这允许 Next.js 用实际对象或值交换引用，这意味着像 `children` 这样的不可序列化输入可以保留其原始、未缓存的值。

这意味着我们可以安全地缓存 _仅_ `<Profile>` 组件，而不包括子组件。在后续渲染中，不再调用 `getUser()`。`children` 的值可能是动态的，或者是一个具有不同缓存生命周期的单独缓存元素。这是组合式缓存。

这看起来很眼熟……...[](#this-seems-familiar)
--------------------------------------------

如果您心想“这感觉和服务端与客户端合成的同一模型一样”——您完全正确。这有时被称为“甜甜圈”模式：

*   甜甜圈的**外部**部分是一个服务端组件，负责数据获取或繁重逻辑。
*   中间的**空洞**是一个子组件，它可能具有一定的交互性

app/page.tsx

```
export default function Page() {
  return (
    <ServerComponent>
      {/* Create a hole to the client */}
      <ClientComponent />
    <ServerComponent />
  );
}
```

`'use cache'` 也是如此。甜甜圈是外部组件的缓存值，空洞是运行时填充的引用。这就是为什么更改 `children` 不会使整个缓存输出失效的原因。子组件只是稍后填充的引用。

关于标记和失效呢？[](#what-about-tagging-and-invalidation)
----------------------------------------------------------------------------

您可以使用不同的 [配置文件](/docs/app/api-reference/functions/cacheLife) 定义缓存的生命周期。我们包含一组默认配置文件，但如果您愿意，可以定义自己的自定义值。

```
async function getUser(id) {
  'use cache';
  cacheLife('hours');
  let res = await fetch(`https://api.vercel.app/user/${id}`);
  return res.json();
}
```

要使特定的缓存条目失效，您可以 [标记缓存](/docs/app/api-reference/functions/cacheTag)，然后调用 `revalidateTag()`。一个强大的模式是在获取数据后（例如从 CMS）标记缓存：

```
async function getPost(postId) {
  'use cache';
  let res = await fetch(`https://api.vercel.app/blog/${postId}`);
  let data = await res.json();
  cacheTag(postId, data.authorId);
  return data;
}
```

简单且强大[](#simple-and-powerful)
-------------------------------------------

我们的目标是让 `use cache` 的缓存逻辑编写变得简单 _且_ 强大。

*   **简单 (Simple)**: 您可以根据本地推理创建缓存条目。您不必担心全局副作用，如遗忘缓存键条目或无意中更改代码库的其他部分。
*   **强大 (Powerful)**: 您可以缓存不仅仅是静态可分析的代码。例如，在运行时可能变化的值，但您仍希望在评估后缓存输出结果。

'use cache' 在 Next.js 中仍处于 **实验性** 阶段。当我们测试它时，我们非常期待您的早期反馈。
