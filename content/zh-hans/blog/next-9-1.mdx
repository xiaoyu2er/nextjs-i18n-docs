今天，我们很兴奋地宣布 Next.js 9.1 支持 `src` 和 `public` 目录。

**本次发布新增**

*   **[`src` 目录支持](#src-directory-support)**: `pages` 目录现在可以嵌套在 `src` 文件夹下，支持更广泛的应用设置。
*   **[`public` 目录支持](#public-directory-support)**: 定义要挂载在应用程序 URL 根目录下的文件（例如 `favicon.ico`）。

**本次预览**

*   **内置 CSS 支持**: 应用程序将很快能够导入全局 CSS，并利用开发热重载和高级生产优化、编译及补全。
*   **静态错误页面**: 静态导出预期的错误页面（如 404），以获得更高的可用性（CDN）。
*   **Module / Nomodule **(模块/非模块): 向运行在现代浏览器上的最终用户提供较小的 JavaScript 捆绑包。
*   **改进的捆绑包拆分**: 向最终用户发送更少的字节，提高 TTI 和页面转换速度。大型库块在部署之间也进行长期缓存。

一如既往，我们要确保所有这些 benefits 都是向后兼容的。您更新所需做的就是运行：

终端

```
npm i next@latest react@latest react-dom@latest
```

如果您的 Next.js 应用版本在 9 之前，您可以参考 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md) 查看可能需要更改的内容。

自从我们上一次主要发布以来，我们很高兴看到像 [TikTok](https://www.tiktok.com/en/)、[Hilton](https://www.hilton.com/en/hilton/)、[Elastic](https://www.elastic.co)、[Realtor](https://www.realtor.com/) 和 [JW Player](https://www.jwplayer.com/) 这样的公司使用 Next.js 发布。查看更多 [展示](https://nextjs.org/showcase)！

### `src` 目录支持 [](#src-directory-support)

Next.js 有一个特殊的 `pages` 目录，其中每个文件都成为一个单独的路由，遵循惯例优于配置的这种做法，该目录必须位于 Next.js 应用的根目录中。

在与使用 Next.js 的公司交谈并检查一些专有代码库后，我们发现有开发人员想要的一种常见模式是为代码创建一个 `src` 目录，并也将 `pages` 目录放在其中。

从 Next.js 9.1 开始，现在可以创建 `src/pages` 目录，而不是在应用根目录中创建 `pages` 目录。

使用 `src` 目录是可选的，适用于公司已经实施此标准的情况。

### `public` 目录支持 [](#public-directory-support)

除了 `pages` 目录外，Next.js 还有另一个名为 `static` 的特殊目录，其中的文件映射到 `/static` 路由。

例如，`static/my-image.png` 将映射到 `/static/my-image.png`。

自 Next.js 第一次发布以来，这种惯例运作良好，并没有特别的问题。

然而，随着时间的推移，我们确定了 `/static` 并不足以涵盖 Web 应用程序所需的一切。例如，`robots.txt` 必须从域名的根目录提供。

从 Next.js 9.1 开始，我们正在引入一个新的名为 `public` 的目录。

`public` 目录中的任何文件将映射到域的根目录。

例如，`public/robots.txt` 将映射到 `/robots.txt`。

由于 `public` 也覆盖了 `static` 目录的使用案例，我们决定弃用 `static` 目录，转而创建具有相同功能的 `public/static` 文件夹。

一如既往，我们力求向后兼容，因此 `static` 目录将继续有效，但会带有一个弃用消息。

### 内置 CSS 支持 [](#built-in-css-support)

目前，Next.js 内置了一个名为 [styled-jsx](http://github.com/vercel/styled-jsx) 的 CSS-in-JS 解决方案。此解决方案对于为单个 React 组件样式设置工作良好。

然而，在与公司交流时，我们发现有一个常见的需求是重用一些基于 CSS 的现有样式或设计系统。

一般来说，这意味着添加 `next-css` 插件以支持 CSS 导入。

我们发现大约 50% 的 Next.js 用户都会在应用中添加此插件。

由于这种广泛的使用，我们添加了 CSS 导入的内置支持，包括在开发环境中样式自动重载以及之前 `next-css` 插件中不可能实现的生产优化。

初始实施目前正在生产 Next.js 应用中进行测试。

首先将引入全局 CSS 导入：

`pages/_app.js`

```
// Global styles can be imported from _app.js
import '../styles/global.css';
import App from 'next/app';
 
export default App;
```

在导入全局 CSS 之后，我们将通过 `.module.css` 扩展支持 CSS 模块：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
// Scoped styles are imported through .module.css
import styles from '../styles/index.module.css';
 
export default function HomePage() {
  return <h1 className={styles.heading}>Hello World</h1>;
}
```

这将允许我们为使用 CSS 导入提供更好的开发者体验。

您可以阅读 GitHub 上的 [RFC](https://github.com/vercel/next.js/issues/8626) 以了解更多正在进行的工作。

### 静态错误页面 [](#static-error-pages)

当发生错误时，Next.js 会渲染一个特殊页面，该页面在内部称为 `/_error`。用户可以通过创建 `pages/_error.js` 文件并导出 React 组件来自定义此页面。

渲染的错误通常分为 2 种情况：预期的错误和非预期的错误。

预期的错误，例如 `404` 页面。

非预期的错误，例如，当错误在 `getInitialProps` 中抛出或在渲染 React 树时发生，这将渲染 `500` 错误。

我们计划为预期错误添加 [自动静态优化](https://nextjs.org/blog/next-9#automatic-static-optimization)，因为通常它们不需要动态渲染。

如果用户希望动态渲染这些页面，将有一个放弃选项，但默认情况下 `404` 将是静态页面。在使用 `server` 目标时这会减少服务器负载，在使用 `serverless` 目标时会降低成本。

使页面静态化的另一个好处是，在使用 CDN 时可以自动缓存。

### Module / Nomodule (模块/非模块) [](#module--nomodule)

在 Next.js 中编写代码时，您通常编写“现代”JavaScript。您可以使用所有最新稳定功能，Next.js 将通过使用 Babel 编译代码，确保自动转换或补全这些功能。

目前，大多数浏览器都支持这些 JavaScript 功能中的一部分。然而，一般来说（包括在 Next.js 中），代码会被打包成一个在应用程序支持的浏览器上运行的单一 JavaScript 捆绑包。

在 Next.js 的情况下，这意味着将现代 JavaScript 编译为兼容 Internet Explorer 11 的格式。

例如，目前，Next.js 必须为 `async/await` 语法提供补全，因为代码可能会在不支持 `async/await` 的浏览器中执行，从而导致错误。

为了不破坏旧浏览器同时向支持新语法浏览器发送现代 JavaScript，Next.js 将利用 [module/nomodule](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/) 模式。module/nomodule 模式为现代浏览器提供现代 JavaScript 的可靠机制，同时允许旧浏览器回退到补全的 ES5 代码。

此新功能当前正在由多个大型 Next.js 应用在生产中进行测试，以收集真实世界的数据。测试结果看起来很有希望，有关此更改的性能改进的更多细节将很快分享。

### 改进的捆绑包拆分 [](#improved-bundle-splitting)

Next.js 目前加载多个 JavaScript 捆绑包以使页面交互。最显著的是：

*   页面 JavaScript 捆绑包。
*   带有公共 JavaScript 的文件。
*   Next.js 客户端运行时捆绑包。
*   动态导入（通常通过 `next/dynamic` 添加）。

为了确保页面可交互，所有这些捆绑包都必须加载，因为它们都彼此依赖以便在浏览器中启动 React。

因为这些捆绑包需要被加载以便 React 启动，所以它们必须尽可能优化，并且不要从应用的其他部分下载过多代码。

出于这个原因，有一个 `commons` 捆绑包用于存储页面之间的公共 JavaScript。当前捆绑包拆分策略的计算以启发式方法为基础，如果一个模块在所有页面中使用 50% 以上，它将被标记为通用模块。

然而，应用可能由许多不同的部分组成。例如，营销页面、博客和管理仪表板。如果营销页面的数量远多于仪表板，那么 commons 的计算可能会对营销页面产生更优化的结果。

我们的目标是在一个应用中同时优化这两者。

Alex Castle 定义了新的代码分层层级（创建单独的 JavaScript 文件），允许更优化的公共代码分块，特别是在涉及多个文件和许多页面时。

类似于 module/nomodule 支持，改进的捆绑包拆分正在由多个大型 Next.js 应用在生产中进行测试，以收集真实数据。测试结果以及有关此更改的性能改进的更多细节将在稍后的博客文章中分享。
