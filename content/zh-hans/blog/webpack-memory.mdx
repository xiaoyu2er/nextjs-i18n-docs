最近发布了 [Next.js 8](/blog/next-8)。该版本包含了显著的构建时内存使用减少。这篇博文将探讨我们如何帮助社区优化 webpack。

Next.js 是零配置的，并且构建在 [webpack](https://webpack.js.org/) 和 [Babel](https://babeljs.io) 等工具之上。其目标是帮助你专注于重要的事情：你的应用程序代码。

现代 Web 应用程序由一个或多个页面组成。例如，一个主页、博客、仪表板或产品列表页面。

在 Next.js 中，这些页面会成为你项目根目录下一个特殊的 `pages` 目录中的文件。

例如：文件 `pages/about.js` 对应 URL `/about`。

该框架的一个关键设计约束是，它必须能够良好地处理单个页面和数千个页面的情况。

在实现 [无服务器 Next.js](/blog/next-8#serverless-nextjs) 时，我们很快发现在一个拥有数百个页面的项目上运行 `next build` 会导致高内存使用。有时会超过 Node.js 大约 1.4 GB 的内存堆限制。

我们开始使用 Chrome 开发者工具分析构建过程的内存使用情况。

在生成的分析文件中，我们发现了一个 webpack 会一次性分配 **548 MB** 内存的点。

分配的内存量与页面数量直接相关，这意味着页面越多，内存使用量就越大。

![Chrome 开发者工具内存分析器显示一次性分配了 548 MB](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/webpack-memory/allocation-before.png)

Chrome 开发者工具内存分析器显示一次性分配了 548 MB

通过查看内存分析的堆栈跟踪，我们能够追踪到导致内存分配峰值的函数。

分配本身来自于调用 [`source.source()` 方法](https://github.com/webpack/webpack/blob/v4.28.4/lib/Compiler.js#L334)，该方法会生成最终文件并将其存储到内存中。

然而，进一步查看调用 `source()` 方法的函数，你可以看到 [`compilation.assets`](https://github.com/webpack/webpack/blob/v4.28.4/lib/Compiler.js#L316) 正被 `asyncLib.forEach` 遍历。这意味着[提供的函数](https://github.com/webpack/webpack/blob/v4.28.4/lib/Compiler.js#L317)会被同时调用，应用于 `compilation.assets` 数组中的每一个文件。

因此，这意味着如果有例如 100 个页面，并且每个页面都需要写入磁盘，上述代码会尝试同时写入所有 100 个页面，包括同时生成所有 100 个文件。

此问题的解决方案是使用[信号量](https://en.wikipedia.org/wiki/Semaphore_\(programming\))来限制并发写入的数量。通常我们使用 [async-sema](https://github.com/vercel/async-sema) 来实现这一点，但在这个案例中，webpack 已经在其 [neo-async](https://github.com/webpack/webpack/blob/v4.28.4/lib/Compiler.js#L8) 上提供了一个合适的方法：

```
asyncLib.forEach(compilation.assets, (source, file, callback) => {
  // etc
});
```

> 之前并发运行函数处理所有资源的代码

```
asyncLib.forEachLimit(compilation.assets, 15, (source, file, callback) => {
  // etc
});
```

> 新代码最多同时运行 15 个函数

实施了此并发限制并再次分析构建内存使用后，我们可以看到内存分配被分割成了更小的 **34 MB** 块。

![分析器现在显示随着时间的推移分配了 34 MB 的块](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/webpack-memory/after-limiting.png)

分析器现在显示随着时间的推移分配了 34 MB 的块

这一改变展示了非常有前景的结果，但在实践中构建仍然会内存不足，因此我们继续分析和调查此问题。

通过进一步检查内存分析，我们注意到在调用 [`source.source()` 方法](https://github.com/webpack/webpack/blob/v4.28.4/lib/Compiler.js#L334)之后，内存并没有被清理（垃圾回收）。

在 webpack 中，资源通常是 [Source 类](https://github.com/webpack/webpack-sources)的实例。这些类都实现了 `source()` 方法，该方法会生成文件源代码。

分析显示许多资源是 [`CachedSource`](https://github.com/webpack/webpack-sources#cachedsource) 的实例。`CachedSource` 的工作方式是，当调用 `source()` 时，结果会缓存在内存中，直到该资源被释放。

检查 Next.js 使用的 webpack 插件发现，我们没有在 webpack 写入文件之后调用 `source()` 的插件，这意味着缓存写入的值没有任何好处。

在与 [Tobias Koppers](https://twitter.com/wSokra) [协作](https://github.com/webpack/webpack/pull/8609)之后，他[实现了一个名为 `output.futureEmitAssets` 的新选项](https://github.com/webpack/webpack/pull/8642)，该选项允许启用新的资源写入行为。

采用这个新行为后，随着时间推移分配的块减少到了 **_182 KB_**。

![在所有优化之后，分析器显示随着时间的推移分配了 184 KB 的块](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/webpack-memory/allocation-after.png)

在所有优化之后，分析器显示随着时间的推移分配了 184 KB 的块

[Next.js 8](/blog/next-8) 已经内置了所有这些优化。使用 Next.js 时无需更改任何配置。

此优化是在 webpack 上引入的，这意味着不仅仅是 Next.js 用户，所有 webpack 用户都将从这些优化中受益。

我们将继续积极改进 Next.js 和 webpack 的内存使用和性能。
