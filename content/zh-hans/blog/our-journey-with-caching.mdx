前端性能很难调优。即使在高度优化的应用中，迄今为止最常见的原因仍然是客户端 - 服务器瀑布流。在推出 Next.js App Router 时，我们就知道想要解决这个问题。为此，我们需要利用 React Server Components 将客户端到服务器的 REST fetch 移动到服务器端，在单次往返中完成。这意味着服务器有时必须保持动态，从而牺牲了 Jamstack 出色的初始加载性能。我们构建了部分预渲染 (partial prerendering) 来解决这个权衡，从而兼得两全优势。

然而，在此过程中，由于我们提供的缓存默认值和控件，开发体验受到了影响。`fetch()` 的默认值变为偏向性能，默认开启缓存，但这损害了快速原型开发和高度动态体验。我们没有提供足够的控制来访问不使用 `fetch()` 的本地数据库。我们虽然有 `unstable_cache()`，但使用体验不佳。这导致需要段级配置作为逃生通道，例如 `export const dynamic, runtime, fetchCache, dynamicParams, revalidate = ...`。

当然，我们为了向后兼容性将继续支持这一点。但现在，我希望你们暂时忘掉所有那些东西。我们认为我们有一个更简单的点子。

我们一直在研发一种新的实验模式，它仅基于两个概念构建：`<Suspense>` 和 `use cache`。

-----------------------------------------------

你会注意到的第一件事是，当向组件添加数据时，你现在会收到错误。

app/page.tsx

```
async function Component() {
  return fetch(...) // error
}
 
export default async function Page() {
  return <Component />
}
```

要使用数据、cookies、headers、当前时间或随机值，你现在有了一个选择：你希望数据是缓存的（服务器端或客户端）还是在每次请求时执行？我以 `fetch()` 为例，但这适用于任何异步 Node API，例如数据库或计时器。

**动态模式(#dynamic)**
-------------------

如果您仍在迭代或构建高度动态的仪表盘，您可以将组件包裹在 `<Suspense>` 边界中。`<Suspense>` 支持动态数据获取和流式传输。

app/page.tsx

```
async function Component() {
  return fetch(...) // no error
}
 
export default async function Page() {
  return <Suspense fallback="..."><Component /></Suspense>
}
```

你也可以在根布局中执行此操作或使用 `loading.tsx`。

这确保了你应用的基础结构保持即时。你可以继续向页面中添加更多数据，知道它们默认都是动态的。默认不缓存任何内容。不再有隐藏的缓存。

**静态模式(#static)**
-----------------

如果你正在构建静态内容且不想使用动态功能，可以使用新的 `use cache` 指令。

app/page.tsx

```
"use cache"
 
export default async function Page() {
  return fetch(...) // no error
}
```

通过在 Page 上标记 `use cache`，你表明整个段都应当被缓存。这意味着你可以获取的任何数据现在都可以被缓存，从而允许页面静态渲染。静态内容不使用 `<Suspense>` 边界。你可以向页面添加更多数据，它们都将被缓存。

**部分预渲染(#partial)**
-------------------

你也可以混合匹配。例如，你可以在根布局中放置 `use cache` 以确保它被缓存。每个布局或页面都可以独立缓存。

app/layout.tsx

```
"use cache"
 
export default async function Layout({ children }) {
  const response = await fetch(...)
  const data = await response.json()
  return <html>
    <body>
      <div>{data.notice}</div>
      {children}
    </body>
  </html>
}
```

在特定 Page 内使用动态数据时：

app/page.tsx

```
import { Suspense } from 'react'
async function Component() {
  return fetch(...) // no error
}
 
export default async function Page() {
  return <Suspense fallback="..."><Component /></Suspense>
}
```

**缓存函数(#cached-functions)**
-------------------------------------

当使用这种混合方法时，在靠近 API 调用的地方添加缓存可能更方便。

你可以像 `use server` 一样将 `use cache` 添加到任何异步函数中。将其视为 Server Action，但你不是调用 Server，而是调用 Cache。它支持除了 JSON 之外的更多丰富类型的参数和返回值。缓存密钥自动包含任何参数和闭包，因此你不需要手动指定缓存密钥。

app/layout.tsx

```
async function getNotice() {
  "use cache"
  const response = await fetch(...)
  const data = await response.json()
  return data.notice;
}
 
export default async function Layout({ children }) {
  return <html>
    <body>
      <h1>{await getNotice()}</h1>
      {children}
    </body>
  </html>
}
```

由于此布局中没有使用其他数据，它可以保持静态。这种方法的一个好处是，如果你不小心向布局添加了新的动态数据，它会在构建期间触发错误，迫使你重新做出选择。如果你将整个布局都添加到 `use cache`，它将被缓存且不会报错。选择哪种方法取决于你的用例。

**标记缓存(#tagging-a-cache)**
-----------------------------------

如果你想通过标签显式清除缓存条目，可以在 `use cache` 函数中使用新的 `cacheTag()` API。

app/utils.ts

```
import { cacheTag } from 'next/cache';
 
async function getNotice() {
  'use cache';
  cacheTag('my-tag');
}
```

然后，只需像以前一样从 Server Action 调用 `revalidateTag('my-tag')`。

由于此 API 可以在数据加载后调用，你现在可以使用数据来标记缓存条目。

app/actions.ts

```
import { cacheTag } from 'next/cache';
 
async function getBlogPosts(page) {
  'use cache';
  const posts = await fetchPosts(page);
  for (let post of posts) {
    cacheTag('blog-post-' + post.id);
  }
  return posts;
}
```

**定义缓存的生命周期(#defining-the-lifetime-of-a-cache)**
---------------------------------------------------------------------

如果你想控制某个条目或页面在缓存中存活多久，可以使用 `cacheLife()` API：

app/page.tsx

```
"use cache"
import { cacheLife } from 'next/cache'
 
export default async function Page() {
  cacheLife("minutes")
  return ...
}
```

默认情况下，它接受以下值：

*   `"seconds"`
*   `"minutes"`
*   `"hours"`
*   `"days"`
*   `"weeks"`
*   `"max"`

选择一个最适合你用例的大致范围。无需精确指定数字并计算一周有多少秒（还是毫秒？）。但也可以指定特定值或配置自己的命名缓存配置文件。

除了 `revalidate`，此 API 还可以控制客户端缓存的 `stale` 时间以及 `expire`，后者规定了如果页面一段时间没有太多流量，它何时应该过期。

**实验性(#experimental)**
-----------------------------

这仍然是一个非常实验性的项目。它尚未准备好用于生产，且仍缺少功能和包含错误。特别是，我们知道需要改进这种新类型错误的错误堆栈。但是，如果你很有探险精神，我们非常希望得到你们的早期反馈。

我们将发布更详细的升级路径。除了早期错误外，这里的主要破坏性更改是撤销 `fetch()` 的默认缓存。话虽如此，我们建议在这种早期实验阶段仅在新项目上进行实验。如果效果良好，我们希望在小版本中推出可选版本，并在未来的主版本中将其设为默认。

要体验它，你必须在 Next.js 的 `canary` 版本上：

```
npx create-next-app@canary
```

你也必须在 `next.config.ts` 中启用实验性 `dynamicIO` 标志：

next.config.ts

```
import type { NextConfig } from 'next';
 
const nextConfig: NextConfig = {
  experimental: {
    dynamicIO: true,
  }
};
 
export default nextConfig;
```

在我们的文档中阅读有关 [`use cache`](/docs/canary/app/api-reference/directives/use-cache)、[`cacheLife`](/docs/canary/app/api-reference/next-config-js/cacheLife) 和 [`cacheTag`](/docs/canary/app/api-reference/functions/cacheTag) 的更多信息。
