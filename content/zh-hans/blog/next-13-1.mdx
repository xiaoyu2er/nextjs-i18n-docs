提升 app 目录的可靠性和支持 [](#improved-reliability-and-support-for-app-directory)
-----------------------------------------------------------------------------------------------------------

在 Next.js 13 中，我们宣布了新的 `app` 目录（测试版）。这个新的路由和数据获取系统可以与现有的 `pages` 目录一起逐步采用。

`app` 目录提供了许多好处，包括增强的布局、组件、测试和样式的本地化、组件级数据获取等。感谢您的反馈和早期测试，我们针对 `app` 目录的可靠性进行了多项改进：

*   **无需布局 `<div>`：** 之前，`app` 目录在导航时添加了额外的 `<div>` 元素以将布局滚动到视图中。在 13.1 中，不再创建这些额外元素。滚动行为保持不变。
*   **TypeScript 插件：** 我们构建了一个新的 TypeScript 插件，它提供页面和布局配置选项的建议，将文档直接带入您的 IDE，并提供有关 Server 和 Client Components 的有用使用提示（例如防止在 Server Components 中使用 `useState`）。[了解更多](https://www.youtube.com/watch?v=pqMqn9fKEf8)。
*   **可靠性改进：** 我们修复了大量错误，包括改善 CSS modules 支持、正确去重布局和页面的 `cache()` 和 `fetch()`、内存泄漏等。
*   **减少客户端 JavaScript：** `app` 目录现在比 `pages` 目录减少了 `9.3kB` 的客户端 JavaScript。无论是在您的应用中添加 1 个还是 1000 个 Server Components，此基线都不会增加。React 运行时暂时稍大，增加是由于 React Server Components 运行时引起的，它处理了 Next.js 之前处理的机制。我们正在努力进一步减少这一点。

|  | `pages/` | `app/` | 差异 |
| --- | --- | --- | --- |
| 总首屏加载 JS | `基准` | `-9.3kB` | `小 12.1%` |
| Next.js 运行时 | `基准` | `-12kB` | `小 56.8%` |
| React 运行时 | `基准` | `+2.7kB` | `大 5.2%` |

我们对继续推进 `app` 目录的稳定性感到兴奋。[beta 文档](https://beta.nextjs.org/) 对 `app` 目录进行了数百次更新 [基于您的反馈](https://vercel.com/blog/using-vercel-comments-to-improve-the-next-js-13-documentation)。

内置模块转译（稳定版）[](#built-in-module-transpilation-stable)
-------------------------------------------------------------------------------

您现在可以标记来自本地包（如 monorepos）或外部依赖（`node_modules`）的依赖项进行转译和打包。这种内置支持取代了流行的 `next-transpile-modules` 包。

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ['@acme/ui', 'lodash-es'],
};
 
module.exports = nextConfig;
```

我们要感谢 Pierre de la Martinière ([@martpie](https://twitter.com/martpie_)) 为此包所做的工作，以及他们协助确保内置支持满足社区需求。

用于更小包的导入解析 [](#import-resolution-for-smaller-bundles)
-------------------------------------------------------------------------------

许多流行的 npm 包使用 "barrel files" 来提供单个文件以重新导出其他模块。例如：

@acme/ui/index.ts

```
export { default as Button } from './dist/Button';
export { default as Slider } from './dist/Slider';
export { default as Dropdown } from './dist/Dropdown';
```

这使得包的使用者可以一次性使用命名导出：

```
import { Button, Slider, Dropdown } from '@acme/ui';
```

虽然打包工具理解这些 barrels 文件并可以移除未使用的重新导出（称为 "dead code elimination"），但此过程涉及解析/编译所有重新导出的文件。对于发布的库，一些 npm 包发布了重新导出数千个模块的 barrel 文件，这减慢了编译时间。这些库推荐 `babel-plugin-transform-imports` 以避免此问题，但对于使用 SWC 的人来说，之前没有支持。我们已添加一个新的内置于 Next.js 的 SWC 转换，称为 `modularizeImports`。

此新设置启用了 SWC 转换，它根据定义的模式更改您的导入语句。例如，上述使用三个组件的代码将自动转换为使用直接导入，而无需开发人员手动编写此代码：

```
// Before (with barrel file)
import { Button, Slider, Dropdown } from '@acme/ui';
 
// After (with modularized imports from plugin)
import Button from '@acme/ui/dist/Button';
import Slider from '@acme/ui/dist/Slider';
import Dropdown from '@acme/ui/dist/Dropdown';
```

使用 `next.config.js` 中的 `modularizeImports` 选项可以实现此转换：

next.config.js

```
module.exports = {
  modularizeImports: {
    '@acme/ui': {
      transform: '@acme/ui/dist/{{member}}',
    },
  },
};
```

利用此转换与 `@mui/icons-material` 或 `lodash` 可以跳过未使用文件的编译。[了解更多](/docs/architecture/nextjs-compiler#modularize-imports)。

[查看演示](https://twitter.com/nextjs/status/1610288783311732737) 以查看其实际操作效果。

用于边缘的轻量级 Node.js 运行时，现在对 API 路由稳定 [](#a-light-nodejs-runtime-for-the-edge-now-stable-for-api-routes)
---------------------------------------------------------------------------------------------------------------------------------

Next.js 内部的 Edge Runtime 使用了一组严格的 Node.js API 子集（如 `Request`、`Response` 等），这些 API 与 Vercel 等边缘计算平台兼容，或者在自托管时使用。这些 API 处处运行，包括在浏览器中，允许开发人员学习一次并在任何地方编写。

pages/api/hello.ts

```
// "experimental-" prefix is no longer needed
export const config = {
  runtime: 'edge',
};
 
export default function handler(req: Request) {
  return new Response('Hello World');
}
```

Next.js Middleware 默认已经使用此轻量边缘运行时以提供更好的性能。由于 Middleware 可以在应用程序的每个请求之前运行，因此拥有轻量级运行时对于确保低延迟至关重要。在 Next.js 12.2 中，我们添加了可选择为 [API Routes](https://nextjs.org/blog/next-12-2#edge-api-routes-experimental) 使用此运行时的能力。

随着 13.1，Next.js 内部的 Edge Runtime 对 API 路由是**现已稳定**的。当自托管时，使用 Edge Runtime 的 Middleware 和 API 路由默认将作为单一区域工作负载运行，作为 `next start` 的一部分。在 Vercel 上，Next.js Middleware 和 API 路由使用 [Vercel Edge Functions](https://vercel.com/docs/concepts/next.js/overview) 进行全球部署以实现最低延迟。Vercel Edge Functions 现在也已 [正式发布](https://vercel.com/blog/edge-functions-generally-available)。

Turbopack 改进 [](#turbopack-improvements)
-------------------------------------------------

在发布随 Next.js 13 的 [Turbopack](https://nextjs.org/docs/app/api-reference/turbopack) alpha 之后，我们一直专注于提高可靠性，添加最受欢迎功能的支持，并定义插件和其他框架中使用的计划。

自 Next.js 13.0.0 以来，Turbopack：

*   支持 PostCSS，包括 Tailwind CSS
*   支持 `next/image`
*   支持 `@next/font`（Google Fonts）
*   支持从动态 `import()` 语句加载 CSS
*   支持 CSS source maps（感谢 `@ahabhgk` 的 [贡献](https://github.com/vercel/turbo/pull/2839)）
*   改进 `next dev` 错误覆盖中的错误处理
*   改进内存使用
*   改进 CSS modules 支持
*   改进 HMR 更新的块划分算法
*   改进 HMR source maps 的可靠性

我们要感谢 Evan You 和 Vite 社区的反馈和贡献，以确保 Turbopack 基准测试尽可能准确。我们与 Vite 团队合作验证了最新的 Turbopack 基准测试，并对我们的测试方法进行了多项改进。

作为此次合作的结果，我们现在使用更精确的指标，其中包括在 React 更新机制上花费的时间。我们能够将 Turbopack 中的 React Fast Refresh 时间提高了 30ms，同时也提高了 Next.js 13.1 在 webpack 上的性能。我们还添加了一个新的 Vite 与 SWC 使用的基准测试，与使用默认 Vite 配合 Babel 相比，显示出性能的提升。查看 [更新后的基准测试](https://nextjs.org/docs/app/api-reference/turbopack) 或阅读 [测试方法](https://nextjs.org/docs/app/api-reference/turbopack)。

今天就在 Next.js 13 中使用 `next dev --turbo` 试用 Turbopack alpha 版本。如果您有任何反馈，请在 [GitHub 讨论](https://github.com/vercel/turbo/discussions/2312) 中告诉我们。

Next.js 高级 Middleware[](#nextjs-advanced-middleware)
----------------------------------------------------------

感谢您的反馈，我们正在使 Next.js Middleware 前所未有地强大。随着 13.1，您现在可以从 Middleware 返回响应，也可以设置请求头。

这些 API 改进赋予您强大的新灵活性来定制 Next.js 路由生命周期的每一部分。`next.config.js` 内的 `experimental.allowMiddlewareResponseBody` 配置选项不再需要。

您现在可以更轻松地设置请求头，以及直接响应，而无需 `rewrite` 或 `redirect`：

middleware.ts

```
import { NextResponse } from 'next/server';
 
export function middleware(request: Request) {
  // Check if a user has access...
  if (!isAuthorized(request)) {
    return NextResponse.json({ message: 'Unauthorized' });
  }
 
  // Add a new header, this will change the incoming request headers
  // that you can read in getServerSideProps and API routes
  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-version', '13.1');
 
  return NextResponse.next({
    request: {
      // Apply new request headers
      headers: requestHeaders,
    },
  });
}
```

了解更多关于 [Next.js 高级 Middleware](/docs/pages/building-your-application/routing/middleware)。

其他改进 [](#other-improvements)
-----------------------------------------

*   `@next/font` 现在支持在同一字体声明中添加多种字体粗细和样式。[了解更多](/docs/pages/building-your-application/optimizing/fonts#google-fonts)。
*   `next/dynamic` 现在使用 React 原语 `lazy()` 和 `<Suspense>`。之前的 `suspense` 选项不再需要。有了这些更改，`next/dynamic` 现在与 `app` 目录兼容。
*   `create-next-app` 已更新为新设计，现在默认包含 [`@next/font`](/docs/pages/building-your-application/optimizing/fonts) 以实现零布局偏移的自动自托管字体。使用 `npx create-next-app@latest` 试用，或 [部署模板](https://vercel.com/templates/next.js/nextjs-boilerplate)。
*   我们对 [App Directory Playground](https://app-dir.vercel.app/) 做出了多项改进，该演示展示了 Next.js 13 中 `app` 目录（测试版）的一些最新功能和约定。[部署您自己的](https://vercel.com/templates/next.js/app-directory)。
*   我们创建了一个 [高性能图片画廊模板](https://vercel.com/templates/next.js/image-gallery-starter)，其中包括图片占位符、懒加载、自动优化、键盘支持等。[部署您自己的](https://vercel.com/templates/next.js/image-gallery-starter)。
*   我们创建了一份资源，用于了解如何 [迁移大型开源 React 和 Express.js 应用](https://vercel.com/blog/migrating-a-large-open-source-react-application-to-next-js-and-vercel) 到 Next.js，包括详细的分步演示和指向特定提交的链接。
