我们非常高兴向全世界介绍 Next.js 5.0。即日起它已在 npm 上可用。要升级，请运行：

终端

```
npm i next@latest react@latest react-dom@latest
```

> 除了升级 Next.js，我们还将 peer 依赖项 `react` 和 `react-dom` 一并升级

Next.js 是一个用于通用 (universal)、服务器渲染 (server-rendered)(或静态预渲染 (statically pre-rendered)) 的 React.js 应用程序工具箱。开始开发任意规模的应用都很简单，只需执行 `next` 即可。([阅读更多](https://vercel.com/blog/next)。)

在每次新版本发布时，我们致力于保留向后兼容性，提供简单的升级路径，并且仅在绝对必要时才进行 API 变更。Next.js 5.0 也不例外。

然而，在底层，Next.js 已经经历了彻底的转变，以启用强大的新用例和扩展性。我们从让 Next.js 在服务器和客户端代码之间共享通用 Webpack 管道开始。

通用 Webpack 与 Next 插件[](#universal-webpack-and-next-plugins)
-------------------------------------------------------------------------

Next.js 利用了现有的强大工具，如 Webpack、Babel 和 Uglify，向最终用户展示了非常简单的接口：`next` (开发)，`next build` (准备生产环境)，`next start` (服务) 或 `next export` (预渲染到静态文件)。

我们要做出的早期决策之一是为这些工具的如何配置提供非常强大的扩展点。我们不仅希望易用性，还希望启用灵活性，按您想要的任何方式扩展工具箱。

例如，您可以通过在 `next.config.js` 中设置 `webpack` 属性 [扩展 Next.js webpack 配置](https://github.com/vercel/next.js#customizing-webpack-config)。

由于 Webpack 在开发和生产环境中的执行方式不同，我们当时决定将其设为 **函数**，用于装饰我们的默认 Webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    // modify it!
    return config;
  },
};
```

> 一个可选 `next.config.js` 文件示例示例

然而，Webpack 只会针对客户端 (浏览器) 绑定执行，您将错失使用该工具链进行服务器渲染的可能性。

我们很高兴地宣布，我们对代码库进行了全面重构，**以使 Webpack 能够通用工作**。

从您的角度来看，唯一的变化是额外传入了一个 `isServer` 属性给上述装饰器函数。但新的语义意味着，广泛存在的 Webpack 加载器生态现在可供您使用了。

### CSS, LESS, SASS, SCSS 以及 CSS 模块[](#css-less-sass-scss-and-css-modules)

我们收到请求最多的功能之一是能够导入 CSS 文件并利用 Webpack 加载器：

```
import './index.css';
 
export default function Index() {
  return (
    <div>
      <p>I love CSS!</p>
    </div>
  );
}
```

> 一个使用 CSS 导入的示例页面 (`pages/index.js`)，这得益于通用 Webpack

要使这一点生效，您可以将所需的加载器引入为 peer 依赖项：

终端

```
npm i --save css-loader style-loader postcss-loader
```

> Next.js 赋予您选择所需加载器并随心所欲升级到不同版本的自由。

然后扩展配置以配置您的加载器。在 `next.config.js` 中：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dev, isServer } = options;
    const extractCSSPlugin = new ExtractTextPlugin({
      filename: 'static/style.css',
      disable: dev,
    });
    config.module.rules.push({
      test: /\\.css$/,
      use: cssLoaderConfig(extractCSSPlugin, {
        cssModules,
        dev,
        isServer,
      }),
    });
    return config;
  },
};
```

> 扩展原生 Webpack 配置可以为您提供极大的灵活性和控制力

虽然我们的总体建议是使用组件级别的样式解决方案，例如 [内置的 `styled-jsx` Babel 插件](https://github.com/vercel/next.js#built-in-css-support)，但我们认为 CSS 加载器具有许多重要的优势，例如轻松重用现有的 CSS 代码库，并极大地简化旧代码库迁移到 Next.js。

与其默认启用每种设想到的功能和加载器，我们正在引入 [**Next.js 插件**](https://github.com/vercel/next-plugins)，它们是 _装饰您配置_ 的简单函数。与其像上面那样手动扩展配置来设置加载器，您只需这样做：

```
const withCss = require('next-css');
module.exports = withCss({
  /* extra optional config */
});
```

> 启用导入 `.css` 文件只需引入 `next-css`

阅读关于 [CSS 加载器](https://github.com/vercel/next-plugins/tree/master/packages/next-css) 与 Next.JS 的用法的更多文章，或者参考我们已经为您创建的一些包：

| 加载器 | 包 |
| --- | --- |
| CSS | [next-css](https://github.com/vercel/next-plugins/tree/master/packages/next-css) |
| LESS | [next-less](https://github.com/vercel/next-plugins/tree/master/packages/next-less) |
| SASS | [next-sass](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) |

我们的目标是赋能社区开发并培养一个实用的简单扩展生态系统。为此，我们打开了 [next-plugins](https://github.com/vercel/next-plugins) monorepo，供 Next.js 社区维护。欢迎所有 PR！

### TypeScript 支持[](#typescript-support)

JavaScript 生态系统中增长最快的技术之一 [之一](https://stateofjs.com/2017/flavors/results) 是 TypeScript。以至于它正在获得 [Babel 7 的官方支持](https://github.com/babel/babel/tree/master/packages/babel-preset-typescript)，这意味着它将通过简单地 [自定义您的 .babelrc](https://github.com/vercel/next.js#customizing-babel-config) 自然被 Next.js 支持。

与此同时，得益于我们新的通用 Webpack 支持，您今天就可以获得完整的 TypeScript 支持！

您可以像这样扩展您的 webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dir, defaultLoaders } = options;
    config.resolve.extensions.push('.ts', '.tsx');
    config.module.rules.push({
      test: /\\.+(ts|tsx)$/,
      include: [dir],
      exclude: /node_modules/,
      use: [
        defaultLoaders.babel,
        { loader: 'ts-loader', options: { transpileOnly: true } },
      ],
    });
    return config;
  },
};
```

> 我们要做的只是启用 `ts-loader`

像 CSS 加载器和预处理器一样，TypeScript 是最受欢迎的功能之一。要将它集成到项目中变得像其他加载器一样容易，我们现在有了您包含在 `next.config.js` 文件中的 [`next-typescript` 插件](/)：

next.config.js

```
const withTs = require('next-typescript');
module.exports = withTs({
  /* additional config*/
});
```

> 插件可以非常简单组合：它们只是函数

### 更好地支持 React 替代库 & 模块重载[](#better-support-for-react-altlibs--module-overloading)

随着时间的推移，出现了许多 React 的直接替换实现。其中一些 notable 的有 [`preact`](https://preactjs.com/)、[`nervjs`](https://github.com/NervJS/nerv) 和 [`inferno`](https://github.com/infernojs/inferno)。

其他库专注于替换 DOM 渲染器，如 [`react-dom-lite`](https://github.com/jquense/react-dom-lite)，它旨在通过引入一些浏览器兼容性上的微小权衡来使 React 构建更小。

通用 Webpack 支持使得 **将这些库作为直接替换进行集成变得更加容易**。与其他插件一样，这是您配合 Preact 使用 Next.js 所需做的全部：

终端

```
npm i @zeit/next-preact preact preact-compat
```

> 我们安装了 preact 插件和必要的 peer 依赖项

```
const withPreact = require('@zeit/next-preact');
module.exports = withPreact();
```

> 我们新的 `next.config.js` 为 preact 准备好了

查看非常简单直接的 [@zeit/next-preact](https://github.com/vercel/next-plugins/tree/master/packages/next-preact) 模块或者创建您自己的！

### 生产环境中的可选外部 Source Maps[](#optional-external-sourcemaps-in-production)

现在 Webpack 被 Next.js 用于客户端和服务器代码，在生产构建中启用 Source Maps 仅需要对其配置进行一小项调整。

在开发环境中 Source Maps 会自动启用，所以我们在生产环境中进行不同的配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
    }
    return config;
  },
};
```

> 一个可选 `next.config.js` 文件示例

### 结论

通用 Webpack 巩固了 Next.js 的基础，使其更具未来适应性。总的来说，不再存在插件或加载器哪些适用于 Next.js 哪些不适用的任何人为分隔。

秉承 _零配置_ 的精神，我们要兴奋引入 [Next Plugins](https://github.com/vercel/next-plugins)，这是社区仓库，收录了自动扩展 Next.js 功能的食谱，而无需调整特定设置。

通过这种方式，我们现在支持整个范围的 CSS 解决方案、像 TypeScript 这样的 compile-to-js 语言以及替代品如 [Nerve](https://github.com/NervJS/nerv)，只需引入额外的模块并在 `next.config.js` 中明确包含它。简单而不晦涩。

区域允许互连不位于同一仓库甚至不在同一服务器上的 Next.js 应用。我们认为这是在“团队扩展性”类别改进中非常重要的一步。

因此 Next.js 成为由多个团队维护的大型应用的绝佳候选者。他们现在可以并行部署改进，减少错误面，提高迭代速度，甚至尝试除我们的核心之外的不同技术，例如状态管理或数据获取的 [许多不同方法](https://github.com/vercel/next.js/tree/canary/examples)。

我们借此机会感谢 Deep Varma 和 [Trulia](https://trulia.com) 工程团队贡献了导致该功能设计的关键见解、代码和测试。

Improved Caching for Dynamic Imports[](#improved-caching-for-dynamic-imports)
-----------------------------------------------------------------------------

Whenever you use dynamic `import()`, this signals to WebPack that a new code-splitting entry-point exists.

At build time, this means generating a specific bundle for the corresponding subtree of modules.

Prior to Next.js 5.0, the dynamic bundles would receive a URL that resembles the following:

```
/_next/1517592683901/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6.js
```

Now, we have turned dynamic imports into content-addressable hashes of the contents of the subtree:

```
/_next/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6-b7874680a9e21fb6eb89.js
```

This means that across deployments, your users and customers won't have to needlessly re-download the code they've already used.

Fragments[](#fragments)
-----------------------

Next.js build a top-level `<Document>` component that gets server-rendered with each page. [Overloading this component](/) puts you in complete control over your markup, enabling many [advanced use cases](https://github.com/vercel/next.js/tree/canary/examples).

Part of that initial markup is the list of scripts that Next.js needs to evaluate on the client side. A custom `_document` looks like this:

pages\_document.js

```
import Document, { Head, Main, NextScript } from 'next/document';
export default class extends Document {
  render() {
    return (
      <html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </html>
    );
  }
}
```

> `Document` 允许您自定义页面的整个服务端渲染输出

Until recently, we were forced to wrap our scripts inside a `<div>`.

With Next.js 5.0, we now take advantage of the new `Fragment` support, which translates into **more lightweight pages** and complete control over the styling of the page, with no extraneous markup.

More Accurate Errors[](#more-accurate-errors)
---------------------------------------------

Node.js doesn't support source maps, errors that happen on the server side are accompanied with a stack trace pointing to the compiled code.

With Next 5 we've improved source map support on the server side. Errors that happen when server rendering now point to the correct function and line number.

结论<a id="conclusion"></a>

结论<a id="conclusion"></a>

Universal Webpack solidifies the foundations of Next.js and makes it even more future proof. By and large, there is no longer an artificial separation of what plugins or loaders are applicable to Next.js and which ones aren't.

In the spirit of _zero configuration_, we are excited to introduce [Next Plugins](https://github.com/vercel/next-plugins), the community repository for recipes that automatically expand the functionality of Next.js, without having to tweak specific knobs.

With this, we now support the entire spectrum of CSS solutions, compile-to-js languages like TypeScript and React alternatives like [Nerve](https://github.com/NervJS/nerv), by just bringing an additional module and being explicit about its inclusion in `next.config.js`. Simplicity without obscurity.

Zones allow interconnecting Next.js apps that are not rooted in the same repository or even server(s). We consider this a very important milestone in the "team scalability" category of improvements.

Next.js thus becomes a great candidate for large applications maintained by multiple teams. They can now deploy improvements concurrently, reducing error surfaces, increasing iteration speed and even trying out different technologies in addition to our core, like the [many different approaches](https://github.com/vercel/next.js/tree/canary/examples) to state management or data fetching.

We want to take this opportunity to thank Deep Varma and the [Trulia](https://trulia.com) engineering team for contributing key insights, code and testing that led to the design of this feature.

As always, this release wouldn't have been possible without the many open-source contributors and our wonderful community.
