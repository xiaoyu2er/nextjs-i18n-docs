我们非常高兴向全世界介绍 Next.js 5.0。即日起它已在 npm 上可用。要升级，请运行：

终端

```
npm i next@latest react@latest react-dom@latest
```

> 除了升级 Next.js，我们还升级了 peer 依赖 `react` 和 `react-dom`

Next.js 是一个用于通用、服务端渲染（静态预渲染）React.js 应用的工具箱。开发任何规模的都同样简单，只需执行 `next`。 ([了解更多](https://vercel.com/blog/next).)

随着每一次新版本的发布，我们都致力于保持向后兼容性，提供简单的迁移路径，并且仅在绝对必要时进行 API 更改。Next.js 5.0 也不例外。

然而，在系统底层，Next.js 经历了彻底的变革，以支持强大的新用例和可扩展性。我们首先让 Next.js 为服务器和客户端代码共享通用的 Webpack 管线。

通用 Webpack 和 Next.js 插件

Next.js 利用了现有的强大工具，如 Webpack、Babel 和 Uglify，并以极其简单的界面呈现给用户：`next`（开发）、`next build`（准备生产）和 `next start`（服务）或 `next export`（预渲染为静态文件）。

我们做出的早期决策之一是为这些工具的配置提供非常强大的可扩展点。我们不仅想要易用性，还希望能让用户根据自己的意愿扩展工具箱。

例如，您可以通过在 `next.config.js` 中设置 `webpack` 属性来 [扩展 Next.js Webpack 配置](https://github.com/vercel/next.js#customizing-webpack-config)。

由于 Webpack 在生产环境和开发环境中的执行方式不同，我们当时决定使其为**一个函数**来修饰我们的默认 Webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    // modify it!
    return config;
  },
};
```

> 一个可选 `next.config.js` 文件示例

然而，Webpack 仅在客户端（浏览器）包上执行，您将失去利用此工具链进行服务端渲染的可能性。

我们很高兴地宣布，我们已经彻底重构了代码库，**让 Webpack 通用**。

从您的角度来看，变化仅在于装饰器函数将额外的 `isServer` 属性传递到了上面。但新的语义意味着广泛的 Webpack 加载器生态系统现在可供您使用。

### CSS, LESS, SASS, SCSS 和 CSS 模块

我们用户请求最多的功能之一是可以导入 CSS 文件并利用 Webpack 加载器：

```
import './index.css';
 
export default function Index() {
  return (
    <div>
      <p>I love CSS!</p>
    </div>
  );
}
```

> 一个使用 CSS 导入的示例页面 (`pages/index.js`)，得益于通用 Webpack

为了让它起作用，您可以将所需加载器作为 peer 依赖引入：

终端

```
npm i --save css-loader style-loader postcss-loader
```

> Next.js 赋予您自由选择所需加载器并根据需要将其升级到不同版本的自由。

然后扩展配置以配置您的加载器。在 `next.config.js` 中：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dev, isServer } = options;
    const extractCSSPlugin = new ExtractTextPlugin({
      filename: 'static/style.css',
      disable: dev,
    });
    config.module.rules.push({
      test: /\\.css$/,
      use: cssLoaderConfig(extractCSSPlugin, {
        cssModules,
        dev,
        isServer,
      }),
    });
    return config;
  },
};
```

> 扩展原始 Webpack 配置赋予了您极大的灵活性和控制力

虽然我们的总体建议是使用组件级别的样式解决方案，例如 [包含的 `styled-jsx` Babel 插件](https://github.com/vercel/next.js#built-in-css-support)，但我们认为 CSS 加载器具有许多重要的优势，例如轻松重用现有 CSS 代码库并极大简化旧代码库迁移到 Next.js。

为了启用几乎所有预想的功能和加载器，我们引入了 [**Next.js 插件**](https://github.com/vercel/next-plugins)，它们是**修饰您配置的简单函数**。不像我们上面那样手动扩展配置来设置加载器，您只需这样做：

```
const withCss = require('next-css');
module.exports = withCss({
  /* extra optional config */
});
```

> 启用导入 `.css` 文件只需引入 `next-css`

了解关于 [CSS 加载器](https://github.com/vercel/next-plugins/tree/master/packages/next-css) 用法与 Next.JS 的使用，或参考我们要已经为您创建的一些包：

| 加载器 | 包 |
| --- | --- |
| CSS | [next-css](https://github.com/vercel/next-plugins/tree/master/packages/next-css) |
| LESS | [next-less](https://github.com/vercel/next-plugins/tree/master/packages/next-less) |
| SASS | [next-sass](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) |

我们的目标是赋能社区开发和增长实用的简单扩展生态系统。为此，我们开放 [next-plugins](https://github.com/vercel/next-plugins) monorepo 供 Next.js 社区维护。所有 PR 都受欢迎！

### TypeScript 支持

JavaScript 生态系统中 [增长最快的技术之一](https://stateofjs.com/2017/flavors/results) 是 TypeScript。以至于它正在成为 [Babel 7 的官方支持](https://github.com/babel/babel/tree/master/packages/babel-preset-typescript)，这意味着它将自然地被 Next.js 支持，只需 [自定义您的 .babelrc](https://github.com/vercel/next.js#customizing-babel-config)。

同时，得益于我们新的通用 Webpack 支持，您现在已经可以获得完整的 TypeScript 支持了！

您可以像这样扩展您的 Webpack 配置：

next.config.js

```
module.exports = {
  webpack(config, options) {
    const { dir, defaultLoaders } = options;
    config.resolve.extensions.push('.ts', '.tsx');
    config.module.rules.push({
      test: /\\.+(ts|tsx)$/,
      include: [dir],
      exclude: /node_modules/,
      use: [
        defaultLoaders.babel,
        { loader: 'ts-loader', options: { transpileOnly: true } },
      ],
    });
    return config;
  },
};
```

> 我们只需要启用 `ts-loader`

像 CSS 加载器和预处理器一样，TypeScript 是最常见的请求功能之一。为了使其像任何其他加载器一样轻松纳入项目，现在我们有了 [`next-typescript` 插件](/)，您可以将其包含在 `next.config.js` 文件中：

next.config.js

```
const withTs = require('next-typescript');
module.exports = withTs({
  /* additional config*/
});
```

> 插件可以轻松组合：它们只是函数

### 更好的 React 替代库和模块重载支持

过去一段时间涌现了许多 React 的即插即用替代实现。其中最突出的一些是 [`preact`](https://preactjs.com/)、[`nervjs`](https://github.com/NervJS/nerv) 和 [`inferno`](https://github.com/infernojs/inferno)。

其他库专注于替换 DOM 渲染器，如 [`react-dom-lite`](https://github.com/jquense/react-dom-lite)，旨在通过引入浏览器兼容性的一些微小权衡来实现更小的 React 构建。

通用 Webpack 支持使得**将这些库作为即插即用替代品集成变得更加容易**。与其他插件一样，这只是使用 Next.js 配合 preact 您需要做的：

终端

```
npm i @zeit/next-preact preact preact-compat
```

> 我们安装 preact 插件和必要的 peer 依赖

```
const withPreact = require('@zeit/next-preact');
module.exports = withPreact();
```

> 我们的新 `next.config.js` 准备好配合 preact 使用

查看非常简单的 [@zeit/next-preact](https://github.com/vercel/next-plugins/tree/master/packages/next-preact) 模块或创建您自己的！

### 生产环境中有可选的 external 源映射

现在 Webpack 已被 Next.js 用于客户端和服务器代码，在生产构建中启用 source maps 只是对其配置的一点小调整。

在开发中 source maps 会自动启用，所以我们针对生产进行不同的配置：

next.config.js

```
module.exports = {
  webpack(config, { dev }) {
    if (!dev) {
      config.devtool = 'source-map';
    }
    return config;
  },
};
```

> 我们只需配置 `devtool` 选项在不处于开发环境时不同

### 区域

从一开始，[Next.js 的既定目标之一](https://deck.vercel.app) 就是恢复并保持 Web 的简洁性。

服务端渲染、一种简单且独立于平台的数据获取方法以及基于文件系统结构的声明式页面，就是我们在这一理念下引入的一些功能。

Web 服务和网站的一个经常被忽视的方面是它们如何 _自然地组合和扩展_。

例如，`mydomain.com/settings` 和 `mydomain.com/` 可以是两个完全不同的应用，独立部署、独立扩展，甚至运行同一软件的不同版本。

要将它们 _"粘合"_" 成一个统一的用户体验，只需 [一些简单的配置](/docs/pages/building-your-application/configuring/absolute-imports-and-module-aliases) 后端路由层或将它们暴露给世界的负载均衡器。我们现在很高兴带来**组合多个使用 Next.js 构建的应用**的能力，通过普通的 `<Link>` 组件连接。我们将该功能称为 **区域**。

作为示例，考虑这两个部署到 [Vercel](https://vercel.com) 的独立 Next.js 应用：

*   [https://front.vercel.app](https://front.vercel.app)
*   [https://zeit-docs-zbqbsrucga.vercel.app/docs](https://zeit-docs-zbqbsrucga.vercel.app/docs) [https://zeit-docs-zbqbsrucga.vercel.app/api](https://zeit-docs-zbqbsrucga.vercel.app/api)

![Both of our pages have a seamless experience, but they belong to separate apps](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/zones.png)
我们的两个页面都有无缝体验，但它们属于独立的应用

当我们改版我们的文档时，我们希望让接受社区贡献尽可能容易。

我们决定将文档“迷你网站”拆分到其 [自己的仓库](https://github.com/vercel/docs) 中。此外，每当提交拉取请求并提出更改时，我们都会自动部署，并在隔离中：

![Every time a change happens inside a PR our bot automatically deploys it](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/github.png)
每当 PR 内部发生变化时，我们的机器人会自动部署它

我们最后得到了 _两个区域_，使用我们的 [路径别名功能](/docs/pages/building-your-application/configuring/absolute-imports-and-module-aliases) 将它们聚合到父域 `https://vercel.com`。看起来像这样：

```
{
  "rules": [
    { "pathname": "/docs", "dest": "our-docs.vercel.app" },
    { "pathname": "/api", "dest": "our-docs.vercel.app" },
    { "dest": "my-main-website.vercel.app" }
  ]
}
```

> 这些简单的规则允许您将微服务和区域组合在一起

剩下的就是调用 `now alias` 命令：

终端

```
now alias -r rules.json my-domain.com
```

我们的使命是让部署尽可能通用和开放。为了协助本地开发，我们最近开源了 `micro-proxy`，这是一个与上述配置格式配合使用的工具。

您同样能够与其他解决方案如 Nginx, HAProxy 或 API Gateway 一起连接区域。

### 动态导入的改进缓存

每次您使用动态 `import()` 时，这向 WebPack 指示存在新的代码拆分入口点。

在构建时，这意味着为模块的相应子树生成特定的包。

在 Next.js 5.0 之前，动态包会接收类似以下 URL：

```
/_next/1517592683901/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6.js
```

现在，我们将动态导入转换为了子树内容的可寻址哈希：

```
/_next/webpack/chunks/components_hello1_1345d10fc951cd6717c5676c467579a6-b7874680a9e21fb6eb89.js
```

这意味着在部署之间，您的用户和客户无需重新下载他们已使用的代码。

### 片段

Next.js 构建一个顶级 `<Document>` 组件，将在每个页面进行服务端渲染。[重载此组件](/) 让您完全控制您的标记，启用许多 [高级用例](https://github.com/vercel/next.js/tree/canary/examples)。

该初始标记的一部分是 Next.js 需要在客户端评估的脚本列表。一个自定义 `_document` 看起来像这样：

pages/_document.js

```
import Document, { Head, Main, NextScript } from 'next/document';
export default class extends Document {
  render() {
    return (
      <html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </html>
    );
  }
}
```

> `Document` 允许您自定义页面的整个服务端渲染输出

直到最近，我们被迫将脚本包裹在 `<div>` 内。

### 更准确的错误

Node.js 不支持 source maps，发生在服务端的错误附带指向编译代码的堆栈跟踪。

通过 Next 5，我们改进了服务端的 Source Map 支持。发生在服务端渲染时的错误现在指向正确的函数和行号。

![Errors now show the correct line, file and function name](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-5/errors.png)
现在错误显示正确的行、文件和函数名

### 结论

通用 Webpack 巩固了 Next.js 的基础，使其更具未来适应性。总的来说，不再存在插件或加载器哪些适用于 Next.js 哪些不适用的任何人为分隔。

秉承 _零配置_ 的精神，我们要兴奋引入 [Next Plugins](https://github.com/vercel/next-plugins)，这是社区仓库，收录了自动扩展 Next.js 功能的食谱，而无需调整特定设置。

通过这种方式，我们现在支持整个范围的 CSS 解决方案、像 TypeScript 这样的 compile-to-js 语言以及替代品如 [Nerve](https://github.com/NervJS/nerv)，只需引入额外的模块并在 `next.config.js` 中明确包含它。简单而不晦涩。

区域允许互连不位于同一仓库甚至不在同一服务器上的 Next.js 应用。我们认为这是在“团队扩展性”类别改进中非常重要的一步。

因此 Next.js 成为由多个团队维护的大型应用的绝佳候选者。他们现在可以并行部署改进，减少错误面，提高迭代速度，甚至尝试除我们的核心之外的不同技术，例如状态管理或数据获取的 [许多不同方法](https://github.com/vercel/next.js/tree/canary/examples)。
