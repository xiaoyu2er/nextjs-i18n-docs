开发者体验 (DX) 改进[](#dx-improvements)
-----------------------------------

Next.js 的首要目标之一是在**尽可能好的开发者体验**下提供最佳的生产性能。此发布为构建和调试流水线带来了许多重大改进

### 编译速度[](#compilation-speed)

得益于 Webpack 4、Babel 7 以及代码库中无数的改进和优化，Next.js 在开发期间的启动速度现在提升了多达 57%。

得益于我们新的增量编译缓存，您对代码所做的更改现在将_以快 40% 的速度构建_。

以下是我们收集的一些示例数据：

|  | 6.0 | 7.0 | 差异 |
| --- | --- | --- | --- |
| 启动时间（基本应用） | 1947ms | 835ms | **57%** _更快_ |
| 代码变更（基本应用） | 304ms | 178ms | **42%** _更快_ |

Webpack 4[](#webpack-4)
-----------------------

从最初的版本开始，Next.js 就一直使用 webpack 来打包您的代码并复用丰富的插件和扩展生态系统。我们很兴奋地宣布，Next.js 现在由最新的 webpack 4 驱动，其中包含大量的改进和错误修复。

其中包括：

*   支持 `.mjs` 源码文件
*   代码拆分改进
*   更好的摇树优化（移除未使用代码）支持

另一个新特性是 WebAssembly 支持，Next.js 甚至可以服务端渲染 WebAssembly，这是一个 [示例](https://github.com/vercel/next.js/tree/canary/examples/with-webassembly)。

**注意：** 此升级**完全向后兼容**。然而，如果您通过 [next.config.js](/docs#customizing-webpack-config) 使用了自定义 webpack loader 或插件，您可能需要升级它们。

### CSS 导入[](#css-imports)

随着 webpack 4 的引入，推出了一种新的从代码包提取 CSS 的方法，称为 [mini-extract-css-plugin](https://github.com/webpack-contrib/mini-css-extract-plugin)。

[`@zeit/next-css`](https://github.com/vercel/next-plugins/tree/master/packages/next-css)、[`@zeit/next-less`](https://github.com/vercel/next-plugins/tree/master/packages/next-less)、[`@zeit/next-sass`](https://github.com/vercel/next-plugins/tree/master/packages/next-sass) 和 [`@zeit/next-stylus`](https://github.com/vercel/next-plugins/tree/master/packages/next-stylus) 现在由 `mini-extract-css-plugin` 驱动。

这些 Next.js 插件的**新版本解决了 20 个与 CSS 导入相关的现有问题**；作为一个例子，在动态 `import()` 中导入 CSS 现已支持：

一项重大改进是您不再需要在 `pages/_document.js` 中添加以下内容：

```
import './my-dynamic-component.css';
 
export default function MyDynamicComponent() {
  return <h1>My dynamic component</h1>;
}
```

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import dynamic from 'next/dynamic'
 
const MyDynamicComponent = dynamic(import('../components/my-dynamic-component'))
 
export default function Index() {
  return () {
    <div>
      <MyDynamicComponent/>
    </div>
  }
}
```

简而言之，要在 Next.js 项目中支持导入 `.css` 文件，您只需要在 `next.config.js` 中**注册 withCSS 插件**即可：

```
<link rel="stylesheet" href="/_next/static/style.css" />
```

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还会自动在 CSS URL 中添加内容哈希，以便如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```

module.exports = withCSS({

```

### 标准统一动态导入[](#standardized-dynamic-imports)

从 Version 3 开始，Next.js 就支持通过 `next/dynamic` 导入动态组件。

作为这一技术的早期采用者，我们不得不编写自己的解决方案来处理 `import()`。

因此，Next.js 开始与 webpack 稍后为其引入的支持偏离，一些功能缺失。

因此 Next.js 不支持一些 webpack 后来引入的 `import()` 功能。

例如，手动将某些文件命名和打包在一起是不可能的：

```
import(/* webpackChunkName: 'my-chunk' */ '../lib/my-library');
```

另一个例子是不被 `next/dynamic` 模块包裹的情况下使用 `import()`。

从 Next.js 7 开始，我们不再干预默认的 `import()` 行为。这意味着您可以开箱即获**完整的 `import()` 支持**。

此更改也完全向后兼容。使用动态组件依然很简单：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import dynamic from 'next/dynamic';
 
const MyComponent = dynamic(import('../components/my-component'));
 
export default function Index() {
  return (
    <div>
      <MyComponent />
    </div>
  );
}
```

最重要的是，如果不渲染，`<script>` 标签**不会包含在初始 HTML 文档负载中**。

为了进一步简化代码库并利用出色的 React 生态系统，Next.js 7 中的 `next/dynamic` 被重写，以在幕后使用 [`react-loadable`](https://github.com/jamiebuilds/react-loadable)（经过一些微小修改）。这也为动态组件引入了两个很棒的特性：

为了进一步简化我们的代码库并利用出色的 React 生态系统，在 Next.js 7 中 `next/dynamic` 被重写，以便在幕后使用 [`react-loadable`](https://github.com/jamiebuilds/react-loadable)（经过一些微小修改）。这也为动态组件引入了两个很棒的特性：

*   使用 `next/dynamic` 上的 `timeout` 选项进行超时
*   加载组件延迟，使用 `next/dynamic` 上的 `delay` 选项。此延迟允许您的 `loading` 组件在渲染加载状态之前等待 x 时间，例如，如果导入非常快。

Babel 7[](#babel-7)
-------------------

Next.js 6 在 Babel 7 仍处于测试版时引入了 Babel 7。此后，Babel 7 的稳定版本已发布，Next.js 7 现在使用此版本。

有关更改的完整列表，您可以参考 Babel 的 [发布说明](http://babeljs.io/blog/2018/08/27/7.0.0)。

一些主要特性包括：

*   TypeScript 支持，对于 Next.js 您可以使用 [`@zeit/next-typescript`](https://github.com/vercel/next-plugins/tree/master/packages/next-typescript)
*   Fragment 语法 `<>` 支持
*   `babel.config.js` 支持
*   `overrides` 属性可仅对部分文件或目录应用预设/插件

如果您的 Next.js 项目没有自定义 Babel 配置，则没有破坏性更改。

但是，如果您确实有自定义 Babel 配置，则必须将相应的自定义插件/预设升级到最新版本。

如果您是从低于 Next.js 6 的版本进行升级，您可以运行出色的 [`babel-upgrade`](http://babeljs.io/blog/2018/08/27/7.0.0#babel-upgrade) 工具。

除了升级到 Babel 7 外，Next.js Babel 预设 ([`next/babel`](https://github.com/vercel/next.js#customizing-babel-config)) 现在默认在 `NODE_ENV` 设置为 `test` 时将 `modules` 选项设置为 `commonjs`。

此配置选项通常是创建自定义 `.babelrc` 在 Next.js 项目中唯一的理由：

在 Next.js 7 中，这变为：

```
{
  "env": {
    "development": {
      "presets": ["next/babel"]
    },
    "production": {
      "presets": ["next/babel"]
    },
    "test": {
      "presets": [["next/babel", { "preset-env": { "modules": "commonjs" } }]]
    }
  }
}
```

此时，可以移除 `.babelrc`，因为如果没有 Babel 配置，Next.js 将自动使用 `next/babel`。

在 Next.js 7 中，这变为：

```
{
  "presets": ["next/babel"]
}
```

此时，可以移除 `.babelrc`，因为如果没有 Babel 配置，Next.js 将自动使用 `next/babel`。

### 更小的初始 HTML 载荷[](#smaller-initial-html-payload)

由于 Next.js 预渲染 HTML，它将页面包装为包含 `<html>`、`<head>`、`<body>` 以及渲染页面所需的 JavaScript 文件的默认结构。

此初始载荷之前约为 1.62kB。使用 Next.js 7，我们优化了初始 HTML 载荷，现在为 1.5kB，减少了 7.4%，使您的页面更轻量。

|  | 6.0 | 7.0 | 差异 |
| --- | --- | --- | --- |
| 文档大小（服务端渲染） | 1.62kb | 1.50kb | **7.4%** _更小_ |

减少大小的主要方式如下：

*   `__next-error` div 被移除
*   内联脚本已压缩，在未来的版本中它们将被完全移除
*   未使用 `__NEXT_DATA__` 属性被编译消除，例如 `nextExport` 和 `assetPrefix` 属性。

### 静态 CDN 支持[](#static-cdn-support)

在 Next.js 5 中，我们引入了 `assetPrefix` 支持，这是一种让 Next.js 自动从某个位置加载资源的方法，通常是 CDN。如果 CDN 支持代理，此选项效果很好，例如您请求以下 URL：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
```

通常，CDN 会检查其缓存中是否有该文件，否则直接从源请求。

代理资源正是 [Edge Network](https://vercel.com/edge-network) 的工作原理。

然而，某些解决方案需要将目录直接预上传到 CDN。这样做的一个问题是，Next.js 的 URL 结构与 `.next` 文件夹中的文件夹结构不匹配。例如我们之前的示例

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/page/index.js
```

在 Next.js 7 中，我们更改了 `.next` 的目录结构，以匹配 URL 结构：

```
https://cdn.example.com/_next/static/<buildid>/pages/index.js
// mapped to:
.next/static/<buildid>/pages/index.js
```

虽然我们建议使用代理型 CDN，但这个新结构允许使用其他类型 CDN 的用户将 `.next` 目录上传到他们的 CDN。

### Styled JSX v3[](#styled-jsx-v3)

我们很高兴推出 styled-jsx 3，Next.js 中默认包含的 CSS-in-JS 解决方案现可为 [React Suspense](https://www.youtube.com/watch?v=z-6JC0_cOns) 提供支持。

一个经常不清楚的问题是，如果组件不属于当前组件的范围，该如何为子组件样式化。例如，如果您包含的子组件仅在父组件内部使用时才需要特定样式：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
const ChildComponent = () => (
  <div>
    <p>some text</p>
  </div>
);
 
export default function Index() {
  return (
    <div>
      <ChildComponent />
      <style jsx>{`
        p {
          color: black;
        }
      `}</style>
    </div>
  );
}
```

在 styled-jsx 3 中，通过引入一个新 API `css.resolve` 解决了此问题，该 API 将为给定的 styled-jsx 字符串生成 `className` 和 `<style>` 标签（`styles` 属性）：

此新 API 允许透明地将自定义样式透传给子组件。

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import css from 'styled-jsx/css';
 
const ChildComponent = ({ className }) => (
  <div>
    <p className={className}>some text</p>
  </div>
);
 
const { className, styles } = css.resolve`
  p {
    color: black;
  }
`;
 
export default function Index() {
  return (
    <div>
      <ChildComponent className={className} />
      {styles}
    </div>
  );
}
```

使用 styled-jsx 3，全局样式必须标记为 `css.global` 而不是 `css`，这样 styled-jsx 就可以优化 bundle 大小。

有关所有更改，请参阅 [发布说明](https://github.com/vercel/styled-jsx/releases/tag/v3.0.0)。

### App 与 Pages 之间的 React Context 和 SSR[](#react-context-with-ssr-between-app-and-pages)

从 Next.js 7 开始，我们现在支持 `pages/_app.js` 和页面组件之间的新 React context API。

以前在服务器端页面之间无法使用 React context。原因是 webpack 保存内部模块缓存而不是使用 require.cache，我们编写了一个自定义 webpack 插件来改变此行为，以便在页面之间共享模块实例。

这样我们不仅允许使用新的 React context，还减少了在页面之间共享代码时 Next.js 的内存占用。

|  | 6.0 | 7.0 | 差异 |
| --- | --- | --- | --- |
| 内存使用 | 57.5MB | 48.1MB | **\-16%** _内存_ |
