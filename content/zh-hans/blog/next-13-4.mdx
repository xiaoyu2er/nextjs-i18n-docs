Next.js 13.4 是一个奠基性的版本，标志着 App Router 的稳定性：

*   [**App Router（稳定版）**](#nextjs-app-router):
    *   React 服务端组件 (React Server Components)
    *   嵌套路由与布局 (Nested Routes & Layouts)
    *   简化的数据获取 (Simplified Data Fetching)
    *   流式传输与 Suspense (Streaming & Suspense)
    *   内置 SEO 支持 (Built-in SEO Support)
*   [**Turbopack（测试版）**](#turbopack-beta): 您的本地开发服务器，速度更快且稳定性更高 (Your local dev server, faster and with improved stability)
*   [**服务端操作 (Server Actions)（测试版）**](#server-actions-alpha): 在服务端修改数据，无需客户端 JavaScript (Mutate data on the server with zero client JavaScript)

自从 [Next.js 13](/blog/next-13) 发布六个月以来，我们致力于构建 Next.js 的未来基础——App Router——使其能够以不破坏现有功能的方式进行增量采用。

今天，随着 13.4 版本的发布，您现在可以开始在prod中使用 App Router 了。

```
npm i next@latest react@latest react-dom@latest eslint-config-next@latest
```

Next.js App Router
----------------------------------------

我们于 2016 年发布了 Next.js，旨在提供一种轻松地为 React 应用服务器渲染的方法，我们的目标是创建一个更加动态、个性化和全球化的 Web。

[在最初的公告文章中](https://vercel.com/blog/next)，我们分享了 Next.js 的一些设计原则：

*   **零配置。将文件系统作为 API 使用 (Zero setup. Use the filesystem as an API)**
*   **仅使用 JavaScript。一切都是函数 (Only JavaScript. Everything is a function)**
*   **自动服务端渲染和代码分割 (Automatic server rendering and code splitting)**
*   **数据获取由开发者决定 (Data fetching is up to the developer)**

Next.js 现在已成立六年。我们的原始设计原则保持不变——随着 Next.js 被更多的开发者和公司采用，我们一直在对框架进行奠基性的升级，以更好地实现这些原则。

我们一直在开发 Next.js 的下一代，今天随着 `13.4`，这个下一代已稳定并可供采用。本文将分享更多关于我们的设计决策和 App Router 的选择。

### 零配置。将文件系统作为 API 使用

<a id="zero-setup-use-the-filesystem-as-an-api"></a>

[基于文件系统的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在最初的文章中，我们展示了从单个 React 组件创建路由的示例：

```
// Pages Router
 
import React from 'react';
export default () => <h1>About us</h1>;
```

pages/about.js

无需额外的配置。将文件放入 `pages/` 目录，Next.js 路由器就会处理其余部分。我们依然喜爱这种路由的简洁性。但随着框架的使用日益增长，开发者希望用它构建的界面类型也越来越多。

开发者请求改进对定义布局的支持，将界面的片段嵌套为布局，并在定义加载和错误状态方面拥有更多灵活性。这不是容易的一件事来改写到现有的 Next.js 路由器。

框架的每部分都必须围绕路由器进行设计。页面过渡、数据获取、缓存、数据修改和重新验证、流式传输、样式和内容等等。

为了让我们的路由器兼容流式传输，并解决这些对布局增强支持的需求，我们着手构建我们的路由器的新版本。

app/layout.js

```
// New: App Router ✨
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

`getInitialProps` 在服务端*和*客户端运行。此 API 扩展了 React 组件，允许您创建一个 `Promise` 并将结果转发给组件的 `props`。

```
export default function Page() {
  return <h1>Hello, Next.js!</h1>;
}
```

app/page.js

比这里所见更重要的是您*没有*看到的东西。这个新路由器（可以通过 `app/` 目录进行增量采用）拥有完全不同的架构，基于 [React 服务端组件](/docs/getting-started/react-essentials) 和 [Suspense](/docs/app/building-your-application/routing/loading-ui-and-streaming) 的基础。

`pages/_app.js`

```
// Pages Router
 
// This "global layout" wraps all routes. There's no way to
// compose other layout components, and you cannot fetch global
// data from this file.
export default function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
```

在使用 Pages Router 时，布局无法被组合，数据获取也不能与组件共存。在使用新的 App Router 时，现在支持这些功能。

app/layout.js

```
// New: App Router ✨
// The root layout is shared for the entire application
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

app/dashboard/layout.js

```
// Layouts can be nested and composed
export default function DashboardLayout({ children }) {
  return (
    <section>
      <h1>Dashboard</h1>
      {children}
    </section>
  );
}
```

在使用 Pages Router 时，`_document` 用于从服务端定制初始负载。

pages\_document.js

```

// This file allows you to customize the <html> and <body> tags
// for the server request, but adds framework-specific features
```

app/layout.js

```

export default function RootLayout({ children }) {
```

构建新文件系统路由器的机会也是解决我们路由系统的许多其他相关特性请求的合适时机。例如：

*   此前，您只能从外部 npm 包（如组件库）在 `_app.js` 中导入全局样式表。这对开发者体验来说不是最佳体验。在使用 App Router 时，您可以在任何组件中导入（并共存）任何 CSS 文件。
*   此前，使用 Next.js 选择服务端渲染（通过 `getServerSideProps`）意味着在整页水合之前无法与您的应用进行交互。在使用 App Router 时，我们重构了架构以与 React Suspense 深度集成，这意味着我们可以选择性地水合页面的部分，而不阻止 UI 中的其他组件进行交互。内容可以即时从服务端流式传输，提高页面的感知加载性能。
*   [路由器](/docs/app/building-your-application/routing) 是让 Next.js 工作的核心。但这不仅仅是路由器本身，而是它如何集成框架的其他部分——如 [数据获取](/docs/app/building-your-application/data-fetching)。

### 自动服务端渲染和代码分割

<a id="automatic-server-rendering-and-code-splitting"></a>

当我们创建 Next.js 时，开发者手动配置 webpack、babel 和其他工具以使 React 应用运行仍然很常见。添加进一步的优化，如服务端渲染或代码分割，往往未在自定义解决方案中实现。Next.js 以及其它 React 框架创建了一个抽象层来实现并强制这些最佳实践。

基于路由的代码分割意味着 `pages/` 目录中的每个文件都将被代码分割为独立的 JavaScript 包，有助于减少文件系统并提高初始页面加载性能。

app/page.tsx

```
import dynamic from 'next/dynamic';
 
const DynamicHeader = dynamic(() => import('../components/header'), {
  loading: () => <p>Loading...</p>,
});
 
export default function Home() {
  return <DynamicHeader />;
}
```

在使用 App Router 时，服务端组件不包含在浏览器的 JavaScript 包中。[客户端组件](/docs/getting-started/react-essentials#client-components) 默认自动代码分割（使用 webpack 或 Next.js 中的 Turbopack）。此外，由于整个路由器架构是流式和 Suspense 启用的，您可以将 UI 的某些部分从服务端逐步发送到客户端。

例如，您可以通过条件逻辑代码分割整个代码路径。在此示例中，您不需要为已登出的用户加载仪表板的服务端 JavaScript。

app/layout.tsx

```
import { getUser } from './auth';
import { Dashboard, Landing } from './components';
 
export default async function Layout() {
  const isLoggedIn = await getUser();
  return isLoggedIn ? <Dashboard /> : <Landing />;
}
```

服务端操作 (Server Actions)（测试版）
-----------------------------------------------

React 生态系统已经看到许多关于表单、管理表单状态和数据缓存及重新验证的想法的创新和探索。随着时间的推移，React 对一些模式变得更加有观点。例如，推荐表单状态的 ["非受控组件"](https://react.dev/learn/sharing-state-between-components#controlled-and-uncontrolled-components)。

当前的解决方案生态系统要么是可重用的客户端解决方案，要么是构建在框架中的原生功能。直到现在，还没有一种方式可以组合服务端修改和数据原生。React 团队 [一直在工作](https://react.dev/blog/2023/03/22/react-labs-what-we-have-been-working-on-march-2023) 关于原生修改解决方案。

我们很兴奋宣布支持 Next.js 中的实验性 **服务端操作 (Server Actions)**，让您能够在服务端修改数据，直接调用函数*无需*创建中间 API 层。

app/post/\[id\]/page.tsx (Server Component)

```
import kv from './kv';
 
export default function Page({ params }) {
  async function increment() {
    'use server';
    await kv.incr(`post:id:${params.id}`);
  }
 
  return (
    <form action={increment}>
      <button type="submit">Like</button>
    </form>
  );
}
```

通过使用服务端操作，您拥有强大的服务端优先数据修改，更少的客户端 JavaScript，以及逐步增强的表单。

app/post/new/page.tsx (Server Component)

```
import db from './db';
import { redirect } from 'next/navigation';
 
async function create(formData: FormData) {
  'use server';
  const post = await db.post.insert({
    title: formData.get('title'),
    content: formData.get('content'),
  });
  redirect(`/blog/${post.slug}`);
}
 
export default function Page() {
  return (
    <form action={create}>
      <input type="text" name="title" />
      <textarea name="content" />
      <button type="submit">Submit</button>
    </form>
  );
}
```

Next.js 中的服务端操作设计为与数据生命周期的其余部分深度集成，包括 Next.js 缓存、增量静态再生 (ISR) 和客户端路由器。

通过新 API `revalidatePath` 和 `revalidateTag` 重新验证数据意味着修改、重新渲染页面或重定向可以在**一个网络往返**中完成，确保客户端显示正确的数据，即使上游提供者慢。

app/dashboard/posts/page.tsx (Server Component)

```
import db from './db';
import { revalidateTag } from 'next/cache';
 
async function update(formData: FormData) {
  'use server';
  await db.post.update({
    title: formData.get('title'),
  });
  revalidateTag('posts');
}
 
export default async function Page() {
  const res = await fetch('https://...', { next: { tags: ['posts'] } });
  const data = await res.json();
  // ...
}
```

服务端操作设计为可组合的。React 社区中的任何人都可以构建和发布服务端操作并在生态系统中分发它们。就像服务端组件一样，我们对客户端和服务端的新一代可组合原生组件感到兴奋。

[服务端操作](/docs/app/building-your-application/data-fetching/server-actions) 今天在第 13.4 版中可用，处于测试版。通过选择使用服务端操作，Next.js 将使用 React 的测试版发布渠道。

next.config.js

```
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    serverActions: true,
  },
};
 
module.exports = nextConfig;
```

### App Router 稳定性意味着什么？

今天标记 App Router 为稳定并不意味着我们的工作已经完成。稳定性意味着 App Router 的核心已准备就绪，可用于生产，并经过了我们要的内部测试以及许多 Next.js 早期采用者的验证。

未来我们仍有一些额外的优化想要进行，包括服务端操作达到完全稳定。重要的是我们要推进核心稳定性，以帮助社区清楚地了解今天从哪里开始学习和构建应用。

App Router 建立在 React `canary` 通道之上，该通道现在已准备好用于 Server Components 等功能的框架采用。[了解更多](https://react.dev/blog/2023/05/03/react-canaries)。

### 这对 Next.js beta 文档意味着什么？

从今天开始，我们建议使用 App Router 构建新应用。用于解释 App Router 的 Next.js beta 文档并从基础重写，现在已合并回 [稳定版 Next.js 文档](/docs)。您现在可以轻松在 App 或 Pages 路由器之间切换。

我们建议阅读 [App Router 增量采用指南](/docs/app/building-your-application/upgrading/app-router-migration) 以了解如何采用 App Router。

### Pages Router 会消失吗？

不会。我们致力于支持 `pages/` 开发，包括错误修复、改进和安全补丁，在多个主要版本中继续前进。我们要确保开发者有足够的时间逐步采用 App Router。

在生产中同时使用 `pages/` 和 `app/` 是受支持和鼓励的。App Router 可以 [按路由basis](/docs/app/building-your-application/upgrading/app-router-migration) 采用。

### 这是否意味着服务端组件是"完整的"？

Next.js 是选择基于 React 架构构建的一个框架，其中包括服务端组件。我们希望 App Router 提供的体验将鼓励其他框架（或新框架）也考虑使用此架构。

该生态系统中仍有许多尚未定义的规范，例如处理无限滚动。现在，我们建议对这种模式使用客户端解决方案，同时生态系统和库正在创建或更新。
