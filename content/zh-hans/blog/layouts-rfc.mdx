本 RFC（意见征求稿）概述了自 2016 年 Next.js 诞生以来最大规模的更新：

*   **嵌套布局：** 构建具有嵌套路由的复杂应用程序。
*   **专为服务器组件设计：** 针对子树导航进行优化。
*   **改进的数据获取：** 在布局中获取数据，同时避免请求瀑布。
*   **使用 React 18 特性：** 流式传输（Streaming）、过渡（Transitions）和悬念（Suspense）。
*   **客户端与服务器路由：** 以服务器为中心的路由，兼具类似单页应用（SPA）的行为。
*   **100% 渐进式采用**：无破坏性更改，您可以逐步采用。
*   **高级路由模式**：并行路由、拦截路由等。

新的 Next.js 路由器将基于 [最近发布的 React 18](https://reactjs.org/blog/2022/03/29/react-v18.html) 特性构建。我们计划引入默认值和约定，让您可以轻松采用这些新特性并利用它们解锁的优势。

> 本 RFC 的工作正在进行中，我们将在新功能可用时进行公告。要提供反馈，请加入 [Github Discussions](https://github.com/vercel/next.js/discussions/37136) 上的讨论。

动机 [](#motivation)
-------------------------

我们一直在通过 GitHub、Discord、Reddit 和开发者调查收集社区对 Next.js 当前路由限制的反馈。我们发现：

*   创建布局的开发者体验有待改进。应该能够轻松创建可嵌套、可在路由间共享并在导航时保留其状态的布局。
*   许多 Next.js 应用是仪表板或控制台，它们将受益于更高级的路由解决方案。

虽然自 Next.js 诞生以来，当前的路由系统一直运行良好，但我们希望让开发者更容易构建性能更高、功能更丰富的 Web 应用程序。

作为框架维护者，我们也希望建立一个向后兼容并与 React 未来发展方向保持一致的路由系统。

> **注意：** 某些路由约定受到 Meta 公司基于 Relay 的路由器（服务器组件的某些特性最初在此开发）以及 React Router 和 Ember.js 等客户端路由器的启发。`layout.js` 文件约定则受到 SvelteKit 工作的启发。我们还要感谢 [Cassidy](https://twitter.com/cassidoo) 提出的[关于布局的早期 RFC](https://github.com/vercel/next.js/discussions/26389)。

术语 [](#terminology)
---------------------------

本 RFC 引入了新的路由约定和语法。术语基于 React 和标准 Web 平台术语。在整个 RFC 中，您会看到这些术语链接回下面的定义。

*   **树 (Tree)：** 一种可视化层次结构约定。例如，一个包含父组件和子组件的组件树、文件夹结构等。
*   **子树 (Subtree)** 树的一部分，从根节点（第一个）开始，到叶节点（最后一个）结束。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/terminology.png)

*   **URL 路径 (URL Path)：** URL 中域名之后的部分。
*   **URL 段 (URL Segment)：** URL 路径中由斜杠分隔的部分。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/url-anatomy.png)

当前路由工作原理 [](#how-routing-currently-works)
-----------------------------------------------------------

如今，Next.js 使用文件系统将 [Pages](/docs/pages/building-your-application/routing/pages-and-layouts) 目录中的各个文件夹和文件映射为可通过 URL 访问的路由。每个**页面**文件导出一个 React 组件，并具有基于其文件名的关联**路由**。例如：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/routing-today.png)

*   **动态路由：** Next.js 支持[动态路由](/docs/pages/building-your-application/routing/dynamic-routes)（包括捕获所有变体），使用 `[param].js`、`[...param].js` 和 `[[...param]].js` 约定。
*   **布局：** Next.js 支持简单的[基于组件的](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern)布局、使用组件[属性模式](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern#per-page-layouts)的每页布局，以及使用[自定义 App](/docs/pages/building-your-application/routing/pages-and-layouts#layout-pattern#single-shared-layout-with-custom-app) 的单一全局布局。
*   **数据获取：** Next.js 提供数据获取方法（[`getStaticProps`](/docs/pages/building-your-application/data-fetching/get-static-props)、[`getServerSideProps`](/docs/pages/building-your-application/data-fetching/get-server-side-props)），可在页面（路由）级别使用。这些方法用于确定页面应静态生成（[`getStaticProps`](/docs/pages/building-your-application/data-fetching/get-static-props)）还是服务器端渲染（[`getServerSideProps`](/docs/pages/building-your-application/data-fetching/get-server-side-props)）。此外，您可以使用[增量静态再生（ISR）](/docs/pages/building-your-application/data-fetching/incremental-static-regeneration)在站点构建后创建或更新静态页面。
*   **渲染：** Next.js 提供三种渲染选项：[静态生成](https://nextjs.org/learn/foundations/how-nextjs-works/rendering)、[服务器端渲染](https://nextjs.org/learn/foundations/how-nextjs-works/rendering)和[客户端渲染](https://nextjs.org/learn/foundations/how-nextjs-works/rendering)。默认情况下，页面是静态生成的，除非它们有阻塞的数据获取需求（`getServerSideProps`）。

介绍 `app` 目录 [](#introducing-the-app-directory)
-----------------------------------------------------------------

为了确保这些新改进可以渐进式采用并避免破坏性更改，我们建议引入一个名为 `app` 的新目录。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/app-folder.png)

`app` 目录将与 `pages` 目录并行工作。您可以将应用程序的部分内容逐步移至新的 `app` 目录以利用新特性。为了向后兼容，`pages` 目录的行为将保持不变并继续受支持。

定义路由 [](#defining-routes)
-----------------------------------

您可以使用 `app` 内部的**文件夹**层次结构来定义路由。**路由**是一条嵌套文件夹的单一路径，从**根文件夹**向下遵循层次结构，直到一个最终的**叶文件夹**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/routes.png)

例如，您可以通过在 `app` 目录中嵌套两个新文件夹来添加一个新的 `/dashboard/settings` 路由。

> **注意：**
> 
> *   使用此系统，您将使用文件夹定义路由，使用文件定义 UI（通过新的文件约定，如 `layout.js`、`page.js`，以及在 RFC 第二部分的 `loading.js`）。
> *   这允许您将项目自己的文件（UI 组件、测试文件、故事文件等）放在 `app` 目录中。目前，这只能通过 [pageExtensions 配置](/docs/pages/api-reference/next-config-js/pageExtensions#including-non-page-files-in-the-pages-directory)实现。

### 路由段 [](#route-segments)

子树中的每个文件夹代表一个**路由段**。每个路由段映射到 **[URL 路径](#术语)** 中的一个相应**段**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/route-segments.png)

例如，`/dashboard/settings` 路由由 3 个段组成：

*   `/` 根段
*   `dashboard` 段
*   `settings` 段

> **注意：** 选择**路由段**这个名称是为了匹配围绕 [URL 路径](#术语)的现有术语。

布局 [](#layouts)
-------------------

**新文件约定：** `layout.js`

到目前为止，我们使用文件夹来定义应用程序的路由。但空文件夹本身不做任何事情。让我们讨论如何使用新的文件约定为这些路由定义将要渲染的 UI。

**布局**是在[子树](#术语)中的路由段之间共享的 UI。布局不会影响 [URL 路径](#术语)，并且当用户在同级段之间导航时不会重新渲染（React 状态被保留）。

布局可以通过从 `layout.js` 文件默认导出一个 React 组件来定义。该组件应接受一个 `children` 属性，该属性将填充布局包装的段。

布局有 2 种类型：

*   **根布局：** 应用于所有路由
*   **常规布局：** 应用于特定路由

您可以将两个或多个布局嵌套在一起以形成**嵌套布局**。

### 根布局 [](#root-layout)

您可以通过在 `app` 文件夹内添加一个 `layout.js` 文件来创建一个根布局，该布局将应用于您应用程序的所有路由。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/root-layout.png)

> **注意：**
> 
> *   根布局取代了对[自定义 App (`_app.js`)](/docs/pages/building-your-application/routing/custom-app) 和[自定义 Document (`_document.js`)](/docs/pages/building-your-application/routing/custom-document) 的需求，因为它应用于所有路由。
> *   您将能够使用根布局来自定义初始文档外壳（例如 `<html>` 和 `<body>` 标签）。
> *   您将能够在根布局（以及其他布局）中获取数据。

### 常规布局 [](#regular-layouts)

您也可以通过在特定文件夹内添加 `layout.js` 文件来创建一个仅应用于应用程序一部分的布局。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/regular-layouts.png)

例如，您可以在 `dashboard` 文件夹内创建一个 `layout.js` 文件，该文件将仅应用于 `dashboard` 内的路由段。

### 嵌套布局 [](#nesting-layouts)

布局默认是**嵌套的**。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/nested-layouts.png)

例如，如果我们要组合上面的两个布局。根布局 (`app/layout.js`) 将应用于 `dashboard` 布局，而 `dashboard` 布局又将应用于 `dashboard/*` 内的所有路由段。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/nested-layouts-example.png)

页面 [](#pages)
---------------

**新文件约定：** `page.js`

页面是路由段独有的 UI。您可以通过在文件夹内添加 `page.js` 文件来创建页面。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page.png)

例如，要为 `/dashboard/*` 路由创建页面，您可以在每个文件夹内添加一个 `page.js` 文件。当用户访问 `/dashboard/settings` 时，Next.js 将渲染 `settings` 文件夹的 `page.js` 文件，并包裹在[子树](#术语)中任何存在的布局内。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page-example.png)

您可以在 dashboard 文件夹内直接创建一个 `page.js` 文件以匹配 `/dashboard` 路由。dashboard 布局也将应用于该页面：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/page-nested.png)

该路由由 2 个段组成：

*   `/` 根段
*   `dashboard` 段

> **注意：**
> 
> *   要使路由有效，其叶段中需要有一个页面。如果没有，该路由将抛出错误。

### 布局和页面行为 [](#layout-and-page-behavior)

*   文件扩展名 `js|jsx|ts|tsx` 可用于页面和布局。
*   页面组件是 `page.js` 的默认导出。
*   布局组件是 `layout.js` 的默认导出。
*   布局组件**必须**接受一个 `children` 属性。

当布局组件被渲染时，`children` 属性将填充一个子布局（如果在[子树](#术语)中进一步存在）或一个页面。

将其可视化为一个布局[树](#术语)可能会更容易理解，其中父布局将选择最近的子布局，直到它到达一个页面。

**示例：**

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/basic-example.png)

app/layout.js

```
// Root layout
// - Applies to all routes
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <Header />
        {children}
        <Footer />
      </body>
    </html>
  );
}
```

app/dashboard/layout.js

```
// Regular layout
// - Applies to route segments in app/dashboard/*
export default function DashboardLayout({ children }) {
  return (
    <>
      <DashboardSidebar />
      {children}
    </>
  );
}
```

app/dashboard/analytics/page.js

```
// Page Component
// - The UI for the `app/dashboard/analytics` segment
// - Matches the `acme.com/dashboard/analytics` URL path
export default function AnalyticsPage() {
  return <main>...</main>;
}
```

上述布局和页面的组合将渲染以下组件层次结构：

### 组件层次结构

```
<RootLayout>
  <Header />
  <DashboardLayout>
    <DashboardSidebar />
    <AnalyticsPage>
      <main>...</main>
    </AnalyticsPage>
  </DashboardLayout>
  <Footer />
</RootLayout>
```

React 服务器组件 [](#react-server-components)
---------------------------------------------------

> **注意：** React 引入了新的组件类型：服务器组件、客户端组件（传统的 React 组件）和共享组件。要了解这些新类型的更多信息，我们建议阅读 React [服务器组件 RFC](https://reactjs.org/blog/2020/12/21/data-fetching-with-react-server-components.html)。

通过本 RFC，您可以开始使用 React 特性并将 React 服务器组件渐进式采用到您的 Next.js 应用程序中。

新路由系统的内部实现也将利用最近发布的 React 特性，如流式传输（Streaming）、悬念（Suspense）和过渡（Transitions）。这些是 React 服务器组件的构建模块。

### 服务器组件作为默认值 [](#server-components-as-the-default)

`pages` 和 `app` 目录之间最大的变化之一是，**默认情况下，`app` 内的文件将作为 React 服务器组件在服务器上渲染。**

这将允许您在从 `pages` 迁移到 `app` 时自动采用 React 服务器组件。

> **注意：** 服务器组件可以在 `app` 文件夹或您自己的文件夹中使用，但不能在 `pages` 目录中使用，这是为了向后兼容。

### 客户端和服务器组件约定 [](#client-and-server-components-convention)

`app` 文件夹将支持服务器组件、客户端组件和共享组件，您将能够[在树中交错这些组件](#在路由中交错客户端和服务器组件)。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/server-components.png)

关于如何定义客户端组件和服务器组件的具体约定，目前有一个[正在进行的讨论](https://github.com/reactjs/rfcs/pull/189#issuecomment-1116482278)。我们将遵循该讨论的解决方案。

*   目前，可以通过在文件名后附加 `.server.js` 来定义服务器组件。例如 `layout.server.js`
*   客户端组件可以通过在文件名后附加 `.client.js` 来定义。例如 `page.client.js`。
*   `.js` 文件被视为共享组件。由于它们可能在服务器和客户端渲染，因此需要遵守每个上下文的约束。

> **注意：**
> 
> *   客户端和服务器组件有需要遵守的[约束](https://github.com/josephsavona/rfcs/blob/server-components/text/0000-server-components.md#capabilities--constraints-of-server-and-client-components)。在决定使用客户端还是服务器组件时，我们建议在需要使用客户端组件之前，使用服务器组件（默认）。

### 钩子 [](#hooks)

我们将添加客户端和服务器组件钩子，允许您访问 headers 对象、cookies、路径名、搜索参数等。未来，我们将提供包含更多信息的文档。

### 渲染环境 [](#rendering-environments)

您将使用客户端和服务器组件约定来精细控制哪些组件将包含在客户端 JavaScript 包中。

默认情况下，`app` 中的路由将使用静态生成，并在路由段使用需要请求上下文的[服务器端钩子](#钩子)时切换到动态渲染。

### 在路由中交错客户端和服务器组件 [](#interleaving-client-and-server-components-in-a-route)

在 React 中，有一个限制是不能在客户端组件中导入服务器组件，因为服务器组件可能包含仅限服务器端的代码（例如数据库或文件系统工具）。

例如，导入服务器组件将不起作用：

ClientComponent.js

```
import ServerComponent from './ServerComponent.js';
 
export default function ClientComponent() {
  return (
    <>
      <ServerComponent />
    </>
  );
}
```

但是，服务器组件可以作为客户端组件的子项传递。您可以通过将它们**包装**在另一个服务器组件中来实现。例如：

ClientComponent.js

```
export default function ClientComponent({ children }) {
  return (
    <>
      <h1>Client Component</h1>
      {children}
    </>
  );
}
 
// ServerComponent.js
export default function ServerComponent() {
  return (
    <>
      <h1>Server Component</h1>
    </>
  );
}
 
// page.js
// It's possible to import Client and Server components inside Server Components
// because this component is rendered on the server
import ClientComponent from "./ClientComponent.js";
import ServerComponent from "./ServerComponent.js";
 
export default function ServerComponentPage() {
  return (
    <>
      <ClientComponent>
        <ServerComponent />
      </ClientComponent>
    </>
  );
}
```

使用此模式，React 将知道需要在服务器上渲染 `ServerComponent`，然后将结果（不包含任何仅限服务器端的代码）发送到客户端。从客户端组件的角度来看，它的子组件将已经渲染好。

在布局中，此模式通过 `children` 属性应用，因此您无需创建额外的包装组件。

例如，`ClientLayout` 组件将接受 `ServerPage` 组件作为其子组件：

app/dashboard/layout.js

```
// The Dashboard Layout is a Client Component
export default function ClientLayout({ children }) {
  // Can use useState / useEffect here
  return (
    <>
      <h1>Layout</h1>
      {children}
    </>
  );
}
 
// The Page is a Server Component that will be passed to Dashboard Layout
// app/dashboard/settings/page.js
export default function ServerPage() {
  return (
    <>
      <h1>Page</h1>
    </>
  );
}
```

> **注意：** 这种组合方式是在客户端组件中渲染服务器组件的重要模式。它确立了一种需要学习的模式的优先级，也是我们决定使用 `children` 属性的原因之一。

数据获取 [](#data-fetching)
-------------------------------

可以在路由的多个段中获取数据。这与 `pages` 目录不同，在 `pages` 目录中，数据获取仅限于页面级别。

### 布局中的数据获取 [](#data-fetching-in-layouts)

您可以通过使用 Next.js 数据获取方法 `getStaticProps` 或 `getServerSideProps` 在 `layout.js` 文件中获取数据。

例如，博客布局可以使用 `getStaticProps` 从 CMS 获取分类，用于填充侧边栏组件：

app/blog/layout.js

```
export async function getStaticProps() {
  const categories = await getCategoriesFromCMS();
 
  return {
    props: { categories },
  };
}
 
export default function BlogLayout({ categories, children }) {
  return (
    <>
      <BlogSidebar categories={categories} />
      {children}
    </>
  );
}
```

### 路由中的多种数据获取方法 [](#multiple-data-fetching-methods-in-a-route)

您也可以在路由的多个段中获取数据。例如，一个获取数据的 `layout` 也可以包装一个获取其自身数据的 `page`。

使用上面的博客示例，单个文章页面可以使用 `getStaticProps` 和 `getStaticPaths` 从 CMS 获取文章数据：

app/blog/[slug]/page.js

```
export async function getStaticPaths() {
  const posts = await getPostSlugsFromCMS();
 
  return {
    paths: posts.map((post) => ({
      params: { slug: post.slug },
    })),
  };
}
 
export async function getStaticProps({ params }) {
  const post = await getPostFromCMS(params.slug);
 
  return {
    props: { post },
  };
}
 
export default function BlogPostPage({ post }) {
  return <Post post={post} />;
}
```

由于 `app/blog/layout.js` 和 `app/blog/[slug]/page.js` 都使用了 `getStaticProps`，Next.js 将在构建时将整个 `/blog/[slug]` 路由静态生成为** React 服务器组件**，从而减少客户端 JavaScript 并加快水合速度。

静态生成的路由进一步改进了这一点，因为客户端导航重用缓存（服务器组件数据）并且不会重新计算工作，从而减少 CPU 时间，因为您正在渲染服务器组件的快照。

### 行为和优先级 [](#behavior-and-priority)

Next.js 数据获取方法（`getServerSideProps` 和 `getStaticProps`）只能在 `app` 文件夹中的服务器组件中使用。单个路由中不同段的不同数据获取方法会相互影响。

在一个段中使用 `getServerSideProps` 将影响其他段中的 `getStaticProps`。由于请求必须为使用 `getServerSideProps` 的段发送到服务器，服务器也会渲染任何使用 `getStaticProps` 的段。它将重用在构建时获取的 props，因此数据仍然是静态的，但**渲染**在每次请求时按需进行，使用 `next build` 期间生成的 props。

在一个段中使用带有 **revalidate (ISR)** 的 `getStaticProps` 将影响其他段中带有 `revalidate` 的 `getStaticProps`。如果一个路由中有两个重新验证周期，较短的重新验证将优先。

> **注意：** 未来，这可能会被优化以允许在路由中实现完整的数据获取粒度。

### 使用 React 服务器组件进行数据获取 [](#data-fetching-with-react-server-components)

服务器端路由、React 服务器组件、悬念和流式传输的组合对 Next.js 中的数据获取和渲染有一些影响：

### 并行数据获取 [](#parallel-data-fetching)

Next.js 将并行地预先启动数据获取，以最小化请求瀑布。例如，如果数据获取是**顺序**的，路由中的每个嵌套段都无法开始获取数据，直到前一个段完成。然而，通过**并行**获取，每个段可以同时预先开始数据获取。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel.png)

由于渲染可能依赖于上下文，每个段的渲染将在其数据已获取且其父段渲染完成后开始。

未来，使用悬念，渲染也可以立即开始，即使数据尚未完全加载。如果在数据可用之前读取数据，将触发悬念。React 将在请求完成之前乐观地开始渲染服务器组件，并在请求解析时插入结果。

### 部分获取和渲染 [](#partial-fetching-and-rendering)

在同级路由段之间导航时，Next.js 仅从该段向下获取和渲染。它不需要重新获取或重新渲染上面的任何内容。这意味着在共享布局的页面中，当用户在同级页面之间导航时，布局将被保留，Next.js 仅从该段向下获取和渲染。

这对于 React 服务器组件尤其有用，因为否则每次导航都会导致整个页面在服务器上重新渲染，而不是仅在服务器上渲染页面中已更改的部分。这减少了传输的数据量和执行时间，从而提高了性能。

例如，如果用户在 `/analytics` 和 `/settings` 页面之间导航，React 将重新渲染页面段但保留布局：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/partial-example.png)

> **注意：** 可以强制重新获取[树](#术语)中更高位置的数据。我们仍在讨论此功能的细节，并将更新 RFC。

以服务器为中心的路由 [](#server-centric-routing)
-------------------------------------------------

目前，Next.js 使用客户端路由。在初始加载和后续导航后，会向服务器发出请求以获取新页面的资源。这包括**每个组件的 JavaScript**（包括仅在特定条件下显示的组件）及其 props（来自 `getServerSideProps` 或 `getStaticProps` 的 JSON 数据）。一旦 JavaScript 和数据都从服务器加载完毕，**React 在客户端渲染组件。**

在这个新模式中，Next.js 将使用以服务器为中心的路由，同时保持客户端过渡。这与在服务器上评估的服务器组件保持一致。

在导航时，数据被获取，React 在**服务器端**渲染组件。来自服务器的输出是用于客户端 React 更新 DOM 的特殊指令（不是 HTML 或 JSON）。这些指令包含已渲染的服务器组件的**结果**，意味着无需在浏览器中加载该组件的 JavaScript 即可渲染结果。

这与当前默认的客户端组件形成对比，后者将组件 JavaScript 发送到浏览器以进行客户端渲染。

使用 React 服务器组件的以服务器为中心的路由的一些好处包括：

*   路由使用与服务器组件相同的请求（不会发出额外的服务器请求）。
*   在服务器上完成的工作更少，因为路由之间的导航仅获取和渲染发生变化的段。
*   在客户端导航时，如果没有使用新的客户端组件，浏览器中不会加载额外的 JavaScript。
*   路由器利用新的流式协议，可以在所有数据加载完毕之前就开始渲染。

当用户在应用中导航时，路由器将把 React 服务器组件 _payload_ 的结果存储在内存中的客户端缓存中。缓存按路由段拆分，允许在任何级别进行失效，并确保跨并发渲染的一致性。这意味着在某些情况下，可以重用先前获取的段的缓存。

> **注意**
> 
> *   静态生成和服务器端缓存可用于优化数据获取。
> *   以上信息描述了后续导航的行为。初始加载是一个涉及服务器端渲染以生成 HTML 的不同过程。
> *   虽然客户端路由在 Next.js 中运行良好，但当潜在路由数量很大时，它的扩展性很差，因为客户端必须下载一个**路由映射**。
> *   总体而言，通过使用 React 服务器组件，客户端导航更快，因为我们在浏览器中加载和渲染的组件更少。

即时加载状态 [](#instant-loading-states)
-------------------------------------------------

使用服务器端路由时，导航发生在数据获取和渲染**之后**，因此在获取数据时显示加载 UI 非常重要，否则应用程序会显得无响应。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/server-side-routing.png)

新的路由器将使用[悬念](https://reactjs.org/docs/react-api.html#reactsuspense)来实现即时加载状态和默认骨架屏。这意味着加载 UI 可以在加载新段的内容时立即显示。然后，一旦服务器上的渲染完成，新内容将被交换进来。

在渲染进行期间：

*   导航到新路由将是即时的。
*   共享布局将在加载新路由段时保持交互性。
*   导航将是可中断的，这意味着用户可以在一个路由的内容加载时在路由之间导航。

### 默认加载骨架屏 [](#default-loading-skeletons)

悬念边界将通过一个名为 `loading.js` 的新文件约定在后台自动处理。

**示例：**

您将能够通过在文件夹内添加 `loading.js` 文件来创建默认的加载骨架屏。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/loading.png)

`loading.js` 应导出一个 React 组件：

loading.js

```
export default function Loading() {
  return <YourSkeleton />
}
 
// layout.js
export default function Layout({children}) {
  return (
    <>
      <Sidebar />
      {children}
    </>
  )
}
 
// Output
<>
  <Sidebar />
  <Suspense fallback={<Loading />}>{children}</Suspense>
</>
```

这将导致文件夹中的所有段被包装在一个悬念边界中。当布局首次加载以及在同级页面之间导航时，将使用默认的骨架屏。

### 错误处理 (Error Handling)

[错误边界](https://reactjs.org/docs/error-boundaries.html) 是 React 组件，可捕获其子组件树中任何位置的 JavaScript 错误。

### 约定 [](#convention-1)

您将能够通过添加 `error.js` 文件并默认导出一个 React 组件来创建一个错误边界，该边界将捕获子树内的错误。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/error.png)

如果在该子树内抛出错误，该组件将显示为回退。此组件可用于记录错误、显示有关错误的有用信息以及尝试从错误中恢复的功能。

由于段和布局的嵌套性质，创建错误边界允许您将错误隔离到 UI 的这些部分。在错误期间，边界之上的布局将保持交互状态，其状态将被保留。

error.js

```
export default function Error({ error, reset }) {
  return (
    <>
      An error occurred: {error.message}
      <button onClick={() => reset()}>Try again</button>
    </>
  );
}
 
// layout.js
export default function Layout({children}) {
  return (
    <>
      <Sidebar />
      {children}
    </>
  )
}
 
// Output
<>
  <Sidebar />
  <ErrorBoundary fallback={<Error />}>{children}</ErrorBoundary>
</>
```

> 注意：
> 
> *   与 `error.js` 位于同一段中的 `layout.js` 文件内的错误将不会被捕获，因为自动错误边界包装的是布局的子组件，而不是布局本身。

模板 [](#templates)
-----------------------

模板类似于布局，它们包装每个子布局或页面。

与跨路由持久化并保持状态的布局不同，模板为其每个子项创建一个新实例。这意味着当用户在共享模板的路由段之间导航时，会挂载该组件的一个新实例。

> **注意：** 除非有特定原因使用模板，否则我们建议使用布局。

### 约定 [](#convention-2)

可以通过从 `template.js` 文件导出默认的 React 组件来定义模板。该组件应接受一个 `children` 属性，该属性将填充嵌套的段。

### [示例](#example)

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/template.png)

template.js

```
export default function Template({ children }) {
  return <Container>{children}</Container>;
}
```

具有布局和模板的路由段的渲染输出如下：

```
<Layout>
  {/* Note that the template is given a unique key. */}
  <Template key={routeParam}>{children}</Template>
</Layout>
```

### 行为 [](#behavior)

在某些情况下，您可能需要挂载和卸载共享 UI，模板将是更合适的选择。例如：

*   使用 CSS 或动画库的进入/退出动画
*   依赖 `useEffect`（例如记录页面浏览量）和 `useState`（例如每页反馈表单）的功能
*   更改默认框架行为。例如，布局内的悬念边界仅在布局首次加载时显示回退，而在切换页面时不显示。对于模板，每次导航时都会显示回退。

例如，考虑一个嵌套布局的设计，其中包含一个带边框的容器，该容器应包装在每个子页面周围。

您可以将容器放在父布局中（`shop/layout.js`）：

shop/layout.js

```
export default function Layout({ children }) {
  return <div className="container">{children}</div>;
}
 
// shop/page.js
export default function Page() {
  return <div>...</div>;
}
 
// shop/categories/layout.js
export default function CategoryLayout({ children }) {
  return <div>{children}</div>;
}
```

但是，切换页面时不会播放任何进入/退出动画，因为共享的父布局不会重新渲染。

您可以将容器放在每个嵌套布局或页面中：

shop/layout.js

```
export default function Layout({ children }) {
  return <div>{children}</div>;
}
 
// shop/page.js
export default function Page() {
  return <div className="container">...</div>;
}
 
// shop/categories/layout.js
export default function CategoryLayout({ children }) {
  return <div className="container">{children}</div>;
}
```

但是，您必须手动将其放入每个嵌套布局或页面中，这在更复杂的应用程序中可能很繁琐且容易出错。

使用此约定，您可以跨路由共享模板，这些模板在导航时创建一个新实例。这意味着 DOM 元素将被重新创建，状态将不会被保留，效果将被重新同步。

高级路由模式 [](#advanced-routing-patterns)
-------------------------------------------------------

我们计划引入约定来涵盖边缘情况，并允许您实现更高级的路由模式。以下是我们一直在积极思考的一些示例：

### 拦截路由 [](#intercepting-routes)

有时，从其他路由内拦截路由段可能很有用。在导航时，URL 将照常更新，但被拦截的段将显示在当前路由的布局内。

#### 示例 [](#example-1)

**之前：** 点击图片会跳转到具有自己布局的新路由。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-routes-before.png)

**之后：** 通过拦截路由，点击图片现在会在当前路由的布局内加载该段。例如，作为一个模态框。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-routes-after.png)

要从 `/[username]` 段内拦截 `/photo/[id]` 路由，请在 `/[username]` 文件夹内创建一个重复的 `/photo/[id]` 文件夹，并使用 `(..)` 约定作为前缀。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-route.png)

#### 约定 [](#convention-3)

*   `(..)` - 将匹配上一级的路由段（父目录的同级）。类似于相对路径中的 `../`。
*   `(..)(..)` - 将匹配上两级的路由段。类似于相对路径中的 `../../`。
*   `(...)` - 将匹配根目录中的路由段。

> **注意：** 刷新或分享页面将使用其默认布局加载路由。

### 动态并行路由 [](#dynamic-parallel-routes)

有时，在同一个视图中显示两个或更多可以独立导航的叶段（`page.js`）可能很有用。

以同一仪表板内的两个或更多标签组为例。导航一个标签组不应影响另一个。向后和向前导航时，标签的组合也应正确恢复。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes.png)

#### 约定 [](#convention-4)

默认情况下，布局接受一个名为 `children` 的属性，该属性将包含一个嵌套的布局或页面。您可以通过创建一个命名的“插槽”（一个以 `@` 为前缀的文件夹）并在其中嵌套段来重命名该属性。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes-children.png)

此更改后，布局将接收一个名为 `customProp` 的属性，而不是 `children`。

analytics/layout.js

```
export default function Layout({ customProp }) {
  return <>{customProp}</>;
}
```

您可以通过在同一级别添加多个命名插槽来创建并行路由。在下面的示例中，`@views` 和 `@audience` 都作为属性传递给 analytics 布局。

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/parallel-routes-complete.png)

您可以使用命名插槽同时显示叶段。

analytics/layout.js

```
export default function Layout({ views, audience }) {
  return (
    <>
      <div>
        <ViewsNav />
        {views}
      </div>
      <div>
        <AudienceNav />
        {audience}
      </div>
    </>
  );
}
```

当用户首次导航到 `/analytics` 时，将显示每个文件夹（`@views` 和 `@audience`）中的 `page.js` 段。

导航到 `/analytics/subscribers` 时，仅更新 `@audience`。同样，导航到 `/analytics/impressions` 时，仅更新 `@views`。

向后和向前导航将恢复并行路由的正确组合。

组合拦截和并行路由 [](#combining-intercepting-and-parallel-routes)
-----------------------------------------------------------------------------------------

您可以组合拦截和并行路由，以在应用程序中实现特定的路由行为。

**示例**

例如，在创建模态框时，您通常需要注意一些常见挑战，例如：

*   模态框无法通过 URL 访问。
*   刷新页面时模态框会关闭。
*   向后导航会转到上一个路由，而不是模态框后面的路由。
*   向前导航不会重新打开模态框。

您可能希望模态框在打开时更新 URL，并且向前/向后导航可以打开和关闭模态框。此外，在分享 URL 时，您可能希望页面加载时模态框打开并显示其背后的上下文，或者您可能希望页面加载时不显示模态框的内容。

社交媒体网站上的照片就是一个很好的例子。通常，照片可以从用户的动态或个人资料中的模态框访问。但在分享照片时，它们会直接显示在自己的页面上。

通过使用约定，我们可以使模态框行为默认映射到路由行为。

考虑以下文件夹结构：

![](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/layouts-rfc/intercepted-route-modal.png)

使用此模式：

*   `/photo/[id]` 的内容可以通过 URL 在其自己的上下文中访问。也可以在 `/[username]` 路由内的模态框中访问。
*   使用客户端导航向前和向后导航应关闭和重新打开模态框。
*   刷新页面（服务器端导航）应将用户带到原始的 `/photo/id` 路由，而不是显示模态框。

在 `/@modal/(..)photo/[id]/page.js` 中，您可以返回包装在模态框组件中的页面内容。

/@modal/(..)photo/[id]/page.js

```
export default function PhotoPage() {
  const router = useRouter();
 
  return (
    <Modal
      // the modal should always be shown on page load
      isOpen={true}
      // closing the modal should take user back to the previous page
      onClose={() => router.back()}
    >
      {/* Page Content */}
    </Modal>
  );
}
```

> **注意：** 此解决方案不是在 Next.js 中创建模态框的唯一方法，但旨在展示如何组合约定以实现更复杂的路由行为。

条件路由 [](#conditional-routes)
-----------------------------------------

有时，您可能需要动态信息（如数据或上下文）来确定要显示的路由。您可以使用并行路由有条件地加载一个或另一个路由。

**示例**

layout.js

```
export async function getServerSideProps({ params }) {
  const { accountType } = await fetchAccount(params.slug);
  return { props: { isUser: accountType === 'user' } };
}
 
export default function UserOrTeamLayout({ isUser, user, team }) {
  return <>{isUser ? user : team}</>;
}
```

在上面的示例中，您可以根据 slug 返回 `user` 或 `team` 路由。这使您可以有条件地加载数据并匹配一个选项或另一个选项的子路由。
