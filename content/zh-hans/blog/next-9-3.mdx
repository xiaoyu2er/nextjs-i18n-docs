我们很高兴今天推出 Next.js 9.3，包含：

*   **[下一代静态网站生成 (SSG) 支持](#next-gen-static-site-generation-ssg-support)**：通过新的数据获取方法实现内置的优化静态生成。

下一代静态网站生成 (SSG) 支持[](#next-gen-static-site-generation-ssg-support)
---------------------------------------------------------------------------------------------

在构建网站或 Web 应用程序时，您通常需要在两种策略之间做出选择：静态生成 (SSG) 或服务器端渲染 (SSR)。

Next.js 是首个混合框架，允许您根据页面为基础选择最适合您用例的技术。

Next.js 9.0 引入了 [自动静态优化](https://nextjs.org/blog/next-9#automatic-static-optimization) 的概念。当一个页面没有 `getInitialProps` 等阻塞式数据获取要求时，它将在构建时自动渲染为 HTML。

还有更多情况让您希望在构建时将页面渲染为静态 HTML，即使有阻塞式数据获取要求。例如，由无头内容管理系统 (CMS) 支持的市场营销页面或站点的博客部分。

我们与 SSG 和 `next export` 的重度用户（如 HashiCorp）合作，并在 Next.js 历史上评论最多的 RFC 中广泛讨论了正确的约束条件，以创建一种新的统一数据获取和静态生成方式。

今天我们非常兴奋地宣布两种新的数据获取方法：[`getStaticProps`](/docs/pages/building-your-application/data-fetching) 和 [`getServerSideProps`](/docs/pages/building-your-application/data-fetching)。我们还包含一种为动态路由提供参数以静态生成静态页面的方法：[`getStaticPaths`](/docs/pages/building-your-application/data-fetching)。

这些新方法相比 `getInitialProps` 模型有许多优势，因为 SSG 和 SSR 之间有明确的区分。

*   `getStaticProps`（静态生成）：在**构建时**获取数据。
    
*   `getStaticPaths`（静态生成）：基于数据指定 [动态路由](/docs/pages/building-your-application/routing/dynamic-routes) 以预渲染。
    
*   `getServerSideProps`（服务器端渲染）：在**每次请求**时获取数据。
    
*   这些改进是 API 的添加。所有新功能都完全向后兼容，可以逐步采用。没有引入弃用，`getInitialProps` 将继续按当前方式运行。我们鼓励在新页面和项目中使用这些新方法。
    

### getStaticProps[](#getstaticprops)

如果您从页面导出一个名为 `getStaticProps` 的 `async` 函数，Next.js 将在构建时预渲染此页面。当您希望从 CMS 渲染特定的静态页面时，这特别有用。

`getStaticProps` 总是在 Node.js 上下文中运行，并且代码会自动从浏览器捆绑包中摇树优化，确保发送给浏览器的代码更少。这样您就不必担心在 Node.js 和浏览器环境（两者存在一些不一致之处）中执行数据获取代码的问题。

这使您能够使用任何异步甚至同步的数据获取技术，包括 `fetch`、REST、GraphQL 或直接访问数据库。

`pages/posts/[id].js`

```
export async function getStaticProps(context) {
  return {
    props: {}, // will be passed to the page component as props
  };
}
```

`context` 参数是一个包含以下键的对象：

*   `params`：`params` 包含使用动态路由页面的路由参数。例如，如果页面名称是 `[id].js`，那么 `params` 将类似于 `{ id: ... }`。要了解更多信息，请查看 [动态路由文档](/docs/pages/building-your-application/routing/dynamic-routes)。您应结合 `getStaticPaths` 使用它，我们将稍后解释。

这是一个使用 `getStaticProps` 从 CMS 获取博客文章列表的示例：

pages/blog.js

```
// You can use any data fetching library
import fetch from 'node-fetch';
 
// posts will be populated at build time by getStaticProps()
function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li>{post.title}</li>
      ))}
    </ul>
  );
}
 
// This function gets called at build time in the Node.js environment.
// It won't be called on client-side, so you can even do
// direct database queries. See the "Technical details" section.
export async function getStaticProps() {
  // Call an external API endpoint to get posts.
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  // By returning { props: posts }, the Blog component
  // will receive `posts` as a prop at build time
  return {
    props: {
      posts,
    },
  };
}
 
export default Blog;
```

### 我应该在什么时候使用 getStaticProps？[](#when-should-i-use-getstaticprops)

在以下情况下应使用 `getStaticProps`：

*   渲染页面所需的数据在用户请求之前于构建时可用。
*   数据来自无头内容管理系统 (CMS)。
*   数据可以被公共缓存（非用户特定）。
*   页面必须预渲染（为了 SEO 优化）并且速度非常快——`getStaticProps` 生成 HTML 和 JSON 文件，两者都可以通过 CDN 缓存以提高性能。

要了解有关 `getStaticProps` 的更多信息，请参阅 [Data Fetching 文档](/docs/pages/building-your-application/data-fetching)。

### getStaticPaths [](#getstaticpaths)

如果页面有 [动态路由](/docs/pages/building-your-application/routing/dynamic-routes) 并使用 `getStaticProps`，它需要定义一组路径，这些路径必须在构建时渲染为 HTML。

如果您从使用动态路由的页面导出一个名为 `getStaticPaths` 的 `async` 函数，Next.js 将静态预渲染 `getStaticPaths` 指定的所有路径。

`pages/posts/[id].js`

```
export async function getStaticPaths() {
  return {
    paths: [
      { params: { ... } } // See the "paths" section below
    ],
    fallback: true or false // See the "fallback" section below
  };
}
```

### paths 键（必需）[](#the-paths-key-required)

paths 键确定哪些路径将预渲染。例如，假设您有一个使用动态路由名为 `pages/posts/[id].js` 的页面。如果您从该页面导出 `getStaticPaths` 并针对 paths 返回以下内容：

```
return {
  paths: [
    { params: { id: 1 } },
    { params: { id: 2 } }
  ],
  fallback: ...
}
```

然后 Next.js 将在构建时使用 `pages/posts/[id].js` 中的页面组件静态生成 `posts/1` 和 `posts/2`。

请注意，每个 `params` 的值必须与页面名称中使用的参数匹配：

*   如果页面名称是 `pages/posts/[postId]/[commentId]`，则 `params` 应包含 `postId` 和 `commentId`。
*   如果页面名称使用捕捉所有路由，例如 `pages/[...slug]`，则 `params` 应包含 `slug`（这是一个数组）。例如，如果该数组是 `['foo', 'bar']`，则 Next.js 将静态生成位于 `/foo/bar` 的页面。

### fallback 键（必需）[](#the-fallback-key-required)

`getStaticPaths` 返回的对象必须包含一个布尔值 fallback 键。

### fallback: false [](#fallback-false)

如果 `fallback` 为 `false`，则 `getStaticPaths` 未返回的任何路径都将是 **404 页面**。如果您知道所有路径将在构建时已知，这很有用。

这是一个示例，为名为 `pages/posts/[id].js` 的每个页面预渲染一篇博客文章。博客文章列表将从内容管理系统 (CMS) 获取并由 `getStaticPaths` 返回。然后，对于每个页面，它使用 `getStaticProps` 从内容管理系统 (CMS) 获取帖子数据。

`pages/posts/[id].js`

```
import fetch from 'node-fetch';
 
function Post({ post }) {
  // Render post...
}
 
// This function gets called at build time
export async function getStaticPaths() {
  // Call an external API endpoint to get posts
  const res = await fetch('https://.../posts');
  const posts = await res.json();
 
  // Get the paths we want to pre-render based on posts
  const paths = posts.map((post) => `/posts/${post.id}`);
 
  // We'll pre-render only these paths at build time.
  // { fallback: false } means other routes should 404.
  return { paths, fallback: false };
}
 
// This also gets called at build time
export async function getStaticProps({ params }) {
  // params contains the post `id`.
  // If the route is like /posts/1, then params.id is 1
  const res = await fetch(`https://.../posts/${params.id}`);
  const post = await res.json();
 
  // Pass post data to the page via props
  return { props: { post } };
}
 
export default Post;
```

### fallback: true [](#fallback-true)

如果 `fallback` 为 `true`，则 `getStaticProps` 的行为会发生变化，Next.js 将在构建时将提供的路径渲染为 HTML。当某条路径未在构建时生成时，它将在用户请求页面时按需生成。

**这对于拥有许多可以静态生成的路由，但您不想仅通过在构建时生成子集来增加页面构建时间的应用程序非常有用。**

触发页面生成的用户将收到一个备用 (fallback) HTML，这通常是一个带有加载状态的页面。这样做的原因是静态 HTML 可以通过 CDN 提供，确保页面始终快速，即使尚未生成它。

按需静态生成其他页面的示例：

`pages/posts/[id].js`

```
import { useRouter } from 'next/router';
import fetch from 'node-fetch';
 
function Post({ post }) {
  const router = useRouter();
 
  // If the page is not yet generated, this will be displayed
  // initially until getStaticProps() finishes running
  if (router.isFallback) {
    return <div>Loading...</div>;
  }
 
  // Render post...
}
 
// This function gets called at build time
export async function getStaticPaths() {
  return {
    // Only `/posts/1` and `/posts/2` are generated at build time
    paths: [{ params: { id: 1 } }, { params: { id: 2 } }],
    // Enable statically generating additional pages
    // For example: `/posts/3`
    fallback: true,
  };
}
 
// This also gets called at build time
export async function getStaticProps({ params }) {
  // params contains the post `id`.
  // If the route is like /posts/1, then params.id is 1
  const res = await fetch(`https://.../posts/${params.id}`);
  const post = await res.json();
 
  // Pass post data to the page via props
  return { props: { post } };
}
 
export default Post;
```

要了解有关 `getStaticPaths` 的更多信息，请参阅 [Data Fetching 文档](/docs/pages/building-your-application/data-fetching)。

### getServerSideProps [](#getserversideprops)

如果您从页面导出一个名为 `getServerSideProps` 的 `async` 函数，Next.js 将在每次请求时渲染此页面（SSR）。

`getServerSideProps` 始终在服务端运行，并且代码会自动从浏览器捆绑包中摇树优化 (tree-shaking)，确保发送给浏览器的代码更少。这样您就不必担心在服务端和浏览器环境（这两者存在一些不一致之处）中执行数据获取代码的问题。这在许多情况下可以提高性能，因为服务端通常与数据源有更快的连接。它还通过减少暴露数据获取逻辑的来增加安全性。

这使您能够使用任何异步甚至同步的数据获取技术，包括 `fetch`、REST、GraphQL 或直接访问数据库。

在页面间使用 `next/link` 而不是在浏览器中执行 `getServerSideProps` 进行导航时，Next.js 将向服务端进行请求，返回调用 `getServerSideProps` 的结果。

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
export async function getServerSideProps(context) {
  return {
    props: {}, // will be passed to the page component as props
  };
}
```

`context` 参数是一个包含以下键的对象：

*   `params`: 如果此页面使用动态路由，`params` 包含路由参数。如果页面名称是 `[id].js`，那么 `params` 将类似于 `{ id: ... }`。要了解更多信息，请查看 [动态路由文档](/docs/pages/building-your-application/routing/dynamic-routes)。
*   `req`: [HTTP 请求对象](https://nodejs.org/api/http.html#http_class_http_incomingmessage)。
*   `res`: [HTTP 响应对象](https://nodejs.org/api/http.html#http_class_http_serverresponse)。
*   `query`: 查询字符串。


预览模式 [](#preview-mode)
-----------------------------

正如本文前面讨论的那样，当您的页面从内容管理系统 (CMS) 获取数据时，静态生成非常有用。但是，当您正在无头 CMS 中编写草稿，并希望在页面上立即预览草稿时，这并不是理想的选择。由于输出是静态的，预览变化变得更加困难，因为您必须重新生成该静态页面。

`getStaticProps` 在 Next.js 中的引入为利用 Next.js 在特定条件下的按需渲染能力打开了新的可能性。

例如，当从您的无头 CMS 预览草稿时，您将想要绕过静态渲染，使用草稿内容按需渲染页面，而不是已发布的内容。您将希望 Next.js 仅针对此特定情况绕过静态生成。

**我们很高兴宣布 Next.js 的一个新内置功能来解决此需求：预览模式 (Preview Mode)。**

预览模式允许用户绕过静态生成的页面，为例如来自内容管理系统 (CMS) 的草稿页面按需渲染 (SSR)。

但是，您并不局限于某些内容管理系统 (CMS)。预览模式直接集成到 `getStaticProps` 和 `getServerSideProps` 中，因此可以与任何类型的数据获取解决方案一起使用。

当您使用 `next start` 或将 [部署](/docs/pages/building-your-application/deploying) 到 [Vercel Edge Network](https://vercel.com/docs/edge-network) 时，预览模式已经可用。

自己在 [https://next-preview.vercel.app/](https://next-preview.vercel.app/) 上体验预览模式。

通过参考 [the documentation](/docs/pages/building-your-application/configuring/preview-mode) 了解更多关于 [预览模式](/docs/pages/building-your-application/configuring/preview-mode) 的信息。

全局样式表的内置 Sass 支持 [](#built-in-sass-support-for-global-stylesheets)
---------------------------------------------------------------------------------------------

[Next.js 9.2](https://nextjs.org/blog/next-9-2) 引入了 [全局 CSS 样式表的内置支持](https://nextjs.org/blog/next-9-2#built-in-css-support-for-global-stylesheets)，用更好的默认值替换 `next-css` 插件，以提供更具优化性的结果。

发布后，我们收到了越来越多的关于集成 Sass 支持的询问，因为许多转向 Next.js 的企业具有基于 Sass 的现有设计系统。

在调查 Next.js 插件使用情况时，我们发现目前约有 30% 的 Next.js 应用程序使用 `next-sass`。相比之下，44% 使用原生 CSS，6% 使用 Less。

此外，`next-sass` 与 `next-css` 具有相同的缺失限制。这意味着您可以在项目的每个文件中导入 Sass 文件，但是此导入的 Sass 文件对于整个应用程序是全局的。

在考虑到这些统计数据和建议后，我们很高兴地宣布 Next.js 现在内置了对导入 Sass 样式表的支持。

要开始让您的应用程序使用全局 Sass 导入，请安装 [`sass`](https://github.com/sass/sass)：

终端

```
npm install sass
```

然后，在 `pages/_app.js` 内导入 Sass 文件。

例如，考虑您的项目根目录中以下名为 `styles.scss` 的样式表：

```
$primary-color: #333;
 
body {
  padding: 20px 20px 60px;
  margin: 0;
  color: $primary-color;
}
```

如果尚不存在，请创建一个 `pages/_app.js` 文件。然后，导入 `styles.scss` 文件：

`pages/_app.js`

```
import '../styles.scss';
 
// This default export is required in a new `pages/_app.js` file.
export default function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
```

由于样式表本质上就是全局的，它们必须在 [Custom `<App>` component](/docs/pages/building-your-application/routing/custom-app) 中导入。这对于避免全局样式的类名和顺序冲突是必要的。

在开发环境中，以这种方式表达样式表允许您在编辑样式时自动更新页面。

在生产环境中，所有 Sass 和 CSS 文件将自动连接到一个单一的最小化 `.css` 文件中。此 CSS 文件将通过 `<link>` 标签加载，并自动注入 Next.js 生成的默认 HTML 标记中。

此新功能完全向后兼容。如果您正在使用 `@zeit/next-sass` 或其他 CSS 相关插件，该功能将被禁用以避免冲突。

如果您当前正在使用 `@zeit/next-sass`，我们建议您从您的 `next.config.js` 和 `package.json` 中删除该插件，从而在升级时转向内置的 Sass 支持。

组件级样式的内置 Sass CSS 模块支持 [](#built-in-sass-css-module-support-for-component-level-styles)
---------------------------------------------------------------------------------------------------------------------------

Next.js 现在支持使用 `[name].module.scss` 文件命名约定的 Sass 文件的 [CSS 模块](https://github.com/css-modules/css-modules)。

与以前 Next.js 5+ 中使用 `next-sass` 提供的支持相比，全局 Sass 和 CSS 模块现在可以**共存**——`next-sass` 要求应用程序中的所有 `.scss` 文件都作为全局或本地处理，但不能两者兼是。

CSS 模块通过自动创建唯一的类名来本地范围化 Sass。这允许您在不同的文件中使用相同的 Sass 类名而无需担心冲突。

这种行为使 CSS 模块成为包含组件级 Sass 的理想方式。CSS 模块文件**可以在您应用程序的任何地方导入**。

要开始让您的应用程序使用 Sass CSS 模块，请确保您已安装 [`sass`](https://github.com/sass/sass)：

终端

```
npm install sass
```

现在，考虑位于 `components/` 文件夹中的可重用 `Button` 组件：

首先，创建 `components/Button.module.scss`，内容如下：

```
/*
You do not need to worry about .error {} colliding with any other `.css` or
`.module.css` files!
*/
$color: white;
 
.error {
  color: $color;
  background-color: red;
}
```

然后，创建 `components/Button.js`，导入并使用上面的 CSS 文件：

`components/Button.js`

```
import styles from './Button.module.scss';
 
export function Button() {
  return (
    <button
      type="button"
      // Note how the "error" class is accessed as a property on the imported
      // `styles` object.
      className={styles.error}
    >
      Destroy
    </button>
  );
}
```

Sass 文件的 CSS 模块是一个**可选功能**，仅针对具有 `.module.scss` 扩展名的文件启用。常规的 [<link> stylesheets](/docs/pages/api-reference/components/head) 和 [全局 Sass 样式](https://nextjs.org/blog/next-9-2#built-in-css-support-for-global-stylesheets) 仍然受支持。

在生产环境中，所有 CSS 模块文件将自动连接为**多个最小化和代码分割的 `.css` 文件**。这些 `.css` 文件代表您应用程序中的热执行路径，确保您的应用程序每次绘制的 CSS 加载量最小化。

如上所述，此新功能完全向后兼容。如果您正在使用 `@zeit/next-sass` 或其他 CSS 相关插件，该功能将被禁用以避免冲突。

如果您当前正在使用 `@zeit/next-sass`，我们建议您从您的 `next.config.js` 和 `package.json` 中删除该插件，从而转向内置的 Sass 支持。

404 页面的自动静态优化 [](#automatic-static-optimization-for-404)
-------------------------------------------------------------------------------

Next.js 9 的发布引入了 [自动静态优化](https://nextjs.org/blog/next-9#automatic-static-optimization) 的概念，当页面没有阻塞式的数据需求时，Next.js 将在构建时自动将页面生成静态 HTML。然而，有一页没有自动渲染为静态 HTML：404 页面。404 页面没有自动转换为静态的主要原因是提供 404 支持的 `/_error` 页面处理的不仅仅是 404，还包括错误。

考虑到 404 页面会在不存在的路由下渲染，如果按需渲染页面会导致成本增加和服务端负载。

我们设定了两个途径来使您更容易成功：

*   默认的 Next.js 体验会生成静态 404 页面。
*   自定义 404 页面时也确保您最终得到一个静态页面。

此功能完全向后兼容，因此如果您目前有一个自定义 `pages/_error.js`，它将继续用于 404 页面，直到您添加了 `pages/404.js`。

### 默认静态 404 页面 [](#static-404-page-by-default)

当您的应用程序没有自定义 `pages/_error.js` 页面时，Next.js 将自动静态生成 404 页面，并在必须提供 404 时使用该页面。这是自动发生的，不需要任何更改。

### 使用 pages/404.js 自定义 404 页面 [](#custom-404-page-using-pages404js)

要覆盖默认 404 页面，您可以现在创建一个 `pages/404.js`，它仍然将在构建时自动静态优化。如果您的应用程序有一个，此页面用于代替 `pages/_error.js` 来渲染 404。

`pages/404.js`

```
export default () => <h1>This is the 404 page</h1>;
```

32 kB+ **更小的运行时 (15 kB+ 压缩)** [](#32-kb-smaller-runtime-15-kb-gzip)
-----------------------------------------------------------------------------

Next.js 支持与 React 本身 [相同的浏览器](https://reactjs.org/docs/react-dom.html#browser-support)，无需任何配置。这包括 Internet Explorer 11 (IE11) 和所有流行浏览器 (Edge, Firefox, Chrome, Safari, Opera 等)。

作为此兼容性的一部分，我们还编译您的应用程序以兼容 IE11：这允许您安全地使用 [ES6](https://github.com/lukehoban/es6features)\+ 语法特性、Async/Await、对象 Rest/Spread 属性等等——所有这一切都不需要任何配置。

此编译过程的一部分还涉及透明地注入必要的功能多实现 (polyfills)（例如 `Array.from` 或 `Symbol`）。但是，这些多实现仅用于 [不到 10% 的 Web 流量](https://caniuse.com/#feat=es6-module)，大多数情况下是为了支持 IE11。

从 Next.js 9.3 开始，Next.js 将自动加载支持旧版浏览器所需的 polyfills，并且仅在这些旧版浏览器中加载 polyfills。

实际上，这意味着对于 [80%+ 的用户](https://caniuse.com/#feat=es6-module)（原文 90%+，此处保持原文比例描述），您**首次加载 (First Load)** 大小中可消除 32 kB 或更多。

这些节省尺寸对于依赖更多浏览器特性的更大应用程序来说更大。

这些优化是完全自动的，不需要更改应用程序即可利用它！
