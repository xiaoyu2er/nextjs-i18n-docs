在我们即将召开的 [Next.js Conf 2025](https://nextjs.org/conf) 之前，Next.js 16 现已发布。

此次发布带来了 Turbopack、缓存和 Next.js 架构的最新改进。自从上一个 Beta 版本发布以来，我们添加了多项新功能和改进：

*   [**缓存组件**](/blog/next-16#cache-components): 基于部分预渲染 (PPR) 的新模型，使用缓存实现即时导航。
*   [**Next.js 开发工具 MCP**](/blog/next-16#nextjs-devtools-mcp): 模型上下文协议集成，用于改进调试和工作流。
*   [**Proxy**](/blog/next-16#proxyts-formerly-middlewarets): `middleware.ts` 被 `proxy.ts` 替代，以明确网络边界。
*   [**开发体验**](/blog/next-16#logging-improvements): 改进了构建和开发请求的日志记录。

提醒一下，以下功能在上一个 Beta 版本发布以来已可用：

*   [**Turbopack (稳定版)**](#turbopack-stable): 所有应用的默认打包工具，Fast Refresh 速度提升 5-10 倍，构建速度提升 2-5 倍
*   [**Turbopack 文件系统缓存（测试版）**](#turbopack-file-system-caching-beta): 针对大型应用，实现更快的启动和编译时间
*   [**React Compiler 支持（稳定版）**](#react-compiler-support-stable): 用于自动缓存的内置集成
*   [**构建适配器 API（Alpha）**](#build-adapters-api-alpha): 创建自定义适配器以修改构建流程
*   [**增强的路由**](#enhanced-routing-and-navigation): 优化的导航和预加载，支持布局去重和增量预加载
*   [**改进的缓存 API**](#improved-caching-apis): 新的 `updateTag()` 和优化的 `revalidateTag()`
*   [**React 19.2**](#react-192-and-canary-features): View Transitions、`useEffectEvent()`、`<Activity/>`
*   [**破坏性更改**](#breaking-changes-and-other-updates): 异步 `params`、`next/image` 默认值，以及更多

### 缓存组件 [](#cache-components)

缓存组件是一组新功能，旨在使 Next.js 中的缓存更加明确和灵活。它们围绕新的 `"use cache"` 指令构建，该指令可用于缓存页面、组件和函数，并利用编译器在其使用的任何位置自动生成缓存键。

与 App Router 先前版本中隐含的缓存不同，使用缓存组件完全是可选的。页面、布局或 API 路由中的任何动态代码默认都将在请求时执行，提供开箱即用的体验，更符合作为全栈应用框架的开发者预期。

缓存组件也完成了部分预渲染 (PPR) 的故事，PPR 最早于 2023 年引入。在 PPR 之前，Next.js 必须选择是静态渲染每个 URL 还是动态渲染；没有中间地带。PPR 消除了这种二元对立，让开发者可以选择静态页面的一部分进行动态渲染（通过 Suspense），而无需牺牲完全静态页面的快速初始加载。

您可以在 `next.config.ts` 文件中启用缓存组件：

next.config.ts

```
const nextConfig = {
  cacheComponents: true,
};
 
export default nextConfig;
```

我们将在 10 月 22 日的 [Next.js Conf 2025](https://nextjs.org/conf) 上分享更多关于缓存组件及其使用方法的内容，我们将在未来几周的更新日志和文档中分享更多内容。

> **注意：** 如此前在 Beta 版本中宣布，之前的实验性 `experimental.ppr` 标志和配置选项已被移除，取而代之的是缓存组件配置。

在文档中了解更多 [此处](/docs/app/api-reference/config/next-config-js/cacheComponents)。

### Next.js 开发工具 MCP [](#nextjs-devtools-mcp)

Next.js 16 推出了 **Next.js 开发工具 MCP**，这是一种模型上下文协议集成，用于 AI 辅助调试，并提供对应用程序的上下文洞察。

Next.js 开发工具 MCP 为 AI 代理提供：

*   **Next.js 知识**：路由、缓存和渲染行为
*   **统一日志**：无需切换上下文即可查看浏览器和服务器日志
*   **自动错误访问**：无需手动复制即可获取详细的堆栈跟踪
*   **页面感知**：对当前路由的上下文理解

这使得 AI 代理能够在您的开发工作流中直接诊断问题、解释行为并提出修复建议。

在文档中了解更多 [此处](/docs/app/guides/mcp)。

### `proxy.ts`（原 `middleware.ts`）[](#proxyts-formerly-middlewarets)

`proxy.ts` 替代了 `middleware.ts`，使应用程序的网络边界更加明确。`proxy.ts` 在 Node.js 运行时上运行。

*   **操作方法**：将 `middleware.ts` 重命名为 `proxy.ts`，并将导出的函数重命名为 `proxy`。逻辑保持不变。
*   **原因**：更清晰的命名和用于请求拦截的单一、可预测的运行时。

proxy.ts

```
export default function proxy(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url));
}
```

> **注意：** `middleware.ts` 文件仍然可用于边缘运行时用例，但已弃用，将在未来版本中移除。

在文档中了解更多 [此处](/docs/app/getting-started/proxy)。

### 开发人员体验 [](#developer-experience)

#### Turbopack (稳定版) [](#turbopack-stable)

Turbopack 的开发和生产构建均已达到稳定状态，现已成为所有新 Next.js 项目的默认打包工具。自今年夏天早期发布测试版以来，采用率迅速增长：Next.js 15.3+ 超过 50% 的开发会话和 20% 的生产构建已运行于 Turbopack。

使用 Turbopack，您可以期待：

*   生产构建速度提升 2 至 5 倍
*   Fast Refresh 速度提升高达 10 倍

我们将 Turbopack 设为默认，以便每位 Next.js 开发者都能获得这些性能提升，无需任何配置。对于具有自定义 webpack 设置的应用，可以通过运行以下命令继续使用 webpack：

终端

```
next dev --webpack
next build --webpack
```

#### Turbopack 文件系统缓存（测试版）[](#turbopack-file-system-caching-beta)

Turbopack 现在支持开发模式的文件系统缓存，在运行之间将编译器产物存储在磁盘上，从而在各次重启之间显著提升编译时间，对于大型项目尤为明显。

在您的配置中启用文件系统缓存：

next.config.ts

```
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
};
 
export default nextConfig;
```

所有内部的 Vercel 应用都在使用此功能，我们在大型存储库中观察到开发人员生产力的显著提升。

随着我们在文件系统缓存上的迭代，我们非常希望听到您的反馈。请尝试使用并分享您的体验。

#### 简化版 `create-next-app` [](#simplified-create-next-app)

`create-next-app` 已重新设计，采用简化的设置流程、更新的项目结构和改进的默认设置。新模板默认包含 App Router、TypeScript 优先配置、Tailwind CSS 和 ESLint。

#### 构建适配器 API（Alpha）[](#build-adapters-api-alpha)

继 [构建适配器 RFC](https://github.com/vercel/next.js/discussions/77740) 之后，我们携手社区和部署平台交付了构建适配器 API 的第一个 Alpha 版本。

构建适配器允许您创建自定义适配器以挂钩到构建流程，使部署平台和自定义构建集成能够修改 Next.js 配置或处理构建输出。

next.config.js

```
const nextConfig = {
  experimental: {
    adapterPath: require.resolve('./my-adapter.js'),
  },
};
 
module.exports = nextConfig;
```

请在 [RFC 讨论](https://github.com/vercel/next.js/discussions/77740) 中分享您的反馈。

#### React Compiler 支持（稳定版）[](#react-compiler-support-stable)

自 React Compiler 1.0 发布以来，React Compiler 的内置支持在 Next.js 16 中已达到稳定。React Compiler 可自动缓存组件，无需手动更改代码即可减少不必要的重新渲染。

`reactCompiler` 配置选项已从 `experimental` 提升为稳定。由于我们仍在收集不同类型应用程序的构建性能数据，该选项默认不启用。启用此选项时，预期开发和构建过程中的编译时间会较长，因为 React Compiler 依赖 Babel。

next.config.ts

```
const nextConfig = {
  reactCompiler: true,
};
 
export default nextConfig;
```

安装最新版本的 React Compiler 插件：

终端

```
npm install babel-plugin-react-compiler@latest
```

### 核心功能与架构 [](#core-features--architecture)

#### 增强的路由和导航 [](#enhanced-routing-and-navigation)

Next.js 16 包含了对路由和导航系统的全面重构，使页面过渡更精简、更快速。

**布局去重**：当使用共享布局预加载多个 URL 时，布局仅下载一次，而不是为每个 Link 单独下载。例如，包含 50 个产品链接的页面现在仅下载一次共享布局，而不是 50 次，大大减少了网络传输大小。

**增量预加载**：Next.js 仅预加载未缓存的部分，而不是整个页面。预加载缓存现在：

*   当链接离开视口时取消请求
*   在悬停或重新进入视口时优先处理链接预加载
*   当其数据无效时重新预加载链接
*   与即将到来的功能（如 Cache Components）无缝配合

**权衡**：您可能会看到更多的独立预加载请求，但总传输大小更低。我们认为这对于几乎所有应用都是正确的权衡。如果请求计数增加导致问题，请告诉我们。我们正在努力实现更有效地内联数据块的其他优化。

这些更改无需修改代码，旨在提升所有应用的性能。

#### 改进的缓存 API [](#improved-caching-apis)

Next.js 16 引入了改进后的缓存 API，用于对缓存行为进行更明确的控制。

##### `revalidateTag()`（更新）[](#revalidatetag-updated)

`revalidateTag()` 现在需要**[`cacheLife`](https://nextjs.org/docs/app/api-reference/functions/cacheLife#reference) profile**作为第二个参数以启用 stale-while-revalidate (SWR) 行为：

```
import { revalidateTag } from 'next/cache';
 
// ✅ Use built-in cacheLife profile (we recommend 'max' for most cases)
revalidateTag('blog-posts', 'max');
 
// Or use other built-in profiles
revalidateTag('news-feed', 'hours');
revalidateTag('analytics', 'days');
 
// Or use an inline object with a custom revalidation time
revalidateTag('products', { expire: 3600 });
 
// ⚠️ Deprecated - single argument form
revalidateTag('blog-posts');
```

配置文件参数接受内置的 `cacheLife` profile 名称（如 `'max'`, `'hours'`, `'days'`）或您在 `next.config` 中定义的 [自定义 profile](https://nextjs.org/docs/app/api-reference/functions/cacheLife#defining-reusable-cache-profiles)。您还可以传递内联 `{ expire: number }` 对象。我们推荐使用 `'max'`，因为它允许长期内容进行后台重新验证。当用户请求带标签的内容时，他们立即收到缓存数据，同时 Next.js 在后台重新验证。

当您希望无效化仅标记且带有 stale-while-revalidate 行为的缓存条目时，请使用 `revalidateTag()`。这适用于可容忍最终一致性的静态内容。

> **迁移指南：** 添加带有 `cacheLife` profile 的第二个参数（我们推荐 `'max'`）以实现 SWR 行为，或者在 Server Actions 中使用 `updateTag()` 如果您需要读后写语义。

##### `updateTag()`（新增）[](#updatetag-new)

`updateTag()` 是一个全新的仅限 Server Actions 的 API，提供 **读后写** 语义，在同一请求中过期并立即读取新数据：

```
'use server';
 
import { updateTag } from 'next/cache';
 
export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile);
 
  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`);
}
```

这确保了交互功能能立即反映更改。非常适合表单、用户设置以及用户期望即时看到更新的工作流程。

##### `refresh()`（新增）[](#refresh-new)

`refresh()` 是一个全新的仅限 Server Actions 的 API，用于刷新 **仅未缓存数据**。它完全不会触碰缓存：

```
'use server';
 
import { refresh } from 'next/cache';
 
export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId);
 
  // Refresh the notification count displayed in the header
  // (which is fetched separately and not cached)
  refresh();
}
```

此 API 与客户端 `router.refresh()` 互补。当您在执行操作后需要刷新页面上其他位置显示的未缓存数据时使用。您的缓存页面外壳和静态内容保持快速，而动态数据如通知计数、实时指标或状态指示器则进行刷新。

#### React 19.2 和 Canary 功能 [](#react-192-and-canary-features)

Next.js 16 中的 App Router 使用了最新的 React [Canary 发布](https://react.dev/blog/2023/05/03/react-canaries)，其中包括刚刚发布的 React 19.2 功能和其他正在逐步稳定的功能。亮点包括：

*   **[View Transitions](https://react.dev/reference/react/ViewTransition)**：动画显示在 Transition 或导航内部更新的元素
*   **[`useEffectEvent`](https://react.dev/reference/react/useEffectEvent)**：将非响应式逻辑从 Effects 中提取到可复用的 Effect Event 函数中
*   **[Activity](https://react.dev/reference/react/Activity)**：通过 `display: none` 隐藏 UI 来渲染“后台活动”，同时保留状态并清理 Effects

有关更多信息，请查看 [React 19.2 宣布](https://react.dev/blog/2025/10/01/react-19-2)。

### 破坏性更改和其他更新 [](#breaking-changes-and-other-updates)

#### 版本要求 [](#version-requirements)

| 更改 | 详细信息 |
| --- | --- |
| **Node.js 20.9+** | 最低版本现在为 20.9.0 (LTS)；不再支持 Node.js 18 |
| **TypeScript 5+** | 最低版本现在为 5.1.0 |
| **浏览器** | Chrome 111+, Edge 111+, Firefox 111+, Safari 16.4+ |

#### 移除 [](#removals)

这些功能之前已被弃用，现已移除：

| Removed | Replacement |
| --- | --- |
| **AMP support** | 所有 AMP APIs 和配置已移除 (`useAmp`, `export const config = { amp: true }`) |
| **`next lint` command** | 直接使用 Biome 或 ESLint；`next build` 不再运行 linting。提供一个 CodeMod：`npx @next/codemod@canary next-lint-to-eslint-cli .` |
| **`devIndicators` options** | `appIsrStatus`, `buildActivity`, `buildActivityPosition` 从配置中移除。指示器保留。 |
| **`serverRuntimeConfig`, `publicRuntimeConfig`** | 使用环境变量 (`.env` 文件) |
| **`experimental.turbopack` location** | 配置移至顶级 `turbopack`（不再位于 `experimental` 中） |
| **`experimental.dynamicIO` flag** | 重命名为 `cacheComponents` |
| **`experimental.ppr` flag** | PPR 标志已移除；正演变为缓存组件编程模型 |
| **`export const experimental_ppr`** | 路由级别的 PPR 导出已移除；正演变为缓存组件编程模型 |
| **Automatic `scroll-behavior: smooth`** | 在 HTML 文档中添加 `data-scroll-behavior="smooth"` 以选回 |
| **`unstable_rootParams()`** | 我们正在研究一个替代 API，我们将在未来的次要版本中发布 |
| **Async `params`, `searchParams` props access** | 必须使用异步：`await params`, `await searchParams` |
| **Async `cookies()`, `headers()`, `draftMode()` access** | 必须使用异步：`await cookies()`, `await headers()`, `await draftMode()` |
| **Metadata image route `params` argument** | 更改为异步 `params`；`generateImageMetadata` 中的 `id` 现在为 `Promise<string>` |
| **`next/image` local src with query strings** | 现在需要 `images.localPatterns` 配置以防止枚举攻击 |

#### 行为更改 [](#behavior-changes)

这些功能在 Next.js 16 中具有新的默认行为：

| 更改的行为 | 详细信息 |
| --- | --- |
| **默认打包工具** | Turbopack 已成为所有应用的默认打包工具；通过 `next build --webpack` 选出不使用 |
| **`images.minimumCacheTTL` 默认** | 从 60 秒更改为 4 小时 (14400 秒)；减少无 cache-control 头的图像的验证成本 |
| **`images.imageSizes` 默认** | 从默认尺寸中移除 `16` (仅 4.2% 的项目使用)；减少 `srcset` 大小和 API 变体 |
| **`images.qualities` 默认** | 从 `[1..100]` 更改为 `[75]`；`quality` prop 现在强制转换为 `images.qualities` 中的最接近值 |
| **`images.dangerouslyAllowLocalIP`** | 新的安全限制默认阻止本地 IP 优化；仅对私有网络设置为 `true` |
| **`images.maximumRedirects` 默认** | 从不限制到最多 3 次重定向；设置为 `0` 以禁用或针对罕见边缘情况增加 |
| **`@next/eslint-plugin-next` 默认** | 现在默认为 ESLint Flat Config 格式，与将弃用旧配置支持的 ESLint v10 保持一致 |
| **预加载缓存行为** | 完全重写，包含布局去重和增量预加载 |
| **`revalidateTag()` 签名** | 现在需要 `cacheLife` profile 作为第二个参数以实现 stale-while-revalidate 行为 |
| **Turbopack 中的 Babel 配置** | 找到 babel 配置时自动启用 Babel (以前会硬编码退出错误) |
| **终端输出** | 重新设计，具有更清晰的格式、更好的错误信息和改进的性能指标 |
| **开发和构建输出目录** | `next dev` 和 `next build` 现在使用独立的输出目录，启用并发执行 |
| **锁定文件行为** | 添加了锁定文件机制以防止同一项目上的多个 `next dev` 或 `next build` 实例 |
| **并行路由 `default.js`** | 所有并行路由槽现在都需要显式的 `default.js` 文件；没有它们会构建失败。创建调用 `notFound()` 或返回 `null` 的 `default.js` 以获得之前的行为 |
| **现代 Sass API** | 将 `sass-loader` 提升至 v16，支持现代 Sass 语法和新功能 |

#### 弃用 [](#deprecations)

这些功能在 Next.js 16 中已被弃用，将在未来版本中移除：

| 弃用项 | 详细信息 |
| --- | --- |
| `middleware.ts` 文件名 | 重命名为 `proxy.ts` 以澄清网络边界和路由重点 |
| `next/legacy/image` 组件 | 使用 `next/image` 以获得更好的性能和功能 |
| `images.domains` 配置 | 使用 `images.remotePatterns` 配置以获得更好的安全限制 |
| `revalidateTag()` 单参数 | 使用 `revalidateTag(tag, profile)` 实现 SWR，或在 Actions 中使用 `updateTag(tag)` 实现读后写 |

#### 其他改进 [](#additional-improvements)

*   **性能改进**：`next dev` 和 `next start` 命令的重大性能优化
*   **Node.js 原生 TypeScript 用于 `next.config.ts`**：使用 `--experimental-next-config-strip-types` 标志运行 `next dev`、`next build` 和 `next start` 命令，为 `next.config.ts` 启用原生 TypeScript。
