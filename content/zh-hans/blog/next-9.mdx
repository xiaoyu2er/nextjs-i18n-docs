经过 70 个试验版发布，我们很高兴地介绍 Next.js 9，特性包括：

*   **[零配置 TypeScript 支持 (Built-in Zero-Config TypeScript Support)](#built-in-zero-config-typescript-support)**：得益于自动 TypeScript 支持和集成类型检查，构建您的应用程序更加充满信心。
*   **[基于文件系统 (File system-Based) 动态路由 (Dynamic Routing)](#dynamic-route-segments)**：通过文件系统表达复杂的应用程序路由需求而无需自定义服务器。
*   **[自动静态优化 (Automatic Static Optimization)](#automatic-static-optimization)**：默认借助 _服务端渲染 (Server-Side Rendering) 和静态预渲染 (Static Prerendering)_ 创建超快网站，同时不牺牲功能。
*   **[API 路由 (API Routes)](#api-routes)**：快速构建后端应用程序端点，利用热重载和统一的构建管道。
*   **[更多生产优化 (More Production Optimizations)](#production-optimizations)**：由于视口内预取和多项其他优化，应用程序比以往更加响应迅速。
*   **[改进的开发体验 (Improved DX)](#developer-experience-improvements)**：不显眼的、易于使用的改进，帮助您以最佳状态开发。

零配置 TypeScript 支持内置 [](built-in-zero-config-typescript-support)
-----------------------------------------------------------------------------------

一年前 [Next.js 6](https://nextjs.org/blog/next-6) 通过名为 `@zeit/next-typescript` 的插件引入了基本的 TypeScript 支持。用户还必须自定义他们的 `.babelrc` 并在 `next.config.js` 中启用它。

配置时，该插件将允许 `.ts` 和 `.tsx` 文件由 Next.js 构建。然而，它没有集成类型检查，Next.js 核心也没有提供类型。这意味着必须在 DefinitelyTyped 中单独维护一个社区软件包，该软件包可能与版本不同步。

在与许多现有和新的用户交谈时，很明显大多数人非常有兴趣使用 TypeScript。他们希望有一个更可靠和标准的解决方案来轻松地将 TypeScript 集成到他们的现有或新代码库中。

出于这个原因，我们要将 TypeScript 支持集成到 Next.js 核心中，从而改善开发体验，并在此过程中使其更快。

##### 自动化设置 [](automated-setup)

在 Next.js 中使用 TypeScript 开始非常简单：将任何文件、页面或组件从 `.js` 重命名为 `.tsx`。然后，运行 `next dev`！

这将导致 Next.js 检测到您的项目正在使用 TypeScript。Next.js CLI 将指导您安装 React 和 Node.js 所需的类型。

如果 `tsconfig.json` 尚不存在，Next.js 还将创建一个具有合理默认值的默认 `tsconfig.json`。此文件允许在 [Visual Studio Code](https://code.visualstudio.com/) 等编辑器中集成类型检查。

动态路由段 [](dynamic-route-segments)
-------------------------------------------------

动态路由（也称为 URL 路径段 (URL Slugs) 或漂亮/整洁 URLs）是在 Next.js 发布 2.5 年后 GitHub 上提出的第一个功能请求之一！

该问题在第 2.0 版的 Next.js 中通过引入用于以编程方式使用 Next.js 的自定义服务器 API 来“解决”。这允许将 Next.js 用作渲染引擎，启用将传入 URL 映射到渲染特定页面的抽象。

我们与用户交谈并检查了许多他们的应用程序，发现其中许多都有自定义服务器。出现了一种模式：自定义服务器最突出的原因是动态路由。

然而，自定义服务器有其自身的陷阱：路由在服务器级别处理而不是代理，它作为单体部署和扩展，并且容易出现性能问题。

由于自定义服务器要求整个应用程序在一个实例中可用，因此通常难以部署到解决这些问题的 Serverless 环境中。Serverless 请求在代理层路由，并且独立进行缩放/执行以避免性能瓶颈。

此外，我们相信我们可以提供更好的开发体验！当您创建一个名为 `pages/blog.js` 的文件并突然有一个页面可通过 `/blog` 访问时，Next.js 的许多魔力就此开始。

为什么用户需要创建自己的服务器并了解 Next.js 的程序化 API 来支持像 `/blog/my-first-post` (`/blog/:id`) 这样的路由？

基于这些反馈和愿景，我们开始研究路由映射解决方案，推动用户已经知道的内容：`pages/` 目录。

##### 创建动态路由页面 [](creating-a-dynamically-routed-page)

Next.js 支持使用基本命名参数创建路由，这一模式由 [`path-to-regexp`](https://www.npmjs.com/package/path-to-regexp) (为 [Express](https://expressjs.com) 提供动力的库) 普及。

通过创建名为 `pages/post/[pid].js` 的文件，现在可以实现匹配路由 `/post/:pid` 的页面！

Next.js 将自动匹配像 `/post/1`、`/post/hello-nextjs` 等请求，并渲染在 `pages/post/[pid].js` 中定义的页面。匹配的路由段将作为名称指定在 `[square-brackets]` 之间的查询参数传递给您的页面。

例如：给定以下页面和请求 `/post/hello-nextjs`，`query` 对象将为 `{ pid: 'hello-nextjs' }`：

```
static async getInitialProps({ query }) {
  // pid = 'hello-nextjs'
  const { pid } = query
 
  const postContent = await fetch(
    `https://api.example.com/post/${encodeURIComponent(pid)}`
  ).then(r => r.text())
 
  return { postContent }
}
```

还支持多个动态 URL 段！

`[param]` 语法支持目录名称和文件名称，这意味着以下示例有效：

```
./pages/blog/[blogId]/comments/[commentId].js
./pages/posts/[pid]/index.js
```

您可以在 [Next.js 文档](https://github.com/vercel/next.js#dynamic-routing) 或 [Next.js 学习部分](https://nextjs.org/learn/basics/dynamic-routes) 中阅读有关此功能的更多信息。

自动静态优化 [](automatic-static-optimization)
---------------------------------------------------------------

Next.js 在 v3 版中增加了对静态网站生成 (static website generation) 的支持，大约两年前发布。当时，这是最请求添加到 Next.js 的功能。

很有道理：无可否认静态网站 _快！_它们不需要服务器端计算，并且可以从 CDN 位置立即流式传输到最终用户。

然而，服务器端渲染或静态生成应用程序之间的选择是二元的，您要么选择服务器端渲染，要么选择静态生成。没有中间地带。

实际上，应用程序可以有不同的需求。这些需求需要不同的渲染策略和权衡。

例如，主页和营销页面通常包含静态内容，是静态优化的绝佳候选。

另一方面，产品仪表板可能受益于服务器端渲染，其中数据经常更新。

我们开始探索如何给予用户两全其美的方案，并**默认快速**。我们如何给予用户静态营销页面和动态服务器渲染页面？

从 Next.js 9 开始，用户不再必须在完全服务器渲染或静态导出应用程序之间做出选择。在每页基础上给予您两全其美的优势。

#### 自动部分静态导出 [](automatic-partial-static-export)

引入了一种启发式算法来自动确定页面是否可以预渲染为静态 HTML。

这种决定是根据页面是否通过 [`getInitialProps`](https://github.com/vercel/next.js#fetching-data-and-component-lifecycle) 具有阻塞的数据要求来确定的。

这种启发式算法允许 Next.js 发出混合应用程序，包含 **服务器端渲染 (server-rendered) 和静态生成 (statically generated) 的页面**。

内置的 Next.js 服务器 (`next start`) 和程序化 API (`app.getRequestHandler()`) 都透明地支持此构建输出。**无需配置或特殊处理**。

静态生成的页面仍然是响应式的：Next.js 将在客户端重新水合 (hydration) 您的应用程序，以给予其完全的交互性。

此外，如果页面依赖于 URL 中的查询参数，Next.js 将在水合后更新您的应用程序。

在开发过程中，如果页面将静态生成，Next.js 将直观地通知您。可以通过点击它来隐藏此视觉伪影。

API 路由 [](api-routes)
-------------------------

在构建 React 应用程序的许多情况下，您最终需要某种后端。要么检索数据库中的数据，要么处理用户提供的数据（例如联系表单）。

我们发现，许多需要后端服务的用户都使用自定义服务器构建他们的 API。在这样做的过程中，他们遇到了很多问题。例如，Next.js 不编译自定义服务器代码，这意味着您不能使用 `import` / `export` 或 TypeScript。

出于这个原因，许多用户最终在自定义服务器之上实现了自己的自定义编译管道。虽然这解决了他们的目标，但它容易引发很多问题：例如，如果配置不正确，摇树 (Tree Shaking) 将被禁用整个应用程序。

这就提出了一个问题：如果我们将在构建 API 后端时 Next.js 提供的开发体验带过来会怎样？

今天，我们很兴奋地介绍 API 路由，这是 Next.js 用于构建后端的最佳开发体验。

要开始使用 API 路由，请在 `pages/` 目录内创建一个名为 `api/` 的目录。

此目录中的任何文件将自动映射到 `/api/<your route>`，就像其他页面文件映射到路由一样。

例如，`pages/api/contact.js` 将映射到 `/api/contact`。

> **注意**：API 路由也支持 [动态路由 (Dynamic Routes)](#dynamic-route-segments)！

`pages/api/` 目录中的所有文件都导出一个请求处理函数，而不是 React 组件：

```
export default function handle(req, res) {
  res.end('Hello World');
}
```

*   `req` 指的是 [NextApiRequest](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L143-L158)，它扩展了 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
*   `res` 指的是 [NextApiResponse](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L168-L178)，它扩展了 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)

通常 API 端点接收一些传入数据，例如查询字符串、请求 body 或 cookies，并响应其他数据。

在调查为 Next.js 添加 API 路由支持时，我们注意到在许多情况下用户不直接使用 Node.js 请求和响应对象。相反，他们使用由 [Express](https://expressjs.com) 等服务器库提供的抽象。

这样做的原因是，在许多情况下，传入数据是某种文本形式，解析后才能有用。因此，这些特定的服务器库有助于减轻手动解析数据的负担，通常通过中间件 (middlewares) 完成。最常用的提供查询字符串、body 和 cookies 解析，但它们仍然需要一些设置才能开始。

Next.js 中的 API 路由将默认提供这些中间件，以便您可以立即开始生产 API 端点：

```
export default function handle(req, res) {
  console.log(req.body); // 请求 body
  console.log(req.query); // Url 查询字符串
  console.log(req.cookies); // 传递的 cookies
  res.end('Hello World');
}
```

除了使用传入数据外，您的 API 端点通常还返回数据。通常此响应为 JSON。Next.js 默认提供 `res.json()` 以方便发送数据：

```
export default function handle(req, res) {
  res.json({ title: 'Hello World' });
}
```

在开发中对 API 端点进行更改时，代码会自动重新加载，因此无需重启服务器。

生产优化 [](production-optimizations)
-----------------------------------------------------

#### 视口内 `<Link>` 预获取 [](prefetching-in-viewport-links)

Next.js 9 将在 `<Link>` 组件出现在视口内时自动预取它们。

此功能通过使新页面的导航更快来提高应用程序的响应能力。

Next.js 使用 [Intersection Observer](https://developer.mozilla.org/docs/Web/API/Intersection_Observer_API) 在后台 [预取 (prefetch)](https://www.w3.org/TR/resource-hints/#prefetch) 所需资源。

这些请求具有低优先级，并让位于 `fetch()` 或 XHR 请求。如果用户启用了数据节省模式，Next 将避免自动预取。

您可以通过将 `prefetch` 属性设置为 `false` 来不选择此功能：

```
<Link href="/terms" prefetch={false}>
  <a>服务条款</a>
</Link>
```

#### 默认优化 AMP [](optimized-amp-by-default)

Next.js 9 现在默认针对 AMP 优先和混合 AMP 页面渲染优化的 AMP。

虽然 AMP 页面是可选加入的，但 Next.js 将自动优化它们的输出。这些优化可能会导致高达 [50% 更快的](https://blog.amp.dev/2018/10/08/how-to-make-amp-even-faster/) 渲染速度！

这项改变是由 Sebastian Benz 在 [AMP Optimizer](https://github.com/ampproject/amp-toolbox/tree/master/packages/optimizer) 上的出色工作所促成的。

#### typeof window 分支的删除死代码 [](dead-code-elimination-for-typeof-window-branches)

Next.js 9 在服务器和客户端构建期间用其适当的值（`undefined` 或 `object`）替换 `typeof window`。此更改允许 Next.js 自动从您的生产构建应用程序中删除死代码。

如果用户有 `getInitialProps` 或其他应用程序部分的服务器端代码，他们应该会看到客户端 bundle 大小减小。

开发者体验改进[](##developer-experience-improvements)

#### 编译指示器 [](compiling-indicator)

在 9 之前的版本中，唯一知道正在进行热代码替换（和 Next.js 编译器工具链正在进行工作）的方法是查看开发人员控制台。

然而，很多时候人们是在查看生成的渲染，这使得很难知道 Next.js 是否仍在进行编译工作。例如，您可能正在对页面上细微的样式进行更改，您不会立即知道它们是否已更新。

出于这个原因，我们创建了一个 [RFC / "适合新手的问题"](https://github.com/vercel/next.js/issues/4626) 来讨论指示工作正在进行的问题的潜在解决方案。

我们从许多设计师和工程师那里收到了有关 RFC 的反馈，例如他们喜欢什么以及指示器设计的潜在方向。

[**Rafael Almeida**](https://github.com/rafaelalmeidatk) 利用这次机会与我们的团队合作，并实现了一个全新的指示器，现在默认可用在 Next.js 9 中。

每当 Next.js 进行编译工作时，您都会看到页面右下角出现一个小三角形！

Next.js 编译指示器

#### 控制台输出 [](console-output)

通常在进行开发更改时，Next.js 会显示一个编译指示器状态，加载状态栏填充，并且随着您进行更改而不断清除屏幕。

这种行为会导致一些问题。最明显的是，它也会清除您应用程序代码的控制台输出，例如，当您向组件添加 `console.log` 时。当使用像 [Vercel CLI](https://vercel.com/cli) 或 `docker-compose` 这样合并日志输出的外部工具时也是如此。

从 Next.js 9 开始，日志输出减少跳跃，不再清除屏幕。这允许更好的整体体验，因为您的终端窗口将拥有更多信息，并且在 Next.js 与您可能已经开始使用的工具集成更好时闪烁更少。

每页配置对象 [](per-page-configuration-object)
---------------------------------------------------------------

每个页面现在都可以导出一个配置对象。最初这个配置允许您选择加入 [AMP](https://nextjs.org/blog/next-8-1)，但未来您将能够配置更多页面特定选项。

[基于文件系统的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在最初的文章中，我们展示了从单个 React 组件创建路由的示例：

```
export const config = { amp: true };
 
export default function AboutPage(props) {
  return <h3>我的 AMP 关于页面！</h3>;
}
```

要选择加入混合 AMP 渲染，您可以使用 `'hybrid'` 值：

[基于文件系统的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在最初的文章中，我们展示了从单个 React 组件创建路由的示例：

```
import { useAmp } from 'next/amp';
 
export const config = { amp: 'hybrid' };
 
export default function AboutPage(props) {
  const isAmp = useAmp();
  return <h3>我的关于页面！{isAmp ? <> Powered by AMP!</> : ''}</h3>;
}
```

`withAmp` 高阶组件已删除以使用此新配置。

我们提供了一个 [codemod](https://github.com/vercel/next-codemod#withamp-to-config) 自动将 `withAmp` 的使用转换为新的配置对象。您可以在 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md) 中阅读更多关于此的内容。

代码库改进 [](codebase-improvements)
-----------------------------------------------

我们最近对我们的工具进行了一些更改，为贡献代码库提供更好的体验，并在代码库增长时确保稳定性。

正如您在 TypeScript 部分所读到的那样，Next.js 核心现在使用 TypeScript 编写，并且为 Next.js 应用程序自动生成类型。除了这对使用 Next.js 构建的应用程序有用外，当在核心代码库上工作时也很有用。当您自动获得类型错误和自动完成时。

Next.js 已经有一个相当大的集成测试套件，由 50+ 个 Next.js 应用程序组成，测试与它们一起运行。这些测试确保当发布新版本时，升级是平滑的，因为之前可用的功能都用相同的测试套件进行了测试。

我们的大多数测试都是集成测试，因为在许多情况下它们复制了开发人员在使用 Next.js 进行开发。例如，我们有一些测试复制对 Next.js 应用程序进行更改，以查看热模块替换是否有效。

我们的集成测试主要基于 Selenium webdriver，结合 chromedriver 在无头 Chrome 中测试。但随着时间的推移，其他浏览器出现了一些问题，特别是像 Internet Explorer 11 这样的较旧浏览器。

因为我们使用了 Selenium，所以能够在多个浏览器上自动运行测试。

目前，我们已经在 Chrome、Firefox、Safari 和 Internet Explorer 11 上运行测试套件。

Google Chrome 合作 [](#google-chrome-collaboration)

Google Chrome 团队一直致力于通过提交 RFC 和 pull-requests 来改进 Next.js。

例如，这些更改将改善小型网站的体验，但也像 [Hulu](https://hulu.com)、[Twitch](https://m.twitch.tv) 和 [Deliveroo](https://deliveroo.co.uk/) 这样的大型应用程序的体验。

### Google Chrome 合作 [](#google-chrome-collaboration)

第一个重点是向支持现代 JavaScript 的浏览器交付现代 JavaScript。

例如，目前 Next.js 必须为 [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 语法提供 polyfills，因为代码可能在不支持 [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 的浏览器中执行，这将导致错误。

![Next.js Module/Nomodule 合作 RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/first-collab-rfc.png)

Next.js Module/Nomodule 合作 RFC

为了避免破坏旧浏览器，同时向支持它的浏览器发送现代 JavaScript，Next.js 将利用 [module/nomodule 模式](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/)。module/nomodule 模式提供了一种可靠的机制来向现代浏览器提供现代 JavaScript，同时允许旧浏览器回退到 polyfilled ES5。

Next.js 中的 module/nomodule 的 RFC 可以在 [此处](https://github.com/vercel/next.js/issues/7563) 找到。

#### 改进的 bundle 拆分 [](improved-bundle-splitting)

Next.js 当前的 bundle 拆分策略基于基于比例启发式的将模块包含在单个“commons”chun 中。因为粒度很少（只有一个 bundle），代码要么不必要地下载（因为 commons bundle 可能包含实际上不是特定路由所需的代码），要么代码在多个页面 bundle 中重复。
