* **[零配置 TypeScript 支持](#built-in-zero-config-typescript-support)**: Build your application with increased confidence, thanks to automatic TypeScript support and integrated type-checking.
* **[基于文件系统的动态路由](#dynamic-route-segments)**: Express complex application routing requirements through the file system without the need for a custom server.

## 内置零配置 TypeScript 支持 [](#built-in-zero-config-typescript-support)
-----------------------------------------------------------------------------------

一年前，[Next.js 6](https://nextjs.org/blog/next-6) 通过名为 `@zeit/next-typescript` 的插件引入了基本的 TypeScript 支持。用户还必须自定义他们的 `.babelrc`，并在 `next.config.js` 中启用它。

配置后，该插件允许 `.ts` 和 `.tsx` 文件由 Next.js 构建。但是，它没有集成类型检查，Next.js 核心也没有提供类型。这意味着必须维护一个独立的社区包（在 DefinitelyTyped 中），该包可能与发布不同步。

在与许多用户交谈后，现有和新用户都显示出了对使用 TypeScript 的浓厚兴趣。他们希望有一种更可靠、更标准的解决方案，以便轻松地将 TypeScript 集成到现有的或新的代码库中。

因此，我们将 TypeScript 支持集成到 Next.js 核心中，以改善开发者体验，并在此过程中加快速度。

##### 自动化设置 [](#automated-setup)

在 Next.js 中使用 TypeScript 很简单：将任何文件、页面或组件从 `.js` 重命名为 `.tsx`。然后，运行 `next dev`！

这将使 Next.js 检测到您的项目正在使用 TypeScript。Next.js CLI 将指导您安装 React 和 Node.js 所需的所有类型。

Next.js 还将创建一个默认的 `tsconfig.json` 文件（如果尚不存在），其中包含合理的默认设置。此文件允许在 [Visual Studio Code](https://code.visualstudio.com/) 等编辑器中进行集成类型检查。

##### 集成类型检查 [](#integrated-type-checking)

Next.js 负责为您处理开发阶段和构建生产版本时的类型检查。

在开发过程中，保存文件后 Next.js 会显示类型错误。类型检查在后台进行，允许您立即在浏览器中与更新后的应用程序进行交互。一旦获取到类型错误，它们将传播到浏览器中。

Next.js 9 开发阶段类型检查

如果存在类型错误，Next.js 还会自动使生产构建（即 `next build`）失败。这有助于防止将损坏的代码发布到生产环境。

## 动态路由片段 [](#dynamic-route-segments)
-------------------------------------------------

动态路由（也称为 URL Slug 或美观 URL）是 Next.js 发布 2.5 年后 GitHub 上最早的增求之一！

这个问题在 Next.js 2.0 中通过引入自定义服务器 API 以编程方式使用 Next.js 而“解决”。这允许将 Next.js 用作渲染引擎，启用抽象并将传入的 URL 映射到以渲染特定页面。

我们与用户交谈并检查了许多应用，发现其中许多使用了自定义服务器。一种模式浮现出来：使用自定义服务器的最主要原因是动态路由。

然而，自定义服务器有其固有的弊端：路由在服务器级别而不是代理级别处理，它作为单体进行部署和扩展，并且容易出现性能问题。

由于自定义服务器需要整个应用程序在一个实例中可用，因此通常难以部署到解决这些问题的 Serverless 环境中。Serverless 请求在代理层进行路由，并独立进行扩展/执行，以避免性能瓶颈。

此外，我们认为我们可以提供更好的开发者体验！Next.js 的许多魔力始于您创建一个名为 `pages/blog.js` 的文件，然后突然就有了在 `/blog` 可访问的页面。

为什么用户需要创建自己的服务器并了解 Next.js 的编程 API 来支持像 `/blog/my-first-post` (`/blog/:id`) 这样的路由？

基于这些反馈和愿景，我们开始研究路由映射解决方案，由用户已知的 `pages/` 目录驱动。

##### 创建动态路由页面 [](#creating-a-dynamically-routed-page)

Next.js 支持使用基本命名参数创建路由，这是由 [`path-to-regexp`](https://www.npmjs.com/package/path-to-regexp) 普及的模式（驱动 [Express](https://expressjs.com) 的库）。

要创建与路由 `/post/:pid` 匹配的页面，现在可以通过在 `pages` 目录中创建一个名为 `pages/post/[pid].js` 的文件来实现！

Next.js 将自动匹配请求，如 `/post/1`、`/post/hello-nextjs` 等，并渲染 `pages/post/[pid].js` 中定义的页面。匹配的 URL 段将作为查询参数传递给您的页面，名称指定在方括号 `[square-brackets]` 之间。

例如：假设给定以下页面和请求 `/post/hello-nextjs`，`query` 对象将为 `{ pid: 'hello-nextjs' }`：

```
static async getInitialProps({ query }) {
  // pid = 'hello-nextjs'
  const { pid } = query
 
  const postContent = await fetch(
    `https://api.example.com/post/${encodeURIComponent(pid)}`
  ).then(r => r.text())
 
  return { postContent }
}
```

还支持多个动态 URL 段！

`[param]` 语法支持目录名和文件名，这意味着以下示例可行：

```
./pages/blog/[blogId]/comments/[commentId].js
./pages/posts/[pid]/index.js
```

您可以阅读 [Next.js 文档中关于此功能的更多信息](https://github.com/vercel/next.js#dynamic-routing) 或 [Next.js Learn 部分](https://nextjs.org/learn/basics/dynamic-routes)。

## 自动静态优化 [](#automatic-static-optimization)
---------------------------------------------------------------

Next.js 在 v3（大约两年前发布）中增加了对静态网站生成的支持。当时，这是被要求添加到 Next.js 中的最广泛请求的功能。

而且有充分的理由：不可否认，静态网站_速度极快_！它们不需要服务器端计算，并且可以从 CDN 位置即时流式传输给最终用户。

然而，选择服务器端渲染还是静态生成应用程序是二选一的，您要么选择服务器端渲染，要么选择静态生成。没有折衷方案。

实际上，应用程序可以有不同的需求。这些需求需要不同的渲染策略和权衡。

例如，首页和营销页面通常包含静态内容，非常适合进行静态优化。

另一方面，产品仪表板可能受益于服务器端渲染，因为数据频繁更新。

我们开始探索如何让用户兼得两者之利，并**默认快速**。我们如何向用户提供静态营销页面和动态服务器渲染页面？

从 Next.js 9 开始，用户不再需要在完全服务器渲染或静态导出应用程序之间做出选择。按页面为您提供两者的最佳之处。

#### 自动部分静态导出 [](#automatic-partial-static-export)

引入了启发式方法来自动确定页面是否可以预渲染为静态 HTML。

此决定取决于页面是否通过使用 [`getInitialProps`](https://github.com/vercel/next.js#fetching-data-and-component-lifecycle) 具有阻塞性数据需求。

此启发式方法允许 Next.js 生成混合应用程序，其中包含**同时包含服务器渲染和静态生成的页面**。

内建 Next.js 服务器（`next start`）和编程 API（`app.getRequestHandler()`）都**透明地**支持此构建输出。无需配置或特殊处理。

静态生成的页面仍然具有响应性：Next.js 将在客户端水合您的应用程序以提供完整的交互性。

此外，如果页面依赖 URL 中的查询参数，Next.js 还将在水合后更新您的应用程序。

在开发期间，如果页面将被静态生成，Next.js 会视觉化地通知您。可以通过点击它来隐藏此视觉提示。

## API 路由 [](#api-routes)
-------------------------

在构建 React 应用程序时，在许多情况下您最终需要某种后端。要么是从数据库检索数据，要么是处理用户提供的数据（例如联系表单）。

我们发现，许多需要后端的用户使用自定义服务器构建他们的 API。在此过程中，他们遇到了许多问题。例如，Next.js 不会编译自定义服务器代码，这意味着您不能使用 `import`/`export` 或 TypeScript。

因此，许多用户最终在自定义服务器上实现了自己的自定义编译流程。虽然这解决了他们的目标，但它容易陷入许多陷阱：例如，如果配置错误，整个应用程序的树摇 (tree shaking) 将被禁用。

这引出了一个问题：如果我们把 Next.js 为构建 API 后端提供的开发者体验应用到后端构建上，怎么样？

今天，我们很兴奋地介绍 API 路由，这是 Next.js 提供的为构建后端服务的顶级开发者体验。

要开始使用 API 路由，请在 `pages/` 目录内创建一个名为 `api/` 的目录。

此目录中的任何文件都将自动映射到 `/api/<your route>`，与其他页面文件映射到路由的方式相同。

例如，`pages/api/contact.js` 将映射到 `/api/contact`。

> **注意：** API 路由也支持 [动态路由 (Dynamic Routes)](#dynamic-route-segments)！

`pages/api/` 目录内的所有文件均导出请求处理函数，而不是 React 组件：

```
export default function handle(req, res) {
  res.end('Hello World');
}
```

* `req` 指向 [NextApiRequest](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L143-L158), 它扩展了 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
* `res` 指向 [NextApiResponse](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L168-L178), 它扩展了 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)

一般来说，API 端点会接收一些传入数据，例如查询字符串、请求正文或 Cookie，并响应其他数据。

在调查是否为 Next.js 添加 API 路由支持时，我们注意到在许多情况下，用户没有直接使用 Node.js 请求和响应对象。相反，他们使用了像 [Express](https://expressjs.com) 这样的服务器库提供的抽象。

这样做的原因是，在许多情况下，传入数据是某种形式的文本，需要先解析才能使用。因此，这些特定的服务器库有助于消除手动解析数据的负担，通常通过中间件来实现。最常用的提供查询字符串、正文和 Cookie 解析，但是启动时仍需要一些设置。

Next.js 中的 API 路由将默认提供这些中间件，以便您可以立即有效地创建 API 端点：

```
export default function handle(req, res) {
  console.log(req.body); // The request body
  console.log(req.query); // The url querystring
  console.log(req.cookies); // The passed cookies
  res.end('Hello World');
}
```

除了使用传入数据，您的 API 端点通常也会返回数据。通常此响应将是 JSON。Next.js 默认提供 `res.json()` 以使发送数据更容易：

```
export default function handle(req, res) {
  res.json({ title: 'Hello World' });
}
```

在开发期间对 API 端点进行更改时，代码会自动重新加载，因此无需重新启动服务器。

## 生产优化 [](#production-optimizations)
-----------------------------------------------------

#### 视口内的 <Link> 预加载 [](#prefetching-in-viewport-links)

Next.js 9 将在 `<Link>` 组件出现在视口内时自动预加载它们。

此功能通过加快新页面的导航速度来提高您应用程序的响应能力。

Next.js 使用 [Intersection Observer](https://developer.mozilla.org/docs/Web/API/Intersection_Observer_API) 来 [预加载必要的资源](https://www.w3.org/TR/resource-hints/#prefetch)。

这些请求优先级较低，并让位于 `fetch()` 或 XHR 请求。如果用户启用了数据节省模式，Next.js 将避免自动预加载。

对于访问频率较低的页面，您可以通过将 `prefetch` 属性设置为 `false` 来选择不使用此功能：

```
<Link href="/terms" prefetch={false}>
  <a>Terms of Service</a>
</Link>
```

#### 默认优化 AMP [](#optimized-amp-by-default)

Next.js 9 现在默认生成优化的 AMP，适用于 AMP 优先和混合 AMP 页面。

虽然 AMP 页面是可选加入的，但 Next.js 会自动优化其输出。这些优化可使渲染速度高达 [50% 更快](https://blog.amp.dev/2018/10/08/how-to-make-amp-even-faster/)！

这一改变之所以成为可能，归功于 [Sebastian Benz](https://twitter.com/sebabenz) 在 [AMP Optimizer](https://github.com/ampproject/amp-toolbox/tree/master/packages/optimizer) 上所做的非凡工作。

#### typeof window 分支的死代码消除 [](#dead-code-elimination-for-typeof-window-branches)

Next.js 9 在服务器和客户端构建期间将 `typeof window` 替换为其适当的值（`undefined` 或 `object`）。此更改允许 Next.js 自动从您的生产构建应用程序中删除死代码。

如果用户在 `getInitialProps` 或应用程序的其他部分有仅服务器端的代码，他们的客户端 bundle 大小应该会减少。

#### 编译指示器 [](#compiling-indicator)

在 9 之前的版本中，知道热代码替换即将发生的唯一方法（以及 Next.js 编译器工具链正在进行工作）是查看开发者控制台。

然而，很多时候人们看的是最终渲染结果，这使得很难知道 Next.js 是否仍在进行编译工作。例如，您可能会更改页面上微妙的样式，您可能无法立即知道它们是否已更新。

为此，我们创建了一个 [RFC / "入门级问题"](https://github.com/vercel/next.js/issues/4626)，以讨论关于指示工作进行状态的潜在解决方案。

我们对 RFC 收到了许多设计师和工程师的反馈，例如他们偏好什么以及指示器设计的方向。

[Rafael Almeida](https://github.com/rafaelalmeidatk) 利用这个机会与我们的团队合作，并实现了一个全新的指示器，现默认可在 Next.js 9 中使用。

只要 Next.js 正在进行编译工作，您就会在页面右下角看到一个小三角形出现！

Next.js 编译指示器

#### 控制台输出 [](#console-output)

传统上，在开发期间进行更改时，Next.js 会显示一个正在编译的指示器状态，加载条会填满，并且您进行更改时会不断清除屏幕。

此行为会导致一些问题。最显著的是它会清除您的应用程序代码的控制台输出，例如当您向组件添加 `console.log` 时。此外，在使用将日志输出拼接在一起的外部工具（如 [Vercel CLI](https://vercel.com/cli) 或 `docker-compose`）时也会如此。

从 Next.js 9 开始，日志输出跳动更少，不再清除屏幕。这提供了更好的整体体验，因为您的终端窗口将有更多相关信息，闪烁更少，同时 Next.js 将与您可能已经使用的工具更好地集成。

## 每页配置对象 [](#per-page-configuration-object)
---------------------------------------------------------------

}
```

export const config = { amp: 'hybrid' };
 
export default function AboutPage(props) {
  const isAmp = useAmp();
  return <h3>My About Page!{isAmp ? <> Powered by AMP!</> : ''}</h3>;
}
```

`withAmp` 高阶组件已被移除，以采用此新配置。

我们提供了一个 [codemod](https://github.com/vercel/next-codemod#withamp-to-config)，可自动将 `withAmp` 用法转换为新的配置对象。您可以在 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md) 中阅读更多相关内容。

## 代码库改进 [](#codebase-improvements)
-----------------------------------------------

我们最近对工具进行了一些更改，以便在贡献代码库时提供更好的体验，并确保随着代码库的增长保持稳定。

正如您在 TypeScript 部分所阅读的，Next.js 核心现在是用 TypeScript 编写的，类型是自动生成的，供 Next.js 应用程序使用。除了这对于使用 Next.js 构建的应用程序有用之外，它在处理核心代码库时也很有用。因为这会让您自动获得类型错误和自动补全。

Next.js 已经拥有一个相当大的集成测试套件，其中包括 50 多个 Next.js 应用程序，其中包含针对这些应用程序运行的测试。这些测试确保当新版本发布时升级是平滑的，因为以前可用的功能都针对相同的测试套件进行了测试。

我们的大多数测试都是集成测试，因为在许多情况下，它们模拟“真实”开发者在开发中使用 Next.js。例如，我们有测试可以模拟对 Next.js 应用程序进行更改，以查看热模块替换是否有效。

我们的集成测试主要基于 Selenium webdriver，我们与 chromedriver 结合使用，在无头 Chrome 中测试。然而，随着时间的推移，其他浏览器中出现了一些问题，特别是像 Internet Explorer 11 这样的旧浏览器。

因为我们使用了 Selenium，我们可以在多个浏览器上自动运行测试。

目前，我们已在 Chrome、Firefox、Safari 和 Internet Explorer 11 上运行测试套件。

### Google Chrome 合作 [](#google-chrome-collaboration)

Google Chrome 团队一直是通过贡献 RFC 和 Pull Request 来改进 Next.js 的。

此次协作的目标是大幅的性能提升，重点在于 bundle 大小、启动和水合时间。

例如，这些更改将改善小型网站的使用体验，同时也将改善像 [Hulu](https://hulu.com)、[Twitch](https://m.twitch.tv) 和 [Deliveroo](https://deliveroo.co.uk/) 这样大规模应用程序的体验。

#### Module / Nomodule[](#module--nomodule)

首先关注的是向支持现代 JavaScript 的浏览器交付现代 JavaScript。

例如，目前 Next.js 必须为 [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 语法提供多填充 (polyfills)，因为代码可能在不支持 [`async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 的浏览器中执行，从而导致错误。

![Next.js Module/Nomodule 协作 RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/first-collab-rfc.png)

Next.js Module/Nomodule 协作 RFC

为了避免破坏旧浏览器，同时向支持它的浏览器发送现代 JavaScript，Next.js 将使用 [module/nomodule 模式](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/)。module/nomodule 模式提供了一种可靠的机制，以便向现代浏览器提供现代 JavaScript，同时允许旧浏览器回退到多填充 (polyfilled) 的 ES5。

Next.js 中 module/nomodule 的 RFC 可以[在此处找到](https://github.com/vercel/next.js/issues/7563)。

#### 改进的 Bundle 分割 [](#improved-bundle-splitting)

Next.js 中当前的 bundle 分割策略围绕一种基于比率的启发式方法，将模块包含在单个 "commons" 块中。由于粒度非常低，只有一个 bundle，代码要么是不必要地下载了（因为 commons chunk 可能包含了特定路由实际上不需要的代码），要么是在多个页面 bundle 之间重复了代码。

![Next.js 分包协作 RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/chunking-rfc.png)

Next.js 分包协作 RFC

关于改进 bundle 分割的 RFC 可以[在此处找到](https://github.com/vercel/next.js/issues/7631)。
