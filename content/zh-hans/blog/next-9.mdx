在 70 次 canary 版本发布之后，我们很高兴推出 Next.js 9，包含：

*   **[内置零配置 TypeScript 支持](#built-in-zero-config-typescript-support)**：通过自动 TypeScript 支持和集成类型检查，更有信心地构建您的应用。
*   **[基于文件系统的动态路由](#dynamic-route-segments)**：通过文件系统表达复杂的应用路由需求，无需自定义服务器。
*   **[自动静态优化](#automatic-static-optimization)**：默认情况下利用 _服务端渲染 (SSR) 和静态预渲染 (Static Prerendering)_ 创建超快速网站，同时不损害功能。
*   **[API 路由](#api-routes)**：利用热重载和统一构建管道，快速构建后端应用端点。
*   **[更多生产优化](#production-optimizations)**：得益于视口预加载和其他优化，应用比以往更加响应灵活。
*   **[开发者体验改进](#developer-experience-improvements)**：非侵入式、易于使用的改进，帮助您以最佳状态开发。

内置零配置 TypeScript 支持  [ ](#built-in-zero-config-typescript-support)

一年前，[Next.js 6](https://nextjs.org/blog/next-6) 通过名为 `@zeit/next-typescript` 的插件引入了基本的 TypeScript 支持。用户还需要自定义 `.babelrc` 并在 `next.config.js` 中启用它。

配置后，该插件将允许 `.ts` 和 `.tsx` 文件由 Next.js 构建。然而，它并未集成类型检查，Next.js 核心也没有提供类型。这意味着必须在 DefinitelyTyped 中单独维护社区软件包，该软件包可能与发布不同步。

在与许多现有和新用户交谈后，很明显大多数人非常有兴趣使用 TypeScript。他们想要一种更可靠和标准的解决方案，以便轻松地将 TypeScript 集成到他们的现有或新代码库中。

因此，我们着手将 TypeScript 支持集成到 Next.js 核心中，从而改善开发者体验，并在此过程中使其更快。

##### 自动化 设置 [ ](#automated-setup)

在 Next.js 中使用 TypeScript 非常容易：将任何文件、页面或组件从 `.js` 重命名为 `.tsx`。然后，运行 `next dev`！

这将导致 Next.js 检测到您的项目中正在使用 TypeScript。Next.js CLI 将指导您安装 React 和 Node.js 所需的类型。

Next.js 还将创建默认 `tsconfig.json`，其中包含合理的默认值（如果尚不存在）。此文件允许在 [Visual Studio Code](https://code.visualstudio.com/) 等编辑器中进行集成类型检查。

动态路由 段 [ ](#dynamic-route-segments)

动态路由（也称为 URL Slugs 或 美化/简洁 URL）是在 Next.js 发布 2.5 年后的 GitHub 上第一批功能请求之一！

这个问题在 Next.js 2.0 中通过引入自定义服务器 API 以程序方式使用 Next.js 来解决。这允许将 Next.js 用作渲染引擎，启用抽象并将传入 URL 映射到渲染特定页面。

我们与用户进行了沟通并检查了他们的大量应用，发现其中许多都使用了自定义服务器。出现了一个模式：自定义服务器的最主要原因是动态路由。

然而，自定义服务器也有其自身的隐患：路由在服务器级别而不是代理级别处理，它以单体方式部署和扩展，并且容易遇到性能问题。

由于自定义服务器要求整个应用在单一实例中可用，因此通常很难部署到 Serverless 环境中，而 Serverless 环境可以解决这些问题。Serverless 请求在代理层进行路由，并独立进行扩展/执行，以避免性能瓶颈。

此外，我们相信我们可以提供更好的开发者体验！Next.js 的许多魔法始于您创建一个名为 `pages/blog.js` 的文件，突然就有一个页面可以在 `/blog` 访问。

为什么用户需要创建自己的服务器并学习 Next.js 的 API 以支持诸如 `/blog/my-first-post` (`/blog/:id`) 这样的路由？

基于这些反馈和愿景，我们开始研究路由映射解决方案，由用户已经 know 的东西驱动：`pages/` 目录。

##### 创建动态路由页面 [ ](#creating-a-dynamically-routed-page)

Next.js 支持使用基本命名参数创建路由，这是一种由 [`path-to-regexp`](https://www.npmjs.com/package/path-to-regexp)（ powering [Express](https://expressjs.com) 的库）普及的模式。

创建与路由 `/post/:pid` 匹配的页面现在可以通过在 `pages` 目录中创建名为 `pages/post/[pid].js` 的文件来实现！

Next.js 将自动匹配诸如 `/post/1`、`/post/hello-nextjs` 等请求，并渲染在 `pages/post/[pid].js` 中定义的页面。匹配的路径段将作为查询参数传递给您的页面，名称指定在 `[square-brackets]` 之间。

例如：给定以下页面和请求 `/post/hello-nextjs`，`query` 对象将为 `{ pid: 'hello-nextjs' }`：

```
static async getInitialProps({ query }) {
  // pid = 'hello-nextjs'
  const { pid } = query
 
  const postContent = await fetch(
    `https://api.example.com/post/${encodeURIComponent(pid)}`
  ).then(r => r.text())
 
  return { postContent }
}
```

还支持多个动态 URL 路径段！

`[param]` 语法支持用于目录名称和文件名称，这意味着以下示例有效：

```
./pages/blog/[blogId]/comments/[commentId].js
./pages/posts/[pid]/index.js
```

您可以在 [Next.js 文档](https://github.com/vercel/next.js#dynamic-routing) 或 [Next.js Learn 部分](https://nextjs.org/learn/basics/dynamic-routes) 中 [了解更多关于此功能的信息](https://github.com/vercel/next.js#dynamic-routing)。

自动静态优化 [ ](#automatic-static-optimization)

Next.js 在 v3 中添加了静态网站生成支持，该版本大约在两年前发布。当时，这是被请求添加到 Next.js 的最多的功能。

这确实有理由：毫不否认静态网站是 _快_！它们不需要服务器端计算，并且可以即时从 CDN 位置流式传输到最终用户。

然而，服务器端渲染或静态生成应用之间的选择是二元化的，您要么选择服务器端渲染，要么选择静态生成。没有中间地带。

实际上，应用可以有不同的需求。这些需求需要不同的渲染策略和权衡。

例如，主页和营销页面通常包含静态内容，是静态优化的绝佳候选者。

另一方面，产品仪表板可能受益于使用服务端渲染，其中数据会频繁更新。

我们开始探索如何既能给用户两全其美，又能 **默认快速**。我们如何能给用户提供静态营销页面和动态服务器渲染页面？

从 Next.js 9 开始，用户不再需要在完全服务器渲染或静态导出应用之间做选择。在按页面基础上给您两全其美。

#### 自动部分静态导出 [ ](#automatic-partial-static-export)

引入了启发式算法来自动确定页面是否可以预渲染为静态 HTML。

这一决定是通过页面是否通过 [ getInitialProps ](https://github.com/vercel/next.js#fetching-data-and-component-lifecycle) 使用阻塞数据需求来确定的。

此启发式算法允许 Next.js 生成混合应用，其中包含 **服务端渲染和静态生成页面**。

内置 Next.js 服务器 (`next start`) 和程序化 API (`app.getRequestHandler()`) 都支持这种构建输出 **透明**。无需配置或特殊处理。

静态生成的页面仍然是响应式的：Next.js 将在客户端水合 (Hydration) 您的应用以赋予其完全交互性。

此外，如果页面依赖于 URL 中的查询参数，Next.js 将在水合后更新您的应用。

Next.js 将在开发期间视觉上告诉您页面是否将被静态生成。通过点击可以隐藏这种视觉提示。

API 路由 [ ](#api-routes)

在构建 React 应用的许多情况下，您最终需要某种后端。要么是从数据库检索数据，要么处理由用户提供的数据（例如联系表格）。

我们发现许多需要后端的用户都使用自定义服务器构建 API。这样做时，他们遇到了一些问题。例如，Next.js 不编译自定义服务器代码，这意味着您不能使用 `import` / `export` 或 TypeScript。

因此，许多用户最终在自定义服务器之上实现了自己的自定义编译管道。虽然这解决了他们的目标，但它容易陷入许多陷阱：例如，如果配置不正确，树摇 (Tree Shaking) 将为整个应用禁用。

这就引出了一个问题：如果我们把 Next.js 为构建 API 后端提供的开发者体验带过来怎么样？

今天，我们很高兴地介绍 API 路由，这是 Next.js 为构建后端提供的最佳开发者体验。

To start using API routes you create a directory called `api/` inside the `pages/` directory.

此目录中的任何文件都将自动映射到 `/api/<your route>`，其他页面文件映射到路由的方式相同。

例如，`pages/api/contact.js` 将映射到 `/api/contact`。

> **注意**：API 路由也支持 [动态路由](#dynamic-route-segments)！

`pages/api/` 目录中的所有文件都导出一个请求处理函数而不是 React 组件：

```
export default function handle(req, res) {
  res.end('Hello World');
}
```

*   `req` 指 [NextApiRequest](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L143-L158)，它扩展了 [http.IncomingMessage](https://nodejs.org/api/http.html#http_class_http_incomingmessage)
*   `res` 指 [NextApiResponse](https://github.com/vercel/next.js/blob/v9.0.0/packages/next-server/lib/utils.ts#L168-L178)，它扩展了 [http.ServerResponse](https://nodejs.org/api/http.html#http_class_http_serverresponse)

通常 API 端点接收一些传入数据，例如查询字符串、请求体或 cookie，并响应其他数据。

在调查将 API 路由支持添加到 Next.js 时，我们注意到在許多情况下，用户不直接使用 Node.js 请求和响应对象。相反，他们使用了由 [Express](https://expressjs.com) 等服务器库提供的抽象。

这样做的原因是，在许多情况下，传入数据是必须首先解析才能有用的某种文本。因此，这些特定的服务器库帮助减轻了手动解析数据的负担，最常见的是通过中间件。最常用的中间件提供查询字符串、正文和 cookie 解析，但是它们仍然需要一些设置才能开始。

Next.js 中的 API 路由将默认提供这些中间件，以便您可以立即高效地创建 API 端点：

```
export default function handle(req, res) {
  console.log(req.body); // The request body
  console.log(req.query); // The url querystring
  console.log(req.cookies); // The passed cookies
  res.end('Hello World');
}
```

除了使用传入数据，您的 API 端点通常还返回数据。通常此响应将是 JSON。Next.js 默认提供 `res.json()` 以简化数据传输：

```
export default function handle(req, res) {
  res.json({ title: 'Hello World' });
}
```

在开发期间修改 API 端点时，代码会自动重新加载，因此无需重新启动服务器。

生产优化 [ ](#production-optimizations)

#### 视口内<Link>预加载 [ ](#prefetching-in-viewport-links)

Next.js 9 将在 `<Link>` 组件出现在视口内时自动预加载它们。

此功能通过使跳转到新页面更快来提高应用的响应速度。

Next.js 使用 [Intersection Observer](https://developer.mozilla.org/docs/Web/API/Intersection_Observer_API) 在后台 [ 预加载 资源 ](https://www.w3.org/TR/resource-hints/#prefetch)。

这些请求优先级较低，并让位于 `fetch()` 或 XHR 请求。如果用户启用了数据节省模式，Next.js 将避免自动预加载。

您可以将 `prefetch` 属性设置为 `false` 以选择不为很少访问的页面使用此功能：

```
<Link href="/terms" prefetch={false}>
  <a>Terms of Service</a>
</Link>
```

#### 默认优化 AMP [ ](#optimized-amp-by-default)

Next.js 9 现在默认为 AMP-first 和混合 AMP 页面渲染优化的 AMP。

虽然 AMP 页面是可选的，但 Next.js 会自动优化它们的输出。这些优化可使 [50% 更快](https://blog.amp.dev/2018/10/08/how-to-make-amp-even-faster/) 的渲染速度！

这一变化归功于 [Sebastian Benz](https://twitter.com/sebabenz) 在 [AMP Optimizer](https://github.com/ampproject/amp-toolbox/tree/master/packages/optimizer) 上的非凡工作。

#### `typeof window` 分支的死代码消除 [ ](#dead-code-elimination-for-typeof-window-branches)

Next.js 9 在服务器和客户端构建期间将 `typeof window` 替换为其适当的值 (`undefined` 或 `object`)。此更改允许 Next.js 自动从生产构建的应用中删除死代码。

如果用户在 `getInitialProps` 或其他应用部分中有仅服务器端的代码，他们应该能看到客户端 bundle 大小减小。

#### 编译指示器 [ ](#compiling-indicator)

在 9 之前的版本中，知道将要进行热代码替换（且 Next.js 编译器工具链正在工作）的唯一方法是查看开发者控制台。

但是，很多时候人们看到的是结果的渲染，这使得很难知道 Next.js 是否仍在进行编译工作。例如，您可能正在对页面上的样式进行微妙更改，而您不会立即知道它们是否已更新。

因此，我们创建了一个 [RFC / “好第一问题”](https://github.com/vercel/next.js/issues/4626) 来讨论解决指示工作进行中问题的潜在方案。

我们在 RFC 上收到了许多设计师和工程师的反馈，例如他们更喜欢什么以及指示器设计的潜在方向。

[Rafael Almeida](https://github.com/rafaelalmeidatk) 抓住这次机会与我们的团队合作，实现了一个全新的指示器，现在可在 Next.js 9 中默认使用。

Whenever Next.js 进行编译工作时，您将在页面右下角看到一个小三角形出现！

#### 控制台输出 [ ](#console-output)

传统上，在开发期间进行更改时，Next.js 会显示编译指示器状态，加载状态条会填满，并且在您进行更改时会不断清除屏幕。

这种行为会导致一些问题。最值得注意的是，它会清除来自您的应用代码的控制台输出，例如当您向组件添加 `console.log` 时。但是，也会清除使用类似 [Vercel CLI](https://vercel.com/cli) 或 `docker-compose` 等外部工具拼接日志输出时的内容。

从 Next.js 9 开始，日志输出跳动更少且不再清除屏幕。这使得您的终端窗口具有更多有用信息且闪烁更少，同时 Next.js 将与您可能已经使用的工具更好地集成。

按页面配置对象 [ ](#per-page-configuration-object)

每个页面现在都可以导出一个配置对象。最初，此配置允许您选择加入 [AMP](https://nextjs.org/blog/next-8-1)，但在未来您将能够配置更多的页面特定选项。

[基于文件系统的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在最初的文章中，我们展示了从单个 React 组件创建路由的示例：

```
export const config = { amp: true };
 
export default function AboutPage(props) {
  return <h3>My AMP About Page!</h3>;
}
```

要选择混合 AMP 渲染，您可以使用值 `'hybrid'`：

[基于文件系统的路由](/docs/app/building-your-application/routing) 一直是 Next.js 的核心功能。在最初的文章中，我们展示了从单个 React 组件创建路由的示例：

```
import { useAmp } from 'next/amp';

```

`withAmp` 高阶组件已被删除，改用此新配置。

我们提供了一个 [codemod](https://github.com/vercel/next-codemod#withamp-to-config)，可以将 `withAmp` 的使用自动转换为新的配置对象。您可以在 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md) 中了解更多关于此信息。

代码库改进 [ ](#codebase-improvements)

我们最近对我们的工具进行了一些更改，以便在贡献代码库时提供更好的体验，并确保随着代码库增长稳定性。

正如您在 TypeScript 部分所读到的，Next.js 核心现在使用 TypeScript 编写，并且类型会自动生成供 Next.js 应用使用。除了这对使用 Next.js 构建的应用有用外，在核心代码库工作时也很有用。因为您会自动获得类型错误和自动补全。

Next.js 已经有一个相当大的集成测试套件，包含 50+ 个 Next.js 应用，并对其运行测试。这些测试确保当发布新版本时，升级是平滑的，因为之前可用的功能都针对相同的测试套件进行了测试。

我们的大多数测试都是集成测试，因为它们在很多情况下复制了“真实”开发者在开发中如何使用 Next.js。例如，我们有测试来复制对 Next.js 应用进行更改，以查看热模块替换是否有效。

我们的集成测试主要基于 Selenium webdriver，我们将其与 chromedriver 结合起来在 headless Chrome 中测试。但是随着时间的推移，其他浏览器（尤其是像 Internet Explorer 11 这样的旧浏览器）会出现某些问题。

由于我们使用了 Selenium，我们能够在多个浏览器上自动运行我们的测试。

目前，我们在 Chrome、Firefox、Safari 和 Internet Explorer 11 上运行测试套件。

### Google Chrome 合作 [](#google-chrome-collaboration)

Google Chrome 团队一直致力于通过贡献 RFC 和 pull-request 来改进 Next.js。

此次合作的目标是大规模性能改进，重点是 bundle 大小、启动和水合 (Hydration) 时间。

例如，这些更改将改善小型网站的体验，也会改善像 [Hulu](https://hulu.com)、[Twitch](https://m.twitch.tv) 和 [Deliveroo](https://deliveroo.co.uk/) 这样的大型应用的体验。

#### Module / Nomodule [ ](#module--nomodule)

首要关注领域是将现代 JavaScript 发送给支持现代 JavaScript 的浏览器。

例如，目前 Next.js 必须为 [ `async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 语法提供 polyfills，因为代码可能在不支持 [ `async`/`await`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/async_function) 的浏览器中执行，这将导致问题。

![Next.js Module/Nomodule 合作 RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/first-collab-rfc.png)

Next.js Module/Nomodule 合作 RFC

为避免破坏旧版浏览器，同时向支持现代 JavaScript 的浏览器发送现代 JavaScript，Next.js 将利用 [module/nomodule 模式](https://philipwalton.com/articles/deploying-es2015-code-in-production-today/)。module/nomodule 模式为向现代浏览器提供现代 JavaScript 提供了可靠的机制，同时允许旧版浏览器回退到 polyfilled ES5。

Next.js 中 module/nomodule 的 RFC 可以 [在此处找到](https://github.com/vercel/next.js/issues/7563)。

#### 改进的包拆分 [ ](#improved-bundle-splitting)

Next.js 中当前的包拆分策略围绕基于比重的启发式算法，用于将模块包含在单个“commons”包中。由于只有一个包，粒度非常有限，代码要么不必要地下载（因为 commons 包可能包含特定路由不需要的代码），要么代码在多个页面包之间重复。

![Next.js Chunking 合作 RFC](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-9/chunking-rfc.png)

Next.js Chunking 合作 RFC

改进的包拆分的 RFC 可以 [在此处找到](https://github.com/vercel/next.js/issues/7631)。
