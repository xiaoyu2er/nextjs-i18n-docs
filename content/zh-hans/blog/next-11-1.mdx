随着 Next.js 11.1 的推出，我们正在改进整个堆栈的构建性能，包括：

*   [**安全补丁**](#security-patch): 防止潜在开放重定向的重要更新。
*   [**ES 模块支持**](#es-modules-support): 使用实验性标志立即启用。
*   [**基于 Rust 的工具**](#adopting-rust-based-swc): 集成 SWC 以替代 JS 工具（Babel 和 Terser）。
*   [**更快的数据获取**](#builds--data-fetching): 预渲染时使用 HTTP `keep-alive`，数据获取速度提升 2 倍。
*   [**更快的 Source 映射**](#source-maps): 使用 Source 映射时构建速度提升 70%，内存使用减少 67%。
*   [**ESLint 集成改进**](#eslint-improvements): 更易访问的默认值以及拼写错误检查。
*   [**`next/image` 改进**](#nextimage-improvements): 可选 Sharp 使用，更好的 `next export` 支持。

今天通过运行 `npm i next@latest` 进行更新。

安全补丁[](#security-patch)
---------------------------------

Next.js 团队与安全研究人员和审计员合作，防止漏洞。我们要感谢 Robinhood 的 Gabriel Benmergui 进行的调查以及发现 `pages/_error.js` 的开放重定向问题，以及随后的负责任的披露。

报告的漏洞并未直接损害用户，但它可能通过从受信任域名重定向到攻击者域名来允许网络钓鱼攻击。我们在 Next.js 11.1 中推出了补丁，防止此开放重定向的发生，以及 [安全回归测试](https://github.com/vercel/next.js/blob/canary/test/integration/repeated-slashes/test/index.test.js)。

有关更多详细信息，请阅读 [公开 CVE](https://github.com/vercel/next.js/security/advisories/GHSA-vxf5-wxwp-m7g9)。我们建议升级到最新版本的 Next.js 以提高应用程序的整体安全性。对于未来的负责任披露，请通过电子邮件联系我们：`security@vercel.com`。

ES 模块支持[](#es-modules-support)
-----------------------------------------

我们正在为 Next.js 开发广泛的 [ES 模块](https://nodejs.org/docs/latest/api/esm.html) 支持，包括作为输入模块和输出目标。从 Next.js 11.1 开始，您现在可以使用实验标志使用 ES 模块导入 npm 包（例如，它们在 `package.json` 中的 `"type": "module"`）。

next.config.js

```
module.exports = {
  // Prefer loading of ES Modules over CommonJS
  experimental: { esmExternals: true },
};
```

ES 模块支持包含向后兼容性以支持之前 [CommonJS](https://nodejs.org/docs/latest/api/modules.html) 的导入行为。在 Next.js 12 中，`esmExternals: true` 将成为默认值。我们建议尝试此新选项，[并在 GitHub Discussions 上留下反馈](https://github.com/vercel/next.js/discussions/27876) 如果您有改进建议。

采用基于 Rust 的 SWC[](#adopting-rust-based-swc)
---------------------------------------------------

我们正在集成 [SWC](https://swc.rs/)，这是一个用 Rust 编写的超快 JavaScript 和 TypeScript 编译器，它将取代 Next.js 中使用的所有两个工具链：用于单个文件的 Babel 和用于输出包压缩的 Terser。

作为用 SWC 替换 Babel 的一部分，我们将 Next.js 使用的所有自定义代码转换移植到 SWC transforms 中使用 Rust 编写，以最大化性能。例如，`getStaticProps`、`getStaticPaths` 和 `getServerSideProps` 内部未使用代码的摇树优化。

作为替换 Terser 的一部分，我们正在确保 SWC 压缩器的输出与 Terser 类似，同时大幅提高压缩性能和并行化。

在早期测试中，之前使用 Babel 进行的代码转换从 **~500ms 降至 ~10ms**，Terser 的代码压缩从 **~250ms 降至 ~30ms**，使用 SWC。总体而言，这带来了 **构建速度提升 2 倍**。

我们兴奋地宣布 [DongYoon Kang](https://twitter.com/kdy1dev)，[SWC](https://swc.rs/) 的创作者，以及 [Maia Teegarden](https://twitter.com/padmaia)，[Parcel](https://parceljs.org/) 的贡献者，已加入 Vercel 的 Next.js 团队，致力于改进 `next dev` 和 `next build` 性能。我们将在下一次发布中分享更多关于我们采用 SWC 的结果，届时它将成为稳定版本。

性能提升[](#improved-performance)
---------------------------------------------

### 构建与数据获取[](#builds--data-fetching)

使用 `next build` 并发起大量 HTTP 请求时，我们的平均性能**提升了约 2 倍**。例如，如果您使用 `getStaticProps` 和 `getStaticPaths` 从无头 CMS 获取内容，您应该注意到构建速度明显加快。

Next.js 会自动 polyfill [node-fetch](/docs/architecture/supported-browsers#polyfills)，现在默认启用 [HTTP Keep-Alive](https://en.wikipedia.org/wiki/HTTP_persistent_connection)。根据 [外部基准测试](https://github.com/Ethan-Arrowood/undici-fetch/blob/main/benchmarks.md#fetch)，这应使预渲染**快约 2 倍**。

要禁用某些 `fetch()` 调用的 HTTP Keep-Alive，您可以添加 agent 选项：

```
import { Agent } from 'https';
const url = '<https://example.com>';
const agent = new Agent({ keepAlive: false });
fetch(url, { agent });
```

要全局覆盖所有 `fetch()` 调用，您可以使用 `next.config.js`：

next.config.js

```
module.exports = {
  httpAgentOptions: {
    keepAlive: false,
  },
};
```

[`next/image` 改进](#nextimage-improvements)
----------------------------------------------------

我们正在收集社区对 `next/image` 和内置的 [图像优化](/docs/pages/building-your-application/optimizing/images) 的反馈，并兴奋地分享多项提升性能、开发体验和用户体验的改进。

### 图像优化[](#image-optimization)

默认情况下，Next.js 使用 WebAssembly 进行图像优化，这通过使用显著更小的文件且没有安装后步骤来抵消 Next.js 包的安装时间。在 Next.js 11.1 中，您可以选择安装 `sharp`，它可以优化未缓存图像生成时间，但安装速度会更慢。

基于 WebAssembly 的图像优化器已更新，以支持 Node.js 16 的 ARM 芯片，如 Apple M1。

内置图像优化器现在会根据响应体的内容自动检测外部图像的内容类型。这允许 Next.js 优化托管在 AWS S3 上的图像，即使响应头是 `Content-Type: application/octet-stream` 。

### 开发中的模糊占位符懒惰生成[](#lazy-generation-of-blur-up-placeholders-in-development)

在 `next dev` 期间，带有 `placeholder="blur"` 的 [静态图像导入](/docs/pages/building-your-application/optimizing/images#image-imports) 现在会自动懒惰生成，改善了具有许多静态图像导入的应用程序的开发服务器启动时间：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Image from 'next/image';
import author from '../public/me.png';
 
export default function Home() {
  return (
    // The placeholder for this image is lazy-generated during development
    <Image src={author} alt="Picture of the author" placeholder="blur" />
  );
}
```

### 其他图像改进[](#other-image-improvements)

*   **以前加载过的图像不再被懒惰加载**：如果图像之前已在页面上加载过（无论是通过客户端导航还是在页面中的其他位置加载它），Next.js 现在将自动跳过懒惰加载，以避免在显示图像前快速闪烁。
*   **`next export` 的自定义图像加载器支持**：`next/image` 现在支持配合 [第三方图像优化服务](/docs/pages/building-your-application/optimizing/images#loader) 使用 `next export`。您可以在 `next.config.js` 中配置 `images.loader: "custom"`，以便向 `next/image` 提供 [自定义 `loader` prop](/docs/pages/api-reference/components/image#loader)。
*   **图像完成加载时的新事件**：基于用户反馈，我们为 `next/image` 添加了一个新属性 [`onLoadingComplete`](/docs/pages/api-reference/components/image#onloadingcomplete)。这允许注册一个当图像完全加载后调用的回调函数。
*   **默认图像缓存 TTL（生存时间）的配置文件**：您现在可以在 `next.config.js` 中配置 [`images.minimumCacheTTL`](/docs/pages/building-your-application/optimizing/images#minimumcachettl) 来更改优化图像的默认缓存 TTL。如有可能，我们建议使用 [静态图像 `import`s](/docs/pages/building-your-application/optimizing/images#image-imports)，因为它们会自动使用最大 TTL，因为图像内容哈希在 URL 中。
