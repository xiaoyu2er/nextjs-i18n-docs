Next.js 16.2 包含性能改进、更好的调试、针对智能体 (Agents) 的改进，以及超过 200 项 Turbopack 修复和改进。

*   [**更快的 URL 就绪时间**](#faster-time-to-url-in-development): `next dev` 启动时间~400% 更快
*   [**更快的渲染**](#faster-rendering): 渲染速度快~50%
*   [**新的默认错误页面**](#new-default-error-page): 重新设计的内置 500 页面
*   [**服务端函数日志记录**](#server-function-logging): Server Function 执行的开发终端日志
*   [**水合差异指示器**](#hydration-diff-indicator): 错误覆盖层中清晰的服务器/客户端差异
*   [**`next start` 的 `--inspect`**](#--inspect-for-next-start): 将 Node.js 调试器附加到您的生产服务器

我们还针对此版本中两个主要领域发布了专门的深度剖析文章：

*   [**Turbopack**](/blog/next-16-2-turbopack): 构建更快，SRI 支持，`postcss.config.ts`，Tree Shaking 改进，Server Fast Refresh，以及 200+ 错误修复
*   [**AI 改进**](/blog/next-16-2-ai): create-next-app 中的 `AGENTS.md`，浏览器日志转发，以及 `next-browser`（实验性）

开发中更快的 URL 就绪时间 [](#faster-time-to-url-in-development)
----------------------------------------------------------------------

我们显著改善了 `next dev` 期间 `localhost:3000` 就绪所需的时间。在同一台机器和项目上，相比 Next.js 16.1，默认应用的启动速度提高了~87%。

更快的渲染 [](#faster-rendering)
-------------------------------------

我们向 React 提交了一项变更 [贡献代码](https://github.com/facebook/react/pull/35776)，使服务端组件负载反序列化速度提高高达 350%。之前的实现使用 `JSON.parse` 再生器回调，这会为解析后的 JSON 中的每个键值对跨越 V8 中的 C++/JavaScript 边界。即使是简单的空操作再生器也会使 `JSON.parse` 比没有使用时慢大约 4 倍。

新方法采用两步过程：普通的 `JSON.parse()` 后跟纯 JavaScript 中的递归遍历。这消除了边界跨越开销，并添加了优化，例如跳过不需要转换的普通字符串。

在实际的 Next.js 应用中，这转化为 25% 到 60% 更快的 HTML 渲染速度，具体取决于 RSC 负载大小。

`next start` 的 `--inspect` [](#--inspect-for-next-start)
---------------------------------------------------------

Next.js 16.1 引入了 `next dev --inspect`，用于在开发期间附加 Node.js 调试器。在 16.2 中，此功能扩展到 `next start`，允许您将 Node.js 调试器附加到生产服务器。

终端

```
next start --inspect
```

这对调试问题或分析 CPU 和内存使用情况非常有用。

有关如何使用 Node.js 调试器的更多信息，请参阅 [Chrome文档](https://developer.chrome.com/docs/devtools/performance/nodejs)。

`transitionTypes` 属性用于 `next/link` [](#transitiontypes-prop-for-nextlink)
----------------------------------------------------------------------------

<Link> 组件现在接受 [`transitionTypes`](https://react.dev/reference/react/addTransitionType) 属性——一个字符串数组，指定导航时应用的视图转换 (View Transitions) 类型。每种类型在导航转换 (Transition) 期间传递给 React.addTransitionType，允许您根据导航方向或上下文触发不同的动画。

```
<Link href="/about" transitionTypes={['slide']}>
  About
</Link>
```

这仅在 App Router 中受支持，因为 Pages Router 不使用 React 转换 (Transitions) 进行导航。Pages Router 链接上的 `transitionTypes` 会被静默忽略，因此共享链接组件可在两个路由器中共同工作。

要了解有关视图转换 (View Transitions) 的更多信息，请参阅 [文档](/docs/app/api-reference/config/next-config-js/viewTransition)。

更快的 ImageResponse [](#faster-imageresponse)
-----------------------------------------------

ImageResponse 已更新，包含重大改进：

*   **2 倍速** `ImageResponse` 对于基本图像，对于复杂图像高达 **20 倍速**
*   改进了 CSS 和 SVG 覆盖范围，包括支持行内 CSS 变量、text-indent、text-decoration-skip-ink、box-sizing、display: contents、position: static，以及 `gap` 的百分比值

了解有关 [ImageResponse](/docs/app/api-reference/functions/image-response) 的更多信息。

多种图标格式 [](#multiple-icon-formats)
-----------------------------------------------

多个具有相同基本名称但扩展名不同的图标文件现在在您的应用目录 (例如 `icon.png` 和 `icon.svg`) 中自动处理。这对浏览器回退支持非常有用。现代浏览器可以使用 SVG 图标，而旧版浏览器回退到 PNG。这两种格式都渲染为单独的 `<link>` 标签。

### unstable_catchError() [](#unstable_catcherror)

通过 [`error.js`](/docs/app/api-reference/file-conventions/error) 文件约定扩展错误恢复体验，[`unstable_catchError()`](/docs/app/api-reference/functions/catchError) 提供了组件级别的错误边界的更细粒度控制。通过使用 `unstable_catchError()` 创建自定义错误边界，您可以将其放在组件树的任何位置。

unstable_catchError() 创建的错误边界接收从调用站点传递的 `props` 和作为第二个参数的 `ErrorInfo` 值，其形状与传递给 `error.js` 文件约定的 `props` 相同。

与 `error.js` 一样，`unstable_catchError()` 只能从 [客户端组件](/docs/app/getting-started/server-and-client-components) 中调用。

app/custom-error-boundary.tsx

```
'use client';
 
import { unstable_catchError, type ErrorInfo } from 'next/error';
 
function CustomErrorBoundary(
  props: { title: string },
  { error, unstable_retry }: ErrorInfo,
) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
 
export default unstable_catchError(CustomErrorBoundary);
```

app/page.tsx

```
import CustomErrorBoundary from './custom-error-boundary';
 
export default function Page() {
  return (
    <CustomErrorBoundary title="Oops! Something went wrong!">
      <Component />
    </CustomErrorBoundary>
  );
}
```

### error.tsx 中的 unstable_retry() [](#unstable_retry-in-errortsx)

在 error.tsx 组件中现在有了一个新的 [`unstable_retry()`](/docs/app/api-reference/file-conventions/error#unstable_retry) 属性。之前，`reset()` 属性仅清除错误状态并重渲染子项，这适用于临时渲染错误，但对于源自数据获取或 RSC 阶段的错误没有帮助。

unstable_retry() 在 startTransition() 内调用 `router.refresh()` 和 `reset()`，提供内置的重试逻辑，重新获取数据并重渲染片段。预计对于大多数错误恢复场景，这比 reset() 更受青睐。

app/error.tsx

```
'use client';
 
import type { ErrorInfo } from 'next/error';
 
export default function Error({ error, unstable_retry }: ErrorInfo) {
  return (
    <div>
      <h2>Something went wrong</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
```

### experimental.prefetchInlining [](#experimentalprefetchinlining)

Next.js 16 引入了每片段预获取，其中客户端为路由树中的每个片段发出单独的请求。这提高了缓存效率，因为同级 (Sibling) 路由之间的共享布局只获取一次并重用，但增加了请求量。

新的 experimental.prefetchInlining 选项将路由的所有片段数据打包到单个响应中，将预获取请求数减少到每链接一个。代价是共享布局数据在行内响应中重复，而不是被缓存和重用。

next.config.ts

```
const nextConfig = {
  experimental: {
    prefetchInlining: true,
  },
};
 
export default nextConfig;
```

这是迈向基于大小的启发式方法的一步，即小片段自动行内，大片段保持单独。

### experimental.cachedNavigations [](#experimentalcachednavigations)

此标志独立控制缓存导航行为，缓存来自导航和初始 HTML 加载的静态和动态服务端组件数据，以便重复访问可以即时提供访问。需要启用 cacheComponents。

### experimental.appNewScrollHandler [](#experimentalappnewscrollhandler)

App Router 的重构滚动和焦点管理系统使用 React Fragment ref。新处理器改进了导航后如何管理焦点。它不再聚焦片段深处第一个可聚焦的子孙元素 (可能会跳过内容)，而是让活动元素失去焦点，匹配浏览器导航的工作原理。

next.config.ts

```
const nextConfig = {
  experimental: {
    appNewScrollHandler: true,
  },
};
 
export default nextConfig;
```
