* [**更快的时间到 URL**](#faster-time-to-url-in-development): 开发环境下的 `next dev` 启动速度快约 400%
* [**更快的渲染**](#faster-rendering): 渲染速度约快 50%
* [**新的默认错误页面**](#new-default-error-page): 重新设计内置的 500 页面
* [**服务器函数日志记录**](#server-function-logging): Server Function 执行的终端日志
* [**水合差异指示器**](#hydration-diff-indicator): 错误覆盖层中清晰显示服务端/客户端差异
* [**`next start` 的 `--inspect`**](#--inspect-for-next-start): 将 Node.js 调试器附加到您的生产服务器

我们还发布了两篇关于本次发布两大主要领域的深度文章：

* [**Turbopack**](/blog/next-16-2-turbopack): 更快的构建、SRI 支持、`postcss.config.ts`、摇树优化、Server Fast Refresh 以及 200 多项 bug 修复

开发环境中的更快时间到 URL (#faster-time-to-url-in-development)
-----------------------------------------------------------------------

我们显著改善了 `next dev` 期间 `localhost:3000` 准备好之前的等待时间。在相同的机器和项目上，与默认应用的 Next.js 16.1 相比，启动速度快了约 87%。

更快的渲染 (#faster-rendering)
-------------------------------------

我们向 React [贡献了一个更改](https://github.com/facebook/react/pull/35776)，使服务端组件 负载 反序列化速度快了高达 350%。之前的实现使用了 `JSON.parse` reviver 回调，这会在 V8 中为解析 JSON 中的每个键值对跨 C++/JavaScript 边界。即使是一个微不足道的无操作 reviver 也使 `JSON.parse` 在没有它的情况下慢了大约 4 倍。

新方法采用两步流程：首先进行普通 `JSON.parse()`，然后在纯 JavaScript 中进行递归遍历。这消除了跨边界开销，并添加了优化，例如对不需要转换的普通字符串进行短路处理。

在实际的 Next.js 应用中，这使到 HTML 的渲染速度提高了 25% 到 60%，具体取决于 RSC 负载大小。

**`--inspect` 用于 `next start`** (#--inspect-for-next-start)
---------------------------------------------------------

Next.js 16.1 引入了 `next dev --inspect` 用于在开发期间附加 Node.js 调试器。在 16.2 中，这扩展到了 `next start`，允许您将 Node.js 调试器附加到您的生产服务器。

终端

```
next start --inspect
```

这对于调试问题或分析 CPU 和内存使用很有用。

有关如何使用 Node.js 调试器的更多信息，请参阅 [Chrome 文档](https://developer.chrome.com/docs/devtools/performance/nodejs)。

next/link 的 `transitionTypes` 属性 (#transitiontypes-prop-for-nextlink)
----------------------------------------------------------------------------

`<Link>` 组件现在接受 [`transitionTypes`](https://react.dev/reference/react/addTransitionType) 属性——一个字符串数组，指定导航时应用哪些类型的视图转换 (View Transitions)。每种类型在导航转换期间传递给 `React.addTransitionType`，允许您根据导航方向或上下文触发不同的动画。 ``` <Link href="/about" transitionTypes={['slide']}> About </Link> ```

```
<Link href="/about" transitionTypes={['slide']}>
  About
</Link>
```

这仅在 App Router 中受支持，因为 Pages Router 不使用 React 转换进行导航。Pages Router 链接上的 `transitionTypes` 会被静默忽略，因此共享链接组件可以在两个路由器中工作。

想了解更多信息，请查看 [文档](/docs/app/api-reference/config/next-config-js/viewTransition) 中的视图转换 (View Transitions) 相关内容。

更快的 ImageResponse (#faster-imageresponse)
-----------------------------------------------

`ImageResponse` 已更新，带来了显著改进：

* **快 2 倍** 的 `ImageResponse` 用于基本图像，对于复杂图像快达 **20 倍**
* 改进了 CSS 和 SVG 覆盖，包括支持内联 CSS 变量、`text-indent`、`text-decoration-skip-ink`、`box-sizing`、`display: contents`、`position: static` 以及 `gap` 的百分比值
* 默认字体从 Noto Sans 更改为 Geist Sans

了解有关 [ImageResponse](/docs/app/api-reference/functions/image-response) 的更多信息。

### `unstable_catchError()` (#unstable_catcherror)

通过 `error.js` 文件约定扩展错误恢复体验，`unstable_catchError()` 提供了更细粒度的组件级别错误边界控制。通过使用 `unstable_catchError()` 创建自定义错误边界，您可以将其放置在组件树中的任意位置。

`unstable_catchError()` 创建的错误边界接收从调用点传递的 props 和第二参数 `ErrorInfo` 值，这与传递给 `error.js` 文件约定的 props 结构相同。

与 `error.js` 一样，`unstable_catchError()` 只能从 [客户端组件](/docs/app/getting-started/server-and-client-components) 中调用。

app/custom-error-boundary.tsx
```
'use client';
 
import { unstable_catchError, type ErrorInfo } from 'next/error';
 
function CustomErrorBoundary(
  props: { title: string },
  { error, unstable_retry }: ErrorInfo,
) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
 
export default unstable_catchError(CustomErrorBoundary);
```
app/page.tsx
```
import CustomErrorBoundary from './custom-error-boundary';
 
export default function Page() {
  return (
    <CustomErrorBoundary title="Oops! Something went wrong!">
      <Component />
    </CustomErrorBoundary>
  );
}
```

### `error.tsx` 中的 `unstable_retry()` (#unstable_retry-in-errortsx)

现在 `error.tsx` 组件中可用一个新的 `unstable_retry()` 属性。此前，`reset()` 属性仅清除错误状态并重渲染子组件，这对于临时渲染错误有效，但在错误源自数据获取或 RSC 阶段时没有帮助。

`unstable_retry()` 在 `startTransition()` 内调用 `router.refresh()` 和 `reset()`，提供内置的重试逻辑，重新获取数据并重渲染段。对于大多数错误恢复场景，预计这将优于 `reset()`。

app/error.tsx
```
'use client';
 
import type { ErrorInfo } from 'next/error';
 
export default function Error({ error, unstable_retry }: ErrorInfo) {
  return (
    <div>
      <h2>Something went wrong</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
```

```
'use client';
 
import type { ErrorInfo } from 'next/error';
 
export default function Error({ error, unstable_retry }: ErrorInfo) {
  return (

      <h2>Something went wrong</h2>

    </div>
```

### `experimental.prefetchInlining` (#experimentalprefetchinlining)

Next.js 16 引入了每段预取 (per-segment prefetching)，其中客户端为路由树中的每个段发出单独请求。这提高了缓存效率，因为兄弟路由间共享的布局被获取一次并重复使用，但增加了请求量。

新的 `experimental.prefetchInlining` 选项将路由的所有段数据打包到一个响应中，将预取请求数量减少到每个链接一个。权衡是共享布局数据在预内联响应中被重复，而不是被缓存和重用。

next.config.ts
```
const nextConfig = {
  experimental: {
    prefetchInlining: true,
  },
};
 
export default nextConfig;
```

next.config.ts

```
const nextConfig = {
  experimental: {
    prefetchInlining: true,
  },
};
 
export default nextConfig;
```

这是迈向基于大小的启发式方法的一步，其中小段会自动内联，较大的段保持分离。

### `experimental.cachedNavigations` (#experimentalcachednavigations)

此标志独立控制 缓存导航 (Cached Navigations) 行为，该行为缓存导航和初始 HTML 加载中的静态和动态 Server Components 数据，以便重复访问可以立即提供服务。需要启用 `cacheComponents`。

### `experimental.appNewScrollHandler` (#experimentalappnewscrollhandler)

App Router 中重新设计的滚动和焦点管理系统，使用 React Fragment refs。新的处理器改善了导航后如何管理焦点。不再是聚焦段深层中的第一个可聚焦后代（这可能会跳过内容），现在它模糊活动元素，与浏览器导航工作的方式相匹配。

next.config.ts
```
const nextConfig = {
  experimental: {
    appNewScrollHandler: true,
  },
};
 
export default nextConfig;
```

next.config.ts

```
const nextConfig = {
  experimental: {
    appNewScrollHandler: true,
  },
};
 
export default nextConfig;
```
