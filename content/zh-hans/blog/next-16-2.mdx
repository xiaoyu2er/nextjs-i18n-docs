开发期间更快的 Time-to-URL[](#faster-time-to-url-in-development)
-----------------------------------------------------------------------

我们显著改进了 `next dev` 运行期间 `localhost:3000` 就绪之前的时间。在同一台机器和项目上，与 Next.js 16.1 相比，默认应用程序的启动速度提高了约 87%。

更快的渲染[](#faster-rendering)
-------------------------------------

我们对 React 贡献了一项更改 [1]，使 Server Components 负载反序列化速度快了 350%。之前的实现使用 `JSON.parse` 更新器回调，这在 V8 中会为解析的 JSON 中的每个键值对跨越 C++/JavaScript 边界。即使是微不足道的空操作更新器也会使 `JSON.parse` 比没有更新器时慢 4 倍。

新方法采用两步流程：纯 `JSON.parse()` 后接纯 JavaScript 中的递归遍历。这消除了跨边界开销，并添加了优化，例如短路不需要转换的普通字符串。

在实际的 Next.js 应用中，这意味着根据 RSC 负载大小，渲染到 HTML 的速度快 25% 到 60%。

`next start` 的 `--inspect`[](#--inspect-for-next-start)
---------------------------------------------------------

Next.js 16.1 引入了 `next dev --inspect` 以便在开发期间附加 Node.js 调试器。在 16.2 中，这扩展到 `next start`，允许您将 Node.js 调试器附加到生产服务器。

终端

```
next start --inspect
```

这对于调试问题或分析 CPU 和内存使用情况很有用。

有关如何使用 Node.js 调试器的更多信息，请查看 [Chrome 文档](https://developer.chrome.com/docs/devtools/performance/nodejs)。

`next/link` 的 `transitionTypes` 属性[](#transitiontypes-prop-for-nextlink)
----------------------------------------------------------------------------

`<Link>` 组件现在接受 [`transitionTypes`](https://react.dev/reference/react/addTransitionType) 属性 — 一个字符串数组，用于指定在导航时应应用的 View Transitions 类型。每种类型在导航 Transition 期间传递给 `React.addTransitionType`，允许您根据导航方向或上下文触发不同的动画。

```
<Link href="/about" transitionTypes={['slide']}>
  About
</Link>
```

这仅支持 App Router，因为 Pages Router 不使用 React Transitions 进行导航。Pages Router 链接上的 `transitionTypes` 会被静默忽略，因此通用链接组件可在两个路由器中使用。

要了解有关视图转换的更多信息，请参阅 [文档](/docs/app/api-reference/config/next-config-js/viewTransition)。

更快的 `ImageResponse`[](#faster-imageresponse)
-----------------------------------------------

`ImageResponse` 已更新，带来重大改进：

*   **2 倍更快** `ImageResponse` 用于基本图像，对于复杂图像快 **20 倍**
*   改进了 CSS 和 SVG 支持，包括支持内联 CSS 变量、`text-indent`、`text-decoration-skip-ink`、`box-sizing`、`display: contents`、`position: static` 和 `gap` 的百分比值
*   默认字体从 Noto Sans 更改为 Geist Sans

了解更多关于 [ImageResponse](/docs/app/api-reference/functions/image-response)。

多种图标格式[](#multiple-icon-formats)
-----------------------------------------------

具有相同基本名称但扩展名不同的多个图标文件现在在您的应用目录中自动处理（例如 `icon.png` 和 `icon.svg`）。这对于浏览器回退支持很有用。现代浏览器可以使用 SVG 图标，而旧浏览器回退到 PNG。两种格式都呈现为单独的 `<link>` 标签。

-----------------------------------------------

### `unstable_catchError()`[](#unstable_catcherror)

扩展 [`error.js`](/docs/app/api-reference/file-conventions/error) 文件约定的错误恢复体验，[`unstable_catchError()`](/docs/app/api-reference/functions/catchError) 提供了更细粒度的错误边界控制。通过使用 `unstable_catchError()` 创建自定义错误边界，您可以将其放置在组件树中的任何位置。

`unstable_catchError()` 创建的错误边界接收从调用点传递的 props 和 `ErrorInfo` 值作为第二个参数，这与传递给 `error.js` 文件约定的 props 形状相同。

与 `error.js` 一样，`unstable_catchError()` 只能从 [Client Components](/docs/app/getting-started/server-and-client-components) 调用。

app/custom-error-boundary.tsx

```
'use client';
 
import { unstable_catchError, type ErrorInfo } from 'next/error';
 
function CustomErrorBoundary(
  props: { title: string },
  { error, unstable_retry }: ErrorInfo,
) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
 
export default unstable_catchError(CustomErrorBoundary);
```

app/page.tsx

```
import CustomErrorBoundary from './custom-error-boundary';
 
export default function Page() {
  return (
    <CustomErrorBoundary title="Oops! Something went wrong!">
      <Component />
    </CustomErrorBoundary>
  );
}
```

### `unstable_retry()` in `error.tsx`[](#unstable_retry-in-errortsx)

新的 [`unstable_retry()`](/docs/app/api-reference/file-conventions/error#unstable_retry) prop 现在可用于 `error.tsx` 组件。之前，`reset()` prop 仅清除错误状态并重新渲染子项，这对于临时渲染错误有效，但在错误源自数据获取或 RSC 阶段时无济于事。

`unstable_retry()` 在 `startTransition()` 内调用 `router.refresh()` 和 `reset()`，提供内置的重试逻辑，重新获取数据并重新渲染片段。预计这对于大多数错误恢复场景比 `reset()` 更受青睐。

app/error.tsx

```
'use client';
 
import type { ErrorInfo } from 'next/error';
 
export default function Error({ error, unstable_retry }: ErrorInfo) {
  return (
    <div>
      <h2>Something went wrong</h2>
      <p>{error.message}</p>
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
```

### `experimental.prefetchInlining`[](#experimentalprefetchinlining)

Next.js 16 引入了每段预取，客户端为路由树中的每个段发出单独的请求。这提高了缓存效率，因为兄弟路由之间的共享布局被获取一次并重新使用，但增加了请求量。

新的 `experimental.prefetchInlining` 选项将路由的所有段数据捆绑到一个响应中，将每个链接的预取请求减少到一个。权衡是共享布局数据在行内响应中复制，而不是被缓存和重新使用。

next.config.ts

```
const nextConfig = {
  experimental: {
    prefetchInlining: true,
  },
};
 
export default nextConfig;
```

这是迈向基于大小的启发式方法的一步，其中小片段自动行内，大片段保持分离。

### `experimental.cachedNavigations`[](#experimentalcachednavigations)

此标志独立控制缓存导航行为，该行为缓存来自导航和初始 HTML 加载的静态和动态 Server Components 数据，以便重复访问可以即时服务。需要启用 `cacheComponents`。

### `experimental.appNewScrollHandler`[](#experimentalappnewscrollhandler)

App Router 的重新设计的滚动和焦点管理系统使用 React Fragment refs。新处理器改进了导航后如何管理焦点。它不再聚焦于深度位于片段内的第一个可聚焦后代（这可能会跳过内容），而是取消聚焦活动元素，匹配浏览器导航的工作原理。

next.config.ts

```
const nextConfig = {
  experimental: {
    appNewScrollHandler: true,
  },
};
 
export default nextConfig;
```
