这是一段漫长的旅程，但我们很高兴宣布 `next dev --turbo` 现已稳定并准备好加快您的开发体验。我们一直在将其用于迭代 [vercel.com](https://vercel.com)、[nextjs.org](https://nextjs.org)、[v0](https://v0.app) 以及我们所有其他的应用，并取得了很好的效果。

自 8 年前发布以来，Next.js 一直用于构建一切，从周末的业余项目到复杂的企业应用。当 Next.js 首次发布时，webpack 显然是该框架打包基础的最佳选择，但随着时间的推移，它一直难以跟上现代网页开发者的需求。我们的社区开始觉得等待路由加载、代码更改反映和生产构建部署的痛苦缓慢迭代。

我们投入了大量 [时间](/blog/next-12#faster-builds-and-fast-refresh-with-rust-compiler) 和 [精力](/blog/next-13-5#improving-startup-and-fast-refresh-time) 优化 webpack，但在某个阶段，我们觉得投入的精力与获得的改进不成正比。我们需要一个新的基础，可以支持许多目前已在生产环境中的 Next.js 应用，以及我们计划中的未来创新，比如 React 服务端组件。

这是我们对这个新打包器的要求：

*   最少的破坏性变更
*   同时支持 App Router 和 Pages Router
*   所有大小代码库的编译时间更快
*   与生产环境紧密匹配的构建
*   高级生产优化（例如，模块内的摇树优化）
*   支持 Node.js 和浏览器等多个环境的模块图
*   给维护者和高级用户的完全可观测性

我们评估了当时所有现有的解决方案，发现每一个都有与我们需求和目标不符的权衡。从头设计一个可以完成 Next.js 今天真正所需功能并拥有路线图以实现未来需求的方案对我们来说是合理的。这就是我们创建 Turbopack 的动机。

我们首先致力于优化开发体验，这就是我们今天发布为稳定的内容。我们通过 Vercel 的应用程序广泛地试用了 Turbopack，并明显提高了开发人员的迭代速度。例如，对于 `vercel.com` 这样一个大型 Next.js 应用，我们看到：

*   本地服务器启动速度提升了 **76.7%**。
*   使用 Fast Refresh 的代码更新速度提升了 **96.3%**。
*   初始路由编译速度提升了 **45.8%**（无缓存情况，Turbopack 尚未有磁盘缓存）。

在本文中，我们将讨论如何实现这些结果以及其它亮点。我们还会澄清从这个发布中可以期待的具体内容，并提供未来预期的路线图。

## 亮点[](#highlights)

### 路由的初始编译更快[](##faster-initial-compile-of-a-route)

我们从社区听到的最大问题之一是路由在开发环境中加载时间过长，这归结为 webpack 的编译速度。Next.js 按需编译路由，以避免在需要之前编译所有可能的路由，这保持了初始启动速度快且内存使用较低，但即便如此，您仍然可能会发现自己在等待单页加载时脚踩脚。

说句公道话，像 webpack 这样的打包程序在幕后做了很多工作。当首次编译路由时，打包程序开始于“入口点”。在 Next.js 的情况下，这是 `page.tsx` 与该路由的所有相关文件（如 `layout.tsx` 和 `loading.tsx` 等）的组合。解析这些入口点以查找 `import` 语句，这些语句被解析为文件，然后以与入口点相同的方式处理，此循环持续直到未找到更多导入。此过程构建模块图，它不仅由 TypeScript / JavaScript 模块（包括 `node_modules`）组成，还包括 CSS 文件（全局和 CSS 模块）以及静态文件（如 `next/image` 的导入图像）。

收集所有模块后，使用模块图创建 JavaScript 捆绑包，通常称为“块”（chunks）。这些块是编译器在服务器（构建时或运行时）或浏览器中运行的输出。

webpack 不支持创建为多环境生成输出的图，因此在现在的 Next.js 中，我们必须使用 webpack 运行至少两个单独的编译器，一个用于服务器，一个用于浏览器。我们必须首先编译服务器模块图，以便找到所有对 `"use client"` 的引用。构建完服务器后，我们遍历其图以创建浏览器编译器的相关入口点。由于这是单独的 webpack 编译器，此过程中存在一些开销，例如在客户端和服务器之间解析相同的代码两次。

在使用 Turbopack 时，我们着手消除运行多个编译器以及它们之间协调的开销。解决方案是使编译器能够感知多个不同的输出目标。内部将这些称为目标“转换”（transitions）。我们可以将导入标记为从服务器到浏览器或从浏览器到服务器的转换。这就是 Turbopack 能够高效地捆绑服务端组件和客户端组件，以及从客户端组件导入的服务端函数的原因。

除了提高性能外，拥有单个编译器可以在一次传递中处理多个环境，还具有可靠性和调试优势，因为我们不再需要在 Next.js 中协调两个单独的编译器进程。

webpack 和 Turbopack 之间的另一个重大区别是 Turbopack 可以在多个 CPU 之间并行化工作，而在 webpack 中，仅使用 SWC 的 TypeScript / JavaScript 转换步骤被并行化。

webpack 不支持跨 CPU 进行并行化，因为为了有效利用并行化，数据必须在线程间容易访问。webpack 的构建方式重度使用大型 JavaScript 对象，这无法在不进行昂贵的序列化和反序列化的情况下在线程间共享。此开销通常抵消了利用多个 CPU 带来的性能改进。Turbopack 是用 Rust 编写的，没有同样的限制，并且从开始就考虑了并行化。

我们还实现了更快的文件系统读写、更快的模块解析以及 [跳过更多无副作用模块的工作](https://gist.github.com/timneutkens/63f95e096a7a060627ca53056720022e) 的性能提升。

在 [`vercel.com`](https://vercel.com) 这样的大型 Next.js 应用程序上使用 Turbopack，我们与 Next.js 和 webpack 相比，初始编译速度最高可提升 **45.8%**。

### 更快的 Fast Refresh[](##faster-fast-refresh)

Fast Refresh 是打包程序用于将更改传播到您当前在浏览器中查看的路由的系统，有时也称为热模块替换（HMR）。

Next.js 有更深入地集成 Fast Refresh 到 React，确保每当您更改组件时 React 不会丢失状态。

使用 webpack，我们发现当您达到一定数量的 JavaScript 模块时，Fast Refresh 的性能存在限制。即使对于未更改的模块，Webpack 也需要进行图遍历和生成输出，随着 JavaScript 模块数量的增加呈线性缩放。

我们发现，在大约 30,000 个模块时，代码更改始终至少有 1 秒的开销来处理更新，无论更改大小如何。例如，更改 CSS 文件中的颜色可能需要 1 秒才能在屏幕上显示。

这种性能对我们来说是不可接受的。我们相信增量构建应该仅随本地更改的大小进行扩展，而不是随路由或应用程序的大小扩展。当 `button.tsx` 更改时，编译器应该只运行与该文件更改相关的工作，而不必重新计算其他不受更改影响的模块和输出文件。为了对抗这个问题，我们在 Turbopack 中优先考虑了允许非常细粒度重新计算工作的基础。

这项努力转化为了底层库 [Turbo Engine](https://nextjs.org/docs/app/api-reference/turbopack)，它使用自动按需增量计算架构，为大规模 Next.js 和 React 应用提供交互式的热重载，耗时仅几十毫秒。此架构基于十多年的研究和现有成果，包括 [webpack](https://webpack.js.org/)、[Salsa](https://github.com/salsa-rs/salsa)、[Parcel](https://parceljs.org/)、[Adapton](http://adapton.org/) 以及 [Rust 编译器的查询系统](https://rustc-dev-guide.rust-lang.org/query.html)。

现在使用 Turbopack，Fast Refresh 速度随您的更改大小进行扩展，这就是我们能够在 [vercel.com](https://vercel.com) 这样的大型 Next.js 应用上实现 **96.3% 更快** 代码更新的原因。

### 高级追踪[](##advanced-tracing)

随着 Next.js 多年来不断普及，我们发现越来越难在 GitHub 上重现报告的问题，特别是与编译器性能和内存使用相关的问题。这是因为大多数人无法分享其应用程序代码，或者当他们分享代码时，应用程序无法运行，因为它需要数据库或其他设置。

为了解决这个问题，我们向 Next.js 内部添加了追踪。这些追踪写入 `.next` 文件夹中的文件，并且不包含应用程序代码——仅包含文件路径、编译器处理它的时间以及其他时间，如单独转换。但是，使用 webpack，我们从来没有一种好的方式来清晰区分编译器的内存使用与框架或应用程序代码的内存使用，因为它们都在同一个 Node.js 实例中运行。

使用 Turbopack，我们能够从一开始就设计了可观测性。我们在 [Turbo Engine](https://nextjs.org/docs/app/api-reference/turbopack) 中实现了可观测性层，可以收集每个单独函数的时间。我们能够扩展这些追踪，还跟踪每个函数的内存分配、释放和持久内存。

这种新的、高级的追踪为我们提供了调查减慢和内存使用所需的所有信息；它只需要一个追踪，而不需要整个代码库。

为了处理这些新的追踪，我们实现了一个自定义追踪查看器，无论应用程序和追踪大小如何都能保持高性能。这是专门为调查 Turbopack 的减慢和内存使用而构建的追踪查看器，它通过缩短反馈循环，允许我们为许多早期采用的应用程序优化性能。

虽然追踪查看器最初是用于内部使用的（并且它旨在用于需要深入技术调查的情况），但我们已将所需部分归入 Next.js，让您自己运行。您可以使用 [这些说明](/docs/canary/architecture/turbopack#generating-trace-files) 生成 Turbopack 追踪。然后，当生成追踪时，您可以使用 `next internal turbo-trace-server .next/trace-turbopack` 启动允许检查追踪的服务器。这里有一段关于追踪查看器的快速视频概览 [可用此处](https://www.youtube.com/watch?v=PGO2szAye7A&t=1s)。

### 编译时间的不一致性减少[](##less-flakiness-in-compile-times)

使用 Next.js 和 webpack 时，编译时间往往不够透明。例如，打开页面可能需要 10 秒，而在另一种情况下，可能需要 20 秒。虽然有缓存存在，但有时它的影响不足以产生一致的结果。即使在没有缓存的编译中，我们也看到了一些波动。

Turbopack 的基础架构确保编译时间的波动要一致得多。路由的编译时间仅变化几个百分点，使我们能够持续优化编译器性能。

### 与生产紧密匹配的开发构建 [](##development-builds-that-closely-match-production)

为了优化 webpack 的编译速度，我们必须接受一些权衡，导致开发和生产环境出现分歧。那些权衡的一些例子是，我们使用 `style-loader`，它向页面注入样式并允许快速刷新它们，而无需重新加载页面。但是，这意味着样式是在开发环境中通过 JavaScript 注入的，导致内容闪烁。我们绕过了这种闪烁，因此您看不到它。另一个例子是，带有 webpack 的 Next.js 使用 `eval-source-map`，这意味着所有代码都包装在 `eval` 中，并且源映射包含在其中，这确保了开发环境中源映射可用，代价是捆绑代码更难检查。虽然 webpack 支持使用 `source-map` 选项输出完整源映射，但这对编译时间和内存使用产生了不成比例的影响。

对于 Turbopack，我们着手默认解决这些问题，输出 CSS 文件和源映射而不使用 `eval`。Turbopack 利用 [`sections` 源映射](https://tc39.es/source-map/#index-map)，这是源映射规范中相对较新的部分，允许更有效地合并源映射输出。我们以前不得不在一个地方生成所有映射，现在我们可以更细粒度地生成和缓存它们。

Turbopack 中的 CSS 处理始终输出 CSS 文件，并且类似于 JavaScript 处理，可以通过 Turbopack 开发运行时的机制在不刷新浏览器的情况下更新 CSS 文件。

我们现在可以自信地说，当某事物在开发环境中与 Turbopack 一起工作时，它在生产环境中也能同样工作。

## 我们的首次稳定发布[](#our-first-stable-release)

两年前，作为 Next.js 13 的一部分，我们推出了作为 alpha 版本的 Turbopack，展示了其性能潜力。虽然最初结果令人鼓舞，但它只支持基本用途——许多 Next.js 功能，如 `basePath`，尚未实现。

在接下来的一年里，我们专注于添加缺失的 Next.js 和打包功能。根据社区反馈，我们决定完全专注于 `next dev` 体验，这样我们可以解决最常见的迭代速度投诉。到去年 Next.js 大会，90% 的开发测试已通过，Vercel 开发人员已经在日常开发中使用 Turbopack。

在四月，我们宣布了 Next.js 14.2，99.8% 的测试通过，不久后达到 100%。从那时起，我们解决了 GitHub 报告的问题，特别是在 npm 包、Fast Refresh 和错误定位准确性方面。

诚然，通往稳定之路花费了很长时间，但这主要归结于 Next.js 广泛的测试套件，它设定了稳定性的最高标准。我们有 8 年的时间来发现边缘情况并添加 6,599 个开发测试，这些测试也需要与 Turbopack 一起通过。另一个因素是我们设计的 Turbopack 具有与 webpack 完全不同的架构。简单地将其移植到 Rust 会更容易，但不会实现我们要实现的性能提升。

现在 Turbopack 通过了所有测试，经过顶级 npm 包验证，并解决了早期采用者的反馈，我们准备好将其标记为稳定。

### 究竟什么是稳定的？[](##what-exactly-is-stable)

这在过去一直是一个混淆点，所以我们将利用此部分澄清此发布为 Next.js 社区解锁了什么。

此发布特别将 `next dev --turbo` 命令标记为稳定。生产构建 (`next build --turbo`) 尚不支持，但随着它们正在进行中，请继续阅读更新。我们最终计划发布独立于 Next.js 之外的 Turbopack 版本，但我们想通过首先增强 Next.js 社区的体验来证明其价值。

除了我们在下一部分中涵盖的不支持功能外，Turbopack 应该适用于 Next.js 的所有稳定功能。为简洁起见，Turbopack 支持 App Router 和 Pages Router。实验性功能可能可以也可能不可以与 Turbopack 一起使用，但在它们被标记为稳定时肯定可以使用。

如果您的应用程序有 webpack 定制但仅添加 webpack loaders，您可能已经通过为 Turbopack 配置 loaders 来使用 Turbopack。您可以 [阅读文档](/docs/canary/app/api-reference/next-config-js/turbo#webpack-loaders) 来了解 Turbopack 中的 webpack loader 支持。

以下是经过验证可与 Turbopack 一起使用的 webpack loaders 列表：

*   `@svgr/webpack`
*   `babel-loader`
*   `url-loader`
*   `file-loader`
*   `raw-loader`
*   `tsconfig-paths-webpack-plugin` —— 开箱即用支持，无需插件。
*   大多数其他 loaders 也能工作，因为我们支持 webpack loader API 的子集。

大多数 CSS 和 CSS-in-JS 库都受支持：

*   支持
    *   Tailwind CSS
    *   @emotion/react
    *   Sass
    *   styled-components
    *   Bootstrap
    *   Antd
    *   node-sass
    *   JSS
    *   Emotion
    *   theme-ui (使用 Emotion)
    *   @chakra-ui/core (使用 Emotion)
    *   aphrodite
*   当前不支持
    *   Less —— 您可以添加 less-loader。带有 webpack 的 Next.js 也不是开箱支持 Less。
    *   @vanilla-extract/css —— 使用自定义 webpack 插件 —— 我们将研究支持所需钩子的情况。
    *   StyleX —— 需要 Babel 转换和 `data:` 属性支持 —— 我们将在 `next build --turbo` 稳定后研究支持 StyleX。

### 性能[](##performance)

我们要指出，今天发布的版本性能显著优于 webpack，但它不是最终性能数字。我们一直在遵循 Kent Beck 的名言“让它工作。让它变好。让它变快。”到目前为止，我们大部分精力都花在了“让它工作”阶段，因为我们必须赶上已经成熟近十年的 Next.js 和 webpack 的范围。

Turbopack 在缓存基础设施上重注一博，但如您所知，缓存是软件开发中仅有的两难之一。根据经验，我们知道将缓存添加到未明确为此构建的架构可能导致不良结果，所以我们为最细粒度的函数开启了缓存。这意味着在冷启动和内存使用方面，重新构建非常快，我们正在努力实现更好的平衡。有趣的是，我们可以使用本文前面提到的高级追踪来查找低效之处并分析哪些函数最值得缓存。

在过去的 3 个月里，我们已经取得了一些显著改进。比较 [Next.js 15 RC 2](/blog/next-15-rc2) 中的 Turbopack 与 [15 RC 1](/blog/next-15-rc) 中的 Turbopack 显示了这些优化的结果：

*   平均 **内存** 使用量减少 25-35%。
*   大型页面（包含数千个模块）的 **初始** **编译** 速度提升 30-50%。

稳定版本的 Turbopack 包含内存缓存，必须在开发服务器每次重启时重建，这可能需要 10 秒或更长时间，对于大型应用程序。令我们非常兴奋的是，在测试磁盘持久化缓存时我们看到的巨大胜利，我们将在本文后面讨论。

### 破坏性变更[](##breaking-changes)

构建我们自己的打包器的一个巨大动机是需要尽可能匹配 webpack 的现有行为，这是我们当时无法保证的任何现有解决方案。包括文件解析的方式和 webpack 的较小功能，如某些 npm 包使用的 [`webpackIgnore` 注释](https://webpack.js.org/api/module-methods/#magic-comments)。

不幸的是，我们确实移除了一些功能以保护 Turbopack 和相关 Next.js 实现的未来。当您使用 webpack 时，这些功能仍将得到支持。

有一些亮点，让我们深入研究为什么我们要改变它们：
**不支持 `webpack()` 配置。** Turbopack 不是 webpack，它没有相同的配置选项结构，尽管它支持许多相同的特性。我们具体实现了 [webpack loaders](/docs/app/api-reference/next-config-js/turbo#webpack-loaders) 和 [resolve aliases](/docs/app/api-reference/next-config-js/turbo#resolve-aliases) 的支持。大多数转换代码的 webpack loaders 开箱即用。一些做奇特事情的 webpack loaders，如 webpack 子编译器和发出文件，不受支持。

**`.babelrc` 不会自动转换代码。** Turbopack 默认利用 [SWC](/docs/architecture/nextjs-compiler#why-swc)。您仍然可以在需要时添加 `babel-loader`，但我们确保默认值始终快速，且从架构角度来看也很合理。即使您配置了 `.babelrc`，我们也必须始终运行 SWC 以处理其他优化。这类似于 webpack 必须始终运行 `acorn` 解析器以进行进一步优化的方式。如果您在 Turbopack 中使用 SWC 而不是 Babel，我们可以在整个 Turbopack 解析一次并端到端利用相同的抽象语法树 (AST)。

**一些少用的 CSS 模块功能。** 我们已经将 CSS 处理从 PostCSS 切换到 [Lightning CSS](https://lightningcss.dev/)。Lightning CSS 是一个快得多的 CSS 编译器，开箱支持 CSS 转换、压缩和 CSS 模块。权衡是未受支持一些少用的功能。具体而言，`:global` 和 `:local` 伪选择器（它们的函数变体 `:global()` 和 `:local()` 仍然有效），`@value` 和 `:import / :export` ICSS 规则。它比其他 CSS 解析器更严格，并会指向代码中的错误而不是忽略它们。

在添加 Lightning CSS 的过程中，我们向该项目做出了贡献。例如，我们为 CSS 模块实现了细粒度选项，以禁用 CSS 网格前缀和 CSS 模块的 `pure` 模式。这使得从 webpack 的 css-loader 切换到 Lightning CSS 用于 CSS 模块更加容易。我们也改进了不受支持的 CSS 模块功能的错误。

我们要感谢 [Devon Govett](https://x.com/devongovett)，Lightning CSS 的作者和维护者，对项目的持续合作。

**实验性功能。** 由于我们专注于 Next.js 中 Turbopack 的稳定性，我们决定首先专注于 Next.js 中可用的稳定功能。

完整列表请参阅 [文档页面](/docs/canary/architecture/turbopack#unsupported-features)。

### 持久化缓存（重启间的 Fast Refresh）[](##persistent-caching-fast-refresh-across-restarts)

持久化缓存意味着以允许在开发服务器重启之间或跨多个生产构建之间重用编译工作的方式存储工作。

简而言之，Turbopack 会避免在执行重启时重复做同样的工作。

正如在“更快的 Fast Refresh"部分中提到的，我们构建了 [Turbo Engine](https://nextjs.org/docs/app/api-reference/turbopack) 以确保工作可以并行化和缓存，因此每当您进行文件更改时，我们只需要运行与该文件更改相关的工作。如果我们可以跨重启和打开路由为您提供这种体验怎么办？我们不需要重新编译在以前开发会话中完成的工作。如果我们可以获得 Fast Refresh 的好处，但用于打开在以前开发会话中编译的路由，以及跨多个 `next build` 构建呢？

这正是我们要处理的内容：[Turbo Engine](https://nextjs.org/docs/app/api-reference/turbopack) 的一个新存储层，支持将编译工作持久化到磁盘并在启动开发服务器或重新构建时恢复它。

虽然 webpack 在 Next.js 中默认开启了磁盘缓存，但它有许多限制。值得注意的是，缓存的很大一部分必须从磁盘恢复并读入内存才能工作。感觉从来就不够细粒度。例如，在 Vercel 的大型应用程序上，我们发现当缓存增长到足够大的大小时，webpack 磁盘缓存甚至比从头开始做所有工作更慢。

与 webpack 的现有磁盘缓存不同，Turbopack 的持久化缓存确实感觉像重启间的 Fast Refresh。第一次编译需要 10 秒以上的路由，一旦编译过一次，从缓存恢复只需 500 毫秒不到。

我们看到 `next build` 与 Turbopack 的类似结果，其中只有更改的文件被重新编译，其余保持不变。在 `next build` 采取的多个步骤中，这将花费的大部分时间从运行编译和打包转移到了运行 TypeScript 类型检查。

持久化缓存目前正在进行中，因为我们想首先使用内部 Next.js 应用程序对其进行验证。初步结果非常令人鼓舞，随着我们保持优化这些热点路径，性能将变得更好。

一旦持久化缓存稳定，它将默认启用。启用持久化缓存不需要更改您的代码库。

如果您有兴趣测试持久化缓存，请联系！

### 生产构建[](##production-builds)

我们很高兴地分享我们正在 Turbopack 中稳步推进稳定的生产构建。目前，我们 96% 的生产测试已通过，这是一个巨大的进步。但是，在我们能够自信地推荐大规模生产环境的 Turbopack 之前，仍有一些领域需要更多工作。

与开发相比，生产构建带来了自己独特的挑战，我们正在积极努力解决它们。下面，我们将概述什么已经完成，什么仍在进行中。

### 生产优化[](##production-optimizations)

### 正确性[](##correctness)

确保正确性对于可靠的生产构建至关重要。以下是当前状态：

*   **CSS 分块**：进行中。此功能对于将 CSS 拆分为较小的块至关重要，允许每个应用程序部分仅加载必要的 CSS，这有助于减少加载时间并确保 CSS 规则的正确顺序。
*   **生产 JS 运行时**：已完成。这确保 JavaScript 运行时在生产环境中按预期运行，提供可靠性和稳定性。
*   **基于内容的文件名哈希**：尚未实现。基于内容的哈希将允许我们根据内容生成文件名，从而实现浏览器中更有效的长期缓存。

### UX 性能优化[](##ux-performance-optimizations)

UX 性能对于交付快速加载时间和高效的资源使用至关重要。以下是我们正在努力的内容：

*   **JS 压缩**：已完成。我们已实现 SWC Minify，Next.js 自 Next.js 13 起已在使用 webpack 时使用。
*   **CSS 压缩**：已完成。使用 Lightning CSS 进行 CSS 压缩，这对于减少样式表大小很重要。
*   **全局信息（全应用优化）**：已完成。Turbopack 可以应用需要有关应用程序中所有路由数据的优化，例如模块 ID 哈希。
*   **摇树优化 (Tree Shaking)**：部分完成。进行中。我们部分支持摇树优化，这有助于消除未使用代码并减少捆绑大小。然而，还有一些场景摇树优化目前效果不佳：
    *   **动态导入**：动态导入（如使用 `next/dynamic`）的摇树优化有限。
    *   **复杂导出**：某些类型的导出，如 `export { foo as "string name" }`。
    *   **非 ES 模块**：CommonJS 模块不可摇树。
    *   **Barrel 文件**：来自 Barrel 文件的重新导出效率低下，在跳过无副作用模块方面存在限制。
    *   **碎片**：在某些情况下，摇树优化会产生太多碎片，导致捆绑效率低。
*   **模块 ID 哈希（部分）**：进行中。模块 ID 哈希部分实现，但我们正在努力改进性能。一旦完全启用，它将帮助减少最终捆绑大小。
*   **导出名称混淆**：进行中。这涉及减少导出名称的大小以减少最终捆绑大小。
*   **作用域提升 (Scope Hoisting)**：尚未实现。作用域提升将帮助通过将较小的 JavaScript 模块合并到单个作用域中来减少捆绑大小，这减少了开销并提高了性能。
*   **生产优化的 JS 分块**：尚未实现。将 JavaScript 分块以最小化重复对于提高加载性能至关重要，尤其对于大型应用程序。
