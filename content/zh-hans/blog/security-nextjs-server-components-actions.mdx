React Server Components (RCS) 在 App Router 中是一种新颖的范式，消除了与常规方法相关的许多冗余和潜在风险。由于它很新，开发人员随后可能会发现安全团队难以将其现有的安全协议与该模型对齐。

本文档旨指出需要注意的几个领域，内置的保护措施，并包括应用程序审核指南。我们特别专注于意外数据暴露的风险。

选择您的数据处理模型 (Choosing Your Data Handling Model)
-----------------------------------------------------------------------

[React 服务器组件 (React Server Components)](/docs/app/building-your-application/rendering/server-components) 模糊了服务器和客户端之间的界限。数据处理对于理解信息在哪里处理并随后可用至关重要。

我们需要做的第一件事是选择适合我们项目的数据处理方法。

*   [HTTP APIs](#http-apis) (适用于现有大型项目/组织的推荐方法)
*   [数据访问层 (Data Access Layer)](#data-access-layer) (新项目的推荐方法)
*   [组件级数据访问 (Component Level Data Access)](#component-level-data-access) (建议用于原型设计和学习)

我们建议您坚持使用一种方法，不要太多地混合搭配。这对于在您代码库中工作的开发人员和审核人员来说，都能清楚地期望什么。例外情况会显得可疑。

### HTTP APIs

如果您在现有项目中采用服务器组件，推荐的方法是在运行时将服务器组件默认为不安全/不受信任，就像 SSR 或在客户端中一样。因此，没有内部网络或信任区域的假设，工程师可以应用零信任概念。相反，像从客户端执行一样，您只需在服务器组件中使用 `fetch()` 调用自定义 API 端点，例如 REST 或 GraphQL。传递任何 cookie。

如果您现有连接数据库的 `getStaticProps`/`getServerSideProps`，您可能希望合并模型并将这些也移动到 API 端点，以便您有一种做事方式。

注意任何假设从内部网络获取的是安全的访问控制。

这种方法让您能够保留现有的组织结构，其中现有的专注于安全的后端团队可以应用现有的安全实践。如果这些团队使用非 JavaScript 语言，这种方法也适用。

这仍然利用了服务器组件的许多好处，通过向客户端发送更少的代码，并且固有的数据瀑布流可以以低延迟执行。

### 数据访问层 (Data Access Layer)

我们推荐的新项目方法是创建一个位于 JavaScript 代码库内的独立数据访问层，并在那里合并所有数据访问。这种方法确保了一致性，并减少了发生授权错误的可能性。鉴于您正在合并到一个单独的库中，维护起来也更简单。可能提供通过单一编程语言实现的更好的团队协作。您还可以通过更低的运行时开销获得更好的性能，能够在请求的不同部分之间共享内存中的缓存。

您构建一个内部 JavaScript 库，在向调用者提供之前提供自定义数据访问检查。类似于 HTTP 端点，但在相同的内存模型中。每个 API 都应该接受当前用户，并在返回之前检查用户是否有权查看此数据。原则是，服务器组件函数体应该只看到请求发起者当前用户授权可以访问的数据。

从此点开始，实施 API 的常规安全实践接管。

data/auth.tsx

```
import { cache } from 'react';
import { cookies } from 'next/headers';
 
// Cached helper methods makes it easy to get the same value in many places
// without manually passing it around. This discourages passing it from Server
// Component to Server Component which minimizes risk of passing it to a Client
// Component.
export const getCurrentUser = cache(async () => {
  const token = cookies().get('AUTH_TOKEN');
  const decodedToken = await decryptAndValidate(token);
  // Don't include secret tokens or private information as public fields.
  // Use classes to avoid accidentally passing the whole object to the client.
  return new User(decodedToken.id);
});
```

data/user-dto.tsx

```
import 'server-only';
import { getCurrentUser } from './auth';
 
function canSeeUsername(viewer: User) {
  // Public info for now, but can change
  return true;
}
 
function canSeePhoneNumber(viewer: User, team: string) {
  // Privacy rules
  return viewer.isAdmin || team === viewer.team;
}
 
export async function getProfileDTO(slug: string) {
  // Don't pass values, read back cached values, also solves context and easier to make it lazy
 
  // use a database API that supports safe templating of queries
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`;
  const userData = rows[0];
 
  const currentUser = await getCurrentUser();
 
  // only return the data relevant for this query and not everything
  // <https://www.w3.org/2001/tag/doc/APIMinimization>
  return {
    username: canSeeUsername(currentUser) ? userData.username : null,
    phonenumber: canSeePhoneNumber(currentUser, userData.team)
      ? userData.phonenumber
      : null,
  };
}
```

这些方法应该暴露可以直接安全传输到客户端的对象。我们喜欢称这些为数据传输对象 (DTO)，以澄清它们已准备好供客户端使用。

在实践中，它们可能只被服务器组件消耗。这创建了一个分层结构，安全审计可以侧重于数据访问层，而 UI 可以快速迭代。更小的攻击面和更少的代码覆盖使得更容易发现安全问题。

```
import {getProfile} from '../../data/user'
export async function Page({ params: { slug } }) {
  // This page can now safely pass around this profile knowing
  // that it shouldn't contain anything sensitive.
  const profile = await getProfile(slug);
  ...
}
```

密钥可以存储在环境变量中，但在这种情况下，只有数据访问层应该访问 `process.env`。

### 组件级数据访问 (Component Level Data Access)

另一种方法是将数据库查询直接放入服务器组件中。这种方法仅适用于快速迭代和原型制作。例如，对于团队较小且每个人都意识到风险及如何监视风险的小型产品。

在这种情况下，您想仔细审核您的 `"use client"` 文件。在审核和审查 PR 时，查看所有导出的函数，如果类型签名接受过于宽泛的对象如 `User`，或者包含 `token` 或 `creditCard` 这样的属性。即使是隐私敏感字段如 `phoneNumber` 也需要仔细检查。客户端组件不应接受比执行工作所需的最小数据更多的数据。

```
import Profile from './components/profile.tsx';
 
export async function Page({ params: { slug } }) {
  const [rows] = await sql`SELECT * FROM user WHERE slug = ${slug}`;
  const userData = rows[0];
  // EXPOSED: This exposes all the fields in userData to the client because
  // we are passing the data from the Server Component to the Client.
  // This is similar to returning `userData` in `getServerSideProps`
  return <Profile user={userData} />;
}
```

```
'use client';
// BAD: This is a bad props interface because it accepts way more data than the
// Client Component needs and it encourages server components to pass all that
// data down. A better solution would be to accept a limited object with just
// the fields necessary for rendering the profile.
export default async function Profile({ user }: { user: User }) {
  return (
    <div>
      <h1>{user.name}</h1>
      ...
    </div>
  );
}
```

始终使用参数化查询，或使用代为您执行的数据库库，以避免 SQL 注入攻击。

### 服务器端专用 (Server Only)

只在服务器端执行代码可以标记为：

```
import 'server-only';
```

如果客户端组件尝试导入此模块，这将导致构建错误。这可用于确保专有/敏感代码或内部业务逻辑不会意外泄露到客户端。

传输数据的主要方法是使用 React 服务器组件协议，这在向客户端组件传递属性时会自动发生。此序列化支持 JSON 的超集。传输自定义类不受支持，会导致错误。

因此，避免意外过大对象泄露到客户端的一个好窍门是，对于数据访问记录使用 `class`。

在即将到来的 Next.js 14 版本中，您也可以在 `next.config.js` 中启用 `taint` 标志来尝试实验性的 [React 污染 API](https://react.dev/reference/react/experimental_taintObjectReference)。

next.config.js

```
module.exports = {
  experimental: {
    taint: true,
  },
};
```

这允许您标记不应允许直接传递给客户端的对象。

app/data.ts

```
import { experimental_taintObjectReference } from 'react';
 
export async function getUserData(id) {
  const data = ...;
  experimental_taintObjectReference(
    'Do not pass user data to the client',
    data
  );
  return data;
}
```

app/page.tsx

```
import { getUserData } from './data';
 
export async function Page({ searchParams }) {
  const userData = getUserData(searchParams.id);
  return <ClientComponent user={userData} />; // error
}
```

这不能防止提取该对象中的数据字段并传递它们：

app/page.tsx

```
export async function Page({ searchParams }) {
  const { name, phone } = getUserData(searchParams.id);
  // Intentionally exposing personal data
  return <ClientComponent name={name} phoneNumber={phone} />;
}
```

对于 token 等唯一字符串，也可以阻止原始值，使用 [`taintUniqueValue`](https://react.dev/reference/react/experimental_taintUniqueValue)。

app/data.ts

```
import { experimental_taintObjectReference, experimental_taintUniqueValue } from 'react';
 
export async function getUserData(id) {
  const data = ...;
  experimental_taintObjectReference(
    'Do not pass user data to the client',
    data
  );
  experimental_taintUniqueValue(
    'Do not pass tokens to the client',
    data,
    data.token
  );
  return data;
}
```

但是，即使这样也阻止不了衍生值。

最好从一开始就避免数据进入服务器组件——使用数据访问层。Taint 检查通过指定值提供了额外的一层保护，请注意，函数和类已经被阻止传递给客户端组件。更多的层可以最小化某些内容遗漏的风险。

默认情况下，环境变量仅在服务器端可用。按照约定，Next.js 还将任何以 `NEXT_PUBLIC_` 为前缀的环境变量暴露给客户端。这允许您向客户端公开某些明确的配置。

### 服务端渲染 (SSR) vs 服务器组件 (RCS)

对于初始加载，Next.js 将在服务器上运行服务器组件和客户端组件以生成 HTML。

服务器组件 (RCS) 在与客户端组件不同的模块系统中执行，以避免在两个模块之间意外暴露信息。

通过服务器端渲染 (SSR) 渲染的客户端组件应被视为与浏览器客户端具有相同的安全策略。它不应访问任何特有权的数据或私有 API。强烈不建议使用黑客手段试图绕过此保护（例如，将数据存储在全局对象上）。原则是这段代码应该能够在服务器和客户端上执行相同的代码。为了与默认安全实践保持一致，如果从客户端组件导入了 `server-only` 模块，Next.js 将导致构建失败。

### 读取 (Read)

在 Next.js App Router 中，从数据库或 API 读取数据是通过渲染服务器组件页面来实现的。

页面的输入是 URL 中的 `searchParams`，从 URL 映射的动态参数和标头。这些可能会被客户端滥用为不同的值。它们不应该被信任，并且每次读取时都需要重新验证。例如，`searchParam` 不应用于跟踪诸如 `?isAdmin=true` 之类的内容。仅仅因为用户在 `/[team]/` 上并不意味着他们有权访问该团队，这需要在读取数据时进行验证。原则是始终在读取数据时重新读取访问控制和 `cookies()`。不要将其作为属性或参数传递。

渲染服务器组件永远不应该执行诸如突变之类的副作用。这并非服务器组件所独有。React 自然不鼓励副作用，甚至在渲染客户端组件时（在 `useEffect` 之外），通过诸如双重渲染等方法。

此外，在 Next.js 中，没有方法可以在渲染期间设置 cookie 或触发缓存重组。这也阻止了使用渲染进行突变。

例如，`searchParams` 不应用于执行诸如保存更改或登出之类的副作用。相反，服务器操作 (Server Actions) 应该用于此目的。

这意味着 Next.js 模型在按预期使用时从不将 GET 请求用于副作用。这有助于避免 CSRF 问题的大量来源。

Next.js 确实支持自定义路由处理器 (`route.tsx`)，可以在 GET 上设置 cookie。它被认为是逃生舱口，不属于一般模型。这些必须明确选择接受 GET 请求。没有可能会意外接收 GET 请求的通配符处理程序。如果您确实决定创建自定义 GET 处理程序，可能需要对这些进行额外审计。

### 写入 (Write)

在 Next.js App Router 中执行写入或突变的习惯用法方法是使用 [服务器操作 (Server Actions)](/docs/app/building-your-application/data-fetching/forms-and-mutations)。

actions.ts

```
'use server';
 
export function logout() {
  cookies().delete('AUTH_TOKEN');
}
```

`'use server'` 注解公开了一个端点，使所有导出的函数可被客户端调用。标识符当前是源代码位置的哈希。只要用户获取到操作的 ID 句柄，您就可以用它调用任何参数。

结果，这些函数应该始终首先验证当前用户是否被允许调用此操作。函数还应该验证每个参数的完整性。这可以手动完成，也可以与 `zod` 等工具一起完成。

actions.ts

```
"use server";
 
export async function deletePost(id: number) {
  if (typeof id !== 'number') {
    // The TypeScript annotations are not enforced so
    // we might need to check that the id is what we
    // think it is.
    throw new Error();
  }
  const user = await getCurrentUser();
  if (!canDeletePost(user, id)) {
    throw new Error();
  }
  ...
}
```

### 闭包 (Closures)

服务器操作 (Server Actions) 也可以编码在 [闭包](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures) 中。这允许操作与渲染时使用的数据快照关联，以便在调用操作时使用：

app/page.tsx

```
export default function Page() {
  const publishVersion = await getLatestVersion();
  async function publish() {
    "use server";
    if (publishVersion !== await getLatestVersion()) {
      throw new Error('The version has changed since pressing publish');
    }
    ...
  }
  return <button action={publish}>Publish</button>;
}
 
```

闭包的状态快照必须在服务器被调用时发送到客户端并返回。

在 Next.js 14 中，闭合变量在与操作 ID 一起发送之前会使用加密。默认情况下，私钥在构建 Next.js 项目期间自动生成。每个重新生成都会生成一个新的私钥，这意味着每个操作只能通过特定的构建调用。您可能希望使用 [偏斜保护 (Skew Protection)](https://vercel.com/blog/version-skew-protection) 来确保在重新部署时始终调用正确的版本。

如果您需要一个旋转频率更高或在多个构建之间持久的密钥，您可以使用 `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 环境变量手动配置它。

通过加密所有闭合变量，您就不会意外在其中泄露任何秘密。通过签名，它使得攻击者更难篡改操作的输入。

使用闭包的另一替代方案是使用 JavaScript 中的 `.bind(...)` 函数。**这些不是加密的。** 这是为了性能而提供的选择退出选项，并且与客户端上的 `.bind()` 一致。

app/page.tsx

```
async function deletePost(id: number) {
  "use server";
  // verify id and that you can still delete it
  ...
}
 
export async function Page({ slug }) {
  const post = await getPost(slug);
  return <button action={deletePost.bind(null, post.id)}>
    Delete
  </button>;
}
```

原则是，服务器操作 (Server Actions) (`"use server"`) 的参数列表必须始终被视为敌对的，并且必须验证输入。

### 跨站请求伪造 (CSRF)

所有服务器操作都可以由纯 `<form>` 调用，这可能会使它们面临 CSRF 攻击。在幕后，服务器操作始终使用 POST 实现，并且只有这个 HTTP 方法允许调用它们。这本身就阻止了现代浏览器中的大多数 CSRF 漏洞，特别是由于 Same-Site Cookie 是默认的。

作为额外保护，Next.js 14 中的服务器操作还比较 `Origin` 标头与 `Host` 标头（或 `X-Forwarded-Host`）。如果不匹配，操作将被拒绝。换句话说，服务器操作只能在托管它的页面的同一主机上调用。非常陈旧的不受支持且不兼容 `Origin` 标头的浏览器可能会有风险。

服务器操作不使用 CSRF token，因此 HTML 清理至关重要。

当改用自定义路由处理器 (`route.tsx`) 时，可能需要额外审计，因为那里必须手动完成 CSRF 保护。那里适用传统规则。

### 错误处理 (Error Handling)

错误难免发生。当错误在服务器上抛出时，它们最终会在客户端代码中重新抛出，以便在 UI 中处理。错误消息和堆栈跟踪可能会包含敏感信息。例如，`[credit card number] is not a valid phone number`。

在生产模式下，React 不会向客户端发出错误或拒绝的 Promise。相反，会发送代表错误的哈希。此哈希可用于将多个相同错误关联在一起并将错误与服务器日志关联。React 用其自己的通用错误消息替换错误消息。

在开发模式下，服务器错误仍会以纯文本形式发送到客户端以协助调试。

始终在 Next.js 的生产模式下运行对于生产负载很重要。开发模式不针对安全性和性能进行优化。

### 自定义路由和中间件 (Custom Routes and Middleware)

[自定义路由处理器](/docs/app/building-your-application/routing/route-handlers) 和 [中间件](/docs/app/building-your-application/routing/middleware) 被视为框架无法通过任何其他内置功能实现的低级逃生舱口。这也打开了框架否则保护免受潜在的漏洞 (footguns)。能力越大，责任越大。

如上所述，`route.tsx` 路由可以实现自定义的 GET 和 POST 处理程序，如果操作不当可能会遭受 CSRF 问题。

中间件可用于限制对某些页面的访问。通常最好用允许列表而不是拒绝列表来完成此操作。这是因为很难知道访问数据的所有不同方式，例如是否存在重写或客户端请求。

例如，通常只想到 HTML 页面。Next.js 还支持客户端导航，可以加载 RSC/JSON 负载。在 Pages Router 中，这曾经在自定义 URL 中。

为了使编写匹配器更容易，Next.js App Router 始终使用页面的简单 URL 进行初始 HTML、客户端导航和服务器操作。客户端导航使用 `?_rsc=...` 搜索参数作为缓存破坏者。

服务器操作驻留在它们所在的页面上，因此继承了相同的访问控制。如果中间件允许读取页面，您也可以调用该页面上的操作。要限制对页面上服务器操作的访问，您可以禁用该页面上的 POST HTTP 方法。

### 审计 (Audit)

如果您正在审计 Next.js App Router 项目，我们建议您额外关注以下几点：

*   **数据访问层 (Data Access Layer)**。是否有针对独立数据访问层的确立实践？验证是否在数据访问层之外导入了数据库包和环境变量。
*   **`"use client"` 文件**。组件属性是否期望私有数据？类型签名是否过于宽泛？
*   **`"use server"` 文件**。操作参数是在操作内还是在数据访问层内验证的？用户是否在操作内重新授权？
*   **`/[param]/`**。带有括号的文件夹是用户输入。参数已验证吗？
*   **`middleware.tsx`** 和 **`route.tsx`** 拥有很大的权力。花费额外时间使用传统技术审核这些。定期进行渗透测试或漏洞扫描，或与团队软件开发生命周期一致。
