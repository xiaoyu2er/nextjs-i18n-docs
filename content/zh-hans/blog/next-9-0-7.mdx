Next.js 9.0 大约两个月前发布。从那时起，我们忙于进行了 7 次较小但相当重要的发布：[9.0.1](https://github.com/vercel/next.js/releases/tag/v9.0.1), [9.0.2](https://github.com/vercel/next.js/releases/tag/v9.0.2), [9.0.3](https://github.com/vercel/next.js/releases/tag/v9.0.3), [9.0.4](https://github.com/vercel/next.js/releases/tag/v9.0.4), [9.0.5](https://github.com/vercel/next.js/releases/tag/v9.0.5), [9.0.6](https://github.com/vercel/next.js/releases/tag/v9.0.6), 和 [9.0.7](https://github.com/vercel/next.js/releases/tag/v9.0.7)。

让我们深入了解这些发布为您的网站和应用程序带来了什么，绝对没有任何破坏性变更。

*   **[Windows 环境下改进的并发情况 (#improved-concurrency-in-windows-environments)](#improved-concurrency-in-windows-environments)**: `next build` 过程现在在 Windows 上更可靠，可以更好地并行处理工作。
*   **[默认启用 Gzip 压缩 (#gzip-compression-by-default)](#gzip-compression-by-default)**: 默认添加了 Gzip 压缩以减少所需的优化步骤。
*   **[仅对活动页面进行 TypeScript 报告 (#typescript-report-on-active-pages-only)](#typescript-report-on-active-pages-only)**: 内置的 TypeScript 支持已扩展为仅监视活动页面的更改，使其更快且更可靠。
*   **[遥测 (Telemetry) (#telemetry)](#telemetry)**: 将帮助我们专注于要优化 Next.js 的哪些方面，并验证优化是否具有预期的效果。
*   **[改进的 `next/head` 元素跟踪 (#improved-nexthead-element-tracking)](#improved-nexthead-element-tracking)**: 已移除 `next-head` 类，这使得实现某些验证其实现的工具变得更简单。
*   **[防止 Pages 目录中的非页面 (#preventing-non-pages-in-the-pages-directory)](#preventing-non-pages-in-the-pages-directory)**: 通过防止意外发布非页面来优化您的应用程序安全性和 `next build` 时间。
*   **[运行时改进 (#runtime-improvements)](#runtime-improvements)**: 当某些 Next.js 部分未被使用时，例如 `next/dynamic`，它们在运行时将不再需要，使打包大小更小。

### Windows 环境下改进的并发情况 (#improved-concurrency-in-windows-environments)

Next.js 在 `next build` 过程中的许多地方都会进行并发工作。主要用途是在并行中使用 Terser 最小化构建输出。

以前，这项工作是通过一个名为 `worker-farm` 的包跨多个 CPU 处理的。然而，我们注意到许多 Windows 用户禁用了最小化，使用了自定义 webpack 配置。进一步检查后，我们发现 `worker-farm` 在 Windows 机器上无法一致工作。

为了解决这个问题，我们从 `worker-farm` 迁移到了 `jest-worker`。这确保了构建在 macOS、Linux 和 Windows 机器上都可靠且一致。

顾名思义，`jest-worker` 是 Jest 测试运行器的一部分。它是 Jest 用于并行化测试用例的包。这意味着该包经过了充分的实战测试，可靠且得到维护。

`jest-worker` 还支持 [`worker_threads`](https://github.com/facebook/jest/tree/master/packages/jest-worker#experimental-worker)，这是 Node 12 中的一个新功能。与 `child_process` 不同，`worker_threads` 可以共享内存 —— 这意味着在新版 Node 上构建时间更快。

切换到 `jest-worker` 后，我们能够重新为 Windows 用户启用构建并发。

### 默认启用 Gzip 压缩 (#gzip-compression-by-default)

在调查公司为何使用自定义服务器时，我们发现最常见的原因是压缩。公司会添加一个叫 `compression` 的 Express 中间件，它会处理 HTTP 响应的 Gzip 压缩。

此中间件会压缩响应，从而减少通过网络传输给用户的字节数。通常，这应由反向代理（如 Nginx）处理。反向代理从单线程 Node 进程中移除 CPU 繁重的任务。

然而，在检查 Web 上的 Next.js 使用时，我们发现很大一部分公司没有配置任何压缩。

在 [Vercel](https://vercel.com) 等平台上，`gzip` 和 `brotli` 是在反向代理级别自动处理的。

在自托管时，公司需要自行添加 gzip（通过 `compression` 或反向代理）。

从 Next.js 9.0.4 开始，当使用 `next start` 或自定义 `server.js` 时，默认包含 `gzip` 压缩。

由于 Node.js 现已原生支持 Brotli，`brotli` 支持[即将推出](https://github.com/expressjs/compression/pull/150)。

如果您的应用程序已通过自定义服务器启用了压缩，Next.js 将不会添加自己的压缩器。

Next.js 默认不为 serverless 目标包含压缩，因为使用 serverless 目标时，资产是单独上传的，不通过 Node.js 服务。

如果您在支持压缩的平台上进行部署，例如 [Vercel](https://vercel.com)，则不需要进行任何更改。

### 仅对活动页面进行 TypeScript 报告 (#typescript-report-on-active-pages-only)

Next.js 9 包含了对 TypeScript 的内置支持。所有需要的就是将单个页面从 `.js` 重命名为 `.tsx`。Next.js 将自动处理或指导您完成剩余的任何设置。

Next.js 还通过在开发过程并行运行 `tsc --watch` 来处理类型检查。在开发过程中，Next.js 有一个称为 [按需条目 (on-demand entries)](https://nextjs.org/blog/next-8#improved-on-demand-entries) 的概念。此功能仅编译您正在积极处理的页面。

![按需条目流程](https://h8DxKfmAPhn8O0p3.public.blob.vercel-storage.com/static/blog/next-8/flow.jpg)

The on-demand entries flow

从 9.0.4 开始，Next.js 现在仅报告由按需条目积极编译的页面的类型错误。这减少了大量类型检查的噪音，同时专注于特定的一组页面。

完整应用程序的类型检查仍会在 `next build` 期间运行，或者可以由您的编辑器处理。

### 遥测 (Telemetry) (#telemetry)

Next.js 发布已经快 3 年了，框架在这 3 年中有了相当大的增长，从新功能到为所有用户提供更好的默认值。

我们一直采用的改进过程方法非常手动。

Vercel 有几个大型的 Next.js 应用程序。例如， [vercel.com](https://vercel.com), [vercel.com/docs](https://vercel.com/docs), 和 [https://nextjs.org](https://nextjs.org)。我们一直在 Vercel 内部试用 Next.js (dogfooding)，并根据我们的经验改进了 Next.js。

除此之外，我们积极与合作社区互动以收集反馈。很有可能您之前曾与 Tim 交谈过，提供关于您在公司如何使用 Next.js 的反馈。

例如，如果您使用自定义服务器，如果您有自定义 webpack 配置，等等。这些反馈对于指导 Next.js 的功能开发具有极高的价值。

然而，这种方法有一个问题，那就是我们只能从一部分用户那里收集反馈。这一部分人可能具有与您/您的公司不同的需求和用例。

例如导入 CSS 文件，这是非标准的，但似乎有很大一部分用户在使用它，无论是通过 `next-css` (或 Sass/Less)，还是自定义配置。如果我们知道使用该方法的用户百分比，我们就可以优先考虑改进它。

出于这个原因，我们引入了一种匿名、更自动化的方法来收集这些反馈点，以便我们在不久的将来进一步改进 Next.js。

这也将允许我们验证对框架所做的改进是否正在改善所有应用程序的基线。

您可以在 [nextjs.org/telemetry](https://nextjs.org/telemetry) 上阅读更多关于遥测的信息。在那里您还可以找到如何可选地退出，如果您不想参与的话。

### 改进的 `next/head` 元素跟踪 (#improved-nexthead-element-tracking)

Next.js 提供了一种内置方法来管理 `<head>` 元素，因为它负责渲染应用程序的 HTML。此 API 通过 `next/head` 模块公开。

```
import Head from 'next/head';
 
export default function MyPage() {

    <>

        <title>My Title</title>

      <h1>Hello World</h1>
    </>
  );
}
```

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Head from 'next/head';
 
export default function MyPage() {
  return (
    <>
      <Head>
        <title>My Title</title>
      </Head>
      <h1>Hello World</h1>
    </>
  );
}
```

当渲染到 HTML 时，Next.js 将收集所有 `<Head>` 中渲染的组件，并将标签渲染到页面的 `<head>` 中。

然而，Next.js 允许使用 `<Link>` 组件进行单页应用程序 (SPA) 类型的路由转换。

当您点击 `<Link>` 时，Next.js 将获取渲染页面所需的 JavaScript 文件。然后，它将渲染与该文件关联的 React 组件。

由于此转换发生在客户端，我们必须清理从上一个页面注入的 `<head>` 元素，否则另一个页面上可能会有过时的元素。

以前，Next.js 通过为每个 `<Head>` 提供的元素添加类名来跟踪这些元素。

以上述示例为例，`<head>` 将如下所示：

```
<head>
  <title class="next-head">My Title</title>
</head>
```

此解决方案工作良好，因为通过 `next/head` 注入的每个元素都有明显标记，易于清理。

然而，一小部分用户报告说，元素上额外的 `class` 属性有时会导致从外部服务添加的脚本无法通过验证。

Google Chrome 团队的 [Gerald Monaco](https://github.com/devknoll) [提出了一个方法](https://github.com/vercel/next.js/pull/8020)，可以在不需要元素类名的情况下保留清理行为。

新行为是 Next.js 将插入一个额外的 `<meta>` 标签，其中包含它 (`next/head`) 渲染的元素数量。通过这一点，Next.js 可以使用此数量来清理现有元素。

结果，当多个元素被注入到页面的 `<head>` 时，此方法减少了初始 HTML 有效载荷的大小。

### 防止 Pages 目录中的非页面 (#preventing-non-pages-in-the-pages-directory)

开始使用 Next.js 时，做的第一件事就是创建一个 `pages` 目录。

约定是 `pages` 目录中的每个文件都将成为应用程序中的一个路由。一个简单的例子是 `pages/about.js` 变为 `/about`。

随着时间推移，我们收到了偶尔的报告，称用户的应用程序（大型和小型）构建性能较差。

经进一步调查，发现这些用户在其 `pages` 目录中创建了它们整个组件结构。

由于 `pages` 目录中的每个文件都被视为页面，因此每个组件都作为应用程序中的页面进行编译。这会造成大量的构建时开销，为无效页面生成 2 个以上的 JavaScript 文件。

此外，这还会部分影响 Next.js 决定生成代码分割块的方式。这是因为 Next.js 使用跨页面的库使用启发式方法。

因此，我们必须确保用户不要在其 Next.js 应用程序中引入这种陷阱。

Next.js 9 现在验证 `pages` 目录中的文件是否导出 React 组件。

实际上，这意味着 Next.js 将向您显示一条消息，警示您在 `pages` 目录中发现了潜在的非页面。

这鼓励用户将不是页面的文件移动到另一个目录。反过来，开发、构建和代码分割速度更快且更准确。

### 运行时改进 (#runtime-improvements)

Next.js 框架由许多部分组成。其中一部分是客户端运行时。此运行时负责处理水合 (hydration)、客户端路由等。

水合 (Hydration) 是此改进的焦点，是使服务器渲染或预渲染的 HTML 具有交互性所必需的。水合添加事件处理程序并调用生命周期方法，如 `useEffect()` 或 `componentDidMount`，使您的应用程序准备好供最终用户使用。

此外，Next.js 处理的内容多于基本的水合 —— 例如，设置客户端路由器、配置 `next/head` 以及通过 `next/dynamic` 加载额外的应用程序逻辑。

每个责任都有其自己的特定运行时部分。

在 `next/dynamic` 的情况下，Next.js 必须确保在服务器上渲染的延迟加载组件已准备好在客户端。每次使用 `next/dynamic` 都会生成额外的 JavaScript 包，这些文件必须在水合 (hydration) 之前加载，以避免水合不匹配。

以前，此运行时始终包含在 Next.js 运行时包中。现在，仅当您在使用 `next/dynamic` 的应用程序中使用它时，才会包含它。这意味着不使用 `next/dynamic` 的应用程序将下载、解析和执行的 JavaScript 更少。

### AppTree 支持 (#apptree-support)

React 生态系统中的某些库以非常特定的方式实现服务器端渲染。最值得注意的是，Apollo 的服务器端渲染解决方案，称为 `getDataFromTree`，通过渲染 React 树并针对找到的每个 `Query` 等待结果，然后再次重新渲染 React 树来实现。

默认情况下，Next.js 会将一些上下文值添加到 React 树中，例如可以使用 `useRouter` 读取的路由器。

`with-apollo` 示例过去渲染 React 树的方式是渲染 `<App>` 并试图手动填写缺失的属性。随着 React Context 的添加，这已不再可能，因为上下文提供者是一个单独的组件。

从 Next.js 9.0.4 开始，向 `getInitialProps` 中的上下文对象添加了名为 `AppTree` 的新属性。它是专门为了需要遍历整个 React 树的外部库（如 Apollo 的 `getDataFromTree`）的情况而添加的。

`with-apollo` 示例已 [更新以反映这些更改](https://github.com/vercel/next.js/blob/v9.0.4/examples/with-apollo/lib/with-apollo-client.js#L20-L34)。如果您已经在应用中实现了 Apollo，建议更新为 `AppTree` 方法，以便 `useRouter` 和未来添加的其他 API 能在您的 Next.js 应用程序中正常工作。

如果您不使用 Apollo 或类似的库，我们建议您尝试并避免使用 AppTree，因为 Next.js 团队通常不建议遍历 React 树。它增加了相当多的性能开销，因为 React 树被渲染了多次，而不是仅一次。

### `next-server` 包的移除 (#removal-of-the-next-server-package)

当我们一年多前开始优化 Next.js 以进行 serverless 部署时，我们创建了一个名为 `next-server` 的包。此包是实验性的，与 `next` 包一起发布。它从未公开文档化，但是创建一个尽可能小的 Next.js 服务器运行时的实验。

最终，该包取得了成功并确实使生产服务器运行时更小。然而，我们想出了一种创新的新方法，使用 Next.js 编译器 和 [静态分析](https://github.com/vercel/node-file-trace) 使运行时变得更小。

在此过程中，`next-server` 变得过时了，并被 [serverless 目标](/docs#serverless-deployment) 取代。此目标的输出比使用 `next-server` 包替代 `next` 具有更优化的输出。

虽然此包已过时且不能直接使用，但我们保留了它。这是因为它具有包之间共享的内部结构，移动代码将需要非微不足道的努力。

我们最近实现了这一努力，将代码从 `next-server` 移回了 `next` 包。这意味着 Next.js 框架的所有代码现在都生活在 `next` 包中。

这使得初学者和资深贡献者都更容易为 Next.js 做出贡献。现在有一个单个编译过程和统一的构建配置。以前，`next` 和 `next-server` 有单独的设置，以及关于每个包属于哪些代码的任意限制。

### 升级 Next.js (#upgrading-nextjs)
