Next.js 9.0 大约两个月前发布。在此期间，我们忙于处理 7 次虽然较小但相当重要的发布：[9.0.1](https://github.com/vercel/next.js/releases/tag/v9.0.1), [9.0.2](https://github.com/vercel/next.js/releases/tag/v9.0.2), [9.0.3](https://github.com/vercel/next.js/releases/tag/v9.0.3), [9.0.4](https://github.com/vercel/next.js/releases/tag/v9.0.4), [9.0.5](https://github.com/vercel/next.js/releases/tag/v9.0.5), [9.0.6](https://github.com/vercel/next.js/releases/tag/v9.0.6), and [9.0.7](https://github.com/vercel/next.js/releases/tag/v9.0.7)。

让我们深入了解这些发布为您的网站和应用程序带来了什么，绝对没有任何破坏性更改。

*   **[Windows 环境下并发性的改进](#improved-concurrency-in-windows-environments)**：`next build` 过程现在在 Windows 上更加可靠，并且可以更好地并行工作。
*   **[默认启用 Gzip 压缩](#gzip-compression-by-default)**：Gzip 压缩现在默认添加，以减少所需的优化步骤。
*   **[TypeScript 报告仅针对活动页面](#typescript-report-on-active-pages-only)**：内置的 TypeScript 支持被扩展为仅监视活动页面的更改，使其更快且更可靠。
*   **[遥测数据](#telemetry)**：将帮助我们关注 Next.js 中需要优化的部分，并验证优化是否产生了预期的效果。
*   **[改进的 `next/head` 元素跟踪](#improved-nexthead-element-tracking)**：`next-head` 类已被移除，这使得实现某些验证其实现的工具更容易。
*   **[防止 Pages 目录中出现非页面文件](#preventing-non-pages-in-the-pages-directory)**：防止意外发布非页面文件，从而提高您应用程序的安全性和 `next build` 时间。
*   **[运行时优化](#runtime-improvements)**：当不使用 Next.js 的某些部分时，例如 `next/dynamic`，它们将不再需要在运行时，从而使捆绑大小更小。

### Windows 环境下并发性的改进[](#improved-concurrency-in-windows-environments)

Next.js 在 `next build` 过程中许多地方都进行并发工作。主要用途是在并行中使用 Terser 最小化构建输出。

以前，这项工作是使用一个名为 `worker-farm` 的包在多个 CPU 上处理的。然而，我们注意到许多 Windows 用户使用了自定义 webpack 配置来禁用最小化。进一步检查后，我们发现 `worker-farm` 在 Windows 机器上不能一致地工作。

为了解决这个问题，我们从 `worker-farm` 迁移到了 `jest-worker`。这确保了在 macOS、Linux 和 Windows 机器上构建都可靠且一致。

`jest-worker`，如其名所示，是 Jest 测试运行器的一部分。它是 Jest 用于并行化测试用例的包。这意味着该包经过非常严格的测试、可靠且得到维护。

`jest-worker` 还支持 [`worker_threads`](https://github.com/facebook/jest/tree/master/packages/jest-worker#experimental-worker)，这是 Node 12 中的一个新功能。与 `child_process` 不同，`worker_threads` 可以共享内存 — 这意味着在新版 Node 上构建时间更快。

通过切换到 `jest-worker`，我们能够为 Windows 用户重新启用构建并发。

### 默认启用 Gzip 压缩[](#gzip-compression-by-default)

在调查公司为何使用自定义服务器时，我们发现最常见的原因是为了压缩。公司会添加一个名为 `compression` 的 Express Middleware，用于处理 HTTP 响应的 Gzip 压缩。

该中间件压缩响应，使得更少的字节通过网络传输到您的用户。通常，这应该由反向代理（例如 Nginx）处理。反向代理从单线程 Node 进程中移除 CPU 密集型工作。

然而，当检查 Web 上的 Next.js 使用情况时，我们发现大量百分比的公司没有配置任何压缩。

在 [Vercel](https://vercel.com) 等平台上，`gzip` 和 `brotli` 在代理级别自动处理。

在自建托管时，公司必须自行添加 gzip（通过 `compression` 或反向代理）。

从 Next.js 9.0.4 开始，使用 `next start` 或自定义 `server.js` 时包含 `gzip` 压缩。

`brotli` 支持 [即将到来](https://github.com/expressjs/compression/pull/150)，因为 Node.js 现在原生支持 Brotli。

如果您的应用程序已通过自定义服务器启用压缩，Next.js 将不会添加自己的压缩器。

Next.js 默认不包含 serverless 目标的压缩，因为使用 serverless 目标时，资产是单独上传的，不通过 Node.js 服务。

如果您部署在像 [Vercel](https://vercel.com) 这样处理压缩的平台上，则不需要任何更改。

### TypeScript 报告仅针对活动页面[](#typescript-report-on-active-pages-only)

Next.js 9 包含内置的 TypeScript 支持。所需要做的只是一个页面的重命名，从 `.js` 改为 `.tsx`。Next.js 将自动处理或通过其余引导的设置。

Next.js 还通过在开发进程并行运行 `tsc --watch` 来处理类型检查。在开发期间，Next.js 有一个称为 [on-demand entries](https://nextjs.org/blog/next-8#improved-on-demand-entries) 的概念。此功能仅编译您正在积极使用的页面。

截至 9.0.4，Next.js 现在仅报告通过按需条目正在积极编译的页面的类型错误。这减少了大量类型检查噪音，同时专注于特定的一组页面。

完整的类型检查是在 `next build` 期间运行的，也可以由/在您的编辑器中处理。

### 遥测数据[](#telemetry)

Next.js 发布快 3 年了，这 3 年来框架有了很大增长，从新功能到所有用户的更好默认值。

我们一直以来处理这个改进过程的方式非常手动。

Vercel 有几个大型 Next.js 应用程序。例如，[vercel.com](https://vercel.com), [vercel.com/docs](https://vercel.com/docs), and [https://nextjs.org](https://nextjs.org)。我们一直在 Vercel 内部 [采用狗粮 (Eating your own dog food)](https://en.wikipedia.org/wiki/Eating_your_own_dog_food) Next.js，并根据我们的经验改进了 Next.js。

除此之外，我们积极参与社区以收集反馈。很有可能您之前已经和 Tim 讨论过如何在您的公司提供 Next.js 的使用反馈。

例如，如果您使用自定义服务器，如果您有自定义 webpack 配置，等等。这些反馈对于引导 Next.js 的功能开发极其有价值。

然而，这种方法有一个问题，即我们只能从一部分用户那里收集反馈。这部分用户可能有与您/您公司不同的需求和用例。

一个例子是导入 CSS 文件，这是非标准的，但相当一部分用户似乎在使用它，要么通过 `next-css` (或 Sass/Less)，要么通过自定义配置。如果我们知道使用这种特定方法的用户百分比，我们就可以优先考虑改进它。

出于这个原因，我们引入了一种匿名、更自动化的方法，收集这些反馈点，以便我们在不久的将来能够进一步改进 Next.js。

这也允许我们验证对框架的改进是否提升了所有应用程序的基线。

您可以在 [nextjs.org/telemetry](https://nextjs.org/telemetry) 阅读有关遥测数据的更多详细信息。在那里，您还将找到如何可选择退出如果您不想参与。

在与 Next.js 用户交谈时，一个小的反馈项是 `next build` 有时看起来没做任何事情（主要是视觉上）。

为了解决这个问题，我们在运行 `next build` 时在控制台输出中添加了一个加载指示器。此输出提供了视觉指示，表明命令仍在运行，且进程未冻结。

我们计划扩展此构建输出，以在可能时显示构建的更多阶段。

### 改进的 `next/head` 元素跟踪[](#improved-nexthead-element-tracking)

Next.js 提供了一种内置方式来管理 `<head>` 元素，因为它负责呈现应用程序的 HTML。此 API 通过 `next/head` 模块公开。

例如，为页面添加标题：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Head from 'next/head';
 
export default function MyPage() {
  return (
    <>
      <Head>
        <title>My Title</title>
      </Head>
      <h1>Hello World</h1>
    </>
  );
}
```

当渲染到 HTML 时，Next.js 将收集在 `<Head>` 内渲染的所有组件，并将标签渲染到页面的 `<head>` 中。

但是，Next.js 允许使用 `<Link>` 组件进行单页应用程序 (SPA) 类型的路由转换。

当您点击 `<Link>` 时，Next.js 将获取渲染页面所需的 JavaScript 文件。然后，它将渲染与文件相关的 React 组件。

由于此转换发生在客户端，我们必须清理从上页注入的 `<head>` 元素，否则另一个页面上可能会有旧元素。

以前，Next.js 通过为每个由 `<Head>` 提供的元素添加类名来跟踪这些元素。

以上示例中，`<head>` 看起来像：

```
<head>
  <title class="next-head">My Title</title>
</head>
```

这个解决方案工作良好，因为通过 `next/head` 注入的每个元素都有清晰的标记，易于清理。

但是，一小部分用户报告称元素上的额外 `class` 属性有时会导致从外部服务添加的脚本验证失败。

[Gerald Monaco](https://github.com/devknoll)，来自 Google Chrome 团队，提出了 [一种方法](https://github.com/vercel/next.js/pull/8020) 来保留清理功能，而无需在元素上使用类名。

新行为是 Next.js 将插入一个额外的 `<meta>` 标签，其中包含它渲染的元素数量 (`next/head`)。有了这个，Next.js 可以使用计数来清理现有元素。

结果，当多个元素注入到页面的 `<head>` 时，这种方法减少了初始 HTML 数据包的大小。

### 防止 Pages 目录中出现非页面文件[](#preventing-non-pages-in-the-pages-directory)

开始使用 Next.js 时，您做的第一件事是创建一个 `pages` 目录。

约定是 `pages` 目录中的每个文件都成为应用程序中的一个路由。一个简单的例子是 `pages/about.js` 变成 `/about`。

随着时间的推移，我们收到偶发报告，称用户的应用程序（无论大小）都具有较差的构建性能。

进一步调查后，发现这些用户在 `pages` 目录中创建了他们的整个组件结构。

由于 `pages` 目录中的每个文件都被视为一个页面，每个组件都在应用程序中作为页面被编译。这导致了大量的构建时开销，为无效页面生成了 2 个以上的 JavaScript 文件。

此外，这会影响 Next.js 决定生成代码拆分块的方式。这是因为 Next.js 使用跨页面的库使用情况启发式规则。

因此，我们必须确保用户不将此陷阱引入他们的 Next.js 应用程序。

Next.js 9 现在验证 `pages` 目录中的文件是否导出了 React 组件。

在操作中，这意味着 Next.js 会向您显示一条消息，提醒您可能发现 `pages` 目录中有一个潜在的页面文件。

这鼓励用户将不是页面的文件移动到另一个目录。反过来，开发、生产和代码拆分更快且更准确。

### 运行时优化[](#runtime-improvements)

Next.js 框架由许多部分组成。其中之一是客户端运行时。此运行时处理数据获取、客户端路由等。

数据获取（这是此改进关注的重点）是使服务器渲染或预渲染 HTML 可交互所必需的。数据获取添加事件处理程序并调用生命周期方法，例如 `useEffect()` 或 `componentDidMount` — 使您的应用程序准备好供最终用户使用。

此外，Next.js 处理比基本数据获取更多内容 — 例如，设置客户端路由器，配置 `next/head`，并通过 `next/dynamic` 加载额外的应用程序逻辑。

这些责任中的每一个都有它们自己的特定运行时部分。

在 `next/dynamic` 的情况下，Next.js 必须确保在服务器上渲染的懒加载组件在客户端准备好。`next/dynamic` 的每次使用都会生成额外的 JavaScript 捆绑，这些文件必须在数据获取之前加载，以避免数据获取不匹配。

以前，此运行时始终包含在 Next.js 运行时捆绑包中。现在，只有当您在应用程序中使用 `next/dynamic` 时才包含它。这意味着不使用 `next/dynamic` 的应用程序下载的 JavaScript 更少，解析和执行的更少。

### AppTree 支持[](#apptree-support)

React 生态系统中的某些库以非常特定的方式实现服务器端渲染。最著名的是，Apollo 的服务器端渲染解决方案，称为 `getDataFromTree`，通过渲染 React 树工作，对于找到的每个 `Query`，它将等待结果，然后再次重新渲染 React 树。

默认情况下，Next.js 在 React 树中添加一些上下文值，例如可以使用 `useRouter` 读取的路由。

`with-apollo` 示例过去渲染 React 树的方式是通过渲染 `<App>` 并尝试手动填充缺失的属性。随着 React Context 的添加，这不再可能，因为上下文提供程序是一个单独的元素。

从 Next.js 9.0.4 开始，在 `getInitialProps` 的上下文对象中添加了一个名为 `AppTree` 的新属性。它 specifically 适用于外部库必须遍历整个 React 树的情况，例如 Apollo 的 `getDataFromTree`。

`with-apollo` 示例已 [更新以反映更改](https://github.com/vercel/next.js/blob/v9.0.4/examples/with-apollo/lib/with-apollo-client.js#L20-L34)。如果您已经在应用中实现了 Apollo，建议更新到 `AppTree` 方法，以便将来添加的 `useRouter` 和其他 API 在您的 Next.js 应用程序中能正常工作。

如果您不使用 Apollo 或类似的库，我们建议尽量避免使用 AppTree，因为 Next.js 团队通常不建议遍历 React 树。由于 React 树被渲染多次而不是一次，因此增加了大量的性能开销。

### 移除 `next-server` 包[](#removal-of-the-next-server-package)

当我们开始一年多前优化 Next.js 用于 serverless 部署时，我们创建了一个名为 `next-server` 的包。该包是实验性的，并随 `next` 包一起发布。它从未在公共中记录，但是一次尝试创建最小的 Next.js 服务器运行时的实验。

最终，该包成功了，确实使生产服务器运行时变小了。然而，我们想出了一个创新的新技术，通过 Next.js 编译器和 [静态分析](https://github.com/vercel/node-file-trace) 使运行时更小。

在此过程中，`next-server` 变得过时，被 [serverless 目标](/docs#serverless-deployment) 替代。此目标在使用 `next-server` 包作为 `next` 的替代品时具有更大的优化输出。

虽然此包已过时且无法直接使用，但我们保留了它。这是因为它具有在包之间共享的内部结构，移动代码需要大量的努力。

我们最近完成了这一努力，将代码从 `next-server` 移回了 `next` 包。这意味着 Next.js 框架的所有代码现在都存在于 `next` 包中。

这使得初学者和有经验贡献者都能更容易地为 Next.js 做出贡献。现在有一个单一的编译过程和统一的构建配置。以前，`next` 和 `next-server` 有单独的设置，以及代码属于每个包的限制。

### 升级 Next.js[](#upgrading-nextjs)

如果您的项目运行在旧版本的 Next.js 上，我们建议升级到 Next.js 9。

在大多数情况下，升级不需要任何更改。您可以遵循 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md#migrating-from-v8-to-v9) 以确保无缝升级体验。

我们要感谢所有社区贡献者自发布以来更新了指南。
