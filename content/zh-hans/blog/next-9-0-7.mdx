Next.js 9.0 大约于两个月前发布。此后，我们一直在忙于 7 次较小但相当重要的发布：[9.0.1](https://github.com/vercel/next.js/releases/tag/v9.0.1)，[9.0.2](https://github.com/vercel/next.js/releases/tag/v9.0.2)，[9.0.3](https://github.com/vercel/next.js/releases/tag/v9.0.3)，[9.0.4](https://github.com/vercel/next.js/releases/tag/v9.0.4)，[9.0.5](https://github.com/vercel/next.js/releases/tag/v9.0.5)，[9.0.6](https://github.com/vercel/next.js/releases/tag/v9.0.6)，以及 [9.0.7](https://github.com/vercel/next.js/releases/tag/v9.0.7)。

让我们深入了解一下这些发布为我们的网站和应用带来了什么，且没有任何破坏性变更。

*   **[Windows 环境下的并发改进](#improved-concurrency-in-windows-environments)**: `next build` 进程现在在 Windows 上更可靠，并能更好地并行化处理工作。
*   **[默认启用 Gzip 压缩](#gzip-compression-by-default)**: Gzip 压缩现在默认添加，以减少所需的优化步骤。
*   **[仅报告激活页面的 TypeScript](#typescript-report-on-active-pages-only)**: 内置的 TypeScript 支持仅限于监视激活页面的更改，使其更快更可靠。
*   **[遥测](#telemetry)**: 这将帮助我们专注于优化 Next.js 的哪些方面，并验证优化是否具有预期效果。
*   **[改进的 `next/head` 元素跟踪](#improved-nexthead-element-tracking)**: 移除了 `next-head` 类，这使得实现某些验证其实现的工具变得更容易。
*   **[防止 Pages 目录中包含非页面](#preventing-non-pages-in-the-pages-directory)**: 通过防止意外发布非页面来优化应用程序安全并减少 `next build` 时间。
*   **[运行时改进](#runtime-improvements)**: 当不使用 Next.js 的某些部分时，例如 `next/dynamic`，它们在运行时将不再需要，从而使打包体积更小。

### Windows 环境下的并发改进 [](#improved-concurrency-in-windows-environments)

Next.js 在 `next build` 过程中在许多地方执行并发工作。主要用途是并行使用 Terser 压缩构建输出。

以前，这项工作是通过 `worker-farm` 这个包在多个 CPU 上处理的。然而，我们注意到许多 Windows 用户使用了自定义的 webpack 配置禁用了压缩。经过进一步检查，我们发现 `worker-farm` 在 Windows 机器上不能一致地工作。

为了解决这个问题，我们从 `worker-farm` 迁移到了 `jest-worker`。这确保了构建在 macOS、Linux 和 Windows 机器上都可靠且一致。

`jest-worker`，如其名称所示，是 Jest 测试运行器的一部分。它是 Jest 用于并行化测试用例的包。这意味着这个包经过了充分的实战测试，可靠且由团队维护。

`jest-worker` 还支持 [`worker_threads`](https://github.com/facebook/jest/tree/master/packages/jest-worker#experimental-worker)，这是 Node 12 中的新特性。与 `child_process` 不同，`worker_threads` 可以共享内存——这意味着在新版本的 Node 上构建速度更快。

通过切换到 `jest-worker`，我们能够重新启用 Windows 用户的构建并发。

### Gzip 压缩默认开启 [](#gzip-compression-by-default)

在调查公司为什么使用自定义服务器时，我们发现大多数情况下是为了压缩。公司会添加一个名为 `compression` 的 Express 中间件，用于处理 HTTP 响应的 Gzip 压缩。

这个中间件会压缩响应，从而减少发送给用户的字节数。通常，这应该由 Nginx 这样的反向代理来处理。反向代理可以从单线程 Node 进程中移除繁重的 CPU 工作。

然而，在检测网络上 Next.js 的使用情况时，我们发现很大比例的公司没有配置任何压缩。

在像 [Vercel](https://vercel.com) 这样的平台上，`gzip` 和 `brotli` 会在代理级别自动处理。

当自行托管时，公司必须自行添加 gzip（通过 `compression` 或反向代理）。

从 Next.js 9.0.4 开始，使用 `next start` 或自定义 `server.js` 时，默认包含 `gzip` 压缩。

`brotli` 支持 [即将到来](https://github.com/expressjs/compression/pull/150)，因为 Node.js 现在原生支持 Brotli。

如果压缩已通过自定义服务器在您的应用程序中启用，Next.js 将不会添加自己的压缩器。

Next.js 默认不为 serverless 目标包含压缩，因为使用 serverless 目标时，资源是单独上传的，而不是通过 Node.js 服务的。

如果您在像 [Vercel](https://vercel.com) 这样处理压缩的平台上部署，则无需更改。

### 仅报告激活页面的 TypeScript[](#typescript-report-on-active-pages-only)

Next.js 9 包含了内置的 TypeScript 支持。只需要将一个页面从 `.js` 重命名为 `.tsx`。Next.js 将自动处理或指导您完成其余的设置。

Next.js 还通过并行于开发进程运行 `tsc --watch` 来处理类型检查。在开发期间，Next.js 有一个概念称为 [按需入口](https://nextjs.org/blog/next-8#improved-on-demand-entries)。该功能仅编译您正在积极工作的页面。

从 9.0.4 开始，Next.js 现在仅报告正在被按需入口积极编译的页面的类型错误。这减少了大量类型检查噪音，同时专注于特定的一组页面。

完整的类型检查仍会在 `next build` 期间运行，或者可以在/由您的编辑器处理。

### 遥测 [](#telemetry)

Next.js 几乎在 3 年前发布，并且在该 3 年间框架有了显著增长，从新功能到为所有用户提供更好的默认值。

我们处理这种改进的方式一直非常手工化。

Vercel 有一些大型的 Next.js 应用程序。例如，[vercel.com](https://vercel.com)，[vercel.com/docs](https://vercel.com/docs)，以及 [https://nextjs.org](https://nextjs.org)。我们一直在 Vercel 内部 [吃狗粮 (dogfooding)](https://en.wikipedia.org/wiki/Eating_your_own_dog_food)Next.js，并根据我们的经验改进了 Next.js。

除此之外，我们积极与社区互动收集反馈。您之前很可能与 Tim 谈过，以提供反馈说明您在公司如何使用 Next.js。

例如，如果您使用自定义服务器，如果您有自定义 webpack 配置等。这些反馈对于指导 Next.js 的功能开发非常有价值。

然而，这种方法有一个问题，即我们只能从一部分用户那里收集反馈。这部分用户可能有不同于您/您公司的需求和用例。

一个例子是导入 CSS 文件，这是非标准的，但似乎相当大一部分用户在使用这种方式，无论是通过 `next-css`（或 Sass/Less）还是自定义配置。如果我们知道使用该特定方法的用户百分比，我们可以优先考虑改进它。

出于这个原因，我们引入了一种匿名、更自动化的方法来收集这些反馈点，以便我们可以进一步在未来改进 Next.js。

这也使我们能够验证对框架的改进是否改善了所有应用程序的基线。

您可以在 [nextjs.org/telemetry](https://nextjs.org/telemetry) 了解更多关于遥测的信息。您还将找到如何可选地退出如果您不想参与。

### 改进的 `next/head` 元素跟踪 [](#improved-nexthead-element-tracking)

Next.js 提供了一种内部管理 `<head>` 元素的方法，因为它负责渲染应用程序的 HTML。此 API 通过 `next/head` 模块公开。

例如，为页面添加标题：

取而代之的是，Next.js 会自动注入 CSS 文件。在生产环境中，Next.js 还自动在 CSS URL 中添加内容哈希，因此如果有更改，可确保您的最终用户永远不会获得文件的过时版本，并且具有引入不可变永久缓存的能力。

```
import Head from 'next/head';
 
export default function MyPage() {
  return (
    <>
      <Head>
        <title>My Title</title>
      </Head>
      <h1>Hello World</h1>
    </>
  );
}
```

在渲染为 HTML 时，Next.js 将收集 `<Head>` 内渲染的所有组件，并将标签渲染到页面的 `<head>` 中。

然而，Next.js 允许使用 `<Link>` 组件进行单页应用程序 (SPA) 类型的路由转换。

当您点击 `<Link>` 时，Next.js 将获取所需的 JavaScript 文件以在客户端端渲染页面。然后，它将渲染与该文件关联的 React 组件。

因为这种转换发生在客户端，所以我们需要清理从前一页面注入的 `<head>` 元素，否则过时元素可能在另一页面上存在。

以前，Next.js 通过在每个 `<Head>`提供的元素上添加类名来跟踪这些元素。

以上面的示例为例，`<head>` 看起来像：

```
<head>
  <title class="next-head">My Title</title>
</head>
```

此解决方案效果很好，因为 `next/head` 注入的每个元素都有明确标记且易于清理。

然而，一小部分用户报告说，元素上的额外 `class` 属性有时会导致从外部服务添加的脚本无法通过验证。

[Gerald Monaco](https://github.com/devknoll)，来自 Google Chrome 团队，[提出了一个方法](https://github.com/vercel/next.js/pull/8020) 在不需要在元素上使用类名的情况下保留清理行为。

新行为是 Next.js 将插入一个额外的 `<meta>` 标签，其中包含它 (`next/head`) 渲染的元素数量。借此，Next.js 可以使用此数量来清理现有元素。

因此，这种方法减少了多个元素注入到页面 `<head>` 时的初始 HTML 负载大小。

### 防止 Pages 目录中包含非页面 [](#preventing-non-pages-in-the-pages-directory)

开始使用 Next.js 时，您做的第一件事是创建一个 `pages` 目录。

约定是将 `pages` 目录中的每个文件变为应用程序中的一个路由。一个简单的例子是 `pages/about.js` 变为 `/about`。

随着时间的推移，我们偶尔收到用户的应用程序，无论大小区，构建性能不佳的报告。

经过进一步调查，发现这些用户将整个组件结构创建在 `pages` 目录中。

由于 `pages` 目录中的每个文件都被视为页面，每个组件都会被编译为应用程序中的页面。这会导致大量的构建时间开销，为无效的页面生成 2 个以上的 JavaScript 文件。

此外，这将部分影响 Next.js 决定生成代码块分割的方式。这是因为 Next.js 使用关于跨页面库使用的启发式方法。

因此，我们必须确保用户不要将这种陷阱引入他们的 Next.js 应用程序。

Next.js 9 现在验证 `pages` 目录内的文件是否导出 React 组件。

在操作中，这意味着 Next.js 将向您显示一条消息，警告您在 `pages` 目录中发现了潜在的非页面。

这鼓励用户将非页面的文件移动到另一个目录。随之，开发、生产和代码分割更快更准确。

### 运行时改进 [](#runtime-improvements)

Next.js 框架由许多部分组成。其中一个是客户端运行时间。该运行时处理水合 (hydration)，客户端端路由等。

水合，这是此次改进的重点，是使服务器渲染或预渲染的 HTML 具有交互性的必要条件。水合添加事件处理器并调用生命周期方法，如 `useEffect()` 或 `componentDidMount`——使您的应用程序为最终用户做好准备。

此外，Next.js 处理比基本水合更多的内容——例如，设置客户端端路由器，配置 `next/head`，以及通过 `next/dynamic` 加载额外的应用程序逻辑。

这些职责中的每一个也有其自己的特定运行时部分。

在 `next/dynamic` 的情况下，Next.js 必须确保在服务端渲染的延迟加载组件在客户端端就绪。每次使用 `next/dynamic` 都会生成额外的 JavaScript 包，并且必须在渲染水合之前加载这些文件，以避免渲染水合不匹配 (hydration mismatch)。

以前，此运行时会始终包含在 Next.js 运行时包中。现在，仅在您在应用程序中使用 `next/dynamic` 时才包含它。这意味着不使用 `next/dynamic` 的应用程序下载、解析和执行的 JavaScript 更少。

### AppTree 支持 [](#apptree-support)

React 生态系统中的某些库以非常特定的方式实现服务端渲染。最值得注意的是，Apollo 的服务端渲染解决方案，称为 `getDataFromTree`，通过渲染 React 树工作，对于找到的每个 `Query`，它将等待结果，然后再次渲染 React 树。

默认情况下，Next.js 会在 React 树中添加一些上下文值，例如可以使用 `useRouter` 读取的路由器。

之前 `with-apollo` 示例渲染 React 树的方式是通过渲染 `<App>` 并尝试手动填充缺失的属性。随着 React Context 的添加，这不再可能，因为上下文提供程序是一个单独的组件。

从 Next.js 9.0.4 开始，在 `getInitialProps` 的上下文对象中添加了一个名为 `AppTree` 的新属性。它是专门为外部库必须遍历整个 React 树的情况而添加的，就像 Apollo 的 `getDataFromTree`。

`with-apollo` 示例已 [更新以反映更改](https://github.com/vercel/next.js/blob/v9.0.4/examples/with-apollo/lib/with-apollo-client.js#L20-L34)。如果您已经在应用程序中实现了 Apollo，建议更新到 `AppTree` 方法，以便将来添加的 `useRouter` 和其他 API 将在您的 Next.js 应用程序中正常运行。

如果您不使用 Apollo 或类似的库，我们建议努力避免使用 AppTree，因为 Next.js 团队通常不建议遍历 React 树。它增加了相当多的性能开销，因为 React 树被多次渲染而不是仅仅一次。

### 移除 `next-server` 包 [](#removal-of-the-next-server-package)

当我们开始通过一年多来优化 Next.js 以用于 serverless 部署时，我们创建了一个名为 `next-server` 的包。该包是实验性的，与 `next` 包一起发布。它从未在公共场合文档化，但是一种实验，旨在创建最小的可能 Next.js 服务器运行时。

最终，该包取得了成功，确实使生产服务器运行时变小。然而，我们想出了一个创新的新方法，通过 Next.js 编译器和 [静态分析](https://github.com/vercel/node-file-trace) 使运行时更小。

在此过程中，`next-server` 变得过时并被 [serverless 目标](/docs#serverless-deployment) 取代。此目标比使用 `next-server` 包作为 `next` 的替代品具有更优化的输出。

虽然此包已过时且不能直接使用，但我们将其保留了下来。这是因为它具有在包之间共享的内部结构，移动代码需要不小的精力。

我们最近进行了这项努力，并将代码从 `next-server` 移回了 `next` 包。这意味着 Next.js 框架的所有代码现在都存在于 `next` 包中。

这使得新手和有经验的贡献者都能更容易地为 Next.js 做出贡献。现在有一个统一的编译过程和构建配置。以前，`next` 和 `next-server` 有单独的设置，以及任意约束规定哪些代码属于哪个包。

### 升级 Next.js [](#upgrading-nextjs)

如果您的项目运行在 Next.js 的旧版本上，我们建议升级到 Next.js 9。

在大多数情况下，升级无需更改。您可以遵循 [升级指南](https://github.com/vercel/next.js/blob/canary/UPGRADING.md#migrating-from-v8-to-v9) 以确保升级体验顺畅。

我们要感谢所有社区贡献者在指南发布后更新了它。
