使用 `@next/codemod` CLI 平滑升级 [](#smooth-upgrades-with-nextcodemod-cli)
----------------------------------------------------------------------------------

我们在每个主要 Next.js 版本发布中都包含 codemod（自动代码转换），以帮助自动化升级破坏性更改。

为了让升级更加平滑，我们发布了增强的 codemod CLI：

终端

```
npx @next/codemod@canary upgrade latest
```

此工具可帮助您将代码库升级到最新的稳定版或预发布版本。CLI 将更新您的依赖项、显示可用的 codemod，并指导您应用它们。

`canary` 标签使用 codemod 的最新版本，而 `latest` 指定 Next.js 版本。我们建议您使用 codemod 的 canary 版本，即使您要升级到最新的 Next.js 版本，因为我们计划根据您的反馈继续改进工具。

了解有关 [Next.js codemod CLI](/docs/app/building-your-application/upgrading/codemods) 的更多信息。

## Async 请求 APIs（破坏性更改）[](#async-request-apis-breaking-change)

在传统的服务端渲染 (Server-Side Rendering) 中，服务器在渲染任何内容之前会等待请求。然而，并非所有组件都依赖特定请求的数据，因此无需等待请求即可渲染它们。理想情况下，服务器会在请求到达之前尽可能多地做好准备。为了实现这一点，并为未来的优化铺平道路，我们需要知道何时等待请求。

因此，我们将依赖特定请求数据的 API——如 `headers`, `cookies`, `params` 和 `searchParams`——过渡为**异步**。

```
import { cookies } from 'next/headers';
 
export async function AdminPanel() {
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
 
  // ...
}
```

这是一个**破坏性更改**，影响以下 APIs：

*   `cookies`
*   `headers`
*   `draftMode`
*   `params` (在 `layout.js`, `page.js`, `route.js`, `default.js`, `generateMetadata` 和 `generateViewport` 中)
*   `searchParams` (在 `page.js` 中)

为了更容易地进行迁移，这些 API 可以暂时同步访问，但在下一个主要版本之前，在开发和生产环境中会显示警告。有一个 [codemod](/docs/app/building-your-application/upgrading/codemods) 可用于自动迁移：

终端

```
npx @next/codemod@canary next-async-request-api .
```

如果 codemod 无法完全迁移您的代码，请阅读 [升级指南](/docs/app/building-your-application/upgrading/version-15)。我们还提供了 [示例](https://github.com/leerob/next-saas-starter/pull/62) 说明如何将 Next.js 应用程序迁移到新的 API。

缓存语义 [](#caching-semantics)
---------------------------------------

Next.js App Router 推出时采用了特定的缓存默认策略。这些策略旨在默认提供性能最佳的选项，并能够在需要时选择不加入。

根据您的反馈，我们重新评估了 [缓存启发式方法](https://x.com/feedthejim/status/1785242054773145636) 以及它们如何将部分预渲染 (PPR) 和使用 `fetch` 的第三方库等项目相互作用。

随着 Next.js 15 的发布，我们正在将 `GET` 路由处理器和客户端路由缓存的缓存默认值从默认缓存更改为默认不缓存。如果您想保留以前的行为，可以继续选择启用缓存。

在接下来的几个月里，我们将继续改进 Next.js 中的缓存，并将很快分享更多细节。

### GET 路由处理器默认不再被缓存[](#get-route-handlers-are-no-longer-cached-by-default)

在 Next 14 中，使用 `GET` HTTP 方法的路由处理器默认被缓存，除非它们使用了动态函数或动态配置选项。在 Next.js 15 中，`GET` 函数默认 **不被缓存**。

您仍然可以选择加入缓存，使用静态路由配置选项，例如 `export dynamic = 'force-static'`。

特殊路由处理器如 [`sitemap.ts`](/docs/app/api-reference/file-conventions/metadata/sitemap)、[`opengraph-image.tsx`](/docs/app/api-reference/file-conventions/metadata/opengraph-image) 和 [`icon.tsx`](/docs/app/api-reference/file-conventions/metadata/app-icons)，以及其他 [元数据文件](/docs/app/api-reference/file-conventions/metadata) 默认保持静态，除非它们使用动态函数或动态配置选项。

### 客户端路由器缓存默认不再缓存页面组件[](#client-router-cache-no-longer-caches-page-components-by-default)

在 Next.js 14.2.0 中，我们引入了实验性的 [`staleTimes`](/docs/app/api-reference/next-config-js/staleTimes) 标志，以允许自定义 [路由缓存](/docs/app/building-your-application/caching#client-side-router-cache)。

在 Next.js 15 中，此标志仍然可访问，但我们正在更改默认行为，使页面段的 `staleTime` 为 `0`。这意味着，当您在应用中导航时，客户端将始终反映作为导航一部分变为活动的页面组件的最新数据。但是，仍有一些重要的行为保持不变：

*   共享布局数据将不会从服务器重新获取，以继续支持 [部分渲染](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)。
*   向后/向前导航仍将从缓存恢复，以确保浏览器可以恢复滚动位置。
*   [`loading.js`](/docs/app/api-reference/file-conventions/loading) 将缓存 5 分钟（或 `staleTimes.static` 配置的值）。

您可以通过以下配置选择加入以前的客户端路由器缓存行为：

next.config.ts

```
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
    },
  },
};
 
export default nextConfig;
```

React 19[](#react-19)
---------------------

作为 Next.js 15 发布的一部分，我们做出了与即将发布的 React 19 保持一致的决定。

在 15 版本中，App Router 使用 React 19 RC，并且我们根据社区反馈也引入了对 React 18 的后向兼容性（基于 Pages Router）。如果您使用 Pages Router，这允许您在准备好时升级到 React 19。

虽然 React 19 仍处于 RC 阶段，但我们在实际应用程序中的广泛测试以及与 React 团队的密切合作让我们对其稳定性充满信心。核心破坏性变更已经过充分测试，不会影响现有的 App Router 用户。因此，我们决定现在发布 Next.js 15 为稳定版，以便您的项目为 React 19 GA 做好充分准备。

为了确保过渡尽可能顺利，我们提供了 [codemods 和自动工具](#smooth-upgrades-with-codemod-cli) 以帮助减轻迁移过程。

阅读 [Next.js 15 升级指南](/docs/app/building-your-application/upgrading/version-15)、[React 19 升级指南](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)，并观看 [React Conf 主题演讲](https://www.youtube.com/live/T8TZQ6k4SLE?t=1788) 以了解更多信息。

### Pages Router 在 React 18 上 [](#pages-router-on-react-18)

Next.js 15 保持了 Pages Router 与 React 18 的后向兼容性，允许用户继续使用 React 18，同时从 Next.js 15 的改进中受益。

自第一个发布候选版本 (RC1) 以来，我们根据社区反馈将重点转向包括对 React 18 的支持。这种灵活性使您能够在使用 React 18 的 Pages Router 时采用 Next.js 15，从而让您对自己的升级路径拥有更大的控制权。

> **注意：** 虽然在同一个应用程序中在 React 18 上运行 Pages Router 和在 React 19 上运行 App Router 是可能的，但我们不推荐这种设置。这样做可能会导致不可预测的行为和类型不一致，因为两个版本之间的底层 API 和渲染逻辑可能不完全一致。

### React 编译器（实验性）[](#react-compiler-experimental)

[React 编译器 (React)](https://react.dev/learn/react-compiler) 是由 Meta React 团队创建的一个新的实验性编译器。该编译器通过其对纯 JavaScript 语义和 [React 规则](https://react.dev/reference/rules) 的理解，深入理解您的代码，从而能够为您的代码添加自动优化。该编译器减少了开发人员通过 `useMemo` 和 `useCallback` 等 API 进行手动 memoization 的数量——使代码更简单、更容易维护，并且更少出错。

随着 Next.js 15 的发布，我们添加了对 [React 编译器](https://react.dev/learn/react-compiler) 的支持。了解有关 React 编译器以及 [可用的 Next.js 配置选项](https://react.dev/learn/react-compiler#usage-with-nextjs) 的更多信息。

> **注意：** React 编译器目前仅作为 Babel 插件提供，这将导致开发和构建时间变慢。

在响应完成后使用 `unstable_after` 执行代码（实验性） [](#executing-code-after-a-response-with-unstable_after-experimental)
--------------------------------------------------------------------------------------------------------------------------------------------

在处理用户请求时，服务器通常会执行与计算响应直接相关的任务。但是，您可能需要执行记录日志、分析以及其他外部系统同步等任务。

由于这些任务与响应没有直接关系，因此用户应该不需要等待它们完成。在响应后延迟工作是一个挑战，因为 serverless 函数在响应关闭后立即停止计算。

`after()` 是一个新的实验性 API，通过允许您调度工作以便在响应完成后处理该问题，从而使次要任务不会阻塞主要响应。

要使用它，请在 `next.config.js` 中添加 `experimental.after`：

next.config.ts

```
const nextConfig = {
  experimental: {
    after: true,
  },
};
 
export default nextConfig;
```

然后，在 Server Components、Server Actions、Route Handlers 或 Middleware 中导入该函数。

```
import { unstable_after as after } from 'next/server';
import { log } from '@/app/utils';
 
export default function Layout({ children }) {
  // Secondary task
  after(() => {
    log();
  });
 
  // Primary task
  return <>{children}</>;
}
```

了解有关 [`unstable_after`](/docs/app/api-reference/functions/unstable_after) 的更多信息。

## `instrumentation.js`（稳定）[](#instrumentationjs-stable)

`instrumentation` 文件具有 `register()` API，允许用户切入 Next.js 服务器生命周期以监控性能、跟踪错误来源以及深度集成到像 [OpenTelemetry](https://opentelemetry.io/) 这样的可观测性库中。

此功能现在**稳定**，可以移除 `experimental.instrumentationHook` 配置选项。

此外，我们与 [Sentry](https://sentry.io/) 合作设计了一个新的 `onRequestError` 钩子，可用于：

*   捕获服务器上传抛出的所有错误的重要上下文，包括：
    *   Router：Pages Router 或 App Router
    *   Server context：Server Component、Server Action、Route Handler 或 Middleware
*   向您喜爱的可观测性提供商报告错误。

```
export async function onRequestError(err, request, context) {
  await fetch('https://...', {
    method: 'POST',
    body: JSON.stringify({ message: err.message, request, context }),
    headers: { 'Content-Type': 'application/json' },
  });
}
 
export async function register() {
  // init your favorite observability provider SDK
}
```

了解有关 `onRequestError` [函数](/docs/app/api-reference/file-conventions/instrumentation#onrequesterror-optional) 的更多信息。

## `<Form>` 组件[](#form-component)

新的 `<Form>` 组件通过 [预取](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)、[客户端导航](/docs/app/building-your-application/routing/linking-and-navigating#5-soft-navigation) 和渐进增强来扩展 HTML `<form>` 元素。

它适用于导航到新页面的表单，例如引导至结果页面的搜索表单。

app/page.jsx

```
import Form from 'next/form';
 
export default function Page() {
  return (
    <Form action="/search">
      <input name="query" />
      <button type="submit">Submit</button>
    </Form>
  );
}
```

`<Form>` 组件带有：

*   **预取：** 当表单在视图中时，[布局](/docs/app/api-reference/file-conventions/layout) 和 [加载](/docs/app/api-reference/file-conventions/loading) UI 会被预取，使导航快速。
*   **客户端导航：** 提交时，共享布局和客户端状态保持不变。
*   **渐进增强：** 如果 JavaScript 尚未加载，表单仍可通过整页导航工作。

以前，实现这些功能需要大量手动样板代码。例如:

### 示例

```
// Note: This is abbreviated for demonstration purposes.
// Not recommended for use in production code.
 
'use client'
 
import { useEffect } from 'react'
import { useRouter } from 'next/navigation'
 
export default function Form(props) {
  const action = props.action
  const router = useRouter()
 
  useEffect(() => {
    // if form target is a URL, prefetch it
    if (typeof action === 'string') {
      router.prefetch(action)
    }
  }, [action, router])
 
  function onSubmit(event) {
    event.preventDefault()
 
    // grab all of the form fields and trigger a `router.push` with the data URL encoded
    const formData = new FormData(event.currentTarget)
    const data = new URLSearchParams()
 
    for (const [name, value] of formData) {
      data.append(name, value as string)
    }
 
    router.push(`${action}?${data.toString()}`)
  }
 
  if (typeof action === 'string') {
    return <form onSubmit={onSubmit} {...props} />
  }
 
  return <form {...props} />
}
```

了解有关 [`<Form>` 组件](/docs/app/api-reference/components/form) 的更多信息。

## 自托管改进[](#improvements-for-self-hosting)

当自托管应用时，您可能需要更多的 `Cache-Control` 指令控制权。

一个常见的用例是控制 ISR 页面发送的 `stale-while-revalidate` 周期。我们实施了两个改进：

1.  您现在可以在 `next.config` 中配置 [`expireTime`](/docs/app/api-reference/next-config-js/expireTime) 值。这之前是 `experimental.swrDelta` 选项。
2.  更新默认值为一年，确保大多数 CDN 可以按照预期完全应用 `stale-while-revalidate` 语义。

我们也不再覆盖自定义 `Cache-Control` 值以使用我们的默认值，实现完全控制并确保与任何 CDN 设置兼容。

最后，我们改进了自托管时的图像优化。以前，我们建议您安装 `sharp` 来优化 Next.js 服务器上的图像。这个建议有时会被忽略。随着 Next.js 15，您不再需要手动安装 `sharp`——在使用 `next start` 或运行 [standalone output mode](/docs/app/api-reference/next-config-js/output) 时，Next.js 将自动使用 `sharp`。

有关更多信息，请参阅我们关于自托管 Next.js 的新 [演示和教程视频](https://x.com/leeerob/status/1843796169173995544)。

## Server Actions 增强安全性[](#enhanced-security-for-server-actions)

[Server Actions](https://react.dev/reference/rsc/server-actions) 是可从客户端调用的服务器端函数。通过在文件顶部添加 `'use server'` 指令并导出一个异步函数来定义它们。

即使 Server Action 或实用功能未在代码的其他地方导入，它仍然是一个可公开访问的 HTTP 端点。虽然此行为在技术上正确，但可能会导致此类功能的意外暴露。

为了提高安全性，我们引入了以下增强功能：

*   **死代码消除：** 未使用的 Server Action 不会将其 ID 暴露给客户端 JavaScript 包，从而减少包大小并提高性能。
*   **安全的操作 ID：** Next.js 现在创建不可猜测的、非确定性的 ID，允许客户端引用并调用 Server Action。这些 ID 在构建之间会定期重新计算以增强安全性。

```
// app/actions.js
'use server';
 
// This action **is** used in our application, so Next.js
// will create a secure ID to allow the client to reference
// and call the Server Action.
export async function updateUserAction(formData) {}
 
// This action **is not** used in our application, so Next.js
// will automatically remove this code during `next build`
// and will not create a public endpoint.
export async function deleteUserAction(formData) {}
```

您仍应将 Server Action 视为公共 HTTP 端点。了解有关 [保护 Server Action](/blog/security-nextjs-server-components-actions#write) 的更多信息。

## 开发和构建改进[](#development-and-build-improvements)

### Server Components HMR[](#server-components-hmr)

开发期间，Server Components 会在保存时重新执行。这意味着，任何对您 API 端点或第三方服务的 `fetch`请求也会被调用。为了提高本地开发性能并减少已计费 API 调用的潜在成本，我们确保热模块替换 (HMR) 可以重用先前渲染的 `fetch` 响应。

为了提高本地开发性能并减少已计费 API 调用的潜在成本，我们现在确保热模块替换 (HMR) 可以重用上次渲染中的 `fetch` 响应。

了解有关 [服务器组件 HMR 缓存](/docs/app/api-reference/next-config-js/serverComponentsHmrCache) 的更多信息。

### App Router 更快的静态生成[](#faster-static-generation-for-the-app-router)

我们优化了静态生成以提高构建时间，特别是对于那些具有缓慢网络请求的页面。

此前，我们的静态优化过程渲染页面两次——一次用于生成客户端导航的数据，第二次用于渲染初始页面访问的 HTML。现在，我们重用第一次渲染，取消了第二次遍历，减少了工作负载和构建时间。

此外，静态生成工作线程现在在页面之间共享 `fetch` 缓存。如果 `fetch` 调用不选择禁用缓存，那么它将由同一工作线程处理的其他页面重用结果。这减少了对相同数据的请求次数。

### 高级静态生成控制（实验）[](#advanced-static-generation-control-experimental)

我们为高级用例添加了对静态生成过程更多控制的实验性支持，这将受益于更大的控制。

除非您有特定需求，否则建议坚持使用当前的默认值，因为这些选项可能会导致资源使用增加以及由于并发增加而导致的潜在内存不足错误。

next.config.ts

```
const nextConfig = {
  experimental: {
    // how many times Next.js will retry failed page generation attempts
    // before failing the build
    staticGenerationRetryCount: 1
    // how many pages will be processed per worker
    staticGenerationMaxConcurrency: 8
    // the minimum number of pages before spinning up a new export worker
    staticGenerationMinPagesPerWorker: 25
  },
}
 
export default nextConfig;
```

了解有关 [静态生成选项](/docs/app/api-reference/next-config-js/staticGeneration) 的更多信息。
