欢迎来到 Next.js 文档贡献指南！我们很高兴有你的加入。

本页面提供关于如何编辑 Next.js 文档的说明。我们的目标是确保社区中的每个人都能感到有力量贡献并改进我们的文档。

## 为什么要贡献？

开源工作永无止境，文档也是如此。为文档做贡献是初学者参与开源、经验丰富的开发者在分享知识的同时澄清更复杂主题的好方法。

通过为 Next.js 文档做贡献，你正在帮助我们为所有开发者构建一个更强大的学习资源。无论你发现了拼写错误、令人困惑的部分，还是意识到某个主题缺失，你的贡献都会受到欢迎和赞赏。

## 如何贡献

文档内容可以在 [Next.js 仓库](https://github.com/vercel/next.js/tree/canary/docs) 找到。要贡献，你可以直接在 GitHub 上编辑文件，或者克隆仓库并在本地编辑文件。

### GitHub 工作流程

如果你是 GitHub 新手，我们建议阅读 [GitHub 开源指南](https://opensource.guide/how-to-contribute/#opening-a-pull-request) 来学习如何 fork 仓库、创建分支和提交拉取请求。

> **提示**：底层文档代码位于一个私有代码库中，该代码库与 Next.js 公共仓库同步。这意味着你无法在本地预览文档。但是，合并拉取请求后，你会在 [nextjs.org](https://nextjs.org/docs) 上看到你的更改。

### 编写 MDX

文档用 [MDX](https://mdxjs.com/) 编写，这是一种支持 JSX 语法的 markdown 格式。这允许我们在文档中嵌入 React 组件。有关 markdown 语法的快速概述，请参阅 [GitHub Markdown 指南](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)。

### VSCode

#### 本地预览更改

VSCode 有一个内置的 markdown 预览器，你可以用它来在本地查看编辑。要为 MDX 文件启用预览器，你需要向用户设置添加一个配置选项。

打开命令面板（Mac 上按 `⌘ + ⇧ + P`，Windows 上按 `Ctrl + Shift + P`），然后搜索 `Preferences: Open User Settings (JSON)`。

然后，向你的 `settings.json` 文件添加以下行：

```json filename="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

接下来，再次打开命令面板，搜索 `Markdown: Preview File` 或 `Markdown: Open Preview to the Side`。这将打开一个预览窗口，你可以在其中查看格式化的更改。

#### 扩展

我们还为 VSCode 用户推荐以下扩展：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx)：为 MDX 提供智能感知和语法高亮。
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode)：保存时格式化 MDX 文件。

### 审查流程

提交贡献后，Next.js 或开发者体验团队将审查你的更改、提供反馈，并在准备好时合并拉取请求。

如果你有任何问题或需要进一步帮助，请在 PR 的评论中告诉我们。感谢你为 Next.js 文档做出贡献并成为我们社区的一部分！

> **提示**：在提交 PR 之前运行 `pnpm prettier-fix` 来运行 Prettier。

## 文件结构

文档使用 **文件系统路由**。 [`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 中的每个文件夹和文件代表一个路由段。这些段用于生成 URL 路径、导航和面包屑。

文件结构反映了你在网站上看到的导航，默认情况下，导航项目按字母顺序排序。但是，我们可以在文件夹或文件名前加上两位数（`00-`）来更改项目的顺序。

例如，在 [functions API 参考](/docs/app/api-reference/functions) 中，页面按字母顺序排序，因为这使开发者更容易找到特定的函数：

```txt
04-functions
├── after.mdx
├── cacheLife.mdx
├── cacheTag.mdx
└── ...
```

但是，在 [应用路由 (app router) 章节](/docs/app) 中，文件的前缀都是两位数，并按开发者学习这些概念的顺序排序：

```txt
01-getting-started
├── 01-installation.mdx
├── 02-project-structure.mdx
├── 03-layouts-and-pages.mdx
└── ...
```

若要快速查找页面，您可以使用 `⌘ + P` (Mac) 或 `Ctrl + P` (Windows) 打开 VSCode 中的搜索栏。然后，输入您正在查找的页面的 slug。例如：`installation`

> **为什么不使用清单？**
>
> 我们考虑过使用清单文件（另一种生成文档导航的流行方式），但我们发现清单会很快与文件不同步。文件系统路由迫使我们必须思考文档的结构，并且感觉更符合 Next.js 的原生方式。

## 元数据

每个页面在文件顶部都有一个由三个破折号分隔的元数据块。

### 必填字段

以下字段是 **必填** 的：

| 字段         | 描述                                                                  |
| ------------- | ---------------------------------------------------------------------------- |
| `title`       | 页面的 `<h1>` 标题，用于 SEO 和 OG 图片。                         |
| `description` | 页面的描述，用于 SEO 的 `<meta name="description">` 标签。 |

```yaml filename="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

最好将页面标题限制为 2-3 个单词（例如：Optimizing Images），描述限制为 1-2 个句子（例如：Learn how to optimize images in Next.js）。

### 可选字段

以下字段是 **可选** 的：

| 字段       | 描述                                                                                                                                        |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | 覆盖导航中页面的标题。当页面标题太长时很有用。如果未提供，则使用 `title` 字段。 |
| `source`    | 将内容拉入共享页面。请参阅 [共享页面](#shared-pages)。                                                                               |
| `related`   | 文档底部的相关页面列表。这些将自动变成卡片。请参阅 [相关链接](#related-links)。         |
| `version`   | 开发阶段。例如：`experimental`, `legacy`, `unstable`, `RC`                                                                               |

```yaml filename="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
version: experimental
---
```

## `App` 和 `Pages` 文档

由于 **App Router** 和 **Pages Router** 中的大多数特性完全不同，它们的文档分别保存在不同的部分（`02-app` 和 `03-pages`）。但是，它们之间共享一些特性。

### 共享页面

为了避免内容重复和内容不同步的风险，我们使用 `source` 字段将内容从一个页面拉入另一个页面。例如，`<Link>` 组件在 **App** 和 **Pages** 中的行为基本相同。我们可以将内容从 `app/.../link.mdx` 拉入 `pages/.../link.mdx`，而不是复制内容：

```mdx filename="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx filename="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* DO NOT EDIT THIS PAGE. */}
{/* The content of this page is pulled from the source above. */}
```

因此，我们可以在一个地方编辑内容，并在两个部分中反映出来。

### 共享内容

在共享页面中，有时可能有 **App Router** 或 **Pages Router** 特定的内容。例如，`<Link>` 组件有一个 `shallow` 属性，仅在 **Pages** 中可用，而在 **App** 中不可用。

为了确保内容仅在正确的路由器中显示，我们可以将内容块包装在 `<AppOnly>` 或 `<PagesOnly>` 组件中：

```mdx filename="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

This content will only be shown on the Pages docs.

</PagesOnly>

This content is shared between App and Pages.
```

你可能会在示例和代码块中使用这些组件。

## 代码块

代码块应该包含一个最小可行示例，可以复制粘贴。这意味着代码应该能够在不进行任何额外配置的情况下运行。

例如，如果你要展示如何使用 `<Link>` 组件，你应该包括 `import` 语句和 `<Link>` 组件本身。

```tsx filename="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

在提交示例之前，请在本地运行它们。这将确保代码是最新的且可运行。

### 语言和文件名

代码块应有一个头部，其中包含语言和 `filename`。例如：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx"
export default function Page() {
  return <h1>Hello, Next.js!</h1>
}
```
````

对于 CLI 命令，请使用 `package` 属性来显示每个包管理器的命令：

````mdx filename="code-example.mdx"
```bash package="pnpm"
pnpm create next-app
```

```bash package="npm"
npx create-next-app@latest
```

```bash package="yarn"
yarn create next-app
```

```bash package="bun"
bun create next-app
```
````

文档中的大多数示例用 `tsx` 和 `jsx` 编写，少数用 `bash`。但是，你可以使用任何支持的语言，这是[完整列表](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)。

在编写 JavaScript 代码块时，我们使用以下语言和扩展组合。

| TypeScript files with JSX      | ```tsx   | .tsx      |
| TypeScript files without JSX   | ```ts    | .ts       |

### TS 和 JS 切换器

添加语言切换器以在 TypeScript 和 JavaScript 之间切换。代码块应该以 TypeScript 为主，并提供 JavaScript 版本以适应用户。

目前，我们逐个编写 TS 和 JS 示例，并使用 `switcher` 属性链接它们：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx" switcher

```

```jsx filename="app/page.js" switcher

```
````

> **提示**：我们计划在未来自动将 TypeScript 片段编译为 JavaScript。同时，你可以使用 [transform.tools](https://transform.tools/typescript-to-javascript)。

### 行高亮

可以高亮代码行。当你想要引起对代码特定部分的注意时，这很有用。你可以通过向 `highlight` 属性传递一个数字来高亮行。

**单行：** `highlight={1}`

```tsx filename="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**多行：** `highlight={1,3}`

```tsx filename="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**行范围：** `highlight={1-5}`

```tsx filename="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## 注意

对于重要但不关键的信息，使用注意。注意是添加信息而不分散用户对主要内容注意力的好方法。

```mdx filename="notes.mdx"
> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.
```

**输出：**

> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.

## 相关链接

相关链接通过添加逻辑下一步的链接来指导用户的学习旅程。

- 链接将显示在页面主要内容下的卡片中。
- 对于有子页面的页面，链接将自动生成。

使用页面元数据中的 `related` 字段创建相关链接。

```yaml filename="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### 嵌套字段

| 字段         | 是否必填？ | 描述                                                                                                                                               |
| ------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | 可选  | 卡片列表的标题。默认为 **下一步**。                                                                                                   |
| `description` | 可选  | 卡片列表的描述。                                                                                                                         |
| `links`       | 必填  | 指向其他文档页面的链接列表。每个列表项应该是相对 URL 路径（不带前导斜杠），例如：`app/api-reference/file-conventions/page` |

## 风格指南

本节包含针对技术写作新手的文档撰写指南。

### 页面模板

虽然我们没有严格的页面模板，但你会看到文档中重复出现的页面部分：

- **概述**：页面的第一段应该告诉用户该功能是什么以及它的用途。接着是最小工作示例或其 API 参考。
- **约定**：如果该功能有约定，应在此处解释。
- **示例**：展示该功能如何用于不同的用例。
- **API 表格**：API 页面应该在页面顶部有一个概述表格，并带有跳转到章节的链接（如果可能）。
- **下一步（相关链接）**：添加指向相关页面的链接以指导用户的学习旅程。

根据需要添加这些部分。

### 页面类型

文档页面也分为两类：概念性和参考性。

- **概念性** 页面用于解释概念或功能。它们通常比参考页面更长，包含更多信息。在 Next.js 文档中，概念性页面位于 **构建应用程序** 部分。
- **参考性** 页面用于解释特定 API。它们通常更短且更集中。在 Next.js 文档中，参考页面位于 **API 参考** 部分。

> **提示**：根据你贡献的页面，你可能需要遵循不同的语气和风格。例如，概念性页面更具指导性，使用单词 _you_ 来称呼用户。参考页面更技术性，它们使用更多命令式词语，如“create, update, accept”，并倾向于省略单词 _you_。

### 语气

以下是一些指南，用于在文档中保持一致的风格和语气：

- 写清晰、简洁的句子。避免离题。
  - 如果你发现自己用了很多逗号，考虑将句子分成多个句子或使用列表。
  - 用简单的词替换复杂的词。例如，用 _use_ 代替 _utilize_。
- 注意单词 _this_。它可能含糊且令人困惑，如果不清楚，不要害怕重复句子的主语。
  - 例如，_Next.js uses React_ 而不是 _Next.js uses this_。
- 使用主动语态而不是被动语态。主动句更容易阅读。
  - 例如，_Next.js uses React_ 而不是 _React is used by Next.js_。如果你发现自己使用 _was_ 和 _by_ 这样的词，你可能是在使用被动语态。
- 避免使用 _easy_、_quick_、_simple_、_just_ 等词。这是主观的，可能会让用户感到沮丧。
- 避免使用 _don't_、_can't_、_won't_ 等否定词。这可能会让读者感到沮丧。
  - 例如，_"You can use the `Link` component to create links between pages"_ 而不是 _"Don't use the `<a>` tag to create links between pages"_。
- 使用第二人称（you/your）。这更个人化和吸引人。
- 使用性别中立语言。在提到观众时，使用 _developers_、_users_ 或 _readers_。
- 如果添加代码示例，确保它们格式正确且可运行。

虽然这些指南并非详尽无遗，但它们应该能帮助你入门。如果你想深入了解技术写作，请查看 [Google Technical Writing Course](https://developers.google.com/tech-writing/overview)。
