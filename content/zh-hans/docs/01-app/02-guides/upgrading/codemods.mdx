---
title: 代码修改工具
description: 当新功能发布时，使用代码修改工具来升级您的 Next.js 代码库。
---

代码修改工具是可以在您的代码库上程序化运行的转换工具。这允许通过程序化方式应用大量更改，而无需手动遍历每个文件。

Next.js 提供了代码修改转换工具，以在 API 更新或弃用时帮助升级您的 Next.js 代码库。

## 使用方法

在您的终端中，导航 (`cd`) 进入项目的文件夹，然后运行：

```bash filename="Terminal"
npx @next/codemod <transform> <path>
```

将 `<transform>` 和 `<path>` 替换为适当的值。

- `transform` - 转换工具的名称
- `path` - 要转换的文件或目录
- `--dry` 进行演练，不会编辑任何代码
- `--print` 打印更改后的输出以进行比较

## 升级

升级您的 Next.js 应用，自动运行 codemods 并更新 Next.js、React 和 React DOM。

```bash filename="Terminal"
npx @next/codemod upgrade [revision]
```

### 选项

- `revision`（可选）：指定升级类型（`patch`、`minor`、`major`）、NPM dist tag（例如 `latest`、`canary`、`rc`）或确切版本（例如 `15.0.0`）。稳定版本默认为 `minor`。
- `--verbose`：在升级过程中显示更详细的输出。

例如：

```bash filename="Terminal"
# Upgrade to the latest patch (e.g. 16.0.7 -> 16.0.8)
npx @next/codemod upgrade patch

# Upgrade to the latest minor (e.g. 15.3.7 -> 15.4.8). This is the default.
npx @next/codemod upgrade minor

# Upgrade to the latest major (e.g. 15.5.7 -> 16.0.7)
npx @next/codemod upgrade major

# Upgrade to a specific version
npx @next/codemod upgrade 16

# Upgrade to the canary release
npx @next/codemod upgrade canary
```

> **温馨提示**：
>
> - 如果目标版本与当前版本相同或更低，命令将退出而不进行更改。
> - 在升级过程中，您可能会被提示选择应用哪些 Next.js codemods，如果升级 React，则运行 React 19 codemods。

## Codemods

### 16.0

#### 从 App Router 页面和布局中移除 `experimental_ppr` Route Segment Config

##### `remove-experimental-ppr`

```bash filename="Terminal"
npx @next/codemod@latest remove-experimental-ppr .
```

此 codemod 从 App Router 页面和布局中移除 `experimental_ppr` Route Segment Config。

```diff filename="app/page.tsx"
- export const experimental_ppr = true;
```

#### 从已稳定 API 中移除 `unstable_` 前缀

##### `remove-unstable-prefix`

```bash filename="Terminal"
npx @next/codemod@latest remove-unstable-prefix .
```

此 codemod 从稳定的 API 中移除 `unstable_` 前缀。

例如：

```ts
import { unstable_cacheTag as cacheTag } from 'next/cache'

cacheTag()
```

转换为：

```ts
import { cacheTag } from 'next/cache'

cacheTag()
```

#### 从已过时的 `middleware` 约定迁移到 `proxy`

##### `middleware-to-proxy`

```bash filename="Terminal"
npx @next/codemod@latest middleware-to-proxy .
```

此 codemod 将项目从使用已过时的 `middleware` 约定迁移到使用 `proxy` 约定。它：

- 将 `middleware.<extension>` 重命名为 `proxy.<extension>`（例如 `middleware.ts` 到 `proxy.ts`）
- 将具名导出 `middleware` 重命名为 `proxy`
- 将 Next.js 配置属性 `experimental.middlewarePrefetch` 重命名为 `experimental.proxyPrefetch`
- 将 Next.js 配置属性 `experimental.middlewareClientMaxBodySize` 重命名为 `experimental.proxyClientMaxBodySize`
- 将 Next.js 配置属性 `experimental.externalMiddlewareRewritesResolve` 重命名为 `experimental.externalProxyRewritesResolve`
- 将 Next.js 配置属性 `skipMiddlewareUrlNormalize` 重命名为 `skipProxyUrlNormalize`

例如：

```ts filename="middleware.ts"
import { NextResponse } from 'next/server'

export function middleware() {
  return NextResponse.next()
}
```

转换为：

```ts filename="proxy.ts"
import { NextResponse } from 'next/server'

export function proxy() {
  return NextResponse.next()
}
```

#### 从 `next lint`迁移到 ESLint CLI

##### `next-lint-to-eslint-cli`

```bash filename="Terminal"
npx @next/codemod@canary next-lint-to-eslint-cli .
```

此代码修改工具将项目从使用 `next lint` 迁移到使用带有本地 ESLint 配置的 ESLint CLI。它执行以下操作：

- 创建包含 Next.js 推荐配置的 `eslint.config.mjs` 文件
- 更新 `package.json` 脚本以使用 `eslint .` 代替 `next lint`
- 将必要的 ESLint 依赖项添加到 `package.json`
- 当找到时保留现有的 ESLint 配置

例如：

```json filename="package.json"
{
  "scripts": {
    "lint": "next lint"
  }
}
```

变为：

```json filename="package.json"
{
  "scripts": {
    "lint": "eslint ."
  }
}
```

并创建：

```js filename="eslint.config.mjs"
import { dirname } from 'path'
import { fileURLToPath } from 'url'
import { FlatCompat } from '@eslint/eslintrc'

const __filename = fileURLToPath(import.meta.url)
const __dirname = dirname(__filename)

const compat = new FlatCompat({
  baseDirectory: __dirname,
})

const eslintConfig = [
  ...compat.extends('next/core-web-vitals', 'next/typescript'),
  {
    ignores: [
      'node_modules/**',
      '.next/**',
      'out/**',
      'build/**',
      'next-env.d.ts',
    ],
  },
]

export default eslintConfig
```

#### 迁移到异步动态 APIs

之前支持同步访问的、选择加入动态渲染的 API 现在变为异步。您可以在 [升级指南](/docs/app/guides/upgrading/version-15) 中阅读有关此重大更改的更多信息。

##### `next-async-request-api`

```bash filename="Terminal"
npx @next/codemod@latest next-async-request-api .
```

此代码修改工具将把现在是异步的动态 API（`next/headers` 中的 `cookies()`、`headers()` 和 `draftMode()`）转换为正确地使用 `await` 或在适用时用 `React.use()` 包装。
当无法进行自动迁移时，代码修改工具将添加类型转换（如果是 TypeScript 文件）或注释，以通知用户需要手动审查和更新。

例如：

```tsx
import { cookies, headers } from 'next/headers'
const token = cookies().get('token')

function useToken() {
  const token = cookies().get('token')
  return token
}

export default function Page() {
  const name = cookies().get('name')
}

function getHeader() {
  return headers().get('x-foo')
}
```

转换为：

```tsx
import { use } from 'react'
import {
  cookies,
  headers,
  type UnsafeUnwrappedCookies,
  type UnsafeUnwrappedHeaders,
} from 'next/headers'
const token = (cookies() as unknown as UnsafeUnwrappedCookies).get('token')

function useToken() {
  const token = use(cookies()).get('token')
  return token
}

export default async function Page() {
  const name = (await cookies()).get('name')
}

function getHeader() {
  return (headers() as unknown as UnsafeUnwrappedHeaders).get('x-foo')
}
```

当我们在页面/路由入口（`page.js`、`layout.js`、`route.js` 或 `default.js`）或 `generateMetadata` / `generateViewport` API 中检测到对 `params` 或 `searchParams` 属性的访问时，
它会尝试将调用点从同步函数转换为异步函数，并等待属性访问。如果无法使其异步化（例如在客户端组件中），它将使用 `React.use` 来解包 Promise。

例如：

```tsx
// page.tsx
export default function Page({
  params,
  searchParams,
}: {
  params: { slug: string }
  searchParams: { [key: string]: string | string[] | undefined }
}) {
  const { value } = searchParams
  if (value === 'foo') {
    // ...
  }
}

export function generateMetadata({ params }: { params: { slug: string } }) {
  const { slug } = params
  return {
    title: `My Page - ${slug}`,
  }
}
```

转换为：

```tsx
// page.tsx
export default async function Page(props: {
  params: Promise<{ slug: string }>
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>
}) {
  const searchParams = await props.searchParams
  const { value } = searchParams
  if (value === 'foo') {
    // ...
  }
}

export async function generateMetadata(props: {
  params: Promise<{ slug: string }>
}) {
  const params = await props.params
  const { slug } = params
  return {
    title: `My Page - ${slug}`,
  }
}
```

> **请注意：** 当此代码修改工具识别到一个可能需要手动干预的位置，但我们无法确定确切的修复方法时，它会在代码中添加注释或类型转换，以通知用户需要手动更新。这些注释以 **@next/codemod** 为前缀，类型转换以 `UnsafeUnwrapped` 为前缀。
> 您的构建将报错，直到这些注释被明确移除。[了解更多](/docs/messages/sync-dynamic-apis)。

#### 替换 `NextRequest` 的 `geo` 和 `ip` 属性为 `@vercel/functions`

##### `next-request-geo-ip`

```bash filename="Terminal"
npx @next/codemod@latest next-request-geo-ip .
```

此代码修改工具安装 `@vercel/functions` 并将 `NextRequest` 的 `geo` 和 `ip` 属性转换为对应的 `@vercel/functions` 功能。

例如：

```ts
import type { NextRequest } from 'next/server'

export function GET(req: NextRequest) {
  const { geo, ip } = req
}
```

转换为：

```ts
import type { NextRequest } from 'next/server'
import { geolocation, ipAddress } from '@vercel/functions'

export function GET(req: NextRequest) {
  const geo = geolocation(req)
  const ip = ipAddress(req)
}
```

### 13.0

#### 重命名 Next 图像导入项

##### `next-image-to-legacy-image`

```bash filename="Terminal"
npx @next/codemod@latest next-image-to-legacy-image .
```

安全地将现有 Next.js 10、11 或 12 应用程序中的 `next/image` 导入重命名为 Next.js 13 中的 `next/legacy/image`。同时将 `next/future/image` 重命名为 `next/image`。

例如：

```jsx filename="pages/index.js"
import Image1 from 'next/image'
import Image2 from 'next/future/image'

export default function Home() {
  return (
    <div>
      <Image1 src="/test.jpg" width="200" height="300" />
      <Image2 src="/test.png" width="500" height="400" />
    </div>
  )
}
```

转换为：

```jsx filename="pages/index.js"
// 'next/image' becomes 'next/legacy/image'
import Image1 from 'next/legacy/image'
// 'next/future/image' becomes 'next/image'
import Image2 from 'next/image'

export default function Home() {
  return (
    <div>
      <Image1 src="/test.jpg" width="200" height="300" />
      <Image2 src="/test.png" width="500" height="400" />
    </div>
  )

```

#### 从 Link 组件中移除 `<a>` 标签

##### `new-link`

```bash filename="Terminal"
npx @next/codemod@latest new-link .
```

<AppOnly>

移除 [链接组件](/docs/app/api-reference/components/link) 内部的 `<a>` 标签。

</AppOnly>

<PagesOnly>

移除 [链接组件](/docs/pages/api-reference/components/link) 内部的 `<a>` 标签。

</PagesOnly>

例如：

```jsx
<Link href="/about">
  <a>About</a>
</Link>
// transforms into
<Link href="/about">
  About
</Link>

<Link href="/about">
  <a onClick={() => console.log('clicked')}>About</a>
</Link>
// transforms into
<Link href="/about" onClick={() => console.log('clicked')}>
  About
</Link>
```

### 9

#### 将匿名组件转换为具名组件

##### `name-default-component`

```bash filename="Terminal"
npx @next/codemod name-default-component
```

**适用于版本 9 及以上。**

将匿名组件转换为具名组件，以确保它们与 [快速刷新](https://nextjs.org/blog/next-9-4#fast-refresh) 兼容。

例如：

```jsx filename="my-component.js"
export default function () {
  return <div>Hello World</div>
}
```

转换为：

```jsx filename="my-component.js"
export default function MyComponent() {
  return <div>Hello World</div>
}
```

该组件将基于文件名采用驼峰式命名，并且也适用于箭头函数。

### 6

#### 使用 `withRouter`

##### `url-to-withrouter`

```bash filename="Terminal"
npx @next/codemod url-to-withrouter
```

将顶层页面上已弃用的自动注入的 `url` 属性转换为使用 `withRouter` 及其注入的 `router` 属性。在此处了解更多信息：[https://nextjs.org/docs/messages/url-deprecated](/docs/messages/url-deprecated)

例如：

```js filename="From"
import React from 'react'
export default class extends React.Component {
  render() {
    const { pathname } = this.props.url
    return <div>Current pathname: {pathname}</div>
  }
}
```

```js filename="To"
import React from 'react'
import { withRouter } from 'next/router'
export default withRouter(
  class extends React.Component {
    render() {
      const { pathname } = this.props.router
      return <div>Current pathname: {pathname}</div>
    }
  }
)
```

这是一个案例。所有已转换（并经过测试）的案例可以在 [`__testfixtures__` 目录](https://github.com/vercel/next.js/tree/canary/packages/next-codemod/transforms/__testfixtures__/url-to-withrouter) 中找到。
