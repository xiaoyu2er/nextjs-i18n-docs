## 从 15 升级到 16

### 在 Next.js DevTools MCP 中使用 AI 代理

如果您使用的 AI 编程助手支持 [模型上下文协议 (MCP)](https://modelcontextprotocol.io)，您可以使用 **Next.js DevTools MCP** 来自动化升级流程和迁移任务。

#### 设置

将以下配置添加到您的 MCP 客户端，对于每个编程代理，您可以阅读 [此部分](https://github.com/vercel/next-devtools-mcp#mcp-client-configuration) 获取配置详情。

**示例：**

```json filename=".mcp.json"
{
  "mcpServers": {
    "next-devtools": {
      "command": "npx",
      "args": ["-y", "next-devtools-mcp@latest"]
    }
  }
}
```

有关更多信息，请访问 [`next-devtools-mcp`](https://github.com/vercel/next-devtools-mcp) 文档以使用您的 MCP 客户端进行配置。

> **注意：** 使用 `next-devtools-mcp@latest` 可确保您的 MCP 客户端始终使用最新版本 Next.js DevTools MCP 服务器。

#### 示例提示

配置完成后，您可以使用自然语言提示来升级您的 Next.js 应用程序：

**升级到 Next.js 16：**

连接到您的编程代理，然后输入提示：

```txt
Next Devtools, help me upgrade my Next.js app to version 16
```

**迁移到缓存组件（升级至 v16 后）：**

连接到您的编程代理，然后输入提示：

```txt
Next Devtools, migrate my Next.js app to cache components
```

在文档中了解更多 [此处](/docs/app/guides/mcp)。

### 使用 Codemod

要更新到 Next.js 版本 16，您可以使用 `upgrade` [codemod](/docs/app/guides/upgrading/codemods#160)：

```bash package="pnpm"
pnpm dlx @next/codemod@canary upgrade latest
```

```bash package="npm"
npx @next/codemod@canary upgrade latest
```

```bash package="yarn"
yarn dlx @next/codemod@canary upgrade latest
```

```bash package="bun"
bunx @next/codemod@canary upgrade latest
```

该 [codemod](/docs/app/guides/upgrading/codemods#160) 能够：

- 更新 `next.config.js` 以使用新的 `turbopack` 配置
- 从 `next lint` 迁移到 ESLint CLI
- 从已弃用的 `middleware` 约定迁移到 `proxy`
- 移除已稳定 API 的 `unstable_` 前缀
- 从页面和布局中移除 `experimental_ppr` 路由段配置

如果您更喜欢手动执行，请安装最新版本的 Next.js 和 React：

```bash package="pnpm"
pnpm add next@latest react@latest react-dom@latest
```

```bash package="npm"
npm install next@latest react@latest react-dom@latest
```

```bash package="yarn"
yarn add next@latest react@latest react-dom@latest
```

```bash package="bun"
bun add next@latest react@latest react-dom@latest
```

如果您正在使用 TypeScript，请确保也将 `@types/react` 和 `@types/react-dom` 升级到它们的最新版本。

## 默认使用 Turbopack

从 **Next.js 16** 开始，Turbopack 已稳定，并默认用于 `next dev` 和 `next build`

以前，您必须使用 `--turbopack` 或 `--turbo` 启用 Turbopack。

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev --turbopack",
    "build": "next build --turbopack",
    "start": "next start"
  }
}
```

这不再必要。您可以更新您的 `package.json` 脚本：

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start"
  }
}
```

如果您的项目有 [自定义 `webpack`](/docs/app/api-reference/config/next-config-js/webpack) 配置，并且您运行 `next build`（现在默认使用 Turbopack），构建将 **失败** 以防止配置错误问题。

您可以采用几种不同方法来解决此问题：

- **无论如何使用 Turbopack：** 运行 `next build --turbopack` 使用 Turbopack 构建并忽略您的 `webpack` 配置。
- **完全切换到 Turbopack：** 将您的 `webpack` 配置迁移到与 Turbopack 兼容的选项。
- **继续使用 Webpack：** 使用 `--webpack` 标志选择不使用 Turbopack 并使用 Webpack 构建。

> **小知识**：如果您看到构建失败，因为找到了 `webpack` 配置，但您自己没有定义，那么很可能是插件添加了 `webpack` 选项

### 选择不使用 Turbopack

如果您需要继续使用 Webpack，您可以使用 `--webpack` 标志选择不使用默认行为。例如，在开发中使用 Turbopack，但在生产构建中使用 Webpack：

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build --webpack",
    "start": "next start"
  }
}
```

我们推荐使用 Turbopack 进行开发和生产。如果您无法切换到 Turbopack，请在此 [线程](https://github.com/vercel/next.js/discussions/77721) 提交评论。

### Turbopack 配置位置

 `experimental.turbopack` 配置已不再处于实验阶段。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

// Next.js 15 - experimental.turbopack
const nextConfig: NextConfig = {
  experimental: {
    turbopack: {
      // options
    },
  },
}

export default nextConfig
```

您可以将其作为顶级 `turbopack` 选项使用：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

// Next.js 16 - turbopack at the top level of nextConfig
const nextConfig: NextConfig = {
  turbopack: {
    // options
  },
}

export default nextConfig
```

请务必查看 `Turbopack` 配置 [选项](/docs/app/api-reference/config/next-config-js/turbopack)。 **Next.js 16** 引入了各种改进和新选项，例如：

- [高级 Webpack loader 条件](/docs/app/api-reference/config/next-config-js/turbopack#advanced-webpack-loader-conditions)
- [debugIds](/docs/app/api-reference/config/next-config-js/turbopack#debug-ids)

### 解析别名回退

在某些项目中，客户端代码可能会导入包含 Node.js 原生模块的文件。这将导致 `Module not found: Can't resolve 'fs'` 类型的错误。

发生这种情况时，您应该重构您的代码，使您的客户端包不引用这些 Node.js 原生模块。

然而，在某些情况下，这可能是不可能的。在 Webpack 中，通常使用 `resolve.fallback` 选项来 **静音**（抑制）该错误。Turbopack 提供了一个类似的选项，使用 `turbopack.resolveAlias`。在这种情况下，告诉 Turbopack 在浏览器请求 `fs` 时加载一个空模块。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  turbopack: {
    resolveAlias: {
      fs: {
        browser: './empty.ts', // We recommend to fix code imports before using this method
      },
    },
  },
}

export default nextConfig
```

最好重构您的模块，使得客户端代码从不导入使用 Node.js 原生模块的模块。

### Sass node_modules 导入

Turbopack 完全支持从 `node_modules` 导入 Sass 文件。注意，虽然 Webpack 允许传统的波浪号 (`~`) 前缀，但 Turbopack 不支持此语法。

在 Webpack 中：

```scss filename="styles/globals.scss"
@import '~bootstrap/dist/css/bootstrap.min.css';
```

在 Turbopack 中：

```scss filename="styles/globals.scss"
@import 'bootstrap/dist/css/bootstrap.min.css';
```

如果更改导入是不可能的，您可以使用 `turbopack.resolveAlias`。例如：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  turbopack: {
    resolveAlias: {
      '~*': '*',
    },
  },
}

export default nextConfig
```

### Turbopack 文件系统缓存 (beta)

Turbopack 现在支持开发环境中的文件系统缓存，在运行之间将编译器工件存储在磁盘上，从而实现跨重启的显著更快的编译时间。

在您的配置中启用文件系统缓存：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },
}

module.exports = nextConfig
```

## 异步请求 API (重大变更)

版本 15 引入了 [异步请求 API](https://nextjs.org/docs/app/guides/upgrading/version-15#async-request-apis-breaking-change) 作为破坏性变更，带有 **临时** 同步兼容性。

从 **Next.js 16** 开始，同步访问已完全移除。这些 API 只能通过异步方式访问。

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout), [`page.js`](/docs/app/api-reference/file-conventions/page), [`route.js`](/docs/app/api-reference/file-conventions/route), [`default.js`](/docs/app/api-reference/file-conventions/default), [`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image#opengraph-image), [`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image#twitter-image), [`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons#icon), and [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons#apple-icon) 中的 `params`
- [`page.js`](/docs/app/api-reference/file-conventions/page) 中的 `searchParams`

使用 [codemod](/docs/app/guides/upgrading/codemods#migrate-to-async-dynamic-apis) 迁移到异步请求时 API。

### 迁移异步请求时 API 的类型

为了帮助迁移到异步 `params` 和 `searchParams`，您可以运行 [`npx next typegen`](/docs/app/api-reference/cli/next#next-typegen-options) 来自动生成这些全局可用的类型辅助函数：

- [`PageProps`](/docs/app/api-reference/file-conventions/page#page-props-helper)
- [`LayoutProps`](/docs/app/api-reference/file-conventions/layout#layout-props-helper)
- [`RouteContext`](/docs/app/api-reference/file-conventions/route#route-context-helper)

> **小知识**：`typegen` 在 Next.js 15.5 中引入

这简化了向新异步 API 模式的类型安全迁移，并允许您以完整的类型安全性更新您的组件，例如：

```tsx filename="/app/blog/[slug]/page.tsx"
export default async function Page(props: PageProps<'/blog/[slug]'>) {
  const { slug } = await props.params
  const query = await props.searchParams
  return <h1>Blog Post: {slug}</h1>
}
```

这种方法使您能够在页面内完全类型安全地访问 `props.params`（包括 `slug`）和 `searchParams`。

## icon 和 Open Graph 图像的异步参数（破坏性变更）

> 传递给 `opengraph-image`、`twitter-image`、`icon` 和 `apple-icon` 中图像生成函数的 props 现在是 Promise。

在以前的版本中，`Image`（图像生成函数）和 `generateImageMetadata` 都接收 `params` 对象。`generateImageMetadata` 返回的 `id` 作为字符串传递给图像生成函数。

```js filename="app/shop/[slug]/opengraph-image.js"
// Next.js 15 - synchronous params access
export function generateImageMetadata({ params }) {
  const { slug } = params
  return [{ id: '1' }, { id: '2' }]
}

// Next.js 15 - synchronous params and id access
export default function Image({ params, id }) {
  const slug = params.slug
  const imageId = id // string
  // ...
}
```

从 **Next.js 16** 开始，为了与 [异步请求 API](#async-request-apis-breaking-change) 变更保持一致，图像生成函数现在将 `params` 和 `id` 作为 Promise 接收。 `generateImageMetadata` 函数继续接收同步 `params`。

```js filename="app/shop/[slug]/opengraph-image.js"
export async function generateImageMetadata({ params }) {
  const { slug } = params
  return [{ id: '1' }, { id: '2' }]
}

// Next.js 16 - asynchronous params and id access
export default async function Image({ params, id }) {
  const { slug } = await params // params now async
  const imageId = await id // id is now Promise<string> when using generateImageMetadata
  // ...
}
```

## `sitemap` 的异步 `id` 参数（破坏性变更）

以前，从 [`generateSitemaps`](/docs/app/api-reference/functions/generate-sitemaps) 返回的 `id` 值直接传递给 `sitemap` 生成函数。

```js filename="app/product/sitemap.js"
export async function generateSitemaps() {
  return [{ id: 0 }, { id: 1 }, { id: 2 }, { id: 3 }]
}

// Next.js 15 - synchronous id access
export default async function sitemap({ id }) {
  const start = id * 50000 // id is a number
  // ...
}
```

从 **Next.js 16** 开始， `sitemap` 生成函数现在将 `id` 作为 Promise 接收。

```js filename="app/product/sitemap.js"
export async function generateSitemaps() {
  return [{ id: 0 }, { id: 1 }, { id: 2 }, { id: 3 }]
}

// Next.js 16 - asynchronous id access
export default async function sitemap({ id }) {
  const resolvedId = await id // id is now Promise<string>
  const start = Number(resolvedId) * 50000
  // ...
}
```

## React Compiler 支持

随着 React Compiler 1.0 版本的发布，React Compiler 的内置支持现在在 **Next.js 16** 中已稳定。React Compiler 自动对组件进行记忆化，减少不必要的重新渲染，无需任何手动代码更改。

 `reactCompiler` 配置选项已从 `experimental` 提升到稳定级别。它默认未启用，因为我们继续在不同应用类型中收集构建性能数据。

/** @type {import('next').NextConfig} */
const nextConfig = {

## 缓存 API

### revalidateTag

[`revalidateTag`](/docs/app/api-reference/functions/revalidateTag) 有了新的函数签名。您可以将 [`cacheLife`](/docs/app/api-reference/functions/cacheLife#reference) 配置文件作为第二个参数传递。

```ts filename="app/actions.ts" switcher
'use server'

import { revalidateTag } from 'next/cache'

export async function updateArticle(articleId: string) {
  // Mark article data as stale - article readers see stale data while it revalidates
  revalidateTag(`article-${articleId}`, 'max')
}
```

```js filename="app/actions.js" switcher
'use server'

import { revalidateTag } from 'next/cache'

export async function updateArticle(articleId) {
  // Mark article data as stale - article readers see stale data while it revalidates
  revalidateTag(`article-${articleId}`, 'max')
}
```

对于更新可接受稍微波动的内容（如文章、产品目录或文档），请使用 `revalidateTag`。用户可能会收到旧内容，同时新数据在后台加载。

### updateTag

[`updateTag`](/docs/app/api-reference/functions/updateTag) 是一个新的仅限 [Server Actions](/docs/app/getting-started/mutating-data#what-are-server-functions) 的 API，提供 **读写一致性 (read-your-writes)** 语义，用户做出更改后 UI 会立即显示更改，而不是旧数据。

它通过在同一个请求中使数据过期并立即刷新数据来实现这一点。

```ts filename="app/actions.ts" switcher
'use server'

import { updateTag } from 'next/cache'

export async function updateUserProfile(userId: string, profile: Profile) {
  await db.users.update(userId, profile)

  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`)
}
```

```js filename="app/actions.js" switcher
'use server'

import { updateTag } from 'next/cache'

export async function updateUserProfile(userId, profile) {
  await db.users.update(userId, profile)

  // Expire cache and refresh immediately - user sees their changes right away
  updateTag(`user-${userId}`)
}
```

这确保了交互功能能立即反映更改。非常适合表单、用户设置以及用户期望即时看到更新的工作流程。

在此处了解更多关于何时使用 `updateTag` 或 `revalidateTag` 的信息 [here](/docs/app/api-reference/functions/updateTag#when-to-use-updatetag)。

### refresh

[`refresh`](/docs/app/api-reference/functions/refresh) 允许您在 Server Action 内部刷新客户端路由。

```ts filename="app/actions.ts" switcher
'use server'

import { refresh } from 'next/cache'

export async function markNotificationAsRead(notificationId: string) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId)

  // Refresh the notification count displayed in the header
  refresh()
}
```

```js filename="app/actions.js" switcher
'use server'

import { refresh } from 'next/cache'

export async function markNotificationAsRead(notificationId) {
  // Update the notification in the database
  await db.notifications.markAsRead(notificationId)

  // Refresh the notification count displayed in the header
  refresh()
}
```

在执行操作后需要刷新客户端路由时使用。

### cacheLife 和 cacheTag

[`cacheLife`](/docs/app/api-reference/functions/cacheLife) 和 [`cacheTag`](/docs/app/api-reference/functions/cacheTag) 现在已稳定。不再需要 `unstable_` 前缀。

在任何您有如下别名导入的地方：

```ts
import {
  unstable_cacheLife as cacheLife,
  unstable_cacheTag as cacheTag,
} from 'next/cache'
```

您可以将导入更新为：

```ts
import { cacheLife, cacheTag } from 'next/cache'
```

## 增强的路由和导航

**Next.js 16** 包含了对路由和导航系统的彻底重构，使页面过渡更简洁、更快速。这优化了 Next.js 预取和缓存导航数据的方式：

- **布局去重**：当预取共享布局的多个 URL 时，布局仅下载一次。
- **增量预取**：Next.js 仅预取缓存中尚未包含的部分，而不是整个页面。

这些更改 **无需代码修改**，旨在提高所有应用的性能。

不过，您可能会看到更多的单独预取请求，但总传输量明显降低。我们相信这对于几乎所有应用来说都是正确的权衡。

如果增加的请求次数导致问题，请通过创建 [问题](https://github.com/vercel/next.js/issues) 或 [讨论](https://github.com/vercel/next.js/discussions) 项来告诉我们。

## 部分预渲染 (PPR)

**Next.js 16** 移除了实验性 **部分预渲染 (PPR)** 标志和配置选项，包括路由级别的 `experimental_ppr` 段。

从 **Next.js 16** 开始，您可以使用 [`cacheComponents`](/docs/app/api-reference/config/next-config-js/cacheComponents) 配置选择加入 PPR。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  cacheComponents: true,
}

module.exports = nextConfig
```

 **Next.js 16** 中的 PPR 与 **Next.js 15** 金丝雀版本 (Canary) 的工作方式不同。如果您今天正在使用 PPR，请保留在您正在使用的当前 Next.js 15 金丝雀版本 (Canary) 中。请参阅 [迁移到缓存组件](/docs/app/guides/migrating-to-cache-components) 以获取迁移模式。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  // If you are using PPR today
  // stay in the current Next.js 15 canary
  experimental: {
    ppr: true,
  },
}

module.exports = nextConfig
```

## `middleware` 改为 `proxy`

 `middleware` 文件名已弃用，并已重命名为 `proxy`，以澄清网络边界和路由重点。

 `proxy` 中 **不支持** `edge` 运行时。`proxy` 运行时是 `nodejs`，且无法配置。如果您想继续使用 `edge` 运行时，请继续使用 `middleware`。我们将在后续的小版本更新中提供进一步的 `edge` 运行时说明。

```bash filename="Terminal"
# Rename your middleware file
mv middleware.ts proxy.ts
# or
mv middleware.js proxy.js
```

具名导出 `middleware` 也已弃用。将函数重命名为 `proxy`。

```ts filename="proxy.ts" switcher
export function proxy(request: Request) {}
```

```js filename="proxy.js" switcher
export function proxy(request) {}
```

我们建议将函数名称更改为 `proxy`，即使您使用的是默认导出。

包含 `middleware` 名称的配置标志也已被重命名。例如， `skipMiddlewareUrlNormalize` 现在是 `skipProxyUrlNormalize`

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  skipProxyUrlNormalize: true,
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  skipProxyUrlNormalize: true,
}

module.exports = nextConfig
```

版本 16 的 [codemod](/docs/app/guides/upgrading/codemods#160) 也能够更新这些标志。

## `next/image` 变更

### 带有查询字符串的本地图像（破坏性变更）

带有查询字符串的本地图像源现在需要 `images.localPatterns.search` 配置以防止枚举攻击。

```tsx filename="app/page.tsx"
import Image from 'next/image'

export default function Page() {
  return <Image src="/assets/photo?v=1" alt="Photo" width="100" height="100" />
}
```

如果您需要与本地图像一起使用查询字符串，请将模式添加到您的配置中：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  images: {
    localPatterns: [
      {
        pathname: '/assets/**',
        search: '?v=1',
      },
    ],
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    localPatterns: [
      {
        pathname: '/assets/**',
        search: '?v=1',
      },
    ],
  },
}

module.exports = nextConfig
```

### `minimumCacheTTL` 默认值（破坏性变更）

 `images.minimumCacheTTL` 的默认值已从 `60 秒` 更改为 `4 小时`（14400 秒）。这减少了没有 cache-control 头的图像的重新验证成本。

对于一些 Next.js 用户，图像重新验证频繁发生，通常是因为上游源图像缺少 `cache-control` 头。这导致重新验证每 `60` 秒发生一次，增加了 CPU 使用和成本。

由于大多数图像不经常更改，这个短间隔并不理想。将默认值设置为 4 小时提供了更持久的缓存，同时仍然允许图像在必要时每天更新几次。

如果您需要之前的行为，将 `minimumCacheTTL` 更改为较低的值，例如改回 `60` 秒：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  images: {
    minimumCacheTTL: 60,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    minimumCacheTTL: 60,
  },
}

module.exports = nextConfig
```

### `imageSizes` 默认值（破坏性变更）

 `16` 的值已从默认 `images.imageSizes` 数组中移除。

我们查看了请求分析，发现很少有项目 ever 提供 16 像素宽度的图像。移除此设置减小了 `next/image` 发送到浏览器的 `srcset` 属性的大小。

如果您需要支持 16px 图像：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  images: {
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
  },
}

module.exports = nextConfig
```

这不是开发者使用不足的问题，我们认为 16 像素宽度的图像变得不太常见，因为 `devicePixelRatio: 2` 实际上获取 32px 图像以防止视网膜显示屏 (Retina displays) 的模糊。

### `qualities` 默认值（破坏性变更）

 `images.qualities` 的默认值已从允许所有质量更改为仅 `[75]`。

如果您需要支持多个质量级别：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  images: {
    qualities: [50, 75, 100],
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    qualities: [50, 75, 100],
  },
}

module.exports = nextConfig
```

如果您指定的 `quality` prop 未包含在 `image.qualities` 数组中，则该质量将被强制转换为 `images.qualities` 中的最接近值。例如，给定上述配置，`quality` prop 为 80，将被强制转换为 75。

### 本地 IP 限制（破坏性变更）

## 并行路由 `default.js` 要求

所有 [并行路由](/docs/app/api-reference/file-conventions/parallel-routes) 槽现在都需要显式的 `default.js` 文件。没有它们构建将失败。

为了保持之前的行为，创建一个 [`default.js`](/docs/app/api-reference/file-conventions/default) 文件，调用 `notFound()` 或返回 `null`。

```tsx filename="app/@modal/default.tsx"
import { notFound } from 'next/navigation'

export default function Default() {
  notFound()
}
```

或返回 `null`：

```tsx filename="app/@modal/default.tsx"
export default function Default() {
  return null
}
```

## 滚动行为覆盖

在 **Next.js 的以前版本中**，如果您通过 CSS 将 `scroll-behavior: smooth` 全局设置到您的 `<html>` 元素上，Next.js 会在 SPA 路由转换期间覆盖此设置，如下所示：

1. 临时设置 `scroll-behavior` 为 `auto`
2. 执行导航（导致立即滚动到顶部）
3. 恢复您原始的 `scroll-behavior` 值

这确保了页面导航始终感觉灵敏和即时，即使您为页面内导航启用了平滑滚动。但是，这种操作可能需要消耗资源，尤其是在每次导航的开始时。

在 **Next.js 16** 中，此行为已更改。默认情况下，Next.js **不再覆盖** 您在导航期间的 `scroll-behavior` 设置。

**如果您希望 Next.js 执行此覆盖**（之前的默认行为），请将 `data-scroll-behavior="smooth"` 属性添加到您的 `<html>` 元素：

```tsx filename="app/layout.tsx"
export default function RootLayout({ children }) {
  return (
    <html lang="en" data-scroll-behavior="smooth">
      <body>{children}</body>
    </html>
  )
}
```

## 性能改进

对 `next dev` 和 `next start` 命令进行了显著的性能优化，并改进了终端输出，使其格式更清晰，错误信息更完善，性能指标更准确。

**Next.js 16** 从 `next build` 输出中移除了 `size` 和 `First Load JS` 指标。我们发现这些指标在使用 React Server Components 的服务器驱动架构中不准确。我们的 Turbopack 和 Webpack 实现都存在问题，并且对如何计算 Client Components 负载存在分歧。

测量实际路由性能最有效的方法是使用 [Chrome Lighthouse](https://developer.chrome.com/docs/lighthouse/overview) 或 Vercel Analytics 等工具，它们专注于 Core Web Vitals 和已下载资源的大小。

### `next dev` 配置加载

在以前的版本中，Next 配置文件在开发期间加载了两次：

- 运行 `next dev` 命令时
- `next dev` 命令启动 Next.js 服务器时

这不够高效，因为 `next dev` 命令启动 Next.js 服务器不需要配置文件。

这一更改的一个后果是，在您的 Next.js 配置文件中运行时，检查 `process.argv` 是否包含 `'dev'` 将返回 `false`。

> **小知识**： `typegen` 和 `build` 命令仍会出现在 `process.argv` 中。

这对在 `next dev` 上触发副作用的插件尤为重要。如果是这种情况，检查 `NODE_ENV` 是否设置为 `development` 可能就足够了。

```js filename="next.config.js"
import { startServer } from 'docs-lib/dev-server'

const isDev = process.env.NODE_ENV === 'development'

if (isDev) {
  startServer()
}

const nextConfig = {
  /* Your config options */
}

module.exports = nextConfig
```

或者，使用 configuration 加载的 [`phase`](/docs/app/api-reference/config/next-config-js#phase)。

这些功能之前已被弃用，现已移除：

### AMP 支持

AMP 采用率显著下降，维护此功能增加了框架的复杂性。所有 AMP API 和配置均已移除：

- `amp` 配置从您的 Next 配置文件中移除
- `next/amp` hook 导入和使用 (`useAmp`)

```tsx
// Removed
import { useAmp } from 'next/amp'

// Removed
export const config = { amp: true }
```

- 来自页面的 `export const config = { amp: true }`

```js filename="next.config.js"
const nextConfig = {
  // Removed
  amp: {
    canonicalBase: 'https://example.com',
  },
}

export default nextConfig
```

评估 AMP 是否仍对您的用例必要。大多数性能优势现在可以通过 Next.js 的内置优化和现代 Web 标准来实现。

### 运行时配置

 `serverRuntimeConfig` 和 `publicRuntimeConfig` 已移除。请改用环境变量。

**之前 (Next.js 15):**

```js filename="next.config.js"
module.exports = {
  serverRuntimeConfig: {
    dbUrl: process.env.DATABASE_URL,
  },
  publicRuntimeConfig: {
    apiUrl: '/api',
  },
}
```

```tsx filename="pages/index.tsx"
import getConfig from 'next/config'

export default function Page() {
  const { publicRuntimeConfig } = getConfig()
  return <p>API URL: {publicRuntimeConfig.apiUrl}</p>
}
```

**之后 (Next.js 16):**

对于仅服务器端的值，在 Server Components (服务端组件) 中直接访问环境变量：

```tsx filename="app/page.tsx"
async function fetchData() {
  const dbUrl = process.env.DATABASE_URL
  // Use for server-side operations only
  return await db.query(dbUrl, 'SELECT * FROM users')
}

export default async function Page() {
  const data = await fetchData()
  return <div>{/* render data */}</div>
}
```

> **小知识**：使用 [taint API](/docs/app/api-reference/config/next-config-js/taint) 防止意外将敏感的服务器值传递给 Client Components (客户端组件)。

对于客户端可访问的值，使用 `NEXT_PUBLIC_` 前缀：

```bash filename=".env.local"
NEXT_PUBLIC_API_URL="/api"
```

```tsx filename="app/components/client-component.tsx"
'use client'

export default function ClientComponent() {
  const apiUrl = process.env.NEXT_PUBLIC_API_URL
  return <p>API URL: {apiUrl}</p>
}
```

为了确保环境变量在运行时读取（而不是在构建时打包），在读取 `process.env` 之前使用 [`connection()`](/docs/app/api-reference/functions/connection) 函数：

```tsx filename="app/page.tsx"
import { connection } from 'next/server'

export default async function Page() {
  await connection()
  const config = process.env.RUNTIME_CONFIG
  return <p>{config}</p>
}
```

了解更多关于 [环境变量](/docs/app/guides/environment-variables)。
