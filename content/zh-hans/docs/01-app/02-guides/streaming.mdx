## 什么是流式传输？

在传统的服务器端渲染中，服务器在发送任何内容之前会生成完整的 HTML 文档。一个缓慢的数据库查询或 API 调用就可能阻塞整个页面。流式传输通过使用[分块传输编码](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Reference/Headers/Transfer-Encoding)改变了这一点，它在响应部分就绪时即可发送。浏览器在服务器仍在生成其余部分时就开始渲染 HTML。

这对于结合了快速静态内容（页头、导航、布局）和较慢动态内容（个性化数据、分析、推荐）的页面尤其有效。静态部分可以预渲染并从 CDN 提供服务，立即呈现，而动态部分则在就绪时从服务器流式传输。

React 的服务器渲染器按照 `<Suspense>` 边界分块生成 HTML。Next.js 将其集成到应用路由器中，因此无需额外配置即可实现流式传输。

## 示例

配套的[流式传输演示](https://streaming-demo.labs.vercel.dev/)（[源代码](https://github.com/vercel-labs/streaming-demo)）让您可以看到本指南中每个概念的实际应用：

- 使用 `loading.tsx` 进行页面级流式传输（骨架屏立即出现，内容在约 2 秒后流入）
- 兄弟 `<Suspense>` 边界独立解析的精细流式传输
- 水合比较：单次阻塞式水合与使用 Suspense 边界的分流水合
- 路由处理程序中的原始 HTML 流式传输，包含早期 CSS 发现
- 可配置的 `ReadableStream` API 端点，用于实验分块大小和浏览器缓冲

## App Router 如何交付页面

当浏览器请求页面时，在初始页面加载期间，有两个流协同工作：

### HTML 流

React 的服务器渲染器生成渐进式 HTML 块。页面的静态部分（布局、导航、 Suspense 后备内容）首先渲染并立即发送。当一个异步[服务器组件](/docs/app/glossary#server-component)解析完成时，React 会流式传输其完成的 HTML 以及内联的 `<script>` 标签：一个标签将后备 DOM 节点替换为新内容，另一个标签携带[组件负载](#the-component-payload)，以便 React 稍后对其进行水合。浏览器会立即执行替换，无需等待页面的 JavaScript 包加载或水合完成。这就是用户_看到的_：页面逐节地渐进呈现。

### 组件负载

组件负载是组件树的序列化表示，React 使用它来[水合](/docs/app/glossary#hydration)页面并处理客户端更新。在初始加载时，它嵌入在 HTML 流中到达（如上所述）。在**客户端导航**时，仅获取组件负载（带有 `rsc: 1` 请求头），根本不传输 HTML。React 使用它来就地更新组件树。

### 静态外壳

在任何异步工作解析之前渲染的所有内容称为**静态外壳**：您的布局、导航以及由 `<Suspense>` 边界定义的后备 UI。它被立即发送，为用户提供了在动态内容流入时可以看到和交互的内容。通过[缓存组件](/docs/app/getting-started/caching)，静态外壳在构建时预渲染并从边缘即时提供。

## 使用 `<Suspense>` 进行细粒度流式传输

`<Suspense>` 让您精确控制页面的哪些部分独立流式传输。您可以将后备内容推送到特定部分，而不是使用全页骨架，这样静态外壳就包含更多真实内容。

### 使用兄弟边界进行并行流式传输

当多个组件执行异步工作（获取数据、从数据库读取）时，将每个组件包装在自己的 `<Suspense>` 边界中。每个边界在其异步工作完成时独立流式传输，无论顺序如何，都不会相互阻塞：

```tsx filename="app/dashboard/page.tsx" switcher
import { Suspense } from 'react'
import { Revenue } from './revenue'
import { RecentOrders } from './recent-orders'
import { Recommendations } from './recommendations'

export default function Dashboard() {
  return (
    <div>
      <h1>Dashboard</h1>
      <div className="grid grid-cols-2 gap-4">
        <Suspense fallback={<p>Loading revenue...</p>}>
          <Revenue />
        </Suspense>
        <Suspense fallback={<p>Loading orders...</p>}>
          <RecentOrders />
        </Suspense>
      </div>
      <Suspense fallback={<p>Loading recommendations...</p>}>
        <Recommendations />
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/dashboard/page.js" switcher
import { Suspense } from 'react'
import { Revenue } from './revenue'
import { RecentOrders } from './recent-orders'
import { Recommendations } from './recommendations'

        <Suspense fallback={<p>Loading orders...</p>}>
```

在此示例中，如果 `Revenue` 在 200 毫秒内解析，`RecentOrders` 在 1 秒内解析，`Recommendations` 在 3 秒内解析，用户会在每个部分的数据就绪时立即看到它出现。

### 嵌套边界实现渐进式细节

您可以嵌套 `<Suspense>` 边界以创建分层加载体验。例如，产品页面可能会立即流式传输标题，接下来是产品详情，最后是评论：

```tsx filename="app/product/[id]/page.tsx" switcher
import { Suspense } from 'react'
import { ProductDetails } from './product-details'
import { Reviews } from './reviews'

export async function generateStaticParams() {
  const products = await getTopProducts()
  return products.map((product) => ({ id: product.id }))
}

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params

  return (
    <div>
      <h1>Product</h1>
      <Suspense fallback={<p>Loading product details...</p>}>
        <ProductDetails id={id} />
        <Suspense fallback={<p>Loading reviews...</p>}>
          <Reviews productId={id} />
        </Suspense>
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/product/[id]/page.js" switcher
import { Suspense } from 'react'
import { ProductDetails } from './product-details'
import { Reviews } from './reviews'

  return products.map((product) => ({ id: product.id }))
}

      <h1>Product</h1>
```

外部边界显示"Loading product details..."，直到 `ProductDetails` 解析完成。一旦完成，内部边界变为可见，显示"Loading reviews..."，直到 `Reviews` 解析完成。这就创建了一个渐进式的展示效果。

### 下推动态访问

最大化即时流式传输内容的关键是将动态数据访问推迟到实际需要它的组件。这适用于 `params`、`searchParams`、`cookies()`、`headers()` 和数据获取。如果您在布局或页面的顶部 `await` 任何这些内容，该点之后的所有内容都会变成动态的，无法作为静态外壳的一部分进行预渲染。

相反，将 Promise 传递下去，并让消费组件在 `<Suspense>` 边界内解析它：

```tsx filename="app/dashboard/layout.tsx" switcher
import { Suspense } from 'react'
import { Nav } from './nav'
import { UserMenu } from './user-menu'
import { cookies } from 'next/headers'

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  const cookieStore = cookies() // Start the work, but don't await

  return (
    <div>
      <Nav>
        <Suspense fallback={<p>Loading user...</p>}>
          <UserMenu cookiePromise={cookieStore} />
        </Suspense>
      </Nav>
      {children}
    </div>
  )
}
```

```jsx filename="app/dashboard/layout.js" switcher
import { Suspense } from 'react'
import { Nav } from './nav'
import { UserMenu } from './user-menu'
import { cookies } from 'next/headers'

export default function DashboardLayout({ children }) {
  const cookieStore = cookies() // Start the work, but don't await

  return (
    <div>
      <Nav>
        <Suspense fallback={<p>Loading user...</p>}>
          <UserMenu cookiePromise={cookieStore} />
        </Suspense>
      </Nav>
      {children}
    </div>
  )
}
```

在此示例中，`<Nav>` 和 `{children}` 作为静态外壳的一部分渲染，因为布局中没有任何地方进行了 await。只有 `<UserMenu>` 在解析 cookie Promise 时挂起。如果布局在顶部调用了 `await cookies()`，则整个布局及其所有子组件都将被阻止进行预渲染。

同样的原则也适用于 `params` 和 `searchParams`。与其在页面级别解构它们，不如将 Promise 传递给需要该值的组件：

```tsx filename="app/shop/[category]/page.tsx" switcher
import { Suspense } from 'react'
import { Hero } from './hero'
import { ProductGrid } from './product-grid'

export async function generateStaticParams() {
  const categories = await getCategories()
  return categories.map((c) => ({ category: c.slug }))
}

export default function ShopPage({
  params,
}: {
  params: Promise<{ category: string }>
}) {
  return (
    <div>
      <Hero />
      <Suspense fallback={<p>Loading products...</p>}>
        <ProductGrid paramsPromise={params} />
      </Suspense>
    </div>
  )
}
```

```jsx filename="app/shop/[category]/page.js" switcher

import { ProductGrid } from './product-grid'

```

`<Hero />` 作为静态外壳的一部分绘制。`<ProductGrid>` 在需要分类时解析 `params`，仅在其边界内挂起。

您也可以使用 `.then()` 内联展开 Promise，这样子组件接收的是普通值而不是 Promise：

```tsx filename="app/shop/[category]/page.tsx" switcher
<Suspense fallback={<p>Loading products...</p>}>
  {params.then(({ category }) => (
    <ProductGrid category={category} />
  ))}
</Suspense>
```

```jsx filename="app/shop/[category]/page.js" switcher
<Suspense fallback={<p>Loading products...</p>}>
  {params.then(({ category }) => (
    <ProductGrid category={category} />
  ))}
</Suspense>
```

这保持了 `ProductGrid` 的简单性（它接受 `string`，而不是 `Promise`），同时仍然将 `params` 访问推迟到 Suspense 边界内。

### 何时使用 `loading.js` 与 `<Suspense>`

|                | `loading.js`                             | `<Suspense>`                     |
| -------------- | ---------------------------------------- | -------------------------------- |
| **作用范围**      | 整个页面                              | 任何组件                    |
| **设置方式**      | 放入一个文件中                           | 显式包装组件       |
| **导航** | 作为即时后备内容预获取           | 默认不预获取        |
| **最适合**   | 没有数据就无法渲染任何内容的页面 | 大多数页面，用于精细控制 |

优先使用靠近动态访问点的显式 `<Suspense>` 边界。当预渲染器遇到动态工作时，它会向上遍历树以寻找最近的 Suspense 边界。如果未找到，构建将失败并出现[阻塞路由错误](/docs/messages/blocking-route)。树中高层的 `loading.js` 是一个有效的边界，因此框架会找到它并停止，但现在整个页面会回退到全页骨架，而不是精细地流式传输。

如果组件在流式传输开始后抛出错误，最近的 [`error.js`](/docs/app/api-reference/file-conventions/error) 边界会捕获它，并在失败组件的位置渲染错误 UI。页面的其余部分保持完好，只有出错的那部分被替换。

因为 HTTP 状态码 (`200 OK`) 已经随第一个数据块发送，所以无法将其更改为 `4xx` 或 `5xx`。错误完全在流式 HTML 内处理。有关此限制的更多信息，请参阅 [HTTP 约定](#the-http-contract)。

## 向客户端流式传输数据

您可以在[服务器组件](/docs/app/glossary#server-component)中启动一个 fetch 请求，并将未解析的 Promise 作为 prop 传递给[客户端组件](/docs/app/glossary#client-component)。这个 Promise 可以根据需要传递任意多层。只有调用 React 的 [`use`](https://react.dev/reference/react/use) API 来读取值的组件才需要一个围绕它的 `<Suspense>` 边界：

```tsx filename="app/dashboard/page.tsx" switcher
import { Suspense } from 'react'
import { StatsChart } from './stats-chart'

type Stats = { revenue: number; orders: number }

async function getStats(): Promise<Stats> {
  const res = await fetch('https://api.example.com/stats')
  return res.json()
}

export default function Dashboard() {
  // Start the fetch during server render, don't await it
  const statsPromise = getStats()

  return (
    <Suspense fallback={<p>Loading chart...</p>}>
      <StatsChart dataPromise={statsPromise} />
    </Suspense>
  )
}
```

```jsx filename="app/dashboard/page.js" switcher
import { Suspense } from 'react'
import { StatsChart } from './stats-chart'

async function getStats() {
  const res = await fetch('https://api.example.com/stats')
  return res.json()
}

export default function Dashboard() {
  // Start the fetch during server render, don't await it
  const statsPromise = getStats()

  return (
    <Suspense fallback={<p>Loading chart...</p>}>
      <StatsChart dataPromise={statsPromise} />
    </Suspense>
  )
}
```

```tsx filename="app/dashboard/stats-chart.tsx" switcher
'use client'

import { use } from 'react'

type Stats = { revenue: number; orders: number }

export function StatsChart({ dataPromise }: { dataPromise: Promise<Stats> }) {
  const stats = use(dataPromise)

  return <div>{/* render chart with stats */}</div>
}
```

```jsx filename="app/dashboard/stats-chart.js" switcher
'use client'

import { use } from 'react'

export function StatsChart({ dataPromise }) {
  const stats = use(dataPromise)

  return <div>{/* render chart with stats */}</div>
}
```

后备内容随静态外壳立即发送。当 Promise 解析时，React 将完成的 HTML 流式传输到页面中。

### 在树中共享 Promise

当多个组件需要相同的数据时，启动一次 fetch，并通过上下文提供程序传递 Promise，这样子树中的任何组件都可以用 `use()` 来解析它：

```tsx filename="app/layout.tsx"
import { getUser } from '@/lib/data'
// Stores the promise in React context for the subtree
import { UserProvider } from './user-provider'

export default function Layout({ children }: { children: React.ReactNode }) {
  const userPromise = getUser()

  return <UserProvider userPromise={userPromise}>{children}</UserProvider>
}
```

请参阅[使用上下文和 React.cache 共享数据](/docs/app/getting-started/fetching-data#sharing-data-with-context-and-reactcache)以获取包括提供程序和消费者组件在内的完整模式。

## 路由处理程序 (Route Handlers) 中的流式传输

上述模式依赖于 React 和 Suspense 来流式传输 UI。在 React 渲染之外，[路由处理程序](/docs/app/api-reference/file-conventions/route)可以使用 Web Streams API 流式传输原始响应。这对于服务器发送事件、大文件生成或任何您希望数据逐步到达的响应非常有用：

```ts filename="app/api/stream/route.ts" switcher
export async function GET() {
  const encoder = new TextEncoder()

  const stream = new ReadableStream({
    async start(controller) {
      for (let i = 0; i < 10; i++) {
        controller.enqueue(encoder.encode(`Chunk ${i + 1}\n`))
        await new Promise((resolve) => setTimeout(resolve, 200))
      }
      controller.close()
    },
  })

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/plain; charset=utf-8',
      'X-Content-Type-Options': 'nosniff',
    },
  })
}
```

```js filename="app/api/download/route.js" switcher
}
```

有关构建流式端点的更多详细信息，请参阅[路由处理程序 API 参考](/docs/app/api-reference/file-conventions/route)。

## 流式传输与 Web 性能指标 (Web Vitals)

[网页指标 (Web Vitals)](https://web.dev/articles/vitals) 是 Google 用来衡量用户体验的指标。流式传输直接影响其中几个指标。

### TTFB (首字节时间) 和 FCP (首次内容绘制)

没有流式传输时，服务器在发送任何 HTML 之前等待所有数据，因此 TTFB 等于最慢的查询。使用流式传输后，服务器在静态外壳就绪后立即发送它。TTFB 降至渲染布局和后备内容所需的时间。浏览器立即绘制静态外壳，因此 FCP 与您的数据获取时间解耦。

### LCP (最大内容绘制)

如果您的 LCP 元素（主图、主标题、产品照片）位于 Suspense 边界内，它必须等到该边界解析后才能绘制。为了保持 LCP 快速：

- 将 LCP 元素放在 Suspense 边界**之外**或**之上**，以便它们作为静态外壳的一部分渲染。
- 对于 LCP 图片，在 `next/image` 上使用 [`preload`](/docs/app/api-reference/components/image#preload) 属性。这会在 `<head>` 中注入 `<link rel="preload">`，因此浏览器从第一个数据块开始就获取图片，甚至在 `<img>` 标签出现在 HTML 中之前。
- 对于非图片 LCP 元素（文本、标题），确保它们没有被包装在依赖于慢速数据的 Suspense 边界中。

### CLS (累积布局偏移)

当 Suspense 后备内容被解析的内容替换时，浏览器会重新排列页面。如果后备内容和解析的内容大小不同，周围的布局就会发生偏移。为了最小化 CLS：

- 设计与它们所代表的内容**尺寸匹配**的骨架后备内容。与最终卡片网格具有相同高度和宽度的骨架可以防止偏移。
- 在 Suspense 边界周围使用固定或最小高度的容器，以便在内容到达之前预留空间。

### INP (交互到下一次绘制)

流式传输支持[选择性水合](https://react.dev/reference/react-dom/client/hydrateRoot)：React 在组件流入时独立水合它们，并优先水合用户正在交互的任何内容。每个 `<Suspense>` 边界都是一个水合单元。没有它们，React 会以一次阻塞式传递水合整个页面。有了它们，水合被分解成更小的任务，这些任务会让出给浏览器，保持主线程响应。[配套演示](https://streaming-demo.labs.vercel.dev/hydration-single)让您比较单次阻塞式水合传递与使用 Suspense 边界的[分流水合](https://streaming-demo.labs.vercel.dev/hydration-split)。

### 早期资源发现

静态外壳在第一个 HTML 数据块中就包含 `<link>` 和 `<script>` 标签。浏览器在服务器仍在生成内容时，就立即发现并开始获取 CSS、JavaScript 和字体。资源在服务器思考期间获取，而不是之后。

在上面的[仪表板示例](#parallel-streaming-with-sibling-boundaries)中，`<h1>` 在外壳中渲染（有利于 LCP），每个数据部分在其自己的 Suspense 边界后独立流式传输（有利于 INP，因为水合被分割），并且骨架后备内容预留了空间（有利于 CLS）。

## HTTP 协议

一旦流式传输开始，HTTP 响应头（包括状态码）就已经发送到客户端。**流式传输开始后，您无法更改状态码或响应头。**本节中的所有内容都源于这一基本约束。

### 状态码

当 `<Suspense>` 后备内容渲染或组件挂起时，服务器必须提交 `200 OK` 以开始发送 HTML 流。如果在流式传输过程中触发了 [`notFound()`](/docs/app/api-reference/functions/not-found)，Next.js 无法返回并将状态更改为 404。相反，它会将 `<meta name="robots" content="noindex">` 注入到流式 HTML 中，以便搜索引擎不索引该页面。类似地，流式传输中的 [`redirect()`](/docs/app/api-reference/functions/redirect) 变为客户端重定向，而不是 HTTP 重定向头。

### 流式传输何时开始？

当 Suspense 后备内容渲染（例如，`loading.tsx`）或当组件在 `<Suspense>` 边界下挂起时，响应正文开始流式传输。要为错误获取真实的 HTTP 状态码，请将 `notFound()` 放在任何 `await` 或 `<Suspense>` 边界**之前**：

```tsx filename="app/post/[slug]/page.tsx" switcher
import { Suspense } from 'react'
import { notFound } from 'next/navigation'
import { PostContent } from './post-content'

export async function generateStaticParams() {
  const posts = await getPublishedPosts()
  return posts.map((post) => ({ slug: post.slug }))
}

export default async function PostPage({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const exists = await checkSlugExists(slug) // Fast existence check
  if (!exists) notFound() // Real 404, before any Suspense boundary

  return (
    <Suspense fallback={<p>Loading post...</p>}>
      <PostContent slug={slug} />
    </Suspense>
  )
}
```


### 元数据和机器人 (Bots)

[`generateMetadata`](/docs/app/api-reference/functions/generate-metadata) 在流式传输开始前解析，用于只抓取静态 HTML 的机器人（例如 Twitterbot 或 Slackbot）。对于完整的浏览器和有能力的爬虫，元数据可以与页面内容一起[流式传输](/docs/app/api-reference/functions/generate-metadata#streaming-metadata)。

Next.js 自动检测用户代理以选择正确的行为。您可以使用 [`htmlLimitedBots`](/docs/app/api-reference/config/next-config-js/htmlLimitedBots) 配置选项自定义哪些机器人接收阻塞式元数据。

## 哪些因素会影响流式传输

您的服务器和客户端之间任何缓冲响应的层都可能削弱流式传输的好处。HTML 可能在服务器上逐步完全生成，但如果代理、CDN 甚至客户端本身在渲染它们之前收集了所有数据块，用户看到的是单一的延迟响应，而不是渐进式渲染。

### 反向代理

Nginx 和类似的反向代理默认缓冲响应。通过将 `X-Accel-Buffering` 头设置为 `no` 来禁用缓冲：

```js filename="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*{/}?',
        headers: [
          {
            key: 'X-Accel-Buffering',
            value: 'no',
          },
        ],
      },
    ]
  },
}
```

### 内容分发网络 (CDNs)

内容分发网络 (CDN) 可能会在将响应转发给客户端之前缓冲整个响应。请查阅您的 CDN 提供商的文档以了解流式传输支持。有些需要特定的配置或计划层级才能传递分块响应。

### 无服务器平台

并非所有无服务器环境都支持流式传输。例如，AWS Lambda 需要显式启用[响应流式传输模式](https://docs.aws.amazon.com/lambda/latest/dg/configuration-response-streaming.html)（这不是默认设置）。Vercel 原生支持流式传输。

### 压缩

Gzip 和 Brotli 压缩在刷新之前可以在内部缓冲数据块，因为压缩算法需要足够的数据才能有效压缩。这可能会增加第一个可见数据块的延迟。如果您注意到流式传输延迟，请检查您的压缩层是否足够积极地刷新。

### 客户端

像 `curl` 这样的命令行工具也默认缓冲。`-N` 标志禁用输出缓冲，但 `curl` 仍然依赖换行符将行刷新到终端。发送不带换行符的数据块的流即使在使用 `-N` 时也可能看起来停滞。

### 验证流式传输是否有效

### 验证流式传输是否生效

**检查网络选项卡。** 在 Chrome 开发者工具中，选择文档请求并查看 "Timing" 分解。长 "Content Download" 阶段加上早的 "Time to First Byte" 确认响应是流式传输的，而不是一次性到达。

**观察原始数据块。** 要准确查看服务器发送的内容以及时间，使用一个小脚本将响应作为流读取。这对于观察定时数据块比 `curl` 更可靠，因为 `curl` 有自己的缓冲行为：

使用 `node stream-observer.mjs` 运行。对于有两个兄弟 Suspense 边界的页面（如[配套演示的 Suspense 页面](https://streaming-demo.labs.vercel.dev/suspense-demo)），您将看到类似以下的输出：

```js filename="stream-observer.mjs"
const res = await fetch(
  'https://streaming-demo.labs.vercel.dev/suspense-demo',
  {
    headers: { 'Accept-Encoding': 'identity' },
  }
)

const reader = res.body.getReader()
const decoder = new TextDecoder()
let i = 0
const start = Date.now()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  console.log(`\nchunk ${i++} (+${Date.now() - start}ms)\n`)
  console.log(decoder.decode(value))
}
```

`<template id="B:0">` 标记是 Suspense 后备占位符。当一个边界解析时，React 流式传输一个包含完成 HTML 的 `<div hidden id="S:0">` 和一个将其交换到页面中的脚本。时间戳显示每个边界独立解析。

```text filename="Terminal"
chunk 0 (+0ms)    # Static shell: <head>, CSS, nav, fallback skeletons,
                  # <template id="B:0"> and <template id="B:1"> placeholders,
                  # bootstrap scripts
chunk 1 (+170ms)  # Component payload (self.__next_f.push) for hydration
chunk 2 (+1000ms) # Weather widget: payload + <div hidden id="S:0"> (swaps B:0)
chunk 3 (+3000ms) # Analytics dashboard: payload + <div hidden id="S:1"> (swaps B:1)
```

> **注意：** `Accept-Encoding: identity` 头禁用压缩，因此数据块不会被压缩层缓冲。

## 总结

## 总结

关键决策是**缓存什么**以及**在哪里放置 Suspense 边界**。使用 [`"use cache"`](/docs/app/api-reference/directives/use-cache) 缓存尽可能多的内容以扩大静态外壳。将动态访问推送到需要它的组件，并用 `<Suspense>` 包装它们。其他所有内容都成为外壳的一部分。
