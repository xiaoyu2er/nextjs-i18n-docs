打包是将应用代码及其依赖项合并为客户端和服务器的优化输出文件的过程。更小的构建加载更快，减少 JavaScript 执行时间，改善 [核心网络指标 (Core Web Vitals)](https://web.dev/articles/vitals)，并降低服务器冷启动时间。

Next.js 通过代码分割、摇树优化等技术自动优化构建。然而，在某些情况下您可能需要手动优化构建。

有两个工具可以分析您的应用构建：

- [Turbopack 专用 Next.js 包分析器 (实验性) (#nextjs-bundle-analyzer-experimental)](#nextjs-bundle-analyzer-experimental)
- [Webpack 的 `@next/bundle-analyzer` 插件](#nextbundle-analyzer-for-webpack)

本指南将介绍如何使用每个工具，以及如何 [优化大型构建](#optimizing-large-bundles)。

## Next.js Bundle Analyzer (实验性)

> 支持 v16.1 及更高版本。您可以在 [专用的 GitHub 讨论](https://github.com/vercel/next.js/discussions/86731) 中分享反馈，并查看 [turbopack-bundle-analyzer-demo.vercel.sh](https://turbopack-bundle-analyzer-demo.vercel.sh/) 上的演示。

Next.js Bundle Analyzer 与 Turbopack 的模块图集成。您可以使用精确的导入追踪来检查服务器和客户端模块，从而更容易找到大型依赖项。打开交互式 Bundle Analyzer 演示以探索模块图。

### 第一步：运行 Turbopack Bundle Analyzer

要开始使用，请运行以下命令并在浏览器中打开交互式视图。

```bash filename="Terminal" package="npm"
npx next experimental-analyze
```

```bash filename="Terminal" package="yarn"
yarn next experimental-analyze
```

```bash filename="Terminal" package="pnpm"
pnpm next experimental-analyze
```

```bash filename="Terminal" package="bun"
bunx next experimental-analyze
```

### 第二步：过滤和检查模块

在 UI 中，您可以按路由、环境（客户端或服务器）和类型（JavaScript, CSS, JSON）进行过滤，或按文件搜索：

树状图将每个模块显示为矩形。模块的大小由矩形的面积表示。

点击模块以查看其大小，检查其完整的导入链，并查看它在应用中确切的使用位置：

### 第四步：将输出写入磁盘以便共享或比较差异

如果您想与队友分享分析结果，或在优化前后比较构建大小，您可以跳过交互式视图，并使用 `--output` 标志将分析保存为静态文件：

```bash filename="Terminal" package="npm"
npx next experimental-analyze --output
```

```bash filename="Terminal" package="yarn"
yarn next experimental-analyze --output
```

```bash filename="Terminal" package="pnpm"
pnpm next experimental-analyze --output
```

```bash filename="Terminal" package="bun"
bunx next experimental-analyze --output
```

此命令将输出写入 `.next/diagnostics/analyze`。您可以将此目录复制到其他位置以比较结果：

```bash filename="Terminal"
cp -r .next/diagnostics/analyze ./analyze-before-refactor
```

## Webpack 的 `@next/bundle-analyzer`

`[`@next/bundle-analyzer`](https://www.npmjs.com/package/@next/bundle-analyzer)` 是一个插件，可帮助您管理应用构建的大小。它会生成每个包及其依赖项大小的可视化报告。您可以使用这些信息来删除大型依赖项、拆分或 [懒加载](/docs/app/guides/lazy-loading) 代码。

### 第一步：安装

运行以下命令来安装该插件：

```bash package="pnpm"
pnpm add @next/bundle-analyzer
```

```bash package="npm"
npm install @next/bundle-analyzer
```

```bash package="yarn"
yarn add @next/bundle-analyzer
```

```bash package="bun"
bun add @next/bundle-analyzer
```

然后，将包分析器的设置添加到您的 `next.config.js` 文件中。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {}

const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})

module.exports = withBundleAnalyzer(nextConfig)
```

### 第二步：生成报告

运行以下命令来分析您的包：

```bash
ANALYZE=true npm run build
# or
ANALYZE=true yarn build
# or
ANALYZE=true pnpm build
```

报告将在浏览器中打开三个新标签页，您可以在其中进行检查。

## 优化大型构建

一旦您识别出大型模块，解决方案将取决于您的用例。以下是常见原因及解决方法：

### 导出项较多的包

如果您正在使用导出数百个模块的包（例如图标和实用程序库），您可以在 `next.config.js` 文件中使用 [`optimizePackageImports`](/docs/app/api-reference/config/next-config-js/optimizePackageImports) 选项来优化这些导入的解析方式。此选项将仅加载您 _实际_ 使用的模块，同时仍提供编写包含许多命名导入的导入语句的便利。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    optimizePackageImports: ['icon-library'],
  },
}

module.exports = nextConfig
```

> **小提示：** Next.js 还会自动优化一些库，因此它们不需要包含在 `optimizePackageImports` 列表中。查看 [完整列表](/docs/app/api-reference/config/next-config-js/optimizePackageImports) 以了解受支持的包。

### 重型客户端负载

客户端构建过大的一个常见原因是在客户端组件中执行昂贵的渲染工作。这通常发生在仅用于将数据转换为 UI 的库中，例如语法高亮、图表渲染或 Markdown 解析。

如果该工作不需要浏览器 API 或用户交互，可以在服务器组件中运行。

在此示例中，基于 Prism 的高亮器在客户端组件中运行。尽管最终输出只是一个 `<code>` 块，但整个高亮库都被打包到客户端 JavaScript 构建中：

```tsx filename="app/blog/[slug]/page.tsx"
'use client'

import Highlight from 'prism-react-renderer'
import theme from 'prism-react-renderer/themes/github'

export default function Page() {
  const code = `export function hello() {
    console.log("hi")
  }`

  return (
    <article>
      <h1>Blog Post Title</h1>

      {/* The prism package and its tokenization logic are shipped to the client */}
      <Highlight code={code} language="tsx" theme={theme}>
        {({ className, style, tokens, getLineProps, getTokenProps }) => (
          <pre className={className} style={style}>
            <code>
              {tokens.map((line, i) => (
                <div key={i} {...getLineProps({ line })}>
                  {line.map((token, key) => (
                    <span key={key} {...getTokenProps({ token })} />
                  ))}
                </div>
              ))}
            </code>
          </pre>
        )}
      </Highlight>
    </article>
  )
}
```

这增加了构建大小，因为客户端必须下载并执行高亮库，即使结果是静态 HTML。

相反，将高亮逻辑移动到服务器组件并在服务器上渲染最终的 HTML。客户端将只接收渲染后的标记。

```tsx filename="app/blog/[slug]/page.tsx"
import { codeToHtml } from 'shiki'

export default async function Page() {
  const code = `export function hello() {
    console.log("hi")
  }`

  // The Shiki package runs on the server and is never bundled for the client.
  const highlightedHtml = await codeToHtml(code, {
    lang: 'tsx',
    theme: 'github-dark',
  })

  return (
    <article>
      <h1>Blog Post Title</h1>

      {/* Client receives plain markup */}
      <pre>
        <code dangerouslySetInnerHTML={{ __html: highlightedHtml }} />
      </pre>
    </article>
  )
}
```

### 特定包排除在构建之外

在服务器组件和路由处理器中导入的包会被 Next.js 自动打包。

您可以在 `next.config.js` 中使用 [`serverExternalPackages`](/docs/app/api-reference/config/next-config-js/serverExternalPackages) 选项来将特定包排除在构建之外。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  serverExternalPackages: ['package-name'],
}

module.exports = nextConfig
```

### 未预打包的外部包

默认情况下，导入到您的应用中的包不会被打包。如果外部包未预打包（例如，从 monorepo 或 `node_modules` 导入），这可能会影响性能。

若要打包特定包，您可以在 `next.config.js` 中使用 [`transpilePackages`](/docs/app/api-reference/config/next-config-js/transpilePackages) 选项。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ['package-name'],
}

module.exports = nextConfig
```

若要自动打包所有包，您可以在 `next.config.js` 中使用 [`bundlePagesRouterDependencies`](/docs/pages/api-reference/config/next-config-js/bundlePagesRouterDependencies) 选项。

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  bundlePagesRouterDependencies: true,
}

module.exports = nextConfig
```

### 特定包排除在构建之外

如果您识别出不应包含在构建中的包，您可以在 `next.config.js` 中使用 [`serverExternalPackages`](/docs/pages/api-reference/config/next-config-js/serverExternalPackages) 选项将特定包排除在自动构建之外：

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  // Automatically bundle external packages:
  bundlePagesRouterDependencies: true,
  // Opt specific packages out of bundling:
  serverExternalPackages: ['package-name'],
}

module.exports = nextConfig
```
