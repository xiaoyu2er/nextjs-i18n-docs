Next.js 旨在提供出色的开发者体验。随着应用程序的增长，您可能会注意到本地开发期间的编译时间变慢。本指南将帮助您识别和解决常见的编译时性能问题。

## 本地开发与生产环境

使用 `next dev` 的开发过程与 `next build` 和 `next start` 不同。

`next dev` 会在您打开或导航到应用程序中的路由时编译它们。这使您可以在不等待应用程序中每个路由都编译完成的情况下启动开发服务器，这既更快也占用更少内存。运行生产构建会应用其他优化，例如压缩文件和创建内容哈希，这些对于本地开发不是必需的。

## 提高本地开发性能

### 1. 检查计算机的防病毒软件

杀毒软件可能会减慢文件访问速度。虽然这在 Windows 机器上更为常见，但安装了杀毒工具的任何系统都可能出现此问题。

在 Windows 上，您可以将您的项目添加到 [Microsoft Defender 杀毒软件排除列表](https://support.microsoft.com/en-us/windows/virus-and-threat-protection-in-the-windows-security-app-1362f4cd-d71a-b52a-0b66-c2820032b65e#bkmk_threat-protection-settings)。

1. 打开 **Windows 安全中心** 应用程序，然后选择 **病毒和威胁防护** → **管理设置** → **管理排除项**。
2. 添加 **文件夹** 排除项。选择您的项目文件夹。

在 macOS 上，您可以在终端中禁用 [门控程序 (Gatekeeper)](https://support.apple.com/guide/security/gatekeeper-and-runtime-protection-sec5599b66df/web)。

1. 在终端中运行 `sudo spctl developer-mode enable-terminal`。
2. 打开 **"系统设置"** 应用程序，然后选择 **"隐私与安全性"** → **"开发者工具"**。
3. 确保您的终端已列出并启用。如果您使用的是 iTerm 或 Ghostty 等第三方终端，请将其添加到列表中。
4. 重启您的终端。

### 2. 更新 Next.js 并使用 Turbopack

确保您使用的是最新版本的 Next.js。每个新版本通常都包含性能改进。

Turbopack 现在是 Next.js 开发默认的打包器，相较于 webpack 提供了显著的性能提升。

```bash package="pnpm"
pnpm add next@latest
pnpm dev  # Turbopack is used by default
```

```bash package="npm"
npm install next@latest
npm run dev  # Turbopack is used by default
```

```bash package="yarn"
yarn add next@latest
yarn dev  # Turbopack is used by default
```

```bash package="bun"
bun add next@latest
bun dev  # Turbopack is used by default
```

如果您需要不使用 Turbopack 而使用 Webpack，您可以手动选择启用：

```bash package="pnpm"
pnpm dev --webpack
```

```bash package="npm"
npm run dev -- --webpack
```

```bash package="yarn"
yarn dev --webpack
```

```bash package="bun"
bun run dev --webpack
```

[了解更多关于 Turbopack](/blog/turbopack-for-development-stable)。更多信息请参阅我们的[升级指南](/docs/app/guides/upgrading)和 codemods。

### 3. 检查你的导入

导入代码的方式会极大地影响编译和打包时间。了解更多关于[优化包打包](/docs/app/guides/package-bundling)的信息，并探索像 [Dependency Cruiser](https://github.com/sverweij/dependency-cruiser) 或 [Madge](https://github.com/pahen/madge) 这样的工具。

#### 图标库

诸如 `@material-ui/icons`、`@phosphor-icons/react` 或 `react-icons` 这样的库可能会导入成千上万个图标，即使你只使用其中少量。尝试只导入你需要的图标：

```jsx
// Instead of this:
import { TriangleIcon } from '@phosphor-icons/react'

// Do this:
import { TriangleIcon } from '@phosphor-icons/react/dist/csr/Triangle'
```

你通常可以在所使用的图标库的文档中找到要采用的导入模式。此示例遵循 [`@phosphor-icons/react`](https://www.npmjs.com/package/@phosphor-icons/react#import-performance-optimization) 建议。

像 `react-icons` 这样的库包含许多不同的图标集。选择一个图标集并坚持使用该集合。

例如，如果您的应用程序使用 `react-icons` 并导入了所有这些：

- `pi` (Phosphor Icons)
- `md` (Material Design Icons)
- `tb` (tabler-icons)
- `cg` (cssgg)

即使您只从每个库中使用单个导入，它们结合起来也将是数万个编译器需要处理的模块。

#### Barrel 文件

"Barrel 文件"是从其他文件导出许多项目的文件。它们会减慢构建速度，因为编译器必须解析它们，以通过导入来查找模块作用域中是否存在副作用。

尽可能尝试直接从特定文件导入。[了解更多关于 barrel 文件](https://vercel.com/blog/how-we-optimized-package-imports-in-next-js)以及 Next.js 中的内置优化。

#### 优化包导入

Next.js 可以自动优化某些包的导入。如果您使用利用 barrel 文件的包，请将它们添加到您的 `next.config.js` 中：

```jsx
module.exports = {
  experimental: {
    optimizePackageImports: ['package-name'],
  },
}
```

Turbopack 会自动分析导入并进行优化。它不需要此配置。

### 4. 检查你的 Tailwind CSS 设置

如果您使用 Tailwind CSS，请确保其设置正确。

一个常见的错误是以包含 `node_modules` 或其他不应扫描的大文件目录的方式配置 `content` 数组。

Tailwind CSS 版本 3.4.8 或更高版本会警告您可能导致构建速度变慢的设置。

1.  在您的 `tailwind.config.js` 中，具体指定要扫描的文件：

    ```jsx
    module.exports = {
      content: [
        './src/**/*.{js,ts,jsx,tsx}', // 好
        // 这可能过于宽泛
        // 它也会匹配 `packages/**/node_modules`
        // '../../packages/**/*.{js,ts,jsx,tsx}',
      ],
    }
    ```

2.  避免扫描不必要的文件：

    ```jsx
    module.exports = {
      content: [
        // 更好 - 只扫描 'src' 文件夹
        '../../packages/ui/src/**/*.{js,ts,jsx,tsx}',
      ],
    }
    ```

### 5. 检查自定义 webpack 设置

如果您添加了自定义 webpack 设置，它们可能会减慢编译速度。

考虑您是否真的需要在本地开发中使用它们。您也可以仅在生产构建中包含特定工具，或者尝试使用默认的 Turbopack 打包器并配置 [加载器 (loaders)](/docs/app/api-reference/config/next-config-js/turbopack#configuring-webpack-loaders)。

### 6. 优化内存使用

如果您的应用程序非常大，它可能需要更多内存。

[了解更多关于优化内存使用](/docs/app/guides/memory-usage)。

### 7. 服务器组件与数据获取

对服务器组件的更改会导致整个页面在本地重新渲染以显示新更改，这包括为组件获取新数据。

实验性的 `serverComponentsHmrCache` 选项允许您在本地开发期间，通过热模块替换 (HMR) 刷新来缓存服务器组件中的 `fetch` 响应。这会导致更快的响应速度并减少计费 API 调用的成本。

[了解更多关于实验性选项](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache)。

### 8. 考虑优先使用本地开发而非 Docker

如果您在 Mac 或 Windows 上使用 Docker 进行开发，您可能会遇到比本地运行 Next.js 显著更慢的性能。

Docker 在 Mac 和 Windows 上的文件系统访问可能导致热模块替换 (HMR) 需要数秒甚至数分钟，而相同的应用程序在本地开发时却能实现快速的 HMR。

这种性能差异是由于 Docker 在 Linux 环境之外处理文件系统操作的方式所致。为了获得最佳开发体验：

-   在开发期间使用本地开发（`npm run dev` 或 `pnpm dev`）而不是 Docker
-   将 Docker 保留用于生产部署和测试生产构建
-   如果您必须在开发中使用 Docker，请考虑在 Linux 机器或虚拟机上使用 Docker

[了解更多关于用于生产的 Docker 部署](/docs/app/getting-started/deploying#docker)。

## 查找问题的工具

### 详细 fetch 日志记录

在您的 `next.config.js` 文件中使用 `logging.fetches` 选项，以查看有关开发期间发生情况的更详细信息：

```js
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

[了解更多关于 fetch 日志记录](/docs/app/api-reference/config/next-config-js/logging)。

### Turbopack 跟踪

Turbopack 跟踪是一个帮助您了解本地开发期间应用程序性能的工具。它提供有关每个模块编译所需时间以及它们如何关联的详细信息。

1. 确保已安装最新版本的 Next.js。
1. 生成 Turbopack 跟踪文件：

   ```bash package="pnpm"
   NEXT_TURBOPACK_TRACING=1 pnpm dev
   ```

   ```bash package="npm"
   NEXT_TURBOPACK_TRACING=1 npm run dev
   ```

   ```bash package="yarn"
   NEXT_TURBOPACK_TRACING=1 yarn dev
   ```

   ```bash package="bun"
   NEXT_TURBOPACK_TRACING=1 bun dev
   ```

1. 导航到应用程序或编辑文件以重现问题。
1. 停止 Next.js 开发服务器。
1. `trace-turbopack` 文件将出现在 `.next/dev` 文件夹中。
1. 您可以使用 `npx next internal trace [path-to-file]` 查看该文件：

   ```bash
   npx next internal trace .next/dev/trace-turbopack
   ```

   在 `trace` 不可用的版本中，命令名为 `turbo-trace-server`：

   ```bash
   npx next internal turbo-trace-server .next/dev/trace-turbopack
   ```

1. 跟踪服务器运行后，您可以在 https://trace.nextjs.org/ 查看跟踪结果。
1. 默认情况下，跟踪查看器会聚合耗时。为了查看每个单独的耗时，您可以在查看器右上角从“按顺序聚合”切换到“按顺序跨度”。

> **温馨提示**：跟踪文件位于开发服务器目录下，默认值为 `.next/dev`。
