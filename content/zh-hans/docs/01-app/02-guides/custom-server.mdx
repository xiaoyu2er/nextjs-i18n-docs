Next.js 默认通过 `next start` 包含自己的服务器。如果您已有后端，仍然可以将其与 Next.js 一起使用（这不属于自定义服务器）。自定义 Next.js 服务器允许您以编程方式启动服务器以实现自定义模式。大多数情况下，您不需要这种方法。但是，如果您需要“弹出”，它是可用的。

> **须知**:
>
> - 在决定使用自定义服务器之前，请注意，只有在 Next.js 的集成路由器无法满足您的应用程序需求时才应使用它。自定义服务器将移除重要的性能优化，例如 **[自动静态优化 (Automatic Static Optimization)](/docs/pages/building-your-application/rendering/automatic-static-optimization)。**
> - 当使用独立输出模式时，它不会追踪自定义服务器文件。此模式会输出一个单独的最小化 `server.js` 文件。这两者不能一起使用。

请查看以下自定义服务器的[示例](https://github.com/vercel/next.js/tree/canary/examples/custom-server)：

```ts filename="server.ts" switcher
import { createServer } from 'http'
import next from 'next'

const port = parseInt(process.env.PORT || '3000', 10)
const dev = process.env.NODE_ENV !== 'production'
const app = next({ dev })
const handle = app.getRequestHandler()

app.prepare().then(() => {
  createServer((req, res) => {
    handle(req, res)
  }).listen(port)

  console.log(
    `> Server listening at http://localhost:${port} as ${
      dev ? 'development' : process.env.NODE_ENV
    }`
  )
})
```

```js filename="server.js" switcher
import { createServer } from 'http'
import next from 'next'

const port = parseInt(process.env.PORT || '3000', 10)
const dev = process.env.NODE_ENV !== 'production'
const app = next({ dev })
const handle = app.getRequestHandler()

app.prepare().then(() => {
  createServer((req, res) => {
    handle(req, res)
  }).listen(port)

  console.log(
    `> Server listening at http://localhost:${port} as ${
      dev ? 'development' : process.env.NODE_ENV
    }`
  )
})
```

> `server.js` 不经过 Next.js 编译器或打包流程。请确保此文件所需的语法和源代码与您当前使用的 Node.js 版本兼容。[查看示例](https://github.com/vercel/next.js/tree/canary/examples/custom-server)。

要运行自定义服务器，您需要像这样更新 `package.json` 中的 `scripts`：

```json filename="package.json"
{
  "scripts": {
    "dev": "node server.js",
    "build": "next build",
    "start": "NODE_ENV=production node server.js"
  }
}
```

或者，您可以设置 `nodemon`（[示例](https://github.com/vercel/next.js/tree/canary/examples/custom-server)）。自定义服务器使用以下导入将服务器与 Next.js 应用程序连接起来：

```js
import next from 'next'

const app = next({})
```

上述 `next` 导入是一个函数，它接收一个包含以下选项的对象：

| 选项         | 类型                | 说明                                                                                 |
| ------------ | ------------------- | ------------------------------------------------------------------------------------ |
| `conf`       | `Object`            | 与 `next.config.js` 相同的配置对象。默认为 `{}`                                      |
| `dev`        | `Boolean`           | （可选）是否以开发模式启动 Next.js。默认为 `false`                                   |
| `dir`        | `String`            | （可选）Next.js 项目路径。默认为 `'.'`                                               |
| `quiet`      | `Boolean`           | （可选）隐藏包含服务器信息的错误消息。默认为 `false`                                 |
| `hostname`   | `String`            | （可选）服务器运行的主机名                                                           |
| `port`       | `Number`            | （可选）服务器运行的端口号                                                           |
| `httpServer` | `node:http#Server`  | （可选）Next.js 运行的 HTTP 服务器实例                                               |
| `turbo`      | `Boolean`           | （可选）启用 Turbopack                                                               |

返回的 `app` 可用于让 Next.js 根据需要处理请求。

## 禁用文件系统路由

默认情况下，`Next` 会为 `pages` 文件夹中的每个文件提供与文件名匹配的路径名。如果您的项目使用自定义服务器，此行为可能导致从多个路径提供相同内容，这可能会带来 SEO 和 UX 问题。

要禁用此行为并防止基于 `pages` 中文件的路由，请打开 `next.config.js` 并禁用 `useFileSystemPublicRoutes` 配置：

```js filename="next.config.js"
module.exports = {
  useFileSystemPublicRoutes: false,
}
```

> 请注意，`useFileSystemPublicRoutes` 会禁用来自 SSR 的文件名路由；客户端路由可能仍能访问这些路径。使用此选项时，您应通过编程方式防止导航到您不希望的路由。

> 您可能还希望配置客户端路由器以禁止客户端重定向到文件名路由；为此请参考 [`router.beforePopState`](/docs/pages/api-reference/functions/use-router#routerbeforepopstate)。
