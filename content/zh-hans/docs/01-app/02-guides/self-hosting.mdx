在[部署](/docs/app/getting-started/deploying)你的 Next.js 应用时，你可能需要根据你的基础设施来配置不同功能的处理方式。

> **🎥 观看视频：** 了解更多关于自行托管 Next.js 的信息 → [YouTube (45 分钟)](https://www.youtube.com/watch?v=sIVL4JMqRfc)。

## 反向代理

自托管时，建议在您的 Next.js 服务器前端使用反向代理 (reverse proxy)（如 nginx），而不是直接暴露在互联网上。反向代理可以处理格式错误的请求、慢连接攻击、负载大小限制、速率限制和其他安全顾虑，将这些任务从 Next.js 服务器卸载。这允许服务器将其资源专门用于渲染，而不是请求验证。

## 图像优化

当使用 `next start` 部署时，通过 `next/image` 进行的[图像优化](/docs/app/api-reference/components/image)在自行托管环境下无需任何配置即可工作。如果你更倾向于使用单独的服务来优化图像，你可以[配置图像加载器](/docs/app/api-reference/components/image#loader)。

通过定义 `next.config.js` 中的自定义图像加载器，图像优化可以与[静态导出](/docs/app/guides/static-exports#image-optimization)一起使用。请注意，图像是在运行时优化的，而不是在构建期间。

> **值得注意的是：**
>
> - 在基于 glibc 的 Linux 系统上，图像优化可能需要[额外配置](https://sharp.pixelplumbing.com/install#linux-memory-allocator)以防止内存使用过高。
> - 了解更多关于[优化图像的缓存行为](/docs/app/api-reference/components/image#minimumcachettl)以及如何配置 TTL。
> - 你也可以[禁用图像优化](/docs/app/api-reference/components/image#unoptimized)，如果你愿意，仍然可以保留使用 `next/image` 的其他好处。例如，如果你自己单独优化图像。

## 代理 (Proxy)

[代理 (Proxy)](/docs/app/api-reference/file-conventions/proxy) 在自托管模式下工作零配置，当使用 `next start` 部署时。由于它需要访问传入请求，因此在采用 [static export](/docs/app/guides/static-exports) 时不受支持。

代理 (Proxy) 使用 [Edge 运行时 (Edge runtime)](/docs/app/api-reference/edge)，它是所有可用 Node.js API 的一个子集，有助于确保低延迟，因为它可能运行在您的应用程序的每个路由或资产前端。如果您不需要此项，可以使用 [完整 Node.js 运行时 (full Node.js runtime)](/blog/next-15-2#nodejs-middleware-experimental) 来运行代理 (Proxy)。

如果你想要添加需要所有 Node.js APIs 的逻辑（或使用外部包），你可能可以将此逻辑移动到[布局](/docs/app/api-reference/file-conventions/layout)中作为[服务器组件](/docs/app/getting-started/server-and-client-components)。例如，检查[请求头](/docs/app/api-reference/functions/headers)和[重定向](/docs/app/api-reference/functions/redirect)。你也可以使用请求头、Cookie 或查询参数通过 `next.config.js` 进行[重定向](/docs/app/api-reference/config/next-config-js/redirects#header-cookie-and-query-matching)或[重写](/docs/app/api-reference/config/next-config-js/rewrites#header-cookie-and-query-matching)。如果这不行，你还可以使用[自定义服务器](/docs/pages/guides/custom-server)。

## 环境变量

Next.js 可以同时支持构建时和运行时环境变量。

**默认情况下，环境变量仅在服务器上可用**。要将环境变量暴露给浏览器，它必须以 `NEXT_PUBLIC_` 为前缀。但是，这些公共环境变量将在 `next build` 期间内联到 JavaScript 包中。

<PagesOnly>

要读取运行时环境变量，我们建议使用 `getServerSideProps` 或[逐步采用应用路由器](/docs/app/guides/migrating/app-router-migration)。

</PagesOnly>

<AppOnly>

在动态渲染期间，你可以在服务器上安全地读取环境变量。

```tsx filename="app/page.ts" switcher
import { connection } from 'next/server'

export default async function Component() {
  await connection()
  // cookies, headers, and other Request-time APIs
  // will also opt into dynamic rendering, meaning
  // this env variable is evaluated at runtime
  const value = process.env.MY_VALUE
  // ...
}
```

```jsx filename="app/page.js" switcher
import { connection } from 'next/server'

export default async function Component() {
  await connection()
  // cookies, headers, and other Request-time APIs
  // will also opt into dynamic rendering, meaning
  // this env variable is evaluated at runtime
  const value = process.env.MY_VALUE
  // ...
}
```

</AppOnly>

这允许你使用一个单一的 Docker 镜像，该镜像可以在具有不同值的多个环境中进行推广。

> **小知识:**
>
> - 您可以使用 [`register`](/docs/app/guides/instrumentation) 函数在服务器启动时运行代码。

## 缓存和 ISR

Next.js 可以缓存响应、生成的静态页面、构建输出以及其他静态资源，如图像、字体和脚本。

缓存和重新验证页面（使用[增量静态再生](/docs/app/guides/incremental-static-regeneration)）使用**同一个共享缓存**。默认情况下，此缓存存储在你的 Next.js 服务器上的文件系统（磁盘）上。**使用页面路由器和应用路由器在自行托管时，这会自动工作**。

如果你想将缓存的页面和数据持久化到持久存储，或者在多个 Next.js 应用程序容器或实例之间共享缓存，你可以配置 Next.js 缓存位置。

### 自动缓存

- Next.js 为真正不可变的资源设置 `Cache-Control` 头为 `public, max-age=31536000, immutable`。它不能被覆盖。这些不可变的文件在文件名中包含 SHA 哈希值，因此可以安全地无限期缓存。例如，[静态图像导入](/docs/app/getting-started/images#local-images)。你可以[配置 TTL](/docs/app/api-reference/components/image#minimumcachettl) 用于图像。
- 增量静态再生 (ISR) 设置 `Cache-Control` 头为 `s-maxage: <revalidate in getStaticProps>, stale-while-revalidate`。这个重新验证时间在你的 [`getStaticProps` 函数](/docs/pages/building-your-application/data-fetching/get-static-props)中以秒为单位定义。如果你设置 `revalidate: false`，它将默认为一年的缓存持续时间。
- 动态渲染的页面设置 `Cache-Control` 头为 `private, no-cache, no-store, max-age=0, must-revalidate`，以防止缓存特定于用户的数据。这适用于应用路由器和页面路由器。这也包括[草稿模式](/docs/app/guides/draft-mode)。

### 静态资源

如果你想将静态资源托管在不同的域或 CDN 上，你可以在 `next.config.js` 中使用 `assetPrefix` [配置](/docs/app/api-reference/config/next-config-js/assetPrefix)。Next.js 在检索 JavaScript 或 CSS 文件时将使用此资源前缀。将资源分离到不同的域确实存在 DNS 和 TLS 解析额外时间的缺点。

[了解更多关于 `assetPrefix`](/docs/app/api-reference/config/next-config-js/assetPrefix)。

### 配置缓存

默认情况下，生成的缓存资源将存储在内存中（默认为 50mb）和磁盘上。如果你使用像 Kubernetes 这样的容器编排平台托管 Next.js，每个 pod 都将有一份缓存的副本。为了防止由于缓存默认在 pod 之间不共享而显示陈旧数据，你可以配置 Next.js 缓存以提供缓存处理程序并禁用内存缓存。

要在自托管时配置缓存位置，您可以在 `next.config.js` 文件中配置自定义处理器:

```jsx filename="next.config.js"
module.exports = {
  cacheHandler: require.resolve('./cache-handler.js'),
  cacheMaxMemorySize: 0, // disable default in-memory caching
}
```

然后，在项目的根目录创建 `cache-handler.js`，例如：

```jsx filename="cache-handler.js"
const cache = new Map()

module.exports = class CacheHandler {
  constructor(options) {
    this.options = options
  }

  async get(key) {
    // This could be stored anywhere, like durable storage
    return cache.get(key)
  }

  async set(key, data, ctx) {
    // This could be stored anywhere, like durable storage
    cache.set(key, {
      value: data,
      lastModified: Date.now(),
      tags: ctx.tags,
    })
  }

  async revalidateTag(tags) {
    // tags is either a string or an array of strings
    tags = [tags].flat()
    // Iterate over all entries in the cache
    for (let [key, value] of cache) {
      // If the value's tags include the specified tag, delete this entry
      if (value.tags.some((tag) => tags.includes(tag))) {
        cache.delete(key)
      }
    }
  }

  // If you want to have temporary in memory cache for a single request that is reset
  // before the next request you can leverage this method
  resetRequestCache() {}
}
```

使用自定义缓存处理程序将使你能够确保托管 Next.js 应用程序的所有 pod 之间的一致性。例如，你可以将缓存值保存在任何地方，比如 [Redis](https://github.com/vercel/next.js/tree/canary/examples/cache-handler-redis) 或 AWS S3。

> **值得注意的是：**
>
> - `revalidatePath` 是缓存标签之上的一个便捷层。调用 `revalidatePath` 将使用为提供的页面设置的特殊默认标签调用 `revalidateTag` 函数。

## 构建缓存

Next.js 在 `next build` 期间生成一个 ID，以标识正在提供服务的应用程序版本。应该使用相同的构建来启动多个容器。

如果你正在为环境的每个阶段重新构建，你将需要生成一个一致的构建 ID 以在容器之间使用。在 `next.config.js` 中使用 `generateBuildId` 命令：

```jsx filename="next.config.js"
module.exports = {
  generateBuildId: async () => {
    // This could be anything, using the latest git hash
    return process.env.GIT_HASH
  },
}
```

## 多服务器部署

当在多个服务器实例上运行 Next.js 时（例如，负载均衡器背后的容器），还有其他注意事项以确保行为一致。

### 服务端函数 (Server Functions) 加密密钥

Next.js 在向客户端发送之前加密 [服务端函数 (Server Function)](/docs/app/getting-started/mutating-data) 闭包变量。默认情况下，每次构建都会生成唯一的加密密钥。

在运行多个服务器实例时，所有实例必须使用相同的加密密钥。否则，一个实例加密的服务端函数 (Server Function)，另一个实例无法解密，导致 "Failed to find Server Action" 错误。

使用 `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 环境变量设置一致的加密密钥。密钥必须是 base64 编码的值，并具有有效的 AES 密钥长度（16、24 或 32 字节）。Next.js 默认生成 32 字节密钥。

```bash
NEXT_SERVER_ACTIONS_ENCRYPTION_KEY=your-generated-key next build
```

密钥嵌入在构建输出中，并在运行时自动使用。在 [数据安全指南](/docs/app/guides/data-security#overwriting-encryption-keys-advanced) 中了解更多。

### 部署标识符

配置 [`deploymentId`](/docs/app/api-reference/config/next-config-js/deploymentId) 以在滚动部署期间启用版本倾斜保护。这确保客户端始终从一致的部署版本接收资产。

### 共享缓存

默认情况下，Next.js 使用未跨实例共享的内存缓存。为了实现一致的缓存行为，使用 [`'use cache: remote'`](/docs/app/api-reference/directives/use-cache-remote) 配合 [自定义缓存处理器](/docs/app/api-reference/config/next-config-js/cacheHandlers)，将数据存储到外部存储中。

## 版本偏移

当跨多个实例自托管或进行滚动部署时，[版本倾斜](/docs/app/glossary#version-skew) 可能导致：

- **缺少资产**：客户端请求不再存在于服务器上的 JavaScript 或 CSS 文件
- **服务端函数 (Server Function) 不匹配**：客户端使用先前构建的 ID 调用服务端函数 (Server Function)，服务器不再识别
- **导航失败**：来自旧部署的预获取页面数据与新的服务器不兼容

Next.js 使用 [`deploymentId`](/docs/app/api-reference/config/next-config-js/deploymentId) 来检测和处理版本倾斜。当配置了部署 ID 时：

- 静态资产包含 `?dpl=<deploymentId>` 查询参数
- 客户端导航请求包含 `x-deployment-id` 请求头
- 服务器将客户端的部署 ID 与自己的进行比较

如果检测到不匹配，Next.js 会触发硬导航（完整页面重新加载）而不是客户端导航。这确保客户端从一致的部署版本获取资产。

```js filename="next.config.js"
module.exports = {
  deploymentId: process.env.DEPLOYMENT_VERSION,
}
```

> **小知识：**当应用程序重新加载时，如果未在页面导航之间设计为持久化，可能会丢失应用程序状态。URL 状态或本地存储将持久化，但像 `useState` 这样的组件状态会丢失。

## 流式传输和 Suspense

Next.js App Router 在自托管时支持 [流式响应 (streaming responses)](/docs/app/api-reference/file-conventions/loading)。如果您正在使用 nginx 或类似代理，则需要配置它以禁用缓冲以启用流式传输。

例如，您可以通过将 `X-Accel-Buffering` 设置为 `no` 来在 nginx 中禁用缓冲：

```js filename="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*{/}?',
        headers: [
          {
            key: 'X-Accel-Buffering',
            value: 'no',
          },
        ],
      },
    ]
  },
}
```

## 缓存组件 (Cache Components)

[缓存组件](/docs/app/getting-started/caching) 默认与 Next.js 配合工作，不仅仅是 CDN 功能。这包括作为 Node.js 服务器部署（通过 `next start`）以及在使用 Docker 容器时。

## 使用 CDN

当在您的 Next.js 应用程序前端使用 CDN 时，当访问动态 API 时，页面将包含 `Cache-Control: private` 响应头。这确保生成的 HTML 页面标记为不可缓存。如果页面完全预渲染为静态，它将包含 `Cache-Control: public` 以允许页面在 CDN 上缓存。

如果你不需要静态和动态组件的混合，你可以让你的整个路由成为静态的，并将输出 HTML 缓存在 CDN 上。如果没有使用动态 API，这种自动静态优化是运行 `next build` 时的默认行为。

随着部分预渲染转向稳定，我们将通过部署适配器 API 提供支持。

</AppOnly>

## `after`

使用 `next start` 自行托管时，完全支持 [`after`](/docs/app/api-reference/functions/after)。

停止服务器时，请通过发送 `SIGINT` 或 `SIGTERM` 信号并等待来确保优雅关闭。这允许 Next.js 服务器等待，直到在 `after` 中使用的待处理回调函数或 Promise 完成。

</AppOnly>

## 手动优雅关闭

在自行托管时，你可能希望在服务器收到 `SIGTERM` 或 `SIGINT` 信号关闭时运行代码。

你可以将环境变量 `NEXT_MANUAL_SIG_HANDLE` 设置为 `true`，然后在你的 `_document.js` 文件中为该信号注册一个处理程序。你需要直接在 `package.json` 脚本中注册环境变量，而不是在 `.env` 文件中。

> **值得注意的是**：手动信号处理在 `next dev` 中不可用。

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "NEXT_MANUAL_SIG_HANDLE=true next start"
  }
}
```

```js filename="pages/_document.js"
if (process.env.NEXT_MANUAL_SIG_HANDLE) {
  process.on('SIGTERM', () => {
    console.log('Received SIGTERM: cleaning up')
    process.exit(0)
  })
  process.on('SIGINT', () => {
    console.log('Received SIGINT: cleaning up')
    process.exit(0)
  })
}
```

</PagesOnly>
