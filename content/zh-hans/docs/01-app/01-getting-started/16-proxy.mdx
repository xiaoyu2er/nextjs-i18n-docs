## 代理 (Proxy)

> **须知**：从 Next.js 16 开始，中间件 (Middleware) 现在被称为代理 (Proxy)，以更好地反映其用途。其功能保持不变。

代理 (Proxy) 允许您在请求完成之前运行代码。然后，根据传入的请求，您可以通过重写、重定向、修改请求或响应头，或者直接响应来修改响应。

### 使用场景

代理 (Proxy) 适用的一些常见场景包括：

- 为所有页面或部分页面修改请求头
- 根据 A/B 测试或实验重写到不同的页面
- 基于传入请求属性进行程序化重定向

对于简单的重定向，请首先考虑在 `next.config.ts` 中使用 [`redirects`](/docs/app/api-reference/config/next-config-js/redirects) 配置。只有当您需要访问请求数据或进行更复杂的逻辑处理时，才应使用代理 (Proxy)。

代理 (Proxy) 并非为获取慢速数据而设计。虽然代理 (Proxy) 对于[乐观检查](/docs/app/guides/authentication#optimistic-checks-with-proxy-optional)（例如基于权限的重定向）可能有帮助，但它不应用作完整的会话管理或授权解决方案。

在代理 (Proxy) 中使用带有 `options.cache`、`options.next.revalidate` 或 `options.next.tags` 的 `fetch` 是无效的。

### 约定

在项目根目录（或 `src` 目录内，如果适用）创建一个 `proxy.ts`（或 `.js`）文件，使其与 `pages` 或 `app` 位于同一层级。

> **注意**：虽然每个项目仅支持一个 `proxy.ts` 文件，但您仍然可以将您的代理逻辑组织到模块中。将代理功能拆分成单独的 `.ts` 或 `.js` 文件，并将它们导入到您的主 `proxy.ts` 文件中。这样可以更清晰地管理特定于路由的代理，并汇总到 `proxy.ts` 中以进行集中控制。通过强制使用单个代理文件，可以简化配置，防止潜在冲突，并通过避免多个代理层来优化性能。

### 示例

您可以将您的代理函数导出为默认导出或命名导出 `proxy`：

```ts filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url))
}

// Alternatively, you can use a default export:
// export default function proxy(request: NextRequest) { ... }

export const config = {
  matcher: '/about/:path*',
}
```

```js filename="proxy.js" switcher
import { NextResponse } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request) {
  return NextResponse.redirect(new URL('/home', request.url))
}

// Alternatively, you can use a default export:
// export default function proxy(request) { ... }

export const config = {
  matcher: '/about/:path*',
}
```

`matcher` 配置允许您过滤代理仅在特定路径上运行。有关路径匹配的更多详细信息，请参阅 [Matcher](/docs/app/api-reference/file-conventions/proxy#matcher) 文档。
