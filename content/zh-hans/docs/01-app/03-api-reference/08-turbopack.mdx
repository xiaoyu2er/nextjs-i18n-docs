Turbopack 是一个为 **JavaScript** 和 **TypeScript** 优化的**增量打包器**，用 **Rust** 编写，并内置于 **Next.js** 中。您可以在 Pages Router 和 App Router 中使用 Turbopack，以获得**快得多**的本地开发体验。

## 为什么选择 Turbopack?

我们构建 Turbopack 是为了推动 Next.js 的性能，包括：

- **统一图谱：** Next.js 支持多种输出环境（例如客户端和服务器）。管理多个编译器并将打包结果拼接在一起可能很繁琐。Turbopack 为所有环境使用**单一、统一的图谱**。
- **打包 vs 原生 ESM：** 一些工具在开发中跳过打包，依赖浏览器的原生 ESM。这对于小型应用程序效果很好，但由于过多的网络请求，可能会减慢大型应用程序的速度。Turbopack 在开发中进行**打包**，但采用优化方式以保持大型应用程序的快速。
- **增量计算：** Turbopack 将工作并行化到多个核心上，并将结果**缓存**到函数级别。一旦某项工作完成，Turbopack 就不会重复执行。
- **延迟打包：** Turbopack 仅打包开发服务器实际请求的内容。这种延迟方法可以减少初始编译时间和内存使用。

## 支持的平台

Turbopack 需要平台特定的原生绑定。目前支持以下平台：

| 平台             | 架构         |
| ---------------- | ------------ |
| macOS (Darwin)   | x64, ARM64   |
| Windows          | x64, ARM64   |
| Linux (glibc)    | x64, ARM64   |
| Linux (musl)     | x64, ARM64   |

在没有原生绑定的平台上（例如 FreeBSD、OpenBSD），Next.js 会回退到 WebAssembly (WASM) 绑定。WASM 绑定支持核心 SWC 功能，如编译和压缩，但**不支持 Turbopack**。在这些平台上，请使用 `--webpack` 标志：

```bash
next dev --webpack
next build --webpack
```

## 快速开始

Turbopack 现在是 Next.js 中的**默认打包器**。使用 Turbopack 无需任何配置：

```json filename="package.json" highlight={3}
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start"
  }
}
```

### 使用 Webpack 替代

如果您需要使用 Webpack 而不是 Turbopack，可以通过 `--webpack` 标志来启用：

```json filename="package.json"
{
  "scripts": {
    "dev": "next dev --webpack",
    "build": "next build --webpack",
    "start": "next start"
  }
}
```

## 支持的功能

Next.js 中的 Turbopack 对于常见用例具有**零配置**。以下是开箱即用支持的功能摘要，以及在需要时如何进一步配置 Turbopack 的一些参考。

### 语言特性

| 功能                          | 状态          | 备注                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------------------- | ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **JavaScript & TypeScript**   | **支持**      | 底层使用 SWC。Turbopack 不执行类型检查（请运行 `tsc --watch` 或依赖您的 IDE 进行类型检查）。                                                                                                                                                                                                                                                                                                                           |
| **ECMAScript (ESNext)**       | **支持**      | Turbopack 支持最新的 ECMAScript 特性，与 SWC 的覆盖范围相匹配。                                                                                                                                                                                                                                                                                                                                                        |

| **ESM**                       | **支持**      | 完全支持静态和动态 `import`。                                                                                                                                                                                                                                                                                                                                                                                         |
| **Babel**                     | **支持**      | 从 Next.js 16 开始，如果检测到[配置文件][babel-config]，Turbopack 会自动使用 Babel。与 webpack 不同，SWC 始终用于 Next.js 的内部转换和降级到旧版 ECMAScript 修订版本。使用 webpack 的 Next.js 会在存在 Babel 配置文件时禁用 SWC。`node_modules` 中的文件会被排除，除非您[手动配置 `babel-loader`][manual-loader]。                                                                                                |

| **Fast Refresh**        | **支持**      | 更新 JavaScript、TypeScript 和 CSS 无需完全刷新。                    |

### 魔法注释

Turbopack 支持 webpack 兼容的魔法注释来控制导入行为。这些注释适用于动态 `import()`、`require()`、`require.resolve()` 和 `new Worker()` 表达式（不适用于静态 `import` 语句）。

| 注释                      | Webpack | Turbopack | 描述                     |
| ------------------------- | ------- | --------- | ------------------------ |
| `webpackIgnore: true`     | ✓       | ✓         | 跳过打包，保留导入       |
| `turbopackIgnore: true`   | ✗       | ✓         | 跳过打包（仅 Turbopack） |
| `turbopackOptional: true` | ✗       | ✓         | 抑制解析错误             |
| `webpackOptional: true`   | ✗       | ✗         | 不支持                   |

有关用法示例，请参见[懒加载](/docs/app/guides/lazy-loading#magic-comments)。

## 与 webpack 的已知差距

在迁移应用程序时，了解 webpack 和 Turbopack 之间许多重要的行为差异非常重要。通常，对于新应用程序来说，这些差异不太重要。

### 文件系统根目录

Turbopack 使用根目录来解析模块。项目根目录之外的文件不会被解析。

例如，当链接项目根目录之外的依赖项时（通过 `npm link`、`yarn link`、`pnpm link` 等），默认情况下这些链接的文件不会被解析。要解析这些文件，您必须将根选项配置为项目和链接依赖项的父目录。

您可以使用 `next.config.js` 中的 [turbopack.root](/docs/app/api-reference/config/next-config-js/turbopack#root-directory) 选项配置文件系统根目录。

### CSS 模块排序

Turbopack 将遵循 JS 导入顺序来排列那些没有其他顺序的 [CSS modules](/docs/app/getting-started/css#css-modules)。例如：

```jsx filename="components/BlogPost.jsx"
import utilStyles from './utils.module.css'
import buttonStyles from './button.module.css'
export default function BlogPost() {
  return (
    <div className={utilStyles.container}>
      <button className={buttonStyles.primary}>Click me</button>
    </div>
  )
}
```

在此示例中，Turbopack 将确保在生成的 CSS 块中，`utils.module.css` 出现在 `button.module.css` 之前，遵循导入顺序。

Webpack 通常也会这样做，但在某些情况下会忽略 JS 推断的顺序，例如如果它推断 JS 文件是无副作用的。

当采用 Turbopack 时，如果应用程序依赖于任意顺序，这可能会导致细微的渲染变化。通常，解决方案很简单，例如让 `button.module.css` `@import utils.module.css` 来强制顺序，或者识别冲突的规则并更改它们，使它们不针对相同的属性。

### Sass node_modules 导入

Turbopack 支持开箱即用的导入 `node_modules` Sass 文件。Webpack 支持用于此目的的传统波浪号 `~` 语法，Turbopack 不支持此语法。

从：

```scss filename="styles/globals.scss"
@import '~bootstrap/dist/css/bootstrap.min.css';
```

改为：

```scss filename="styles/globals.scss"
@import 'bootstrap/dist/css/bootstrap.min.css';
```

如果您无法更新导入，可以添加 `turbopack.resolveAlias` 配置来将 `~` 语法映射到实际路径：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    resolveAlias: {
      '~*': '*',
    },
  },
}
```

### 构建缓存

Webpack 支持[磁盘构建缓存](https://webpack.js.org/configuration/cache/#cache)以提高构建性能。Turbopack 提供了类似的功能，目前处于测试阶段。从 Next.js 16 开始，您可以通过设置以下实验性标志来启用 Turbopack 的文件系统缓存：

- [`experimental.turbopackFileSystemCacheForDev`](/docs/app/api-reference/config/next-config-js/turbopackFileSystemCache) 默认启用
- [`experimental.turbopackFileSystemCacheForBuild`](/docs/app/api-reference/config/next-config-js/turbopackFileSystemCache) 目前需要手动启用

> **值得注意的是：** 因此，在比较 webpack 和 Turbopack 性能时，请确保在构建之间删除 `.next` 文件夹以进行公平的冷构建比较，或者启用 Turbopack 文件系统缓存功能以比较热构建。

### Webpack 插件

Turbopack 不支持 webpack 插件。这会影响依赖 webpack 插件系统进行集成的第三方工具。我们支持 [webpack 加载器](/docs/app/api-reference/config/next-config-js/turbopack#configuring-webpack-loaders)。如果您依赖 webpack 插件，则需要寻找 Turbopack 兼容的替代方案，或者继续使用 webpack，直到可用等效功能。

## 不支持且未计划的功能

一些功能尚未实现或未计划：

- **遗留 CSS Modules 功能**
  - 独立的 `:local` 和 `:global` 伪类（仅支持函数变体 `:global(...)`）。
  - `@value` 规则（已被 CSS 变量取代）。
  - `:import` 和 `:export` ICSS 规则。
  - `.module.css` 中的 `composes` 用于组合 `.css` 文件。在 webpack 中会将 `.css` 文件视为 CSS 模块，而在 Turbopack 中 `.css` 文件将始终为全局。这意味着如果你要在 CSS 模块中使用 `composes`，需要将 `.css` 文件更改为 `.module.css` 文件。
  - CSS 模块中的 `@import` 导入 `.css` 作为 CSS 模块。在 webpack 中会将 `.css` 文件视为 CSS 模块，而在 Turbopack 中 `.css` 文件将始终为全局。这意味着如果你要在 CSS 模块中使用 `@import`，需要将 `.css` 文件更改为 `.module.css` 文件。
- **`sassOptions.functions`**
  在 `sassOptions.functions` 中定义的自定义 Sass 函数不受支持。此功能允许在编译期间定义可从 Sass 代码调用的 JavaScript 函数。Turbopack 基于 Rust 的架构无法直接执行通过 `sassOptions.functions` 传递的 JavaScript 函数，而 webpack 基于 Node.js 的 sass-loader 则完全运行在 JavaScript 中。如果你使用自定义 Sass 函数，则需要使用 webpack 而不是 Turbopack。
- **`next.config.js` 中的 `webpack()` 配置**
  Turbopack 替换了 webpack，因此不识别 `webpack()` 配置。请使用 [`turbopack` 配置](/docs/app/api-reference/config/next-config-js/turbopack) 代替。
- **Yarn PnP**
  尚未计划在 Next.js 中支持 Turbopack。
- **`experimental.urlImports`**
  Turbopack 暂无支持计划。
- **`experimental.esmExternals`**
  暂无计划。Turbopack 不支持 Next.js 中的旧版 `esmExternals` 配置。
- **部分 Next.js 实验性标志**
  - `experimental.nextScriptWorkers`
  - `experimental.fallbackNodePolyfills`
    我们计划在未来实现这些功能。

有关每个功能标志及其状态的完整详细细分，请参阅 [Turbopack API 参考](/docs/app/api-reference/config/next-config-js/turbopack)。

## 配置

可以通过 `next.config.js`（或 `next.config.ts`）中的 `turbopack` 键来配置 Turbopack。配置选项包括：

- **`rules`**
  为文件转换定义额外的 [webpack 加载器](/docs/app/api-reference/config/next-config-js/turbopack#configuring-webpack-loaders)。
- **`resolveAlias`**
  创建手动别名（类似于 webpack 中的 `resolve.alias`）。
- **`resolveExtensions`**
  更改或扩展用于模块解析的文件扩展名。
- **[`ignoreIssue`](/docs/app/api-reference/config/next-config-js/turbopackIgnoreIssue)**
  从 CLI 输出和错误覆盖中抑制特定的 Turbopack 错误和警告。

此外，`next.config.js` 中的 `experimental` 下提供以下实验性选项：

| 选项                                                                                                         | 描述                                                                 | 默认值 (开发) | 默认值 (构建)             |
| ------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------- | ------------- | ------------------------- |
| [`turbopackFileSystemCacheForDev`](/docs/app/api-reference/config/next-config-js/turbopackFileSystemCache)   | 为开发服务器启用文件系统缓存。                                       | `true`        | 不适用                    |
| [`turbopackFileSystemCacheForBuild`](/docs/app/api-reference/config/next-config-js/turbopackFileSystemCache) | 为构建启用文件系统缓存。                                             | 不适用        | `false`                   |
| `turbopackMinify`                                                                                            | 启用压缩。                                                           | `false`       | `true`                    |
| `turbopackSourceMaps`                                                                                        | 启用源映射。                                                         | `true`        | `productionBrowserSourceMaps` |
| `turbopackInputSourceMaps`                                                                                   | 启用从输入文件提取源映射。                                           | `true`        | `true`                    |
| `turbopackTreeShaking`                                                                                       | 使用高级模块片段树摇，而不是默认的仅重新导出模式。                   | `false`       | `false`                   |
| `turbopackRemoveUnusedImports`                                                                               | 启用移除未使用的导入。需要 `turbopackRemoveUnusedExports`。          | `false`       | `true`                    |
| `turbopackRemoveUnusedExports`                                                                               | 启用移除未使用的导出。                                               | `false`       | `true`                    |
| `turbopackInferModuleSideEffects`                                                                            | 启用局部分析以推断无副作用模块，以实现更好的树摇。                   | `true`        | `true`                    |
| `turbopackScopeHoisting`                                                                                     | 启用作用域提升。在开发模式下始终禁用。                               | `false`       | `true`                    |
| `turbopackClientSideNestedAsyncChunking`                                                                     | 为客户端资产启用嵌套异步分块。                                       | `false`       | `true`                    |
| `turbopackServerSideNestedAsyncChunking`                                                                     | 为服务器端资产启用嵌套异步分块。                                     | `false`       | `false`                   |
| `turbopackImportTypeBytes`                                                                                   | 为 ESM 导入启用 `with {type: "bytes"}` 支持。                        | `false`       | `false`                   |
| `turbopackUseBuiltinBabel`                                                                                   | 当存在 Babel 配置文件时，启用自动 Babel 加载器配置。                 | `true`        | `true`                    |
| `turbopackUseBuiltinSass`                                                                                    | 启用自动 Sass 加载器配置。                                           | `true`        | `true`                    |
| `turbopackModuleIds`                                                                                         | 模块 ID 策略：`'named'` 或 `'deterministic'`。                       | `'named'`     | `'deterministic'`         |

```js filename="next.config.js"
module.exports = {
  turbopack: {
    // Example: adding an alias and custom file extension
    resolveAlias: {
      underscore: 'lodash',
    },
    resolveExtensions: ['.mdx', '.tsx', '.ts', '.jsx', '.js', '.json'],
  },
}
```

有关更深入的配置示例，请参阅 [Turbopack 配置文档](/docs/app/api-reference/config/next-config-js/turbopack)。

## 生成跟踪文件以进行性能调试

如果您遇到性能或内存问题，并希望帮助 Next.js 团队进行诊断，您可以通过在开发命令后附加 `NEXT_TURBOPACK_TRACING=1` 来生成跟踪文件：

```bash
NEXT_TURBOPACK_TRACING=1 next dev
```

这将生成一个 `.next/dev/trace-turbopack` 文件。在 [Next.js 仓库](https://github.com/vercel/next.js) 上创建 GitHub issue 时请附上该文件，以帮助我们进行调查。

默认情况下，开发服务器输出到 `.next/dev`。

## 总结

Turbopack 是一个基于 **Rust** 的、**增量式**打包器，旨在使本地开发和构建变得快速——尤其适用于大型应用程序。它集成在 Next.js 中，提供零配置的 CSS、React 和 TypeScript 支持。
