`after` 允许您安排在响应（或预渲染）完成后执行的工作。这对于不应阻塞响应的任务和其他副作用（例如日志记录和分析）非常有用。

它可在 [服务端组件 (Server Components)](/docs/app/getting-started/server-and-client-components)（包括 [`generateMetadata`](/docs/app/api-reference/functions/generate-metadata)）、[服务器函数 (Server Functions)](/docs/app/getting-started/mutating-data)、[路由处理器 (Route Handlers)](/docs/app/api-reference/file-conventions/route) 和 [代理 (Proxy)](/docs/app/api-reference/file-conventions/proxy) 中使用。

该函数接受一个回调函数，该函数将在响应（或预渲染）完成后执行：

```tsx filename="app/layout.tsx" switcher
import { after } from 'next/server'
// Custom logging function
import { log } from '@/app/utils'

export default function Layout({ children }: { children: React.ReactNode }) {
  after(() => {
    // Execute after the layout is rendered and sent to the user
    log()
  })
  return <>{children}</>
}
```

```jsx filename="app/layout.jsx" switcher
import { after } from 'next/server'
// Custom logging function
import { log } from '@/app/utils'

export default function Layout({ children }) {
  after(() => {
    // Execute after the layout is rendered and sent to the user
    log()
  })
  return <>{children}</>
}
```

> **要知道：** `after` 不是 [请求时 API (Request-time API)](/docs/app/glossary#request-time-apis)，调用它不会导致路由变为动态。如果它用于静态页面中，回调将在构建时或当页面重新验证时执行。

## 参考

### 参数

- 一个将在响应（或预渲染）完成后执行的回调函数。

### 持续时间

`after` 会根据平台默认或配置的路由最大持续时间运行。如果平台支持，你可以使用 [`maxDuration`](/docs/app/api-reference/file-conventions/route-segment-config/maxDuration) 路由段配置来设置超时限制。

## 小知识

- 即使响应没有成功完成，`after` 也会被执行。包括抛出错误或调用 `notFound` 或 `redirect` 的情况。
- 您可以使用 React `cache` 来对 `after` 内部调用的函数进行去重。
- `after` 可以嵌套在其他 `after` 调用中，例如，您可以创建包装 `after` 调用的工具函数以添加额外功能。

## 示例

### 使用请求 API

你能否在 `after` 内部使用像 [`cookies`](/docs/app/api-reference/functions/cookies) 和 [`headers`](/docs/app/api-reference/functions/headers) 这样的请求 API ，取决于在哪里调用 `after`。

#### 在路由处理器 (Route Handlers) 和服务器函数 (Server Functions) 中

当用于 [路由处理器 (Route Handlers)](/docs/app/api-reference/file-conventions/route) 和 [服务器函数 (Server Functions)](/docs/app/getting-started/mutating-data) 时，你可以直接在 `after` 回调中调用 `cookies` 和 `headers`。这在记录变更或 API 请求后的活动方面很有用。例如：

```ts filename="app/api/route.ts" highlight={2,10-16} switcher
import { after } from 'next/server'
import { cookies, headers } from 'next/headers'
import { logUserAction } from '@/app/utils'

export async function POST(request: Request) {
  // Perform mutation
  // ...

  // Log user activity for analytics
  after(async () => {
    const userAgent = (await headers()).get('user-agent') || 'unknown'
    const sessionCookie =
      (await cookies()).get('session-id')?.value || 'anonymous'

    logUserAction({ sessionCookie, userAgent })
  })

  return new Response(JSON.stringify({ status: 'success' }), {
    status: 200,
    headers: { 'Content-Type': 'application/json' },
  })
}
```

```js filename="app/api/route.js" highlight={2,10-16} switcher
import { after } from 'next/server'
import { cookies, headers } from 'next/headers'

```

#### 在服务端组件 (Server Components) 中（页面和布局）

[服务端组件 (Server Components)](/docs/app/getting-started/server-and-client-components)（包括页面、布局以及 `generateMetadata`）**不能**在 `after` 内部使用 `cookies`、`headers` 或其他请求时 API。这是因为 Next.js 需要知道组件树的哪一部分访问请求数据以支持 [部分预渲染 (Partial Prerendering)](/docs/app/glossary#partial-prerendering-ppr) 和 [缓存组件 (Cache Components)](/docs/app/getting-started/caching)，但 `after` 在 React 渲染生命周期之后运行。

如果你需要在服务器组件 (Server Component) 的 `after` 回调内获取请求数据，请预先读取并将值传进去：

```tsx filename="app/page.tsx" highlight={8-10,12} switcher
import { after } from 'next/server'
import { cookies, headers } from 'next/headers'
import { logUserAction } from '@/app/utils'

export default async function Page() {
  // Read request data before `after` — this is allowed
  // These calls will be read during the component's rendering lifecycle
  const userAgent = (await headers()).get('user-agent') || 'unknown'
  const sessionCookie =
    (await cookies()).get('session-id')?.value || 'anonymous'

  after(() => {
    // Use the values read above
    logUserAction({ sessionCookie, userAgent })
  })

  return <h1>My Page</h1>
}
```

```jsx filename="app/page.jsx" highlight={8-10,12} switcher
import { after } from 'next/server'
import { cookies, headers } from 'next/headers'
import { logUserAction } from '@/app/utils'

export default async function Page() {
  // Read request data before `after` — this is allowed
  // These calls will be read during the component's rendering lifecycle
  const userAgent = (await headers()).get('user-agent') || 'unknown'
  const sessionCookie =
    (await cookies()).get('session-id')?.value || 'anonymous'

  after(() => {
    // Use the values read above
    logUserAction({ sessionCookie, userAgent })
  })

  return <h1>My Page</h1>
}
```

在服务器组件的 `after` 回调中调用 `cookies()` 或 `headers()` 将引发运行时错误。

#### 使用 缓存 组件

当使用 [缓存组件 (Cache Components)](/docs/app/getting-started/caching) 时，访问 `cookies` 或 `headers` 等请求数据的组件必须包裹在 [`<Suspense>`](https://react.dev/reference/react/Suspense) 中，以便页面的其余部分可以被预渲染为静态外壳。

你可以通过在动态组件中读取请求数据并将其传递给 `after` 来将此模式与 `after` 结合使用：

```tsx filename="app/page.tsx" highlight={18-19,22-24} switcher
import { Suspense } from 'react'
import { after } from 'next/server'
import { cookies } from 'next/headers'
import { logUserAction } from '@/app/utils'

export default function Page() {
  return (
    <>
      <h1>Part of the static shell</h1>
      <Suspense fallback={<p>Loading...</p>}>
        <DynamicContent />
      </Suspense>
    </>
  )
}

async function DynamicContent() {
  const sessionCookie =
    (await cookies()).get('session-id')?.value || 'anonymous'

  // Schedule work after the response is sent
  after(() => {
    logUserAction({ sessionCookie })
  })

  return <p>Your session: {sessionCookie}</p>
}
```

```jsx filename="app/page.jsx" highlight={18-19,22-24} switcher
import { Suspense } from 'react'
import { after } from 'next/server'
import { cookies } from 'next/headers'
import { logUserAction } from '@/app/utils'

export default function Page() {
  return (
    <>
      <h1>Part of the static shell</h1>
      <Suspense fallback={<p>Loading...</p>}>
        <DynamicContent />
      </Suspense>
    </>
  )
}

async function DynamicContent() {
  const sessionCookie =
    (await cookies()).get('session-id')?.value || 'anonymous'

  // Schedule work after the response is sent
  after(() => {
    logUserAction({ sessionCookie })
  })

  return <p>Your session: {sessionCookie}</p>
}
```

在此示例中，`<h1>` 和 `<Suspense>` 的降级内容 (fallback) 包含在静态外壳 (static shell) 中。`DynamicContent` 在渲染期间读取 cookie 并通过闭包将其传递给 `after`。由于 `cookies()` 是在 `after` 回调 _ 外部 (_outside_)_ 被调用的（在组件的渲染期间），因此这可以正常工作。

<details id="after-serverless">

  <summary>参考：支持 serverless 平台的 `after`</summary>

在 serverless 上下文中使用 `after` 需要在响应发送后等待异步任务完成。在 Next.js 和 Vercel 中，这是通过使用一个名为 `waitUntil(promise)` 的原语来实现的，该原语会延长 serverless 调用的生命周期，直到传递给 [`waitUntil`](https://vercel.com/docs/functions/functions-api-reference#waituntil) 的所有 Promise 都已处理完毕。

如果您希望用户能够运行 `after`，您必须提供自己实现的 `waitUntil`，其行为方式类似。

当 `after` 被调用时，Next.js 将像这样访问 `waitUntil`：

```jsx
const RequestContext = globalThis[Symbol.for('@next/request-context')]
const contextValue = RequestContext?.get()
const waitUntil = contextValue?.waitUntil
```

这意味着 `globalThis[Symbol.for('@next/request-context')]` 预计包含一个像这样的对象：

```tsx
type NextRequestContext = {
  get(): NextRequestContextValue | undefined
}

type NextRequestContextValue = {
  waitUntil?: (promise: Promise<any>) => void
}
```

下面是一个实现示例。

```tsx
import { AsyncLocalStorage } from 'node:async_hooks'

const RequestContextStorage = new AsyncLocalStorage<NextRequestContextValue>()

// Define and inject the accessor that next.js will use
const RequestContext: NextRequestContext = {
  get() {
    return RequestContextStorage.getStore()
  },
}
globalThis[Symbol.for('@next/request-context')] = RequestContext

const handler = (req, res) => {
  const contextValue = { waitUntil: YOUR_WAITUNTIL }
  // Provide the value
  return RequestContextStorage.run(contextValue, () => nextJsHandler(req, res))
}
```
