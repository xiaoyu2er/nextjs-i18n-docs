`cacheLife` 函数用于设置函数或组件的缓存生命周期。它应与 [`use cache`](/docs/app/api-reference/directives/use-cache) 指令一起使用，并位于函数或组件的作用域内。

## 使用方法

### 基本设置

要使用 `cacheLife`，首先在你的 `next.config.js` 文件中启用 [`cacheComponents` 标志](/docs/app/api-reference/config/next-config-js/cacheComponents)：

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  cacheComponents: true,
}

export default nextConfig
```

```js filename="next.config.js" switcher
const nextConfig = {
  cacheComponents: true,
}

export default nextConfig
```

`cacheLife` 需要 `use cache` 指令，该指令必须放在文件级别或异步函数/组件的顶部。

> **了解**：
>
> - 如果使用，`cacheLife` 应放置在正在缓存其输出的函数内，即使 `use cache` 指令位于文件级别
> - 每次函数调用只应执行一次 `cacheLife`。你可以在不同的控制流分支中调用 `cacheLife`，但确保每次运行只执行一次。参见 [条件缓存生命周期](#conditional-cache-lifetimes) 示例

### 使用预设配置文件

Next.js 提供涵盖常见缓存需求的预设缓存配置文件。每个配置文件平衡三个因素：

- 用户看到缓存内容而无需检查更新的时间（客户端）
- 服务器生成新内容的频率
- 旧内容完全过期的时间

根据内容更改频率选择配置文件：

- **`seconds`** - 实时数据（股票价格、实时比分）
- **`minutes`** - 频繁更新（社交动态、新闻）
- **`hours`** - 每日多次更新（产品库存、天气）
- **`days`** - 每日更新（博客文章、文章）
- **`weeks`** - 每周更新（播客、新闻通讯）
- **`max`** - 极少更改（法律页面、归档内容）

导入 `cacheLife` 并传入配置文件名称：

```tsx filename="app/blog/page.tsx" highlight={1,5}
'use cache'
import { cacheLife } from 'next/cache'

export default async function BlogPage() {
  cacheLife('days') // Blog content updated daily

  const posts = await getBlogPosts()
  return <div>{/* render posts */}</div>
}
```

配置文件名称告诉 Next.js 如何缓存整个函数的输出。如果你不调用 `cacheLife`，则使用 `default` 配置文件。详见 [预设缓存配置文件](#preset-cache-profiles) 以获取时间详情。

## 参考

### 缓存配置文件属性

缓存配置文件通过三个时间属性控制缓存行为：

- **[`stale`](#stale)**：客户端可以在不检查服务器的情况下使用缓存数据的时间
- **[`revalidate`](#revalidate)**：经过此时间后，下一个请求将触发后台刷新
- **[`expire`](#expire)**：经过此时间且无请求后，下一个请求将等待新内容

#### `stale`

**客户端**：客户端可以在不检查服务器的情况下使用缓存数据的时间。

在此期间，客户端路由器立即显示缓存内容而无需任何网络请求。在此期间过后，路由器必须在下次导航或请求时检查服务器。这提供了来自客户端缓存的即时页面加载，但数据可能已过期。

- 如果省略，默认为 `default` 配置文件的 `stale` 值（5 分钟，参见 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes)）

```tsx
cacheLife({ stale: 300 }) // 5 minutes
```

#### `revalidate`

服务器在后台重新生成缓存内容的频率。

- 当请求到达该时间段后，服务器：
  1. 立即提供缓存版本（如果可用）
  2. 在后台重新生成内容
  3. 使用新内容更新缓存
- 类似于 [增量静态生成 (ISR)](/docs/app/guides/incremental-static-regeneration)
- 如果省略，默认为 `default` 配置文件的 `revalidate` 值（15 分钟）

```tsx
cacheLife({ revalidate: 900 }) // 15 minutes
```

#### `expire`

服务器必须重新生成缓存内容的最大时间。

- 在没有流量的这段时间过后，服务器在下一个请求时同步重新生成内容
- 当你同时设置 `revalidate` 和 `expire` 时，`expire` 必须长于 `revalidate`。Next.js 会验证这一点，并为无效配置抛出一个错误。
- 如果省略，默认为 `default` 配置文件的 `expire` 值（永不过期）

```tsx
cacheLife({ expire: 3600 }) // 1 hour
```

### 预设缓存配置文件

如果你不指定配置文件，Next.js 将使用 `default` 配置文件。我们建议明确设置配置文件以使缓存行为清晰。

| **配置文件** | **用例**                           | `stale`    | `revalidate` | `expire` |
| ----------- | -------------------------------------- | ---------- | ------------ | -------- |
| `default`   | 标准内容                       | 5 分钟  | 15 分钟   | 永不过期    |
| `seconds`   | 实时数据                         | 30 秒 | 1 秒     | 1 分钟 |
| `minutes`   | 频繁更新的内容             | 5 分钟  | 1 分钟     | 1 小时   |
| `hours`     | 每天多次更新的内容 | 5 分钟  | 1 小时       | 1 天    |
| `days`      | 每日更新的内容                  | 5 分钟  | 1 天        | 1 周   |
| `weeks`     | 每周更新的内容                 | 5 分钟  | 1 周       | 30 天  |
| `max`       | 稳定且极少更改的内容     | 5 分钟  | 30 天      | 1 年   |

### 自定义缓存配置

在 `next.config.ts` 文件中定义可重用的缓存配置文件：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  cacheComponents: true,
  cacheLife: {
    biweekly: {
      stale: 60 * 60 * 24 * 14, // 14 days
      revalidate: 60 * 60 * 24, // 1 day
      expire: 60 * 60 * 24 * 14, // 14 days
    },
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
const nextConfig = {
  cacheComponents: true,
  cacheLife: {
    biweekly: {
      stale: 60 * 60 * 24 * 14, // 14 days
      revalidate: 60 * 60 * 24, // 1 day
      expire: 60 * 60 * 24 * 14, // 14 days
    },
  },
}

module.exports = nextConfig
```

上面的示例将缓存 14 天，每天检查更新，并在 14 天后使缓存过期。然后，您可以在整个应用程序中通过其名称引用此配置：

> **了解**：自定义配置文件中省略的任何属性都继承自 `default` 配置文件。这也适用于直接传递给 `cacheLife()` 的内置配置文件对象。

```tsx filename="app/page.tsx" highlight={5}
'use cache'
import { cacheLife } from 'next/cache'

export default async function Page() {
  cacheLife('biweekly')
  return <div>Page</div>
}
```

### 覆盖默认缓存配置

虽然默认缓存配置提供了一种有用的方式来思考任何可缓存输出的任何部分可以有多新鲜或多陈旧，但您可能更喜欢不同的命名配置，以更好地与您的应用程序缓存策略保持一致。

您可以通过创建与默认配置同名的新配置来覆盖默认的命名缓存配置。

下面的示例展示了如何覆盖默认的 `"days"` 缓存配置文件：

```ts filename="next.config.ts"
const nextConfig = {
  cacheComponents: true,
  cacheLife: {
    // Override the 'days' profile
    days: {
      stale: 3600, // 1 hour
      revalidate: 900, // 15 minutes
      expire: 86400, // 1 day
    },
  },
}

export default nextConfig
```

### 内置缓存配置文件

对于一次性情况，直接将配置文件对象传递给 `cacheLife`：

```tsx filename="app/page.tsx"
'use cache'
import { cacheLife } from 'next/cache'

export default async function Page() {
  cacheLife({
    stale: 3600,
    revalidate: 900,
    expire: 86400,
  })

  return <div>Page</div>
}
```

内置配置文件仅应用于特定的函数或组件。对于可重用的配置，请在 `next.config.ts` 中定义自定义配置文件。

使用 `cacheLife({})` 且对象为空时，应用 `default` 配置文件的值。

### 客户端缓存行为

`stale` 属性控制 [客户端缓存](/docs/app/glossary#client-cache)，而不是 `Cache-Control` 标头：

- 服务器通过 `x-nextjs-stale-time` 响应标头发送过期时间
- 客户端路由器使用此值确定何时重新验证
- **强制执行最少 30 秒的过期时间**，以确保预取链接保持可用

这 30 秒的最小值可防止预取数据在用户能够点击链接前过期。它仅适用于基于时间的过期。

当你从服务器操作 ([`revalidateTag`](/docs/app/api-reference/functions/revalidateTag), [`revalidatePath`](/docs/app/api-reference/functions/revalidatePath), [`updateTag`](/docs/app/api-reference/functions/updateTag), 或 [`refresh`](/docs/app/api-reference/functions/refresh)) 调用重新验证函数时，整个客户端缓存会立即清除，绕过过期时间。

> **了解**：`cacheLife` 中的 `stale` 属性与 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 不同。虽然 `staleTimes` 是影响所有路由的全局设置，但 `cacheLife` 允许按函数或按路由配置。更新 `staleTimes.static` 也会更新 `default` 缓存配置文件的 `stale` 值。

### 预渲染行为

生命周期很短的缓存——`revalidate` 为零或 `expire` 在 5 分钟以下——将被自动排除在预渲染之外，变为 "动态空洞"。这包括 `seconds` 配置文件。

此行为允许你在同一页面中混合静态和动态内容。静态部分会被预渲染，而短期缓存会在数据在请求时获取而不是构建时获取的位置创建边界。使用 `<Suspense>` 边界包裹动态缓存，以便在内容加载时提供回退。

## 示例

### 使用预设配置文件

配置缓存最简单的方法是使用预设配置文件。选择一个匹配你的内容更新模式的配置文件：

```tsx filename="app/blog/[slug]/page.tsx"
import { cacheLife } from 'next/cache'

export default async function BlogPost() {
  'use cache'
  cacheLife('days') // Blog posts updated daily

  const post = await fetchBlogPost()
  return <article>{post.content}</article>
}
```

```tsx filename="app/products/[id]/page.tsx"
import { cacheLife } from 'next/cache'

export default async function ProductPage() {
  'use cache'
  cacheLife('hours') // Product data updated multiple times per day

  const product = await fetchProduct()
  return <div>{product.name}</div>
}
```

### 特定需求的自定义配置文件

当预设选项不满足你的需求时，定义自定义配置文件：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  cacheComponents: true,
  cacheLife: {
    editorial: {
      stale: 600, // 10 minutes
      revalidate: 3600, // 1 hour
      expire: 86400, // 1 day
    },
    marketing: {
      stale: 300, // 5 minutes
      revalidate: 1800, // 30 minutes
      expire: 43200, // 12 hours
    },
  },
}

export default nextConfig
```

然后在整个应用程序中使用这些配置文件：

```tsx filename="app/editorial/page.tsx"
import { cacheLife } from 'next/cache'

export default async function EditorialPage() {
  'use cache'
  cacheLife('editorial')
  // ...
}
```

### 独特情况的内置配置文件

当特定函数需要一次性缓存行为时，使用内置配置文件：

```tsx filename="app/api/limited-offer/route.ts"
import { cacheLife } from 'next/cache'
import { getDb } from '@lib/db'

async function getLimitedOffer() {
  'use cache'

  cacheLife({
    stale: 60, // 1 minute
    revalidate: 300, // 5 minutes
    expire: 3600, // 1 hour
  })

  const offer = await getDb().offer.findFirst({
    where: { type: 'limited' },
    orderBy: { created_at: 'desc' },
  })

  return offer
}

export async function GET() {
  const offer = await getLimitedOffer()

  return Response.json(offer)
}
```

### 缓存单个函数

对实用函数应用缓存以实现细粒度控制：

```tsx filename="lib/api.ts"
import { cacheLife } from 'next/cache'

export async function getSettings() {
  'use cache'
  cacheLife('max') // Settings rarely change

  return await fetchSettings()
}
```

```tsx filename="lib/stats.ts"
import { cacheLife } from 'next/cache'

export async function getRealtimeStats() {
  'use cache'
  cacheLife('seconds') // Stats update constantly

  return await fetchStats()
}
```

### 嵌套缓存行为

当你嵌套 `use cache` 指令时（一个缓存函数或组件使用另一个缓存函数或组件），外层缓存的行为取决于它是否有显式的 `cacheLife`。

#### 具有显式外层 cacheLife

外层缓存使用其自己的生命周期，无论内层缓存的生命周期如何。当时外层缓存命中，它返回包含所有嵌套数据的完整输出。显式的 `cacheLife` 总是优先于内层生命周期，无论它更长还是更短。

```tsx filename="app/dashboard/page.tsx"
import { cacheLife } from 'next/cache'
import { Widget } from './widget'

export default async function Dashboard() {
  'use cache'
  cacheLife('hours') // Outer scope sets its own lifetime

  return (
    <div>
      <h1>Dashboard</h1>
      <Widget /> {/* Inner scope has 'minutes' lifetime */}
    </div>
  )
}
```

#### 没有显式外层 cacheLife

如果你在外层缓存中没有调用 `cacheLife`，则使用 `default` 配置文件（15 分钟重新验证）。生命周期较短的内层缓存可以减少外层缓存的 `default` 生命周期。生命周期较长的内层缓存无法将其扩展超出默认值。

```tsx filename="app/dashboard/page.tsx"
import { Widget } from './widget'

export default async function Dashboard() {
  'use cache'
  // No cacheLife call - uses default (15 min)
  // If Widget has 5 min → Dashboard becomes 5 min
  // If Widget has 1 hour → Dashboard stays 15 min

  return (
    <div>
      <h1>Dashboard</h1>
      <Widget />
    </div>
  )
}
```

**建议指定显式的 `cacheLife`**。使用显式生命周期值，你可以检查缓存函数或组件并立即了解其行为，而无需追踪嵌套缓存。如果没有显式生命周期值，行为将依赖于内层缓存生命周期，使得推理变得更加困难。

#### 嵌套短期缓存

如 [预渲染行为](#prerendering-behavior) 中所描述的那样，短期缓存（`revalidate` 为零或 `expire` 在 5 分钟以下）变为排除在预渲染之外的动态空洞。

当短期缓存嵌套在不带显式 `cacheLife` 的另一个 `use cache` 中时，外层缓存的生命周期会通过传播而静默变短。为了防止这种意外配置错误，Next.js 会在预渲染期间抛出一个错误。

请注意，嵌套的缓存可能不明显——它可能在导入的模块中，甚至在第三方依赖项中：

```tsx filename="components/short-lived-widget.tsx" highlight={5}
import { cacheLife } from 'next/cache'

export async function ShortLivedWidget() {
  'use cache'
  cacheLife('seconds')
  const data = await fetchRealtimeData()
  return <div>{data}</div>
}
```

从另一个不带显式 `cacheLife` 的 `use cache` 中使用此组件将在预渲染期间报错：

```tsx filename="app/page.tsx"
import { ShortLivedWidget } from '@/components/short-lived-widget'

export default async function Page() {
  'use cache'
  // Error: no explicit cacheLife on outer cache
  return (
    <div>
      <h1>Dashboard</h1>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </div>
  )
}
```

要修复此错误，请为外层 `use cache` 添加显式的 `cacheLife()`：

**如果你希望外层缓存保持静态（预渲染）**，请设置较长的缓存生命周期：

```tsx filename="app/page.tsx" highlight={6}
import { cacheLife } from 'next/cache'
import { ShortLivedWidget } from '@/components/short-lived-widget'

export default async function Page() {
  'use cache'
  cacheLife('default') // Explicit cacheLife prevents the error
  return (
    <div>
      <h1>Dashboard</h1>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </div>
  )
}
```

**如果你希望外层缓存也是短期的**，请显式设置短期缓存生命周期以确认这是有意为之。将该组件包裹在 `<Suspense>` 边界中以在内容加载时提供回退：

```tsx filename="app/page.tsx" highlight={7,17-19}
import { Suspense } from 'react'
import { cacheLife } from 'next/cache'
import { ShortLivedWidget } from '@/components/short-lived-widget'

async function Content() {
  'use cache: remote'
  cacheLife('seconds') // Explicit cacheLife confirms this is intentionally short-lived
  return (
    <>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </>
  )
}

export default function Page() {
  return (
    <div>
      <h1>Dashboard</h1>
      <Suspense fallback={<p>Loading...</p>}>
        <Content />
      </Suspense>
    </div>
  )
}
```

> **注意：** 此示例使用 `"use cache: remote"`，因为 serverless 部署中的运行时缓存不会像默认的内存缓存那样跨请求持久化。对于自托管环境，`"use cache"` 可能已足够。详见 [运行时缓存注意事项](/docs/app/api-reference/directives/use-cache#runtime-caching-considerations) 了解更多详情。

### 条件缓存生命周期

你可以在不同的代码路径中条件调用 `cacheLife`，根据你的应用程序逻辑设置不同的缓存持续时间：

```tsx filename="lib/posts.ts" highlight={14,19}
import { cacheLife, cacheTag } from 'next/cache'

async function getPostContent(slug: string) {
  'use cache'

  const post = await fetchPost(slug)

  // Tag the cache entry for targeted revalidation
  cacheTag(`post-${slug}`)

  if (!post) {
    // Content may not be published yet or could be in draft
    // Cache briefly to reduce database load
    cacheLife('minutes')
    return null
  }

  // Published content can be cached longer
  cacheLife('days')

  // Return only the necessary data to keep cache size minimal
  return post.data
}
```

此模式在不同结果需要不同缓存持续时间时非常有用，例如，当某件物品缺失但该物品稍后可能可用时。

#### 使用数据中的动态缓存生命周期

如果你希望在运行时计算缓存生命周期，例如通过从获取的数据中读取它，请使用 [内置缓存配置文件](#inline-cache-profiles) 对象：

```tsx filename="lib/posts.ts" highlight={15,16,17,18}
import { cacheLife, cacheTag } from 'next/cache'

async function getPostContent(slug: string) {
  'use cache'

  const post = await fetchPost(slug)
  cacheTag(`post-${slug}`)

  if (!post) {
    cacheLife('minutes')
    return null
  }

  // Use cache timing from CMS data directly as an object
  cacheLife({
    // Ensure post.revalidateSeconds is a number in seconds
    // stale and expire inherit from 'default' profile
    revalidate: post.revalidateSeconds ?? 3600,
  })

  return post.data
}
```
