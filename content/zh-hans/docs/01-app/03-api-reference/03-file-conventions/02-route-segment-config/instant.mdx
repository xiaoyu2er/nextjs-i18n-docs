The `unstable_instant` 路由段配置将路由纳入验证，以支持即时的客户端-side 导航。Next.js 在开发和构建时检查缓存结构是否在进入路由的每个可能入口点都生成即时的 [静态外壳](/docs/app/glossary#static-shell)。

> **Good to know**:
>
> - The `unstable_instant` 导出仅在启用了 [`cacheComponents`](/docs/app/api-reference/config/next-config-js/cacheComponents) 时有效。
> - `unstable_instant` 不能在 Client Components 中使用。它将抛出错误。

```tsx filename="layout.tsx | page.tsx" switcher
export const unstable_instant = {
  prefetch: 'static',
}

export default function Page() {
  return <div>...</div>
}
```

```jsx filename="layout.js | page.js" switcher
export const unstable_instant = {
  prefetch: 'static',
}

export default function Page() {
  return <div>...</div>
}
```

## 参考

### `prefetch`

Controls 验证和预取模式。

```tsx filename="page.tsx"
export const unstable_instant = {
  prefetch: 'static',
}
```

- **`'static'`**: Enables 验证。Prefetching 行为保持不变（默认为静态）。Components 读取 cookies 或 headers 被视为动态组件，必须位于 Suspense 之后。

### Disabling instant

Set `false` 以豁免段验证：

```tsx filename="app/dashboard/layout.tsx"
export const unstable_instant = false
```

## How validation works

`unstable_instant` 触发验证，在路由中的每个共享布局边界。Validation 在开发期间（在页面加载和 HMR updates 期间）和在构建时运行。Errors 出现在 dev error overlay 或失败构建。

Each error 标识将阻止导航的组件。The fix 通常是用 `use cache` 缓存数据，或将其包裹在 `<Suspense>` 边界中。

## Inspecting loading states

Enable 使用实验性标志的 DevTools toggle:

```js filename="next.config.js"
module.exports = {
  experimental: {
    instantNavigationDevToolsToggle: true,
  },
}
```

Open the Next.js DevTools 并选择 **Instant Navs**。有两个选项可用：

- **Page load**: click **Reload** 刷新页面并冻结在它生成的初始静态 UI 上，在此路由中，在动态数据流进之前。
- **Client navigation**: 启用后，点击应用中的任意链接将显示该页面的预取 UI，而不是完整结果。

Use both to check that your loading states look right on first visit and on navigation.

## Testing instant navigation

The `@next/playwright` package exports an `instant()` helper that holds back dynamic content while the callback runs against the static shell. See the [guide](/docs/app/guides/instant-navigation#prevent-regressions-with-e2e-tests) for a full example.

## Known issue: shared cookie across projects

The DevTools use a `next-instant-navigation-testing` cookie to freeze the UI at the static shell. Because cookies are scoped to the domain and not the port, running multiple projects on the same domain (typically `localhost`) means the cookie is shared across them and can cause unexpected behavior. Clear the cookie or close the Instant Navs panel when switching between projects to avoid issues.

> **Good to know**: This will be fixed as part of stabilizing the feature.

## TypeScript

```tsx
type RuntimeSample = {
  cookies?: Array<{ name: string; value: string }>
  headers?: Array<[string, string]>
  params?: Record<string, string | string[]>
  searchParams?: Record<string, string | string[]>
}

type InstantConfig =
  | false
  | {
      prefetch: 'static'
      from?: string[]
      unstable_disableValidation?: boolean
    }
  | {
      prefetch: 'runtime'
      samples: RuntimeSample[]
      from?: string[]
      unstable_disableValidation?: boolean
    }

export const unstable_instant: InstantConfig = {
  prefetch: 'static',
}
```
