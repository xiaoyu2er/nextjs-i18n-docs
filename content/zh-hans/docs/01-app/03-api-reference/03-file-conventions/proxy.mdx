> **注意**：`middleware` 文件约定已弃用并更名为 `proxy`。更多详情请参阅[迁移到代理](#migration-to-proxy)。

`proxy.js|ts` 文件用于编写[代理](/docs/app/getting-started/proxy)，并在请求完成前在服务器上运行代码。然后，根据传入的请求，你可以通过重写、重定向、修改请求或响应头或直接响应来修改响应。

代理在路由渲染前执行。它对于实现自定义服务器端逻辑（如身份验证、日志记录或处理重定向）特别有用。

> **须知**：
>
> 代理旨在与你的渲染代码分开调用，在优化的情况下部署到你的 CDN 以快速处理重定向/重写，你不应尝试依赖共享模块或全局变量。
>
> 要将信息从代理传递到你的应用程序，请使用 [headers](#setting-headers)、[cookies](#using-cookies)、[rewrites](/docs/app/api-reference/functions/next-response#rewrite)、[redirects](/docs/app/api-reference/functions/next-response#redirect) 或 URL。

在项目根目录（或 `src` 目录内，如果适用）创建一个 `proxy.ts`（或 `.js`）文件，使其与 `pages` 或 `app` 位于同一层级。

如果你已经自定义了 [`pageExtensions`](/docs/app/api-reference/config/next-config-js/pageExtensions)，例如设置为 `.page.ts` 或 `.page.js`，请相应地将你的文件命名为 `proxy.page.ts` 或 `proxy.page.js`。

```tsx filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

```js filename="proxy.js" switcher
import { NextResponse } from 'next/server'

// This function can be marked `async` if using `await` inside
export function proxy(request) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

## 导出

### 代理 (Proxy) 函数

该文件必须导出一个单一函数，可以是默认导出，也可以命名为 `proxy`。请注意，不支持从同一文件导出多个代理。

```js filename="proxy.js"
// Example of default export
export default function proxy(request) {
  // Proxy logic
}
```

### 配置对象（可选）

可选择性地，可以与代理函数一起导出一个配置对象。此对象包括用于指定代理适用路径的 [matcher](#matcher)。

### 匹配器

`matcher` 选项允许你针对特定路径运行代理。你可以通过以下几种方式指定这些路径：

- 对于单个路径：直接使用字符串定义路径，如 `'/about'`。
- 对于多个路径：使用数组列出多个路径，例如 `matcher: ['/about', '/contact']`，这将使代理应用于 `/about` 和 `/contact`。

```js filename="proxy.js"
export const config = {
  matcher: ['/about/:path*', '/dashboard/:path*'],
}
```

此外，`matcher` 选项支持使用正则表达式进行复杂的路径规范。例如，你可以使用正则表达式匹配器排除某些路径：

```js filename="proxy.js"
export const config = {
  matcher: [
    // Exclude API routes, static files, image optimizations, and .png files
    '/((?!api|_next/static|_next/image|.*\\.png$).*)',
  ],
}
```

这使得精确控制要包含或排除哪些路径成为可能。

`matcher` 选项接受一个具有以下键的对象数组：

- `source`：用于匹配请求路径的路径或模式。它可以是用于直接路径匹配的字符串，也可以是用于更复杂匹配的模式。
- `locale`（可选）：一个布尔值，设置为 `false` 时，在路径匹配中忽略基于区域设置的路由。
- `has`（可选）：指定基于特定请求元素（如请求头、查询参数或 Cookie）存在的条件。
- `missing`（可选）：专注于某些请求元素（如缺少请求头或 Cookie）不存在的条件。

```js filename="proxy.js"
export const config = {
  matcher: [
    {
      source: '/api/:path*',
      locale: false,
      has: [
        { type: 'header', key: 'Authorization', value: 'Bearer Token' },
        { type: 'query', key: 'userId', value: '123' },
      ],
      missing: [{ type: 'cookie', key: 'session', value: 'active' }],
    },
  ],
}
```

`source` 路径模式：

1. **必须**以 `/` 开头
2. 可以包含命名参数：`/about/:path` 匹配 `/about/a` 和 `/about/b`，但不匹配 `/about/a/c`
3. 可以对命名参数（以 `:` 开头）使用修饰符：`/about/:path*` 匹配 `/about/a/b/c`，因为 `*` 表示 _零个或多个_。`?` 表示 _零个或一个_，`+` 表示 _一个或多个_
4. 可以使用括号括起来的正则表达式：`/about/(.*)` 与 `/about/:path*` 相同
5. 锚定在路径的开头：`/about` 匹配 `/about` 和 `/about/team`，但不匹配 `/blog/about`

在 [path-to-regexp](https://github.com/pillarjs/path-to-regexp#path-to-regexp-1) 文档中阅读更多细节。

> **须知**：
>
> - `matcher` 值必须是常量，以便可以在构建时进行静态分析。动态值（如变量）将被忽略。
> - 为了向后兼容，Next.js 始终将 `/public` 视为 `/public/index`。因此，匹配器 `/public/:path` 将会匹配。

## 参数

### 'request'

定义代理时，默认导出函数接受一个参数 `request`。此参数是 `NextRequest` 的实例，代表传入的 HTTP 请求。

```tsx filename="proxy.ts" switcher
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Proxy logic goes here
}
```

```js filename="proxy.js" switcher
export function proxy(request) {
  // Proxy logic goes here
}
```

如果你喜欢简写，可以使用 `NextProxy` 类型。它会自动推断 `request` (`NextRequest`) 和 `event` (`NextFetchEvent`) 的参数类型：

```tsx filename="proxy.ts"
import type { NextProxy } from 'next/server'

export const proxy: NextProxy = (request, event) => {
  event.waitUntil(Promise.resolve())
  return Response.json({ pathname: request.nextUrl.pathname })
}
```

> **须知**：
>
> - `NextRequest` 是表示 Next.js 代理中传入 HTTP 请求的类型，而 [`NextResponse`](#nextresponse) 是用于操作和发送回 HTTP 响应的类。

## NextResponse

`NextResponse` API 允许你：

- `redirect` 传入的请求到不同的 URL
- 通过显示给定的 URL 来 `rewrite` 响应
- 为 API 路由、`getServerSideProps` 和 `rewrite` 目标设置请求头
- 设置响应 Cookie
- 设置响应头

<AppOnly>

要从代理生成响应，你可以：

1. `rewrite` 到一个生成响应的路由（[页面](/docs/app/api-reference/file-conventions/page) 或 [路由处理程序](/docs/app/api-reference/file-conventions/route)）
2. 直接返回一个 `NextResponse`。参阅[生成响应](#producing-a-response)

> **须知**：对于重定向，你也可以使用 `Response.redirect` 代替 `NextResponse.redirect`。

</AppOnly>

<PagesOnly>

要从代理生成响应，你可以：

1. `rewrite` 到一个生成响应的路由（[页面](/docs/pages/building-your-application/routing/pages-and-layouts) 或 [Edge API 路由](/docs/pages/building-your-application/routing/api-routes)）
2. 直接返回一个 `NextResponse`。参阅[生成响应](#producing-a-response)

</PagesOnly>

## 执行顺序

代理将为**项目中的每个路由**被调用。因此，使用 [匹配器](#matcher) 来精确针对或排除特定路由至关重要。以下是执行顺序：

1. 来自 `next.config.js` 的 `headers`
2. 来自 `next.config.js` 的 `redirects`
3. 代理（`rewrites`、`redirects` 等）
4. 来自 `next.config.js` 的 `beforeFiles`（`rewrites`）
5. 文件系统路由（`public/`、`_next/static/`、`pages/`、`app/` 等）
6. 来自 `next.config.js` 的 `afterFiles`（`rewrites`）
7. 动态路由（`/blog/[slug]`）
8. 来自 `next.config.js` 的 `fallback`（`rewrites`）

> **须知：**[服务器函数](/docs/app/api-reference/directives/use-server) 在此链中不是独立的路由。它们作为 POST 请求处理到使用它们的路由，因此，排除某个路径的代理匹配器也将跳过该路径上的服务器函数调用。
>
> 匹配器的更改或将服务器函数移动到不同路由的重构可能会静默地移除代理覆盖。始终在每个服务器函数内部验证身份验证和授权，而不是仅依赖代理。请参阅[数据安全指南](/docs/app/guides/data-security#authentication-and-authorization)了解推荐的模式。

## 运行时

代理默认使用 Node.js 运行时。[`runtime`](/docs/app/api-reference/file-conventions/route-segment-config/runtime) 配置选项在代理文件中不可用。在代理中设置 `runtime` 配置选项将引发错误。

## 高级代理 (Proxy) 选项

在 Next.js 的 `v13.1` 中，为代理引入了两个额外的标志，`skipProxyUrlNormalize`（以前称为 `skipMiddlewareUrlNormalize`）和 `skipTrailingSlashRedirect`，用于处理高级用例。

`skipTrailingSlashRedirect` 禁用 Next.js 用于添加或删除尾部斜杠的重定向。这允许在代理内部进行自定义处理，为某些路径保留尾部斜杠，但其他路径不保留，这可以使增量迁移更容易。

```js filename="next.config.js"
module.exports = {
  skipTrailingSlashRedirect: true,
}
```

```js filename="proxy.js"
const legacyPrefixes = ['/docs', '/blog']

export default async function proxy(req) {
  const { pathname } = req.nextUrl

  if (legacyPrefixes.some((prefix) => pathname.startsWith(prefix))) {
    return NextResponse.next()
  }

  // apply trailing slash handling
  if (
    !pathname.endsWith('/') &&
    !pathname.match(/((?!\.well-known(?:\/.*)?)(?:[^/]+\/)*[^/]+\.\w+)/)
  ) {
    return NextResponse.redirect(
      new URL(`${req.nextUrl.pathname}/`, req.nextUrl)
    )
  }
}
```

`skipProxyUrlNormalize` 允许禁用 Next.js 中的 URL 规范化，使处理直接访问和客户端转换相同。在某些高级情况下，此选项通过使用原始 URL 提供完全控制。

```js filename="next.config.js"
module.exports = {
  skipProxyUrlNormalize: true,
}
```

```js filename="proxy.js"
export default async function proxy(req) {
  const { pathname } = req.nextUrl

  // GET /_next/data/build-id/hello.json

  console.log(pathname)
  // with the flag this now /_next/data/build-id/hello.json
  // without the flag this would be normalized to /hello
}
```

### Using Cookies

Cookie 是常规的请求头。在 `Request` 上，它们存储在 `Cookie` 头中。在 `Response` 上，它们在 `Set-Cookie` 头中。Next.js 通过 `NextRequest` 和 `NextResponse` 上的 `cookies` 扩展提供了便捷的方式来访问和操作这些 Cookie。

1. 对于传入的请求，`cookies` 提供以下方法：`get`、`getAll`、`set` 和 `delete`。你可以使用 `has` 检查 Cookie 是否存在，或使用 `clear` 删除所有 Cookie。
2. 对于传出的响应，`cookies` 提供以下方法：`get`、`getAll`、`set` 和 `delete`。

```ts filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Assume a "Cookie:nextjs=fast" header to be present on the incoming request
  // Getting cookies from the request using the `RequestCookies` API
  let cookie = request.cookies.get('nextjs')
  console.log(cookie) // => { name: 'nextjs', value: 'fast', Path: '/' }
  const allCookies = request.cookies.getAll()
  console.log(allCookies) // => [{ name: 'nextjs', value: 'fast' }]

  request.cookies.has('nextjs') // => true
  request.cookies.delete('nextjs')
  request.cookies.has('nextjs') // => false

  // Setting cookies on the response using the `ResponseCookies` API
  const response = NextResponse.next()
  response.cookies.set('vercel', 'fast')
  response.cookies.set({
    name: 'vercel',
    value: 'fast',
    path: '/',
  })
  cookie = response.cookies.get('vercel')
  console.log(cookie) // => { name: 'vercel', value: 'fast', Path: '/' }
  // The outgoing response will have a `Set-Cookie:vercel=fast;path=/` header.

  return response
}
```


### Setting Headers

你可以使用 `NextResponse` API 设置请求和响应头（自 Next.js v13.0.0 起支持设置*请求*头）。

```ts filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  // Clone the request headers and set a new header `x-hello-from-proxy1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-proxy1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-proxy2`
  response.headers.set('x-hello-from-proxy2', 'hello')
  return response
}
```

```js filename="proxy.js" switcher
import { NextResponse } from 'next/server'

export function proxy(request) {
  // Clone the request headers and set a new header `x-hello-from-proxy1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-proxy1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-proxy2`
  response.headers.set('x-hello-from-proxy2', 'hello')
  return response
}
```

请注意，该代码片段使用了：

- `NextResponse.next({ request: { headers: requestHeaders } })` 使 `requestHeaders` 在上游可用
- **不是** `NextResponse.next({ headers: requestHeaders })`，这会使 `requestHeaders` 对客户端可用

了解更多关于 [代理中的 NextResponse 头部](/docs/app/api-reference/functions/next-response#next)。

> **注意**：避免设置过大的请求头，根据你的后端 Web 服务器配置，这可能导致 [431 请求头字段过大](https://developer.mozilla.org/docs/Web/HTTP/Status/431) 错误。

#### RSC 请求与重写

在 RSC 请求期间，Next.js 会从代理中的 `request` 实例中剥离内部的 Flight 头。例如，`rsc`、`next-router-state-tree` 和 `next-router-prefetch` 等头不会通过 `request.headers` 暴露。这是为了防止意外地以不同方式处理 RSC 请求和 HTML 请求，因为两者需要保持一致。

当你使用 `NextResponse.rewrite()` 时，Next.js 会自动将所需的 RSC 重写头向上传播。

如果你使用 `fetch()` 而不是 `NextResponse.rewrite()` 实现自定义重写逻辑，除非手动转发，否则可能会遇到缺少 RSC 头的问题。

对于自定义 `fetch` 重写设置，你还可以在 `next.config.js` 中启用 `skipProxyUrlNormalize`，以便你的重写逻辑可以从提供的请求对象接收必要的 URL 形状和 RSC 头：

```js filename="next.config.js"
module.exports = {
  skipProxyUrlNormalize: true,
}
```

### CORS

你可以在代理中设置 CORS 头以允许跨域请求，包括[简单](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#simple_requests)和[预检](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#preflighted_requests)请求。

```tsx filename="proxy.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

const allowedOrigins = ['https://acme.com', 'https://my-app.org']

const corsOptions = {
  'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type, Authorization',
}

export function proxy(request: NextRequest) {
  // Check the origin from the request
  const origin = request.headers.get('origin') ?? ''
  const isAllowedOrigin = allowedOrigins.includes(origin)

  // Handle preflighted requests
  const isPreflight = request.method === 'OPTIONS'

  if (isPreflight) {
    const preflightHeaders = {
      ...(isAllowedOrigin && { 'Access-Control-Allow-Origin': origin }),
      ...corsOptions,
    }
    return NextResponse.json({}, { headers: preflightHeaders })
  }

  // Handle simple requests
  const response = NextResponse.next()

  if (isAllowedOrigin) {
    response.headers.set('Access-Control-Allow-Origin', origin)
  }

  Object.entries(corsOptions).forEach(([key, value]) => {
    response.headers.set(key, value)
  })

  return response
}

export const config = {
  matcher: '/api/:path*',
}
```

### 生成响应

你可以通过返回 `Response` 或 `NextResponse` 实例直接从代理进行响应。（此功能自 [Next.js v13.1.0](https://nextjs.org/blog/next-13-1#nextjs-advanced-proxy) 起可用）

```ts filename="proxy.ts" switcher
import type { NextRequest } from 'next/server'
import { isAuthenticated } from '@lib/auth'

// Limit the proxy to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
}

export function proxy(request: NextRequest) {
  // Call our authentication function to check the request
  if (!isAuthenticated(request)) {
    // Respond with JSON indicating an error message
    return Response.json(
      { success: false, message: 'authentication failed' },
      { status: 401 }
    )
  }
}
```

```js filename="proxy.js" switcher
import { isAuthenticated } from '@lib/auth'

// Limit the proxy to paths starting with `/api/`

  matcher: '/api/:function*',

    )
  }
}
```

### 负向匹配

`matcher` 配置允许完整的正则表达式，因此支持负向前瞻或字符匹配等。这里有一个使用负向前瞻来匹配除特定路径外的所有路径的示例：

```js filename="proxy.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
  ],
}
```

你还可以使用 `missing` 或 `has` 数组，或两者的组合，为某些请求绕过代理：

```js filename="proxy.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      missing: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [{ type: 'header', key: 'x-present' }],
      missing: [{ type: 'header', key: 'x-missing', value: 'prefetch' }],
    },
  ],
}
```

> **须知**：
>
> 即使在负向匹配模式中排除了 `_next/data`，代理仍然会为 `_next/data` 路由调用。这是有意的行为，以防止意外的安全问题，例如你可能保护了一个页面，却忘了保护相应的数据路由。

```js filename="proxy.js"
export const config = {
  matcher:
    '/((?!api|_next/data|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
}

// Proxy will still run for /_next/data/* routes despite being excluded
```

### 'waitUntil' 和 'NextFetchEvent'

`NextFetchEvent` 对象扩展了原生的 [`FetchEvent`](https://developer.mozilla.org/docs/Web/API/FetchEvent) 对象，并包含 [`waitUntil()`](https://developer.mozilla.org/docs/Web/API/ExtendableEvent/waitUntil) 方法。

`waitUntil()` 方法接受一个 Promise 作为参数，并扩展代理的生命周期，直到该 Promise 解决。这对于在后台执行工作很有用。

```ts filename="proxy.ts"
import { NextResponse } from 'next/server'
import type { NextFetchEvent, NextRequest } from 'next/server'

export function proxy(req: NextRequest, event: NextFetchEvent) {
  event.waitUntil(
    fetch('https://my-analytics-platform.com', {
      method: 'POST',
      body: JSON.stringify({ pathname: req.nextUrl.pathname }),
    })
  )

  return NextResponse.next()
}
```

### 单元测试（实验性）

从 Next.js 15.1 开始，`next/experimental/testing/server` 包包含一些工具，可帮助对代理文件进行单元测试。对代理进行单元测试有助于确保它只在期望的路径上运行，并且自定义路由逻辑在代码进入生产环境之前按预期工作。

`unstable_doesProxyMatch` 函数可用于断言代理是否会为提供的 URL、请求头和 Cookie 运行。

```js
import { unstable_doesProxyMatch } from 'next/experimental/testing/server'

expect(
  unstable_doesProxyMatch({
    config,
    nextConfig,
    url: '/test',
  })
).toEqual(false)
```

整个代理函数也可以被测试。

```js
import { isRewrite, getRewrittenUrl } from 'next/experimental/testing/server'

const request = new NextRequest('https://nextjs.org/docs')
const response = await proxy(request)
expect(isRewrite(response)).toEqual(true)
expect(getRewrittenUrl(response)).toEqual('https://other-domain.com/docs')
// getRedirectUrl could also be used if the response were a redirect
```

## 迁移至代理 (Proxy)

### 为何做出更改

`middleware` 更名的原因是，“middleware”这个术语经常与 Express.js 中间件混淆，导致对其用途的误解。此外，中间件功能强大，可能会鼓励使用；然而，此功能建议作为最后的手段使用。

Next.js 正在向前发展，提供更好、更易用的 API，以便开发人员无需中间件即可实现目标。这就是 `middleware` 更名的原因。

### 为何使用 "Proxy"

名称“代理”明确了中间件的功能。术语“代理”意味着它在网络边界前，这是中间件的行为。此外，中间件默认在 [Edge Runtime](/docs/app/api-reference/edge) 上运行，它可以更靠近客户端运行，与应用的区域分离。这些行为与“代理”一词更加契合，并提供了该功能更清晰的用途。

### 如何迁移

我们建议用户避免依赖中间件，除非没有其他选择。我们的目标是提供更好、更易用的 API，以便他们无需中间件即可实现目标。

术语“中间件”常常使用户与 Express.js 中间件混淆，这可能导致误用。为了明确我们的方向，我们将文件约定重命名为“代理”。这突显出我们正在远离中间件，分解其过度加载的功能，并使代理的目的清晰。

Next.js 提供了一个代码迁移工具（codemod），用于将 `middleware.ts` 迁移到 `proxy.ts`。你可以运行以下命令进行迁移：

```bash
npx @next/codemod@canary middleware-to-proxy .
```

该代码迁移工具将把文件和函数名从 `middleware` 重命名为 `proxy`。

```diff
// middleware.ts -> proxy.ts

- export function middleware() {
+ export function proxy() {
```
