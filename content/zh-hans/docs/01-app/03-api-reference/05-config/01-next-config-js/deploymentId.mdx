“deploymentId”选项允许您为部署设置标识符。此标识符用于 [版本倾斜 (version skew)](/docs/app/guides/self-hosting#version-skew) 保护及滚动部署期间的缓存清除。

```js filename="next.config.js"
module.exports = {
  deploymentId: 'my-deployment-id',
}
```

您还可以使用 `NEXT_DEPLOYMENT_ID` 环境变量设置部署 ID：

```bash
NEXT_DEPLOYMENT_ID=my-deployment-id next build
```

> **小知识：** 如果两者都设置了，`next.config.js` 中的 `deploymentId` 值优先于 `NEXT_DEPLOYMENT_ID` 环境变量。

## 工作原理

配置了 `deploymentId` 后，Next.js 会：

1. 将 `?dpl=<deploymentId>` 追加到静态资源 URL（JavaScript、CSS、图片）
2. 在客户端导航请求中添加 `x-deployment-id` 头部
3. 在导航响应中添加 `x-nextjs-deployment-id` 头部
4. 在 `<html>` 元素上注入 `data-dpl-id` 属性

当客户端检测到其部署 ID 与服务器端（通过响应头）不匹配时，它将触发硬导航（完整页面加载），而不是客户端导航。这确保用户始终从一致的部署版本获取资源 <AppOnly>和服务端函数</AppOnly>。

> **小知识：** Next.js 不会读取传入请求中的 `?dpl=` 查询参数。该查询参数用于缓存清除（确保浏览器和 CDN 获取最新资源），而不是用于路由。如果您需要版本感知路由，请咨询您的托管提供商或 CDN 文档以实现基于部署的路由。

## 使用场景

### 滚动部署

在滚动部署期间，部分服务器实例可能正在运行新版本，而其他实例仍在运行旧版本。如果没有部署 ID，用户可能会收到新旧混合的资源，从而导致错误。

为每次部署设置一致的 `deploymentId` 可确保：

<AppOnly>

- 客户端总是从匹配的部署版本请求资源
- 不匹配时触发完整重新加载以获取正确的资源
- 服务端函数在部署边界间正常工作

</AppOnly>

<PagesOnly>

- 客户端总是从匹配的部署版本请求资源
- 不匹配时触发完整重新加载以获取正确的资源

</PagesOnly>

### 多服务器环境

当在负载均衡器后运行多个 Next.js 应用实例时，同一部署的所有实例应使用相同的 `deploymentId`。

```js filename="next.config.js"
module.exports = {
  deploymentId: process.env.DEPLOYMENT_VERSION || process.env.GIT_SHA,
}
```
