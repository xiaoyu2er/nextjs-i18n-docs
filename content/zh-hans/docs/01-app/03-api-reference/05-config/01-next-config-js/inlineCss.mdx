## 使用方法

实验性支持在 `<head>` 中内联 CSS。启用此标志后，所有通常生成 `<link>` 标签的位置都将改为生成 `<style>` 标签。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    inlineCss: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    inlineCss: true,
  },
}

module.exports = nextConfig
```

## 权衡

- **启用** 如果您使用原子 CSS（例如 Tailwind）并希望优化新访问者的初始加载性能
- **跳过** 如果回访用户很常见，并且您希望他们受益于缓存的样式表

### 何时内联 CSS 有益

通常情况下，浏览器必须先下载 HTML，解析它，发现 CSS <link> 标签，然后请求样式表，之后才能渲染。内联 [消除了这个请求级联](https://web.dev/learn/performance/optimize-resource-loading#inline_critical_css)，这样样式会随着 HTML 一起到达，浏览器可以立即渲染。

这种优势在以下情况下最明显：

- **首次访问者**: 由于 CSS 文件是渲染阻塞的，内联消除了首次访问者面临的初始下载延迟。拥有缓存样式表的回访用户不会体验到这个好处。

- **性能指标**: 通过移除 CSS 文件的其他网络连接请求，内联可以显著改善 [首屏内容绘制 (FCP)](https://web.dev/articles/fcp) 和 [最大内容绘制 (LCP)](https://web.dev/articles/lcp)。

- **缓慢连接**: 对于高延迟网络的用户，每个额外的请求都会增加延迟。内联减少了往返次数，这在连接缓慢时尤为重要。

- **原子 CSS (Tailwind)**: Utility-first 框架仅生成您使用的类，保持 CSS 很小。页面的样式不会与页面复杂度成比例增长——无论构建多少 UI，它们通常都很紧凑。这使得内联非常实用，因为您可以获得性能优势而不会显著膨胀 HTML。

### 何时外部 CSS 更佳

内联样式无法与 HTML 分开缓存。每次页面加载都会重新下载相同的 CSS。

这种权衡在以下情况下最明显：

- **回访用户**: 重复访问您网站的用户将从缓存的外部样式表中受益。使用内联方式时，他们每次访问都会重新下载样式。

- **大型 CSS 包**: 外部样式表独立缓存，并在现代基础设施上高效加载。内联的 CSS 随每次 HTML 响应到达，增加了 [首字节时间 (TTFB)](https://web.dev/articles/ttfb) 并防止浏览器独立缓存样式。这种权衡适用于小型 CSS（如 Tailwind 的原子框架），但对于更大的包（如 Bootstrap 或 Material UI 等组件库）会增加开销。

- **许多共享样式的页面**: 在一个页面上缓存的外部样式表可加快导航到其他页面的速度。内联样式不提供跨页面缓存优势。

> **要知道**:
>
> 此功能目前处于实验阶段，并有一些已知限制：
>
> - CSS 内联是全局应用的，无法按页面进行配置
> - 样式在初始页面加载期间会重复——一次在 `<style>` 标签中用于 SSR，一次在 RSC 负载中
> - 当导航到预渲染页面时，样式将使用 `<link>` 标签而不是内联 CSS 以避免重复
> - 此功能在开发模式下不可用，仅在生产构建中有效
