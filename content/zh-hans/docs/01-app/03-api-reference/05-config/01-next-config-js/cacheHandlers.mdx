`cacheHandlers` 配置允许你为 [`'use cache'`](/docs/app/api-reference/directives/use-cache) 和 [`'use cache: remote'`](/docs/app/api-reference/directives/use-cache-remote) 定义自定义缓存存储实现。这使你能够将缓存的组件和函数存储在外部服务中，或自定义缓存行为。[`'use cache: private'`](/docs/app/api-reference/directives/use-cache-private) 不可配置。

## 何时使用自定义缓存处理器

**大多数应用程序不需要自定义缓存处理器。** 默认内存缓存通常在典型用例中表现良好。

自定义缓存处理器用于高级场景，你需要在多个实例之间共享缓存，或更改缓存存储位置。例如，你可以为外部存储（如键值存储）配置自定义 `remote` 处理器，然后在代码中使用 `'use cache'` 进行内存缓存，使用 `'use cache: remote'` 用于外部存储，从而在同一应用程序中使用不同的缓存策略。

**跨实例共享缓存**

默认内存缓存隔离于每个 Next.js 进程。如果你在运行多个服务器或容器，每个实例将拥有自己的缓存，不与其他实例共享，并且在重启后会丢失。

自定义处理器允许你集成共享存储系统（如 Redis、Memcached 或 DynamoDB），所有这些 Next.js 实例都可以访问。

**更改存储类型**

你可能希望使用不同于默认内存缓存的方式存储缓存。你可以实现自定义处理器，将缓存存储在磁盘、数据库或外部缓存服务中。原因包括：重启后的持久性、减少内存使用，或与现有基础设施集成。

## 使用方法

要配置自定义缓存处理器：

1. 在单独的文件中定义你的缓存处理器，实现细节参见 [示例](#examples)。
2. 在 Next 配置文件中引用文件路径

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  cacheHandlers: {
    default: require.resolve('./cache-handlers/default-handler.js'),
    remote: require.resolve('./cache-handlers/remote-handler.js'),
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
module.exports = {
  cacheHandlers: {
    default: require.resolve('./cache-handlers/default-handler.js'),
    remote: require.resolve('./cache-handlers/remote-handler.js'),
  },
}
```

### 处理器类型

- **`default`**：由 `'use cache'` 指令使用
- **`remote`**：由 `'use cache: remote'` 指令使用

如果不配置 `cacheHandlers`，Next.js 会为 `default` 和 `remote` 都使用内存 LRU（最近最少使用）缓存。你可以查看 [默认实现](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/default.ts) 作为参考。

你还可以定义其他命名处理器（例如 `sessions`、`analytics`），并通过 `'use cache: <name>'` 引用它们。

请注意，`'use cache: private'` 不使用缓存处理器且无法自定义。

## API 参考

缓存处理器必须实现 [`CacheHandler`](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/types.ts) 接口，包含以下方法：

### `get()`

检索给定缓存键的缓存条目。

```ts
get(cacheKey: string, softTags: string[]): Promise<CacheEntry | undefined>
```

| 参数  | 类型       | 描述                                                  |
| ---------- | ---------- | ------------------------------------------------------------ |
| `cacheKey` | `string`   | 缓存条目的唯一键。                          |
| `softTags` | `string[]` | 用于检查过期状态的标签（用于某些缓存策略）。 |

如果找到则返回 `CacheEntry` 对象，如果未找到或已过期则返回 `undefined`。

你的 `get` 方法应从存储中检索缓存条目，根据 `revalidate` 时间检查是否已过期，并为缺失或已过期的条目返回 `undefined`。

```js
const cacheHandler = {
  async get(cacheKey, softTags) {
    const entry = cache.get(cacheKey)
    if (!entry) return undefined

    // Check if expired
    const now = Date.now()
    if (now > entry.timestamp + entry.revalidate * 1000) {
      return undefined
    }

    return entry
  },
}
```

### `set()`

存储给定缓存键的缓存条目。

```ts
set(cacheKey: string, pendingEntry: Promise<CacheEntry>): Promise<void>
```

| 参数      | 类型                  | 描述                                 |
| -------------- | --------------------- | ------------------------------------------- |
| `cacheKey`     | `string`              | 存储条目的唯一键。    |
| `pendingEntry` | `Promise<CacheEntry>` | 解析为缓存条目的承诺。 |

调用时条目可能仍处于挂起状态（即，其值流可能仍在写入中）。你的处理器应在处理条目之前等待该承诺。

返回 `Promise<void>`。

你的 `set` 方法必须在存储之前等待 `pendingEntry` 承诺，因为当此方法被调用时，缓存条目可能仍在生成。一旦解析，将条目存储到缓存系统中。

```js
const cacheHandler = {
  async set(cacheKey, pendingEntry) {
    // Wait for the entry to be ready
    const entry = await pendingEntry

    // Store in your cache system
    cache.set(cacheKey, entry)
  },
}
```

### `refreshTags()`

在新请求启动前定期调用，以便与外部标签服务同步。

```ts
refreshTags(): Promise<void>
```

如果你正在跨多个实例或服务协调缓存失效，这很有用。对于内存缓存，这可能是空操作 (no-op)。

返回 `Promise<void>`。

对于内存缓存，这可能是空操作 (no-op)。对于分布式缓存，在处理请求前，使用此方法从外部服务或数据库同步标签状态。

```js
const cacheHandler = {
  async refreshTags() {
    // For in-memory cache, no action needed
    // For distributed cache, sync tag state from external service
  },
}
```

### `getExpiration()`

获取一组标签的最大重验证时间戳。

```ts
getExpiration(tags: string[]): Promise<number>
```

| 参数 | 类型       | 描述                            |
| --------- | ---------- | -------------------------------------- |
| `tags`    | `string[]` | 用于检查过期的标签数组。 |

返回：

- 如果没有任何标签曾经历过重验证，则为 `0`
- 表示最近一次重验证的时间戳（毫秒）
- `Infinity`，表示应通过 `get` 方法检查标签

如果你不跟踪标签重验证时间戳，返回 `0`。否则，在所有提供的标签中找到最近的重新验证时间戳。如果你更喜欢在 `get` 方法中处理标签检查，请返回 `Infinity`。

```js
const cacheHandler = {
  async getExpiration(tags) {
    // Return 0 if not tracking tag revalidation
    return 0

    // Or return the most recent revalidation timestamp
    // return Math.max(...tags.map(tag => tagTimestamps.get(tag) || 0));
  },
}
```

### `updateTags()`

当标签发生重验证或过期时调用。

```ts
updateTags(tags: string[], durations?: { expire?: number }): Promise<void>
```

| 参数   | 类型                  | 描述                              |
| ----------- | --------------------- | ---------------------------------------- |
| `tags`      | `string[]`            | 要更新的标签数组。                 |
| `durations` | `{ expire?: number }` | 可选的过期时长（秒）。 |

你的处理器应更新其内部状态以将这些标签标记为无效。

返回 `Promise<void>`。

当标签发生重验证时，你的处理器应使所有拥有这些标签的缓存条目失效。遍历你的缓存并移除标签与提供的列表匹配的条目。

```js
const cacheHandler = {
  async updateTags(tags, durations) {
    // Invalidate all cache entries with matching tags
    for (const [key, entry] of cache.entries()) {
      if (entry.tags.some((tag) => tags.includes(tag))) {
        cache.delete(key)
      }
    }
  },
}
```

## CacheEntry 类型

[`CacheEntry`](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/types.ts) 对象具有以下结构：

```ts
interface CacheEntry {
  value: ReadableStream<Uint8Array>
  tags: string[]
  stale: number
  timestamp: number
  expire: number
  revalidate: number
}
```

| 属性     | 类型                         | 描述                                                  |
| ------------ | ---------------------------- | ------------------------------------------------------------ |
| `value`      | `ReadableStream<Uint8Array>` | 存储的数据流。                                 |
| `tags`       | `string[]`                   | 缓存标签（不包括软标签）。                            |
| `stale`      | `number`                     | 客户端过期时长（秒）。               |
| `timestamp`  | `number`                     | 条目创建时间（毫秒时间戳）。      |
| `expire`     | `number`                     | 允许使用条目的时长（秒）。       |
| `revalidate` | `number`                     | 条目应重新验证的时长（秒）。 |

> **值得了解**：
>
> - `value` 是一个 [`ReadableStream`](https://developer.mozilla.org/docs/Web/API/ReadableStream)。如果你需要读取和存储数据流，请使用 [`.tee()`](https://developer.mozilla.org/docs/Web/API/ReadableStream/tee)。
> - 如果流在部分数据时出错，你的处理器必须决定是否保留部分缓存还是丢弃它。

## 示例

### 基础内存缓存处理器

下面是一个使用 `Map` 存储的极简实现。此示例演示了核心概念，但对于包含 LRU 淘汰、错误处理和标签管理的生产就绪实现，请参见 [默认缓存处理器](https://github.com/vercel/next.js/blob/canary/packages/next/src/server/lib/cache-handlers/default.ts)。

```js filename="cache-handlers/memory-handler.js"
const cache = new Map()
const pendingSets = new Map()

module.exports = {
  async get(cacheKey, softTags) {
    // Wait for any pending set operation to complete
    const pendingPromise = pendingSets.get(cacheKey)
    if (pendingPromise) {
      await pendingPromise
    }

    const entry = cache.get(cacheKey)
    if (!entry) {
      return undefined
    }

    // Check if entry has expired
    const now = Date.now()
    if (now > entry.timestamp + entry.revalidate * 1000) {
      return undefined
    }

    return entry
  },

  async set(cacheKey, pendingEntry) {
    // Create a promise to track this set operation
    let resolvePending
    const pendingPromise = new Promise((resolve) => {
      resolvePending = resolve
    })
    pendingSets.set(cacheKey, pendingPromise)

    try {
      // Wait for the entry to be ready
      const entry = await pendingEntry

      // Store the entry in the cache
      cache.set(cacheKey, entry)
    } finally {
      resolvePending()
      pendingSets.delete(cacheKey)
    }
  },

  async refreshTags() {
    // No-op for in-memory cache
  },

  async getExpiration(tags) {
    // Return 0 to indicate no tags have been revalidated
    return 0
  },

  async updateTags(tags, durations) {
    // Implement tag-based invalidation
    for (const [key, entry] of cache.entries()) {
      if (entry.tags.some((tag) => tags.includes(tag))) {
        cache.delete(key)
      }
    }
  },
}
```

### 外部存储模式

对于持久化存储（如 Redis 或数据库），你需要序列化缓存条目。这里有一个简单的 Redis 示例：

```js filename="cache-handlers/redis-handler.js"
const { createClient } = require('redis')

const client = createClient({ url: process.env.REDIS_URL })
client.connect()

module.exports = {
  async get(cacheKey, softTags) {
    // Retrieve from Redis
    const stored = await client.get(cacheKey)
    if (!stored) return undefined

    // Deserialize the entry
    const data = JSON.parse(stored)

    // Reconstruct the ReadableStream from stored data
    return {
      value: new ReadableStream({
        start(controller) {
          controller.enqueue(Buffer.from(data.value, 'base64'))
          controller.close()
        },
      }),
      tags: data.tags,
      stale: data.stale,
      timestamp: data.timestamp,
      expire: data.expire,
      revalidate: data.revalidate,
    }
  },

  async set(cacheKey, pendingEntry) {
    const entry = await pendingEntry

    // Read the stream to get the data
    const reader = entry.value.getReader()
    const chunks = []

    try {
      while (true) {
        const { done, value } = await reader.read()
        if (done) break
        chunks.push(value)
      }
    } finally {
      reader.releaseLock()
    }

    // Combine chunks and serialize for Redis storage
    const data = Buffer.concat(chunks.map((chunk) => Buffer.from(chunk)))

    await client.set(
      cacheKey,
      JSON.stringify({
        value: data.toString('base64'),
        tags: entry.tags,
        stale: entry.stale,
        timestamp: entry.timestamp,
        expire: entry.expire,
        revalidate: entry.revalidate,
      }),
      { EX: entry.expire } // Use Redis TTL for automatic expiration
    )
  },

  async refreshTags() {
    // No-op for basic Redis implementation
    // Could sync with external tag service if needed
  },

  async getExpiration(tags) {
    // Return 0 to indicate no tags have been revalidated
    // Could query Redis for tag expiration timestamps if tracking them
    return 0
  },

  async updateTags(tags, durations) {
    // Implement tag-based invalidation if needed
    // Could iterate over keys with matching tags and delete them
  },
}
```
