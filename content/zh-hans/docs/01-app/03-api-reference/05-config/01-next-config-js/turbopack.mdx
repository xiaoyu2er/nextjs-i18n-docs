`turbopack` 选项允许您自定义 [Turbopack](/docs/app/api-reference/turbopack) 以转换不同文件并更改模块的解析方式。

> **小贴士**：`turbopack` 选项在 Next.js 13.0.0 至 15.2.x 版本中曾命名为 `experimental.turbo`。`experimental.turbo` 选项将在 Next.js 16 中移除。
>
> 如果您使用的是旧版本的 Next.js，请运行 `npx @next/codemod@latest next-experimental-turbo-to-turbopack .` 来自动迁移您的配置。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  turbopack: {
    // ...
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  turbopack: {
    // ...
  },
}

module.exports = nextConfig
```

> **Good to know**:
>
> - 用于 Next.js 的 Turbopack 对内置功能不需要加载器或加载器配置。Turbopack 内置支持 CSS 和编译现代 JavaScript，因此如果您使用 `@babel/preset-env`，则不需要 `css-loader`、`postcss-loader` 或 `babel-loader`。

## 参考

### 选项

适用于 `turbopack` 配置的选项如下：

| 选项              | 描述                                                                                                                              |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `root`              | 设置应用根目录。应为绝对路径。                                                                         |
| `rules`             | 列出在 Turbopack 运行时支持适用的 Webpack loaders。                                                                  |
| `resolveAlias`      | 将别名导入映射到要加载的模块。                                                                                   |
| `resolveExtensions` | 导入文件时解析的扩展名列表。                                                                                      |
| `debugIds`          | 启用 JavaScript bundles 和 source maps 中的 [调试 ID (Debug IDs)](https://github.com/tc39/ecma426/blob/main/proposals/debug-id.md)。 |

### Supported loaders

以下加载器已测试可与 Turbopack 的 webpack 加载器实现配合使用，但许多其他 webpack 加载器即使未在此处列出也应该可以工作：

- [`babel-loader`](https://www.npmjs.com/package/babel-loader) [_(若找到 Babel 配置文件，则自动配置)_](/docs/app/api-reference/turbopack#language-features)
- [`@svgr/webpack`](https://www.npmjs.com/package/@svgr/webpack)
- [`svg-inline-loader`](https://www.npmjs.com/package/svg-inline-loader)
- [`yaml-loader`](https://www.npmjs.com/package/yaml-loader)
- [`string-replace-loader`](https://www.npmjs.com/package/string-replace-loader)
- [`raw-loader`](https://www.npmjs.com/package/raw-loader)
- [`sass-loader`](https://www.npmjs.com/package/sass-loader) [_(自动配置)_](/docs/app/api-reference/turbopack#css-and-styling)
- [`graphql-tag/loader`](https://www.npmjs.com/package/graphql-tag)

#### 缺少的 Webpack loader 功能

Turbopack 使用 [`loader-runner`](https://github.com/webpack/loader-runner) 库来执行 Webpack loaders，它提供了大多数标准 loader API。但是，某些功能不支持：

**模块加载：**

- [`importModule`](https://webpack.js.org/api/loaders/#thisimportmodule) - 不支持
- [`loadModule`](https://webpack.js.org/api/loaders/#thisloadmodule) - 不支持

**文件系统与输出：**

- [`fs`](https://webpack.js.org/api/loaders/#thisfs) - 部分支持：目前仅实现了 `fs.readFile`。
- [`emitFile`](https://webpack.js.org/api/loaders/#thisemitfile) - 不支持

**上下文属性：**

- [`version`](https://webpack.js.org/api/loaders/#thisversion) - 不支持
- [`mode`](https://webpack.js.org/api/loaders/#thismode) - 不支持
- [`target`](https://webpack.js.org/api/loaders/#thistarget) - 不支持

**工具：**

- [`utils`](https://webpack.js.org/api/loaders/#thisutils) - 不支持
- [`resolve`](https://webpack.js.org/api/loaders/#thisresolve) - 不支持 (请使用 [`getResolve`](https://webpack.js.org/api/loaders/#thisgetresolve) 代替)

如果您有一个 loader 严重依赖这些功能中的某一个，请提交一个问题。

## 示例

### Root directory

Turbopack 使用根目录来解析模块。项目根目录之外的文件不会被解析。

文件不在项目根目录外被解析的原因是：为了提高缓存验证效率，减少文件系统监控开销，并减少所需的解析步骤数量。

Next.js 会自动检测您项目的根目录。它通过查找以下文件之一来实现：

- `pnpm-lock.yaml`
- `package-lock.json`
- `yarn.lock`
- `bun.lock`
- `bun.lockb`

如果您有不同的项目结构，例如不使用工作区，可以手动设置 `root` 选项：

```js filename="next.config.js"
const path = require('path')
module.exports = {
  turbopack: {
    root: path.join(__dirname, '..'),
  },
}
```

要解析项目根目录外链接依赖中的文件（通过 `npm link`, `yarn link`, `pnpm link` 等），您必须将 `turbopack.root` 配置为项目及其所链接依赖的父目录。

虽然这会扩大文件系统监控的范围，但通常仅在开发时积极处理链接包时才需要。

### Configuring webpack loaders

如果您需要超出内置范围的加载器支持，许多 webpack 加载器已经可以与 Turbopack 一起工作。目前存在一些限制：

- 仅实现了 webpack 加载器 API 的核心子集。目前，这足以覆盖一些流行的加载器，我们将在未来扩展我们的 API 支持。
- 仅支持返回 JavaScript 代码的加载器。目前不支持转换样式表或图像等文件的加载器。
- 传递给 webpack 加载器的选项必须是纯 JavaScript 原语、对象和数组。例如，不能将 `require()` 插件模块作为选项值传递。

要配置 loaders，请在 `next.config.js` 中添加已安装的 loader 名称及任何选项，将文件扩展名映射到 loader 列表。规则按顺序评估。

下面是一个使用 [`@svgr/webpack`](https://www.npmjs.com/package/@svgr/webpack) 加载器的示例，它允许导入 `.svg` 文件并将其渲染为 React 组件。

```js filename="next.config.js"
module.exports = {
  turbopack: {
    rules: {
      '*.svg': {
        loaders: ['@svgr/webpack'],
        as: '*.js',
      },
    },
  },
}
```

> **小贴士**：`rules` 对象中使用的通配符基于文件名进行匹配，除非通配符包含 `/` 字符，这将导致其基于完整的相对项目文件路径进行匹配。Windows 文件路径已规范化为使用 Unix 风格的 `/` 路径分隔符。
>
> Turbopack 使用修改版的 [Rust `globset` 库](https://docs.rs/globset/latest/globset/)。

对于需要配置选项的加载器，您可以使用对象格式而不是字符串：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    rules: {
      '*.svg': {
        loaders: [
          {
            loader: '@svgr/webpack',
            options: {
              icon: true,
            },
          },
        ],
        as: '*.js',
      },
    },
  },
}
```

> **小贴士**：在 Next.js 13.4.4 版本之前，`turbopack.rules` 名为 `turbo.loaders`，且仅接受 `.mdx` 等文件扩展名，而非 `*.mdx`。

### 高级 Webpack loader 条件

您可以使用高级 `condition` 语法进一步限制 loader 的运行位置：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    rules: {
      // '*' will match all file paths, but we restrict where our
      // rule runs with a condition.
      '*': {
        condition: {
          all: [
            // 'foreign' is a built-in condition.
            { not: 'foreign' },
            // 'path' can be a RegExp or a glob string. A RegExp matches
            // anywhere in the full project-relative file path.
            { path: /^img\/[0-9]{3}\// },
            {
              any: [
                { path: '*.svg' },
                // 'query' matches anywhere in the full query string,
                // which can be empty, or start with `?`.
                { query: /[?&]svgr(?=&|$)/ },
                // 'content' is always a RegExp, and can match
                // anywhere in the file.
                { content: /\<svg\W/ },
              ],
            },
          ],
        },
        loaders: ['@svgr/webpack'],
        as: '*.js',
      },
    },
  },
}
```

- 支持的布尔运算符为 `{all: [...]}`, `{any: [...]}` 和 `{not: ...}`。
- 支持的自定义运算符为 `{path: string | RegExp}`, `{content: RegExp}`, `{query: string | RegExp}`, 和 `{contentType: string | RegExp}`。如果在同一对象中指定多个运算符，则充当隐式的 `and`。
  - `path` 匹配相对项目的文件路径。字符串被视为通配符模式，而 RegExp 可用于部分匹配路径。
  - `content` 匹配文件内容中的任何位置。
  - `query` 匹配导入的查询字符串（例如 `import './file?foo'` 中的 `?foo`）。字符串必须完全匹配，而 RegExp 可用于部分匹配查询字符串。
  - `contentType` 匹配资源的 MIME 内容类型（例如来自 data URLs，如 `data:text/plain,...`）。字符串被视为通配符模式（例如 `text/*`, `image/*`），而 RegExp 可用于部分匹配内容类型。

此外，还支持一些内置条件：

- `browser`：匹配将在客户端执行的代码。服务器代码可以使用 `{not: 'browser'}` 匹配。
- `foreign`：匹配 `node_modules` 中的代码以及部分 Next.js 内部代码。通常您希望将 loaders 限制为 `{not: 'foreign'}`。这可以通过减少调用 loader 的文件数量来提高性能。
- `development`：当使用 `next dev` 时匹配。
- `production`：当使用 `next build` 时匹配。
- `node`：匹配将在默认 Node.js 运行时运行的代码。
- `edge-light`：匹配将在 [Edge runtime](/docs/app/api-reference/edge) 上运行的代码。

规则可以是对象或对象数组。数组通常用于建模不相交的条件：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    rules: {
      '*.svg': [
        {
          condition: 'browser',
          loaders: ['@svgr/webpack'],
          as: '*.js',
        },
        {
          condition: { not: 'browser' },
          loaders: [require.resolve('./custom-svg-loader.js')],
          as: '*.js',
        },
      ],
    },
  },
}
```

> **小贴士**：所有匹配规则按顺序执行。

### 模块类型

您可以直接设置模块类型而无需使用 loader。这对于更改文件处理方式很有用，类似于 Webpack 的 [`type`](https://webpack.js.org/configuration/module/#ruletype) 选项。

```js filename="next.config.js"
module.exports = {
  turbopack: {
    rules: {
      '*.svg': {
        type: 'asset',
      },
    },
  },
}
```

当使用 `type: 'asset'` 时，导入文件会返回其 URL：

```tsx filename="app/page.tsx"
import svgUrl from './icon.svg'

export default function Page() {
  return <img src={svgUrl} alt="Icon" />
}
```

`type` 选项可与 `loaders` 结合使用 - loader 先运行，然后结果根据指定的类型进行处理。

可用的模块类型：

| 类型         | 描述                                              |
| ------------ | -------------------------------------------------------- |
| `asset`      | 输出文件并返回 URL（类似 Webpack `asset/resource`） |
| `ecmascript` | 作为 JavaScript 处理                                    |
| `typescript` | 作为 TypeScript 处理                                    |
| `css`        | 作为 CSS 处理                                           |
| `css-module` | 作为 CSS 模块处理                                    |
| `wasm`       | 作为 WebAssembly 处理                                   |
| `raw`        | 返回原始内容作为字符串                            |
| `bytes`      | 以字节形式内联内容                                 |

### 使用导入特性进行内联 loader 配置

您可以使用 `with` 子句（导入特性）将 Turbopack loader 应用于单个导入。这是按导入指定，而非通过 `turbopack.rules` 全球指定。

当您想将 loader 应用于特定导入而不影响该类型的所有文件时，这非常有用。

```tsx filename="app/page.tsx"
// Apply a raw loader to import a .txt file as a JavaScript module
import rawText from '../data.txt' with { turbopackLoader: 'raw-loader', turbopackAs: '*.js' }

export default function Page() {
  return <p>{rawText}</p>
}
```

支持以下导入特性：

| 特性                | 描述                                                                                                                   |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| `turbopackLoader`        | 要应用的 loader（例如 `'raw-loader'`）。                                                                                    |
| `turbopackLoaderOptions` | loader 选项的 JSON 字符串（例如 `'{"search":"X","replace":"Y"}'`）。                                                        |
| `turbopackAs`            | 输出重命名模式（与 `turbopack.rules[].as` 相同）。例如 `'*.js'` 将 loader 输出视为 JavaScript。 |
| `turbopackModuleType`    | 设置输出的模块类型（与 `turbopack.rules[].type` 相同）。                                                        |

带有选项的 loader 通过 `turbopackLoaderOptions` 传递 JSON 编码的字符串：

```tsx filename="app/page.tsx"
import value from '../data.js' with { turbopackLoader: 'string-replace-loader', turbopackLoaderOptions: '{"search":"PLACEHOLDER","replace":"replaced value"}' }
```

> **小贴士**：带有 `turbopackLoader` 的导入特性是 Turbopack 特有的，Webpack 不支持。此功能要求导入语句中使用 `with` 关键字（而非 `assert`）。

### Resolving aliases

可以配置 Turbopack 通过别名修改模块解析，类似于 webpack 的 [`resolve.alias`](https://webpack.js.org/configuration/resolve/#resolvealias) 配置。

要配置解析别名，请在 `next.config.js` 中将导入模式映射到它们的新目标位置：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    resolveAlias: {
      underscore: 'lodash',
      mocha: { browser: 'mocha/browser-entry.js' },
    },
  },
}
```

这会将 `underscore` 包的导入别名指向 `lodash` 包。换句话说，`import underscore from 'underscore'` 将加载 `lodash` 模块而不是 `underscore`。

Turbopack 还通过此字段支持条件别名，类似于 Node.js 的 [条件导出](https://nodejs.org/docs/latest-v18.x/api/packages.html#conditional-exports)。目前仅支持 `browser` 条件。在上面的例子中，当 Turbopack 以浏览器环境为目标时，`mocha` 模块的导入将被别名指向 `mocha/browser-entry.js`。

### Resolving custom extensions

可以配置 Turbopack 解析具有自定义扩展名的模块，类似于 webpack 的 [`resolve.extensions`](https://webpack.js.org/configuration/resolve/#resolveextensions) 配置。

要配置解析扩展名，请在 `next.config.js` 中使用 `resolveExtensions` 字段：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    resolveExtensions: ['.mdx', '.tsx', '.ts', '.jsx', '.js', '.mjs', '.json'],
  },
}
```

这会使用提供的列表覆盖原始的解析扩展名。请确保包含默认扩展名。

有关如何将您的应用从 webpack 迁移到 Turbopack 的更多信息和指南，请参阅 [Turbopack 关于 webpack 兼容性的文档](https://turbo.build/pack/docs/migrating-from-webpack)。

### 调试 ID (Debug IDs)

Turbopack 可以配置为在 JavaScript bundles 和 source maps 中生成 [调试 ID (Debug IDs)](https://github.com/tc39/ecma426/blob/main/proposals/debug-id.md)。

要配置调试 ID，请在 `next.config.js` 中使用 `debugIds` 字段：

```js filename="next.config.js"
module.exports = {
  turbopack: {
    debugIds: true,
  },
}
```

该选项会自动将调试 ID 的 polyfill 添加到 JavaScript bundle 以确保兼容性。调试 ID 可在 `globalThis._debugIds` 全局变量中获取。
