# 增量翻译优化方案设计

## 背景与痛点

1. **API 调用耗时** — 当前使用 DeepSeek API 翻译 1118 个文件 × 8 种语言，成本高、速度慢
2. **目录结构频繁变化** — Next.js 官方文档经常重组目录，导致增量翻译难以追踪
3. **文件级粒度浪费** — 改 1 个字就要重新翻译整个文件

## 核心思路

**将翻译缓存从"文件级"下沉到"AST 节点级"，用 MD5 哈希做增量追踪，用 pi agent 在本地翻译。**

- 不再维护目录结构映射，每次直接下载最新 Next.js 文档
- 不再依赖 git 时间戳判断变化，用内容哈希判断
- 多版本（v13、v14、canary）天然复用相同内容的翻译

## 方案设计

### 1. 文档解析 — 基于 remark AST

使用 remark 解析器将 MDX 文件解析为 AST，按顶层节点切分。

**预处理 normalize：** 在 JSX 标签（`<AppOnly>`、`<PagesOnly>`、`<details>`、`<div>`）后强制插入空行，确保 remark 将标签和后续内容解析为独立节点。

**节点分类：**

| 策略 | 节点类型 | 占比 | 说明 |
|---|---|---|---|
| 🔤 翻译 | `paragraph`, `heading`, `list`, `blockquote` | ~55% | 自然语言，需要翻译并缓存 |
| 📦 保留 | `code`, `thematicBreak` | ~40% | 代码和分隔线，原样保留 |
| 🏷️ 标签 | `html`（纯标签） | ~4% | `<AppOnly>` 等，原样保留 |
| 🏷️ 标签+文本 | `html`（含 summary 等） | ~1% | 提取文本翻译，保留标签结构 |

**数据验证：**

- 全部 1118 个文件，34,253 个顶层 AST 节点
- 只有 7 种顶层节点类型：`paragraph`、`heading`、`code`、`list`、`html`、`blockquote`、`thematicBreak`
- 代码块完整性 100%（remark 保证不会撕裂代码块）
- 预处理后 `<AppOnly>`/`<PagesOnly>` 内容被吞的问题完全解决

### 2. MD5 哈希缓存

**哈希策略：**

- 对每个需要翻译的 AST 节点，用其原始文本内容生成 MD5
- heading 节点哈希时去掉 `#` 前缀（normalize），使 `## Title` 和 `### Title` 生成相同哈希，避免 heading 层级变化导致缓存失效

**缓存结构：**

```
translation-cache/
├── zh-hans.json    # { "md5hash": "翻译后的文本", ... }
├── ja.json
├── es.json
└── ...
```

或使用更结构化的存储（可后续优化为 SQLite）。

**缓存数据量估算：**

| 指标 | 值 |
|---|---|
| 需要翻译的节点总数 | 25,406 |
| 去重后唯一节点 | 14,899 |
| 跨版本去重率 | 41% |
| 平均节点大小 | 182 字符 |

### 3. 翻译流程 — 逐文件 + 节点级增量

#### 步骤 1：下载最新文档

从 Next.js 官方仓库下载最新的 canary/v14/v13 文档到 `content/en/`。

#### 步骤 2：解析并拼装目标文件

对每个英文源文件：

1. 预处理 normalize
2. remark 解析为 AST
3. 遍历顶层节点：
   - `code` / `thematicBreak` / 纯标签 `html` → 原样保留
   - 需要翻译的节点 → 计算 MD5，查缓存：
     - **命中** → 替换为缓存中的翻译
     - **未命中** → 保留英文原文，用标记包裹

```markdown
已翻译的中文内容...

<!-- NEEDS_TRANSLATION -->
This is untranslated English content.
<!-- /NEEDS_TRANSLATION -->

另一段已翻译的中文内容...
```

4. 检查文件是否全部已翻译 → 全部命中则直接写入，跳过翻译

#### 步骤 3：pi 翻译未命中的文件

对包含 `<!-- NEEDS_TRANSLATION -->` 标记的文件，调用 pi：

- prompt："请翻译所有 `NEEDS_TRANSLATION` 标记内的内容为 {目标语言}，其余已翻译部分原样保留，输出完整文件"
- pi 有完整的文件上下文，翻译质量有保障

#### 步骤 4：校验 + 更新缓存

pi 输出全文后：

1. 重新用 remark 解析输出
2. 逐节点与缓存比对：
   - 已缓存的节点：对比输出是否与缓存一致
     - **一致** → ✅
     - **不一致** → 用缓存内容覆盖，同时记录差异日志
   - 新翻译的节点：写入 MD5 缓存
3. 差异日志用于后续优化 prompt（哪些地方 pi 擅自修改了已翻译内容）

**差异日志格式示例：**

```json
{
  "file": "docs/01-app/02-guides/authentication.mdx",
  "lang": "zh-hans",
  "node_index": 5,
  "node_type": "paragraph",
  "md5": "a1b2c3d4...",
  "cached": "已缓存的翻译内容",
  "pi_output": "pi 输出的不同内容",
  "timestamp": "2026-03-14T10:00:00Z"
}
```

### 4. 增量翻译场景估算

| 场景 | 变更节点/语言 | pi 调用次数/语言 | 说明 |
|---|---|---|---|
| 冷启动 | 14,899 | ~1,118（逐文件） | 首次全量翻译 |
| 重度更新 (10%) | ~1,490 | ~100-200 | 大版本更新 |
| 常规更新 (5%) | ~745 | ~50-100 | 日常同步 |
| 轻度更新 (1%) | ~149 | ~10-30 | 小改动 |

### 5. 目录结构同步

**彻底简化：** 不再需要 `process-file-changes.js` 追踪 rename/delete。

**全量替换策略：** 翻译结果先写入临时目录，最后整体替换 `content/{lang}/`。

每次同步时：
1. 下载最新 Next.js 文档到 `content/en/`
2. 创建临时目录 `tmp/{lang}/`
3. 遍历 `content/en/` 所有文件，用 AST + MD5 缓存拼装到 `tmp/{lang}/`
4. 需要翻译的文件交给 pi 处理，结果也写入 `tmp/{lang}/`
5. 全部完成后，`rm -rf content/{lang}/ && mv tmp/{lang}/ content/{lang}/`

**好处：**
- 不需要检测哪些文件被删除/重命名
- 目录结构天然与最新英文文档一致
- 如果翻译中途失败，原有的 `content/{lang}/` 不受影响（原子替换）

## 技术栈

| 组件 | 技术 | 说明 |
|---|---|---|
| MDX 解析 | remark（项目已有依赖） | AST 节点切分 |
| 哈希 | Node.js crypto MD5 | 节点内容指纹 |
| 缓存存储 | JSON 文件（MVP） | 后续可迁移 SQLite |
| 翻译引擎 | pi agent（SDK 或 CLI） | 本地调用，支持多 provider |
| CI/CD | GitHub Actions | 定时同步 + 翻译 |

## 多版本复用验证

| 切分方式 | 唯一哈希 | 跨版本共享 | 共享率 |
|---|---|---|---|
| AST 节点级 | 18,572 | 6,583 | 35% |
| heading 级 | 4,425 | 1,378 | 31% |

AST 节点级的更细粒度带来了更高的跨版本复用率。

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| pi 修改已翻译内容 | 缓存不一致 | 校验 + 差异日志 + 用缓存覆盖 |
| remark 版本升级改变 AST | 缓存大面积失效 | 锁定 remark 版本；失效时等同冷启动 |
| 节点太小缺乏上下文 | 翻译质量差 | 以文件为单位给 pi，上下文充足 |
| 超大 html 节点（div+表格） | 解析复杂 | 仅 3 个，特殊处理或标记为需翻译 |
| prompt 标记被 pi 误解 | 输出格式错误 | 校验 + 重试机制 |

## 实施策略 — 本地小范围验证优先

**原则：先本地跑通，再改造 CI/CD。小范围试，避免消耗过多时间和 token。**

### Phase 1：本地 Prototype（单文件验证）

1. 选 1-2 个文件（如 `installation.mdx`），手动走通完整流程：
   - normalize → remark 解析 → MD5 生成 → 拼装半成品 → pi 翻译 → 校验 + 缓存写入
2. 验证：输出文件能否被 fumadocs 正常渲染
3. 验证：校验逻辑和差异日志是否正常工作

### Phase 2：本地批量测试（单语言 + 少量文件）

1. 选 10 个多样性文件（前面分析选出的覆盖全类型的文件集）
2. 只翻译 zh-hans 一种语言
3. 验证增量逻辑：修改 1 个文件，重新运行，确认只有变化的节点被翻译

### Phase 3：本地全量（单语言）

1. 对 `content/en/` 全部 1118 个文件运行
2. 只翻译 zh-hans
3. 验证临时目录 → 全量替换流程
4. 对比翻译质量与现有方案

### Phase 4：改造 CI/CD

1. 将本地脚本打包到 `packages/translate/`
2. 替换现有的 `translate-docs.yml` workflow
3. 支持多语言并行

## 与现有方案对比

| | 现有方案 | 新方案 |
|---|---|---|
| 变更检测 | git 时间戳 | MD5 内容哈希 |
| 翻译粒度 | 文件级 | AST 节点级 |
| 翻译引擎 | DeepSeek API | pi agent（本地） |
| 目录结构同步 | git rename/delete 追踪 | 每次从 en/ 重新生成 |
| 跨版本复用 | 无 | 35% 节点共享 |
| 增量能力 | 文件级（改一字翻全文） | 节点级（只翻变化的节点） |
