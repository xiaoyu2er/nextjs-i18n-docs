`middleware.js|ts` 檔案用於撰寫 [Middleware](/docs/app/getting-started/route-handlers-and-middleware#middleware)，並在請求完成之前在伺服器上執行程式碼。接著，根據傳入的請求，您可以透過重寫 (rewrite)、重導向 (redirect)、修改請求或回應標頭，或直接回應來修改回應。

Middleware 在路由渲染之前執行。它特別適合用於實作自訂伺服器邏輯，例如身份驗證、記錄記錄或處理重導向。

> **重要資訊**：
>
> 中間件 (Middleware) 設計為獨立於渲染程式碼調用，且在最佳化情況下會部署到您的 CDN 以快速處理重導向/重寫，您不應嘗試依賴共用模組或全域變數。
>
> 要將資訊從中間件傳遞到您的應用程式中，請使用 [標頭](#setting-headers)、[Cookie](#using-cookies)、[重寫](/docs/app/api-reference/functions/next-response#rewrite)、[重導向](/docs/app/api-reference/functions/next-response#redirect)，或 URL。

在專案根目錄或 `src` 目錄（若適用）建立一個 `middleware.ts`（或 `.js`）檔案，使其位於與 `pages` 或 `app` 相同的層級。

如果您已自訂 [`pageExtensions`](/docs/app/api-reference/config/next-config-js/pageExtensions)，例如設定為 `.page.ts` 或 `.page.js`，請將檔案命名為 `middleware.page.ts` 或 `middleware.page.js`。

```tsx filename="middleware.ts" switcher
import { NextResponse, NextRequest } from 'next/server'

// This function can be marked `async` if using `await` inside
export function middleware(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

```js filename="middleware.js" switcher
import { NextResponse } from 'next/server'

// This function can be marked `async` if using `await` inside
export function middleware(request) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

## 匯出

### Middleware 函式

檔案必須匯出單一函式，作為預設匯出或命名為 `middleware`。注意不支援來自同一個檔案的多個 Middleware。

```js filename="middleware.js"
// Example of default export
export default function middleware(request) {
  // Middleware logic
}
```

### 設定物件（選用）

可選地，可以與 Middleware 函式一起匯出一個設定物件。此物件包含 [matcher](#matcher) 以指定 Middleware 適用的路徑。

### 比對器 (Matcher)

`matcher` 選項允許您鎖定特定路徑供 Middleware 運行。您可以透過幾種方式指定這些路徑：

- 對於單一路徑：直接使用字串來定義路徑，例如 `'/about'`。
- 對於多個路徑：使用陣列列出多個路徑，例如 `matcher: ['/about', '/contact']`，這將把 Middleware 應用於 `/about` 和 `/contact`。

```js filename="middleware.js"
export const config = {
  matcher: ['/about/:path*', '/dashboard/:path*'],
}
```

此外，`matcher` 選項支援透過正則表達式的複雜路徑規範，例如 `matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)']`，從而精確控制包含或排除哪些路徑。

`matcher` 選項接受包含以下鍵的物件陣列：

- `source`: 用於比對請求路徑的路徑或模式。可以是直接比對路徑的字串，或是用於更複雜比對的模式。
- `regexp`（選用）：一個正則表達式字串，根據 `source` 微調匹配。對於包含或排除哪些路徑提供額外的控制。
- `locale`（選用）：一個布林值，設定為 `false` 時會忽略路徑匹配中的基於地區的路由。
- `has`（選用）：指定基於請求元素（例如標頭、查詢參數或 Cookie）的存在條件。
- `missing`（選用）：專注於某些請求元素（如缺失標頭或 Cookie）缺失的情況。

```js filename="middleware.js"
export const config = {
  matcher: [
    {
      source: '/api/*',
      regexp: '^/api/(.*)',
      locale: false,
      has: [
        { type: 'header', key: 'Authorization', value: 'Bearer Token' },
        { type: 'query', key: 'userId', value: '123' },
      ],
      missing: [{ type: 'cookie', key: 'session', value: 'active' }],
    },
  ],
}
```

已設定比對器：

1. 必須以 `/` 開頭
2. 可包含命名參數：`/about/:path` 匹配 `/about/a` 和 `/about/b` 但不匹配 `/about/a/c`
3. 命名參數上可有修飾符（以 `:` 開頭）：`/about/:path*` 匹配 `/about/a/b/c`，因為 `*` 是 _零或多個_。`?` 是 _零或一個_，`+` 是 _一個或多個_
4. 可使用括號內的正則表達式：`/about/(.*)` 等同於 `/about/:path*`

有關更多詳細資訊，請參閱 [path-to-regexp](https://github.com/pillarjs/path-to-regexp#path-to-regexp-1) 文件。

> **重要資訊**：
>
> - `matcher` 值必須是常數，以便在編譯時進行靜態分析。動態值（如變數）將被忽略。
> - 為了向後相容性，Next.js 總是將 `/public` 視為 `/public/index`。因此，`/public/:path` 的比對器會匹配。

## Params

### `request`

定義 Middleware 時，預設匯出函式接受單一參數 `request`。此參數是 `NextRequest` 的實例，代表傳入的 HTTP 請求。

```tsx filename="middleware.ts" switcher
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  // Middleware logic goes here
}
```

```js filename="middleware.js" switcher
export function middleware(request) {
  // Middleware logic goes here
}
```

> **重要資訊**：
>
> - `NextRequest` 是代表 Next.js Middleware 中傳入 HTTP 請求的類型，而 [`NextResponse`](#nextresponse) 是用來操作和發送 HTTP 回應的類別。

## NextResponse

`NextResponse` API 允許您：

- `redirect` 將傳入請求重導向到不同的 URL
- `rewrite` 重寫回應，顯示給定的 URL
- 設定 API Routes、`getServerSideProps` 和 `rewrite` 目的地的請求標頭
- 設定回應 Cookie
- 設定回應標頭

<AppOnly>

要從 Middleware 產生回應，您可以：

1. `rewrite` 重寫到產生回應的路由 ([Page](/docs/app/api-reference/file-conventions/page) 或 [Route Handler](/docs/app/api-reference/file-conventions/route))
2. 直接返回 `NextResponse`。請參閱 [產生回應](#producing-a-response)

> **重要資訊**：對於重導向，您也可以使用 `Response.redirect` 代替 `NextResponse.redirect`。

</AppOnly>

<PagesOnly>

要從 Middleware 產生回應，您可以：

1. `rewrite` 重寫到產生回應的路由 ([Page](/docs/pages/building-your-application/routing/pages-and-layouts) 或 [Edge API Route](/docs/pages/building-your-application/routing/api-routes))
2. 直接返回 `NextResponse`。請參閱 [產生回應](#producing-a-response)

</PagesOnly>

## 執行順序

Middleware 將被調用於您的專案中的**每一個路由**。因此，務必使用 [比對器](#matcher) 來精確鎖定或排除特定路由。以下是執行順序：

1. `headers` 從 `next.config.js`
2. `redirects` 從 `next.config.js`
3. Middleware (`rewrites`, `redirects` 等)
4. `beforeFiles` (`rewrites`) 從 `next.config.js`
5. 檔案系統路由 (`public/`, `_next/static/`, `pages/`, `app/` 等)
6. `afterFiles` (`rewrites`) 從 `next.config.js`
7. 動態路由 (`/blog/[slug]`)
8. `fallback` (`rewrites`) 從 `next.config.js`

## 執行階段

Middleware 預設使用 Edge 執行階段。從 v15.5 開始，我們支援使用 Node.js 執行階段。要在啟用，在您的 middleware 檔案中，在 `config` 物件中將 `runtime` 設定為 `nodejs`：

```js highlight={2} filename="middleware.js" switcher
export const config = {
  runtime: 'nodejs',
}
```

```ts highlight={2} filename="middleware.ts" switcher
export const config = {
  runtime: 'nodejs',
}
```

## 進階 Middleware 標誌

在 Next.js v13.1 中，為 Middleware 引入了兩個額外的標誌：`skipMiddlewareUrlNormalize` 和 `skipTrailingSlashRedirect`，用於處理進階使用情境。

`skipTrailingSlashRedirect` 停用 Next.js 添加或移除結尾斜線的重導向。這允許在 Middleware 內自訂處理，以保留某些路徑的結尾斜線而不保留其他路徑的結尾斜線，這可以讓增量遷移更輕鬆。

```js filename="next.config.js"
module.exports = {
  skipTrailingSlashRedirect: true,
}
```

```js filename="middleware.js"
const legacyPrefixes = ['/docs', '/blog']

export default async function middleware(req) {
  const { pathname } = req.nextUrl

  if (legacyPrefixes.some((prefix) => pathname.startsWith(prefix))) {
    return NextResponse.next()
  }

  // apply trailing slash handling
  if (
    !pathname.endsWith('/') &&
    !pathname.match(/((?!\.well-known(?:\/.*)?)(?:[^/]+\/)*[^/]+\.\w+)/)
  ) {
    return NextResponse.redirect(
      new URL(`${req.nextUrl.pathname}/`, req.nextUrl)
    )
  }
}
```

`skipMiddlewareUrlNormalize` 允許停用 Next.js 中的 URL 正規化，使直接訪問和客戶端轉換的處理方式相同。在某些進階情況下，此選項提供完全控制，使用原始 URL。

```js filename="next.config.js"
module.exports = {
  skipMiddlewareUrlNormalize: true,
}
```

```js filename="middleware.js"
export default async function middleware(req) {
  const { pathname } = req.nextUrl

  // GET /_next/data/build-id/hello.json

  console.log(pathname)
  // with the flag this now /_next/data/build-id/hello.json
  // without the flag this would be normalized to /hello
}
```

### 使用 Cookie

Cookie 是常規標頭。在 `Request` 上，它們儲存在 `Cookie` 標頭中。在 `Response` 上，它們位於 `Set-Cookie` 標頭中。Next.js 提供了通過 `NextRequest` 和 `NextResponse` 上的 `cookies` 擴充功能來訪問和操作這些 Cookie 的便捷方式。

1. 對於傳入請求，`cookies` 帶有以下方法：`get`, `getAll`, `set` 和 `delete` Cookie。您可以使用 `has` 檢查 Cookie 是否存在，或與 `clear` 移除所有 Cookie。
2. 對於傳出回應，`cookies` 具有 `get`, `getAll`, `set` 和 `delete` 方法。

```ts filename="middleware.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  // Assume a "Cookie:nextjs=fast" header to be present on the incoming request
  // Getting cookies from the request using the `RequestCookies` API
  let cookie = request.cookies.get('nextjs')
  console.log(cookie) // => { name: 'nextjs', value: 'fast', Path: '/' }
  const allCookies = request.cookies.getAll()
  console.log(allCookies) // => [{ name: 'nextjs', value: 'fast' }]

  request.cookies.has('nextjs') // => true
  request.cookies.delete('nextjs')
  request.cookies.has('nextjs') // => false

  // Setting cookies on the response using the `ResponseCookies` API
  const response = NextResponse.next()
  response.cookies.set('vercel', 'fast')
  response.cookies.set({
    name: 'vercel',
    value: 'fast',
    path: '/',
  })
  cookie = response.cookies.get('vercel')
  console.log(cookie) // => { name: 'vercel', value: 'fast', Path: '/' }
  // The outgoing response will have a `Set-Cookie:vercel=fast;path=/` header.

  return response
}
```

```js filename="middleware.js" switcher
import { NextResponse } from 'next/server'

  response.cookies.set({
    name: 'vercel',

    path: '/',
  })

  return response
}
```

### 設定標頭

您可以使用 `NextResponse` API 設定請求和回應標頭（設定 _請求_ 標頭自 Next.js v13.0.0 起可用）。

```ts filename="middleware.ts" switcher
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  // Clone the request headers and set a new header `x-hello-from-middleware1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-middleware1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-middleware2`
  response.headers.set('x-hello-from-middleware2', 'hello')
  return response
}
```

```js filename="middleware.js" switcher
import { NextResponse } from 'next/server'

export function middleware(request) {
  // Clone the request headers and set a new header `x-hello-from-middleware1`
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-hello-from-middleware1', 'hello')

  // You can also set request headers in NextResponse.next
  const response = NextResponse.next({
    request: {
      // New request headers
      headers: requestHeaders,
    },
  })

  // Set a new response header `x-hello-from-middleware2`
  response.headers.set('x-hello-from-middleware2', 'hello')
  return response
}
```

注意此程式碼片段使用：

- `NextResponse.next({ request: { headers: requestHeaders } })` 使 `requestHeaders` 對上游可用
- **非** `NextResponse.next({ headers: requestHeaders })`，這使 `requestHeaders` 對客戶端可用

深入了解 [NextResponse headers in Middleware](/docs/app/api-reference/functions/next-response#next)。

> **重要資訊**：避免設定大型標頭，這可能會導致根據您的後端 web 伺服器設定而出現 [431 Request Header Fields Too Large](https://developer.mozilla.org/docs/Web/HTTP/Status/431) 錯誤。

### CORS

您可以在 Middleware 中設定 CORS 標頭以允許跨 origin 請求，包括 [簡單](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#simple_requests) 和 [預處理](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#preflighted_requests) 請求。

```tsx filename="middleware.ts" switcher
import { NextRequest, NextResponse } from 'next/server'

const allowedOrigins = ['https://acme.com', 'https://my-app.org']

const corsOptions = {
  'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type, Authorization',
}

export function middleware(request: NextRequest) {
  // Check the origin from the request
  const origin = request.headers.get('origin') ?? ''
  const isAllowedOrigin = allowedOrigins.includes(origin)

  // Handle preflighted requests
  const isPreflight = request.method === 'OPTIONS'

  if (isPreflight) {
    const preflightHeaders = {
      ...(isAllowedOrigin && { 'Access-Control-Allow-Origin': origin }),
      ...corsOptions,
    }
    return NextResponse.json({}, { headers: preflightHeaders })
  }

  // Handle simple requests
  const response = NextResponse.next()

  if (isAllowedOrigin) {
    response.headers.set('Access-Control-Allow-Origin', origin)
  }

  Object.entries(corsOptions).forEach(([key, value]) => {
    response.headers.set(key, value)
  })

  return response
}

export const config = {
  matcher: '/api/:path*',
}
```


### 產生回應

您可以直接通過返回 `Response` 或 `NextResponse` 實例來從 Middleware 回應 (這在 [Next.js v13.1.0](https://nextjs.org/blog/next-13-1#nextjs-advanced-middleware) 後可用)。

```ts filename="middleware.ts" switcher
import type { NextRequest } from 'next/server'
import { isAuthenticated } from '@lib/auth'

// Limit the middleware to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
}

export function middleware(request: NextRequest) {
  // Call our authentication function to check the request
  if (!isAuthenticated(request)) {
    // Respond with JSON indicating an error message
    return Response.json(
      { success: false, message: 'authentication failed' },
      { status: 401 }
    )
  }
}
```

```js filename="middleware.js" switcher
import { isAuthenticated } from '@lib/auth'

// Limit the middleware to paths starting with `/api/`
export const config = {
  matcher: '/api/:function*',
}

export function middleware(request) {

    // Respond with JSON indicating an error message
    return Response.json(
      { success: false, message: 'authentication failed' },
      { status: 401 }
    )
  }
}
```

### 負向匹配

`matcher` 設定允許完整的正則表達式，因此支援像負向預覽或字元匹配的匹配。以下是用於匹配特定路徑以外所有路徑的負向預覽範例：

```js filename="middleware.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
  ],
}
```

您也可以透過使用 `missing` 或 `has` 陣列，或兩者組合，來跳過特定請求的 Middleware：

```js filename="middleware.js"
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico, sitemap.xml, robots.txt (metadata files)
     */
    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      missing: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },

    {
      source:
        '/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)',
      has: [{ type: 'header', key: 'x-present' }],
      missing: [{ type: 'header', key: 'x-missing', value: 'prefetch' }],
    },
  ],
}
```

### `waitUntil` 和 `NextFetchEvent`

`NextFetchEvent` 物件擴充了原生的 [`FetchEvent`](https://developer.mozilla.org/docs/Web/API/FetchEvent) 物件，並包含 [`waitUntil()`](https://developer.mozilla.org/docs/Web/API/ExtendableEvent/waitUntil) 方法。

`waitUntil()` 方法接受一個 Promise 作為引數，並將 Middleware 的生命週期延伸至該 Promise 解決為止。這對於在背景執行工作很有用。

```ts filename="middleware.ts"
import { NextResponse } from 'next/server'
import type { NextFetchEvent, NextRequest } from 'next/server'

export function middleware(req: NextRequest, event: NextFetchEvent) {
  event.waitUntil(
    fetch('https://my-analytics-platform.com', {
      method: 'POST',
      body: JSON.stringify({ pathname: req.nextUrl.pathname }),
    })
  )

  return NextResponse.next()
}
```

### 單元測試 (實驗性)

從 Next.js 15.1 開始，`next/experimental/testing/server` 套件包含有助於單元測試 Middleware 檔案的工具。單元測試 Middleware 可以確保它僅在您期望的路徑上運行，並且在程式碼到達生產環境之前，自訂路由邏輯能按預期運作。

可以使用 `unstable_doesMiddlewareMatch` 函數來斷言 Middleware 是否會在提供的 URL、標頭和 Cookie 上運行。

```js
import { unstable_doesMiddlewareMatch } from 'next/experimental/testing/server'

expect(
  unstable_doesMiddlewareMatch({
    config,
    nextConfig,
    url: '/test',
  })
).toEqual(false)
```

整個 Middleware 函式也可以進行測試。

```js
import { isRewrite, getRewrittenUrl } from 'next/experimental/testing/server'

const request = new NextRequest('https://nextjs.org/docs')
const response = await middleware(request)
expect(isRewrite(response)).toEqual(true)
expect(getRewrittenUrl(response)).toEqual('https://other-domain.com/docs')
// getRedirectUrl could also be used if the response were a redirect
```
