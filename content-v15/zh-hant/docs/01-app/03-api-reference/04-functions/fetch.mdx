Next.js 擴充了 [Web `fetch()` API](https://developer.mozilla.org/docs/Web/API/Fetch_API) 以允許伺服器的每個請求設定其自身的持久快取和重新驗證語意。

在瀏覽器中，`cache` 選項表示 fetch 請求如何與 _瀏覽器_ 的 HTTP 快取互動。透過此擴充功能，`cache` 表示 _伺服器端_ 的 fetch 請求如何與架構的持久 [Data Cache](/docs/app/guides/caching#data-cache) 互動。

您可以在 伺服器組件 (Server Components) 內部直接使用 `async` 和 `await` 調用 `fetch`。

```tsx filename="app/page.tsx" switcher
export default async function Page() {
  let data = await fetch('https://api.vercel.app/blog')
  let posts = await data.json()
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

```jsx filename="app/page.js" switcher
export default async function Page() {
  let data = await fetch('https://api.vercel.app/blog')
  let posts = await data.json()
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

## `fetch(url, options)`

由於 Next.js 擴充了 [Web `fetch()` API](https://developer.mozilla.org/docs/Web/API/Fetch_API)，您可以使用任何 [可用的原生選項](https://developer.mozilla.org/docs/Web/API/fetch#parameters)。

### `options.cache`

配置請求如何與 Next.js [Data Cache](/docs/app/guides/caching#data-cache) 互動。

```ts
fetch(`https://...`, { cache: 'force-cache' | 'no-store' })
```

- **`auto no cache`** (default): Next.js 在開發環境的每次請求中從遠端伺服器擷取資源，但在 `next build` 期間僅獲取一次，因為路由將被靜態預先渲染。如果路由上偵測到 [Dynamic APIs](/docs/app/getting-started/partial-prerendering#dynamic-rendering)，Next.js 將在每次請求時擷取資源。
- **`no-store`**: Next.js 在每次請求中從遠端伺服器擷取資源，即使路由上未偵測到 Dynamic APIs。
- **`force-cache`**: Next.js 在其 Data Cache 中尋找匹配的請求。
  - 如果有匹配且是新資源，則從快取中返回。
  - 如果沒有匹配或匹配已過期，Next.js 將從遠端伺服器獲取資源並用下載的資源更新快取。

### `options.next.revalidate`

```ts
fetch(`https://...`, { next: { revalidate: false | 0 | number } })
```

設定資源的快存取活時間（以秒為單位）。[Data Cache](/docs/app/guides/caching#data-cache)。

- **`false`** - 無限期緩存資源。語意上等同於 `revalidate: Infinity`。HTTP 快取可能會隨時間淘汰舊資源。
- **`0`** - 防止資源被緩存。
- **`number`** - （以秒為單位）指定資源應具有最多 `n` 秒的快取出生存時間。

> **要了解的事**：
>
> - 如果單獨的 `fetch()` 請求設置的 `revalidate` 數值 低于路由的 [默認 `revalidate`](/docs/app/api-reference/file-conventions/route-segment-config#revalidate)，則整個路由的重新驗證間隔將減少。
> - 如果同一路由中相同 URL 的兩個 fetch 請求具有不同的 `revalidate` 值，則使用較低的值。
> - 衝突的選項，例如 `{ revalidate: 3600, cache: 'no-store' }` 不被允許，兩者都將被忽略，在開發模式下警報將打印到終端。

### `options.next.tags`

```ts
fetch(`https://...`, { next: { tags: ['collection'] } })
```

設定資源的快取標籤。之後可以透過 [`revalidateTag`](/docs/app/api-reference/functions/revalidateTag) 按需重新驗證資料。自定義標籤的最大長度為 256 個字符，標籤數量的最大項目為 128。

## 疑難排解

### Fetch 預設 `auto no store` 和 `cache: 'no-store'` 在開發模式下未顯示最新資料

在本地開發環境中，為了加快響應速度並降低計費 API 調用的成本，Next.js 會在伺服器組件 (Server Components) 中跨熱模塊替換 (HMR) 緩存 `fetch` 響應。

預設情況下，[HMR 快取](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache) 適用於所有 fetch 請求，包括那些具有預設 `auto no cache` 和 `cache: 'no-store'` 選項的請求。這意味著未緩存的請求在 HMR 刷新之間將不會顯示最新資料。然而，快取將在導航或整頁重新加載時被清除。

請參閱 [`serverComponentsHmrCache`](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache) 文件以獲取更多資訊。

### 開發模式下的硬重新整理與快取

在開發模式下，如果請求包含 `cache-control: no-cache` 標頭，則忽略 `options.cache`、`options.next.revalidate` 和 `options.next.tags`，並且 `fetch` 請求將從原始來源提供。

當開發者工具中禁用快取或進行硬重新整理時，瀏覽器通常會包含 `cache-control: no-cache`。
