Next.js 內建 TypeScript，使用 `create-next-app` 建立新專案時，會自動安裝必要的套件並設定正確的設定。

要在現有專案中新增 TypeScript，請將檔案重新命名為 `.ts` / `.tsx`。執行 `next dev` 和 `next build` 以自動安裝必要的相依套件，並新增帶有建議設定選項的 `tsconfig.json` 檔案。

> **值得注意**: 如果您已經有 `jsconfig.json` 檔案，請從舊的 `jsconfig.json` 複製 `paths` 編譯器選項到新 `tsconfig.json` 檔案中，並刪除舊的 `jsconfig.json` 檔案。

## IDE 外掛程式

Next.js 包含自定义 TypeScript 外掛程式和類型檢查器，VSCode 和其他程式碼編輯器可用於進階類型檢查和自動完成。

您可以在 VS Code 中透過以下方式啟用外掛程式：

1. 開啟指令選單 (`Ctrl/⌘` + `Shift` + `P`)
2. 搜尋 "TypeScript: Select TypeScript Version"
3. 選擇 "Use Workspace Version"

TypeScript 外掛程式可協助以下事項：

- 傳遞無效的 [區塊設定選項](/docs/app/api-reference/file-conventions/route-segment-config) 值時會發出警告。
- 顯示可用選項及上下文中說明文件。
- 確保 `'use client'` 指令使用正確。
- 確保客戶端掛鉤（如 `useState`）僅在 Client Components 中使用。

## 端到端類型安全性

Next.js App Router 具有**增強的類型安全性**。這包括：

1. **獲取函式與頁面之間的資料不再需要序列化**：您可以在伺服器的組件、佈局和頁面中直接 `fetch`。此資料_不需要_被序列化（轉換為字串）即可傳遞至客戶端供 React 使用。相反地，由於 `app` 預設使用伺服器組件，因此我們可以無需額外的步驟直接使用 `Date`、`Map`、`Set` 等值。過去，您需要使用 Next.js 專屬類型來手動設定伺服器與客戶端之間的邊界。
2. **簡化組件之間的資料流程**：隨著 `_app` 移除並改用根佈局，現在更容易視覺化組件與頁面之間的資料流程。過去，在個別 `pages` 和 `_app` 之間流動的資料難以類型化，且可能引入令人困惑的錯誤。透過 App Router 中的 [共置資料獲取](/docs/app/getting-started/fetching-data)，此問題已不再存在。

[Next.js 中的資料獲取](/docs/app/getting-started/fetching-data) 現在提供幾乎完整的端到端類型安全性，且無須對您選擇的資料庫或內容提供程式進行規定。

我們能夠按照正常 TypeScript 的預期方式為回應資料進行類型化。例如：

```tsx filename="app/page.tsx" switcher
async function getData() {
  const res = await fetch('https://api.example.com/...')
  // The return value is *not* serialized
  // You can return Date, Map, Set, etc.
  return res.json()
}

export default async function Page() {
  const name = await getData()

  return '...'
}
```

若要達到_完整的端到端類型安全性，這也要求您的資料庫或內容提供者支援 TypeScript。這可以透過使用 [ORM](https://en.wikipedia.org/wiki/Object%E2%80%93relational_mapping) 或型別安全查詢建構器來實現。

### 型別檢查 `next.config.ts`

您可以使用 `next.config.ts` 在 Next.js 設定中使用 TypeScript 並匯入型別。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  /* config options here */
}

export default nextConfig
```

> **值得注意**: Module resolution in `next.config.ts` is currently limited to `CommonJS`. This may cause incompatibilities with ESM only packages being loaded in `next.config.ts`.

這可透過 `tsconfig.json` 保持編輯器嚴謹，同時允許生產環境建置使用較寬鬆的設定。

```js filename="next.config.js"
// @ts-check

/** @type {import('next').NextConfig} */
const nextConfig = {
  /* config options here */
}

module.exports = nextConfig
```

### 靜態類型連結 (Statically Typed Links)

Next.js 可以靜態類型化連結，在使用 `next/link` 時防止打字錯誤和其他錯誤，改善頁面之間導覽的安全性。

適用於 Pages 和 App Router 中的 `next/link` 的 `href` prop。在 App Router 中，它也會類型化 `next/navigation` 方法，例如 `push`、`replace` 和 `prefetch`。它不會類型化 Pages Router 中的 `next/router` 方法。

字串 `href` 字面值 (Literal `href` strings) 會經過驗證，但非字面值 `href` 可能需要以 `as Route` 強制轉換。

要參與此功能，需要啟用 `typedRoutes` 並且專案需要使用 TypeScript。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  typedRoutes: true,
}

export default nextConfig
```

Next.js 會在 `.next/types` 中產生連結定義，其中包含應用中現有所有路由的資訊，TypeScript 隨後可據此在編輯器中提供關於無效連結的反饋。

> **小知識 (Good to know)**: 如果未使用 `create-next-app` 設定專案，請在 `tsconfig.json` 的 `include` 陣列中加入 `.next/types/**/*.ts`，以確保包含生成的 Next.js 類型：

{/* 格式忽略開始 */}

```json filename="tsconfig.json" highlight={4}
{
  "include": [
    "next-env.d.ts",
    ".next/types/**/*.ts",
    "**/*.ts",
    "**/*.tsx"
  ],
  "exclude": ["node_modules"]
}
```
{/* 格式忽略結束 */}

目前，支援包含任何字串字面值，包括動態區塊。對於非字串字面值，您需要以 `as Route` 進行手動轉換。以下範例顯示 `next/link` 和 `next/navigation` 的用途：

```tsx filename="app/example-client.tsx"
'use client'

import type { Route } from 'next'
import Link from 'next/link'
import { useRouter } from 'next/navigation'

export default function Example() {
  const router = useRouter()
  const slug = 'nextjs'

  return (
    <>
      {/* Link: literal and dynamic */}
      <Link href="/about" />
      <Link href={`/blog/${slug}`} />
      <Link href={('/blog/' + slug) as Route} />
      {/* TypeScript error if href is not a valid route */}
      <Link href="/aboot" />

      {/* Router: literal and dynamic strings are validated */}
      <button onClick={() => router.push('/about')}>Push About</button>
      <button onClick={() => router.replace(`/blog/${slug}`)}>
        Replace Blog
      </button>
      <button onClick={() => router.prefetch('/contact')}>
        Prefetch Contact
      </button>

      {/* For non-literal strings, cast to Route */}
      <button onClick={() => router.push(('/blog/' + slug) as Route)}>
        Push Non-literal Blog
      </button>
    </>
  )
}
```

這同樣適用於由 middleware 定義的重導向路由：

```ts filename="middleware.ts"
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  if (request.nextUrl.pathname === '/middleware-redirect') {
    return NextResponse.redirect(new URL('/', request.url))
  }

  return NextResponse.next()
}
```

```tsx filename="app/some/page.tsx"
import type { Route } from 'next'

export default function Page() {
  return <Link href={'/middleware-redirect' as Route}>Link Text</Link>
}
```

若要接受在封裝 `next/link` 的自訂元件中的 `href` 參數，請使用泛型：

```tsx
import type { Route } from 'next'
import Link from 'next/link'

function Card<T extends string>({ href }: { href: Route<T> | URL }) {
  return (
    <Link href={href}>
      <div>My Card</div>
    </Link>
  )
}
```

您也可以類型化簡單的資料結構並迭代以呈現連結：

```ts filename="components/nav-items.ts"
import type { Route } from 'next'

type NavItem<T extends string = string> = {
  href: T
  label: string
}

export const navItems: NavItem<Route>[] = [
  { href: '/', label: 'Home' },
  { href: '/about', label: 'About' },
  { href: '/blog', label: 'Blog' },
]
```

接著，迭代項目以呈現 `Link`：

```tsx filename="components/nav.tsx"
import Link from 'next/link'
import { navItems } from './nav-items'

export function Nav() {
  return (
    <nav>
      {navItems.map((item) => (
        <Link key={item.href} href={item.href}>
          {item.label}
        </Link>
      ))}
    </nav>
  )
}
```

> **它是如何運作的？**
>
> 執行 `next dev` 或 `next build` 時，Next.js 會在 `.next` 內部產生一個隱藏的 `.d.ts` 檔案，其中包含您的應用中現有所有路由的資訊（所有有效的路由作為 `Link` 的 `href` 型別）。此 `.d.ts` 檔案包含在 `tsconfig.json` 中，TypeScript 編譯器將檢查此 `.d.ts` 檔案並在您的編輯器中提供關於無效連結的反饋。

### 針對環境變數的類型完整提示 (Type IntelliSense for Environment Variables)

開發期間，Next.js 會在 `.next/types` 中產生一個 `.d.ts` 檔案，其中包含您編輯器的 IntelliSense 所需的外部變數載入資訊。如果同一個環境變數鍵定在多份檔案中，則會根據 [環境變數載入順序](/docs/app/guides/environment-variables#environment-variable-load-order) 進行去重。

要參與此功能，需要啟用 `experimental.typedEnv` 並且專案需要使用 TypeScript。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    typedEnv: true,
  },
}

export default nextConfig
```

> **小知識 (Good to know)**: 類型是根據開發時載入的 environnement 變數產生的，預設排除來自有 `.env.production*` 檔案的變數。若要包含生產專用變數，請執行 `next dev` 並帶上 `NODE_ENV=production`。

### 遞增類型檢查

自 `v10.2.1` 以來，Next.js 支援 [遞增類型檢查](https://www.typescriptlang.org/tsconfig#incremental)（在您的 `tsconfig.json` 中啟用時）這可以在大型應用程式中幫助加快類型檢查速度。

### 自訂 `tsconfig` 路徑

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  typescript: {
    tsconfigPath: 'tsconfig.build.json',
  },
}

export default nextConfig
```

在某些情況下，您可能想要為建置或工具使用不同的 TypeScript 設定。若要執行此操作，請在 `next.config.ts` 中設定 `typescript.tsconfigPath` 以指向 Next.js 的另一个 `tsconfig` 檔案。

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const isProd = process.env.NODE_ENV === 'production'

const nextConfig: NextConfig = {
  typescript: {
    tsconfigPath: isProd ? 'tsconfig.build.json' : 'tsconfig.json',
  },
}

export default nextConfig
```

> **Good to know**:
>
> - IDEs typically read `tsconfig.json` for diagnostics and IntelliSense, so you can still see IDE warnings while production builds use the alternate config. Mirror critical options if you want parity in the editor.

### 停用生產環境中的 TypeScript 錯誤

Next.js 在您的專案中存在 TypeScript 錯誤時，會失敗您的**生產建置**（`next build`）。

如果您希望在應用程式有錯誤的情況下仍危險地產出生產代碼，您可以停用內建的類型檢查步驟。

若停用，請確保您正在建置或部署流程中執行類型檢查，否則這可能會非常危險。

開啟 `next.config.ts` 並在 `typescript` 設定中啟用 `ignoreBuildErrors` 選項：

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  typescript: {
    // !! WARN !!
    // Dangerously allow production builds to successfully complete even if
    // your project has type errors.
    // !! WARN !!
    ignoreBuildErrors: true,
  },
}

export default nextConfig
```

> **小知識 (Good to know)**: 您可以在建置前執行 `tsc --noEmit` 來自行檢查 TypeScript 錯誤。這在 CI/CD 流程中有用，您希望在下載部署前檢查 TypeScript 錯誤。

### 自訂類型宣告

當您需要宣告自訂類型時，您可能會想修改 `next-env.d.ts`。然而，此檔案是自動產生的，因此您進行的任何變更都會被覆蓋。相反地，您應該建立一個新檔案，比方說 `new-types.d.ts`，並在您的 `tsconfig.json` 中參考它：

```json filename="tsconfig.json"
{
  "compilerOptions": {
    "skipLibCheck": true
    //...truncated...
  },
  "include": [
    "new-types.d.ts",
    "next-env.d.ts",
    ".next/types/**/*.ts",
    "**/*.ts",
    "**/*.tsx"
  ],
  "exclude": ["node_modules"]
}
```
