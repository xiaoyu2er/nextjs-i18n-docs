## 使用方法

關於在 `<head>` 中內嵌 CSS 的實驗性支援。啟用此標誌時，所有原本會生成 `<link>` 標籤的地方將改為使用生成的 `<style>` 標籤。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    inlineCss: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    inlineCss: true,
  },
}

module.exports = nextConfig
```

## 取捨 (Trade-Offs)

### 何時使用內嵌 CSS (Inline CSS)

在很多情況下，內嵌 CSS 都能帶來好處：

- **首次訪客**：由於 CSS 檔案是渲染阻斷資源，內嵌消除了首次訪客遇到的初始下載延遲，從而提高頁面載入效能。

- **效能指標**：通過移除 CSS 檔案的額外網路請求，內嵌可以顯著改善關鍵指標，例如內容首次繪製 (FCP) 和最大內容繪製 (LCP)。

- **慢速連線**：對於網路較慢的用戶，每個請求都會增加顯著的延遲，在這種情況下，內嵌 CSS 可以通過減少網路往返來提供顯著的效能提升。

- **原子 CSS 捆綁 (例如 Tailwind)**：對於像 Tailwind CSS 這麼的工具優先框架，頁面所需的樣式大小通常相對於設計複雜度是 O(1)。這使得內嵌成為一個很有吸引力的選擇，因為當前頁面的整套樣式輕量級且不會隨頁面大小增長。內嵌 Tailwind 樣式可確保負載最小化，無需額外的網路請求，這可以進一步提升效能。

### 何時不使用內嵌 CSS

雖然內嵌 CSS 為效能帶來了顯著的好處，但也有一些情況它可能不是最佳選擇：

- **大型 CSS 捆綁**：如果 CSS 捆綁过大，內嵌它可能會顯著增加 HTML 的大小，導致首次字節傳輸時間 (TTFB) 變慢，對於網路連線較慢的用戶的效能可能會變差。
- **動態或頁面特定 CSS**：對於具有高度動態樣式或頁面使用不同 CSS 集的應用，內嵌可能會導致重複和膨脹，因為所有頁面的完整 CSS 可能需要反覆內嵌。

- **瀏覽器快取**：在用戶經常返回您的網站的情況下，外部 CSS 檔案允許瀏覽器有效地快取樣式，減少後續訪問的資料傳輸。內嵌 CSS 消除了這項優點。

請慎重評估這些取捨，並考慮將內嵌與其他策略結合使用，例如關鍵 CSS 提取 (critical CSS extraction) 或混合方法，以獲得最適合您網站需求的效果。

> **值得了解**：
>
> 此功能目前處於實驗階段，並存在一些已知限制：
>
> - CSS 內嵌是全局應用的，無法在頁面基礎上配置
> - 樣式在初始頁面載入期間會被複製 —— 一次在 SSR 的 `<style>` 標籤內，一次在 RSC payload 中
> - 當導航到靜態渲染的頁面時，樣式將使用 `<link>` 標籤而不是內嵌 CSS 以避免重複
> - 此功能在開發模式下不可用，僅在生產構建中有效
