在 Next.js 中，路由預設會在伺服器上渲染。這通常意味著客戶端必須等待伺服器回應後才能顯示新路由。Next.js 內建了 [預先取用](#prefetching)、[串流](#streaming) 和 [客戶端轉換](#client-side-transitions)，確保導航保持快速且具回應性。

本指南解釋 Next.js 中導航的運作方式，以及您如何為 [動態路由](#dynamic-routes-without-loadingtsx) 和 [慢網路](#slow-networks) 最佳化它。

## 導航如何運作

要了解 Next.js 中如何導航，熟悉以下概念會有所幫助：

- [伺服器渲染](#server-rendering)
- [預先取用](#prefetching)
- [串流](#streaming)
- [客戶端轉換](#client-side-transitions)

### 伺服器渲染

在 Next.js 中，[佈局和頁面](/docs/app/getting-started/layouts-and-pages) 預設為 [React 伺服器元件](https://react.dev/reference/rsc/server-components)。在所有初始和後續導航中，[伺服器元件負載](/docs/app/getting-started/server-and-client-components#how-do-server-and-client-components-work-in-nextjs) 會在伺服器上生成，然後再發送給客戶端。

伺服器渲染有兩種類型，基於 *_發生時刻*_：

- **靜態渲染（或預渲染）** 發生在建置時間或 [重新驗證](/docs/app/getting-started/caching-and-revalidating) 期間，結果會遭到快取。
- **動態渲染** 發生在請求時間，作為對客戶端請求的回應。

伺服器渲染的取捨是客戶端必須等待伺服器回應後才能顯示新路由。Next.js 透過預先取用 [用戶可能造訪的路由](#prefetching) 並執行 [客戶端轉換](#client-side-transitions) 來解決這延遲。

> **值得注意**：HTML 也會為首次造訪而生成。

### 預先取用

預先取用是在用戶導航到該路由之前，在背景中載入路由的過程。這讓應用程式之間的路由導航感覺是即時的，因為當用戶點擊連結時，渲染下一個路由的資料已經在客戶端可用。

Next.js 會自動預先取用與 [`<Link>` 元件](/docs/app/api-reference/components/link) 連結的路由，當它們進入用戶的視窗範圍時。

```tsx filename="app/layout.tsx" switcher
import Link from 'next/link'

export default function Layout({ children }: { children: React.ReactNode }) {
  return (
    <html>
      <body>
        <nav>
          {/* Prefetched when the link is hovered or enters the viewport */}
          <Link href="/blog">Blog</Link>
          {/* No prefetching */}
          <a href="/contact">Contact</a>
        </nav>
        {children}
      </body>
    </html>
  )
}
```

```jsx filename="app/layout.js" switcher
import Link from 'next/link'

export default function Layout() {
  return (
    <html>
      <body>
        <nav>
          {/* Prefetched when the link is hovered or enters the viewport */}
          <Link href="/blog">Blog</Link>
          {/* No prefetching */}
          <a href="/contact">Contact</a>
        </nav>
        {children}
      </body>
    </html>
  )
}
```

預先取用的路由量取決於它是靜態還是動態的：

- **靜態路線**：完整的路由會遭到預先取用。
- **動態路線**：預先取用會被跳過，或者如果存在 [`loading.tsx`](/docs/app/api-reference/file-conventions/loading)，則會部分預先取用。

透過跳過或部分預先取用動態路由，Next.js 可避免為用戶可能永遠不會造訪的路由在伺服器上執行不必要的作業。然而，等待伺服器回應後才能導航可能會給用戶一種應用程式沒有回應的印象。

### 串流

串流允許伺服器在資料準備好時，立即發送動態路由的部分給客戶端，而不必等待整個路由渲染完畢。這意味著用戶能更早看到某些內容，即使頁面部分區域仍在載入。

對於動態路由，這意味著它們可以 **部分預先取用**。也就是說，共用佈局和載入骨架圖案可以提前請求。

在幕後，Next.js 會自動將 `page.tsx` 的內容包裝在 `<Suspense>` 邊界中。在路由載入期間，預先取用的 fallback UI 將被顯示，一旦就緒則替換為實際內容。

> **值得注意**：您也可以使用 [`<Suspense>`](https://react.dev/reference/react/Suspense) 為巢狀元件建立載入 UI。

`loading.tsx` 的好處：

- 立即導航並為用戶提供視覺回饋。
- 共用佈局保持互動性，導航可中斷。
- 改善核心網頁指標：[TTFB](https://web.dev/articles/ttfb)、[FCP](https://web.dev/articles/fcp) 和 [TTI](https://web.dev/articles/tti)。

為了進一步改善導航體驗，Next.js 使用 `<Link>` 元件執行 [客戶端轉換](#client-side-transitions)。

### 客戶端轉換

傳統上，導航至伺服器渲染的頁面會觸發完整頁面載入。這會清除狀態、重置捲軸位置，並阻斷互動功能。

Next.js 使用 `<Link>` 元件的客戶端轉換來避免此情況。它不是重新載入頁面，而是更新內容，方式如下：

- 保留任何共用佈局和 UI。
- 用預先取用的載入狀態或新頁面替換目前頁面（如有）。

客戶端轉換讓伺服器渲染的應用程式 *_感覺*_ 像是客戶端渲染的應用程式。當與 [預先取用](#prefetching) 和 [串流](#streaming) 搭配時，它可實現快速轉換，即使是針對動態路由。

## 什麼可能導致轉換變慢？

這些 Next.js 最佳化可讓導航快速且具回應性。然而，在某些情況下，轉換仍可能 *_感覺*_ 變慢。以下是一些常見原因及如何改善用戶體驗：

### 沒有 `loading.tsx` 的動態路由

當導航至動態路由時，客戶端必須等待伺服器回應後才能顯示結果。這可能會給用戶一種應用程式沒有回應的印象。

我們建議將 `loading.tsx` 新增至動態路由，以啟用部分預先取用、觸發立即導航，並在路由渲染時顯示載入 UI。

```tsx filename="app/blog/[slug]/loading.tsx" switcher
export default function Loading() {
  return <LoadingSkeleton />
}
```

```jsx filename="app/blog/[slug]/loading.js" switcher
export default function Loading() {
  return <LoadingSkeleton />
}
```

> **值得注意**：在開發模式下，您可以使用 Next.js Devtools 來識別路由是靜態還是動態。請參閱 [`devIndicators`](/docs/app/api-reference/config/next-config-js/devIndicators) 以取得更多資訊。

### 缺少 `generateStaticParams` 的動態區段

如果 [動態區段](/docs/app/api-reference/file-conventions/dynamic-routes) 本可以被預渲染，但因為缺少 [`generateStaticParams`](/docs/app/api-reference/functions/generate-static-params) 而未能預渲染，則路由會 fallback 至請求時間的動態渲染。

確保路由在建置時間以靜態方式生成，新增 `generateStaticParams`：

```tsx filename="app/blog/[slug]/page.tsx" switcher
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  // ...
}
```

```jsx filename="app/blog/[slug]/page.js" switcher
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))

export default async function Page({ params }) {
  const { slug } = await params
  // ...
}
```

### 慢網路

在慢或網路不穩的情況下，預先取用可能無法在用戶點擊連結前完成。這會影響靜態和動態路由。在這些情況下，`loading.js` fallback 可能不會立即出現，因為它尚未被預先取用。

為了改善感知效能，您可以使用 [`useLinkStatus` hook](/docs/app/api-reference/functions/use-link-status) 以在轉換進行中時為用戶顯示行內視覺回饋（如連結上的旋轉器或文字光暈）。

```tsx filename="app/ui/loading-indicator.tsx" switcher
'use client'

import { useLinkStatus } from 'next/link'

export default function LoadingIndicator() {
  const { pending } = useLinkStatus()
  return pending ? (
    <div role="status" aria-label="Loading" className="spinner" />
  ) : null
}
```

```jsx filename="app/ui/loading-indicator.js" switcher
'use client'

import { useLinkStatus } from 'next/link'

export default function LoadingIndicator() {
  const { pending } = useLinkStatus()
  return pending ? (
    <div role="status" aria-label="Loading" className="spinner" />
  ) : null
}
```

您可以透過新增初始動畫延遲（例如 100ms）並開始動畫為不可見狀態（例如 `opacity: 0`）來「延遲」載入指示器。這意味著僅當導航時間長於指定的延遲時，才會顯示載入指示器。

```css
.spinner {
  /* ... */
  opacity: 0;
  animation:
    fadeIn 500ms 100ms forwards,
    rotate 1s linear infinite;
}

@keyframes fadeIn {
  from {
    opacity: 0;
  }
  to {
    opacity: 1;
  }
}

@keyframes rotate {
  to {
    transform: rotate(360deg);
  }
}
```

> **值得注意**：您可以使用其他視覺回饋模式，例如進度條。在 [此處](https://github.com/vercel/react-transition-progress) 檢視範例。

### 停用預先取用

您可以將 `<Link>` 元件的 `prefetch` prop 設為 `false` 以取消選擇預先取用。這有助於在渲染大量連結列表（例如無限滾動表）時避免不必要的資源使用。

```tsx
<Link prefetch={false} href="/blog">
  Blog
</Link>
```

然而，停用預先取用也會帶來取捨：

- **靜態路由** 僅會在用戶點擊連結時才會被取用。
- **動態路由** 必須先在伺服器上渲染，然後客戶端才能導航至該路由。

為了在不完全停用預先取用的情況下減少資源使用，您可以僅在懸停時預先取用。這將預先取用限制為用戶 *_更大可能*_ 造訪的路由，而非視窗中的所有連結。

```tsx filename="app/ui/hover-prefetch-link.tsx" switcher
'use client'

import Link from 'next/link'
import { useState } from 'react'

function HoverPrefetchLink({
  href,
  children,
}: {
  href: string
  children: React.ReactNode
}) {
  const [active, setActive] = useState(false)

  return (
    <Link
      href={href}
      prefetch={active ? null : false}
      onMouseEnter={() => setActive(true)}
    >
      {children}
    </Link>
  )
}
```


### 水化未完成

`<Link>` 是一個客戶端元件，必須先水化才能預先取用路由。在首次造訪時，大型 JavaScript 套件可能會延遲水化，導致預先取用無法立即開始。

React 透過选择性水化 (Selective Hydration) 減輕此情況，您可以透過以下方式進一步改善：

- 使用 [`@next/bundle-analyzer`](/docs/app/guides/package-bundling#analyzing-javascript-bundles) 插件來識別並透過移除大型套件來減少套件大小。
- 盡可能將邏輯從客戶端移至伺服器。請參閱 [伺服器和客戶端元件](/docs/app/getting-started/server-and-client-components) 文件以取得指導。

## 範例

### Native History API

Next.js 允許您使用原生 [`window.history.pushState`](https://developer.mozilla.org/en-US/docs/Web/API/History/pushState) 和 [`window.history.replaceState`](https://developer.mozilla.org/en-US/docs/Web/API/History/replaceState) 方法來更新瀏覽器的歷史堆疊而無需重新載入頁面。

`pushState` 和 `replaceState` 呼叫會整合到 Next.js 路由器中，允許您與 [`usePathname`](/docs/app/api-reference/functions/use-pathname) 和 [`useSearchParams`](/docs/app/api-reference/functions/use-search-params) 同步。

#### `window.history.pushState`

Use it to add a new entry to the browser's history stack. The user can navigate back to the previous state. For example, to sort a list of products:

```tsx fileName="app/ui/sort-products.tsx" switcher
'use client'

import { useSearchParams } from 'next/navigation'

export default function SortProducts() {
  const searchParams = useSearchParams()

  function updateSorting(sortOrder: string) {
    const params = new URLSearchParams(searchParams.toString())
    params.set('sort', sortOrder)
    window.history.pushState(null, '', `?${params.toString()}`)
  }

  return (
    <>
      <button onClick={() => updateSorting('asc')}>Sort Ascending</button>
      <button onClick={() => updateSorting('desc')}>Sort Descending</button>
    </>
  )
}
```


#### `window.history.replaceState`

Use it to replace the current entry on the browser's history stack. The user is not able to navigate back to the previous state. For example, to switch the application's locale:

```tsx fileName="app/ui/locale-switcher.tsx" switcher
'use client'

import { usePathname } from 'next/navigation'

export function LocaleSwitcher() {
  const pathname = usePathname()

  function switchLocale(locale: string) {
    // e.g. '/en/about' or '/fr/contact'
    const newPath = `/${locale}${pathname}`
    window.history.replaceState(null, '', newPath)
  }

  return (
    <>
      <button onClick={() => switchLocale('en')}>English</button>
      <button onClick={() => switchLocale('fr')}>French</button>
    </>
  )
}
```
