### 路由檔案

### 巢狀路由

資料夾定義 URL 區段。巢狀資料夾會巢狀區段。任何層級的佈局會包圍其子區段。當存在 `page` 或 `route` 檔案時，路由即公開。

| Path                        | URL pattern     | Notes                         |
| --------------------------- | --------------- | ----------------------------- |
| `app/layout.tsx`            | —               | 根佈局包圍所有路由            |
| `app/blog/layout.tsx`       | —               | 包圍 /blog 及其子項目          |
| `app/page.tsx`              | `/`             | 公開路由                      |
| `app/blog/page.tsx`         | `/blog`         | 公開路由                      |
| `app/blog/authors/page.tsx` | `/blog/authors` | 公開路由                      |

### 動態路由

使用方括號參數化區段。使用 `[segment]` 為單一參數，`[...segment]` 為萬能捕捉，`[[...segment]]` 為可選的萬能捕捉。透過 [`params`](/docs/app/api-reference/file-conventions/page#params-optional) prop 存取值。

| Path                            | URL pattern                                                          |
| ------------------------------- | -------------------------------------------------------------------- |
| `app/blog/[slug]/page.tsx`      | `/blog/my-first-post`                                                |
| `app/shop/[...slug]/page.tsx`   | `/shop/clothing`, `/shop/clothing/shirts`                            |
| `app/docs/[[...slug]]/page.tsx` | `/docs`, `/docs/layouts-and-pages`, `/docs/api-reference/use-router` |

### 路由群組與私密資料夾

透過路由群組 [`(group)`](/docs/app/api-reference/file-conventions/route-groups#convention) 在不變更 URL 的情況下組織程式碼，並透過私密資料夾 [`_folder`](#private-folders) 共置不可路由的檔案。

| Path                            | URL pattern | Notes                                     |
| ------------------------------- | ----------- | ----------------------------------------- |
| `app/(marketing)/page.tsx`      | `/`         | 群組從 URL 中省略                           |
| `app/(shop)/cart/page.tsx`      | `/cart`     | 在 (shop) 內共享佈局                       |
| `app/blog/_components/Post.tsx` | —           | 不可路由；UI 工具的安全位置                |
| `app/blog/_lib/data.ts`         | —           | 不可路由；工具的安全位置                  |

### 平行與攔截路由

這些功能適合特定的 UI 模式，例如基於 slot 的佈局或模態框路由。

使用 `@slot` 供父層佈局渲染具名稱的 slot。使用攔截模式在当前佈局中渲染另一路由而不變更 URL，例如在列表上方以模態框顯示詳細資訊檢視。

| Pattern (docs)                                                                              | Meaning              | Typical use case                     |
| ------------------------------------------------------------------------------------------- | -------------------- | ------------------------------------ |
| [`@folder`](/docs/app/api-reference/file-conventions/parallel-routes#slots)                 | 具名稱的 slot        | 側邊欄 + 主要內容                    |
| [`(.)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)      | 攔截同一層級         | 在模態框中預覽夥伴路由               |
| [`(..)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)     | 攔截上層             | 以覆蓋層開啟子層                     |
| [`(..)(..)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention) | 攔截兩層級           | 深度巢狀覆蓋層                       |
| [`(...)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)    | 從根攔截             | 在當前檢視中顯示任意路由             |

## 組織你的專案

Next.js 對於你如何組織和共置專案檔案沒有強制定義。但它確實提供了多項功能來協助你組織專案。

### 元件層級

在特殊檔案中定義的元件會依特定層級渲染：

- `layout.js`
- `template.js`
- `error.js` (React 錯誤邊界)
- `loading.js` (React suspense 邊界)
- `not-found.js` (React 錯誤邊界)
- `page.js` 或巢狀 `layout.js`

元件會依嵌套路由遞迴渲染，意味著路由區段的元件將**嵌套在**其父層區段元件**內部**。

### 共置

在 `app` 資料夾中，巢狀資料夾定義路由結構。每個資料夾代表一個路由區段，該區段對應到 URL 路徑中的相對區段。

然而，即使路由結構透過資料夾定義，直到向路由區段新增 `page.js` 或 `route.js` 檔案之前，路由都不可公開存取。

<Image
  alt="A diagram showing how a route is not publicly accessible until a page.js or route.js file is added to a route segment."
  srcLight="/docs/light/project-organization-not-routable.png"
  srcDark="/docs/dark/project-organization-not-routable.png"
  width="1600"
  height="444"
/>

而且，即使路由已公開存取，只有 `page.js` 或 `route.js` 回傳的內容才會傳送給客戶端。

<Image
  alt="A diagram showing how page.js and route.js files make routes publicly accessible."
  srcLight="/docs/light/project-organization-routable.png"
  srcDark="/docs/dark/project-organization-routable.png"
  width="1600"
  height="687"
/>

這意味著專案檔案可以在 `app` 資料夾的路由區段內安全共置，而不會意外變得可路由。

<Image
  alt="A diagram showing colocated project files are not routable even when a segment contains a page.js or route.js file."
  srcLight="/docs/light/project-organization-colocation.png"
  srcDark="/docs/dark/project-organization-colocation.png"
  width="1600"
  height="1011"
/>

> **小知識**: 雖然你**可以**在 `app` 中**共置**你的專案檔案，但你並**不需要**。如果你偏好，可以 [將它們保留在 `app` 資料夾之外](#store-project-files-outside-of-app)。

### 私密資料夾

私密資料夾可透過在資料夾前加上底線建立：`_folderName`

這表示該資料夾是私有實現細節，路由系統應忽略它，從而使該資料夾及其所有子資料夾退出路由機制。

<Image
  alt="An example folder structure using private folders"
  srcLight="/docs/light/project-organization-private-folders.png"
  srcDark="/docs/dark/project-organization-private-folders.png"
  width="1600"
  height="849"
/>

由於 `app` 資料夾中的檔案可 [安全共置](#colocation) 預設，私密資料夾並非共置所需。然而，它們可針對以下用途：

- 將 UI 邏輯與路由邏輯分離。
- 一致地組織專案與 Next.js 生態系統中的內部檔案。
- 在程式碼編輯器中排序與分組檔案。
- 避免與未來 Next.js 檔案慣例名稱衝突的潛在風險。

> **小知識**:
>
> - 雖然這不是框架慣例，但你可能也會考慮使用相同的底線模式將私密資料夾外的檔案標記為「私密」。
> - 你可以透過在資料夾名稱前加上 `%5F`（底線的 URL 編碼形式）來建立以底線開頭的 URL 區段：%5FfolderName。
> - 如果你不使用私密資料夾，了解 Next.js [特殊檔案慣例](/docs/app/getting-started/project-structure#routing-files) 會有幫助，以避免意外的名稱衝突。

### 路由群組

路由群組可透過在資料夾外加括弧建立：`(folderName)`

這表示該資料夾用於組織目的，不應包含在路由的 URL 路徑中。

<Image
  alt="An example folder structure using route groups"
  srcLight="/docs/light/project-organization-route-groups.png"
  srcDark="/docs/dark/project-organization-route-groups.png"
  width="1600"
  height="849"
/>

路由群組適用於：

- 依網站區塊、意圖或團隊組織路由。例如行銷頁面、後台頁面等。
- 在同一路由層級啟用巢狀佈局：
  - [在同一區段建立多個巢狀佈局，包括多個根佈局](#creating-multiple-root-layouts)
  - [在同一共用區段中向部分路由新增佈局](#opting-specific-segments-into-a-layout)

### `src` 資料夾

Next.js 支援將應用程式程式碼（包括 `app`）儲存在可選的 [`src` 資料夾](/docs/app/api-reference/file-conventions/src-folder) 中。這將應用程式程式碼與大部分位於專案根目錄的專案配置文件分離。

### 巢狀路由

資料夾定義 URL 區段。巢狀資料夾會巢狀區段。任何層級的佈局會包圍其子區段。當存在 `page` 或 `route` 檔案時，路由即公開。

| Path                        | URL pattern     | Notes                         |
| --------------------------- | --------------- | ----------------------------- |
| `app/layout.tsx`            | —               | 根佈局包圍所有路由            |
| `app/blog/layout.tsx`       | —               | 包圍 /blog 及其子項目          |
| `app/page.tsx`              | `/`             | 公開路由                      |
| `app/blog/page.tsx`         | `/blog`         | 公開路由                      |
| `app/blog/authors/page.tsx` | `/blog/authors` | 公開路由                      |

### 動態路由

使用方括號參數化區段。使用 `[segment]` 為單一參數，`[...segment]` 為萬能捕捉，`[[...segment]]` 為可選的萬能捕捉。透過 [`params`](/docs/app/api-reference/file-conventions/page#params-optional) prop 存取值。

| Path                            | URL pattern                                                          |

| `app/docs/[[...slug]]/page.tsx` | `/docs`, `/docs/layouts-and-pages`, `/docs/api-reference/use-router` |

## 組織你的專案

Next.js 對於你如何組織和共置專案檔案沒有強制定義。但它確實提供了多項功能來協助你組織專案。

### 元件層級

在特殊檔案中定義的元件會依特定層級渲染：

- `layout.js`
- `template.js`
- `error.js` (React 錯誤邊界)
- `loading.js` (React suspense 邊界)
- `not-found.js` (React 錯誤邊界)
- `page.js` 或巢狀 `layout.js`

### 共置

在 `app` 資料夾中，巢狀資料夾定義路由結構。每個資料夾代表一個路由區段，該區段對應到 URL 路徑中的相對區段。

然而，即使路由結構透過資料夾定義，直到向路由區段新增 `page.js` 或 `route.js` 檔案之前，路由都不可公開存取。

### 私密資料夾

私密資料夾可透過在資料夾前加上底線建立：`_folderName`

這表示該資料夾是私有實現細節，路由系統應忽略它，從而使該資料夾及其所有子資料夾退出路由機制。

<Image
  alt="An example folder structure using private folders"
  srcLight="/docs/light/project-organization-private-folders.png"
  srcDark="/docs/dark/project-organization-private-folders.png"
  width="1600"
  height="849"
/>

由於 `app` 資料夾中的檔案可 [安全共置](#colocation) 預設，私密資料夾並非共置所需。然而，它們可針對以下用途：

- 將 UI 邏輯與路由邏輯分離。
- 一致地組織專案與 Next.js 生態系統中的內部檔案。
- 在程式碼編輯器中排序與分組檔案。
- 避免與未來 Next.js 檔案慣例名稱衝突的潛在風險。

> **小知識**:
>
> - 雖然這不是框架慣例，但你可能也會考慮使用相同的底線模式將私密資料夾外的檔案標記為「私密」。
> - 你可以透過在資料夾名稱前加上 `%5F`（底線的 URL 編碼形式）來建立以底線開頭的 URL 區段：%5FfolderName。
> - 如果你不使用私密資料夾，了解 Next.js [特殊檔案慣例](/docs/app/getting-started/project-structure#routing-files) 會有幫助，以避免意外的名稱衝突。

### 路由群組

路由群組可透過在資料夾外加括弧建立：`(folderName)`

這表示該資料夾用於組織目的，不應包含在路由的 URL 路徑中。

<Image
  alt="An example folder structure using route groups"
  srcLight="/docs/light/project-organization-route-groups.png"
  srcDark="/docs/dark/project-organization-route-groups.png"
  width="1600"
  height="849"

路由群組適用於：

- 依網站區塊、意圖或團隊組織路由。例如行銷頁面、後台頁面等。
- 在同一路由層級啟用巢狀佈局：
  - [在同一區段建立多個巢狀佈局，包括多個根佈局](#creating-multiple-root-layouts)
  - [在同一共用區段中向部分路由新增佈局](#opting-specific-segments-into-a-layout)

## 範例

The following section lists a very high-level overview of common strategies. The simplest takeaway is to choose a strategy that works for you and your team and be consistent across the project.

> **小知識**: In our examples below, we're using `components` and `lib` folders as generalized placeholders, their naming has no special framework significance and your projects might use other folders like `ui`, `utils`, `hooks`, `styles`, etc.

### Store project files outside of `app`

This strategy stores all application code in shared folders in the root of your project and keeps the `app` directory purely for routing purposes.

### Store project files in top-level folders inside of `app`

This strategy stores all application code in shared folders in the root of the `app` directory.

This strategy stores globally shared application code in the root `app` directory and splits more specific application code into the route segments that use them.

### Organize routes without affecting the URL path

To organize routes without affecting the URL, create a group to keep related routes together. The folders in parenthesis will be omitted from the URL (e.g. `(marketing)` or `(shop)`).

### Opting specific segments into a layout

To opt specific routes into a layout, create a new route group (e.g. `(shop)`) and move the routes that share the same layout into the group (e.g. `account` and `cart`). The routes outside of the group will not share the layout (e.g. `checkout`).

  alt="Route Groups with Opt-in Layouts"

### Opting for loading skeletons on a specific route

To apply a [loading skeleton](/docs/app/api-reference/file-conventions/loading) via a `loading.js` file to a specific route, create a new route group (e.g., `(overview)`) and then move your `loading.tsx` inside that route group.

<Image
  alt="Folder structure showing a loading.tsx and a page.tsx inside the route group"
  srcLight="/docs/light/route-group-loading.png"
  srcDark="/docs/dark/route-group-loading.png"
  width="1600"
  height="444"
/>

Now, the `loading.tsx` file will only apply to your dashboard → overview page instead of all your dashboard pages without affecting the URL path structure.

### Creating multiple root layouts

To create multiple [root layouts](/docs/app/api-reference/file-conventions/layout#root-layout), remove the top-level `layout.js` file, and add a `layout.js` file inside each route group. This is useful for partitioning an application into sections that have a completely different UI or experience. The `<html>` and `<body>` tags need to be added to each root layout.

<Image
  alt="Route Groups with Multiple Root Layouts"
  srcLight="/docs/light/route-group-multiple-root-layouts.png"
  srcDark="/docs/dark/route-group-multiple-root-layouts.png"
  width="1600"
  height="687"
/>

In the example above, both `(marketing)` and `(shop)` have their own root layout.
