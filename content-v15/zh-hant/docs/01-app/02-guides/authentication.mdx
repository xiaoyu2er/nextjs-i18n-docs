理解身份驗證對於保護您的應用程式至關重要。這頁將指導您使用哪些 React 和 Next.js 功能來實現身份驗證。

在開始之前，將過程分解為三個概念會有所幫助：

1. **[身份驗證](#authentication)**：驗證用戶是否為其聲稱的本人。這需要用戶證明其身份，例如透過他們擁有的東西，如用戶名和密碼。
2. **[會話管理](#session-management)**：在請求之間跟蹤用戶的身分驗證狀態。
3. **[授權](#authorization)**：決定用戶可以訪問哪些路由和數據。

## 身份驗證

<AppOnly>

### 註冊和登入功能

您可以使用 [`<form>`](https://react.dev/reference/react-dom/components/form) 元素配合 React 的 [伺服器操作](/docs/app/getting-started/updating-data) 和 `useActionState` 來捕獲用戶憑據、驗證表單欄位，並呼叫您的身份驗證提供者的 API 或資料庫。

由於伺服器操作一律在伺服器上執行，因此它為處理身份驗證邏輯提供了安全環境。

以下是實施註冊/登入功能的步驟：

#### 1. 捕獲用戶憑據

要捕獲用戶憑據，請建立一個在提交時調用伺服器操作的表單。例如，一個接受用戶姓名、電子郵件和密碼的註冊表單：

```tsx filename="app/ui/signup-form.tsx" switcher
import { signup } from '@/app/actions/auth'

export function SignupForm() {

      <div>
        <label htmlFor="email">Email</label>
        <input id="email" name="email" type="email" placeholder="Email" />
      </div>
      <div>
        <label htmlFor="password">Password</label>
        <input id="password" name="password" type="password" />
      </div>
      <button type="submit">Sign Up</button>

  )
}
```

#### 2. 在伺服器上驗證表單欄位

使用伺服器操作在伺服器上驗證表單欄位。如果您的身份驗證提供者不提供表單驗證，您可以使用如 [Zod](https://zod.dev/) 或 [Yup](https://github.com/jquense/yup) 之類的表單驗證庫。

以 Zod 為例，您可以定義包含適當錯誤訊息的表單結構：

```ts filename="app/lib/definitions.ts" switcher
import { z } from 'zod'

export const SignupFormSchema = z.object({
  name: z
    .string()
    .min(2, { message: 'Name must be at least 2 characters long.' })
    .trim(),
  email: z.string().email({ message: 'Please enter a valid email.' }).trim(),
  password: z
    .string()
    .min(8, { message: 'Be at least 8 characters long' })
    .regex(/[a-zA-Z]/, { message: 'Contain at least one letter.' })
    .regex(/[0-9]/, { message: 'Contain at least one number.' })
    .regex(/[^a-zA-Z0-9]/, {
      message: 'Contain at least one special character.',
    })
    .trim(),
})

export type FormState =
  | {
      errors?: {
        name?: string[]
        email?: string[]
        password?: string[]
      }
      message?: string
    }
  | undefined
```

為了避免不必要的身份驗證提供者 API 或資料庫調用，如果任何表單欄位不匹配定義的結構，您可以在伺服器操作中 `return` 提前結束。

```ts filename="app/actions/auth.ts" switcher
import { SignupFormSchema, FormState } from '@/app/lib/definitions'

export async function signup(state: FormState, formData: FormData) {
  // Validate form fields
  const validatedFields = SignupFormSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email'),
    password: formData.get('password'),
  })

  // If any form fields are invalid, return early
  if (!validatedFields.success) {
    return {
      errors: validatedFields.error.flatten().fieldErrors,
    }
  }

  // Call the provider or db to create a user...
}
```

回到您的 `<SignupForm />`，您可以使用 React 的 `useActionState` hook 在表單提交時顯示驗證錯誤：

```tsx filename="app/ui/signup-form.tsx" switcher highlight={7,15,21,27-36}
'use client'

import { signup } from '@/app/actions/auth'
import { useActionState } from 'react'

export default function SignupForm() {
  const [state, action, pending] = useActionState(signup, undefined)

  return (
    <form action={action}>
      <div>
        <label htmlFor="name">Name</label>
        <input id="name" name="name" placeholder="Name" />
      </div>
      {state?.errors?.name && <p>{state.errors.name}</p>}

      <div>
        <label htmlFor="email">Email</label>
        <input id="email" name="email" placeholder="Email" />
      </div>
      {state?.errors?.email && <p>{state.errors.email}</p>}

      <div>
        <label htmlFor="password">Password</label>
        <input id="password" name="password" type="password" />
      </div>
      {state?.errors?.password && (
        <div>
          <p>Password must:</p>
          <ul>
            {state.errors.password.map((error) => (
              <li key={error}>- {error}</li>
            ))}
          </ul>
        </div>
      )}
      <button disabled={pending} type="submit">
        Sign Up
      </button>
    </form>
  )
}
```

以下是實施註冊和/或登入表單的步驟：

1. 用戶通過表單提交憑據。
2. 表單發送請求，由 API 路由處理。
3. 驗證成功後，流程完成，標記用戶驗證成功。
4. 如果驗證失敗，則顯示錯誤訊息。

```tsx filename="pages/login.tsx" switcher
import { FormEvent } from 'react'
import { useRouter } from 'next/router'

export default function LoginPage() {
  const router = useRouter()

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault()

    const formData = new FormData(event.currentTarget)
    const email = formData.get('email')
    const password = formData.get('password')

    const response = await fetch('/api/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    })

    if (response.ok) {
      router.push('/profile')
    } else {
      // Handle errors
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input type="email" name="email" placeholder="Email" required />
      <input type="password" name="password" placeholder="Password" required />
      <button type="submit">Login</button>
    </form>
  )
}
```

```ts filename="pages/api/auth/login.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'
import { signIn } from '@/auth'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  try {
    const { email, password } = req.body
    await signIn('credentials', { email, password })

    res.status(200).json({ success: true })
  } catch (error) {
    if (error.type === 'CredentialsSignin') {
      res.status(401).json({ error: 'Invalid credentials.' })
    } else {
      res.status(500).json({ error: 'Something went wrong.' })
    }
  }
}
```

## 會話管理

會話管理確保用戶的身份驗證狀態在請求之間得以保留。它涉及創建、存儲、刷新和刪除會話或標識符。

會話有兩種類型：

1. [**無狀態**](#stateless-sessions)：會話數據（或標識符）存儲在瀏覽器的 cookie 中。cookie 隨每個請求發送，允許在伺服器上驗證會話。此方法更簡單，但如果實現不當可能較不安全。
2. [**資料庫**](#database-sessions)：會話數據存儲在資料庫中，用戶的瀏覽器僅接收加密的會話 ID。此方法更安全，但可能更複雜且使用更多伺服器資源。

> **值得了解：** 雖然您可以使用任一方法或兩者，我們建議使用會話管理庫，例如 [iron-session](https://github.com/vvo/iron-session) 或 [Jose](https://github.com/panva/jose)。

### 無狀態會話

<AppOnly>

要創建和管理無狀態會話，需要遵循以下步驟：

1. 生成一個密鑰，用於簽署會話，並將其存儲為 [環境變數](/docs/app/guides/environment-variables)。
2. 使用會話管理庫編寫加密/解密會話數據的邏輯。
3. 使用 Next.js [`cookies`](/docs/app/api-reference/functions/cookies) API 管理 cookie。

除了上述內容外，請考慮添加功能以在用戶返回應用程式時 [更新或刷新](#updating-or-refreshing-sessions) 會話，並在用戶登出時 [刪除](#deleting-the-session) 會話。

> **值得了解：** 檢查您的 [身份驗證庫](#auth-libraries) 是否包含會話管理。

#### 1. 生成密鑰

生成簽署會話的密鑰有幾種方法。例如，您可能選擇在終端中使用 `openssl` 命令：

```bash filename="terminal"
openssl rand -base64 32
```

此命令生成一個 32 個字符的隨機字串，您可以將其作為密鑰並存儲在 [環境變數文件](/docs/app/guides/environment-variables) 中：

```bash filename=".env"
SESSION_SECRET=your_secret_key
```

然後您可以在會話管理邏輯中引用此密鑰：

```js filename="app/lib/session.js"
const secretKey = process.env.SESSION_SECRET
```

#### 2. 加密和解密會話

接下來，您可以使用首選的 [會話管理庫](#session-management-libraries) 來加密和解密會話。繼續使用之前的示例，我們將使用 [Jose](https://www.npmjs.com/package/jose)（與 [邊緣運行時](/docs/app/api-reference/edge) 兼容）和 React 的 [`server-only`](https://www.npmjs.com/package/server-only) 包，以確保您的會話管理邏輯僅在伺服器上執行。

```tsx filename="app/lib/session.ts" switcher
import 'server-only'
import { SignJWT, jwtVerify } from 'jose'
import { SessionPayload } from '@/app/lib/definitions'

const secretKey = process.env.SESSION_SECRET
const encodedKey = new TextEncoder().encode(secretKey)

export async function encrypt(payload: SessionPayload) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('7d')
    .sign(encodedKey)
}

export async function decrypt(session: string | undefined = '') {
  try {
    const { payload } = await jwtVerify(session, encodedKey, {
      algorithms: ['HS256'],
    })
    return payload
  } catch (error) {
    console.log('Failed to verify session')
  }
}
```


>
> - 負載應包含將在後續請求中使用且**最小化**的唯一用戶數據，例如用戶 ID、角色等。它不應包含個人可識別信息（如電話號碼、電子郵件地址、信用卡信息等）或敏感數據（如密碼）。

#### 3. 設置 cookie（推薦選項）

要將會話存儲在 cookie 中，請使用 Next.js [`cookies`](/docs/app/api-reference/functions/cookies) API。cookie 應在伺服器上設置，並包含推薦選項：

- **HttpOnly**：防止客戶端 JavaScript 訪問 cookie。
- **Secure**：使用 https 發送 cookie。
- **SameSite**：指定 cookie 是否可以跨站請求發送。
- **Max-Age or Expires**：經過一定時間後刪除 cookie。
- **Path**：定義 cookie 的 URL 路徑。

請參閱 [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies) 以獲取每個選項的更多信息。


#### 更新（或刷新）會話

您也可以延長會話的過期時間。這對於在用戶再次訪問應用程式後保持登入狀態很有用。例如：

```ts filename="app/lib/session.ts" switcher
  const cookieStore = await cookies()
  cookieStore.set('session', session, {
    httpOnly: true,
    secure: true,
    expires: expires,

    path: '/',
  })
}
```

### 資料庫會話

要創建和管理資料庫會話，您需要遵循以下步驟：

1. 在資料庫中創建一個表以存儲會話和數據（或檢查您的身份驗證庫是否處理此問題）。
2. 實現插入、更新和刪除會話的功能
3. 在存儲到用戶的瀏覽器之前加密會話 ID，並確保資料庫和 cookie 保持同步（這可選，但建議對於 [中介層](#optimistic-checks-with-middleware-optional) 中的樂觀身份驗證檢查使用）。

## 權限管理

一旦用戶經過身份驗證且會話已創建，您可以實施權限管理以控制用戶可以在您的應用程式中訪問和執行什麼內容。

主要有兩種權限管理檢查類型：

1. **樂觀檢查 (Optimistic)**：使用 cookie 中存儲的會話數據檢查用戶是否有權訪問路由或執行操作。這些檢查有助於快速操作，例如顯示/隱藏 UI 元素或根據權限或角色重定向用戶。
2. **安全檢查 (Secure)**：使用資料庫中存儲的會話數據檢查用戶是否有權訪問路由或執行操作。這些檢查更安全，並用於需要訪問敏感數據或操作的操作。

對於兩種情況，我們建議：

- 創建 [數據訪問層 (DAL)](#creating-a-data-access-layer-dal) 集中您的權限管理邏輯
- 使用 [數據傳輸物件 (DTO)](#using-data-transfer-objects-dto) 僅返回必要的數據
- 可選使用 [中介層 (Middleware)](#optimistic-checks-with-middleware-optional) 執行樂觀檢查。

### 使用中介層進行樂觀檢查（可選）

在某些情況下，您可能需要使用 [中介層](/docs/app/api-reference/file-conventions/middleware) 並根據權限重定向用戶：

- 執行樂觀檢查。由於中介層在每個路由上運行，這是集中重定向邏輯和預過濾未授權用戶的好方法。
- 保護在用戶之間共享數據的靜態路由（例如，付費牆後的內容）。

但是，由於中介層在每個路由上運行，包括 [預取](/docs/app/getting-started/linking-and-navigating#prefetching) 路由，因此僅從 cookie 讀取會話（樂觀檢查）並避免資料庫檢查以防止性能問題很重要。

例如：

```tsx filename="middleware.ts" switcher
import { NextRequest, NextResponse } from 'next/server'
import { decrypt } from '@/app/lib/session'
import { cookies } from 'next/headers'

// 1. Specify protected and public routes
const protectedRoutes = ['/dashboard']
const publicRoutes = ['/login', '/signup', '/']

export default async function middleware(req: NextRequest) {
  // 2. Check if the current route is protected or public
  const path = req.nextUrl.pathname
  const isProtectedRoute = protectedRoutes.includes(path)
  const isPublicRoute = publicRoutes.includes(path)

  // 3. Decrypt the session from the cookie
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  // 4. Redirect to /login if the user is not authenticated
  if (isProtectedRoute && !session?.userId) {
    return NextResponse.redirect(new URL('/login', req.nextUrl))
  }

  // 5. Redirect to /dashboard if the user is authenticated
  if (
    isPublicRoute &&
    session?.userId &&
    !req.nextUrl.pathname.startsWith('/dashboard')
  ) {
    return NextResponse.redirect(new URL('/dashboard', req.nextUrl))
  }

  return NextResponse.next()
}

// Routes Middleware should not run on
export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

雖然中介層對於初步檢查可能很有用，但它不應該是保護數據的唯一防線。大多數安全檢查應盡可能接近數據源執行，更多信息請參閱 [數據訪問層](#creating-a-data-access-layer-dal)。

> **提示：**
>
> - 在中介層中，您還可以通過 `req.cookies.get('session').value` 讀取 cookie。
> - 中介層使用 [邊緣運行時](/docs/app/api-reference/edge)，檢查您的身份驗證庫和會話管理库是否兼容。
> - 您可以在中介層中使用 `matcher` 屬性指定中介層應運行的路由。不過，對於身份驗證，建議中介層在所有路由上運行。

### 創建數據訪問層 (DAL)

我們建議創建 DAL 以集中您的數據請求和權限管理邏輯。

DAL 應包含一個函數，在用戶與您的應用程式互動時驗證用戶的會話。至少，該函數應檢查會話是否有效，然後重定向或返回進行進一步請求所需的用戶信息。

例如，為您的 DAL 創建一個單獨的文件，其中包含 `verifySession()` 函數。然後使用 React 的 [緩存](https://react.dev/reference/react/cache) API 在 React 渲染遍歷期間緩存函數的返回值：

```tsx filename="app/lib/dal.ts" switcher
import 'server-only'

import { cookies } from 'next/headers'
import { decrypt } from '@/app/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  if (!session?.userId) {
    redirect('/login')
  }

  return { isAuth: true, userId: session.userId }
})
```

```js filename="app/lib/dal.js" switcher
import 'server-only'

import { decrypt } from '@/app/lib/session'
```

然後您可以在數據請求、伺服器操作、路由處理器中調用 `verifySession()` 函數：

```tsx filename="app/lib/dal.ts" switcher
export const getUser = cache(async () => {
  const session = await verifySession()
  if (!session) return null

  try {
    const data = await db.query.users.findMany({

      // Explicitly return the columns you need rather than the whole user object
      columns: {
        id: true,
        name: true,
        email: true,
      },
    })

    const user = data[0]

    return user
  } catch (error) {
    console.log('Failed to fetch user')
    return null
  }
})
```

> - DAL 可用於保護請求時獲取的數據。不過，對於在用戶之間共享靜態數據的靜態路由，數據將在構建時獲取，而非請求時獲取。使用 [中介層](#optimistic-checks-with-middleware-optional) 來保護靜態路由。
> - 對於安全檢查，您可以通過將會話 ID 與您的資料庫進行比較來檢查會話是否有效。使用 React 的 [緩存](https://react.dev/reference/react/cache) 函數可避免在一次渲染遍歷中對資料庫進行不必要重複請求。
> - 您可能希望在運行方法的 JavaScript 類中合併相關數據請求，並在執行任何方法之前運行 `verifySession()`。

### 使用數據傳輸物件 (DTO)

在检索數據時，建議您僅返回應用程式中所需使用的數據，而非整個對象。例如，如果您正在获取用戶數據，您可能只返回用戶的 ID 和姓名，而非整個用戶對象（可能包含密碼、電話號碼等）。

不過，如果您無法控制返回的數據結構，或者在團隊中希望避免將整個對象傳遞給客戶端，您可以使用策略，例如指定哪些欄位可安全向客戶端公開。

```tsx filename="app/lib/dto.ts" switcher
import 'server-only'
import { getUser } from '@/app/lib/dal'

function canSeeUsername(viewer: User) {
  return true
}

function canSeePhoneNumber(viewer: User, team: string) {
  return viewer.isAdmin || team === viewer.team
}

export async function getProfileDTO(slug: string) {
  const data = await db.query.users.findMany({
    where: eq(users.slug, slug),
    // Return specific columns here
  })
  const user = data[0]

  const currentUser = await getUser(user.id)

  // Or return only what's specific to the query here
  return {
    username: canSeeUsername(currentUser) ? user.username : null,
    phonenumber: canSeePhoneNumber(currentUser, user.team)
      ? user.phonenumber
      : null,
  }
}
```

### 伺服器元件

在 [伺服器元件](/docs/app/getting-started/server-and-client-components) 中進行的身份驗證檢查對於基於角色的訪問很有用。例如，根據用戶的角色來條件渲染組件：

```tsx filename="app/dashboard/page.tsx" switcher
import { verifySession } from '@/app/lib/dal'

export default async function Dashboard() {
  const session = await verifySession()
  const userRole = session?.user?.role // Assuming 'role' is part of the session object

  if (userRole === 'admin') {
    return <AdminDashboard />
  } else if (userRole === 'user') {
    return <UserDashboard />
  } else {
    redirect('/login')
  }
}
```

在示例中，我們使用 DAL 中的 `verifySession()` 函數檢查 'admin', 'user' 和未授權的角色。這種模式確保每個用戶僅與其角色適當的組件互動。

### 佈局和身份驗證檢查

由於 [部分渲染](/docs/app/getting-started/linking-and-navigating#client-side-transitions)，在 [佈局](/docs/app/api-reference/file-conventions/layout) 中進行檢查時要謹慎，因為這些在導航時不會重新渲染，這意味著用戶會話不會在每次路由變更時被檢查。

相反，您應該在接近數據源或將條件渲染組件的位置進行檢查。

例如，考慮一個共享佈局，它獲取用戶數據並在導航中顯示用戶頭像。與在佈局中進行身份驗證檢查相反，您應該在佈局中獲取用戶數據 (`getUser()`) 並在您的 DAL 中進行身份驗證檢查。

這保證了 wherever `getUser()` 在您的應用程式中被調用，身份驗證檢查都會執行，並防止開發人員忘記檢查用戶是否有權訪問數據。

```tsx filename="app/layout.tsx" switcher
export default async function Layout({
  children,
}: {
  children: React.ReactNode;
}) {
  const user = await getUser();

  return (
    // ...
  )
}
```

```jsx filename="app/layout.js" switcher
export default async function Layout({ children }) {
  const user = await getUser();

  return (
    // ...
  )
}
```

```ts filename="app/lib/dal.ts" switcher
export const getUser = cache(async () => {
```

```js filename="app/lib/dal.js" switcher
  // Get user ID from session and fetch data
})
```

> **值得了解：**
>
> - SPA 中的一個常見模式是如果用戶未獲授權，則在佈局或頂級組件中 `return null`。此模式並**不建議**在 Next.js 應用程式中使用，因為它們擁有多個入口點，這將無法防止嵌套路由段和伺服器操作被訪問。

### 路由處理器

以與公共 API 端點相同的安全考慮來處理 [路由處理器](/docs/app/api-reference/file-conventions/route)，並驗證用戶是否有權訪問路由處理器。

例如：

```ts filename="app/api/route.ts" switcher
import { verifySession } from '@/app/lib/dal'

export async function GET() {
  // User authentication and role verification
  const session = await verifySession()

  // Check if the user is authenticated
  if (!session) {
    // User is not authenticated
    return new Response(null, { status: 401 })
  }

  // Check if the user has the 'admin' role
  if (session.user.role !== 'admin') {
    // User is authenticated but does not have the right permissions
    return new Response(null, { status: 403 })
  }

  // Continue for authorized users
}
```

#### 保護 API 路由

Next.js 中的 API 路由對於處理服務器端邏輯和數據管理至關重要。保護這些路由至關重要，以確保只有授權的用戶才能訪問特定功能。這通常涉及驗證用戶的身份驗證地位和基於角色的權限。

以下是保護 API 路由的示例：

```ts filename="pages/api/route.ts" switcher
import { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  const session = await getSession(req)

  // Check if the user is authenticated
  if (!session) {
    res.status(401).json({
      error: 'User is not authenticated',
    })
    return
  }

  // Check if the user has the 'admin' role
  if (session.user.role !== 'admin') {
    res.status(401).json({
      error: 'Unauthorized access: User does not have admin privileges.',
    })
    return
  }

  // Proceed with the route for authorized users
  // ... implementation of the API Route
}
```
