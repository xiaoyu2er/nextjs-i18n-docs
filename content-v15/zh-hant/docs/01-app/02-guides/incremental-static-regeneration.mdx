增量靜態再生 (ISR) 允許您：

- 無需重建整個網站即可更新靜態內容
- 為大多數請求提供預先渲染的靜態頁面，從而減輕伺服器的負載
- 確保適當的 `cache-control` 標頭會自動添加到頁面
- 無需長 `next build` 時間即可處理大量內容頁面

這是一個範例：

<AppOnly>

```tsx filename="app/blog/[id]/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

// Next.js will invalidate the cache when a
// request comes in, at most once every 60 seconds.
export const revalidate = 60

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```


範例運作方式說明：

1. 在 `next build` 期間，生成所有已知的部落格文章
2. 對這些頁面（例如 `/blog/1`）的所有請求都會被快取且立即可用
3. 60 秒後，下一個請求仍會返回已快取的（現已過期的）頁面
4. 快取無效，新版本頁面開始在背景生成
5. 成功生成後，下一個請求將返回更新的頁面，並將該頁面快取以供後續請求使用
6. 如果請求 `/blog/26` 且存在，頁面將按需生成。此行為可透過使用不同之 [dynamicParams](/docs/app/api-reference/file-conventions/route-segment-config#dynamicparams) 值來變更。然而，如果文章不存在，則會返回 404。

</AppOnly>

```tsx filename="pages/blog/[id].tsx" switcher
    props: { post },

    // request comes in, at most once every 60 seconds.
    revalidate: 60,

export default function Page({ post }: Props) {

}
```

## 範例

<AppOnly>

### 基於時間的重新驗證

此處會擷取並顯示 /blog 上的文章列表。經過一小時後，下一位訪客仍會立即收到快取的（過期的）版本以獲得快速回應。同時，Next.js 會在背景觸發新版本的重建。一旦新版本成功生成，它將替換快取版本，後續訪客將收到更新的內容。

```tsx filename="app/blog/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

export const revalidate = 3600 // invalidate every hour

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

```jsx filename="app/blog/page.js" switcher
export const revalidate = 3600 // invalidate every hour

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

建議設定較長的重新驗證時間。例如，1 小時而非 1 秒。如果需要更高精確度，請考慮使用按需重新驗證。如果需要即時資料，請考慮切換至 [動態渲染](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。

### 使用 `revalidatePath` 進行按需重新驗證

若要使用更精確的重新驗證方法，請使用 `revalidatePath` 函式來按需無效快取頁面。

例如，此伺服器動作將在新增新文章後被呼叫。無論您如何在伺服器元件中擷取資料，無論是使用 `fetch` 或連接資料庫，這都將無效整個路由的快取。對該路由的下一次請求將會觸發重建並提供新鮮資料，這些資料隨後將被快取以供後續請求使用。

> **注意事項：** `revalidatePath` 會無效快取項目，但重建會在下一個請求時發生。如果您希望立即熱情地重建快取項目，而不必等待下一個請求，則可改用 Pages Router [`res.revalidate`](docs/pages/guides/incremental-static-regeneration#on-demand-validation-with-resrevalidate) 方法。我們正在努力新增新方法，以提供 App Router 的熱情重建功能。

```ts filename="app/actions.ts" switcher
'use server'

import { revalidatePath } from 'next/cache'

export async function createPost() {
  // Invalidate the cache for the /posts route
  revalidatePath('/posts')
}
```

```js filename="app/actions.js" switcher
'use server'

import { revalidatePath } from 'next/cache'

export async function createPost() {
  // Invalidate the cache for the /posts route
  revalidatePath('/posts')
}
```

[檢視演示](https://on-demand-isr.vercel.app) 和 [檢視原始碼](https://github.com/vercel/on-demand-isr)。

### 使用 `revalidateTag` 進行按需重新驗證

對於大多數使用情況，請優先重新驗證整個路徑。如果您需要更細粒度的控制，可使用 `revalidateTag` 函式。例如，您可以標記個別 `fetch` 呼叫：

```tsx filename="app/blog/page.tsx" switcher
export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog', {
    next: { tags: ['posts'] },
  })
  const posts = await data.json()
  // ...
}
```

```jsx filename="app/blog/page.js" switcher
export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog', {
    next: { tags: ['posts'] },
  })
  const posts = await data.json()
  // ...
}
```

如果您正在使用 ORM 或連接資料庫，可使用 `unstable_cache`：

```tsx filename="app/blog/page.tsx" switcher
import { unstable_cache } from 'next/cache'
import { db, posts } from '@/lib/db'

const getCachedPosts = unstable_cache(
  async () => {
    return await db.select().from(posts)
  },
  ['posts'],
  { revalidate: 3600, tags: ['posts'] }
)

export default async function Page() {
  const posts = getCachedPosts()
  // ...
}
```

```jsx filename="app/blog/page.js" switcher

```

```ts filename="app/actions.ts" switcher
import { revalidateTag } from 'next/cache'

  revalidateTag('posts')
}
```

### 使用 `res.revalidate()` 進行按需驗證

若要使用更精確的重新驗證方法，請使用 `res.revalidate` 從 API 路由器按需生成新頁面。

例如，此 API 路由可在 `/api/revalidate?secret=<token>` 處被呼叫，以重新驗證特定的部落格文章。建立一個僅為您的 Next.js 應用程式所知的秘密金鑰。此秘密將用於防止未經授權存取重新驗證 API 路由。

```ts filename="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // Check for secret to confirm this is a valid request
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // This should be the actual path not a rewritten path
    // e.g. for "/posts/[id]" this should be "/posts/1"
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // If there was an error, Next.js will continue
    // to show the last successfully generated page
    return res.status(500).send('Error revalidating')
  }
}
```

```js filename="pages/api/revalidate.js" switcher

```

如果您正在使用按需重新驗證，則不需要在 `getStaticProps` 內部指定 `revalidate` 時間。Next.js 將使用 `false` (不重新驗證) 的預設值，並且僅在呼叫 `res.revalidate()` 時按需重新驗證頁面。

### 處理未捕捉的例外狀況

<AppOnly>

如果在嘗試重新驗證資料時發生錯誤，則會繼續從快取服務最後成功產生的資料。在後續的下一次請求中，Next.js 將重試重新驗證資料。[了解有關錯誤處理的更多資訊](/docs/app/getting-started/error-handling)。

</AppOnly>

<PagesOnly>

如果在處理背景重建期間 `getStaticProps` 內部發生錯誤，或您手動拋出錯誤，則最後成功產生的頁面將繼續顯示。在後續的下一次請求中，Next.js 將重試呼叫 `getStaticProps`。

```tsx filename="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // If this request throws an uncaught error, Next.js will
  // not invalidate the currently shown page and
  // retry getStaticProps on the next request.
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // If there is a server error, you might want to
    // throw an error instead of returning so that the cache is not updated
    // until the next successful request.
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.js will invalidate the cache when a
    // request comes in, at most once every 60 seconds.
    revalidate: 60,
  }
}
```

## 疑難排解

### 在本地開發中除錯快取資料

如果您正在使用 `fetch` API，則可以新增其他記錄以了解哪些請求已快取或未快取。[了解有關 `logging` 選項的更多資訊](/docs/app/api-reference/config/next-config-js/logging)。

```jsx filename="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 驗證生產環境中的正確行為

為了驗證您的頁面在生產環境中快取和重新驗證是否正確，您可以透過執行 `next build`，然後執行 `next start` 來在本地測試以執行生產環境 Next.js 伺服器。

這將允許您測試生產環境中運作的 ISR 行為。對於進一步除錯，請將下列環境變數新增到您的 `.env` 檔案中：

```bash filename=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

這將使 Next.js 伺服器主控台記錄 ISR 快取命中和未命中。您可以檢查輸出以了解在 `next build` 期間生成哪些頁面，以及在對路徑進行按需存取時頁面如何更新。

## 注意事項

<AppOnly>

- 僅在使用 Node.js 執行階段（預設值）時支援 ISR。
- 建立靜態匯出時不支援 ISR。
- 如果在靜態渲染路由中有多個 `fetch` 請求，且每個請求都有不同的 `revalidate` 頻率，則 ISR 將使用最短時間。然而，這些重新驗證頻率仍將為 [數據快取](/docs/app/guides/caching#data-cache) 所尊重。
- 如果在路由上使用的任何 `fetch` 請求的 `revalidate` 時間為 `0` 或明確設定 `no-store`，該路由將 [動態渲染](/docs/app/getting-started/partial-prerendering#dynamic-rendering)。
- Middleware 不會執行於按需 ISR 請求，意味著 Middleware 中的任何路徑重新編寫或邏輯將不會套用。請務必要重新驗證精確的路徑。例如 `/post/1` 而非重新編寫的 `/post-1`。

</AppOnly>

<PagesOnly>

- 僅在使用 Node.js 執行階段（預設值）時支援 ISR。
- 建立靜態匯出時不支援 ISR。
