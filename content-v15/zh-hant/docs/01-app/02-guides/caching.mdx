Next.js 透過快取渲染工作和資料請求來提升你的應用程式效能並降低成本。本頁深入介紹了 Next.js 的快取機制、可用於設定它們的 API，以及它們如何互相交互。

> **值得了解**: 本頁協助你理解 Next.js 在底層如何運作，但對於使用 Next.js 進行生產工作而言，這**並非**必要知識。大多數 Next.js 的快取啟發式規則取決於你的 API 用法，並有預設值以提供零或少量配置的最優效能。如果你想直接跳轉到範例，請 [開始於此](/docs/app/getting-started/fetching-data)。

## 總覽

以下是不同快取機制及其用途的層級概述：

| 機制 | 說明 | 位置 | 目的 | 持續時間 |
| :--- | :--- | :--- | :--- | :--- |
| [請求記憶體化 (Request Memoization)](#request-memoization) | 函數的回傳值 | 伺服器 | 重用在 React 元件樹中的資料 | 每個請求的生命週期 |
| [資料快取 (Data Cache)](#data-cache) | 資料 | 伺服器 | 跨使用者請求和部署儲存資料 | 持久 (可重新驗證) |
| [完整路由快取 (Full Route Cache)](#full-route-cache) | HTML 和 RSC 負載 | 伺服器 | 降低渲染成本並提升效能 | 持久 (可重新驗證) |
| [路由快取 (Router Cache)](#client-side-router-cache) | RSC 負載 | 客戶端 | 減少導航時的伺服器請求 | 使用者工作階段或基於時間 |

預設情況下，Next.js 會快取盡可能多的內容以提升效能並降低成本。這意味著路徑會以**靜態渲染 (Statically Rendered)**，資料請求會被**快取 (Cached)**，除非你選擇退出。下表圖示預設快取行為：路徑在建構時如何靜態渲染，以及在首次訪問靜態路由時。

快取行為會根據路徑是以靜態或動態方式渲染、資料是否快取，以及請求是初次訪問還是後續導航而有所不同。根據你的使用案例，你可以針對個別路徑和資料請求設定快取行為。

## 請求記憶體化 (Request Memoization)

Next.js 擴充了 [`fetch` API](#fetch)，會自動 **記憶體化 (Memoize)**具有相同 URL 和選項的請求。這意味著你可以在 React 元件樹中的多個位置呼叫相同的資料 fetch 函數，同時只執行一次。

<Image
  alt="去重複的 Fetch 請求"
  srcLight="/docs/light/deduplicated-fetch-requests.png"
  srcDark="/docs/dark/deduplicated-fetch-requests.png"
  width="1600"
  height="857"
/>

例如，如果你需要在整個路徑中（例如在佈局、頁面和多個元件中）使用相同的資料，則無需在樹頂部取得資料，並在元件之間傳遞 props。相反，你可以在需要它們的元件中取得資料，而無需擔心在網路上多次取得相同資料的效能影響。

```tsx filename="app/example.tsx" switcher
async function getItem() {
  // The `fetch` function is automatically memoized and the result
  // is cached
  const res = await fetch('https://.../item/1')
  return res.json()
}

// This function is called twice, but only executed the first time
const item = await getItem() // cache MISS

// The second call could be anywhere in your route
const item = await getItem() // cache HIT
```

```jsx filename="app/example.js" switcher
async function getItem() {
  // The `fetch` function is automatically memoized and the result
  // is cached
  const res = await fetch('https://.../item/1')
  return res.json()
}

// This function is called twice, but only executed the first time
const item = await getItem() // cache MISS

// The second call could be anywhere in your route
const item = await getItem() // cache HIT
```

**請求記憶體化如何運作**

<Image
  alt="圖示顯示在 React 渲染期間 fetch 記憶體化如何運作。"
  srcLight="/docs/light/request-memoization.png"
  srcDark="/docs/dark/request-memoization.png"
  width="1600"
  height="742"
/>

- 在渲染路徑時，首次呼叫特定請求時，其結果將不在記憶體中，並且會是快取 `MISS`（未命中）。
- 因此，函數將會執行，資料將從外部來源取得，結果將儲存於記憶體中。
- 在同一批次渲染期間的後續函數呼叫將是快取 `HIT`（命中），並將從記憶體回傳資料而無需執行函數。
- 一旦路徑渲染完成且渲染過程結束，記憶體將被「重置」，所有請求記憶體化條目都會被清除。

> **值得了解**:
>
> - 請求記憶體化 (Request Memoization) 是 React 功能，而非 Next.js 功能。此處包含它是為了展示它如何與其他快取機制交互。
> - 記憶體化僅適用於 `fetch` 請求中的 `GET` 方法。
> - 記憶體化僅適用於 React 元件樹，這意味著：
>   - 它適用於 `generateMetadata`、`generateStaticParams`、佈局、頁面和其他伺服器元件中的 `fetch` 請求。
>   - 它不適用於路由處理器 (Route Handlers) 中的 `fetch` 請求，因為它們不是 React 元件樹的一部分。
> - 在 `fetch` 不適用的情況下（例如某些資料庫客戶端、CMS 客戶端或 GraphQL 客戶端），你可以使用 [React `cache` 函數](#react-cache-function) 來記憶體化函數。

### 持續時間

快取持續整個伺服器請求的生命週期，直到 React 元件樹完成渲染。

### 重新驗證

由於記憶體化不會跨伺服器請求共用，並且僅適用於渲染期間，因此無需重新驗證它。

### 退出

記憶體化僅適用於 `fetch` 請求中的 `GET` 方法，其他方法如 `POST` 和 `DELETE` 則不被記憶體化。此預設行為是 React 優化，我們不建議退出它。

若要管理個別請求，你可以使用 [`AbortController`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController) 中的 [`signal`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController/signal) 屬性。

```js filename="app/example.js"
const { signal } = new AbortController()
fetch(url, { signal })
```

## 資料快取 (Data Cache)

Next.js 內建資料快取 (Data Cache)，會跨多個**伺服器請求**和**部署**持久化資料取得的結果。這是因為 Next.js 擴充了原生 `fetch` API，允許伺服器的每個請求設定其自己的持久化快取語義。

> **值得了解**: 在瀏覽器中，`fetch` 的 `cache` 選項表示請求將如何與瀏覽器的 HTTP 快取交互，在 Next.js 中，`cache` 選項表示伺服器端請求將如何與伺服器的資料快取交互。

你可以使用 `fetch` 的 [`cache`](#fetch-optionscache) 和 [`next.revalidate`](#fetch-optionsnextrevalidate) 選項來設定快取行為。

在開發模式下，`fetch` 資料會 [為熱模組替換 (HMR)](/docs/app/api-reference/functions/fetch#fetch-default-auto-no-store-and-cache-no-store-not-showing-fresh-data-in-development) 重複使用，並且快取選項對於 [硬重新整理](/docs/app/api-reference/functions/fetch#hard-refresh-and-caching-in-development) 會受到忽略。

**資料快取如何運作**

<Image
  alt="圖示顯示快取和未快取的 Fetch 請求如何與資料快取交互。快取請求儲存於資料快取中，未快取且記憶體化的請求則從資料來源取得、不儲存於資料快取，並記憶體化。"
  srcLight="/docs/light/data-cache.png"
  srcDark="/docs/dark/data-cache.png"
  width="1600"
  height="661"
/>

- 在渲染期間，首次呼叫具有 `'force-cache'` 選項的 `fetch` 請求時，Next.js 會在資料快取中檢查快取的回應。
- 如果找到快取的回應，將立即回傳並 [記憶體化](#request-memoization)。
- 如果未找到快取的回應，則向資料來源發出請求，結果將儲存於資料快取，並記憶體化。
- 對於未快取的資料（例如未定義 `cache` 選項或使用 `{ cache: 'no-store' }`），結果將總是從資料來源取得，並記憶體化。
- 無論資料是否快取，請求都會始終被記憶體化，以避免在 React 渲染期間發出相同資料的重複請求。

> **相關術語**:
>
> 你可能看到**自動靜態優化 (Automatic Static Optimization)**、**靜態網站生成 (Static Site Generation)** 或 **靜態渲染 (Static Rendering)** 這些術語被交替使用來指代建置時渲染和快取應用程式路徑的過程。

### 4. 客戶端上的 Next.js 快取 (路由快取)

React 伺服器元件負載儲存於客戶端 [路由快取 (Router Cache)](#client-side-router-cache) – 一個分開的記憶體內快取，按個別路徑區塊分。此路由快取用於透過儲存已訪問的路徑並預取未來路徑來改善導航體驗。

```js
// Revalidate at most every hour
fetch('https://...', { next: { revalidate: 3600 } })
```

在後續導航或預取期間，Next.js 會檢查 React 伺服器元件負載是否儲存於路由快取中。如果是，它將跳過向伺服器發出新請求。

如果路徑區塊不在快取中，Next.js 將從伺服器取得新 React 伺服器元件負載，並填充客戶端上的路由快取。

路徑在建構時是否快取，取決於它是以靜態或動態方式渲染。靜態路徑預設會快取，而動態路徑則在請求時渲染，且不會快取。

- **[重新驗證資料 (Revalidating Data)](/docs/app/guides/caching#revalidating)**：重新驗證 [資料快取 (Data Cache)](#data-cache)，將進而無效化路由快取，透過在伺服器上重新渲染元件並快取新的渲染輸出。
- **重新部署 (Redeploying)**：與跨部署持久化的資料快取不同，在新增部署期間會清除完整路由快取。

## 客戶端路由快取 (Client-side Router Cache)

Next.js 有一個記憶體內客戶端路由器快取，儲存路徑區塊的 RSC 負載，按佈局、載入狀態和頁面區分。

當使用者在路徑之間導航時，Next.js 會快取已訪問的路徑區塊，並 [預取](/docs/app/getting-started/linking-and-navigating#prefetching) 使用者可能導航至的路徑。這導致立即回/正向導航，沒有導航之間的全頁面重新整理，並保留共用佈景中的瀏覽器狀態和 React 狀態。

使用路由快取 (Router Cache)：

- **佈局 (Layouts)** 會被快取並在導航時重用 ([部分渲染](/docs/app/getting-started/linking-and-navigating#client-side-transitions))。
- **載入狀態 (Loading states)** 會在導航時快取並重用，用於 [立即載入狀態](/docs/app/api-reference/file-conventions/loading#instant-loading-states)。
- **頁面 (Pages)** 預設不被快取，但在瀏覽器倒退和前進導航期間會被重用。你可以透過使用實驗性 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定選項為頁面區塊啟用快取。

### 退出

快取儲存於瀏覽器的臨時記憶體中。兩個因素決定路由器快取持續多久：

```js
let data = await fetch('https://api.vercel.app/blog', { cache: 'no-store' })
```

### 無效化

> **值得了解**: 你也可以透過將 `<Link>` 元件的 `prefetch` prop 設定為 `false` 來停用 [預取](/docs/app/getting-started/linking-and-navigating#prefetching)。

在設定不同快取機制時，了解它們如何互相交互很重要：

下表提供了不同 Next.js API 如何影響快取的概述：

### `<Link>`

預設情況下，`<Link>` 元件會從完整路由快取中自動預取路徑，並將 React 伺服器元件負載新增至路由快取。

若要停用預取，你可以將 `prefetch` prop 設定為 `false`。但這不會永久跳過快取，當使用者訪問路徑時，路徑區塊仍會被快取在客戶端。

1. 使用 `fetch` 或 [`unstable_cache`](/docs/app/api-reference/functions/unstable_cache) 時，你有選項為快取條目標籤設定一或多個標籤。
2. 接著，你可以呼叫 `revalidateTag` 以清除關聯於該標籤的快取條目。

例如，你可以在取得資料時設定標籤：

> **`revalidatePath`** 與 **`router.refresh`**:
>
> 呼叫 `router.refresh` 將清除路由快取，並在伺服器上重新渲染路徑區塊而無需無效化資料快取或完整路由快取。
>
> 差別在於 `revalidatePath` 清除資料快取和完整路由快取，而 `router.refresh()` 不改變資料快取和完整路由快取，因為它是一個客戶端 API。

### 動態 API

`cookies` 和 `headers` 等動態 API，以及「頁面 (Pages)」中的 `searchParams` prop，取決於執行時 incoming 請求資訊。使用它們將使路徑退出完整路由快取，換句話說，將動態渲染路徑。

在伺服器動作中使用 `cookies.set` 或 `cookies.delete` 會無效化路由快取，以防止使用快取的路徑變過期（例如反映身份驗證變更）。

### 區塊設定選項 (Segment Config Options)

路由區塊設定選項可用於覆蓋路徑區塊預設值，或當你無法使用 `fetch` API 時（例如資料庫客戶端或 3rd 方函式庫）。

- `const dynamic = 'force-dynamic'`

### `generateStaticParams`

對於 [動態區塊 (Dynamic Segments)](/docs/app/api-reference/file-conventions/dynamic-routes)（例如 `app/blog/[slug]/page.js`），`generateStaticParams` 提供的路徑在建構時儲存於完整路由快取中。在請求時，Next.js 還會在首次訪問時快取在建構時未知的路徑。

### React `cache` 函數

React `cache` 函數允許你記憶體化函數的回傳值，允許你多次呼叫同一函數，但只執行一次。

`GET` 或 `HEAD` 方法的 `fetch` 請求會自動進行記憶體化，因此你不需要將它包裝在 React `cache` 中。然而，對於其他 `fetch` 方法，或使用不 inherently 記憶體化請求的資料取得函式庫（如某些資料庫、CMS 或 GraphQL 客戶端），你可以使用 `cache` 來手動記憶體化資料請求。

### 靜態與動態渲染

建構時路徑是否被快取取決於它是靜態還是動態渲染。靜態路徑預設會被快取，而動態路徑則在請求時渲染且不會被快取。

### 持續時間

預設情況下，完整路由快取是持久的。這意味著渲染輸出會在用戶請求間被快取。

### 無效化

有兩種方法可以無效化完整路由快取：

- [重新驗證資料](/docs/app/guides/caching#revalidating)：重新驗證 [資料快取](#data-cache) 將進而無效化路由快取，透過在伺服器上重新渲染元件並快取新的渲染輸出。
- 重新部署：與跨部署持久化的資料快取不同，完整路由快取在新部署期間會被清除。

### 退出

你可以選擇退出完整路由快取，換句話說，透過以下方式為每個請求動態渲染元件：

- [使用動態 API](#dynamic-apis)：這將退出完整路由快取並請求時動態渲染它。資料快取仍然可以使用。
- [使用 `dynamic = 'force-dynamic'` 或 `revalidate = 0` 路由區塊設定選項]：這將跳過完整路由快取和資料快取。意味著元件將在每次請求時在伺服器上渲染並取得資料。路由快取仍然適用，因為它是客戶端快取。
- [退出資料快取](#data-cache)：如果路徑有一個未快取的 `fetch` 請求，這將退出完整路由快取。特定 `fetch` 請求的資料將在每次請求時取得。明確啟用快取的其它 `fetch` 請求仍將快取在資料快取中。這允許快取和未快取資料的混合。

## 客戶端路由快取

Next.js 具備一個記憶體中的客戶端路由快取，儲存路由區塊的 RSC 負載，依佈局、載入狀態和頁面區分。

當使用者在路徑之間導航時，Next.js 會快取已訪問的路徑區塊並 [預取](/docs/app/getting-started/linking-and-navigating#prefetching) 使用者可能導航至的路徑。這導致立即的返回/正向導航，導航之間無全頁面重新整理，並保留共用佈局中的瀏覽器狀態和 React 狀態。

使用路由快取：

- **佈局** 會被快取並在導航時重用 ([部分渲染](/docs/app/getting-started/linking-and-navigating#client-side-transitions))。
- **載入狀態** 會在導航時快取並重用，用於 [即時導航](/docs/app/api-reference/file-conventions/loading#instant-loading-states)。
- **頁面** 預設不被快取，但在瀏覽器倒退和正向導航期間會被重用。你可以透過使用實驗性 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定選項為頁面區塊啟用快取。

<!-- TODO: Update diagram to match v15 behavior -->

> **值得了解**：此快取特定應用於 Next.js 和伺服器元件，與瀏覽器的 [bfcache](https://web.dev/bfcache/) 不同，儘管它有類似的結果。

### 持續時間

快取儲存於瀏覽器的臨時記憶體中。兩個因素決定路由器快取持續多久：

- **工作階段 (Session)**：快取會在導航間持續。然而，它會在頁面重新整理時被清除。
- **自動無效化期間**：佈局和載入狀態的快取會在特定時間後自動無效化。持續時間取決於資源如何 [預取](/docs/app/api-reference/components/link#prefetch)，以及資源是否 [靜態生成](/docs/app/getting-started/partial-prerendering#static-rendering)：
  - **預設預取** (`prefetch={null}` 或未指定)：動態頁面不快取，靜態頁面 5 分鐘。
  - **完整預取** (`prefetch={true}` 或 `router.prefetch`)：靜態和動態頁面均為 5 分鐘。

頁面重新整理將清除**所有**快取區塊，但自動無效化期間僅影響從它被預取時開始的單獨區塊。

> **值得了解**：實驗性 [`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定選項可用於調整上述的自動無效化時間。

### 無效化

有兩種方法可以無效化路由快取：

- 在 **伺服器動作** 中：
  - 透過路徑 ([`revalidatePath`](/docs/app/api-reference/functions/revalidatePath)) 或透過快取標籤 ([`revalidateTag`](/docs/app/api-reference/functions/revalidateTag)) 按需重新驗證資料。
  - 使用 [`cookies.set`](/docs/app/api-reference/functions/cookies#setting-a-cookie) 或 [`cookies.delete`](/docs/app/api-reference/functions/cookies#deleting-cookies) 會無效化路由快取，以防止使用 cookies 的路徑變得過期（例如身份驗證）。
- 呼叫 [`router.refresh`](/docs/app/api-reference/functions/use-router) 將無效化路由快取並為當前路徑向伺服器發出新請求。

### 退出

自 Next.js 15 起，頁面區塊預設已選擇退出。

> **值得了解**：你也可以將 `<Link>` 元件的 `prefetch` prop 設定為 `false` 來停用 [預取](/docs/app/getting-started/linking-and-navigating#prefetching)。

## 快取交互

設定不同快取機制時，了解它們如何互相交互很重要：

### 資料快取與完整路由快取

- 重新驗證或退出資料快取 **會** 無效化完整路由快取，因為渲染輸出取決於資料。
- 無效化或退出完整路由快取 **不會** 影響資料快取。你可以動態渲染同時擁有快取和未快取資料的路徑。這非常有用，當你的頁面大部分使用快取資料時，但有一些元件依賴於需要在請求時取得的資料。你可以動態渲染而無需擔心重新取得所有資料的效能影響。

### 資料快取與客戶端路由快取

- 要立即無效化資料快取和路由快取，你可以在 [伺服器動作](/docs/app/getting-started/updating-data) 中使用 [`revalidatePath`](#revalidatepath) 或 [`revalidateTag`](#fetch-optionsnexttags-and-revalidatetag)。
- 在 [路由處理器](/docs/app/api-reference/file-conventions/route) 中重新驗證資料快取 **不會** 立即無效化路由快取，因為路由處理器不與特定路徑綁定。這意味著路由快取將繼續提供先前的負載，直到硬重新整理或自動無效化期間已結束。

### `<Link>`

預設情況下，`<Link>` 元件會自動從完整路由快取中預取路徑，並將 React 伺服器元件負載新增至路由快取。

若要停用預取，你可以將 `prefetch` prop 設定為 `false`。但這不會永久跳過快取，當使用者訪問路徑時，路徑區塊仍會被快取在客戶端。

了解更多關於 [`<Link>` 元件](/docs/app/api-reference/components/link)。

### `router.prefetch`

`useRouter` hook 的 `prefetch` 選項可用於手動預取路徑。這將 React 伺服器元件負載新增至路由快取。

參閱 [`useRouter` hook](/docs/app/api-reference/functions/use-router) API 參考文件。

### `router.refresh`

`useRouter` hook 的 `refresh` 選項可用於手動重新整理路徑。這將完全清除路由快取，並為當前路徑向伺服器發出新請求。`refresh` 不會影響資料快取或完整路由快取。

渲染結果將在客戶端進行比對，同時保留 React 狀態和瀏覽器狀態。

參閱 [`useRouter` hook](/docs/app/api-reference/functions/use-router) API 參考文件。

### `fetch`

從 `fetch` 回傳的資料**不會**自動快取在資料快取中。

`fetch` 的預設快取行為（例如當未指定 `cache` 選項時）等於將 `cache` 選項設定為 `no-store`：

```js
let data = await fetch('https://api.vercel.app/blog', { cache: 'no-store' })
```

參見 [`fetch` API 參考文件](/docs/app/api-reference/functions/fetch) 以瞭解更多選項。

### `fetch options.cache`

你可以透過將 `cache` 選項設定為 `force-cache` 來讓個別 `fetch` 加入快取：

```jsx
// Opt into caching
fetch(`https://...`, { cache: 'force-cache' })
```

參見 [`fetch` API 參考文件](/docs/app/api-reference/functions/fetch) 以瞭解更多選項。

### `fetch options.next.revalidate`

你可以使用 `fetch` 的 `next.revalidate` 選項來設定個別 `fetch` 請求的重新驗證週期（以秒為單位）。這將重新驗證資料快取，進而重新驗證完整路由快取。將取得新資料，並且元件將在伺服器上重新渲染。

```jsx
// Revalidate at most after 1 hour
fetch(`https://...`, { next: { revalidate: 3600 } })
```

參見 [`fetch` API 參考文件](/docs/app/api-reference/functions/fetch) 以瞭解更多選項。

### `fetch options.next.tags` and `revalidateTag`

Next.js 擁有快取標籤系統，用於細粒度的資料快取和重新驗證。

1. 使用 `fetch` 或 [`unstable_cache`](/docs/app/api-reference/functions/unstable_cache) 時，你有選項為快取條目設定一個或多個標籤。
2. 接著，你可以呼叫 `revalidateTag` 以清除與該標籤相關的快取條目。

例如，你可以在取得資料時設定標籤：

```jsx
// Cache data with a tag
fetch(`https://...`, { next: { tags: ['a', 'b', 'c'] } })
```

接著，使用帶有標籤的 `revalidateTag` 呼叫以清除快取條目：

```jsx
// Revalidate entries with a specific tag
revalidateTag('a')
```

根據你想達成的目標，有以下兩個地方可以使用 `revalidateTag`：

1. [路由處理器](/docs/app/api-reference/file-conventions/route) - 響應第三方事件（例如 webhook）重新驗證資料。這將不會立即無效化路由快取，因為路由處理器不與特定路徑綁定。
2. [伺服器動作](/docs/app/getting-started/updating-data) - 在使用者操作後重新驗證資料（例如表單提交）。這將無效化關聯路徑的路由快取。

### `revalidatePath`

`revalidatePath` 允許你在單一操作中手動重新驗證資料，並重新渲染特定路徑下方的路由區塊。呼叫 `revalidatePath` 方法會重新驗證資料快取，進而無效化完整路由快取。

```jsx
revalidatePath('/')
```

根據你想達成的目標，有以下兩個地方可以使用 `revalidatePath`：

1. [路由處理器](/docs/app/api-reference/file-conventions/route) - 響應第三方事件（例如 webhook）重新驗證資料。
2. [伺服器動作](/docs/app/getting-started/updating-data) - 在使用者互動後重新驗證資料（例如表單提交、點擊按鈕）。

參閱 [`revalidatePath` API 參考文件](/docs/app/api-reference/functions/revalidatePath) 以瞭解更多資訊。

> **`revalidatePath`** 與 **`router.refresh`**:
>
> 呼叫 `router.refresh` 將清除路由快取，並在伺服器上重新渲染路由區塊而無需無效化資料快取或完整路由快取。
>
> 差別在於 `revalidatePath` 清除資料快取和完整路由快取，而 `router.refresh()` 不改變資料快取和完整路由快取，因為它是一個客戶端 API。

### 動態 API

`cookies` 和 `headers` 等動態 API，以及頁面中的 `searchParams` prop，取決於執行時的請求資訊。使用它們將使路徑退出完整路由快取，換句話說，將動態渲染路徑。

#### `cookies`

在伺服器動作中使用 `cookies.set` 或 `cookies.delete` 會無效化路由快取，以防止使用 cookies 的路徑變得過期（例如以反映身份驗證變更）。

參見 [`cookies`](/docs/app/api-reference/functions/cookies) API 參考文件。

### 區塊設定選項

路由區塊設定選項可用於覆蓋路由區塊預設值，或當你無法使用 `fetch` API 時（例如資料庫客戶端或第三方函式庫）。

以下路由區塊設定選項將退出完整路由快取：

- `const dynamic = 'force-dynamic'`

此設定選項將使所有 `fetch` 退出資料快取（即 `no-store`）：

- `const fetchCache = 'default-no-store'`

參見 [`fetchCache`](/docs/app/api-reference/file-conventions/route-segment-config#fetchcache) 以查看更多進階選項。

參見 [路由區塊設定](/docs/app/api-reference/file-conventions/route-segment-config) 文件以查看更多選項。

### `generateStaticParams`

對於 [動態區塊](/docs/app/api-reference/file-conventions/dynamic-routes)（例如 `app/blog/[slug]/page.js`），`generateStaticParams` 提供的路徑在建構時儲存於完整路由快取中。在請求時，Next.js 也會在首次訪問時快取在建構時未知的路徑。

若要於建構時靜態渲染所有路徑，請提供完整的 `generateStaticParams` 路徑列表：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  return posts.map((post) => ({
    slug: post.slug,
  }))
}
```

若要於建構時靜態渲染部分路徑，並在執行時於首次訪問時渲染其餘路徑，請返回部分路徑列表：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  const posts = await fetch('https://.../posts').then((res) => res.json())

  // Render the first 10 posts at build time
  return posts.slice(0, 10).map((post) => ({
    slug: post.slug,
  }))
}
```

若要於首次訪問時靜態渲染所有路徑，請返回空陣列（建構時不渲染任何路徑）或使用 [`export const dynamic = 'force-static'`](/docs/app/api-reference/file-conventions/route-segment-config#dynamic)：

```jsx filename="app/blog/[slug]/page.js"
export async function generateStaticParams() {
  return []
}
```

> **值得了解**: 你必須從 `generateStaticParams` 返回陣列，即使它是空的。否則，該路徑將被動態渲染。

```jsx filename="app/changelog/[slug]/page.js"
export const dynamic = 'force-static'
```

若要停用請求時的快取，請在路由區塊中添加 `export const dynamicParams = false` 選項。使用此設定選項時，只有 `generateStaticParams` 提供的路徑會被提供，其他路由將產生 404 或匹配（在 [catch-all routes](/docs/app/api-reference/file-conventions/dynamic-routes#catch-all-segments) 的情況下）。

### React `cache` function

React `cache` 函數允許你記憶體化函數的回傳值，允許你多次呼叫同一函數，但只執行一次。

使用 `GET` 或 `HEAD` 方法的 `fetch` 請求會自動進行記憶體化，因此你不需要將它包裝在 React `cache` 中。然而，對於其他 `fetch` 方法，或使用不 inherently 記憶體化請求的資料取得函式庫（如某些資料庫、CMS 或 GraphQL 客戶端），你可以使用 `cache` 來手動記憶體化資料請求。

```ts filename="utils/get-item.ts" switcher
import { cache } from 'react'
import db from '@/lib/db'

export const getItem = cache(async (id: string) => {
  const item = await db.item.findUnique({ id })
  return item
})
```

```js filename="utils/get-item.js" switcher
import { cache } from 'react'
import db from '@/lib/db'

export const getItem = cache(async (id) => {
  const item = await db.item.findUnique({ id })
  return item
})
```
