Next.js 旨在提供優異的開發者體驗。隨著您的應用程序增長，您可能會注意到本地開發期間編譯時間較慢。本指南將幫助您識別並修復常見的編譯期性能問題。

## 本地開發與生產環境
---
使用 `next dev` 的開發過程與 `next build` 和 `next start` 不同。
---
`next dev` 在您打開或導航到應用程序中的路由時編譯該路由。這使您可以啟動開發服務器而無需等待應用程序中的所有路由都完成編譯，這既更快又使用更少的內存。運行生產構建會應用其他優化，如縮小文件大小和創建內容雜湊，這些在本地開發中不需要。
---
## 提升本地開發效能
---
### 1. 檢查電腦的防毒軟體
---
防毒軟體可能會減緩檔案存取速度。
---
請嘗試將您的專案資料夾加入防毒排除清單。雖然這在 Windows 機器上更為常見，但我們建議任何安裝了防毒工具的系統都採用此方法。
---
### 2. 更新 Next.js 並啟用 Turbopack
---
請確保您使用的是最新版本。每個新版本通常都包含效能改進。
---
Turbopack 是整合到 Next.js 中的新打包器，可以改進本地效能。
---
```bash
npm install next@latest
npm run dev --turbopack
```

---
`[深入了解 Turbopack](/blog/turbopack-for-development-stable)`。參閱我們的 `[升級指南](/docs/app/guides/upgrading)` 和 codemods 以獲取更多資訊。
---
### 3. 檢查您的引入方式
---
您引入程式碼的方式會大大影響編譯和打包時間。更多內容請參閱 `[優化套件打包](/docs/app/guides/package-bundling)` 並探索 `[Dependency Cruiser](https://github.com/sverweij/dependency-cruiser)` 或 `[Madge](https://github.com/pahen/madge)` 等工具。
---
#### 圖示庫
---
像 `@material-ui/icons`, `@phosphor-icons/react`, 或 `react-icons` 這樣的庫會引入數千個圖示，即使您只使用少量圖示。請嘗試只引入您需要的圖示：
---
```jsx
//  Instead of this:
import { TriangleIcon } from '@phosphor-icons/react'

// Do this:
import { TriangleIcon } from '@phosphor-icons/react/dist/csr/Triangle'
```

---
您通常可以在所使用的圖示庫的文檔中找到應使用什麼引入模式。此範例遵循 [`@phosphor-icons/react`](https://www.npmjs.com/package/@phosphor-icons/react#import-performance-optimization) 建議。
---
像 `react-icons` 這樣的庫包含許多不同的圖示集。選擇一個集合併堅持使用該集。
---
例如，如果您的應用程序使用 `react-icons` 並導入以下所有：
---
- `pi` (Phosphor 圖示)
- `md` (Material Design 圖示)
- `tb` (Tabler Icons)
- `cg` (Css gg)
---
總計將是數萬個模組，即使您只從每個模組導入一個。
---
#### Barrel 檔案 (Barrel files)
---
“Barrel 檔案 (Barrel files)” 是用於從其他檔案匯出許多項目的檔案。它們可能會減緩構建速度，因為編譯器必須解析它們來檢查模組範圍內的副作用（`import` 是否會導致副作用）。
---
請盡可能嘗試直接從特定檔案引入。[`深入了解 Barrel 檔案 (Barrel files)`](https://vercel.com/blog/how-we-optimized-package-imports-in-next-js) 和 Next.js 內建的優化。
---
#### 優化套件引入
---
Next.js 可以自動優化某些套件的引入。如果您正在使用利用 Barrel 檔案的套件，請將它們加入您的 `next.config.js`：
---
```jsx
module.exports = {
  experimental: {
    optimizePackageImports: ['package-name'],
  },
}
```

---
Turbopack 會自動分析引入並進行優化。它不需要此設定。
---
### 4. 檢查您的 Tailwind CSS 設定
---
如果您使用 Tailwind CSS，請確保設定正確。
---
一個常見的錯誤是您配置 `content` 陣列的方式包含了 `node_modules` 或其他不應該掃描的大型目錄。
---
Tailwind CSS 3.4.8 或更新版本將警告您關於可能減慢構建的設定。
---
1. 在您的 `tailwind.config.js` 中，明確指定要掃描的檔案：

   ```jsx
   module.exports = {
     content: [
       './src/**/*.{js,ts,jsx,tsx}', // Good
       // This might be too broad
       // It will match `packages/**/node_modules` too
       // '../../packages/**/*.{js,ts,jsx,tsx}',
     ],
   }
   ```

2. 避免掃描不必要檔案：

   ```jsx
   module.exports = {
     content: [
       // Better - only scans the 'src' folder
       '../../packages/ui/src/**/*.{js,ts,jsx,tsx}',
     ],
   }
   ```
---
### 5. 檢查自訂 webpack 設定
---
如果您添加了自訂 webpack 設定，它們可能減慢編譯速度。
---
考慮您是否真的需要它們用於本地開發。您可以選擇性地只在生產構建中包含某些工具，或探索遷移到 Turbopack 並使用 [loaders](/docs/app/api-reference/config/next-config-js/turbopack#supported-loaders)。
---
### 6. 優化記憶體使用
---
如果您的應用程序非常大，可能需要更多記憶體。
---
[深入了解優化記憶體使用](/docs/app/guides/memory-usage)。
---
### 7. 伺服器組件 (Server Components) 和數據獲取
---
Server Components 的變更會導致整個頁面在本地重新渲染，以顯示新變更，這包括獲取元件的新數據。
---
實驗性 `serverComponentsHmrCache` 選項允許您在本地開發期間的熱模組替換 (HMR) 重新整理之間緩存 Server Components 中的 `fetch` 回應。這導致更快的回應和減少已計費 API 呼叫的成本。
---
`[深入了解此實驗性選項](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache)`。
---
### 8. 考慮使用本地開發而非 Docker
---
如果您在 Mac 或 Windows 上使用 Docker 進行開發，可能會比在本地運行 Next.js 經歷顯著較慢的效能。
---
Docker 在 Mac 和 Windows 上的檔案系統存取可能會導致熱模組替換 (HMR) 需要數秒甚至數分鐘，而同一應用程序在本地開發時運行快速 HMR。
---
此效能差異歸因於 Docker 如何處理 Linux 環境以外的檔案系統操作。為了獲得最佳開發體驗：
---
- 開發期間使用本地開發 (`npm run dev` 或 `pnpm dev`) 而非 Docker
- 保留 Docker 用於生產部署和測試生產構建
- 如果您必須使用 Docker 進行開發，考慮在 Linux 機器或 VM 上使用 Docker
---
[深入了解 Docker 部署](/docs/app/getting-started/deploying#docker) 以供生產使用。
---
## 查找問題的工具
---
### 詳細數據獲取日誌記錄
---
在您的 `next.config.js` 檔案中使用 `logging.fetches` 選項，以獲得有關開發期間發生什麼事情的更多詳細資訊：
---
```js
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

---
[深入了解數據獲取日誌記錄](/docs/app/api-reference/config/next-config-js/logging)。

## Turbopack 追蹤
---
Turbopack 追蹤是一種工具，可幫助您瞭解本地開發期間應用程序的效能。
它提供有關每個模組編譯耗時及其關係的詳細資訊。
---
1. 請確保您已安裝最新版本的 Next.js。
1. 生成 Turbopack 追蹤檔案：

   ```bash
   NEXT_TURBOPACK_TRACING=1 npm run dev
   ```

1. 在您的應用程序中導航或進行檔案編輯以複現問題。
1. 停止 Next.js 開發服務器。
1. `trace-turbopack` 檔案將可用的 `.next` 資料夾中。
1. 您可以使用 `npx next internal trace [path-to-file]` 解析該檔案：

   ```bash
   npx next internal trace .next/trace-turbopack
   ```

   在 `trace` 不可用的版本中，命令名為 `turbo-trace-server`：

   ```bash
   npx next internal turbo-trace-server .next/trace-turbopack
   ```

1. 一旦追蹤服務器運行，您可以在 https://trace.nextjs.org/ 查看追蹤。
1. 默認情況下追蹤檢視器會彙總時機，若要查看每個個別時間，您可以將檢視器右上角的 "Aggregated in order" 切換為 "Spans in order"。
---
