## 使用方法

实验性支持在 `<head>` 中内联 CSS。启用此标志后，所有通常生成 `<link>` 标签的位置都将改为生成 `<style>` 标签。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    inlineCss: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    inlineCss: true,
  },
}

module.exports = nextConfig
```

## 权衡

### 何时使用行内 CSS

在某些场景下，行内 CSS 可以带来以下益处：



- **首次访问者**: 由于 CSS 文件是渲染阻塞资源，内联可以消除首次访问者经历的首次下载延迟，从而提升页面加载性能。

- **性能指标**: 通过移除针对 CSS 文件的额外网络请求，内联可以显著提高关键指标，例如首次内容绘制 (FCP) 和最大内容绘制 (LCP)。

- **慢速连接**: 对于在网络较慢的用户，每个请求都会增加明显的延迟，内联 CSS 可以通过减少网络往返次数提供显著的性能提升。

- **原子 CSS 包 (例如 Tailwind)**: 对于像 Tailwind CSS 这样的工具优先框架，页面所需的样式大小相对于设计复杂度通常是 O(1) 的。这使得内联成为一个有吸引力的选择，因为当前页面的整套样式都轻量且不会随页面大小增长。内联 Tailwind 样式可确保极小的负载大小，并消除额外网络请求的需求，这可以进一步提升性能。

### 何时不使用行内 CSS

虽然内联 CSS 为性能提供了显著的益处，但在某些场景下它可能不是最佳选择：



- **大型 CSS 包**: 如果您的 CSS 包过大，内联它可能会显著增加 HTML 的大小，导致首次字节时间 (TTFB) 变慢，并且对于慢速连接的用户来说性能可能更差。
- **动态或页面特定 CSS**: 对于具有高度动态样式或使用不同 CSS 集合的页面的应用程序，内联可能会导致冗余和膨胀，因为所有页面的完整 CSS 可能需要反复内联。

- **浏览器缓存**: 在访客频繁返回您网站的情况下，外部 CSS 文件允许浏览器高效地缓存样式，减少后续访问的数据传输。内联 CSS 消除了这一优势。

请仔细权衡这些利弊，并考虑将内联与其他策略结合使用，例如提取关键 CSS 或采用混合方案，以获得最适合您站点需求的最佳结果。

> **小知识**:
>
> 此功能目前是实验性的，并存在一些已知限制：
>
> - CSS 内联是全局应用的，无法在每页基础上进行配置
> - 样式在初次页面加载期间会复制一份 —— 一次在 `<style>` 标签中用于 SSR，一次在 RSC 负载中
> - 当导航到静态渲染的页面时，样式将使用 `<link>` 标签而不是内联 CSS 以避免重复
> - 此功能在开发模式下不可用，仅在生产构建中有效
