## 使用方法

内联 CSS 的实验性支持（在 `<head>` 中）。启用此标志后，所有通常生成 `<link>` 标签的位置将改为生成 `<style>` 标签。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    inlineCss: true,
  },
}

export default nextConfig
```

```js filename="next.config.js" switcher
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    inlineCss: true,
  },
}

module.exports = nextConfig
```

## 权衡 (Trade-Offs)

### 何时使用行内 CSS

内联 CSS 在以下几种情况下很有益：

- **首次访问者**：由于 CSS 文件是渲染阻塞资源，内联消除了首次访问者经历的初始下载延迟，从而提高了页面加载性能。

- **性能指标**：通过去除对 CSS 文件的额外网络请求，内联可以显著改善首次内容绘制 (FCP) 和最大内容绘制 (LCP) 等关键指标。

- **缓慢连接**：对于处于较慢网络的用户，每个请求都会增加相当的延迟，内联 CSS 通过减少网络往返次数可以提供显著的性能提升。

- **Atomic CSS 包 (例如 Tailwind)**：对于 Tailwind CSS 等实用优先框架，页面所需的样式大小相对于设计复杂度通常是 O(1)。这使得内联成为一个有吸引力的选择，因为当前页面的整套样式都很轻量，且不会随页面大小增长。内联 Tailwind 样式可确保最小有效载荷，并消除对额外网络请求的需求，从而进一步增强性能。

### 何时不使用行内 CSS

虽然内联 CSS 为性能提供了显著的益处，但在某些情况下它可能不是最佳选择：

- **大型 CSS 包**：如果您的 CSS 包太大，内联它可能会显著增加 HTML 的大小，导致首字节时间 (TTFB) 变慢，并可能导致缓慢连接的用户性能更差。
- **动态或页面特定 CSS**：对于具有高动态样式或使用不同 CSS 集页面的应用程序，内联可能会导致冗余和膨胀，因为可能需要重复内联所有页面的完整 CSS。

- **浏览器缓存**：对于经常返回您站点的访问者，外部 CSS 文件允许浏览器有效地缓存样式，从而减少后续访问的数据传输。内联 CSS 消除了这一好处。

仔细评估这些权衡，并考虑将内联与其他策略（例如关键 CSS 提取或混合方法）相结合，根据您的网站需求获得最佳结果。

> **值得注意**：
>
> 此功能目前处于实验阶段，且有一些已知的限制：
>
> - CSS 内联是全局应用的，无法针对每个页面进行配置
> - 样式在初始页面加载期间会重复 - 一次用于 SSR 的 `<style>` 标签内，另一次在 RSC 负载中
> - 当导航到静态渲染的页面时，样式将使用 `<link>` 标签而不是内联 CSS 以避免重复
> - 此功能在开发模式下不可用，仅在生产构建中有效
