可观测性 (observability) 对于理解和优化 Next.js 应用的行为及性能至关重要。

{/* md5:838ad41afc23a62cead94bad92ac4270 */}
随着应用程序变得越来越复杂，识别和诊断可能出现的问题也变得愈发困难。通过利用可观测性工具，例如日志记录和指标，开发人员可以深入了解应用程序的行为并找到优化的方向。借助可观测性，开发人员可以在问题演变成重大故障之前主动解决它们，并提供更好的用户体验。因此，强烈建议在您的 Next.js 应用程序中使用可观测性来提升性能、优化资源并增强用户体验。

{/* md5:8aa410fb5091fb31ff083b41c90992ce */}
我们推荐使用 OpenTelemetry 来检测您的应用。这是一种与平台无关的应用检测方法，允许您在不更改代码的情况下更换可观测性提供商。请阅读 [OpenTelemetry 官方文档](https://opentelemetry.io/docs/)，了解更多关于 OpenTelemetry 及其工作原理的信息。

{/* md5:d707c4ebd46f951d63ec1f221839eef9 */}
本文档中使用了 _跨度 (Span)_、_追踪 (Trace)_ 或 _导出器 (Exporter)_ 等术语，所有这些都可以在 [OpenTelemetry 可观测性入门](https://opentelemetry.io/docs/concepts/observability-primer/) 中找到。

### 手动 OpenTelemetry 配置

{/* md5:ee6f91fa92dd3186e8e86dfeb07e5e8d */}
`@vercel/otel` 包提供了许多配置选项，应该能满足大多数常见用例。但如果它不能满足您的需求，您可以手动配置 OpenTelemetry。

{/* md5:2870f63af8e42e7f1208ca3cb17e6f1f */}
首先，您需要安装 OpenTelemetry 软件包：

```bash filename="Terminal"
npm install @opentelemetry/sdk-node @opentelemetry/resources @opentelemetry/semantic-conventions @opentelemetry/sdk-trace-node @opentelemetry/exporter-trace-otlp-http
```

{/* md5:8cae28447f01af3255f9e0bdfdaaac3d */}
现在您可以在 `instrumentation.ts` 中初始化 `NodeSDK`。与 `@vercel/otel` 不同，`NodeSDK` 与 edge runtime 不兼容，因此您需要确保仅在 `process.env.NEXT_RUNTIME === 'nodejs'` 时才导入它们。我们建议创建一个新文件 `instrumentation.node.ts`，您仅在使用 Node.js 时才条件性地导入它：

```ts filename="instrumentation.ts" switcher
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    await import('./instrumentation.node.ts')
  }
}
```

```js filename="instrumentation.js" switcher
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    await import('./instrumentation.node.js')
  }
}
```

```ts filename="instrumentation.node.ts" switcher
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { Resource } from '@opentelemetry/resources'
import { NodeSDK } from '@opentelemetry/sdk-node'
import { SimpleSpanProcessor } from '@opentelemetry/sdk-trace-node'
import { ATTR_SERVICE_NAME } from '@opentelemetry/semantic-conventions'

const sdk = new NodeSDK({
  resource: new Resource({
    [ATTR_SERVICE_NAME]: 'next-app',
  }),
  spanProcessor: new SimpleSpanProcessor(new OTLPTraceExporter()),
})
sdk.start()
```

## 自定义跨度

{/* md5:5633f0f9f925e728a786cce9d82f6e66 */}
您可以使用 [OpenTelemetry API](https://opentelemetry.io/docs/instrumentation/js/instrumentation) 添加自定义跨度。

```bash filename="Terminal"
npm install @opentelemetry/api
```

{/* md5:db83f25662529af7c9c57647dccb4cac */}
以下示例演示了一个获取 GitHub star 数的函数，并添加了一个自定义的 `fetchGithubStars` 跨度来跟踪 fetch 请求的结果：

```ts
import { trace } from '@opentelemetry/api'

export async function fetchGithubStars() {
  return await trace
    .getTracer('nextjs-example')
    .startActiveSpan('fetchGithubStars', async (span) => {
      try {
        return await getValue()
      } finally {
        span.end()
      }
    })
}
```

`register` 函数将在您的代码在新环境中运行之前执行。您可以开始创建新的跨度，它们应该会被正确添加到导出的追踪中。

## Next.js 中的默认跨度

{/* md5:a1b0c23b358f64e581a5cce1a8e9ec04 */}
Next.js 会自动检测多个跨度，为您提供有关应用程序性能的有用见解。

{/* md5:3aacc49a519c2c0028e38bd09811d5b9 */}
跨度上的属性遵循 [OpenTelemetry 语义约定](https://opentelemetry.io/docs/reference/specification/trace/semantic_conventions/)。我们还在 `next` 命名空间下添加了一些自定义属性：

{/* md5:febcf2db8ebbb0039995c9369dfd8155 */}
- `next.span_name` - 跨度名称的副本
- `next.span_type` - 每个跨度类型都有唯一的标识符
- `next.route` - 请求的路由模式（例如 `/[param]/user`）。
- `next.rsc` (true/false) - 请求是否为 RSC 请求，例如预取。
- `next.page`
  - 这是 app router 使用的一个内部值。
  - 您可以将其视为指向特殊文件的路由（如 `page.ts`、`layout.ts`、`loading.ts` 等）
  - 只有与 `next.route` 配对时，它才能用作唯一标识符，因为 `/layout` 可用于识别 `/(groupA)/layout.ts` 和 `/(groupB)/layout.ts`
