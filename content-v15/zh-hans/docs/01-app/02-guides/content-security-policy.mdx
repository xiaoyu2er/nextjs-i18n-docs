## 随机数

{/* md5:51b75da0c6fd3df52e8c59a3d88b75a2 */}
[随机数 (nonce)](https://developer.mozilla.org/docs/Web/HTML/Global_attributes/nonce) 是一个为单次使用而创建的唯一、随机字符串。它与 CSP 结合使用，以选择性地允许某些内联脚本或样式执行，从而绕过严格的 CSP 指令。

{/* md5:6951d5ca989664094a16889660dac48a */}
### 为什么要使用随机数？

{/* md5:382519edb2d3c7ae413a9e5e53fedf7d */}
CSP 可以阻止内联和外部脚本，以防止攻击。随机数允许您安全地允许特定的脚本运行——前提是该脚本包含匹配的随机数值。

{/* md5:ae0aa3c64651ea088a9a4e07801d905f */}
如果攻击者试图在你的页面中加载脚本，他们需要猜测随机数值。这就是为什么随机数对于每个请求都必须是不可预测且唯一的。

{/* md5:092c979d32b722e72fca82bd80162c42 */}
### 使用中间件添加随机数

{/* md5:1c648a8efa16c597ec64f494e02c896b */}
[中间件](/docs/app/api-reference/file-conventions/middleware) 允许你在页面渲染之前添加头信息并生成随机数。

{/* md5:06aba33981bde0a63330d9c4f331bdaa */}
每次查看页面时，都应生成一个新的随机数。这意味着你**必须使用 [动态渲染](/docs/app/getting-started/partial-prerendering#dynamic-rendering) 来添加随机数**。

{/* md5:506c2c0c7f5b70af3df68c45c46f45a7 */}
例如：

```ts filename="middleware.ts" switcher
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  const nonce = Buffer.from(crypto.randomUUID()).toString('base64')
  const cspHeader = `
    default-src 'self';
    script-src 'self' 'nonce-${nonce}' 'strict-dynamic';
    style-src 'self' 'nonce-${nonce}';
    img-src 'self' blob: data:;
    font-src 'self';
    object-src 'none';
    base-uri 'self';
    form-action 'self';
    frame-ancestors 'none';
    upgrade-insecure-requests;
`
  // Replace newline characters and spaces
  const contentSecurityPolicyHeaderValue = cspHeader
    .replace(/\s{2,}/g, ' ')
    .trim()

  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-nonce', nonce)

  requestHeaders.set(
    'Content-Security-Policy',
    contentSecurityPolicyHeaderValue
  )

  const response = NextResponse.next({
    request: {
      headers: requestHeaders,
    },
  })
  response.headers.set(
    'Content-Security-Policy',
    contentSecurityPolicyHeaderValue
  )

  return response
}
```

默认情况下，中间件在所有请求上运行。你可以使用 [`matcher`](/docs/app/api-reference/file-conventions/middleware#matcher) 过滤中间件以在特定路径上运行。

{/* md5:04b850d935a2061af1af8c83e6929c01 */}
我们建议忽略匹配预取请求（来自 `next/link`）和不需要 CSP 头的静态资源。

```ts filename="middleware.ts" switcher
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico (favicon file)
     */
    {
      source: '/((?!api|_next/static|_next/image|favicon.ico).*)',
      missing: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },
  ],
}
```

### 随机数在 Next.js 中如何工作

{/* md5:2a8532aa7e77ff2c03bee72de4c0d81e */}
要使用随机数，你的页面必须是**动态渲染**的。这是因为 Next.js 在**服务器端渲染**期间应用随机数，基于请求中存在的 CSP 头。静态页面是在构建时生成的，当时不存在请求或响应头——因此无法注入随机数。

{/* md5:14bcc8e4f4404f1d92f4556b8b4783d7 */}
以下是动态渲染页面中随机数支持的工作原理：

{/* md5:74c5112b906a9cd943d82e513d6620ca */}
1. **中间件生成随机数**：你的中间件为请求创建唯一的随机数，将其添加到你的 `Content-Security-Policy` 头中，并将其设置在自定义的 `x-nonce` 头中。
2. **Next.js 提取随机数**：在渲染期间，Next.js 解析 `Content-Security-Policy` 头并使用 `'nonce-{value}'` 模式提取随机数。
3. **随机数自动应用**：Next.js 将随机数附加到：
   - 框架脚本（React、Next.js 运行时）
   - 页面特定的 JavaScript 包
   - Next.js 生成的内联样式和脚本
   - 使用 `nonce` prop 的任何 `<Script>` 组件

{/* md5:b2ec267538d0bfa92b60a693625782d8 */}
由于这种自动行为，你不需要手动为每个标签添加随机数。

{/* md5:55ac3c379a3774d710d3008992400812 */}

```tsx filename="pages/_document.tsx" switcher
import Document, {
  Html,
  Head,
  Main,
  NextScript,
  DocumentContext,
  DocumentInitialProps,
} from 'next/document'

interface ExtendedDocumentProps extends DocumentInitialProps {
  nonce?: string
}

class MyDocument extends Document<ExtendedDocumentProps> {
  static async getInitialProps(
    ctx: DocumentContext
  ): Promise<ExtendedDocumentProps> {
    const initialProps = await Document.getInitialProps(ctx)

    return {
      ...initialProps,
      nonce,
    }
  }

  render() {
    const { nonce } = this.props

    return (
      <Html lang="en">
        <Head nonce={nonce} />
        <body>
          <Main />
          <NextScript nonce={nonce} />
        </body>
      </Html>
    )
  }
}

export default MyDocument
```

## Static vs Dynamic Rendering with CSP

{/* md5:5613f7ce10432829a5608ec8659e17e8 */}
Using nonces has important implications for how your Next.js application renders:

{/* md5:f0bc83f747b04a7610b6663ca0acaf1d */}
### Dynamic Rendering Requirement

{/* md5:977a7576e51e6ef9df334530a1480297 */}
When you use nonces in your CSP, **all pages must be dynamically rendered**. This means:

{/* md5:786b55f4b44a04f15bf6f3e7749c57b8 */}
- Pages will build successfully but may encounter runtime errors if not properly configured for dynamic rendering
- Each request generates a fresh page with a new nonce
- Static optimization and Incremental Static Regeneration (ISR) are disabled
- Pages cannot be cached by CDNs without additional configuration
- **Partial Prerendering (PPR) is incompatible** with nonce-based CSP since static shell scripts won't have access to the nonce

{/* md5:b48aff143c821805a7bdac9935e0abf3 */}
### Performance Implications

{/* md5:f45f93fb02d6b8cb635ad725f5e78498 */}
The shift from static to dynamic rendering affects performance:

{/* md5:ae0776d5fc051caded3fa9399324bd71 */}
- **Slower initial page loads**: Pages must be generated on each request
- **Increased server load**: Every request requires server-side rendering
- **No CDN caching**: Dynamic pages cannot be cached at the edge by default
- **Higher hosting costs**: More server resources needed for dynamic rendering

{/* md5:50e9d33948c44363df84f9d53a350b37 */}
### When to use nonces

{/* md5:ed6f2dd0b9d6f932b472eea9fed3cb7a */}
Consider nonces when:

{/* md5:5f7b765b3fe28fea0bba2dd55e095ee0 */}
- You have strict security requirements that prohibit `'unsafe-inline'`
- Your application handles sensitive data
- You need to allow specific inline scripts while blocking others
- Compliance requirements mandate strict CSP

{/* md5:b6b0f4c1f276276b7428090dc73cb8a8 */}
## Without Nonces

{/* md5:5e41d00583e937c38ce712a5f63e1287 */}
对于不需要随机数 (nonce) 的应用，你可以直接在 [`next.config.js`](/docs/app/api-reference/config/next-config-js) 文件中设置 CSP 头：

```js filename="next.config.js"
    object-src 'none';
    base-uri 'self';
    form-action 'self';

  },
```

## Subresource Integrity (Experimental)

{/* md5:34e38e91637eabc1751291f24bbdea18 */}
As an alternative to nonces, Next.js offers experimental support for hash-based CSP using Subresource Integrity (SRI). This approach allows you to maintain static generation while still having a strict CSP.

{/* md5:662abf733a8070bab47b6573e804b307 */}
> **Good to know**: This feature is experimental and only available with webpack bundler in App Router applications.

{/* md5:932910153aedad43b8b2ad8e1c39a41f */}
### How SRI works

{/* md5:89e54c30d34c9e74251e739baae5de15 */}
Instead of using nonces, SRI generates cryptographic hashes of your JavaScript files at build time. These hashes are added as `integrity` attributes to script tags, allowing browsers to verify that files haven't been modified during transit.

{/* md5:5e010d76f8fcc5b44cd94d43d54d5827 */}
### Enabling SRI

{/* md5:b0269b977b250b5e25521a8863c64326 */}
Add the experimental SRI configuration to your `next.config.js`:

```js filename="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    sri: {
      algorithm: 'sha256', // or 'sha384' or 'sha512'
    },
  },
}

module.exports = nextConfig
```

{/* md5:7442bf716aeb9ddafa59283e582aba5d */}
### CSP configuration with SRI

{/* md5:790969067f552fcfd55046b2c6242c94 */}
When SRI is enabled, you can continue using your existing CSP policies. SRI works independently by adding `integrity` attributes to your assets:

{/* md5:4f470b30ef9a0e5904a6e897895efcdc */}
> **Good to know**: For dynamic rendering scenarios, you can still generate nonces with middleware if needed, combining both SRI integrity attributes and nonce-based CSP approaches.

```js filename="next.config.js"
const cspHeader = `
    default-src 'self';
    script-src 'self';
    style-src 'self';
    img-src 'self' blob: data:;
    font-src 'self';
    object-src 'none';
    base-uri 'self';
    form-action 'self';
    frame-ancestors 'none';
    upgrade-insecure-requests;
`

module.exports = {
  experimental: {
    sri: {
      algorithm: 'sha256',
    },
  },
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'Content-Security-Policy',
            value: cspHeader.replace(/\n/g, ''),
          },
        ],
      },
    ]
  },
}
```

{/* md5:16d6ed9337bff724ae0f7887ec108eb0 */}
### Benefits of SRI over nonces

{/* md5:8cbdbc7a3acbe1bf2a347e9661f9fd67 */}
- **Static generation**: Pages can be statically generated and cached
- **CDN compatibility**: Static pages work with CDN caching
- **Better performance**: No server-side rendering required for each request
- **Build-time security**: Hashes are generated at build time, ensuring integrity

{/* md5:e7e2db927ddc312e3dd34bcc0947d4a4 */}
### Limitations of SRI

{/* md5:6b926a409a0cb4f0eff73db3b729a838 */}
- **Experimental**: Feature may change or be removed
- **Webpack only**: Not available with Turbopack
- **App Router only**: Not supported in Pages Router
- **Build-time only**: Cannot handle dynamically generated scripts

## Development vs Production Considerations

{/* md5:57bb00f43118e74d7c588137f3e8938d */}

{/* md5:cb8ed2910cfdbc160d78d47427d046ce */}
### Development Environment

{/* md5:6f89e6f81bb8f297c04b15ebb13cdbed */}
In development, you will need to enable `'unsafe-eval'` to support APIs that provide additional debugging information:

```ts filename="middleware.ts" switcher
export function middleware(request: NextRequest) {

    object-src 'none';
    base-uri 'self';
    form-action 'self';
    frame-ancestors 'none';
    upgrade-insecure-requests;
`

  // Rest of middleware implementation
}
```

2. **Dynamic imports**: Ensure dynamic imports are allowed in your script-src policy
