{/* md5:98cf071dce7fddc4af7a2fc496fbc2e1 */}
随着应用程序变得更大、功能更丰富，它们在本地开发或创建生产构建时可能会占用更多资源。

{/* md5:d786a8ba412d04d446674c5f01e9c5fa */}
让我们探讨一些策略和技巧，用于在 Next.js 中优化内存并解决常见的内存问题。

{/* md5:13a3d58bd29dd1742b18257a96b131aa */}
## 减少依赖数量

{/* md5:a93fcce081ab785971040d87ca356569 */}
拥有大量依赖的应用程序会使用更多内存。

{/* md5:66641de8f459768bcc47faf49e48d67f */}
[包分析工具](/docs/app/guides/package-bundling) 可以帮助您调查应用程序中可能被移除以改善性能和内存使用的大型依赖。

{/* md5:183bf982158ef628e005ca27b63de58a */}
## 尝试 `experimental.webpackMemoryOptimizations`

{/* md5:79cdba58b3e8a1e27ade53a203f1d870 */}
从 `v15.0.0` 开始，您可以在 `next.config.js` 文件中添加 `experimental.webpackMemoryOptimizations: true`，以更改 Webpack 中的行为，这会降低最大内存使用量，但可能会使编译时间略微增加。

{/* md5:ff1622951eaf7d429cf7993a918e30e3 */}
> **须知**：此功能目前是实验性的，旨在先在更多项目上进行测试，但被认为是低风险的。

{/* md5:8752b34654e9daa1d2990151e250288f */}
## 使用 `--experimental-debug-memory-usage` 运行 `next build`

{/* md5:de85b0082350cb01056554da1726f1e1 */}
从 `14.2.0` 开始，您可以运行 `next build --experimental-debug-memory-usage`，以在一种模式下运行构建，在此模式下，Next.js 将在构建过程中持续打印内存使用信息，例如堆使用情况和垃圾回收统计信息。当内存使用量接近配置的限制时，也会自动拍摄堆快照。

{/* md5:61ab45bbd3cf403798a5b891cc65bdfe */}
> **须知**：此功能与 Webpack 构建工作器选项不兼容，该选项在您没有自定义 webpack 配置时会自动启用。

{/* md5:44a073d7c53abc1db8c96d3b828463d2 */}
## 记录堆配置文件

{/* md5:d9f3e39128685551dd8fb520b71ed700 */}
为了查找内存问题，您可以从 Node.js 记录一个堆配置文件，并在 Chrome 开发者工具中加载它，以识别潜在的内存泄漏源。

{/* md5:8b6ccd09312c0b493af8bfc7024f5365 */}
在您的终端中，在启动 Next.js 构建时，将 `--heap-prof` 标志传递给 Node.js：

```sh
node --heap-prof node_modules/next/dist/bin/next build
```

{/* md5:e6331560b816171be1e3f54aae03476b */}
在构建结束时，Node.js 将创建一个 `.heapprofile` 文件。

{/* md5:54299dee42896765900413ff615aee04 */}
在 Chrome 开发者工具中，您可以打开 Memory 标签页，然后点击 "Load Profile" 按钮来可视化该文件。

{/* md5:09fafceffec5fea7fa00649d56d6c545 */}
## 分析堆快照

{/* md5:4de185afdc5b80672fa750219308a883 */}
您可以使用检查器工具来分析应用程序的内存使用情况。

{/* md5:a118473637ef8c7c14b7d5901b2ad374 */}
在运行 `next build` 或 `next dev` 命令时，在命令开头添加 `NODE_OPTIONS=--inspect`。这将在默认端口暴露检查器代理。
如果您希望在任何用户代码开始之前中断，可以改为传递 `--inspect-brk`。当进程运行时，您可以使用诸如 Chrome 开发者工具之类的工具连接到调试端口，以记录和分析堆的快照，查看哪些内存被保留。

{/* md5:7acd934c40b219ecb621cabc5b89a554 */}
从 `14.2.0` 开始，您还可以使用 `--experimental-debug-memory-usage` 标志运行 `next build`，以便更轻松地拍摄堆快照。

{/* md5:867a267aa7072faf08681723acda82b9 */}
在此模式下运行时，您可以随时向进程发送 `SIGUSR2` 信号，进程将拍摄堆快照。

{/* md5:a6d96ea0d2118ce6c3da16cadf80bb9a */}
堆快照将保存到 Next.js 应用程序的项目根目录，并且可以加载到任何堆分析器（如 Chrome 开发者工具）中，以查看保留了哪些内存。此模式目前与 Webpack 构建工作器不兼容。

{/* md5:925eab04d65e4cf783ef8d6fadb3b368 */}
有关更多信息，请参阅 [如何记录和分析堆快照](https://developer.chrome.com/docs/devtools/memory-problems/heap-snapshots)。

{/* md5:6a6b9603b1e577c95e07c916dcc0936b */}
## Webpack 构建工作器

{/* md5:5274eb9ef0f15933e9c14959c0335bba */}
Webpack 构建工作器允许您在单独的 Node.js 工作器中运行 Webpack 编译，这将减少构建期间应用程序的内存使用量。

{/* md5:acb30ad9931c1f9890fd7a0b53c2cd13 */}
如果您的应用程序没有自定义的 Webpack 配置，则从 `v14.1.0` 开始，此选项默认启用。

{/* md5:5bbf0151d6491ef06bbceb1beac168af */}
如果您使用的是旧版 Next.js 或您有自定义的 Webpack 配置，您可以通过在 `next.config.js` 中设置 `experimental.webpackBuildWorker: true` 来启用此选项。

{/* md5:ea792c9bbe14d545a7f310b88ea245cf */}
> **须知**：此功能可能与所有自定义 Webpack 插件不兼容。

{/* md5:377bcb8c046d46f3396b2d7b2fbdccbc */}
## 禁用 Webpack 缓存

{/* md5:1981595c9ba6840a9be1b6fa65a1112e */}
[Webpack 缓存](https://webpack.js.org/configuration/cache/) 将生成的 Webpack 模块保存在内存和/或磁盘上，以提高构建速度。这有助于提升性能，但它也会增加应用程序的内存使用量以存储缓存数据。

{/* md5:10262aab3f8cb3d0f4e6bab996639a15 */}
您可以通过向应用程序添加 [自定义 Webpack 配置](/docs/app/api-reference/config/next-config-js/webpack) 来禁用此行为：

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  webpack: (
    config,
    { buildId, dev, isServer, defaultLoaders, nextRuntime, webpack }
  ) => {
    if (config.cache && !dev) {
      config.cache = Object.freeze({
        type: 'memory',
      })
    }
    // Important: return the modified config
    return config
  },
}

export default nextConfig
```

{/* md5:8739e01d672f71666ca6ce92d294666a */}
## 禁用静态分析

{/* md5:507e02a89c37ce43c859072c4b3f371c */}
类型检查和代码检查可能需要大量内存，尤其是在大型项目中。然而，大多数项目都有一个专用的 CI 运行器来处理这些任务。当构建在 "Linting and checking validity of types"（代码检查和类型有效性检查）步骤出现内存不足问题时，您可以在构建期间禁用这些任务：

```js filename="next.config.mjs"
/** @type {import('next').NextConfig} */
const nextConfig = {
  eslint: {
    // Warning: This allows production builds to successfully complete even if
    // your project has ESLint errors.
    ignoreDuringBuilds: true,
  },
  typescript: {
    // !! WARN !!
    // Dangerously allow production builds to successfully complete even if
    // your project has type errors.
    // !! WARN !!
    ignoreBuildErrors: true,
  },
}

export default nextConfig
```

{/* md5:e84423947eae03a58324f763dd444d2c */}
- [忽略 TypeScript 错误](/docs/app/api-reference/config/typescript#disabling-typescript-errors-in-production)
- [Next.js 配置中的 ESLint](/docs/pages/api-reference/config/next-config-js/eslint)

{/* md5:f5c8380c3f80fef8ce400ab00cca4b4c */}
请记住，这可能会由于类型错误或代码检查问题而产生有缺陷的部署。
我们强烈建议仅在静态分析完成后才将构建提升到生产环境。
如果您部署到 Vercel，可以查看 [关于阶段性部署的指南](https://vercel.com/docs/deployments/managing-deployments#staging-and-promoting-a-production-deployment)，以了解如何在自定义任务成功完成后将构建提升到生产环境。

{/* md5:5fc92c4c62d17ab227eb6b165a7ad9a5 */}
## 禁用源映射

{/* md5:0a05b0731d286d50cf8a75ecb5461c3a */}
生成源映射会在构建过程中消耗额外的内存。

{/* md5:081da78f7e2c61c4c6fb4c52b61611c7 */}
您可以通过在 Next.js 配置中添加 `productionBrowserSourceMaps: false` 和 `experimental.serverSourceMaps: false` 来禁用源映射生成。

{/* md5:58e925543f4c4f7492fad5aba9c9de37 */}
> **须知**：某些插件可能会打开源映射，并且可能需要自定义配置来禁用。

{/* md5:a88aabe4b46fc3c0a83f41e6790ce801 */}
## Edge 内存问题

{/* md5:6269dbc9f6c4fc7c4b0303bbcdf3676f */}
Next.js `v14.1.3` 修复了使用 Edge 运行时的一个内存问题。请更新到此版本（或更高版本），看看它是否能解决您的问题。

{/* md5:0b9c5fbb3cd6488cc65197b359206939 */}
## 预加载入口

{/* md5:b7033bbf43b395a7f78efda71fe645b4 */}
当 Next.js 服务器启动时，它会将每个页面的 JavaScript 模块预加载到内存中，而不是在请求时加载。

{/* md5:4e114f14df48572ecfe3dcf043fbc531 */}
这种优化可以带来更快的响应时间，代价是初始内存占用更大。

{/* md5:728d112acb18964ffb7b50ee2af75734 */}
要禁用此优化，请将 `experimental.preloadEntriesOnStart` 标志设置为 `false`。

```ts filename="next.config.ts" switcher
import type { NextConfig } from 'next'

const config: NextConfig = {
  experimental: {
    preloadEntriesOnStart: false,
  },
}

export default config
```

```js filename="next.config.mjs" switcher
/** @type {import('next').NextConfig} */
const config = {
  experimental: {
    preloadEntriesOnStart: false,
  },
}

export default config
```

{/* md5:5b72673bf40182928f33dbedf07b6296 */}
Next.js 不会卸载这些 JavaScript 模块，这意味着即使禁用此优化，如果所有页面最终都被请求，您的 Next.js 服务器的内存占用最终也会相同。
