如果页面使用**静态生成**，页面 HTML 将在**构建时**生成。这意味着在生产环境中，当你运行 `next build` 时就会生成页面 HTML。该 HTML 随后会在每次请求时被重复使用，并且可以被 CDN 缓存。

在 Next.js 中，你可以**带或不带数据**静态生成页面。让我们分别看看这两种情况。

### 不带数据的静态生成

默认情况下，Next.js 使用静态生成 (Static Generation) 预渲染页面，无需获取数据。示例如下：

```jsx
function About() {
  return <div>About</div>
}

export default About
```

注意，此页面无需获取任何外部数据即可进行预渲染。在这种情况下，Next.js 在构建时为每个页面生成一个单个 HTML 文件。

### 带数据的静态生成

有些页面需要获取外部数据进行预渲染。有两种情况，可能适用其中一种或两种。在每种情况下，你可以使用 Next.js 提供的这些函数：

1. 你的页面**内容**依赖于外部数据：使用 `getStaticProps`。
2. 你的页面**路径**依赖于外部数据：使用 `getStaticPaths`（通常与 `getStaticProps` 一起使用）。

#### 情况 1：页面内容依赖于外部数据

**示例**：你的博客页面可能需要从 CMS（内容管理系统）获取博客文章列表。

```jsx
// TODO: Need to fetch `posts` (by calling some API endpoint)
//       before this page can be pre-rendered.
export default function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li>{post.title}</li>
      ))}
    </ul>
  )
}
```

为了在预渲染时获取此数据，Next.js 允许你从同一个文件 `export` 一个名为 `getStaticProps` 的 `async` 函数。该函数在构建时被调用，并允许你在预渲染时将获取的数据传递给页面的 `props`。

```jsx
export default function Blog({ posts }) {
  // Render posts...
}

// This function gets called at build time
export async function getStaticProps() {
  // Call an external API endpoint to get posts
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  // By returning { props: { posts } }, the Blog component
  // will receive `posts` as a prop at build time
  return {
    props: {
      posts,
    },
  }
}
```

要了解更多关于 `getStaticProps` 的工作原理，请查看[数据获取文档](/docs/pages/building-your-application/data-fetching/get-static-props)。

#### 情况 2：页面路径依赖于外部数据

Next.js 允许你创建具有**动态路由**的页面。例如，你可以创建一个名为 `pages/posts/[id].js` 的文件来显示基于 `id` 的单个博客文章。这将允许你在访问 `posts/1` 时显示 `id: 1` 的博客文章。

> 要了解更多关于动态路由的信息，请查看[动态路由文档](/docs/pages/building-your-application/routing/dynamic-routes)。

然而，你在构建时想要预渲染的哪个 `id` 可能取决于外部数据。

**示例**：假设你向数据库中只添加了一篇博客文章（`id: 1`）。在这种情况下，你只想在构建时预渲染 `posts/1`。

之后，你可能会添加第二篇文章，`id: 2`。那么你也想预渲染 `posts/2`。

所以，你要预渲染的页面 **路径** 依赖于外部数据。为了解决这个问题，Next.js 允许你从动态页面（此处为 `pages/posts/[id].js`）`export` 一个名为 `getStaticPaths` 的 `async` 函数。该函数在构建时被调用，并让你指定要预渲染的路径。

```jsx
// This function gets called at build time
export async function getStaticPaths() {
  // Call an external API endpoint to get posts
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  // Get the paths we want to pre-render based on posts
  const paths = posts.map((post) => ({
    params: { id: post.id },
  }))

  // We'll pre-render only these paths at build time.
  // { fallback: false } means other routes should 404.
  return { paths, fallback: false }
}
```

另外，在 `pages/posts/[id].js` 文件中，你还需要 `export` `getStaticProps`，以便你可以获取关于该 `id` 的文章数据并使用它来预渲染页面：

```jsx
export default function Post({ post }) {
  // Render post...
}

export async function getStaticPaths() {
  // ...
}

// This also gets called at build time
export async function getStaticProps({ params }) {
  // params contains the post `id`.
  // If the route is like /posts/1, then params.id is 1
  const res = await fetch(`https://.../posts/${params.id}`)
  const post = await res.json()

  // Pass post data to the page via props
  return { props: { post } }
}
```

要了解更多关于 `getStaticPaths` 的工作原理，请查看[数据获取文档](/docs/pages/building-your-application/data-fetching/get-static-paths)。

### 何时应该使用静态生成？

我们建议尽可能使用**静态生成**（带或不带数据），因为你的页面可以构建一次并由 CDN 提供服务，这比让服务器在每次请求时渲染页面要快得多。

你可以将静态生成用于多种类型的页面，包括：

- 营销页面
- 博客文章和作品集
- 电子商务产品列表
- 帮助和文档

你应该问自己：“我可以在用户请求 **之前** 预渲染此页面吗？”如果答案是肯定的，那么你应该选择静态生成 (Static Generation)。

另一方面，如果你无法在用户请求 **之前** 预渲染页面，那么静态生成 (Static Generation) **不是**个好主意。也许你的页面显示频繁更新的数据，并且页面内容在每次请求时都会发生变化。

在这种情况下，你可以采取以下措施之一：

- 使用带有 **客户端数据获取 (Client-side data fetching)** 的静态生成：你可以跳过页面某些部分的预渲染，然后使用客户端 JavaScript 来填充它们。要了解有关此方法的更多信息，请查看 [数据获取文档](/docs/pages/building-your-application/data-fetching/client-side)。
- 使用 **服务器端渲染 (Server-Side Rendering)**：Next.js 在每次请求时预渲染页面。这会比较慢，因为页面无法被 CDN 缓存，但预渲染的页面将始终保持最新。我们将在下面讨论这种方法。
